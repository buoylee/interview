# 04 故障注入

## 目标

故障注入要放在最容易产生事实分叉的位置。验证实验室不按工具命名故障，而按事实边界命名故障：本地事务、Outbox、broker、消费者、Activity、渠道、供应商、对账和人工修复。

本章的主语仍然是 History 和 Fact。一次故障注入不是证明系统能抗住所有生产故障，而是在一个明确注入点上制造可复现异常，检查故障前后留下的 Fact 是否仍能被 Invariant、状态机 oracle、资金 oracle 和外部事实 oracle 解释。

## 注入点矩阵

| 注入点 | 故障前事实 | 允许恢复动作 | 禁止重复动作 | 合格终态 |
| --- | --- | --- | --- | --- |
| 本地事务提交前失败 | 没有业务 Fact、没有 Outbox 事件、没有账本分录 | 用同一 request id 重试 Command，或返回未创建 | 发布不存在的事件，写入账本，调用外部渠道 | 未创建，或幂等重试后成功创建 |
| 本地事务提交后、Outbox sent 前失败 | 业务 Fact 和 Outbox 事件已在同一事务提交 | publisher 扫描 Outbox 并重投事件 | 重做扣款、出票、退款或重复创建业务 Fact | 事件 eventually sent，下游 Fact 可继续生成 |
| broker 暂停或重复投递 | Outbox 事件存在，可能已发送但未稳定消费 | 延迟投递、重复投递、消费者幂等处理 | 消费者产生第二次业务效果，或用 broker offset 代替处理记录 | 消费者处理记录存在且业务效果最多一次 |
| 消费者外部副作用后、处理记录前失败 | event id 已收到，外部副作用可能已成功，消费者处理记录缺失 | 用 event id 和业务 request id 查询、补记或幂等重试 | 第二次不可关联的扣款、退款、出票、预订或记账 | 同一业务效果被记录，重复消息不再产生副作用 |
| Activity completion 未写入 workflow history | Activity 已调用外部系统，外部副作用可能已成功 | Activity 用同一 request id 重试、查询或等待回调 | 重复扣款、重复退款、重复供应商订单、用 workflow history 直接当业务 Fact | 外部事实、领域 Fact 和 workflow history 能互相解释 |
| 渠道响应超时 | 渠道请求已发出，本地没有确定成功或失败 | 标记 unknown、查询渠道、等待回调、进入对账任务 | 本地直接失败后再次发起不可关联扣款或退款 | unknown、已确认成功、已确认失败、长期对账或人工挂起 |
| 供应商成功回调迟到 | 供应商请求已发出，可能已成功，本地状态可能已取消或 unknown | 状态机接收迟到成功、生成差错、转人工审批 | 用迟到失败覆盖成功，删除供应商订单 Fact，重复下单 | 成功、补偿中、差错记录、人工挂起或可审计失败 |
| 对账任务中断后重跑 | 部分匹配、差错或修复命令可能已生成 | 按批次、渠道流水和业务单号幂等重跑 | 重复生成修复命令，直接改领域 Fact，关闭未复核差错 | 差错稳定分类，修复命令唯一，复核结果可追踪 |
| 人工审批后修复命令重复 | 审批 Fact、证据快照和修复命令可能已存在 | 幂等返回同一修复结果，或追加复核 Fact | 重复冲正、重复补分录、绕过 maker-checker 改状态 | 修复 Fact 可复核，账本和外部事实可解释 |

## 注入用例规则

每个故障注入用例都必须写清：

1. 故障发生前已经提交了哪些 Fact，包括领域 Fact、Outbox 事件、消费者处理记录、渠道流水、供应商订单、账本分录、对账差错和人工审批。
2. 故障发生后系统能安全重试哪些动作，例如重投事件、查询外部结果、补记处理记录、重跑对账任务或重复提交同一个人工修复 request id。
3. 哪些外部动作或资金动作绝对不能重复，例如 capture、refund、出票、供应商预订、账户入账、冲正和补分录。
4. 终态是成功、失败、unknown、长期对账终态还是人工挂起终态。
5. 由哪个 oracle 判断合格，并指出失败时破坏的是状态机、幂等、账本、外部事实、对账还是人工修复边界。

通过标准必须绑定 Fact，而不是绑定工具输出。进程被 kill、容器重启、broker 暂停或 workflow task 重试只是触发方式；真正要验证的是故障前 Fact、恢复动作、禁止重复动作和最终 Fact 集合。

## 分场景注入规则

### 本地事务

本地事务边界验证的是“业务 Fact 和传播 Fact 是否同生共死”。如果事务提交前失败，系统不能留下半个业务事实；如果事务提交后失败，Outbox 必须能恢复传播。

- 注入位置：写领域 Fact 前、写账本分录前、写 Outbox 前、事务提交后但进程返回前。
- 允许恢复：同一 request id 重试 Command，或由唯一约束返回已存在业务结果。
- 禁止行为：事务未提交却发布事件，或事务已提交后重新执行扣款、退款、出票和记账。
- oracle 检查：业务 Fact、账本 Fact 和 Outbox Fact 必须能通过 business id、request id 或 event id 互相关联；需要账本分录的业务效果必须有平衡且可追溯的账本 Fact；需要传播的本地 Fact 必须有对应 Outbox Fact。

### Outbox

Outbox 边界验证的是“本地事实提交后，传播路径是否可恢复”。Publisher 可以重发事件，但不能重新执行业务命令。

- 注入位置：Outbox 写入后 publisher 未扫描、发送成功但 sent 标记未更新、sent 标记更新前宕机。
- 允许恢复：按 Outbox id 重投，消费者按 event id 幂等处理。
- 禁止行为：把 Outbox `sent` 当成下游业务完成，或因发送失败回滚已提交业务 Fact。
- oracle 检查：本地 Fact 有对应 Outbox 事件；消费者最终要么完成幂等处理，要么留下可恢复未处理状态。

### Broker 和消费者

broker 边界验证的是“消息传输状态不能替代业务处理状态”。消费者边界验证的是“重复、乱序和重投不能产生第二次业务效果”。

- 注入位置：broker 暂停、重复投递、乱序投递、offset 提交前失败、offset 提交后业务失败。
- 允许恢复：重复消费同一 event id，补写消费者处理记录，查询下游幂等结果。
- 禁止行为：只看 broker offset 断言业务成功，或在处理记录缺失时直接重复外部副作用。
- oracle 检查：消费者处理记录、业务唯一约束和下游 Fact 共同证明业务效果最多一次。

### Temporal Activity

Temporal Activity 边界验证的是“workflow history 记录的是编排过程，不是最终业务 Fact”。外部副作用成功但 Activity completion 未写入 workflow history 时，工作流引擎可能重试 Activity。

- 注入位置：Activity 调用外部渠道后、Activity completion 上报前、worker 返回结果后但 workflow history 未持久化前。
- 允许恢复：Activity 用同一个 request id 重试，先查外部结果，再决定补记成功、保持 unknown 或转对账。
- 禁止行为：每次 Activity retry 生成新的外部 request id，或把 workflow history 的完成状态当成资金、供应商或人工审批事实。
- oracle 检查：外部渠道流水或供应商订单、领域 Fact、账本 Fact 和 workflow history 能互相解释。

### 外部渠道

外部渠道边界验证的是“响应超时不是失败”。渠道响应超时后，渠道可能已经完成扣款、退款或授权。

- 注入位置：请求已发出但响应丢失、响应超时、回调先于查询、查询返回处理中。
- 允许恢复：本地进入 unknown，使用同一 request id 查询，等待回调，或进入对账任务。
- 禁止行为：本地直接判失败后重新发起不可关联扣款，或用迟到失败覆盖已确认成功。
- oracle 检查：渠道流水、支付请求、账本分录、对账差错和人工修复 Fact 必须能解释同一业务效果。

### 供应商

供应商边界验证的是“供应商 Fact 可能比本地状态晚到，但不能被删除或覆盖”。供应商成功回调迟到尤其危险，因为本地可能已经取消、补偿或转人工。

- 注入位置：供应商响应超时、供应商成功回调迟到、供应商失败回调迟到、查询和回调结果冲突。
- 允许恢复：状态机接收迟到成功、生成差错、触发补偿、进入人工审批或人工挂起。
- 禁止行为：删除供应商订单 Fact，重复预订，或用本地失败覆盖供应商成功。
- oracle 检查：供应商订单、退款或罚金、补差、人工审批和本地订单终态必须形成可审计闭环。

### 对账任务

对账任务边界验证的是“对账只能分类和发起修复，不能悄悄改写领域事实”。重跑对账应得到稳定差错集合。

- 注入位置：账单下载后、匹配一半后、差错生成后、修复命令发出前、复核关闭前。
- 允许恢复：按对账批次、渠道流水、供应商单号和业务单号幂等重跑。
- 禁止行为：重复创建修复命令，直接改领域状态，或关闭没有证据链的差错。
- oracle 检查：对账差错、修复命令、审批、复核和最终修复 Fact 必须唯一且可追踪。

### 人工审批

人工审批边界验证的是“人工动作也是 Fact，也必须幂等和可复核”。人工修复不能成为绕过不变量的新入口。

- 注入位置：审批通过后修复命令未执行、修复命令执行后响应丢失、审批回调重复、复核关闭前失败。
- 允许恢复：同一 request id 重提修复命令，返回同一修复结果，或追加复核说明。
- 禁止行为：重复冲正、重复补分录、无 maker-checker 直接改状态。
- oracle 检查：审批 Fact、证据快照、修复 Fact、账本调整和对账关闭原因必须互相解释。

## 关键边界

| 边界 | 要保留的事实 | 恢复依据 | 不能依赖 |
| --- | --- | --- | --- |
| Outbox | 业务 Fact、Outbox id、event id、发送尝试 | publisher 扫描和消费者幂等 | `sent` 标记证明业务完成 |
| Consumer | event id、consumer group、处理记录、业务唯一键 | 处理表、下游查询、业务 request id | broker offset 证明业务副作用成功 |
| Temporal Activity | Activity attempt、request id、外部请求记录、workflow history | 同 request id 重试、查询、回调、对账 | workflow history 直接替代资金或供应商 Fact |
| 外部渠道 | 渠道 request id、渠道流水、回调、查询结果、账单 | unknown、查询、回调、对账任务 | 超时等于失败 |
| 供应商 | 供应商 request id、供应商订单、回调、罚金或补差 | 查询、补偿、人工审批、人工挂起 | 本地取消能删除供应商成功 |

## 危险误用

| 误用 | 后果 | 正确做法 |
| --- | --- | --- |
| 用 kill process 代替故障模型 | 不知道故障前有哪些 Fact，恢复是否正确不可判断 | 先定义注入点、故障前 Fact、允许恢复和禁止重复动作 |
| 只验证服务能恢复启动 | 服务健康但事实分叉仍然存在 | 检查恢复后业务 Fact、账本 Fact 和外部 Fact 是否可解释 |
| 故障后简单重试外部请求 | 重复扣款、重复出票、重复退款或重复供应商订单 | 使用 request id、查询、回调、幂等重试和对账 |
| 把 broker offset 当消费者完成 | 业务副作用失败或处理记录缺失被掩盖 | 消费者必须落处理记录并满足业务唯一约束 |
| 把 workflow history 当资金事实 | 编排成功掩盖渠道、供应商或账本缺口 | workflow history 只作为执行线索，最终读取领域 Fact |
| 对账失败后直接改数据 | 修复不可审计，历史回放无法解释 | 生成差错、修复命令、人工审批、复核和修复 Fact |
| 人工审批绕过不变量 | 人工动作制造新的不可解释事实 | 人工修复进入 History，并被 oracle 检查 |
| 宣称故障注入覆盖全部生产故障 | 过度承诺，忽略生成范围之外的网络、时钟、容量和人为风险 | 只声明在已定义注入点和 History 范围内降低风险 |

## 输出结论

故障注入用例的输出应该包括：注入点、故障前 Fact、恢复动作、禁止重复动作、最终终态和 oracle 检查结果。只有这些信息齐全，故障注入才是在验证一致性，而不是只是在测试系统会不会重启。

一次故障注入通过只能说明：在这个注入点、这段 History 和这些 Invariant 范围内，没有发现不可解释 Fact。它不能证明所有生产故障都被覆盖，也不能替代属性测试、历史回放、对账验证、生产监控、审计和人工复核。
