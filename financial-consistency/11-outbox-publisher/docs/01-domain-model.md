# 01 领域模型

本章的模型故意保持小：只覆盖内部转账资金内核的最小事实集合。每张表都对应一种业务事实，不用缓存、日志或接口返回替代数据库事实。

## account

`account` 是账户当前状态。它保存账户号、币种、可用余额、冻结余额和版本号。

服务在转账时会锁定转出和转入账户行，然后更新 `available_balance`。余额表的作用是支撑在线判断：是否余额充足、转账后页面应该显示多少、后续风控或限额判断应该基于哪个当前值。

## transfer_order

`transfer_order` 是转账业务单。它保存转账 ID、幂等请求 ID、转出账户、转入账户、币种、金额、状态和失败原因。

它回答的是“这笔转账业务是否成立，最终是什么结果”。`SUCCEEDED` 代表业务成功，`FAILED` 代表业务已终止，例如余额不足。它不是账本本身，也不直接证明钱已经一借一贷地落账。

## idempotency_record

`idempotency_record` 保存调用方传入的幂等键、请求摘要、业务类型、业务 ID、状态、响应码和响应体。

同一个幂等键重复提交相同 payload 时，服务返回第一次完成时存下来的响应；同一个幂等键提交不同 payload 时，服务拒绝请求。这个表保护的是写接口语义：客户端重试不能制造第二笔转账，也不能用旧 key 改写新的业务意图。

## ledger_entry

`ledger_entry` 是账本分录。成功转账必须有两行：

- 转出账户一行 `DEBIT`。
- 转入账户一行 `CREDIT`。

两行必须属于同一 `transfer_id`，金额和币种必须彼此一致，并且与 `transfer_order` 的金额和币种一致。Task 6 的验证器已经把这些规则收紧为 `LEDGER_DOUBLE_ENTRY_REQUIRED` 和 `LEDGER_BALANCED` 等不变量。

## outbox_message

`outbox_message` 是本地消息事实。成功转账在同一个 MySQL 事务里写入一行 `PENDING` 的 `TransferSucceeded` 事件，`aggregate_type` 为 `TRANSFER`，`aggregate_id` 为转账 ID。

这张表不是 broker，也不表示消息已经发布。它表示“业务成功后有一条必须被发布的事件事实”。后续发布器可以扫描 `PENDING` 行，成功发布后再更新状态。

## 为什么余额和账本都必须存在

余额和账本解决的是不同问题。

余额是当前状态，适合高频读取和交易前判断。如果每次都从所有账本分录重算余额，接口路径会变长，也更难控制锁和性能。

账本是审计事实，适合追溯、对账和重建。如果只有余额，余额错了只能看到结果，无法解释是哪一笔交易造成的。

因此资金系统通常同时保留余额和账本：在线交易用余额，事后验证用账本和业务单反推。当前验证器会检查成功转账必须有双分录，账本必须与转账金额和币种一致，账本也必须引用已成功转账。余额和账本不是重复字段，而是互相校验的两类事实。
