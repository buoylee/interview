# 08 模式组合

## 目标

真实系统通常不是单模式。一个金融级流程会同时使用状态机、幂等、Outbox、Saga、TCC-like 资源预留、Temporal、事务消息或 CDC、账本、对账和人工处理。

本章的重点不是再解释每个模式的定义，而是说明它们如何组合。模式组合必须先回答事实来源、状态边界、外部未知、可逆性、事件传播和修复闭环，再决定哪些执行机制适合承载流程。

不存在一个 magic pattern 能同时解决资金入账、外部渠道超时、库存预留、供应商出票、消息发布、workflow 恢复和账本对账。真实设计必须把不同问题交给不同模式，并明确每个模式不能越界。

## 内部转账

推荐组合：

```text
状态机 + 幂等 + 账户预留/TCC-like + Saga/编排 + 账本 + Outbox + 对账 + 人工处理
```

内部转账的核心边界不是一个跨 `account-service` 和 `ledger-service` 的共享本地事务。`account-service` 拥有余额、冻结和 account movement，在自己的本地事务里完成冻结、扣减、入账、幂等记录和 Outbox；`ledger-service` 拥有不可变 posting 和 double-entry，在自己的本地事务里完成借贷分录、posting 幂等和 Outbox。orchestrator 或 Saga-like 状态机协调这些已落库事实，并在 account movement 和 ledger posting 分叉时通过重试、补充分录、冲正/调整、对账或人工处理收敛。

关键事实：

- 账户余额。
- 冻结金额。
- 资金流水。
- 会计分录。
- account movement。
- ledger posting。
- 转账状态。
- 幂等请求号和转账单号。
- Outbox 事件和消费者处理记录。

不能做：

- 用 Saga/编排替代 `account-service` 和 `ledger-service` 各自的本地事务边界。
- 只靠消息最终一致性更新账本。
- 没有借贷平衡检查、余额守恒检查和唯一约束。
- 把已过账转账删除或覆盖成失败；修复必须追加冲正、调整或人工处理事实。

## 充值提现

推荐组合：

```text
状态机 + 渠道幂等 + 查询/回调 + Outbox + 对账 + 人工处理
```

充值提现多了外部渠道。渠道请求超时、回调乱序、主动查询和渠道账单可能给出不同时间点的事实。本地状态机必须保留 `PAYMENT_UNKNOWN`、`WITHDRAW_UNKNOWN` 或 `REFUND_UNKNOWN` 这类未知状态，并通过查询、回调、渠道账单、对账和人工处理收敛。

关键事实：

- 本地充值单、提现单或支付单。
- 渠道请求号和渠道交易号。
- 渠道回调。
- 主动查询结果。
- 支付、提现或退款状态。
- 账户流水和会计分录。
- 渠道账单。
- 对账差错和人工处理记录。

不能做：

- 把渠道超时当失败。
- 重复发起无关联扣款、出款或退款。
- 用本地订单状态覆盖渠道成功事实。
- 只看本地订单不看渠道账单、账户流水和账本分录。
- 让人工处理直接改历史状态，而不生成可审计修复事实。

## 电商下单

推荐组合：

```text
库存预留/TCC-like + Saga + Outbox + 支付退款状态机 + 对账
```

电商下单不是单纯的消息最终一致性问题。库存适合 TCC-like 资源预留，因为 Try 能真实占用库存，Confirm 能扣减预留，Cancel 能释放预留。订单、支付、库存确认、取消和退款属于跨服务长流程，适合 Saga 编排，并通过 Outbox 发布每个服务的本地事实。

关键事实：

- 订单和订单状态。
- 库存预留、确认和释放记录。
- 支付请求、支付成功和支付未知状态。
- 退款请求、退款成功和退款未知状态。
- 发货、取消或售后状态。
- 账本分录和商户结算依据。
- Outbox 事件和消费者幂等记录。

不能做：

- 支付成功后直接删除订单或把订单改成不可解释失败。
- 库存预留失败后继续扣款。
- 消息重复导致重复确认库存、重复释放库存或重复退款。
- 把 Saga 补偿理解成数据库回滚。
- 已支付取消后没有退款、冲正、账本和对账路径。

## 旅行组合预订

推荐组合：

```text
Temporal Saga + 状态机 + Outbox + 付款策略 + 供应商查询 + 退款 + 对账 + 人工处理
```

旅行组合预订同时包含核心项、附加项、供应商未知、不可逆动作、价格变化、支付授权、capture、void、refund、罚金和补差。Temporal 适合承载长流程 Saga、timer、重试和人工等待，但 workflow history 不是供应商事实、支付事实、退款事实或账本事实。

关键事实：

- 报价快照。
- 组合订单和子订单状态。
- 机票出票。
- 酒店确认。
- 租车确认。
- 保险生效。
- 供应商请求号、查询结果、回调和账单。
- 支付授权、capture、void、refund。
- 罚金、补差、退款和账本分录。
- 人工处理工单、审批和复核记录。
- 对账差错和修复验证结果。

不能做：

- 附加项失败取消核心行程。
- 供应商未知直接失败。
- 已出票或已生效动作被本地字段回滚。
- workflow history 替代供应商、支付、退款和账本事实。
- 把核心项部分成功伪装成全失败。
- 自动流程无法判断用户权益、供应商责任或资金风险时不转人工处理。

## 组合原则

- 状态机定义合法业务推进，拒绝重复、乱序、迟到成功和迟到失败造成的非法迁移。
- 幂等保护重复命令、重复消息、重复回调、重复 Activity 和重复人工修复动作。
- Outbox 保护本地事实和事件发布，不能证明下游业务已经完成。
- Saga 或 Temporal 编排长流程、补偿顺序、超时等待和人工等待。
- TCC-like 模型只用于可预留资源，Try 必须真实冻结、占用或授权资源。
- 事务消息或 CDC 是事件传播选项，不是业务一致性本身。
- 账本保存资金事实，workflow、broker offset、BPMN 实例或接口返回不能替代账本。
- 外部未知状态必须通过查询、回调、对账或人工处理收敛，不能被本地流程直接裁决。
- 对账发现事实分叉，修复完成后必须重新对账。
- 人工修复处理不可逆、未知和高风险差错，但人工结论也必须落成领域事实、审计记录和可验证分录。

## 验证方法

- 对每个组合场景列出事实来源，包括本地事实、外部事实、资金事实、事件事实、人工事实和对账事实。
- 对每个事实来源列出唯一约束、幂等键、状态机终态和审计字段。
- 对每个跨服务事件验证 Outbox 或等价机制，确认本地事务提交后事件可恢复发布。
- 对每个消费者验证重复消息、乱序消息和迟到消息不会产生第二次业务效果。
- 对每个外部系统结果验证查询、回调、超时、未知状态和对账路径。
- 对每个补偿动作验证它是新的业务事实，不是删除或覆盖历史事实。
- 对每个 Temporal Workflow 验证 Activity 幂等、request id、timer 恢复和人工等待恢复；同时验证 workflow history 不被当成业务事实来源。
- 对每个账本相关终态验证余额、冻结、流水、借贷分录、罚金、补差和退款能互相解释。
- 对每个人工处理 case 验证 maker-checker、证据快照、修复命令、审批结果和复核记录。
- 对每个终态验证是否能解释资金、供应商、库存、订单、账本和对账事实。

## 面试表达

金融级一致性不是在 Outbox、Saga、TCC、Temporal 里选一个万能答案。内部转账先守住账户和账本边界；充值提现先承认外部渠道未知；电商下单把库存预留、支付退款状态机和 Saga 组合起来；旅行组合预订用 Temporal 承载长流程，但供应商、支付、退款、账本、对账和人工处理事实必须独立存在。

可以进一步强调：模式组合的判断标准是事实是否可解释，而不是框架是否成功。只要终态无法解释资金、外部渠道、供应商、库存、账本和修复记录，系统就没有真正完成一致性闭环。
