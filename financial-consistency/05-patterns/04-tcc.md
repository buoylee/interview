# 04 TCC

## 解决的问题

TCC 解决可预留资源的一致性问题。它把业务动作拆成 Try、Confirm、Cancel：

- Try：真实预留资源，并把资源从可用池移到冻结、占用或授权状态。
- Confirm：只确认已经成功 Try 的资源，把预留事实转成最终业务事实。
- Cancel：释放 Try 阶段预留的资源，让资源回到可用或授权释放状态。

TCC 成立的前提是业务资源真的支持预留、确认和取消。如果 Try 只是参数校验、余额查询、库存检查或调用前置校验，没有冻结余额、预留库存或创建支付预授权，TCC 只是接口命名，不能保护一致性。

金融级 TCC 保护的是资源状态，而不是接口调用顺序。每个分支都必须有全局事务 ID、分支 ID、业务幂等键、明确状态机和审计记录；Try、Confirm、Cancel 都必须幂等，并且能解释重复、乱序、超时、宕机和并发请求。

## 基本模型

```text
Try
  -> 校验业务状态
  -> 预留资源
  -> 记录 TRY_SUCCEEDED

Confirm
  -> 只接受 TRY_SUCCEEDED
  -> 确认预留资源
  -> 记录 CONFIRMED

Cancel
  -> 释放 TRY_SUCCEEDED 的预留资源
  -> 或记录空回滚
  -> 记录 CANCELED
```

Confirm 不能跳过 Try，也不能确认失败、取消或不存在的 Try。Cancel 的职责是释放 Try 留下的占用，不是删除历史事实；如果业务已经 Confirm，后续 Cancel 必须被拒绝或保持幂等返回已完成状态，不能覆盖已经完成的资金、库存或授权事实。

## 场景映射

| 场景 | Try | Confirm | Cancel |
| --- | --- | --- | --- |
| 内部账户转账 | 冻结余额，记录冻结流水和转账分支 | 扣减冻结、增加收款方余额、写账本分录 | 解冻余额，记录取消原因 |
| 库存 | 预留库存，减少可售并增加 reserved | 扣减预留库存，生成出库或履约事实 | 释放预留库存，恢复可售 |
| 支付预授权 | 向渠道创建授权并冻结资金 | capture 授权完成扣款 | void/release 授权释放冻结 |

内部账户转账中，Try 可以冻结付款方资金，让这笔资金不可再被其他交易使用；但 Confirm 发起的扣减冻结、收款方入账、流水和会计分录仍然需要一个本地账本一致性边界，不能用 TCC 替代本地账务事务、复式记账不变量或原子入账。库存场景中，Try 必须减少可售或建立不可重复消费的 reservation，不能只读取库存数量。支付预授权场景中，Try 对应渠道授权成功，Confirm 对应 capture，Cancel 对应 void 或 release；渠道返回未知时不能简单重试扣款，必须进入查询、对账或人工处理。

## 必须处理的问题

### 幂等

Try、Confirm、Cancel 都必须支持重复调用。重复 Try 只能返回同一笔预留结果，不能冻结两次；重复 Confirm 只能确认一次，不能重复扣款、重复出库或重复入账；重复 Cancel 只能释放一次，不能把余额或库存释放成超过原始可用量。

### 空回滚

Cancel 可能先于 Try 到达，例如协调者超时后触发取消，但 Try 请求在网络中延迟。系统必须在分支级记录空回滚状态，至少以全局事务 ID + 分支 ID 保存“这个分支已经取消”的事实。业务幂等键可以作为补充索引，不能替代全局事务 ID + 分支 ID。

### 悬挂

空回滚之后迟到的 Try 不能再成功，否则资源会被预留但没有后续 Confirm 或 Cancel，这就是悬挂。Try 必须先用全局事务 ID + 分支 ID 检查分支是否已经 CANCELED、EMPTY_CANCELED 或已终结；如果已经取消，应拒绝或幂等返回取消事实。业务幂等键只能帮助定位业务请求，不能作为防悬挂的唯一身份。

### 并发 Confirm 和 Cancel

Confirm 和 Cancel 不能同时成功。状态转换必须用条件更新、版本号、唯一约束或数据库事务保护，例如只允许 `TRY_SUCCEEDED -> CONFIRMED` 或 `TRY_SUCCEEDED -> CANCELED` 其中一个更新成功。失败的一方只能读取最终状态并幂等返回，不能再次修改资源。

## 不适合场景

- 外部供应商不提供受控的预留、确认和取消语义。
- 机票已经出票、票号已经产生，Cancel 只能退票且可能有罚金。
- 保险已经生效，取消不是释放预留资源，而是退保、批改或人工处理。
- Try 阶段无法隔离资源，Confirm 时资源可能已经被其他流程消费。
- Cancel 没有可靠结果，或者供应商不保证取消幂等。
- Confirm 或 Cancel 依赖人工、邮件、批处理或不可预测异步结果。

这些场景更适合 Saga、状态机、查询、对账、退款、罚金、替代供应商或人工工单。不能因为接口名字写成 Try/Confirm/Cancel，就把不可逆外部动作包装成 TCC。

## 危险误用

| 误用 | 后果 | 正确做法 |
| --- | --- | --- |
| Try 只做参数校验，没有预留资源 | Confirm 时资源可能已经不存在 | Try 必须冻结余额、预留库存或创建授权 |
| Confirm 允许处理不存在的 Try | 越过资源预留直接完成业务 | Confirm 只能确认成功 Try |
| Cancel 不幂等 | 重复释放导致余额、库存或授权错误 | 以分支状态和唯一约束保护释放动作 |
| 空回滚不落库 | 迟到 Try 成功后形成悬挂 | Cancel 先到也要记录终结事实 |
| Confirm 和 Cancel 并发无保护 | 同一笔资源既确认又释放 | 用状态机和条件更新保证单一终态 |
| 对不可逆外部动作套 TCC | Cancel 无法真实撤销 | 改用 Saga、退款、对账或人工处理 |

## 验证方法

- 重复调用 Try，验证只预留一次，冻结余额、预留库存或支付预授权金额不重复增加。
- 重复调用 Confirm，验证只确认一次，账本分录、扣款、出库或 capture 不重复。
- 重复调用 Cancel，验证只释放一次，可用余额、可售库存或授权释放不超过原始资源。
- 先 Cancel 后 Try，验证系统记录空回滚，并拒绝迟到 Try，避免悬挂。
- 并发执行 Confirm 和 Cancel，验证只有一个状态转换成功，另一个按最终状态幂等返回。
- 在 Try 成功后宕机，验证恢复后能继续 Confirm 或 Cancel，且审计记录完整。
- 对账冻结、确认、释放和账本分录，验证资源总量守恒。

## 面试表达

TCC 的关键不在三段式接口，而在 Try 是否真实预留资源。内部账户、库存和支付预授权适合 TCC-like 模型；不受控外部供应商、已经出票、已生效保险和 Cancel 不可靠的动作不适合 TCC。

可以进一步强调：金融级 TCC 必须把 Try、Confirm、Cancel 都做成幂等状态迁移，并显式处理空回滚、悬挂以及 Confirm/Cancel 并发。否则它不是一致性模式，只是把失败风险拆到了三个接口里。
