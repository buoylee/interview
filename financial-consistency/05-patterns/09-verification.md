# 09 验证方法

## 目标

验证方法要降低一致性模式在异常历史下产生不可解释事实的风险。它不能证明所有可能异常历史都安全，但必须通过指定异常历史和不变量检查，覆盖超时、重复、乱序、宕机、回调延迟、人工修复和对账差错。

更重要的是，验证对象不是“框架有没有返回成功”，而是系统是否始终只产生可解释事实。`SUPPLIER_UNKNOWN`、`PAYMENT_UNKNOWN`、`REFUND_UNKNOWN` 这类未知状态不能被直接裁决为失败；补偿不是回滚或删除历史；Temporal workflow history 也不能替代供应商、支付、退款或账本事实。

## 按模式验证

| 模式 | 必须验证 |
| --- | --- |
| Outbox | 本地事务提交后 publisher 崩溃，恢复后事件补发且外部动作不重复 |
| Saga | 任意步骤失败、未知或超时后，已成功步骤进入合法补偿、重试、对账或人工处理 |
| TCC | Try / Confirm / Cancel 幂等，支持空回滚、悬挂、防重复确认和 Confirm / Cancel 互斥 |
| Temporal | worker 重启、Activity 重试、timer 恢复、人工等待恢复后，业务事实不重复且 workflow history 不被当成账本 |
| 事务消息 | 半消息、提交、回查、重复投递和消费者幂等都能收敛到同一业务事实 |
| CDC | 重复捕获、乱序投影、schema 变化和下游幂等不会改变领域事实 |
| 状态机 | 非法状态转换被拒绝，重复、迟到和乱序事件不能覆盖已确认成功事实 |
| 对账 | 本地事实、外部事实、资金事实、账本事实、人工事实和事件事实可互相解释 |

这些验证要覆盖异常历史，而不是只覆盖 happy path。比如 Outbox 不能只测消息能发出，还要测本地事务提交后进程崩溃；Saga 不能只测补偿函数被调用，还要测补偿失败后是否进入可追踪的重试、对账或人工处理；Temporal 不能只测 workflow 执行完成，还要测 Activity 重试是否依赖幂等键保护外部动作。

## 不变量检查

- 同一业务幂等键最多产生一个业务效果。
- `SUPPLIER_UNKNOWN`、`PAYMENT_UNKNOWN`、`REFUND_UNKNOWN` 不能直接当失败。
- 已 capture 的资金不能 void，只能 refund、冲正、调整或进入人工处理。
- TCC 的 Confirm 和 Cancel 不能同时成功。
- 成功事实不能被迟到失败、重复回调或人工修复直接覆盖。
- 消息重复投递不能重复扣款、重复退款、重复出票、重复释放库存或重复记账。
- 补偿必须追加新的业务事实，不能删除历史事实。
- 对账服务不能直接修改领域事实，只能生成差错、修复命令、工单、审批和复核记录。
- workflow history、broker offset、接口返回值和日志不能替代账本分录。
- 每个终态都必须能解释相关业务事实、账本事实，以及实际采用的传播机制事实，例如 Outbox、事务消息或 CDC，加上消费者、投影和对账事实。

不变量应该放在状态机、唯一约束、账本规则、消费者幂等表和对账校验中重复表达。只在单元测试里断言一次不够，因为异常历史通常来自跨服务时序、重试和人工修复。

## 属性测试

属性测试适合在大量随机异常组合下检查不变量是否仍然成立。它不能穷尽所有历史，但随机生成命令、消息、回调、查询结果、人工修复和系统故障，比枚举少量固定 happy path 更接近金融系统的真实风险。

随机生成：

- 命令重复。
- 消息重复。
- 回调乱序。
- 迟到成功和迟到失败。
- 外部请求超时。
- 本地事务提交后进程宕机。
- workflow worker 重启。
- Activity 重试。
- TCC Cancel 先于 Try。
- TCC Confirm 和 Cancel 并发。
- CDC 事件重复、乱序和投影延迟。
- 对账批次重复执行。
- 人工修复命令重复提交。

检查：

- 状态机终态合法。
- 每个状态转换都有前置事实。
- 幂等键没有产生第二次业务效果。
- 资金金额可解释。
- 账本分录借贷平衡且能解释原始业务事实。
- 外部事实和本地事实冲突时进入对账或人工处理。
- 补偿产生的是新事实，不是删除或覆盖旧事实。
- 未知状态不能由本地直接裁决，必须由查询、回调、对账或人工处理推进，或者稳定进入可审计的长期对账终态或人工挂起终态。

属性测试的价值不在于生成复杂随机数据本身，而在于把系统承诺写成不变量。只要已生成的随机历史能产生不可解释事实，就说明模式设计还没有守住这些异常场景下的一致性边界。

## 故障注入

故障注入要放在最容易产生事实分叉的位置：

- Outbox publisher 在本地事务提交后、标记 sent 前崩溃。
- broker 暂停、重复投递或延迟投递。
- 消费者处理一半宕机。
- payment activity 超时后重试。
- refund request 已提交但结果未知。
- supplier request 超时后晚到成功回调。
- TCC Confirm 和 Cancel 并发。
- Temporal worker 重启、timer 恢复和人工等待恢复。
- CDC 下游投影落后、重复或乱序。
- 对账任务执行到一半失败后重跑。
- 人工修复审批通过后修复命令重复执行。

每个故障注入用例都要检查两个结果：第一，业务事实没有丢失；第二，恢复动作不会重复执行不可逆外部动作。比如 publisher 可以重发 Outbox 事件，但不能重新发起扣款、出票或退款；Activity 可以重试，但必须带同一个 request id 或业务幂等键。

## 历史回放

历史回放把真实或构造的历史事件按不同顺序重放，检查异常历史是否仍然可解释。它特别适合验证回调乱序、迟到消息、重复 CDC 投影、人工修复和对账差错。

回放时检查：

- 每个状态转换都有前置事实。
- 每个资金动作都有原始请求、幂等键和渠道流水。
- 每个退款都有原支付、退款请求、退款结果和账本分录。
- 每个供应商成功事实都有供应商请求号、查询结果、回调或账单证据。
- 每个补偿动作都有触发原因、幂等键和审计记录。
- 每个人工修复都有证据快照、审批、修复命令和复核结果。
- 每个对账差错都有分类、owner、修复动作和重新对账结果。
- 采用 Outbox、事务消息或 CDC 时，每个传播事实和投影结果都能追溯到已经落库的领域事实。

历史回放不能把 workflow history 当成业务事实来源。Temporal 的历史可以说明编排曾经执行过哪些步骤，但支付是否成功、退款是否成功、供应商是否确认、账本是否入账，仍然要回到各自领域事实和账本事实。

## 对账验证

对账验证要主动构造差异，而不是只验证无差异批次可以通过。

构造差异：

- 本地显示成功但供应商无单。
- 供应商成功但本地未推进。
- 支付成功但订单失败未退款。
- 退款成功但账本无分录。
- 账本有分录但领域单据缺失。
- 采用 Outbox 时事件缺失。
- 采用 CDC 时投影漏数据。
- 消费者重复处理造成重复业务效果。
- 人工修复无审计。
- 对账修复后未重新对账。

每个差异必须稳定分类，路由到 owner，执行修复命令或人工工单，修复后重新对账。对账的目标不是把报表调平，而是让本地事实、外部事实、资金事实、账本事实和修复事实重新变得可解释。

## 面试表达

验证一致性模式的关键不是展示正常流程能跑通，而是用异常历史降低不可解释事实出现的风险。金融级系统必须把未知状态、重复请求、乱序回调、迟到消息、宕机恢复、补偿失败、人工修复和对账差错都纳入验证范围。

可以进一步强调：Outbox、Saga、TCC、Temporal、事务消息、CDC、状态机和对账都不是单独的银弹。模式只有能通过不变量、属性测试、故障注入、历史回放和对账验证，才适合进入金融级主线。
