# 02 Outbox 和本地消息表

## 解决的问题

Outbox 解决的是本地事实和事件发布之间的不一致：业务表已经提交，但进程在发消息前宕机；或者消息发出后，本地事务回滚。金融级系统不能让这两种情况变成不可解释状态。

它的核心目标是让“本地业务事实已经发生”和“对应事件等待发布”具备同一个提交边界。只要本地事务提交成功，系统就可以通过 Outbox 记录恢复发布动作；如果本地事务回滚，事件也不会被保存为可发布消息。

## 基本模型

```text
本地事务
  -> 写业务事实
  -> 写 Outbox 事件
提交成功
  -> publisher 异步投递 broker
  -> 消费者按业务幂等键处理
```

业务表和 Outbox 表必须在同一数据库事务中提交。Outbox 事件代表本地事实已经落库，不代表下游消费者已经成功，也不代表外部供应商、支付渠道或退款渠道已经完成。

常见实现会有一张本地消息表，至少包含事件 ID、业务幂等键、事件类型、payload、状态、重试次数、创建时间、更新时间和最后错误。publisher 轮询 pending 事件或读取数据库变更日志，将事件投递到 broker；投递确认后再把 Outbox 记录标记为 sent。这个标记只说明消息已进入发布通道，不说明消费者业务已经完成。

## 保护的事实

- 订单创建后一定有 `BookingCreated`。
- 支付扣款落库后一定有 `PaymentCaptured`。
- 库存确认落库后一定有 `InventoryConfirmed`。
- 旅行供应商确认落库后一定有 `FlightTicketed` 或 `HotelConfirmed`。
- 对账差错落库后一定有 `TravelMismatchDetected`。

这些事件都是业务事实的外部可观察表达。它们应该从已经落库的事实生成，而不是从内存里的中间状态生成。比如 `PaymentCaptured` 只能说明本系统记录了支付成功事实，`FlightTicketed` 只能说明出票确认事实已被本系统保存，`TravelMismatchDetected` 只能说明差错事实已落库并可被后续流程处理。

## 不能解决的问题

- 不能证明消费者完成业务处理。
- 不能证明供应商确认、支付到账或退款成功。
- 不能自动处理重复消息。
- 不能替代状态机、账本、对账或人工处理。
- 不能让外部不可逆动作变成可回滚。

Outbox 解决消息不丢，不解决业务闭环。消费者拿到事件后仍可能失败、超时、重复执行或只处理了一半。broker offset 只证明消费者客户端推进了消息读取位置，不能证明数据库写入、账本分录、状态迁移、供应商确认或通知发送已经完成。

金融系统不能把 Outbox 当成最终一致性的全部方案。状态机仍然负责约束状态迁移，账本仍然负责资金事实不可变记录，对账仍然负责发现跨系统差异，人工处理仍然负责处理自动流程无法消除的悬挂状态。

## 危险误用

| 误用 | 后果 | 正确做法 |
| --- | --- | --- |
| 业务提交后直接发消息，不写 Outbox | 进程宕机导致事件丢失 | 业务事实和 Outbox 同事务提交 |
| 发布失败后重做外部扣款或供应商动作 | 重复扣款、重复出票或重复退款 | 只重试事件投递，不重做外部动作 |
| 消费者只依赖 broker offset | 业务重复或漏处理不可解释 | 消费者使用业务幂等键和处理表 |
| 把消息投递成功当业务完成 | 下游失败但本地误判成功 | 下游完成必须有自己的事实事件 |

消费者幂等通常需要独立的处理记录或业务唯一约束。比如消费 `PaymentCaptured` 时，以支付单号、捕获流水号或事件 ID 建立幂等键；消费 `FlightTicketed` 时，以订单号、航段和票号建立幂等约束；消费 `TravelMismatchDetected` 时，以对账批次、供应商流水和差错类型避免重复建单。

## 验证方法

- 在本地事务提交后、publisher 投递前杀进程，验证恢复后事件能补发。
- 暂停 broker，验证业务事实仍能提交，Outbox 保持 pending。
- 重放同一事件，验证消费者只产生一次业务效果。
- 让消费者处理失败，验证 broker offset 不被当作业务完成证明。
- 对账业务事实和 Outbox 事件，验证每个关键事实都有对应事件。

还应验证失败恢复路径：publisher 重启后只能重投 Outbox 事件，不能重新调用支付捕获、供应商出票或退款接口；消费者失败后应保留可追踪的错误事实或重试记录，而不是只依赖消息系统的消费位置。

## 面试表达

Outbox 的价值是把“本地事实”和“待发布事件”放进同一个事务。它解决消息不丢的问题，但不解决下游是否成功的问题。金融系统使用 Outbox 后仍然需要消费者幂等、状态机、账本、对账和人工修复。

可以进一步强调：Outbox 不是分布式事务，也不是业务完成证明。它只是保证本地事务提交后有一条可恢复、可重试、可对账的事件发布记录；真正的业务一致性要靠每个下游系统保存自己的完成事实，并通过状态机、账本、对账和人工处理把异常状态闭环。
