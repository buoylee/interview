# 06 事务消息和 CDC

## 解决的问题

事务消息、CDC 和 Outbox 都常被用来连接数据库事实和消息流，但它们解决的问题不同。本章比较三者的边界，防止把消息传播机制、数据库变更捕获机制误认为完整的业务一致性方案。

金融级系统真正关心的是事实是否可解释：订单、支付、退款、账本、供应商确认和对账差错必须有明确来源、状态机约束、幂等保护和审计记录。事务消息、CDC 和 Outbox 只能帮助事实向外传播，不能替代业务事实本身。

## 事务消息

事务消息通常依赖特定消息中间件语义，把本地事务状态和消息发送状态绑定起来。典型流程是先发送半消息，消费者暂时不可见；应用执行本地事务；本地事务成功后提交消息，失败后回滚消息；如果应用在提交或回滚前宕机，broker 通过回查接口查询本地事务状态，再决定消息应该提交还是回滚。

```text
发送半消息
  -> 执行业务本地事务
  -> 写入订单、支付、退款或账本事实
  -> 提交或回滚消息
异常中断
  -> broker 回查本地事务状态
  -> 根据本地事实提交或回滚消息
```

事务消息适合解决本地事务和消息发送之间的原子可见性问题，尤其是在消息中间件原生支持半消息、提交、回滚和回查时。它可以减少一部分 Outbox publisher 逻辑，但仍然要求本地业务事实可查询、回查接口幂等，并且能够在重启后准确回答这笔事务到底成功、失败还是未知。

它不能解决：

- 消费者业务是否成功。
- 消费者幂等。
- 下游状态机是否合法转换。
- 外部供应商、支付渠道或退款渠道是否真实成功。
- 账本分录是否平衡。
- 对账差异是否收敛。
- 跨多个服务数据库的强原子提交。

事务消息的主要风险来自 broker-specific semantics。不同中间件对半消息保存时间、回查次数、回查并发、事务状态超时、提交可见性和失败处理的定义不同。系统不能只写“用了事务消息”就结束设计，必须说明 broker 如何回查、回查接口如何读本地事实、未知状态如何处理，以及重复投递时消费者如何保持幂等。

## CDC

CDC 从数据库提交日志捕获变化，把已经提交的数据库事实流出到下游。它常用于构建读模型、同步搜索索引、审计、数据集成、报表链路，或者从 legacy 数据库抽取事实流。

```text
数据库事务提交
  -> binlog / WAL / redo log 产生变化
  -> CDC connector 捕获变更
  -> 投递到 broker 或下游存储
  -> 消费者更新投影、索引或审计表
```

CDC 的优势是对应用侵入较低，尤其适合已有系统需要把数据变化同步出去的场景。它捕获的是数据变化，不天然等于领域事件。一次支付成功可能修改支付单、流水、账本分录和订单状态；一次字段更新也可能只是技术修正、补录、迁移或重算。下游如果直接把表级变更当作业务事件，容易把“字段变化”误读成“业务事实发生”。

CDC 不能解决：

- 领域事件语义建模。
- 补偿规则。
- 消费者幂等。
- 下游投影乱序。
- schema evolution 带来的兼容问题。
- 多表变化如何合成为一个业务事实。
- 外部系统是否成功。

CDC 的主要风险是 data-change 与 domain-event gap。数据库变更说明某些行已经提交，不说明业务事件边界、意图、因果关系和补偿语义。下游投影还必须处理重复捕获、乱序到达、connector 重启、offset 回退、schema 变更、历史回放和字段语义变化。金融系统尤其不能把 CDC 投影完成当成支付到账、退款成功或账本已平。

## Outbox、事务消息、CDC 对比

| 机制 | 事实来源 | 优点 | 主要风险 |
| --- | --- | --- | --- |
| Outbox | 应用在本地事务中显式写业务事实和事件 | 业务语义清楚，和领域事件天然一致，便于对账 | 需要 publisher、重试、清理和事件表治理 |
| 事务消息 | broker 和本地事务状态协作 | 中间件支持时流程简洁，半消息和回查可减少发布窗口 | 依赖 broker-specific semantics，回查实现错误会放大不一致 |
| CDC | 数据库提交日志 | 对已有数据库侵入较低，适合读模型、搜索、审计和集成 | 数据变化不等于领域事件，schema evolution、排序和投影风险高 |

Outbox 的边界是应用主动声明“这个业务事实对应这个待发布事件”，并把两者放进同一个本地事务。事务消息的边界是依赖 broker 协议协调消息可见性和本地事务状态。CDC 的边界是从数据库日志捕获已提交变化，再由下游解释这些变化。

三者都不是完整业务一致性方案。它们只处理事实传播或变化捕获的一段链路，不负责判断支付、退款、供应商确认、库存履约或账本是否真的完成。完整的一致性仍然需要领域服务保存事实，状态机保护迁移，消费者幂等处理重复，对账发现差异，人工修复处理自动流程无法收敛的异常。

## 危险误用

| 误用 | 后果 | 正确做法 |
| --- | --- | --- |
| 把事务消息当跨服务分布式事务 | 下游失败、重复消费或外部未知状态被误判为成功 | 事务消息只保护本地事务和消息可见性，下游仍用状态机、幂等和对账 |
| 回查接口只看内存或临时状态 | 服务重启后无法判断半消息对应业务是否成功 | 回查必须读取持久化业务事实或事务记录 |
| 消费者没有幂等记录 | 重复投递导致重复扣款、重复退款、重复建单或重复投影 | 使用事件 ID、业务 request id、唯一约束和处理表保护消费者幂等 |
| 把 CDC 行变更直接当领域事件 | 下游误解字段修正、迁移或多表更新的业务含义 | 对关键业务显式建模领域事件，或在 CDC 下游做事件语义转换和验证 |
| 忽略 schema evolution | 字段新增、重命名、拆分或语义变化导致消费者解析错误 | 版本化 payload、兼容发布、回放测试和消费者契约验证 |
| 假设消息天然有全局顺序 | 投影被迟到事件覆盖，读模型和账本视图不一致 | 使用版本号、业务时间、条件更新和状态机拒绝倒退 |

## 金融级结论

金融级系统更关心事实可解释性，而不是只关心消息是否送达。不管选择 Outbox、事务消息还是 CDC，都不能省略：

- 消费者幂等。
- 状态机合法转换。
- 业务唯一约束。
- 账本分录和资金不变量。
- 对账和差错处理。
- 人工修复审计。

事务消息依赖 broker 的事务协议，CDC 依赖数据库日志，Outbox 依赖应用显式写事件。选择哪一种机制，要看事实来源在哪里、业务语义是否需要显式表达、现有系统是否能改造，以及下游是否能够承受重复、乱序、延迟和 schema 演进。

## 验证方法

- 事务消息：模拟半消息发送成功后服务宕机，验证 broker 回查会读取持久化本地事实，而不是重复执行业务动作。
- 事务消息：模拟本地事务成功但提交消息超时，验证回查最终提交消息，并且重复投递只产生一次消费者业务效果。
- 事务消息：模拟本地事务失败但半消息仍存在，验证回查会回滚或丢弃消息，下游不会看到不存在的业务事实。
- CDC：重复捕获同一数据库变化，验证下游投影幂等。
- CDC：乱序到达或 offset 回退，验证投影使用版本号、业务时间或状态机拒绝倒退更新。
- CDC：执行 schema evolution 演练，验证新增字段、字段重命名、payload 版本变化和历史回放不会破坏消费者。
- Outbox：publisher 崩溃恢复后只补发事件，不重做支付捕获、退款、供应商出票或其他外部动作。
- 三者共同验证：消息传播成功不能被当作支付成功、退款成功、供应商确认或账本已平，必须通过业务事实、账本事实和对账结果确认。

## 面试表达

事务消息和 CDC 是消息传播、变化捕获机制，不是完整业务一致性方案。Outbox 更贴近领域事件，因为应用在本地事务里显式写出业务事实对应的事件；事务消息依赖 broker 的半消息、提交和回查语义；CDC 依赖数据库提交日志，适合读模型、搜索、审计和集成。

可以进一步强调：三者都不能替代消费者幂等、状态机、账本、对账和人工修复。金融系统不能把“消息到了”当成“业务完成”，也不能把“数据库行变了”直接当成“领域事件发生”。真正可靠的设计必须同时说明事实来源、重复处理、乱序处理、schema 演进、对账验证和异常闭环。
