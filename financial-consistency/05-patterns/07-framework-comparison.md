# 07 框架比较

## 目标

本章从国际视角比较 Temporal、Camunda 和 Seata。比较目的不是排名，而是判断它们分别适合解决什么问题、不能解决什么问题，以及在金融级一致性教程中的位置。

这三个框架的核心定位不同：

- Temporal 更接近 durable execution 和 workflow orchestration。
- Camunda 更接近 BPMN、流程治理和 human task 编排。
- Seata 更接近分布式事务模式框架，覆盖 AT、TCC、Saga 等工程模式。

它们都不能替代领域服务所有权、状态机、幂等、账本、Outbox、对账和人工修复。框架只能承载一部分执行机制，不能替业务系统定义事实、资金规则和异常闭环。

## 对比表

| 框架 | 国际视角下的核心定位 | 适合解决 | 不适合解决 |
| --- | --- | --- | --- |
| Temporal | durable execution 和 workflow 编排 | 长流程 Saga、超时、重试、人工等待、恢复执行 | 业务事实、账本、对账、跨服务强原子提交 |
| Camunda | BPMN 流程建模、流程自动化和人工任务治理 | 审批流、运营处理流、流程可视化、业务流程治理 | 资金事实保护、支付幂等、账本一致性 |
| Seata | 分布式事务模式框架 | 理解 AT、TCC、Saga 等模式，部分受控服务间事务 | 外部不可控供应商、国际金融架构主线、替代对账 |

## Temporal

Temporal 推荐作为本教程的长流程编排主线。它适合把旅行组合预订、支付查询、退款轮询、人工等待和补偿动作编排成可恢复 workflow。Temporal 的价值在于 durable execution：Workflow 可以在 worker 重启、Activity 超时、timer 到期和进程崩溃后恢复执行。

适合场景：

- 旅行组合预订的多供应商 Saga。
- 支付超时后的查询、等待回调和人工升级。
- 退款未知后的轮询、补偿和转人工。
- 需要数小时、数天甚至更久等待窗口的流程。
- 需要可靠重试和恢复执行的编排层。

边界：

- Activity 必须幂等，并携带业务 request id。
- workflow history 不是账本，也不是订单、支付、退款或供应商事实来源。
- Workflow 成功不代表渠道账单、供应商账单、内部账本和对账全部一致。
- Temporal 不能绕过领域服务状态机，不能直接修改多个服务数据库。
- 仍然需要 Outbox、状态机、幂等、账本、对账和人工修复。

Temporal 解决“流程下一步如何可靠执行”，不解决“资金事实是否真实成立”。金融系统不能把 workflow completion 当成 accounting close。

## Camunda

Camunda 更偏 BPMN、流程建模、人工任务和业务流程治理。它适合表达审批流、运营处理流、人工任务、跨部门流程和需要可视化治理的业务流程。对于需要业务人员理解流程图、审计人工节点、管理任务队列的场景，Camunda 的位置比纯代码 workflow 更接近 process governance。

适合场景：

- 客服、风控、财务运营等人工审批流。
- 退款争议、异常订单、供应商差错的人工处理流。
- 跨部门流程治理和流程图审计。
- 需要 BPMN 语义表达网关、事件、人工任务和服务任务的场景。

边界：

- BPMN 流程图不是资金事实。
- 人工任务完成不代表支付成功、退款到账、供应商确认或账本已平。
- Camunda 可以治理流程，但不能替代幂等键、唯一约束、账本分录和对账。
- 如果服务任务调用外部系统，仍然必须由下游领域服务保护状态机和幂等。
- Camunda 的流程实例状态不能成为财务事实的唯一来源。

Camunda 适合帮助团队看清“流程怎么走、谁处理、在哪里等待”，但资金一致性仍要落在领域服务、账本、Outbox、对账和人工修复记录上。

## Seata

Seata 适合理解分布式事务框架中的 AT、TCC、Saga 等模式，尤其能帮助学习者认识不同事务模式的工程取舍。在受控服务、受控数据库和明确资源模型内，Seata 的模式可以作为学习和局部工程方案参考。

适合场景：

- 学习 AT、TCC、Saga 的差异和代价。
- 服务和数据库都在统一技术栈、统一治理域内的事务协作。
- Try 能真实预留资源、Confirm/Cancel 语义清楚的 TCC 场景。
- 对历史系统或内部系统做受控范围的一致性改造。

边界：

- AT 模式依赖受控数据库和代理语义，不适合不可控外部供应商、支付渠道和国际清算网络。
- TCC 仍要求 Try 真实预留资源，不能把普通接口包装成 TCC。
- Saga 仍然是补偿语义，不是强原子提交，也不是删除历史事实。
- Seata 不能替代领域服务所有权、Outbox、账本、对账和人工修复。
- 在国际金融架构表达中，不应把 Seata 当主线答案。

Seata 更适合作为“分布式事务模式框架”的对比材料。它能帮助解释 AT、TCC、Saga 的模式边界，但不应被描述成跨支付渠道、跨供应商、跨国家或跨清算系统的通用金融一致性答案。

## 推荐位置

| 需求 | 推荐 | 原因 |
| --- | --- | --- |
| 长流程可靠执行 | Temporal | durable execution 适合 Saga 编排、timer、重试、等待和恢复执行 |
| 人工审批和 BPMN 流程治理 | Camunda | BPMN 和 human task 更适合流程可视化、任务队列和运营治理 |
| 理解 AT/TCC/Saga 框架思想 | Seata | 适合作为分布式事务模式学习材料和受控服务内的局部方案 |
| 本地事实和消息发布一致 | Outbox | 业务表和事件表同事务提交，表达领域事实发布 |
| 业务资源预留 | TCC-like 设计 | Try 必须真实预留资源，Confirm/Cancel 必须有明确语义 |
| 资金事实和账本一致 | 状态机、幂等、账本、对账 | 资金事实必须由领域事实、不可变流水、会计分录和核对闭环保护 |
| 国际支付、供应商和清算场景 | 状态机、Outbox、Saga/Temporal、账本、对账、人工修复 | 外部系统不可控，未知状态和差错必须可解释、可追踪、可修复 |

## 不能替代什么

任何框架都不能替代以下设计：

- 领域服务所有权：订单、支付、退款、库存、供应商和账本服务必须拥有自己的事实和状态机。
- 幂等：外部调用、Activity、服务任务、消费者和人工修复动作都必须有业务幂等键。
- 状态机：迟到成功、迟到失败、重复回调、乱序事件和人工结论都必须经过合法迁移校验。
- 账本：资金事实必须有不可变流水、会计分录、币种、借贷方向和审计依据。
- Outbox：本地事实和事件发布必须用可靠机制绑定，不能依赖进程内发送。
- 对账：渠道、供应商、账本和订单事实必须周期性核对，差错必须进入修复闭环。
- 人工修复：自动流程无法收敛时，必须有工单、审批、证据、操作记录和复核。

## 危险误用

| 误用 | 后果 | 正确做法 |
| --- | --- | --- |
| 用了 Temporal 就不用状态机 | Workflow 重试或回调乱序时可能重复扣款、重复退款或覆盖成功事实 | Workflow 只编排，领域服务用状态机和幂等保护事实 |
| 把 workflow history 当账本 | 财务事实无法审计，渠道账单和内部账本无法解释 | 账本独立保存不可变资金事实，Temporal 只保存执行历史 |
| 用了 Camunda 就代表资金一致 | 人工任务完成被误当成支付、退款或账本完成 | 人工结论必须落领域事实，并经过账本和对账验证 |
| 把 BPMN 图当系统边界 | 流程图漂亮但服务所有权、幂等和补偿语义不清 | BPMN 只表达流程，领域服务仍负责业务命令和事实 |
| 用了 Seata 就能覆盖外部供应商 | 外部支付、供应商和清算系统不可控，AT/TCC 语义无法成立 | 外部系统用状态机、查询、回调、Saga、对账和人工修复 |
| 把 AT 当国际金融架构主线 | 受控数据库事务假设扩展到不可控跨机构链路 | 只在受控服务上下文讨论 AT，跨机构场景强调事实和对账 |
| 把 Saga 补偿当回滚 | 删除历史事实，资金和审计不可解释 | 补偿是新业务动作，例如退款、void、释放、罚金或人工处理 |

## 评审底线

- 不接受“用了 Temporal 就不用状态机”。
- 不接受“用了 Camunda 就代表资金一致”。
- 不接受“用了 Seata 就能覆盖外部供应商不可控问题”。
- 不接受“用了 AT 就能解决跨机构金融一致性”。
- 不接受“用了 TCC 就不需要 Try 真实预留资源”。
- 不接受“用了 Saga 就能回滚已经成功的外部事实”。
- 不接受“框架成功就代表账平”。
- 不接受任何框架绕过领域服务、幂等、账本、Outbox、对账和人工修复。

## 面试表达

Temporal、Camunda 和 Seata 不在同一个抽象层上。Temporal 更适合作为 durable execution 和 workflow orchestration，用来承载长流程 Saga、重试、timer 和人工等待；Camunda 更适合 BPMN、流程可视化、human task 和流程治理；Seata 更适合理解 AT、TCC、Saga 等分布式事务模式，在受控服务上下文中可以作为局部方案或学习材料。

金融级一致性不能靠框架名字兜底。真正的主线仍然是领域服务保存事实，状态机拒绝非法迁移，幂等阻止重复副作用，Outbox 可靠发布事件，账本记录资金事实，对账发现差异，人工修复处理自动流程无法收敛的异常。
