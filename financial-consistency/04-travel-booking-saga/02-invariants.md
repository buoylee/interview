# 02 正确性不变量

旅行组合预订不是单库事务问题。一次行程可能同时包含机票、酒店、租车、保险、优惠、支付授权、扣款、退款、罚金、补差和人工工单；其中一些事实发生在供应商系统或资金渠道中，本地数据库只能记录和解释它们，不能假装它们不存在。

本章定义系统必须长期保持的正确性不变量。它们的目标不是让所有子产品严格全有全无，而是保证组合订单状态、供应商事实、资金事实、消息事实、账本事实和人工修复事实最终能够互相解释。

## 事实来源

### 本地事实

- 组合订单事实由订单服务维护，包括用户、行程、子订单集合、组合状态、解释性备注和审计轨迹。
- 子订单事实由对应领域服务维护，包括产品类型、供应商、供应商请求号、报价快照、付款策略、可逆性快照和状态机历史。
- Outbox 事件必须和本地状态变更在同一服务事务中提交。没有本地事实的事件不能成为业务事实。
- broker offset 只能证明消费者读到某条消息，不能证明供应商确认、付款、退款、补偿或账务修复已经完成。

### 供应商事实

- 供应商确认号、出票号、酒店确认号、保险保单号、租车确认号、取消确认号和供应商账单是供应商事实。
- `SUPPLIER_UNKNOWN` 表示本地不知道供应商最终结果，不能直接当失败，也不能直接重试到产生重复预订。
- 供应商成功事实不能被后到达的失败回调覆盖。只能追加冲正、取消、退款、人工核实或供应商差错事实。
- 不可逆或半不可逆动作不能被数据库字段回滚掩盖。机票已出票、酒店已确认不可免费取消、保险已生效时，本地只能追加后续事实，不能删除原事实。

### 资金事实

- 支付授权、capture、void、refund、拒付、渠道对账单和资金流水是资金事实。
- `PAYMENT_UNKNOWN` 表示支付渠道结果未知，不能直接当失败，也不能立即发起新的无关联扣款。
- `REFUND_UNKNOWN` 表示退款提交后的渠道结果未知，不能重复提交外部退款；只能查询、对账、补记或人工处理。
- 同一授权只能 capture 或 void 一次。已 capture 的资金不能 void，只能 refund、冲正、补差或进入人工处理。

### 账本事实

- 会计分录是财务解释的最终内部事实，必须能解释订单应收、渠道实收、供应商应付、退款、罚金、补差、优惠和坏账。
- 账本不能用覆盖更新掩盖错误。修复必须通过反向分录、调整分录或审批后的补充分录完成。
- 账本事实不能单独改变订单或供应商事实，只能解释资金和履约事实。

## 组合订单不变量

- 同一用户、同一业务幂等键只能创建一个组合订单。
- 每个组合订单必须有可解释的子订单集合。组合状态必须能从子订单状态、供应商事实、资金事实和人工处理记录解释出来。
- 组合订单 `CONFIRMED` 必须要求机票和酒店等核心项全部成功，并且所需资金事实已达到该状态要求。
- 附加项失败不能把核心成功订单改成整体失败。核心行程成立时，租车或保险失败应进入解释性状态，而不是取消已经成功的机票和酒店。
- `CONFIRMED_WITH_ADDON_FAILURES` 必须明确哪些附加项失败、失败发生在哪个事实边界、是否已退款、是否尝试替代供应商、是否通知用户以及是否需要人工跟进。
- `PARTIALLY_CONFIRMED` 必须有 owner、下一步动作、截止时间和审计记录。它不能是无人负责的长期悬挂状态。
- 组合订单状态不能直接修改。任何状态变化都必须来自领域命令、状态机转换、对账结果、补偿动作或人工工单，并保留审计记录。

## 子订单和供应商不变量

- 每个子订单必须记录供应商、供应商请求号、报价快照、付款策略、可逆性快照、供应商状态和本地状态。
- 搜索价不能直接当成交价。扣款和确认必须基于仍有效的报价、重新定价结果或用户重新确认。
- 报价过期后不能按旧价扣款，除非用户重新确认价格、罚金、补差和取消规则。
- 供应商请求必须使用业务幂等键或供应商侧可识别的 request id，超时后先查询或对账，不能盲目重放。
- `SUPPLIER_UNKNOWN` 不能直接当失败。它必须进入查询、供应商对账、人工核实或受控补偿流程。
- 供应商已确认的子订单不能被本地失败状态覆盖。后续只能追加取消、退款、罚金、替代供应商或人工修复事实。
- 供应商事实与本地事实冲突时，系统必须保留冲突记录，并由对账或人工工单解释最终处理结果。

## 核心项和附加项不变量

- 机票和酒店是核心项，核心项未全部成功时组合订单不能进入 `CONFIRMED`。
- 租车和保险是附加项，附加项失败默认保留核心成功，除非用户事前选择严格全有全无且供应商动作仍可逆。
- 附加项退款、替代供应商、用户通知或人工处理必须能解释组合订单状态。
- 保险已生效不能被本地删除掩盖，只能追加取消、退款、作废或人工处理事实。
- 附加项失败后的补偿不能破坏核心项事实。不能因为租车失败而取消已出票机票，除非有明确用户授权和可解释的资金处理。

## 付款和退款不变量

- 付款策略必须按子产品记录，不能用全局付款策略覆盖所有子订单。机票、酒店、租车和保险可能分别要求预授权、即时 capture、延迟 capture 或供应商代收。
- 支付授权、capture、void 和 refund 必须有幂等键，并且幂等键必须绑定组合订单、子订单、金额、币种和业务目的。
- 同一授权只能 capture 或 void 一次。重复消息、重试或人工操作不能突破该约束。
- 已 capture 的资金不能 void，只能 refund、冲正或人工处理。
- `PAYMENT_UNKNOWN` 不能直接当失败。系统必须查询渠道、等待回调、对账或进入人工处理，避免重复扣款。
- `REFUND_UNKNOWN` 不能重复提交外部退款。系统必须用原退款请求号查询或对账，必要时通过人工工单处理。
- 退款金额、罚金和补差必须能被原支付、供应商规则、用户确认和会计分录解释。
- 组合订单显示给用户的金额必须能从子订单价格、税费、优惠、罚金、补差、退款和渠道结果解释出来。

## 消息和事件不变量

- Outbox 事件必须和本地事实在同一服务事务中提交，避免状态已变但事件丢失，或事件已发但状态未变。
- 消息消费者必须按业务幂等键去重。重复事件只能产生一次业务效果。
- broker offset 不能作为业务完成证明。它不能证明供应商确认、付款成功、退款成功、补偿完成或账本已平。
- 消息重放必须安全。重放只能恢复漏处理事实，不能重复 capture、重复退款、重复创建供应商订单或重复入账。
- 消息处理失败必须可观测，至少记录失败原因、重试次数、下一次处理时间、死信记录和 owner。

## 对账和修复不变量

- 组合订单、子订单、供应商事实、支付单、退款单、罚金、补差、会计分录和 Outbox 消费记录最终必须能解释彼此。
- 对账服务负责发现差异、分类差异、生成修复建议、触发受控领域命令或创建人工工单。
- `reconciliation-service` 不能直接修改订单、供应商、支付、退款或分录历史。它不是跨域后台脚本，不能绕过领域服务状态机和审计。
- 修复只能通过领域服务幂等命令、冲正、退款、补差、人工工单或审批完成。
- 差错必须有分类、owner、状态、修复动作、证据链接和审计记录。无人负责的差错不能被标记为已解决。
- 人工修复必须记录操作者、审批人、原因、前后事实、外部证据、影响金额和关联工单。
- 对账不能只看本地订单。它必须覆盖供应商订单、渠道支付、渠道退款、供应商账单和账本分录。

## 常见误区

- 把旅行组合预订设计成严格全有全无，忽略机票、酒店、保险等动作的不可逆性。
- 只有组合订单状态，没有子订单状态、供应商事实和资金事实。
- 把搜索价当最终成交价。
- 供应商超时后直接当失败，导致重复预订或错误退款。
- 支付未知后再次扣款，导致重复扣款。
- 退款未知后再次提交退款，导致重复退款。
- 机票已出票或保险已生效后用本地字段回滚。
- 附加项失败后取消核心成功行程，破坏用户已经成立的核心旅程。
- 已扣款失败后没有退款、罚金、补差或人工处理路径。
- 只对账本地订单，不对账供应商订单、支付、退款和账本。
- 让 `reconciliation-service` 直接改库，绕过领域服务、状态机和审计。
