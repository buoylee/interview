# Yjs 同步算法解析

## 核心概念

每个客户端维护两个东西：

1. **clock**：整数，从 0 开始，自己每产生一个操作 +1，收到别人的操作**不变**。所有操作共享一个计数器，不区分属性、不区分节点。

2. **状态向量**：一张表，记录"我已经收到了每个客户端的多少个操作"。只有两种情况会更新：自己产生操作（自己那行 +1）、收到别人的操作（别人那行更新）。

每个操作携带三个信息：

```
操作 = {
  id: (clientId, clock),              ← 谁的第几个操作
  content: { title: "方案一" },       ← 修改内容
  origin_state: { A: 1000, B: 5 }    ← 创建时的状态向量快照，表示"我见过谁的哪些操作"
}
```

## 并发判断

两个操作 X 和 Y 修改了同一个属性，Yjs 通过 origin_state 判断因果关系：

- **Y 见过 X**：Y 的 origin_state 中，X 创建者的计数覆盖了 X 的 clock → Y 因果上晚于 X → Y 直接覆盖 X
- **X 见过 Y**：反过来同理 → X 直接覆盖 Y
- **互相没见过**：真正的并发 → 走 LWW（先比 clock，再比 clientId）

## 例子

### 场景

- 用户 A 活跃了很久，做过 1000 个操作，之前在第 800 个操作时把 title 改成了"方案一"
- 用户 B 刚加入，做过 5 个操作

### 正常流程：B 先同步再修改 → B 的修改生效

```
B 加入白板，同步 A 的全部 1000 个操作
B 的状态向量变成：{ A: 1000, B: 5 }

B 修改 title，产生操作Y：
  id: (B, 5)
  content: { title: "方案二" }
  origin_state: { A: 1000, B: 5 }    ← B 见过 A 的前 1000 个操作

A 之前修改 title 的操作X：
  id: (A, 800)

判断：Y 的 origin_state 中 A 的计数是 1000，X 的 clock 是 800
      800 在前 1000 个里面 → Y 见过 X → Y 直接覆盖 X
结果：title = "方案二" ✓（B 的修改生效，与 clock 大小无关）
```

### 对比：B 没同步就修改 → B 的修改不生效

```
B 没有同步，直接修改 title，产生操作Y：
  id: (B, 5)
  content: { title: "方案二" }
  origin_state: { B: 5 }             ← 没有 A，B 不知道 A 的存在

A 之前修改 title 的操作X：
  id: (A, 800)
  origin_state: { A: 800 }           ← 没有 B，A 也不知道 B 的存在

判断：互相没见过 → 真正的并发 → 走 LWW
      X.clock=800 > Y.clock=5 → X 赢
结果：title = "方案一"（B 的修改不生效）
```

**两个例子的唯一区别是 B 是否先同步了 A 的操作。** 正常流程下 B 一定会先同步再操作，所以 origin_state 一定包含 A 的历史，不会走到 LWW。

## 重连同步

```
1. 客户端把自己的状态向量发给服务端：{ A: 50, B: 1000 }
2. 服务端对比后计算差量：A 缺第 51~200 个，C 缺全部 30 个
3. 服务端发送差量操作给客户端
4. 客户端应用后，状态向量更新为：{ A: 200, B: 1000, C: 30 }
5. 后续操作都"见过"所有人的历史，不会走到 LWW
```
