## A. 概念與關係

**Q1. 什麼是可觀測性?metrics / logs / traces 各回答什麼問題?**(→ 第 1 章)
> 三種信號:**metrics「有沒有事、趨勢如何」**(數字、可聚合、便宜、怕高基數);**logs「到底發生什麼」**(離散事件、細節最全、最貴);**traces「事卡在哪一段」**(一個請求的跨服務全鏈路)。三者靠 **`trace_id`** 串成一個畫面。可順口加第四支柱:continuous profiling(哪行程式碼燒資源)。

**Q2. 請在白板上畫出一套可觀測性架構。**(→ README + 第 2、4 章)
> 畫四階段管線:**產生(OTel SDK / agent)→ 採集處理(OTel Collector)→ 儲存(Prometheus / Tempo / Loki 或 ES)→ 視覺化(Grafana / Kibana)**,旁邊標「告警:Alertmanager」。強調 **OTel 是產生+採集的中立層,後端可換**。

**Q3. 為什麼有這麼多看起來重複的工具(Prometheus / Loki / Tempo / ES / Jaeger…)?**(→ README + 第 5 章)
> 因為**每種信號都要走完「產生→採集→存→看」四階段**,每階段各需一個工具,於是名字看起來重疊。再用「信號 × 階段」二維表把每個名字歸位,混亂就解了。

## B. OpenTelemetry

**Q4. OpenTelemetry 是什麼?解決什麼問題?**(→ 第 2 章)
> 廠商中立的「**產生 + 傳輸信號**」標準(API + SDK + OTLP 線格式 + 語意慣例)。解決「**埋點與後端綁死**」:程式只依賴 OTel,換後端只改設定、不動業務碼。身世:OpenTracing + OpenCensus 合併。

**Q5. OTel 的 API 和 SDK 差在哪?**(→ 第 2 章)
> **API 是 no-op 介面**(定義形狀,函式庫可放心依賴它埋點、沒裝 SDK 也零開銷);**SDK 是實作**(取樣、批次、匯出)。換後端只動 SDK/Collector,業務碼不變。

**Q6. OTel Collector 是什麼?為什麼要它?**(→ 第 2 章)
> 獨立進程,管線 **receiver → processor → exporter**。價值:**解耦**(應用只丟 OTLP 給本地 Collector,後端隨便換不重啟應用)、**集中加工**(批次、過濾敏感欄位、補環境標籤、**尾取樣**)、**減負**。

## C. 分散式追蹤(最常深挖)

**Q7. 一個請求跨好幾個服務,你怎麼知道它們是同一個請求?**(→ 第 3 章)
> 靠 **context propagation**:發送方把當前 span 的 `trace_id` + `span_id` **inject** 進請求 header(W3C **`traceparent`**),接收方 **extract** 出來,生的新 span 共用 `trace_id`、parent 指向上游。後端靠 `trace_id` + `parent_span_id` 重建整棵樹。

**Q8. `traceparent` 長什麼樣?B3 是什麼?**(→ 第 3 章)
> `00-{trace_id}-{parent_span_id}-{flags}`(版本-追蹤id-上游span-旗標,旗標含「已取樣」位)。**B3** 是 Zipkin 的舊式 header(`X-B3-TraceId` 等),對接舊系統時 OTel propagator 可同時支援 W3C 與 B3。

**Q9. 為什麼開了新執行緒/丟線程池後 trace 斷了?**(→ 第 3 章)
> 進程內 context 存在隱式上下文(Java ThreadLocal / Python contextvars / Go context),**新執行緒不會自動繼承**。要用 OTel 的 context 包裝在切換點手動傳遞。

**Q10. head-based 與 tail-based 取樣差在哪?**(→ 第 3 章)
> **Head**:請求一開始就決定留不留,簡單、開銷小,但可能丟掉出問題的那條。**Tail**:在 Collector 等整條 trace 跑完再決定,能「**錯誤/慢請求 100% 留、正常的只留 1%**」,但要緩存、較重。常見:head 粗篩 + tail 保底。

**Q11. trace 比「人工拼日誌」強在哪?**(→ 第 1、3 章)
> 純 log 下,一個請求的線索散在多個服務的日誌裡,要靠時間戳肉眼拼、跨服務還對不上;trace 用**同一個 `trace_id` 在請求發生當下就自動縫好**,附帶父子順序與各段耗時,瓶頸 span 直接標紅。

## D. 後端選型

**Q12. Prometheus 的 pull 模型?pull vs push 取捨?**(→ 第 4 章)
> Prometheus **定期主動抓**各目標的 `/metrics`(靠服務發現列出目標)。**Pull 優勢**:目標掛了抓不到=立刻知道 down、抓取頻率統一可控;**弱點**:短命批次任務來不及被抓(用 Pushgateway 補)。**Push**(OTel/StatsD)適合短命任務,但收集端可能被灌爆、難判斷「沒推=掛了還是沒事」。OTel(push)與 Prometheus(pull)用 Collector 橋接。

**Q13. Loki 和 Elasticsearch 存日誌差在哪?怎麼選?**(→ 第 4 章)
> **ES 把整條日誌全文建倒排索引**——任意搜尋極快但**貴**;**Loki 只索引標籤、本體壓縮丟物件儲存**——**便宜**,但只能先用標籤圈範圍再 grep(LogQL)。要任意全文搜尋/分析選 ES;要省成本、靠已知標籤定位、跟 Tempo/Prometheus 在 Grafana 關聯選 Loki。

**Q14. Jaeger、Zipkin、Tempo 怎麼選?**(→ 第 4 章)
> **Zipkin** 最老牌、簡單,舊系統常見;**Jaeger** 功能全、自帶 UI、雲原生主流;**Tempo** 最省成本(只用物件儲存、幾乎不建索引),但**只能用 `trace_id` 直接撈**,要靠 Loki/Prometheus 的 exemplar 先給你 `trace_id`。

**Q15. Grafana 和 Kibana 差在哪?**(→ 第 4 章)
> **Grafana 資料源無關**,能在一個面板同時放 metrics+logs+traces 做跨信號關聯;**Kibana 綁定 ES**,是 Elastic Stack 的眼睛。

## E. 生態系與 APM

**Q16. LGTM、ELK、SkyWalking 三套怎麼定位、何時選哪套?**(→ 第 5 章)
> **LGTM**(Loki+Grafana+Tempo+Mimir/Prometheus)=專用後端 + Grafana 統一關聯,省成本、雲原生、無鎖定。**ELK**(ES+Logstash+Kibana+Beats)=搜尋引擎為核心,全文搜尋強但重。**SkyWalking**=一體化 APM,Java 微服務開箱即用、有拓撲圖。選型收斂成一句:**埋點交給 OTel(中立可逆),後端按處境挑一套。**

**Q17. APM 是什麼?跟 OTel+後端的 DIY 路線差在哪?**(→ 第 6 章)
> APM 是**產品類別**:把 traces/metrics(/logs)打包 + 有主見的 agent + 拓撲圖 + 告警,用「開箱即用」換「彈性」。DIY(OTel+LGTM)則自由組裝、無鎖定但要懂較多。兩者可結合:OTel 埋點餵 APM 後端。

**Q18. SkyWalking 的架構?**(→ 第 6 章)
> **Agent**(自動採集 trace/metrics)→ **OAP**(大腦:接收、用 OAL 串流聚合算指標、自動畫服務拓撲、寫儲存)→ **Storage**(ES / BanyanDB)→ **UI**(拓撲圖/鏈路/告警)。OAP 是「邊收邊算」的串流分析引擎,也能收 OTLP。

**Q19.(高分題)SkyWalking / OTel 的 Java agent 怎麼做到零改碼自動埋點?**(→ 第 6 章)
> `-javaagent` 走 JDK 的 **`premain` + `java.lang.instrument`**,用 **ByteBuddy** 在**類別載入時改寫字節碼**:對 agent 外掛認得的**已知框架方法**(Spring 入口、JDBC、HTTP client…)織入 start/stop span,並對出站呼叫 inject context。**自寫業務方法不會自動有 span,要手動埋點。** OTel Java agent 同源,差在資料送 OTLP(不鎖定)。

## F. 開發者實務

**Q20. 身為開發者,可觀測性你負責哪塊?**(→ 第 7 章)
> 讓服務**發得出高品質信號**(埋點、加 span 屬性、**結構化日誌帶 `trace_id`**、暴露/自訂業務指標)並**能用它定位問題**;平台的部署/擴展由 SRE,取樣與告警策略一起定。

**Q21. 三種信號實務上怎麼串起來定位問題?**(→ 第 7 章)
> **single pane of glass**:metric 告警 → 儀表板點 exemplar(帶 `trace_id`)→ 跳 trace 找最慢的標紅 span → 用 `trace_id` 過濾跳到該服務該時段的 log → 讀出根因。串起來的線就是 **`trace_id`**。

**Q22. 自訂指標的三種型別?高基數要注意什麼?**(→ 第 1、7 章)
> **Counter**(只增:`orders_total`)、**Gauge**(可上下:`queue_length`)、**Histogram**(分佈算 p99:`request_duration`)。**高基數(如 user_id)別當 metrics 標籤**(會撐爆時間序列),但放 **span 屬性 / log 欄位 OK**。

## 一分鐘總綱(臨場開場白)

> 「可觀測性就三種信號:**metrics 知道有沒有事、logs 知道發生什麼事、traces 知道事在哪一段**,靠 `trace_id` 串成一個畫面。我會用 **OpenTelemetry 做中立埋點**,信號送進後端——可以是 **Grafana LGTM** 那種專用後端組合、**ELK** 那種搜尋引擎核心,或 **SkyWalking** 那種一體化 APM;埋點層中立化後,**後端是可換的決定**。身為開發者,我負責讓服務發得出帶 `trace_id` 的高品質信號,並能從告警一路追到出問題的那一行。」
