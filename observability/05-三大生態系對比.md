# 第 5 章 · 三大生態系對比

> 你最初的混亂——「為什麼有這麼多看似重複的工具」——根源在這裡:業界有**三套哲學不同的套裝**,每套都得自備「產生→採集→存→看」整條管線,所以名字才會看起來重疊。這章把三套擺一起,講清楚**它們為何重疊、各自取捨、何時選哪套**。

---

## 為什麼會「看似重複」?先講清這件事

回到 README 那張表:**每一種信號,都需要「產生→採集→存→看」四個階段各一個工具**。於是任何一套完整方案,都得湊齊一組工具。三大生態系不是「做不同的事」,而是「**用不同哲學做同一件事**」:

- **LGTM**:每種信號各用一個**專用、省成本**的後端,靠 Grafana 統一關聯。
- **ELK**:用**一個強大的搜尋引擎(ES)**為核心,什麼信號都塞進去搜。
- **SkyWalking**:**一體化 APM**,agent 自動抓,開箱即用,少操心組裝。

---

## 三套並排看

| | **Grafana LGTM** | **ELK / Elastic Stack** | **SkyWalking** |
|---|---|---|---|
| Metrics | Prometheus / **M**imir | Metricbeat → ES | OAP 內建(OAL 聚合) |
| Logs | **L**oki | **E**lasticsearch + **L**ogstash | (可接 ES,非主打) |
| Traces | **T**empo | Elastic APM | **主打**:agent 自動鏈路 |
| 視覺化 | **G**rafana | **K**ibana | 自帶 UI(拓撲圖) |
| 採集 | Alloy / Promtail / OTel | Beats / Logstash / OTel | **自家 agent(自動埋點)** |
| 哲學 | 專用後端 + 統一面板 | 搜尋引擎為核心 | 一體化 APM |
| 自己組裝程度 | 高(挑零件) | 中 | **低(開箱即用)** |

---

## LGTM:best-of-breed,用標籤與 trace_id 縫起來

名字就是縮寫:**L**oki + **G**rafana + **T**empo + **M**imir。(實務上 metrics 常直接用 Prometheus,Mimir 是它的長期儲存擴展。)

**哲學**:每種信號交給「為它而生、最省成本」的專用後端(logs→Loki、traces→Tempo、metrics→Prometheus/Mimir),再用 **Grafana 一個面板把三者關聯**。關聯的線就是前面講過的 **`trace_id` 與 exemplar**:
```
Grafana 面板:metric 圖上一個 exemplar 點(帶 trace_id)
  → 點開跳到 Tempo 的那條 trace
  → trace 裡某個 span → 跳到 Loki 過濾出該時段該服務的 log
```
**取捨**:彈性高、成本低、無廠商鎖定;代價是**你要自己挑零件、組管線**(雖然現在 Grafana 把整合做得很順)。**雲原生、K8s、成本敏感**的團隊愛用。

---

## ELK / Elastic Stack:以搜尋引擎為核心

組成記憶法 **ELK**:**E**lasticsearch(存+全文搜尋)+ **L**ogstash(收集與轉換管線)+ **K**ibana(UI)。再加 **Beats**(輕量採集器:Filebeat 收日誌、Metricbeat 收指標)。加上 Beats 後其實叫 **Elastic Stack**(官方已不只叫 ELK)。

```
應用日誌 → Filebeat → Logstash(grok 解析/清洗/打標) → Elasticsearch(索引) → Kibana(搜尋/儀表)
                      └ 也可不經 Logstash,Beats 直送 ES
```
- **Logstash**:重量級的「**收集 + 轉換**」管線,能 grok 正則解析非結構化日誌、改欄位、過濾。重,常被輕量的 Beats / Fluentd / Fluent Bit 取代部分角色。
- **Elastic APM**:Elastic 後來補的 traces 方案,讓 ELK 也能做三支柱。

**哲學**:把一切丟進一個**強大的全文搜尋引擎**,用搜尋解決所有問題。**日誌量大、需要任意全文搜尋與分析(也常被當搜尋/SIEM 平台)**的場景強。**取捨**:功能強、搜尋無敵;代價是**ES 吃資源、運維重、成本高**(索引全文的代價,見第 4 章)。

---

## SkyWalking:一體化 APM(你的 Java 主場)

**組成**:Agent(自動埋點)+ **OAP**(分析平台/大腦)+ Storage + 自帶 UI。它不是「一堆工具拼起來」,而是**一個完整產品**。(架構與 agent 字節碼注入內幕在第 6 章專講。)

**哲學**:**開箱即用**。掛上 agent,不改一行業務碼,就自動拿到:服務調用**拓撲圖**、鏈路追蹤、各服務/端點的指標、告警。**Apache 頂級專案,在 Java/Spring Cloud 微服務、中文社群特別流行**。

**取捨**:省事、開箱即用、拓撲圖直觀;代價是**比較一體化、沒 LGTM 那麼自由換零件**,且 agent 對非 Java 語言支援不如 Java 成熟。**團隊想「快速上一套 APM、少操心組裝」**就選它。

---

## 該怎麼選(面試講得出這套決策樹就夠了)

```
你的處境是?
├─ Java/Spring Cloud 微服務,想最快有「調用拓撲 + 鏈路」,少組裝
│     → SkyWalking(一體化 APM)
├─ 日誌量很大,需要任意全文搜尋/分析,或要當 SIEM
│     → ELK / Elastic Stack
├─ 雲原生/K8s,成本敏感,想要無廠商鎖定、信號自由組裝
│     → Grafana LGTM(+ OpenTelemetry 埋點)
└─ 都想要、又不想被綁死
      → 用 OpenTelemetry 埋點(中立層),後端先接一套、日後能換
```

**最關鍵的一句**:**先用 OpenTelemetry 做埋點(產生信號這層中立化),後端選哪套都行、日後還能換。** 這正是本文件以 OTel 為主軸的原因——它讓「選哪個生態系」變成一個**可逆**的決定,而不是綁死業務碼的賭注。

---

## 它們跟 OTel 的關係(收束回主軸)

三套現在**都相容 OpenTelemetry**,只是程度不同:
- **LGTM**:原生擁抱 OTLP,Grafana Alloy 就是 OTel Collector 的發行版,最契合。
- **ELK**:Elastic 也支援接收 OTLP,可用 OTel 取代部分 Beats/APM agent。
- **SkyWalking**:OAP 能**接收 OTLP**(metrics 等),也能與 OTel 資料互通;但它自家 agent 與 OTel agent 功能重疊,通常二選一。

所以「選生態系」≈「**選後端與 UI**」,而**埋點層交給 OTel 就對了**。

---

## 本章小結

- 三套之所以看似重複,是因為**每種信號都要走完「產生→採集→存→看」**,於是每套都得湊一組工具。
- **LGTM**=專用後端 + Grafana 統一關聯(省成本、雲原生);**ELK**=搜尋引擎為核心(全文搜尋強、重);**SkyWalking**=一體化 APM(開箱即用、Java 主場)。
- 選型用一句話收斂:**埋點交給 OTel(中立、可逆),後端按團隊處境挑一套。**
