# 第 3 章 · 分散式追蹤內幕

> 🔬 這章是面試最愛深挖的一塊:「**一個請求跨好幾個服務,你怎麼知道它們是同一個請求?context 怎麼傳過去的?**」能把這章畫在白板上,你在「分散式系統」的印象分會立刻拉高。

---

## 先把資料模型釘死:Trace 與 Span

- **Trace**:一個請求的完整路徑,有一個全域唯一的 **`trace_id`**。
- **Span**:路徑上的一段工作(一次 HTTP 處理、一次 DB 查詢、一次跨服務呼叫)。一個 trace 由很多 span 組成一棵樹。

每個 span 帶這些欄位(白板上要畫得出來):

```
span {
  trace_id      : 0af7651916cd43dd8448eb211c80319c   ← 整條 trace 共用,把所有 span 綁在一起
  span_id       : b7ad6b7169203331                   ← 這個 span 自己的 id
  parent_span_id: a1b2c3d4e5f60718                   ← 父 span 是誰(決定樹的形狀)
  name          : "POST /order"
  start / end   : 起訖時間 → 相減就是耗時
  attributes    : { http.method:"POST", user_id:"u_123", ... }  ← 可過濾的標籤
  events        : [ {t, "cache miss"}, ... ]         ← span 內的時間點事件
  status        : OK / ERROR
}
```

**三個 id 的關係決定一切**:
- `trace_id` 相同 → 屬於同一個請求。
- `parent_span_id` 指向誰 → 決定父子順序(誰呼叫誰)。
- 後端拿到一堆 span,**光靠這兩個欄位就能把樹重建出來**,不需要服務之間額外協調。

```
trace_id = 0af765...
span A  (parent=none)         GET /order            ← root span,網關
 └ span B (parent=A)          訂單服務.create
    ├ span C (parent=B)       SELECT inventory      ← DB
    └ span D (parent=B)       呼叫 支付服務          ← 跨服務的「客戶端」span
       └ span E (parent=D)    支付服務.charge        ← 跨服務的「伺服器」span,在另一台機器!
```

注意 D→E:**跨服務時,span E 是在另一個進程/機器產生的,但它的 `parent_span_id` 卻指向 D**。這怎麼做到的?答案就是下一節——**context propagation**。

---

## 核心內幕:Context 怎麼跨服務傳 🔬

問題:服務 A 呼叫服務 B,B 在**另一個進程**,它怎麼知道「我這個 span 的父親是 A 的某個 span、我們屬於同一個 trace」?

答案:**A 在發出請求時,把當前 span 的身分塞進請求的 header;B 收到後從 header 把它讀出來,接著往下生 span。** 這對動作叫 **inject(注入)** 和 **extract(抽取)**。

```
服務 A (span D 正在進行)
   │  準備呼叫 B,OTel 把「當前 context」inject 進 HTTP header:
   │     traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
   │                  └ver┘ └──────── trace_id ────────┘ └── span_id(=D)┘ └flags┘
   ▼
HTTP 請求 ───────────────────────────────────────────────►  服務 B
                                                              │ OTel 從 header extract:
                                                              │  讀到 trace_id 與 parent=D
                                                              ▼
                                                          建立 span E
                                                          {trace_id=0af765..., parent=D}
```

### W3C Trace Context:那個 header 的標準格式
跨服務傳的就是一個 HTTP header,叫 **`traceparent`**(W3C 標準,現在的業界共識):

```
traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
             │   │                                │                │
           版本  trace_id(16 bytes)              parent span_id    trace-flags
                                                  (8 bytes)         (01 = 已取樣,要記錄)
```

- 還有一個 `tracestate` header,給各廠商放自己的附加資料。
- **舊系統會看到別的格式**:Zipkin 用的是 **B3 headers**(`X-B3-TraceId`、`X-B3-SpanId`、`X-B3-Sampled`)。面試若提到「跟舊系統對接」,答:OTel 的 propagator 可設定同時支援 W3C 與 B3。

### 不同傳輸協定,注入的位置不同
「塞進 header」這件事按協定而異,但概念一樣(把 context 夾帶在請求裡):
- **HTTP**:放進 HTTP headers(`traceparent`)。
- **gRPC**:放進 metadata。
- **訊息佇列(Kafka/RabbitMQ)**:放進 message headers——所以**非同步、跨 MQ 的鏈路也能串起來**(生產者注入,消費者抽取)。

### 進程內又是怎麼傳的?(同一個服務裡)
跨服務靠 header;**同一個進程內**,當前 span 是放在一個「**隱式上下文**」裡,沿著呼叫鏈自動帶:
- **Go**:顯式用 `context.Context` 一路傳下去。
- **Java**:`ThreadLocal`(OTel 用 `Context` + `Scope`);跨執行緒要小心手動傳遞。
- **Python**:`contextvars`(對 async/await 友善)。
- **Node.js**:`AsyncLocalStorage`。

**面試陷阱題**:「為什麼我開了新執行緒/丟到執行緒池後 trace 就斷了?」——因為 context 存在 ThreadLocal,沒跟著新執行緒走;要用 OTel 提供的 context 包裝(`Context.wrap`)或在切換點手動傳遞。能答出這題很加分。

## 兄弟 span 怎麼排序?時鐘偏移的坑 🔬

回到資料模型:`parent_span_id` 只定義**樹的形狀**(誰是誰的爸),它**不帶任何兄弟排序資訊**。那同一個 parent 底下的好幾個子 span,後端怎麼知道誰先誰後?

答案:**靠 `start` 時間戳排,沒有專門的順序欄位**——span 裡並沒有 `child_index` / `sequence` 這種東西。後端拿到一堆 span,做兩件事:

1. 用 `parent_span_id` 把**層級**搭出來(縱向:誰在誰底下)。
2. 對「同一個 parent 的子 span」**按 `start` 由小到大排序**,由上而下畫成 waterfall。

```
parent B
 ├ C  start=10ms   ├────┤              ← start 小,排上面
 └ D  start=25ms            ├──────┤   ← start 大,排下面
       (沒有 sequence 欄位,純看 start;parent_span_id 只定層級,不定先後)
```

### 排序只是副產品,真正的價值是看出「串行 vs 並行」
有了每段的 `start / end`,兄弟之間的時間軸關係一眼就分得出來:

- C 的 `end` < D 的 `start` → **串行**(做完 C 才做 D)。
- C、D 的時間區間**重疊** → **並行/併發**(同時打兩個下游、`Promise.all`、goroutine 併發)。

```
串行:  C ├────┤
        D        ├──────┤        ← D 等 C 做完才開始

並行:  C ├────────┤
        D    ├──────────┤        ← 兩條 bar 疊在一起 = 同時在跑
```

這是 trace 完勝「人工拼日誌」的關鍵之一:並行關係用肉眼拼日誌幾乎看不出來,waterfall 上兩條 bar 一疊就懂。

### 跨機器的兄弟:時鐘偏移(clock skew)會讓排序騙人
**這是面試陷阱**。兄弟若在**同一個進程**裡產生,共用同一個時鐘,排序可靠。但若兩個子 span 在**不同機器**上產生(各記各的 wall clock),機器間時鐘有漂移,就會出事:

- 兩個 sibling 的先後**被排反**;
- 更經典的鬼畫面:**子 span 看起來比父 span 還早開始**(子在另一台、時鐘快了幾十 ms,例如前面 D→E 的 E 在支付服務那台機器)。

對策:
- NTP/PTP 對時只能**縮小**偏移,**不能消除**。
- 所以各家 UI(Jaeger/Tempo)用**啟發式修正**:既然因果上「子一定在父的區間內」,就把越界的子 span 夾回父的 `[start, end]` 再畫。
- 一句話:**真正可信的因果靠 `parent_span_id`(指針),時間戳只是輔助排序、且跨機不可全信。**

> 比 span 更細的順序,看 span 內的 `events`(各帶時間戳),那是 span *內部*的時間軸,跟「兄弟 span 之間」是兩個層次。

**白板一句話**:`parent_span_id` 定樹形,`start` 定兄弟先後;同進程可信,跨機器要防時鐘偏移,真因果永遠以 parent 指針為準。

## 取樣(Sampling):省錢與保真的取捨 🔬

全部請求都記 trace,量太大也太貴。於是要**取樣**——只留一部分。兩種策略,面試常考差異:

| | **Head-based(頭部取樣)** | **Tail-based(尾部取樣)** |
|---|---|---|
| 何時決定 | **請求一開始**就擲骰子決定留不留 | **等整條 trace 跑完**、看過全貌再決定 |
| 在哪做 | 在應用 SDK(`traceparent` 的 flags 帶著走) | 在 **OTel Collector**(要先暫存整條 trace) |
| 優點 | 簡單、開銷小、決定一致(整條要嘛全留要嘛全丟) | 能「**錯誤和慢請求 100% 留、正常的只留 1%**」 |
| 缺點 | 可能剛好把出問題的那條丟掉 | 要緩存、耗資源,部署較複雜 |

**白話**:head-based 像進場前抽籤,公平但盲目;tail-based 像看完比賽再決定錄影留哪段,聰明但要先全部錄下來暫存。**生產級常見做法:應用端 head-based 粗篩,Collector 端 tail-based 保證「壞的都留下」。**

> 那個 `traceparent` 結尾的 `-01`,最後一個 bit 就是「**這條已被取樣、請記錄**」的旗標,跟著請求一路傳,確保整條 trace 的取樣決定一致。

## 回扣第 1 章:為什麼 trace 完勝「人工拼日誌」

```
純日誌世界:                         有 trace:
服務A.log  ─┐                       同一個 trace_id 自動把五段縫成一棵樹,
服務B.log  ─┤  你用時間戳+肉眼       附帶父子順序與每段耗時。
服務C.log  ─┤  硬拼,跨服務還對不上   你只要在 UI 點開,瓶頸 span 直接標紅。
服務D.log  ─┤
服務E.log  ─┘
```

trace 的本質,就是**靠 `trace_id` + context 傳播,把散落各服務的因果關係,在請求發生的當下就自動縫好**。

---

## 本章小結

- Span 靠 **`trace_id`(綁同一請求)+ `parent_span_id`(定父子)** 兩個欄位,就能讓後端重建整棵樹。
- **兄弟 span 沒有順序欄位**:同一 parent 下的先後靠 **`start` 時間戳**排;同進程可信,**跨機器要防時鐘偏移**(子可能看起來比父早開始),真因果以 `parent_span_id` 為準。
- 跨服務的關鍵是 **context propagation**:發送方 **inject** 到 header(W3C `traceparent`)、接收方 **extract**。舊系統用 B3。
- 進程內靠隱式上下文(ThreadLocal / contextvars / Go context);**切執行緒會斷**是經典坑。
- **取樣**:head-based(開頭擲骰、便宜)vs tail-based(看完再決定、能保證留下錯誤與慢請求)。
- 下一章:trace/metric/log 各自要存去哪——**後端怎麼選**。
