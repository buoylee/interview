# 第 7 章 · 開發者實際要做什麼

> 你說自己是 SWE 不是運維——這章就把鏡頭拉回你身上:整個體系裡,**哪幾件事是你會親手做的**?其餘(部署 Prometheus 叢集、調 ES 分片、設保留期)交給運維。記住一句:**運維搭舞台,你負責「讓程式發得出信號」和「會看信號」。**

開發者只碰四件事:**埋點、帶 trace_id 的結構化日誌、暴露/自訂指標、會讀會關聯。**

---

## 1. 埋點(Instrumentation):讓程式發得出 trace

優先順序由省力到費力:

1. **先開自動埋點**(投報率最高):掛上 OTel 或 SkyWalking agent,零改碼先拿到一張全鏈路圖與服務拓撲。**這步通常你只要加一個啟動參數。**
2. **對業務關鍵操作補手動 span**:自動埋點只認得框架方法,**你自寫的核心邏輯(下單、扣款、對帳)不會自動有 span**,要手動補:

   ```python
   # OTel Python 範例
   from opentelemetry import trace
   tracer = trace.get_tracer(__name__)

   with tracer.start_as_current_span("create_order") as span:
       span.set_attribute("order.id", order_id)
       span.set_attribute("user.id", user_id)      # 加屬性,之後能用它過濾 trace
       do_business_logic()
   ```
3. **加有用的屬性(attributes)**:`order_id`、`user_id`、`tenant`……讓你日後能在 UI 用「`order.id = A-998`」直接撈出那條 trace。
   - ⚠️ **別把超高基數的東西塞進 metrics 標籤**(第 1 章的坑);但塞進 **span attributes 或 log 欄位是 OK 的**——它們本來就是高基數設計。這條區分面試常考。

---

## 2. 結構化日誌 + 注入 trace_id(最關鍵的接縫)🔬

這是**開發者最該做對的一件事**,也是三支柱能串起來的接縫。

- **結構化**:輸出 JSON 而非純字串,欄位化(`level`、`msg`、`order_id`…)。
- **注入 `trace_id`**:讓**每一條 log 自動帶上當前的 `trace_id` / `span_id`**。這樣你看到一條錯誤日誌,就能一鍵跳到它所屬的完整 trace;反之從 trace 也能跳到那段 log。

各語言怎麼把 trace_id 塞進每條 log(機制就是第 3 章的「進程內隱式上下文」):

| 語言 | 機制 |
|---|---|
| Java | **MDC**(Mapped Diagnostic Context);OTel 提供 log appender 自動注入 `trace_id`/`span_id` |
| Python | `contextvars` + logging filter;OTel logging instrumentation 自動注入 |
| Go | 從 `context.Context` 取出 span context,寫進結構化 logger 欄位 |

```json
{"ts":"...","level":"ERROR","msg":"charge failed","order_id":"A-998",
 "trace_id":"0af7651916cd43dd8448eb211c80319c","span_id":"b7ad6b71"}
                                                  ▲ 有了這個,log ↔ trace 一鍵互跳
```

> 你 repo 的 `log/01-日志格式规范.md`、`log/02-FastAPI日志配置.md` 就是這一層的落地。回去看時,重點檢查一條:**每條 log 有沒有帶 `trace_id`**。沒有的話,三支柱就串不起來。

## 3. 暴露 / 自訂指標(Metrics)

- **暴露端點**:讓服務有 `/metrics`(Prometheus 抓)或用 OTel metrics push。框架通常有現成套件(Spring 用 Micrometer、FastAPI 用 prometheus-client / OTel)。
- **自訂業務指標**:除了系統指標,你常要加業務指標。三種型別要分清(面試會問):

  | 型別 | 用途 | 例子 |
  |---|---|---|
  | **Counter**(計數器) | 只增不減的累計 | `orders_total`、`errors_total` |
  | **Gauge**(量表) | 可上可下的瞬時值 | `queue_length`、`active_connections` |
  | **Histogram**(直方圖) | 分佈,算 p50/p95/p99 | `request_duration_seconds` |

- **方法論**:對外部服務盯 **RED**(Rate / Errors / Duration),對資源盯 **USE**(第 1 章)。建儀表板與告警時照這個來,不會漏。

## 4. 會讀、會關聯(single pane of glass)🔬

**「會看」本身就是一項開發技能**。整個體系的回報,就在這條「從告警一路點到出問題那行程式」的路徑上:

```
① metric 告警:支付服務 p99 > 2s、錯誤率飆高
        │  打開 Grafana 儀表板,在延遲圖上點一個 exemplar(帶 trace_id)
        ▼
② trace:這條 trace 攤開,某個 span「呼叫第三方支付」佔了 1.8s,標紅
        │  用該 span 的 trace_id 過濾日誌
        ▼
③ log:那段時間支付服務狂噴 "connection pool exhausted"
        │
        ▼
④ 定位:連線池打滿 → 回去看是哪段程式沒釋放連線。收工。
```

這就是 **single pane of glass(單一視窗)**:三種信號用 `trace_id` / exemplar 串在一個畫面,**不用在五個系統之間切換複製貼上**。LGTM 在 Grafana 裡做這件事、SkyWalking 在自家 UI 做、ELK 在 Kibana 做——形式不同,**內核都是 `trace_id` 這條線**。

要練的具體動作:
- 在儀表板上看懂 RED 三圖、找出異常時段。
- 從圖上的 exemplar 或一條 log 跳進一條 trace。
- 在 trace 的甘特圖/火焰圖上,一眼找出最長/標紅的 span。
- 從那個 span 跳到對應時段、對應服務的 log,讀出根因。

**面試若問「可觀測性你負責哪塊」**:答「我負責**讓服務發得出高品質信號**(埋點、結構化日誌帶 trace_id、業務指標)**並能用它定位問題**;平台的部署與擴展由 SRE/平台團隊,我們一起定取樣與告警策略。」——這個答法精準、不越界、顯示你懂邊界。

## 本章小結

- 開發者只碰四件事:**埋點(自動 + 業務關鍵手動)、帶 `trace_id` 的結構化日誌、暴露/自訂指標、會讀會關聯**。
- 最關鍵的接縫是 **每條 log 帶 `trace_id`**——沒有它,三支柱串不起來。
- 指標要分清 Counter / Gauge / Histogram,照 RED / USE 設計。
- 回報全在 **single pane of glass**:從告警→trace→log,一路點到出問題那行。
