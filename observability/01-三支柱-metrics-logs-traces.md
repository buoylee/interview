# 第 1 章 · 三支柱:Metrics / Logs / Traces

> 整個可觀測性體系,底層只有三種「信號(signal)」。所有工具都是在搬運這三種信號之一。先把這三個分清楚,後面所有名字才有歸宿。

---

## 一句話區分

| 信號 | 回答的問題 | 一句話 |
|---|---|---|
| **Metrics**(指標) | 「**有沒有事?趨勢如何?**」 | 隨時間變化的**數字**,可聚合,便宜 |
| **Logs**(日誌) | 「**到底發生了什麼?**」 | 一條條帶時間戳的**離散事件**,細節最全,最貴 |
| **Traces**(追蹤) | 「**事情卡在哪一段?**」 | **一個請求**走過所有服務的完整路徑 |

記憶法:**儀表板上的紅燈是 metrics,你點進去看的細節是 logs,你想知道這個請求為什麼慢就看 traces。**

---

## Metrics:數字與趨勢

**是什麼**:在固定時間點對某個量的測量,存成「時間序列(time series)」。例如 `http_requests_total`、`cpu_usage`、`order_amount`。每個點是 `(時間戳, 數值, 一組標籤)`。

```
http_requests_total{method="POST", path="/order", status="500"}  →  1247 (at 10:00:00)
                    └────────────── 標籤(labels)──────────────┘     └ 值 ┘
```

**特性**:
- **可聚合**:可以求和、平均、百分位(p99 延遲),這是它最強的地方。
- **便宜**:一條時間序列無論一天來幾百萬個請求,存的只是定期取樣的數字,儲存量固定。
- **低基數(low cardinality)友善**:標籤的組合數要可控。`status` 只有幾種值沒問題;但如果你把 `user_id` 當標籤,幾百萬用戶 = 幾百萬條時間序列,會把儲存撐爆。**這條規則面試常考**。

**弱點**:它只告訴你「p99 延遲漲到 2 秒」,**不能告訴你是哪一個請求、為什麼**。要往下查就得靠 logs / traces。

**開發者常用的兩套指標方法論**(面試可順口提):
- **RED**(面向請求的服務):**R**ate(每秒請求數)、**E**rrors(錯誤率)、**D**uration(延遲分佈)。
- **USE**(面向資源):**U**tilization(使用率)、**S**aturation(飽和/排隊)、**E**rrors。

---

## Logs:事件與細節

**是什麼**:程式在某個時刻輸出的一條離散記錄。最原始是純文字,**現代做法是結構化日誌(structured logging)**——輸出 JSON,欄位化:

```json
{"ts":"2026-06-18T10:00:01Z","level":"ERROR","msg":"payment failed",
 "order_id":"A-998","user_id":"u_123","trace_id":"0af7651916cd43dd","err":"timeout"}
```

**特性**:
- **細節最全**:可以塞任意上下文(哪個訂單、哪個用戶、什麼錯誤)。排查具體事件時不可取代。
- **高基數天然**:每條都不一樣,正是它的價值。
- **最貴**:量大(每請求可能十幾條)、又要能搜尋。儲存與索引成本是三者最高的——第 4 章 Loki vs ES 就是在解這個成本問題。

**結構化 vs 純文字**:純文字 `"payment failed for order A-998"` 機器難解析;JSON 讓你能用 `order_id="A-998"` 精準過濾。**面試問「日誌規範你怎麼做」,先答結構化 + 統一欄位 + 帶 `trace_id`。**(你 repo 的 `log/01-日志格式规范.md` 就是這層的落地。)

**關鍵伏筆**:每條 log 都帶上 `trace_id`,你才能從「這條錯誤日誌」一鍵跳到「這個請求的完整 trace」。這是三支柱能串起來的接縫,第 7 章詳談。

---

## Traces:一個請求的全鏈路

**是什麼**:在微服務世界,一個用戶請求會穿過好幾個服務(網關→訂單→庫存→支付)。**Trace 就是這一個請求走過所有服務的完整路徑**,由一串 **span** 組成。

```
Trace (trace_id = 0af765...)        ← 整個請求
└─ span: GET /order        [120ms]  ← 網關
   └─ span: 訂單服務.create [100ms]
      ├─ span: SELECT 庫存   [10ms]  ← DB 查詢
      └─ span: 呼叫 支付服務  [80ms]  ← 跨服務!瓶頸在這
         └─ span: 第三方支付  [75ms]
```

每個 **span** = 一段有名字、有起訖時間的工作,記著:`trace_id`(整條共用)、`span_id`、`parent_span_id`、名稱、耗時、屬性(attributes)、狀態。把這些 span 按父子關係組起來就是一棵樹。

**特性**:
- **看因果與延遲分佈**:一眼看出 120ms 裡 80ms 花在支付,75ms 又花在第三方——**跨服務的瓶頸定位,只有 trace 做得到**。
- metrics 說「慢」、logs 說「這條請求報了什麼錯」,但**「慢在哪一段、誰拖累誰」是 trace 的主場**。

**為什麼 trace 比 log 更能定位跨服務問題**(你上次問的對照):
- 在純 log 世界,一個請求的線索散落在 5 個服務的 5 份日誌裡,你得靠時間戳和肉眼去「拼」,跨服務還會對不上。
- trace 用**同一個 `trace_id` 自動把這 5 段縫起來**,並保留父子順序與各段耗時。等於有人替你把散落的日誌拼成了一張時間軸。
- 第 3 章講的「context 怎麼跨服務傳」,就是這個 `trace_id` 能跟著請求跑的機制。

## 三者如何互補(也是面試金句)

```
告警響了(metric: p99 延遲 > 2s)
   ↓ 點進儀表板,找到一個慢的樣本請求(exemplar 帶 trace_id)
看 trace(哪一段慢:支付服務的某 span 80ms)
   ↓ 用該 span 的 trace_id 過濾
看 logs(那段時間支付服務報 "connection pool exhausted")
   ↓
定位:連線池打滿。三種信號各補一塊,缺一不可。
```

- **Metrics**:廣度與趨勢,便宜,適合儀表板與告警,但說不出細節。
- **Logs**:深度與細節,貴,適合查具體事件,但跨服務要靠人拼。
- **Traces**:把一個請求的跨服務路徑縫起來,專治「慢在哪/誰拖累誰」。

**串起三者的線就是 `trace_id`**:logs 帶它、traces 由它定義、metrics 用 exemplar 指向它。記住這條線,第 7 章的「single pane of glass」就通了。

## (補充)正在成形的第四支柱:Continuous Profiling

面試講完三支柱,可以順口加一句顯示你跟得上趨勢:
> 「現在業界把**持續性能剖析(continuous profiling)**當第四支柱——回答『**到底是哪一行程式碼在燒 CPU/記憶體**』。工具像 Pyroscope、Parca,OpenTelemetry 也已經把 profiling 納入信號。」

三支柱定位「哪個服務、哪一段、什麼事件」,profiling 再往下鑽到「哪一行程式碼」。知道有這層即可,不是本文件重點。

---

## 本章小結

- 只有三種信號:**metrics(有沒有事)、logs(發生什麼事)、traces(事在哪一段)**。
- 各自的取捨:metrics 便宜怕高基數、logs 最貴最全、traces 專治跨服務。
- 它們靠 **`trace_id`** 串成一個畫面。
- 下一章:這三種信號要怎麼「產生」——**OpenTelemetry** 就是那個統一標準。
