# 第 4 章 · 後端怎麼選(選型決策層)

> 信號用 OTel 產生並送出後,要找地方**存起來、查得到、看得見**。每種信號各有幾個候選後端,它們的差別就是面試的「選型理由」。這章把你列的名字一個個歸位,並給出取捨。

整章對著這張表看:

| 信號 | 候選後端 | 一句話差異 |
|---|---|---|
| Traces | **Jaeger** / **Zipkin** / **Tempo** | 老牌自帶 UI vs 最簡單 vs 最省成本 |
| Metrics | **Prometheus**(+ Thanos/Mimir) | **pull 模型**是它的靈魂 |
| Logs | **Elasticsearch** / **Loki** | 全文索引貴而強 vs 只索引標籤、便宜 |
| 視覺化 | **Grafana** / **Kibana** | 多信號統一面板 vs ES 專屬 |

---

## Traces 後端:Jaeger / Zipkin / Tempo

| | **Zipkin** | **Jaeger** | **Tempo**(Grafana) |
|---|---|---|---|
| 出身 | Twitter,最老牌 | Uber → CNCF 畢業 | Grafana Labs |
| 語言 | Java | Go | Go |
| 儲存 | 記憶體 / ES / Cassandra | 記憶體 / ES / Cassandra / Badger | **物件儲存(S3/GCS)** |
| 特點 | 簡單、輕、生態老 | 功能全、自帶查詢 UI、社群大 | **超省成本**:不建索引 |
| UI | 自帶 Zipkin UI | 自帶 Jaeger UI | **沒有自己的 UI,用 Grafana** |

**選型理由(面試講這幾句)**:
- **Zipkin**:最早的開源分散式追蹤(Google Dapper 論文的開源實作之一),簡單、輕量,老系統常見;新專案少選了。
- **Jaeger**:CNCF 畢業專案,功能完整、自帶好用 UI,**雲原生環境最常見的獨立 trace 後端**。現在也全面相容 OTLP。
- **Tempo**:Grafana 的解法,**最大賣點是便宜**——它**只用物件儲存、幾乎不建索引**:你不能用任意條件搜 trace,只能**用 `trace_id` 直接撈**。
  - 「那我怎麼先知道 `trace_id`?」→ 從 **metrics 的 exemplar 或 logs 裡拿到 `trace_id`,再來 Tempo 撈整條**。這正是 LGTM 套件「三信號互相關聯」的設計(第 5 章)。
  - 取捨一句話:**Tempo 用「放棄全文檢索 trace」換「儲存成本極低」**,前提是你有 Loki/Prometheus 幫你找到入口 `trace_id`。

## Metrics 後端:Prometheus 與它的 pull 模型 🔬

**Prometheus** 幾乎是 metrics 的事實標準:一個 TSDB(時間序列資料庫)+ 查詢語言 **PromQL** + 抓取器 + 告警(Alertmanager)。它最該講清楚的是 **pull 模型**。

### Pull vs Push——面試高頻題

```
Pull(Prometheus 預設):              Push(OTel / StatsD):
                                     
 ┌────────────┐  定期主動來抓        ┌────────┐  主動推出去
 │ Prometheus │ ───GET /metrics──►   │  App   │ ───metrics──► 收集端
 └────────────┘     每 15s            └────────┘
   它掌握「該抓誰」                     App 掌握「往哪推」
```

| | **Pull(拉)** | **Push(推)** |
|---|---|---|
| 誰主動 | 監控系統定期來抓應用的 `/metrics` | 應用主動把指標推給收集端 |
| 代表 | **Prometheus** | OTel(OTLP)、StatsD、GraphIndia |
| 怎麼發現目標 | **服務發現**(K8s/Consul)自動列出要抓的端點 | 應用得知道往哪推 |
| 天然優勢 | **目標掛了立刻看得出**(抓不到=down);抓取方統一控制頻率 | 適合**短命任務**(批次 job 跑完就沒了,來不及被抓) |
| 弱點 | 短命任務抓不到(要 **Pushgateway** 補) | 收集端可能被流量灌爆,且難判斷「誰沒推=掛了還是沒事」 |

**Prometheus 與 OTel 的橋接**(這題能答出來很加分):OTel 預設是 **push(OTLP)**,Prometheus 預設是 **pull**,兩者哲學相反。實務上的接法:
- 讓應用暴露 `/metrics`,**OTel Collector 用 Prometheus receiver 去抓**,再轉送;或
- 用 **Prometheus remote write**,把 OTLP metrics 寫進 Prometheus 相容後端。
- 一句話:**OTel 管「產生」,Prometheus 管「存與查」,中間用 Collector 橋接 pull/push 的差異。**

### 為什麼要 Thanos / Mimir / VictoriaMetrics(知道即可,屬運維)
單機 Prometheus 存不下長期資料、也不好做跨多台聚合。**Thanos / Mimir / Cortex / VictoriaMetrics** 是在 Prometheus 之上做**長期儲存 + 水平擴展 + 多叢集統一查詢**的方案。LGTM 裡的 **M** 就是 Mimir。面試知道「它們是 Prometheus 的長期儲存/擴展層」即可。

## Logs 後端:Elasticsearch vs Loki——成本對決 🔬

兩者都存 logs,但**索引哲學相反**,這是面試最值得講的對比。

| | **Elasticsearch(ELK)** | **Loki(Grafana)** |
|---|---|---|
| 索引什麼 | **整條日誌內容全部建倒排索引** | **只索引標籤(labels)**,日誌本體壓縮丟物件儲存 |
| 能怎麼查 | 任意全文搜尋(`error AND payment`),極快 | 先用標籤選出日誌流,**再在流裡 grep**(LogQL) |
| 成本 | **高**(索引很佔空間與 CPU) | **低**(只索引少量標籤,本體放 S3) |
| 自我定位 | 強大的搜尋引擎 | 「**Prometheus for logs**」 |
| 配套 | Logstash/Beats 收集、Kibana 看 | Promtail/Alloy 收集、Grafana 看 |

**取捨一句話**:
- **ES**:你需要**對任意欄位、任意關鍵字做臨時全文搜尋**(像查案),願意付索引的錢 → 選 ES。
- **Loki**:你的日誌大多**靠已知標籤(service、level、namespace)圈定範圍**,要的是**省成本**、又想跟 Tempo/Prometheus 在 Grafana 裡用 `trace_id` 串起來 → 選 Loki。

**設計同源的妙處**:Loki 故意模仿 Prometheus 的標籤模型(`{service="order", level="error"}`),所以**同一套標籤思維可同時用在 metrics 和 logs**,在 Grafana 裡切換信號時心智一致——這是 LGTM 套件的設計巧思。

## 視覺化:Grafana vs Kibana

| | **Grafana** | **Kibana** |
|---|---|---|
| 綁定 | **資料源無關**,插上各種後端 | **ES 專屬**(Elastic Stack 的 UI) |
| 強項 | **一個面板同時放 metrics+logs+traces**,跨信號關聯 | ES 的日誌搜尋、Discover、APM 視覺化 |
| 適合 | 多後端、多信號的統一觀測台 | 已經 all-in ES/Elastic Stack |

**一句話**:**Grafana 是「多信號統一面板」的代名詞**——它能同時連 Prometheus(metrics)、Loki(logs)、Tempo(traces),讓你在一個畫面從 metric 點到 trace 再點到 log。**Kibana 則和 Elasticsearch 綁定**,是 ELK 的眼睛。

## 本章小結

- **Traces**:Jaeger(功能全、自帶 UI、雲原生主流)/ Zipkin(老牌簡單)/ Tempo(超省成本,但要靠別人給 `trace_id`)。
- **Metrics**:Prometheus,靈魂是 **pull 模型**(目標掛了立刻看得出;短命任務要 Pushgateway);OTel 是 push,靠 Collector 橋接。
- **Logs**:**ES 索引全文(強而貴)vs Loki 只索引標籤(便宜、跟 Prometheus 同思維)**。
- **視覺化**:Grafana 多信號統一、資料源無關;Kibana 綁 ES。
