# 00 · 执行模型 —— 跑起来时机器到底在干什么

> 这是全课的**地基**。前面的章节讲「用什么、怎么选」，但如果你不知道一行并发代码跑起来时，**CPU、操作系统、Python 解释器各自在干什么**，那些选型规则就只是结论，原理落不了地。本章用一条连贯的主线，从 CPU 一路讲到协程，把散落的机制缝成一张全局图。

## 1. 核心问题

1. 一个「线程」「进程」在操作系统眼里到底是什么实体？

3. 「用户态 / 内核态」「系统调用」是什么？为什么一次 I/O 要进内核？

6. 把这些串起来：你的一行代码，从敲下去到在 CPU 上跑完，经过了哪些层？

## 2. 直觉理解：你的代码站在一摞抽象之上

写 Python 时你以为「代码直接在跑」，其实它站在一摞抽象上，每一层都在帮你（也在产生开销）：

```
   你的 Python 代码  (def / async def / threading...)
        │  编译
   字节码 bytecode    (LOAD_FAST, CALL, ...)            ← Python 这台"虚拟机"的指令
        │  由解释器逐条执行
   CPython 解释器 eval 主循环   (持有 GIL 才能跑)        ← 一个 C 写的大 while 循环
        │  解释器本身是一个 C 程序，跑在
   OS 线程 (pthread, 和内核线程 1:1)                    ← 操作系统调度的单位
        │  被
   内核调度器  (决定哪个线程此刻上哪个核)
        │  最终在
   CPU 核心 (寄存器 + PC 真正执行机器指令)
```

**理解并发，本质就是理解「这摞抽象里，谁在什么时候占着 CPU、谁在等」。** 三大模型的区别，就是它们在这摞抽象里切换的位置不同：

- **线程切换**发生在「内核调度器」那层（贵，要进内核）。
- **进程切换**也在内核那层，但还要换地址空间（更贵）。
- **协程切换**发生在最上面「解释器」那层（极快，纯用户态，连内核都不惊动）。

下面逐层拆开。

## 3. 原理深入

### 视角 A · 机器视角：OS 和 CPU 到底怎么跑并发

#### A.1 进程和线程是 OS 的「抽象」，不是物理实体

CPU 只会做一件事：从「程序计数器（PC）」指的地址取一条指令、执行、PC 前进。它根本不知道「进程」「线程」是什么。

- **进程**是操作系统造出来的抽象：一份**独立的地址空间**（虚拟内存）+ 打开的文件 + 一堆资源。是「资源分配的单位」。
- **线程**是「CPU 调度的单位」：一个执行流，有自己的**栈**和**寄存器现场**，但**和同进程的其他线程共享地址空间**（堆、全局变量）。

> 一句话：**进程之间隔着墙（独立内存），线程之间在一个屋里（共享内存）。** 这就是第 02/03 章「线程共享变量要加锁、进程传数据要序列化」的物理根源。

#### A.2 调度器与时间片：单核也能「同时」跑很多线程

机器上线程数远多于核数（几千个线程，8 个核）。**内核调度器**用「时间片」轮转：给线程 A 几毫秒，时间到了把它踢下来、换线程 B 上。宏观上看起来都在跑（并发），微观上每个核同一刻只跑一个线程。

- 多核 = 多个核**真的同时**各跑一个线程（并行）。
- 单核 = 一个核快速轮换（只能并发，不能并行）。

> 呼应第 01 章「并发 vs 并行」——这里是它的物理实现。

#### A.2.1 一条线程会在核间「跳」，但一个进程封顶一个核

A.2 说「多核能真的同时跑」，但这里藏着一个特别容易混的点（也是很多人对 GIL、对「单 worker 用几个核」想不通的根）：**「线程跑在哪个核上」和「能不能同时用多个核」是两件不同的事，答案相反。**

- **跑在哪个核上？——会跳。** 内核调度器不会把一条线程钉死在某个核上：这次时间片在核 0 跑，被抢下来再上来，可能就排到核 3 了——这叫**线程迁移（migration）**。所以「一条线程只能用某一个核」是错的，它一直在核之间跳（除非你用 `taskset` / 设 CPU 亲和性 affinity 手动把它钉住）。
- **能不能同时用多个核？——看有没有 GIL。** 没有 GIL 的世界（Java/Go），一个进程里的多条线程能被调度器**在同一瞬间**摆到不同核上真跑 → 单进程吃满多核。有 GIL 的世界（CPython），同一进程同一瞬间只有一条线程能跑 Python 字节码（B.2）→ **不管这进程开多少线程、机器多少核，它的 Python 部分都封顶在「一个核的吞吐」**。

一个比喻把这两件事焊死。把机器想成一栋有 4 个炉台（= 4 核）的厨房，一个进程 = 一支厨师团队 + **一支专属对讲机（= 那把 GIL）**：

- 规矩：**握着对讲机的厨师才能下厨。** 团队里就算有 3 个厨师（3 条线程），同一时间只有握着对讲机那一个能动手。
- 那个厨师这道菜站炉台 1、下道菜可能站炉台 3（**线程迁移／跳核**）——但无论站哪，**整个团队同一瞬间只有一道菜在进行**（≈ 一个核的产出）。
- 想 4 道菜同时做？**开 4 支团队、4 支对讲机** = 4 个进程（第 03 章多进程／第 07 章多 worker），各占一个炉台同时开火 → 吃满 4 核。

**`top` 里能直接看到这件事**：给单进程的 Python 灌 CPU 活，它的 CPU% 顶到 **~100% 就上不去**（100% = 一个核满载，尽管它一直在跳核）；换成 4 进程，总和能冲到 **~400%**。进程能不能超过 100%，就是「它有没有真的同时用多核」最直接的证据。

> 一句话：**炉台会换（跳核）≠ 同时用多个炉台（并行）。** 跳核是内核调度的自由，封顶一核是 GIL 的限制——把这两件事分开，「单 worker 到底用几个核」（第 07 章）就不再绕。

#### A.3 上下文切换：到底切了什么，为什么贵

调度器把线程 A 换成线程 B，叫**上下文切换**。具体动作：

1. 把 A 的**寄存器现场 + 程序计数器（PC）** 保存到内存（A 的内核栈/控制块）——「记下 A 执行到哪、手里的值是什么」。
2. 从内存恢复 B 的寄存器现场 + PC——「把 B 上次的现场摆回 CPU」。
3. PC 跳到 B 上次中断的位置，B 继续跑。

**为什么贵**：
- 整个过程必须**进内核**（用户态做不了）。
- **进程**切换还要换地址空间（切页表 CR3）→ **TLB（地址翻译缓存）失效**，之后内存访问要重新翻译，更慢。
- 切换后 CPU 缓存（L1/L2）里全是上一个线程的数据，新线程要重新预热——**隐性成本常比切换本身还大**。

量级（具体数字，建立直觉）：线程切换约 **1–10 微秒**，进程切换更贵（多了切页表/刷 TLB）。所以**线程不是越多越好**——线程多了，光上下文切换就吃掉大量 CPU。

> 对比协程切换（视角 B）：约 **100–200 纳秒**，快了一两个数量级——因为它不进内核、不刷 TLB，只是解释器层面的一次函数返回。这就是协程能开几十万、线程只能开几千的根本原因。

#### A.4 用户态 / 内核态 / 系统调用：为什么一次 I/O 要「进内核」

CPU 有两种运行级别：

- **用户态**：跑你的普通代码，权限受限，**不能直接碰硬件**（磁盘、网卡）。
- **内核态**：操作系统代码运行的级别，能操作硬件、管理内存和进程。

你的程序想读网络/磁盘，自己没权限，必须**请操作系统代劳**——这就是**系统调用（syscall）**：程序通过一条特殊指令「陷入」内核，切到内核态，内核替你操作硬件，做完切回用户态。

```
用户态：你的代码  ──read() 系统调用──▶  陷入内核
                                          内核态：内核操作网卡/磁盘
用户态：拿到数据继续  ◀──返回──────────  切回用户态
```

一次 syscall 有成本（模式切换 + 内核处理，约几百纳秒~微秒级）。这解释了为什么「频繁小 I/O」慢、要批量；也是理解「阻塞」的前提。

#### A.5 「阻塞」在系统层面是什么

你调 `data = sock.recv(4096)`，但数据还没到。会发生什么？

> **没学过操作系统？先装 4 个名词就够读懂下面。** 底层事实：**一个 CPU 核一个瞬间只能跑一个线程**，你机器上几百个线程绝大多数此刻都没在跑。OS 靠两个名单和一个门铃来管：
> - **运行队列（run queue）**：「我有活干，给我 CPU 就行」的线程排这，调度器从这里挑下一个上 CPU。
> - **等待队列（wait queue）**：「我卡住了，在等某个**具体东西**（这个 socket 的数据 / 这把锁 / 这个定时器）」的线程挂这，**不抢 CPU，纯睡**——这就是「阻塞」的真身。
> - **中断（interrupt）= 硬件的门铃**：CPU 正忙着，网卡数据到了没法排队等，就「按门铃」打断 CPU；CPU 立刻停下、跳进内核处理一下、再回去。对比 CPU 自己每隔一会去问网卡「到了没」（轮询，浪费），门铃是「**来了才吵你**」。
> - **唤醒（wakeup）没有魔法**：就是把一个线程**从「等待队列」搬回「运行队列」**，让它重新有资格抢 CPU，仅此而已。

1. 线程发起 `recv` 系统调用，进内核。
2. 数据没准备好 → **内核把这个线程从「运行队列」拿下，挂到这个 socket 的「等待队列」上，线程进入睡眠**（不占 CPU）。
3. 调度器趁机让别的线程上 CPU 跑。
4. 网卡收到数据 → 硬件中断 → 内核把数据备好，**把该线程从等待队列移回运行队列**（唤醒）。
5. 线程重新被调度，从 `recv` 返回，拿到数据。

**关键点**：阻塞的线程**不占 CPU**（它在睡觉），但它**占着一个线程**。所以「一连接一线程」模型在高并发下，瓶颈是线程太多（内存 + 上下文切换），不是 CPU。

> 关联 Python：CPython 在执行这种阻塞 syscall 期间会**释放 GIL**（第 01 章 3.4），所以别的 Python 线程能趁机跑——这就是「I/O 密集多线程有效」的底层机制。

#### A.6 I/O 多路复用（select/poll/epoll）：一个线程盯住上万连接

「一连接一线程」开几万线程扛不住。换个思路：**一个线程，同时盯着上万个 socket，谁有数据了就处理谁**。这就是 I/O 多路复用，靠这几个系统调用：

- **select / poll**（老）：每次调用都把「我关心的这 N 个 fd」整个列表传给内核，内核**逐个扫一遍**看谁就绪，返回。N 大时每次都 O(N) 扫描，慢。
- **epoll**（Linux）/ **kqueue**（macOS）（新）：先用 `epoll_ctl` 把 fd **一次性注册**进内核维护的数据结构；之后 `epoll_wait` 直接返回**已就绪的那几个**（内核靠 fd 上的回调维护一个就绪列表，**不用每次全扫**）。所以监听 1 万个 fd 也几乎 O(1)。

```
一个线程：
   epoll_wait()  ──阻塞等待──▶ 内核：这 10000 个 socket 里，3 号、78 号、902 号有数据了
   ◀── 返回就绪的 3 个 ──
   依次处理这 3 个，再回去 epoll_wait()
```

> 这就是著名的 **C10k 问题**（单机万级并发连接）的解法，也是**事件循环的内核基础**（视角 B 会接上）。Java 的 NIO `Selector`、Netty 的 EventLoop 用的是同一套东西。

#### A.7 黑盒里发生什么：epoll 怎么把「逐个扫」变成「主动通知」

上面说 epoll「靠回调维护就绪列表」，这句话是整个魔法的核心。拆开看其实只有一个 trick：**epoll 没有发明新的通知机制，它复用了 A.5 那条「网卡中断 → 唤醒等待队列」的现成通路**——只不过等待队列上挂的不是「一个睡着的线程」，而是「一个回调」。所以根本没有「内核主动去扫谁好了」这回事，是**事件（中断）推它**，不是它去拉。

内核里一个 epoll 实例（`struct eventpoll`）就两个关键结构：

- **兴趣列表 = 一棵红黑树**：每次 `epoll_ctl(ADD)` 一个 fd，就往树里塞一个节点。红黑树让「加 / 删 / 查重」都是 O(log N)——**这就是为什么注册一次就够，不用像 select 那样每次重传整个 fd 列表**。
- **就绪列表 = 一条双向链表**：谁就绪了就被挂上来。`epoll_wait` 只看这条链表。

连起来三步：

```
① epoll_ctl(ADD, fd)
   建节点插进红黑树;
   在「这个 socket 自己的等待队列」上挂一个回调 ep_poll_callback
   (对比 A.5：那里挂的是睡着的线程，这里挂的是回调)

② 数据到达 (和 A.5 完全同一条路)
   网卡中断 → 数据进 socket 缓冲区 → 唤醒该 socket 等待队列
   → 触发 ep_poll_callback → 把这个 fd 挪进「就绪链表」→ 顺手唤醒睡在 epoll_wait 的线程

③ epoll_wait() 醒来
   就绪链表空不空？空 → 继续睡;
   不空 → 把链表里这几个拷给用户态，返回
```

全程**没有任何轮询 / 扫描**，纯中断驱动。「主动通知」= 事件来了顺着唤醒链把 fd 记到一张小名单上而已。

**为什么 select/poll 慢、epoll 快**，对比就清楚了：

| | select / poll | epoll |
|---|---|---|
| 内核记不记得你关心哪些 fd | **不记**，每次调用重传整个列表 | 红黑树持久保存，**注册一次** |
| 每次调用的扫描 | 遍历全部 N 个 fd 挨个问「你好了没」 | **不扫**，直接读就绪链表 |
| 复杂度 | O(N)，N = 总连接数 | O(就绪数)，**和总连接数无关** |

select/poll 的本质问题是**无状态**：它和内核之间没有持久记忆，所以每次都得「重传 + 全扫」。epoll 把这件事拆成「注册一次（摊销）+ 收割就绪（很便宜）」。

> **常见疑惑：epoll 是「新」的，那中断是不是也是后来才有的？** 不是——**中断是计算机最古老的机制之一（1950s）**，比 Unix、select、epoll 都早几十年。select 和 epoll 底下用的是**同一套中断 + 唤醒**，区别根本不在「谁用了中断」，而在**中断那一刻有没有人记下「是谁好了」**：
> - **select**：中断只负责把你叫醒，「有东西好了，但没记是谁」→ 醒来还得把 N 个 fd 全扫一遍找出来 → O(N)。
> - **epoll**：中断触发 `ep_poll_callback`，**当场**把「3 号好了」记进就绪链表 → 醒来名单已就位，不用扫 → O(就绪数)。
>
> 所以 epoll 不是发明了新机制，而是**在同一套老机制上换了个更聪明的记账法**。它晚到 2002 年（kqueue 2000）不是技术不到位（中断/红黑树/回调早就有），而是**动机晚到**：直到 1999 年 **C10k 问题**（单机要扛上万并发连接）被提出，select 每次 O(N) 的代价才真的开始痛，需求才逼出了 epoll / kqueue。一句话：**中断是地基（老），epoll 只是地基上更聪明的记账软件（新）**。

> **破个常见迷思**：epoll 的快**不是**靠 mmap 共享内存（网上流传的错误说法）。`epoll_wait` 照样把就绪事件拷贝到用户态。真正省的是两件事：① 不用每次重传整个 fd 列表；② 不用每次全扫。

**LT vs ET（边沿触发，面试常问，也是上面机制的直接产物）**：

- **水平触发 LT（默认）**：`epoll_wait` 返回后你没把数据读干净，这个 fd 会被**重新挂回就绪链表**，下次还报。安全、好写。
- **边沿触发 ET**：只在 `ep_poll_callback` 触发那一刻（状态翻转的「边沿」）通知一次，没读干净也不再提醒，直到**新数据再来**。所以 ET 必须配非阻塞 fd + 循环读到 `EAGAIN`。省一次 wait，但写错会丢事件。

> 关联你熟的东西：Java NIO 的 `Selector`、Netty 的 `EpollEventLoop`、Go runtime 的 **netpoller**、Redis 的事件循环、Nginx，在 Linux 上底层都是这套 epoll。Go 能开几百万 goroutine 跑网络 I/O，正是因为每个 goroutine 阻塞在网络上时让出 P，由 netpoller 统一 `epoll_wait`——你写的是同步阻塞风格，runtime 偷偷帮你做了多路复用。

**epoll 之后呢？——`io_uring`（2019，内核 5.1）**。两者是不同模型，这是面试的高分点：

- **epoll = readiness（就绪通知）**：只告诉你「可以读了」，真正的 `read()` 还得你自己调 syscall。它干掉了「扫 N 个 fd」，但**没干掉「每次 I/O 一个 syscall」**——高 IOPS 下 syscall 本身成了新瓶颈。
- **io_uring = completion（完成通知）**：你把「读这个 fd 到这块 buffer」**提交**给内核，内核**干完**再通知你「已读好」。靠**两个用户/内核共享的环形缓冲区**（mmap 出来的；前面说 epoll 不用 mmap，真正用共享内存省 syscall 的是 io_uring），可**一次 syscall 批量提交**甚至 polling 模式 0 syscall；还**通用到文件 I/O**（epoll 对普通文件无效，普通文件「永远就绪」）。

为什么 io_uring 还没取代 epoll：① 编程模型复杂得多；② 要够新内核（5.x+，好功能 5.10/5.15/6.x）；③ 安全攻击面大，**Google 在 ChromeOS/Android/GKE 默认禁用**（2023）；④ 普通服务瓶颈在业务/DB，epoll 根本不是瓶颈。**结论：网络多路复用 epoll 仍是默认天下；io_uring 是更强的完成模型后继，正从高性能存储 / 文件 I/O 渐进渗透（ScyllaDB、Rust monoio/glommio、Netty io_uring incubator、libuv 的文件 I/O 已切 io_uring 但网络仍走 epoll），尚未登基。** 跨平台对照：BSD/macOS = `kqueue`，Windows = `IOCP`（完成模型，比 Linux 早近二十年）。

### 视角 B · 解释器内部：CPython 怎么执行、GIL 怎么实现、协程怎么暂停

> 适度深入，够面试用。不抠 C 源码。

#### B.1 字节码 + eval 主循环

Python 不是逐字符解释源码。它先把源码**编译成字节码**（一组类似汇编的虚拟指令），再由解释器执行。你能亲眼看到：

```python
# 需要: 仅标准库
import dis
def add(a, b):
    return a + b
dis.dis(add)
#   LOAD_FAST  a
#   LOAD_FAST  b
#   BINARY_OP  +        ← "a+b 不是原子的"就是因为它是好几条字节码（第 01 章 3.5）
#   RETURN_VALUE
```

**CPython 解释器的核心，是一个巨大的 C 语言 `while` 循环**（俗称 eval loop）：取一条字节码 → 跳到对应的处理逻辑 → 执行 → 取下一条。你的程序跑起来，本质就是这个循环在一条条吞字节码。

> 心智：**Python 是一台「虚拟机」，字节码是它的指令集，eval loop 是它的 CPU。** 而这台虚拟机本身，是跑在一个真实 OS 线程上的 C 程序。

#### B.2 GIL 到底怎么实现的（面试常深挖）

GIL 本质就是**一把互斥锁**。规则：**一个线程必须先拿到 GIL，才能进 eval loop 执行字节码。** 怎么协调多个线程轮流拿：

1. **持有**：当前线程拿着 GIL，在 eval loop 里跑字节码。
2. **等待方催促**：另一个线程想跑，等了一会儿（默认 **5ms**，`sys.setswitchinterval()`）还没轮到，就设一个标志 `gil_drop_request`（"该让让了"）。
3. **检查点**：持有线程的 eval loop 里有个「检查点」（eval breaker），**在字节码之间**会瞄一眼这个标志。看到了，就释放 GIL，唤醒等待线程去抢。
4. **I/O 时主动让**：线程要做阻塞 syscall（A.5）前，用 `Py_BEGIN_ALLOW_THREADS` **主动释放 GIL**，做完再 `Py_END_ALLOW_THREADS` 重新获取。

> 两个推论，正好解释第 01 章的现象：
> - CPU 密集多线程为何没用：大家在 eval loop 里抢同一把 GIL，同一刻只一个能跑，还多了「催促—释放—重抢」的开销。
> - 为何 `x+=1` 要加锁：检查点在**字节码之间**，而 `x+=1` 是多条字节码，可能跑到一半就被要求让出 GIL，另一线程读到旧值。

#### B.3 协程其实是「可暂停的函数」

普通函数调用：进去、跑完、返回，中途停不下来。**协程是能在中途「暂停、保存现场、之后从原地恢复」的函数**——它建立在生成器（generator）的机制上。

- `async def` 创建的协程对象，能被「驱动」一步步执行。
- 跑到 `await something` 时，协程**把控制权交还给调用它的事件循环**（连同「我执行到哪」的现场），自己挂起。
- 等 `something` 就绪，事件循环再「驱动」它从 `await` 处继续。

**关键：这个暂停/恢复完全在用户态、在解释器层面完成**——就是一次函数返回 + 一次 `.send()` 恢复，**不进内核、不做 OS 上下文切换**。这就是 A.3 里说的「协程切换 100–200ns」的来源。

> 对比：线程切换是内核把寄存器现场存进内存再换一套（A.3）；协程切换是 Python 对象层面记住「执行到第几行、局部变量是什么」，由事件循环用一次函数调用恢复。一个动用 OS，一个纯解释器内部。

> 想把这个机制彻底拆开（generator → 堆上的 frame → `send`/`StopIteration`/`throw` → `await` 协议怎么一路冒泡到事件循环 → 函数染色的物理根因），资深面试深挖向，见 [04 章 3.6](../04-asyncio-core/README.md)。

#### B.4 事件循环到底是什么

把 A.6（epoll）和 B.3（可暂停的协程）接起来——但**得先说清它俩到底什么关系**，这是 asyncio 最容易含糊、面试也最爱追问的一环。

**核心事实：epoll 和协程互不相识。**

- **epoll（A.6）只认整数 fd。** 它在内核里，管的是「3 号 socket 就绪了」，根本不知道「协程」「`await`」为何物。
- **协程（B.3）只认 `await` / `.send()`。** 它是解释器里的 Python 对象，知道「我 `await` 一下被挂起，回头有人 `.send()` 我就从原地继续」，根本不知道 fd、更不知道 epoll。

这两个世界**从不直接对话**。把它们接起来的，是夹在中间的翻译兼经纪人——**事件循环**。它手里攥着一张**登记表**，每一行把内核侧的一个 fd 一路绑到解释器侧的一个协程：

```
fd 3  →  回调 cb  →  Future  →  协程 C（卡在某个 await）
```

一次完整来回（盯住「fd 3」和「协程 C」怎么找到对方）：

```
① 协程 C 跑到  await sock.recv()
       └底层落到→ loop.add_reader(fd3, cb)：把 fd 和「醒来调谁」一起登记
              · 登记表记一行：  fd3 → cb → Future → 协程C
              · 顺手 epoll_ctl 把 fd3 加进内核监视名单
② 协程 C 让出、挂起                         ← B.3：本质是一次函数返回
③ 事件循环调 selector.select(timeout)（就是 epoll_wait），阻塞等内核
④ 内核：fd3 来数据 → 中断 → 就绪链表记上 fd3 → epoll_wait 返回「fd3 好了」
⑤ 事件循环拿 fd3 查登记表 → 找到 cb，调 cb；cb 把 Future 置为完成 → 调度 协程C 这个 Task
⑥ 事件循环对协程 C 执行 .send() → 它从 await 原地醒来继续   ← B.3
       跑到下一个 await（回到 ①）或结束
```

**一句话关系**：`fd ↔ 回调 ↔ Future ↔ 协程` 是登记表串起来的一条链——**epoll 只碰最左边的 fd，协程只碰最右边的自己，事件循环是唯一同时认识两头、负责来回翻译的中间人**。没有这张表，「fd3 好了」就找不到该叫醒谁。（②→⑥ 那条「协程怎么一路冒泡到事件循环、又怎么被 `.send()` 回去」的解释器内幕，深挖见 [04 章 3.6](../04-asyncio-core/README.md)。）

把上面 ③④⑤⑥ 收成一个循环，事件循环本体就一点不神秘了——它就是**一个线程上的一个 `while` 循环**，干这三件事：

```
while 还有任务:
    1. 算出最近的超时时间
    2. selector.select(timeout)        ← 就是 epoll_wait！阻塞等"哪些 socket 就绪了"
    3. 对每个就绪的 fd / 到期的回调：
          恢复对应的协程（.send()），让它从 await 处继续跑，
          直到它下一次 await（又让出）或结束
```

> 所以「单线程 asyncio 扛上万并发」= **epoll 一次性盯住上万 socket（A.6）+ 协程在用户态极速切换（B.3）**。整个过程一个 OS 线程、一个 GIL，GIL 根本不构成竞争（只有一个线程）。也正因为单线程，**任何一个协程里的阻塞/死算都会卡住这个 while 循环**（第 05 章头号坑的物理解释）。

看懂上面整条链之后，初学最容易顺手画两个等号——**`fd ↔ 协程` 是 1:1**、**`事件循环` 就是 `epoll`**。两个都错，而且都是面试追问点。下面各破一个。

#### B.4.1 「fd ↔ 协程」是一一对应吗？——是临时租约，不是户口

方向是对的（epoll 报「fd3 好了」，事件循环确实得**唯一地**知道该 `.send()` 叫醒哪个协程），但**不是干净的 1:1**，有两道裂缝。

先说**几乎成立的那一面**：当某协程正卡在这个 socket 的 `await sock.recv()` 上时，登记表确实是整齐一行 `fd3 → cb → Future → 协程C`。而且 asyncio **强制**「同一个 fd、同一个方向只能挂一个回调」——你对同一 socket 同时发两个 `await sock.recv()` 会直接 `RuntimeError`。所以**在「读」这一个方向上，fd → 协程 就是 1:1**，不会一个 fd 喂两个读协程。否则 epoll 报「fd3 好了」，查表也分不清该叫醒谁。

**裂缝一：一个 fd 可以同时绑两个协程（读 + 写）。** epoll 对 fd 是**分方向**监视的（可读 / 可写），asyncio 对应有 `add_reader` 和 `add_writer` 两个独立的槽：

```
fd 3 ─读方向→ 协程A（卡在 await recv）
     ─写方向→ 协程B（卡在 await send，发送缓冲区满了在等可写）
```

一个 fd，两个协程。所以严格说是 **fd → 最多 2 个协程**，不是 1:1。

**裂缝二：这条绑定是临时的，不是 fd 的固有属性。** fd 并不「拥有」一个协程：

- 协程跑到 `await recv()` 那刻才建（底层 `loop.add_reader(fd, cb)` 才往表里写一行）；
- 协程一从 `await` 醒来继续往下跑，立刻解绑（`remove_reader`，这行从表里删掉）。

所以同一个 fd3，这一轮绑协程C，C 处理完去 `await` 别的，下一轮可能换成绑协程D。**fd 和协程之间没有户口关系，只有「此刻谁卡在你身上」的临时租约**，由事件循环按 `add_reader`/`remove_reader` 随时增删。

> 一句话：**不是天生 1:1。准确说——「当且仅当某协程此刻正卡在这个 fd 的某个方向上」，登记表里才临时存在一条 `fd(方向) → 协程` 的绑定；同方向唯一（1:1），但一个 fd 可跨读/写绑到两个协程，且醒来即解绑。** 映射的动作和状态全在那张登记表里，不在 epoll 也不在协程自己身上。

#### B.4.2 「事件循环 = epoll」吗？——epoll 只是它的一步

另一个等号也不对——**epoll ⊊ 事件循环**。回看上面那个 while，epoll 只占其中一行：

```
while 还有任务:
    1. 算出最近的超时时间            ← 定时器堆，跟 epoll 无关
    2. selector.select(timeout)     ← 这一行才是 epoll_wait
    3. 对每个就绪的 fd / 到期回调:
          .send() 恢复对应协程       ← 解释器侧，epoll 不认识
```

epoll 是**第 2 步**那一个 syscall；事件循环是**整个 while + 第 1、3 步**。它还干了三件 epoll 根本不知道的事：

| 事件循环的职责 | epoll 管不管 |
|---|---|
| 等网络 I/O（哪些 fd 就绪） | ✅ 这一步就是 epoll |
| 计时（`sleep`/超时/`call_later`）→ 维护一个定时器小顶堆，算出 `select` 该等多久 | ❌ epoll 只认 fd，不认时间 |
| 就绪回调队列（`call_soon`、被 `put`/`set` 唤醒的协程、刚完成的 Task） | ❌ 纯进程内，没 fd |
| 用 `.send()` 真正驱动协程从 `await` 处继续 | ❌ 解释器侧，epoll 连协程是什么都不知道 |
| 把阻塞活/普通磁盘文件丢线程池（`run_in_executor`） | ❌ 绕开 epoll |

这也回头解释了 B.4.1：**「绑上 fd 的那批」只是协程里的一小撮**——大量协程此刻压根没有对应的 fd，醒来根本不靠 epoll：

| 协程在 await 什么 | 走 epoll？ | 靠什么醒 |
|---|---|---|
| `await sock.recv()`（网络 I/O） | ✅ | epoll_wait 返回该 fd（见 B.4.1） |
| `await asyncio.sleep(1)` | ❌ | 定时器堆（就是 `select` 的 timeout） |
| `await queue.get()` / `Event` / `Lock` | ❌ | 别的协程 `put`/`set` 时直接 `.send()` 唤醒 |
| `await other_coro()` | ❌ | 普通函数调用链，没 fd |
| 读写普通磁盘文件 | ❌ | epoll 对普通文件无效，asyncio 丢线程池 |

**一个硬证据：换掉 epoll，事件循环照跑。** epoll 是 **Linux 专属**；同一套 asyncio 事件循环，在 macOS 上第 2 步调的是 **kqueue**，Windows 上是 **IOCP / select**——靠 `selectors` 模块把这层抽象掉，底下换零件，循环逻辑一行不改。要是「事件循环 = epoll」，macOS 上就没有事件循环了，显然不是。甚至更细：第 2 步 `select(timeout)` 里那个 `timeout`，还是第 1 步的定时器堆算出来的——**连 epoll 该等多久，都是事件循环说了算**。

> 一句话：**epoll 是事件循环的「耳朵」，只负责回答「现在哪几个 socket 有动静」。一个完整的事件循环还得有闹钟（定时器堆）、待办板（就绪队列）、一只手（`.send()` 驱动协程）、一张登记表（B.4.1）——epoll 只是被这个大循环在第 2 步调用一下的可替换零件（Linux 上是它，别处换 kqueue/IOCP）。所以事件循环必然比 epoll 大一圈。**

### 视角 C · 全链路贯通：三张图把一切串起来

#### C.1 多线程跑 I/O 密集

```
4 个 Python 线程，1 个进程，1 把 GIL：
  线程1：持 GIL 跑字节码 →到 recv()→ 释放 GIL，进内核睡觉等数据（不占核、不占GIL）
  线程2：抢到 GIL，跑字节码 →到 recv()→ 释放 GIL……
  线程3、4 同理
  → 大家轮流持 GIL，但都在"等 I/O"上重叠了 → 总时间被压缩 → 多线程有效 ✓
```

#### C.2 多进程跑 CPU 密集

```
4 个进程，各自独立解释器 + 独立 GIL：
  进程1 在核0 上跑 eval loop（自己的 GIL）┐
  进程2 在核1 上跑 eval loop（自己的 GIL）├ 真·同时跑 → 吃满 4 核 ✓
  进程3 在核2 ...                          │  代价：进程间传数据要序列化（隔着墙）
  进程4 在核3 ...                          ┘
```

#### C.3 asyncio 单线程跑海量 I/O

```
1 个线程，1 个事件循环，1 把 GIL（无竞争）：
  while:
    epoll_wait 一次性问内核：上万 socket 里谁就绪了？   ← 视角 A.6
    就绪的几个 → 恢复对应协程，从 await 跑到下一个 await ← 视角 B.3
  → 一个线程、一个核，靠"从不阻塞 + 极速协程切换"扛上万连接 ✓
  → 但不吃多核（单线程）；要多核得叠多进程（第 07 章多 worker）
```

#### C.4 一张总图

```
        你的代码 ─编译→ 字节码 ─→ CPython eval loop ─需要→ [GIL]
                                          │ 是个 C 程序，跑在
                                       OS 线程(1:1 内核线程)
                                          │ 被
                                       内核调度器 ─放到→ CPU 核
        ┌──────────────┬──────────────────────────────┬─────────────────┐
   threading        multiprocessing                 asyncio
   多线程共享1把GIL    多进程各1把GIL→真多核           单线程1把GIL无竞争
   切换在内核(μs)      切换在内核+换地址空间            切换在解释器(ns)+epoll盯fd
```

#### C.5 线程 / I/O / event loop 的边界速记

这几个词最容易搅在一起，复习时按层拆：

```text
Python Thread      = 通常是真 OS 线程；跑 Python 字节码前要抢 GIL
用户态/内核态       = 同一个 OS 线程执行时的 CPU 权限状态，不是两种线程
阻塞 I/O           = 当前线程进入 syscall；数据没好时被内核挂起，不占 CPU
event loop         = 跑在某个 OS 线程里的 while 调度循环
coroutine / Task   = event loop 管理的用户态任务，不是 OS 线程
```

所以可以精确回答：

- Python 多线程「不能并行」指的是：普通 CPython 下多个线程不能同时执行 Python 字节码。
- 一个线程释放 GIL 去等 I/O，另一个线程执行字节码，二者可以重叠推进。
- 如果释放 GIL 的线程正在 kernel/native 代码里占 CPU，同时另一个线程在另一个核上跑字节码，这也可能是真 CPU 并行。
- asyncio 的 Task 不等于 Java Thread；更像「coroutine + Future 调度句柄」，由 event loop 在线程内恢复。

完整复习卡见：[线程 / I/O / event loop：谁是系统线程，谁在并行](../99-interview-cards/q-thread-io-event-loop.md)。

## 4. 日常开发应用：有了这层视角，很多结论变成「显然」

- **「同步阻塞会卡死事件循环」**（第 05 章）：因为事件循环是 B.4 那个单线程 while；阻塞 syscall 不释放控制权回 while，整个循环停转。一看 B.4 就懂，不用背。
- **「线程不是越多越好」**：每多一个线程，调度器多一份上下文切换（A.3，μs 级）+ 一份栈内存（MB 级）。线程数有甜点，过了就被切换开销吃掉。
- **「协程能开几十万」**：切换 ns 级、栈 KB 级（A.3 + B.3），所以能海量。
- **「CPU 密集必须多进程」**：单进程一把 GIL，eval loop 同刻只一个线程跑（B.2）；多进程才有多把 GIL、多核并行（C.2）。
- **「I/O 要带超时、要批量」**：每次 I/O 是一次进内核的 syscall（A.4），有固定成本，碎 I/O 贵。

## 5. 生产 & 调优实战：怎么「亲眼看到」这些机制

这些不是玄学，Linux 下都能观测：

```bash
# 上下文切换次数（A.3）—— cs 列：每秒上下文切换数，飙高说明线程太多/争用严重
vmstat 1

# 某进程的自愿/非自愿上下文切换（非自愿高 = 被时间片抢占多 = CPU 争用）
pidstat -w -p <pid> 1
cat /proc/<pid>/status | grep ctxt        # voluntary / nonvoluntary_ctxt_switches

# 看一个进程到底在做哪些系统调用（A.4/A.5）—— 能看到 recv/epoll_wait/read...
strace -p <pid> -f -e trace=network,read,write

# 看线程当前卡在哪个调用栈（含 C 层）—— 定位"卡在哪个阻塞 syscall"
py-spy dump --pid <pid>

# 看字节码（B.1）
python -c "import dis; dis.dis('a+b')"
```

## 6. 面试高频考点（资深向）

**Q1：进程和线程在操作系统层面的本质区别？**
进程是资源分配单位，有独立地址空间（隔离）；线程是 CPU 调度单位，有独立栈和寄存器现场，但共享所在进程的地址空间。所以线程通信靠共享内存+锁，进程通信要序列化/IPC。

**Q2：上下文切换具体切了什么？为什么贵？**
保存当前线程的寄存器现场+PC 到内存、恢复下一个线程的现场。必须进内核；进程切换还要换页表、刷 TLB；切换后 CPU 缓存失效要重新预热。量级线程约 1–10μs，所以线程过多时切换开销显著。

**Q3：用户态和内核态、系统调用是什么？**
用户态权限受限不能直接操作硬件，内核态可以。程序要做 I/O 等特权操作须通过系统调用陷入内核，由内核代劳再返回。一次 syscall 有模式切换成本。

**Q4：阻塞 I/O 在系统层面发生了什么？**
线程发起 syscall 进内核，数据没好就被移出运行队列、挂到该 fd 的等待队列睡眠（不占 CPU 但占线程），数据到达由中断唤醒、移回运行队列再被调度返回。CPython 在此期间释放 GIL。

**Q5：select/poll 和 epoll 的区别？epoll 为什么高效？**
select/poll 每次调用把全部关心的 fd 传给内核并被逐个扫描（O(N)）；epoll 先注册 fd 到内核结构，靠 fd 上的回调维护就绪列表，`epoll_wait` 直接返回就绪的少数 fd（近 O(1)），所以能高效监听上万连接。是 C10k 的解法、事件循环的基础。

**Q5.1：epoll 的「主动通知」到底怎么实现的？回调挂在哪、就绪列表怎么来？**
内核 epoll 实例有两个结构：兴趣列表（红黑树，`epoll_ctl` 注册的全部 fd，加/删/查 O(log N)）和就绪列表（双向链表）。注册 fd 时在「该 socket 的等待队列」上挂一个回调 `ep_poll_callback`（对比阻塞 I/O：那里挂的是睡着的线程，这里挂的是回调）。数据到达走的是同一条路——网卡中断 → 数据进 socket 缓冲区 → 唤醒该 socket 等待队列 → 触发回调把这个 fd 挪进就绪链表 → 唤醒睡在 `epoll_wait` 的线程。所以 `epoll_wait` 只收割就绪链表，O(就绪数) 而非 O(总连接)，全程无轮询。补 LT/ET：LT（默认）没读干净的 fd 会被重新挂回就绪链表、下次还报；ET 只在回调触发那一刻通知一次，必须配非阻塞 fd + 循环读到 `EAGAIN`。

**Q5.2：中断是很老的机制，epoll 为什么 2002 年才出现？select 和 epoll 用了不同的底层机制吗？**
不是不同机制——select 和 epoll 底下是**同一套中断 + 唤醒**，中断是 1950s 就有的硬件地基。区别只在「中断那一刻有没有记下是谁好了」：select 被唤醒后只知道「有东西好了」，得把 N 个 fd 全扫一遍找出是谁（O(N)）；epoll 的回调在中断那一刻就把就绪 fd 记进就绪链表，醒来直接拿（O(就绪数)）。所以 epoll 不是新机制，是同一套老机制上更聪明的记账法。它晚到只因**动机晚到**：1999 年 C10k 问题（单机上万连接）出现，select 每次 O(N) 才真的痛，才逼出 epoll(2002)/kqueue(2000)。

**Q5.3：epoll 之后有没有更好的方案？它和 io_uring 的本质区别？**
epoll 仍是网络 I/O 多路复用的默认主流。`io_uring`（2019，内核 5.1）是更强的后继，区别在模型：epoll 是 **readiness（就绪通知）**——只告诉你「能读了」，`read()` 还得你自己调，每次 I/O 仍一个 syscall；io_uring 是 **completion（完成通知）**——你提交「读这个 fd 到 buffer」，内核干完再通知你，靠用户/内核共享的两个 mmap 环形缓冲区批量提交、甚至 0 syscall，还覆盖文件 I/O（epoll 对普通文件无效）。但 io_uring 未取代 epoll：编程复杂、要新内核、安全攻击面大（Google 在 ChromeOS/Android/GKE 默认禁用）、多数服务瓶颈不在 epoll。跨平台：BSD/macOS = `kqueue`，Windows = `IOCP`（完成模型，比 Linux 早近 20 年）。

**Q6：GIL 是怎么实现的？什么时候让出？**
本质是一把互斥锁，线程须持有它才能在 eval loop 执行字节码。等待线程超过切换间隔（默认 5ms）会设置 drop 请求，持有线程在字节码之间的检查点看到后释放；做阻塞 syscall 前也会主动释放（Py_BEGIN_ALLOW_THREADS）。

**Q7：协程切换为什么比线程切换快这么多？**
线程切换由内核完成：存/恢复寄存器现场、可能刷 TLB、缓存失效（μs 级）。协程切换在用户态、解释器层面完成：协程基于生成器，暂停=把控制权连同现场交回事件循环，恢复=一次 `.send()`，不进内核不刷 TLB（ns 级）。

**Q8：事件循环的本质？单线程怎么扛上万连接？**
事件循环是单线程上的一个 while：`selector.select()`（即 epoll_wait）一次性问内核哪些 fd 就绪，再恢复对应协程跑到下一个 await。靠「epoll 盯住上万 fd + 协程用户态极速切换 + 从不阻塞」实现高并发；但单线程不吃多核，且一个阻塞操作会卡死整个循环。

**Q8.1：事件循环就是 epoll 吗？「fd ↔ 协程」是一一对应吗？**
都不是。① **事件循环 ⊋ epoll**：epoll 只是 while 里 `selector.select()` 那一步（等网络 fd 就绪），事件循环还管定时器堆（`sleep`/超时）、就绪回调队列（`queue`/`Event`/`call_soon`）、用 `.send()` 驱动协程、`run_in_executor` 丢线程池；而且 epoll 是 Linux 专属，换 macOS 是 kqueue、Windows 是 IOCP，循环逻辑不变——所以它只是事件循环的一个可替换零件。② **fd ↔ 协程不是天生 1:1**：只有某协程此刻正卡在这个 fd 的某方向 `await` 上，登记表里才临时存在 `fd(方向) → 协程` 一行；同方向唯一，但一个 fd 可跨读/写绑两个协程（`add_reader`/`add_writer`），且醒来即解绑；反过来，大量协程（`sleep`/等队列/纯计算）压根没有对应 fd。

**Q9：把三大模型用一句话从执行层面区分。**
线程：一进程多线程共享一把 GIL，切换在内核（μs），适合 I/O 密集；进程：多进程多把 GIL，真多核并行，切换最贵且要序列化通信，适合 CPU 密集；协程：单线程一把 GIL 无竞争，切换在解释器（ns）+epoll 多路复用，适合海量 I/O。

## 7. 一句话总结

并发的一切，归结为「**在『代码→字节码→解释器(GIL)→OS 线程/进程→内核调度→CPU 核』这摞抽象里，谁在占 CPU、谁在等、在哪一层切换**」。线程/进程切换发生在内核（贵，μs 级，要存恢复现场甚至刷 TLB）；协程切换发生在解释器（廉，ns 级，纯用户态）；阻塞是线程进内核睡觉，epoll 是一个线程盯住上万 fd；GIL 是「进 eval loop 执行字节码必须先拿的那把锁」。把这张图刻在脑子里，后面所有选型结论都变成「显然」，而不是背下来的规则。
