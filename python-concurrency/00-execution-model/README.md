## 2. 直觉理解：你的代码站在一摞抽象之上

写 Python 时你以为「代码直接在跑」，其实它站在一摞抽象上，每一层都在帮你（也在产生开销）：

```
   你的 Python 代码  (def / async def / threading...)
        │  编译
   字节码 bytecode    (LOAD_FAST, CALL, ...)            ← Python 这台"虚拟机"的指令
        │  由解释器逐条执行
   CPython 解释器 eval 主循环   (持有 GIL 才能跑)        ← 一个 C 写的大 while 循环
        │  解释器本身是一个 C 程序，跑在
   OS 线程 (pthread, 和内核线程 1:1)                    ← 操作系统调度的单位
        │  被
   内核调度器  (决定哪个线程此刻上哪个核)
        │  最终在
   CPU 核心 (寄存器 + PC 真正执行机器指令)
```

**理解并发，本质就是理解「这摞抽象里，谁在什么时候占着 CPU、谁在等」。** 三大模型的区别，就是它们在这摞抽象里切换的位置不同：

- **线程切换**发生在「内核调度器」那层（贵，要进内核）。
- **进程切换**也在内核那层，但还要换地址空间（更贵）。
- **协程切换**发生在最上面「解释器」那层（极快，纯用户态，连内核都不惊动）。

下面逐层拆开。

## 3. 原理深入

### 视角 A · 机器视角：OS 和 CPU 到底怎么跑并发

#### A.1 进程和线程是 OS 的「抽象」，不是物理实体

CPU 只会做一件事：从「程序计数器（PC）」指的地址取一条指令、执行、PC 前进。它根本不知道「进程」「线程」是什么。

- **进程**是操作系统造出来的抽象：一份**独立的地址空间**（虚拟内存）+ 打开的文件 + 一堆资源。是「资源分配的单位」。
- **线程**是「CPU 调度的单位」：一个执行流，有自己的**栈**和**寄存器现场**，但**和同进程的其他线程共享地址空间**（堆、全局变量）。

> 一句话：**进程之间隔着墙（独立内存），线程之间在一个屋里（共享内存）。** 这就是第 02/03 章「线程共享变量要加锁、进程传数据要序列化」的物理根源。

#### A.2 调度器与时间片：单核也能「同时」跑很多线程

机器上线程数远多于核数（几千个线程，8 个核）。**内核调度器**用「时间片」轮转：给线程 A 几毫秒，时间到了把它踢下来、换线程 B 上。宏观上看起来都在跑（并发），微观上每个核同一刻只跑一个线程。

- 多核 = 多个核**真的同时**各跑一个线程（并行）。
- 单核 = 一个核快速轮换（只能并发，不能并行）。

> 呼应第 01 章「并发 vs 并行」——这里是它的物理实现。

#### A.3 上下文切换：到底切了什么，为什么贵

调度器把线程 A 换成线程 B，叫**上下文切换**。具体动作：

1. 把 A 的**寄存器现场 + 程序计数器（PC）** 保存到内存（A 的内核栈/控制块）——「记下 A 执行到哪、手里的值是什么」。
2. 从内存恢复 B 的寄存器现场 + PC——「把 B 上次的现场摆回 CPU」。
3. PC 跳到 B 上次中断的位置，B 继续跑。

**为什么贵**：
- 整个过程必须**进内核**（用户态做不了）。
- **进程**切换还要换地址空间（切页表 CR3）→ **TLB（地址翻译缓存）失效**，之后内存访问要重新翻译，更慢。
- 切换后 CPU 缓存（L1/L2）里全是上一个线程的数据，新线程要重新预热——**隐性成本常比切换本身还大**。

量级（具体数字，建立直觉）：线程切换约 **1–10 微秒**，进程切换更贵（多了切页表/刷 TLB）。所以**线程不是越多越好**——线程多了，光上下文切换就吃掉大量 CPU。

> 对比协程切换（视角 B）：约 **100–200 纳秒**，快了一两个数量级——因为它不进内核、不刷 TLB，只是解释器层面的一次函数返回。这就是协程能开几十万、线程只能开几千的根本原因。

#### A.4 用户态 / 内核态 / 系统调用：为什么一次 I/O 要「进内核」

CPU 有两种运行级别：

- **用户态**：跑你的普通代码，权限受限，**不能直接碰硬件**（磁盘、网卡）。
- **内核态**：操作系统代码运行的级别，能操作硬件、管理内存和进程。

你的程序想读网络/磁盘，自己没权限，必须**请操作系统代劳**——这就是**系统调用（syscall）**：程序通过一条特殊指令「陷入」内核，切到内核态，内核替你操作硬件，做完切回用户态。

```
用户态：你的代码  ──read() 系统调用──▶  陷入内核
                                          内核态：内核操作网卡/磁盘
用户态：拿到数据继续  ◀──返回──────────  切回用户态
```

一次 syscall 有成本（模式切换 + 内核处理，约几百纳秒~微秒级）。这解释了为什么「频繁小 I/O」慢、要批量；也是理解「阻塞」的前提。

#### A.5 「阻塞」在系统层面是什么

你调 `data = sock.recv(4096)`，但数据还没到。会发生什么？

1. 线程发起 `recv` 系统调用，进内核。
2. 数据没准备好 → **内核把这个线程从「运行队列」拿下，挂到这个 socket 的「等待队列」上，线程进入睡眠**（不占 CPU）。
3. 调度器趁机让别的线程上 CPU 跑。
4. 网卡收到数据 → 硬件中断 → 内核把数据备好，**把该线程从等待队列移回运行队列**（唤醒）。
5. 线程重新被调度，从 `recv` 返回，拿到数据。

**关键点**：阻塞的线程**不占 CPU**（它在睡觉），但它**占着一个线程**。所以「一连接一线程」模型在高并发下，瓶颈是线程太多（内存 + 上下文切换），不是 CPU。

> 关联 Python：CPython 在执行这种阻塞 syscall 期间会**释放 GIL**（第 01 章 3.4），所以别的 Python 线程能趁机跑——这就是「I/O 密集多线程有效」的底层机制。

#### A.6 I/O 多路复用（select/poll/epoll）：一个线程盯住上万连接

「一连接一线程」开几万线程扛不住。换个思路：**一个线程，同时盯着上万个 socket，谁有数据了就处理谁**。这就是 I/O 多路复用，靠这几个系统调用：

- **select / poll**（老）：每次调用都把「我关心的这 N 个 fd」整个列表传给内核，内核**逐个扫一遍**看谁就绪，返回。N 大时每次都 O(N) 扫描，慢。
- **epoll**（Linux）/ **kqueue**（macOS）（新）：先用 `epoll_ctl` 把 fd **一次性注册**进内核维护的数据结构；之后 `epoll_wait` 直接返回**已就绪的那几个**（内核靠 fd 上的回调维护一个就绪列表，**不用每次全扫**）。所以监听 1 万个 fd 也几乎 O(1)。

```
一个线程：
   epoll_wait()  ──阻塞等待──▶ 内核：这 10000 个 socket 里，3 号、78 号、902 号有数据了
   ◀── 返回就绪的 3 个 ──
   依次处理这 3 个，再回去 epoll_wait()
```

> 这就是著名的 **C10k 问题**（单机万级并发连接）的解法，也是**事件循环的内核基础**（视角 B 会接上）。Java 的 NIO `Selector`、Netty 的 EventLoop 用的是同一套东西。

### 视角 B · 解释器内部：CPython 怎么执行、GIL 怎么实现、协程怎么暂停

> 适度深入，够面试用。不抠 C 源码。

#### B.1 字节码 + eval 主循环

Python 不是逐字符解释源码。它先把源码**编译成字节码**（一组类似汇编的虚拟指令），再由解释器执行。你能亲眼看到：

```python
# 需要: 仅标准库
import dis
def add(a, b):
    return a + b
dis.dis(add)
#   LOAD_FAST  a
#   LOAD_FAST  b
#   BINARY_OP  +        ← "a+b 不是原子的"就是因为它是好几条字节码（第 01 章 3.5）
#   RETURN_VALUE
```

**CPython 解释器的核心，是一个巨大的 C 语言 `while` 循环**（俗称 eval loop）：取一条字节码 → 跳到对应的处理逻辑 → 执行 → 取下一条。你的程序跑起来，本质就是这个循环在一条条吞字节码。

> 心智：**Python 是一台「虚拟机」，字节码是它的指令集，eval loop 是它的 CPU。** 而这台虚拟机本身，是跑在一个真实 OS 线程上的 C 程序。

#### B.2 GIL 到底怎么实现的（面试常深挖）

GIL 本质就是**一把互斥锁**。规则：**一个线程必须先拿到 GIL，才能进 eval loop 执行字节码。** 怎么协调多个线程轮流拿：

1. **持有**：当前线程拿着 GIL，在 eval loop 里跑字节码。
2. **等待方催促**：另一个线程想跑，等了一会儿（默认 **5ms**，`sys.setswitchinterval()`）还没轮到，就设一个标志 `gil_drop_request`（"该让让了"）。
3. **检查点**：持有线程的 eval loop 里有个「检查点」（eval breaker），**在字节码之间**会瞄一眼这个标志。看到了，就释放 GIL，唤醒等待线程去抢。
4. **I/O 时主动让**：线程要做阻塞 syscall（A.5）前，用 `Py_BEGIN_ALLOW_THREADS` **主动释放 GIL**，做完再 `Py_END_ALLOW_THREADS` 重新获取。

> 两个推论，正好解释第 01 章的现象：
> - CPU 密集多线程为何没用：大家在 eval loop 里抢同一把 GIL，同一刻只一个能跑，还多了「催促—释放—重抢」的开销。
> - 为何 `x+=1` 要加锁：检查点在**字节码之间**，而 `x+=1` 是多条字节码，可能跑到一半就被要求让出 GIL，另一线程读到旧值。

#### B.3 协程其实是「可暂停的函数」

普通函数调用：进去、跑完、返回，中途停不下来。**协程是能在中途「暂停、保存现场、之后从原地恢复」的函数**——它建立在生成器（generator）的机制上。

- `async def` 创建的协程对象，能被「驱动」一步步执行。
- 跑到 `await something` 时，协程**把控制权交还给调用它的事件循环**（连同「我执行到哪」的现场），自己挂起。
- 等 `something` 就绪，事件循环再「驱动」它从 `await` 处继续。

**关键：这个暂停/恢复完全在用户态、在解释器层面完成**——就是一次函数返回 + 一次 `.send()` 恢复，**不进内核、不做 OS 上下文切换**。这就是 A.3 里说的「协程切换 100–200ns」的来源。

> 对比：线程切换是内核把寄存器现场存进内存再换一套（A.3）；协程切换是 Python 对象层面记住「执行到第几行、局部变量是什么」，由事件循环用一次函数调用恢复。一个动用 OS，一个纯解释器内部。

#### B.4 事件循环到底是什么

把 A.6（epoll）和 B.3（可暂停的协程）接起来，事件循环就是**一个线程上的一个 `while` 循环**，干这三件事：

```
while 还有任务:
    1. 算出最近的超时时间
    2. selector.select(timeout)        ← 就是 epoll_wait！阻塞等"哪些 socket 就绪了"
    3. 对每个就绪的 fd / 到期的回调：
          恢复对应的协程（.send()），让它从 await 处继续跑，
          直到它下一次 await（又让出）或结束
```

> 所以「单线程 asyncio 扛上万并发」= **epoll 一次性盯住上万 socket（A.6）+ 协程在用户态极速切换（B.3）**。整个过程一个 OS 线程、一个 GIL，GIL 根本不构成竞争（只有一个线程）。也正因为单线程，**任何一个协程里的阻塞/死算都会卡住这个 while 循环**（第 05 章头号坑的物理解释）。

### 视角 C · 全链路贯通：三张图把一切串起来

#### C.1 多线程跑 I/O 密集

```
4 个 Python 线程，1 个进程，1 把 GIL：
  线程1：持 GIL 跑字节码 →到 recv()→ 释放 GIL，进内核睡觉等数据（不占核、不占GIL）
  线程2：抢到 GIL，跑字节码 →到 recv()→ 释放 GIL……
  线程3、4 同理
  → 大家轮流持 GIL，但都在"等 I/O"上重叠了 → 总时间被压缩 → 多线程有效 ✓
```

#### C.2 多进程跑 CPU 密集

```
4 个进程，各自独立解释器 + 独立 GIL：
  进程1 在核0 上跑 eval loop（自己的 GIL）┐
  进程2 在核1 上跑 eval loop（自己的 GIL）├ 真·同时跑 → 吃满 4 核 ✓
  进程3 在核2 ...                          │  代价：进程间传数据要序列化（隔着墙）
  进程4 在核3 ...                          ┘
```

#### C.3 asyncio 单线程跑海量 I/O

```
1 个线程，1 个事件循环，1 把 GIL（无竞争）：
  while:
    epoll_wait 一次性问内核：上万 socket 里谁就绪了？   ← 视角 A.6
    就绪的几个 → 恢复对应协程，从 await 跑到下一个 await ← 视角 B.3
  → 一个线程、一个核，靠"从不阻塞 + 极速协程切换"扛上万连接 ✓
  → 但不吃多核（单线程）；要多核得叠多进程（第 07 章多 worker）
```

#### C.4 一张总图

```
        你的代码 ─编译→ 字节码 ─→ CPython eval loop ─需要→ [GIL]
                                          │ 是个 C 程序，跑在
                                       OS 线程(1:1 内核线程)
                                          │ 被
                                       内核调度器 ─放到→ CPU 核
        ┌──────────────┬──────────────────────────────┬─────────────────┐
   threading        multiprocessing                 asyncio
   多线程共享1把GIL    多进程各1把GIL→真多核           单线程1把GIL无竞争
   切换在内核(μs)      切换在内核+换地址空间            切换在解释器(ns)+epoll盯fd
```

## 4. 日常开发应用：有了这层视角，很多结论变成「显然」

- **「同步阻塞会卡死事件循环」**（第 05 章）：因为事件循环是 B.4 那个单线程 while；阻塞 syscall 不释放控制权回 while，整个循环停转。一看 B.4 就懂，不用背。
- **「线程不是越多越好」**：每多一个线程，调度器多一份上下文切换（A.3，μs 级）+ 一份栈内存（MB 级）。线程数有甜点，过了就被切换开销吃掉。
- **「协程能开几十万」**：切换 ns 级、栈 KB 级（A.3 + B.3），所以能海量。
- **「CPU 密集必须多进程」**：单进程一把 GIL，eval loop 同刻只一个线程跑（B.2）；多进程才有多把 GIL、多核并行（C.2）。
- **「I/O 要带超时、要批量」**：每次 I/O 是一次进内核的 syscall（A.4），有固定成本，碎 I/O 贵。

## 5. 生产 & 调优实战：怎么「亲眼看到」这些机制

这些不是玄学，Linux 下都能观测：

```bash
# 上下文切换次数（A.3）—— cs 列：每秒上下文切换数，飙高说明线程太多/争用严重
vmstat 1

# 某进程的自愿/非自愿上下文切换（非自愿高 = 被时间片抢占多 = CPU 争用）
pidstat -w -p <pid> 1
cat /proc/<pid>/status | grep ctxt        # voluntary / nonvoluntary_ctxt_switches

# 看一个进程到底在做哪些系统调用（A.4/A.5）—— 能看到 recv/epoll_wait/read...
strace -p <pid> -f -e trace=network,read,write

# 看线程当前卡在哪个调用栈（含 C 层）—— 定位"卡在哪个阻塞 syscall"
py-spy dump --pid <pid>

# 看字节码（B.1）
python -c "import dis; dis.dis('a+b')"
```

> 面试加分：能说出「我用 `pidstat -w` 看到非自愿上下文切换暴增，定位到线程池开太大」「用 `strace` 看到大量 `epoll_wait` 短超时空转」这类**把现象落到系统调用层**的排查，远比只会说「CPU 高」有说服力。更系统的工具见 `performance-tuning-roadmap/02-linux-tools/` 和 `00-os-fundamentals/`。

## 6. 面试高频考点（资深向）

**Q1：进程和线程在操作系统层面的本质区别？**
进程是资源分配单位，有独立地址空间（隔离）；线程是 CPU 调度单位，有独立栈和寄存器现场，但共享所在进程的地址空间。所以线程通信靠共享内存+锁，进程通信要序列化/IPC。

**Q2：上下文切换具体切了什么？为什么贵？**
保存当前线程的寄存器现场+PC 到内存、恢复下一个线程的现场。必须进内核；进程切换还要换页表、刷 TLB；切换后 CPU 缓存失效要重新预热。量级线程约 1–10μs，所以线程过多时切换开销显著。

**Q3：用户态和内核态、系统调用是什么？**
用户态权限受限不能直接操作硬件，内核态可以。程序要做 I/O 等特权操作须通过系统调用陷入内核，由内核代劳再返回。一次 syscall 有模式切换成本。

**Q4：阻塞 I/O 在系统层面发生了什么？**
线程发起 syscall 进内核，数据没好就被移出运行队列、挂到该 fd 的等待队列睡眠（不占 CPU 但占线程），数据到达由中断唤醒、移回运行队列再被调度返回。CPython 在此期间释放 GIL。

**Q5：select/poll 和 epoll 的区别？epoll 为什么高效？**
select/poll 每次调用把全部关心的 fd 传给内核并被逐个扫描（O(N)）；epoll 先注册 fd 到内核结构，靠 fd 上的回调维护就绪列表，`epoll_wait` 直接返回就绪的少数 fd（近 O(1)），所以能高效监听上万连接。是 C10k 的解法、事件循环的基础。

**Q6：GIL 是怎么实现的？什么时候让出？**
本质是一把互斥锁，线程须持有它才能在 eval loop 执行字节码。等待线程超过切换间隔（默认 5ms）会设置 drop 请求，持有线程在字节码之间的检查点看到后释放；做阻塞 syscall 前也会主动释放（Py_BEGIN_ALLOW_THREADS）。

**Q7：协程切换为什么比线程切换快这么多？**
线程切换由内核完成：存/恢复寄存器现场、可能刷 TLB、缓存失效（μs 级）。协程切换在用户态、解释器层面完成：协程基于生成器，暂停=把控制权连同现场交回事件循环，恢复=一次 `.send()`，不进内核不刷 TLB（ns 级）。

**Q8：事件循环的本质？单线程怎么扛上万连接？**
事件循环是单线程上的一个 while：`selector.select()`（即 epoll_wait）一次性问内核哪些 fd 就绪，再恢复对应协程跑到下一个 await。靠「epoll 盯住上万 fd + 协程用户态极速切换 + 从不阻塞」实现高并发；但单线程不吃多核，且一个阻塞操作会卡死整个循环。

**Q9：把三大模型用一句话从执行层面区分。**
线程：一进程多线程共享一把 GIL，切换在内核（μs），适合 I/O 密集；进程：多进程多把 GIL，真多核并行，切换最贵且要序列化通信，适合 CPU 密集；协程：单线程一把 GIL 无竞争，切换在解释器（ns）+epoll 多路复用，适合海量 I/O。

---

## 7. 一句话总结

并发的一切，归结为「**在『代码→字节码→解释器(GIL)→OS 线程/进程→内核调度→CPU 核』这摞抽象里，谁在占 CPU、谁在等、在哪一层切换**」。线程/进程切换发生在内核（贵，μs 级，要存恢复现场甚至刷 TLB）；协程切换发生在解释器（廉，ns 级，纯用户态）；阻塞是线程进内核睡觉，epoll 是一个线程盯住上万 fd；GIL 是「进 eval loop 执行字节码必须先拿的那把锁」。把这张图刻在脑子里，后面所有选型结论都变成「显然」，而不是背下来的规则。
