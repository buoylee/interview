## 2. 直觉理解

### 2.1 并发 vs 并行（先把这两个词分清）

- **并发（concurrency）**：同一段时间内，多个任务都在「推进中」，但不一定同时刻在跑。像一个厨师同时照看 3 口锅——他在锅之间快速切换，宏观上 3 道菜都在做。
- **并行（parallelism）**：同一时刻，多个任务真的在「同时跑」。像 3 个厨师各看 1 口锅。

> 关键：**并行需要多个执行单元（多核 CPU）；并发不需要**。单核也能并发（快速切换），但单核永远无法并行。
>
> 这个区分对理解 GIL 至关重要：**Python 的多线程能做到「并发」，但做不到「计算上的并行」**。

### 2.2 进程 / 线程 / 协程（用一张图 + 类比）

```
┌──────────────────────────── 进程（独立地址空间，像一栋独栋房子）─────────────┐
│  独立的内存、文件描述符；崩溃了不连累别人                                      │
│                                                                              │
│   ┌──────────── 线程 1（共享房子里的家具）────────────┐  ┌──── 线程 2 ────┐  │
│   │ 独立的栈 + 寄存器；共享进程的堆/全局变量            │  │ 同左            │  │
│   │   ┌── 协程 A ──┐  ┌── 协程 B ──┐  ← 在一个线程内    │  │                 │  │
│   │   │ 用户态调度  │  │ 用户态调度  │     由程序自己切换 │  │                 │  │
│   │   └───────────┘  └───────────┘                    │  │                 │  │
│   └───────────────────────────────────────────────────┘  └────────────────┘  │
└──────────────────────────────────────────────────────────────────────────────┘
```

- **进程**：操作系统分配资源的单位。独立内存空间，互相隔离。创建/切换最贵。
- **线程**：CPU 调度的单位。同一进程内的线程**共享内存**（堆、全局变量），各自有独立的栈。由**操作系统内核**调度。
- **协程**：比线程更轻，在**一个线程内部**由**程序自己（用户态）**调度的「微任务」。切换不进内核，所以极快极省。

这三者你在 Java/Go 里都见过，只是名字/形态不同。先建立对应关系（细节后面章节展开）：

| | Java | Go | Python |
|---|---|---|---|
| 进程 | 起多个 JVM | 起多个进程 | `multiprocessing` |
| 线程 | `Thread`（1:1 内核线程） | runtime 管理的 M 个 OS 线程 | `threading`（1:1 内核线程） |
| 协程 | 虚拟线程（JDK 21+） | **goroutine**（语言核心） | `asyncio` 的 coroutine |

### 2.3 三者的代价（具体数字，建立量级直觉）

| 维度 | 进程 | 线程 | 协程 |
|---|---|---|---|
| 栈大小 | 独立，通常 8MB | 独立，通常 1–8MB | KB 级（按需增长） |
| 创建代价 | 高（~毫秒，复制页表/启动） | 中（~10 微秒） | 低（~1 微秒） |
| 切换代价 | 高（3–5 微秒，切页表+刷 TLB） | 中（1–10 微秒，要进内核） | **极低（100–200 纳秒，纯用户态）** |
| 调度者 | 内核 | 内核 | 程序自己（事件循环） |
| 能开多少 | 百级 | 千级 | **十万~百万级** |
| 隔离性 | 强（崩溃不连累） | 弱（一个挂全进程挂） | 弱（同线程） |

> 一句话直觉：**协程之所以能开几十万个、切换快百倍**，是因为它不麻烦操作系统——切换就是个用户态的函数跳转，栈也是按需长的几 KB，而不是线程那种固定几 MB、还要陷入内核。这正是 Go 能轻松开百万 goroutine、Python asyncio 能扛上万连接的根本原因。

## 3. 原理深入：GIL

这是 Python 并发**唯一一个**和你 Java/Go 直觉相悖的地方。讲透它，后面都是 API 差异。

> 说明：GIL 是 **CPython**（官方默认实现，你 `python` 跑的就是它）的特性。Jython、PyPy 等另说。下文「Python」均指 CPython。

### 3.1 GIL 是什么

**GIL = Global Interpreter Lock，全局解释器锁。** 它是 CPython 解释器里的一把进程级互斥锁，规则只有一句：

> **同一时刻，同一个进程内，只有一个线程能执行 Python 字节码。**

类比：把整个解释器想成「只有一支笔的办公室」。你可以雇 4 个员工（线程），但**要写字（执行 Python 代码）必须先抢到那支笔**。任何时刻只有一个人在写，其他人干等。

> 对比 Java：JVM **没有**这种全局锁。4 个 Java 线程能真的在 4 个核上同时执行字节码。Go 同理（runtime 面向多核设计）。**这把「笔」是 Python 独有的**。

### 3.2 现象：多线程跑计算，不加速反而变慢

```python
# 需要: 仅标准库
import threading, time

def cpu_bound():
    total = 0
    for _ in range(20_000_000):
        total += 1

# 串行跑 4 次
t0 = time.perf_counter()
for _ in range(4):
    cpu_bound()
print(f"串行: {time.perf_counter()-t0:.2f}s")     # 约 4 秒

# 4 个线程"并行"跑
t0 = time.perf_counter()
ts = [threading.Thread(target=cpu_bound) for _ in range(4)]
for t in ts: t.start()
for t in ts: t.join()
print(f"4线程: {time.perf_counter()-t0:.2f}s")    # 不是 1 秒，而是 ≥4 秒，往往更慢
```

4 个线程在「执行字节码」这件事上被 GIL 串行化了，多核形同虚设；还多了抢笔（GIL 争用）的切换开销，所以更慢。

### 3.3 为什么 CPython 要有 GIL（面试必问）

不是失误，是历史包袱 + 工程权衡，记三点：

1. **引用计数的线程安全**：CPython 靠「引用计数」管理内存（每个对象记着「有几个地方在引用我」，归零就回收）。没有 GIL 的话，每次计数 +1/−1 都得加锁或用原子操作，**所有对象操作都变慢**，连单线程程序都被拖累。GIL 让这个计数不用逐个加锁。
2. **C 扩展生态**：numpy、pandas 等海量 C 扩展默认假设「有 GIL」来保证线程安全。去掉 GIL 会冲击整个生态。
3. **单线程性能 & 实现简单**：有 GIL，单线程代码完全不背锁开销，解释器实现也简单。

> 对标：Java 没走这条路，它靠 JMM（内存模型）+ 每对象 monitor + GC 屏障来保证多线程安全，代价是实现复杂得多。CPython 当年选了「简单 + 单线程快」，GIL 就是那个选择的遗产。

### 3.4 GIL 什么时候释放（决定了「哪些场景多线程有用」）

GIL 不是焊死的。当前持笔的线程在这些时刻会**主动把笔交出去**，让别的线程跑：

```python
data = sock.recv(4096)     # ① 阻塞 I/O（网络/文件/数据库）—— 等待时放笔
time.sleep(1)              # ② sleep —— 放笔
np.dot(a, b)               # ③ 设计良好的 C 扩展计算时显式放笔（numpy 会）
lock.acquire()             # ④ 等待锁等同步原语时放笔
# ⑤ 纯 Python 死算时：解释器每隔约 5ms 强制检查一次，可能把笔切给别人
#    这个间隔 = sys.getswitchinterval()，默认 0.005 秒
```

**这就解释了那条核心规律**：

- **I/O 密集**（网络请求、读文件、查数据库）：线程一发起 I/O 就放笔 → 别的线程趁机跑 → **多线程能重叠掉等待时间** → 有效，能提速。
- **CPU 密集**（纯计算）：线程一直在执行字节码、死攥着笔，每 5ms 才被迫交一次 → 4 个线程轮流用一支笔 → **没并行还多了切换开销** → 没用，甚至更慢。

```python
# I/O 密集：多线程真有效（对比上面的 CPU 例子）
# 需要: pip install requests
import threading, requests, time
urls = ["https://httpbin.org/delay/1"] * 10   # 每个请求服务端故意等 1 秒

t0 = time.perf_counter()
for u in urls: requests.get(u)                 # 串行 ≈ 10 秒
print(f"串行: {time.perf_counter()-t0:.1f}s")

t0 = time.perf_counter()
ts = [threading.Thread(target=requests.get, args=(u,)) for u in urls]
for t in ts: t.start()
for t in ts: t.join()                          # 并发 ≈ 1 秒（等待被重叠）
print(f"多线程: {time.perf_counter()-t0:.1f}s")
```

### 3.5 GIL ≠ 线程安全（经典坑，别被坑）

很多人以为「有 GIL，那 Python 多线程就不用加锁了」——**错**。GIL 只保证**单条字节码**不被打断，但一条 Python 语句往往是多条字节码：

```python
counter += 1
# 编译成大致三步字节码：
#   LOAD   counter   ← 读
#   ADD    1
#   STORE  counter   ← 写
# GIL 可能在 LOAD 之后、STORE 之前把笔交走 → 另一个线程也读到旧值 → 丢更新
```

所以 `counter += 1` 这种「读-改-写」在多线程下**依然要加锁**。

> 对标 Java：完全同理。`i++` 在 Java 里也不是原子的（`getfield/iadd/putfield`），所以才有 `AtomicInteger`、`synchronized`。**你的 Java 直觉在这里是对的**——只是别被「Python 有 GIL」误导成「天然原子」。（细节和 Python 的锁原语在第 02 章。）

### 3.6 前沿：no-GIL / free-threaded（了解即可）

Python 3.13 引入了实验性的 **free-threaded 模式**（PEP 703），能编译出一个真正去掉 GIL 的解释器，让多线程吃满多核。但到 2026 年仍不成熟：单线程性能有回退、大量 C 扩展未适配。**面试提一句「了解 PEP 703 在去 GIL」即可，生产默认仍按有 GIL 来设计。**

---

### 3.7 精确说法：不能并行的是「Python 字节码」

把这句话背准，很多追问就不会绕晕：

```text
Thread A 执行 Python 字节码 + Thread B 执行 Python 字节码
=> 普通 CPython 下不能同时并行

Thread A 阻塞 I/O / sleep / 等锁 / native 代码释放 GIL
Thread B 执行 Python 字节码
=> 可以同时推进
```

阻塞 I/O 时，Thread A 通常已经进入系统调用并被内核挂起，不占 CPU；但网络/磁盘/驱动/内核仍在推进 I/O，Thread B 可以拿到 GIL 继续执行 Python 字节码。更精确地说，这是 **I/O 与 Python 计算重叠**。如果 Thread A 正在执行已释放 GIL 的 kernel/native 代码，同时 Thread B 在另一个核心执行字节码，那也可能是真 CPU 并行。

所以「Python 单进程完全不能利用多核」是不准确的；准确说法是：**普通 CPython 单进程不能靠纯 Python 多线程 CPU 计算正常吃满多核**。多进程、释放 GIL 的 native 库、以及 Python 3.13+ free-threaded 构建是例外路径。

复习这组边界见：[线程 / I/O / event loop 面试卡](../99-interview-cards/q-thread-io-event-loop.md)。

## 4. 日常开发应用：三大武器与选型

GIL 锁的是「同一进程内的线程」，于是 Python 给了你三套武器去绕它/利用它：

```
                  ┌──────────────────────────────────────────────┐
                  │ GIL：一个进程内，同一刻只有 1 个线程跑字节码    │
                  └──────────────────────────────────────────────┘
        ┌──────────────────────┼───────────────────────┐
   threading            multiprocessing              asyncio
   （多线程）            （多进程）                   （协程/单线程）
   受 GIL 限制           每个进程一把独立的"笔"        单线程，GIL 根本无关
   I/O 密集有效          CPU 密集真并行（吃多核）      海量 I/O 等待最优
   对标 Java Thread      对标"多开 JVM 进程"           对标 goroutine(但更弱)
```

**选型只问一句话：你的任务是 CPU 密集还是 I/O 密集？**

| 任务类型 | 选什么 | 理由 |
|---|---|---|
| CPU 密集（计算/图像/加解密/序列化） | **multiprocessing** | 多进程 = 多支笔，绕开 GIL 吃满多核 |
| I/O 密集，并发不大（<100） | **ThreadPoolExecutor**（线程池） | 简单够用，I/O 时自动放笔 |
| I/O 密集，海量连接（上千上万） | **asyncio** | 单线程事件循环，开销最低，GIL 无关 |
| 必须用同步库（没有 async 版本） | **ThreadPoolExecutor** | 同步阻塞丢线程池里，不卡主逻辑 |
| 混合（I/O 为主 + 偶尔 CPU 重活） | **asyncio + 进程池** | 事件循环管 I/O，CPU 活外包给进程池 |

> 记住这张表，你已经能答出 80% 的「Python 并发选型」面试题。三大武器各自的 API、原语、坑，分别在第 02 / 03 / 04-05 章展开。

## 5. 生产 & 调优实战（预告：你会在第 07-08 章看到全貌）

你说「生产怎么用 Python 并发完全没概念」——先给你看终点长啥样，建立期待，细节在生产专章讲。

**关键认知：单个 Python 进程受 GIL 限制，吃不满多核。所以 Python 后端线上几乎都靠「多 worker（多进程）」来利用多核 + 故障隔离。** 这正是 GIL 给部署架构带来的直接后果——而 Java 单进程多线程就能吃满多核，所以 Java 世界很少需要这种「多进程 worker」形态。

```bash
# 同步框架（Flask/Django）：起多个进程 worker，每个进程一把独立 GIL
gunicorn app:wsgi --workers 4

# 异步框架（FastAPI）：每个 uvicorn worker 内部跑一个 asyncio 事件循环
uvicorn app:asgi --workers 4

# 后台/定时任务：交给 Celery，多 worker 消费任务队列
celery -A app worker --concurrency=8
```

心智模型一句话：**worker 数 ≈ 进程数（吃多核），每个 worker 内部再用线程或 asyncio 处理并发请求。** 这就是「第 03 章多进程绕 GIL」+「第 02/04 章进程内并发」的组合拳。

> 对标 Java：gunicorn 多 worker ≈ 起多个 JVM 实例 + 前面挂个负载均衡；单个 worker 内部的并发 ≈ 一个 Tomcat 进程里的线程池。Java 一个进程就能吃满多核，Python 得靠多进程凑。

## 6. 面试高频考点

**Q1：Python 多线程能利用多核吗？为什么？**
不能（指 CPython 的 CPU 密集场景）。因为 GIL 限制同一时刻只有一个线程执行字节码。但 I/O 密集场景下，I/O 会释放 GIL，多线程仍能提升吞吐。要吃多核做计算得用多进程。

**Q2：GIL 锁住了什么？有了 GIL 还要加锁吗？**
锁住的是「同一时刻只有一个线程执行字节码」。仍要加锁——GIL 只保证单条字节码不被打断，而 `x += 1` 是「读-改-写」多条字节码，中间可能被切走导致丢更新，和 Java 的 `i++` 非原子一个道理。

**Q3：GIL 什么时候释放？**
阻塞 I/O、`time.sleep`、等待锁、设计良好的 C 扩展计算（如 numpy）时会显式释放；纯计算时每约 5ms（`sys.getswitchinterval()`）强制检查一次是否要切换。

**Q4：为什么有 GIL？**
为简化 CPython 的内存管理：引用计数无需逐对象加锁、兼容大量假设有 GIL 的 C 扩展、保证单线程不背锁开销。是历史 + 工程权衡，不是 bug。

**Q5：CPU 密集任务在 Python 怎么做？**
multiprocessing 多进程绕开 GIL；或把热点下沉到会释放 GIL 的 C 扩展（numpy/Cython）；前沿可提 no-GIL（PEP 703）。

**Q6：并发和并行的区别？**
并发是「一段时间内多个任务都在推进」（单核也行，靠切换）；并行是「同一时刻真的同时跑」（需要多核）。Python 多线程能并发，但（计算上）不能并行。

**Q7：协程为什么比线程轻？**
协程切换是用户态的函数跳转级（约百纳秒），不进内核、不刷 TLB；线程切换要陷入内核（约 1–10 微秒）。协程栈按需增长（KB 级），线程栈固定 MB 级，所以同样内存能开多得多。

**Q8：为什么 Python Web 服务要配多个 worker？**
单进程受 GIL 限制吃不满多核，多 worker（多进程）才能利用多核 + 故障隔离。这是 Java（单进程多线程即可吃满多核）所不需要、由 GIL 导致的部署差异。

## 7. 一句话总结

你那套 Java/Go 并发知识（线程、池、锁、队列、协程、生产者-消费者）**基本能直接平移到 Python**，只需换 API 名字。唯一要重装的认知是 **GIL**：它让「Python 多线程 = 真并行」这个等式不成立，于是逼出了「**CPU 密集走多进程、海量 I/O 走 asyncio、线上靠多 worker**」这套 Python 特有的打法。抓住 GIL 这把钥匙，后面每一章都是它的展开。
