# 日志格式规范

## 1. 日志级别

| 级别 | 用途 | 示例 |
|------|------|------|
| **CRITICAL** | 服务无法继续运行 | 数据库连接池耗尽、关键配置缺失 |
| **ERROR** | 业务流程失败，需要人工介入 | 支付回调处理失败、第三方 API 持续超时 |
| **WARNING** | 异常但可自动恢复，需要关注 | 请求重试成功、缓存穿透触发降级 |
| **INFO** | 关键业务流程节点 | 用户登录、订单创建、服务启动/关闭 |
| **DEBUG** | 开发调试信息 | 函数入参出参、SQL 语句、缓存命中情况 |

> 级别名称沿用 Python logging 标准（CRITICAL 而非 FATAL，WARNING 而非 WARN）。

**生产环境**：INFO 及以上。DEBUG 仅在排查问题时动态开启，不需要重新部署。

### 如何选择日志级别

不要从"严重程度"去想，从**"谁需要看到这条日志，需要做什么"**去判断：

| 级别 | 谁需要处理 | 处理方式 |
|------|-----------|---------|
| **ERROR** | 值班人员/开发者，**现在就要处理** | 收到告警，打开电脑排查 |
| **WARNING** | 开发者，**下次上班时看看** | 看看 Grafana 面板，趋势是否恶化 |
| **INFO** | 没人需要处理，但**排查问题时需要回溯** | 出了问题后按 trace_id 查上下文 |
| **DEBUG** | 开发者，**正在排查某个具体问题** | 对指定模块临时开启，查完关掉 |

决策流程：

```
这条日志触发时，需要有人半夜爬起来处理吗？
  ├── 是 → ERROR
  └── 否 → 明天上班需要关注趋势吗？
        ├── 是 → WARNING
        └── 否 → 出问题后回溯时有用吗？
              ├── 是 → INFO
              └── 否 → DEBUG（或者不打）
```

### 常见场景级别参考

```python
# ------- ERROR: 非预期异常，需要人立刻来排查 -------
# 数据库写入失败、第三方支付回调处理异常
logger.error("inventory deduction failed", sku=sku_id, error=str(e))

# ------- WARNING: 有异常但系统还在正常运转 -------
# 库存不足走了降级逻辑、请求重试后成功、缓存穿透触发兜底
logger.warning("inventory insufficient, fallback triggered", sku=sku_id)

# ------- INFO: 关键业务节点，排查时用来回溯上下文 -------
# 用户登录、订单创建成功、服务启动
logger.info("order created", order_id=order_id, user_id=user_id)

# ------- DEBUG: 临时排查用，按模块开启而非全局开启 -------
# 中间变量值、缓存命中情况、SQL 执行结果
# DEBUG 日志正常写在代码里，不要注释/反注释，通过级别控制输出
logger.debug("checking inventory result", sku=sku_id, available=count, requested=qty)
```

### 常见错误

- **所有异常都标 ERROR** —— 参数校验失败、用户输入不合法是 WARNING 甚至 INFO，用户犯错不需要开发者半夜爬起来。ERROR 应该是低频的，每一条都值得认真看
- **循环里打 INFO** —— 批量处理 1000 条数据不要打 1000 条 INFO，汇总后打一条
- **ERROR 不带上下文** —— `"order failed"` 毫无价值，必须带 request_id、入参、完整错误信息
- **ERROR 日志每天几百条且没人处理** —— 说明大部分不该是 ERROR，降级为 WARNING 或 INFO

## 2. 日志格式

### 生产环境：结构化 JSON（强制）

```json
{
  "timestamp": "2026-03-05T10:15:30.123Z",
  "level": "error",
  "service": "order-service",
  "instance": "vm-prod-01",
  "trace_id": "abc123def456",
  "request_id": "req-20260305-001",
  "caller": "app/services/order.py:142",
  "message": "create order failed",
  "user_id": "u_10086",
  "order_id": "ord_20260305_001",
  "error": "insufficient inventory: sku_12345 requested 10, available 3",
  "duration_ms": 230
}
```

### 开发环境：人类可读格式

```
2026-03-05 18:15:30.123 [error] create order failed  trace_id=abc123 user_id=u_10086 error="insufficient inventory"
```

通过环境变量切换，代码不需要改动（详见 [02-FastAPI日志配置](./02-FastAPI日志配置.md)）。

## 3. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `timestamp` | string | 是 | ISO 8601，UTC 时区，毫秒精度 |
| `level` | string | 是 | 小写：critical / error / warning / info / debug |
| `service` | string | 是 | 服务名称，与部署单元一致 |
| `instance` | string | 是 | 实例标识（主机名或容器 ID） |
| `trace_id` | string | 是 | 贯穿整个请求链的唯一 ID |
| `request_id` | string | 否 | 单次请求的唯一 ID |
| `caller` | string | 是 | 代码位置 `文件:行号` |
| `message` | string | 是 | 日志描述，简明扼要 |
| `error` | string | 否 | 错误详情，ERROR 级别必填 |
| `duration_ms` | number | 否 | 操作耗时（毫秒） |

### 关键要求

1. **timestamp 用 UTC** —— 团队成员在不同时区排查问题时不会混乱
2. **trace_id 跨服务传递** —— 通过 HTTP Header `X-Trace-Id` 在服务间传递，是串联调用链的核心
3. **message 不拼接动态变量** —— 写 `"create order failed"` 而不是 `f"user {user_id} create order failed"`，动态数据放独立字段，这样才能按 message 聚合统计

## 4. 敏感信息脱敏

日志中**禁止**出现以下信息：

- 用户密码（包括哈希后的）
- 完整身份证号、银行卡号
- API 密钥、Token、Secret
- 完整手机号（应脱敏为 `138****8888`）

在日志库层面统一处理，不依赖开发者自觉（配置方式见 [02-FastAPI日志配置](./02-FastAPI日志配置.md#敏感信息过滤)）。
