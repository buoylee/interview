## 一句話總綱(背起來)

> **雲原生 ≠ k8s。** k8s 是**編排內核(圓心)**;雲原生是「為了在彈性基礎設施上、持續可靠地交付可演進系統」的**一整套工程實踐**——
> 圓心外面一圈圈是:**運行時 → 網絡 → 服務間 → 可觀測 → 交付 → 安全 → 平台 → AI 工作負載 → 彈性成本 → 前沿**。
> 你 2019-2020 的認知停在圓心;2020 後的主流變化,幾乎全在外環。

## 🧵 貫穿全書的示例服務

為了讓「圈與圈之間的流動」具體,全書用同一個小系統當例子:

```
            ┌─────────────┐         ┌──────────────┐
  使用者 ──▶ │ order-api    │ ──────▶ │ inventory    │
            │ (HTTP 訂單)  │         │ (HTTP 庫存)   │
            └──────┬───────┘         └──────┬───────┘
                   │                        │
                   ▼                        ▼
              ┌─────────┐              ┌─────────┐
              │ Postgres│              │ Redis   │
              └─────────┘              └─────────┘
```

- **order-api**:對外的訂單服務(HTTP),寫 Postgres,呼叫 inventory。
- **inventory**:內部庫存服務(HTTP),讀寫 Redis。
- **AI 章擴展**:到 09-10 章,給這個系統加一個 **推薦/風控模型**(order-api 調用模型推理服務),用來講 GPU 調度與模型服務。

每一環都問同一個問題:**「這個系統在這一環該怎麼做、2026 的主流答案是什麼?」**

## 一頁速記:2019-2020 → 2026 變了什麼

| 維度 | 2019-2020 | 2026 主流 |
|---|---|---|
| 容器運行時 | Docker | **containerd / CRI-O**(dockershim 1.24 移除) |
| 入口網關 | Ingress + ingress-nginx | **Gateway API**(ingress-nginx 2026-03 退役) |
| Pod 網絡 / kube-proxy | iptables | **eBPF / Cilium**(可換掉 kube-proxy) |
| 服務間通信 | 自己接 SDK | **服務網格**(sidecar → **ambient 無 sidecar**) |
| 可觀測 | 各組件各搞各的 | **OpenTelemetry** 統一三支柱 |
| 部署方式 | CI 跑 `kubectl apply` | **GitOps**(Git 為唯一真相源) |
| 發布策略 | 滾動更新 | **漸進發布**(金絲雀 / 藍綠 / Argo Rollouts) |
| 安全重點 | 鏡像掃描 | **供應鏈安全**(SBOM / 簽名 / SLSA)+ **Policy as Code** |
| 使用姿勢 | 人人手寫 k8s YAML | **平台工程 / IDP**(黃金路徑,別讓開發直接寫 YAML) |
| AI workload | 跑在 VM / 裸機,手工排 GPU | **k8s 成 AI 底座**(DRA/MIG GPU 調度、Kueue/Volcano、KServe/vLLM) |
| 彈性與成本 | 固定副本 + 預留機器、沒人算錢 | **多維 autoscaling + Karpenter 節點彈性 + FinOps/Kubecost** |
| k8s 的定位 | 跑微服務 | **一切 workload 的底座**(含 AI/GPU、serverless、邊緣) |
