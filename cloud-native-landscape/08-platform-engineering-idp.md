# 08 · 平台環:平台工程 / IDP(別讓開發直接寫 YAML)

> **一句話定位**:到這一環,外圈已經堆了一大堆——網格(04)、可觀測(05)、GitOps(06)、策略(07)。**沒人能要求每個業務開發都精通這一切**。平台工程的答案:把它們**包成一個內部開發平台(IDP),給開發一條「黃金路徑」**,自助、抽象、帶護欄。

> **本章深度層**:外環,**寫滿深度**。這格和你「工程規範 / 帶團隊 / 往架構師走」直接相關。

## 🕰 變遷盒

| | 舊世界(2019-2020) | 新世界(2026) |
|---|---|---|
| 開發怎麼上服務 | **人人手寫** Deployment/Service/Ingress YAML | 走**黃金路徑**:模板生成,不碰底層 YAML |
| 要資源 | 提工單給運維,等 | **自助**:聲明意圖,平台自動供給 |
| 認知負荷 | 每個開發都得懂 k8s + 網格 + 策略… | 平台**抽象掉**,開發只關心「我要一個帶 DB 的 HTTP 服務」 |
| 運維/平台團隊角色 | 救火 + 處理工單(瓶頸) | **把平台當產品做**(賦能型團隊) |
| 護欄 | 靠人 review / 口頭規範 | 平台內置策略(07)+ GitOps(06),**規範即默認** |

**一句話**:平台工程把「一堆雲原生組件」**產品化**成一條開發者自助的黃金路徑——讓「做對的事」變成「最省力的默認」,而不是要求人人都成為 k8s 專家。

## 1. 核心敘事

### 1.1 為什麼需要平台:認知負荷到頂了

你前七環學下來應該有感:要把一個服務**正確**上線,涉及鏡像簽名、Deployment、HTTPRoute、網格策略、資源 limits、Pod 安全、監控面板、GitOps 接入……**這對業務開發是不合理的負擔**。結果要麼開發被拖慢,要麼到處抄錯配置、繞過護欄。

平台工程的核心理念:**Platform as a Product**——平台團隊把這些能力打包成一個「內部開發平台(IDP)」,像對待產品一樣對待它(有用戶、有體驗、有迭代),給開發**黃金路徑(golden path / paved road)**:一條鋪好的、自助的、默認合規的路。

> 組織視角(Team Topologies):**平台團隊是「賦能型」**,不是接工單的;**流團隊(業務)自助消費平台**,不用等。

### 1.2 黃金路徑長什麼樣

開發想要的是表達**意圖**:「我要一個 HTTP 服務,帶一個 Postgres,對外暴露 /payments」。平台把這個高層意圖**展開**成底層一切:

```
開發聲明意圖(高層)                平台自動展開(底層,開發看不到)
┌─────────────────┐               ┌────────────────────────────────┐
│ kind: Service   │   平台控制器   │ Deployment + Service + HTTPRoute │
│ name: payment   │ ──reconcile──▶ │ + 網格策略 + Pod 安全 + 資源limits│
│ db: postgres    │  (回扣 01!)    │ + 監控面板 + GitOps 接入          │
│ expose: /payments│              │ + Crossplane 供給的 Postgres      │
└─────────────────┘               └────────────────────────────────┘
```

**注意這又是第 01 章的模型**:平台本身就是「一組 CRD + 控制器」,把高層抽象 reconcile 成低層 k8s 對象。你學的 Operator 模式,在這裡用來造平台。

### 1.3 平台的積木

| 能力 | 工具 | 做什麼 |
|---|---|---|
| **開發者門戶 / 服務目錄** | **Backstage** | 服務目錄、文檔、軟件模板(scaffolding)、一站式入口 |
| **基礎設施抽象** | **Crossplane** | 用 k8s CRD 描述並供給雲資源(DB/隊列/桶),把雲 API 變成聲明式 |
| **工作負載抽象** | **score.dev / Kratix** | 用一份與環境無關的 workload spec,各環境展開 |
| **GitOps 底座** | Argo CD / Flux(06) | 平台生成的清單走 GitOps 落地 |
| **護欄** | Kyverno / VAP(07) | 策略內置,開發想繞也繞不過 |

### 1.4 IDP 不是 Heroku

老 PaaS(Heroku)也抽象,但是**封閉黑盒**。雲原生 IDP 是**你用開源組件組裝、k8s 原生、可組合、可逃生**的——抽象但不鎖死,需要時能掀開蓋子看底層(這也呼應你「靠開源逃生票」的偏好)。

## 🏛 架構師視角

### 🔬 黑盒內幕

- **平台 = CRD + 控制器**:定義一個高層 CRD(如 `Application`/`Service`),寫一個控制器把它 reconcile 成 Deployment+Service+HTTPRoute+策略+監控(回扣 01)。
- **Crossplane composition**:把「一個 Postgres」定義成一個 `Composition`,聲明時平台調雲 API 真供給一個 RDS,並把連接信息注入為 Secret。
- **Backstage scaffolder**:模板生成代碼倉庫 + 預接好的 CI/清單,push 進 GitOps 倉庫 → Argo CD 落地。

### 💥 失敗模式 / 故障域

| 故障 | 現象 | 根因 / 架構含義 |
|---|---|---|
| **抽象洩漏(leaky abstraction)** | 底層一出問題,開發還是得懂 k8s 才能 debug | 抽象能藏複雜度,但**藏不住故障**;平台要保留「掀蓋子」能力 + 好的可觀測 |
| **黃金牢籠(golden cage)** | 資深開發被卡死,路徑外的需求做不了 | 過度抽象、只有 happy path;要留**逃生艙口**(允許降級到原始 k8s) |
| **平台團隊成瓶頸** | 任何新需求都要平台團隊改,又變工單 | 沒做到真自助;平台要可擴展、可自助 |
| **造了沒人用的平台** | 投入大、採用率低 | 沒當產品做、沒聽用戶;**Platform as Product 的反面教材** |

### 📈 規模化極限 / 投入

- **平台是長期投入**:它是產品,要持續維護、版本化平台 API、處理向後兼容——**不是一次性項目**。小團隊養不起專職平台團隊。
- **抽象面積**:抽象越多,平台維護成本越高、洩漏點越多。**只抽象「大多數服務都一樣」的部分**,長尾留逃生艙口。
- **一個平台團隊服務 N 個流團隊**:這是槓桿所在,但平台 API 的設計質量決定它是賦能還是新瓶頸。

### ⚖️ 選型論證(含「何時不需要平台」)

| 維度 | 選項 | 怎麼選 |
|---|---|---|
| 門戶 | **Backstage** | 事實標準;要服務目錄 + 模板 |
| 基礎設施抽象 | **Crossplane** / Terraform + 包裝 | 要 k8s 原生聲明式選 Crossplane |
| 工作負載抽象 | score.dev / Kratix / 自寫 CRD | 看抽象需求複雜度 |
| 買 vs 造 | 託管 IDP(如 Humanitec)vs 開源組裝 | 小團隊買省心;要可控/逃生用開源組裝 |
| **不做平台** | 直接 Helm/Kustomize + 文檔 | **小團隊(你 <20 人?)往往不需要完整 IDP**——一套好的 Helm chart 模板 + 規範文檔 + GitOps 就夠 |

> **架構師判斷**:平台工程是**規模到了才划算**的事。團隊小、服務少時,一個維護良好的 Helm/Kustomize 模板 + GitOps + 清晰規範(你的 Engineering Handbook 就是這個)比一個半成品 IDP 強。**別為了「大廠都在搞平台工程」就造平台**——先問「我的開發認知負荷真的到頂了嗎」。

### 🧭 演進路徑

1. **裸 YAML → 模板**:先把重複的 Deployment/Service/HTTPRoute 抽成 Helm/Kustomize 模板 + 規範。
2. **模板 → 腳手架**:Backstage 軟件模板,一鍵生成預接好 CI/GitOps/監控的新服務。
3. **腳手架 → 自助供給**:Crossplane 讓開發自助申請 DB/隊列,不提工單。
4. **護欄內置**:策略(07)、GitOps(06)、可觀測(05)默認接好,合規成為默認。

### 🏭 生產事故 / 教訓

- **黃金牢籠逼走資深**:平台只支持標準路徑,一個有特殊需求的團隊被卡死、最後自己繞過平台另起爐灶 → 平台失去信任。教訓:**永遠留逃生艙口**。
- **抽象在事故中洩漏**:平時開發不用懂 k8s,半夜出事卻發現沒人會 debug 底層。教訓:抽象要配**極好的可觀測 + 故障時的下鑽能力**,不能真把底層藏死。

## 2. 現在主流怎麼選

| 決策 | 2026 主流答案 |
|---|---|
| 要不要 IDP | **先質疑**:小團隊用模板 + GitOps + 規範即可;認知負荷真到頂才造平台 |
| 門戶 | Backstage |
| 基礎設施自助 | Crossplane(k8s 原生聲明式) |
| 落地底座 | GitOps(06)+ 策略護欄(07)+ 可觀測(05) |
| 抽象原則 | 抽象共性、**留逃生艙口**,別造黃金牢籠 |
| 組織 | 平台當產品做,平台團隊賦能而非接工單 |

## 🧵 示例服務在這一環

一個開發要新增 `payment` 服務:

- **無平台時(2019)**:手寫 Deployment + Service + Ingress + 配監控 + 申請 DB 工單等兩天 + 自己琢磨網格策略……容易抄錯、漏護欄。
- **有平台時(2026)**:在 Backstage 選「HTTP 服務 + Postgres」模板,填名字 `payment`、暴露 `/payments` → 平台**自動**生成代碼倉庫(預接 CI、簽名、GitOps)、生成 Deployment/HTTPRoute/網格策略/Pod 安全/監控面板,Crossplane 供給 Postgres 並注入連接 Secret。**幾分鐘上線,開發全程沒手寫一行 k8s YAML,且默認就合規**(簽名、limits、非 root、mTLS 全有)。

需要做路徑外的特殊配置時,平台允許他**掀蓋子**直接調底層——不被鎖死。
