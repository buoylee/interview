# 00 · 開篇:雲原生 ≠ k8s

> **一句話定位**:在學任何組件之前,先把「雲原生」這個詞放回正確的尺寸——k8s 只是圓心,不是全部。這一章給你一張地圖,後面每一章都是地圖上的一格。

---

## 🕰 變遷盒(這一章是「元變遷」)

| | 舊世界(2019-2020) | 新世界(2026) |
|---|---|---|
| 「雲原生」約等於 | **k8s**;會 pod/service/deployment 就算懂 | **一整套交付實踐**;k8s 只是編排內核 |
| 心智模型 | 一個點(k8s) | 一張同心圓(圓心 + 七環) |
| 主要難點 | 「怎麼把服務跑在 k8s 上」 | 「怎麼**持續、安全、可觀測地**交付一個會演進的系統」 |

如果你也曾經把「雲原生」和「k8s」當同義詞——這很正常,2019 年那會兒大家都這樣。但這正是你說的「認知面窄」的根源,我們先把它拆開。

---

## 1. 先給「雲原生」一個準確定義

CNCF(雲原生計算基金會)官方的定義,翻成人話是這樣:

> 雲原生技術讓組織能在**彈性、動態的環境**(公有雲/私有雲/混合雲)裡,**構建並運行可擴展的應用**。
> 代表性手段:**容器、服務網格、微服務、不可變基礎設施、聲明式 API**。

注意定義裡**根本沒提 k8s** —— 它提的是一組**特性**和**手段**。k8s 只是「實現這些特性」目前最主流的那個底座。

再把它濃縮成你要記住的三個關鍵詞:

1. **彈性可恢復(resilient)** —— 掛了能自愈、能擴縮。
2. **可觀測(observable)** —— 系統內部狀態能被外部看見。
3. **可頻繁安全地變更(frequent & safe change)** —— 高頻發布,但風險可控、可回滾。

**雲原生的本質,是「為了高頻安全地交付可演進系統」而生的一整套技術與實踐。** 容器和 k8s 是讓這件事成立的**地基**,但地基不是房子。

---

## 2. 為什麼是「同心圓」而不是「一個點」

把雲原生畫成同心圓,是這條線的核心心智模型。由內而外,每一環都在解決「上一環沒解決的問題」:

```
圓心   01 內核      k8s 編排底座        —— 「怎麼聲明我要什麼,讓系統自己維持」
 │     02 運行時    containerd/CRI      —— 「容器到底怎麼被跑起來」
 │     03 網絡      eBPF/Gateway API    —— 「Pod 之間、外部進來怎麼連」
 │     04 服務間    服務網格            —— 「服務之間的加密/重試/可觀測誰來管」
 │     05 可觀測    OpenTelemetry       —— 「出事了我怎麼看見」
 │     06 交付      GitOps              —— 「怎麼把變更安全地推上去」
 │     07 安全      供應鏈/Policy       —— 「我跑的東西可信嗎、誰能做什麼」
 │     08 平台      IDP/平台工程        —— 「別讓每個開發都重學上面這一切」
 │   09-10 AI 基建  GPU調度/模型服務     —— 「訓練與推理這種特殊 workload 怎麼跑」
 │     11 彈性成本  autoscaling/Karpenter—— 「怎麼隨負載伸縮又不燒錢」
 ▼     12 前沿      serverless/WASM/邊緣 —— 「k8s 還能往哪長」
       13 收口      選型總圖 + 變遷對照  —— 「把整張地圖收成一頁」
```

**關鍵直覺:越靠圓心越「基礎設施」,越往外越「交付、組織與工作負載」。**

- 你現有的知識(pod/service/deployment、調度、存儲)集中在**最裡面兩環**。
- 而 2020 之後雲原生真正爆發的領域——**服務網格、GitOps、供應鏈安全、平台工程、AI 基建**——全在**外圈**。
- 這就是為什麼你「會 k8s」卻覺得「跟不上」:你會圓心,但外環是新長出來的。
- 而且這條線不只帶你「認得」外環,還會把每一環講到**架構師深度**(黑盒內幕 / 失敗模式 / 選型論證)——見每章的「🏛 架構師視角」段。

> 圈與圈不是孤立的:一個請求進來,要穿過網絡環(03)→ 服務間環(04)→ 被可觀測環(05)記錄;而這套東西怎麼上線,是交付環(06)+ 安全環(07)的事;最後平台環(08)把這一切包成「開發不用懂細節也能用」。我們用一個貫穿全書的示例服務把這條流動串起來(見第 5 節)。

## 3. 怎麼讀 CNCF Landscape(別被那張「幾百個 logo」嚇到)

你大概見過那張密密麻麻、幾百個 logo 的 [CNCF Landscape](https://landscape.cncf.io) 圖,第一反應是「這怎麼學得完」。**答案是:不用學完,要學會「歸位」。** 那張圖的每個 logo,都能塞進我們同心圓的某一環。

**第一步:按「環/類別」歸位。** Landscape 橫向分了幾大類,基本對得上我們的環:

| Landscape 類別 | 對應本線哪一環 |
|---|---|
| Runtime(運行時)/ Container Runtime | 02 運行時 |
| Cloud Native Network / Service Mesh | 03 網絡 / 04 服務間 |
| Observability and Analysis | 05 可觀測 |
| App Definition & Development / CD | 06 交付 |
| Provisioning / Security & Compliance | 07 安全 |
| Platform / PaaS | 08 平台 |
| Scheduling & Orchestration | 01 內核(k8s 在這) |

> 看到一個沒聽過的工具,先問:**「它在哪一環?它和這環的主流選手比,差在哪?」** ——這就是「歸位」式閱讀,比逐個記名字有用一百倍。

**第二步:按「成熟度」過濾。** CNCF 項目分三級,這直接告訴你「能不能押注生產」:

| 級別 | 含義 | 你該怎麼對待 |
|---|---|---|
| **Graduated(畢業)** | 成熟、廣泛生產採用、治理健全 | 選型默認從這裡挑(k8s、containerd、Cilium、Prometheus、Argo、OpenTelemetry…) |
| **Incubating(孵化)** | 已有實際生產用戶,仍在成長 | 可用,評估社區活躍度與你的容錯度 |
| **Sandbox(沙箱)** | 早期、實驗性 | 觀望 / 玩具,別押生產 |

**一句話:讀 Landscape = 先「歸位到某一環」,再「看成熟度決定信任度」。** 這兩刀下去,幾百個 logo 就變成「七環 × 三級」的一張可管理的表。

## 4. 2020 → 2026,到底變了什麼(總覽)

下面這張是全書的「為什麼你覺得舊」總賬。每一行都會在對應章展開「為什麼變、新主流怎麼選」;這裡先讓你一眼看到全貌:

| 環 | 維度 | 2019-2020 | 2026 主流 | 為什麼變(一句話) |
|---|---|---|---|---|
| 02 | 容器運行時 | Docker | **containerd / CRI-O** | k8s 1.24 移除 dockershim,不再內建 Docker 適配 |
| 03 | 入口網關 | Ingress + ingress-nginx | **Gateway API** | Ingress 表達力太弱、annotation 地獄;ingress-nginx 2026-03 退役 |
| 03 | Pod 網絡 | iptables / kube-proxy | **eBPF / Cilium** | iptables 規則數爆炸、性能差;eBPF 在內核直接轉發 |
| 04 | 服務間 | 自己接 SDK(重試/熔斷/追蹤) | **服務網格(sidecar→ambient)** | 把通用治理從業務代碼下沉到基礎設施 |
| 05 | 可觀測 | 每種信號各搞各的 | **OpenTelemetry** | 統一指標/日誌/追蹤的數據模型與採集,免廠商鎖定 |
| 06 | 部署 | CI 跑 `kubectl apply` | **GitOps** | 集群狀態要可審計、可回滾、與 Git 一致 |
| 06 | 發布 | 滾動更新 | **漸進發布(金絲雀/藍綠)** | 高頻發布需要「小步、可觀測、可自動回滾」 |
| 07 | 安全 | 鏡像掃描 | **供應鏈安全(SBOM/簽名/SLSA)** | SolarWinds 等事件後,「你跑的東西可信嗎」成為焦點 |
| 07 | 策略 | 人工審查 / 散落腳本 | **Policy as Code(OPA/Kyverno)** | 規則要可版本化、自動 enforce |
| 08 | 使用姿勢 | 人人手寫 k8s YAML | **平台工程 / IDP** | k8s 太複雜,不該讓每個開發都重學;抽象出黃金路徑 |
| 09-10 | AI workload | 跑在 VM/裸機,手工排 GPU | **k8s 成 AI 底座**(DRA/MIG、Kueue/Volcano、KServe/vLLM) | 訓練/推理都往 k8s 收斂,GPU 調度與彈性成核心 |
| 11 | 彈性與成本 | 固定副本 + 預留機器、沒人算錢 | **多維 autoscaling + Karpenter + FinOps** | 高頻彈性 + 雲成本治理成架構師核心責任 |
| 12 | k8s 定位 | 跑微服務 | **一切 workload 的底座** | serverless、AI/GPU、邊緣都往 k8s 上收斂 |

> 如果你只有 30 分鐘:把這張表看懂,再去每章讀對應的「🕰 變遷盒」,你的「2019 認知」就更新到 2026 了。

## 5. 🧵 引入:貫穿全書的示例服務

抽象的環很難記,所以全書用同一個小系統當「導遊」。每一環我們都問它同一個問題:**「這個系統在這一環該怎麼做、2026 的答案是什麼?」**

```
            ┌─────────────┐         ┌──────────────┐
  使用者 ──▶ │ order-api    │ ──────▶ │ inventory    │
            │ (HTTP 訂單)  │         │ (HTTP 庫存)   │
            └──────┬───────┘         └──────┬───────┘
                   │                        │
                   ▼                        ▼
              ┌─────────┐              ┌─────────┐
              │ Postgres│              │ Redis   │
              └─────────┘              └─────────┘
```

- **order-api**:對外訂單服務,寫 Postgres,呼叫 inventory。
- **inventory**:內部庫存服務,讀寫 Redis。

帶著這個系統往下走,你會看到同一個東西在每一環的「現代答案」:

| 環 | 對示例服務問什麼 | 預告答案 |
|---|---|---|
| 02 運行時 | 它的容器被誰跑起來? | containerd(不是 Docker) |
| 03 網絡 | 外部流量怎麼進到 order-api? | Gateway API(不是 Ingress) |
| 04 服務間 | order→inventory 的 mTLS/重試誰管? | 服務網格,業務代碼不用寫 |
| 05 可觀測 | 一筆訂單慢了怎麼查? | OTel 串起跨服務的 trace |
| 06 交付 | 改一行配置怎麼上線? | 改 Git → Argo CD 自動同步 |
| 07 安全 | 怎麼確定鏡像沒被掉包? | cosign 簽名 + SBOM 校驗 |
| 08 平台 | 新人怎麼快速發一個新服務? | 平台模板,一鍵生成,不手寫 YAML |
| 09-10 AI | order-api 要調一個推薦模型,GPU 怎麼排、模型怎麼上線? | DRA 排 GPU、KServe/vLLM 服務化、按請求量 scale-to-zero |
