# 04 · 服務間環:服務網格(sidecar → ambient)

> **一句話定位**:第 03 環把 L3/L4 連通了,但服務之間的 **mTLS 加密、重試、超時、熔斷、流量切分、細粒度追蹤**——這些「每個服務都需要、但不該寫進業務代碼」的橫切關注點——誰來做?答案是**服務網格**。這是你 2019 認知裡**完全沒有的一格**。

> **本章深度層**:外環,**寫滿深度**。這格 repo 裡沒有,內幕/失敗模式/規模化/選型全寫進正文。

---

## 🕰 變遷盒

| | 舊世界(2019-2020) | 新世界(2026) |
|---|---|---|
| 重試/熔斷/超時 | 每個服務**自己接 SDK**(Hystrix / Resilience4j / Ribbon) | **下沉到網格**,業務代碼不寫 |
| 服務間加密 | 多數明文 / 自己折騰 TLS | **自動 mTLS**(網格透明加密 + 身份) |
| 跨服務追蹤 | 手工埋點、各語言各搞 | 網格自動產生 L7 遙測(配合 05 OTel) |
| 網格形態 | **sidecar**(每 Pod 一個 Envoy) | **ambient / 無 sidecar**(L4 下沉到節點,L7 按需) |
| 多語言一致性 | 每種語言重寫一套治理庫 | **語言無關**:治理在代理層,Java/Go/Python 一視同仁 |

**一句話**:服務網格把「微服務治理」從**業務代碼裡的庫**,搬到了**基礎設施裡的代理**——而 2024 後最大的變化,是這層代理從「每 Pod 一個 sidecar」演進到「**ambient 無 sidecar**」,大幅砍掉開銷。

---

## 1. 核心敘事

### 1.1 網格解決什麼:橫切關注點的下沉

微服務裡每個服務都要:加密通信、調用失敗重試、慢了超時、對方掛了熔斷、灰度時按比例切流、出問題能看到調用鏈。2019 年這些靠**每個服務引一套庫**(還得每種語言各寫一遍,版本還不一致)。

服務網格的核心思想:**把這些塞進一個和你的服務並排跑的代理(proxy),所有進出流量都過它**。於是治理邏輯從業務代碼裡消失,變成基礎設施配置。

### 1.2 經典架構:數據面 + 控制面

```
控制面 (istiod)  ── 下發配置 (xDS) ──▶  數據面代理們
   │ watch Service/路由/策略 對象              │
   └─(又是一個控制器 reconcile,回扣 01)        │ 真正轉發 + 加密 + 重試 + 遙測
                                              ▼
                              order-api ◀──proxy──▶ proxy──▶ inventory
```

- **數據面**:真正攔截、轉發、加密、重試、產生遙測的代理(多為 **Envoy**)。
- **控制面**:把你聲明的路由/策略對象,翻譯成代理配置下發(Istio 是 **istiod**,協議是 **xDS**)。本質還是第 01 章那個「watch 對象 → reconcile → 下發」的控制器模型。

### 1.3 sidecar 模式,以及它的痛

經典 Istio:**每個 Pod 注入一個 Envoy sidecar**,用 iptables 把 Pod 的進出流量劫持到 sidecar。功能強,但代價硬:

- **資源開銷**:每個 Pod 多一個 Envoy(幾十~上百 MB)。**1 萬個 Pod = 1 萬個 Envoy**,光內存就幾百 GB。
- **延遲**:每次調用多兩跳代理(出端 + 入端各一次)。
- **生命週期耦合**:sidecar 要比應用先就緒,否則應用啟動瞬間的請求失敗;Pod 關閉時 sidecar 要比應用後退出。早期這是經典 race(k8s 1.33 的**原生 sidecar 容器**才從根上修好——回扣 01 章 k8s 自身演進)。
- **升級痛**:升級 Envoy = 重啟所有 Pod。

### 1.4 ambient(無 sidecar):2024 後的主流方向

Istio **ambient mesh** 把網格拆成兩層,按需付費:

```
L4 層(ztunnel)：每節點一個,常駐,負責 mTLS + L4 路由 + 身份
                 —— 便宜、默認全開,所有 Pod 自動有零信任 mTLS

L7 層(waypoint)：每 namespace/服務按需部署的 Envoy
                 —— 要 L7 重試/流量切分/細粒度策略時才加,不要就不付這個錢
```

- **ztunnel**(per-node):用 **HBONE**(HTTP/2 CONNECT 隧道)在節點間建 mTLS,給所有 Pod **零侵入的 L4 加密 + 身份**,不再每 Pod 塞代理。
- **waypoint**(per-namespace/service,可選):需要 L7 能力(七層路由、重試、切流)時才部署一個共享 Envoy。

**好處**:沒有 per-Pod sidecar → 開銷大降、無注入、無生命週期 race、升級不動業務 Pod;**L4 安全默認全覆蓋,L7 成本按需**。這就是為什麼 2024 後新上網格多直接走 ambient。

### 1.5 東西向也歸 Gateway API:GAMMA

第 03 章的 Gateway API 本來管「南北向(外部進來)」;**GAMMA 倡議**把它擴展到「東西向(服務間)」——於是入口和網格用**同一套 API 心智**(Gateway/HTTPRoute)描述,不用再學兩套。這是「Envoy 系生態收斂」的一部分。

## 🏛 架構師視角

### 🔬 黑盒內幕

- **sidecar 流量劫持**:注入的 init 容器寫 iptables,把 Pod 所有進出流量重定向到 Envoy 的端口;應用無感。
- **xDS**:控制面到數據面的配置協議(LDS/RDS/CDS/EDS——監聽器/路由/集群/端點)。控制面 watch k8s 對象,算出每個代理該有的配置,**增量推送**。
- **mTLS 身份**:基於 **SPIFFE** 風格的工作負載身份(`spiffe://...`),證書自動簽發輪換;對端身份來自證書,不靠 IP。
- **ambient HBONE**:ztunnel 之間用 HTTP/2 CONNECT 隧道封裝原始流量並加 mTLS;L7 需要時流量再繞經 waypoint 的 Envoy。

### 💥 失敗模式 / 故障域

| 故障 | 現象 | 根因 / 架構含義 |
|---|---|---|
| **sidecar 啟動 race** | 應用啟動瞬間請求 5xx / 連不上 | 應用先於 Envoy 就緒;原生 sidecar(1.33)前的經典坑 |
| 控制面(istiod)掛 | **不能下發新配置**,但**現有流量照走** | 數據面緩存了配置;控制面非數據路徑上的 SPOF(優雅降級) |
| 全網格錯配 | 一條壞策略推給所有代理 → 大面積故障 | **爆炸半徑 = 整個網格**;改動要灰度 + 限定範圍 |
| sidecar OOM | Pod 整個被連坐重啟 | Envoy 內存沒限好;per-Pod 模型放大資源風險 |
| 調試複雜度 | 「明明代碼沒問題卻 503」 | 多了代理層,故障可能在 mTLS/策略/Envoy 配置;要看代理日誌與遙測 |

### 📈 規模化極限

- **sidecar 內存** ≈ 每 Pod 數十~上百 MB。**萬級 Pod 時這是幾百 GB 的純開銷**——這正是 ambient 要解決的核心痛點。
- **xDS 配置規模**:服務/端點越多,每個代理要接收的配置越大、推送越慢;大網格要做**命名空間隔離 / Sidecar 資源裁剪**(只下發該 Pod 需要的配置),否則每個 Envoy 都裝下全網格信息會爆。
- **延遲預算**:每跳代理增加亞毫秒~毫秒級延遲;延遲敏感鏈路要算進預算(ambient 的 L4-only 路徑更輕)。

### ⚖️ 選型論證(含「何時不要網格」)

| 方案 | 何時選 | 取捨 |
|---|---|---|
| **Istio ambient** | 新上網格的默認;要全功能 + 低開銷 | 較新,L7 waypoint 拓撲要規劃 |
| **Istio sidecar** | 需要成熟穩定的全功能、已有大量 sidecar 部署 | 開銷大、生命週期複雜 |
| **Linkerd** | 要**簡單輕量**,Rust 微代理,運維心智小 | 功能比 Istio 少;極致定制不如 Envoy 系 |
| **Cilium service mesh** | 已用 Cilium CNI,想 eBPF 無 sidecar 統一 L3-L7 | L7 仍嵌 Envoy;網格功能廣度不如 Istio |
| **不上網格** | 小集群 / 服務少 / Gateway API + 少量庫夠用 | **省掉一整層複雜度**——這常是對的選擇 |

> **架構師判斷**:網格不是免費的,它加了一整層需要運維、調試、升級的基礎設施。**先問「我是不是真的需要全網格的 mTLS + L7 治理」**:如果只要服務間加密,ambient 的 L4(ztunnel)或 Cilium 就夠;如果只要入口路由,Gateway API 就夠。**別因為「大廠都用 Istio」就上 Istio。**

### 🧭 演進路徑

1. **SDK → 網格**:把重試/熔斷/mTLS 從業務庫逐步下沉到網格,業務代碼瘦身。
2. **sidecar → ambient**:新集群直接 ambient;老 sidecar 集群可漸進遷移(ambient 與 sidecar 可共存過渡)。
3. **先 L4 後 L7**:先用 ztunnel 拿「全網零信任 mTLS」這個最高性價比的收益,L7(waypoint)等真有需求再按服務加。
4. **統一到 Gateway API**:南北向 + 東西向(GAMMA)共用一套路由 API。

### 🏭 生產事故 / 教訓

- **sidecar 啟動競態導致間歇 5xx**:部署後零星失敗,查半天發現是應用在 Envoy 就緒前發了請求。教訓:用原生 sidecar(1.33+)或 `holdApplicationUntilProxyStarts`;這也說明**網格把故障點藏進了基礎設施層**。
- **一條 VirtualService 錯配打掛半個網格**:全局推送 = 全局爆炸半徑。教訓:**網格變更要當生產變更**,灰度 + 限範圍 + 可快速回滾(回扣 06 章 GitOps/漸進發布)。

## 2. 現在主流怎麼選

| 決策 | 2026 主流答案 |
|---|---|
| 要不要網格 | **先質疑**:小規模/需求簡單就別上;真需要 mTLS+L7 治理才上 |
| 網格形態 | **ambient(無 sidecar)** 優先;新集群默認 |
| 實現 | Istio(ambient,功能全)/ Linkerd(輕)/ Cilium(已用 eBPF) |
| 第一步收益 | 全網 **L4 mTLS 零信任**(ztunnel),L7 按服務加 |
| API 心智 | Gateway API(南北 + 東西,GAMMA) |

## 🧵 示例服務在這一環

`order-api → inventory` 現在交給網格:

- **mTLS**:ztunnel 在節點間自動建立 mTLS,兩個服務的流量全程加密且雙向身份校驗——**order-api 和 inventory 的代碼一行都不用改**。策略:「只有 `order-api` 身份能調 `inventory`」基於工作負載身份,不靠 IP。
- **重試/超時**:給 inventory 配 waypoint,聲明「調用失敗重試 2 次、超時 200ms」——同樣不進業務代碼。
- **金絲雀**:發 `inventory:v2` 時,HTTPRoute 把 10% 流量切到 v2、90% 留 v1,觀測無誤再放大(這條和 06 章漸進發布合流)。

對比 2019:這些當年要在 order-api 裡引 Resilience4j + 自己配 TLS + 埋追蹤點,還只在 Java 端生效;現在是基礎設施統一給,Python 寫的服務也一視同仁。
