# 前期准备 · AI Prompt 清单(可复制)

> 配合 [`../ai-front-phase-playbook.md`](../ai-front-phase-playbook.md) 使用。
> 节奏:**挖(采访)→ 写(起草)→ 验(红队)→ 你收敛**。AI 给选项,你拍板。

---

## A. 产品需求

**A1 · 结构化(brain-dump)**
```
以下是我对【功能名】的零散想法:
【倒出你脑子里的一切】
帮我整理成结构:用户/场景、要解决的痛、成功标准、初步边界。
信息不足的地方,明确标出来。
```

**A2 · ⭐ 让 AI 采访你**
```
我要做【功能名】。在你能写出一份完整、无歧义的需求之前,
一直问我澄清问题:一次只问一个,优先问最影响方案的。
我答完你再问下一个。你觉得够了,就说"够了"。
```

**A3 · 起草 spec + AC**
```
基于以上问答,写一份 spec:
- Problem(一句话)
- 业务规则
- 验收标准 AC(每条用"给定…当…则…",可独立验收)
- Scope:in(这版做)/ out(这版不做)
```

**A4 · 生成边界 case**
```
针对这份需求,列出所有边界和异常场景:
空 / 错误输入 / 越权 / 并发 / 极端值 / 网络失败。
每个给出期望行为。
```

**A5 · ⭐ 红队(攻击需求)**
```
站在三个角色挑这份需求的刺:
① 怀疑的工程师 ② 未来的维护者 ③ 找茬的用户
分别指出:哪里有歧义?哪条 AC 没法验收?漏了什么?
最后回答:我是不是在解决一个假问题?
```

**A6 · 砍范围**
```
如果只保留"能解决核心痛点"的最小集合,这份需求该砍成什么?
列出每个可砍项 + 砍掉的风险。(我来决定砍哪个)
```

---

## B. 原型

**B1 · 需求 → 原型**
```
基于这份需求,生成一个可交互原型,覆盖:
主流程 + 这些状态:加载 / 空 / 错误 / 无权限。
```

**B2 · ⭐ 用原型反向查需求缺口**
```
看这个原型,列出它隐含、但需求文档没写明的规则和状态。
哪些是我还没决定的?(我会把这些回填进需求)
```

**B3 · 标注 normative / incidental**
```
这个原型里,哪些细节是产品意图(必须照做 normative),
哪些只是占位/随手(可不照 incidental)?分两类列出。
```

---

## C. 收敛(这步没有 prompt)
- 砍范围、定取舍、冻结需求和原型 —— **由你做**。
- 冻结后才进 DoR。还在改 = 没 ready。
