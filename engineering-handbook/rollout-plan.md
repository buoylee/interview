# Rollout 计划:让规范真正"被用起来"

> 本计划的核心不是"部署文档",而是**变更管理**——小团队最怕"突然变重"。目标:让团队觉得这套是"帮我少踩坑",而不是"又多一堆流程"。
> ⚙️ 只用现有工具(GitHub + 团队的 AI 编码工具 + Claude),**不依赖未采用的新工具**,零采购摩擦。

---

## 核心策略

> ❌ 把整本手册一次性甩给团队 → 抵触 → 几天后回到老路。
> ✅ **一次只推一样,每样绑一个"团队已经痛的点";你(tech lead)先示范,而不是先命令。**

---

## 四个阶段

### Phase 0 · 第 0 周:地基 + 共识(你一个人做)

| 做什么 | 具体 |
|---|---|
| 建看板 | **GitHub Projects**(零新工具,贴 PR),八个泳道:Inbox→Discovery→Shaping→Ready→Build→Review→Verify→Done |
| CI wave-1 | GitHub Actions:lint / 测试 / 构建 **红了挡合并**;PR 模板入库(`.github/pull_request_template.md`) |
| **拿下 PM** | 单独和 PM 对齐"加厚前期"——改动最大的是 TA 的活。用 [examples/](./examples/) 给 TA 看"前期被挖出 6-8 个盲区" |
| 不碰 | 其他章节先按住不动 |

✅ **完成标志**:看板能用、PR 红了会挡、PM 认同前期要加厚。

#### Phase 0 当天可执行清单
- [ ] 建 GitHub Project,加八个泳道,设 Build 列 WIP 上限(= 工程人数)
- [ ] 加 GitHub Actions:lint + 测试 + 构建;开 branch protection(≥1 approve + CI 绿)
- [ ] 把 [`templates/pr-template.md`](./templates/pr-template.md) 放到 `.github/pull_request_template.md`
- [ ] 和 PM 一对一:讲"AI 让建造变便宜→前期是新杠杆",一起读一个 example
- [ ] 把本手册仓库地址发给全员(只说"先看 README,别急着全用")

### Phase 1 · 第 1-2 周:挑一个试点功能,跑完整闭环

- 选一个 **M 大小、不太急** 的真实功能当试点(急的功能没耐心走流程)。
- 你 **+ PM 一起**用 [AI 前期 playbook](./ai-front-phase-playbook.md) 走 `Discovery → Shaping → DoR`,再让工程走 `Build → Review → Verify → DoD`。
- 关键:**你亲自示范**那两招(让 AI 采访你 / 用原型反查),做给大家看。

> 🎯 **本次试点功能**:_____________________(待填)

✅ **完成标志**:试点功能**全程没有"边写边改"**、AC 验收一次过、团队普遍感觉"这次清楚多了"。

### Phase 2 · 第 3-4 周:变成所有新功能的默认

- 所有新需求都走流水线;**DoR/DoD 成为硬门**。
- 上自动化 wave-2(AI 自动 review、自动 changelog、依赖升级)。
- **双周复盘里专门留 10 分钟迭代流程本身。**

✅ **完成标志**:不用你催,大家自然走流程。

### Phase 3 · 第 2 个月起:成熟

自动化 wave-3、ADR 习惯、Onboarding 文档、开始看指标(cycle time / 返工次数)。

---

## 治"写了没人用"的 5 个钉子

1. **一次只推一样**,且绑一个已经痛的点。
2. **让模板成为最省力路径**——放进 repo / PR 模板 / 看板,顺手就用,比"另开文档"省事。
3. **把闸门焊进自动化**——CI 挡合并、PR 模板强制字段,**想跳过都难**(纪律靠工具,不靠自觉)。
4. **流程进双周复盘**——它是活的、团队能改它 → 有归属感,才不会当成强加。
5. **你先示范,别先命令**——tech lead 自己天天用,团队才信。

---

## 量一个数,证明它有用

选 **1 个指标**:`需求返工次数` 或 `开工后需求变更次数`。前期做仔细了,这个数会降。**给团队看降幅,胜过任何说教。**

> 基线:Phase 0 前先粗记当前水平(凭印象也行) → Phase 2 末对比。

---

## 两个最大阻力(提前想好对策)

| 阻力 | 对策 |
|---|---|
| **PM** 觉得前期工作量上升 | Phase 0 的共识 + 让 AI 帮 TA 起草(采访/红队)降低实际负担 |
| 赶进度时**想跳过 DoR** | 摆出"跳过的代价"——上次返工花的时间/数据,比讲道理管用 |

---

## 一句话

> **规范不是发布出来的,是被一次成功的试点"带"出来的。** 先用一个功能证明"这次清楚多了",团队自己就会想要下一次。
