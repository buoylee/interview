# 工程开发规范 · Engineering Handbook

> **适用**:<20 人小团队 · Web/SaaS + AI/Agent 产品 · 持续交付(硬日期很少)
> **双重用途**:① 团队的执行标准;② 作者(tech lead → PM/架构)的成长教材。
> 因此本手册每条规范都讲 **"为什么"**——规则可以背,取舍才是能力。

---

## 0. 总览

### 0.1 三层分工(全书总纲)

整套系统把每件事分给三层,判据只有一句:**"这件事需不需要判断对错?"**

| 层 | 谁干 | 接什么任务 | 例子 |
|---|---|---|---|
| **人** | 你 / PM / 工程 | **判断、把关**(两道闸的放行权) | 过 DoR/DoD、定方案、挑刺 |
| **AI** | Cursor/Codex/Claude | **需要智能、但人定稿**(起草/整理/挑刺) | 写 spec 草稿、写代码、写 ADR |
| **自动化** | CI / bot / 脚本 | **确定性+重复+不需判断**(无人值守) | 跑测试、移卡片、发预览 |

> 把所有"不需要判断"的事榨干给机器、能起草的交给 AI,人的精力才能全压到**唯一不能糊弄的地方:前期想清楚**。

### 0.2 一条贯穿全书的真相

> **AI 让"建造"几乎免费,于是瓶颈和价值全部移到了"前期想清楚"。**

过去写代码贵,大家自然舍不得返工;现在建造快到飞起,省下的时间若不投回前期,就变成"极快地造出错的东西"。**正确反应:把 AI 省下的建造时间,全部投回前期。** 这是本手册"加厚前期"的根本理由。

### 0.3 流程全景

```
Inbox → Discovery → Shaping →[Shaping Review]═[DoR]═> Ready → Build → Review → Verify ═[DoD]═> Done
想法    问题澄清    方案打磨    打磨评审    闸门①    待开发  开发    评审     验证    闸门②   完成
        └──────── 加厚的前期 ────────┘          └──── 流动(不加仪式)────┘
        治痛点 1/E                                治痛点 2/D            治痛点 4
```

**质量压力全集中在两道闸(DoR/DoD),中间环节保持轻快。** 这是小团队不靠 QA/PM 兜底也能稳的核心。

## 1. 骨架与节奏

### 1.1 一块看板,八个泳道

全队**只用一块看板**(推荐 **Linear**;零成本用 **GitHub Projects**,可直接关联 PR)。列就是流程阶段:

```
Inbox → Discovery → Shaping → Ready → Build → Review → Verify → Done
```

> *Shaping Review 是 Shaping → Ready 之间的"放行活动",并入 DoR,不单独占列。*

### 1.2 WIP 上限(持续流的命根子)

- **Build 列设 WIP 上限**:每人最多同时 **1–2 个**(4 工程 → Build 最多 6 张)。
- 规则:**想拉新任务,先把手上的推到下一列。** 强制"做完再开新的",根治"啥都在做、啥都没完成"。

### 1.3 胃口 Appetite(治范围蔓延)

每张卡进 Discovery/Shaping 时**必须打尺寸**——是"愿意花多少时间"的上限,不是估时:

| 尺寸 | 上限 | 含义 |
|---|---|---|
| **S** | ≤ 2 天 | 小改动 |
| **M** | ≤ 1 周 | 标准功能 |
| **L** | ≤ 2 周 | 大功能,已是上限 |
| 超过 L | — | **必须拆**,不许有无底洞任务 |

> 铁律:**胃口是先定死的"盒子",需求往里塞;塞不下就砍范围,而不是撑大盒子。**

### 1.4 三个会(仪式最少)

| 会议 | 频率 | 时长 | 干什么 |
|---|---|---|---|
| **每周规划** | 周一 | 30 min | 排 Ready 队列优先级、定本周重点、确认胃口 |
| **每日站会** | 每天 | **异步文字** | 三句话:昨天推进啥 / 今天拉啥 / 卡在哪 |
| **双周复盘** | 隔周五 | 30 min | 哪里能改进(流程也在这里迭代) |

> 站会用**异步文字**是刻意的:小团队+持续流不需要每天站起来开会,文字留痕还能沉淀。

## 2. 流水线与关卡(心脏)

核心:**你们没专职 QA、没专职 PM 兜底,所以质量靠两道闸门锁在流程里,而不是靠某个英雄。**

### 2.1 加厚的前期:Discovery → Shaping → Shaping Review

#### ① Discovery(问题澄清)— 画任何原型/代码之前
产出一页纸,**只谈问题、不谈方案**:
- 谁是用户 / 什么场景
- 要解决什么痛 + **凭什么相信这痛是真的**
- 成功长什么样(什么指标/行为会变)
- 为什么现在做
- 初步"不做什么"的边界

> 🔒 **铁律:这一步禁止画原型、禁止写代码。** 先把"问题"钉死。最大的陷阱是**跳过问题、直接给一个漂亮方案,然后发现它在精确回答一个错的问题。**

#### ② Shaping(方案打磨)— 问题钉死后才动手
产出三样:
1. **精简 spec**:problem(来自 Discovery)+ 业务规则 + **验收标准 AC** + **scope 边界(in/out)**
2. **设计**:手绘线稿 → vibe-code 原型;**标注哪些细节是 normative(必须照做)、哪些是 incidental(随手画的)**
3. **任务拆分**:拆成胃口 ≤L 的卡

> 铁律:**原型回答"长什么样",spec 回答"规则/边界/为什么";原型必须冻结才能进下一步**(还在天天改 = 需求没定稿)。
> 概念辨析见 [附录 A:原型 / 设计稿 / PRD / MVP](#附录-a原型--设计稿--prd--mvp-的区别)。

#### ③ Shaping Review(打磨评审)— 进 DoR 前 15 分钟挑刺
全队(或你+PM)**专门找茬**:
- 这真是用户的问题吗?(回头对 Discovery)
- 边界/AC 想全了吗?能验收吗?
- **最简做法是什么?能砍掉什么?**(治范围蔓延)
- 有没有更简单的方案被忽略?

> 本质是"上线前先自我攻击"。在这里 catch 住疏漏花 15 分钟,漏到 Build 之后要几天。

### 2.2 后段:Ready → Build → Review → Verify → Done

| 阶段 | 谁 | 干什么 | 过关条件 |
|---|---|---|---|
| **Ready** | — | 过 DoR 的卡按优先级排队 | 工程从这里 pull |
| **Build** | 工程(守 WIP) | feature 分支、写代码(AI 重度)、写测试 | 开 PR |
| **Review** | 非作者 ≥1 人 | AI 先过一遍 + 人审业务逻辑 | approve + CI 绿 |
| **Verify** | 作者本人 | 对 AC 逐条验收 + 跑测试 + 走边界 | **过 DoD** |
| **Done** | 作者 | 上线、写 ADR、更新文档、关卡片 | 沉淀完成 |

### 2.3 🚪 闸门①:DoR(Definition of Ready)— 治"需求→开发对不齐"

进 Build 前**全勾**才放行:
- [ ] spec 有:problem + 业务规则 + AC + scope 边界(in/out)
- [ ] 原型**已冻结**,且标注 normative vs incidental
- [ ] 已拆成胃口 ≤L 的任务
- [ ] **工程看过并点头:"我看得懂、能做、没有大歧义"** ← 灵魂条,工程不点头就不算 ready
- [ ] 有 AC 能判定"做完算对"

### 2.4 🚪 闸门②:DoD(Definition of Done)— 治"质量没保障"+"沉淀乱"

上线前**全勾**才算完成:
- [ ] 代码合并、CI 绿、已上线(或在 feature flag 后)
- [ ] **AC 逐条验收通过**
- [ ] 关键路径有测试覆盖
- [ ] happy path + 关键边界(空/错/越权)手动走过一遍
- [ ] 重要决策写了 **ADR**
- [ ] 文档已更新,PR ↔ spec ↔ 卡片互相链接(可追溯)

> **为什么是两道闸而不是一份大文档**:DoR 守输入质量(想清楚再做),DoD 守输出质量(做扎实再走),中间保持流动不加仪式。**用两个 checklist,顶替别人家一整个 QA+PM 团队的兜底。**

## 3. AI 整合层

**总原则:AI 做"第一稿"和"整理工",人做"判断"和"把关"。两道闸门(DoR/DoD)的放行权,永远在人手里。**

| 阶段 | AI 干什么 | 工具 | 谁定稿/放行 |
|---|---|---|---|
| **Inbox** | 把粗想法整理成结构;快速调研 | Claude 对话 | 提出人 |
| **Discovery** | **唱反调**:"这问题成立吗?我是否在臆想需求?" | Claude | PM(过) |
| **Shaping** ⭐ | ①对话+线稿→**spec 草稿** ②生成 **AC + 边界 case** ③线稿→可点原型 ④**挑刺**:哪有歧义、漏了哪些边界 | Claude(spec)+ v0/Figma MCP/Pencil(原型) | PM + 你 |
| **Build** ⭐ | 写代码 + **对着 AC 生成测试** | Cursor / Codex / Claude Code | 工程 |
| **Review** | 先过一遍 PR,标可疑点 | CodeRabbit / Claude | 人审业务逻辑 |
| **Verify** | 帮想"还有什么没测到" | Claude Code | 作者(过 DoD) |
| **Done** ⭐ | diff+讨论 → **ADR/changelog/文档草稿** | Claude Code | 作者 |

### 三大杠杆(打 ⭐)
1. **Shaping 起草 spec + 生成 AC**:补强 DoR 输入质量,你们没专职 PM/QA,这是最大杠杆。
2. **Build 对着 AC 生成测试**:**对着 AC 写,不对着实现写**——否则代码错测试跟着错,测了个寂寞。
3. **Done 自动沉淀**:把 ADR/文档成本降到接近零,DoD 那条"写 ADR"才真能落地。

### 前期 AI 的特殊用法:一正一反
AI **起草**(给初稿)+ AI **挑刺**(攻击初稿),几分钟把前期质量拉高一档——正是最该加厚的地方。

> 保命规则:**AI 永远不点 DoR/DoD 的勾。** 它能列 checklist、写草稿、标红可疑点,但"放行"是人的判断。
>
> 📌 *未来钩子*:做 AI 产品时,要在 **Build/Verify** 补一层 **prompt/模型版本化 + eval 回归**(现在先空着)。

## 4. 自动化层

判据:**需要判断对错 → 人/AI;只是机械重复 → 自动化。**

| 阶段 | 能全自动的事 | 工具 |
|---|---|---|
| **Build** | PR:lint/格式化/类型检查/单测/构建,**红了禁止合并** | GitHub Actions |
| **流转** | 建分支→Build;开 PR→Review;合并→Done | Linear/GH Projects 自动化 |
| **Review** | AI 自动 review bot 留评论(第一道筛) | CodeRabbit / Claude |
| **Verify** | 每个 PR 自动部署**预览环境 URL** | Vercel / Netlify |
| **Ship** | 合并 main→自动 staging→(点一下)→prod;或持续部署+flag | GH Actions + flag |
| **Done** | commit→自动 changelog/版本;合并→AI 起草 ADR | conventional commits + release-please |
| **节奏** | commit/PR→站会摘要;复盘自动拉 cycle time/吞吐 | bot + Linear 指标 |
| **杂项** | 依赖自动升级、secret 扫描 | Renovate / gitleaks |

### 分三波上(别一口吃成胖子)
- **🌊 第一波(第一周,回报最高)**:CI 挡合并 + PR 模板 + 卡片自动流转 + PR 预览环境。
- **🌊 第二波**:AI 自动 review + 自动 changelog/版本 + 依赖升级 + 站会摘要。
- **🌊 第三波**:持续部署 + feature flag + 复盘指标自动化 + 合并自动起草 ADR。

> 自动化是**一次性投入、长期收租**。第一波花一两天搭好,之后每个 PR 都省人力、挡 bug——小团队"用工具顶人头"最划算的杠杆。

## 5. 工程规范(地基纪律)

### 5.1 分支策略:GitHub Flow
- `main` **永远可发布**
- feature 分支**短命**(理想 <2 天,呼应胃口 S/M)
- 从 main 切,PR 回 main,**合并即可部署**
- **大功能用 feature flag,不用长命分支**(见 6.3)

> **为什么不用 Git Flow**:它为"攒批、定期发版"设计,和你们"持续小步上线"冲突;多一条 develop = 多一份合并地狱。**分支模型必须匹配发布节奏。**

### 5.2 commit 约定:Conventional Commits
`feat: / fix: / chore: / docs: / refactor:`
> 副作用是**解锁自动化**(自动 changelog/版本)。规范真正的复利在这里。

### 5.3 PR 规则
- **小 PR**(diff 尽量 <400 行)——大 PR 没人审得动
- **PR 模板**(见 templates)
- **branch protection**:≥1 人 approve + CI 全绿 才能合并;**不能自己批自己**

### 5.4 代码风格:不争论,交给工具
- formatter + linter 自动执行(JS: Prettier+ESLint;Py: Ruff;Go: gofmt),pre-commit/CI 强制。
> **能机器决定的事,不要用人的意志去争。**

### 5.5 测试标准
- **测试金字塔**:多单元、少集成、极少 e2e
- **关键路径必须有测试**(写进 DoD)
- **测试对着 AC 写,不对着实现写**
- **不追 100% 覆盖率**,追"关键路径 + 易错点"
> 覆盖率是手段不是目的;架构师看的是"**有没有覆盖会真正出事的地方**"。

## 6. 环境与发布

### 6.1 四套环境
| 环境 | 用途 |
|---|---|
| **dev(本地)** | 工程开发 |
| **PR 预览(临时)** | 每个 PR 自动起,验收用 |
| **staging(预发)** | 贴近 prod,上线最后一道 |
| **prod(生产)** | 真实用户 |

> 铁律:**prod 数据脱敏后才能下到其他环境。**

### 6.2 密钥管理
- **永不进代码、永不进 git**(gitleaks 扫)
- 本地 `.env`(进 .gitignore)+ 线上 secret manager(Vercel env / Doppler / GH Secrets)
- 最小权限

### 6.3 Feature Flag(发布神器)
> **核心:把"部署 deploy"和"发布 release"解耦。**
- 大功能**先合并、先上线,但 flag 关着**(用户看不到)
- 想发布→打开 flag;出问题→关掉 flag 即回滚
- 这是 GitHub Flow + 持续合并 main 的前提
- 起步用一张配置表;长大上 PostHog / Unleash

### 6.4 发布与回滚
- **持续部署**:合并 main → 自动 staging → 冒烟 → prod(或点一下)
- **回滚优先于修复**:线上出事先**止血**(rollback / 关 flag),再复盘
- **可观测**:基本日志 + 错误监控(Sentry)

> **为什么"回滚优先"**:事故时**恢复服务 > 查清原因**。先让用户恢复,再慢慢查。写进规范,慌乱时救你。

## 7. 角色与协作(RACI)

> 本章是 tech lead → PM/架构 最该吃透的一章:**谁对什么负最终责任。**

### 7.1 三空间模型(决定边界)
- **问题空间(What & Why)** → PM:需求、目标、AC、优先级
- **体验空间(Look & Feel)** → 设计(你们 PM 兼):手绘+vibe-code 原型
- **解决方案空间(How)** → 你/工程:技术方案、架构、接口、schema

### 7.2 RACI 表(A=最终负责 R=执行 C=被咨询 I=知会)

| 交付物 / 活动 | PM(兼设计) | 工程 | Tech Lead(你) |
|---|---|---|---|
| Discovery 一页纸 | **A/R** | C | C |
| 原型(线稿+vibe-code) | **A/R** | I | C |
| spec + AC + scope | **A/R** | C | C |
| **技术方案 / 架构** | I | C | **A/R** |
| 任务拆分 + 胃口 | C | C | **A/R** |
| **DoR 放行** | R(产品侧) | R(工程点头) | **A** |
| 写代码 | I | **A/R** | C |
| code review | I | R | A |
| **DoD 放行** | C | R | **A** |
| 上线 | I | R | A |
| ADR | C | C | **A/R** |

### 7.3 三条铁律
1. **每件事只能有一个 A**——"大家一起负责" = 没人负责。
2. **一人多帽,但角色不消失**——别让角色塌成"人人都干所有事"。
3. **守住空间边界**——PM 给问题,你给方案,谁也别替对方拍板。

> 职业地图:你现在是"解决方案空间"的 A;往 PM 走 = 逐步接管"问题空间"的 A。

## 8. 知识沉淀

| 类型 | 答什么 | 放哪 | 何时写 |
|---|---|---|---|
| **ADR(架构决策记录)** | **为什么这么选**(否决了什么) | `docs/adr/`,编号 | 做重要技术决策时 |
| **README / 文档** | 怎么跑、怎么用 | 跟代码一起 | DoD 时更新 |
| **Runbook** | 怎么部署/回滚、坏了怎么办 | `docs/runbook/` | 上线前 |

### 四条原则
1. **文档随代码走(docs-as-code)**:进 repo、有版本、PR 里一起审。别丢进没人看的 wiki。
2. **写"为什么",不写"是什么"**:代码说 what,文档补 **why/决策/取舍/踩过的坑**。
3. **AI 起草,人定稿**:PR diff → AI 出 ADR 草稿。
4. **沉淀写进 DoD**:不强制就永远被跳过。

> ADR 是 aspiring architect 的金矿——"**架构思考的存档**"。一年后翻 ADR,能看到自己决策水平的成长。

## 9. 新人 Onboarding

| 时间 | 目标 |
|---|---|
| **Day 1** | 照 setup 文档,**半天内本地把项目跑起来** |
| **Week 1** | 读 README + 本手册 + 最近 5 条 ADR;**跟一个 S 任务走完整流程一遍** |
| **全程** | 配一个 **buddy**(治 bus factor) |

> **Onboarding 是整份规范的"验收测试"**:新人多久能独立交付第一个任务?很久 = 规范/文档不成体系。它逼你把隐性知识显性化——治 bus factor 的根本。

## 10. 痛点对照表(证明规范是闭环)

| 痛点 | 被哪些机制治住 |
|---|---|
| **需求→开发对不齐** | Discovery + Shaping + **DoR(工程点头)** + 原型冻结/normative 标注 |
| **质量没保障** | **DoD** + CI 挡合并 + Review(AI+人)+ 测试标准 + 对 AC 验收 |
| **进度/任务不透明** | 看板 + WIP 上限 + 异步站会 + 流动指标 |
| **发布/知识沉淀乱** | 持续部署/回滚 + **ADR** + docs-as-code + DoD 强制沉淀 |
| **review 瓶颈 / bus factor** | AI 预审 + 评审轮换 + buddy + Onboarding + ADR |
| **范围蔓延** | **胃口上限** + scope 边界 + Shaping Review("能砍什么") |

> "完整"不是面面俱到,是**闭环**:每个痛点都有明确机制兜底。

## 附录 A:原型 / 设计稿 / PRD / MVP 的区别

团队高频用"原型"一词,易混淆。统一定义:

- **原型 Prototype** = 正式建造前做的"近似品",为**验证某事**、用完即弃(throwaway)。按两轴理解:
  - **轴 A 验证什么**:体验对不对(设计原型,设计师/PM 产)/ 技术行不行(技术原型/PoC,工程产)
  - **轴 B 保真度**:低保真(线框)↔ 高保真(可点击 / vibe-code 可运行)
  - "vibe-code 高保真原型" = 用代码做的高保真设计原型(可能兼做技术验证)。
- **设计稿 Mockup**:静态,只答"长什么样",**不验证交互**——严格说不是原型,是原型的材料。
- **PRD / Spec**:答"为什么做 + 规则 + 边界 + NFR"。**原型是一个例子,推不出完整规格;且分不清哪些细节是故意的。** 故有原型仍需 spec,只是 spec 更薄。
- **MVP**:会**真上线给用户**的最小产品,是**生产代码**;原型是内部 throwaway。
- ⚠️ **vibe-code 原型 ≠ 生产代码**:它没处理错误/边界/安全/可维护;定位是"验证",不是上线起点。
