## 0. 三十秒全局心智图

Python 给你三套并发武器，外加一个搅局的 GIL：

```
                        ┌─────────────────────────────────────────┐
                        │  GIL：同一进程内，同一时刻只有 1 个线程   │
                        │       能执行 Python 字节码                │
                        └─────────────────────────────────────────┘
                                          │
   ┌──────────────────────┬──────────────┴───────────────┬──────────────────────┐
   │                      │                              │                      │
threading            multiprocessing                 asyncio              concurrent.futures
(多线程)              (多进程)                        (协程/单线程)         (上面两者的统一壳)
   │                      │                              │
受 GIL 限制            每个进程独立 GIL               单线程，GIL 非问题
CPU 密集没用           CPU 密集真并行                 海量 I/O 等待最优
I/O 密集有效           代价高(序列化+进程启动)        一个阻塞卡死全场
对标 Java Thread       对标「多开 JVM 进程」          对标 goroutine(但更弱)
```

选型口诀（先记住，后面展开）：
- **CPU 密集 → multiprocessing**（绕开 GIL，吃满多核）
- **I/O 密集、并发不大 → threading / ThreadPoolExecutor**（简单够用）
- **I/O 密集、海量连接（上千上万）→ asyncio**（单线程事件循环，开销最低）

## 1. 核心差异：GIL（为什么 Python 并发和你熟的不一样）

### 1.1 现象：Java/Go 多线程吃满多核，Python 多线程吃不满

在 Java 里你开 4 个线程跑纯计算，4 核机器能接近 4 倍加速（线程 1:1 映射内核线程，真并行）。
在 Go 里 goroutine 由 runtime 调度到多个 OS 线程（GOMAXPROCS），同样真并行。

Python（指 CPython，下同）里你开 4 个线程跑纯计算，**不但没有 4 倍，反而比单线程还慢**：

```python
import threading, time

def cpu_bound():
    total = 0
    for i in range(10_000_000):
        total += i * i

# 单线程跑 4 次 ≈ 4.5s
# 4 线程并行    ≈ 4.9s（更慢！）—— 因为线程在抢 GIL，多了切换开销
```

原因就是 **GIL（Global Interpreter Lock，全局解释器锁）**：CPython 解释器里有一把全局互斥锁，**同一时刻只允许一个线程执行 Python 字节码**。所以 4 个线程在「执行 Python 代码」这件事上是串行的，多核形同虚设。

> 类比：把 GIL 想成「整个解释器只有一支笔」。你可以有 4 个人（线程），但要写字（执行字节码）必须先抢到那支笔。任何时刻只有一个人在写。

### 1.2 为什么 CPython 要有 GIL（面试常问）

不是设计失误，是历史包袱 + 工程权衡：

1. **引用计数的线程安全**：CPython 用引用计数管理内存（每个对象有 `ob_refcnt`）。如果没有 GIL，每次 `incref/decref` 都要加锁或用原子操作，所有对象操作都变慢——单线程程序也被拖累。
2. **C 扩展生态**：numpy、pandas 等海量 C 扩展默认依赖「有 GIL」这个前提来保证线程安全。去掉 GIL 会冲击整个生态。
3. **单线程性能**：有了 GIL，单线程代码完全不需要承担锁开销。

> 对比 Java：JVM 没有全局锁，靠 JMM（内存模型）+ 每对象的 monitor + GC 屏障来保证多线程内存安全，代价是实现复杂得多。Go 同理，runtime 直接面向多核设计。CPython 选了「简单」这条路。

### 1.3 GIL 什么时候释放（关键，决定了哪些场景多线程有用）

GIL 不是焊死的，它在这些时刻会被当前线程主动让出：

```python
data = sock.recv(4096)        # ① 阻塞 I/O（网络/文件）—— 等待时释放 GIL
time.sleep(1)                 # ② sleep —— 释放 GIL
np.dot(big_a, big_b)          # ③ C 扩展显式释放（Py_BEGIN_ALLOW_THREADS）
lock.acquire()                # ④ 等待同步原语时释放
# ⑤ 纯 Python 计算时：每隔约 5ms 强制检查一次，可能切给别的线程
#    （sys.getswitchinterval() 默认 0.005 秒）
```

**这就是为什么「I/O 密集用多线程有效，CPU 密集没用」**：
- I/O 密集：线程一发起 I/O 就交出 GIL，别的线程趁机执行 → 多线程能重叠等待时间 → 有效。
- CPU 密集：线程一直在执行字节码，每 5ms 才被迫交出一次，4 个线程轮流用同一支笔 → 没并行还多了切换 → 没用甚至更慢。

### 1.4 GIL 不等于「线程安全」（经典坑）

很多人以为「有 GIL，那 Python 多线程就不用加锁了」——**错**。GIL 保证的是「字节码级别」的串行，但一条 Python 语句往往编译成多条字节码，GIL 可以在中间切走：

```python
counter += 1
# 编译为大致三条字节码：
#   LOAD  counter   ← 读
#   ADD   1
#   STORE counter   ← 写
# GIL 可能在 LOAD 之后、STORE 之前切到别的线程 → 丢更新
```

所以 `counter += 1` 这种「读-改-写」在多线程下依然要加锁。

> 对标 Java：完全一样的道理。`i++` 在 Java 里也不是原子的（`getfield/iadd/putfield`），所以才有 `AtomicInteger`、`synchronized`。Python 这里的直觉和你的 Java 经验一致，只是别被「有 GIL」误导成「天然原子」。

### 1.5 free-threaded Python（前沿，了解即可）

Python 3.13 引入了实验性的 **no-GIL / free-threaded 模式**（PEP 703），可以编译出一个真正去掉 GIL 的解释器，让多线程吃满多核。但目前（2026 年）仍不成熟：单线程性能有回退、大量 C 扩展未适配。**面试可以提一句「了解 PEP 703 在去 GIL」，但生产默认还是按有 GIL 来设计。**

## 3. threading —— 对标 Java Thread / ExecutorService

### 3.1 基本用法

```python
import threading

def work(name):
    print(f"hello from {name}")

# 方式一：直接传函数（最常用，类似传 Runnable lambda）
t = threading.Thread(target=work, args=("t1",))
t.start()
t.join()          # 等它结束，语义同 Java Thread.join()

# 方式二：继承 Thread 重写 run（类似 extends Thread）
class Worker(threading.Thread):
    def run(self):
        print("running")
Worker().start()
```

直接用裸线程的场景其实不多，**生产里几乎都用线程池**：

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def fetch(url):
    return requests.get(url).status_code   # I/O 密集，等待时释放 GIL

with ThreadPoolExecutor(max_workers=20) as pool:   # 类似 newFixedThreadPool(20)
    futures = [pool.submit(fetch, u) for u in urls] # submit 返回 Future
    for f in as_completed(futures):                 # 按完成顺序取，类似 CompletionService
        print(f.result())                           # result() 阻塞取值，异常会在这里重新抛出
```

> 对标：`ThreadPoolExecutor` ≈ `ExecutorService`，`submit` ≈ `submit(Callable)`，`Future.result()` ≈ `Future.get()`，`as_completed` ≈ `ExecutorCompletionService`。`with` 块退出 = 自动 `shutdown(wait=True)`。

### 3.2 锁与同步原语（直觉和 Java 几乎一一对应）

```python
import threading

lock = threading.Lock()          # ≈ ReentrantLock（不可重入）
rlock = threading.RLock()        # ≈ 可重入锁（同一线程可多次 acquire）
sem = threading.Semaphore(5)     # ≈ Semaphore，5 个许可
cond = threading.Condition()     # ≈ wait/notify
event = threading.Event()        # ≈ 一次性闸门，set() 后所有 wait() 放行

# 推荐用 with，等价于 try/finally 里 acquire/release —— 对应你「unlock 要放 finally」的肌肉记忆
with lock:
    counter += 1                 # 临界区
```

**线程间通信首选 `queue.Queue`**（线程安全的阻塞队列，对标 `BlockingQueue`），天然解决生产者-消费者，不用自己玩锁：

```python
import queue, threading

q = queue.Queue(maxsize=100)     # 有界队列，满了 put 会阻塞（≈ ArrayBlockingQueue）

def producer():
    for item in source:
        q.put(item)              # 阻塞式投递
    q.put(None)                  # 毒丸，通知消费者结束

def consumer():
    while True:
        item = q.get()           # 阻塞式获取
        if item is None: break
        handle(item)
```

### 3.3 你需要重新记住的几个「不一样」

- **没有现成的 `AtomicInteger`/CAS**：标准库不提供原子类，计数器之类老老实实用 `Lock`。（CPython 某些单字节码操作恰好原子，但别依赖这种隐式行为。）
- **守护线程**：`t.daemon = True` 表示主线程退出时它跟着死，类似 Java 的 `setDaemon(true)`。
- **没有真正的「优雅中断」**：Python 线程**不能从外部强行 kill**，也没有 `Thread.interrupt()` 那套中断标志位机制。优雅停止靠你自己设一个 `threading.Event` 标志位，线程循环里检查它。这点比 Java 更原始。

## 4. multiprocessing —— Python 吃满多核的正路

GIL 锁的是「同一进程内」。所以想吃满多核做 CPU 密集计算，就**多开进程，每个进程一个独立解释器、独立 GIL**。

> 类比：既然一个解释器只有一支笔（GIL），那就多开几个解释器（进程），每人发一支笔。代价是这些人不在一个屋子里（独立地址空间），传东西得打包快递（序列化）。

```python
from concurrent.futures import ProcessPoolExecutor

def heavy(n):
    return sum(i * i for i in range(n))   # 纯 CPU

with ProcessPoolExecutor(max_workers=4) as pool:   # 一般设为 CPU 核数
    results = list(pool.map(heavy, [10**7] * 4))    # 这次是真·4 倍加速
```

### 4.1 进程的代价（和线程的本质区别）

| 维度 | threading（线程） | multiprocessing（进程） |
|---|---|---|
| 地址空间 | 共享，直接读写同一对象 | **独立**，对象要序列化传递 |
| 参数/返回值 | 直接传引用 | **pickle 序列化**，大对象很贵 |
| 启动开销 | 轻（~10μs） | 重（fork ~ms / spawn 更慢，要重启解释器） |
| 通信 | 共享变量 + 锁 / `queue.Queue` | `multiprocessing.Queue`、`Pipe`、共享内存 |
| 崩溃隔离 | 弱（一个挂全进程挂） | **强**（一个子进程崩溃不影响别人） |
| 吃多核 | ❌（GIL） | ✅ |

### 4.2 两个一定会踩的坑

**坑一：fork vs spawn（跨平台行为不同）**

- Linux 默认 `fork`：子进程复制父进程内存（写时复制 COW），快。
- **macOS（3.8+）和 Windows 默认 `spawn`**：子进程从头启动一个新解释器、重新 import 你的模块。

这导致 spawn 下你**必须**把入口代码放进 `if __name__ == "__main__":`，否则子进程重新 import 时会递归创建进程：

```python
def heavy(n): ...

if __name__ == "__main__":      # spawn 平台必须有这个守卫
    with ProcessPoolExecutor() as pool:
        pool.map(heavy, range(4))
```

**坑二：序列化开销**。给子进程传一个几百 MB 的 DataFrame，光 pickle 来回就能把多核收益吃掉。大数据共享用 `multiprocessing.shared_memory`（或干脆用 numpy/pandas 这类内部释放 GIL 的 C 扩展，连多进程都省了）。

## 5. asyncio —— 像 goroutine，但是「单线程协作式」

这是从 Java/Go 过来最容易误解的部分。先把关键差异钉死：

> **asyncio 不是 goroutine。** goroutine 是 Go runtime 在**多个 OS 线程**上**抢占式**调度的、可真并行的轻量线程；asyncio 是**单线程**上**协作式**调度的事件循环。它更像 Node.js 的事件循环，而不是 Go 的 GMP。

| | Go goroutine | Python asyncio |
|---|---|---|
| 线程数 | 多个 OS 线程（M:N） | **单线程**（1 个事件循环） |
| 调度 | 抢占式（runtime 强制切换） | **协作式**（只在 `await` 处让出） |
| 并行 | 真并行（吃多核） | 并发不并行（一核） |
| 阻塞一个 | 别的 goroutine 照跑 | **整个 loop 卡死** |
| 函数染色 | 无，goroutine 透明 | 有，`async`/`await` 会传染（见 §5.4） |

为什么 asyncio 还值得用？因为对**海量 I/O 等待**场景，它开销最低：单线程、无线程切换、无锁、一个协程几百字节~KB 级（对比线程栈 MB 级），轻松扛上万并发连接（C10k）。而且**单线程 = GIL 完全不是问题**。

### 5.1 基本结构

```python
import asyncio, httpx

async def fetch(client, url):
    r = await client.get(url)        # await：发起 I/O 并「让出」事件循环，去跑别的协程
    return r.status_code

async def main():
    async with httpx.AsyncClient() as client:
        tasks = [fetch(client, u) for u in urls]   # 创建一堆协程
        results = await asyncio.gather(*tasks)      # 并发跑，等全部完成（≈ WaitGroup）
    return results

asyncio.run(main())                  # 启动事件循环，跑到 main 结束
```

理解 `await` 的本质：**它是「主动让出执行权」的标记**。协程跑到 `await some_io()` 时说「我要等 I/O 了，事件循环你先去伺候别人」，I/O 就绪后再被调度回来。整个过程在一个线程里反复横跳。

> 对标 Go：`await client.get(url)` ≈ Go 里一个会阻塞的网络调用，但 Go 的阻塞由 runtime 透明处理（你写同步代码，runtime 帮你切 goroutine）；Python 要你**显式**用 `await` 标出每个让出点。这就是「协作式」——你不写 `await`，loop 永远不会从这里切走。

### 5.2 概念对照

| asyncio | 说明 | Go/Java 类比 |
|---|---|---|
| `async def` | 定义协程函数 | — |
| `await x` | 等待并让出 | Go 透明阻塞 / `Future.get` |
| `asyncio.create_task(coro)` | 后台并发跑一个协程 | `go func()` |
| `asyncio.gather(*coros)` | 并发跑一组并收集结果 | `WaitGroup` / `CompletableFuture.allOf` |
| `asyncio.TaskGroup`（3.11+）| 结构化并发，子任务异常自动取消同组 | — |
| `asyncio.Queue` | 协程间队列 | Go `channel` / `BlockingQueue` |
| `asyncio.Lock/Semaphore` | 协程级锁（不是线程锁！）| 限并发 |
| `task.cancel()` / `asyncio.timeout()` | 取消/超时 | Go `context.Context` |

`asyncio.Queue` 当 channel 用的生产者-消费者：

```python
async def producer(q):
    for item in source:
        await q.put(item)
    await q.put(None)

async def consumer(q):
    while (item := await q.get()) is not None:
        await handle(item)
```

### 5.3 头号杀手：在协程里调用同步阻塞代码

这是 asyncio 最常见的生产事故。因为是单线程协作式，**任何不让出的耗时操作都会冻结整个事件循环**，所有并发连接一起卡住：

```python
async def handler():
    time.sleep(1)              # ❌ 同步 sleep，冻结整个 loop 1 秒
    requests.get(url)          # ❌ 同步 HTTP，冻结直到返回
    data = open("f").read()    # ❌ 同步文件 I/O
```

正确做法二选一：
1. **换 async 库**：`asyncio.sleep`、`httpx`/`aiohttp`、`asyncpg`、`aiomysql`、`aiofiles`、`redis.asyncio`……
2. **没有 async 版本的库（或 CPU 密集）→ 丢线程池/进程池**，用 `run_in_executor` 桥接，不阻塞 loop：

```python
loop = asyncio.get_running_loop()
result = await loop.run_in_executor(None, blocking_fn, arg)   # 默认线程池
# CPU 密集则传一个 ProcessPoolExecutor 进去
```

> 详细排查手段（debug 模式、慢回调检测、看门狗、py-spy dump）见 `06b-python-debugging/02-asyncio-debugging.md`，不在此重复。

### 5.4 函数染色问题（function color，Go 开发会不适应）

`async` 函数只能被 `await`，而 `await` 只能写在 `async` 函数里。于是 async 会沿调用链一路「传染」：一个底层函数改成 async，所有调用它的上层都得跟着变 async。Go 没有这个问题（goroutine 透明），这是 asyncio 在工程上最被诟病的一点。**实务上的取舍：要么整条链路 async 到底，要么干脆用线程池，不要在两个世界之间反复横跳。**

## 6. concurrent.futures —— 线程池/进程池的统一外壳

它把 threading 和 multiprocessing 包成同一套 `Executor` 接口，**切换只改一个类名**：

```python
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

Executor = ProcessPoolExecutor if cpu_bound else ThreadPoolExecutor
with Executor(max_workers=8) as pool:
    results = list(pool.map(task, items))     # 有序返回
    # 或 submit + as_completed 按完成顺序取
```

记住对应关系即可：
- `ThreadPoolExecutor` → 受 GIL 限制，**I/O 密集**用。
- `ProcessPoolExecutor` → 独立 GIL，**CPU 密集**用，但有序列化/启动开销。

这是日常写并发代码最顺手的入口，比裸 `threading.Thread` / `multiprocessing.Process` 都推荐。

## 7. 实战：Web 服务里的「worker」是什么

你问到 "worker"——在 Python 后端里这个词通常指**部署层面的进程/线程模型**，正好把上面三套原语串起来。这是面试和实战的高频点：

### 7.1 gunicorn / uvicorn 的 worker

WSGI/ASGI 服务器靠「多 worker」绕开单进程 GIL 的吞吐瓶颈：

```bash
# WSGI（同步框架 Flask/Django）：多进程 worker，每进程一个解释器/GIL
gunicorn app:wsgi --workers 4 --worker-class sync          # 4 进程，CPU 核数级别

# 同步框架要扛高并发 I/O，可换 gevent/eventlet（协程 worker，monkey-patch 阻塞调用）
gunicorn app:wsgi --workers 4 --worker-class gevent --worker-connections 1000

# ASGI（异步框架 FastAPI）：每个 uvicorn worker 内部跑一个 asyncio 事件循环
gunicorn app:asgi --workers 4 --worker-class uvicorn.workers.UvicornWorker
uvicorn app:asgi --workers 4
```

心智模型：
- **worker 数 = 进程数**，用来吃多核 + 故障隔离（一个 worker 崩了 master 重启它）。经验值 `2 * 核数 + 1`（同步）或 `= 核数`（异步，单 worker 已能扛海量连接）。
- **每个 worker 内部**：同步 worker 靠线程/gevent 协程处理并发；异步 worker（uvicorn）靠 asyncio 事件循环。
- 这正是「多进程绕 GIL（§4）+ 进程内并发（§3/§5）」的组合拳。

> 对标 Java：gunicorn 多 worker ≈ 起多个 JVM 实例 + 前面挂个 Nginx/负载均衡；而单个 Tomcat 进程内的线程池，对应这里「一个 worker 内部的并发处理」。Java 单进程多线程就能吃满多核（无 GIL），所以 Java 世界很少需要「多进程 worker」这种部署形态——**这是 GIL 给 Python 部署架构带来的直接差异**。

### 7.2 Celery worker（异步任务队列）

后台任务、定时任务的事实标准，对标 Java 的 Spring `@Async` + 消息队列消费者：

```bash
celery -A app worker --concurrency=8           # 8 个 worker 进程/线程消费任务
celery -A app worker --pool=prefork            # 默认 prefork（多进程，CPU 密集任务）
celery -A app worker --pool=gevent -c 1000     # 协程池（I/O 密集任务，高并发）
```

`--pool` 选 prefork(进程)/threads(线程)/gevent(协程)，本质又回到「任务是 CPU 密集还是 I/O 密集」这个根本问题。

## 8. 选型决策树

```
任务是什么类型？
│
├── CPU 密集（计算/图像/序列化/加解密）
│   └── multiprocessing / ProcessPoolExecutor（核数个进程）
│       └── 共享大数据？→ shared_memory；或换 numpy/pandas（C 扩展内部已释放 GIL）
│
├── I/O 密集（HTTP/DB/文件/RPC）
│   ├── 并发连接 > 1000，且能用 async 库 → asyncio
│   ├── 要调用的库只有同步版本     → ThreadPoolExecutor
│   └── 并发连接 < 100，图省事     → ThreadPoolExecutor
│
└── 混合（I/O + 偶尔 CPU 密集）
    └── asyncio 主体 + run_in_executor(ProcessPoolExecutor) 把 CPU 活外包出去
```

混合型示例（事件循环管 I/O，CPU 活丢进程池）：

```python
async def handle_upload(req):
    data = await req.body()                                   # I/O：异步读
    loop = asyncio.get_running_loop()
    with ProcessPoolExecutor() as pool:
        out = await loop.run_in_executor(pool, resize_image, data)  # CPU：外包给进程池
    await s3.put_object(Body=out)                             # I/O：异步写
```

## 9. 三套模型速查总表

| 模型 | 适用 | GIL 影响 | 开销 | 真并行 | 通信 | Java/Go 类比 |
|---|---|---|---|---|---|---|
| threading | I/O 密集、并发不大 | I/O 时释放 | 低 | ❌ | 共享内存+锁 / `queue.Queue` | Java Thread/ExecutorService |
| multiprocessing | CPU 密集 | 每进程独立 | 高（pickle+启动） | ✅ | Queue/Pipe/共享内存 | 多开 JVM 进程 |
| asyncio | 海量 I/O 等待 | 单线程，无关 | 最低 | ❌ | `asyncio.Queue` | Go goroutine（但单线程协作式） |

## 10. 面试高频问答（你是为面试准备的）

**Q1：Python 多线程能利用多核吗？为什么？**
不能（指 CPython 的 CPU 密集场景）。因为 GIL 限制同一时刻只有一个线程执行字节码。I/O 密集场景下因为 I/O 会释放 GIL，多线程仍能提升吞吐。要吃多核做计算得用多进程。

**Q2：有了 GIL 还需要加锁吗？**
需要。GIL 只保证单条字节码不被打断，但 `x += 1` 这种「读-改-写」是多条字节码，GIL 可能在中间切走，仍会丢更新。和 Java 的 `i++` 非原子是一个道理。

**Q3：GIL 什么时候释放？**
阻塞 I/O、`time.sleep`、等待锁、C 扩展显式释放（如 numpy 计算），以及纯计算时每约 5ms（`sys.getswitchinterval()`）强制检查一次。

**Q4：asyncio 和多线程区别？什么时候用哪个？**
asyncio 单线程协作式（只在 await 让出），无线程切换/锁开销，适合海量 I/O 并发；多线程是内核抢占式调度，适合并发量不大或必须用同步库的 I/O 场景。CPU 密集两者都不行，用多进程。

**Q5：asyncio 和 Go 的 goroutine 像吗？**
表面像（都写「并发」代码），本质不同：goroutine 是多 OS 线程上抢占式调度、能真并行；asyncio 是单线程协作式、不并行。goroutine 阻塞一个不影响别的，asyncio 一个阻塞操作会冻结整个事件循环。

**Q6：协程为什么比线程「轻」？**
协程切换是用户态的函数调用级（约百纳秒级），不进内核、不刷 TLB；线程切换要陷入内核（约 1–10 微秒）。协程栈按需增长（KB 级），线程栈固定 MB 级，所以同样内存能开多得多的协程。

**Q7：CPU 密集任务在 Python 怎么做？**
multiprocessing 多进程绕开 GIL；或把热点计算下沉到释放 GIL 的 C 扩展（numpy/Cython）；或用 `ProcessPoolExecutor`；前沿可关注 no-GIL（PEP 703）。

**Q8：Web 服务为什么要配多个 worker？**
单个 Python 进程受 GIL 限制吃不满多核，多 worker（多进程）才能利用多核 + 故障隔离。这是 Java（单进程多线程即可吃满多核）所不需要的、由 GIL 导致的部署差异。

## 11. 一句话收尾

你脑子里那套 Java/Go 的并发知识（线程、池、锁、队列、协程、生产者-消费者）**基本可以直接迁移**，API 名字换一下即可。唯一要重装的认知是 **GIL**：它让「Python 多线程 = 真并行」这个等式不成立，于是逼出了「CPU 密集走多进程、I/O 海量走 asyncio、部署靠多 worker」这套 Python 特有的打法。抓住 GIL 这把钥匙，其余都是查文档的事。
