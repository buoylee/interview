# 00 · 心智模型与起步

> **为什么这章重要**:你已经会写代码了,缺的不是语法,而是 Python 的两条"世界观"——它怎么跑(执行模型),以及它怎么看待数据(一切皆对象 + 名字即绑定)。这两条立住了,后面九成的"陷阱"都不再是陷阱,而是同一条规则的自然推论。

## 一、Python 的设计哲学

在 REPL 里敲 `import this`,会打印出 "The Zen of Python"。不用背,记住几条会直接影响你写出的代码风格:

- **可读性至上**(Readability counts):代码是写给人看的,顺便给机器跑。
- **显式优于隐式**(Explicit is better than implicit):别玩隐藏的魔法,把意图写出来。
- **应该有一种(最好只有一种)显而易见的做法**:这跟 Perl/JS "条条大路" 的文化相反,也是为什么 Python 社区对"地道写法(Pythonic)"特别较真。

> 这不是鸡汤。面试官问"你觉得这段代码哪里不 Pythonic",考的就是你有没有内化这套价值观。第 16 章会把它落到具体写法。

## 二、执行模型:Python 是怎么跑起来的

很多 Java/Go 的人第一反应是问:"Python 是解释型还是编译型?" 这个二分法本身就不准确。真实流程是:

```
你的 .py 源码
   │  ① 编译(compile)——发生在导入/运行时,自动,无独立构建步骤
   ▼
字节码 bytecode(平台无关的指令)
   │  ② CPython 虚拟机逐条解释执行(eval loop)
   ▼
运行结果
```

所以准确说法是:**Python 先编译成字节码,再由虚拟机解释执行**。它有"编译"这一步,只是编译目标不是机器码,而是字节码;而且这步是隐式的,没有 `javac` 或 `go build` 那样的独立命令。

### `.pyc` 与 `__pycache__`

当你 `import` 一个模块时,CPython 会把它编译出的字节码缓存成 `.pyc` 文件,放在 `__pycache__/` 目录里。下次再导入,如果源码没变,就直接加载 `.pyc`,省掉重新编译。

- **只有被 `import` 的模块会生成 `.pyc`**;你直接 `python main.py` 运行的入口脚本本身不会缓存(它每次都重新编译,但这点开销可忽略)。
- `.pyc` 是优化缓存,不是"编译产物交付物"。删掉它程序照常跑,只是下次导入慢一点点。
- 文件名形如 `foo.cpython-311.pyc`,带上了解释器版本——不同版本的字节码不通用。

### 解释器 / REPL / 脚本 / 模块

- **REPL**:命令行直接敲 `python3`,进入交互式环境,一行一行求值。学习和验证假设的利器——本教程让你"猜输出"时,就该去 REPL 或临时脚本里跑一遍。
- **脚本**:`python3 app.py`,把文件当程序入口跑。
- **模块**:`python3 -m pkg.tool`,把它当模块运行(会正确设置包上下文)。`-m` 和直接给路径在导入行为上有区别,第 10 章细讲。

## 三、实现版图:CPython 只是其中一个

"Python"是**语言规范**,实现有好几个:

| 实现 | 是什么 | 用在哪 |
|------|--------|--------|
| **CPython** | 官方参考实现,C 写的,你 `python3` 装的几乎都是它 | 默认,生态最全 |
| PyPy | 带 JIT 的实现,纯 Python 计算可快数倍 | 计算密集、长时运行的服务 |
| 其他 | Jython(JVM)、GraalPy、MicroPython 等 | 特定平台/嵌入 |

**本教程一律以 CPython 为准**。很多"陷阱"(小整数缓存、引用计数、GIL)其实是 *CPython 的实现细节*,换个实现可能不成立——这点会在相关章节明确点出,因为面试官爱追问"这是语言规范还是实现细节"。

> **版本**:用 3.11 或更新。3.10 给了 `match`、`X | Y` 联合类型;3.11 大幅提速并加了 `except*`;3.12 给了新泛型语法 `class Foo[T]`;3.13 引入实验性 JIT 和可关 GIL 的 free-threaded 构建(第 13、15 章会提)。本教程示例在 3.11 实测,3.12+ 专属语法会标注。

## 四、总纲心智:一切皆对象 + 名字即绑定

这是整本教程的地基,务必现在就建立直觉(第 01 章展开细节)。

**(1) 一切皆对象。** 整数、字符串、函数、类、模块——全是对象。每个对象有三样东西:

- **身份(identity)**:它在内存里是哪一个,用 `id()` 看,`is` 比较的就是它;
- **类型(type)**:它是什么,用 `type()` 看;
- **值(value)**:它装了什么。

```python
x = 42
print(id(x))      # 某个整数,代表对象身份(地址)
print(type(x))    # <class 'int'>
print(type(print))  # <class 'builtin_function_or_method'>  ← 函数也是对象
```

**(2) 名字只是绑定,不是盒子。** 这是跟 Java/Go 最根本的思维差异。

在 Java 里 `int x = 42;` 你会想象一个叫 `x` 的盒子,里面装着 42。**在 Python 里没有盒子**。`x = 42` 的含义是:先有一个值为 42 的 int 对象,然后让名字 `x` *指向(绑定到)* 它。名字是贴在对象上的标签,赋值就是把标签贴到另一个对象上。

```python
x = 42      # 名字 x 绑定到 int 对象 42
y = x       # 名字 y 也绑定到同一个对象,不是“复制一份”
x = "hi"    # x 改贴到一个 str 对象;y 还指着 42,纹丝不动
```

为什么这条这么重要?因为它直接解释了后面一连串"为什么":为什么 `b = a` 之后改 `a` 会影响 `b`(当对象可变时)、为什么可变默认参数会累积、为什么 `[[0]*3]*3` 三行一起变。**它们不是孤立的怪癖,而是"名字是标签、对象可共享"这一条规则的不同表现**。第 01 章把这条规则讲透。

## Java/Go 对照框

| | Java / Go | Python(CPython) |
|--|-----------|------------------|
| 构建步骤 | `javac` / `go build` 显式编译 | 无独立步骤,导入/运行时自动编译成字节码 |
| 编译目标 | JVM 字节码 / 原生机器码 | Python 字节码(`.pyc` 缓存) |
| 执行 | JVM(含 JIT)/ 直接跑机器码 | CPython 虚拟机解释(3.13 起有实验 JIT) |
| 变量 | `int x` 是盒子;对象类型存引用 | 名字一律是绑定/标签,指向堆上对象 |
| `==` 语义 | 基本类型比值,对象比引用 | `==` 比值,`is` 才比身份(≈ Java 的 `==` 对象比较) |

一句话:**Java/Go 的"变量是带类型的盒子"直觉,在 Python 里要换成"名字是贴在对象上的标签"**。带着盒子直觉读 Python,迟早被别名和可变性坑到。

## 章末面试卡

**Q1. Python 是解释型还是编译型语言?**
都沾边但说法要准:Python 源码会先被**编译成字节码**,再由解释器(CPython 虚拟机)**逐条解释执行**。它有编译步骤,只是目标是平台无关的字节码、且隐式自动完成,没有独立的构建命令。

**Q2. `.pyc` 文件是什么?为什么有时有有时没有?**
是模块编译后的字节码缓存,存在 `__pycache__/`,源码没变时直接复用以省去重编译。只有被 `import` 的模块会生成;直接运行的入口脚本不缓存。删掉不影响正确性,只影响下次导入速度。

**Q3. CPython 和 PyPy 什么区别?**
CPython 是官方 C 实现(默认、生态最全、用引用计数 + GIL);PyPy 带 JIT,纯计算场景快数倍但生态/扩展兼容性弱一些。"Python"是规范,二者是不同实现——小整数缓存、GIL 这类是 *CPython 实现细节*,不是语言规范。

**Q4(心智题). `y = x` 是复制了一份吗?**
不是。`x`、`y` 是两个名字,绑定到**同一个对象**。是否"互不影响"取决于对象**可变还是不可变**:不可变(int/str)时改 `x` 只是让 `x` 改指别处,`y` 不变;可变(list/dict)时通过任一名字就地修改,另一个看得到。详见第 01 章。
