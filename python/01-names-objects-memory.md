# 01 · 名字、对象与内存模型

> **为什么这章重要**:这是全书地基。Python 里几乎所有"诡异行为"——别名互相影响、可变默认参数累积、`[[0]*3]*3` 三行一起变——都不是孤立怪癖,而是**同一条规则**的推论:*名字是贴在对象上的标签,对象可以被多个标签共享,而对象有的可变有的不可变*。把这条吃透,后面就没有"陷阱",只有"意料之中"。

## 一、名字是绑定,不是盒子

Java 里 `int x = 42;`,你脑子里是一个叫 `x` 的盒子装着 42。**Python 没有盒子**。`x = 42` 的真实含义是:

1. 先在堆上有一个值为 42 的 `int` 对象;
2. 让名字 `x` *绑定到(指向)* 这个对象。

名字是一张贴在对象上的标签。赋值 = 把标签贴到某个对象上。于是:

```python
x = 42       # 标签 x 贴到对象 42 上
y = x        # 标签 y 贴到“同一个对象”上,不是复制一份
x = "hi"     # 把 x 撕下来改贴到对象 "hi" 上;y 还贴在 42 上,没动
```

执行完 `x` 是 `"hi"`,`y` 还是 `42`。重点:**`y = x` 从不复制对象**,它只是让两个名字指向同一个对象。会不会"互相影响",取决于那个对象能不能被原地改——这就引出可变性。

## 二、可变 vs 不可变

| 不可变(immutable) | 可变(mutable) |
|---|---|
| `int`、`float`、`bool`、`str`、`bytes`、`tuple`、`frozenset`、`None` | `list`、`dict`、`set`、`bytearray`、绝大多数自定义对象 |

- **不可变**:对象一旦创建,值不能改。你以为"改了"它,其实是新建了一个对象再让名字改指。`x += 1` 不是把 42 改成 43,而是算出新对象 43 让 `x` 改指。
- **可变**:可以原地修改内容(`list.append`、`dict[k]=v`),对象身份不变,所有指向它的名字都看得到变化。

```python
s = "abc"
print(id(s))
s += "d"          # 看似修改,其实造了新 str "abcd"
print(id(s))      # id 变了——是新对象

lst = [1, 2]
print(id(lst))
lst.append(3)     # 原地修改
print(id(lst))    # id 不变——还是同一个对象
```

## 三、`is` 与 `==`:身份 vs 值

- `==` 比**值**(背后调 `__eq__`):两个对象内容相不相等。
- `is` 比**身份**:是不是内存里同一个对象(等价于 `id(a) == id(b)`)。

**99% 的场景你要的是 `==`。** `is` 只该用来跟单例比:`x is None`、`x is True`、`x is False`。

```python
a = [1, 2, 3]
b = [1, 2, 3]
print(a == b)    # True  值相等
print(a is b)    # False 两个不同对象
c = a
print(a is c)    # True  同一个对象(别名)
```

### 小整数缓存与字符串驻留(CPython 实现细节,别依赖)

你可能见过面试题"`a is b` 对小数成立、对大数不一定",根源是 CPython 的两个优化:

```python
a = 256; b = 256
print(a is b)            # True —— -5~256 的小整数被解释器预先缓存,全局共享

x = 1000
y = int("1000")          # 运行时构造,绕开编译期优化
print(x is y)            # False —— 1000 不在缓存里,是两个不同对象
print(x == y)            # True  —— 但值相等
```

- **小整数缓存**:CPython 启动时把 `-5` 到 `256` 这些常用整数预先建好、全局复用。所以这个区间内 `is` 永远 True。
- **字符串驻留(interning)**:像标识符的字符串字面量(`"hello"`)会被驻留复用,`"hello" is "hello"` 常为 True。
- **`257` 的坑**:`a = 257; b = 257; a is b` 的结果**看上下文**——在*同一段被一起编译的代码*里(脚本、函数体),编译器会把相等的常量去重,于是是 `True`;但在*真交互式 REPL 里逐行敲*(每行单独编译)或跨独立模块,就是 `False`。

**结论**:`is` 对非缓存对象的结果是实现细节,会随版本/上下文变。**判断值相等永远用 `==`**;`is` 留给 `None`。这正是面试想考的"你知不知道这是实现细节"。

## 四、别名:可变对象的"牵一发动全身"

既然 `b = a` 让两个名字指同一个对象,那么对**可变**对象,通过任一名字的原地修改,另一个都看得见:

```python
lst = [1, 2]
alias = lst
alias.append(3)
print(lst)        # [1, 2, 3] —— 通过 alias 改的,lst 也变了
```

这不是 bug,是"同一个对象、两个标签"的必然。函数传参同理:把 list 传进函数,函数里 `append` 会改到外面的那个 list(Python 是"传对象引用"——准确说是 *call by sharing*)。

### 经典坑:`[[0]*3]*3`

```python
grid = [[0]*3]*3
grid[0][0] = 1
print(grid)       # [[1, 0, 0], [1, 0, 0], [1, 0, 0]] —— 三行一起变!
```

为什么?`[0]*3` 先造出**一个**内层 list `[0,0,0]`;外层 `*3` 复制的是**这个 list 的引用三次**,于是三行其实是同一个对象的三个别名。改一行 = 改三行。正解是让每行独立新建:

```python
grid = [[0]*3 for _ in range(3)]
grid[0][0] = 1
print(grid)       # [[1, 0, 0], [0, 0, 0], [0, 0, 0]] —— 各自独立
```

> 记住根因:`*` 复制引用,推导式每次迭代新建对象。这跟可变默认参数(第 04 章)、类变量共享(第 05 章)是同一个"共享可变对象"母题的不同马甲。

## 五、浅拷贝 vs 深拷贝

想要"独立的一份"而不是别名,就得拷贝。但拷贝分两层:

```python
import copy
orig = [[1, 2], [3, 4]]
shallow = copy.copy(orig)       # 浅:新外层 list,但内层还是共享的引用
deep = copy.deepcopy(orig)      # 深:递归复制,内层也是新的

orig[0].append(99)
print(shallow)   # [[1, 2, 99], [3, 4]] —— 浅拷贝共享内层,看得到改动
print(deep)      # [[1, 2], [3, 4]]     —— 深拷贝完全隔离
```

- **浅拷贝**(`copy.copy`、`list(x)`、`x[:]`、`dict(x)`):只复制最外层容器,里面的元素仍是原对象的引用。
- **深拷贝**(`copy.deepcopy`):递归复制到底,彻底隔离。代价是慢、且要处理循环引用(`deepcopy` 自己能处理)。

经验:**只要容器里装的是可变对象,又想要独立副本,就得想清楚浅还是深**。多数 bug 来自"以为拷了一份,其实共享内层"。

## 六、内存怎么回收:引用计数 + 分代 GC(入门)

CPython 主要靠**引用计数**管理内存:每个对象记着"有多少个引用指向我",归零立刻回收。

```python
import sys
a = []
# getrefcount 自己会临时持有一个引用,所以读数比你想的多 1
print(sys.getrefcount(a))   # 比如 2(a 一个 + getrefcount 参数一个)
b = a
print(sys.getrefcount(a))   # 3(多了 b)
del b
print(sys.getrefcount(a))   # 回到 2
```

引用计数有个致命短板:**循环引用**。`a.ref = b; b.ref = a`,两者引用计数都不为 0,即使外部再也够不到它们,也不会被回收。为此 CPython 额外配了一个**分代垃圾回收器(generational GC)**,专门检测并回收这类循环垃圾。

这里建立直觉即可——*引用计数管大头、GC 兜循环引用* 的细节(三代、`gc` 模块、为什么这影响 `__del__`)放到[第 15 章](15-cpython-internals-performance.md)讲透。

## Java/Go 对照框

| | Java / Go | Python |
|--|-----------|--------|
| 变量 | 基本类型是值(盒子);对象/struct 指针是引用 | 名字一律是绑定,指向堆对象;无"基本类型盒子" |
| 相等 | `==` 对对象比引用,`.equals()` 比值 | 反过来:`==` 比值,`is` 才比引用/身份 |
| 传参 | 基本类型传值;对象传引用值(Java)/ 值或指针(Go) | 一律传对象引用(call by sharing):传可变对象进函数,函数内修改外部可见 |
| 拷贝 | `clone()`/拷贝构造,需自己区分深浅 | `copy.copy` / `copy.deepcopy` 显式区分 |
| 回收 | 纯 GC(分代/并发标记清除),无引用计数 | 引用计数为主 + 分代 GC 兜循环引用 |

最容易栽的两点:(1) 把 `==`/`.equals` 的直觉套过来,误用 `is` 比值;(2) 以为传 list 进函数是传副本,结果函数改到了外面的数据。

## 章末面试卡

**Q1. `is` 和 `==` 有什么区别?什么时候用哪个?**
`==` 比值(调 `__eq__`),`is` 比身份(是否同一对象,等价 `id` 相等)。判断相等用 `==`;`is` 只用于跟 `None`/`True`/`False` 这类单例比较。

**Q2. 为什么 `a=256;b=256; a is b` 是 True,换成大数就不一定?**
CPython 缓存了 `-5~256` 的小整数对象全局复用,所以这区间 `is` 恒 True;区间外的整数是不同对象,`is` 结果取决于编译上下文(同一编译单元内常量会被去重,跨单元/交互式逐行则不会)。这是 *CPython 实现细节*,不能依赖,值比较一律用 `==`。

**Q3(猜输出).**
```python
a = [1, 2]
b = a
b.append(3)
print(a)          # ?
```
`[1, 2, 3]`。`b = a` 没复制,`a`、`b` 是同一个 list 的两个名字,`b.append` 就是改那个 list。

**Q4(猜输出).**
```python
grid = [[0] * 2] * 2
grid[0][0] = 9
print(grid)       # ?
```
`[[9, 0], [9, 0]]`。外层 `*2` 复制的是同一个内层 list 的引用,两行是别名。要独立用 `[[0]*2 for _ in range(2)]`。

**Q5. 浅拷贝和深拷贝的区别?什么时候必须深拷贝?**
浅拷贝(`copy.copy`/`list()`/`[:]`)只复制最外层,内层元素仍共享引用;深拷贝(`copy.deepcopy`)递归复制彻底隔离。当容器里装着可变对象、且你要的副本必须与原件完全独立时,才需要深拷贝。

**Q6. Python 怎么管理内存?会内存泄漏吗?**
以引用计数为主(计数归零立即回收),辅以分代 GC 回收引用计数处理不了的循环引用。仍可能"泄漏":比如全局容器一直持有对象、缓存不清理,或 C 扩展里的引用问题——这些引用计数不为零,GC 也不会动。
