# 00 · 镜头：从"能答"到"能设计"

> 这一章不教任何具体资源，只教一种**读法**。把它装进脑子，后面 5 个资源章都是它的实例化。

---

## 一、盲点：你卡在"能答"，架构师在"能设计"

同样懂 OS，资深和架构师的差别不在"知道多少机制"，而在**什么时候用**：

- **资深 · 能答 · 反应式**：服务慢了 / 报错了 → 看指标 → 解释是哪类资源的病 → 排查修复。这是 `linux-handson` 教你的，是**事后**的能力。
- **架构师 · 能设计 · 预判式**：还没写代码 / 还没定架构 → **先想清楚这套设计会怎么消耗资源、会撞上哪条 OS 约束** → 让问题根本不发生。这是**事前**的能力。

举个对照你立刻明白：

| 同一个原语 | 能答（事后） | 能设计（事前） |
|---|---|---|
| "一个连接 = 一个 fd + 内核缓冲 + 队列" | 线上 `CLOSE_WAIT` 堆积 → 定位到没关连接 | 设计时就决定用连接池、长连接，**算好** fd 上限和池子大小 |

左边是被问题推着走，右边是设计时就把问题挡在门外。**本课就是教你把任何 OS 原语从左边用法翻到右边用法。**

---

## 二、统一反射模板（核心）

拿到**任何一个 OS 原语**，条件反射地问三件事：

```
①  它在运行期表现成什么指标？        ← 反应式（linux-handson 教你读）
        ↓
②  它在设计期约束了什么？            ← 预判式（本课的起点）
        ↓
③  这个约束逼出哪个具体决策？怎么定量？  ← 架构决策 + 指向深矿的容量公式
```

- **第①问**你已经会了（或回 `linux-handson` 补）。本课从**第②问**起步。
- **第③问**是落点：一个**能写进设计文档的决策**（选什么并发模型、池子多大、limit 设多少、选哪种存储引擎），外加"怎么用数字定这个参数"的反射。完整的数学推导**不在本课展开**，指向 `concurrency-capacity` / `perf` 深矿。

每个资源章的第二段"原语 → 决策映射表"，就是把这个模板套在该资源的几条核心原语上，列成三列：`OS 原语 | 资深·能答 | 架构师·能设计`。

## 四、怎么配套用

- **往下打地基**（机制不熟时）：[`linux-handson/`](../../linux-handson/) —— 每条原语的运行期机制 + 动手实验。本课只引用，不重讲。
- **往深挖容量**（要完整数学/深决策时）：[`concurrency-capacity/`](../../concurrency-capacity/)（Little 定律推容量）、`performance-tuning-roadmap/11·12`、`system-design/`。本课的指针会精确指到对应章。

记住一句话：**本课薄，是因为深矿已经挖好了。它的价值是那张"原语 → 决策"的反射地图，不是重新挖矿。**

## 小结

- 资深=能答=反应式（事后解释），架构师=能设计=预判式（事前挡住）。
- 反射模板三问：①运行期什么指标 → ②设计期约束什么 → ③逼出哪个决策、怎么定量。
- 后面 5 章 = 这个模板分别套在 CPU / 内存 / 存储 / 网络 / 隔离 上。
