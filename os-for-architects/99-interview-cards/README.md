# 99 · 面试卡：OS → 设计（能答 → 能设计）

> 临考速记。每张卡两层：**能答**（资深都会说的，反应式）→ **能设计**（架构师再往上一层，预判式）。面试官真正想听的是第二层。回对应章深读。

---

## CPU / 并发 → [`../01`](../01-cpu-and-scheduling/)

**Q：线程池/worker 大小怎么定？**
- 能答：根据 CPU 核数和负载来调。
- 能设计：先分清 CPU 密集还是 IO 密集。CPU 密集 ≈ 核数；IO 密集用 Little 反推 —— 在途请求 = QPS × 单请求延迟，据此定能扛住这么多等待的并发（线程数 ≈ 核数 ÷ (1−阻塞占比)，或换协程/async 模型）。不是拍脑袋 200。

**Q：为什么线程不是越多越好？**
- 能答：线程多了上下文切换开销大。
- 能设计：每个线程有栈内存 + 调度成本；超过承载点后，加线程只增加 cs 和内存、不增吞吐。容量应由 Little 和核数推出，不是越大越好。

## 内存 → [`../02`](../02-memory-and-paging/)

**Q：容器 memory limit 怎么设？为什么会被 OOMKill？**
- 能答：设得比应用用量大一点。
- 能设计：limit = 堆 + 堆外 + 栈 × 线程数 + page cache 余量。最常见错误是 limit ≈ 堆大小 —— 堆外和 cache 一涨就触顶被 OOMKill。要按总内存预算留余量。

**Q：page cache 算不算你的内存？**
- 能答：page cache 是可回收的文件缓存。
- 能设计：在 cgroup 里它**算进**容器内存统计、会触发 limit。但它也是"免费缓存层" —— 读密集服务靠它把热点常驻、点查不落盘。设计时既要给它留空间，又要把它当缓存资产用。

## 存储 IO → [`../03`](../03-storage-io/)

**Q：顺序写 vs 随机写怎么影响存储选型？**
- 能答：顺序快、随机慢。
- 能设计：写密集且能容忍读放大 → LSM（把随机写转顺序写）；读多、要点查/范围查 → B+ 树 + 覆盖索引。WAL 是"把随机写变顺序写"的通用手法。选引擎本质是选你的 IO 形态。

**Q：acks / fsync 怎么权衡？**
- 能答：fsync 保证落盘但慢。
- 能设计：持久化强度 vs 吞吐的权衡。每写都 fsync 最安全但吞吐崩；组提交/批量 fsync、或 acks=1 在丢失窗口和吞吐间取平衡。按业务能容忍的丢失窗口来定，而不是默认最强。

## 网络 / 连接 → [`../04`](../04-network-and-connections/)

**Q：连接池大小怎么定？**
- 能答：根据并发调。
- 能设计：用 Little —— 到下游的连接数 = 下游 QPS × 下游单次延迟。池子大一点点留余量即可；太大反而压垮下游（连接也吃下游内存/CPU）。不是越大越好。

**Q：为什么用长连接？TIME_WAIT 堆积怎么设计避免？**
- 能答：长连接省握手开销，TIME_WAIT 是主动关闭方的正常状态。
- 能设计：短连接每次新建 = 握手 + TIME_WAIT 占用源端口，高 QPS 下端口耗尽。设计上用连接池 + keep-alive 复用连接，从根上不产生大量 TIME_WAIT，而不是事后调内核参数。

## 隔离 / cgroup → [`../05`](../05-isolation-and-cgroups/)

**Q：limit / request 怎么设？**
- 能答：request 是保证量，limit 是上限。
- 能设计：request = 稳态用量（调度按它分配、保证不被饿死），limit = 峰值 + 余量。延迟敏感服务 CPU limit 别设太低 —— CFS throttling 会在峰值把 P99 拉爆。关键服务给 Guaranteed QoS（request=limit）。

**Q：怎么避免 noisy neighbor？**
- 能答：给容器设资源限制。
- 能设计：用 cgroups 做资源隔离 + bulkhead 隔板，延迟敏感服务别和批处理/离线任务混部抢 CPU；关键路径独占或绑核，并设计故障传播边界（线程池/连接隔离），让一个租户的过载不拖垮其他。
