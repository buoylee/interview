# 00 - 架构总览：OpenCode 核心数据流

> OpenCode 是一个开源的 AI 编程助手 CLI，核心代码量不大（~7K 行核心逻辑），但架构设计精巧。
> 本文是整个深度剖析系列的索引与导读，用一条完整的数据流把 5 篇文章串起来。

---

## 一句话定位

OpenCode 的本质是一个 **消息驱动的 while(true) 循环**：

```
用户说一句话 → 组装 prompt → 选模型 → 调 LLM → 流式处理响应 → 执行工具 → 循环回去
```

没有复杂的 Agent 编排框架，没有 DAG 调度引擎。一个循环 + 数据库持久化 = 全部核心逻辑。

## 核心源文件地图

所有核心代码在 `packages/opencode/src/` 下，按职责分布：

```
session/

  processor.ts     (430 行)    ← 流事件消费者：fullStream → 持久化 parts
  llm.ts           (277 行)    ← LLM 调用层：prompt 拼接 + streamText()
  index.ts         (893 行)    ← Session CRUD、消息持久化、费用计算
  instruction.ts   (192 行)    ← AGENTS.md / CLAUDE.md 发现与加载
  compaction.ts    (331 行)    ← 上下文压缩：prune + LLM 摘要
  system.ts        (72 行)     ← 模型专用 prompt + 环境信息

agent/
  agent.ts         (343 行)    ← Agent 定义：7 个内置 agent，权限合并链

  provider.ts      (1,405 行)  ← 20+ 厂商统一路由：SDK 管理、模型查找、费用
```

## 数据流详解：一次完整的对话循环

以用户输入 `"帮我修复 src/index.ts 的类型错误"` 为例：

### Phase 1: 消息创建 → [01-主循环与调度](./01-主循环与调度.md)

```
SessionPrompt.prompt()
  │
  ├── createUserMessage("帮我修复 src/index.ts 的类型错误")
  │     → 检测无文件引用、无 @agent、无 MCP 资源
  │     → 写入数据库，生成 UserMessage
  │
  └── loop({ sessionID })  ← 进入主循环
```

### Phase 2: Prompt 组装 → [02-Prompt组装与系统指令](./02-Prompt组装与系统指令.md)

```
loop() 第 1 轮:
  │
  ├── 加载消息历史 (MessageV2.stream → filterCompacted)
  │
  ├── 解析当前 Agent (默认: "general")
  │     → Agent.info: { mode: "chat", permission: [...], model: undefined }
  │
  └── 组装 system prompt:
        system[0] = PROMPT_ANTHROPIC (claude 模型)
                    + "You are an AI assistant..."
        system[1] = "Platform: darwin, Model: claude-sonnet-4-..., CWD: /Users/..."
        system[2] = "Available skills: commit, review-pr, ..."
        system[3] = CLAUDE.md 内容 (findUp 查找)
```

### Phase 3: Provider 路由 → [03-Provider模型路由](./03-Provider模型路由.md)

```
LLM.stream():
  │
  ├── Provider.getModel("anthropic", "claude-sonnet-4-...") → Model 元数据
  │     → { id, cost, limit: { context: 200000 }, capabilities: [...] }
  │
  ├── Provider.getLanguage(model) → LanguageModelV2
  │     → BUNDLED_PROVIDERS["@anthropic-ai/sdk"]()  (缓存命中)
  │     → CUSTOM_LOADERS["anthropic"](sdk)           (注入 headers)
  │     → sdk.languageModel(modelID)
  │
  └── streamText({ model, system, messages, tools, ... })
```

### Phase 4: 工具桥接 → [04-工具系统](./04-工具系统.md)

```
resolveTools():
  │
  ├── ToolRegistry.tools(model)
  │     → [read, edit, write, bash, grep, glob, ls, ...]  (模型适配后的工具列表)
  │
  ├── 为每个工具创建 AI SDK 包装:
  │     tool.init() → { description, parameters, execute }
  │     → Zod schema 转 JSON Schema
  │     → execute 包装: 注入 ctx (sessionID, messageID, ask, metadata)
  │
  └── Plugin.toolHooks() → 允许插件修改工具行为
```

### Phase 5: 流式处理 → [05-流式处理与上下文管理](./05-流式处理与上下文管理.md)

```
SessionProcessor.process():
  │
  ├── LLM 返回: "让我读取文件来分析错误..."
  │     text-start → 创建 TextPart
  │     text-delta → 增量推送到 UI
  │     text-end   → 完成
  │
  ├── LLM 调用工具: read({ path: "src/index.ts" })
  │     tool-input-start → 创建 ToolPart (pending)
  │     tool-call        → 状态变为 running
  │       └── execute:
  │             ctx.ask({ permission: "read", patterns: ["src/index.ts"] })
  │             → Permission 评估 → allow
  │             → 读取文件内容 → 返回结果
  │     tool-result      → 状态变为 completed
  │
  ├── finish-step → 记录 token 用量、费用
  │
  └── 返回 "continue" (因为有 tool-calls)
```

### Phase 6: 循环继续

```
loop() 第 2 轮:
  │
  ├── 消息历史现在包含: [user, assistant(text + tool_call + tool_result)]
  │
  ├── LLM 看到文件内容，分析类型错误
  │
  ├── LLM 调用: edit({ path: "src/index.ts", old_string: "...", new_string: "..." })
  │     → 9 级模糊匹配找到目标位置
  │     → 执行替换
  │     → LSP 诊断反馈
  │
  └── 返回 "continue"

loop() 第 3 轮:
  │
  ├── LLM 确认修复完成，输出总结
  │
  ├── finish_reason = "end_turn" (不是 "tool-calls")
  │
  └── 退出循环，返回最终 assistant message
```

## 5 个关键设计决策

### 1. 消息即状态（Messages-as-State）

OpenCode 没有内存中的 `RunState` 对象。所有状态都是数据库里的消息序列。
每次循环开始都重新加载消息，扫描 `lastUser`/`lastAssistant`/`lastFinished`。

**好处**：进程崩溃后可以恢复；多端（CLI + Desktop）看到一致状态。
**代价**：每轮循环都要遍历消息，O(n) 扫描。

→ 详见 [01-主循环与调度 § 消息扫描](./01-主循环与调度.md)

### 2. 单 Agent 多模式（vs 多 Agent 编排）

7 个内置 Agent 不是独立的运行时实体，而是**配置对象**：

| Agent | 作用 | 关键差异 |
|-------|------|----------|
| general | 通用对话 | 完整工具集 |
| plan | 规划模式 | 只有 read/grep/glob/web 工具 |
| build | 代码生成 | 步骤限制 + 自动验证 |
| explore | 代码探索 | 精简工具集 + 专用 prompt |
| compaction | 上下文压缩 | 无工具，只做摘要 |
| title | 标题生成 | 无工具，单次调用 |
| summary | 摘要生成 | 无工具，单次调用 |

同一个 `loop()` 循环，不同 Agent 只是换了 prompt + 工具集 + 模型。

→ 详见 [02-Prompt组装与系统指令 § Agent 系统](./02-Prompt组装与系统指令.md)

### 3. 工具由 SDK 自动执行

OpenCode 不自己管理工具调用的生命周期。Vercel AI SDK 的 `streamText()` 在收到 LLM 的 tool_call 后**自动执行工具并将结果发回 LLM**（通过 `maxSteps`）。

OpenCode 只负责：
- 注册工具（`resolveTools()`）
- 监听执行事件（`tool-call` / `tool-result`）
- 持久化结果到数据库

→ 详见 [04-工具系统 § resolveTools 桥接](./04-工具系统.md)

### 4. 双层上下文管理：Prune + Compaction

```
上下文快满了？
  │
  ├── 第 1 层: Prune（零成本）
  │     标记旧的工具输出为 compacted
  │     保护最近 2 轮 + 40K token
  │     不调用 LLM
  │
  └── 第 2 层: Compaction（有成本）
        用 compaction agent 生成摘要
        替换被标记的消息
        需要调用 LLM
```

先 prune 腾空间，不够再 compaction。大部分场景下 prune 就够了。

→ 详见 [05-流式处理与上下文管理 § Compaction](./05-流式处理与上下文管理.md)

### 5. Provider 的 6 步加载链

```
models.dev 数据库 → 用户 config 合并 → 环境变量检测 → 认证信息 → 插件扩展 → Custom Loaders
```

20+ 家厂商的差异全部在 `provider.ts` 一个文件里抹平。模型实例和 SDK 实例双层缓存，避免重复创建。

→ 详见 [03-Provider模型路由 § 6 步加载链](./03-Provider模型路由.md)

## 与 OpenAI Agents SDK 的设计对比

| 维度 | OpenCode | OpenAI Agents SDK |
|------|----------|-------------------|
| **状态管理** | 消息即状态（数据库持久化） | RunState 内存对象 |
| **Agent 模型** | 配置对象，共享同一循环 | 独立 Agent 类，可编排 handoff |
| **工具执行** | Vercel AI SDK 自动执行 | Runner 手动管理工具调用循环 |
| **上下文管理** | Prune + Compaction 双层 | 无内置方案 |
| **多模型支持** | 20+ Provider 统一路由 | 仅 OpenAI API |
| **持久化** | SQLite 数据库 | 无（纯内存） |
| **UI 集成** | 流式事件 → 实时 UI 更新 | 无（SDK only） |
| **核心代码量** | ~7K 行 | ~3K 行 |
| **设计哲学** | 产品级 CLI 工具 | 轻量级 SDK 框架 |

**核心差异**：OpenAI Agents SDK 是给开发者用的**构建框架**（你用它来搭 agent），OpenCode 是一个**完整产品**（它自己就是 agent）。所以 OpenCode 需要处理持久化、多模型、UI、权限等产品级问题，而 SDK 只需要提供编排原语。
