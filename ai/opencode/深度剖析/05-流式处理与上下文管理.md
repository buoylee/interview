# 05 - 流式处理与上下文管理：SessionProcessor 与 Compaction

> SessionProcessor 是 LLM 流式响应的事件处理中枢。
> 它将 Vercel AI SDK 的 `fullStream` 事件转化为持久化的消息 parts。
> Compaction 则是 OpenCode 的上下文窗口管理方案——在消息溢出前自动压缩。
> 本文剖析这两个核心机制。

---

## SessionProcessor 架构

源码：`session/processor.ts:1-430`

```
LLM.stream()
    │
    └── streamText().fullStream  ← Vercel AI SDK 的流式事件
            │
            ▼
SessionProcessor.process()  ← while(true) 消费流事件
    │
    ├── "start"              → 设置 session 状态为 busy
    ├── "reasoning-start"    → 创建 reasoning part
    ├── "reasoning-delta"    → 增量更新 reasoning 文本
    ├── "reasoning-end"      → 完成 reasoning part
    ├── "text-start"         → 创建 text part
    ├── "text-delta"         → 增量更新文本（实时推送到 UI）
    ├── "text-end"           → 完成 text part
    ├── "tool-input-start"   → 创建 tool part (pending)
    ├── "tool-call"          → 更新为 running，检测 doom loop
    ├── "tool-result"        → 更新为 completed
    ├── "tool-error"         → 更新为 error，检测权限拒绝
    ├── "start-step"         → 创建 git snapshot
    ├── "finish-step"        → 记录 usage/cost，生成 file patch
    ├── "error"              → 抛出异常
    └── "finish"             → 忽略
            │
            ▼
    返回 "continue" | "stop" | "compact"
```

## process() 主循环

```typescript
export function create(input: {
    assistantMessage: MessageV2.Assistant
    sessionID: SessionID
    model: Provider.Model
    abort: AbortSignal
}) {
    const toolcalls: Record<string, MessageV2.ToolPart> = {}
    let snapshot: string | undefined
    let blocked = false
    let attempt = 0
    let needsCompaction = false

    return {
        get message() { return input.assistantMessage },

        async process(streamInput: LLM.StreamInput) {
            needsCompaction = false
            const shouldBreak = (await Config.get()).experimental?.continue_loop_on_deny !== true

            while (true) {
                try {
                    let currentText: MessageV2.TextPart | undefined
                    let reasoningMap: Record<string, MessageV2.ReasoningPart> = {}
                    const stream = await LLM.stream(streamInput)    // ← 发起 LLM 调用

                    for await (const value of stream.fullStream) {  // ← 消费流事件
                        input.abort.throwIfAborted()
                        switch (value.type) {
                            // ... 事件处理（下文详述）
                        }
                        if (needsCompaction) break
                    }
                } catch (e) {
                    // ... 错误处理
                }
                // ... 清理和返回值
            }
        },
    }
}
```

**关键状态**：
- `toolcalls` — 追踪当前正在执行的工具调用（callID → ToolPart）
- `snapshot` — git snapshot hash，用于计算文件 diff
- `blocked` — 工具被权限拒绝时设为 true
- `needsCompaction` — 检测到上下文溢出时设为 true
- `attempt` — 重试计数

## 事件处理详解

### Reasoning 事件

```typescript
case "reasoning-start":
    const reasoningPart = {
        id: PartID.ascending(),
        messageID: input.assistantMessage.id,
        type: "reasoning" as const,
        text: "",
        time: { start: Date.now() },
        metadata: value.providerMetadata,
    }
    reasoningMap[value.id] = reasoningPart
    await Session.updatePart(reasoningPart)     // 写入 DB
    break

case "reasoning-delta":
    const part = reasoningMap[value.id]
    part.text += value.text
    await Session.updatePartDelta({             // 增量推送
        partID: part.id,
        field: "text",
        delta: value.text,
    })
    break

case "reasoning-end":
    part.text = part.text.trimEnd()
    part.time = { ...part.time, end: Date.now() }
    await Session.updatePart(part)              // 最终更新
    delete reasoningMap[value.id]
    break
```

reasoning 事件来自支持 extended thinking 的模型（如 Claude）。每个 reasoning block 有独立的 ID，支持 interleaved thinking（与文本交错）。

### Text 事件

```typescript
case "text-start":
    currentText = {
        id: PartID.ascending(),
        type: "text",
        text: "",
        time: { start: Date.now() },
        metadata: value.providerMetadata,
    }
    await Session.updatePart(currentText)
    break

case "text-delta":
    currentText.text += value.text
    await Session.updatePartDelta({
        partID: currentText.id,
        field: "text",
        delta: value.text,    // ← 仅发送增量，不是全量
    })
    break

case "text-end":
    currentText.text = currentText.text.trimEnd()
    // Plugin hook：允许插件修改最终文本
    const textOutput = await Plugin.trigger(
        "experimental.text.complete",
        { sessionID, messageID, partID },
        { text: currentText.text },
    )
    currentText.text = textOutput.text
    await Session.updatePart(currentText)
    currentText = undefined
    break
```

**增量推送**：`updatePartDelta` 只发送 delta 文本，UI 端拼接。这比每次发送全量文本高效得多。

### Tool 事件

```typescript
case "tool-input-start":
    // 创建 pending 状态的 tool part
    const part = await Session.updatePart({
        type: "tool",
        tool: value.toolName,
        callID: value.id,
        state: { status: "pending", input: {}, raw: "" },
    })
    toolcalls[value.id] = part
    break

case "tool-call":
    // 更新为 running，注入实际参数
    await Session.updatePart({
        ...match,
        state: { status: "running", input: value.input, time: { start: Date.now() } },
    })

    // Doom loop 检测
    const lastThree = parts.slice(-DOOM_LOOP_THRESHOLD)
    if (
        lastThree.length === DOOM_LOOP_THRESHOLD &&
        lastThree.every((p) =>
            p.type === "tool" &&
            p.tool === value.toolName &&
            JSON.stringify(p.state.input) === JSON.stringify(value.input),
        )
    ) {
        await PermissionNext.ask({ permission: "doom_loop", patterns: [value.toolName], ... })
    }
    break

case "tool-result":
    await Session.updatePart({
        ...match,
        state: {
            status: "completed",
            input: value.input ?? match.state.input,
            output: value.output.output,
            metadata: value.output.metadata,
            title: value.output.title,
            time: { start: match.state.time.start, end: Date.now() },
            attachments: value.output.attachments,
        },
    })
    delete toolcalls[value.toolCallId]
    break

case "tool-error":
    await Session.updatePart({
        ...match,
        state: { status: "error", error: value.error.message, ... },
    })
    // 权限拒绝 → 设置 blocked
    if (value.error instanceof PermissionNext.RejectedError || value.error instanceof Question.RejectedError) {
        blocked = shouldBreak
    }
    delete toolcalls[value.toolCallId]
    break
```

工具的生命周期：`pending → running → completed/error`。每个状态变化都持久化到 DB 并推送到 UI。

### Step 事件

```typescript
case "start-step":
    snapshot = await Snapshot.track()      // 记录当前 git 状态
    await Session.updatePart({
        type: "step-start",
        snapshot,
    })
    break

case "finish-step":
    // ① 计算 usage 和 cost
    const usage = Session.getUsage({ model, usage: value.usage, metadata: value.providerMetadata })
    input.assistantMessage.cost += usage.cost
    input.assistantMessage.tokens = usage.tokens

    // ② 记录 step 完成
    await Session.updatePart({
        type: "step-finish",
        reason: value.finishReason,
        snapshot: await Snapshot.track(),
        tokens: usage.tokens,
        cost: usage.cost,
    })
    await Session.updateMessage(input.assistantMessage)

    // ③ 生成文件 diff
    if (snapshot) {
        const patch = await Snapshot.patch(snapshot)
        if (patch.files.length) {
            await Session.updatePart({
                type: "patch",
                hash: patch.hash,
                files: patch.files,
            })
        }
    }

    // ④ 触发摘要生成
    SessionSummary.summarize({ sessionID, messageID: input.assistantMessage.parentID })

    // ⑤ 检测上下文溢出
    if (!input.assistantMessage.summary && await SessionCompaction.isOverflow({ tokens: usage.tokens, model })) {
        needsCompaction = true
    }
    break
```

每个 "step" 对应一次 LLM 调用（一轮可能包含 LLM 响应 + 工具执行）。`finish-step` 是最重要的事件——它汇总了这一步的所有信息。

## 错误处理与重试

```typescript
catch (e) {
    const error = MessageV2.fromError(e, { providerID: input.model.providerID })

    if (MessageV2.ContextOverflowError.isInstance(error)) {
        // 上下文溢出 → 标记需要压缩
        needsCompaction = true
        Bus.publish(Session.Event.Error, { sessionID, error })
    } else {
        const retry = SessionRetry.retryable(error)
        if (retry !== undefined) {
            // 可重试错误（rate limit、server error 等）
            attempt++
            const delay = SessionRetry.delay(attempt, error)
            SessionStatus.set(sessionID, {
                type: "retry",
                attempt,
                message: retry,
                next: Date.now() + delay,
            })
            await SessionRetry.sleep(delay, input.abort)
            continue   // ← 重试：回到 while(true) 顶部
        }
        // 不可重试 → 记录错误，停止
        input.assistantMessage.error = error
        Bus.publish(Session.Event.Error, { sessionID, error })
    }
}
```

错误分三类：
1. **上下文溢出** → 标记 `needsCompaction`，process 返回 `"compact"`
2. **可重试** → 指数退避重试（rate limit、5xx 等）
3. **不可重试** → 记录错误，process 返回 `"stop"`

for (const part of parts) {

## Compaction 系统

源码：`session/compaction.ts:1-331`

### 溢出检测

```typescript
export async function isOverflow(input: { tokens; model: Provider.Model }) {
    if (config.compaction?.auto === false) return false
    const context = input.model.limit.context
    if (context === 0) return false

    const count = input.tokens.total || input.tokens.input + input.tokens.output + ...
    const reserved = config.compaction?.reserved ?? Math.min(COMPACTION_BUFFER, maxOutputTokens)
    const usable = input.model.limit.input
        ? input.model.limit.input - reserved
        : context - maxOutputTokens
    return count >= usable
}
```

**判断公式**：`当前 token 数 ≥ 可用空间`，其中可用空间 = 输入上限 - 预留缓冲（默认 20K tokens）。

### Compaction 流程

```
主循环检测到溢出
    │
    ▼
SessionCompaction.create()
    │  写入 compaction part 到消息流
    ▼
主循环下次迭代发现 compaction task
    │
    ▼
SessionCompaction.process()
    │
    ├── 加载 "compaction" agent（无工具、专用 prompt）
    ├── 构建压缩请求：历史消息 + "请总结这段对话"
    ├── SessionProcessor.create() + process()
    │     └── LLM 生成摘要
    │
    ├── 如果是 overflow 模式：
    │     ├── 找到最近的用户消息
    │     ├── 截断历史到该消息之前
    │     └── 压缩后重放该用户消息
    │
    └── 如果是普通 compaction：
          └── 插入 "Continue if you have next steps" 消息
```

### Prune：增量清理

```typescript
export async function prune(input: { sessionID }) {
    const msgs = await Session.messages({ sessionID })
    let total = 0
    let turns = 0

    for (let msgIndex = msgs.length - 1; msgIndex >= 0; msgIndex--) {
        if (msg.info.role === "user") turns++
        if (turns < 2) continue    // 保护最近 2 轮

        for (const part of msg.parts) {
            if (part.type === "tool" && part.state.status === "completed") {
                if (part.state.time.compacted) break   // 已经 prune 过
                const estimate = Token.estimate(part.state.output)
                total += estimate
                if (total > PRUNE_PROTECT) {   // 超过 40K tokens
                    toPrune.push(part)         // 标记需要清理
                }
            }
        }
    }

    if (pruned > PRUNE_MINIMUM) {   // 至少清理 20K tokens 才值得
        for (const part of toPrune) {
            part.state.time.compacted = Date.now()   // 标记为已压缩
            await Session.updatePart(part)
        }
    }
}
```

**Prune 策略**：
1. 保护最近 2 轮的所有内容
2. 更早的工具输出，保留最近 40K tokens 的
3. 超过 40K tokens 的旧工具输出，标记为 `compacted`
4. `compacted` 的工具输出在 `filterCompacted` 中被跳过，不发给 LLM

Prune 是轻量级的——不需要调用 LLM，只是标记旧输出为"已压缩"。它在每次主循环结束时自动执行。

## Part 类型总览

一条 assistant message 可以包含多种 parts：

| Part 类型 | 含义 | 持久化时机 |
|-----------|------|-----------|
| `text` | LLM 文本输出 | text-start → text-delta → text-end |
| `reasoning` | 推理过程 | reasoning-start → delta → end |
| `tool` | 工具调用及结果 | tool-input-start → tool-call → tool-result/error |
| `step-start` | 步骤开始（含 git snapshot） | start-step 事件 |
| `step-finish` | 步骤结束（含 usage/cost） | finish-step 事件 |
| `patch` | 文件变更 diff | finish-step 后计算 |
| `compaction` | 压缩请求标记 | SessionCompaction.create() |
| `subtask` | 子任务请求标记 | 用户消息中指定 |

## 设计洞察

### 1. 事件驱动 vs 请求-响应

OpenCode 不是等 LLM 完成后才处理——而是逐个事件处理。这带来：
- 实时 UI 更新（每个 text-delta 都推送）
- 细粒度的状态追踪（工具 pending → running → completed）
- 及时的错误处理（不用等整个响应完成）

### 2. 一切皆 Part

消息不是一个简单的字符串，而是一个 `Part[]` 数组。这个设计允许：
- 同一条消息中交错 text、reasoning、tool call
- 每个 part 有独立的 ID、时间戳、状态
- Part 可以被单独更新（增量推送）

### 3. Compaction 的两层策略

| 层级 | 机制 | 触发 | 成本 |
|------|------|------|------|
| Prune | 标记旧工具输出为 compacted | 每轮结束 | 零（无 LLM 调用） |
| Compaction | LLM 生成历史摘要 | 上下文溢出时 | 一次 LLM 调用 |

Prune 是常态化的轻量清理，Compaction 是溢出时的重度处理。两层配合使用，最大化上下文利用率。

### 4. 与 OpenAI Agents SDK 的对比

| | OpenAI Agents SDK | OpenCode |
|---|---|---|
| **流式处理** | asyncio.Queue 事件队列 | Vercel AI SDK fullStream |
| **上下文管理** | 无内置 | Prune + Compaction 双层 |
| **工具状态** | RunItem 内存对象 | Part 持久化到 DB |
| **错误重试** | 无内置 | 指数退避 + retry 状态展示 |
| **Doom loop** | 无 | 连续 3 次相同调用检测 |

## 小结

SessionProcessor 和 Compaction 是 OpenCode 流式处理和上下文管理的核心：

1. **SessionProcessor** 消费 AI SDK 的 fullStream 事件，将每个事件转化为持久化的 Part
2. **三种返回值** `"continue"` / `"stop"` / `"compact"` 驱动主循环的决策
3. **错误处理**：上下文溢出 → compaction，可重试 → 退避重试，其他 → 停止
4. **Prune** 每轮清理旧工具输出（零成本），**Compaction** 溢出时 LLM 生成摘要
5. **增量推送**：text-delta / reasoning-delta 只发送差量，UI 实时拼接
