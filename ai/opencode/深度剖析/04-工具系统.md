# 04 - 工具系统：注册、执行与权限

> 工具是 LLM 与真实世界交互的唯一通道。
> OpenCode 的工具系统包含：统一的 Tool 接口、注册表、权限检查、输出截断、以及 Vercel AI SDK 的桥接。
> 本文以 bash 和 edit 两个典型工具为例，剖析从注册到执行的完整生命周期。

## 工具接口定义

源码：`tool/tool.ts:1-90`

```typescript
export namespace Tool {
    export type Context = {
        sessionID: SessionID
        messageID: MessageID
        agent: string
        abort: AbortSignal
        callID?: string
        extra?: { [key: string]: any }
        messages: MessageV2.WithParts[]
        metadata(input: { title?: string; metadata?: any }): void   // 更新 UI 元数据
        ask(input: Omit<PermissionNext.Request, ...>): Promise<void>  // 请求权限
    }

    export interface Info<Parameters extends z.ZodType = z.ZodType> {
        id: string
        init: (ctx?: InitContext) => Promise<{
            description: string
            parameters: Parameters
            execute(args: z.infer<Parameters>, ctx: Context): Promise<{
                title: string
                metadata: any
                output: string
                attachments?: Omit<MessageV2.FilePart, ...>[]
            }>
        }>
    }
}
```

**核心设计**：
- **延迟初始化**：`init()` 在需要时才调用，可以根据 agent 配置动态调整 description 和 parameters
- **Context 注入**：每次工具执行都有完整上下文（session、message、abort signal...）
- **metadata 回调**：工具可以在执行过程中实时更新 UI 状态（如 bash 的 output streaming）
- **ask 回调**：工具通过 `ctx.ask()` 请求权限，被拒绝时抛出 `PermissionNext.RejectedError`

### Tool.define 包装器

```typescript
export function define(id, init) {
    return {
        id,
        init: async (initCtx) => {
            const toolInfo = init instanceof Function ? await init(initCtx) : init
            const execute = toolInfo.execute

            toolInfo.execute = async (args, ctx) => {
                // ① 参数校验
                try {
                    toolInfo.parameters.parse(args)
                } catch (error) {
                    if (error instanceof z.ZodError && toolInfo.formatValidationError) {
                        throw new Error(toolInfo.formatValidationError(error))
                    }
                    throw new Error(`The ${id} tool was called with invalid arguments: ${error}.`)
                }

                // ② 执行
                const result = await execute(args, ctx)

                // ③ 自动输出截断
                if (result.metadata.truncated !== undefined) return result  // 工具自己处理了截断
                const truncated = await Truncate.output(result.output, {}, initCtx?.agent)
                return {
                    ...result,
                    output: truncated.content,
                    metadata: {
                        ...result.metadata,
                        truncated: truncated.truncated,
                        ...(truncated.truncated && { outputPath: truncated.outputPath }),
                    },
                }
            }
            return toolInfo
        },
    }
}
```

`Tool.define` 自动包装每个工具的 `execute`，添加：
1. **参数校验** — Zod schema 验证，失败时返回友好错误消息
2. **输出截断** — 防止工具输出过大撑爆上下文窗口

### 模型相关的工具过滤

```typescript
export async function tools(model, agent?) {
    const tools = await all()
    const result = await Promise.all(
        tools.filter((t) => {
            // websearch/codesearch 仅对特定 provider 启用
            if (t.id === "codesearch" || t.id === "websearch") {
                return model.providerID === ProviderID.opencode || Flag.OPENCODE_ENABLE_EXA
            }
            // GPT-5+ 用 apply_patch，其他用 edit/write
            const usePatch = model.modelID.includes("gpt-") && !model.modelID.includes("gpt-4")
            if (t.id === "apply_patch") return usePatch
            if (t.id === "edit" || t.id === "write") return !usePatch
            return true
        })
        .map(async (t) => {
            const tool = await t.init({ agent })
            // Plugin 可以修改工具描述和参数
            await Plugin.trigger("tool.definition", { toolID: t.id }, output)
            return { id: t.id, ...tool }
        }),
    )
    return result
}
```

**关键设计**：edit/write 和 apply_patch 是互斥的——GPT-5+ 使用 patch 格式（类似 Codex），其他模型使用传统的 string replace。

## 工具到 AI SDK 的桥接

源码：`session/prompt.ts:746-933`

`resolveTools()` 将 OpenCode 的 `Tool.Info` 转换为 Vercel AI SDK 的 `Tool` 格式：

```typescript
async function resolveTools(input) {
    const tools: Record<string, AITool> = {}

    // 构建 context 工厂
    const context = (args, options: ToolCallOptions): Tool.Context => ({
        sessionID: input.session.id,
        abort: options.abortSignal!,
        messageID: input.processor.message.id,
        callID: options.toolCallId,
        agent: input.agent.name,
        messages: input.messages,
        metadata: async (val) => {
            const match = input.processor.partFromToolCall(options.toolCallId)
            if (match && match.state.status === "running") {
                await Session.updatePart({ ...match, state: { ...val, status: "running", ... } })
            }
        },
        async ask(req) {
            await PermissionNext.ask({
                ...req,
                sessionID: input.session.id,
                tool: { messageID: input.processor.message.id, callID: options.toolCallId },
                ruleset: PermissionNext.merge(input.agent.permission, input.session.permission ?? []),
            })
        },
    })

    // 注册内置工具
    for (const item of await ToolRegistry.tools(model, agent)) {
        tools[item.id] = tool({
            id: item.id,
            description: item.description,
            inputSchema: jsonSchema(schema),
            async execute(args, options) {
                const ctx = context(args, options)
                await Plugin.trigger("tool.execute.before", { tool: item.id, ... }, { args })
                const result = await item.execute(args, ctx)
                await Plugin.trigger("tool.execute.after", { tool: item.id, ... }, result)
                return result
            },
        })
    }

    // 注册 MCP 工具
    for (const [key, item] of Object.entries(await MCP.tools())) {
        // MCP 工具也包装上权限检查和 Plugin hooks
        item.execute = async (args, opts) => {
            await ctx.ask({ permission: key, ... })
            const result = await execute(args, opts)
            // 转换 MCP 格式 → OpenCode 格式
            return { title: "", metadata, output: truncated.content, attachments }
        }
        tools[key] = item
    }

    return tools
}
```

**桥接做了什么**：
1. **Context 构建**：每次工具调用创建一个 `Tool.Context`，注入 session/message/abort/ask
2. **Plugin hooks**：在工具执行前后触发 `tool.execute.before` 和 `tool.execute.after`
3. **MCP 工具**：外部 MCP server 提供的工具也被统一包装，注入权限检查

## Bash 工具：深度剖析

源码：`tool/bash.ts:55-270`

### 命令解析与权限

```typescript
export const BashTool = Tool.define("bash", async () => {
    const shell = Shell.acceptable()
    return {
        parameters: z.object({
            command: z.string(),
            timeout: z.number().optional(),
            workdir: z.string().optional(),
            description: z.string(),
        }),
        async execute(params, ctx) {
            // ① 用 tree-sitter 解析命令的 AST
            const tree = await parser().then((p) => p.parse(params.command))

            const directories = new Set<string>()
            const patterns = new Set<string>()
            const always = new Set<string>()

            for (const node of tree.rootNode.descendantsOfType("command")) {
                const command = []
                for (let i = 0; i < node.childCount; i++) {
                    const child = node.child(i)
                    if (["command_name", "word", "string", ...].includes(child.type)) {
                        command.push(child.text)
                    }
                }

                // ② 检测外部目录访问
                if (["cd", "rm", "cp", "mv", "mkdir", ...].includes(command[0])) {
                    for (const arg of command.slice(1)) {
                        const resolved = await fs.realpath(path.resolve(cwd, arg))
                        if (!Instance.containsPath(resolved)) {
                            directories.add(dir)
                        }
                    }
                }

                // ③ 收集权限模式
                patterns.add(commandText)
                always.add(BashArity.prefix(command).join(" ") + " *")
            }

            // ④ 请求权限
            if (directories.size > 0) {
                await ctx.ask({ permission: "external_directory", patterns: globs, ... })
            }
            if (patterns.size > 0) {
                await ctx.ask({ permission: "bash", patterns: Array.from(patterns), ... })
            }
```

**Bash 的权限检查非常精细**：
1. 用 **tree-sitter** 解析 bash AST（不是正则匹配！）
2. 提取每个命令及其参数
3. 对涉及文件操作的命令（cd/rm/cp/mv...），解析路径判断是否在项目外
4. 用命令前缀生成 `always` 模式（如 `git push *`），用于"总是允许"的记忆

### 执行与 streaming

```typescript
            // ⑤ 执行命令
            const proc = spawn(params.command, {
                shell,
                cwd,
                env: { ...process.env, ...shellEnv.env },
                stdio: ["ignore", "pipe", "pipe"],
                detached: process.platform !== "win32",
            })

            // ⑥ 实时输出到 UI
            const append = (chunk: Buffer) => {
                output += chunk.toString()
                ctx.metadata({
                    metadata: {
                        output: output.length > MAX_METADATA_LENGTH
                            ? output.slice(0, MAX_METADATA_LENGTH) + "\n\n..."
                            : output,
                        description: params.description,
                    },
                })
            }
            proc.stdout?.on("data", append)
            proc.stderr?.on("data", append)

            // ⑦ 超时和中止
            const abortHandler = () => { aborted = true; void kill() }
            ctx.abort.addEventListener("abort", abortHandler, { once: true })
            const timeoutTimer = setTimeout(() => { timedOut = true; void kill() }, timeout + 100)

            await new Promise<void>((resolve) => {
                proc.once("exit", () => { exited = true; cleanup(); resolve() })
                proc.once("error", (error) => { exited = true; cleanup(); reject(error) })
            })
```

**关键**：
- `ctx.metadata()` 在每次 stdout/stderr 输出时被调用，实现了 UI 上的实时 output 展示
- metadata 被截断到 30KB（`MAX_METADATA_LENGTH`），但完整 output 仍然返回给 LLM
- 三种终止方式：超时 / 用户中止（abort signal）/ 正常退出

## Edit 工具：模糊匹配引擎

源码：`tool/edit.ts:36-668`

### 基本流程

```typescript
export const EditTool = Tool.define("edit", {
    parameters: z.object({
        filePath: z.string(),
        oldString: z.string(),
        newString: z.string(),
        replaceAll: z.boolean().optional(),
    }),
    async execute(params, ctx) {
        // ① 路径处理
        const filePath = path.isAbsolute(params.filePath)
            ? params.filePath
            : path.join(Instance.directory, params.filePath)
        await assertExternalDirectory(ctx, filePath)

        await FileTime.withLock(filePath, async () => {
            // ② 空 oldString = 创建新文件
            if (params.oldString === "") {
                await Filesystem.write(filePath, params.newString)
                return
            }

            // ③ 读取并匹配
            await FileTime.assert(ctx.sessionID, filePath)  // 文件时间戳校验
            contentOld = await Filesystem.readText(filePath)
            contentNew = replace(contentOld, old, next, params.replaceAll)

            // ④ 权限检查
            diff = trimDiff(createTwoFilesPatch(...))
            await ctx.ask({
                permission: "edit",
                patterns: [path.relative(Instance.worktree, filePath)],
                metadata: { filepath: filePath, diff },
            })

            // ⑤ 写入
            await Filesystem.write(filePath, contentNew)
        })

        // ⑥ LSP 诊断
        await LSP.touchFile(filePath, true)
        const diagnostics = await LSP.diagnostics()
        const errors = issues.filter((item) => item.severity === 1)
        if (errors.length > 0) {
            output += `\nLSP errors detected in this file, please fix:\n...`
        }
    },
})
```

### 9 级模糊匹配

`replace()` 函数依次尝试 9 种匹配策略：

```typescript
export function replace(content, oldString, newString, replaceAll = false) {
    for (const replacer of [
        SimpleReplacer,                // 1. 精确匹配
        LineTrimmedReplacer,           // 2. 行级 trim 后匹配
        BlockAnchorReplacer,           // 3. 首尾行锚定 + 中间模糊
        WhitespaceNormalizedReplacer,  // 4. 空白符归一化
        IndentationFlexibleReplacer,   // 5. 缩进无关匹配
        EscapeNormalizedReplacer,      // 6. 转义字符归一化
        TrimmedBoundaryReplacer,       // 7. 首尾 trim
        ContextAwareReplacer,          // 8. 上下文感知（首尾锚定 + 50% 相似度）
        MultiOccurrenceReplacer,       // 9. 多次出现（仅用于 replaceAll）
    ]) {
        for (const search of replacer(content, oldString)) {
            const index = content.indexOf(search)
            if (index === -1) continue
            // 非 replaceAll 模式下，必须唯一匹配
            if (!replaceAll) {
                const lastIndex = content.lastIndexOf(search)
                if (index !== lastIndex) continue  // 多个匹配 → 跳过
            }
            return replaceAll
                ? content.replaceAll(search, newString)
                : content.substring(0, index) + newString + content.substring(index + search.length)
        }
    }
    throw new Error("Could not find oldString in the file.")
}
```

**BlockAnchorReplacer 详解**（最有趣的一个）：

```typescript
export const BlockAnchorReplacer: Replacer = function* (content, find) {
    const firstLineSearch = searchLines[0].trim()
    const lastLineSearch = searchLines[searchLines.length - 1].trim()

    // 找所有 "首行匹配 + 尾行匹配" 的候选区间
    for (let i = 0; i < originalLines.length; i++) {
        if (originalLines[i].trim() !== firstLineSearch) continue
        for (let j = i + 2; j < originalLines.length; j++) {
            if (originalLines[j].trim() === lastLineSearch) {
                candidates.push({ startLine: i, endLine: j })
                break
            }
        }
    }

    // 单候选：Levenshtein 相似度 > 0（几乎总是接受）
    // 多候选：取相似度最高的，且 > 0.3
    for (let j = 1; j < searchBlockSize - 1; j++) {
        const distance = levenshtein(originalLine, searchLine)
        similarity += (1 - distance / maxLen) / linesToCheck
    }
}
```

LLM 经常给出的 oldString 不完全精确（缩进差异、空白差异、多余空行等）。这 9 级匹配策略极大提升了编辑成功率。

## 权限系统集成

工具通过 `ctx.ask()` 请求权限：

```typescript
await ctx.ask({
    permission: "bash",           // 权限类型
    patterns: ["git push origin main"],  // 具体操作
    always: ["git push *"],       // "总是允许" 的模式
    metadata: {},                  // 展示给用户的额外信息
})
```

权限判定链：

```
ctx.ask()
  │
  ├── PermissionNext.ask()
  │     │
  │     ├── 检查 agent.permission（agent 级规则）
  │     ├── 检查 session.permission（session 级规则）
  │     ├── 检查 config.permission（用户配置）
  │     │
  │     ├── action = "allow" → 通过
  │     ├── action = "deny" → 抛出 RejectedError
  │     └── action = "ask" → 弹出 UI 确认
  │           ├── 用户允许 → 通过，可选 "always" 记住
  │           └── 用户拒绝 → 抛出 RejectedError
  │
  └── RejectedError 在 processor.ts 中被捕获
        → blocked = true → process() 返回 "stop"
```

## 输出截断

```typescript
export async function output(content: string, options: {}, agent?: Agent.Info) {
    if (content.length <= MAX_BYTES && content.split("\n").length <= MAX_LINES) {
        return { content, truncated: false }
    }
    // 保留前 N 行 + 后 N 行，中间截断

    const outputPath = path.join(Truncate.GLOB, `truncated-${hash}.txt`)

        truncated: true,
        outputPath,
    }
}
```

## 设计洞察

### 1. 工具执行不在主循环内

与 OpenAI Agents SDK 不同（它有显式的 `execute_tools_and_side_effects`），OpenCode 的工具执行由 **Vercel AI SDK 自动驱动**。`streamText()` 内部处理 tool call → execute → 将结果发回 LLM。主循环只关心最终的 finish reason。

### 2. 权限即配置

权限不是硬编码的，而是通过规则集配置。同一个工具在不同 agent 下可以有不同的权限：
- `build` agent：bash = ask（默认需要确认）
- `explore` agent：bash = allow（总是允许，因为 explore 被限制为只读）
- `plan` agent：bash = deny（完全禁用）

### 3. Edit 的 9 级匹配是生产级的解决方案

这不是学术论文里的模糊匹配——而是在实际使用中积累的 9 种 LLM 常犯错误的修复策略。来源注释表明它借鉴了 Cline 和 Gemini CLI 的实现。

## 小结

工具系统的数据流：

1. **注册**：`ToolRegistry.tools()` 收集内置 + 自定义 + MCP 工具，按 model 过滤
2. **桥接**：`resolveTools()` 将 OpenCode Tool 转为 AI SDK Tool，注入 context/ask/metadata
3. **执行**：AI SDK 自动处理 tool call → execute → result feedback
4. **权限**：每次执行前通过 `ctx.ask()` 检查，被拒绝则中断
5. **截断**：自动截断过长输出，保存完整版到临时文件
6. **匹配**：Edit 工具的 9 级模糊匹配提升 LLM 编辑成功率
