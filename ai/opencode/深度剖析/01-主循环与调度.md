# 01 - 主循环与调度：从用户输入到 LLM 响应

> OpenCode 的核心循环藏在 `session/prompt.ts` 的 `loop()` 函数中。
> 它不是一个简单的"发送-接收"，而是一个带状态感知、自动压缩、子任务调度的 while(true) 循环。
> 本文按数据流顺序，逐行剖析这条主链路。

---

## 架构总览

```
用户输入
  │
  ▼
SessionPrompt.prompt()          ← 入口：创建 user message，启动循环
  │
  ▼
SessionPrompt.loop()            ← 主循环 while(true)
  │
  ├── 1. 加载历史消息（MessageV2.stream → filterCompacted）
  │
  ├── 2. 定位 lastUser / lastAssistant / lastFinished
  │
  ├── 3. 判断退出条件（finish reason 不是 tool-calls）
  │
  ├── 4. 处理待执行的 subtask / compaction
  │
  ├── 5. 检测上下文溢出 → 触发 compaction
  │
  ├── 6. 正常处理：
  │   ├── 解析 Agent
  │   ├── 解析 Tools（resolveTools）
  │   ├── 组装 system prompt
  │   ├── 构建 assistant message
  │   ├── 创建 SessionProcessor
  │   └── processor.process(streamInput) → LLM.stream()
  │
  ├── 7. 根据 process 返回值决定下一步
  │   ├── "continue" → 继续循环（工具调用后再次请求 LLM）
  │   ├── "stop" → 退出（出错或被阻断）
  │   └── "compact" → 触发压缩后继续
  │
  └── 8. 循环结束 → 返回最终 assistant message
```

## 入口：SessionPrompt.prompt()

源码：`session/prompt.ts:162-189`

```typescript
export const prompt = fn(PromptInput, async (input) => {
    const session = await Session.get(input.sessionID)
    await SessionRevert.cleanup(session)

    const message = await createUserMessage(input)   // ① 创建用户消息
    await Session.touch(input.sessionID)              // ② 更新 session 时间戳

    // 权限向后兼容
    const permissions: PermissionNext.Ruleset = []
    for (const [tool, enabled] of Object.entries(input.tools ?? {})) {
        permissions.push({
            permission: tool,
            action: enabled ? "allow" : "deny",
            pattern: "*",
        })
    }
    if (permissions.length > 0) {
        session.permission = permissions
        await Session.setPermission({ sessionID: session.id, permission: permissions })
    }

    if (input.noReply === true) {          // ③ noReply 模式：只存消息，不调 LLM
        return message
    }

    return loop({ sessionID: input.sessionID })  // ④ 启动主循环
})
```

**关键点**：
1. `createUserMessage` 做了大量工作：解析 `@file` 引用、读取文件内容、处理 MCP 资源、注入 agent 信息
2. `noReply` 模式用于只写入消息但不触发 LLM，比如用户手动执行 shell 命令时
3. 真正的核心在 `loop()`

## 主循环：SessionPrompt.loop()

源码：`session/prompt.ts:278-736`

### 并发控制

```typescript
export const loop = fn(LoopInput, async (input) => {
    const { sessionID, resume_existing } = input

    const abort = resume_existing ? resume(sessionID) : start(sessionID)
    if (!abort) {
        // 已经有循环在运行 → 排队等待
        return new Promise<MessageV2.WithParts>((resolve, reject) => {
            const callbacks = state()[sessionID].callbacks
            callbacks.push({ resolve, reject })
        })
    }

    using _ = defer(() => cancel(sessionID))
    // ...
})
```

**设计**：
- 每个 session 同时只允许一个 loop 运行（通过 `state()` 全局状态追踪）
- 如果已有循环在运行，新请求被放入 `callbacks` 队列，等循环结束时统一 resolve
- `using _ = defer(() => cancel(sessionID))` 确保循环结束时清理状态（TC39 Explicit Resource Management）

### 消息扫描

```typescript
while (true) {
    SessionStatus.set(sessionID, { type: "busy" })
    if (abort.aborted) break

    // ① 从数据库加载所有消息（已压缩的自动跳过）
    let msgs = await MessageV2.filterCompacted(MessageV2.stream(sessionID))

    // ② 从后往前扫描，定位关键消息
    let lastUser: MessageV2.User | undefined
    let lastAssistant: MessageV2.Assistant | undefined
    let lastFinished: MessageV2.Assistant | undefined
    let tasks: (MessageV2.CompactionPart | MessageV2.SubtaskPart)[] = []

    for (let i = msgs.length - 1; i >= 0; i--) {
        const msg = msgs[i]
        if (!lastUser && msg.info.role === "user") lastUser = msg.info as MessageV2.User
        if (!lastAssistant && msg.info.role === "assistant")
            lastAssistant = msg.info as MessageV2.Assistant
        if (!lastFinished && msg.info.role === "assistant" && msg.info.finish)
            lastFinished = msg.info as MessageV2.Assistant
        if (lastUser && lastFinished) break
        // 收集未完成的 task
        const task = msg.parts.filter((part) => part.type === "compaction" || part.type === "subtask")
        if (task && !lastFinished) {
            tasks.push(...task)
        }
    }
```

**为什么从后往前扫描？** 因为循环需要知道的是"最近的状态"：
- `lastUser`：最近的用户消息（包含 model、agent 等配置）
- `lastAssistant`：最近的助手消息（可能还在运行中）
- `lastFinished`：最近完成的助手消息（用于判断是否该退出）
- `tasks`：待处理的任务（压缩或子任务）

### 退出判断

```typescript
if (!lastUser) throw new Error("No user message found in stream.")

if (
    lastAssistant?.finish &&
    !["tool-calls", "unknown"].includes(lastAssistant.finish) &&
    lastUser.id < lastAssistant.id
) {
    log.info("exiting loop", { sessionID })
    break
}
```

**核心逻辑**：如果最近的 assistant 消息已经完成（finish reason 不是 `tool-calls` 或 `unknown`），且它在最近的 user 消息之后，说明 LLM 已经给出最终回复 → 退出循环。

`tool-calls` 表示 LLM 调用了工具，需要继续循环执行工具然后再次调用 LLM。

### 三种特殊处理路径

#### 路径 A：子任务（Subtask）

```typescript
if (task?.type === "subtask") {
    const taskTool = await TaskTool.init()
    const taskModel = task.model
        ? await Provider.getModel(task.model.providerID, task.model.modelID)
        : model

    // 创建 assistant message
    const assistantMessage = await Session.updateMessage({
        id: MessageID.ascending(),
        parentID: lastUser.id,
        role: "assistant",
        mode: task.agent,
        agent: task.agent,
        // ...
    })

    // 执行子任务工具
    const result = await taskTool.execute(taskArgs, taskCtx)

    // 保存结果
    continue  // ← 回到循环顶部
}
```

子任务是 OpenCode 的核心特性之一——agent 可以派生子 agent 去执行独立任务。子任务的结果会写入消息历史，然后主循环继续。

#### 路径 B：上下文压缩（Compaction）

```typescript
if (task?.type === "compaction") {
    const result = await SessionCompaction.process({
        messages: msgs,
        parentID: lastUser.id,
        abort,
        sessionID,
        auto: task.auto,
        overflow: task.overflow,
    })
    if (result === "stop") break
    continue
}
```

当消息历史超过模型上下文窗口时，系统会自动触发 compaction：用一个专门的 compaction agent 把历史总结成摘要。

#### 路径 C：溢出检测

```typescript
if (
    lastFinished &&
    lastFinished.summary !== true &&
    (await SessionCompaction.isOverflow({ tokens: lastFinished.tokens, model }))
) {
    await SessionCompaction.create({
        sessionID,
        agent: lastUser.agent,
        model: lastUser.model,
        auto: true,
    })
    continue
}
```

即使没有显式的 compaction 任务，如果检测到 token 数已经接近上限，也会主动创建一个 compaction 任务插入队列。

### 正常处理路径

```typescript
// ① 解析 agent 配置
const agent = await Agent.get(lastUser.agent)
const maxSteps = agent.steps ?? Infinity
const isLastStep = step >= maxSteps

// ② 注入 reminders（plan/build 模式切换提示等）
msgs = await insertReminders({ messages: msgs, agent, session })

// ③ 创建 SessionProcessor
const processor = SessionProcessor.create({
    assistantMessage: await Session.updateMessage({
        id: MessageID.ascending(),
        parentID: lastUser.id,
        role: "assistant",
        agent: agent.name,
        // ...
    }),
    sessionID,
    model,
    abort,
})

// ④ 解析可用工具
const tools = await resolveTools({
    agent, session, model,
    tools: lastUser.tools,
    processor,
    bypassAgentCheck,
    messages: msgs,
})

// ⑤ 组装 system prompt
const system = [
    ...(await SystemPrompt.environment(model)),
    ...(skills ? [skills] : []),
    ...(await InstructionPrompt.system()),
]

// ⑥ 调用 processor.process() → 内部调用 LLM.stream()
const result = await processor.process({
    user: lastUser,
    agent,
    permission: session.permission,
    abort,
    sessionID,
    system,
    messages: [
        ...MessageV2.toModelMessages(msgs, model),
        ...(isLastStep ? [{ role: "assistant", content: MAX_STEPS }] : []),
    ],
    tools,
    model,
    toolChoice: format.type === "json_schema" ? "required" : undefined,
})
```

**第 ⑥ 步是核心**：`processor.process()` 启动 LLM 流式调用，处理所有事件（文本、工具调用、reasoning），并将结果写入数据库。详见第 02 章。

### 循环控制

```typescript
if (result === "stop") break
if (result === "compact") {
    await SessionCompaction.create({
        sessionID,
        agent: lastUser.agent,
        model: lastUser.model,
        auto: true,
        overflow: !processor.message.finish,
    })
}
continue
```

`processor.process()` 返回三种结果：
- `"continue"` — 工具执行完毕，需要再次调用 LLM（继续循环）
- `"stop"` — 出错、被用户中断、或权限拒绝
- `"compact"` — 上下文溢出，需要先压缩再继续

### 循环结束

```typescript
}  // while(true) 结束

SessionCompaction.prune({ sessionID })  // 清理旧的工具输出

// 从消息流中找到最后一条 assistant 消息返回
for await (const item of MessageV2.stream(sessionID)) {
    if (item.info.role === "user") continue
    const queued = state()[sessionID]?.callbacks ?? []
    for (const q of queued) {
        q.resolve(item)  // 通知所有排队的请求
    }
    return item
}
```

注意 `prune()` 的调用——它在循环结束后回收旧的工具输出以控制存储大小。

## 关键设计洞察

### 1. 消息即状态

OpenCode 没有独立的 "RunState" 对象。整个循环的状态完全由数据库中的消息序列决定：

- **要不要继续循环？** → 看 `lastAssistant.finish`
- **要不要执行子任务？** → 看消息 parts 中有没有 `subtask` 类型
- **要不要压缩？** → 看消息 parts 中有没有 `compaction` 类型
- **工具执行到哪了？** → 看 tool part 的 `state.status`

这种设计让恢复（resume）变得非常简单——只要消息还在，循环可以从任何点继续。

### 2. Doom Loop 检测

```typescript
// processor.ts:153-177
const lastThree = parts.slice(-DOOM_LOOP_THRESHOLD)
if (
    lastThree.length === DOOM_LOOP_THRESHOLD &&
    lastThree.every(
        (p) =>
            p.type === "tool" &&
            p.tool === value.toolName &&
            p.state.status !== "pending" &&
            JSON.stringify(p.state.input) === JSON.stringify(value.input),
    )
) {
    await PermissionNext.ask({
        permission: "doom_loop",
        patterns: [value.toolName],
        // ...
    })
}
```

如果连续 3 次调用同一个工具、参数完全相同，说明 LLM 陷入了死循环。此时触发 `doom_loop` 权限检查，让用户决定是否继续。

### 3. 并发控制模型

```
Session A: prompt() → loop() 正在运行
Session A: prompt() → 发现已有循环 → 放入 callbacks 队列 → 等待
Session A: loop() 完成 → 遍历 callbacks → 全部 resolve
```

同一个 session 的并发请求不会启动多个循环，而是共享同一个循环的结果。这避免了消息混乱。

### 4. 与 OpenAI Agents SDK 的对比

| | OpenAI Agents SDK | OpenCode |
|---|---|---|
| **循环位置** | `run_loop.py` — 独立的运行循环 | `prompt.ts` — 与 session 管理合一 |
| **状态管理** | `RunState` 对象（内存中） | 数据库消息（持久化） |
| **工具执行** | 循环内同步执行 | 流事件驱动，Vercel AI SDK 自动执行 |
| **多 agent** | Handoff（状态机切换） | Subtask（子 session 并行） |
| **上下文管理** | 无内置压缩 | 自动 compaction + prune |
| **退出判断** | `NextStep` 枚举 | `finish` 字段 + 消息扫描 |

## 小结

OpenCode 的主循环是一个**消息驱动的状态机**：

1. **入口** `prompt()` 创建用户消息，启动循环
2. **循环** `loop()` 每次迭代扫描消息历史，决定下一步动作
3. **三条路径**：正常 LLM 调用、子任务执行、上下文压缩
4. **退出条件**：LLM finish reason 不是 `tool-calls`
5. **核心调度**在 `processor.process()` 中完成（下一章详述）

所有状态都通过消息持久化，没有内存中的运行状态对象——这是 OpenCode 与 OpenAI Agents SDK 最大的架构差异。
