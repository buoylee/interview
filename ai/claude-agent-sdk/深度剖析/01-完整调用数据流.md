# 01 - 完整调用数据流：逐行走读

> 本文用一个具体例子，跟踪 `query()` 从进入到返回的**完整数据流**。
> 每个关键步骤标注源码位置，并打印当前核心变量状态。

---

## 我们的例子

```python
from claude_agent_sdk import query, ClaudeAgentOptions

options = ClaudeAgentOptions(
    system_prompt="你是一个计算助手",
    permission_mode="bypassPermissions",
)

async for message in query(prompt="计算 2+2", options=options):
    print(message)
```

这个例子会经历：
1. 启动 Claude Code CLI 子进程
2. 通过 stdin 发送初始化请求和用户消息
3. CLI 处理请求，通过 stdout 返回消息流
4. SDK 解析消息，yield 给用户

---

## 阶段一：入口 — query() 到 InternalClient

### 1.1 query() — 薄包装层

源码：`query.py:11-124`

```python
async def query(
    *,
    prompt: str | AsyncIterable[dict[str, Any]],
    options: ClaudeAgentOptions | None = None,
    transport: Transport | None = None,
) -> AsyncIterator[Message]:
    if options is None:
        options = ClaudeAgentOptions()

    client = InternalClient()

    async for message in client.process_query(
        prompt=prompt, options=options, transport=transport
    ):
        yield message
```

`query()` 做的唯一一件事：创建 `InternalClient` 并转发。

> **为什么有 InternalClient？** 它封装了 Transport 创建、Query 管理、生命周期控制，
> 使 `query()` 保持简洁。`ClaudeSDKClient` 也内部创建类似的 Query，但有更多状态管理。

### 1.2 InternalClient.process_query() — 参数校验与配置

源码：`_internal/client.py:44-146`

```python
async def process_query(self, prompt, options, transport=None):
    # 1. 权限校验
    configured_options = options
    if options.can_use_tool:
        if isinstance(prompt, str):
            raise ValueError("can_use_tool requires streaming mode")
        if options.permission_prompt_tool_name:
            raise ValueError("can_use_tool and permission_prompt_tool_name are mutually exclusive")
        configured_options = replace(options, permission_prompt_tool_name="stdio")
```

**此时变量状态**：
```
prompt = "计算 2+2"（str 类型）
options.system_prompt = "你是一个计算助手"
options.permission_mode = "bypassPermissions"
options.can_use_tool = None → 跳过权限校验
```

> **互斥关系**：`can_use_tool`（Python 回调）和 `permission_prompt_tool_name`（CLI 内置）
> 不能同时使用。如果设了 `can_use_tool`，SDK 自动设 `permission_prompt_tool_name="stdio"`，
> 让 CLI 通过控制协议回调 SDK。

### 2.2 Transport.connect() — 查找 CLI 并启动进程

源码：`_internal/transport/subprocess_cli.py:335-412`

```python
async def connect(self):
    if self._process:
        return

    # 1. 版本检查
    if not os.environ.get("CLAUDE_AGENT_SDK_SKIP_VERSION_CHECK"):
        await self._check_claude_version()

    # 2. 构建命令
    cmd = self._build_command()

    # 3. 启动子进程
    process_env = {
        **os.environ,
        "CLAUDE_CODE_ENTRYPOINT": "sdk-py",
        **self._options.env,
        "CLAUDE_AGENT_SDK_VERSION": __version__,
    }

    self._process = await anyio.open_process(
        cmd,
        stdin=PIPE,
        stdout=PIPE,
        stderr=stderr_dest,  # 如果有 debug/stderr 回调
        cwd=self._cwd,
        env=process_env,
    )

    # 4. 包装流
    self._stdout_stream = TextReceiveStream(self._process.stdout)
    self._stdin_stream = TextSendStream(self._process.stdin)
    self._ready = True
```

### 2.3 _build_command() — 从 Options 到 CLI 参数

源码：`_internal/transport/subprocess_cli.py:166-333`

这个函数把 `ClaudeAgentOptions` 的 40+ 字段翻译成 CLI 命令行参数：

```python
def _build_command(self):
    cmd = [self._cli_path, "--output-format", "stream-json", "--verbose"]

    # system_prompt → --system-prompt
    if self._options.system_prompt is None:
        cmd.extend(["--system-prompt", ""])
    elif isinstance(self._options.system_prompt, str):
        cmd.extend(["--system-prompt", self._options.system_prompt])

    # permission_mode → --permission-mode
    if self._options.permission_mode:
        cmd.extend(["--permission-mode", self._options.permission_mode])

    # 始终使用 streaming 模式
    cmd.extend(["--input-format", "stream-json"])

    return cmd
```

**我们例子生成的命令**：
```bash
claude --output-format stream-json --verbose \
  --system-prompt "你是一个计算助手" \
  --permission-mode bypassPermissions \
  --setting-sources "" \
  --input-format stream-json
```

> **关键设计**：SDK **始终**使用 `--input-format stream-json`，即使是单次 query。
> 这让 agents 和 hooks 等大配置可以通过 initialize 请求发送（而不是命令行参数）。
> 这与 TypeScript SDK 的行为保持一致。

### 2.4 CLI 查找顺序

源码：`_internal/transport/subprocess_cli.py:64-95`

```python
def _find_cli(self):

```

> **捆绑 CLI 优先**：如果 SDK 包内有 `_bundled/claude`，直接使用。
> 这允许 SDK 发布时自带 CLI，无需用户额外安装。

`_read_messages` 作为后台任务持续运行，从 transport 读取消息并路由：

```python

            break

        msg_type = message.get("type")

        if msg_type == "control_response":
            # → 匹配 pending 请求，唤醒等待者
            ...

            # → 分发到 _handle_control_request
            ...
        elif msg_type == "control_cancel_request":
            # → TODO: 取消支持
            ...
        else:
            # → 普通消息推入 stream
            if msg_type == "result":
                self._first_result_event.set()
            await self._message_send.send(message)
```

> **三路分流**：这是整个 SDK 的消息路由核心。
> control_response 走同步等待路径，control_request 走异步处理路径，
> 其余消息走 memory_object_stream 给用户消费。

### 4.2 等待结果并关闭 stdin

源码：`_internal/query.py:614-630`

```python
async def wait_for_result_and_end_input(self):
    if self.sdk_mcp_servers or self.hooks:
        # 有 MCP servers 或 hooks 时，需要保持 stdin 开放直到第一个 result
        # 因为 CLI 可能需要通过 stdin 发送控制请求
        with anyio.move_on_after(self._stream_close_timeout):
            await self._first_result_event.wait()

    await self.transport.end_input()
```

> **为什么要等 result？** 如果有 SDK MCP servers 或 hooks，CLI 可能在执行过程中
> 通过控制协议回调 SDK（比如工具权限请求）。如果过早关闭 stdin，CLI 就无法发送
> 控制请求了。所以要等到第一个 result（表示 CLI 执行完成）才关闭。
>
> 在我们的例子中，没有 MCP servers 也没有 hooks，所以直接关闭 stdin。

## 阶段五：消息流 — CLI 返回结果

### 5.1 Transport 读取 stdout

源码：`_internal/transport/subprocess_cli.py:527-578`

```python
async def _read_messages_impl(self):
    json_buffer = ""

    async for line in self._stdout_stream:
        line_str = line.strip()
        if not line_str:
            continue

        json_lines = line_str.split("\n")

        for json_line in json_lines:
            json_buffer += json_line.strip()

            if len(json_buffer) > self._max_buffer_size:
                json_buffer = ""
                raise SDKJSONDecodeError(...)

            try:
                data = json.loads(json_buffer)
                json_buffer = ""
                yield data
            except json.JSONDecodeError:
                continue  # 继续累积，直到完整 JSON
```

> **JSON 缓冲机制**：`TextReceiveStream` 可能截断长行，所以需要缓冲。
> 每次收到数据就尝试 `json.loads()`，成功了就 yield，失败了就继续累积。
> 如果超过 1MB 限制，抛出错误。

## 阶段六：清理

源码：`_internal/client.py:144-145`

```python
finally:

```

`query.close()` 的内部逻辑：

```python
async def close(self):
    self._closed = True                   # 停止消息路由
    if self._tg:
        self._tg.cancel_scope.cancel()    # 取消后台任务
        await self._tg.__aexit__(...)     # 等待退出
    await self.transport.close()          # 关闭子进程
```

`transport.close()` 的内部逻辑：

```python
async def close(self):
    # 关闭 stdin
    async with self._write_lock:
        self._ready = False
        if self._stdin_stream:
            await self._stdin_stream.aclose()

    # 等待子进程优雅退出（5 秒超时）
    if self._process.returncode is None:
        try:
            with anyio.fail_after(5):
                await self._process.wait()
        except TimeoutError:
            self._process.terminate()  # 强制终止
```

> **5 秒优雅退出**：CLI 收到 stdin EOF 后需要时间刷新 session 文件。
> 如果直接 SIGTERM，可能丢失最后的 assistant 消息。

## 完整数据流总结图

```
用户调用: query(prompt="计算 2+2", options=...)
│
├── 阶段 1: 入口
│     query() → InternalClient.process_query()
│     参数校验 → 创建 Transport → 创建 Query
│
├── 阶段 2: 启动 CLI
│     Transport.connect()
│     ├── _find_cli() → /usr/local/bin/claude
│     ├── _build_command() → ["claude", "--output-format", "stream-json", ...]
│     └── anyio.open_process() → 子进程 PID=12345
│
├── 阶段 3: 初始化
│     Query.start() → 后台 _read_messages 启动
│     Query.initialize() → stdin: {control_request: "initialize"}
│                        ← stdout: {control_response: "success"}
│
├── 阶段 4: 发送消息
│     transport.write({type: "user", message: "计算 2+2"})
│     wait_for_result_and_end_input() → stdin EOF
│
├── 阶段 5: 接收结果
│     ← stdout: {type: "assistant", content: "2 + 2 = 4"}
│     ← stdout: {type: "result", is_error: false, ...}
│     │
│     ├── _read_messages → message_stream
│     ├── parse_message → AssistantMessage / ResultMessage
│     └── yield → 用户代码
│
└── 阶段 6: 清理
      Query.close() → cancel tasks
      Transport.close() → wait(5s) → terminate process
```

## 更复杂的场景：带工具调用

如果 CLI 需要使用工具（比如执行 Bash 命令），数据流变化在于 CLI 内部：

```
CLI 内部：
  收到用户消息 → 调用 Claude API → 返回 tool_use(Bash)
  → 执行 Bash → 结果喂回 API → 返回最终文本

SDK 看到的消息流：
  ← {type: "assistant", content: [tool_use(Bash, "echo 4")]}
  ← {type: "assistant", content: [tool_result(...)]}
  ← {type: "assistant", content: [text("结果是 4")]}
  ← {type: "result", ...}
```

SDK **不参与工具执行**——它只是观察者。工具执行完全在 CLI 内部完成。

唯一例外是 **SDK MCP Server**：如果用户注册了进程内工具，CLI 会通过控制协议回调 SDK 执行。这在第 06 章详细分析。

## 与 OpenAI Agents SDK 的对比

| 环节 | OpenAI Agents SDK | Claude Agent SDK |
|------|-------------------|------------------|
| 启动 | 直接调用，无子进程 | 启动 CLI 子进程 + 初始化握手 |
| 发送 prompt | 构建 API 请求体 | 写入 stdin JSON |
| LLM 调用 | SDK 内 `model.get_response()` | CLI 内部调用 Claude API |
| 工具执行 | SDK 内执行 Python 函数 | CLI 内部执行（Read/Write/Bash 等） |
| 消息解析 | `process_model_response()` 400+ 行分类器 | `parse_message()` 250 行 match-case |
| 状态管理 | `RunState` + `generated_items` | CLI 维护，SDK 只读 session |
| 循环控制 | SDK 内 `NextStepRunAgain/FinalOutput` | CLI 内部循环，SDK 只观察 |
| 清理 | 关闭 span | 终止子进程 |
