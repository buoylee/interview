## 第一章：整体架构——CLI 子进程 + 控制协议

### 1.1 与 OpenAI Agents SDK 的根本区别

Claude Agent SDK 和 OpenAI Agents SDK 的架构差异是**根本性的**：

| | OpenAI Agents SDK | Claude Agent SDK |
|---|---|---|
| **执行位置** | SDK 自身是执行引擎，直接调用 API | SDK 是**驱动层**，实际执行在 Claude Code CLI 子进程 |
| **核心循环** | SDK 内实现 while-loop + tool execution | CLI 内部运行主循环，SDK 只做消息收发 |
| **通信方式** | 直接 HTTP 调 OpenAI API | stdin/stdout JSONL 与 CLI 子进程通信 |
| **工具执行** | SDK 内执行 Python 函数 | CLI 内执行工具（Read/Write/Bash 等），SDK 只接收结果 |
| **状态管理** | SDK 内维护 RunState | CLI 内维护会话状态，SDK 只读取 |

> **一句话概括**：OpenAI SDK 是"引擎"，Claude SDK 是"遥控器"。

### 1.2 分层架构

```
用户代码
  │
  ├── query()              ← 单次查询（fire-and-forget）
  │     └── InternalClient
  │
  └── ClaudeSDKClient      ← 双向交互（多轮对话）
        │
        ▼
  ┌─ Query ──────────────────────────────────────────────────────┐
  │  控制协议层                                                    │
  │  · control_request / control_response 路由                    │
  │  · Hook 回调分发                                              │
  │  · 工具权限回调 (can_use_tool)                                 │
  │  · SDK MCP Server 桥接                                        │
  │  · 消息流 → memory_object_stream                              │
  └──────────────────────┬───────────────────────────────────────┘
                         │
  ┌─ Transport ──────────▼───────────────────────────────────────┐
  │  SubprocessCLITransport                                       │
  │  · 启动 Claude Code CLI 子进程                                 │
  │  · stdin → TextSendStream（写入 JSON 消息）                    │
  │  · stdout → TextReceiveStream（读取 JSON 消息）                │
  │  · stderr → 可选调试输出                                       │
  │  · JSON 缓冲区（处理截断行）                                   │
  └──────────────────────┬───────────────────────────────────────┘
                         │
              Claude Code CLI 子进程
              （真正的 Agent 执行引擎）
```

### 1.3 两种使用模式

**模式一：`query()` — 单次查询**

```python
async for message in query(prompt="What is 2+2?"):
    print(message)
```

数据流：`query()` → `InternalClient.process_query()` → 创建 Transport → 创建 Query → initialize → 写入 prompt → 读取消息流 → 关闭

**模式二：`ClaudeSDKClient` — 双向交互**

```python
async with ClaudeSDKClient(options) as client:
    await client.query("Hello")
    async for msg in client.receive_response():
        print(msg)
    await client.query("Follow up")
    async for msg in client.receive_response():
        print(msg)
```

数据流：`connect()` → 创建 Transport → 创建 Query → `start()` + `initialize()` → 多次 `query()` + `receive_response()` → `disconnect()`

### 1.4 核心类关系

```
query()（顶层函数）
  └── InternalClient.process_query()
        ├── SubprocessCLITransport    ← Transport 接口实现
        └── Query                     ← 控制协议处理
              ├── _read_messages()    ← 后台任务：读取并路由消息
              ├── _handle_control_request()  ← 处理 CLI 发来的请求
              └── _send_control_request()    ← 发送请求给 CLI

ClaudeSDKClient（有状态客户端）
  └── connect() / disconnect() / query() / receive_messages()
        ├── SubprocessCLITransport
        └── Query（同上）
```

## 第二章：一次完整调用的数据流

详见 [深度剖析/01-完整调用数据流.md](深度剖析/01-完整调用数据流.md)

一个 `query()` 调用经过的完整路径：

```
query(prompt="计算 2+2")
│
├── 1. InternalClient.process_query()
│     ├── 参数校验（can_use_tool 互斥检查等）
│     ├── 创建 SubprocessCLITransport
│     └── 创建 Query
│
├── 2. Transport.connect()
│     ├── 查找 CLI（bundled → which → 常见路径）
│     ├── 构建命令行参数
│     └── anyio.open_process() 启动子进程
│
├── 3. Query.start()
│     └── 后台任务：_read_messages() 开始读取 stdout
│
├── 4. Query.initialize()
│     ├── 构建 initialize control_request
│     ├── 注册 hook callbacks
│     └── 等待 CLI 返回 control_response
│
├── 5. 写入用户消息
│     └── transport.write(JSON.dumps({type: "user", message: {...}}))
│
├── 6. 等待结果并关闭 stdin
│     └── wait_for_result_and_end_input()
│
├── 7. 消息流（_read_messages 持续路由）
│     ├── control_response → 匹配 pending 请求
│     ├── control_request → 分发到 handler
│     └── 其他消息 → memory_object_stream → 用户迭代
│
├── 8. 用户迭代消息
│     ├── assistant → AssistantMessage
│     ├── system → SystemMessage
│     ├── result → ResultMessage（标志完成）
│     └── ...
│
└── 9. 清理
      └── Query.close() → Transport.close() → 终止子进程
```

## 第四章：Query 控制协议——双向消息路由

详见 [深度剖析/03-Query控制协议.md](深度剖析/03-Query控制协议.md)

Query 是整个 SDK 的"交换机"：

## 第六章：Hook 系统——拦截与扩展

详见 [深度剖析/05-Hook系统.md](深度剖析/05-Hook系统.md)

Hook 系统允许 SDK 用户在 CLI 执行过程中注入逻辑：
- 10 种 Hook 事件：PreToolUse / PostToolUse / PostToolUseFailure / Stop / SubagentStart 等
- 回调注册：`HookMatcher` + `HookCallback` → callback_id → CLI 按 id 回调
- 异步支持：`async_: True` 延迟执行
- 字段转换：Python `async_` / `continue_` → CLI `async` / `continue`

## 第七章：SDK MCP Server——进程内工具

详见 [深度剖析/06-SDK-MCP-Server.md](深度剖析/06-SDK-MCP-Server.md)

SDK MCP Server 让工具在 Python 进程内运行（而非外部进程）：
- `@tool` 装饰器 → `SdkMcpTool` → MCP Server 实例
- `create_sdk_mcp_server()` → 注册 list_tools / call_tool handler
- Query 作为 JSONRPC 桥接：CLI 发 mcp_message → Query 路由到 Server → 返回结果
- 局限：Python MCP SDK 缺少 Transport 抽象，必须手动路由方法

## 第八章：Session 管理——JSONL 读写与对话链重建

详见 [深度剖析/07-Session管理.md](深度剖析/07-Session管理.md)

Session 系统从磁盘 JSONL 文件读取会话历史：
- 路径：`~/.claude/projects/<sanitized-cwd>/<uuid>.jsonl`
- 轻量读取：只读 head(64KB) + tail(64KB)，不解析整个文件
- 对话链重建：`parentUuid` 链 → 找叶子节点 → 回溯到根
- 变更操作：rename/tag → 原子追加 JSONL 行（O_WRONLY | O_APPEND）

### 架构关键点

1. **SDK 不是引擎** — 真正的 Agent 循环在 Claude Code CLI 内部运行
2. **控制协议** — SDK 通过 stdin/stdout JSONL 与 CLI 双向通信
3. **三种工具路径** — CLI 内置工具（直接执行）、外部 MCP Server（子进程）、SDK MCP Server（进程内）
4. **Hook 桥接** — Hook 回调在 Python 进程执行，但通过控制协议与 CLI 同步
5. **Session 在磁盘** — JSONL 文件存储，SDK 只做读取和追加，不维护内存状态
