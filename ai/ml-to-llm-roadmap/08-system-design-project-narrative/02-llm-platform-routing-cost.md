# LLM 平台、模型路由与成本治理

## 这篇解决什么问题

当公司里只有一个 LLM demo 时，直接在业务代码里调用模型 API 也许够用。但一旦多个团队、多个模型、多个版本、不同预算和不同安全要求同时存在，问题会变成平台问题：谁能调用什么模型，失败后怎么降级，日志怎么留，成本怎么控，质量怎么评估。

这篇解决的问题是：如何从平台层设计 LLM 调用能力，特别是模型路由和成本治理。

## 概念为什么出现

LLM 平台出现，是因为模型调用从“一个接口”变成了“组织级共享能力”。业务团队关心的是任务结果，平台团队要管理的是一组横切问题：

```text
模型网关: 统一鉴权、调用、超时、重试和供应商适配
路由策略: 根据任务、成本、上下文长度和风险选择模型
fallback: 主模型失败、超时或质量不足时如何降级
限流: 防止单团队或异常流量打爆预算和供应商额度
缓存: 复用稳定请求或中间结果，降低成本和延迟
版本管理: prompt、schema、model、routing rule 都要可追踪
日志与评估钩子: 支持离线 eval、线上监控和回归定位
预算控制: 把 token、重试、延迟和失败率变成可治理指标
```

平台层不是为了炫技，而是为了让多个业务在同一套规则下安全、稳定、可审计地使用模型。

## 最小心智模型

可以把 LLM 平台想成一个带策略的模型网关：

```text
business request
  -> auth/quota
  -> prompt/schema version
  -> routing policy
  -> model call
  -> validation/eval hooks
  -> logging/cost accounting
  -> response or fallback
```

业务方只提交任务类型、输入和期望输出；平台负责选择模型、控制调用、记录证据和执行预算规则。

## 最小例子

仍然使用客服工单助手。不同工单不需要同一个模型：

| 路由目标 | 适用场景 | 优点 | 风险 |
|----------|----------|------|------|
| cheap model | 简单分类、短工单、低风险模板回复 | 低成本、低延迟 | 复杂语义可能漏判 |
| strong model | 高价值客户、复杂投诉、多约束回复 | 质量更高、推理能力更强 | 单次成本和延迟更高 |
| long-context model | 长对话历史、长邮件串、需要完整上下文 | 可读入更多信息 | TTFT 和输入成本高 |
| local/edge model | 隐私敏感、离线、低延迟本地分类 | 数据不出端、可控部署 | 能力和上下文通常受限 |

一个简单路由策略可以是：

```text
短文本 + 普通分类 -> cheap model
低置信度或高风险类别 -> strong model
上下文超过阈值且不能压缩 -> long-context model
隐私字段不能出端 -> local/edge model
```

如果 strong model 超时，fallback 不一定是“换便宜模型直接答”。更稳妥的降级可能是只返回分类、不生成草稿，或者升级人工处理。

## 原理层

模型网关统一处理供应商差异。不同供应商的参数名、错误码、流式协议、结构化输出能力和限流规则可能不同。网关把这些差异封装起来，让业务代码依赖内部稳定接口。

路由策略把任务需求映射到模型能力。常见信号包括任务类型、输入 token、输出长度、风险等级、用户层级、历史失败率、目标延迟、预算剩余额度和模型健康状态。路由不是一次性配置，应根据 eval 和线上指标迭代。

Fallback 要区分基础设施失败和质量失败。超时、429、供应商故障可以切供应商或降级模型；低置信度、schema failure、safety risk 更可能需要重试、改参数、收窄输出或人工复核。错误类型不同，fallback 才能不同。

Rate limiting 和 budget control 解决的是组织治理。平台应能按团队、产品、用户、任务类型或环境限制 QPS、并发、token 总量和月度预算。否则一次循环重试或异常长输入就可能迅速放大账单。

缓存可以缓存完全相同的稳定请求，也可以缓存规则分类、prompt 编译结果、系统消息或评估结果。但要注意个性化输入、隐私数据、模型版本和 prompt 版本，否则缓存会返回过期或越权内容。

Prompt/version management 是可回归的前提。一次输出变化可能来自模型版本、prompt 模板、schema、routing rule、temperature 或安全策略。每次请求都要记录这些版本，才能复现线上问题。

成本治理不能只看模型单价。更完整的拆解是：

```text
单请求成本 =
  输入 token 成本
  + 输出 token 成本
  + 重试和 fallback 成本
  + 缓存未命中成本
  + 评估/校验成本
  + 人工复核或失败处理成本
```

如果 eval failure 上升，成本也会隐性上升：系统可能增加重试、触发人工复核、导致用户重复提交，甚至需要回滚和补偿。

## 和应用/面试的连接

应用里，当多个功能都在调用 LLM 时，应尽早把模型调用抽到平台层，至少统一日志、版本、成本和限流。否则每个团队各自处理超时、重试和 prompt 版本，线上回归会很难定位。

面试里，如果被问“如何设计 LLM 平台”或“如何降低模型成本”，不要只说换便宜模型。更完整的回答是：先按任务和风险做模型路由，再用缓存、输入压缩、输出预算、重试控制、限流、eval gating 和预算面板治理全链路成本。

## 常见误区

| 误区 | 更准确的说法 |
|------|--------------|
| 模型路由就是便宜模型优先 | 路由要同时看质量、风险、上下文、延迟、预算和健康状态 |
| Fallback 就是失败后换另一个模型 | 不同失败类型需要不同降级策略，可能是人工复核或只返回部分结果 |
| 成本治理只看 token 单价 | 重试、长输出、缓存未命中、eval failure 和人工处理也会放大成本 |
| 日志只记录最终答案就够了 | 还要记录模型、prompt、schema、路由、参数、延迟、token 和评估结果 |
| 平台层会拖慢业务迭代 | 合理的平台接口能减少重复造轮子，并提高回归定位能力 |
