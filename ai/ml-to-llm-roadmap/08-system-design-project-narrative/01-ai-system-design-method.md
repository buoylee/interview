# AI 系统设计方法论

## 这篇解决什么问题

很多人做 AI 系统设计时会直接回答“我会接一个大模型，然后加 prompt”。这在面试和真实项目里都不够，因为 LLM 系统的风险不只来自服务可用性，还来自输出质量、格式稳定性、延迟、成本、安全和模型行为变化。

这篇解决的问题是：如何用一套通用方法设计 AI 应用，而不是只会背 RAG 或 Agent 模板。默认例子是结构化客服工单分类和回复草稿生成；RAG 与 Agent 题型有独立模块和面试路径，不作为这里的默认方案。

## 概念为什么出现

普通后端系统设计通常重点关注接口、存储、缓存、并发、可靠性和扩展性。AI 系统也需要这些，但还多了模型相关的不确定性：

```text
输出不确定: 同一类输入可能得到不同表达或不同质量
评估困难: HTTP 200 不代表答案正确、有用或安全
延迟/成本波动: 输入长度、输出长度、重试和模型选择都会影响账单
安全风险: 模型可能泄露、编造、越权或生成不合规内容
行为变化: 模型版本、prompt、解码参数和供应商策略都会改变结果
```

所以 AI 系统设计要把“模型可能不稳定”当成系统约束，而不是上线后再靠人工感觉修补。

## 最小心智模型

一个通用 AI 系统设计可以按六步展开：

1. 明确用户、任务和成功指标。
2. 定义输入/输出契约。
3. 选择模型和生成控制策略。
4. 设计评估、校验和 Guardrails。
5. 设计服务路径、延迟和成本控制。
6. 规划监控、回归测试和迭代机制。

这六步对应一条从需求到生产的链路：

```text
who/why -> contract -> model/control -> eval/safety -> serving/cost -> monitor/iterate
```

回答系统设计题时，不要一开始就跳到“用什么模型”。先定义任务和成功标准，再说明为什么这个模型和控制方式适合这个任务。

## 最小例子

假设要设计一个客服工单助手：用户提交一段投诉文本，系统输出结构化分类，并给客服一个回复草稿。

输入：

```text
订单 12345 昨天显示已送达，但我没有收到。请尽快退款。
```

期望输出：

```json
{
  "category": "delivery_issue",
  "urgency": "high",
  "needs_refund_review": true,
  "draft_reply": "抱歉给你带来不便。我们会先核查配送记录，并为你发起退款审核。"
}
```

用六步拆：

1. 用户是客服团队；任务是减少分单和初稿时间；成功指标是分类准确率、草稿采纳率、人工修改率和平均处理时长。
2. 输入契约包括工单正文、订单状态、用户地区和历史投诉次数；输出契约是固定 JSON schema。
3. 模型可以先用便宜模型做分类和草稿，低温度生成，必要字段用结构化输出约束。
4. 评估集覆盖配送、退款、发票、账号、辱骂、敏感信息等样本；Guardrails 检查退款承诺、敏感信息和越权话术。
5. 服务路径要控制超时、重试、缓存常见分类规则，并记录每次 token、延迟和失败原因。
6. 上线后监控分类准确率、schema failure、草稿采纳率、投诉升级率、成本和模型版本回归。

这个例子不需要检索外部知识库，也不需要 Agent 自动调用工具。先把通用链路做好，遇到 RAG/Agent 题时再加对应组件。

## 原理层

第一步是把需求翻译成可评估任务。AI 系统最怕目标含糊，例如“让客服更智能”。更好的定义是：对每条工单输出分类、紧急程度、是否需要退款审核和回复草稿；分类准确率达到某阈值，草稿被客服采纳或少量修改，不能承诺未授权退款。

第二步是输入/输出契约。输入缺字段会让模型猜测，输出无结构会让下游系统难以消费。结构化输出、schema、字段解释和错误处理应该在设计阶段确定，而不是把自然语言答案交给业务系统再解析。

第三步是模型与控制策略。模型选择不只是“越强越好”：分类任务可能适合便宜低延迟模型，复杂投诉解释可能需要更强模型，敏感场景可能需要更严格的解码、模板和后处理。控制策略包括 prompt、temperature、schema-constrained decoding、max tokens、fallback 和人工复核。

第四步是评估和 Guardrails。评估要覆盖正常样本、边界样本和失败样本；Guardrails 要处理格式错误、危险承诺、隐私泄露、政策越界和低置信度升级。没有评估，系统设计就无法证明“这个方案比人工或旧规则更好”。

第五步是服务和成本。LLM 调用的成本由输入 token、输出 token、模型单价、重试次数、缓存命中、并发排队和人工复核共同决定。系统设计要说明 p95 延迟目标、超时策略、限流策略和降级路径。

第六步是监控与迭代。模型、prompt、业务政策和用户输入分布都会变。要记录 prompt 版本、模型版本、schema 版本、输出、评估结果、人工修改和线上反馈，用 regression set 防止一次修复破坏旧能力。

## 和应用/面试的连接

应用里，这套方法能帮助你避免“先接模型再补监控”的返工。每个设计决策都应该能连接到指标：为什么用结构化输出，为什么低温度，为什么某些样本升级人工，为什么要记录失败类型。

面试里，可以先给出六步框架，再用一个具体业务例子展开。被追问 RAG 或 Agent 时，先说明它们是可选扩展：如果问题需要外部知识证据，再引入检索；如果问题需要多步行动和工具调用，再进入 Agent 设计。本题默认先完成通用 AI 系统设计。

## 常见误区

| 误区 | 更准确的说法 |
|------|--------------|
| AI 系统设计就是选一个最强模型 | 还要定义任务、契约、评估、安全、成本和迭代 |
| HTTP 成功就代表 AI 调用成功 | 输出可能格式正确但语义错误、越权或不可用 |
| 面试系统设计必须默认 RAG | RAG 是需要外部知识时的方案，不是所有 AI 系统的起点 |
| Agent 能解决所有复杂流程 | Agent 工具循环会增加权限、观测、失败恢复和成本问题，应单独设计 |
| 评估可以上线后再补 | 没有评估就无法判断模型、prompt 或路由变更是否变好 |

## 自测

1. AI 系统设计相比普通后端系统设计多了哪些不确定性？
2. 六步方法分别解决什么风险？
3. 为什么输入/输出契约要在模型选择前明确？
4. 客服工单例子里，哪些指标能证明系统真的有用？
5. 遇到 RAG/Agent 追问时，为什么要先说明它们是可选扩展？
