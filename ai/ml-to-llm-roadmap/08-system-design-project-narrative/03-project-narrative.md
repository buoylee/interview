# 项目叙事：把经历讲成可追问的技术故事

## 这篇解决什么问题

很多候选人做过 LLM 项目，但讲出来像功能列表：接入模型、写 prompt、上线页面、效果还不错。面试官很难从这种表达里判断你的技术深度、工程判断和复盘能力。

这篇解决的问题是：如何把项目经历讲成“问题 -> 约束 -> 设计决策 -> 评估 -> 生产结果 -> 经验教训”的技术故事，并且能承接追问。

## 学前检查

读这篇前，建议先理解：

- AI 系统设计的六步框架：[AI 系统设计方法论](./01-ai-system-design-method.md)
- 生产问题如何监控和回归定位：[生产排查、监控与回归定位](../07-evaluation-safety-production/03-production-debugging-monitoring.md)
- 模型路由和成本治理如何影响设计取舍：[LLM 平台、模型路由与成本治理](./02-llm-platform-routing-cost.md)

## 概念为什么出现

项目叙事出现，是因为面试不是让你背技术名词，而是判断你是否能在真实约束下做工程取舍。一个好故事要让听者看到：

```text
你解决的问题是什么
你面对哪些约束
你为什么做这些设计选择
你如何证明它有效
上线后发生了什么
你从失败和迭代里学到了什么
```

如果只讲“用了某个模型”，项目会显得浅；如果能讲清评估、延迟、成本、安全、失败样本和下一轮迭代，项目才有技术可信度。

## 最小心智模型

项目叙事可以用 STAR + 技术深度混合模板：

```text
Situation: 业务背景和痛点
Task: 你的目标、边界和成功指标
Action:
  - 输入/输出契约
  - 模型和控制策略
  - 评估与安全
  - 服务、成本和监控
Result: 指标变化、上线结果和影响
Reflection: 失败案例、取舍和下一步
```

更短的面试版本是：

```text
问题 -> 约束 -> 设计决策 -> eval -> 生产结果 -> lessons
```

## 最小例子

一个通用 LLM 输出质量改进项目可以这样讲，不依赖 RAG，也不依赖 Agent：

```text
问题: 客服回复草稿经常语气不稳定，有时遗漏退款审核条件，客服需要大幅修改。
约束: p95 延迟不能超过 3 秒，单请求成本不能明显上涨，不能承诺未授权退款，输出要进入内部工单系统。
设计: 把自由文本改成 JSON schema，字段包括 category、risk_flags、draft_reply 和 confidence；低温度生成；高风险类别触发更强模型或人工复核；加入 forbidden claims 检查。
eval: 建立 300 条工单 golden set，rubric 包含分类正确性、政策合规、语气、完整性和可编辑性；上线前对比旧 prompt。
生产结果: schema failure 降低，客服平均修改字数下降，高风险误承诺减少，但复杂长工单仍需要人工处理。
lessons: 质量提升不只靠换模型，输出契约、评估集、风险分流和监控更关键。下一步是补长工单压缩和更细粒度的失败分类。
```

这个故事的重点不是“我用了 LLM”，而是你如何把输出质量问题拆成可设计、可评估、可上线的工程问题。

## 原理层

“问题”要具体。不要说“提升客服效率”，而要说“回复草稿需要大量人工修改，特别是退款类工单容易遗漏政策条件”。具体问题才能导出指标和设计。

“约束”要真实。常见约束包括数据隐私、延迟、成本、业务政策、安全、人工审核、系统集成、可解释性和上线风险。没有约束的方案通常像空想架构。

“设计决策”要解释取舍。比如为什么不用最强模型处理所有请求：成本高、延迟高、简单请求浪费；为什么用 schema：下游系统需要稳定字段；为什么引入人工复核：高风险承诺不能只靠模型决定。

“评估”要能证明改进。可以包含 golden set、regression set、人工标注、LLM-as-Judge、线上 A/B、用户反馈和人工修改率。关键是说明指标怎么定义、样本怎么覆盖、失败样本怎么进入下一轮。

“生产结果”要讲指标和副作用。好的叙事不仅说成功，也说代价：延迟是否上升，成本是否变化，哪些场景仍失败，哪些策略后来被回滚或调整。

“经验教训”要体现成长。比如：prompt 不是唯一杠杆；低质量样本分析比盲目调参更有效；模型路由需要 eval 支撑；监控 schema failure 和人工修改率比只看 API error 更有用。

## 和应用/面试的连接

应用里，这个模板能倒逼你在项目过程中记录可复盘证据：问题定义、指标、版本、失败案例、上线结果和后续计划。项目做完才补故事，往往会缺关键数据。

面试里，可以先用 60 到 90 秒讲主线，再准备追问清单：

| 追问方向 | 你要能回答 |
|----------|------------|
| eval | 样本怎么选，rubric 怎么定，如何防止只优化少数样本 |
| latency | p50/p95 是多少，慢在哪里，如何降级 |
| cost | token、重试、模型选择和人工复核成本如何拆 |
| safety | 哪些输出不能出现，如何检测和升级 |
| tradeoffs | 为什么不用更强模型、更多上下文或更复杂架构 |
| failure case | 最典型失败样本是什么，后来如何处理 |
| next iteration | 如果再做一版，会优先改什么 |

如果面试官追问 RAG 或 Agent，可以明确说明：这个项目的核心问题是输出质量和结构化契约，不是外部知识召回或多步工具执行；如果未来需要引入知识库或自动操作，再单独设计 RAG/Agent 模块。

## 常见误区

| 误区 | 更准确的说法 |
|------|--------------|
| 项目叙事就是 STAR 背稿 | STAR 只是结构，技术深度来自约束、决策、评估和复盘 |
| 只要讲模型能力强就够了 | 面试官更关心你如何定义问题、验证质量和处理失败 |
| 不能讲失败 | 可信项目通常都有失败样本和迭代，关键是你如何定位和修复 |
| 指标越多越好 | 指标要服务问题，少量关键指标比堆列表更有说服力 |
| 所有 LLM 项目都要包装成 RAG/Agent | 不需要。分类、改写、摘要、结构化输出和质量改进同样可以有深度 |

## 自测

1. 用一句话说清你的项目解决了什么具体问题。
2. 这个项目最大的三个约束是什么？
3. 你做过的一个关键设计决策是什么，放弃了什么替代方案？
4. 你如何证明输出质量变好了？
5. 上线后最典型的失败案例是什么？
6. 如果再迭代一版，你会优先改什么？

## 回到主线

到这里，你已经能把系统设计和项目经历连起来：先用工程框架设计 AI 系统，再用项目叙事表达你的决策、证据和复盘。后续面试路径和速记材料会把这些内容压缩成冲刺版，但默认理解仍来自本模块的系统学习。
