# 状态、记忆与任务规划

## 这篇解决什么问题

Agent 不只是在每轮重新读一遍聊天记录。它要知道目标是什么、已经做了什么、哪些工具调用还没完成、哪些约束不能违反，以及什么时候应该停止。没有状态，Agent 会重复调用工具、忘记用户限制，或把旧观察当成最新事实。

这一篇解决的问题是：Agent 里的 state、memory 和 planning 分别是什么，以及它们和 RAG、长上下文有什么边界。

## 学前检查

读这篇前，建议先理解：

- RAG 解决外部知识接入：[RAG 解决什么问题](../01-rag-retrieval-systems/01-rag-problem-boundary.md)
- 上下文长度会带来成本和缓存影响：[Prefill、Decode 与 KV Cache](../06-inference-deployment-cost/01-prefill-decode-kv-cache.md)

如果还不熟，可以先记住一句话：状态是当前任务的工作台，记忆是可按需取用的历史材料。

## 概念为什么出现

多步任务会产生多种信息：

```text
用户约束: 预算、时间、偏好、禁忌
工具观察: 查询结果、失败原因、创建的工单号
执行轨迹: 计划过什么、调用过什么、为什么改路线
停止条件: 何时算完成，何时必须转人工
```

如果把这些都混在原始聊天记录里，模型每轮都要从噪声里重新推断，成本高，也容易漏掉关键状态。

## 最小心智模型

Agent 常见信息层可以这样分：

| 信息层 | 含义 | 用途 |
|--------|------|------|
| working state | 当前任务的关键变量和约束 | 决定下一步 |
| conversation history | 用户和助手原始对话 | 保留语境 |
| scratchpad/trajectory | 中间推理、工具调用和观察轨迹 | 调试和决策依据 |
| summarized memory | 历史对话或任务摘要 | 降低上下文成本 |
| retrieved memory | 从外部存储检索回来的相关记忆 | 只在相关时注入 |
| task state | 任务阶段、完成项、待处理项和停止条件 | 控制流程 |

规划则回答“下一步怎么走”。常见模式包括：

- direct next-action：每轮直接决定下一步。
- plan-and-execute：先生成高层计划，再逐步执行和修正。
- task decomposition：把大任务拆成子任务。
- multi-agent：把不同子任务交给不同角色，只有在职责确实独立时才需要。

## 最小例子

用户让旅行助手规划一次上海到成都的周末旅行：

```text
预算 3000 元，不要红眼航班，周六早上出发，想吃川菜，周日晚上回。
```

working state 可以记录：

```json
{
  "origin": "上海",
  "destination": "成都",
  "budget": 3000,
  "constraints": ["不要红眼航班", "周六早上出发", "周日晚上返回"],
  "completed_steps": ["收集用户约束"],
  "pending_tool_calls": ["search_flights", "search_hotels"],
  "stop_condition": "给出满足预算和时间约束的行程"
}
```

如果 `search_flights` 返回周六早上机票已经超预算，Agent 应更新状态：预算冲突、需要询问用户是否提高预算或改出发时间，而不是直接编一个便宜航班。

## 原理层

working state 和长期记忆的区别在于生命周期。working state 服务当前任务，结束后可以丢弃或摘要；长期记忆用于跨会话偏好、历史案件和用户资料，但必须有权限和过期策略。

不能把所有历史都塞进上下文，原因有三个：

```text
成本: prefill token 越多，延迟和费用越高
干扰: 无关历史会影响当前决策
安全: 旧工具输出和用户文本可能携带不该继续生效的指令
```

记忆检索常复用 RAG-like retrieval：把历史摘要、用户偏好或案例片段存入索引，需要时按当前任务召回。但 Agent 不是 RAG。RAG 的核心是证据增强回答；Agent 的核心是循环决策和行动。Agent 可以用 RAG 找记忆，不能因此把两者画等号。

Tree-of-Thought、Graph-of-Thought 和 MCTS 都可以看成更复杂的搜索或规划参考。第一遍学习 Agent 时，不需要先掌握这些方法；先把状态、工具观察、停止条件和恢复路径设计清楚。

## 和应用/面试的连接

应用里，状态通常要显式建模，而不是只靠 prompt 里一句“请记住”。例如客服升级 Agent 应记录：用户身份、订单号、已查工具、失败原因、是否创建工单、是否需要人工、最终回复是否已发送。

面试里，可以这样说：短任务用 direct next-action；长任务或依赖多步验证时用 plan-and-execute；多 agent 只有在子任务需要不同工具、权限或评估标准时才有价值，否则只是增加同步成本和排查难度。

下一篇会看这些状态和轨迹如何进入评估、安全和生产日志：[Agent 评估、安全与生产排查](./04-agent-evaluation-safety-production.md)。

## 常见误区

| 误区 | 更准确的理解 |
|------|--------------|
| memory 就是聊天历史 | 记忆包括摘要、检索结果、偏好和任务记录 |
| 上下文越长越好 | 长上下文会增加成本、噪声和安全风险 |
| plan 一旦生成就不能改 | Agent 计划应根据观察结果更新 |
| multi-agent 总是更强 | 多角色会增加协调、权限和评估复杂度 |
| Agent 和记忆检索等于 RAG | 检索可复用 RAG 技术，但 Agent 关注行动循环 |
