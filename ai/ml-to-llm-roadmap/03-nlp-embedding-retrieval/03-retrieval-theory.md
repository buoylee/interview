## 1. 检索全景：一条 RAG 流水线

后面会冒出一堆名词（BM25、Dense、ANN、HNSW、IVF、PQ、Rerank、Hybrid）。**它们不是并列的知识点，而是一条流水线上不同工位的零件。** 先把这张图刻进脑子，后面就不会糊。

### 1.1 检索到底在解决什么问题？

```
RAG 的检索 = 从海量文档里，快速找出和 query 最相关的几条，喂给 LLM。

这其实是你当后端时天天解决的问题：
  "数据太多，怎么不全表扫描就找到最相关的几条？"
你在 MySQL 里的答案是「建索引」。这一节所有技术，本质都是这个答案的变体。
```

### 1.2 流水线总图 ⭐ 记住这张

```
┌──────────────── 离线（建库，一次性）────────────────┐
   文档 → 切块(chunk) → Embedding 编码 → 建索引(HNSW/IVF/PQ)
                                              ↓
                                        向量数据库 + 倒排索引
└─────────────────────────────────────────────────────┘

┌──────────────── 在线（每次查询）────────────────────┐
   query
     ↓
   ① 粗筛 / 召回（要快，捞出 top-100 候选）
      ├─ 稀疏：BM25      → 关键词，跑在倒排索引上
      ├─ 稠密：Dense+ANN → 语义，跑在向量索引上
      └─ 两路都要 → Hybrid，用 RRF 融合两边排名
     ↓
   ② 精排 / 重排（要准，top-100 → top-10）
      └─ Cross-Encoder reranker
     ↓
   ③ 把 top-K 文档塞进 prompt → 交给 LLM 生成答案
└─────────────────────────────────────────────────────┘
```

### 1.3 收成三类，就不乱了

整节的技术其实只回答三个问题：

```
① 怎么不全表扫描就召回？   → 索引：BM25(倒排) / HNSW / IVF
② 怎么在内存里塞下海量向量？ → 压缩：PQ / IVF-PQ
③ 怎么又快又准？           → 两段式：Bi-Encoder 粗筛 → Cross-Encoder 精排
                                       Hybrid：关键词 + 语义两路融合
```

### 1.4 名词翻译表：吓人词 → 你早就会的

| 名词 | 在流水线的位置 | 后端类比 | 一句话 |
|------|--------------|---------|--------|
| 暴力搜索 | 粗筛（慢） | 全表扫描 O(N) | 慢但 100% 准 |
| **ANN** | 粗筛 | 给向量"建索引" | 牺牲一点点准，换大幅加速 |
| **HNSW** | 向量索引 | 跳表 / 多层索引 | 默认选项 |
| **IVF** | 向量索引 | 分库分表 / 分区 | 只搜相关的桶 |
| **PQ** | 索引存储 | 有损压缩（JPEG） | 省内存 |
| **BM25** | 粗筛（稀疏） | 全文索引 / 倒排索引 | 关键词精确匹配 |
| **Bi-Encoder** | 粗筛（稠密） | 预先算好"指纹"再比 | 快 |
| **Cross-Encoder / Rerank** | 精排 | 取出后逐条精算 | 准但慢 |
| **Hybrid** | 粗筛融合 | 两个索引取长补短 | BM25 + Dense |

> 🔑 **一句话**：检索 = 先用便宜的方法（索引）粗筛出一批候选，再用贵的方法（rerank）精排出最终几条。和你后端「先 SQL 粗过滤，再应用层精算」「先查 L1 cache 再查 L2」是同一个套路。

## 2. 稀疏检索 BM25

> 流水线位置：**① 粗筛 · 稀疏路**。跑在倒排索引上，负责"关键词精确匹配"。

### 2.1 BM25 是什么？

```
BM25 = TF-IDF 的改进版，是最经典的关键词检索算法

BM25(q, d) = Σ IDF(qᵢ) × [f(qᵢ,d) × (k₁+1)] / [f(qᵢ,d) + k₁ × (1-b+b×|d|/avgdl)]

qᵢ = query 中的第 i 个词
f(qᵢ, d) = 词 qᵢ 在文档 d 中的词频
|d| = 文档长度
avgdl = 平均文档长度
k₁, b = 超参数（通常 k₁=1.2, b=0.75）
```

### 2.2 人话解释：BM25 到底在算什么？

**不用记公式，记住三个核心思想就够：**

**① IDF：越稀有的词越重要**

```
搜索 "Python 安装教程"

"的" → 几乎每篇文档都有 → 不重要，权重低
"Python" → 只有部分文档有 → 有区分度，权重高
"安装" → 也不是到处都有 → 有区分度，权重高
```

**② TF 饱和：词频有上限，不会无限涨**

```
"Python" 在文档中出现 1次 → 分数 1.0
"Python" 在文档中出现 2次 → 分数 1.5
"Python" 在文档中出现 5次 → 分数 1.8
"Python" 在文档中出现100次 → 分数 1.9  ← 不会无限涨，有天花板
```

**③ 长度归一化：长文档会被适当惩罚**

```
100字的文档出现 5次 "Python" → 密度高 → 更相关
10000字的文档出现 5次 "Python" → 密度低 → 可能只是顺带提了一嘴
```

> 🔑 **一句话总结**：对 query 中的每个词，算 "这个词有多稀有 × 在文档里出现多频繁（有上限，且考虑文档长度）"，所有词的分加起来就是最终得分。

### 2.3 BM25 对 TF-IDF 的改进

| 改进 | TF-IDF 的问题 | BM25 的解决 |
|------|-------------|-------------|
| TF 饱和 | 词出现 100 次 vs 10 次，TF-IDF 差 10 倍 | BM25 用对数增长，有上界 |
| 文档长度 | 长文档天然有更多词 → 总是偏高 | 用 b 参数做文档长度归一化 |

### 2.4 BM25 的优缺点

| 优点 | 缺点 |
|------|------|
| 精确关键词匹配 | 不理解语义（"猫" ≠ "小猫"）|
| 不需要训练 | 不理解同义词 |
| 非常快 | 不理解上下文 |
| 倒排索引成熟 | |

> 🔑 **为什么 BM25 仍然重要？** 搜索 "Python 3.12 release date"，如果文档中确实有 "Python 3.12"，BM25 精确匹配比任何 Embedding 都可靠。这就是为什么 §6 要把它和 Dense 混着用。

## 3. 稠密检索 Dense Retrieval

> 流水线位置：**① 粗筛 · 稠密路**。跑在向量索引（下一节的 ANN）上，负责"语义匹配"。

### 3.1 核心思想

```
文档和 query 都编码为 Embedding 向量
检索 = 在向量空间中找最相似的向量

Query: "LLM 怎么训练"
  → Embedding → [0.3, -0.1, ...] (768D)
  → 在向量库中找余弦最近的文档

文档: "大模型的预训练流程包括..."
  → Embedding → [0.28, -0.12, ...] (768D)
  → 余弦相似度 0.92 ← 高！
```

> 🔑 这里"编码成向量"用的就是上一节 [02-Embedding](./02-embedding-theory.md) 的 Bi-Encoder。文档离线编码好存进库，查询时只编码 query —— 这就是流水线"离线建库"的那一步。

### 3.2 优缺点

| 优点 | 缺点 |
|------|------|
| 理解语义（"猫" ≈ "小猫"）| 可能遗漏精确关键词 |
| 跨语言检索 | 需要 Embedding 模型 |
| 可以处理同义词 | 向量存储和计算成本 |

> 那个"向量存储和计算成本"的缺点，正是下一节 ANN 要解决的问题。

## 4. ANN 近似最近邻

> 流水线位置：**让 §3 的 Dense 检索能在海量向量上跑得动**。这是本节最容易被名词淹没的一段，所以先补 3 个后端原语，再逐个讲。

### 4.1 为什么需要 ANN？

```
精确最近邻（暴力搜索）：和每个向量计算相似度 → O(N)
  N = 100 万文档，768 维 → 每次查询需要 7.68 亿次乘法 → 太慢！

ANN：牺牲一点精度换取大幅加速 → 近似 O(log N)
```

类比：你不会为了找一个人，把全国 14 亿人挨个问一遍。你会先缩范围（哪个省、哪个市）。ANN 就是给向量空间建这种"缩范围"的索引。

**为什么不直接用 B 树 / KD 树？——维度灾难（curse of dimensionality）**

你后端找数据的第一反应大概是「建棵树」（B 树、KD 树），二分一下不就 O(log N) 了？低维确实行，但向量是 768 维，树会失效：

- KD 树每层只按一个维度切一刀。768 维要切很多刀才轮得到同一个维度第二次，树深到失去意义。
- 更要命的是高维下「距离」会塌缩：随机两点的最远距离和最近距离几乎一样大，「最近邻」本身就没什么区分度，基于划分的剪枝几乎剪不掉任何分支 → 退化成全表扫描。

这就是**维度灾难**：低维好用的精确索引（树）到高维全线失效。所以高维只能换思路——要么**用图**（HNSW，靠邻居一步步逼近），要么**靠聚类分桶 + 量化**（IVF / PQ）。下面四种方法，本质都是在绕开维度灾难。

### 4.2 先补 3 个原语（后面全靠它们）

ANN 的几种方法只用到三个后端概念，先花 90 秒铺垫，后面就不卡了：

**① 倒排索引（Inverted Index）—— 你在 ES / MySQL 全文索引里见过**

```
正排：文档 → 它含哪些词       （doc3 → [Python, 安装, 教程]）
倒排：词 → 含它的文档列表      （Python → [doc3, doc7, doc12]）

查 "Python" 时只看挂在它下面的几篇，不用全扫。BM25 就跑在它上面。
一句：从「词」反查「文档」，所以叫"倒排"。记住"建一张 key→候选列表 的表"这个套路，IVF 等会儿照抄。
```

**② 聚类（Clustering）—— 把相近的向量分组**

```
做法：像 K-Means，把所有向量分成 K 组，每组算一个中心点（质心 centroid）。
每个向量归到离它最近的质心。

后端类比：hash 分区 / 分库分表。只不过分区键不是 user_id % 16，
         而是"离哪个质心最近"。
```

**③ 量化（Quantization）—— 用少量比特近似表示，有损压缩**

```
一个 768 维 float32 向量 = 768 × 4 = 3072 字节。
1 亿个向量 ≈ 286 GB → 内存放不下。

量化 = 用近似值换体积，就像：
  - JPEG 把图片压小
  - 音频 320kbps → 128kbps

代价：存的是近似值 → 算出来的距离也是近似的（精度下降）。
```

> 🔑 有了这三个原语，下面四种 ANN 方法就是它们的排列组合：HNSW 靠图、IVF 靠聚类+倒排、PQ 靠量化、IVF-PQ 把后两个合起来。

### 4.3 HNSW（分层导航小世界图）⭐⭐ 面试高频

> 思路：**多层图 + 贪婪搜索**。是目前效果最好、最常用的 ANN，几乎所有向量库的默认索引。

```
构建一个多层图结构：

顶层(稀疏): 少量节点，长距离连接 → 快速定位大致区域
  ↓
中层:       更多节点，中距离连接
  ↓
底层(密集): 所有节点，短距离连接 → 精确搜索

查询过程：
1. 从顶层开始，贪婪搜索找到最近邻居
2. 进入下一层，从上层的最近邻开始继续搜索
3. 层层细化，最终在底层找到精确结果

类比：找一个人
  顶层 = 先确定在哪个国家（亚洲→中国）
  中层 = 哪个城市（北京→海淀）
  底层 = 哪条街哪栋楼
```

**底层长什么样（后端视角：邻接表图 + 一个贪婪循环）**

别被「图」吓到，它就是你写过无数次的**邻接表**：

```java
class Node {
    float[] vec;             // 这个节点的向量
    List<Node>[] neighbors;  // neighbors[L] = 它在第 L 层的邻居指针
    int topLayer;            // 它最高出现在第几层
}
Node entryPoint;            // 全局唯一入口 = 层数最高的那个点
```

「多层」的真相：**同一批节点，越上层留得越少**。每个点插入时抛一枚「越抛越小」的硬币决定它最高爬到第几层 → 顶层只剩零星几个点（快速通道），底层 0 含全部点。

查询就是一个贪婪走法，读起来像 BFS：

```java
// 在某一层里，从 cur 出发，一步步走向离 query 最近的邻居
Node greedyOnLayer(query, cur, L) {
    while (true) {
        Node best = cur;
        for (Node n : cur.neighbors[L])          // 只看当前点的邻居，不全扫
            if (dist(query, n.vec) < dist(query, best.vec))
                best = n;
        if (best == cur) return cur;             // 邻居都没我近 → 卡住，这层到头
        cur = best;                              // 否则迈过去，继续
    }
}

Node search(query) {
    Node cur = entryPoint;
    for (int L = topLayer; L >= 1; L--)          // 上面几层：快速通道，每层只留 1 个最优
        cur = greedyOnLayer(query, cur, L);
    return beamOnLayer0(query, cur, ef_search);  // 第 0 层：同时留 ef 个候选，不止 1 个
}
```

这段代码把三件文档常常没讲透的事讲清了：

- **「贪婪搜索」具体指什么** → `while` 里每次只跳到「比我更近的邻居」，跳不动就停。
- **为什么是近似、会漏** → 贪婪会卡在**局部最优**（当前点的邻居都没 query 近，但全局还有更近的在别处）。
- **`ef_search` 的物理意义** → 第 0 层不止留 1 个最优，而是同时留 ef 个候选一起往前探，降低卡在局部最优的概率。**这就是 recall 旋钮的真身**：ef 越大、留的候选越多、越不容易漏，但每个候选都要算距离，所以越慢。

**和你后端的连接**：这就是「跳表（skip list）」的多层思想搬到图上 —— 上层稀疏指针大步跳，下层密集指针精确定位。复杂度也一样是 ≈ O(log N)。

**线上唯一常调的旋钮**：`ef_search` = 查询时的搜索宽度。调大 → recall 高但慢；调小 → 快但可能漏。建图参数 `M`（每个节点连几条边）和 `ef_construction`（建图时搜多宽）一般建库时定好。

> 🔑 **代价**：HNSW 效果好，但**全图必须在内存里**（指针密集），内存占用大、建索引慢。这正是下面 IVF / PQ 要解决的问题。

### 4.4 IVF（Inverted File）—— 分桶，只搜相关的桶

> 思路：把"倒排索引"的套路搬到向量上 —— 先聚类分桶，查询时只搜最近的几个桶。

```
问题：HNSW 全图在内存，十亿级向量扛不住。能不能别每次都搜全部？

朴素想法：先把向量空间切成若干区域，查询时只搜 query 落在的那几个区域。

IVF 怎么做：
  1. 离线：对所有向量做聚类（K-Means），得到比如 1000 个质心
     → 形成 1000 个"桶"，每个质心下挂着归属它的那批向量
     （这张「质心 → 向量列表」的表，就是倒排索引的翻版，故名 Inverted File）
  2. 查询：先算 query 离哪几个质心最近 → 只在最近的 nprobe 个桶里精搜
```

**底层长什么样**：那张「质心 → 向量列表」的表，就是一个 `HashMap`（这就是它叫 Inverted File 的原因，套路和 §4.2 的倒排索引一模一样）：

```java
Map<Integer, List<VectorId>> invertedLists;  // 质心id → 挂在它下面的向量
float[][] centroids;                          // K 个质心向量，比如 K=1000

List<VectorId> search(query, nprobe) {
    // 1. 算 query 到每个质心的距离，挑最近的 nprobe 个桶（K 很小，这步很便宜）
    int[] nearestBuckets = topNprobe(query, centroids, nprobe);
    // 2. 只在这几个桶里暴力扫，其余 990 个桶完全不碰
    List<VectorId> cand = new ArrayList<>();
    for (int c : nearestBuckets)
        cand.addAll(invertedLists.get(c));
    return bruteForceTopK(query, cand);
}
```

看清楚就一句话：**先花 O(K) 算到各质心的距离、挑几个桶，再只暴力扫这几个桶里的向量。** K=1000 个质心的距离计算，比起扫 100 万向量便宜三个数量级——省的就是这部分。

**旋钮 `nprobe`**：搜几个桶。

```
nprobe = 1   → 最快，但可能漏（query 落在桶边界，真正的最近邻在隔壁桶）
nprobe = 10  → 慢一点，召回更高
nprobe = 全部 → 退化成暴力搜索
```

**后端类比**：分库分表只查命中的分区，不全库扫。`nprobe` 就是"多查几个相邻分区以防边界遗漏"。

> 🔑 **一句话**：聚类分桶 + 只搜最近几桶，用"漏一点召回"换"少搜很多向量"。

### 4.5 PQ（Product Quantization）—— 压缩向量，省内存

> 思路：把"量化（有损压缩）"用到极致，让海量向量塞进内存。

```
问题：向量本身太占内存。1 亿个 768 维 float32 ≈ 286 GB，放不进内存。

PQ 怎么做：
  1. 把 768 维切成比如 8 段，每段 96 维
  2. 每段单独训一个小"码本"（codebook），比如 256 个代表向量
  3. 每个向量的每一段，只存"最像码本里哪个代表"的编号（1 字节）
     → 一个向量从 3072 字节压到 8 字节，约 360 倍！

"Product"（笛卡尔积）：8 段各从 256 个里选 1 个，
  组合出 256⁸ ≈ 1.8×10¹⁹ 种可能 → 表达力够用。
```

**查询怎么不解压就算距离（ADC，非对称距离计算）**：这是 PQ 最妙的机关。query **不压缩**、保持全精度；要算它和某个「8 字节向量」的距离，分两步：

```
① 每次查询开头做一次：建一张距离查找表
   query 也切成 8 段。对第 j 段，预先算出「query 第 j 段」到该段码本里 256 个代表的距离
   → 得到一张 8 × 256 的表  LUT[段][码本编号]

② 对库里每个向量（它就是 8 个字节 = 8 个编号 b0..b7）：
   近似距离 = LUT[0][b0] + LUT[1][b1] + ... + LUT[7][b7]
   → 8 次查表 + 7 次加法，全程不碰原始 768 维向量
```

对比一下成本：暴力是「每个向量都算一次 768 维距离」；PQ 是「每次查询只建一次 8×256 的表，之后每个向量只要 8 次查表相加」。**把『算距离』摊薄成了『查表』**——这才是 PQ 又省内存又快的真正原因，省内存反倒只是顺带的（之所以叫"非对称"：query 是精确值、库向量是量化值，两边精度不对称）。

**代价**：存的是近似值，距离不精确 → 精度下降。所以实战常配 rerank 或对 top 候选重算精确距离来补救。

**类比**：JPEG 分块压缩；每个小块用调色板里最接近的颜色编号代替原始像素。

> 🔑 **一句话**：切段 + 码本编号，约 360 倍压缩换一点精度。

### 4.6 IVF-PQ —— 合体，十亿级标配

```
IVF 解决"少搜"（分桶缩范围）         → 解决速度
PQ  解决"省内存"（压缩向量）          → 解决容量

合体 IVF-PQ：
  先用 IVF 分桶定位到最近的几个桶，
  桶内的向量再用 PQ 压缩存储 + 近似距离计算。

→ 十亿级 + 内存受限场景的工业标配（FAISS 的 IndexIVFPQ）。
```

> **一个值得知道的细节（面试加分）**：IVF-PQ 里 PQ 压的通常不是原向量，而是**残差**（向量 − 所属质心）。同一个桶里的向量都离质心很近，残差数值范围小，量化误差更低——等于先用质心扣掉大头，PQ 只需编码剩下的小尾巴。能说出「PQ on residual」这一句就显得懂。

> 🔑 **一句话**：分桶缩范围 + 压缩省内存，**十亿级才需要**。百万千万级直接 HNSW 就好。

### 4.7 选型对照（复习用）

讲完原理，这张表才好读 —— 它是上面四节的浓缩对比：

| 方法 | 思路（对应原语） | 特点 | 什么时候用 |
|------|------|------|-----------|
| **HNSW** ⭐ | 多层图 + 贪婪搜索 | 效果最好、**内存占用大** | 默认。百万~千万级、内存够 |
| **IVF** | 聚类 + 倒排分桶 | 缩小搜索范围 | 常和 PQ 结合 |
| **PQ** | 向量量化压缩 | 省内存、精度降 | 内存吃紧 |
| **IVF-PQ** | IVF + PQ | 又快又省内存 | 十亿级、内存受限 |
| **ScaNN** (Google) | 各向异性量化 | Google 内部用 | 认得名字即可，可跳过 |

> **应用工程师怎么选（记这三句就够）**：
> - 默认 **HNSW**（百万~千万级，内存够）
> - 内存紧 / 十亿级 → **IVF-PQ**
> - **ScaNN** → 认得名字、知道"Google 自研的各向异性量化"即可，可跳过

### 4.8 选型决策（实战）：从需求 → 索引 → 参数

上面 §4.7 是**静态对照表**（哪个索引有什么特点）。真到项目里，选型是个**带阈值的决策流程**：先看规模、再看内存、落到索引，最后还得调参数。这一节就是那套流程。

**第一步：先量三个数，选型 80% 就定了**

```
1. 规模 N        ：多少向量？      百万 / 千万 / 亿 / 十亿
2. 内存预算       ：放得下原始向量吗？  N × 维度 × 4 字节 ≈ ?
                    （100 万 × 768 × 4 ≈ 3 GB；10 亿 ≈ 2.8 TB → 放不下，必须压）
3. recall / 延迟  ：要多准、p99 几 ms？
```

**第二步：照着这个决策流走**

```
N < 1 万？
  → 别上 ANN！直接 Flat 暴力扫（FAISS IndexFlat）。
     N 太小，ANN 省的那点时间，不值它掉的 recall。← 常见反模式：小数据硬上 HNSW

1 万 ~ 千万，内存放得下原始向量？
  → HNSW。默认选项，recall/延迟最好，旋钮少。

千万 ~ 亿，内存开始紧？
  → HNSW 撑得住就继续；撑不住 → IVF（分桶少搜），再紧 → IVF + PQ。

十亿级 / 内存明显放不下原始向量？
  → IVF-PQ。分桶缩范围 + 压缩省内存，工业标配（FAISS IndexIVFPQ）。
```

**第三步：recall / 延迟 / 内存 是个三角，只能要两个**

```
              recall 高
               /    \
              /      \
        延迟低 ────── 内存省

想三个全要 → 做不到。选型本质就是决定牺牲哪个角：
  - 要快 + 准   → 牺牲内存（HNSW 全图驻内存）
  - 要省内存 + 准 → 牺牲速度（PQ 查表近似 + 对 top 候选 rerank 重算）
  - 要快 + 省内存 → 牺牲 recall（IVF 把 nprobe 调小）
```

**第四步：选完索引，旋钮往哪调（这步最实战，面试常追问）**

前面散落各处的旋钮，收在一张表里：

| 索引 | 旋钮 | 调大 → | 何时定 | 起点 |
|------|------|--------|--------|------|
| HNSW | `M`（每点边数） | recall↑、内存↑、建库慢 | 建库 | 16~32 |
| HNSW | `ef_construction` | 建图质量↑、建库慢 | 建库 | ~200 |
| HNSW | `ef_search` | recall↑、查询慢 | **查询时** | 压着 recall 调 |
| IVF | `nlist`（质心数） | 桶更细、单桶更快 | 建库 | √N ~ 4√N |
| IVF | `nprobe`（搜几个桶） | recall↑、查询慢 | **查询时** | 几 ~ 几十 |
| PQ | `m`（切几段） | 更准、码更长更占内存 | 建库 | 维度能整除即可 |
| PQ | `nbits`（每段比特） | 码本更大更准 | 建库 | 8（=256 码本） |

**调参套路一句话**：建库参数（M / ef_construction / nlist / m）建库时定死；**线上只动查询时旋钮**（`ef_search` / `nprobe`）——压着 recall 目标往上调，达标且延迟能接受就停。

**第五步：几个面试爱问的 gotcha**

- **频繁增删改？** HNSW 删除是「软删除 + 标记」，删多了图会退化、得定期重建；大量更新时 IVF 也要重训质心。**纯增量、很少删 → HNSW**；**频繁全量重建 → IVF 系**。
- **要带元数据过滤？**（如"只在 user=X、近 30 天的文档里搜"）这叫 filtered search，会影响选型：过滤太狠时 HNSW 的图会被"打断"导致 recall 掉，Qdrant / Milvus 专门优化了 filtered HNSW。面试能点出 **pre-filter（先筛元数据再搜）vs post-filter（先搜再筛）** 的取舍就够。
- **GPU 在手？** FAISS 的 IVF-PQ 有 GPU 实现，十亿级离线建库快很多；HNSW 主要吃 CPU + 内存。

> 🔑 **一页纸记忆**：先看**规模**定大类（小→Flat，中→HNSW，大→IVF-PQ），再用**三角权衡**决定牺牲哪个角，最后**只调查询时旋钮**（`ef_search`/`nprobe`）压着 recall 目标走。

### 4.9 向量数据库

你不会自己实现上面这些索引，而是用现成的向量库。知道它们底层用什么索引即可：

| 数据库 | 核心索引 | 特点 |
|--------|---------|------|
| **Milvus** | IVF/HNSW/PQ | 功能丰富、分布式 |
| **Pinecone** | 自研 | 全托管 SaaS |
| **Weaviate** | HNSW | 内置多模态 |
| **Qdrant** | HNSW | Rust、高性能 |
| **Chroma** | HNSW | 轻量、易上手 |
| **pgvector** | IVFFlat/HNSW | PostgreSQL 插件 |
| **FAISS** (Meta) | IVF/PQ/HNSW | 库而非数据库 |

## 5. Reranking 重排

> 流水线位置：**② 精排**。粗筛召回 top-100 后，对这 100 个候选做精细排序，挑出真正的 top-10。

### 5.1 为什么需要重排？

```
Bi-Encoder (初检): 快但不够准（句子独立编码，没有交叉信息）
Cross-Encoder (重排): 准但慢（两个句子一起编码，有交叉注意力）

两阶段检索 (Two-Stage Retrieval):
  Step 1: Bi-Encoder 快速检索 top-100 候选   ← §3 + §4 干的活
  Step 2: Cross-Encoder 对 100 个候选精细排序 → top-10
```

**为什么要分两段——一句话根因**：Bi-Encoder 编码一篇 doc 时**不需要 query**，所以能离线把全库编码好、建索引（§4 那套）；Cross-Encoder 必须把 query 和 doc **拼在一起**才出分，没法离线预算，只能在查询时对每个 (query, doc) 对**现跑一次模型**。于是：

- 能 scale 的（Bi-Encoder）不够准 → 拿来**海捞** top-100。
- 够准的（Cross-Encoder）跑不起百万次 → 只拿来精排那 **100 个**。

把这两个一头一尾夹起来，就是两段式。和"先 SQL 粗过滤、再应用层精算"一模一样——但这里分两段是**被迫**的，下面 §5.2 看代码就知道为什么没法合成一段。

### 5.2 Cross-Encoder 原理

```
Bi-Encoder:   句子 A → [BERT] → vecA   句子 B → [BERT] → vecB → cosine(vecA, vecB)
              两个句子独立编码，没有交互 → 快但信息不足

Cross-Encoder: [CLS] 句子 A [SEP] 句子 B → [BERT] → 相关性分数
               两个句子一起编码，有交叉注意力 → 慢但更准

Cross-Encoder 为什么更准：
  "苹果公司股价" 和 "苹果营养成分"
  Bi-Encoder: 两者都和"苹果"接近 → 分不清
  Cross-Encoder: 看到完整的上下文 → "公司股价" vs "营养成分" → 分得清
```

**底层：两个函数签名的差别就是一切**

把两者写成代码，差别一眼看穿——一个只吃 doc、另一个吃 (query, doc)：

```java
// Bi-Encoder：参数里没有 query → doc 的向量可以离线算好、存库、建 ANN 索引
float[] biEncoder.encode(String text);              // 输出：一个向量(如 768 维)

// Cross-Encoder：参数里 query 和 doc 同时在 → 没法离线预算，查询时每对现跑
float crossEncoder.score(String query, String doc); // 输出：一个分数(0~1 相关度)
```

完整流程串起来，粗排/精排各干嘛就清楚了：

```java
// 离线一次：把每篇 doc 编码成固定向量
for (Doc d : allDocs) index.put(d.id, biEncoder.encode(d.text));

// 在线每次查询：
float[] q = biEncoder.encode(query);              // query 只编码一次
List<Doc> cand = index.topK(q, 100);              // 粗排：和库向量比 cosine / 走 ANN，每篇成本=一次点积
for (Doc d : cand)
    d.score = crossEncoder.score(query, d.text);  // 精排：100 个 (q,doc) 对，各跑一次完整模型
return topK(cand, 10);
```

注意 `biEncoder.encode(d.text)` 的参数里**没有 query** → 可以离线算好；`crossEncoder.score(query, d.text)` **两个都在** → 只能查询时现算。**这一个差别，就是两段式的全部理由。**

**「交叉注意力」到底买到了什么**（30 秒补一下）：Transformer 里的注意力 = 每个 token 能去「看」输入里所有别的 token、按相关度把信息拿过来。

- **Bi-Encoder**：query 和 doc 关在**两次独立的前向**里跑，doc 的 token 只看得到 doc 自己的 token。query 和 doc 在进 cosine 之前**从没见过面**——它俩各自被压成一个向量，再比一下夹角，细粒度信息早被压没了。
- **Cross-Encoder**：query 和 doc **拼成一条序列**喂进去，于是 query 的每个词能**直接 attend 到 doc 的每个词**，算出「query 第 i 词 配不配 doc 第 j 词」这种细粒度匹配。这就是「交叉」——两段文本在同一次前向里互相看。「苹果公司股价 vs 苹果营养成分」能分清，靠的就是「公司」这个词能跨过边界去看 doc 里写的是「股价」还是「营养」。

**成本为什么逼你分两段**（拿 100 万 doc 算）：

| | Bi-Encoder | Cross-Encoder |
|---|---|---|
| 离线 | 编码 100 万次（建库时一次性） | 0（doc 表示依赖 query，没法预算） |
| 每次查询 | 编码 query 1 次 + 比向量（ANN ≈ log N） | 要算全库就得跑 100 万次前向 → 不可能 |
| 所以 | 粗排海捞 top-100 | 只喂它这 100 个 → 跑 100 次前向，几十毫秒搞定 |

> 🔑 一句话：**Bi-Encoder 的 doc 编码不含 query，所以能预算 + 建索引、能 scale 但糙；Cross-Encoder 必须 query+doc 一起进，所以准但每对现跑、跑不起全库。** 两段式不是为了好看，是被这个「可预算 / 不可预算」的鸿沟逼出来的。

## 6. Hybrid Search 混合检索

> 流水线位置：**① 粗筛的融合层**。让 BM25（关键词）和 Dense（语义）两路一起召回，再把排名合并。

### 6.1 为什么混合？

```
BM25 强在：精确关键词匹配
  "Python 3.12 release date" → BM25 精确找到含这些词的文档

Dense 强在：语义理解
  "最新版 Python 什么时候发布" → Dense 理解语义找到相关文档

两者互补 → 混合检索效果 > 任何单独方案
```

这就是为什么 §2 结尾说"BM25 仍然重要"—— 它补的正是 Dense 漏关键词的短板。

### 6.2 RRF (Reciprocal Rank Fusion) ⭐

```
融合多个检索结果排列的方法：

RRF_score(d) = Σ 1/(k + rank_i(d))

k = 常数（通常 60）
rank_i(d) = 文档 d 在第 i 个检索系统中的排名

例：文档 X 在 BM25 排第 3，在 Dense 排第 5
  RRF = 1/(60+3) + 1/(60+5) = 0.0159 + 0.0154 = 0.0313

好处：
  - 不需要分数归一化（只用排名）← BM25 分数和余弦分数量纲不同，没法直接加
  - 对异常值鲁棒
  - 实现简单
```

## 7. 高级检索技术

> 这些是在基础流水线之上的增强，面试能说出 HyDE 和 Query Expansion 即可。

### 7.1 HyDE (Hypothetical Document Embeddings)

```
问题：query 通常很短，Embedding 信息量不够
解决：让 LLM 先生成一个假设性的答案文档 → 用这个文档的 Embedding 去检索

流程：
  Query: "什么是 HNSW？"
    ↓
  LLM 生成假设答案: "HNSW 是一种分层导航小世界图的近似最近邻算法..."
    ↓
  用假设答案的 Embedding 去检索 → 找到真正相关的文档

直觉：把 query 扩展成和文档类似的形式 → query 和文档在同一语义空间
```

### 7.2 Query Expansion / Rewriting

```
原始 query: "LLM 训练"
  ↓ LLM 改写
改写 queries:
  - "大语言模型的预训练流程"
  - "LLM 的 SFT 和 RLHF 训练过程"
  - "如何训练大模型"
  ↓
用多个改写 query 检索 → 合并结果 → 覆盖更全面
```

### 7.3 检索评估指标 ⭐

| 指标 | 含义 | 用途 |
|------|------|------|
| **Recall@K** | top-K 中含正确答案的比例 | 初检阶段衡量 |
| **MRR** (Mean Reciprocal Rank) | 正确答案首次出现位置的倒数 | 排序质量 |
| **NDCG@K** | 考虑位置加权的排序质量 | 综合评估 |
| **Hit Rate@K** | top-K 中至少有 1 个正确结果 | 简单判断 |

> 怎么对应流水线：**粗筛**阶段看 Recall@K（捞得全不全），**精排**阶段看 MRR / NDCG（排得准不准）。

## 8. 工程实现：这条流水线谁帮你做了

> 理论讲完了。真到落地，上面每一步几乎都不用你手写 —— 但要先分清「谁负责哪一段」，否则容易以为「装个 LlamaIndex 就全包了」。这一节把前 7 节的流水线对上现实里的工具。

### 8.1 三层分工 ⭐ 先记这张

落地时有**三层**在分工，叠在一起就是你前面那条流水线：

```
你前面读的流水线          谁帮你做了                  属于哪一层
────────────────────────────────────────────────────────────
建索引 HNSW/IVF/PQ  ┐
BM25 倒排索引        ├─→ 向量数据库 / 搜索引擎        ① 存储·索引层
ANN 召回             ┘   (Qdrant/Milvus/pgvector/ES)
Hybrid + RRF 融合   ──→  越来越多也沉到 ① 里了        ①（新趋势）
────────────────────────────────────────────────────────────
切块 / 编排 / HyDE   ┐
query 改写 / 串流程   ├─→ LangChain / LlamaIndex       ② 编排·框架层
两阶段串起来          ┘   / Haystack
────────────────────────────────────────────────────────────
Embedding 编码      ┐
Cross-Encoder 重排   ├─→ 模型                         ③ 模型层
                     ┘   (bge / sentence-transformers / Cohere)
```

> 🔑 **最容易搞混的点**：HNSW、IVF、PQ、BM25、甚至 Hybrid+RRF，**不在框架层**，它们是**向量库 / 搜索引擎本身**内建的能力。框架（LlamaIndex/LangChain）不自己实现 HNSW，它只是去**调**底下那个库。所以「方便检索」的第一功臣其实是 ①，不是 ②。

### 8.2 流水线每一步，现在谁干

| 流水线步骤 | 谁帮你做 | 你只需要 |
|---|---|---|
| ANN（HNSW/IVF/PQ） | 向量库（Qdrant/Milvus/pgvector） | 选索引类型 + 调 `ef_search`/`nprobe` |
| BM25 / 倒排 | ES/OpenSearch，或向量库的稀疏向量 | 配字段、分词器 |
| **Hybrid + RRF** | Qdrant/Weaviate/pgvector/ES 8.x **已内建** | 一个 query 跑两路，DB 帮你 RRF |
| Embedding 编码 | bge / sentence-transformers / OpenAI | 选模型，调 `encode` |
| Rerank | bge-reranker（本地）/ Cohere（API） | 一次调用，传 query + 候选 |
| 切块 / HyDE / query 改写 / 串流程 | LangChain / LlamaIndex | 拼积木 |

### 8.3 要不要上框架？LangChain vs LlamaIndex

| | 强在哪 | 定位 |
|---|---|---|
| **LlamaIndex** | 检索/索引/RAG 开箱即用，抽象最贴 RAG | 「快速搭一个 RAG 查询」 |
| **LangChain + LangGraph** | Agent 编排、工具调用、状态图 | 「agentic、多步、有控制流」 |

- 两者高度重叠，且**可以混用** —— LlamaIndex 的 retriever 能当一个工具塞进 LangGraph 的节点里。
- 纯 RAG 查询、想最快出活 → **LlamaIndex**；要做 agent（多步推理、工具调用、supervisor）→ **LangChain + LangGraph**。
- 也可以**不上框架**：召回交给向量库、rerank 调模型，自己写几十行胶水。生产里这反而常见 —— 框架的抽象有时比业务还重。

LlamaIndex 把流水线拆成几个可拼积木，几乎和你前面读的逐一对应：

| LlamaIndex 积木 | 对应流水线 |
|---|---|
| `Retriever`（向量 / BM25 / 混合） | ① 粗筛召回 |
| `NodePostprocessor` | ② rerank 挂在这里 |
| `QueryTransform` / HyDE | §7 query 改写、HyDE |
| `QueryEngine` | 把召回 + 合成 + 塞 prompt 串起来 |

### 8.4 资深视角：方便 ≠ 不用懂

1. **框架的活在变少。** Hybrid+RRF、ANN 正在往向量库里沉。更漂亮的系统设计答法：「**检索重活交给向量库（hybrid + ANN 内建），rerank 用独立模型，框架只在需要编排 / agent 时才上**」—— 显示你知道边界在哪。
2. **框架会藏旋钮。** 真正决定效果的是 `ef_search`/`nprobe`/chunk 大小/rerank top-k，框架的默认值经常是糊的。出问题你得能掀开框架去调底层库。
3. **越方便 = 越黑盒。** 一行 `index.as_query_engine()` 背后把「切块 → 编码 → 建 HNSW → 召回 → rerank」全跑了。出了 bad case，你要能定位是哪一段崩的（召回没捞到？还是 rerank 排错？）—— 这正是这篇文档的价值：**框架让你不用手写，但 debug 还是得懂这条流水线。**

> 🔑 **一句话**：检索的「重活」沉在 ① 向量库、③ 模型里；② 框架只是把它们串起来的胶水。会用框架是 60 分；知道框架在调什么、旋钮在哪、出事掀开哪一层，才是资深。

## 9. 面试常问

> 这一节是**复习自检层**——新知识都在前面正文里讲过了，这里只是检查你能不能用自己的话讲出来。

### Q0: 完整描述一下 RAG 的检索流水线？

**答**：
- **离线**：文档切块 → Embedding 编码 → 建索引（向量索引 + 倒排索引）
- **在线**：query → ① 粗筛召回 top-100（BM25 关键词 + Dense 语义，Hybrid 用 RRF 融合）→ ② Cross-Encoder 重排出 top-10 → ③ 塞进 prompt 交给 LLM
- 一句话：**先用便宜的索引粗筛，再用贵的 reranker 精排**

### Q1: BM25 和向量检索的区别？什么时候用哪个？

**答**：
- **BM25**：基于关键词匹配（词频 + 逆文档频率），不理解语义
- **向量检索**：基于语义相似度（Embedding 余弦距离），理解同义词
- 精确关键词场景用 BM25，语义理解场景用向量检索
- **最佳实践：Hybrid Search（RRF 融合两者）**

### Q2: HNSW 是怎么工作的？为什么快？

**答**：
- 构建多层图：顶层稀疏（快速导航），底层密集（精确搜索）
- 查询从顶层开始贪婪搜索，逐层下降（就是跳表的多层思想搬到图上）
- 复杂度约 O(log N) vs 暴力 O(N)
- 代价：构建索引需要时间，且全图常驻内存、占用大

### Q3: HNSW、IVF、PQ 怎么选？

**答**：
- **HNSW**：多层图，效果最好但内存占用大 → 百万~千万级、内存够时的默认选项
- **IVF**：聚类分桶，查询只搜最近的几个桶（nprobe 控制）→ 缩小搜索范围
- **PQ**：向量量化压缩（切段 + 码本编号）→ 省内存但精度降
- **IVF-PQ**：分桶 + 压缩，十亿级 + 内存受限场景的标配

### Q4: 为什么需要两阶段检索（初检 + 重排）？

**答**：
- **初检 (Bi-Encoder)**：速度快（毫秒级）但不够精确 → 检出候选集
- **重排 (Cross-Encoder)**：精度高但速度慢 → 只对候选集重排
- Cross-Encoder 看到 query 和 doc 的交叉信息，比独立编码更准
- 典型流程：top-100 初检 → top-10 重排

### Q5: Hybrid Search 怎么融合 BM25 和向量检索结果？

**答**：
- 最常用的是 **RRF (Reciprocal Rank Fusion)**
- 只用排名（不用分数）→ 不需要归一化（BM25 分和余弦分量纲不同，没法直接加）
- 每个文档的 RRF 分数 = Σ 1/(k + rank_i)
- 简单有效，是工业界标准

### Q6: 什么是 HyDE？为什么有效？

**答**：让 LLM 先生成假设性的答案文档，用这个文档的 Embedding 去检索。有效是因为假设文档和真实文档在语义上更接近（而不是和简短 query 接近），解决了 query 和 document 之间的语义鸿沟问题。

### Q7: 这套检索流水线，工程上你会怎么落地？各部分用什么？

**答**：分三层，别混在一起：
- **召回 + ANN（HNSW/IVF/PQ）+ BM25 + Hybrid/RRF** → 交给**向量库 / 搜索引擎**（Qdrant/Milvus/pgvector/ES），这些是它内建能力，不自己写
- **Embedding 和 Cross-Encoder rerank** → 独立**模型**（bge / sentence-transformers / Cohere）
- **编排、切块、HyDE、query 改写** → **框架**（LlamaIndex 偏纯 RAG，LangChain+LangGraph 偏 agent），简单场景甚至不用框架
- 一句话：**重活在向量库和模型里，框架只是胶水**；会用框架是基础，知道旋钮在哪、出事能掀开底层，才是资深

## 📖 推荐学习路径

1. **先建骨架**：把 §1 的流水线总图和翻译表记牢，这是后面一切的地图
2. **重点理解**：BM25 vs Dense → Hybrid Search → Reranking 的完整流程
3. **HNSW 原理**是面试常问，要能讲清"多层图 + 贪婪搜索 + 逐层细化"
4. **IVF / PQ** 先记住"分桶缩范围 / 压缩省内存"两句定位；想答出深度，就看 §4.4–4.5 的底层小节（IVF = 一个 HashMap、挑几个桶来扫；PQ = 把算距离摊薄成查表）
5. 检索评估指标至少知道 Recall@K 和 MRR
6. **落地选型**（§8）：记住「重活在向量库 + 模型，框架只是胶水」，能说清 LlamaIndex / LangChain 各自的定位
