## 1. 演进全景

```
One-Hot → BoW/TF-IDF → Word2Vec/GloVe → ELMo → BERT/GPT
  稀疏      统计         静态向量        上下文    预训练

每一步解决什么问题：
One-Hot: 能表示词了 → 但维度太高、无语义
BoW:     能表示文档了 → 但忽略词序
TF-IDF:  考虑重要性了 → 但仍然稀疏、无语义
Word2Vec: 有语义了！  → 但"bank"在所有句子里向量一样（静态）
ELMo:    上下文相关了 → 但基于 LSTM，效果有限
BERT/GPT: Transformer + 大规模预训练 → 当前标准
```

## 2. One-Hot 编码

### 2.1 方法

```
词表 = ["猫", "狗", "鱼", "鸟"]

"猫" = [1, 0, 0, 0]
"狗" = [0, 1, 0, 0]
"鱼" = [0, 0, 1, 0]
"鸟" = [0, 0, 0, 1]
```

### 2.2 问题

| 问题 | 说明 |
|------|------|
| 维度爆炸 | 词表 50000 → 每个词是 50000 维向量 |
| 无语义 | "猫"和"狗"的距离 = "猫"和"量子力学"的距离（都正交）|
| 稀疏 | 几乎全是 0 |

## 3. BoW 和 TF-IDF

### 3.1 Bag of Words (BoW)

```
"我 爱 机器 学习 我 爱"  → {"我":2, "爱":2, "机器":1, "学习":1}

文档 → 词频向量 → 可以计算文档相似度
```

**问题**：
- 忽略词序（"狗咬人" = "人咬狗"）
- 常见词（"的"、"是"）权重太高

### 3.2 TF-IDF ⭐

```
TF (词频): 这个词在本文档中出多频繁
IDF (逆文档频率): 这个词在多少文档中出现过

TF-IDF = TF × IDF = 词频 × log(总文档数 / 包含该词的文档数)

"的" → TF 高但 IDF 低（每个文档都有）→ TF-IDF 低
"量子" → TF 适中但 IDF 高（罕见）→ TF-IDF 高
```

**直觉**：在本文中常见 + 在其他文中罕见 = 这个词对本文很重要。

> 🔑 **ML 关联**：
> - BM25（稀疏检索）就是 TF-IDF 的改进版
> - 至今仍在 RAG 的 Hybrid Search 中使用

## 4. Word2Vec

### 4.1 核心突破

**把词映射到低维稠密向量空间，语义相近的词向量距离近。**

```
One-Hot:  "猫" = [1, 0, 0, ..., 0]  (50000维, 稀疏)
Word2Vec: "猫" = [0.3, -0.1, 0.8, ..., 0.2]  (300维, 稠密)

且：vec("国王") - vec("男") + vec("女") ≈ vec("女王")  ← 向量算术！
```

### 4.2 两种训练方式

#### CBOW (Continuous Bag of Words)

```
用上下文预测中心词：
  输入: "今天 天气 [?] 好"
  预测: "真"
```

#### Skip-gram ⭐ 更常用

```
用中心词预测上下文：
  输入: "真"
  预测: "今天", "天气", "好"
```

### 4.3 负采样 (Negative Sampling)

```
问题：词表 50000，每次 softmax 太慢（O(V)）
解决：只对少量负样本计算，而不是全词表

正样本：("天气", "真") → 真的上下文
负样本：("天气", "量子"), ("天气", "披萨") → 随机词

训练目标：正样本分数高，负样本分数低
```

> 🔑 **思想关联**：负采样 = 对比学习的早期形式！和 InfoNCE Loss 思路一致。

### 4.4 Word2Vec 的局限

```
"bank" 在 "river bank"（河岸）和 "bank account"（银行）中
→ 向量完全一样！

Word2Vec 是静态的：一个词只有一个向量，不管上下文
```

## 5. GloVe

### 5.1 和 Word2Vec 的区别

```
Word2Vec: 局部上下文窗口（预测任务）
GloVe:    全局词共现矩阵（矩阵分解）

GloVe = Global Vectors
基于词和词在整个语料中共同出现的频率来训练
```

### 5.2 实际差异

- 效果与 Word2Vec 接近
- 都是静态向量
- 现在都被上下文表示（BERT）取代

## 6. 上下文表示 ELMo → BERT

### 6.1 ELMo (2018)

```
用双向 LSTM 生成词的上下文表示：

"I went to the bank to deposit money"  → "bank" = 银行向量
"I sat by the river bank"              → "bank" = 河岸向量

突破：同一个词在不同上下文有不同向量！
局限：基于 LSTM，不够强
```

### 6.2 BERT (2018) ⭐

```
用 Transformer Encoder 生成上下文表示：
  - 双向：同时看左边和右边
  - 预训练任务：MLM（完形填空）+ NSP
  - 生成的表示比 ELMo 好得多

这是"预训练语言模型"时代的开始
→ 阶段 5 会详细展开 BERT 和 GPT
```

### 6.3 静态 vs 上下文表示总结

| | 静态表示 | 上下文表示 |
|--|---------|-----------|
| 代表 | Word2Vec, GloVe | ELMo, BERT, GPT |
| 同一词不同义 | ❌ 同一向量 | ✅ 不同向量 |
| 训练方式 | 浅层（1-2层）| 深层（12-48层）|
| 效果 | 基础 | ⭐ 大幅提升 |
| 是否需要预训练 | 单独训练后冻结 | 预训练+微调 |

## 7. Tokenization 分词

### 7.1 为什么需要分词？

```
模型不能直接处理文本字符串 → 需要切成 token → 转成数字 ID

"I love machine learning"
  → ["I", "love", "machine", "learning"]  ← 词级分词
  → [32, 156, 4523, 7891]                ← token ID
  → Embedding 矩阵查找 → 向量
```

### 7.2 分词粒度

```
字符级: "hello" → ["h", "e", "l", "l", "o"]
  ✅ 没有 OOV    ❌ 序列太长，语义太碎

词级:   "hello" → ["hello"]
  ✅ 语义完整    ❌ OOV 问题（没见过的词怎么办？）

子词级: "unhappiness" → ["un", "happiness"]  ⭐ 现代标准
  ✅ 语义有意义   ✅ 几乎没有 OOV
```

### 7.3 四种子词分词方法 ⭐

| 方法 | 思路 | 用在 |
|------|------|------|
| **BPE** (Byte Pair Encoding) | 从字符开始，合并最高频的相邻对 | GPT 系列 |
| **WordPiece** | 类似 BPE，用似然而非频率选合并 | BERT |
| **SentencePiece** | 语言无关（不依赖空格分词）| T5, LLaMA |
| **Unigram** | 从大词表开始，剪枝低概率的子词 | T5 (结合 SentencePiece) |

### 7.4 BPE 详解

```
初始: "low" = ['l', 'o', 'w']

Step 1: 统计字符对频率 → ('l','o') 最频繁 → 合并为 'lo'
Step 2: 统计 → ('lo','w') 最频繁 → 合并为 'low'
Step 3: 继续直到达到目标词表大小...

例: "unhappiness" → "un" + "happiness" (如果这两个子词在词表中)
```

### 7.5 词表大小权衡

```
太小（如 1000）:
  ✅ 高频token覆盖好
  ❌ 罕见词被拆成很多碎片 → 序列过长

太大（如 500000）:
  ✅ 大多数词是完整的
  ❌ Embedding 矩阵巨大 → 参数浪费
  ❌ 罕见 token 训练数据不足

常见大小：
  GPT-2:    50257
  LLaMA:    32000
  GPT-4:    100000+
```

> 🔑 **面试关联**：分词方式影响模型能力（编程、多语言等）。LLaMA 对中文分词效果差因为训练时中文数据少，词表中中文子词少。

## 8. 面试常问

### Q1: Word2Vec 和 BERT Embedding 的区别？

**答**：
- **Word2Vec**：静态向量，一词一向量，不考虑上下文
- **BERT**：上下文相关，同一词在不同句子有不同向量
- BERT 用 12 层 Transformer 编码，表示能力远超 Word2Vec
- Word2Vec 是 2013 年的技术，BERT 是 2018 年

### Q2: BPE 分词是怎么工作的？

**答**：从字符级词表开始，反复统计最高频的相邻 token 对并合并，直到词表达到目标大小。好处是任何词都能表示（最差回退到字符级），常见词作为完整 token。

### Q3: TF-IDF 和 Embedding 的区别？

**答**：
| | TF-IDF | Embedding |
|--|--------|-----------|
| 维度 | 高（=词表大小）| 低（768/1024）|
| 稀疏/稠密 | 稀疏 | 稠密 |
| 语义 | 关键词匹配 | 语义理解 |
| "猫"≈"小猫" | ❌ 不同维度 | ✅ 向量接近 |
| 用途 | BM25 检索 | 语义检索(RAG) |

### Q4: 为什么 RAG 需要 Hybrid Search（BM25 + Embedding）？

**答**：
- BM25 擅长精确关键词匹配（搜「Python 3.12 release date」）
- Embedding 擅长语义理解（搜「最新版 Python 什么时候出的」）
- 纯 Embedding 可能遗漏含有确切关键词的文档
- 纯 BM25 不理解同义词和语义
- 混合两者 → 互补优势
