# 7.5 Compound AI Systems（复合 AI 系统）

> **一句话定位**：LLM 不是单独使用的——生产系统中它和检索器、分类器、验证器等组件协作，组合成复合系统。理解这种设计思想是你应用经验的理论升级。

---

## 1. 什么是 Compound AI System？

```
单体 LLM 调用：
  用户输入 → LLM → 输出
  问题：幻觉、知识过时、推理能力有限、不可控

Compound AI System：
  用户输入 → [路由 → 检索 → LLM → 验证 → 后处理] → 输出

  不是一个模型解决所有问题
  而是多个组件各司其职，协作完成任务

你已经在做的 RAG 就是一个典型的 Compound AI System：
  Query → Embedding → 向量检索 → Rerank → LLM 生成 → 输出
  每个环节是独立组件，可以单独优化
```

## 2. 核心组件

```
┌─────────────────────────────────────────────┐
│              Compound AI System              │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐  │
│  │  Router   │  │Retriever │  │   LLM     │  │
│  │ (路由器)  │→│ (检索器)  │→│ (生成器)   │  │
│  └──────────┘  └──────────┘  └───────────┘  │
│       ↑              ↑              ↓        │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐  │
│  │Classifier│  │  Cache   │  │ Verifier  │  │
│  │ (分类器) │  │ (缓存层) │  │ (验证器)  │  │
│  └──────────┘  └──────────┘  └───────────┘  │
│                      ↓                       │
│              ┌───────────┐                   │
│              │Guardrails │                   │
│              │ (安全护栏) │                   │
│              └───────────┘                   │
└─────────────────────────────────────────────┘
```

| 组件 | 功能 | 实现 |
|------|------|------|
| **Router** | 判断用什么模型/策略处理 | 分类器 / LLM / 规则 |
| **Retriever** | 获取相关知识 | 向量检索 / BM25 / 混合 |
| **Generator** | 生成回答 | LLM (大/小模型可切换) |
| **Verifier** | 验证输出正确性 | LLM-as-Judge / 规则 / 代码执行 |
| **Guardrails** | 安全过滤 | 输入/输出检测模型 |
| **Cache** | 缓存重复请求 | 语义缓存 / 精确匹配 |

## 3. 设计模式

### 3.1 Router Pattern（路由模式）

```
思想：不同的请求用不同的处理策略

用户请求
  ↓
Router（分类器）
  ├── 简单问答 → 小模型 / 缓存 → 快速响应
  ├── 知识问答 → RAG Pipeline → 检索+生成
  ├── 复杂推理 → 大模型 + Long CoT → 高质量响应
  └── 代码生成 → 代码专用模型 → 专业输出

实现方式：
  - 意图分类器（BERT/小模型，延迟 <10ms）
  - LLM 路由（让 LLM 判断，更灵活但更慢）
  - 规则路由（关键词匹配，最快但最粗糙）

价值：
  - 降低成本（80% 简单请求用小模型）
  - 提高质量（复杂请求用大模型）
  - 降低延迟（简单请求快速响应）
```

### 3.2 Fallback Pattern（兜底模式）

```
思想：检测失败 → 自动切换备选方案

尝试方案A
  ↓ 成功 → 返回结果
  ↓ 失败/低置信度
切换方案B
  ↓ 成功 → 返回结果
  ↓ 失败
兜底方案（人工/模板/拒绝回答）

例：
  1. 先用小模型回答
  2. 验证器判断回答质量 < 阈值
  3. 切换到大模型重新回答
  4. 仍不满意 → 返回"我不确定，建议人工确认"
```

### 3.3 Verify-then-Output（先验后输出）

```
思想：LLM 的输出经过验证才返回给用户

LLM 生成回答
  ↓
验证环节：
  1. 事实性验证：检索知识库，确认关键事实
  2. 安全验证：检测有害/偏见内容
  3. 格式验证：确保输出符合 JSON Schema / 业务规则
  4. 代码验证：生成的代码实际执行看是否报错
  ↓ 全部通过
返回给用户

应用：
  - 医疗问答：必须验证药品名称和剂量
  - 金融报告：必须验证数据准确性
  - 代码生成：必须通过测试用例
```

### 3.4 Ensemble Pattern（集成模式）

```
思想：多个模型/方法的结果综合

方式 1: 多模型投票
  Model A → Answer A ─┐
  Model B → Answer B ──→ 投票/综合 → 最终答案
  Model C → Answer C ─┘

方式 2: 多路径 RAG
  Query → BM25 检索 ─────→ 合并去重 → Rerank → LLM
  Query → Dense 检索 ────→
  Query → Knowledge Graph →

方式 3: 自一致性 (Self-Consistency)
  同一模型多次采样 → 多数投票
  （这就是 Test-time Compute Scaling 的一种）
```

## 4. 系统设计中的关键决策 ⭐

### 4.1 何时用 RAG vs 微调 vs Prompt

```
                成本    延迟    知识更新   准确性
RAG            中等     较高     实时       高（有来源）
微调            高      低      需重训      中等
Prompt         低      中      受限窗口    低

决策框架：
  知识频繁更新？ → RAG
  需要改变模型行为风格？ → 微调
  一次性任务？ → Prompt Engineering
  以上都需要？ → 组合使用（微调 + RAG 是常见方案）
```

### 4.2 模型选择策略

```
不是所有任务都需要最强模型：

GPT-4 / Claude Opus：
  ← 复杂推理、创意写作、系统设计

GPT-4o-mini / Claude Haiku：
  ← 分类、提取、简单问答、路由判断

专用模型（代码、Embedding）：
  ← 特定领域表现更好

端侧模型（Phi-3, Gemma-2）：
  ← 隐私敏感 + 低延迟 + 离线

规则/正则/传统ML：
  ← 能用规则解决的不要用 LLM（如格式校验、关键词匹配）
```

### 4.3 观测与评估

```
生产系统需要 Observability（可观测性）：

1. 链路追踪
   每个请求经过 Router → Retriever → LLM → Verifier
   记录每个环节的输入/输出/延迟/token 数

2. 质量监控
   LLM-as-Judge 定期抽检输出质量
   用户反馈（👍/👎）作为信号

3. 成本追踪
   每个请求的 token 消耗 + API 成本
   按路由路径统计（大模型 vs 小模型的比例）

4. 报警
   幻觉率上升 / 延迟突增 / 错误率异常

工具：LangSmith, Langfuse, Phoenix, Helicone
```

## 5. 与你的实战经验连接

```
你的 RAG 经验                    Compound AI 视角
─────────────                   ────────────────
向量检索 + Rerank               Retriever 组件
LLM 生成回答                    Generator 组件
Chunk 策略优化                  数据预处理 Pipeline
Prompt Template                 Orchestration 层
评估 RAG 效果                   Observability 层

升级思路：
  1. 加 Router → 简单问题不走 RAG（省检索开销）
  2. 加 Verifier → 检查 LLM 是否忠实于检索结果
  3. 加 Fallback → 检索失败时切换策略
  4. 加 Cache → 高频问题缓存答案
```

## 6. 面试常问

### Q1: 什么是 Compound AI System？相比单独用 LLM 有什么优势？

**答**：Compound AI System 是多个 AI 组件（LLM、检索器、分类器、验证器等）协作的系统。优势：(1) 可以接入外部知识（RAG）减少幻觉 (2) 路由机制降低成本（简单请求用小模型）(3) 验证环节提高可靠性 (4) 每个组件可独立优化和替换。

### Q2: 你会怎么设计一个生产级的 LLM 应用系统？

**答**：核心组件：(1) Router 判断请求类型和复杂度 (2) 简单请求 → 缓存或小模型 (3) 知识类请求 → RAG Pipeline (4) 复杂推理 → 大模型 + CoT (5) 所有输出经过安全 Guardrails (6) 关键输出加 Verifier。配套 Observability：链路追踪、质量监控、成本追踪。

### Q3: RAG vs 微调，怎么选？

**答**：知识频繁更新用 RAG（实时检索）；需要改变模型行为风格用微调（如特定语气、格式）；一次性任务用 Prompt。实践中常组合使用：微调让模型更好地利用检索结果 + RAG 提供最新知识。选择依据：更新频率、延迟要求、成本预算。
