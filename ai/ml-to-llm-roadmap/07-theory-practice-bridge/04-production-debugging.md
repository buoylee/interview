> **一句话定位**：面试官问"你遇到过什么问题？怎么解决的？"——这份 checklist 让你的回答有条理、有深度、有说服力。

## 1. RAG 质量问题排查

### 1.1 系统化排查流程 ⭐

```
用户说"回答不对" → 先定位是哪个环节出了问题：

Step 1: 检查检索结果
  → 检索到了正确的文档吗？
  YES → 问题在生成（Step 3）
  NO  → 问题在检索（Step 2）

Step 2: 检索问题定位
  → Query 和文档有语义匹配吗？ → Embedding 问题
  → 文档存在于知识库中吗？     → 数据入库问题
  → 分块后相关内容被拆散了吗？ → 分块策略问题
  → Reranker 排序对吗？        → 重排问题

Step 3: 生成问题定位
  → 正确文档在 Context 中但 LLM 忽略了？ → Lost in the Middle
  → LLM 编造了不在 Context 中的信息？    → 幻觉，优化 Prompt
  → LLM 理解了但回答格式不对？           → 指令问题
```

### 1.2 常见问题 → 解决方案

| 症状 | 原因 | 解决方案 |
|------|------|---------|
| 搜不到相关文档 | Embedding 语义不匹配 | 换模型 / 领域微调 / 加 BM25 |
| 搜到了但排在后面 | 初筛不够准 | 加 Reranker / 调 top-K |
| 关键信息被拆散 | chunk 太小 | 增大 chunk / 加 overlap / 父子分块 |
| 正确文档在中间被忽略 | Lost in the Middle | 相关文档放开头 / 压缩上下文 |
| 模型编造内容 | 幻觉 | 降 Temperature / 强化「基于文档回答」指令 |
| 多个文档信息矛盾 | 数据质量 | 加时间戳优先新版 / 来源可信度排序 |

### 1.3 评估指标体系

```
离线评估（开发阶段）：
  - Recall@K: 前 K 个检索结果中包含正确答案的比例
  - MRR: 正确答案首次出现位置的倒数
  - Faithfulness: 回答是否忠实于检索内容（RAGAS）
  - Answer Relevancy: 回答是否回答了问题（RAGAS）

在线监控（生产阶段）：
  - 用户满意度（👍/👎 比率）
  - 平均延迟 (P50, P95, P99)
  - 引用命中率（回答引用了哪些文档）
  - 空回答率（没找到相关信息的比例）
  - LLM 调用成本 (tokens/请求)
```

## 2. 延迟问题排查

### 2.1 延迟拆解

```
总延迟 = Query处理 + 检索 + Reranking + LLM生成

典型耗时分布：
  Query 处理（Embedding）:  50-100ms
  向量检索:                 10-50ms
  BM25 检索:                5-20ms
  Reranker (100 docs):      200-500ms    ← 常见瓶颈
  LLM 生成:                 500-3000ms   ← 最大瓶颈（取决于输出长度）

总计: 约 1-4 秒
```

### 2.2 优化手段

| 瓶颈 | 优化 | 效果 |
|------|------|------|
| **LLM 延迟** | 流式输出 | 感知延迟 ↓ 80% |
| **LLM 延迟** | 语义缓存 | 命中时 ↓ 95% |
| **LLM 延迟** | 小模型 + 路由 | 简单问题 ↓ 70% |
| **Reranker** | 减少候选数（100→20） | ↓ 60-80% |
| **Reranker** | 用更小的 Reranker | ↓ 50% |
| **Embedding** | 批量处理 + GPU | ↓ 5-10x |
| **全链路** | 并行检索（BM25 ‖ Dense） | ↓ 30-50% |

## 3. 成本问题排查

### 3.1 成本拆解

```
RAG 系统月成本 (估算):
  LLM API 调用:     60-80% ← 主要成本
  Embedding 计算:    5-10%
  向量库存储/查询:   5-10%
  Reranker 推理:     5-10%
  基础设施:          5-10%
```

### 3.2 降本优先级

```
优先级 1（最高 ROI）:
  ✅ 语义缓存 → 减少 50-80% LLM 调用
  ✅ Prompt 精简 → 每减 100 token ≈ 省 $0.001/次 × 100万次 = $1000/月

优先级 2:
  ✅ 模型路由 → 简单问题用便宜模型
  ✅ max_tokens 控制 → 限制输出长度

优先级 3:
  ✅ 自部署开源模型 → 高流量时比 API 经济
  ✅ 批量处理 → 非实时任务用 batch API（便宜 50%）
```

## 4. 模型效果回退排查

### 4.1 "模型今天变笨了"

```
常见原因：
  1. API 模型版本更新（OpenAI 默认用 latest → 行为可能变化）
     → 固定模型版本号（如 gpt-4-0613）
  
  2. Prompt 被修改（团队成员改了 System Prompt）
     → Prompt 版本管理 + 变更审计
  
  3. 检索数据变化（新入库的文档质量差 / 旧文档被删）
     → 数据入库的质量检查 + 回滚机制
  
  4. 并发增加 → 超时重试 → 用了降级模型
     → 监控降级率

排查步骤：
  1. 检查用的模型版本是否变了
  2. 检查 Prompt 是否被修改（Git 记录）
  3. 检查近期入库的文档
  4. 复现问题 → 对比历史相同 Query 的回答
```

## 5. LLM 应用监控清单

```
必备监控项：

性能指标:
  □ P50 / P95 / P99 延迟
  □ 每分钟请求量 (RPM)
  □ 错误率 / 超时率
  □ 吞吐量 (tokens/sec)

质量指标:
  □ 用户反馈（👍/👎）
  □ 幻觉检测率
  □ 空回答率（检索失败）
  □ 引用准确率

成本指标:
  □ 每次请求的 token 消耗
  □ 日/月 API 调用成本
  □ 缓存命中率
  □ 各模型的使用占比

安全指标:
  □ Prompt Injection 检测率
  □ 敏感信息过滤次数
  □ 内容安全拦截次数
```

## 6. 面试示例

### Q: 你的 RAG 系统准确率只有 60%，怎么提升？

```
回答框架：

"首先我会定位问题在哪个环节——检索、排序还是生成。

方法是抽样 100 个 bad case：
  - 30% 是检索没找到正确文档 → 检索问题
  - 40% 是找到了但 LLM 忽略或编造 → 生成问题
  - 30% 是文档信息不完整 → 数据问题

针对检索问题：
  我会先加 Hybrid Search（当时只用了 Dense），BM25 能补上精确关键词匹配。
  然后评估分块策略——某些case是关键信息被切到两个chunk里。

针对生成问题：
  优化 Prompt，强化'只基于提供的文档回答'。
  加入 Reranker 提升 Context 中相关文档的排名。
  降低 Temperature 到 0.1。

这套优化让准确率从 60% 提升到了 85%。"
```
