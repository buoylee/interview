## 1. Prompt Engineering 的理论框架

### 1.1 为什么 Prompt 有效？

```
GPT-3 的核心发现：In-Context Learning (ICL)
  模型通过输入中的示例，隐式地"学会"当前任务
  不需要修改模型参数！

理论假说（回顾阶段 5）：
  1. 隐式梯度下降：Transformer 前向传播 ≈ 在示例上做了一步梯度更新
  2. 贝叶斯推断：示例帮助模型识别是"哪一类任务"
  3. 任务定位：从预训练知识中"检索"出对应能力

本质：Prompt 是和模型沟通的"接口"
  好的 Prompt = 让模型正确定位到要使用的能力
  差的 Prompt = 模型不知道你要什么 → 随机输出
```

## 2. 核心 Prompt 技术

### 2.1 Zero-shot / Few-shot

```
Zero-shot（0 示例）:
  "将以下文本分类为正面或负面: '这部电影太棒了'"

One-shot（1 示例）:
  "文本: '食物很难吃' → 负面
   文本: '这部电影太棒了' → "

Few-shot（多个示例）:
  "文本: '食物很难吃' → 负面
   文本: '服务态度很好' → 正面
   文本: '价格太贵了' → 负面
   文本: '这部电影太棒了' → "

一般规律：更多示例 → 效果更好（但有上限，通常 3-8 个足够）
```

### 2.2 Chain-of-Thought (CoT) ⭐⭐

```
直接回答:
  Q: "一家商店有 23 个苹果，卖了 15 个，又进了 6 个，有多少个？"
  A: "14" ← 直接给答案，经常出错

CoT:
  Q: "一家商店有 23 个苹果，卖了 15 个，又进了 6 个，有多少个？"
  A: "让我一步步思考：
      1. 开始有 23 个
      2. 卖了 15 个: 23 - 15 = 8 个
      3. 又进了 6 个: 8 + 6 = 14 个
      答案是 14。"

为什么 CoT 有效？
  1. 分解复杂问题 → 每步更简单
  2. 中间步骤提供了额外的"工作记忆"
  3. 自回归模型一步只能做有限计算 → 更多步骤 = 更多计算
  4. 本质上是 Test-time Compute Scaling
```

### 2.3 Zero-shot CoT

```
不需要提供推理示例，只加一句：

"让我们一步步思考。" (Let's think step by step.)

这个简单的魔法指令就能触发模型的推理能力！
原因：预训练数据中有大量"步骤化思考"的文本 → 这句话激活了这个模式
```

## 3. System Prompt 设计

### 3.1 结构化框架

```
一个好的 System Prompt 通常包含：

1. 角色定义：你是一个 [专业角色]
2. 能力边界：你可以/不可以做什么
3. 输出格式：用 [格式] 回答
4. 行为准则：回答时应该 [准则]
5. 上下文：你正在 [场景] 中工作
```

### 3.2 示例

```
你是一位资深后端工程师，专精于分布式系统设计。

规则：
- 回答时先给出结论，再解释原因
- 如果问题涉及你不确定的领域，明确说明
- 代码示例使用 Python，除非用户指定其他语言
- 考虑可扩展性、可维护性和性能

输出格式：
- 使用 Markdown 格式
- 代码块标注语言
- 关键决策用加粗标注
```

## 4. 高级 Prompt 技术

### 4.1 Self-Consistency

```
思路：多次推理 → 投票取最多的答案

1. 让模型用 CoT 推理同一问题 5 次（Temperature > 0）
2. 每次可能走不同的推理路径
3. 取出现最多的最终答案

类似 Ensemble（集成学习）的思想
效果：显著提升数学/逻辑问题的准确率
```

### 4.2 ReAct (回顾)

```
交替 思考(Thought) → 行动(Action) → 观察(Observation)
= CoT + Tool Use 的结合
详见 阶段 7 Agent 架构
```

### 4.3 Tree of Thoughts (ToT)

```
把推理过程看作搜索树：
  每步生成多个候选想法 → 评估每个 → 选择最好的继续
  = CoT + 树搜索

类似 Beam Search 但在推理层面
适用：需要探索和回溯的复杂问题
```

### 4.4 Prompt Chaining

```
复杂任务 → 分解为多步 Prompt：

Prompt 1: "分析这段代码的功能" → 输出 1
Prompt 2: "根据以上分析，找出潜在 bug" → 输出 2
Prompt 3: "根据以上 bug，生成修复方案" → 输出 3

每步的输出作为下一步的输入
= Plan-and-Execute Agent 的手动版
```

## 5. Prompt 与模型能力的关系

### 5.1 Prompt 不能创造能力

```
Prompt 只能激活模型已有的能力，不能凭空创造：

模型没有的能力 → 再好的 Prompt 也不行
  例：7B 模型做复杂数学 → Prompt 再好也难

模型有但没激活的能力 → 好的 Prompt 能激活
  例：70B 模型有推理能力 → CoT 激活它

模型误用的能力 → 好的 Prompt 能引导
  例：模型知道答案但格式不对 → 格式指令修正
```

### 5.2 模型大小与 Prompt 技术

```
小模型 (<7B):
  Few-shot 帮助大
  CoT 效果有限（推理能力弱）
  简单明确的指令最好

中等模型 (7B-70B):
  CoT 开始有效
  Few-shot 效果好
  System Prompt 能显著影响风格

大模型 (70B+):
  Zero-shot CoT 就很好
  复杂指令能遵循
  Self-Consistency 等高级技术有效
```

## 6. 面试常问

### Q1: CoT 为什么有效？

**答**：(1) 分解复杂问题为简单子步骤 (2) 中间步骤提供额外工作记忆（自回归模型一步计算有限）(3) 本质是 Test-time Compute Scaling（用更多 token 进行更多计算）。

### Q2: Few-shot 的示例数量和选择怎么影响效果？

**答**：通常 3-8 个示例即可。示例应该多样化（覆盖不同类型），格式一致，和目标任务相关。太少可能不够模型定位任务模式，太多可能占满上下文窗口。选择顺序也有影响（最相关的放最后效果通常更好）。

### Q3: System Prompt 和 User Prompt 有什么区别？

**答**：System Prompt 设定模型的角色和行为准则（持久性指令），User Prompt 是具体的用户请求。在训练中，System Prompt 是 SFT 数据格式的一部分。模型学会把 System 视为持久约束、User 视为即时请求。
