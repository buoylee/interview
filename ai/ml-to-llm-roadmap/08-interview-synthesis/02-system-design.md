## 1. 系统设计方法论

### 1.1 面试回答框架（4 步）

```
Step 1: 需求澄清（2-3 分钟）
  - 用户量级？QPS？
  - 延迟要求？（实时对话 <2s vs 异步报告 <30s）
  - 数据量？更新频率？
  - 准确性要求？（医疗 vs 闲聊）
  - 成本约束？

Step 2: 高层架构（5 分钟）
  - 画出核心组件和数据流
  - 先不纠结细节

Step 3: 深入关键组件（10-15 分钟）
  - 面试官感兴趣哪个就展开哪个
  - 每个组件说：方案选择 + 为什么 + 权衡

Step 4: 扩展性 & 监控（3-5 分钟）
  - 怎么扩展到 10x 流量？
  - 怎么监控和告警？
  - 故障降级策略？
```

## 2. RAG 系统设计

### 2.1 题目示例

> "设计一个企业知识库问答系统，支持 10 万篇文档，日均 50 万次查询，延迟 < 3 秒。"

### 2.2 高层架构

```
┌─────────────┐    ┌──────────────┐    ┌───────────────┐
│   用户请求   │───→│    API 网关    │───→│   查询处理层   │
└─────────────┘    │ (限流/认证)   │    │ (改写/路由)    │
                   └──────────────┘    └───────┬───────┘
                                               │
                          ┌────────────────────┼────────────────────┐
                          ↓                    ↓                    ↓
                   ┌────────────┐      ┌────────────┐      ┌────────────┐
                   │  BM25 检索  │      │ 向量检索    │      │ 语义缓存   │
                   │ (Elasticsearch)│   │ (Milvus)   │      │ (Redis)    │
                   └──────┬─────┘      └──────┬─────┘      └──────┬─────┘
                          └────────────────────┼───────────────────┘
                                               ↓
                                      ┌────────────────┐
                                      │  Reranker 重排  │
                                      │ (Cross-Encoder) │
                                      └────────┬───────┘
                                               ↓
                                      ┌────────────────┐
                                      │   LLM 生成      │
                                      │ (GPT-4 / 自部署)│
                                      └────────┬───────┘
                                               ↓
                                      ┌────────────────┐
                                      │  后处理 & 返回   │
                                      │ (引用/过滤/日志) │
                                      └────────────────┘
```

### 2.3 关键设计决策

#### 文档处理管线

```
文档入库（离线）：
  文档上传 → 格式解析(PDF/Word/HTML) → 清洗
    → 分块(512 tokens, overlap 50)
    → Embedding(BGE-large, batch 推理)
    → 写入向量库(Milvus) + 搜索引擎(ES)

设计要点：
  - 异步处理（消息队列 + Worker）
  - 增量更新（不重建全量索引）
  - 版本管理（文档更新时旧版本标记删除）
```

#### 查询处理

```
用户 Query 进来后：
  1. 语义缓存检查 → 命中直接返回（节省 70%+ 成本）
  2. Query 改写（LLM 改写为更好的检索 Query）
  3. 并行检索: BM25 + Dense → RRF 融合
  4. Reranker 精排 top-10
  5. Context 组装 + LLM 生成
  6. 后处理：引用标注、安全过滤
```

#### 语义缓存 ⭐ 成本杀手

```
相似问题不需要重新查询：
  "什么是 RAG？" 和 "RAG 是什么意思？" → 同一个答案

实现：
  Query → Embedding → 在缓存中搜相似 Query
  相似度 > 0.95 → 直接返回缓存的答案
  否则 → 正常流程 → 结果写入缓存

效果：减少 50-80% 的 LLM 调用（巨大的成本节省）
```

#### 扩展性设计

```
50 万 QPS 的考量：
  - API 网关：Nginx/Kong 负载均衡
  - 向量库：Milvus 分片（按文档集合分片）
  - LLM：多实例 + 请求排队 + 优先级
  - 缓存：Redis Cluster

故障降级：
  - LLM 不可用 → 直接返回检索结果 + "AI 暂时不可用"
  - 向量库不可用 → 降级为纯 BM25
  - 全部不可用 → 返回热门 FAQ
```

## 3. LLM 服务平台设计

### 3.1 题目示例

> "设计一个多租户 LLM 平台，支持多模型（GPT-4、Claude、自部署模型），需要 token 计费、限流、审计。"

### 3.2 核心组件

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ 多租户    │───→│  统一 API 层   │───→│  模型路由     │
│ (API Key) │    │ (格式转换)    │    │ (负载/策略)   │
└──────────┘    └──────────────┘    └──────┬───────┘
                                          │
                     ┌────────────────────┼────────────────────┐
                     ↓                    ↓                    ↓
              ┌────────────┐      ┌────────────┐      ┌────────────┐
              │  OpenAI     │      │  Claude     │      │  自部署     │
              │  Proxy      │      │  Proxy      │      │  vLLM      │
              └────────────┘      └────────────┘      └────────────┘

横切关注点：
  ├── Token 计量 & 计费
  ├── 限流 (per tenant / per model)
  ├── 审计日志（请求/响应记录）
  ├── 内容安全过滤（输入+输出）
  └── 监控告警
```

### 3.3 模型路由策略

```
简单路由：用户指定模型
智能路由：根据请求特征自动选择

  简单问题 → 小模型（快、便宜）
  复杂推理 → 大模型（准、贵）
  代码任务 → 代码模型

路由判断：
  - 基于 Query 复杂度（长度、关键词）
  - 基于用户套餐（免费用户 → 小模型）
  - 基于延迟要求（实时 → 已部署的模型）
```

## 4. Agent 系统设计

### 4.1 关键挑战

```
Agent 比 RAG 复杂得多：
  1. 多步执行 → 需要状态管理和容错
  2. 工具调用 → 需要权限控制和超时处理
  3. 成本不可控 → 可能无限循环
  4. 结果不确定 → 同样的输入可能不同的执行路径
```

### 4.2 设计要点

```
关键设计：
  - 最大步数限制（防无限循环，如 max_steps=15）
  - 每步 token 预算（防单步用太多 token）
  - 工具调用超时（每个工具 30s 超时）
  - 人工审核节点（高风险操作需要人工确认）
  - 执行日志（每步的 thought/action/observation 都记录）
  - 回滚机制（工具调用失败时的恢复策略）
```

## 5. 通用设计模式

### 5.1 LLM 应用的共性模式

| 模式 | 用途 | 为什么 |
|------|------|--------|
| **语义缓存** | 相似请求复用 | 降低 50-80% 成本 |
| **请求排队** | 控制并发 | LLM 推理是 GPU 密集型 |
| **流式输出** | 降低感知延迟 | 第一个 token 快速返回 |
| **降级策略** | 容错 | LLM API 可能不可用 |
| **输入/输出过滤** | 安全 | 防 prompt injection |
| **多级缓存** | 性能 | Prompt Cache → 语义缓存 → KV-Cache |
| **异步处理** | 长任务 | 报告生成等不需要实时 |
| **A/B 测试** | 迭代 | 比较不同 prompt/模型的效果 |

### 5.2 成本优化清单 ⭐

```
1. 语义缓存（最有效，减少重复 LLM 调用）
2. Prompt 精简（少一半 token = 少一半钱）
3. 模型路由（简单问题用便宜模型）
4. 批量处理（批量请求比逐个请求便宜）
5. 自部署开源模型（高流量时比 API 便宜）
6. Token 限制（max_tokens 控制输出长度）
```

## 6. 面试模拟

### 题目 1: 设计一个客服问答系统

```
面试官: 设计一个电商客服系统，需要回答商品问题、订单查询、退换货流程。

你的回答框架:

1. 澄清需求:
   - QPS? 高峰时多少？
   - 多少种商品？知识库多大？
   - 需要查询实时订单数据吗？
   - 延迟要求？成本预算？

2. 架构设计:
   - 意图识别（分流）→ FAQ 直接回答 / RAG 查知识库 / Agent 查订单
   - 知识库: 商品信息 + 政策文档 → RAG
   - 订单查询: Function Calling → 调用订单 API
   - 兜底: 转人工客服

3. 深入 RAG 部分:
   - 分块策略: 按商品/政策分块
   - Hybrid Search: 商品名精确匹配(BM25) + 语义(Dense)
   - Reranker: 提升准确率
   - 语义缓存: 高频问题命中率高

4. 扩展性:
   - 新增商品 → 增量索引
   - 流量激增(大促) → 自动扩容 + 降级策略
   - 监控: 回答准确率、用户满意度、延迟
```

### 题目 2: 你怎么优化现有 RAG 系统？

```
面试官: 你的 RAG 系统准确率只有 60%，怎么提升？

你的排查思路 (下一节详细展开):

1. 先定位问题在哪个环节:
   检索 → 排序 → 生成 → 哪个环节出了问题？

2. 检索阶段优化:
   - Embedding 模型换更好的？领域微调？
   - 分块策略调整？chunk size？overlap？
   - 加 Hybrid Search？
   
3. 排序阶段优化:
   - 加 Reranker？
   - 过滤低相关性文档？

4. 生成阶段优化:
   - 换更强的 LLM？
   - 优化 Prompt？
   - 降低 Temperature？
```
