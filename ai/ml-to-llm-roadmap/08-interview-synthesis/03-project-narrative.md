# 项目经验叙事模板（面试讲故事）⭐⭐

> **一句话定位**：技术能力相同的两个人，讲故事好的那个拿 offer。这份模板帮你把 RAG/Agent 经验讲出深度和影响力。

---

## 1. STAR 框架 + 技术深度

```
一个完整的项目经验应包含：

S (Situation)  → 业务背景 + 技术痛点（30 秒）
T (Task)       → 你的角色和目标 （15 秒）
A (Action)     → 你做了什么（技术方案 + 决策理由）（2-3 分钟）← 重点
R (Result)     → 量化成果 （30 秒）

总时长控制在 3-4 分钟，然后等面试官追问。
```

---

## 2. RAG 项目叙事模板 ⭐

### 2.1 完整示例

```
[S] 我们公司有一个企业知识库系统，员工需要在数千份内部文档中
    找到准确的信息回答客户问题。传统的关键词搜索准确率只有 40%，
    员工平均需要 15 分钟才能找到答案。

[T] 我负责设计和实现基于 RAG 的智能问答系统，目标是将准确率
    提升到 80% 以上，响应时间控制在 3 秒以内。

[A] 我的技术方案分为三个阶段：

    第一阶段：基础 RAG
    - 选择 BGE-large 做 Embedding，因为中文基准测试排名最高
    - 用 Milvus 做向量存储，HNSW 索引保证毫秒级检索
    - 分块策略：512 tokens + 50 overlap，通过实验发现这个配置
      在我们文档类型上 Recall@10 最高
    - 第一版准确率达到 65%

    第二阶段：检索优化
    - 分析了 100 个 bad case，发现 30% 是精确术语匹配失败
    - 加入 BM25 + RRF Hybrid Search → 准确率提升到 75%
    - 加入 BGE-Reranker → 提升到 82%
    
    第三阶段：生成和成本优化
    - 优化 Prompt（限制基于文档回答 + 引用来源）
    - 实现语义缓存（Redis + Embedding 相似搜索）→ 缓存命中率 60%
    - API 调用成本降低 55%

[R] 最终准确率 85%，平均延迟 2.1 秒，月成本控制在 $X 以内。
    员工找答案时间从 15 分钟降到 30 秒，Client NPS 提升 20 点。
```

### 2.2 面试官可能追问的点

```
Q: "Embedding 为什么选 BGE 不选 OpenAI？"
A: (体现决策能力)
   "对比了三个方案：OpenAI ada-002 效果好但有数据出境限制；
    E5-large 在通用 benchmark 上略好；BGE 在中文 benchmark 上
    排第一且支持本地部署。考虑到我们是中文场景 + 合规要求 → 选 BGE。"

Q: "分块大小为什么是 512？"
A: (体现实验方法论)
   "做了 A/B 实验：256/512/1024 三种 chunk size，
    在 200 个标注 query 上评估 Recall@10：
      256 tokens: Recall=0.72（太碎，语义不完整）
      512 tokens: Recall=0.84（语义完整 + 噪声可控）
      1024 tokens: Recall=0.79（噪声太多稀释了关键信息）
    512 + overlap 50 是最优配置。"

Q: "语义缓存怎么实现的？命中率怎么保证？"
A: (体现工程能力)
   "Query → Embedding → 在 Redis 中搜相似 Query（余弦 > 0.95）。
    命中 → 直接返回缓存的答案 + 标记 'from cache'。
    未命中 → 正常 RAG 流程 → 结果和 Query Embedding 一起写入缓存。
    TTL 设 24 小时，知识库更新时清空相关缓存。
    60% 命中率是因为企业场景有大量重复问题。"
```

## 3. Agent 项目叙事模板

```
[S] 客服场景中，很多问题需要查询实时数据（订单状态、库存、物流），
    纯 RAG 只能回答静态知识库的问题。

[T] 扩展系统为 Agent 模式，支持意图识别 → 工具调用 → 多步推理。

[A] 
    架构选择：
    - 用 LangGraph 构建状态机式 Agent（比纯 ReAct 更可控）
    - 定义工具集：订单查询 API、库存查询 API、退换货流程 API
    - 意图识别路由：简单知识问题 → RAG，需要工具的 → Agent

    关键设计：
    - 最大执行步数限制（15 步）→ 防止无限循环
    - 每个工具调用超时 10s → 防止 API 卡死
    - Fallback 策略：Agent 失败 → 降级为 RAG → 再失败 → 转人工
    
    挑战 & 解决：
    - Function Calling 参数解析错误率高 → 加了参数 validation + retry
    - 多步推理容易偏离 → 每步检查是否和原始意图一致

[R] 支持了 15 种工具调用，端到端成功率 92%，
    客服转人工率从 45% 降到 20%。
```

## 4. 讲故事的技巧

### 4.1 DO ✅

```
✅ 用数字说话 → "准确率从 60% 提升到 85%"
✅ 讲技术决策的原因 → "选 BGE 因为中文 benchmark 第一 + 支持本地部署"
✅ 讲遇到的问题和解决过程 → "发现 30% 的 bad case 是术语匹配问题"
✅ 主动暴露权衡 → "语义缓存提升了速度但需要处理缓存一致性"
✅ 保持简洁 → 先讲结论，面试官追问再展开
```

### 4.2 DON'T ❌

```
❌ 只说做了什么不说为什么 → "我用了 Milvus"（为什么不用 Pinecone？）
❌ 过度细节 → 面试官没问就别展开 HNSW 的数学原理
❌ 回避不确定的部分 → 不知道就说"这个我没深入研究过"
❌ 只讲成功 → 适当讲失败和迭代过程更真实
❌ 不量化 → "效果变好了"（好了多少？怎么衡量的？）
```

## 5. 面试高频追问清单

```
准备以下问题的答案（每个 30-60 秒）：

关于 RAG:
  □ 为什么选这个 Embedding 模型？对比了哪些？
  □ 分块策略怎么确定的？做了什么实验？
  □ Hybrid Search 的 RRF 参数怎么调的？  
  □ 检索准确率怎么评估？用了什么指标？
  □ 遇到过什么坑？怎么解决的？
  □ 如果重新做，你会怎么改进？

关于 Agent:
  □ 工具调用失败怎么处理？
  □ 怎么防止 Agent 无限循环？
  □ LangChain 和 LangGraph 的区别？为什么选这个？
  □ 多 Agent 之间怎么协作？
  □ 成本怎么控制？

关于工程:
  □ 系统日均处理多少请求？
  □ 延迟怎么优化的？
  □ 怎么监控系统健康？
  □ 怎么处理知识库更新？
  □ 遇到过线上故障吗？怎么处理？
```
