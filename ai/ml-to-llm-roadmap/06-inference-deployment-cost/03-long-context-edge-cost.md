# 长上下文、端侧部署与成本估算

## 这篇解决什么问题

很多 LLM 应用一开始会把问题简化成“上下文窗口越大越好”。但上线后你会遇到更实际的问题：输入变长后 TTFT 变高，答案反而更容易漏重点，账单上涨，端侧设备跑不动，失败重试又进一步放大成本。

这一篇解决的问题是：长上下文为什么同时是成本问题和质量问题，端侧部署为什么依赖小模型与量化，以及如何用一个简单公式拆解 LLM 应用的总成本。

## 概念为什么出现

长上下文解决的是“模型需要看到更多信息”的问题，例如长文档问答、多轮对话、代码仓库理解、复杂工具调用轨迹。但它也引入三个新问题：

```text
成本: 输入 token 更多，prefill 和 KV Cache 更贵
质量: 信息太多，模型可能注意不到关键证据
部署: 端侧设备的内存、算力和功耗无法承受大模型长上下文
```

所以长上下文不是简单追求 max tokens，而是要管理“哪些信息值得进入上下文、以什么形式进入、保留多久”。

## 最小心智模型

长上下文策略可以分成三类：

```text
扩展位置/训练能力: RoPE scaling、长上下文训练
控制长序列成本: sliding window/local attention
减少无关上下文: chunking、retrieval-assisted context、summarization、prompt compression
```

端侧部署则是在资源受限的机器上运行模型。它通常依赖更小模型、量化权重、紧凑文件格式和本地推理引擎。GGUF/llama.cpp 这类组合可以理解成“把模型变成更适合本地 CPU/GPU/NPU 加载和推理的形态”，核心目标是降低内存和部署门槛。

成本估算要把一次应用调用拆开，而不是只看模型单价：

```text
每次尝试成本 ≈ 输入 token 成本 + 输出 token 成本 + 检索/重排/工具调用成本
总成本 ≈ 每次尝试成本 × 期望尝试次数 + 固定部署/监控/人工复核成本
```

## 最小例子

假设你做一个合同审阅助手。用户上传 80 页合同，问题是：

```text
这份合同里自动续约条款的风险是什么？
```

粗暴做法是把整份合同塞进 prompt。这可能能跑，但会带来高输入成本、长 TTFT，并且模型可能在大量无关条款里漏掉关键句。

更可控的做法是：

```text
1. 先按章节或条款 chunking。
2. 用检索找出 renewal、termination、notice period 相关片段。
3. 对长片段做摘要或压缩，保留条款编号和原文证据。
4. 只把关键片段、问题和输出要求放进上下文。
5. 如果答案置信度不足，再追加检索或让用户缩小范围。
```

这不是否定长上下文，而是把长上下文当成稀缺资源使用。

## 原理层

RoPE scaling 试图让使用旋转位置编码的模型外推或适配更长位置范围。它解决的是位置表示范围问题，但不自动保证模型在很长上下文里稳定找到关键信息。真正可靠的长上下文能力还和训练数据、指令数据、注意力实现和评估有关。

Sliding window attention 让模型主要关注最近一段上下文，降低长序列的计算和缓存压力。它适合局部依赖强的任务，但如果关键证据在窗口外，就需要摘要、检索或显式记忆机制补上。

Chunking 把长文档切成可管理片段；retrieval-assisted context 用检索选择相关片段；summarization 把历史或文档压缩成短文本；prompt compression 删除冗余表达、保留约束和证据。这些方法共同解决“上下文窗口有限且昂贵”的问题。

端侧部署的约束更硬：设备内存小，算力弱，功耗敏感，还可能要求离线和隐私。量化可以降低模型权重占用，小模型降低计算量，本地推理引擎负责把模型格式、算子和硬件能力连接起来。代价是模型能力、上下文长度、生成速度和输出质量都可能受限。

成本调试要看全链路。一个看似便宜的模型，如果经常失败重试，实际上会把输入、输出、检索、重排和工具调用成本一起重复放大；如果还需要长期占用机器、监控告警或人工复核大量坏答案，总成本仍然可能很高。

## 和应用/面试的连接

工程上，长上下文优化经常从问题定位开始：

- TTFT 高：检查输入 token、prefill、排队、长文档拼接。
- 答案漏证据：检查 chunking、检索召回、排序、摘要是否丢信息。
- 账单高：拆输入、输出、检索/重排、工具调用和重试。
- 端侧慢：检查模型大小、量化精度、上下文长度和硬件后端。

面试里，回答长上下文问题时要强调取舍：更长窗口提高可见信息上限，但不会自动带来更好质量；端侧部署提高隐私和离线能力，但通常要牺牲模型规模、速度或任务覆盖。

## 常见误区

| 误区 | 更准确的说法 |
|------|--------------|
| 长上下文越长越好 | 更长窗口提高容量，也提高成本、延迟和注意力分散风险 |
| RoPE scaling 等于解决长上下文 | 它主要处理位置外推或适配，质量仍需训练和评估验证 |
| RAG 和长上下文二选一 | RAG 可以帮长上下文选择更相关的信息 |
| 端侧部署只是把云端模型下载下来 | 端侧通常需要小模型、量化、格式转换和本地推理引擎 |
| 只看模型 token 单价就能估成本 | 还要算检索、重排、工具调用、失败重试和人工处理 |
