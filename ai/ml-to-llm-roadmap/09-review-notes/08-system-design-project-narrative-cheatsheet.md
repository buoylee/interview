# 系统设计项目叙事面试速记

> 这份笔记用于复习，不适合作为第一次学习入口。第一次学习先读 [系统设计与项目叙事](../08-system-design-project-narrative/)。

## 30 秒答案

AI 系统设计要从任务目标、输入输出契约、模型与控制策略、评估安全、服务成本、监控迭代六步展开。模型路由按任务、风险、上下文、延迟和预算选择 cheap/strong/long-context/local 模型；fallback、缓存、限流和成本治理要进平台层。项目叙事用“问题、约束、设计、eval、生产结果、复盘”讲清工程判断。

## 2 分钟展开

系统设计不要一上来就说“接一个大模型”。先定义用户是谁、任务是什么、成功指标是什么，再定义输入和输出契约。输出如果要进下游系统，优先设计 schema、字段解释、错误处理和人工升级边界。模型选择要服务任务：简单分类可用便宜低延迟模型，复杂高风险请求再路由到更强模型。

平台层要把模型调用变成可治理能力：统一鉴权、超时、重试、路由、fallback、限流、缓存、版本记录、日志、评估钩子和成本核算。Fallback 要区分超时、限流、schema failure、低质量输出和 safety risk，不是失败后随便换一个模型继续答。缓存要注意个性化、隐私、模型版本和 prompt 版本。

项目叙事不是功能列表。用 Situation/Task 说清业务问题和约束，用 Action 讲输入输出契约、模型控制、评估安全、服务成本和监控，用 Result 给指标变化和上线影响，用 Reflection 讲失败样本、取舍和下一步。RAG/Agent 是独立题型边界：只有外部知识证据或多步工具执行需要时再引入。

## 高频追问

| 追问 | 回答 |
|------|------|
| AI 系统设计六步是什么？ | 任务指标、输入输出契约、模型控制、评估安全、服务成本、监控迭代。 |
| 为什么输入/输出契约要先定义？ | 契约决定模型是否需要结构化输出、校验、下游集成和失败处理。 |
| 模型路由看哪些信号？ | 任务类型、输入长度、风险等级、目标延迟、预算、模型健康和历史失败率。 |
| Fallback 怎么设计？ | 按失败类型处理：超时可换供应商，schema failure 可重试修复，高风险可人工复核或只返回部分结果。 |
| 缓存能缓存什么？ | 稳定请求、规则分类、prompt 编译结果或评估结果；要带版本、隐私和个性化边界。 |
| 成本治理怎么拆？ | 输入 token、输出 token、重试、fallback、缓存未命中、校验、安全过滤、评估和人工复核。 |
| 项目叙事怎么讲得有技术深度？ | 讲具体问题、真实约束、关键取舍、评估证据、生产指标、失败案例和下一轮计划。 |

## 易混点

| 概念 | 容易混的点 | 正确理解 |
|------|------------|----------|
| AI 系统设计 vs 模型选择 | 以为选最强模型就够 | 还要契约、评估、安全、成本、监控和迭代 |
| 路由 vs fallback | 都是换模型 | 路由是调用前选择，fallback 是失败或低质量后的降级路径 |
| 成本治理 vs 单价优化 | 以为换便宜模型即可 | 重试、长上下文、长输出、人工复核和 eval failure 都会放大成本 |
| 项目叙事 vs STAR 背稿 | 以为按模板讲就够 | 技术深度来自约束、决策、评估和复盘 |
| RAG/Agent | 以为系统设计默认要加 | 它们是可选扩展，不是所有 AI 系统起点 |

## 项目连接

- 客服工单分类：用固定 JSON schema、低温度、风险字段、eval set 和人工升级讲清可上线性。
- 模型平台：讲统一网关、版本管理、路由规则、预算面板、日志和评估钩子。
- 成本优化：先拆 token、重试、长上下文、fallback、缓存命中和人工复核，再谈模型替换。
- 面试项目：准备 60 到 90 秒主线，再准备 eval、latency、cost、safety、tradeoff、failure case 追问。
