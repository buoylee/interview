## 2. 预训练 Pre-training

### 2.1 训练目标

```
Causal Language Modeling: P(next_token | context)
损失 = -Σ log P(wₜ | w₁, ..., wₜ₋₁) = 交叉熵
```

### 2.2 训练数据

```
来源（TB 级）：
  - Common Crawl（互联网爬虫）→ 需要大量清洗
  - 书籍（BookCorpus、Gutenberg）
  - 维基百科
  - 代码（GitHub）
  - 学术论文（ArXiv）
  - 对话数据（Reddit 等）

数据处理：
  1. 过滤: 去除低质量/有害内容
  2. 去重: MinHash / SimHash 去除重复文档
  3. 配比: 不同来源的混合比例（关键调参）
  4. Tokenization: BPE/SentencePiece
```

### 2.3 预训练学到了什么？

```
语言能力:  语法、语义、上下文理解
世界知识:  事实、常识、历史
推理能力:  逻辑推理（有限）
代码能力:  编程语言的语法和模式

没有学到:
  ❌ 如何对话（只是预测下一个词）
  ❌ 如何遵循指令
  ❌ 什么该说什么不该说
```

### 2.4 预训练的成本

```
LLaMA-65B: ~2048 A100 GPU × 21天 ≈ 约 200-300 万美元
GPT-4: 估计 $50M-100M+
DeepSeek-V3: ~$5.6M（用 MoE+MLA 降低了成本）
```

## 3. 监督微调 SFT

### 3.1 目的

```
预训练后的模型是一个"续写机器"：
  输入: "世界上最高的山是"
  输出: "世界上最高的山是珠穆朗玛峰，位于..." ← 不错

  输入: "请告诉我世界上最高的山是什么"
  输出: "请告诉我世界上最高的山是什么？这是一个很好的问题..." ← 在续写而不是回答！

SFT 教模型学会对话格式：
  输入: [User] 请告诉我世界上最高的山是什么
  输出: [Assistant] 世界上最高的山是珠穆朗玛峰(8848.86米)，位于中国和尼泊尔边境。
```

### 3.2 训练数据格式

```json
{
  "messages": [
    {"role": "system", "content": "你是一个有帮助的助手。"},
    {"role": "user", "content": "请告诉我世界上最高的山是什么"},
    {"role": "assistant", "content": "世界上最高的山是珠穆朗玛峰..."}
  ]
}
```

### 3.3 数据质量 vs 数量 ⭐

```
LIMA 论文的发现 (2023)：
  1000 条高质量数据的 SFT ≈ 大量低质量数据的 SFT

质量 > 数量！
  - 精心编写的回答 > 自动生成的回答
  - 多样性很重要（覆盖不同任务类型）
  - 通常 1万-10万 条高质量数据就够了
```

### 3.4 SFT 的局限

```
SFT 只教模型"模仿"回答格式
  ❌ 不知道哪个回答更好（多个合理回答中无法区分）
  ❌ 不知道什么不该说（没有负面反馈）
  ❌ 可能学会"讨好"但不诚实

→ 需要对齐（Alignment）来解决
```

## 4. 对齐 Alignment

### 4.1 为什么需要对齐？

```
SFT 后的模型会：
  ❌ 生成有害内容（如果预训练数据有）
  ❌ 自信地说错话（幻觉）
  ❌ 不会拒绝不当请求
  ❌ 过于啰嗦或机械

对齐的目标（HHH 准则）：
  Helpful: 有帮助的
  Honest: 诚实的
  Harmless: 无害的
```

### 4.2 RLHF 概览

```
RLHF = Reinforcement Learning from Human Feedback

Step 1: 训练奖励模型 (Reward Model)
  给 SFT 模型生成多个回答 → 人类标注排序
  用排序数据训练 RM：好回答得高分，差回答得低分

Step 2: 用 PPO 优化策略
  模型生成回答 → RM 评分 → PPO 更新模型参数
  + KL 约束（不要偏离 SFT 模型太远）

目标 = max [RM(回答)] - β × D_KL(当前模型 || SFT模型)
```

## 5. RL 基础（RLHF 前置）

### 5.1 核心概念

```
Agent（智能体）→ 在 Environment（环境）中执行 Action（动作）
              → 获得 Reward（奖励）
              → 根据 State（状态）选择下一个 Action

在 RLHF 中：
  Agent = LLM（语言模型）
  Action = 生成下一个 token
  State = 当前已生成的文本
  Reward = 奖励模型(RM)的评分
  Environment = 对话上下文
```

### 5.2 策略 (Policy)

```
π(a|s) = 在状态 s 下选择动作 a 的概率

在 LLM 中：
  π(token | context) = P(next_token | 前面的文本)
  
LLM 本身就是一个策略！
```

### 5.3 PPO (Proximal Policy Optimization)

```
核心思想：限制每步更新的幅度，防止训练不稳定

PPO 约束：
  ratio = π_new(a|s) / π_old(a|s)
  
  clip(ratio, 1-ε, 1+ε)  ← 限制比率在 [1-ε, 1+ε] 范围内

直觉：不能一步改太多
  - 如果新策略比旧策略好 → 更新，但不能太大步
  - 如果新策略比旧策略差 → 回退
```

### 5.4 KL 散度约束

```
目标 = max E[RM(response)] - β × D_KL(π_θ || π_ref)

π_θ = 当前训练的模型
π_ref = 参考模型（SFT 后的模型）

为什么需要 KL 约束？
  ❌ 没有约束：模型可能找到"hack"RM 的方式
     例：重复说"我很抱歉"可能得高分
  ✅ 有约束：模型不能偏离正常行为太远
```

## 6. 面试常问

### Q1: 大模型训练的三个阶段分别是什么？各做什么？

**答**：
1. **预训练 (PT)**：在 TB 级无标注文本上做下一个词预测，学会语言和知识
2. **SFT**：在指令-回复对上微调，学会对话格式和遵循指令
3. **对齐 (RLHF/DPO)**：用人类偏好数据训练，让模型更有帮助、诚实、安全

### Q2: SFT 的数据质量和数量哪个更重要？

**答**：质量远比数量重要（LIMA 论文证明 1000 条高质量数据就能训出好效果）。但需要保证多样性（覆盖不同任务类型）。实际中通常用 1-10 万条高质量数据。

### Q3: 为什么 RLHF 需要 KL 约束？不加会怎样？

**答**：不加 KL 约束，模型会过度优化 RM 的分数（reward hacking），找到欺骗 RM 的方式获得高分但实际回答很差。KL 约束确保模型不偏离 SFT 模型太远，保留通用能力。
