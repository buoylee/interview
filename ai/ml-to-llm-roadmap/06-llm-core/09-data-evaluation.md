# 6.9 数据工程 + LLM 评估（Day 18-19）

> **一句话定位**：数据质量决定模型质量上限，评估方法决定我们怎么衡量"好不好"。

---

## 1. 数据工程

### 1.1 预训练数据处理

```
原始互联网文本 → 高质量训练语料

处理流水线：
1. URL过滤 → 去除已知低质量网站
2. 语言检测 → 按语言分类
3. 去重
   - Exact dedup: 精确去重（hash）
   - Fuzzy dedup: 近似去重（MinHash/SimHash）
   - 文档级和段落级都做
4. 质量过滤
   - 规则过滤（长度、特殊字符比例）
   - 分类器过滤（训练质量判别模型）
5. 有害内容过滤 → 去除色情/暴力/偏见
6. PII 去除 → 个人信息脱敏
```

### 1.2 数据配比（Data Mix）⭐

```
不同数据源的混合比例直接影响模型能力：

  代码比例高 → 代码能力强但对话可能差
  对话比例高 → 对话好但推理可能弱
  学术比例高 → 知识准确但风格可能僵

LLaMA 的数据配比（参考）:
  CommonCrawl: 67%
  C4: 15%
  GitHub: 4.5%
  Wikipedia: 4.5%
  Books: 4.5%
  ArXiv: 2.5%
  StackExchange: 2%
```

### 1.3 SFT 数据构建

```
方法 1: 人工编写 → 质量最高但贵
方法 2: 模型生成（蒸馏）→ 用 GPT-4 生成训练数据
方法 3: 改写/增强 → 从种子数据扩展

Self-Instruct 方法：
  种子任务 → LLM 生成新任务 → 过滤低质量 → 循环

关键：多样性 > 数量
```

## 2. LLM 评估

### 2.1 评估维度

| 维度 | 评估标准 | 代表基准 |
|------|---------|---------|
| **知识** | 事实准确性 | MMLU, TriviaQA |
| **推理** | 逻辑/数学能力 | GSM8K, MATH, ARC |
| **代码** | 编程能力 | HumanEval, MBPP |
| **对话** | 自然度/帮助度 | MT-Bench, Chatbot Arena |
| **安全** | 有害内容 | TruthfulQA |
| **长上下文** | 长文理解 | RULER, Needle-in-Haystack |

### 2.2 Chatbot Arena ⭐ 面试知识点

```
ELO 评分系统（类似于国际象棋）

流程：
1. 用户提问 → 两个匿名模型回答
2. 用户投票哪个更好
3. 根据投票更新 ELO 分数

优点：最接近真实用户体验的评估
缺点：需要大量人类评测，成本高
```

### 2.3 LLM as Judge

```
用 LLM（如 GPT-4）来评估其他模型的输出：

给定 question, answer, reference_answer
GPT-4 评分: 1-10 分 + 理由

优点：成本低，可规模化
缺点：GPT-4 有偏好（更喜欢长回答、格式化回答）
```

## 3. 面试常问

### Q1: 数据质量和数量哪个重要？

**答**：预训练阶段两者都重要（Scaling Law），但数据质量是上限。SFT 阶段质量远比数量重要（LIMA 论文：1000 条高质量 > 100 万条低质量）。

### Q2: 怎么评估 LLM？

**答**：多维度评估：知识(MMLU)、推理(GSM8K)、代码(HumanEval)等。最可靠的是 Chatbot Arena（人类盲评 ELO 排名），自动评估可以用 LLM as Judge 但有偏见。
