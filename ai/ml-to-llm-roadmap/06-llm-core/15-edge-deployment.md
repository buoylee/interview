# 6.15 端侧 / 小模型部署

> **一句话定位**：如何将 LLM 部署到手机、PC、嵌入式设备上——涉及模型压缩全流程和端侧推理引擎。

---

## 1. 为什么要端侧部署？

```
服务端部署的问题：
  - 延迟：网络往返 100-500ms，流式输出也有首 token 延迟
  - 成本：GPU 推理按 token 收费
  - 隐私：用户数据必须发送到服务器
  - 离线：无网络环境无法使用

端侧部署的价值：
  - 隐私敏感场景（医疗、金融、个人助手）
  - 低延迟（本地推理无网络开销）
  - 无需 API 成本
  - 离线可用（飞机上、网络不好的地方）

挑战：
  手机内存 ≈ 6-12 GB（模型 + 系统 + App 共用）
  PC 内存 ≈ 16-32 GB
  LLaMA-7B FP16 ≈ 14 GB → 手机跑不了
  → 必须压缩模型
```

## 2. 模型压缩 Pipeline

```
完整压缩流程：

原始模型 (7B FP16, 14GB)
  ↓ 知识蒸馏 → 更小的模型 (3B)
  ↓ 剪枝 → 去掉不重要的参数
  ↓ 量化 → INT4/INT8 → 最终 2-4GB

三种技术可以组合使用，不是互斥的
```

### 2.1 量化 (Quantization) ⭐

```
核心思想：降低参数精度，减小模型体积

精度层级：
  FP32 (4 bytes) → FP16/BF16 (2 bytes) → INT8 (1 byte) → INT4 (0.5 byte)

  7B 模型体积：
  FP32: 28 GB
  FP16: 14 GB
  INT8: 7 GB
  INT4: 3.5 GB  ← 手机可以跑

量化方法分类：

1. 训练后量化 (Post-Training Quantization, PTQ)
   训练完成后直接量化，不需要重新训练
   - GPTQ：基于 Hessian 信息的分层量化
   - AWQ：保护激活值大的"重要"权重不量化
   - GGUF/GGML：llama.cpp 使用的量化格式，CPU 友好
   - SmoothQuant：平滑激活值分布，使量化更均匀

2. 量化感知训练 (Quantization-Aware Training, QAT)
   训练时模拟量化误差 → 模型学会适应低精度
   效果更好但需要训练资源
   - QLoRA：量化 + LoRA 微调

量化粒度：
  Per-tensor: 整个张量一个 scale → 快但粗糙
  Per-channel: 每个通道一个 scale → 常用
  Per-group: 每 128 个参数一个 scale → 精细但慢
```

### 2.2 剪枝 (Pruning)

```
核心思想：去掉"不重要"的参数

1. 非结构化剪枝 (Unstructured)
   将小于阈值的权重置零 → 稀疏矩阵
   - SparseGPT：针对 LLM 的一次性剪枝
   - 问题：普通硬件不擅长稀疏计算 → 实际加速有限

2. 结构化剪枝 (Structured)
   删掉整个注意力头、整层、或 FFN 的整行/列
   - LLM-Pruner：删掉 LLM 中不重要的结构
   - 优点：直接减小模型尺寸 → 不依赖稀疏硬件
   - 缺点：精度下降更大

3. 深度剪枝
   直接删掉中间的 Transformer 层
   研究发现：LLM 中间层有较大冗余
   删掉 25-50% 的层 → 效果下降有限
```

### 2.3 知识蒸馏 (Distillation)

```
在端侧部署的角色：训练一个小模型来模仿大模型

教师模型 (70B) → 学生模型 (3B)
  学生学习教师的输出分布（soft label），不只是答案

端侧蒸馏的典型方案：
  1. 用大模型生成高质量数据
  2. 用生成的数据 SFT 小模型
  3. 可选：对齐蒸馏（学习大模型的偏好）

代表：
  Phi-3 (3.8B)：用 GPT-4 生成的合成数据训练
  Gemma-2 (2B)：Google 的小模型蒸馏
  Qwen-2.5 (0.5B/1.5B/3B)：阿里的多尺寸蒸馏
```

## 3. 端侧推理引擎

### 3.1 llama.cpp / GGUF ⭐

```
最流行的端侧 LLM 推理引擎

特点：
  - 纯 C/C++ 实现 → 无依赖
  - 支持 CPU 推理（AVX/AVX2/NEON 加速）
  - 支持 Metal (macOS/iOS)、CUDA、Vulkan
  - GGUF 格式：自包含的量化模型格式

GGUF 量化级别：
  Q2_K: ~2 bit → 最小但质量差
  Q4_K_M: ~4.5 bit → 性价比最优 ⭐
  Q5_K_M: ~5.5 bit → 质量好
  Q8_0: ~8 bit → 接近原始精度

性能（7B Q4 模型）：
  MacBook M2: ~30 tokens/s
  iPhone 15: ~10-15 tokens/s
  高端安卓: ~5-10 tokens/s
```

### 3.2 其他推理引擎

| 引擎 | 平台 | 特点 |
|------|------|------|
| **MLC-LLM** | 全平台 | 基于 TVM，支持 iOS/Android/Web |
| **ExecuTorch** | Meta 移动端 | PyTorch 官方的端侧方案 |
| **MediaPipe LLM** | Google 移动端 | 集成到 Google 生态 |
| **ONNX Runtime** | 全平台 | 工业标准格式 |
| **TensorRT-LLM** | NVIDIA GPU | 服务端但也支持 PC |

## 4. 端侧 LLM 的特殊约束

```
1. 内存管理
   手机应用通常限制在 2-4 GB 内存
   → 模型必须 < 2 GB（和 KV-Cache 共用内存）
   → INT4 量化几乎是必须的

2. 热管理
   持续推理 → 手机发热 → 降频 → 速度变慢
   → 需要 token 预算（限制单次生成长度）

3. 电池消耗
   LLM 推理非常耗电
   → 需要考虑何时用端侧、何时用云端

4. 首 token 延迟
   模型加载 + 首次 prefill 可能需要数秒
   → 模型预加载、KV-Cache 预热

5. 上下文长度限制
   端侧内存有限 → KV-Cache 占内存
   → 通常限制 2K-4K context（不像云端可以 128K）
```

## 5. 小模型 vs 大模型的 Trade-off

```
关键问题：什么时候该用小模型端侧推理？什么时候该调用云端大模型？

适合端侧的场景：
  ✅ 简单任务（文本分类、情感分析、简短问答）
  ✅ 隐私敏感（个人笔记、健康数据）
  ✅ 离线场景
  ✅ 高频低延迟需求（输入联想、自动补全）

适合云端的场景：
  ✅ 复杂推理（数学、代码生成、长文档分析）
  ✅ 长上下文（>4K tokens）
  ✅ 最高质量要求
  ✅ 多模态（图像理解等）

混合方案：
  Router → 判断任务难度 → 简单任务端侧处理 → 复杂任务调用云端
  → 这就是 "Compound AI Systems" 的一个应用
```

## 6. 面试常问

### Q1: 如何将 7B 模型部署到手机上？

**答**：核心是量化。用 GPTQ 或 AWQ 将 FP16 量化到 INT4，模型从 14GB 压缩到 ~3.5GB。用 GGUF 格式 + llama.cpp 推理引擎，利用手机 CPU/GPU 加速。需要限制上下文长度（2-4K）以控制 KV-Cache 内存。Q4_K_M 是性价比最优的量化级别。

### Q2: 量化 INT4 后模型质量掉多少？

**答**：取决于量化方法。GPTQ INT4 在 perplexity 上通常只增加 0.5-1.0（相对 FP16）。AWQ 对重要权重保护更好，掉点更少。关键是：简单任务（分类、短文本）几乎不受影响；复杂推理任务（数学、代码）掉点更明显。INT8 和 FP16 几乎无差异。

### Q3: 端侧部署和服务端部署的核心区别？

**答**：(1) 内存：端侧通常 < 4GB vs 服务端 80GB+ GPU (2) 精度：端侧几乎必须 INT4 量化 (3) 上下文：端侧限制 2-4K vs 服务端 128K+ (4) 并发：端侧单用户 vs 服务端批量处理 (5) 优化重点：端侧关注模型体积和首 token 延迟，服务端关注吞吐量和批处理效率。
