# 6.3 Scaling Law & 涌现能力（Day 7）

> **一句话定位**：Scaling Law 回答了「模型多大才够好」，涌现能力解释了「为什么大到某个点突然变好」。

---

## 1. Scaling Law

### 1.1 Kaplan Scaling Law (2020, OpenAI)

```
L(N, D, C) ∝ N^(-0.076) + D^(-0.095) + C^(-0.050)

L = 测试损失
N = 模型参数量
D = 训练数据量
C = 训练计算量(FLOPs)

核心发现：增大任何一个因素都能降低损失，且关系是幂律(power law)
```

### 1.2 Chinchilla Law (2022, DeepMind) ⭐

```
修正：Kaplan Law 低估了数据的重要性

Chinchilla 最优：模型参数 N 和训练 token 数 D 应该同比增长
  D ≈ 20 × N

例：
  10B 参数模型 → 应该用 200B tokens 训练
  70B 参数模型 → 应该用 1.4T tokens 训练

实际影响：
  GPT-3 (175B) 用 300B tokens → 欠训了！
  LLaMA (65B) 用 1.4T tokens → 按 Chinchilla 配比 → 效果更好
```

### 1.3 Chinchilla 公式 ⭐

```
给定计算预算 C (FLOPs)，最优的参数量 N 和数据量 D：

  N_opt ∝ C^0.50
  D_opt ∝ C^0.50

  即：N 和 D 应该以相同的速率增长

  具体系数：D ≈ 20 × N

  验证：
    Chinchilla (70B params, 1.4T tokens)
    N = 70B, D = 1.4T → D/N = 20 ✓

  对比 GPT-3：
    N = 175B, D = 300B → D/N ≈ 1.7 → 严重欠训！
    按 Chinchilla，175B 应该用 3.5T tokens
```

### 1.4 Over-training（过度训练）策略

```
Chinchilla 说 D ≈ 20N，但现在的趋势是 D >> 20N：

  LLaMA-2 (70B) 用 2T tokens    → D/N ≈ 29
  LLaMA-3 (70B) 用 15T tokens   → D/N ≈ 214
  Qwen-2.5 (72B) 用 18T tokens  → D/N ≈ 250

为什么要"过度训练"？

  Chinchilla 优化的是：固定预算下最低的训练 loss
  → 但忽略了部署成本！

  实际情况：
    训练成本是一次性的
    推理成本是持续的（每天百万次调用）
    → 小模型推理更便宜
    → 值得花更多算力把小模型训到更好

  新公式（实践导向）：
    不是"给定预算的最优模型"
    而是"给定目标质量的最便宜推理模型"
    → 训练一个更小但训练更充分的模型
```

### 1.5 实际指导意义

```
确定预算 → 分配最优的参数量和数据量

学术最优（Chinchilla）：
  D ≈ 20N，追求训练效率

工业最优（Over-training）：
  D >> 20N，追求推理性价比
  "宁可训练时多花钱，也要部署时少花钱"

数据墙问题：
  高质量文本数据有限（互联网文本约 10-15T tokens）
  → 数据重复使用 → 效果递减
  → 合成数据（用强模型生成）成为新方向
  → 多模态数据（图像、视频、代码）补充
```

## 2. 涌现能力 (Emergent Abilities)

### 2.1 什么是涌现？

```
某些能力在小模型中不存在，但当模型达到一定规模后突然出现：

  小模型: 简单问答 ✅, 思维链推理 ❌, 代码生成 ❌
  中模型: 简单问答 ✅, 思维链推理 🔶, 代码生成 ❌
  大模型: 简单问答 ✅, 思维链推理 ✅, 代码生成 ✅ ← 涌现！

不是线性提升，而是"相变"（跳跃式出现）
```

### 2.2 涌现的争议

```
有人认为"涌现"是度量方式的伪影：
  - 用连续指标（如 token 级准确率）→ 能力是平滑增长的
  - 用非连续指标（如完全正确率）→ 看起来是突然出现的

但即使如此，大模型确实能做小模型做不到的事情
关键可能不是"涌现"而是"量变到质变"
```

## 3. Test-time Compute Scaling ⭐ 最新趋势

```
传统 Scaling: 模型更大 → 更好（训练阶段扩展）
新的 Scaling: 推理时多想一会 → 更好（推理阶段扩展）

OpenAI o1/o3:
  不是用更大的模型，而是让模型在推理时生成更长的思维链
  推理时多花 10x 计算 → 效果提升显著

这开辟了新的 Scaling 维度：
  训练 Compute Scaling (传统) + Test-time Compute Scaling (新)
```

## 4. 面试常问

### Q1: 什么是 Chinchilla Scaling Law？

**答**：给定计算预算，最优策略是让参数量 N 和训练 token 数 D 同比增长（D ≈ 20N）。修正了之前只追求更大模型的做法，强调数据量同样重要。

### Q2: 什么是涌现能力？

**答**：某些能力只在模型超过一定规模后才出现（如 CoT 推理、代码生成）。存在争议——可能是度量方式造成的错觉，但大模型确实能做小模型做不到的复杂任务。

### Q3: Test-time Compute Scaling 是什么？

**答**：不增大模型，而是让模型在推理时生成更长的思维过程。OpenAI o1 通过延长推理时间（更多 token）来提升复杂任务的表现，开辟了训练之外的新 Scaling 维度。详见 [12-test-time-compute.md](./12-test-time-compute.md)。

### Q4: 为什么现在的模型都"过度训练"？Chinchilla 过时了吗？

**答**：Chinchilla Law 本身没错，但它优化的是训练效率（固定预算下最低 loss），没考虑推理成本。工业界训练是一次性的，推理是持续的——所以宁可多花训练预算把小模型训好（D >> 20N），也要降低推理成本。LLaMA-3 用 15T tokens 训 70B 参数就是典型案例。

### Q5: 数据墙问题是什么？怎么解决？

**答**：高质量文本数据有限（互联网约 10-15T tokens），但模型训练需要的数据量越来越大。解决方向：(1) 数据质量提升（更好的过滤和去重）(2) 合成数据（用强模型生成训练数据）(3) 多模态数据（图像、视频、代码）补充 (4) 课程学习（优化数据使用顺序）。
