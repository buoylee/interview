# 6.12 Test-time Compute Scaling（推理时计算扩展）⭐

> **一句话定位**：传统 Scaling 是"训练更大的模型"，新范式是"让模型在推理时多想一会"——不改模型大小，通过增加推理计算量来提升效果。

---

## 1. 两种 Scaling 范式

```
传统 Scaling（训练阶段）：
  更多参数 + 更多数据 + 更多计算 → 更好的模型
  代价：训练一次几千万美元，效果提升越来越慢

新范式 Scaling（推理阶段）：
  同一模型 + 推理时多花计算 → 更好的回答
  代价：每次推理更贵，但可以按需调节

类比：
  训练 Scaling = 培养一个更聪明的人（投入教育）
  推理 Scaling = 给同一个人更多时间思考（投入解题时间）
```

## 2. Test-time Compute 的主要方法

### 2.1 Long Chain-of-Thought（长思维链）

```
核心思想：让模型生成更长的推理过程

普通回答：
  Q: 7 × 8 + 3 × 5 = ?
  A: 71

长 CoT 回答：
  Q: 7 × 8 + 3 × 5 = ?
  <thinking>
  首先计算 7 × 8 = 56
  然后计算 3 × 5 = 15
  最后 56 + 15 = 71
  让我验证：56 + 15 = 71 ✓
  </thinking>
  A: 71

为什么有效：
  - 中间 token 提供"工作记忆"（Transformer 没有内部状态，靠生成的 token 当草稿纸）
  - 更多推理步骤 → 更低的单步推理复杂度
  - 自我验证 → 减少错误
```

### 2.2 Best-of-N Sampling（多次采样取最优）

```
方法：
  1. 对同一问题生成 N 个回答（N = 8, 16, 64...）
  2. 用某种方法选出最好的一个

选择方法：
  - Majority Voting（多数投票）：选出现最多的答案
  - Reward Model 打分：用奖励模型给每个回答评分
  - Verifier 验证：用验证器检查答案正确性

例：数学题生成 64 个回答 → 多数投票 → 准确率大幅提升

计算成本：线性增长 → N 越大越好，但收益递减
```

### 2.3 Tree Search（树搜索）

```
方法：将推理过程建模为搜索树

          问题
         /    \
      方案A    方案B
      /  \       \
   步骤A1 步骤A2  步骤B1
    |       |      |
   ...     ...    ...

搜索策略：
  - Beam Search：保留 top-K 路径
  - Monte Carlo Tree Search (MCTS)：随机模拟 + 回溯更新
  - Best-First Search：按 Reward Model 打分展开最优节点

AlphaGo 的思想在 LLM 中的应用：
  AlphaGo: 用 MCTS 搜索棋盘状态
  LLM:     用 MCTS 搜索推理路径
```

### 2.4 Self-Refinement（自我修正）

```
方法：
  1. 生成初始回答
  2. 让模型审视自己的回答，找出问题
  3. 生成改进后的回答
  4. 重复 2-3 直到满意

Prompt 模式：
  Round 1: "解决这个问题" → 回答 v1
  Round 2: "检查你的回答是否有错误" → 发现问题
  Round 3: "修正这些错误" → 回答 v2

局限：
  - 模型可能"越改越错"（没有外部信号指导方向）
  - 需要 Reward Model 或 Verifier 判断是否真的在改进
```

## 3. 代表系统

### 3.1 OpenAI o1 / o3

```
核心机制：
  - 训练模型在推理时生成长"思维链"（可能几千 token）
  - 思维链对用户不可见（<thinking> block）
  - 通过 RL 训练模型学会何时思考更多、何时停止

训练方法（推测）：
  1. 用 RL (PPO/类似方法) 训练策略
  2. 奖励信号 = 最终答案是否正确
  3. 模型学会在困难问题上生成更长的推理
  4. Process Reward Model (PRM) 可能用于每步打分

关键创新：
  - 不是 prompt 工程让模型"一步步想"
  - 是通过 RL 训练模型自主决定推理深度
  - 困难问题 → 自动思考更久，简单问题 → 快速回答
```

### 3.2 DeepSeek-R1

```
核心贡献：
  - 开源的推理模型，效果接近 o1
  - 揭示了训练方法：GRPO (Group Relative Policy Optimization)

训练流程：
  1. 冷启动：少量 long-CoT 数据做 SFT
  2. RL 阶段：用 GRPO 训练
     - 同一问题生成一组回答
     - 组内排序（好的奖励高，差的奖励低）
     - 不需要单独的 Reward Model
  3. 拒绝采样 + SFT：筛选 RL 阶段的好样本再 SFT

涌现的行为：
  - 自我反思 ("Wait, let me reconsider...")
  - 回溯 ("Actually, my previous step was wrong...")
  - 验证 ("Let me verify this by...")
  这些行为不是人工设计的，是 RL 训练中自然涌现的
```

## 4. Process Reward Model vs Outcome Reward Model

```
Outcome Reward Model (ORM)：
  只看最终答案对不对 → 给整个推理链一个分数
  简单但信号稀疏

Process Reward Model (PRM)：
  给推理过程的每一步打分 → "这一步推理正确吗？"
  更细粒度，能指导搜索方向

对比：
  ORM: [Step1→Step2→Step3→答案] → 分数: 0.8
  PRM: [Step1:0.9→Step2:0.7→Step3:0.3→答案:0.2] → 能发现 Step3 出错

PRM 的价值：
  - 配合 Tree Search：用 PRM 分数决定展开哪个节点
  - 配合 Best-of-N：用 PRM 选择推理过程最好的回答
  - 训练信号：告诉模型"哪一步"出了问题，而非只知道"结果"错了
```

## 5. Scaling Law: 训练 vs 推理

```
传统观点：
  固定推理成本 → 尽量在训练时 scale

新观点：
  训练计算 + 推理计算 是一个统一的计算预算分配问题

  场景 A：训练 1000B 参数模型 + 快速推理
  场景 B：训练 70B 参数模型 + 推理时多花 10x 计算

  如果总预算一样，场景 B 可能在复杂任务上更好！

关键洞察：
  - 简单任务：大模型直接回答更高效（不需要长推理）
  - 复杂任务：小模型 + 多推理计算可能更好（推理 scaling 收益高）

  → 未来可能是"动态分配"：根据问题难度决定推理计算量
```

## 6. 工程实现考量

```
延迟影响：
  - Long CoT 可能生成 2000+ token → 延迟从 1s 变成 30s+
  - 用户体验需要权衡（流式输出 + 隐藏思维链）

成本影响：
  - Best-of-64 = 64x 推理成本
  - 需要 Reward Model 打分 = 额外模型调用
  - Tree Search = 分支生成 + 评估

优化方向：
  - 难度自适应：简单问题少想，复杂问题多想
  - 并行推理：N 个采样可以并行生成
  - 提前终止：Reward 已经很高就不继续搜索
  - 推理蒸馏：把长 CoT 的能力蒸馏到短 CoT 模型
```

## 7. 面试常问

### Q1: 什么是 Test-time Compute Scaling？和传统 Scaling 有什么区别？

**答**：传统 Scaling 通过增大模型参数量和训练数据来提升能力。Test-time Compute Scaling 不改变模型本身，而是在推理时投入更多计算——让模型生成更长的思维链、多次采样取最优、或用树搜索探索推理路径。两者是互补的 Scaling 维度。

### Q2: OpenAI o1 是怎么实现的？

**答**：o1 通过 RL 训练模型学会在推理时生成长思维链。不同于手工 prompt 的 CoT，模型自主决定推理深度——复杂问题自动思考更多步骤。训练时可能用了 Process Reward Model 对每步推理打分，通过 RL 优化整个推理策略。

### Q3: Process Reward Model 和 Outcome Reward Model 的区别？

**答**：ORM 只看最终答案给分（信号稀疏），PRM 对推理过程的每一步打分（信号密集）。PRM 能发现推理链在哪一步出错，配合树搜索可以更高效地引导推理方向。代价是 PRM 需要步骤级别的标注数据来训练。

### Q4: 推理 Scaling 和训练 Scaling 应该怎么分配计算预算？

**答**：取决于任务难度。简单任务用大模型直接回答更高效；复杂推理任务用中等模型+推理时多花计算可能性价比更高。未来趋势可能是动态分配：根据问题难度自适应调整推理计算量。
