# RAG 解决什么问题：边界与基本链路

## 这篇解决什么问题

LLM 的权重里有通用知识，但它不是一个可靠的私有知识库。公司制度、产品文档、合同条款和工单状态会变化，也可能只对部分用户可见。如果模型只靠训练记忆或 prompt 猜答案，就容易把“看起来合理”说成“有依据”。

这一篇解决的问题是：RAG 为什么存在，它和长上下文、微调、搜索、工具调用的边界是什么，以及一个最小 RAG 链路应该包含哪些步骤。

- 微调更适合改变行为风格和任务格式：[LoRA、QLoRA 与蒸馏](../05-training-alignment-finetuning/03-lora-qlora-distillation.md)

如果还不熟，可以先记住一句话：RAG 不是让模型“记住更多”，而是在回答前把可验证的外部知识放进上下文。

## 概念为什么出现

知识进入 LLM 输出有三个常见困难：

```text
知识会变: 政策、价格、库存和权限每天都可能变化
知识私有: 公司内部文档不应该训练进公开模型权重
回答要有依据: 用户和业务方需要知道答案来自哪条文档
```

RAG 出现，是为了把外部知识从“系统里有一堆文档”变成“模型回答时能看到相关证据，并受证据约束”。

## 最小心智模型

一个最小 RAG 链路是：

```text
User query -> query understanding -> retrieve -> assemble context -> generate -> check grounding/citation
```

各步职责：

- query understanding：理解用户真正问什么，必要时改写查询或提取过滤条件。
- retrieve：从知识库里召回可能相关的文档片段。
- assemble context：把证据按顺序、长度和去重规则放进 prompt。
- generate：让 LLM 基于上下文回答。
- check grounding/citation：检查回答是否被上下文支持，引用是否对应原文。

RAG 是一种 knowledge grounding pattern，不是 Agent loop。它不要求模型规划多步行动、循环调用工具或自主决定下一步，只要求回答能被外部证据支撑。

## 最小例子

公司政策里有一条：

```text
未休年假只在员工离职结算时可折现。仍在职员工不得将未休年假直接折现。
```

用户问：

```text
我今年还有 5 天年假没休，可以直接折现提现吗？
```

不接知识库时，模型可能根据常识回答“可以咨询 HR 申请折现”。RAG 应该先检索到政策条款，再回答：

```text
如果你仍在职，未休年假不能直接折现；只有离职结算时才可按政策折现。建议确认你的雇佣状态和当地 HR 细则。
引用：未休年假折现政策，第 2 条。
```

这里重点不是答案更长，而是模型明确说出政策边界：在职不行，离职结算才行。

## 原理层

RAG 把“知识存在哪里”和“模型如何表达”分开。外部知识留在文档、数据库或搜索索引中；模型只在当前请求里读取相关片段，并基于片段生成回答。

它和几种方案的边界如下：

| 方案 | 适合解决 | 不适合解决 |
|------|----------|------------|
| RAG | 动态、私有、需要引用的知识接入 | 改变模型基础能力或长期行为习惯 |
| 长上下文 | 单次任务需要阅读大量已知材料 | 每次都把全库塞进 prompt，成本高且容易干扰 |
| 微调 | 输出风格、格式、领域表达和任务适配 | 高频变化的政策、库存、权限和事实 |
| 搜索 | 找到相关文档或网页 | 自动把证据转成受约束的最终回答 |
| 工具调用 | 查询实时状态、执行操作或访问 API | 替代证据组织和忠实回答检查 |

长上下文和 RAG 可以配合：RAG 先缩小候选材料，再把关键片段放进上下文。微调也可以配合：微调让模型更会按公司格式回答，但事实仍从检索证据来。

## 和应用/面试的连接

应用里，RAG 常用于企业问答、客服知识库、合同审阅、产品文档助手和研究材料摘要。设计时要先说明知识更新频率、权限边界、引用要求和失败处理，而不是直接说“用向量数据库”。

面试里，可以这样回答边界题：RAG 解决的是外部知识接入和 grounding，不是把知识永久写进模型；长上下文解决单次输入容量，微调解决行为和格式，搜索解决找文档，Agent 解决多步行动。RAG 可以被 Agent 使用，但 RAG 本身不等于 Agent。

## 常见误区

| 误区 | 更准确的理解 |
|------|--------------|
| RAG 是为了让模型更聪明 | RAG 主要让模型使用外部证据回答 |
| RAG 等于接向量库 | 向量库只是检索实现之一，还需要排序、上下文组装和评估 |
| 长上下文能替代所有 RAG | 全量塞上下文成本高，且相关证据可能被噪声淹没 |
| 微调能解决私有知识更新 | 高频变化知识不适合频繁写进权重 |
| RAG 就是 Agent | RAG 是证据增强生成链路，不要求自主规划和工具循环 |
