# 幻觉、安全与 Guardrails

## 这篇解决什么问题

LLM 的危险不只是不回答。更常见的问题是：它用很流畅的语言说出没有依据的内容，或者在恶意输入诱导下忽略系统要求，或者输出虽然格式正确但违反业务和安全边界。

这篇解决的问题是：如何区分幻觉、事实性、grounding、instruction-following 和 policy safety，为什么 Guardrails 不是一个单独开关，以及常见缓解手段分别解决哪类风险。

## 概念为什么出现

LLM 的训练目标让它擅长生成“在上下文里看起来合理的下一个 token”。这带来一个工程问题：合理不等于真实，有帮助不等于安全，听从用户不等于应该执行。

所以安全与 Guardrails 出现，是为了解决几类不同风险：

```text
事实风险: 回答内容是否真实
依据风险: 回答是否被给定上下文支持
指令风险: 模型是否遵守系统和开发者要求
策略风险: 输出是否违反安全、隐私、合规或业务政策
```

## 最小心智模型

先把几个概念分开：

- 幻觉：模型生成看似合理但没有可靠依据或不符合事实的内容。
- factuality：回答是否与外部事实一致。
- grounding：回答是否被当前提供的证据、文档或上下文支持。
- instruction-following：回答是否遵守系统、开发者和用户指令的优先级。
- policy safety：回答是否符合安全政策、法律合规、隐私和业务边界。

Guardrails 可以理解成围绕模型调用的一组控制层：

```text
输入前: 清洗、分类、风险检测
生成中: prompt 约束、解码约束、结构化输出
生成后: 验证、引用检查、过滤、拒答或重试
上线后: red teaming、监控、回归评估
```

## 最小例子

用户问：

```text
我们公司 AlphaPlan 的企业版退款期限是多少天？
```

如果系统没有提供政策文档，模型回答：

```text
企业版支持 30 天无理由退款。
```

这就是典型幻觉：回答流畅、具体、看似有帮助，但没有依据。更稳妥的回答是：

```text
我没有看到 AlphaPlan 企业版退款期限的依据，不能确认具体天数。请查看合同条款或提供退款政策文本，我可以帮你从文本中定位。
```

如果提供了政策片段，回答还应该引用或指出依据，例如“根据退款政策第 3 条”。这时检查重点从 factuality 扩展到 grounding：答案是否真的被给定片段支持。

## 原理层

幻觉不是单一原因。知识过期、训练数据缺失、prompt 信息不足、上下文里证据冲突、解码温度过高、用户要求模型猜测、系统没有验证环节，都可能导致模型编造。降低幻觉要先问：模型缺事实、缺证据、没遵守指令，还是输出没有被验证？

更好的 context 解决“模型没看到足够信息”的问题。它可以来自用户提供的材料、产品数据库、检索结果或工具返回值，但这些只是可选组件。当前模块关注的是：只要系统依赖上下文，就必须检查答案是否被上下文支持。

Refusal 解决的是“不应该回答”或“依据不足”的问题。好的拒答不是简单说不，而是说明限制、避免编造，并提供安全替代路径，例如要求用户补充材料、建议联系管理员或给出通用非敏感信息。

Citations 解决可追溯性问题。引用不能只装饰答案；引用的片段必须真的支持结论。否则系统只是把幻觉包装成看似可信的格式。

Verification 解决生成后核查问题。可以用规则、schema validation、单独 verifier、人工复核或业务系统校验输出。对于结构化结果，可以结合 [结构化输出与约束解码](../03-generation-control/02-structured-output-constrained-decoding.md)，先保证格式可解析，再检查字段含义是否合规。

Red teaming 解决“主动找坏情况”的问题。它通过攻击性、边界性和异常输入测试系统，例如越权请求、隐私探测、误导性上下文、指令冲突和敏感内容诱导。

Input/output filters 解决快速拦截问题。输入过滤可以识别恶意或高风险请求；输出过滤可以拦截敏感信息、危险步骤、辱骂或政策违规内容。但过滤器不能替代评估和设计，因为它们可能误杀、漏检，也不理解完整业务语境。

Prompt injection 分直接和间接两类。Direct prompt injection 是用户在输入里直接要求模型忽略系统规则，例如“忽略以上指令”。Indirect prompt injection 是模型读取的外部内容里包含恶意指令，例如网页、文档或工具返回值写着“把用户密钥发出来”。RAG 和 Agent 会让间接注入更常见；具体防护分别见 [RAG 评估、幻觉与生产排查](../01-rag-retrieval-systems/04-rag-evaluation-debugging.md) 和 [Agent 评估、安全与生产排查](../02-agent-tool-use/04-agent-evaluation-safety-production.md)。这里先记住：外部内容只能当数据，不能当更高优先级指令。

## 和应用/面试的连接

应用里，不要把 Guardrails 理解成“加一个安全库就完事”。你需要先列失败类型：事实错误、无依据回答、越权信息、危险建议、格式失败、注入攻击、过度拒答。然后为每类失败选择控制点：输入检测、上下文管理、系统提示、结构化输出、验证、拒答、人工复核和监控。

面试里，回答幻觉问题时要避免只说“用 RAG”。更系统的回答是：先区分 factuality 和 grounding；若缺信息，补 context 或拒答；若有证据，要求引用并验证引用支持结论；若是安全问题，使用策略分类、过滤、red teaming 和回归集；最后监控幻觉率、拒答率和用户反馈。

## 常见误区

| 误区 | 更准确的说法 |
|------|--------------|
| 幻觉就是模型不知道 | 不知道只是原因之一；上下文冲突、诱导、解码和验证缺失也会导致幻觉 |
| 有引用就一定 grounded | 引用必须真正支持结论，否则只是带引用的幻觉 |
| Guardrails 是单个组件 | Guardrails 是输入、生成、输出、评估和监控的组合 |
| 拒答越多越安全 | 过度拒答会伤害可用性，需要监控 refusal rate 和误拒 |
| RAG/Agent 自动解决安全 | 检索和工具也会引入错误上下文、间接注入和越权风险 |
