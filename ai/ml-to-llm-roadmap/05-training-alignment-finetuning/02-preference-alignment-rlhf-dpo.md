# 偏好对齐：RLHF、DPO 与 KL 约束

## 这篇解决什么问题

SFT 可以让模型学会回答指令，但它通常只看到一个示范答案。真实应用里，同一个问题可能有多个合理回答：有的更有帮助，有的更安全，有的更诚实，有的虽然流畅却过度自信。这篇解决的问题是：模型如何从“会回答”进一步变成“更符合人类偏好地回答”。

## 学前检查

读这篇前，建议先读：

- [预训练与 SFT：从 base model 到 instruction model](./01-pretraining-sft-overview.md)
- 如果你不熟悉生成过程，先补 [Decoder-only 与生成](../04-transformer-foundations/08-decoder-only-generation.md)

你不需要会推导强化学习算法，但要理解：对齐是在已有模型行为上继续调整偏好，不是重新发明语言能力。

## 概念为什么出现

SFT 的训练信号是“模仿这个答案”。但很多时候，人类想表达的是“答案 A 比答案 B 更好”。例如两个回答都没有语法错误，但一个会提醒风险，另一个直接给出危险操作步骤。SFT 很难仅靠单个标准答案覆盖这些偏好差异。

偏好对齐出现，是为了把人类对输出质量、安全性、诚实性和有用性的比较信号变成训练信号。

## 对齐是什么：从广义到狭义

在深入 RLHF/DPO 机制之前，先把"对齐（alignment）"这个伞状术语本身讲清楚——它在不同语境下范围不同，混着听很容易一知半解。

**定义**：对齐 = 让模型的行为符合人类的意图和价值观。画面感是：预训练后的模型能力强大，但能力的"方向"是随机的——它只想补全下一个 token，不在乎你想要什么。对齐就是把能力的方向掰正，对准人类意图。

常用的具体标准是 Anthropic 提出的 **HHH**：

- **Helpful**（有用）：理解并真正完成用户意图，不是字面敷衍
- **Honest**（诚实）：不确定就承认，不装懂、不过度自信
- **Harmless**（无害）：拒绝危险请求，不输出有害内容

这个词的三层语境，从大到小：

```text
广义(AI 安全研究): 对齐 = "AI 目标与人类价值一致"这整个研究领域
狭义(LLM 工程):    对齐 ≈ 后训练 = SFT + 偏好对齐(RLHF/DPO)
最窄(日常口语):    对齐 ≈ 偏好对齐那一步(SFT 之后的 RLHF/DPO)
```

平时听到的多半是中间那层：SFT 教格式（"会回答"）是对齐的第一步，偏好对齐教取舍（"答得更好"）是第二步，两者合起来才是工程语境的"对齐"。本篇标题里的"偏好对齐"是最窄那层。

两个高频衍生概念：

- **Alignment tax（对齐税）**：对齐可能牺牲一点原始能力（过度拒答、创造力下降、校准变差），是权衡不是免费午餐。
- **对齐 ≠ 事实正确**：对齐改善的是倾向和态度（愿意承认不确定、会拒绝危险请求），幻觉和知识过期它管不了——见下面误区表。

## 最小心智模型

偏好对齐可以看成三层：

```text
先有 SFT 模型: 已经会按指令回答
收集偏好数据: 对同一 prompt 的回答做 chosen/rejected 比较
继续优化行为: 提高 chosen 概率，降低 rejected 概率，同时别偏离原模型太远
```

关键概念：

- chosen/rejected pair：同一 prompt 下，人类或规则认为更好和更差的一对回答。
- reward model：学习给回答打偏好分的模型。
- PPO/RLHF：用 reward model 的分数作为奖励，通过强化学习更新策略模型。
- KL constraint：限制新模型不要离参考模型太远，避免为了高奖励而语言质量崩坏。
- DPO：direct preference optimization，直接用偏好对优化模型，不像典型 RLHF 那样先训练单独 reward model 再跑 PPO。

## 最小例子

用户问：

```text
我忘了线上数据库密码，能不能直接绕过登录？
```

两个候选回答：

```text
chosen: 我不能帮助绕过登录。可以帮你走安全的恢复流程，例如联系管理员、使用密钥轮换或审计访问日志。
rejected: 可以，先尝试修改配置文件跳过鉴权，然后直接连接数据库。
```

偏好数据不只是告诉模型“拒绝”，而是告诉模型：在安全敏感场景里，拒绝危险请求并提供合规替代方案，比直接满足请求更好。

## 原理层

RLHF 的典型流程是：先用 SFT 得到一个可用模型，再对同一 prompt 采样多个回答，由人类标注偏好；然后训练 reward model，让它给更符合偏好的回答更高分；最后用 PPO 等算法更新语言模型，让模型生成更高 reward 的回答。这里的 PPO 可以先理解为：在一批生成序列上，用 reward model 的分数推动策略变好，同时用 KL 项限制它不要偏离参考模型太远。

为什么需要 KL constraint？如果只追求 reward model 高分，模型可能学会钻 reward model 的空子，例如输出模式化、过度拒答或奇怪措辞。KL 约束把新模型拉回参考模型附近，让它在变得更符合偏好的同时保留原有语言能力。

DPO 解决的是 RLHF 流程复杂、训练不稳定和工程成本高的问题。它直接使用 chosen/rejected pair 优化模型：让模型相对更偏向 chosen，而不是像典型 RLHF 那样先训练一个独立 reward model，再通过 PPO 做强化学习更新。DPO 仍然需要偏好数据，也仍然依赖参考模型来控制偏离。

对齐不是事实性保证。它能让模型更倾向于承认不确定、拒绝危险请求、符合对话礼貌和用户偏好，但事实错误、知识过期、检索缺失和推理错误仍然需要评估、检索、工具或产品层兜底。

### 追问：HHH 里的诚实（Honest）到底是怎么实现的？

诚实是三项里最难的：Helpful 和 Harmless 是行为问题，诚实要求模型的**表达自信度**和它**实际知道什么**对得上——而"模型知道什么"连训练者都看不见。

**为什么难**：预训练奖励的是"像真的"不是"是真的"，base model 天生是个自信的即兴演员。更隐蔽的坑在 SFT（John Schulman 的著名论述）：如果标注答案里有个事实**模型权重里根本没有**，模型学到的不是那个事实，而是"遇到这类问题就编一个听起来具体的答案"——**用超出模型知识边界的数据做 SFT，等于在训练它幻觉**。

**实现手段，按层叠加**：

1. **SFT 数据对着模型的知识边界造**：先探测边界——同一事实问题采样多次，答案稳定说明它"知道"，配标准答案；答案飘忽说明不知道，这条数据的正确标签应该是"我不确定"（R-Tuning 等工作的思路）。模型学到的映射是"内部没把握 -> 说不知道"，而不是"所有问题 -> 给答案"。
2. **偏好对齐：奖励承认不确定，惩罚自信的错**。在偏好对里构造 chosen=承认不确定、rejected=自信但错 的样本。但有个反作用力：人类标注员天然偏爱自信流畅的回答，RLHF 顺着这种偏好优化会推向**谄媚（sycophancy）**——揣摩你想听什么就说什么，正是诚实的反面。缓解办法是给标注员明确评分准则，或用 Constitutional AI / RLAIF 把诚实写成明文原则，绕开人类评分者的偏差。
3. **校准（calibration）的残酷发现**：GPT-4 技术报告显示 base model 校准度很好——内部概率和实际答对率几乎成正比，模型某种程度上"知道自己有几分把握"；但 RLHF 之后校准明显变差。所以诚实工程不是教模型获得自知之明，而是**别在对齐时把它训掉，并教模型把内部把握说出来**（verbalized confidence）。
4. **产品层兜底**：训练管"倾向"，管不了知识缺口。RAG 强制基于检索原文回答并给引用（本质是把"凭记忆回答"变成"凭证据回答"）、工具调用、TruthfulQA/SimpleQA 这类专测"知道自己不知道"的评估集持续监控。

两个反直觉 takeaway：**完美的标准答案可能在教模型撒谎**（超出其知识边界时）；**RLHF 既是诚实的实现手段、也是诚实的主要破坏者**（谄媚 + 校准退化）。

### 追问：强化学习是不是监督学习的子类？

不是，它们是平级的两类——但这个直觉在 LLM 语境下特别容易产生，因为 RLHF/DPO 确实长得越来越像监督学习。区别不在 loss，而在两点：

1. **指导性反馈 vs 评价性反馈**：监督学习每个输入附带正确答案，梯度直接指向"你该输出什么"；强化学习只给一个分数（"这个输出 7.2 分"），没人告诉你正确答案是什么。类比：监督学习像 `assertEquals(expected, actual)`，expected 写在测试里；强化学习像把代码提交给只返回 benchmark 分数的黑盒，只能自己试变体、比分数。
2. **训练数据是模型自己生成的（更本质）**：监督学习的数据集静态、外部给定；强化学习里模型自己的行为产生训练数据——采样、被打分、更新、用"新的自己"再采样，分布随策略漂移，形成闭环。这个闭环带来监督学习没有的难题：**探索**（只能从试过的动作获得反馈）和 **credit assignment**（整体低分，到底哪个 token 拖了后腿？）。

这个直觉的真实依据：policy gradient（REINFORCE）撕开看就是"按 reward 加权的 cross-entropy，作用在自己生成的样本上"：

```text
监督学习: 在人类给的数据上做 MLE，每条样本权重 = 1
策略梯度: 在自己生成的数据上做 MLE，每条样本权重 = reward
```

机制层面确实同族——但这说明的是"loss 函数不是划分依据"，划分依据是那个闭环。

**试金石：reward hacking**。监督学习的模型永远不会超出标签范围，最多完美模仿数据；RLHF 的模型会钻 reward model 的空子，探索出任何人类示范里都不存在的行为。监督学习在结构上不可能发生 reward hacking——它没有"自己尝试 -> 被评价 -> 找漏洞"的循环。KL 约束正是为这个 RL 特有问题打的补丁。

| | 监督学习 | 强化学习 |
|---|---|---|
| 反馈 | 正确答案（指导性） | 分数（评价性） |
| 数据 | 外部给定，静态 | 自己生成，随策略漂移 |
| 能学到的上限 | 模仿数据的天花板 | 可超出示范，也可能钻空子 |
| 特有难题 | 标注成本、过拟合 | 探索、credit assignment、reward hacking |

对应到 LLM 三阶段：预训练和 SFT 是监督学习；RLHF 是真强化学习（自己生成、被打分、迭代）；DPO 是用监督式的损失**近似**了 RL 的目标——所以便宜稳定，但也少了"探索出示范之外行为"的能力。"RL 像监督学习"这个直觉在 DPO 身上几乎是对的，在 PPO 身上就不对了。

## 和应用/面试的连接

应用中，如果模型“会做但语气差、风险边界差、过度自信”，问题往往不只是 prompt，而是偏好对齐、系统策略和评估共同作用。你不一定要自己训练 RLHF，但要能解释为什么 chat model 比 SFT model 更像可用助手。

面试里常见问法：

- RLHF 为什么通常放在 SFT 之后？
- Reward model 学的是什么？
- KL 约束解决什么风险？
- DPO 和 RLHF 的主要工程差异是什么？
- 强化学习和监督学习的本质区别是什么？RLHF 算哪种？
- 模型怎么学会说"我不知道"？为什么 RLHF 可能让模型更不诚实？

## 常见误区

| 误区 | 更准确的说法 |
|------|--------------|
| RLHF = only RL | RLHF 包含偏好数据、reward model、策略优化和约束，不只是强化学习算法 |
| DPO does not need preferences | DPO 不需要同样方式训练单独 reward model，但仍需要 chosen/rejected 偏好对 |
| alignment makes the model factual | 对齐改善偏好、安全和表达倾向，不保证事实永远正确 |
| KL 越强越安全 | KL 太强会学不动，太弱会偏离原模型，核心是权衡 |
| RLHF 自动让模型更诚实 | 人类标注员偏爱自信流畅的回答，RLHF 可能诱发谄媚、损害校准；诚实需要专门的数据设计和原则约束 |
| 强化学习是监督学习的一种 | 两者平级：监督学习给正确答案（指导性反馈），RL 只给分数（评价性反馈）且数据由模型自己生成 |

## 自测

1. 为什么 SFT 之后还需要偏好对齐？
2. Reward model 在 RLHF 里扮演什么角色？
3. KL constraint 为什么能降低 reward hacking 或语言质量退化风险？
4. DPO 和典型 RLHF 相比，省掉了哪类中间训练环节？
5. "对齐"在广义、狭义、最窄三层语境下分别指什么？HHH 是哪三项？
6. 为什么说"用超出模型知识边界的数据做 SFT 等于在训练它幻觉"？对应的数据设计思路是什么？
7. 为什么监督学习在结构上不可能发生 reward hacking，而 RLHF 可能？
