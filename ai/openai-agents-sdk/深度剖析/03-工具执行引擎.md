# 03 - 工具执行引擎：从 Python 函数到 JSON Schema 再到执行

> 本文追踪一个 `@function_tool` 装饰的 Python 函数，从定义、注册、Schema 生成、到 LLM 调用后的执行全过程。
> 同时覆盖审批（approval）、guardrail、超时、并行执行等机制。

---

```
开发者定义                注册到 Agent             发送给 LLM              LLM 返回调用            执行
──────────              ──────────              ──────────              ──────────              ──────
```

## 第一步：@function_tool 装饰器

源码：`tool.py:1589-1724`

### 入口

```python
@function_tool
def get_weather(city: str) -> str:
    """查询城市天气"""
    return f"{city}今天晴，25°C"
```

装饰器支持两种用法：
- `@function_tool`（无括号）— `func` 直接传入
- `@function_tool(name_override="weather")`（带括号）— 返回一个装饰器

### 核心：_create_function_tool

```python
def _create_function_tool(the_func):
    # 1. 判断是同步还是异步函数
    is_sync_function_tool = not inspect.iscoroutinefunction(the_func)

    # 2. 从函数签名生成 Schema
    schema = function_schema(
        func=the_func,
        name_override=name_override,
        description_override=description_override,
        docstring_style=docstring_style,
        strict_json_schema=strict_mode,
    )

    # 3. 构建 on_invoke_tool 实现
    async def _on_invoke_tool_impl(ctx, input):
        json_data = _parse_function_tool_json_input(tool_name, input)
        parsed = schema.params_pydantic_model(**json_data)
        args, kwargs = schema.to_call_args(parsed)

        if not is_sync_function_tool:
            if schema.takes_context:
                result = await the_func(ctx, *args, **kwargs)
            else:
                result = await the_func(*args, **kwargs)
        else:
            # 同步函数用 asyncio.to_thread 包装
            if schema.takes_context:
                result = await asyncio.to_thread(the_func, ctx, *args, **kwargs)
            else:
                result = await asyncio.to_thread(the_func, *args, **kwargs)
        return result

    # 4. 用 _build_wrapped_function_tool 构造 FunctionTool
    return _build_wrapped_function_tool(
        name=schema.name,
        description=schema.description or "",
        params_json_schema=schema.params_json_schema,
        invoke_tool_impl=_on_invoke_tool_impl,
        on_handled_error=...,
        strict_json_schema=strict_mode,
        ...
    )
```

**关键洞察**：

1. **同步函数自动包装**：如果你的工具函数是同步的（非 async），SDK 会用 `asyncio.to_thread` 在线程池中执行，**不会阻塞事件循环**。
2. **上下文注入**：如果第一个参数类型是 `RunContextWrapper` 或 `ToolContext`，会自动注入运行上下文。
3. **错误容忍**：`_FailureHandlingFunctionToolInvoker` 包装器会捕获异常，如果配置了 `failure_error_function` 则返回错误消息给 LLM（而不是崩溃）。

## 第二步：function_schema — 从函数签名到 JSON Schema

源码：`function_schema.py:222-400+`

这是最精巧的部分。SDK 需要把一个 Python 函数签名变成 LLM 能理解的 JSON Schema。

### 处理流程

```
Python 函数
    ↓
1. 提取 docstring（自动检测 google/numpy/sphinx 风格）
    ↓
2. 获取 type hints（含 Annotated 元数据）
    ↓
3. 遍历参数，为每个参数生成 Pydantic Field
    ↓
4. 用 pydantic.create_model() 动态创建 Model
    ↓
5. 从 Model 导出 JSON Schema
    ↓
6. 如果 strict_mode=True，调用 ensure_strict_json_schema() 修正
```

### 具体示例

我们的 `get_weather(city: str)` 函数会生成：

```python
FuncSchema(
    name="get_weather",
    description="查询城市天气",
    params_pydantic_model=<动态生成的 Model>,
    params_json_schema={
        "type": "object",
        "properties": {
            "city": {
                "type": "string",
                "description": None  # docstring 中没有参数描述
            }
        },
        "required": ["city"],
        "additionalProperties": False  # strict mode
    },
    signature=<Signature (city: str) -> str>,
    takes_context=False,
    strict_json_schema=True,
)
```

### Docstring 风格自动检测

源码：`function_schema.py:94-131`

```python
def _detect_docstring_style(doc: str) -> DocstringStyle:
    scores = {"sphinx": 0, "numpy": 0, "google": 0}

    # Sphinx: :param xxx:
    # Numpy: Parameters\n---
    # Google: Args:

    max_score = max(scores.values())
    if max_score == 0:
        return "google"  # 默认 Google 风格
```

SDK 会检测 docstring 中的模式来判断风格。如果所有得分都是 0（比如一行简单描述），默认用 Google 风格解析。

### 参数类型处理的特殊情况

| Python 类型 | JSON Schema 处理 |
|-------------|-----------------|
| `str`, `int`, `float`, `bool` | 直接映射 |
| `Optional[X]` | `anyOf: [X_schema, null]` |
| `*args: int` | `list[int]`，默认空列表 |
| `**kwargs: int` | `dict[str, int]`，默认空字典 |
| `Annotated[str, "城市名"]` | 提取 "城市名" 作为 description |
| `Annotated[int, Field(ge=1)]` | 提取 Pydantic FieldInfo |
| `RunContextWrapper` (第一参数) | **跳过**，不加入 schema |
| `ToolContext` (第一参数) | **跳过**，不加入 schema |

### to_call_args：Schema 逆向还原

当 LLM 返回 JSON 参数后，SDK 需要把它还原成函数调用的 args/kwargs：

```python
# function_schema.py:42-74
def to_call_args(self, data: BaseModel) -> tuple[list[Any], dict[str, Any]]:
    positional_args = []
    keyword_args = {}

    for idx, (name, param) in enumerate(self.signature.parameters.items()):
        if self.takes_context and idx == 0:
            continue  # 跳过 context 参数

        value = getattr(data, name, None)

        if param.kind == param.VAR_POSITIONAL:
            positional_args.extend(value or [])
        elif param.kind == param.VAR_KEYWORD:
            keyword_args.update(value or {})
        elif param.kind in (POSITIONAL_ONLY, POSITIONAL_OR_KEYWORD):
            if not seen_var_positional:
                positional_args.append(value)
            else:
                keyword_args[name] = value
        else:
            keyword_args[name] = value

    return positional_args, keyword_args
```

这段代码保证了不管函数用什么参数风格（位置参数、关键字参数、*args、**kwargs），都能正确还原调用。

### 4.2 执行计划（_execute_tool_plan）

源码：`tool_planning.py:518-640`

这是真正的执行入口。**默认并行执行所有工具类型**：

```python
async def _execute_tool_plan(*, plan, agent, hooks, context_wrapper, run_config, parallel=True):
    if parallel:
        (
            (function_results, input_guardrail_results, output_guardrail_results),
            computer_results,
            shell_results,
            apply_patch_results,
            local_shell_results,
        ) = await asyncio.gather(
            execute_function_tool_calls(...),    # 函数工具
            execute_computer_actions(...),       # 电脑操作
            execute_shell_calls(...),            # Shell 调用
            execute_apply_patch_calls(...),      # 代码补丁
            execute_local_shell_calls(...),      # 本地 Shell
        )
    else:
        # 串行执行（parallel=False 时）
        ...
```

> **设计洞察**：不同类型的工具之间是**并行**的（用 `asyncio.gather`），但同类型内部的行为不同：
> - 函数工具：**并行**执行（在 `_FunctionToolBatchExecutor` 内部）
> - Shell/apply_patch/computer：**串行**执行（for 循环 + await）

### 4.3 函数工具执行：_FunctionToolBatchExecutor

源码：`tool_execution.py:1793-1812`

```python
async def execute_function_tool_calls(*, agent, tool_runs, hooks, context_wrapper, config, ...):
    return await _FunctionToolBatchExecutor(
        agent=agent,
        tool_runs=tool_runs,
        ...
    ).execute()
```

`_FunctionToolBatchExecutor` 是一个复杂的执行器，处理：

1. **审批检查**：对每个工具调用检查 `needs_approval`
2. **Tool Input Guardrail**：执行前的输入验证
3. **并行执行**：多个函数工具用 `asyncio.gather` 并发
4. **失败隔离**：如果多个工具并行，一个失败不影响其他（当 `isolate_parallel_failures=True`）
5. **Tool Output Guardrail**：执行后的输出验证
6. **超时控制**：每个工具调用的超时
7. **钩子调用**：执行前后的 hooks

### 4.4 invoke_function_tool — 单个工具调用

源码：`tool.py:1498-1539`

```python
async def invoke_function_tool(*, function_tool, context, arguments):
    invoke_context = _get_function_tool_invoke_context(function_tool, context)

    # 无超时：直接调用
    if function_tool.timeout_seconds is None:
        return await function_tool.on_invoke_tool(invoke_context, arguments)

    # 有超时：用 asyncio.wait_for 包装
    tool_task = asyncio.ensure_future(
        function_tool.on_invoke_tool(invoke_context, arguments)
    )
    try:
        return await asyncio.wait_for(tool_task, timeout=timeout_seconds)
    except asyncio.TimeoutError as exc:
        # 任务已完成（竞态情况）：返回结果或抛出异常
        if tool_task.done() and not tool_task.cancelled():
            tool_exception = tool_task.exception()
            if tool_exception is None:
                return tool_task.result()
            raise tool_exception from None

        # 真正超时
        if function_tool.timeout_behavior == "raise_exception":
            raise ToolTimeoutError(...)

        # "error_as_result"：返回错误消息给 LLM
        return default_tool_timeout_error_message(...)
```

**超时处理的两种策略**：

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `error_as_result`（默认） | 向 LLM 返回超时错误消息 | LLM 可以重试或换策略 |
| `raise_exception` | 抛出异常，终止运行 | 关键操作不能继续 |

## 第五步：工具输出处理

`on_invoke_tool` 返回后，SDK 需要把结果包装成 LLM 能理解的格式。

### 输出类型

工具可以返回：

| 返回类型 | 处理方式 |
|---------|---------|
| `str` | 直接作为文本发给 LLM |
| `ToolOutputText(text="...")` | 文本输出 |
| `ToolOutputImage(image_url="...")` | 图片输出（多模态） |
| `ToolOutputFileContent(file_data="...")` | 文件输出 |
| 其他 | 调用 `str()` 转换 |
| `list[上述类型]` | 多段输出 |

### 结果包装

执行结果最终被包装为 `FunctionToolResult`：

```python
@dataclass
class FunctionToolResult:
    tool: FunctionTool            # 哪个工具
    output: Any                   # 工具返回值
    run_item: RunItem | None      # ToolCallOutputItem（结果项）
    interruptions: list           # 嵌套 agent 的中断（agent-as-tool）
    agent_run_result: Any         # 嵌套 agent 运行结果（agent-as-tool）
```

然后通过 `_build_tool_result_items` 提取 `run_item` 加入 `new_step_items`：

```python
# tool_planning.py:319-338
def _build_tool_result_items(*, function_results, computer_results, ...):
    results = []
    for result in function_results:
        run_item = getattr(result, "run_item", None)
        if isinstance(run_item, RunItemBase):
            results.append(run_item)
    results.extend(computer_results)
    results.extend(shell_results)
    results.extend(apply_patch_results)
    return results
```

## 工具审批（Approval）机制

### 静态审批

```python
@function_tool(needs_approval=True)
def delete_file(path: str) -> str:
    ...
```

当 `needs_approval=True` 时：
1. `execute_tools_and_side_effects` 会创建 `ToolApprovalItem` 而不是执行工具
2. 返回 `NextStepInterruption(interruptions=[...])`
3. 调用方拿到 `RunState`，调用 `run_state.approve(call_id)` 或 `run_state.reject(call_id)`
4. 下次 `Runner.run(run_state)` 时，走 `resolve_interrupted_turn` 路径恢复执行

### 动态审批

```python
async def check_approval(ctx, params, call_id):
    return params.get("path", "").startswith("/critical/")

@function_tool(needs_approval=check_approval)
def delete_file(path: str) -> str:
    ...
```

只有满足条件的调用才需要审批。

## 工具级别的 Guardrail

### Input Guardrail

在工具执行**之前**运行，可以拒绝不安全的输入：

```python
@function_tool(
    tool_input_guardrails=[
        ToolInputGuardrail(guardrail_function=check_input)
    ]
)
def dangerous_tool(command: str) -> str:
    ...
```

如果 guardrail 触发了 `tripwire`，会抛出 `ToolInputGuardrailTripwireTriggered`。

### Output Guardrail

在工具执行**之后**运行，可以过滤/修改输出：

```python
@function_tool(
    tool_output_guardrails=[
        ToolOutputGuardrail(guardrail_function=check_output)
    ]
)
def query_db(sql: str) -> str:
    ...
```

## 并行执行与失败隔离

当一个 turn 中有多个函数工具调用时（例如 LLM 同时调用 `get_weather` 和 `get_time`）：

```python
# tool_planning.py:536-544
isolate_function_tool_failures = len(plan.function_runs) > 1 or (
    parallel and (
        bool(plan.computer_actions) or
        bool(plan.shell_calls) or
        bool(plan.apply_patch_calls) or
        bool(plan.local_shell_calls)
    )
)
```

**隔离策略**：
- 如果有多个函数工具，或者函数工具与其他类型工具并行，开启 `isolate_parallel_failures`
- 开启后，一个工具的失败会被捕获并作为错误消息返回给 LLM，其他工具继续执行
- 如果只有单个函数工具且无其他工具，异常直接抛出（fail-fast）

## tool_choice 重置机制

源码：`tool_execution.py` 中的 `maybe_reset_tool_choice`

当 agent 配置了 `model_settings.tool_choice`（如 `"required"`）时，可能导致 LLM **无限循环调用工具**。SDK 的解决方案：

```python
# 在 AgentToolUseTracker 记录工具使用后
if agent has used tools and tool_choice forces tool use:
    reset tool_choice to "auto"  # 允许 LLM 不调用工具
```

`AgentToolUseTracker`（详见 `tool_use_tracker.py`）记录每个 agent 使用过哪些工具。一旦检测到工具已被使用，就把 `tool_choice` 重置为 `"auto"`，让 LLM 可以选择不调用工具，从而结束循环。

## 工具类型体系总览

```
Tool（抽象基类）
├── FunctionTool      — 用户定义的函数工具
├── FileSearchTool    — hosted 文件搜索
├── WebSearchTool     — hosted 网页搜索
├── CodeInterpreterTool — hosted 代码解释器
├── ImageGenerationTool — hosted 图像生成
├── ComputerTool      — 电脑操作（点击/截图）
├── LocalShellTool    — 本地 Shell
├── ShellTool         — Shell 工具（hosted 或 local）
├── ApplyPatchTool    — 代码补丁

```

只有 `FunctionTool` 和 `MCPServerTool` 通过 `@function_tool` / MCP 协议注册。其余都是框架内置类型，通过 Agent 配置启用。

## 与学习路径伪代码的对比

简化版可能只展示：

```python
schema = generate_json_schema(func)
result = func(**json.loads(arguments))
```

**真实代码多了什么？**

1. **自动 docstring 提取**：3 种风格检测 + 参数描述合并
2. **Annotated 元数据提取**：从 `Annotated[str, "描述"]` 或 `Field(ge=1)` 获取约束
3. **同步函数线程池执行**：`asyncio.to_thread` 防止阻塞
4. **超时控制**：两种策略 + 竞态处理
5. **审批机制**：静态/动态审批 → 中断/恢复流程
6. **工具级 Guardrail**：输入/输出两阶段校验
7. **并行执行与失败隔离**：多工具并发 + 错误不扩散
8. **tool_choice 重置**：防止无限工具调用循环
9. **错误容忍**：`failure_error_function` 捕获异常返回给 LLM

## 小结

工具执行引擎是 SDK 中最"厚"的层之一。它的复杂性来自于需要处理各种边界情况：

1. **注册阶段**：`function_schema` 把任意 Python 函数签名变成 JSON Schema
2. **分类阶段**：`process_model_response` 区分 15+ 种工具类型（第 02 章）
