# 02 - 响应解析与分发：process_model_response 逐行走读

> `process_model_response` 是整个 SDK 最核心的"分类器"。
> LLM 返回的原始 output 是一个异构列表，这个函数负责将每个 output item 归类到正确的执行管道。
> 本文逐行走读其 400+ 行实现，并用 3 个场景追踪完整数据流。

---

## 函数签名与定位

源码：`run_internal/turn_resolution.py:1273-1682`

```python
def process_model_response(
    *,
    agent: Agent[Any],
    all_tools: list[Tool],
    response: ModelResponse,
    output_schema: AgentOutputSchemaBase | None,
    handoffs: list[Handoff],
    existing_items: Sequence[RunItem] | None = None,
) -> ProcessedResponse:
```

**关键点**：这是一个**同步函数**（不是 async）。它只做分类和数据结构转换，不执行任何 I/O。

### 调用位置

```
get_single_step_result_from_response()     ← turn_resolution.py:1700
  └── process_model_response()             ← 分类
  └── tool_use_tracker.record_processed_response()  ← 记录工具使用
  └── execute_tools_and_side_effects()     ← 执行
```

上游在 `run_single_turn()` 拿到 LLM 响应后调用 `get_single_step_result_from_response()`，而 `process_model_response` 是该函数的第一步。

---

## 返回值：ProcessedResponse

源码：`run_internal/run_steps.py:100-131`

```python
@dataclass
class ProcessedResponse:
    new_items: list[RunItem]           # 所有新生成的 RunItem（无论类型）
    handoffs: list[ToolRunHandoff]     # handoff 调用
    functions: list[ToolRunFunction]   # 普通函数工具调用
    computer_actions: list[ToolRunComputerAction]   # 电脑操作工具
    local_shell_calls: list[ToolRunLocalShellCall]  # 本地 shell 调用
    shell_calls: list[ToolRunShellCall]             # shell 工具调用
    apply_patch_calls: list[ToolRunApplyPatchCall]  # apply_patch 工具
    tools_used: list[str]              # 所有工具名称列表（含 hosted tool）
    mcp_approval_requests: list[ToolRunMCPApprovalRequest]  # MCP 审批请求
    interruptions: list[ToolApprovalItem]  # 初始化为 []，后续由 execute_tools_and_side_effects 填充
```

**设计洞察**：`new_items` 包含**所有**输出项，而各类型列表只包含**需要本地执行**的项。例如 hosted MCP 调用会出现在 `new_items` 里（作为 ToolCallItem），但**不会**出现在 `functions` 里——因为它已经在服务端执行过了。

## 初始化阶段（1282-1307）

```python
items: list[RunItem] = []

functions = []
computer_actions = []
local_shell_calls = []
shell_calls = []
apply_patch_calls = []
mcp_approval_requests = []
tools_used: list[str] = []
```

然后构建 4 个查找表：

```python
# 1. Handoff 查找表：tool_name → Handoff 对象
handoff_map = {handoff.tool_name: handoff for handoff in handoffs}

# 2. 函数工具查找表：FunctionToolLookupKey → FunctionTool
function_map = build_function_tool_lookup_map(
    [tool for tool in all_tools if isinstance(tool, FunctionTool)]
)

# 3. 特殊工具：每种最多一个
computer_tool = next((t for t in all_tools if isinstance(t, ComputerTool)), None)
local_shell_tool = next((t for t in all_tools if isinstance(t, LocalShellTool)), None)
shell_tool = next((t for t in all_tools if isinstance(t, ShellTool)), None)
apply_patch_tool = next((t for t in all_tools if isinstance(t, ApplyPatchTool)), None)

# 4. Hosted MCP 服务器映射：server_label → HostedMCPTool
hosted_mcp_server_map = {
    tool.tool_config["server_label"]: tool
    for tool in all_tools
    if isinstance(tool, HostedMCPTool)
}
```

### 工具查找键（FunctionToolLookupKey）

源码：`_tool_identity.py:83-94`

```python
def get_function_tool_lookup_key(tool_name, tool_namespace=None):
    if is_reserved_synthetic_tool_namespace(tool_name, tool_namespace):
        return ("deferred_top_level", tool_name)
    if isinstance(tool_namespace, str) and tool_namespace:
        return ("namespaced", tool_namespace, tool_name)
    return ("bare", tool_name)
```

三种键类型：
| 键类型 | 格式 | 场景 |
|--------|------|------|
| `bare` | `("bare", "get_weather")` | 普通函数工具 |
| `namespaced` | `("namespaced", "ns", "tool")` | 带命名空间的 MCP 工具 |
| `deferred_top_level` | `("deferred_top_level", "name")` | 延迟加载的顶层工具 |

> **为什么不用字符串？** 因为 `namespace.name` 格式会和名字本身包含 `.` 的工具冲突。用 tuple 做键可以保证无碰撞。

## 核心循环：遍历 response.output（1321-1670）

```python
for output in response.output:
    output_type = get_mapping_or_attr(output, "type")
```

`response.output` 是 LLM 返回的原始输出列表。每个元素可能是十几种不同类型之一。这个 for 循环是一个**巨大的类型匹配器（type matcher）**。

下面按照源码中的分支顺序逐一分析。

---

## 分支 1：shell_call（1328-1374）

```python
if output_type == "shell_call":
```

处理 LLM 请求执行 shell 命令的场景。

**数据流**：
1. 将原始 output 序列化为 dict（处理三种输入形式：dict / Pydantic model / 其他对象）
2. 移除 `created_by` 字段（隐私保护）
3. 创建 `ToolCallItem` 加入 `items`
4. 如果没有配置 `shell_tool`，**抛出 ModelBehaviorError**
5. 如果 shell 工具的环境不是 `local`，**跳过本地执行**（hosted shell 已由服务端处理）
6. 如果是 local 但没有 executor，**抛出错误**
7. 创建 `ToolRunShellCall` 加入 `shell_calls`

```python
# 关键判断：hosted vs local
shell_environment = shell_tool.environment
if shell_environment is None or shell_environment["type"] != "local":
    logger.debug("Skipping local shell execution for hosted shell tool")
    continue  # ← hosted shell：只记录 item，不安排本地执行
```

> **这里体现了 hosted 与 local 工具的根本区别**：hosted 工具的输出已经在 response 里了（以 shell_call_output 的形式），所以只需要记录 ToolCallItem；local 工具则需要本地安排执行。

---

### 6l. ResponseCustomToolCall + apply_patch（1556-1612）

两个特殊的 apply_patch 兼容分支：

1. `ResponseCustomToolCall` 且名称匹配 apply_patch → 解析自定义输入，构造伪 apply_patch_call
2. `ResponseFunctionToolCall` 且名称匹配 apply_patch 且**不在 function_map 中** → 解析参数，构造伪 apply_patch_call

> **为什么有这两个分支？** 因为 API 可能以不同格式返回 apply_patch 调用，SDK 需要兼容处理。

## 分支 7：ResponseFunctionToolCall — 最重要的分支（1615-1669）

如果走到这里，output 已经确认是 `ResponseFunctionToolCall` 类型。这是**最常见的场景**：LLM 调用了一个自定义函数工具。

```python
if not isinstance(output, ResponseFunctionToolCall):
    continue  # ← 安全检查，前面 elif 链可能遗漏某种类型

tools_used.append(get_tool_call_trace_name(output) or output.name)
qualified_output_name = get_tool_call_qualified_name(output)
```

### 7a. 判断是否是 Handoff

```python
if qualified_output_name == output.name and output.name in handoff_map:
    items.append(HandoffCallItem(raw_item=output, agent=agent))
    handoff = ToolRunHandoff(
        tool_call=output,
        handoff=handoff_map[output.name],
    )
    run_handoffs.append(handoff)
```

**判断逻辑**：如果 `qualified_name == name`（即没有命名空间），并且名字在 handoff_map 中，则这是一个 handoff 调用。

> **关键设计决策**：Handoff 在 LLM 看来就是一个普通的 function_call。SDK 通过名字匹配来区分它是"真正的工具调用"还是"agent 切换请求"。

### 7b. 普通函数工具

```python
else:
    lookup_key = get_function_tool_lookup_key_for_call(output)
    func_tool = function_map.get(lookup_key)

    if func_tool is None:
        # 特例：LiteLLM 的 json_tool_call
        if output_schema is not None and output.name == "json_tool_call":
            # ...构造特殊的 ToolRunFunction
            continue
        # 找不到工具 → 报错
        raise ModelBehaviorError(f"Tool {name} not found in agent {agent.name}")

    items.append(ToolCallItem(
        raw_item=output, agent=agent,
        description=func_tool.description,
        title=func_tool._mcp_title,
    ))
    functions.append(
        ToolRunFunction(tool_call=output, function_tool=func_tool)
    )
```

这就是我们在第 01 章第一轮循环中的场景：`get_weather` 工具被 LLM 调用。

## 场景走读

### 场景 A：天气助手调用 get_weather

用我们第 01 章的例子。LLM 返回：

```python
response.output = [
    ResponseFunctionToolCall(
        type="function_call",
        name="get_weather",
        call_id="call_abc123",
        arguments='{"city": "北京"}',
    )
]
```

**走读**：

1. 进入循环，`output_type = "function_call"`
2. 字符串匹配的分支（shell_call、apply_patch_call 等）全部跳过
3. isinstance 检查：不是 ResponseOutputMessage、不是 ResponseFileSearchToolCall...
4. 最后命中 `isinstance(output, ResponseFunctionToolCall)` → 进入分支 7
5. `qualified_output_name = "get_weather"` = `output.name`，检查 `handoff_map` → 不在里面
6. 进入 else 分支：`lookup_key = ("bare", "get_weather")`
7. `func_tool = function_map[("bare", "get_weather")]` → 找到 `get_weather` 的 FunctionTool
8. 创建 `ToolCallItem`，创建 `ToolRunFunction`

**结果**：
```python
ProcessedResponse(
    new_items=[ToolCallItem(raw_item=..., agent=天气助手)],
    handoffs=[],
    functions=[ToolRunFunction(tool_call=..., function_tool=get_weather)],
    computer_actions=[],
    # ... 其余全空
    tools_used=["get_weather"],
    interruptions=[],
)
```

### 场景 C：Handoff 到另一个 Agent

假设配置了两个 Agent，agent_a 有一个 handoff 到 agent_b：

```python

agent_a = Agent(name="路由", handoffs=[agent_b])
```

```python
        call_id="call_xyz",
```

1. 命中 `isinstance(output, ResponseFunctionToolCall)` → 分支 7


## 下游：从 ProcessedResponse 到 NextStep

`process_model_response` 完成分类后，控制流回到 `get_single_step_result_from_response`：

```python
# turn_resolution.py:1700-1728
def get_single_step_result_from_response(...):
    # 第一步：分类
    processed_response = process_model_response(...)

    # 第二步：记录工具使用（用于 tool_choice 重置）
    tool_use_tracker.record_processed_response(agent, processed_response)

    # 第三步：流式场景下立即推送 handoff 事件
    if event_queue is not None and processed_response.new_items:
        handoff_items = [item for item in processed_response.new_items
                        if isinstance(item, HandoffCallItem)]
        if handoff_items:
            stream_step_items_to_queue(cast(list[RunItem], handoff_items), event_queue)

    # 第四步：执行工具并确定 NextStep
    return await execute_tools_and_side_effects(
        agent=agent,
        processed_response=processed_response,
        ...
    )
```

`execute_tools_and_side_effects` 的决策树（详见第 01 章和后续章节）：

```
ProcessedResponse
│
├── interruptions 不为空? → NextStepInterruption
├── 有 handoffs? → execute_handoffs() → NextStepHandoff
├── tool_use_behavior 触发? → NextStepFinalOutput
├── 无工具可执行 + 有消息?
│   ├── 有 output_schema? → validate_json() → NextStepFinalOutput
│   └── 无 output_schema? → 文本即输出 → NextStepFinalOutput
└── 其余情况 → NextStepRunAgain
```

## 设计洞察

### 1. Hosted vs Local 的本质区别

SDK 处理两类完全不同的工具：

| | Hosted 工具 | Local 工具 |
|---|---|---|
| **执行位置** | OpenAI 服务端 | 用户机器 |
| **结果来源** | response.output 包含结果 | 需要本地执行后回传 |
| **process_model_response 行为** | 只创建 RunItem | 创建 RunItem + 加入执行列表 |
| **例子** | file_search, web_search, MCP, code_interpreter | FunctionTool, ShellTool, ComputerTool |

### 2. Handoff 伪装成 FunctionCall

Handoff 在 LLM 眼中就是一个普通的 `function_call`（名字如 `transfer_to_xxx`）。SDK 在 `process_model_response` 中通过 `handoff_map` 查找来区分。这种设计的好处是不需要 LLM 支持特殊的 handoff 指令——任何支持 function calling 的模型都能做 handoff。

### 3. 分类与执行的分离

`process_model_response` **只分类不执行**，是一个纯同步函数。这种设计使得：
- 易于测试（输入/输出确定）
- 流式和非流式复用同一个分类器
- `tool_use_tracker` 可以在执行前记录工具使用

### 4. 防御性编程模式

注意两个安全检查：
```python
# 1. 循环末尾的 warning
else:
    logger.warning("Unexpected output type, ignoring: %s", type(output))
    continue

# 2. 循环后的二次检查
if not isinstance(output, ResponseFunctionToolCall):
    continue
```

第 2 个看起来冗余，但它是防止 `elif` 链中某个分支忘记 `continue` 导致 fallthrough 到 FunctionToolCall 处理逻辑。

## 与学习路径伪代码的对比

简化伪代码大概是：

```python
def process_model_response(response):
    for output in response:
        if is_message(output):
            items.append(MessageItem)
        elif is_tool_call(output):
            items.append(ToolCallItem)
            functions.append(ToolRun)
        elif is_handoff(output):
            handoffs.append(Handoff)
    return ProcessedResponse(items, functions, handoffs)
```

**真实代码复杂在哪？**

1. **15+ 种 output 类型**（不是 3 种）
2. **Hosted vs Local 的区分**（hosted 只记录，local 要执行）
3. **工具查找用 tuple 键**（防止命名空间碰撞）
4. **apply_patch 有 3 种入口格式**（原生、CustomToolCall、FunctionToolCall 伪装）
5. **MCP 审批的特殊处理**
6. **shell_call 的 hosted/local 双路径**
7. **LocalShellCall 到 ShellTool 的 fallback**

## 小结

`process_model_response` 是 SDK 的"邮件分拣中心"：

1. 接收 LLM 返回的混合 output 列表
2. 逐项分类为 15+ 种类型
3. 为每种类型创建对应的 RunItem（记录用）
4. 将需要本地执行的项放入各自的执行列表
5. 不执行任何实际工作，只返回分类结果

下游的 `execute_tools_and_side_effects` 拿到这个分类结果后，才开始真正的工具执行、审批检查、handoff 触发等操作——这是下一章的内容。
