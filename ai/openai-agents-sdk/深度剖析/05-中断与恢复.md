# 05 - 中断与恢复：Human-in-the-Loop 的完整实现

> 当工具调用需要人类审批时，SDK 如何暂停一次运行、序列化状态、等待审批、再恢复执行？
> 本文追踪从 `NextStepInterruption` 产生到 `resolve_interrupted_turn` 恢复的完整流程。

---

## 总览：中断-恢复的生命周期

```
Runner.run(agent, "删除所有数据")
    │
    ├── LLM 返回 tool_call(delete_all, {})
    │
    ├── process_model_response → ToolRunFunction
    │
    ├── execute_tools_and_side_effects
    │   ├── 检查 needs_approval → True
    │   ├── 创建 ToolApprovalItem
    │   └── return NextStepInterruption(interruptions=[...])
    │
    ├── 主循环捕获 NextStepInterruption
    │   ├── 构建 RunState（序列化快照）
    │   └── return RunResult(interruptions=[...])
    │
    │   ← 暂停：应用层展示给用户 →
    │   ← 用户决定：approve / reject →
    │
    ├── run_state.approve(item)  # 或 reject
    │
    └── Runner.run(run_state)   # 用 RunState 恢复
        │
        ├── 识别 is_resumed_state = True
        │
        └── resolve_interrupted_turn()
            ├── 重放审批状态
            ├── 执行已批准的工具
            └── return NextStepRunAgain / NextStepHandoff / NextStepFinalOutput
```

## 第一步：中断的产生

### 1.1 needs_approval 触发

当函数工具配置了 `needs_approval=True`：

```python
@function_tool(needs_approval=True)
def delete_all(confirm: bool) -> str:
    """删除所有用户数据"""
    ...
```

在 `_build_plan_for_fresh_turn` → `_execute_tool_plan` → `execute_function_tool_calls` 中，SDK 会检查每个工具调用是否需要审批。

### 1.2 ToolApprovalItem 的创建

```python
ToolApprovalItem(
    agent=agent,
    raw_item=tool_call,           # ResponseFunctionToolCall
    tool_name="delete_all",
    tool_namespace=None,
    tool_lookup_key=("bare", "delete_all"),
)
```

### 1.3 中断收集

源码：`tool_planning.py:291-316`

```python
def _collect_tool_interruptions(*, function_results, shell_results, apply_patch_results):
    interruptions = []

    for result in function_results:
        # 直接的审批中断
        if isinstance(result.run_item, ToolApprovalItem):
            interruptions.append(result.run_item)
        # 嵌套 agent (agent-as-tool) 的中断
        if getattr(result, "interruptions", None):
            interruptions.extend(result.interruptions)
        elif getattr(result, "agent_run_result", None):
            nested = result.agent_run_result.interruptions
            if nested:
                interruptions.extend(nested)

    # Shell 和 apply_patch 也可能需要审批
    for shell_result in shell_results:
        if isinstance(shell_result, ToolApprovalItem):
            interruptions.append(shell_result)
    for ap_result in apply_patch_results:
        if isinstance(ap_result, ToolApprovalItem):
            interruptions.append(ap_result)

    return interruptions
```

> **注意嵌套中断**：如果工具是 agent-as-tool，那个嵌套 agent 内部的审批需求也会冒泡到外层。

### 1.4 返回 NextStepInterruption

源码：`turn_resolution.py:569-579`

```python
if interruptions:
    return SingleStepResult(
        ...
        next_step=NextStepInterruption(interruptions=interruptions),
        processed_response=processed_response,  # ← 保存，恢复时需要
    )
```

**关键**：`processed_response` 被保存在 `SingleStepResult` 中，因为恢复时需要知道原始的 LLM 响应包含了哪些工具调用。

## 第三步：RunState — 序列化快照

源码：`run_state.py:138-260`

### 核心字段

```python
@dataclass
class RunState:
    _current_turn: int                          # 当前轮次
    _current_agent: Agent | None                # 当前 agent
    _original_input: str | list[Any]            # 原始输入
    _model_responses: list[ModelResponse]       # 所有 LLM 响应
    _context: RunContextWrapper | None          # 运行上下文（含审批记录）
    _generated_items: list[RunItem]             # 已生成的 items
    _session_items: list[RunItem]               # session 完整历史
    _max_turns: int                             # 最大轮次
    _current_step: NextStepInterruption | None  # 中断步骤
    _last_processed_response: ProcessedResponse | None  # 最近的分类结果
    _tool_use_tracker_snapshot: dict[str, list[str]]  # 工具使用记录
    _conversation_id: str | None                # 服务端对话 ID
    _previous_response_id: str | None           # 上一个响应 ID
```

### Schema 版本控制

```python
CURRENT_SCHEMA_VERSION = "1.6"
SUPPORTED_SCHEMA_VERSIONS = frozenset(
    {"1.0", "1.1", "1.2", "1.3", "1.4", "1.5", CURRENT_SCHEMA_VERSION}
)
```

`RunState` 支持序列化到 JSON（`to_json()`）和反序列化（`from_json()`），使用版本号确保兼容性。这意味着你可以：
1. 将中断的运行状态存储到数据库
2. 后续用另一个进程加载并恢复

### approve() / reject() API

```python
def approve(self, approval_item, always_approve=False):
    self._context.approve_tool(approval_item, always_approve=always_approve)

def reject(self, approval_item, always_reject=False, *, rejection_message=None):
    self._context.reject_tool(approval_item, always_reject=always_reject, ...)
```

审批/拒绝记录存储在 `RunContextWrapper._approvals` 中，是一个 `dict[str, _ApprovalRecord]`。

`always_approve=True` 会将该工具**永久**标记为已批准（后续调用不再询问）。

## 第四步：恢复执行

### 4.1 用 RunState 调用 Runner.run()

```python
# 应用层代码
result = await Runner.run(agent, "删除所有数据")

if result.interruptions:
    # 展示给用户
    for item in result.interruptions:
        print(f"工具 {item.tool_name} 需要审批")

    # 用户批准
    result.state.approve(result.interruptions[0])

    # 恢复执行
    result = await Runner.run(result.state)
```

### 4.2 AgentRunner.run() 识别恢复模式

```python
async def run(self, starting_agent_or_state, input=None, **kwargs):
    if isinstance(starting_agent_or_state, RunState):
        is_resumed_state = True
        run_state = starting_agent_or_state
        # 从 RunState 恢复所有状态
        current_agent = run_state._current_agent
        original_input = run_state._original_input
        # ...
    else:
        is_resumed_state = False
        # 正常初始化
```

### 4.3 主循环中的恢复分支

```python
# run.py 主循环
while True:
    if is_resumed_state and run_state._current_step is not None:
        # 恢复模式：调用 resolve_interrupted_turn
        step_result = await resolve_interrupted_turn(
            agent=current_agent,
            original_input=original_input,
            original_pre_step_items=generated_items,
            new_response=last_model_response,
            processed_response=last_processed_response,
            hooks=hooks,
            context_wrapper=context_wrapper,
            run_config=run_config,
            run_state=run_state,
        )
        is_resumed_state = False  # 只恢复一次
    else:
        # 正常模式：调用 LLM
        step_result = await run_single_turn(...)
```

## 第五步：resolve_interrupted_turn — 恢复执行的核心

源码：`turn_resolution.py:665-1270`

这是 SDK 中最复杂的函数之一（600+ 行）。它需要处理所有可能的恢复场景。

### 整体结构

```
resolve_interrupted_turn()
│
├── 1. 加载审批状态
│   ├── _pending_approvals_from_state()
│   └── approval_items_by_call_id
│
├── 2. 处理函数工具
│   ├── _select_function_tool_runs_for_resume()
│   │   ├── 已有输出? → 跳过
│   │   ├── 已批准? → 加入执行列表
│   │   ├── 已拒绝? → 记录拒绝消息
│   │   └── 待审批? → 加入 pending_interruptions
│   └── _rebuild_function_runs_from_approvals()
│       └── 从 pending_approval_items 重建工具调用
│
├── 3. 处理 computer/shell/apply_patch
│   ├── _collect_runs_by_approval()
│   └── 同样的 approve/reject/pending 逻辑
│
├── 4. 构建执行计划
│   └── _build_plan_for_resume_turn()
│
├── 5. 执行已批准的工具
│   └── _execute_tool_plan()
│
├── 6. 收集新的中断
│   └── _collect_tool_interruptions()
│
├── 7. 如果还有中断 → 再次返回 NextStepInterruption
│
├── 8. 处理 MCP 审批
│
├── 9. 处理 handoff（如果工具执行后有 handoff）
│
└── 10. 检查是否是最终输出
    └── → NextStepRunAgain / NextStepFinalOutput
```

### 关键细节

#### 函数工具的恢复选择

源码：`tool_planning.py:467-515`

```python
async def _select_function_tool_runs_for_resume(runs, ...):
    selected = []
    for run in runs:
        # 已有输出（上一次中断前执行过的）→ 跳过
        if output_exists_checker(run):
            continue

        # 查询审批状态
        approval_status = context_wrapper.get_approval_status(...)

        # 已拒绝 → 记录拒绝消息
        if approval_status is False:
            await record_rejection(...)
            continue

        # 已批准 → 加入执行列表
        if approval_status is True:
            selected.append(run)
            continue

        # 不需要审批 → 直接执行
        if not requires_approval:
            selected.append(run)
            continue

        # 待审批 → 加入 pending
        if approval_status is None:
            pending_interruption_adder(...)
            continue

    return selected
```

#### 多轮中断

如果第一次恢复后，仍有工具需要审批（比如 LLM 调用了 3 个需要审批的工具，用户只批准了 1 个），`resolve_interrupted_turn` 会再次返回 `NextStepInterruption`：

```python
if pending_interruptions:
    return SingleStepResult(
        ...
        next_step=NextStepInterruption(
            interruptions=[item for item in pending_interruptions if item]
        ),
        processed_response=processed_response,
    )
```

应用层需要再次展示给用户审批，形成循环。

#### 嵌套 Agent 的中断恢复

如果中断来自 agent-as-tool 内部的嵌套 agent：

```python
def _function_output_exists(run):
    # 检查是否有嵌套 agent 的中断
    pending_run_result = peek_agent_tool_run_result(run.tool_call, scope_id=...)
    if pending_run_result and getattr(pending_run_result, "interruptions", None):
        status = _nested_interruptions_status(pending_run_result.interruptions)
        if status in ("approved", "rejected"):
            rerun_function_call_ids.add(call_id)
            return False  # 需要重新执行
        return True  # 仍然有 pending 的嵌套中断
```

## RunState 的序列化/反序列化

### to_json()

```python
run_state.to_json()  # → JSON 字符串
run_state.to_json_dict()  # → Python dict
```

序列化内容包括：
- schema_version
- current_turn, max_turns
- current_agent_name
- original_input
- model_responses（每个响应的 output items + usage）
- generated_items（每个 RunItem → raw_item + type 标记）
- approvals（工具名 → approved/rejected 状态）
- conversation_id, previous_response_id
- tool_use_tracker_snapshot
- context（mapping 类型直接序列化，自定义类型需要 context_serializer）

### from_json()

```python
run_state = RunState.from_json(json_str, starting_agent=agent)
```

**注意**：反序列化需要传入 `starting_agent`，因为 Agent 对象（包含 Python 函数引用）无法被 JSON 序列化。SDK 用 `_build_agent_map(starting_agent)` 通过名字查找来重建 agent 引用。

### 上下文序列化策略

| 上下文类型 | 序列化方式 | 反序列化 |
|-----------|-----------|---------|
| `None` | 不序列化 | `None` |
| `dict` / `Mapping` | 直接存为 JSON | 直接还原 |
| 自定义类型 | 需要 `context_serializer` | 需要 `context_deserializer` |
| 自定义类型（无 serializer） | 存为 `None` + 警告 | 需要 `context_override` |

## 完整场景走读

```python
# 定义需要审批的工具
@function_tool(needs_approval=True)
def transfer_funds(from_account: str, to_account: str, amount: float) -> str:
    return f"已从 {from_account} 转账 {amount} 到 {to_account}"

agent = Agent(name="银行助手", tools=[transfer_funds])
```

### Round 1：初始调用

```python
result = await Runner.run(agent, "帮我从 A 账户转 1000 到 B 账户")
```

1. LLM 返回 `tool_call(transfer_funds, {"from_account": "A", "to_account": "B", "amount": 1000})`
2. `process_model_response` → `ToolRunFunction`
3. `execute_tools_and_side_effects` → 检查 `needs_approval` → True
4. 创建 `ToolApprovalItem`
5. → `NextStepInterruption`
6. 主循环构建 `RunState`，返回 `RunResult`

```python
assert result.final_output is None
assert len(result.interruptions) == 1
assert result.interruptions[0].tool_name == "transfer_funds"
```

### Round 2：用户审批并恢复

```python
result.state.approve(result.interruptions[0])
result = await Runner.run(result.state)
```

1. `Runner.run` 检测到 `isinstance(input, RunState)` → `is_resumed_state = True`
2. 主循环进入恢复分支 → `resolve_interrupted_turn()`
3. `_select_function_tool_runs_for_resume`：
   - `output_exists_checker(transfer_funds)` → False（没有输出）
   - `approval_status` → True（刚刚 approve 过）
   - → 加入 selected
4. `_execute_tool_plan`：执行 `transfer_funds("A", "B", 1000.0)`
5. 结果：`"已从 A 转账 1000.0 到 B"`
6. 无更多中断 → 无 handoff → `NextStepRunAgain`
7. 下一轮：LLM 看到工具结果，返回最终文本
8. → `NextStepFinalOutput`

```python
assert result.final_output is not None
```

### 如果用户拒绝

```python
result.state.reject(result.interruptions[0], rejection_message="不允许此操作")
result = await Runner.run(result.state)
```

1. `resolve_interrupted_turn` 中：
   - `approval_status` → False
   - `_record_function_rejection()` → 创建拒绝输出
   - 拒绝消息 "不允许此操作" 作为 tool_call_output 发给 LLM
2. LLM 收到拒绝消息，可能回复 "好的，已取消操作"

## 设计洞察

### 1. 状态快照，不是暂停点

`RunState` 是一个完整的状态快照（snapshot），不是一个暂停的协程。这意味着：
- 可以序列化到 JSON，存储到数据库
- 可以在不同进程、不同机器上恢复
- 但也意味着需要重建所有运行时状态（agent 引用、tool 函数等）

### 2. 重放而非继续

恢复时，SDK 不是"继续执行"，而是"重放"。它用 `processed_response`（上一次 LLM 的分类结果）重建工具调用列表，然后根据审批状态决定执行、跳过还是拒绝。

### 3. 多级中断

中断可以嵌套（agent-as-tool 内部的审批）和多次（多个工具需要逐一审批）。每次只需要处理一部分审批，未处理的继续作为中断返回。

### 4. 审批的粒度

- `approve(item)` — 批准单次调用
- `approve(item, always_approve=True)` — 永久批准该工具
- `reject(item)` — 拒绝单次调用
- `reject(item, always_reject=True)` — 永久拒绝该工具

永久审批存储在 `_ApprovalRecord.approved: bool | set[str]`（bool 表示全局，set 表示按 call_id）。

---

## 小结

中断与恢复机制由三个核心组件构成：

1. **中断产生**：`needs_approval` 检查 → `ToolApprovalItem` → `NextStepInterruption`
2. **状态保存**：`RunState` 序列化所有运行时状态 → JSON 可存储
3. **恢复执行**：`resolve_interrupted_turn` 根据审批状态重放工具调用

整个机制的设计目标是让中断/恢复对调用方透明——只需要 `approve/reject` + `Runner.run(state)`，SDK 处理所有复杂的状态重建和重放逻辑。
