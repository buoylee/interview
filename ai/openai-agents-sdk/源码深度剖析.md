## 第一章：整体架构——不是简单的 while 循环

### 1.1 入口类关系

```
Runner（静态类，公开 API）
  └── 代理到 DEFAULT_AGENT_RUNNER（AgentRunner 实例）
        ├── run()          → RunResult（非流式）
        ├── run_sync()     → RunResult（同步包装）
        └── run_streamed() → RunResultStreaming（流式）
```

源码位置：`run.py:156-388`

`Runner` 是一个薄代理层，所有方法都转发到 `AgentRunner` 实例：

```python
# run.py:156
class Runner:
    @classmethod
    async def run(cls, starting_agent, input, ...):
        runner = DEFAULT_AGENT_RUNNER  # AgentRunner()
        return await runner.run(...)
```

**设计意图**：允许通过 `set_default_agent_runner()` 替换全局默认行为（实验性 API）。

### 1.2 真实的执行流程（非流式）

```

│
├── 1. 判断是否是恢复状态

│
├── 2. 准备输入

│
├── 3. 初始化 conversation tracker（可选）

│
├── 4. 开启 tracing
│     TraceCtxManager(...)
│
├── 5. 初始化 RunState
│     RunState(context, original_input, starting_agent, max_turns)
│
├── 6. 保存初始输入到 session（如果启用）
│     save_result_to_session(session, input_items, [], run_state)
│
└── 7. 主循环 while True:                     ← run.py:638
      │
      ├── [恢复态] resolve_interrupted_turn()
      │
      ├── get_all_tools()                      ← 获取所有工具（含 MCP 发现）
      ├── initialize_computer_tools()          ← 初始化 computer 工具
      ├── agent_span()                         ← 创建 tracing span
      │
      ├── current_turn++
      ├── if current_turn > max_turns → 错误处理或抛异常
      │
      ├── [第一轮特殊] Input Guardrails
      │     ├── 顺序 guardrails → run_input_guardrails()

      │     └── 并行 guardrails + model_task → asyncio.gather(...)
      │
      ├── [非第一轮] 直接 run_single_turn()
      │
      ├── 处理 turn_result.next_step:
      │     ├── NextStepFinalOutput

      │     │
```

## 第二章：一次完整调用的数据流

### 2.1 run_single_turn —— 一轮的完整流程

源码位置：`run_loop.py:1481-1561`

```python
async def run_single_turn(*, agent, all_tools, original_input, generated_items, ...):
    # 1. Agent Start Hooks（并行执行 run hooks + agent hooks）
    if should_run_agent_start_hooks:
        await asyncio.gather(
            hooks.on_agent_start(agent_hook_context, agent),
            agent.hooks.on_start(agent_hook_context, agent) if agent.hooks else noop,
        )

    # 2. 并行获取 system_prompt 和 prompt
    system_prompt, prompt_config = await asyncio.gather(
        agent.get_system_prompt(context_wrapper),
        agent.get_prompt(context_wrapper),
    )

    # 3. 获取 handoffs 和 output_schema
    output_schema = get_output_schema(agent)
    handoffs = await get_handoffs(agent, context_wrapper)

    # 4. 准备 input（合并 original_input + generated_items）
    input = _prepare_turn_input_items(original_input, generated_items, ...)

    # 5. 调用 LLM
    new_response = await get_new_response(agent, system_prompt, input, ...)

    # 6. 处理响应并执行工具
    return await get_single_step_result_from_response(
        agent, original_input, pre_step_items, new_response, ...
    )
```

### 2.3 input 的组装过程

这是理解整个框架的关键。每轮调用 LLM 时，input 是这样组装的：

```
第 1 轮:
  input = session_history + user_input
       = [历史消息...] + ["hello"]

第 2 轮（tool call 后）:
  input = original_input + generated_items
       = [历史消息...] + ["hello"] + [LLM响应, tool_call, tool_result]

第 3 轮（又一次 tool call 后）:
  input = original_input + generated_items
       = [历史消息...] + ["hello"] + [LLM响应1, tool_call1, tool_result1, LLM响应2, tool_call2, tool_result2]
```

源码位置：`run_loop.py:245-252`


## 第三章：process_model_response——LLM 输出的分类器

源码位置：`turn_resolution.py:1273-1682`

这是整个框架最核心的函数之一（约 400 行）。它做一件事：**把 LLM 的原始输出分类到不同的桶里**。

### 3.1 输入和输出

```python
def process_model_response(
    agent, all_tools, response, output_schema, handoffs, existing_items
) -> ProcessedResponse:
    # 输入：LLM 返回的 response.output（一个列表）
    # 输出：分类后的 ProcessedResponse
```

```python
@dataclass
class ProcessedResponse:
    new_items: list[RunItem]                          # 所有生成的 items
    handoffs: list[ToolRunHandoff]                    # handoff 调用
    functions: list[ToolRunFunction]                  # 函数工具调用
    computer_actions: list[ToolRunComputerAction]      # computer 工具
    local_shell_calls: list[ToolRunLocalShellCall]     # 本地 shell
    shell_calls: list[ToolRunShellCall]                # 远程 shell
    apply_patch_calls: list[ToolRunApplyPatchCall]     # 代码补丁
    tools_used: list[str]                              # 所有用到的工具名
    mcp_approval_requests: list[ToolRunMCPApprovalRequest]  # MCP 审批
    interruptions: list[ToolApprovalItem]               # 等待审批的工具
```

### 3.2 分类逻辑

```
遍历 response.output 中的每一项:
│
├── type == "shell_call"
│     → ToolCallItem + ToolRunShellCall
│
├── type == "shell_call_output"
│     → ToolCallOutputItem（服务端已执行的 shell）
│
├── type == "apply_patch_call"
│     → ToolCallItem + ToolRunApplyPatchCall
│
├── type == "compaction"
│     → CompactionItem（上下文压缩标记）
│
├── type == "tool_search_call"
│     → ToolSearchCallItem
│
├── type == "tool_search_output"
│     → ToolSearchOutputItem
│
├── isinstance(ResponseOutputMessage)
│     → MessageOutputItem（纯文本回复）
│
├── isinstance(ResponseFileSearchToolCall)
│     → ToolCallItem（hosted 工具，服务端执行）
│
├── isinstance(ResponseFunctionWebSearch)
│     → ToolCallItem（hosted 工具，服务端执行）
│
├── isinstance(ResponseReasoningItem)
│     → ReasoningItem（推理过程）
│
├── isinstance(ResponseComputerToolCall)
│     → ToolCallItem + ToolRunComputerAction
│
├── isinstance(McpApprovalRequest)
│     → MCPApprovalRequestItem + ToolRunMCPApprovalRequest
│
├── isinstance(McpCall)
│     → ToolCallItem（hosted MCP，服务端执行）
│
├── isinstance(LocalShellCall)
│     → ToolCallItem + ToolRunLocalShellCall
│
├── isinstance(ResponseFunctionToolCall) ← 最常见
│     ├── name 在 handoff_map 中?
│     │     → HandoffCallItem + ToolRunHandoff
│     └── 否则查 function_map
│           → ToolCallItem + ToolRunFunction
│
└── 其他 → 忽略，打 warning
```

**关键洞察**：

1. **Handoff 和 Function Tool 在 LLM 层面是同一种东西**（都是 `ResponseFunctionToolCall`），框架通过 `handoff_map` 查找来区分
2. **Hosted 工具**（file_search、web_search、MCP call）只生成 `ToolCallItem`，不需要本地执行
3. 函数查找使用 `build_function_tool_lookup_map`，支持 namespace 和别名解析

### 3.3 function_map 的构建

```python
# turn_resolution.py:1293
function_map = build_function_tool_lookup_map(
    [tool for tool in all_tools if isinstance(tool, FunctionTool)]
)
```

这个 map 的 key 不是简单的工具名，而是 `NamedToolLookupKey`，考虑了 namespace：

```
普通工具:    "search_kb" → FunctionTool
带 namespace: ("my_namespace", "search_kb") → FunctionTool
MCP 工具:    ("mcp_server_name", "tool_name") → FunctionTool
```

## 第四章：execute_tools_and_side_effects——状态机的心脏

源码位置：`turn_resolution.py:499-662`

这个函数决定**每轮之后下一步做什么**。

### 4.1 执行流程

```python
async def execute_tools_and_side_effects(
    agent, original_input, pre_step_items, new_response,
    processed_response, output_schema, hooks, context_wrapper, run_config,
) -> SingleStepResult:

    # 1. 构建执行计划
    plan = _build_plan_for_fresh_turn(processed_response, agent, context_wrapper, ...)

    # 2. 收集新的 step items（去重）
    new_step_items = _dedupe_tool_call_items(pre_step_items, processed_response.new_items)

    # 3. 按计划执行所有工具
    (function_results, ..., computer_results, shell_results, ...) = \
        await _execute_tool_plan(plan, agent, hooks, context_wrapper, run_config)

    # 4. 构建工具结果 items
    new_step_items.extend(_build_tool_result_items(function_results, ...))

    # 5. 检查是否有中断（需要审批的工具）
    interruptions = _collect_tool_interruptions(function_results, shell_results, ...)
    if interruptions:
        return SingleStepResult(next_step=NextStepInterruption(interruptions))

    # 6. 检查 handoffs
    if processed_response.handoffs:
        return await execute_handoffs(...)  # → NextStepHandoff

    # 7. 检查 tool_use_behavior（工具结果是否直接作为最终输出）
    tool_final_output = await _maybe_finalize_from_tool_results(...)
    if tool_final_output:
        return tool_final_output  # → NextStepFinalOutput

    # 8. 检查是否有最终文本输出
    if not processed_response.has_tools_or_approvals_to_run():
        if output_schema and not output_schema.is_plain_text():
            final_output = output_schema.validate_json(text)
            return ...  # → NextStepFinalOutput
        if not output_schema or output_schema.is_plain_text():
            return ...  # → NextStepFinalOutput（纯文本）

    # 9. 都不是 → 继续循环
    return SingleStepResult(next_step=NextStepRunAgain())
```

### 4.2 判断优先级（关键！）

```
1. 有 interruptions（工具需要审批）    → NextStepInterruption → 立即返回
2. 有 handoffs（切换 agent）            → NextStepHandoff → 执行 handoff
3. tool_use_behavior 匹配              → NextStepFinalOutput → 工具结果当输出
4. 没有 tool/approval，有文本输出       → NextStepFinalOutput → 最终输出
5. 以上都不满足                        → NextStepRunAgain → 继续循环
```

### 4.3 tool_use_behavior 的 4 种策略

源码位置：`turn_resolution.py:464-496`

```python
async def check_for_final_output_from_tools(agent, tool_results, context_wrapper):
    if agent.tool_use_behavior == "run_llm_again":
        # 默认：工具结果喂回 LLM，LLM 再次生成
        return NOT_FINAL_OUTPUT

    elif agent.tool_use_behavior == "stop_on_first_tool":
        # 第一个工具的输出直接作为最终结果
        return ToolsToFinalOutputResult(is_final_output=True, final_output=tool_results[0].output)

    elif isinstance(agent.tool_use_behavior, dict):
        # 指定工具名列表，匹配的工具输出作为最终结果
        names = agent.tool_use_behavior.get("stop_at_tool_names", [])
        for result in tool_results:
            if result.tool.name in names:
                return ToolsToFinalOutputResult(is_final_output=True, final_output=result.output)
        return NOT_FINAL_OUTPUT

    elif callable(agent.tool_use_behavior):
        # 自定义函数决定
        return await agent.tool_use_behavior(context_wrapper, tool_results)
```

## 第五章：Tool 系统——从函数签名到 JSON Schema

### 5.1 @function_tool 的工作原理

源码位置：`tool.py`、`function_schema.py`

```
用户定义:
  @function_tool
  def search_kb(query: str) -> str:
      """搜索知识库"""
      ...

内部发生了什么:
  1. function_schema(search_kb)          ← function_schema.py
     ├── 解析函数签名（inspect.signature）
     ├── 解析 type hints（get_type_hints）
     ├── 解析 docstring（griffe 库，支持 google/numpy/sphinx 风格）
     ├── 检查第一个参数是否是 RunContextWrapper/ToolContext → takes_context
     ├── 为每个参数创建 Pydantic Field
     ├── 动态创建 Pydantic Model（create_model）
     └── 生成 JSON Schema + ensure_strict_json_schema

  2. 构建 FunctionTool(
       name="search_kb",
       description="搜索知识库",                    ← 从 docstring 提取
       params_json_schema={...},                     ← 自动生成
       on_invoke_tool=<wrapped function>,
       strict_json_schema=True,
     )
```

### 5.2 FunctionTool 的关键字段

```python
@dataclass
class FunctionTool:
    name: str                          # 工具名（LLM 可见）
    description: str                   # 工具描述（LLM 可见）
    params_json_schema: dict           # 参数的 JSON Schema（LLM 可见）
    on_invoke_tool: Callable           # 执行函数（本地）
    strict_json_schema: bool = True    # 严格模式
    is_enabled: bool | Callable = True # 动态启用/禁用
    needs_approval: bool | Callable = False   # 是否需要人工审批
    timeout_seconds: float | None = None      # 超时时间
    tool_input_guardrails: list | None = None # 工具输入护栏
    tool_output_guardrails: list | None = None # 工具输出护栏
```

### 5.3 工具执行流程

```
LLM 返回 tool_call(name="search_kb", arguments='{"query": "AI"}')
│
├── process_model_response 中查找 function_map
│     lookup_key = get_function_tool_lookup_key_for_call(tool_call)
│     func_tool = function_map[lookup_key]
│
├── _build_plan_for_fresh_turn 中检查 needs_approval
│     if needs_approval → 加入 interruptions
│     else → 加入执行计划
│
├── _execute_tool_plan 中执行
│     ├── 检查 tool_input_guardrails（输入护栏）
│     ├── invoke_function_tool(func_tool, context, arguments_json)
│     │     ├── JSON 解析 arguments
│     │     ├── Pydantic 验证参数
│     │     ├── 构建 ToolContext
│     │     ├── 调用 on_invoke_tool(context, arguments_json)
│     │     └── 返回结果字符串
│     ├── 检查 tool_output_guardrails（输出护栏）
│     └── 构建 FunctionToolResult
│
└── 结果变成 ToolCallOutputItem，追加到 new_step_items
```

### 5.5 Agent as Tool 的实现

源码位置：`agent.py:470-873`

```python
def as_tool(self, tool_name, tool_description, ...):
    async def _run_agent_impl(context, input_json):
        # 1. 解析输入
        resolved_input = await resolve_agent_tool_input(...)

        # 2. 检查是否有待恢复的嵌套中断
        pending_run_result = peek_agent_tool_run_result(context.tool_call, ...)
        if pending_run_result and pending_run_result.interruptions:
            # 有中断 → 检查父级审批状态 → 可能恢复执行
            resume_state = pending_run_result.to_state()

        # 3. 执行嵌套 agent
        if on_stream:
            run_result = Runner.run_streamed(self, input=resume_state or resolved_input, ...)
        else:
            run_result = await Runner.run(self, input=resume_state or resolved_input, ...)

        # 4. 如果嵌套 agent 也有中断 → 记录供父级 agent 后续恢复
        if run_result.interruptions:
            record_agent_tool_run_result(context.tool_call, run_result, ...)

        # 5. 提取输出
        if custom_output_extractor:
            return await custom_output_extractor(run_result)
        return run_result.final_output

    return FunctionTool(name=tool_name, on_invoke_tool=_run_agent_impl, ...)
```

**关键区别：Agent as Tool vs Handoff**

| | Handoff | Agent as Tool |
|--|---------|---------------|
| 控制权 | 转移到目标 agent | 始终在主 agent |
| 对话上下文 | 全部传递（或过滤） | 不传递，只传工具输入 |
| 用户感知 | 在和另一个 agent 对话 | 无感，像调工具 |
| 实现 | `process_model_response` 中分类为 handoff | 包装成 `FunctionTool`，嵌套 `Runner.run()` |
| 嵌套中断 | 不适用 | 支持（通过 `agent_tool_state`） |

### 6.2 execute_handoffs 的内部流程

源码位置：`turn_resolution.py:285-461`

```
LLM 返回: function_call(name="transfer_to_billing_agent", arguments='{"reason": "..."}')
│
├── process_model_response:
│     name 在 handoff_map 中 → 分类为 ToolRunHandoff
│
├── execute_tools_and_side_effects:
│     有 handoffs → 调用 execute_handoffs()
│

      │

      │
```

### 6.3 消息历史处理的 3 种方式

```python

```

## 第七章：Guardrail 系统——并行还是阻塞

### 7.1 三种 Guardrail

| 类型 | 执行时机 | 配置方式 |
|------|---------|---------|
| Input Guardrail | 第一轮 agent 执行前/并行 | `agent.input_guardrails` 或 `run_config.input_guardrails` |
| Output Guardrail | agent 产出最终输出后 | `agent.output_guardrails` 或 `run_config.output_guardrails` |
| Tool Guardrail | 工具执行前/后 | `FunctionTool.tool_input_guardrails` / `tool_output_guardrails` |

### 7.2 Input Guardrail 的执行模式

源码位置：`run.py:985-1068`

```python
# 第一轮特殊处理
all_input_guardrails = starting_agent.input_guardrails + (run_config.input_guardrails or [])

# 分为两组:
sequential_guardrails = [g for g in all_input_guardrails if not g.run_in_parallel]
parallel_guardrails = [g for g in all_input_guardrails if g.run_in_parallel]

# 第一步: 顺序 guardrails 先执行（阻塞）
if sequential_guardrails:
    sequential_results = await run_input_guardrails(agent, sequential_guardrails, input, context)
    # 如果 tripwire 触发 → 立即抛异常，不调 LLM

# 第二步: 并行 guardrails 和 LLM 调用同时启动
model_task = asyncio.create_task(run_single_turn(...))

if parallel_guardrails:
    parallel_results, turn_result = await asyncio.gather(
        run_input_guardrails(agent, parallel_guardrails, input, context),
        model_task,
    )
    # 如果 guardrail 先完成且触发 tripwire:
    #   → 取消 model_task
    #   → 抛异常
```

**设计意图**：

- `run_in_parallel=True`（默认）：guardrail 和 LLM 同时跑，不增加延迟
- `run_in_parallel=False`：guardrail 先跑完，通过了才调 LLM

### 7.3 Guardrail 的数据模型

```python
@dataclass
class InputGuardrail(Generic[TContext]):
    guardrail_function: Callable[
        [RunContextWrapper[TContext], Agent, str | list[TResponseInputItem]],
        MaybeAwaitable[GuardrailFunctionOutput],
    ]
    name: str | None = None
    run_in_parallel: bool = True  # ← 控制执行模式

@dataclass
class GuardrailFunctionOutput:
    output_info: Any              # 护栏的详细信息
    tripwire_triggered: bool      # True = 立即中断整个 run
```

### 7.4 重要细节

- Input Guardrails **只在第一轮执行**（`current_turn <= 1`）
- Output Guardrails **每次产出 final_output 都执行**
- Guardrail 触发后，抛出 `InputGuardrailTripwireTriggered` 异常
- 异常的 `run_data` 字段包含完整的运行上下文，方便调试

## 第八章：中断与恢复（Human-in-the-Loop）

### 8.1 中断的触发条件

当工具设置了 `needs_approval=True`：

```python
@function_tool(needs_approval=True)
async def delete_account(user_id: str) -> str:
    """删除用户账号"""
    ...

# 或者条件审批:
@function_tool(needs_approval=lambda ctx, args, call_id: args["amount"] > 10000)
async def transfer_funds(amount: float) -> str: ...
```

### 8.2 中断发生时的内部流程

```
LLM 返回: function_call(name="delete_account", arguments='{"user_id": "123"}')
│
├── process_model_response:
│     → ToolRunFunction(tool_call=..., function_tool=delete_account_tool)
│
├── _build_plan_for_fresh_turn:
│     检查 needs_approval → True
│     → 不加入执行计划，生成 ToolApprovalItem
│
├── execute_tools_and_side_effects:
│     interruptions 非空
│     → 返回 SingleStepResult(next_step=NextStepInterruption(interruptions))
│
├── 主循环:
│     ├── 保存 session
│     ├── 更新 RunState:
│     │     run_state._model_responses = [...]
│     │     run_state._last_processed_response = processed_response
│     │     run_state._generated_items = [...]
│     │     run_state._current_step = NextStepInterruption(...)
│     └── 返回 RunResult(interruptions=[ToolApprovalItem(...)])
│
└── 用户拿到:
      result.interruptions = [ToolApprovalItem(tool_name="delete_account", ...)]
```

### 8.3 恢复流程

```python
# 用户代码
result = await Runner.run(agent, "删除我的账号")

if result.interruptions:
    # 序列化状态
    state = result.to_state()

    # 审批（approve 或 reject）
    state.approve(result.interruptions[0])
    # 或: state.reject(result.interruptions[0])

    # 恢复执行
    result = await Runner.run(agent, state)
```

### 8.4 恢复时的内部流程

源码位置：`turn_resolution.py:665-1270` (`resolve_interrupted_turn`)

```
Runner.run(agent, state)
│
├── isinstance(input, RunState) → is_resumed_state = True
├── run_state = input  （直接用传入的 RunState）
│
├── 主循环:
│     run_state._current_step 是 NextStepInterruption
│     → 进入 resolve_interrupted_turn()
│
└── resolve_interrupted_turn():
      │
      ├── 1. 从 RunState 恢复 pending_approval_items
      │
      ├── 2. 对每个 function_tool_run:
      │     ├── 检查 approval 状态:
      │     │     context.get_approval_status(tool_name, call_id)
      │     │     → True (已批准) / False (已拒绝) / None (待定)
      │     │
      │     ├── 已批准 → 加入执行列表
      │     ├── 已拒绝 → 生成 rejection_item（告诉 LLM "工具被拒绝"）
      │     └── 待定 → 加入 pending_interruptions（再次中断）
      │
      ├── 3. 执行已批准的工具
      │     _execute_tool_plan(...)
      │
      ├── 4. 如果还有 pending_interruptions
      │     → 再次返回 NextStepInterruption
      │
      ├── 5. 检查 handoffs（可能之前被中断的 handoff 也在这里恢复）
      │
      └── 6. 否则 → NextStepRunAgain（继续循环）
```

### 8.5 RunState 的序列化

源码位置：`run_state.py`

```python
@dataclass

    @classmethod
```

### 9.2 start_streaming 的结构

源码位置：`run_loop.py:400-1084`

streaming 版本的主循环和非流式版本**结构一致**，但有以下关键差异：

```
start_streaming()
│
├── 通过 event_queue 派发事件（asyncio.Queue）
│     streamed_result._event_queue.put_nowait(event)
│
├── run_single_turn_streamed()（而不是 run_single_turn）
│     ├── 使用 stream_response_with_retry() 代替 get_response_with_retry()
│     ├── async for event in retry_stream:
│     │     ├── 每个 event 立即放入 event_queue（RawResponsesStreamEvent）
│     │     ├── ResponseOutputItemDoneEvent:
│     │     │     ├── tool_call → emit RunItemStreamEvent(name="tool_called")
│     │     │     ├── reasoning → emit RunItemStreamEvent(name="reasoning_item_created")
│     │     │     └── tool_search → emit RunItemStreamEvent(name="tool_search_called")
│     │     └── ResponseCompletedEvent:
│     │           → 构建 final_response
│     │
│     └── 最后调用 get_single_step_result_from_response()（和非流式相同）
│
├── 增量保存 session（通过 _current_turn_persisted_item_count 跟踪）
│
└── 结束时放入 QueueCompleteSentinel
```

### 9.3 三种 Stream 事件

```python
# 1. 原始 LLM 事件（token 级别）
@dataclass
class RawResponsesStreamEvent:
    data: TResponseStreamEvent    # OpenAI API 的原始事件
    type = "raw_response_event"

# 2. 语义级事件（RunItem）
@dataclass
class RunItemStreamEvent:
    name: Literal[
        "message_output_created",
        "handoff_requested",
        "handoff_occured",           # 注意拼写错误，已知 bug 不能改
        "tool_called",
        "tool_output",
        "reasoning_item_created",
        "mcp_approval_requested",
        ...
    ]
    item: RunItem
    type = "run_item_stream_event"

# 3. Agent 切换事件
@dataclass

    new_agent: Agent
    type = "agent_updated_stream_event"
```

### 9.4 Streaming 的去重机制

`run_single_turn_streamed` 中维护了三个已发送 ID 集合：

```python
emitted_tool_call_ids: set[str] = set()
emitted_reasoning_item_ids: set[str] = set()
emitted_tool_search_fingerprints: set[str] = set()
```

因为：
- Streaming 时，tool_call 在 `ResponseOutputItemDoneEvent` 中就被 emit 了
- 但 `get_single_step_result_from_response` 处理后也会生成这些 items
- 所以需要去重，避免同一个 tool_call 被 emit 两次

## 第十章：Session 持久化——状态在哪存，何时存

### 10.1 Session 接口

```python
class Session(Protocol):
    async def get_items(self) -> list[TResponseInputItem]: ...
    async def add_items(self, items: list[TResponseInputItem]) -> None: ...
```

### 10.2 可用的实现

| 实现 | 适用场景 |
|------|---------|
| `SQLiteSession` | 本地开发 |
| `AsyncSQLiteSession` | 异步本地开发 |
| `RedisSession` | 分布式系统 |
| `SQLAlchemySession` | 生产环境（任何 SQL 数据库） |
| `DaprSession` | 云原生 |
| `EncryptedSession` | 加密存储 |
| `OpenAIConversationsSession` | OpenAI 托管 |

### 10.3 保存时机

```
1. run 开始前:
   prepare_input_with_session()
   ├── session.get_items() → 加载历史
   └── 合并到 prepared_input

2. 第一轮 LLM 调用前:
   save_result_to_session(session, input_items, [], run_state)
   └── 保存用户输入

3. 每轮结束后:
   save_result_to_session(session, [], turn_items, run_state)
   └── 保存本轮生成的 items

4. 特殊情况:
   ├── guardrail 触发: persist_session_items_for_guardrail_trip()
   ├── interruption: 保存到中断点
   └── streaming: 增量保存（跟踪 _current_turn_persisted_item_count）

5. 两种模式互斥:
   ├── session != None && conversation_tracker == None → 本地 session 管理
   └── conversation_tracker != None → 服务端会话管理（不保存本地 session）
```

### 10.4 Server-side Conversation Tracking

当使用 `conversation_id` 或 `previous_response_id` 时，框架切换到服务端会话模式：

```python
if conversation_id or previous_response_id or auto_previous_response_id:
    server_conversation_tracker = OpenAIServerConversationTracker(...)
    # 此模式下:
    # - 不保存到本地 session
    # - input 只发送增量（新的 items）
    # - 服务端维护完整历史
```

## 关键设计模式总结

### 1. 状态机模式
`NextStep` 类型族（`NextStepRunAgain` / `NextStepFinalOutput` / `NextStepHandoff` / `NextStepInterruption`）是一个经典的**代数数据类型**，用 dataclass + isinstance 模式匹配实现。

### 2. 管道/过滤器模式
每轮执行：`input → LLM → process_response → execute_tools → next_step`，每一步都可以通过 hooks 和 filters 拦截。

### 3. 策略模式
`tool_use_behavior` 是一个策略字段，支持字符串常量、字典配置和自定义函数三种方式。

### 4. 观察者模式
Hooks（`RunHooks` / `AgentHooks`）和 Stream Events 都是观察者模式的实现。

### 5. 依赖注入
`RunContextWrapper[TContext]` 是一个泛型容器，用户注入任意上下文对象，工具函数通过类型标注声明依赖。

### 6. 序列化/反序列化
`RunState` 支持 JSON 序列化，带版本号（`CURRENT_SCHEMA_VERSION = "1.6"`），实现跨进程、跨服务的状态传递。
