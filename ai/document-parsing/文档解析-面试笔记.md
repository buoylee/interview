## 0. 开篇:为什么解析是 RAG 的命门

一句话:**RAG 的质量上限,由整条链路里最弱的那一环决定,而那一环通常就是 parser。**

```
文档 → [解析/提取] → 分块 → 嵌入 → 向量库 → 检索 → 重排 → Prompt → LLM → 回答
         ↑ 最容易被忽视、却最容易毁掉一切的一步
```

你的检索、重排、chunk_size、prompt 调得再漂亮,只要 parser 把表格拆烂了、把扫描件读漏了,后面全是 **garbage in → garbage out**。所以面试官特别爱用解析题筛人——一句"你怎么处理 PDF 里的表格 / 扫描件"基本就能分辨有没有真做过 RAG。

**本笔记的定位**
- 不是工具 API 手册,是**建立一套贯穿所有格式的心智模型** + **能在面试里讲清原理和选型**。
- 深度:原理够你解释"为什么",代码只给**最小可用片段**(主线工具配 3-8 行,长尾工具只列名)。

**怎么用这份笔记**
1. **先读第 1 章「统一心智模型」**——这是地基,后面每种格式都在复用它。
2. 后面按文档类型(Word / PDF / Excel / CSV)速查。
3. 每节末的 `▸ 面试速答` 可直接当 flashcard 背。

## 1. 统一心智模型(地基,所有格式通用)

### 1.1 核心认知:文件格式是"为展示而存",不是"为计算而存"

这是整份笔记最重要、也最反直觉的一点。

绝大多数文档格式(尤其 PDF)的设计目标是**「在屏幕/纸上长成某个样子」**,不是**「让程序方便读取数据」**。所以文件里存的往往是:

- 一堆字符,每个带 **(x, y) 坐标**和字体,而不是"段落""标题""表格"这种语义;
- 一些线条、矩形(可能是表格框,也可能只是装饰);
- 图片、字形轮廓……

> **所以"解析/提取"本质是逆向工程**:从"为人眼排版的呈现",反推回"机器能用的结构"(纯文本 + 结构 + 元数据)。
>
> 既然是**推断**,就**不保证 100% 正确**——这是一切解析问题(表格错位、阅读顺序乱、扫描件读漏)的总根源。

记住这句话,后面所有格式的难点都是它的具体表现。

### 1.2 三条路线的光谱:A 纯提取 / B1 流水线视觉 / B2 端到端 VLM

把一页文档变成文字,业界其实只有三种干法,它们是一条**光谱**而不是三个对立阵营:

```
忠实转写像素 ←──────────────────────────────────→ 靠语言先验"理解"
  纯提取        CTC-OCR    TrOCR     GOT-OCR      端到端 VLM
 (pypdf)      (传统OCR)  (序列模型)  (小型VLM)   (GPT-4o/Qwen-VL)
  无幻觉 ──────────────────────────────────────→ 幻觉风险最高
  最便宜快 ────────────────────────────────────→ 最贵最慢、最懂难版面
```

| 路线 | 怎么干 | 代表 | 优势 | 死穴 |
|---|---|---|---|---|
| **A. 纯提取** | 直接读文件里**已有的文字层**(字符+坐标),不看图、不用 AI | pypdf、PyMuPDF、pdfplumber、python-docx、openpyxl | 极快、极便宜、**100% 忠实**(读到啥是啥) | 扫描件/图片型**没有文字层**就完全失效;复杂版面顺序乱 |
| **B1. 流水线视觉** | 把页面当图像,**一串专用小模型**各司其职:版面检测→OCR→表格识别→按阅读顺序拼回 | Unstructured、Docling、MinerU、PaddleOCR | 每步可控、可本地、相对便宜、输出带结构(坐标/类型) | 管线长,某环错会传播;要把多个模型串好 |
| **B2. 端到端 VLM** | 页面图片→**一个多模态大模型**→直接吐 Markdown/HTML | GPT-4o / Claude / Qwen-VL;专用 Nougat、GOT-OCR2.0、olmOCR、Mistral OCR | 复杂版面/表格/手写理解力最强,一步到位 | 贵、慢、**会幻觉**、长文档成本爆炸、不一定给精确坐标 |

**关键:这串工具本身就横跨光谱**。比如 Docling/MinerU 属于 B1,但它们内部也用视觉模型——只是拆成多个"各做一件事"的小模型;而 GOT-OCR2.0 挂着 OCR 的名字,本质已经是个小型 VLM(滑到 B2 那端了)。

### 1.3 生成式 vs 幻觉:为什么 OCR 给乱码、VLM 会"编"

新手常以为"OCR 不生成、VLM 才生成",其实**现代 OCR 的认字环节(CRNN+CTC、TrOCR)技术上也是逐字"预测"出来的**。真正决定"会不会编造"的不是生成与否,而是三件事:

| | 传统 OCR | 端到端 VLM |
|---|---|---|
| 带多少**语言先验** | 几乎没有(只学"像素→字符") | 一整个大语言模型的知识 |
| 一次**看多大范围** | 一个裁剪出来的小条(一行字) | 整页 + 理解语义 |
| **看不清时的行为** | 给乱码/空白/**低置信度分数**(它没能力编) | 用语言先验**脑补**一个通顺但可能错的答案,且通常不给可用置信度 |

> **直觉:幻觉是"聪明"的副作用。**
> - OCR 认字模型 = 一个**不懂字义、只会照着小纸条描字**的人,看不清就空着——他没本事编,因为他不懂上下文。
> - VLM = 一个**读过万卷书的博士**,看整页,模糊处用常识补全,补得很顺——**正因为他懂,才编得出以假乱真的东西**。

实务含义:表格里一个数字印糊了,传统 OCR 给你 `??` 或低分(你能阈值过滤、人工复核);VLM 可能"猜"一个看起来对的数字。**金融/医疗/法律场景,后者更危险**,因为错得很像对的。

### 1.4 决策流程图:拿到一个文件先干什么

```
              一个文件进来
                   │
          先试「提取文字层」
                   │
        ┌──────────┴───────────┐
     有文字                  没文字 / 几乎为空
        │                  (图片型 PDF、扫描件、拍照)
   走「提取」路线              只能走「视觉」路线
   A(便宜、确定)            B1 流水线 / B2 VLM
        │                        │
   表格子问题:              图像预处理 → 版面分析
   规则法(Camelot/         → OCR → 表格识别 → 拼接
   pdfplumber)                  │
   搞不定 → TableFormer     手写/超乱 → 上 VLM(会幻觉)
   再不行 → VLM
```

核心判断只有一条:**先尝试提取文字,提取出来是空的 → 它就是图片型,必须走视觉路线。** 生产管道通常就是这么自动分流的(try 提取,空了 fallback 到 OCR)。

### 1.5 部署形态:本地 / 自托管 / 云 API(设备要求 & 取舍)

"本地还是 API"展开其实是**三种形态**:

1. **本地库**:`pip install` 在你机器上跑。数据不出域,你掌控硬件。
2. **自托管服务**:把本地库包成容器/API,部在自己 GPU 服务器,供团队调用。仍是你的硬件,只是长成 API 的样子。
3. **云 API**:别人托管,按页计费,零运维、不用硬件,但数据出域。

**设备要求——B1 比你想的低很多**(这是它相对"只能调商用 VLM"的核心优势):

| | 显存需求 | 能 CPU 跑? | 笔记本可行? |
|---|---|---|---|
| **B1 本地**(Docling/MinerU/PaddleOCR) | 消费级 GPU 几 GB 就够,**不强制 GPU** | ✅ 可以,只是慢 | ✅ 中等量可行 |
| **B2 本地 VLM**(7B+ 多模态) | 通常要 **16–24GB+** 独显 | ❌ 基本不现实 | ❌ 扛不住 |

**本地 vs 云 API 取舍:**

| 维度 | 本地 / 自托管 | 云 API |
|---|---|---|
| 数据隐私 | ✅ 不出域,合规友好 | ❌ 文档要传给第三方 |
| 成本 | 一次性硬件,量大摊薄便宜 | 按页计费,**量大会贵** |
| 运维 | 要自己管环境/GPU/依赖 | ✅ 零运维 |
| 质量/稳定 | 看你选的模型 | ✅ 通常调校好 |

**选型速记**:数据敏感(企业内部知识库)/ 量大稳定 / 想省钱 → **本地**(轻量首选 Docling,中文用 MinerU);量小或波动大 / 不想养 GPU → **云 API**(Unstructured API、LlamaParse、云厂商 Document AI)。

### ▸ 面试速答(第 1 章)

> **Q1:解析文档,业界有哪几种技术路线?**
> A:三条,是一条光谱。**A 纯提取**——直接读文件已有的文字层(pypdf/pdfplumber),快、准、便宜,但扫描件失效;**B1 流水线视觉**——版面检测+OCR+表格识别等一串专用小模型(Docling/MinerU),可本地、可控;**B2 端到端 VLM**——一个多模态大模型看图直接吐结构化文本,最懂难版面但贵、慢、会幻觉。

> **Q2:为什么 PDF 这么难解析?**
> A:因为 PDF 是"为展示/打印而存"的格式,内部只存带坐标的字符和线条,**不存"段落/表格/单元格"这种语义**。解析本质是从呈现逆向重建结构,是推断,所以不保证 100% 正确。

> **Q3:OCR 和视觉大模型(VLM)解析,本质区别是什么?会不会幻觉?**
> A:区别不在"生成与否"(现代 OCR 也是逐字预测),而在**语言先验的多少**和**看不清时的行为**。OCR 几乎无语言先验、只转写眼前小条,看不清就给乱码/低置信度;VLM 带整个大模型的知识、看整页,模糊处会**脑补**一个通顺但可能错的答案。所以 **VLM 会幻觉,OCR 基本不会**——幻觉是"理解能力"的副作用,敏感数据场景要警惕。

> **Q4:拿到一个 PDF,你的处理流程第一步是什么?**
> A:先**试着提取文字层**。提取得到文字 → 走纯提取路线(便宜);提取为空 → 说明是图片型/扫描件,fallback 到 OCR/视觉路线。

> **Q5:解析服务本地部署对硬件要求高吗?**
> A:看路线。**B1 流水线**用的是小模型,消费级 GPU 几 GB 甚至纯 CPU 就能跑,笔记本都行;**本地跑 7B+ VLM(B2)** 才需要 16–24GB+ 显存的独显。所以数据敏感又想本地化时,B1(Docling/MinerU)是性价比之选。

## 2. Word:doc vs docx

### 2.1 核心区别:一个是"二进制黑盒",一个是"压缩包+XML"

| | `.doc`(2003 及以前) | `.docx`(2007 起) |
|---|---|---|
| 格式本质 | 微软**二进制复合格式**(OLE2 / Compound File) | **Office Open XML**:本质是个 **ZIP 压缩包**,里面是 XML |
| 能直接读吗 | 难,结构不透明,要专门库逆向 | **容易**——`unzip a.docx` 就能看到 `word/document.xml` 等 |
| 解析策略 | 通常**先转换**再解析 | 直接用 `python-docx` 读段落/表格/样式 |

> **一句话**:`.docx` 把内容、结构、样式都摊成 XML 放进 zip,所以好解析;`.doc` 是老二进制黑盒,生产里一般**先转成 docx 或纯文本**(LibreOffice headless / antiword)再处理。

### 2.2 怎么用

```python
from docx import Document          # pip install python-docx(只认 .docx)
doc = Document("a.docx")

text = "\n".join(p.text for p in doc.paragraphs)   # 正文段落

for table in doc.tables:           # ★ Word 表格是结构化的,直接拿行列
    for row in table.rows:
        cells = [c.text for c in row.cells]

# 长尾:docx2txt(一行 dump 纯文本);Unstructured 的 partition_docx(出结构化元素)
# .doc 老格式:libreoffice --headless --convert-to docx a.doc(或 antiword)
```

> **注意一个小坑**:`python-docx` 把段落和表格分成两个集合,按文档**真实先后顺序**遍历需要走 body 的 XML 元素;只读 `paragraphs` 会漏掉夹在中间的表格内容。

### 2.3 和 PDF 的关键对比(面试加分点)

**Word 表格 ≠ PDF 表格**。docx 里表格是带"行/列/单元格"语义的结构化 XML,`doc.tables` 直接给你;而 PDF 表格要从坐标**逆向猜**(见第 3 章)。所以同样是"提取表格",Word 是读结构、PDF 是重建结构,难度差一个数量级。

### ▸ 面试速答(第 2 章)

> **Q1:doc 和 docx 的区别?对解析有什么影响?**
> A:`.doc` 是老的二进制 OLE 复合格式,黑盒难直接解析;`.docx` 是 Office Open XML,**本质是 zip+XML**,解压就能看到结构化内容,好解析。生产里遇到 `.doc` 通常先用 LibreOffice headless 转成 docx/纯文本再处理。

> **Q2:Word 的表格好提取吗?**
> A:好得多。docx 把表格作为**结构化 XML**(行/列/单元格语义都在),`python-docx` 的 `doc.tables` 直接给行列;不像 PDF 要从坐标逆向重建。

> **Q3:用什么库解析 Word?**
> A:`.docx` 用 `python-docx`(读段落、表格、样式),快速 dump 文本用 `docx2txt`,要结构化元素用 Unstructured;`.doc` 先转换(libreoffice / antiword)。

## 3. PDF(含 OCR)— 最难、面试最爱问

### 3.1 PDF 的本质:只存"带坐标的字 + 线",不存结构

PDF 是为"在任何设备上长得一模一样"设计的**打印格式**。内部是一串绘制指令:"在 (x, y) 用某字体画字符 'A'"、"从这画条线到那"。**没有"段落 / 表格 / 阅读顺序"这些语义。**

更要命的是 PDF 分两种,处理方式天差地别:

| 类型 | 怎么来的 | 有没有文字层 | 怎么处理 |
|---|---|---|---|
| **文字层 PDF** | Word/网页等电子导出 | ✅ 有可提取字符 | 纯提取(3.2 / 3.3) |
| **图片型 PDF** | 扫描、拍照 | ❌ 整页是图 | OCR / 视觉(3.4 / 3.5) |

### 3.2 有文字层:纯提取(便宜、确定)

| 工具 | 特点 | 何时用 |
|---|---|---|
| **pypdf** | 纯 Python、轻,基础文本提取(LangChain `PyPDFLoader` 包它) | 只要轻量基本文本 |
| **PyMuPDF (fitz)** | C 底层,**快、质量高**,文本+坐标+图片全拿;`pymupdf4llm` 直接出 Markdown | 生产首选;喂 RAG |
| **pdfplumber** | 基于 pdfminer,**按坐标精细提取 + 抽表格 + 可视化调试**(能画出检测到的线和字框) | 抽表格、调试版面 |

```python
import fitz                      # PyMuPDF,pip install pymupdf
doc = fitz.open("a.pdf")
text = "\n".join(page.get_text() for page in doc)   # 快、质量好

import pymupdf4llm               # 想直接出 Markdown 喂 RAG
md = pymupdf4llm.to_markdown("a.pdf")
```

### 3.3 表格提取:从坐标"逆向重建"出行列

回到 1.1:**PDF 不存表格**,所以抽表是最难的子问题。规则法两招:

| 招式 | 原理 | 适用 |
|---|---|---|
| **lattice(靠框线)** | 检测横竖线 → 围成的格子当单元格 | 表格有**完整边框** |
| **stream(靠对齐)** | 看文字 x 坐标对齐 + 空白间隙猜列 | **无框线**的三线表、财报表 |

```python
import pdfplumber                # 对"有文字层"的 PDF 有效
with pdfplumber.open("a.pdf") as pdf:
    for page in pdf.pages:
        for table in page.extract_tables():   # list[list[cell]]
            ...
# Camelot:flavor="lattice"(有框) / "stream"(无框)
# 长尾:tabula-py(包 Java);复杂表上 Docling 的 TableFormer;再不行 VLM
```

**升级路径**:规则法(Camelot/pdfplumber)搞不定**合并单元格 / 多级表头 / 跨页表 / 无框表** → 上**表格结构模型 TableFormer**(Docling 内置)→ 再不行 → **VLM**。

**喂给 LLM 的格式**:简单表用 **Markdown 表格**;**有合并单元格用 HTML**(`rowspan/colspan` 更忠实);也可把每行转成一句自然语言。

### 3.4 图片型 / 扫描件:怎么判断 + 处理流程

**怎么判断**:试提取文字,空 / 几乎为空 → 它是图片型,必须走 OCR。

```python
if not page.get_text().strip():   # 文字层为空 → fallback 到 OCR
    ...
```

**处理流程(就是 B1 流水线)**:

```
图像预处理        → 版面分析        → OCR 认字      → 表格结构还原 → 按阅读顺序拼回
(去噪/纠偏deskew/    (框出标题/        (Tesseract/      (TableFormer)   (多栏要排对顺序)
 二值化)            正文/表格/图)      PaddleOCR)
```

印刷体清晰扫描:准确率 99%+;手写 / 模糊 / 超乱版面:上 VLM(但注意幻觉,见 1.3)。

### 3.5 OCR / 视觉工具谱

| 形态 | 工具 | 备注 |
|---|---|---|
| **本地开源** | Tesseract(老牌、多语言、CPU)、**PaddleOCR(中文强)**、Surya(较新,版面+OCR+阅读顺序)、Docling(整套,CPU 友好)、**MinerU(整套,中文/公式强,GPU 推荐)** | 数据不出域、可控 |
| **云 API** | Azure Document Intelligence、AWS Textract、Google Document AI、百度/腾讯 OCR、Mathpix(公式) | 零运维、按页计费 |
| **端到端 VLM** | GPT-4o / Claude / Qwen-VL;专用 GOT-OCR2.0、olmOCR、Nougat(论文) | 最难版面、有幻觉 |

```python
from paddleocr import PaddleOCR   # 中文场景首选
ocr = PaddleOCR(lang="ch")
result = ocr.ocr("scan.png")      # 每块:文字 + 坐标 + 置信度(可阈值过滤)
# 想一步到位(版面+OCR+表格):用 Docling 或 MinerU
```

### 3.6 中文场景注意

- **选中文友好的**:PaddleOCR、MinerU;英文向的 Tesseract/Nougat 对中文、竖排、公式容易翻车。
- **暗坑:文字层乱码**。有些中文 PDF 字体子集/编码映射坏掉,**看着有文字层、提取出来却是乱码**——这种虽然"有文字层"也得当图片型走 OCR。排查时:提取出的文字若大量是乱码/方块,就走 OCR。

### ▸ 面试速答(第 3 章)

> **Q1:有文字层的 PDF,表格怎么抽?能准吗?**
> A:规则法两招——**lattice 靠框线、stream 靠文字对齐**,工具 Camelot / pdfplumber。简单规整表能准;复杂表(合并单元格/多级表头/跨页/无框)常错,这时上 **TableFormer** 这类表格结构模型,再不行才上 VLM。喂 LLM 时合并单元格用 **HTML** 最忠实。

> **Q2:怎么判断一个 PDF 是文字版还是扫描件?**
> A:**试提取文字**,空 / 几乎为空就是扫描件(图片型),得走 OCR;有文字就走纯提取。

> **Q3:扫描件 / 图片型 PDF 怎么处理?**
> A:走视觉路线:图像预处理(纠偏/去噪)→ 版面分析 → OCR → 表格识别 → 按阅读顺序拼。本地用 PaddleOCR(中文)/Tesseract,整套用 Docling / MinerU,超难版面上 VLM(注意幻觉)。

> **Q4:pypdf、PyMuPDF、pdfplumber 怎么选?**
> A:要快+质量用 **PyMuPDF**(还有 `pymupdf4llm` 直接出 Markdown);要精细按坐标提取/抽表格/调试用 **pdfplumber**;只要轻量基本文本 **pypdf** 够(LangChain 默认包它)。

> **Q5:中文 PDF 有什么坑?**
> A:① 选中文 OCR(PaddleOCR/MinerU);② 暗坑——有些中文 PDF 字体子集/编码坏,**文字层提取出乱码**,虽"有文字层"也得当扫描件走 OCR。

## 4. Excel(xlsx / xls)

### 4.1 本质:数据是结构化的,坑在"喂给 LLM"

- **`.xlsx`** 和 docx 一样是 **Office Open XML**,本质 zip+XML(每个 sheet 一份 XML);**`.xls`** 是老二进制(BIFF)。
- 所以 Excel **读出来不难**(数据本就是行列结构);难点在**转成 LLM 能用的形态**:多 sheet、合并单元格、多级表头(表头不在第一行)、公式 vs 值、宽表过大。

### 4.2 怎么用

```python
import pandas as pd
df = pd.read_excel("a.xlsx", sheet_name=None)          # None=读所有sheet → dict[name, DataFrame]
df = pd.read_excel("a.xlsx", sheet_name="Sheet1", header=2)   # 表头在第3行时指定

import openpyxl                                          # 要读合并单元格/公式/样式时
wb = openpyxl.load_workbook("a.xlsx", data_only=True)    # data_only=True 取公式算出的值
ws = wb["Sheet1"]
print(ws.merged_cells.ranges)                           # 合并单元格区域
# 长尾:老 .xls 用 xlrd(新版 xlrd 已不支持 xlsx)
```

### 4.3 喂给 LLM 的坑(重点)

| 坑 | 现象 | 处理 |
|---|---|---|
| 多 sheet | 糊成一坨没法区分 | 按 sheet 拆开 + 标注 sheet 名 |
| **合并单元格** | 读出来只有左上角有值,其余是 `None` | **forward-fill 向下/向右填充**,否则 LLM 看不懂 |
| 表头不在第一行 | 前几行是标题/说明 | `header=` 指定,或手动跳过 |
| 公式 vs 值 | 读到 `=A1+B1` 而非结果 | `data_only=True`(注意:从没被 Excel 打开存过的文件可能无缓存值) |
| 宽表/大表 | 整张塞爆 prompt | 转"每行一句自然语言记录"或 Markdown;**超大表别塞 RAG,走 Text-to-SQL / pandas agent** |

### ▸ 面试速答(第 4 章)

> **Q1:Excel 解析难在哪?**
> A:不是读不出(xlsx 是 zip+XML,pandas/openpyxl 直接读),而是给 LLM 用的**结构坑**:多 sheet、合并单元格(只有左上角有值,要 ffill)、表头不在第一行、公式 vs 值、宽表太大。

> **Q2:怎么把 Excel 喂给 LLM / RAG?**
> A:按 sheet 拆、合并单元格 forward-fill、表头对齐,再转成 Markdown 表或"每行一句记录";超大表别塞 RAG,走 **Text-to-SQL / pandas agent**。

> **Q3:pandas 还是 openpyxl?**
> A:要直接拿 DataFrame 做分析用 `pandas.read_excel`;要读合并单元格、公式、样式等底层信息用 `openpyxl`(`data_only=True` 取值)。

## 5. CSV / 纯文本

### 5.1 本质:看似最简单,坑全在边界情况

CSV 是纯文本:行=记录,列用分隔符分开。但"直接 `line.split(",")`"会让你踩一片雷:

- **编码**:UTF-8 / UTF-8 BOM / **GBK**(中文 Windows 导出常见)——乱码的头号根源。
- **分隔符**:不一定是逗号(欧洲常用 `;`,TSV 用 `\t`)。
- **引号转义**:字段内含逗号/换行/引号时,用引号包裹 + 双引号转义(RFC 4180)——`split(",")` 会把一个字段拆碎。
- **大文件**:几 GB 要流式/分块,别一次性 load。

### 5.2 怎么用

```python
import pandas as pd
df = pd.read_csv("a.csv", encoding="utf-8", sep=",")    # 中文Win导出常要 encoding="gbk"
for chunk in pd.read_csv("big.csv", chunksize=10000):   # 大文件流式
    ...

import csv                                               # 标准库:正确处理引号内的逗号/换行
with open("a.csv", newline="", encoding="utf-8") as f:
    for row in csv.reader(f):                            # ★ 别用 line.split(",")!
        ...
# LangChain:CSVLoader → 每行一个 Document(适合"一行一条知识")
```

### ▸ 面试速答(第 5 章)

> **Q1:读 CSV 为什么不能直接 `split(",")`?**
> A:字段内可能含逗号/换行/引号(RFC 4180 用引号包裹 + 双引号转义),`split` 会把一个字段拆碎。要用 `csv` 标准库或 `pandas`。

> **Q2:CSV 常见坑?**
> A:编码(中文 Windows 常是 GBK,要 `encoding="gbk"`)、分隔符不一定是逗号、引号转义、大文件用 `chunksize` 流式。

> **Q3:CSV 怎么进 RAG?**
> A:一行一条知识用 LangChain `CSVLoader`(每行一个 Document);结构化数值查询别硬塞 RAG,走 Text-to-SQL / pandas。

## 6. 落回 RAG / LangChain:解析完之后

> 衔接 [第08章 RAG with LangChain](../langchain/08-rag-with-langchain.md)。

不管前面用什么工具,**解析的终点都是统一成一个标准结构**,接进"分块 → 嵌入 → 向量库"。在 LangChain 里就是 `Document(page_content + metadata)`。

**关键认知:你不必用 LangChain 自带的 DocumentLoader。** 它们大多是第三方库的**薄包装**(`PyPDFLoader` 包 pypdf,`WebBaseLoader` 包 requests+BeautifulSoup,`Unstructured*Loader` 包 unstructured),价值只在"统一成 Document 接口"。生产里常**自建解析管道**(PyMuPDF / Docling / MinerU 等),再手动包成 Document:

```python
from langchain_core.documents import Document
# 任何解析器的输出,手动包成 Document 就能接进 LangChain 后续链路
docs = [Document(page_content=text, metadata={"source": "a.pdf", "page": i})
        for i, text in enumerate(pages)]
# → splitter.split_documents(docs) → embeddings → vectorstore(见第08章)
```

**metadata 别偷懒**:`source`、`page`/`sheet`、元素类型(标题/表格)、必要时坐标——后面做**检索过滤**和**引用溯源**全靠它。

### ▸ 面试速答(第 6 章)

> **Q1:必须用 LangChain 的 DocumentLoader 吗?**
> A:不必。它们多是第三方库的**薄包装**,价值只在统一成 `Document` 接口。生产常自建解析管道(PyMuPDF/Docling/MinerU),输出手动包成 `Document(page_content + metadata)` 即可接进后续分块/嵌入。

> **Q2:解析阶段要存什么 metadata?**
> A:`source`、`page`/`sheet`、元素类型(标题/表格)、必要时坐标——给检索过滤和引用溯源用。
