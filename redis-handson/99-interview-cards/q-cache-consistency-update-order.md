# 缓存和数据库怎么保证一致？更新时先删缓存还是先更库？

## 一句话回答

用 **Cache-Aside + 先更 DB 再删缓存**，做到**最终一致**。「先删缓存再更库」有脏读窗口（读线程会把旧值回填）；补救用**延迟双删**，兜底用 **binlog 订阅删缓存**。要强一致就别用缓存。

## 四种顺序对比

| 策略 | 问题 | 评价 |
|---|---|---|
| 先更 DB，再删缓存 | 删缓存前极短窗口读到旧值；删失败才长期脏 | **推荐**，窗口最小 |
| 先删缓存，再更 DB | 读线程在「删缓存」与「更 DB」间 miss → 读旧 DB → 回填旧值 → 缓存长期旧 | 危险（sc04 实测脏读） |
| 先更缓存（而非删） | 并发写乱序覆盖；算出的值未必被读到（浪费） | 宁删勿更 |
| 先删缓存 + 延迟双删 | 更库后延迟再删一次，清掉被回填的旧值 | 「先删」的补丁，延迟难精确估 |

## 易追问的延伸

- **延迟双删的「延迟」为什么必须有？** 要等并发读把旧值回填完再删；延迟 < 回填耗时就盖不住。难精确，所以 binlog 兜底更可靠。
- **为什么删而不是更新缓存？** 更新有并发乱序覆盖风险，且写多读少时白算；删除让下次读按需回填。
- **删缓存失败怎么办？** 重试队列 / 订阅 DB binlog（Canal）异步补删。
- **能强一致吗？** 缓存天然最终一致；要强一致得读写都走缓存（Write-Through）或干脆不用缓存。
