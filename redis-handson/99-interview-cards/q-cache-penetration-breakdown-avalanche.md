# 缓存穿透 / 击穿 / 雪崩 是什么，怎么防？

## 一句话回答

三者都是「请求穿过缓存砸到 DB」：**穿透**=查 DB 也不存在的 key；**击穿**=单个热点 key 失效瞬间惊群；**雪崩**=大批 key 同时失效。分别用 **布隆/缓存空值**、**互斥重建/逻辑过期/永不过期**、**TTL 随机抖动/多级缓存/熔断** 防。

## 区分 + 解法表

| | 成因 | 主解法 | 实测证据 |
|---|---|---|---|
| 穿透 | key 在 DB 也不存在，缓存永远 miss | 布隆过滤器前置（零假阴性，挡得准）/ 缓存空值短 TTL | [sc01](../06-caching-patterns/scenarios/01-penetration-bloom.md)：假阴性=0、假阳性 71/300 |
| 击穿 | 单个热 key 失效瞬间大量并发回源 | `SET NX` 互斥重建 / 逻辑过期 / 热点永不过期 | [sc02](../06-caching-patterns/scenarios/02-breakdown-mutex-rebuild.md)：无锁=10 → 互斥=1 |
| 雪崩 | 大批 key 同一时刻失效 | TTL 加随机抖动 / 多级缓存 / 限流熔断 | [sc03](../06-caching-patterns/scenarios/03-avalanche-ttl-jitter.md)：1 个到期秒 → 100 个 |

## 易追问的延伸

- **布隆假阳性会怎样？** 放个别不存在的 key 到 DB（可接受），但**零假阴性**保证不漏真实 key——这是它能安全挡穿透的前提。要支持删除用 Counting Bloom / 布谷鸟过滤器。
- **击穿三解法怎么选？** 互斥=强一致但其余请求要等；逻辑过期=不阻塞但可能短暂返回旧值；永不过期=最简单、靠后台刷新。
- **互斥锁为什么要带 EX？** 防重建方崩溃后锁永不释放（引出分布式锁，见 07 章）。
- **雪崩只能靠抖动吗？** 抖动是第一道；还要多级缓存（本地+Redis）和 DB 前限流/熔断兜底。
