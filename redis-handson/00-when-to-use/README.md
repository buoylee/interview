# Redis 选型 · 什么时候该用、什么时候别用(全 repo 导读)

> 这一章回答的**不是**「Redis 怎么用」,而是「**面对一个需求,要不要上 Redis、用哪种姿势、代价是什么、什么时候它是错的选择**」。
>
> 1～13 章是 **bottom-up**:把每个特性内部怎么运作挖到见底。这一章是缺的那条 **top-down** 轴:从「问题」出发反推「选型」。面试官问「**你们为什么用 Redis**」「**这个场景要不要上 Redis**」,考的就是这一章——而这恰恰是最容易答虚的地方:你内部懂得很深,一到「该不该用」就接不上,因为这层从没被单独组织过。
>
> 场景目录里每一条都会**链回对应章节的实机 scenario** 当证据,top-down 的判断扣回 bottom-up 的原理,整个 repo 就闭环了。

---

## 0. 先把 Redis 的本质定死(一切判断的地基)

> **Redis = 一个「内存中的 · 单线程的 · 带丰富数据结构的 · 可选持久化的」远程数据存储。**

拆开看,它只给你三样别人难同时给的东西:

| 它的本质 | 给你的能力 | 代价 |
|---|---|---|
| **内存中** | 内存级速度(微秒~毫秒) | 贵(内存)、容量有限、**默认可丢** |
| **远程的** | 多个应用实例**共享**同一份状态 | 多一次网络往返、多一个依赖 |
| **带数据结构** | zset/hash/bitmap/HLL/GEO/Stream 开箱即用 | —— |

> **核心判据(背下来):你该不该用 Redis,就看你的问题需不需要〔内存级速度 + 跨实例共享 + 现成高级结构〕里的至少一个,以及你愿不愿意付〔可能丢数据 + 多一个有状态依赖 + 一致性复杂度〕的代价。**
>
> 架构师的水平,体现在能清楚讲出**后半句**——尤其是「什么时候不该用」。只会讲「Redis 快所以用」的是初级。

## 1. 决策树:要不要引入 Redis?

面对一个需求,按这棵树走一遍,答案基本就出来了:

```
需求来了
  │
  ├─ 这份数据需要「跨进程 / 跨实例」共享吗?
  │     否 ─────────────► 本地缓存(Caffeine/Guava)就够,别上 Redis(省一个依赖+一次网络)
  │     是
  │     ▼
  ├─ 现有数据库(加索引/读库)扛不住,或做不到吗?
  │     做得到且够快 ──► 先别加缓存(过早缓存=白白引入一致性复杂度)
  │     扛不住 / 做不到
  │     ▼
  ├─ 能容忍「可能丢一点 / 最终一致」吗?
  │     不能,要强一致、不能丢 ──► Redis 不是主库,别当 single source of truth(选 DB;协调类选 etcd/ZK)
  │     能容忍
  │     ▼
  ├─ 数据量会撑爆内存预算吗?(海量冷数据?)
  │     会 ──► 内存太贵,不适合(冷数据放 DB/对象存储,Redis 只放热的)
  │     不会
  │     ▼
  └─ 是不是「读多写少 / 需要 zset·HLL·bitmap 这种结构 / 需要内存级延迟」?
        是 ──► ✅ 这就是 Redis 的主场,往下看「场景目录」对号入座
```

**三个最常见的「停手」信号**(面试能主动说出来就是加分):
1. 数据不跨实例 → 本地缓存,不需要 Redis。
2. 不能丢 / 要强一致 → Redis 不背这个锅(它默认 AP、可丢)。
3. 海量冷数据 → 内存成本爆炸,放错地方。

## 2. 场景目录:Redis 的看家本领

每条 = *为何适合 / 代价或坑 / 什么时候别用 / 替代品 / 链回章节*。这些是「Redis 一拿出来就对」的场景,也是面试报得出来才显得「真用过」。

| 场景 | 为何适合 Redis | 代价 / 坑 | 什么时候别用 | 链回 |
|---|---|---|---|---|
| **对象/查询结果缓存** | 内存级读 + 跨实例共享,挡住 DB 读压力 | 一致性(双写/删除时序)、穿透击穿雪崩 | 数据不跨实例→本地缓存;命中率低→没收益 | [`06-caching-patterns`](../06-caching-patterns/) |
| **Session / Token 共享** | 多实例无状态化的前提:会话集中存 | Redis 挂=全员掉线→要高可用 | 单机服务、或用 JWT 无状态方案 | [`04`](../04-expiry-eviction/) TTL |
| **排行榜 / Top-N** | `zset` 按 score 排序天生就是干这个的 | 大 key(榜太长)、热 key | 排序需求复杂(多维)→DB/ES | [`02`](../02-data-structures/) zset |
| **计数器 / 点赞 / 库存** | `INCR` 原子、单线程无锁、极快 | 要落库对账(内存可丢) | 强一致扣减(钱)→DB 事务兜底 | [`02`](../02-data-structures/) / [`08`](../08-transactions-scripting/) |
| **限流** | 原子计数 + Lua,天然适合令牌桶/滑窗 | 分布式限流要 Redis+Lua 保原子 | 单机限流→本地 RateLimiter 即可 | [`13-rate-limiting`](../13-rate-limiting/) |
| **去重 / UV 统计** | `HyperLogLog`(省内存估算)/ `bitmap` / `set` | HLL 有误差(~0.81%) | 要精确且量小→`set`;要明细→DB | [`03`](../03-advanced-types/) |
| **签到 / 活跃位图** | `bitmap` 一个 bit 一天,极省内存 | key 设计(按用户/按天) | —— | [`03`](../03-advanced-types/) |
| **附近的人 / 地理** | `GEO`(底层 zset)算距离、范围 | —— | 复杂地理查询→PostGIS | [`03`](../03-advanced-types/) |
| **分布式锁** | `SET NX` + Lua,快、简单 | 故障转移可能丢锁(AP)、误删 | 要强一致锁/选主→etcd/ZK(CP) | [`07-distributed-locks`](../07-distributed-locks/) |
| **消息队列 / 延迟队列** | `Stream`(消费组)/ `zset`(score=到期时间) | 弱于专业 MQ:堆积、重放、吞吐有限 | 高吞吐/持久/严格顺序→Kafka | [`09-pubsub-streams-mq`](../09-pubsub-streams-mq/) |
| **Feed 流 / 时间线** | `list`/`zset` 存最近 N 条,推拉结合 | 大 V 扇出写放大 | 超大规模→专门的 feed 系统 | [`02`](../02-data-structures/) |
| **分布式 ID** | `INCR` / `INCRBY` 原子发号,简单 | 依赖 Redis 可用性 | 要趋势递增+高可用→雪花算法 | [`02`](../02-data-structures/) |

> 注意一个共性:**几乎每个场景的「代价」最后都指向同一件事——Redis 是一个会丢、会挂、会变慢的外部依赖。** 这就把选型问题接回了「韧性」(见 `system-design/01-韧性`):用了它,就得为它的失效做兜底。

## 3. Redis vs 替代品:边界画在哪

「该不该用 Redis」常常等价于「Redis vs 另一个东西,选谁」。把这几条边界讲清,选型题就稳了。

### vs 本地缓存(Caffeine / Guava Cache)— 最容易用错的一组
- **本地缓存赢**:数据不需跨实例共享、量小、要纳秒级、能容忍各实例略有不一致。**很多人一上来就上 Redis,其实本地缓存就够,白白多一个依赖和一次网络往返。**
- **Redis 赢**:要跨实例一致、容量大、要持久/可观测。
- **常见最优解 = 多级缓存**:本地(L1,挡大部分)+ Redis(L2,跨实例兜底)。面试能说出「为什么要两级」很值钱。

### vs Memcached
- Memcached:纯 KV、多线程、极简,大对象缓存吞吐略有优势。
- Redis:有数据结构 + 持久化 + 复制 + 高级功能。**绝大多数新项目直接选 Redis**,Memcached 基本是历史包袱场景才碰。

### vs Kafka / RabbitMQ(Stream 当 MQ 的边界)
- **Redis Stream 够用**:量不大、要轻量、不想多引一套中间件、能接受相对弱的保证。
- **必须上 Kafka/RabbitMQ**:高吞吐、消息要持久化长期保留、要重放、要严格分区顺序、要海量堆积不丢。
- 一句话:**Redis 做 MQ 是「顺手」,不是「专业」**;量级上来必须换专业 MQ。

### vs Zookeeper / etcd(分布式锁 · 选主 · 配置)
- Redis 锁:**快,但是 AP** —— 主从故障转移瞬间,锁可能丢(见 [`07/03-failover-lock-loss`](../07-distributed-locks/))。适合「锁丢了顶多重复做一次、能容忍」的场景。
- ZK/etcd:**CP** —— 强一致,适合选主、配置下发、不能容忍双重持有的锁。RedLock 的争议(见 [`q-redlock-controversy`](../99-interview-cards/q-redlock-controversy.md))本质就是「能不能用 AP 系统造一个声称强一致的锁」。

### vs 主数据库(MySQL/PG)
- **Redis 不是主库,别当 single source of truth。** 它默认可丢数据(持久化也只是「尽量少丢」)、内存贵、没有复杂查询/范围扫描/多表 join/可回滚事务。
- 正确姿势:**DB 是真相,Redis 是 DB 前面的加速层 / 派生视图**。

### vs Elasticsearch
- 复杂检索、全文搜索、多条件聚合 → ES。
- Redis 只擅长「精确 key 取值」和「特定结构操作(排序/计数/位)」。**别拿 Redis 硬做检索。**

## 4. 什么时候「不该用」Redis(架构师段位)🔬

会讲「什么时候用」是中级,会讲「什么时候**不**用」才是架构师。这张反模式清单背下来:

| 反模式 | 为什么错 | 该用什么 |
|---|---|---|
| 数据不跨实例也上 Redis | 凭空多一个依赖 + 一次网络往返 | 本地缓存 |
| 把 Redis 当主库(强一致、不能丢) | 它默认可丢,持久化也只是「尽量」 | DB 当真相,Redis 当加速层 |
| 海量冷数据塞 Redis | 内存成本爆炸 | 冷数据进 DB/对象存储,只缓存热的 |
| 用 Redis 做复杂查询/检索 | 没有范围/聚合/全文能力 | DB / ES |
| 还没有性能问题就加缓存(过早缓存) | 白白引入一致性复杂度,收益为零 | 先压测,有瓶颈再加 |
| 用 `pub/sub` 当可靠 MQ | pub/sub 消息**会丢**(无持久、无确认) | Redis Stream 或专业 MQ([`09`](../09-pubsub-streams-mq/)) |
| 把 Redis 锁当强一致锁用 | 故障转移会丢锁(AP) | etcd/ZK,或接受「可能重复执行」 |

## 5. 引入 Redis 的「代价」(讲给面试官听的成本意识)

架构师不会说「快就上」,而是把**收益和代价放天平**。引入 Redis 的代价有四笔,每笔都接回这个 repo 别的章节:

1. **多一个有状态的网络依赖 → 新的故障模式**。Redis 挂了/变慢,你的服务跟着出事。这正是缓存穿透/击穿/雪崩的来源,要用熔断/限流/降级/本地兜底防住(接 [`06`](../06-caching-patterns/) + `system-design/01-韧性`)。
2. **缓存一致性复杂度**。DB 和缓存两份数据 = 双源真相,双写时序、删除策略都是坑(接 [`06/04-consistency-delete-order`](../06-caching-patterns/))。
3. **运维负担**。主从/哨兵/集群的部署与故障转移、内存监控、大 key / 热 key 治理(接 [`10`](../10-replication-sentinel/) / [`11`](../11-cluster/) / [`12`](../12-production-ops/))。
4. **数据可丢的心智负担**。任何放进 Redis 的「真相」都要想清楚:它没了我能不能从 DB 重建?不能重建的东西别只放 Redis。

## 6. 面试话术模板(把判断讲出来)

### Q「你们为什么用 Redis?」—— 别只说「快」
满分结构 = **场景 + 收益 + 代价 + 兜底**,四件套:
> 「我们的 ___ 场景是〔读多写少 / 需跨实例共享 / 需要 zset 这种结构〕,直接打 DB〔扛不住 / 做不到〕,所以用 Redis 做〔缓存 / 排行榜 / 限流〕。代价是〔一致性 / 多一个依赖〕,我们用〔删除时序 + TTL 加抖动 + 熔断降级〕兜底。」

只要能把后半句「代价 + 兜底」讲出来,层次立刻和「就是快」拉开。

### Q「这个场景要不要上 Redis?」
当场走**第 1 节的决策树**:跨实例共享吗?DB 扛得住吗?能容忍最终一致吗?会爆内存吗?—— 给出**判断 + 理由**,而不是直接「上/不上」。

### Q「Redis 挂了会怎样?」
回韧性:缓存失效 → 请求穿透打到 DB → DB 被打垮 → 雪崩。**所以要有**:多级缓存兜底、熔断快速失败、限流保 DB、降级返回默认值(接 `system-design/01-韧性` + [`06`](../06-caching-patterns/))。能把「一个缓存依赖的失效如何级联」讲清,就是资深。

## 一句话总结

> Redis 是「**内存里的共享数据结构服务器**」。
> **该用它**,是因为你需要〔内存级速度 + 跨实例共享 + 现成高级结构〕中的至少一个;
> **慎用/别用它**,是因为你要付〔可能丢数据 + 多一个有状态依赖 + 一致性复杂度〕的代价。
> 初级讲得出前半句,**架构师讲得清后半句、尤其是「什么时候不该用」**——这一章就是把那半句给你装上。
