# 事务与脚本（Transactions & Scripting）

## 1. 核心问题

Redis 怎么把「多个操作」做成一个不可分割的单元？`MULTI/EXEC` 是不是「事务」？`WATCH` 怎么做乐观锁？为什么很多场景宁可用 **Lua** 也不用 `MULTI`？`pipeline` 和事务又是两回事吗？

## 2. 直觉理解

- **`MULTI/EXEC`**:把命令**排队**,`EXEC` 时**一次性顺序执行、中间不被插入别的命令**(原子性=不被打断)。但它**没有回滚**——队列里某条命令运行时报错,其余照常执行(sc01 实测)。
- **`WATCH`**:乐观锁。`WATCH k` 后到 `EXEC` 之间 k 被别人改了,`EXEC` 整体作废返回 nil(sc01 实测),靠它做 CAS。
- **Lua**:整个脚本作为**一个命令**原子执行(单线程不被打断),还能**读了再根据结果决定写**(`MULTI` 做不到「执行中拿到值再分支」)——所以「检查库存再扣减」这种用 Lua 最干净(sc02 实测防超卖)。
- **pipeline**:只是**批量发送命令省 RTT**,**不保证原子/隔离**——不是事务。

## 3. 原理深入

### 3.1 MULTI / EXEC / DISCARD
- `MULTI` 开始排队,后续命令返回 `QUEUED`(不执行);`EXEC` 一次性按序执行;`DISCARD` 放弃队列。
- **原子性**:EXEC 执行期间不会插入其他客户端的命令(单线程串行)。
- **无回滚**(关键区别于 RDBMS):若队列里某命令**运行时**出错(如对字符串 `INCR`),Redis **不回滚**已执行的命令——其余命令照常生效(sc01b 实测 a、b 仍写入)。**入队时**的语法错误才会让整个 EXEC 拒绝。
- 没有「隔离级别」「保存点」这些 RDBMS 概念。

### 3.2 WATCH —— 乐观锁 CAS
- `WATCH k1 k2...` 监视若干 key;若在 `EXEC` 前**任一被监视的 key 被改动**,`EXEC` 返回 nil(事务作废),客户端可重试。
- 典型 CAS 模式:`WATCH k` → 读 k → 算新值 → `MULTI` → 写新值 → `EXEC`(失败则重试)。sc01a 实测:WATCH 后被另一连接改 k,EXEC 整体作废。

### 3.3 Lua 脚本(EVAL）
- `EVAL <script> <numkeys> <keys...> <args...>`,脚本**整体原子执行**(占单线程,期间不被打断)。
- 比 `MULTI` 强在:**能在执行中读值并据此分支**(if/循环),做「检查-再-修改」的原子逻辑(sc02 原子扣库存防超卖)。
- 限制:脚本里**不能有副作用随机性/非确定性**(老版本要求,7.x 用 effects replication 放松了);**别写重循环/大遍历**(它占着单线程,会卡全场,呼应 01 章)。

### 3.4 FUNCTION（7.0+) vs EVAL
- `FUNCTION` 把脚本**注册成命名函数库**常驻服务端(`FCALL` 调用),不像 `EVAL` 每次传脚本(靠 SHA 缓存)。更适合复用、版本管理。

### 3.5 pipeline ≠ 事务
- pipeline 把多条命令一次性发出去、一次性收回复,**只省网络往返(RTT)**,大幅提吞吐。
- 但 pipeline **不保证原子性、不保证隔离**:命令之间可能被别的客户端的命令插入;一条失败不影响其他。要原子用 `MULTI`/Lua,要吞吐用 pipeline,两者可叠加(`MULTI` 本身也走 pipeline)。

## 4. 日常开发应用

- 「读了再改」的原子逻辑(扣库存、限流、CAS)→ **Lua** 最干净(sc02)。
- 简单的「一组命令要么都排队执行」→ `MULTI`,但记住**无回滚**,别指望它当 RDBMS 事务用。
- 高并发争用同一 key 的乐观更新 → `WATCH` + 重试。
- 批量写入想提吞吐 → pipeline(但别当事务)。

## 5. 调优实战

- **WATCH 重试风暴**:热点 key 上大量 WATCH-CAS 会频繁 abort、空转重试 → 改用 Lua 原子操作或分布式锁(07 章)。
- **Lua 让实例卡顿** → 脚本里有大遍历/重循环,占着单线程;拆小或用 `SCAN` 思路。
- **批量操作慢** → 是不是一条条 RTT,改 pipeline。

## 6. 面试高频考点

- **`MULTI/EXEC` 是事务吗?有回滚吗?** 是「打包顺序执行不被打断」,但**无回滚**(运行时错误不回滚已执行命令)。
- **WATCH 怎么做乐观锁?** 监视 key,EXEC 前被改则整体作废,重试。
- **为什么用 Lua 不用 MULTI?** Lua 能「执行中读值再分支」,做原子的检查-修改(防超卖,sc02)。
- **pipeline 和事务的区别?** pipeline 只省 RTT、不保证原子隔离;事务保证不被打断。
- **EVAL vs FUNCTION?** FUNCTION(7.0)注册命名函数常驻,EVAL 每次传脚本(SHA 缓存)。

## 7. 一句话总结

`MULTI/EXEC` 是「打包顺序执行、不被打断,但**不回滚**」;`WATCH` 加乐观锁 CAS;真正灵活的原子逻辑(检查-再-改、防超卖)用 **Lua**;`pipeline` 只是省 RTT 的批量、**不是事务**。
