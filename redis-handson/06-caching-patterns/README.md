# 缓存模式与一致性（Caching Patterns & Consistency）

## 1. 核心问题

把 Redis 当 DB 前面的缓存时，要解决四件事：
**(a)** 读写怎么组织（Cache-Aside / Read-Through / Write-Through / Write-Back）；
**(b)** 并发读写下，缓存和 DB 怎么尽量保持一致；
**(c)** 三大失效场景——**穿透 / 击穿 / 雪崩**——怎么防；
**(d)** 这些方案的代价是什么（一致性强度、内存、复杂度）。

## 2. 直觉理解

缓存是「DB 的一份快照副本」。**只要副本和正本之间存在「读到旧值又写回」或「更新先后错位」的时间窗口，就可能留下不一致。** 缓存不可能做到和 DB 强一致，能做的是「最终一致 + 把不一致窗口缩到最小」。

三大失效是「请求穿过缓存砸到 DB」的三种姿势：
- **穿透**：查一个**DB 里也不存在**的 key，缓存永远 miss，每次都打 DB（常被恶意刷）。
- **击穿**：一个**热点 key 失效的瞬间**，大量并发同时回源重建（惊群）。
- **雪崩**：**大批 key 在同一时刻一起失效**，流量瞬间全压到 DB。

## 3. 原理深入

### 3.1 四种读写模式

| 模式 | 读 | 写 | 谁回填/写库 | 特点 |
|---|---|---|---|---|
| **Cache-Aside（旁路，最常用）** | miss 时应用查 DB 再回填缓存 | 应用写 DB，然后**删缓存** | 应用 | 简单、可控；下次读才回填 |
| Read-Through | 缓存层负责 miss 时查 DB | —— | 缓存中间件 | 应用无感，但要缓存层支持 |
| Write-Through | —— | 写穿透：同时写缓存和 DB（同步） | 缓存中间件 | 强一致些，写慢 |
| Write-Back（回写） | —— | 只写缓存，异步刷 DB | 缓存中间件 | 写最快，**宕机丢数据** |

默认选 **Cache-Aside**：读 miss 回填、写「先更 DB 后删缓存」。

### 3.2 一致性核心：更新顺序与延迟双删

**为什么是「先更 DB，再删缓存」而不是别的：**

| 策略 | 脏读窗口 | 评价 |
|---|---|---|
| 先更 DB，再删缓存 | 删缓存前的极短窗口读到旧值；删失败才长期脏 | **推荐**，窗口最小 |
| 先删缓存，再更 DB | **读线程在「删缓存」与「更 DB」之间 miss → 读到旧 DB → 回填旧值**，之后 DB 更新，缓存长期是旧值 | 危险，见 scenario 04 |
| 更新缓存（而非删除） | 并发写时两次更新可能乱序覆盖；且算出的缓存值未必被读到（浪费） | 一般不用，宁删勿更 |

**延迟双删**：补「先删缓存」策略的窗口——更完 DB 后，**延迟一小段**（要 > 读线程「读 DB + 回填」耗时）再删一次缓存，把可能被回填的旧值清掉。scenario 04 实测：不双删 → 缓存=旧/DB=新；双删 → 旧缓存被二次清除，下次读重建为新值。
**binlog 订阅兜底**：用 Canal 订阅 DB binlog，变更后异步删缓存，作为应用层删缓存失败的补偿。

> 结论：缓存一致性只能做到**最终一致**。要强一致就别用缓存（或用 Write-Through + 读写都走缓存）。

### 3.3 三大失效与解法

| 场景 | 成因 | 解法 |
|---|---|---|
| **穿透** | 查 DB 也没有的 key | **布隆过滤器**前置（不存在直接拒，见 sc01）；或**缓存空值**（短 TTL）|
| **击穿** | 单个热 key 失效瞬间惊群 | **互斥重建**（`SET NX` 抢锁，只 1 个回源，见 sc02）；**逻辑过期**（值里存过期戳，过期了异步重建、旧值先返回）；**热点永不过期** |
| **雪崩** | 大批 key 同时失效 | **TTL 加随机抖动**（见 sc03）；多级缓存；限流/熔断兜底 DB |

## 4. 日常开发应用

- 默认 **Cache-Aside + 先更 DB 后删缓存**；删缓存失败要重试或上 binlog 兜底。
- **TTL 一律加随机抖动**（`base + rand`），这是最便宜的雪崩防护。
- 查询型接口（按 id 查）前置**布隆**或对不存在的 id **缓存空值**，防穿透/刷库。
- 高 QPS 热 key 用**互斥重建**或**逻辑过期**，别让它失效瞬间惊群。
- 用 `INFO stats` 的 `keyspace_hits / keyspace_misses` 算命中率；命中率低先查是不是 TTL 太短或穿透。

## 5. 调优实战

**Case A：「上线后 DB 偶发被打满」** → 看是不是穿透（大量查不存在 id）或雪崩（监控 DB QPS 是否周期性尖刺，对齐缓存 TTL）。穿透加布隆/空值；雪崩加 TTL 抖动。
**Case B：「缓存命中率忽高忽低」** → `INFO stats` 看 hits/misses；多半是热 key 集中失效（击穿）或 TTL 设得太齐（雪崩前兆）。
**Case C：「改了 DB，缓存还是旧的」** → 检查更新顺序（是不是「先删缓存再更库」被并发读回填了，见 sc04），改「先更库后删缓存」+ 必要时延迟双删。

## 6. 面试高频考点

- **先删缓存 vs 先更库**：先更库后删缓存窗口最小；先删缓存有「读回填旧值」脏读（sc04 实测）。
- **延迟双删为什么「延迟」**：要等并发读把旧值回填完再删，延迟 < 回填耗时就盖不住。
- **布隆假阳性的业务影响**：假阳性会放个别不存在的 key 到 DB（可接受），但**零假阴性**保证不漏真实 key（sc01 实测 假阴性=0）。
- **击穿三解法对比**：互斥（强一致、有等待）/ 逻辑过期（不阻塞、可能短暂返回旧值）/ 永不过期（最简单、靠后台刷新）。
- **雪崩为什么加随机数**：把「同一秒 1000 个过期」摊成「每秒几十个」（sc03 实测 1→100 个到期秒）。

## 7. 一句话总结

缓存默认 **Cache-Aside + 先更库后删缓存 + TTL 随机抖动 + 布隆挡穿透 + 互斥/逻辑过期挡击穿**。一致性只能做到「最终一致 + 缩小窗口」，强一致就别上缓存。所有结论都在 scenarios 01-04 实跑验证过。

## 对照镜像（reference，不在 lab 跑）

scenario 用 cli/Lua 演示原理；生产落地常用客户端封装：

```java
// Java / Redisson —— 穿透:布隆;击穿:分布式锁保护重建
RBloomFilter<String> bf = redisson.getBloomFilter("user:bf");
bf.tryInit(1_000_000L, 0.01);           // 预估 100w 元素,1% 假阳性
if (!bf.contains(id)) return null;       // 不存在直接返回,挡穿透

RLock lock = redisson.getLock("rebuild:" + id);
if (lock.tryLock(0, 5, TimeUnit.SECONDS)) {   // 只 1 个线程回源重建,挡击穿
    try { v = db.load(id); cache.set(id, v, ttlWithJitter()); }
    finally { lock.unlock(); }
}
```

```python
# redis-py —— 互斥重建 + bitmap 布隆
if r.set(f"lock:{id}", 1, nx=True, ex=5):   # 抢到锁才回源
    try:
        v = db_load(id)
        r.set(f"cache:{id}", v, ex=100 + random.randint(0, 100))  # TTL 抖动
    finally:
        r.delete(f"lock:{id}")
# 布隆位:r.setbit("bf", h1, 1); r.getbit("bf", h1)
```
