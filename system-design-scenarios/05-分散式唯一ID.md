# 05 · 分散式唯一 ID(Distributed Unique ID)

> **T1 基本功第三題。** 表面是「發個號」,實際是一道**約束題**:你要在「全局唯一 / 高吞吐 / 有序 / 不依賴單點 / 不暴露業務量」這幾個彼此打架的需求裡,挑出這個業務真正要的那幾個,並說清你**犧牲了哪些**。
>
> 它幾乎不出現在主題列表裡,卻是**分庫分表的前置零件**——一旦你說「按 userId 分 64 個庫」,主鍵就不能再用單庫自增了(`system-design/07`)。所以這題的真正考點是:**「你憑什麼說這個發號方案在你估出的 QPS 下扛得住、且不會在某個凌晨因為 NTP 校時而發出重複 ID。」**

> **本題吃哪幾包(對照 `00` 依賴表):** A 框架 ● · B 存儲 ● · D 一致性 ●——D 在這題的化身就是**時鐘回撥**(物理時鐘靠不住,錨 `system-design/02`)。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個分散式唯一 ID 生成服務,給全公司的訂單、消息、支付流水等發主鍵。

不要一上來就喊「Snowflake!」。**先問清楚,因為下面每個答案都會換掉你的方案:**

| 維度 | 要問什麼 | 為什麼這個答案會換方案 |
|---|---|---|
| **唯一性範圍** | 全局唯一?還是只要單表/單業務線唯一? | 只要單表唯一 → DB 自增就夠,不必上發號器 |
| **有序性要求** | 需要單調遞增?趨勢遞增?還是完全無序也行? | 要排序/分頁/做主鍵 → 排除隨機 UUID;不在乎 → UUID 最省事 |
| **位寬約束** | 能不能塞進 64-bit `bigint`?還是字符串也行? | 要當 InnoDB 主鍵 → 必須 64-bit 數值(見 deep dive ③);允許字符串 → UUID/雪花文本都行 |
| **吞吐(QPS)** | 全局發號峰值多少?單業務線多少? | 幾千 → DB 號段就夠;幾十萬 → 必須本地生成(雪花) |
| **可用性 SLA** | 發號掛了,業務能不能短暫拿不到 ID? | 不能容忍 → 排除「強依賴一次 DB/Redis 往返」的方案 |
| **業務量保密** | ID 能不能讓人從相鄰兩個值推出「你一天賣多少單」? | 不能 → 排除嚴格 +1 連續遞增(見 deep dive ③ 的代價) |
| **是否需要嵌業務信息** | ID 裡要不要帶分片位 / 業務類型 / 時間? | 要 → 自定義位段(雪花的變體),不能用純隨機 |

**面試話術(把澄清變成分流):**
> 「在動手前我想先分流——這題的方案完全取決於兩個問題:**要不要有序**、以及**峰值 QPS 多大**。如果只要全局唯一、不在乎順序,UUID 三行代碼搞定我們就不用聊了;但如果這 ID 要當訂單主鍵、要支撐分庫分表、峰值幾十萬,那才有意思——我先按『訂單主鍵、趨勢遞增、峰值 20 萬 QPS、64-bit、發號不能掛』這組假設往下走,可以嗎?」

**常見陷阱:**
- 🚩 不問就喊「Snowflake」——顯得你只背了一個答案,沒在 reason。**先問再選**才是資深信號。
- 🚩 把「唯一」和「有序」混為一談——這是兩個正交需求,UUID 給你前者不給後者。

## 2. 容量估算(把規模算成「需要幾個發號器」)

調用 `02` 的數字感。發號題的估算很短,但**直接決定方案分流**:

**算三樣:**

1. **全局發號峰值 QPS。** 假設公司日訂單 + 消息 + 流水 合計需要發 ID 的寫操作 ≈ 20 億次/天。
   - 平均 QPS = `2e9 / 1e5` ≈ **2 萬**(用 `02` 的「一天 ≈ 1e5 秒」)。
   - 峰值 ×3(`02` 的放大係數)≈ **6 萬 QPS**。再給大促 headroom 算到 **20 萬 QPS** 做容量上限。

2. **單發號器吞吐 vs 需要幾個節點。** 這步是分流的關鍵:

   | 方案 | 單節點吞吐(數量級) | 撐 20 萬 QPS 要幾個 | 結論 |
   |---|---|---|---|
   | DB 自增(每次一行) | 單機 MySQL 寫 ~幾千 QPS(`02`) | ~50+ 個庫,且強依賴 DB | ❌ 直接出局 |
   | Redis INCR | 單 Redis ~10 萬 QPS(`02`) | 2–3 個分片 | ⚠️ 引入 Redis 強依賴 + 一次網絡往返 |
   | **號段模式** | 一次 DB 往返取一段(如 1000 個),**之後純內存發** | 內存發號 ≈ 千萬級/s,DB 壓力 = QPS/段長 | ✅ DB 壓力被段長攤薄 |
   | **Snowflake(本地生成)** | **純本地計算,單序列位 4096/ms = 409 萬/s** | 1 個節點理論就夠,實務按業務分組部署 | ✅ 不依賴任何遠端 |

3. **ID 位寬怎麼分配(64-bit 預算)。** 這是發號題唯一要記死的「位帳」(deep dive ① 拆解):
   - 64-bit `bigint` = 1 符號位 + 41 時間戳位 + 10 機器位 + 12 序列位。
   - **41 位毫秒 ≈ 69.7 年**(`2^41 / (1000×86400×365)`);**10 位機器 = 1024 個節點**;**12 位序列 = 同一毫秒內單機 4096 個**。
   - 所以**單機理論上限 = 4096 個/ms = 409 萬/s**——對照上面 20 萬全局 QPS,**單機都綽綽有餘**,你會分多機是為了**容災和按業務隔離**,不是為了吞吐。

**面試話術(把數字焊到方案上):**
> 「峰值我算到 20 萬 QPS。關鍵發現是:**這個量級單台雪花機就能扛**(序列位 12 bit = 單機 409 萬/s),所以我不是因為吞吐才分多機,是為了**容災**——一台掛了還有別台。反過來看 DB 自增,單機幾千 QPS,撐這個量要 50 個庫還強依賴 DB,直接淘汰。所以我會在『雪花本地生成』和『號段模式攤薄 DB』之間選,而不是『每發一個號打一次 DB』。」

**常見陷阱:** 算半天 QPS 卻不接「所以選哪個方案」。發號題的估算**唯一意義**就是把方案從 5 個砍到 2 個。

## 3. API + 數據模型

### 對外接口

發號服務的 API 極簡,但**有沒有批量接口是資深信號**——批量把網絡往返攤薄:

```
GET  /id                         → { id: 7234567890123456789 }      // 單個
GET  /ids?count=100              → { ids: [...] }                    // 批量(攤薄往返)
GET  /id?bizTag=order            → { id }                            // 按業務線(號段模式用)
```

> **要點:** 高頻場景**別讓業務每發一個號就走一次 RPC**——要嘛批量領,要嘛把生成邏輯做成**客戶端 SDK 本地算**(雪花的常見部署形態:不是中心服務,是嵌進每個應用進程的庫)。這是雪花相對 Redis/DB 的核心優勢:**零網絡往返**。

### 數據模型(只有「號段模式」需要存)

UUID 和 Snowflake **不需要任何存儲**(純算)。只有號段模式要一張 DB 表:

```sql
-- 號段分配表(Leaf-segment 風格)
CREATE TABLE id_segment (
  biz_tag     VARCHAR(64) PRIMARY KEY,  -- 業務線(order / message / ...)
  max_id      BIGINT NOT NULL,          -- 當前已分配到的最大值
  step        INT    NOT NULL,          -- 步長(一次領多少,如 1000)
  version     BIGINT NOT NULL,          -- 樂觀鎖,防併發領段衝突
  updated_at  TIMESTAMP
);
```

**領一段的核心 SQL(樂觀鎖,一次 DB 往返領 `step` 個):**
```sql
UPDATE id_segment SET max_id = max_id + step, version = version + 1
WHERE biz_tag = 'order' AND version = #{oldVersion};
-- 領到的區間 = [舊 max_id + 1, 新 max_id],之後 step 個號全在內存發
```

> **分片鍵視角:** 注意這張表**按 `biz_tag` 分行**,每個業務線一行、各領各的段,天然隔離——一個業務線把 DB 拖慢不影響另一個。這就是為什麼 ID 服務常按業務線水平拆。

## 4. 高層架構(兩種主力部署,一鏡到底)

發號題沒有單一「標準架構」,它是**選型題**。把兩種主力方案各走一遍請求路徑:

### 方案 A:Snowflake 本地生成(無中心、零往返)

```
  ┌─────────────── 應用進程 A ───────────────┐
  │  業務代碼 → IdGenerator SDK(本地算)      │     workerId 由
  │            ↑ workerId=17                   │ ←── ZK/etcd 啟動時分配
  └────────────────────────────────────────────┘     (保證機器位不撞)
  ┌─────────────── 應用進程 B ───────────────┐
  │  業務代碼 → IdGenerator SDK(本地算)      │
  │            ↑ workerId=42                   │
  └────────────────────────────────────────────┘
        發號路徑:純 CPU 位運算,0 次網絡往返
```
> **請求一鏡到底:** 業務調 `nextId()` → SDK 讀本機毫秒時間戳 → 同毫秒內序列號 +1 → 三段拼成 64-bit 返回。**全程不出進程。** 唯一的外部依賴是**啟動時**去 ZK/etcd 領一個不重複的 `workerId`(運行期不依賴)。

### 方案 B:號段模式(中心 DB,但壓力被攤薄)

```
  業務進程 → ID-SDK ─(99.9% 命中本地內存段)→ 直接返回
                │
                └─(段快用完時,異步)→ ID Service → DB(領下一段)
                                                      ↑
                                  雙 buffer:當前段用到 ~90% 就預取下一段
```
> **請求一鏡到底:** 業務調 `nextId()` → SDK 從**內存當前段**取下一個(`AtomicLong.incrementAndGet`)→ 返回。**只有當段快耗盡時**,後台線程才異步去 DB 領下一段(deep dive ②)。所以 999/1000 次發號是純內存,DB QPS = 全局 QPS ÷ 段長。

**選型一句話(白板上寫這張表):**

| 方案 | 唯一性 | 有序性 | 依賴 | 吞吐 | 何時選 |
|---|---|---|---|---|---|
| **UUID v4** | 全局 | ❌ 完全無序 | 無 | 極高(本地) | 不當主鍵、不需排序、要最省事(如 traceId / 冪等鍵) |
| **DB 自增** | 單庫 | ✅ 嚴格 +1 | 強依賴 DB | 低(幾千) | 單庫小系統;**分庫後失效** |
| **Redis INCR** | 全局 | ✅ 趨勢 | 強依賴 Redis | ~10 萬 | 已有 Redis、量中等、能接受一次往返 |
| **號段模式** | 全局 | ✅ 趨勢遞增 | 弱依賴 DB(段內不依賴) | 極高(內存) | 要趨勢遞增 + 高吞吐 + 弱化 DB 依賴 |
| **Snowflake** | 全局 | ✅ 趨勢遞增 | 依賴**時鐘** | 極高(本地) | 要趨勢遞增 + 高吞吐 + 零往返 + 不想引入 DB/Redis |

> **面試話術(暴露你在做選型):** 「我會在號段和雪花之間二選一,兩者都給趨勢遞增 + 高吞吐。差別在**它們各自賭什麼**:號段賭 DB 偶爾可用就行(段內不依賴 DB),雪花賭**機器時鐘不會回撥**。我們的機房 NTP 配置怎樣、能不能容忍發號短暫不可用,決定我選哪個——這正好是接下來的 deep dive。」

## 5. Deep Dive(挑 3 個最會被往死裡追問的點)

### Deep dive ①:Snowflake 64 位結構拆解 + 時鐘回撥為什麼致命

**黑盒裡發生什麼——64 位怎麼拼成一個號:**

```
 0 | 0000000000 0000000000 0000000000 0000000000 0 | 0000000000 | 0000000000 00
─┬─  ──────────────────┬─────────────────────────    ────┬─────   ──────┬──────
 │            41 位時間戳(毫秒)                       10 位機器位     12 位序列號
符號位                ≈ 69.7 年                       = 1024 台      = 同毫秒 4096 個
(恆為 0,保證正數)   (相對某 epoch 起點)
```

生成一個 ID 的**完整內部步驟**(這是面試官想聽的「黑盒」):
```
1. now = 當前毫秒時間戳
2. if now == lastTimestamp:                    // 同一毫秒內
      sequence = (sequence + 1) & 4095          // 序列 +1,到 4096 就溢出歸 0
      if sequence == 0: now = 自旋等到下一毫秒    // 這毫秒發滿了,等下一毫秒
   else:
      sequence = 0                              // 新的毫秒,序列歸 0
3. if now < lastTimestamp:  →→→ 時鐘回撥!(見下)
4. lastTimestamp = now
5. id = (now - epoch) << 22 | workerId << 12 | sequence   // 三段拼接
```

> **為什麼是趨勢遞增而非嚴格遞增:** 高位是時間戳,所以**整體隨時間單調往上**(趨勢);但同一毫秒內由序列號排序、跨機器之間誰先誰後不保證,所以**不是全局嚴格 +1**。這個「趨勢」恰好是它對 InnoDB 友好的原因(deep dive ③)。

**時鐘回撥為什麼致命 🔬:** 第 5 步把「時間戳」當高位。如果機器時鐘**往回跳**(NTP 校時、閏秒、虛機遷移、運維手動改時間),那麼 `now < lastTimestamp`,於是你會**用一個過去的時間戳 + 可能相同的序列號**,生成一個**和歷史上某個 ID 完全相同的值**——重複主鍵,訂單覆蓋,災難。這是雪花唯一的、也是最常被追問的**失效模式**。本質是:**它把「物理時鐘單調」當成了正確性前提,而物理時鐘根本不可靠**(這正是 `system-design/02` 講「分布式裡沒有現在幾點」的後果)。

**怎麼處理回撥(三條策略,給 tradeoff):**

| 策略 | 怎麼做 | 代價 / 何時用 |
|---|---|---|
| **等待(wait)** | 檢測到回撥就**阻塞自旋**,等物理時鐘追回到 `lastTimestamp` 再發 | 簡單,但回撥期間**發號不可用**;只適合小回撥(幾 ms,如閏秒抖動) |
| **拒絕(reject)** | 回撥超過閾值(如 >5ms)直接**拋異常**,讓上游重試/切到別的發號機 | 快速失敗,把問題暴露給調用方;配合多發號機可降影響面 |
| **借位 / 邏輯時鐘**(進階) | 不再死綁物理時鐘:用「上次時間戳 + 1」往前推(借未來的時間),或預留幾位做「回撥序號」 | 不會不可用,但**透支了未來時間**,且實現複雜;百度 UidGenerator 走這條路 |

> **面試話術(這是把雪花講到資深的關鍵句):** 「雪花的阿喀琉斯之踵是時鐘回撥——它把『時鐘單調』當成了正確性前提,而 NTP 校時隨時會打破這個前提。生產上我會:小回撥(幾毫秒)就**等**,大回撥就**拒絕並告警**、把這台發號機摘掉,而不是冒重複的險。如果完全不能接受發號不可用,那我就**不選雪花**——這是它和號段的根本分水嶺:**雪花賭時鐘,號段賭 DB。**」

**錨點:** 時鐘為什麼不可靠、物理時鐘 vs 邏輯時鐘的完整光譜,回 `system-design/02 一致性模型與時鐘 · Part B 邏輯時鐘`;位段拆解和 MAC/PID 混淆的工程細節,回既有筆記 `distribution/分布式id.md`(本章不重寫它,只用它的位帳)。

### Deep dive ②:號段模式 —— 從 DB 取一段、雙 buffer 預取、可用性帳

**核心思路(黑盒裡發生什麼):** 不要「每發一個號打一次 DB」(那退化成 DB 自增的低吞吐),而是**一次 DB 往返領一整段**(如 `[1001, 2000]`),**之後 1000 個號全在內存用 `AtomicLong` 發**。DB QPS 因此被**段長攤薄 1000 倍**:全局 20 萬發號 QPS → DB 只承受 200 QPS。

**樸素號段的問題——「取段抖動」:** 當段用完(發到 2000)、要去 DB 領下一段時,**這次發號要同步等一次 DB 往返**(~1ms 甚至更久)。於是每隔 1000 個號就有一個請求延遲飆高,P99 出現規律性毛刺;更糟的是若這一刻 DB 慢/抖,所有等段的請求一起卡住。

**雙 buffer 預取怎麼消掉抖動 🔬(這是面試官想聽的精髓):**

```
  ┌── buffer0(當前正在發,如 [1001,2000])── 發到 ~1900(90%)時觸發 ──┐
  │                                                                      ▼
  │                                              後台線程異步去 DB 領 buffer1 = [2001,3000]
  │                                                                      │
  └── buffer0 發完 → 無縫切到已備好的 buffer1 ←─────────────────────────┘
                     (切換瞬間 0 等待,因為下一段早就預取好了)
```

關鍵在**「提前」**:不等段耗盡才領,而是**用到 ~90% 就在後台異步預取下一段**,放進備用 buffer。等當前段發完,下一段早已躺在內存裡,**切換是零等待的**——抖動被消除,P99 平滑。

**可用性帳(號段對 DB 的依賴有多脆):**

| 場景 | 號段能不能扛住 | 為什麼 |
|---|---|---|
| DB 短暫不可用(秒級) | ✅ 能 | 內存裡還有未發完的段 + 預取好的備用段,**段內完全不碰 DB** |
| DB 長時間掛(超過段耗盡) | ❌ 不能 | 兩個 buffer 都發完還領不到新段 → 發號停 |
| DB 主從切換抖動 | ✅ 通常能 | 預取機制把切換窗口藏在備用段後面 |

> **可用性的本質對比:** 雪花在 DB 全掛時**照發不誤**(它根本不碰 DB);號段在 DB 短掛時靠緩存的段**撐一陣**,長掛則停。所以**號段用「段長 × 緩存」換來了對 DB 故障的容忍窗口**——段越長、buffer 越多,窗口越大,但代價是 ID **跳號越多**(進程重啟丟掉未發完的段,號就不連續了)、且**消耗號空間越快**。

> **面試話術:** 「號段的精髓是雙 buffer 預取——用到 90% 就後台異步領下一段,把『取段』這次 DB 往返**藏在發號路徑之外**,所以 P99 沒有毛刺。它對 DB 的依賴是**弱**的:DB 抖一下,我內存裡的段照發;但如果段長設太短、或 DB 長掛超過兩段,還是會停。我會把段長按『DB 故障窗口能容忍多久』來定——不是越長越好,長了重啟丟段更多、號跳得更厲害。」

**錨點:** 號段表的樂觀鎖併發領段,本質是 `system-design/07` 講的「分庫分表後主鍵不能自增」的解法之一;DB 弱依賴 + 內存緩存的可用性思路,呼應 `system-design/01 韌性`。

---

### Deep dive ③:有序性的代價與價值 —— 趨勢遞增對 B+ 樹的恩賜,嚴格遞增的詛咒

這是發號題**最容易被忽略、卻最體現架構師深度**的點:**「ID 有不有序,不是美學問題,是直接決定 InnoDB 寫性能。」**

**黑盒裡發生什麼——InnoDB 聚簇索引按主鍵物理排序 🔬:**

InnoDB 的表數據**就是**按主鍵組織的 B+ 樹(聚簇索引),葉子節點按主鍵順序物理排布。**新插入一行,要放到「主鍵應該在的那一頁」。** 於是主鍵的「有序性」直接決定插入落在哪:

| 主鍵類型 | 插入落點 | B+ 樹發生什麼 | 後果 |
|---|---|---|---|
| **趨勢遞增**(雪花/號段) | 永遠落在**最右那一頁的尾部**(因為新 ID 總比舊的大) | 順序追加,寫滿一頁開新頁,**不動已有頁** | ✅ 順序寫、頁利用率高、buffer pool 命中好、幾乎無頁分裂 |
| **隨機 UUID** | 落在 B+ 樹**任意中間位置**(UUID 無序) | 為了在中間插入,要**把已寫滿的頁從中間劈開**(頁分裂),騰位置 | ❌ 隨機 IO、頁分裂、頁利用率掉到 ~50%、寫放大、buffer pool 被打髒 |

> **這就是「不要用隨機 UUID 當 InnoDB 主鍵」的真正原因**——不是因為它長(16 字節 vs 8),而是因為它**無序導致頁分裂和隨機寫放大**。在高寫入量下,UUID 主鍵的表插入吞吐可能比雪花主鍵**差幾倍**,且磁盤碎片化嚴重。如果非用 UUID 不可,要用 **UUID v7**(時間戳前綴、趨勢遞增)而不是 v4(純隨機),把有序性補回來。

**但「有序」也有代價——嚴格單調遞增暴露業務量 🔬:**

有序性不是越強越好。**嚴格 +1 連續遞增(如 DB 自增)會洩密:** 競爭對手今天下單拿到訂單號 `10000`,明天同一時刻下單拿到 `10500`,就能推算出你**一天賣了 500 單**——這叫「德國坦克問題」(盟軍靠坦克序列號估算德軍產量)。電商、發號類業務常因此被爬蟲反推 GMV。

| 有序強度 | 寫入友好度 | 保密性 | 代表 |
|---|---|---|---|
| 嚴格遞增 +1 | 最友好 | ❌ 完全洩露業務量 | DB 自增、Redis INCR |
| **趨勢遞增**(時間有序、局部跳號) | 友好 | ⚠️ 能推時間、難推精確量 | 雪花、號段 |
| 完全無序 | ❌ 頁分裂 | ✅ 推不出任何信息 | UUID v4 |

> **所以「趨勢遞增」是甜蜜點:** 它對 B+ 樹**夠順**(高位是時間,整體往上走),又因為**有跳號、跨機器交錯、序列不連續**而**不洩露精確業務量**——同時拿到了寫性能和保密性的大部分好處。這就是為什麼雪花/號段成為主流,而不是兩個極端。

> **面試話術(這段講出來就是架構師信號):** 「有序性看著是個小細節,其實直接決定 InnoDB 寫性能——聚簇索引是按主鍵排的 B+ 樹,趨勢遞增的 ID 永遠追加在最右頁、幾乎不頁分裂;隨機 UUID 插在中間導致頁分裂、寫放大、頁只用一半。**但反過來,嚴格 +1 又太有序了,會被爬蟲反推業務量(德國坦克問題)。** 所以雪花的『趨勢遞增』是甜蜜點:對 B+ 樹夠順,又因為跳號和機器交錯而不洩密。如果業務真的要藏訂單量,我還會在 ID 外面再包一層不可枚舉的外部編號。」

**錨點:** InnoDB 聚簇索引 / B+ 樹頁分裂的存儲層細節,屬於存儲零件,回 `system-design/06 存儲選型` 與 `mysql-handson/`(索引內幕);這裡只用「主鍵有序性 → 寫入落點 → 頁分裂」這條因果鏈。

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效 / 何時不該)

IC 線到第 5 段就夠了。架構師線要回答「**這個發號決策,代價誰買單、怎麼演進、會怎麼掛、什麼時候根本不該上發號器**」。

### 成本與「何時不該做」

| 反模式:過度設計 | 為什麼不該 | 該怎麼做 |
|---|---|---|
| 小系統一上來就部署中心化雪花服務 + ZK 集群 | 為了發號引入 ZK,**新增一整套要運維的有狀態組件**,故障面反而變大 | 量不大、沒分庫 → **DB 自增就夠**,別發明問題 |
| 全公司強推一個發號方案 | 不同業務約束不同(traceId 要無序、訂單要保密、消息只要趨勢) | **按業務選**:traceId 用 UUID,訂單用雪花/號段 |
| 為「全局嚴格單調」硬上中心發號 | 嚴格單調 = 中心化 = 單點 = 吞吐瓶頸 + 暴露業務量 | 先問「真的需要嚴格單調嗎」,99% 場景**趨勢遞增就夠** |

> **何時根本不該上發號器:** 沒分庫分表、單表自增能滿足、且不需要全局唯一時——**自增主鍵是最簡單、最有序、最省運維的選擇**。發號器是「分庫分表 / 跨服務唯一」逼出來的成本,不是默認項。

### 容量規劃(發號題特有的「位耗盡」帳)

發號器有一般服務沒有的容量維度——**號空間和位段會耗盡:**

| 耗盡風險 | 觸發條件 | 怎麼規劃 |
|---|---|---|
| **41 位時間戳耗盡** | epoch 起點後 ~69.7 年 | epoch 設成「項目上線那天」而非 1970,把 69 年留給未來;心裡有數即可 |
| **12 位序列耗盡** | 單機單毫秒 > 4096 個(409 萬/s) | 超了就自旋等下一毫秒;真要更高吞吐 → 重新切位段(借時間位給序列) |
| **10 位機器位耗盡** | 部署機器 > 1024 台 | 大廠會重切位段(如 5+5 拆數據中心+機器,或縮時間位換機器位) |
| **號段 bigint 耗盡** | 號段 max_id 漲到 2^63 | 幾乎不可能,但要監控告警 |

> **位段是固定 64-bit 的零和遊戲:** 給序列多 1 位(吞吐翻倍),就得從時間或機器位扣 1 位(壽命減半或機器數減半)。**容量規劃 = 按你的「機器數 × 單機吞吐 × 期望壽命」反推位段分配**——這是架構師要 own 的決策,不是抄默認 41/10/12。

### 演進路徑

```
單庫自增 ──分庫了──► 號段/雪花 ──量再大/要零往返──► 雪花 SDK 嵌進每個應用
   │                      │
   │                      └──要容忍時鐘回撥──► 邏輯時鐘變體(借位)/ 號段(賭 DB 不賭時鐘)
   └──只要唯一不要有序──► UUID(traceId / 冪等鍵這類根本不進主鍵的場景)
```

### 失效模式(每個方案會怎麼掛)

| 方案 | 失效模式 | 爆炸半徑 | 防護 |
|---|---|---|---|
| **雪花** | 時鐘回撥 → **發出重複 ID** | 訂單覆蓋、數據錯亂(最嚴重) | 回撥檢測 + 等待/拒絕 + 摘除告警;NTP 用 slew 模式(漸進校時不跳變) |
| **雪花** | workerId 撞號(兩台拿到同 ID) | 重複 ID | 啟動時 ZK/etcd 強保證唯一,不靠手配 |
| **號段** | DB 長掛超過緩存段 | 發號**停止**(但不重複) | 多 buffer + 長段;DB 高可用 |
| **號段** | 進程重啟 | 丟未發完的段 → **跳號**(不重複,可接受) | 接受跳號,別追求連續 |
| **Redis INCR** | Redis 掛 / 主從切換丟數據 | 重複或停號 | AOF + 主從;或乾脆別用 Redis 發主鍵 |

> **失效模式的分水嶺:** 雪花的失效是「**可能發重複**」(正確性問題,最可怕);號段的失效是「**可能停號 / 跳號**」(可用性問題,沒那麼致命)。**寧可停號也不要重號**——這條原則決定了在「不能容忍重複主鍵」的支付場景,我會更傾向號段(賭 DB)而非裸雪花(賭時鐘),或給雪花配最嚴格的回撥拒絕策略。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 分散式 ID 五種方案各在什麼時候選? | UUID(只要唯一不要序、不進主鍵)/ DB 自增(單庫小系統)/ Redis INCR(已有 Redis、量中)/ 號段(趨勢遞增 + 弱化 DB + 高吞吐)/ 雪花(趨勢遞增 + 零往返 + 不想引入 DB)。**先問「要不要有序」和「峰值 QPS」分流。** |
| 2 | Snowflake 64 位怎麼分?單機吞吐多少? | 1 符號 + 41 時間戳(≈69.7 年)+ 10 機器(1024 台)+ 12 序列(同毫秒 4096 個)= **單機 409 萬/s**。位段是固定 64-bit 的零和:序列多 1 位 = 時間或機器扣 1 位。 |
| 3 | 雪花時鐘回撥為什麼致命?怎麼處理? | 時鐘往回跳 → 用過去時間戳 + 可能相同序列 → **發出重複 ID**。處理:小回撥**等**、大回撥**拒絕+告警摘機**、進階**借位/邏輯時鐘**。根因:它把「物理時鐘單調」當正確性前提,而 NTP 會打破它。 |
| 4 | 號段模式怎麼把 DB 壓力攤薄?雙 buffer 解決什麼? | 一次 DB 往返領一整段(如 1000 個),之後純內存發 → DB QPS = 全局 QPS ÷ 段長。**雙 buffer:用到 ~90% 就後台異步預取下一段**,把「取段往返」藏在發號路徑外,消掉 P99 毛刺。 |
| 5 | 為什麼隨機 UUID 不適合當 InnoDB 主鍵? | 聚簇索引是按主鍵排的 B+ 樹;隨機 UUID 插在中間 → **頁分裂、隨機寫、頁利用率~50%、寫放大**。趨勢遞增 ID 永遠追加最右頁,順序寫無分裂。要用 UUID 就用 **v7**(時間前綴)。 |
| 6 | 「趨勢遞增」為什麼是甜蜜點? | 對 B+ 樹夠順(高位時間戳,整體往上)→ 寫性能好;又因跳號/機器交錯而不洩露精確業務量(避開**德國坦克問題**)。嚴格 +1 太洩密,完全無序太傷寫——趨勢遞增兩頭好處都拿一半。 |
| 7 | 雪花 vs 號段,最根本的分水嶺是什麼? | **雪花賭時鐘(掛時可能發重複=正確性問題),號段賭 DB(掛時可能停號/跳號=可用性問題)。** 寧可停號不可重號 → 支付類更傾向號段或最嚴格的雪花回撥拒絕。 |

## 一句話總綱

> **分散式唯一 ID 是一道約束題:在「唯一/有序/高吞吐/不依賴單點/不洩業務量」裡挑出這個業務真要的那幾個。先用「要不要有序 + 峰值 QPS」把五個方案砍到二,主力是雪花(本地生成、賭時鐘不回撥、64 位 = 41 時間+10 機器+12 序列、單機 409 萬/s)和號段(一次 DB 領一段、雙 buffer 預取消抖動、賭 DB 偶爾可用)。有序性不是美學:趨勢遞增讓 InnoDB 聚簇索引順序寫、不頁分裂,又不像嚴格 +1 那樣洩露業務量——這是甜蜜點。失效分水嶺一句話:雪花掛了可能發重複(最怕),號段掛了只是停號或跳號;寧可停號不可重號。**
