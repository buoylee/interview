# 系統設計場景題 · scenario-driven SDI track

> 面試導向 · 資深→架構師視角 · **這是「整機裝配線」,不是零件庫**
>
> 你的 repo 裡 `system-design/`(L0–L9)是**零件庫**:把 CAP/共識/複製/分片/韌性/容災 每個零件的 tradeoff 講透,是一張 *reference 地圖*,教你「能 reason about 每個零件」。但面試真正考的是另一件事:**給你一道「設計 Twitter / 限流器 / 支付」,限時 45 分鐘,從需求澄清一鏡到底走完整題。** 那是「整機裝配」的肌肉,零件庫不訓練它。
>
> 這個 track 就是那條裝配線。它**不重寫任何零件**——每道題在需要深挖處,直接「指進」`system-design/` L0–L9 與既有深礦(`financial-consistency` / `perf` / `observability` / `distr-tx` …)。

---

## 三層一體:一條 scenario 主線,同時給你三樣東西

你說「全部都要」——做題肌肉、流暢路線、架構師深度。這三者不是三個 track,是**同一條場景線的三個切面**:

| 你要的 | 在這條線的哪裡 |
|---|---|
| **做題肌肉**(整題、限時打法) | Part B 場景題庫本身——一題一章,每章跑同一套七步框架 |
| **流暢學習路線**(從 0 到能上白板) | `00` 總覽是一張**線性學習地圖**,把散在 6+ 目錄的 L0–L9 與深礦,按「第幾題會用到」串成一條路,解掉 hub-and-spoke 的發散 |
| **架構師深度**(推過面試線) | 每道題在「白板 45min 版」之外,壓一層**架構師深挖**(成本/容量/演進/失效模式),並錨回深礦 |

> **一句話定位:** `system-design/` 教你「每個零件防什麼失效、付什麼代價」;這個 track 教你「面對一道真題,怎麼把零件**裝**成一個能扛住流量、容災、演進的系統,並在白板上**講**清楚」。

## 三段結構

### Part A · 方法與數字感(地基,3 章)

| 章 | 內容 | 解決 |
|---|---|---|
| `00` 總覽 + **學習路線地圖** | 這條 track 怎麼讀;把 L0–L9 + 深礦串成「上白板前要會什麼」的線性路徑;IC 線 vs 架構師線 | 流暢性 |
| `01` **SDI 七步框架** | 需求澄清(功能/非功能/約束)→ 容量估算 → API/接口 → 數據模型 → 高層架構 → deep dive → 瓶頸與取捨;每步給「面試話術 + 常見陷阱」 | 做題的骨架 |
| `02` **容量估算數字感** | 背誦級數字(一行多少 byte、一天 86400 秒、1M DAU 怎麼推 QPS、緩存命中率、讀寫比、5/10/100 倍冗餘)+ 5 個 30 秒估算 drill | 估算肌肉 |

### Part B · 場景題庫(主體,17 題,分三檔,一題一章)

每題覆蓋一類「子模式」,合起來覆蓋幾乎所有 SDI 出題面:讀密集 / 寫密集 / 扇出 / 地理 / 實時流 / 強一致交易 / 協同。

| 章 | 題目 | 子模式 / 為什麼選它 |
|---|---|---|
| **T1 基本功** ||
| `03` | 短鏈 URL(TinyURL) | 編碼/發號、讀密集緩存、301 vs 302、redirect 鏈路——最經典暖身題 |
| `04` | 分散式限流器 Rate Limiter | 固定窗/滑動窗/令牌桶/漏桶、單機 vs 分散式(Redis Lua)、邊界毛刺 |
| `05` | 分散式唯一 ID | Snowflake/號段/UUID、時鐘回撥、單調遞增 vs 趨勢遞增 |
| `06` | 分散式緩存 | 一致性哈希 vs slot、緩存四件套(穿透/擊穿/雪崩/旁路)、熱點打散、LRU/LFU |
| `07` | 排行榜 Leaderboard | ZSet、Top-N、分頁、近實時更新、海量用戶分桶 |
| **T2 進階** ||
| `08` | 信息流 News Feed | **寫擴散 vs 讀擴散 vs 混合**、大 V 問題、feed 合併排序——SDI 第一高頻 |
| `09` | 即時通訊 IM / 群聊 | 長連接、在線狀態、已讀回執、群消息擴散、離線推送、消息有序 |
| `10` | 通知系統 Notification | fan-out、多渠道(push/SMS/email)、去重/限頻、優先級、模板 |
| `11` | 搜索自動補全 Typeahead | Trie、Top-K、前綴聚合、近實時更新、分散式合併 |
| `12` | 對象存儲 / 網盤(Dropbox) | 分塊上傳、去重(內容哈希)、metadata vs blob、斷點續傳、同步衝突 |
| `13` | 網頁爬蟲 Web Crawler | URL 去重(布隆)、禮貌性、優先級隊列、可擴展抓取、陷阱規避 |
| **T3 高階** ||
| `14` | 支付 / 錢包 | 冪等、對賬、狀態機、強一致 + 最終一致混用;**錨 `financial-consistency`** |
| `15` | 網約車 / 外賣派單 | 地理空間索引(GeoHash/QuadTree)、實時撮合、位置上報洪峰 |
| `16` | 秒殺 / 票務 | 超賣防護、庫存扣減、限流削峰、排隊、防刷 |
| `17` | 視頻點播 / 直播 | CDN、轉碼流水線、推拉流、碼率自適應、首屏優化 |
| `18` | 廣告點擊實時聚合 | 流式(Kafka+Flink)、近似計數(HLL/Count-Min)、亂序/遲到、精確 vs 近似 |
| `19` | 協同編輯(Google Docs) | OT vs CRDT、實時同步、衝突解決;**錨 `distr-tx/CRDT`** |

### Part C · 收口(2 章 + 卡)

| 章 | 內容 | 解決 |
|---|---|---|
| `20` **設計模式速查** | 把 17 題裡反覆出現的模式抽出來收口:fan-out(寫/讀)、CQRS、CDC、冪等鍵、消息去重、熱點打散、讀寫分離、緩存四件套、地理分片、近似算法(HLL/Count-Min)、分塊+去重 | 「覺得雜亂」時的收口地圖 |

## 每道場景章 · 固定模板(8 段)

> 模板固定 = 練的是**可遷移的打法**,不是背 17 個孤立答案。

1. **題目 + 澄清問題清單** — 上來先問什麼(功能邊界 / 非功能 SLA / 規模 / 約束)
2. **容量估算** — 帶真實數字推 QPS / 存儲 / 帶寬 / 連接數(用 `02` 的數字感)
3. **API + 數據模型** — 對外接口 + 核心表/KV 結構 + 分片鍵
4. **高層架構**(ASCII 圖)— 請求一鏡到底走完
5. **Deep dive** — 挑 2–3 個「最會被追問」的點往下鑽
6. **架構師深挖層** — 成本 / 容量規劃 / 演進路徑 / 失效模式 / 何時不該這麼設計(推過面試線)
7. **錨點表** — 這題用到的零件，回 `system-design/` L0–L9 與深礦的哪一章(不重寫)
8. **面試卡** — 4–6 張抽認卡

## 怎麼讀(閱讀順序)

> 檔名編號 = 閱讀順序。**先 Part A 打地基,再按 T1→T2→T3 爬難度。**

- **完全新手 / 趕面試:** `00 → 01 → 02`(地基)→ T1 全做一遍練框架 → T2 高頻三題(`08 News Feed`、`09 IM`、`14 支付`)→ `21 playbook`。
- **概念脊柱已熟(你的情況):** `00`(只看學習路線地圖,確認哪些零件已會)→ `01`/`02` 快過 → 直接從 T2 開始練整題打法 → 缺零件時順著**錨點表**回 `system-design/` 補。
- **架構師成長:** 每題重點讀第 6 段「架構師深挖層」+ 第 7 段錨點，把「面試答法」升級成「真實決策」。

## 一句話總綱(背起來)

> **系統設計面試 = 用一套固定七步(澄清→估算→API→數據→架構→deep dive→取捨),把 `system-design/` 的零件，限時裝成一台能扛流量、容災、演進的整機，並在白板上講清每個取捨的 why。** 這個 track 練的就是這套「裝 + 講」的肌肉;零件不夠時，順著錨點回零件庫補。
