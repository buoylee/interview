# 18 · 廣告點擊實時聚合

> **T3 高階,也是整個 track 唯一一道「流式」題。** 前面 16 道題的核心都是「一個請求進來、同步算完、回出去」(讀路徑、寫路徑、扇出);這題第一次把重心整個搬到**離線的、源源不斷的事件流**上——點擊不是「請求-回應」,是**每秒幾百萬條、永不停止、還會亂序遲到**的數據河。你要設計的不是一個 API,是一條**管道**。
>
> 這題的教學價值集中在三處,每處都是一整類面試題的母題:**① 流式聚合管道**(Kafka → Flink 窗口聚合,exactly-once 怎麼保證——這是所有「實時數倉 / 監控 / 推薦特徵」題的骨架);**② 近似算法**(HyperLogLog 算 UV、Count-Min Sketch 算頻次——「為什麼能用幾 KB 估億級基數」是分水嶺問題,布隆過濾器在 `06 緩存`、`13 爬蟲` 已埋過種子,這裡長成大樹);**③ 精確 vs 近似的雙層架構**(看板可以近似,但**廣告計費一分錢不能錯**——Lambda 架構、watermark 處理遲到、去重防刷)。把這三個鑽透,你就拿到了「流式系統」這個 SDI 大類的通票。
>
> 用 `00` 依賴表看:廣告聚合吃 **A(框架)+ E ●●(MQ/事件流扇出)+ D(一致性:exactly-once / 對賬)+ C(背壓削峰)**——它是 `08 News Feed` / `10 通知`「扇出家族」的**流式變體**:同樣是 Kafka 當主幹,但 News Feed 扇出給「人」、這題扇出給「聚合算子」。

## 1. 題目 + 澄清問題清單

**題目:** 設計一個廣告點擊(及曝光)實時聚合系統。每當用戶在某廣告上產生**曝光(impression)/ 點擊(click)/ 轉化(conversion)**,系統要近實時地按多個維度(廣告主、廣告位、活動、地域、時間窗)聚合出 **PV(總次數)、UV(去重用戶數)、CTR、花費**,既要餵給**廣告主實時看板**,也要餵給**計費系統**結算。

這題的陷阱是:它**長著兩張臉**,你必須在澄清階段就把它們拆開,否則整個架構會做錯。**先用 `01` Step 1 問三類**:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 聚合哪些指標?哪些維度?看板和計費是不是同一條路?要不要去重(UV、防刷)? | 指標:PV / UV / CTR / 花費;維度:廣告主 × 廣告位 × 活動 × 地域 × 時間窗(滾動 1min / 1h / 1day)。**看板和計費是兩條不同 SLA 的路**(這是全題題眼)。UV 要去重、點擊要防刷 |
| **非功能(SLA)** | 實時性要多高?準確性要多準?容忍丟/重嗎? | **看板:秒級延遲、近似可接受**(UV 差 1% 沒人死);**計費:可以延遲幾分鐘,但金額必須精確、不丟不重(exactly-once)、可對賬**——這兩條 SLA **天差地別**,逼出雙層架構 |
| **規模 / 約束** | 點擊事件 QPS?維度組合基數?保留多久? | **峰值 ~100 萬 events/s**(曝光遠多於點擊);維度笛卡爾積基數百萬級;明細存 90 天、聚合結果存 2 年 |

**面試話術(把兩張臉拆開,這是這題拿高分的第一句):**
> 「設計前我要先確認一件最關鍵的事:這個系統是**餵看板**還是**餵計費**?因為這倆的 SLA 完全相反——看板要的是**秒級、能容忍 1% 誤差**(UV 用近似算法估就行);計費要的是**一分錢不錯、不丟不重、能對賬**,但可以接受幾分鐘延遲。如果兩個都要,我會做成**雙層架構**:一條近似快路餵看板,一條精確慢路餵計費。我先按『兩個都要』來設計,可以嗎?峰值我假設 100 萬 events/s。」

**這幾個澄清直接決定後面:**

- 「看板近似 / 計費精確」→ 直接定下第 5 步是 **Lambda(雙層)架構**,deep dive ③ 的命門。
- 「UV 去重 + 防刷」→ deep dive ② 的 **HyperLogLog**(估 UV)和去重(防刷)。
- 「100 萬 events/s」→ 第 2 步估算的結論會是:**這題的矛盾不是存儲、不是 QPS 扛不住,是「百萬流怎麼不丟不重地聚合」+「億級基數怎麼不爆內存地去重」。**
- 「亂序遲到」(沒人問你也要主動提)→ deep dive ③ 的 **watermark + 允許遲到窗口**。

**常見陷阱:** 上來就畫「Kafka → Flink → 數據庫」一條線,沒分清看板和計費。✅ 第一句就把「兩張臉」拆出來,直接展示你知道這題的題眼是 **SLA 分裂**。

## 2. 容量估算(直接調用 `02` 的數字)

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 = 平均 ×3、單 Kafka 分區 ~10 萬 msg/s、單 Redis ~10 萬 QPS、行記錄 ~1KB、副本 ×3**。

### ① 事件 QPS(管道入口的命門)

題設峰值 **100 萬 events/s**。先把這個數翻成架構:

```
100 萬 events/s ÷ 單 Kafka 分區 ~10 萬 msg/s ≈ 至少 10+ 分區(留 headroom 取 ~32~64)
```
> **所以:入口必須是分區化的 Kafka,且分區數 ≥ 流量 ÷ 單分區吞吐。** 這也立刻引出 deep dive ① 的核心問題:**按什麼 key 分區?** 分區 key 決定了下游 Flink 能不能正確按維度聚合(同一聚合 key 必須進同一分區/同一算子實例)。

### ② 聚合維度基數(決定「去重」用什麼算法)

UV 要去重——關鍵是**要去重多少個不同的值**:

```
假設一個大活動一天觸達 1 億不同用戶(deviceId/userId)
精確去重 = 把 1 億個 ID 放進一個 Set
  1 億 × 16 B(UUID)= 1.6 GB —— 只是「一個」維度組合的「一天」
維度組合有百萬個 → 精確去重 = 百萬 × GB 級 = 直接爆內存
```
> **所以:這是這題第一個真矛盾——精確去重的內存隨基數線性增長,億級基數 × 百萬維度組合 = 不可能用 Set 硬扛。** 這正是 deep dive ② 的 HyperLogLog 要解的:**用固定 ~12KB 估億級 UV,誤差 ~2%。** 把這個賬算出來,才有底氣說「UV 必須上近似算法」。

### ③ 存儲(明細 vs 聚合,差好幾個量級)

```
明細(raw events,計費對賬要留)
  100 萬/s × 1e5 s/天 = 1e11 條/天 = 1000 億條/天
  × ~200 B/條 = 20 TB/天(裸量)× 副本3 = 60 TB/天 —— 只能留短期(90 天熱 + 冷歸檔)

聚合結果(看板讀的)
  維度組合 ~百萬 × 時間窗(分/時/天)× 指標 ≈ 行數可控(GB~TB 級)
  → 落 OLAP(ClickHouse),小到能全進熱層
```
> **所以:明細是 PB 級洪流(計費對賬需要、但只能短留 + 冷歸檔),聚合結果反而很小(看板讀的就這層)。** 這條「**明細海量、聚合精煉**」的落差,直接決定第 3 步的數據分層:**Kafka(明細流)→ Flink(聚合)→ ClickHouse(聚合結果熱層) + 對象存儲(明細冷歸檔對賬用)。**

### ④ 帶寬 / 背壓(流式特有的第四問)

流式題的「第四問」不是帶寬,是**背壓(backpressure)**:

```
入口 100 萬/s,如果下游 Flink 算子處理不過來(GC / 落庫慢)→ 算子變慢
  → Kafka 消費滯後(consumer lag)堆積 → 但 Kafka 是緩衝,扛得住短時積壓
  → 持續扛不住 → lag 無限增長 → 看板數據延遲拉大
```
> **所以:Kafka 在這題不只是「消息隊列」,它是天然的削峰緩衝(`02` 小抄『峰值尖 → MQ 削峰』)。** 真正要監控的容量指標是 **consumer lag(消費滯後)**——它是流式系統的「水位線」,lag 漲就是處理跟不上了,要擴 Flink 並行度。這條在第 6 步架構師深挖層展開。

### 一句話估算結論(面試一定要說出來)

> 「**這題的矛盾不在 QPS 扛不住——100 萬/s 用 32~64 個 Kafka 分區就吞下了。三個真矛盾是:(1) 億級基數去重會爆內存 → 逼出 HyperLogLog 近似;(2) 明細是 PB 級洪流但聚合結果很小 → 數據分層(Kafka 流 + ClickHouse 聚合 + 冷歸檔對賬);(3) 看板和計費 SLA 分裂 → 近似快路 + 精確慢路雙層。我的設計重心放這三處,而不是堆機器扛 QPS。**」

這句「把估算焊到架構」的三段式結論,就是 `01` 說的資深 vs 初級的分水嶺。

## 3. API + 數據模型

### API(`01` Step 3:這題的「API」是兩端——上報端 + 查詢端)

流式題的 API 跟前面題不一樣:**寫端是「埋點上報」(高頻、單向、可丟一點),讀端是「看板查詢」(低頻、聚合查詢)。**

```
# 上報端(高頻寫,~100 萬/s)—— 要極快、可批量、可容忍少量丟
POST /track
  body: { eventType: "click", adId, campaignId, slotId,
          userId, deviceId, geo, ts, ... }      # ts = 事件發生時間(關鍵!)
  → 202 Accepted                                # 立刻回,異步進 Kafka,不等聚合

# 查詢端(低頻讀,看板)—— 走 OLAP 聚合查詢
GET /stats?adId=X&metric=uv,ctr&window=1h&from=...&to=...
  → { uv: 1.2e6, pv: 5.3e6, ctr: 0.045, spend: 12345.67 }
```

**要點:**
- 上報接口回 **202 Accepted**(收下了、異步處理),**不是 200+聚合結果**——這是「流式上報」和「同步 API」的本質區別,呼應 `03 短鏈`「跳轉回 3xx 不回 200+body」的同類洞察。
- **`ts`(事件發生時間)必須由客戶端帶上、且是聚合的依據**——這是流式系統最容易踩的坑:你必須按 **event time(事件發生時間)** 而不是 **processing time(到達服務器時間)** 來聚合,否則網絡延遲一抖,「下午 3 點的點擊」就被算進「3 點 05 分的窗口」。event time vs processing time 是 deep dive ③ watermark 的前置概念。

### 數據模型(`01` Step 4:選型 + 分區/分片鍵)

流式題的「數據模型」是**三層**,每層用不同存儲:

| 層 | 存什麼 | 選型 | 為什麼 |
|---|---|---|---|
| **L1 明細流** | 原始 events(計費對賬源、可重放) | **Kafka**(+ 冷歸檔到對象存儲/數據湖) | 順序寫盤快、可重放(replay)是 exactly-once 和對賬的根;`02` 單分區 10 萬 msg/s |
| **L2 聚合結果** | 各維度 × 時間窗的 PV/UV/CTR/花費 | **ClickHouse**(列存 OLAP) | 看板是「按維度聚合 + 時間範圍掃描」的 OLAP 查詢,列存掃描極快;聚合結果小,全進熱層 |
| **L3 實時看板熱緩存** | 最近 N 分鐘的熱榜/大盤數 | **Redis**(ZSet / String) | 看板首屏的「當前 QPS / 實時花費」要 sub-ms,Flink 算完直接寫 Redis |

**核心:OLAP 的聚合表結構(看板讀這張):**

| 字段 | 類型 | 說明 |
|---|---|---|
| `window_start` | DateTime | 時間窗起點(滾動窗對齊到 min/hour/day) |
| `ad_id` / `campaign_id` / `slot_id` / `geo` | 維度列 | 聚合維度(ClickHouse 的排序鍵/分區鍵走這幾列) |
| `pv` | UInt64 | 總次數(可加,直接 SUM) |
| `uv_hll` | **AggregateFunction(uniqHLL)** | **UV 的 HLL 草圖(sketch),不是數字!** 存的是 HyperLogLog 的桶狀態,查詢時合併估算 |
| `spend` | Decimal | 花費(計費維度,精確 Decimal 不用 float) |

> **黑盒裡的關鍵設計(資深信號):** UV 那一列**存的不是「12345 這個數」,是 HyperLogLog 的草圖(sketch)狀態**。為什麼?因為 **UV 不可加**——「廣告 A 上午 100 萬 UV + 下午 100 萬 UV ≠ 全天 200 萬 UV」(同一個人上下午都來)。如果存數字,你沒法把分鐘窗合併成小時窗。但 HLL 草圖**可以合併**(取每個桶的 max),合併後再估,誤差不變。**「存 sketch 不存數字」是讓近似 UV 可累加的關鍵一招**——ClickHouse 的 `AggregateFunction(uniq)` 就是幹這個的(錨 `big-data/clickhouse-roadmap.md`)。這個點講出來,deep dive ② 的 HLL 就過了資深線。

**分區鍵(`01` Step 4 鐵律:流式題必講「Kafka 按什麼分區」,這直接決定能不能正確聚合):**

> **Kafka 按「聚合主鍵」(如 `adId` 或 `campaignId`)分區。** 為什麼不能隨機分區?因為 Flink 要**按 key 做窗口聚合**(`keyBy(adId)`),**同一個 `adId` 的所有事件必須進同一個算子實例**才能正確累加——如果隨機分區,同一個廣告的點擊散在所有算子上,每個算子只看到一部分,聚合就錯了。**所以 Kafka 的分區 key 必須等於 Flink 的聚合 key**(或其前綴),這樣「同 key 同分區同算子」,聚合才正確。

> **熱點分區的隱患(主動暴露邊界,對比 `03 短鏈`「隨機短碼無熱點」):** 按 `adId` 分區會有熱點——**一個爆款廣告的點擊全壓在一個分區/一個算子上**(就像 News Feed 的大 V)。緩解:對超熱 key 做**兩階段聚合**(先按 `adId + 隨機後綴` 打散做局部聚合,再去掉後綴做全局合併)——這正是 `08 News Feed` 大 V 熱點打散在流式場景的同款手法。這個呼應講出來,展示你看得到題與題之間的同構。

## 4. 高層架構(`01` Step 5:沿一條事件流一鏡到底)

**先畫能跑的最簡流式管道,再疊上「計費精確層」**(`01` 鐵律:別一上來就 Lambda 雙層,先單條流跑通)。

```
                                    ┌─────────── 快路 / 近似層(餵看板,秒級)───────────┐
                                    │                                                    │
 埋點 SDK ──POST /track──► 上報網關 ──► Kafka ──► Flink(streaming)──► ClickHouse(聚合熱層)
 (曝光/點擊/轉化)         (202,極快)  (分區×64)   │ keyBy(adId)            │            ▲
                                       明細流      │ 滾動/滑動窗口聚合       │     看板 ◄──┘
                                       (可重放)    │ PV=count                │   GET /stats
                                       │           │ UV=HyperLogLog(估)      └──► Redis(實時大盤)
                                       │           │ watermark 處理亂序/遲到
                                       │           └── exactly-once(checkpoint)
                                       │
                                       └─────────── 慢路 / 精確層(餵計費,分鐘級)─────────┐
                                                     │                                    │
                                            冷歸檔(對象存儲/數據湖)──► 批量精確聚合 ──► 計費對賬
                                            (全量明細,可重放)         (去重防刷、Decimal)   │
                                                                                    與快路結果對賬 ◄┘
```

**一條事件一鏡到底(`01` Step 5:這題只有一條主流,但要走完雙層分叉):**

- **上報(寫,高頻):** SDK 產生點擊事件(帶 event-time `ts`)→ POST /track → 上報網關**只做最輕的校驗,立刻回 202**,把事件批量寫進 **Kafka**(按 `adId` 分區)。**網關零聚合、零落庫,只當入口**——這是百萬流能扛住的前提。
- **快路(近似,餵看板):** Flink 從 Kafka 消費 → `keyBy(adId)` → 開**滾動/滑動窗口** → 窗口內 `count` 算 PV、用 **HyperLogLog 估 UV** → 窗口觸發時把結果寫 **ClickHouse**(聚合熱層)和 **Redis**(實時大盤)→ 看板查 ClickHouse/Redis,**秒級看到數,UV 是近似的**。
- **慢路(精確,餵計費):** 同一份 Kafka 明細**冷歸檔**到對象存儲/數據湖(全量、可重放)→ **批量精確聚合**(精確去重 / 防刷規則 / Decimal 金額)→ 出計費結果 → **與快路結果對賬**(快路用來實時監控,慢路用來真結算)。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **Kafka(分區化)** | 百萬流入口削峰 + 可重放(exactly-once / 對賬的根) | ①「分區吞百萬流」④「削峰緩衝」 |
| **Flink + 窗口聚合** | 按 event-time 開窗、按 key 聚合,流式核心 | ①「按 key 聚合」 |
| **HyperLogLog(在 Flink/CH 裡)** | 億級 UV 去重不爆內存 | ②「基數爆內存」 |
| **ClickHouse** | 聚合結果的 OLAP 熱層,看板掃描快 | ③「聚合精煉」 |
| **冷歸檔 + 批量精確層** | 計費要精確、可對賬、可重放 | ③「明細海量、計費精確」 |
| **watermark** | 亂序/遲到事件正確歸窗 | deep dive ③ |

**注意這張圖的「克制」:** **MVP 階段只跑上半條快路就行**(Kafka → Flink → ClickHouse)。計費精確層(下半條 Lambda)是**等真要計費、要對賬時才疊上去**的——一上來就鋪雙層,是 `01` 紅旗清單「過度設計」。**「先單流跑通看板,計費再疊精確層」**這句演進,比一口氣畫滿雙層更顯資深。

## 5. Deep Dive(挑 3 個最會被往死裡鑽的點)

### Deep Dive ① · 流式聚合管道:窗口、exactly-once、checkpoint(這題的骨架)

這是「流式系統」題的母題——所有實時數倉/監控/特徵計算都是這個骨架。要鑽三層:**窗口怎麼開、exactly-once 怎麼保證、checkpoint 是什麼。**

**(a) 窗口聚合:滾動 vs 滑動(先講清「按什麼開窗」)**

| 窗口類型 | 怎麼開 | 適用 | 例 |
|---|---|---|---|
| **滾動窗(Tumbling)** | 固定長度、不重疊,事件只屬於一個窗 | 「每分鐘/每小時的總點擊」這種**對齊的、不重疊**的統計 | 1min 滾動:`[3:00,3:01)`、`[3:01,3:02)`… 算每分鐘 PV |
| **滑動窗(Sliding)** | 固定長度、按步長滑動、**會重疊**,事件可屬於多個窗 | 「最近 5 分鐘的點擊率」這種**移動平均** | 窗長 5min、步長 1min:每分鐘輸出「過去 5min」 |
| **會話窗(Session)** | 按「活動間隔」切分,無固定邊界 | 用戶一次瀏覽會話(這題少用,廣告場景偶用) | 兩次點擊隔 >30min 算兩個會話 |

> **面試話術:** 「看板的『每分鐘大盤』用**滾動窗**(不重疊、對齊整分);『最近 5 分鐘 CTR』這種移動指標用**滑動窗**(重疊、按步長輸出)。窗口的時間軸我用 **event-time**(事件帶的 `ts`),不是事件到達時間——否則網絡抖一下統計就串窗了。」

**(b) exactly-once 怎麼保證(這題的真考點:計費不能丟不能重)**

「至少一次 / 至多一次 / 恰好一次」是流式系統的核心一致性問題,**錨 `00` 能力包 D / `financial-consistency` 的 exactly-once 語義**:

| 語義 | 含義 | 後果(對計費) |
|---|---|---|
| **at-most-once** | 可能丟,不會重 | ❌ 計費少算,廣告主少付,平台虧 |
| **at-least-once** | 不丟,可能重 | ❌ 計費多算,廣告主多付,投訴 |
| **exactly-once** | 不丟不重 | ✅ 計費精確,唯一可接受 |

**黑盒裡 Flink 怎麼做到 exactly-once(這是 deep dive 的肉,講出來過資深→架構師線):**

exactly-once **不是「每條只處理一次」**(物理上做不到,失敗會重放),而是 **「對結果的影響恰好一次」(effectively-once)**。靠兩根支柱:

1. **checkpoint(分布式快照 / Chandy-Lamport 算法):** Flink 週期性(如每 30s)給整個流的狀態打一個**一致性快照**——包括「每個算子的窗口聚合狀態」+「每個 Kafka 分區消費到哪個 offset」。實現上靠一個特殊的 **checkpoint barrier(屏障)** 隨數據流往下飄,算子收到 barrier 就快照自己的狀態。失敗時:**從上一個 checkpoint 恢復狀態 + 把 Kafka offset 倒回快照點重放**——狀態和 offset 是**原子綁定**的,所以「重放的那段事件」對應的「狀態更新」也一起回滾了,不會重複累加。
2. **two-phase commit(兩階段提交)到 sink:** 光恢復還不夠——如果 Flink 已經把結果寫進 ClickHouse 才崩,重放會寫第二次。所以 sink 要**事務化**:Flink 把「寫 sink」做成 **預提交(pre-commit)→ checkpoint 成功才正式 commit** 的兩階段。崩了就回滾未提交的,重放後重新寫,**對外只可見一次**。

> **面試話術(把機制講透):** 「Flink 的 exactly-once 不是物理上每條只算一次,是**對結果恰好一次**。靠兩件事:**checkpoint** 把『算子狀態 + Kafka offset』原子快照,崩了一起恢復+重放,狀態不會重複累加;**sink 兩階段提交**保證寫下游(ClickHouse/計費庫)也是『checkpoint 成功才 commit』。前提是 **source 可重放(Kafka 能按 offset 倒回)+ sink 可事務/冪等**——這兩個前提缺一個,exactly-once 就退化成 at-least-once。」

> **資深加分一句(主動暴露代價):** 「exactly-once 不免費——checkpoint 越頻繁、延遲越低但開銷越大;兩階段提交會讓結果有『commit 延遲』(下游要等 checkpoint 才見到數)。**所以看板那條快路我可能用 at-least-once + 下游冪等去重**(看板差一點沒事、要低延遲),**只在計費那條慢路用嚴格 exactly-once**——這又回到雙層架構:不同 SLA 用不同一致性級別。」

### Deep Dive ② · 近似算法:HyperLogLog 算 UV、Count-Min Sketch 算頻次(這題的「魔法」)

第 2 步算出「億級基數去重會爆內存」——這節就是解法。**核心思想:用「可控誤差」換「常數級內存」**,布隆過濾器(`06 緩存` / `13 爬蟲`)是這個家族的入門款,HLL/CMS 是進階款。

**(a) HyperLogLog(HLL):為什麼能用 ~12KB 估億級 UV?**

精確去重要存所有 ID(內存隨基數線性增長);HLL **完全不存 ID**,只存「一組桶的最大前導零數」,內存**固定**:

| | 精確 Set | HyperLogLog |
|---|---|---|
| 存什麼 | 全部不同的 ID | **m 個桶,每桶一個小整數(前導零的最大值)** |
| 1 億 UV 內存 | ~1.6 GB(線性增長) | **~12 KB(固定!)** |
| 結果 | 精確 | **估計值,誤差 ~2%**(`1.04/√m`) |
| 可合併? | 可(Set 並集,但大) | **可(每桶取 max),且合併後誤差不變** ← 關鍵 |

> **黑盒裡的直覺(別只背公式,給程序員視角):** HLL 的洞察是——**「我見過的不同 ID 越多,就越可能見過某個 hash 值開頭有很多個 0 的 ID」**(拋硬幣連續 k 個正面很罕見,見到了說明拋了很多次)。它把每個 ID 哈希成一串隨機 bit,記錄「見過的最長前導零」,前導零越長 → 估計見過的不同 ID 越多(≈ 2^前導零)。單個估計方差大,所以分成 m 個桶各自估、再調和平均,把誤差壓到 `1.04/√m`。**用「最罕見事件的稀有度」反推「樣本量」,所以不用存樣本本身——這就是它省內存的根。**

> **「可合併且誤差不變」為什麼是殺手鐧(呼應第 3 步 sketch 設計):** 因為窗口聚合要**把小窗合併成大窗**——分鐘 UV 合成小時 UV 合成天 UV。Set 也能合併(求並集)但越並越大;HLL 合併只是「每個桶取兩邊的 max」,**內存恆定、誤差恆定**。所以 ClickHouse/Flink 存的是 **HLL sketch 而不是 UV 數字**(第 3 步那個設計),查詢時把涉及的所有 sketch 合併再估。**這是近似算法在流式窗口裡能用的根本原因。**

**(b) Count-Min Sketch(CMS):估「某個 key 出現幾次」(頻次)**

HLL 答「有多少**不同**的」(基數);CMS 答「**某個具體的**出現**幾次**」(頻次):

| | 精確計數(HashMap) | Count-Min Sketch |
|---|---|---|
| 存什麼 | 每個 key 一個計數器 | **d×w 的二維小矩陣 + d 個 hash 函數** |
| 海量 key 內存 | 隨不同 key 數線性增長 | **固定(幾 MB)** |
| 查 key 頻次 | 精確 | **估計,只會高估不會低估**(取 d 行的 min) |
| 場景 | key 不多時 | **海量 key、找 Top-K 熱點 / 防刷計頻** |

> **黑盒裡發生什麼:** CMS 把每個 key 用 d 個不同 hash 映到 d 行的某一格,計數時 d 格全 +1;查詢時取**這 d 格的最小值**(因為 hash 衝突只會讓某格被別的 key 也加過 → 偏大,取 min 過濾掉一部分衝突)。所以 **CMS 只會高估、不會低估**。這題用它做什麼?**找「點擊頻次異常高的 deviceId」(防刷)** 和 **Top-K 熱門廣告**——海量 device,不可能每個都精確計數,CMS 用幾 MB 估出「誰點得異常多」。

> **面試話術(把「什麼時候近似夠用」講清,這是分水嶺):** 「我對指標分兩類:**看板的 UV / 熱榜 / 頻次,用近似算法**——UV 用 **HyperLogLog**(~12KB 估億級,誤差 2%,且可合併所以能跨窗累加),Top-K 熱點和防刷計頻用 **Count-Min Sketch**(幾 MB 估海量 key 頻次)。判斷標準是:**這個數字用來『看趨勢、做監控』就近似夠用,差 2% 沒人在意;用來『結算金錢』就絕不能近似。** 所以計費那條路我走精確去重(批量、或用 RoaringBitmap 壓縮位圖精確存),看板這條路全用近似。」

> **近似 vs 精確 vs 內存的取捨表(主動給全景,架構師信號):**

| 訴求 | 算法 | 內存 | 誤差 | 用在哪 |
|---|---|---|---|---|
| 「在不在」(存在性) | 布隆過濾器 | 小 | 有假陽性、無假陰性 | 防穿透(`06`)、URL 去重(`13`) |
| 「有多少**不同**的」(基數/UV) | **HyperLogLog** | ~12KB | ~2% | 看板 UV |
| 「**這個**出現幾次」(頻次/Top-K) | **Count-Min Sketch** | 幾 MB | 只高估 | 熱榜、防刷計頻 |
| 「精確不同的」(計費 UV) | RoaringBitmap / 批量精確 | 隨基數(但壓縮) | 0 | 計費對賬 |

### Deep Dive ③ · 精確計費 + 亂序遲到 + 防刷(雙層架構的命門)

這是這題和 `03 短鏈`「秒級異步聚合夠用」的本質區別——**廣告計費一分錢不能錯**,逼出三個前面題沒有的硬約束。

**(a) Lambda 架構:近似看板 + 精確對賬「雙層」**

| | 快路(Speed Layer) | 慢路(Batch Layer) |
|---|---|---|
| 餵 | 實時看板 | 計費結算 / 對賬 |
| SLA | 秒級延遲 | 可延遲幾分鐘~小時 |
| 準確 | 近似(HLL/CMS)+ at-least-once | **精確 + exactly-once + 去重防刷** |
| 技術 | Flink streaming → ClickHouse/Redis | 全量明細重放 → 批量精確聚合 |
| 角色 | 給人「現在大概多少」 | 給錢「到底是多少」 |

> **面試話術:** 「這就是經典 **Lambda 架構**:**快路**用 Flink 流式 + 近似,秒級餵看板,容忍小誤差;**慢路**從全量明細重放、批量精確聚合,出真正的計費數。**看板看快路的,結算用慢路的,慢路還反過來校驗快路有沒有飄。** 為什麼不只用一條?因為快路為了低延遲犧牲了精確(近似 + at-least-once),計費受不了;慢路為了精確犧牲了延遲,看板等不及。**兩個 SLA 物理上不可兼得,所以分兩層。**」

> **主動暴露 Lambda 的代價 + 演進(架構師線):** 「Lambda 的痛點是**兩套代碼維護**——同樣的聚合邏輯快路寫一遍(Flink streaming)、慢路寫一遍(批),容易不一致。業界的演進是 **Kappa 架構:只保留一條流式管道,靠 Kafka 可重放,要『精確版』就把歷史 offset 重放一遍用同一套流式代碼算**——一套邏輯、靠重放補精確。**我會說明白:Lambda 雙層是穩妥起步,但若團隊能把流式算子做到 exactly-once + 可重放,Kappa 能省掉一套代碼。** 選哪個看團隊對流式 exactly-once 的掌控力。」

**(b) 亂序 / 遲到數據:watermark(水位線)**

事件**不會按順序到**——3:00:59 的點擊可能因為手機弱網,3:05 才上報。如果 3:01 就關掉「3:00~3:01」這個窗,這條遲到的點擊就**丟了**(計費少算)。怎麼辦?

**watermark(水位線)= 一個「我認為 event-time 已經推進到這裡了」的標記:**

```
window [3:00, 3:01)
  ├─ 正常:event-time 到 3:01 就該關窗
  ├─ 但允許遲到:設 watermark = max(event-time) - 2min(容忍 2 分鐘亂序)
  │    → watermark 推到 3:01 時(即見過 3:03 的事件),才觸發 [3:00,3:01) 計算
  │    → 但窗口狀態保留到 allowedLateness(再等 N 分鐘),這期間遲到的還能補進來
  └─ 超過 allowedLateness 才到的「太遲事件」→ 進側輸出流(side output),走計費補償
```

> **黑盒裡發生什麼(把 watermark 講成「等多久才結帳」):** watermark 是流式系統的**「結帳時機」決策**——關早了丟遲到數據,關晚了延遲變大。它是一個**「容忍亂序」和「結果延遲」的取捨旋鈕**:watermark 設 `max_event_time - 2min`,意思是「我等到看見 3:03 的事件,才相信 3:01 的數據都到齊了,才結 3:00 那一分鐘的帳」。**`allowedLateness`** 再給一層保險:窗口算完還不立刻丟狀態,再等 N 分鐘,這期間遲到的能**增量更新**已輸出的結果。**實在太遲的進 side output 單獨處理**(計費場景:走補償對賬,不能直接丟)。

> **面試話術(把旋鈕的取捨攤開,`01` Step 7 加分核心):** 「亂序我用 **watermark** 處理——它本質是『等多久才相信數據到齊了』的旋鈕:容忍 2 分鐘亂序就設 `watermark = max_ts - 2min`,代價是結果晚 2 分鐘出。再加 `allowedLateness` 給遲到數據一個增量補窗的窗口期。**計費場景關鍵是:超期太遲的事件不能丟,要走 side output 進補償流,最後對賬時補回去**——這就是計費和看板的差別,看板丟一條無所謂,計費丟一條就是少收一筆錢。」

**(c) 去重防刷:點擊不能被刷**

廣告計費按真實點擊算,**機器人/作弊刷點擊 = 廣告主白付錢**,所以「點擊」進計費前要過兩道去重:

| 去重類型 | 防什麼 | 怎麼做 |
|---|---|---|
| **技術去重(exactly-once 級)** | 同一條點擊因重試/重放被算兩次 | 每個事件帶**唯一 `eventId`(冪等鍵)**,聚合前按 eventId 去重(Flink 狀態 / RoaringBitmap);呼應 `14 支付`冪等鍵 |
| **業務去重 / 防刷** | 真人/機器人的**異常重複點擊**(同 device 1 秒點 100 次) | 規則 + CMS 計頻:同 `deviceId`/`ip` 在窗口內點擊 >閾值 → 標記可疑,進反作弊;短時重複點擊只算一次(去重窗口) |

> **面試話術:** 「點擊去重分兩層:**技術層**靠每事件帶 `eventId` 冪等鍵,防重試/重放重複計數(和 `14 支付` 的冪等鍵同款);**業務層**防作弊——用 Count-Min Sketch 實時統計每個 device/ip 的點擊頻次,異常高的進反作弊規則,計費只認去重後的『有效點擊』。**這也是為什麼計費必須走慢路精確層——防刷規則複雜、要回溯,流式快路來不及。**」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:這題花在「流式計算」和「明細存儲」,不在 QPS

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **流式計算(Flink 集群)** | 7×24 常駐、有狀態(窗口 state 駐內存)、checkpoint 開銷 | **主要成本項**。狀態大小 = 並發 key 數 × 每 key state;UV 用 HLL 而非 Set,**直接把 state 從 GB 壓到 MB**——近似算法在這裡省的不只內存,是 Flink 整個集群規模 |
| **明細存儲** | PB 級洪流(20TB/天 ×3) | 明細**只熱留 90 天 + 冷歸檔到對象存儲/數據湖**(便宜冷存),計費對賬要時從冷層重放 |
| **聚合結果(ClickHouse)** | 聚合後很小(GB~TB) | 便宜,全進熱層;列存壓縮比高 |
| **看板查詢** | 低頻 OLAP | 加結果緩存(Redis),看板首屏不每次掃 CH |

> **成本的核心 insight:** **近似算法(HLL/CMS)在這題省的是 Flink 狀態內存 → 直接砍集群成本。** 把「1 億 UV 的 1.6GB Set」換成「12KB HLL」,乘以百萬維度組合,是 TB → MB 級的差別——這就是為什麼「看板用近似」不只是「夠用」,而是**讓系統在成本上可行**。

### 容量規劃:流式系統盯「consumer lag」不是「CPU」

> 傳統系統盯 CPU/QPS,**流式系統的命脈是 Kafka consumer lag(消費滯後)**——它是「處理速度跟不跟得上入流」的水位線:
> - **lag 穩定(處理 = 入流)→ 健康。**
> - **lag 持續增長 → 處理跟不上,看板延遲會無限拉大。** 對策:**擴 Flink 並行度(parallelism)+ 加 Kafka 分區**(但分區數一旦定大很難縮,要預留)。
> - **容量規劃要算「重放追趕能力」:** 如果 Flink 掛了 1 小時,積壓 1 小時的數據,**集群追得上嗎?** 追趕速度必須 > 入流速度,否則永遠追不上。架構師要按「故障後 N 分鐘內追平」反推 Flink 並行度餘量。

### 演進路徑:從單流看板到 Lambda 到 Kappa

```
階段 1(MVP):Kafka → Flink → ClickHouse,只跑看板快路
   UV 用近似(HLL),at-least-once + 下游冪等,先讓看板有數
        │  業務要計費 / 要對賬
        ▼
階段 2:疊計費精確層(Lambda 雙層)
   明細冷歸檔 + 批量精確聚合(精確去重、防刷、Decimal、exactly-once)
   快路餵看板,慢路出計費,慢路校驗快路
        │  兩套代碼維護成本痛、團隊能 hold 住流式 exactly-once
        ▼
階段 3:收斂到 Kappa(可選)
   只留一條流式管道,靠 Kafka 重放補精確,砍掉批處理那套代碼
```

> **這條演進的關鍵 insight:** 和 `03 短鏈`「第一天就知道終局但不該第一天就上」同一個道理——**別 MVP 就鋪 Lambda 雙層**。先單流跑通看板(這已經能 demo、能拿數),計費精確層是「真要結算」才疊。**講演進**(「現在單流看板,要計費我再疊精確層,團隊熟了考慮收斂 Kappa」)比一上來畫滿雙層**更顯資深**——這是 `01` 紅旗「過度設計」在流式題上的對應。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **Flink 作業崩潰** | 看板停更、lag 堆積 | **從 checkpoint 恢復 + Kafka offset 重放**,exactly-once 保證不丟不重;靠 Kafka 緩衝扛住恢復期積壓 |
| **Kafka 分區熱點**(爆款廣告) | 一個分區/算子被打爆,該廣告聚合延遲 | **兩階段聚合打散熱 key**(`adId+隨機後綴` 局部聚合再合併),同 `08` 大 V 手法 |
| **遲到數據超 watermark** | 計費可能少算 | 太遲事件進 **side output 走補償對賬**,絕不直接丟(計費場景紅線) |
| **ClickHouse 寫入跟不上** | 看板數據延遲 | Flink 批量寫 + 攢批;CH 寫入是瓶頸時降低落庫頻率(窗口觸發間隔調大) |
| **快路慢路結果對不上** | 計費爭議 | 這正是 Lambda 的價值——**慢路是真相源(source of truth),快路飄了以慢路為準**,並告警查快路 bug |

> **架構師話術:** 「我的核心單點是 **Flink 有狀態作業** —— 它掛了看板停更、lag 堆積。但流式的好處是**狀態 + offset 綁定在 checkpoint 裡,恢復後重放就能 exactly-once 追平**,Kafka 當緩衝扛住恢復期。真正不能容忍的失效是**計費少算**——所以遲到超期的事件我絕不丟,進 side output 走補償;且**慢路批量結果是計費的真相源,快路只是實時近似**,兩者對不上以慢路為準。這套『近似快路 + 精確慢路 + 慢路兜底』就是 Lambda 的全部意義。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **小流量統計**(每天幾萬點擊) | 別上 Kafka+Flink+HLL 全家桶 | 直接 `INSERT` + 定時 `GROUP BY` 聚合,甚至 ClickHouse 物化視圖搞定,別過度設計 |
| **只要看板、不要計費** | 別鋪 Lambda 雙層 | 只跑單流快路 + 近似,at-least-once 夠了,砍掉精確慢路 |
| **基數很小的 UV**(就幾千用戶) | 別用 HyperLogLog(近似誤差反而搶眼) | 直接精確 Set / `COUNT(DISTINCT)`,小基數精確比近似還省心 |
| **強事務一致**(扣庫存、轉賬) | 別用「最終一致的流式聚合」 | 那是 `14 支付` / `16 秒殺` 的強一致題,流式聚合是最終一致,別混 |
| **要查單條明細**(某用戶某次點擊詳情) | 別只存聚合結果 | 聚合 ≠ 明細;要明細查詢另留明細存儲(這題明細在冷層,點查慢) |

> **這張表是架構師線的精華**——「知道近似算法的邊界(小基數別用 HLL)、知道 Lambda 的邊界(只要看板別雙層)、知道流式的邊界(強事務別用流式聚合)」,比「知道 HLL 怎麼算」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 廣告聚合的容量估算結論一句話? | **矛盾不在 QPS**(100 萬/s 用 32~64 Kafka 分區吞下)——三個真矛盾:**億級基數去重爆內存**(→HLL)、**明細 PB 級但聚合很小**(→分層:Kafka+ClickHouse+冷歸檔)、**看板/計費 SLA 分裂**(→Lambda 雙層)。 |
| 2 | 這題的「題眼」是什麼?澄清第一句該說啥? | **同一題長著兩張臉:看板(秒級、近似可接受)vs 計費(精確、不丟不重、可對賬)。** 第一句就拆開兩種 SLA,定下**近似快路 + 精確慢路**雙層。 |
| 3 | Kafka 為什麼按 `adId` 分區?有什麼隱患? | Flink 要 `keyBy(adId)` 聚合,**同 key 必須同分區同算子**才聚合正確;隨機分區會散開算錯。隱患:**爆款廣告熱點分區**(同大 V)→ 兩階段聚合 `adId+隨機後綴` 打散。 |
| 4 | Flink 的 exactly-once 怎麼做到?前提是什麼? | **不是物理只算一次,是對結果恰好一次**。靠 ①**checkpoint** 原子快照「算子狀態 + Kafka offset」,崩了一起恢復+重放不重複累加;②**sink 兩階段提交**。前提:**source 可重放 + sink 可事務/冪等**,缺一退化成 at-least-once。 |
| 5 | HyperLogLog 為什麼能用 ~12KB 估億級 UV?殺手鐧? | **不存 ID,只記「見過的最長前導零」反推樣本量**(罕見事件稀有度→樣本數),內存固定 ~12KB、誤差 ~2%。殺手鐧:**可合併(每桶取 max)且誤差不變** → 所以存 **HLL sketch 不存數字**,讓 UV 能跨窗累加(分鐘→小時→天)。 |
| 6 | Count-Min Sketch 估什麼?和 HLL 的分工? | **HLL 答「有多少不同的」(基數/UV);CMS 答「這個出現幾次」(頻次)。** CMS 用 d×w 小矩陣 + d 個 hash,查時取 d 格 min(**只高估不低估**)。這題 CMS 用於 **Top-K 熱榜 + 防刷計頻**(海量 device)。 |
| 7 | 亂序/遲到數據怎麼處理?watermark 是什麼旋鈕? | **watermark = 「等多久才相信數據到齊、才結帳」的旋鈕**:`max_event_time - 2min` 容忍 2 分亂序、代價是晚 2 分出數;`allowedLateness` 給遲到增量補窗;**太遲事件進 side output 走補償(計費不能丟)**。按 **event-time 不是 processing-time** 開窗。 |
| 8 | 為什麼要 Lambda 雙層?何時收斂 Kappa?何時別用這套? | **看板低延遲犧牲精確、計費精確犧牲延遲,兩 SLA 不可兼得→雙層**(慢路是真相源校驗快路)。團隊 hold 住流式 exactly-once+重放 → **收斂 Kappa**(一套代碼靠重放補精確)。**別用:** 小流量(物化視圖即可)、只要看板(單流)、小基數(精確 Set 別用 HLL)、強事務(走支付題)。 |

## 一句話總綱(背起來)

> **廣告點擊聚合是「一題兩張臉」題——看板要秒級近似、計費要精確不丟不重,SLA 分裂逼出 Lambda 雙層。骨架是 Kafka(分區吞百萬流+可重放)→ Flink(按 key 開窗聚合,checkpoint+兩階段提交保 exactly-once)→ ClickHouse(聚合熱層)。魔法在近似算法:UV 用 HyperLogLog(~12KB 估億級、可合併所以存 sketch 不存數字)、頻次/防刷用 Count-Min Sketch,把 Flink 狀態從 GB 壓到 MB 直接砍集群成本。亂序用 watermark(等多久結帳的旋鈕)+ side output 補償,計費的事件絕不丟。架構師線:MVP 只跑單流看板別鋪雙層、流式盯 consumer lag 不盯 CPU、慢路是計費真相源、近似只用在『看趨勢』不用在『算錢』。流式細節錨 `big-data/Flink+ClickHouse`、計費精確錨 `financial-consistency`、扇出家族對比 `08`/`10`。**
