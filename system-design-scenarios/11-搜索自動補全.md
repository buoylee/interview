# 11 · 搜索自動補全(Typeahead / Autocomplete)

> **T2 裡最「反直覺」的一題。** 你在 Google 搜索框敲一個字母,提示就跳出來——看起來只是「查一下熱門詞」,但它藏著兩個極端條件:**讀 QPS 高到離譜**(每敲一個字母發一次請求,一次搜索敲 20 下 = 20 個請求)+ **延遲必須壓到人察覺不到**(<100ms,人在打字,慢一點提示就「追不上手」)。這兩個條件一疊,**通用 DB / 模糊查詢 / `LIKE 'abc%'` 全部出局**——這題逼你走「**為一種訪問模式定制一個專用數據結構**」的路,Trie 就是答案。
>
> 它的教學價值高度集中在三處:**Trie 每節點預存 Top-K**(為什麼不能查詢時實時算 top-k——`06 緩存` 的「預計算」思想在這題的極致版)、**離線聚合 + 近實時更新**(讀寫分離的母題:服務只讀快照,熱度離線重算)、**按前綴分片不需要跨片合併**(`07 數據規模化` 的分片鍵在這題乾淨得出奇——前綴定片,單片命中)。把這三個鑽透,自動補全就通了。
>
> 用 `00` 依賴表看:自動補全吃 **A(框架數字感)+ B(存儲)**,C/D/E/F 幾乎不沾——它和 `03 短鏈` 同屬「**純框架 + 存儲選型**」題,但**比短鏈難一檔**:短鏈的存儲是「拿什麼存」,自動補全的存儲是「**為訪問模式設計一個結構**」,這是架構師選型的進階肌肉。

## 1. 題目 + 澄清問題清單

**題目:** 設計一個搜索自動補全(像 Google / 淘寶搜索框)。使用者在輸入框敲字,系統根據**已輸入的前綴**,實時返回 Top-K 個最可能的補全建議(通常按熱度排序)。

**別一上來就畫**——先用 `01` Step 1 的「問三類」把這道開放題收斂:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 按什麼排序——熱度?個性化?要不要拼寫糾錯?要不要中途插詞?返回幾個? | 核心:**前綴 → Top-K 熱門補全**。排序按**全局熱度**(個性化算加分項,先不做);**拼寫糾錯**做輕量版(差一個字母也能命中);返回 **Top 5~10** 條 |
| **非功能(SLA)** | 延遲?讀寫比?提示要多新鮮?可用性? | **延遲是命門:P99 <100ms**(人在打字,提示要跟得上手);**讀 >>> 寫**(讀是「每敲一鍵一次」,寫是「熱度離線更新」,比短鏈更懸殊);**提示可以不新鮮**——昨天的熱度榜今天用完全 OK(突發熱點除外);**高可用 > 強一致**(提示晚幾小時更新沒人死) |
| **規模 / 約束** | 詞庫多大?支援幾種語言?QPS 多大?更新多頻繁? | 詞庫 **~千萬級**查詢短語;先做**英文**(中文/拼音是擴展);搜索 QPS 大盤按 **Google 級**估(下面算);熱度**離線批量**重算(天級),突發熱點**近實時**插入(分鐘級) |

**面試話術(把假設說出來讓對方確認):**
> 「我先框一下——核心就一件事:給定使用者敲的前綴,返回 Top-K 個最熱門的補全。我假設按**全局熱度**排序,個性化先不做、算加分項;拼寫糾錯做個輕量版。最關鍵的非功能約束是**延遲**:它在使用者打字的關鍵路徑上,P99 得壓在 100ms 內,不然提示追不上手。讀寫極度不對稱——讀是『每敲一個字母一次請求』,寫是『熱度離線重算』,所以這題我會**讀寫徹底分離**:服務只讀一份預計算好的快照,熱度在後台離線重算。提示**可以不新鮮**,昨天的榜今天用沒問題,但突發熱點要能近實時進庫。這樣可以嗎?」

**這幾個澄清直接決定後面:**

- 「P99 <100ms + 每鍵一請求」→ 第 2 步估算會算出**讀 QPS 比搜索本身還高一個量級**,且第 5 步架構**重心全在讀延遲**——這是 deep dive ① **預存 Top-K** 和 deep dive ③ **CDN/防抖** 要解的。
- 「按熱度 Top-K」→ 排除「查詢時把命中前綴的所有詞撈出來再排序」,逼出 deep dive ① 的「**每個前綴節點預存排好序的 Top-K**」。
- 「讀寫分離 + 提示可不新鮮」→ deep dive ② 的「離線聚合 + 只讀快照」整條路成立。
- 「突發熱點近實時」→ deep dive ② 的「離線為主 + 實時插入旁路」雙軌。
- 「按前綴查」→ 第 4 步分片鍵乾淨地落在「**按前綴分片**」上(deep dive ③),沒有跨片合併。

**常見陷阱:** 在「要不要支援中間詞匹配(輸入 `book` 匹配 `the book of`)」「要不要做個性化」上糾結太久。✅ 抓大放小——先把「前綴 → Top-K 熱門」這條主幹做透,中間詞 / 個性化作為演進方向點一句即可。

## 2. 容量估算(直接調用 `02` 的數字)

> 這題的估算**最反直覺**——它的讀 QPS 不是「搜索 QPS」,是「搜索 QPS **× 每次搜索敲的鍵數**」。這個放大係數是這題的靈魂,一定要算出來、說出來。

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 = 平均 ×3、單 Redis ~10 萬 QPS、Redis ~0.5ms vs DB 走盤幾 ms**。

### ① 讀 QPS(真正的怪獸,且帶放大係數)

假設大盤 **40 億次搜索/天**(Google 級量級;面試說「假設搜索量級 N」即可,重點是推導):

```
搜索 QPS:4e9 / 1e5 = 4 萬 QPS(平均)
                    ↓  關鍵放大:每次搜索敲 ~20 個字母,每敲一鍵一次補全請求
補全 QPS:4 萬 × 20 = 80 萬 QPS(平均)
                    ↓  峰值 ×3
補全峰值:≈ 240 萬 QPS
```

> **所以:這是這題的第一個矛盾——補全請求是搜索請求的 ~20 倍,輕鬆到百萬 QPS 級。** 對照 `02` 單 Redis ~10 萬 QPS,光扛這個讀量就要 **20+ 個分片/副本**。而且每個請求 P99 還得 <100ms。**這直接判了「查詢時臨場算」的死刑**——百萬 QPS 下你連一次 DB 查詢都做不起,更別說排序。**唯一活路是把答案預計算好、放進內存、就近返回**(deep dive ①③)。

**放大係數能不能砍?能——這是優化的第一刀(`01` Step 2 估完接「所以」):**

| 砍法 | 機制 | 省多少 |
|---|---|---|
| **客戶端防抖(debounce)** | 敲鍵後等 ~50–100ms 沒新輸入才發請求;連打不發 | 把「每鍵一請求」壓成「每停頓一請求」,**QPS 砍掉一半以上** |
| **前綴增量(只在前綴變長時請求)** | 已有 `ab` 的結果,使用者繼續敲 `abc`,客戶端可先用本地過濾 | 進一步減請求 |
| **客戶端/CDN 緩存熱前綴** | `a`、`ab`、`fa` 這種熱前綴的結果就近緩存 | 熱前綴根本不到源站 |

> **所以:估算結論不是「我要扛 240 萬 QPS」,而是「我要先用防抖 + 邊緣緩存把它砍下來,剩下的用預計算 Trie + 分片扛」。** 這個「先砍需求再扛」的順序,就是 `01` 說的資深話術。

### ② 寫 QPS(低到可以離線做)

熱度更新不是「每次搜索實時改榜」——那會把寫變成另一個百萬 QPS 怪獸。**寫是離線的:**

```
搜索日誌全量收集(這是寫,但寫進日誌/Kafka,不是寫進服務)
   → 離線批量聚合(天級 / 小時級)重算熱度
   → 重建 Trie 快照,灰度推給服務
```

> **所以:服務側的「寫」幾乎為零——它讀的是一份只讀快照。真正的「寫」是離線管線的批處理,跑在 Hadoop/Spark 上,不在線上關鍵路徑。** 這就是 deep dive ② 的讀寫分離:**線上只讀,熱度離線算**。突發熱點是這條規則的例外(分鐘級實時插入),單獨處理。

### ③ 存儲(詞庫 + Trie,內存能不能裝下是關鍵)

千萬級短語,算 Trie 的內存佔用(這題的存儲問題不是「總量多大」,是「**能不能塞進內存**」——因為要 <100ms,必須在內存):

```
1000 萬短語 × 平均 ~20 B/短語(短語本身)= 200 MB(裸短語)
Trie 結構開銷(每字符一節點 + 指針 + 每節點預存 Top-K)
   → 預存 Top-K 是內存大頭:每個前綴節點存 K=10 個候選短語
   → 樸素估:節點數 ~數億級,每節點掛 Top-K → 數 GB ~ 數十 GB
```

> **所以:第二個矛盾——預存 Top-K 雖然換來查詢飛快,但內存會膨脹到數十 GB 級。** 單機內存裝得下(現代機器幾百 GB),但「裝得下」≠「該裝一台」——這逼出兩件事:(1) **壓縮 Trie / Radix Tree** 把節點數壓下來(deep dive ①),(2) **按前綴分片**,每片只裝一部分前綴的 Trie(deep dive ③)。這條結論在第 6 步架構師深挖層展開。

### ④ 帶寬

每個補全請求的回應就是 Top 5~10 條短語(幾百字節到 1KB),帶寬本身不是瓶頸——但**請求數是怪獸**,所以帶寬壓力來自「百萬 QPS × 小 payload」,解法和 QPS 同源(邊緣緩存 + 防抖)。**這題沒有媒體帶寬問題**(對比 `02` Drill 5 直播)。

### 一句話估算結論(面試一定要說出來)

> 「**這題的數字有個放大陷阱——補全 QPS 不是搜索 QPS,是它的 ~20 倍,輕鬆到百萬級,而且每個請求還要 <100ms。所以我的設計分兩刀:先用客戶端防抖 + CDN/邊緣緩存把請求量砍下來;剩下的用『每個前綴節點預存好 Top-K 的 Trie』把查詢壓成一次內存讀,再按前綴分片扛總量。寫是離線的——熱度靠批量聚合搜索日誌重算,線上服務只讀一份快照,所以這題本質是『極致讀優化 + 讀寫徹底分離』。**」

這句把「百萬 QPS + 100ms」翻成「砍 + 預計算 + 分片 + 讀寫分離」的總結,就是 `01` 說的資深 vs 初級的分水嶺。

## 3. API + 數據模型

### API(`01` Step 3:快過,點到為止)

```
GET /autocomplete?q={prefix}&limit=10&lang=en
  → 200 { suggestions: ["facebook", "facetime", "facebook login", ...] }
       # 按熱度降序;這是被每秒呼叫百萬次的那個接口,必須極輕

# 數據收集(不在關鍵路徑,異步)
POST /log/search   { query, userId?, ts }   → 202 Accepted
       # 使用者真正按下搜索後才記;埋進日誌/Kafka,離線聚合用
```

**要點:**
- 補全接口是**唯讀、無副作用、可重複**——這是它能被狠狠緩存(CDN/瀏覽器)的前提。**絕不能在這個接口裡做任何寫**(連「記一下這個前綴被查了」都不行——那會把唯讀接口變成百萬 QPS 的寫)。
- 熱度數據來自「使用者**真正提交的搜索**」(`POST /log/search`),不是「敲過的每個前綴」——**前綴本身不是熱度信號,完整搜索詞才是**。這個區分是 deep dive ② 的前提。

### 數據模型(`01` Step 4:選型 + 為訪問模式設計結構)

**核心訪問模式:給定前綴,返回按熱度排序的 Top-K 短語。** 圍繞它選結構——

**選型(決策樹錨 `system-design/06 存儲選型`):這題的選型是反面教材式的——先說「為什麼通用方案全錯」:**

| 候選 | 適不適合 | 黑盒裡為什麼(這是這題選型的肉) |
|---|---|---|
| **SQL `LIKE 'abc%'`** | ❌ 致命 | 前綴 `LIKE` 能走 B-Tree 索引範圍掃,但**命中前綴的詞可能成千上萬**,撈出來**還要實時按熱度排序**——百萬 QPS × 每次掃+排,DB 秒爆。且 `<100ms` 做不到 |
| **Elasticsearch(completion suggester)** | △ 能用但重 | ES 確實有專門的 completion suggester(底層就是 FST/有限狀態機,和 Trie 同源),**中小規模直接用它是合理工程選擇**。但在 Google 級百萬 QPS + 極致延遲下,通用 ES 集群的成本/延遲尾巴不如**專用 Trie 服務**;且 ES 強在「全文搜索」,自動補全只用到它的一小塊,殺雞用牛刀 |
| **Trie + 每節點預存 Top-K(專用內存結構)** | ✅ 主力 | **為這個訪問模式量身定制**:前綴查找天然是「沿樹走 N 步」O(prefix長度),走到節點**直接讀預存好的 Top-K**(已排序),零實時計算。這是 deep dive ① 的核心 |
| **Redis(輔助:緩存熱前綴結果)** | ✅ 配角 | 不是主存,是**把熱前綴的 Top-K 結果緩存**在 Redis,擋掉打到 Trie 服務的重複熱前綴(錨 `redis-handson/`) |

> **面試話術(把「為什麼不用通用方案」講透——這是這題最值錢的信號):** 「直覺會想用 `LIKE 'abc%'` 或 Elasticsearch,但這題的訪問模式太特殊:**前綴查找 + Top-K 熱度排序 + 百萬 QPS + 100ms**。`LIKE` 命中的詞可能上千,還要實時排序,DB 扛不住;ES 中小規模能用(它的 completion suggester 底層也是 FST/Trie 同源),但 Google 級下成本和尾延遲不划算。所以我會用**專用的 Trie + 每個前綴節點預存好排序的 Top-K**——把『查詢時的搜索 + 排序』全部挪到『離線建樹時』做掉,線上只剩一次 O(前綴長度)的樹遍歷 + 讀一個現成數組。這就是用『為訪問模式定制結構』換『極致讀性能』。」

**核心結構:Trie(前綴樹)+ 每節點 Top-K**

```
詞庫(離線聚合出的「短語 → 熱度分」):
  facebook → 9.2M    facetime → 1.1M    face → 3.3M    factory → 0.4M ...

Trie(每節點掛一個預排好序的 Top-K 候選列表):
            (root)
              │
              f
              │
              a
              │
              c
        ┌─────┴─────┐
        e           t
        │           │
       [Top-K @ "fac" 節點]   ← 走到這就直接返回,不再往下搜
        = [facebook, facetime, face, ...]  (離線已按熱度排好)
```

**分片鍵(`01` Step 4 鐵律:海量題必講「按什麼分」,錨 `system-design/07 數據規模化`):**

> **按前綴分片(prefix-based sharding)——這題的分片鍵乾淨得出奇。** 按前綴的頭一個或頭兩個字母路由:`a*` 的短語落 shard 0、`b*` 落 shard 1……查詢前綴 `fac` 時,根據首字母 `f` **直接定位到唯一一個分片**,在那片的 Trie 裡查。**關鍵 insight:前綴決定分片 = 查詢永遠單片命中,不需要跨片合併**(對比 `News Feed` 要合併多人 feed、`廣告聚合` 要跨片匯總——這題沒有,因為一個前綴的所有補全候選必然在同一片)。

> **唯一的坑是分片不均**(`s`/`a`/`t` 開頭的詞遠多於 `x`/`z`),所以不能簡單按首字母 26 等分,要**按前綴的詞頻做不均勻劃分**(熱門首字母拆細、冷門首字母合併)——這個 deep dive ③ 細講。

這就是為什麼自動補全是「好題」:分片鍵的選擇展示「按查詢維度分 → 單片命中無合併」這個漂亮信號,同時又埋了「分片傾斜」這個可以展開深度的點。

## 4. 高層架構(`01` Step 5:沿一個請求一鏡到底)

**先畫能跑的最簡版,再按需加件**(`01` 鐵律:別一上來拆 20 個微服務)。這題的圖天然分成**兩條完全解耦的路**——讀路徑(熱、百萬 QPS)和離線管線(冷、批處理),它倆只透過「一份只讀快照」相連。

```
   讀路徑(熱,百萬 QPS,P99 <100ms)— 這是主菜
 ┌──────────────────────────────────────────────────────────────┐
 │ Client(輸入框)                                                │
 │   │ 敲鍵 → 防抖 50–100ms → (本地/CDN 命中熱前綴就直接返回)      │
 │   ▼                                                            │
 │ CDN / 邊緣節點 ──hit── 直接回 Top-K(熱前綴根本不到源站)        │
 │   │ miss                                                       │
 │   ▼                                                            │
 │ API GW ──► Autocomplete Svc ──► Redis(熱前綴結果緩存)         │
 │                  │ miss              ▲                          │
 │                  ▼                   │ 回填                      │
 │            按首字母路由到分片 ──► Trie Shard(內存,只讀快照)    │
 │                  └─ 沿樹走 prefix → 讀節點預存的 Top-K → 回      │
 └──────────────────────────────────────────────────────────────┘

   離線管線(冷,批處理,天級/小時級)— 喂給上面那份「只讀快照」
 ┌──────────────────────────────────────────────────────────────┐
 │ 使用者真正搜索 ──► 搜索日誌 / Kafka                             │
 │        │                                                       │
 │        ▼                                                       │
 │  批量聚合(Spark/MapReduce):統計每個查詢詞的出現次數=熱度     │
 │        │                                                       │
 │        ▼                                                       │
 │  重建 Trie(每節點預算 Top-K)→ 產出新快照                      │
 │        │                                                       │
 │        ▼                                                       │
 │  灰度推送新快照 → Trie Shard 原子切換(讀不中斷)               │
 └──────────────────────────────────────────────────────────────┘

   實時旁路(突發熱點,分鐘級)— 繞過離線管線的快車道
   流式統計(Flink)發現某詞暴漲 ──► 直接增量插入 Trie / 改 Top-K
```

**兩條典型路徑走一遍(`01` Step 5:讀 + 寫各一條):**

- **補全(讀,熱路徑):** Client 敲鍵 → **防抖**(等使用者停頓)→ 先試 **CDN/客戶端緩存**(熱前綴直接命中)→ miss 才到源 → GW → Autocomplete Service → 查 **Redis 熱前綴緩存** → miss 才**按首字母路由到對應 Trie 分片** → 在那片的內存 Trie 裡**沿前綴走樹** → 走到節點**直接讀預存好的 Top-K** → 回。**全程零實時排序、零回源 DB、絕大多數請求在 CDN/Redis 就返回。**
- **更新(寫,冷路徑):** 使用者真正搜索 → 寫進**日誌/Kafka**(不碰服務)→ **離線批量聚合**統計熱度 → **重建 Trie 快照**(離線把每個節點的 Top-K 算好排好)→ **灰度推送 + 原子切換**新快照給各分片。**線上服務從頭到尾沒被寫過,它只是某個時刻換了一份只讀快照。**

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **客戶端防抖** | 把「每鍵一請求」砍成「每停頓一請求」 | ①「放大係數能砍」 |
| **CDN / 邊緣緩存** | 熱前綴(`a`/`fa`/`how`)就近返回,不回源 | ①「百萬 QPS 先砍再扛」 |
| **Redis 熱前綴緩存** | 擋掉打到 Trie 服務的重複熱前綴 | ①「讀極熱、熱點集中」 |
| **Trie 分片(內存只讀)** | 預存 Top-K → 查詢 = 一次內存樹遍歷,扛 <100ms | ①「判了臨場算的死刑」 |
| **按首字母路由** | 前綴定片,單片命中無合併 | ③「按前綴分片」 |
| **離線聚合管線** | 熱度離線重算,線上只讀快照 | ②「寫是離線的」 |
| **實時旁路(Flink)** | 突發熱點繞過天級管線,分鐘級進庫 | ②「突發熱點近實時」 |

**注意這張圖的「克制」與「解耦」:** 讀路徑和離線管線**完全不互相阻塞**——這是這題架構最漂亮的地方。讀路徑再忙,離線管線在後台慢慢算;離線管線在重算,讀路徑照樣用舊快照服務。它倆唯一的接觸點是「**原子換快照**」這一瞬。這種「**讀寫物理隔離,靠快照交接**」的形態,值得在白板上特別點出來(`01` 紅旗的反面:沒有把寫塞進讀路徑造成耦合)。

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · Trie 結構 + 每節點預存 Top-K(這題的真正考點)

**核心問題:走到前綴節點後,Top-K 是「現算」還是「預存」?** 這道選擇題決定整個架構的成敗。

**為什麼不能查詢時實時算 Top-K(先把錯誤方案的代價講透):**

樸素 Trie 的做法是:走到前綴節點 `fac`,然後**遍歷它底下的整棵子樹**,收集所有以 `fac` 開頭的短語(`facebook`、`facetime`、`face`、`factory`...),**再按熱度排序取 Top-K**。

```
查詢 "a" → 子樹下可能有【幾百萬】個詞 → 全遍歷 + 全排序 → 取 Top 10
```

> **黑盒裡的災難:** 前綴越短(`a`、`s`),子樹越大,要遍歷+排序的詞越多——**最熱的查詢(短前綴)恰好是最慢的查詢**。`a` 底下可能掛幾百萬個詞,每次查 `a` 都要遍歷幾百萬節點再排序,**百萬 QPS × 每次掃幾百萬 = 直接爆炸**,P99 別說 100ms,100s 都打不住。**這就是「查詢時算 Top-K」必須被槍斃的原因。**

**正確方案:每個前綴節點預存好排序的 Top-K(把計算挪到離線建樹時):**

```
建樹時(離線,一次性):
  自底向上算每個節點的 Top-K
    → 葉子節點的候選 = 它自己
    → 父節點的 Top-K = 合併所有子節點的 Top-K,再取前 K
    → 一路 merge 到 root
  每個前綴節點存好一個【已排序的 K 個候選】數組

查詢時(線上,百萬 QPS):
  走到節點 "fac" → 直接 return 它預存的 Top-K 數組
    → O(前綴長度) 走樹 + O(1) 讀數組 → 零排序、零子樹遍歷
```

| | 查詢時實時算 Top-K | **預存 Top-K(選它)** |
|---|---|---|
| 查詢複雜度 | O(子樹大小 × log) — 短前綴是災難 | **O(前綴長度)** — 與子樹大小無關 |
| 短前綴(`a`)性能 | 最慢(子樹最大) | **和長前綴一樣快** |
| 寫/建樹成本 | 低(不預算) | 高(離線自底向上 merge) |
| 內存 | 省(不存 Top-K) | **貴(每節點掛 K 個候選)** |
| 適配場景 | 寫多讀少、實時性高 | **讀多寫少、可離線預算**(正是這題) |

> **這就是 `06 緩存` 的「預計算」思想在這題的極致版:** 把昂貴的計算(遍歷+排序)從「每次查詢」挪到「離線建樹一次」,用**內存膨脹**和**建樹成本**換**查詢飛快**。這題讀寫比百萬:幾乎零,**用空間換時間是穩賺**。

**內存膨脹怎麼治——壓縮 Trie / Radix Tree(資深必答):**

樸素 Trie「每字符一節點」,`facebook` 要 8 個節點,大量單鏈路徑(`f→a→c→e→b→o→o→k` 中間全是只有一個孩子的節點)極度浪費。

| 優化 | 機制 | 黑盒裡發生什麼 |
|---|---|---|
| **壓縮 Trie / Radix Tree(基數樹)** | 把「只有一個孩子的連續節點」壓成一條邊存整個子串 | `facebook` 沒有分叉的部分壓成一條邊,節點數從 8 降到 2~3,**內存大降** |
| **只在「有意義的前綴節點」存 Top-K** | 不是每個字符節點都存 Top-K,只在常被查的前綴深度存 | 短前綴(1~5 字符)是查詢主力,深前綴查得少,可不預存、退化成現算 |
| **共享候選引用** | Top-K 存的是短語的引用/ID,不是短語全文 | 同一短語在多個前綴節點的 Top-K 裡只存一份 ID,去重 |
| **分層存儲** | 熱前綴的 Trie 常駐內存,冷前綴可下沉 | 長尾前綴沒人查,不必全裝內存 |

> **面試話術:** 「Top-K 我**絕不查詢時算**——短前綴的子樹有幾百萬個詞,而短前綴恰恰是最熱的查詢,實時算等於『最熱的請求最慢』,百萬 QPS 直接爆。我**離線建樹時自底向上把每個前綴節點的 Top-K 預算好排好**,線上查詢就退化成『走樹 O(前綴長度) + 讀一個現成數組』,和子樹多大無關。代價是內存膨脹,我用 **Radix Tree 壓掉單鏈路徑 + Top-K 只存短語 ID 引用 + 只在熱前綴深度預存**,把內存壓回可接受。」

### Deep Dive ② · 數據收集與近實時更新(讀寫分離的母題)

**核心問題:熱度從哪來、怎麼更新,又不拖垮百萬 QPS 的讀路徑?** 答案是**雙軌制**:離線批量為主(天級,扛全量),實時旁路為輔(分鐘級,追突發)。

**主軌:離線批量聚合 + 重建 Trie(讀寫分離的精髓)**

```
① 收集:使用者真正提交的搜索 → 寫進日誌 / Kafka
         (注意:只記完整搜索詞,不記敲過的每個前綴——前綴不是熱度信號)
② 聚合:Spark/MapReduce 批量統計每個查詢詞在窗口內的出現次數 = 熱度分
         (可加時間衰減:近期搜索權重高,讓榜跟得上趨勢)
③ 建樹:自底向上重建整棵 Trie,每節點預算 Top-K(deep dive ①)
④ 切換:新快照灰度推給各 Trie 分片 → 原子替換舊快照
```

> **為什麼必須讀寫分離(這題的核心架構決策):** 如果「每次搜索實時更新熱度榜」,寫就變成另一個百萬 QPS 怪獸,且**讀(查 Top-K)和寫(改 Top-K)會搶同一個節點的鎖**——熱前綴 `a` 的節點會成為**單點鎖熱點**(對比 `03 短鏈` 的點擊計數單行熱點)。所以**線上服務持有的 Trie 是純只讀的不可變快照**,熱度更新全在離線完成,做好一份新的、**整體原子換上**。讀永遠不被寫阻塞,這是百萬 QPS 下唯一可行的形態。

**只讀快照怎麼原子切換(讀不中斷):**

| 機制 | 怎麼做 |
|---|---|
| **雙快照 + 指針切換** | 內存裡同時放 old/new 兩份 Trie,新樹建好後,讀指針一次性指向 new,old 等在飛請求走完再回收(類似 RCU / 雙緩衝) |
| **不可變 + 版本號** | 每份快照帶版本號,服務啟動拉最新版;CDN/Redis 緩存可帶版本失效 |
| **灰度推送** | 新快照先推一個分片觀察(命中率/延遲沒退化)再全量,防壞快照打崩全站 |

**副軌:突發熱點的近實時旁路(離線天級追不上突發事件)**

離線管線是天級/小時級的,但**突發事件**(某明星突然上熱搜、某產品突然爆)的熱度在**分鐘級**就起來了,等不到下次離線重算。

| 做法 | 機制 | 取捨 |
|---|---|---|
| **流式統計(Flink)+ 增量插入** | 流式實時統計查詢詞,發現某詞暴漲,**直接增量更新 Trie 的對應 Top-K**(只改受影響的少數前綴節點) | ✅ 分鐘級追上突發;只動少數節點,不重建全樹;代價是增量寫和讀有短暫並發,要小心並發控制(或只追加到一個「實時 overlay 層」,查詢時 overlay 優先) |
| **實時 overlay 層** | 不碰只讀快照,在它上面疊一個小的「實時熱詞層」,查詢先查 overlay 再查快照 | ✅ 完全不破壞只讀快照的不可變性;overlay 小、可頻繁更新;離線重算時 overlay 併入主樹後清空 |

> **面試話術:** 「熱度更新我走**雙軌**:主軌是**離線批量**——搜索日誌灌進 Kafka,Spark 天級聚合熱度、重建整棵 Trie,做成一份**只讀不可變快照**原子換給線上。為什麼一定要讀寫分離?因為實時改榜會讓熱前綴節點變成讀寫搶鎖的單點熱點,百萬 QPS 下必死。突發熱點離線追不上,我加一條**實時旁路**:Flink 流式發現暴漲詞,寫進一個小的 **overlay 層**,查詢時 overlay 優先——既追上了突發,又不破壞主快照的不可變性。離線重算時把 overlay 併回去。」

### Deep Dive ③ · 分布式分片 + 延遲優化(扛住百萬 QPS × 100ms)

**核心問題:百萬 QPS 怎麼分,前綴查詢要不要跨片合併,延遲怎麼壓到 100ms?**

**按前綴分片:前綴定片 = 單片命中無合併(這題分片的漂亮處)**

```
查詢前綴 "fac"
   │ 取首字母(或前 2 字母)做路由
   ▼
路由到 shard("f")  ← 唯一確定的一個分片
   │
   ▼
在該片的 Trie 裡查 "fac" → 返回 Top-K
   (絕不需要問別的分片——以 "fac" 開頭的詞不可能在別片)
```

> **關鍵 insight(和別的海量題對比著講):** 「以某前綴開頭的所有補全候選,**必然在同一個分片**」——因為分片就是按前綴分的。所以**查詢永遠單片命中,零跨片合併**。對比 `News Feed`(合併多人 feed 要 scatter-gather)、`廣告聚合`(跨片匯總)——那些題的分片要付「合併」的代價,**這題的分片鍵和查詢維度完美對齊,合併代價為零**。這是「選對分片鍵」最乾淨的範例。

**分片傾斜:按首字母 26 等分是錯的(這題分片的真坑)**

| 問題 | 為什麼 | 解法 |
|---|---|---|
| **首字母分布極不均** | `s`/`a`/`t`/`c` 開頭的詞遠多於 `x`/`z`/`q` | **按詞頻不均勻劃分**:熱門首字母(`s`、`a`)拆到多片,冷門(`x`、`z`、`q`)合併到一片 |
| **單個超熱前綴**(如 `a`、`how to`) | 某些前綴的查詢 QPS 遠超其它,單片扛不住 | 該前綴的 Trie **多副本**橫向擴讀;或熱前綴結果直接進 CDN/Redis,根本不到分片 |
| **語言/字符集** | 中文按拼音首字母、CJK 字符空間不同 | 分片策略要按語言定制,別用一套 ASCII 規則套全球 |

**延遲優化:把 100ms 預算花在刀刃上(這題延遲是命門,單列)**

延遲優化是一條「**從客戶端到分片層層攔截**」的縱深防線,越靠前攔下收益越大:

| 層 | 手段 | 省下什麼 |
|---|---|---|
| **客戶端** | **防抖(debounce)**:停頓 50–100ms 才發;前綴變長時本地先過濾舊結果 | 砍掉一半以上請求;最快的請求是「不發的請求」 |
| **邊緣/CDN** | 熱前綴(`a`、`fa`、`how`)的 Top-K 結果緩存在 CDN 邊緣,就近返回 | 熱前綴根本不到源站,省一整趟回源 RTT |
| **服務層 Redis** | 熱前綴結果緩存在 Redis,擋掉重複查 Trie | Trie 服務只處理長尾/未緩存前綴 |
| **Trie 內存只讀** | 預存 Top-K → 查詢是純內存樹遍歷,無盤、無排序、無回源 | 把單次查詢壓到亞毫秒 |
| **就近接入** | 多區域部署,使用者連最近的接入點 | 省跨地域 RTT(`02`:跨洲 100ms,這題 100ms 預算經不起一次跨洲) |

> **面試話術(把延遲講成縱深防線):** 「100ms 預算很緊,我從客戶端到分片**層層攔**:客戶端**防抖**——停頓才發,最快的請求是不發的請求;熱前綴在 **CDN 邊緣**就近返回,根本不回源;回了源先查 **Redis** 熱前綴緩存;再不中才到 **Trie 分片**,而 Trie 因為**預存了 Top-K**,查詢就是一次亞毫秒的內存遍歷。分片我**按前綴分**,前綴定片所以單片命中、零合併,但要注意**首字母傾斜**——不能 26 等分,得按詞頻不均勻劃分,超熱前綴還要多副本或直接進 CDN。」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:這題花在「內存」和「離線算力」,不花在線上 CPU

回到第 2/3 步——這題的成本結構很特別:**線上查詢便宜到可笑(一次內存讀),貴的是兩頭——預存 Top-K 撐起的內存,和離線重建 Trie 的批算力。**

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **內存** | 預存 Top-K 把 Trie 撐到數十 GB 級,且要分片 × 副本 | 主要成本項。**Radix Tree 壓縮 + Top-K 存 ID 引用 + 只在熱前綴深度預存 + 冷前綴分層下沉**,把內存壓到該花的地方 |
| **離線算力** | 天級重建千萬級短語的全樹,自底向上 merge Top-K | 批處理算力(Spark 集群),但**不在線上、可錯峰跑**,成本可控 |
| **線上 CPU** | 百萬 QPS,但每次只是內存樹遍歷 | 幾乎不花錢,別過度配——**反直覺:百萬 QPS 的服務 CPU 卻很閒**,因為重活全離線做了 |
| **邊緣/CDN** | 熱前綴緩存 | 換掉的是「百萬 QPS 回源」,CDN 成本遠低於擴源站,**穩賺** |

> **這題成本的核心 insight:** 它把成本從「線上、實時、貴」搬到了「離線、批量、便宜」。線上百萬 QPS 卻 CPU 閒——因為 Top-K 的計算成本**一次性付在離線**,線上每次查詢攤到的成本趨近於零。這是「**預計算 + 讀寫分離**」最划算的場景。

### 容量規劃:讀靠副本擴,寫(離線)靠批處理擴

- **讀容量**:百萬 QPS / 單 Trie 副本吞吐 → 算需要幾個副本。因為 Trie 是**只讀不可變**的,**讀副本可以無限水平複製**(沒有寫一致性問題,加副本零成本)——這是只讀快照的最大紅利。熱前綴額外多副本 + CDN 兜底。
- **內存容量**:詞庫增長 → Trie 變大 → 算單片內存上限,超了就**加分片**(細分前綴空間)。要算「詞庫增長 vs 單機內存」的耗盡時間。
- **離線管線容量**:詞庫 × 重建頻率 → 批處理跑得完嗎?跑不完就**增量重建**(只重算熱度變化大的子樹)而非全量重建。

### 演進路徑:從 MVP 到 Google 級

```
階段 1(MVP):直接用 Elasticsearch completion suggester 或單機 Trie
   詞庫小、QPS 不高,別自己造 Trie 服務,ES 開箱即用——別過度設計
        │  QPS 上來、延遲尾巴變長、ES 成本不划算
        ▼
階段 2:自建 Trie 服務(預存 Top-K)+ Redis 緩存熱前綴 + 客戶端防抖
   把查詢壓成內存遍歷,讀寫分離,離線批量重建
        │  單機內存/QPS 撐不住
        ▼
階段 3:按前綴分片 + 多副本 + CDN 邊緣緩存 + 實時 overlay 旁路
   前綴定片無合併;熱前綴進 CDN;突發熱點走實時層
        │  全球使用者、要就近低延遲
        ▼
階段 4:多區域部署 + 個性化/拼寫糾錯/多語言
   每區域一套分片副本;個性化是新維度(下面「何時不該」會提它的代價)
```

> **這條演進的關鍵 insight:** 自動補全**從第一天就知道終局是 Trie + 分片 + CDN**,但**不該第一天就自建**。MVP 用 ES completion suggester 開箱即用——`01` 紅旗「上來就拆微服務 / 造輪子」的反面教材就是在這種題上一上來手寫分片 Trie 服務。**講演進**(「現在 ES 夠用,QPS 上來我自建 Trie 服務、再分片、再上 CDN」)比一上來堆滿組件**更顯資深**。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **某 Trie 分片掛** | 該前綴段(如 `f*`)的補全失效,其它前綴不受影響 | 每分片**多副本**;掛了路由到副本;**自動補全是錦上添花功能,單片失效可降級為「無提示」,不阻斷搜索本身** |
| **壞快照推上線**(Top-K 算錯/截斷) | 全站提示質量崩(返回亂序/空) | **灰度推送 + 快照健康檢查**(對比關鍵前綴的 Top-K 是否合理)+ **一鍵回滾**到上一版只讀快照 |
| **超熱前綴打爆單副本** | `a`/`how to` 的查詢壓垮一個副本 | 熱前綴**多副本 + 直接進 CDN/Redis**,根本不到分片;這正是邊緣緩存的價值 |
| **離線管線跑掛/延遲** | 熱度榜不更新,提示變「陳舊」但**仍可用** | **非關鍵路徑,可容忍延遲**——舊快照繼續服務,提示只是不夠新,不影響可用性。這是讀寫分離最大的好處 |
| **CDN/Redis 緩存集群掛** | 熱前綴回源,Trie 分片瞬間扛全量 | Trie 分片要能**降級 + 限流**保護自己;多級緩存避免單點 |

> **架構師話術:** 「我的設計裡**最不怕掛的恰恰是離線管線**——它掛了提示只是不更新、變陳舊,但**舊只讀快照繼續服務,可用性不受影響**,這是讀寫分離的核心紅利。最該防的是**壞快照**:一份算錯的快照推上線會讓全站提示崩,所以必須**灰度 + 健康檢查 + 一鍵回滾**。另外自動補全是**錦上添花功能**——分片掛了大不了『沒提示』,絕不能讓它掛掉拖垮搜索主流程,所以它和搜索主路徑要**故障隔離**。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **詞庫小 / QPS 不高**(企業內搜、站內搜) | 別自建分片 Trie + CDN + 離線管線 | **ES completion suggester** 或單機內存 Trie 搞定,別過度設計 |
| **強個性化**(每人補全不同) | 別用「全局共享的預存 Top-K」 | 全局 Trie **沒法為每人預存**——個性化要疊一層用戶信號(歷史/上下文)在全局候選上**輕量 rerank**,或全局召回 + 個性化精排。這是**完全不同的一道題**(召回-排序架構),別硬塞進這套 |
| **提示必須極度實時**(股票代碼、實時庫存) | 別用「天級離線重建快照」 | 那是「實時數據 + 前綴查詢」,該用實時索引(如 Redis ZSet 按前綴 + 實時更新),犧牲預計算換新鮮度 |
| **重拼寫糾錯 / 語義補全**(「想買蘋果手機」→ iPhone) | 別只靠 Trie 前綴精確匹配 | 那要**模糊匹配(編輯距離/BK-tree)+ 語義向量召回**——Trie 只解前綴精確補全,糾錯是它上面疊的另一層,語義補全更是另一個系統(向量檢索) |

> **這張表是架構師線的精華**——「知道 Trie 預存 Top-K 這套的邊界:它只解『全局熱度 + 前綴精確 + 可離線』這個甜區,一旦要個性化、要實時、要語義,就得換架構」。知道一個方案何時是錯的,比知道它怎麼做高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 自動補全的容量估算為什麼反直覺?結論一句話? | **補全 QPS ≠ 搜索 QPS**,是它的 **~20 倍**(每敲一鍵一請求),輕鬆百萬級 + 每請求 <100ms。結論:**先用防抖 + CDN 砍請求,再用預存 Top-K 的 Trie + 按前綴分片扛;寫是離線的,線上只讀快照**。 |
| 2 | 為什麼不能查詢時實時算 Top-K? | 走到前綴節點要遍歷整棵子樹 + 排序;**短前綴(`a`)子樹有幾百萬詞,而短前綴恰是最熱查詢** → 最熱的請求最慢,百萬 QPS 直接爆。解:**離線建樹時自底向上把每節點 Top-K 預算好排好**,查詢退化成 O(前綴長度) 走樹 + 讀現成數組。 |
| 3 | 為什麼不用 `LIKE 'abc%'` 或 ES? | `LIKE` 命中詞可能上千還要實時排序,百萬 QPS×100ms 扛不住;**ES completion suggester 中小規模能用**(底層 FST/Trie 同源),但 Google 級成本/尾延遲不划算。用**專用 Trie + 預存 Top-K** 把搜索+排序全挪到離線。 |
| 4 | 預存 Top-K 換來什麼、代價是什麼、怎麼治代價? | 換**查詢 O(前綴長度) 與子樹大小無關**(短前綴和長前綴一樣快);代價是**內存膨脹數十 GB + 建樹成本**。治:**Radix Tree 壓單鏈路徑 + Top-K 存短語 ID 引用 + 只在熱前綴深度預存 + 冷前綴分層下沉**。 |
| 5 | 熱度怎麼更新又不拖垮讀路徑?為什麼必須讀寫分離? | **雙軌**:主軌離線批量(Kafka→Spark 聚合熱度→重建 Trie→**只讀快照原子切換**);副軌實時 overlay(Flink 發現突發詞,寫小 overlay 層,查詢 overlay 優先)。必須讀寫分離因為**實時改榜會讓熱前綴節點成讀寫搶鎖的單點熱點**,百萬 QPS 下必死。 |
| 6 | 按前綴分片好在哪、坑在哪? | 好:**前綴定片 → 查詢單片命中、零跨片合併**(對比 feed 要 scatter-gather)——分片鍵與查詢維度完美對齊。坑:**首字母傾斜**(`s`/`a` 遠多於 `x`/`z`),不能 26 等分,要**按詞頻不均勻劃分**;超熱前綴要多副本或直接進 CDN。 |
| 7 | 100ms 延遲預算怎麼花?(縱深防線) | 層層攔:客戶端**防抖**(最快的請求是不發的)→ **CDN 邊緣**緩存熱前綴(不回源)→ **Redis** 熱前綴緩存 → **Trie 內存只讀**(預存 Top-K,亞毫秒遍歷)→ **就近接入**省跨洲 RTT。 |
| 8 | 自動補全何時**不該**套這套方案? | 詞庫小/QPS 低(ES 即可)、**強個性化**(全局 Top-K 沒法為每人預存,要召回+個性化精排,是另一道題)、**極度實時**(別用天級離線快照)、**糾錯/語義補全**(Trie 只解前綴精確,要疊模糊匹配/向量召回)。 |

---

## 一句話總綱(背起來)

> **自動補全是「讀 QPS 放大」題——補全請求是搜索的 ~20 倍(每鍵一請求)、百萬級、還要 <100ms,逼你為這個訪問模式定制專用結構:Trie + 每個前綴節點離線預存好排序的 Top-K(絕不查詢時算,否則最熱的短前綴最慢必爆),用空間換查詢飛快,再用 Radix 壓縮治內存膨脹。熱度走讀寫分離雙軌:離線批量聚合搜索日誌重建只讀快照原子切換(主軌)+ 實時 overlay 追突發熱點(副軌),線上服務從不被寫。分片按前綴分——前綴定片所以單片命中零合併,但要治首字母傾斜。延遲靠『防抖→CDN→Redis→內存 Trie』縱深攔截。架構師線:成本全在離線算力與內存(線上百萬 QPS 卻 CPU 閒)、只讀副本可無限擴、最該防壞快照(灰度+回滾)、自動補全是錦上添花要和搜索主路徑故障隔離、MVP 先用 ES 別自造輪子。**
