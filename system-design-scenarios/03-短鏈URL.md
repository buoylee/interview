# 03 · 短鏈 URL(TinyURL)

> **T1 第一題,最經典的暖身。** 它看起來簡單到沒東西好設計——但正因為簡單,面試官用它測你**有沒有七步的肌肉記憶**:你會不會在「明明可以一句話帶過」的題上,照樣老老實實澄清、估算、暴露 tradeoff。
>
> 這題的「教學價值」高度集中在兩處:**短碼怎麼發**(編碼題的母題,後面 `05 分散式 ID` 是它的放大版)和**讀路徑怎麼扛**(讀密集 + 緩存的母題,後面 `06 緩存`、`08 News Feed` 都復用)。把這兩個鑽透,半個 T1 就通了。
>
> 用 `00` 的依賴表看:短鏈吃 **A(框架數字感)+ B(存儲)**,C/D/E/F 幾乎不沾——所以它是**純框架暖身**,別在它身上堆微服務。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個短鏈服務(像 bit.ly / TinyURL)。使用者貼一條長 URL,系統回一條短 URL;別人點短 URL 就 302/301 跳轉到原長 URL。

**別一上來就畫**——先用 `01` Step 1 的「問三類」把開放題收斂成有邊界的題:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 核心用例?要不要自訂短碼、過期、點擊統計? | 核心:長→短、短→跳轉。**自訂別名**(custom alias)算加分項;**點擊統計**要(但可異步);**TTL 過期**可選 |
| **非功能(SLA)** | 讀寫比?延遲?可用性?短碼能不能被枚舉? | **讀 >> 寫**(~10:1);跳轉 P99 要低(<100ms,它在使用者點擊的關鍵路徑上);**高可用優先於強一致**(短鏈晚幾秒生效沒人死);短碼**要不可枚舉**(防爬全庫) |
| **規模 / 約束** | 新短鏈量級?短碼長度上限?保留多久? | ~**1 億新短鏈/月**;短碼越短越好(7 位以內);存 **5 年**起 |

**面試話術(把假設說出來讓對方確認):**
> 「我先框一下範圍——核心就兩件事:生成短鏈、跳轉。自訂別名和點擊統計我打算做,但統計走異步、不堵跳轉路徑。規模我假設一個月 1 億條新短鏈,讀寫比大概 10:1,跳轉延遲要壓在 100ms 內。短碼我會設計成**不可枚舉**,免得有人按順序爬光全庫。這樣可以嗎?」

**這幾個澄清直接決定後面**:

- 「讀 >> 寫」→ 第 5 步架構**重心在讀路徑 + 緩存**,寫路徑可以很樸素。
- 「不可枚舉」→ 第 6 步 deep dive ① 直接排除「自增 ID 明文當短碼」。
- 「統計可異步」→ deep dive ③ 把點擊計數從關鍵路徑摘出去。
- 「保留 5 年 + 1 億/月」→ 第 2 步估算的結論會是**存儲增長才是真瓶頸,不是 QPS**。

**常見陷阱:** 在「要不要支援短碼自訂大小寫敏感」這種細節上糾結。✅ 抓大放小,5 分鐘內收斂。

## 2. 容量估算(直接調用 `02` 的數字)

> 這題的估算 `02 容量估算數字感` 的 **Drill 2** 已經算過一遍了——這裡把它擴成完整版,並且**每步都接一句「所以架構上……」**(`01` Step 2 的鐵律)。

用 `02` 小抄:**一天 ≈ 1e5 s、一個月 ≈ 2.6e6 s、峰值 = 平均 ×3、副本 ×3、單 Redis ~10 萬 QPS、單 MySQL 讀 ~1 萬 QPS**。

### ① 寫 QPS(生成短鏈)

```
1 億/月 ÷ 2.6e6 s ≈ 40 QPS(平均)
峰值 ×3 ≈ 120 QPS
```
> **所以:寫 QPS 低到單機 MySQL 一根手指就扛住(它能寫幾千)。寫路徑不需要任何花活。**

### ② 讀 QPS(跳轉)

讀寫 ~10:1 → 讀平均 ≈ 400 QPS,峰值 ≈ **1200 QPS**。

> **所以:讀也不高。但別被這個數騙了**——這是**平均攤到全庫**的數。真實流量極度傾斜:某條短鏈進了一條爆款推文,瞬間可能上萬 QPS 砸到**同一個 key**。所以重點不是「讀總量大」,是**熱點集中**——這正是 deep dive ② 緩存要解的。

### ③ 存儲(真正的瓶頸)

單條記錄估 ~**500 B**(短碼 + 長 URL,長 URL 可能上百字節,加 metadata):

```
1 億/月 × 500 B = 50 GB/月
5 年 = 60 個月 × 50 GB = 3 TB(裸量)
× 副本 3 ≈ 9 TB
（再 × 索引開銷 1.2~2 → 實際佔用更大）
```
> **所以:這才是這題的真矛盾——QPS 低到單機綽綽有餘,但 5 年滾到 ~10 TB 級。一台機器的盤放得下,但「能放下」≠「該放一台」:單點、備份、增長都逼你想冷熱分離 / 分庫。** 這條結論要在第 6 步架構師深挖層展開。

### ④ 帶寬

跳轉回應就是一個 301/302 header(幾百字節),帶寬可忽略——**這題沒有帶寬問題**(對比 `02` Drill 3/5 的圖片/直播,媒體才有帶寬瓶頸)。

### 一句話估算結論(面試一定要說出來)

> 「**QPS 不是瓶頸——寫 40、讀峰值 1000 出頭,單機都能扛。真正的兩個矛盾是:(1) 5 年存儲滾到 10 TB 級的增長,(2) 流量極度傾斜、熱鏈接會打爆單 key。所以我的設計重心放在『存儲怎麼長期擴』和『熱鏈接怎麼緩存』,而不是堆 QPS。**」

這句「把估算焊到架構」的總結,就是 `01` 說的資深 vs 初級的分水嶺。

## 3. API + 數據模型

### API(`01` Step 3:快過,點到為止)

```
POST /shorten
  body: { longUrl, customAlias?, ttl? }
  → 201 { shortUrl: "https://sho.rt/aB3xK9" }

GET  /{shortCode}
  → 301/302 Location: <longUrl>        # 跳轉,不是 200+body
```

**要點:**
- 寫接口想**冪等**:同一個 `longUrl` 重複提交,要不要回同一個短碼?→ 看產品決策(見下「去重」),但接口層可帶 `idempotency-key` 為重試鋪路(`01` Step 3 的習慣)。
- 跳轉接口回的是 **3xx + Location header**,不是 200——這是「短鏈服務」和「普通 API」的本質區別,deep dive ③ 會鑽 301 vs 302。

### 數據模型(`01` Step 4:選型 + 分片鍵)

**核心就一張映射表/KV**,結構極簡:

| 字段 | 類型 | 說明 |
|---|---|---|
| `short_code` | varchar(7) **PK** | 短碼,主鍵,查詢入口 |
| `long_url` | text | 原始長 URL |
| `creator_id` | bigint | 誰建的(統計/配額用) |
| `created_at` | timestamp | 建立時間 |
| `expire_at` | timestamp? | TTL,可空 |

**選型(決策樹錨 `system-design/06 存儲選型`):**

| 候選 | 適不適合 | 理由 |
|---|---|---|
| **KV(Redis / DynamoDB / Cassandra)** | ✅ 主力 | 訪問模式就是**純按 `short_code` 點查**,沒有複雜查詢、沒有事務、沒有 join——這是 KV 的完美場景 |
| **SQL(MySQL/PG)** | ✅ 也行 | 數據量單表撐不住要分庫;但小規模起步用 SQL 最簡單,後面再遷 |
| ES / 寬列 | ❌ | 沒有全文搜索、沒有寬列訪問模式,殺雞用牛刀 |

> **面試話術:** 「訪問模式是純 KV 點查,所以我傾向 KV 存儲——要嘛 Redis 當主存(數據量大時成本高),要嘛 Cassandra/DynamoDB 這種水平擴展的 KV,把 5 年 TB 級增長交給它的自動分片。起步階段用 MySQL 也完全 OK,反正寫 QPS 才 40。」

**分片鍵(`01` Step 4 鐵律:海量題必講「按什麼分」,錨 `system-design/07 數據規模化`):**

> **按 `short_code` 做 hash 分片。** 短碼本身就是隨機/編碼出來的高基數值,hash 後天然均勻,**不存在熱點分片問題**(對比 `News Feed` 按 userId 分片會有大 V 熱點——這題沒有,因為短碼是隨機的)。查詢也永遠是「給定 short_code 查 long_url」,**單片命中,沒有跨片查詢**。

這就是為什麼短鏈是「好題」:分片鍵的選擇沒有陷阱,可以乾淨地展示「我知道要講分片鍵」這個信號,而不用陷進熱點分片的泥潭。

## 4. 高層架構(`01` Step 5:沿一個請求一鏡到底)

**先畫能跑的最簡版,再按需加件**(`01` 鐵律:別一上來拆 20 個微服務)。

```
                         ┌──────────────────────────────┐
   寫路徑(冷,40 QPS)    │                              │
 Client ──POST /shorten──► API GW ──► Shorten Svc ──► ID 發號器(取序號)
                                          │ base62 編碼成 short_code
                                          ▼
                                      寫入 KV/DB(short_code → long_url)
                                          │
                                          ▼
                                     回 short URL


   讀路徑(熱,1000+ QPS,且傾斜)
 Client ──GET /{code}──► API GW ──► Redirect Svc ──► Redis(熱鏈接緩存)
                                          │  ▲          │
                                     hit  │  │ miss     │
                                  301/302 │  └── 回源 ──► KV/DB ──► 回填 Redis
                                          ▼
                                  瀏覽器跳轉到 longUrl
                                          │
                                          └── 點擊事件 ──► MQ ──► 統計 Worker(異步)
```

**兩條典型路徑走一遍(`01` Step 5:讀 + 寫各一條):**

- **生成(寫,冷路徑):** Client → GW → Shorten Service → 跟發號器要一個唯一序號 → base62 編碼成 `short_code` → 寫 KV/DB → 回短 URL。**寫 40 QPS,樸素到不需要任何緩存。**
- **跳轉(讀,熱路徑):** Client → GW → Redirect Service → 先查 **Redis 緩存**(熱鏈接命中率極高)→ 命中直接回 301/302 Location → miss 才回源 KV/DB 並**回填緩存** → 同時把點擊事件**異步**丟進 MQ,不堵跳轉。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **Redis 緩存(讀路徑)** | 流量極度傾斜,熱鏈接砸同一 key | ②「熱點集中」 |
| **發號器**(或隨機生成) | 短碼要唯一 + 不可枚舉 | ①「不可枚舉」 |
| **MQ + 統計 Worker** | 點擊統計不能堵跳轉的關鍵路徑 | ①「統計可異步」 |
| **KV/DB 水平分片** | 5 年滾到 10 TB 級 | ③「存儲增長」 |

**注意這張圖的「克制」:** 沒有 CDN(沒媒體)、沒有複雜服務網格、沒有強一致組件。**短鏈的架構本該樸素**——把複雜度堆在這題上,反而是 `01` 紅旗清單裡的「過度設計」。

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · 短碼怎麼生成(這題的真正考點)

短碼長度怎麼定?先算清楚,**62^7 是這題要背的數**:

base62 = `[0-9a-zA-Z]` = 62 個字符。

| 長度 | 空間 | 夠不夠 |
|---|---|---|
| 6 位 | 62⁶ ≈ **568 億** | 5 年 ~60 億條,夠但餘量不大 |
| **7 位** | 62⁷ ≈ **3.5 萬億** | 5 年用不到 0.2%,**選它** |
| 8 位 | 62⁸ ≈ 218 萬億 | 浪費 |

> **面試話術:** 「我估 5 年 ~60 億條,62⁷ ≈ 3.5 萬億,佔用率不到 0.2%,所以**短碼選 7 位 base62**,既短又留足餘量。」

**三種生成方案對比(這是 deep dive 的肉):**

| 方案 | 怎麼做 | 優點 | 致命傷 |
|---|---|---|---|
| **A. 發號器 + base62** | 取一個全局唯一遞增序號,**base62 編碼**成短碼 | 無碰撞(序號天然唯一)、無需查重、寫路徑一次 IO | 序號若**明文遞增 → 短碼可枚舉**(`bit.ly/1,2,3…` 爬光全庫) |
| **B. 隨機生成 + 查重** | 隨機產 7 位 base62,**查 DB 有沒有撞**,撞了重試 | 天然不可枚舉、實現簡單 | 每次寫**多一次查重 IO**;庫越滿碰撞越頻繁(生日問題) |
| **C. hash(longUrl) 截斷** | 對 longUrl 取 MD5/SHA,**截前 7~8 位** base62 | 同一 URL 天然映同一碼(自帶去重) | 截斷後**碰撞率高**(不同 URL 撞同碼),仍要查重 + 重試;且**不可枚舉但可被反推**(已知 URL 能算出碼) |

**黑盒裡發生什麼——三個方案的真實取捨:**

- **方案 A「明文自增」為什麼不行(澄清裡已埋雷):** 如果短碼是 `base62(1), base62(2)…`,攻擊者按順序遍歷就能**爬光所有人的短鏈**(隱私洩漏 + 競品扒數據),還能**估算你的業務量**(看最大序號)。這是 `01` 微範例裡 Pastebin「不可枚舉」要求的同一個坑。

- **怎麼把 A 救活(生產常用):** 發號器**仍然發遞增序號**(無碰撞的好處保留),但編碼前做**一次可逆打散**——對序號做**位置置換 / 異或一個密鑰 / Feistel 輕量加密**,再 base62。這樣外部看到的短碼**跳躍不可預測**,內部仍是無碰撞的遞增空間。**「無碰撞」和「不可枚舉」就同時拿到了。** 這是把 A 的 deal-breaker 補掉的關鍵一招,講出來就過了資深線。

- **發號器本身怎麼來(錨,不重寫):** 全局唯一遞增序號的方案(號段模式 / Snowflake / Redis `INCR`)是**獨立的一道題**——`05 分散式 ID` 專門講,深礦在 `distribution/分布式id.md`。這題只需要說「我要一個發號器,細節錨那邊」,**別在短鏈題裡重講一遍 Snowflake**,那是搶 `05` 的戲、也浪費白板時間。

- **碰撞處理(B/C 必答):** 隨機/hash 方案靠「**生成 → 條件寫入(`INSERT ... IF NOT EXISTS` / Redis `SETNX`)→ 撞了就換一個重試**」。關鍵是**用存儲的原子性兜底**,而不是「先查再寫」——「先 SELECT 再 INSERT」在並發下有 TOCTOU 競態(兩個請求同時查到「沒撞」然後都寫)。庫越滿碰撞越頻繁(生日問題),所以方案 B 的**重試次數會隨佔用率上升**,這是它的隱性成本。

> **面試話術(主動給結論):** 「我會選**發號器 + base62 + 對序號做一次打散**:拿到無碰撞、無查重 IO 的好處,又靠打散拿到不可枚舉。發號器本身怎麼做我可以展開,但它其實是分散式 ID 那道獨立的題。如果不想引入發號器,退而求其次用隨機 7 位 + `SETNX` 重試也行,代價是每次寫多一次條件寫、且庫滿後重試變多。」

### Deep Dive ② · 緩存與熱點(讀路徑的命門)

第 2 步算出「流量極度傾斜」——這題 80% 的讀打在 20%(甚至更少)的熱鏈接上。**讀路徑 = 緩存題**(緩存四件套的完整版錨 `redis-handson/06-caching-patterns` 與後面 `06 分散式緩存`,這裡只講短鏈用到的)。

**讀路徑(旁路緩存 cache-aside):**

```
GET /{code}
  → Redis GET code
      ├─ HIT  → 直接回 301/302 Location        ← 絕大多數請求走這裡
      └─ MISS → 回源 KV/DB
                 ├─ 查到 → 回填 Redis(set + TTL)→ 回 301/302
                 └─ 查無 → 回 404
```

**為什麼緩存收益巨大:** `02` 延遲表——Redis ~0.5ms vs DB 點查(走盤)幾 ms,差一個量級;且熱鏈接命中率可到 95%+,等於**讓 95% 的讀根本不碰 DB**。短鏈是「一次寫、永久讀、值幾乎不變」的數據,**緩存友好度拉滿**(不像 feed 那樣值會變,要操心一致性)。

**穿透防護(短鏈特有的一個坑):** 有人拿**不存在的隨機短碼**狂打(`sho.rt/zzzzzzz`),每次都 MISS → 每次都回源 DB → DB 被打穿。短碼空間 3.5 萬億,瞎猜全是 miss。兩道防線:

| 防護 | 怎麼做 | 黑盒裡發生什麼 |
|---|---|---|
| **緩存空值** | 查無的 code 也寫進 Redis(值=空,短 TTL) | 第二次同樣的瞎猜直接被緩存擋下,不回源;短 TTL 防止把整個無效空間都緩存爆 |
| **布隆過濾器** | 所有存在的 code 灌進 Bloom filter,查前先過濾 | Bloom「說不存在」一定不存在 → 直接 404 不回源;「說存在」可能誤判(再走緩存/DB)。用極小內存擋掉海量瞎猜 |

> **面試話術:** 「短鏈的緩存穿透很真實——短碼空間巨大,有人拿隨機碼狂掃全是 miss,會把 DB 打穿。我用**布隆過濾器 + 緩存空值**雙保險:Bloom 先擋掉絕大多數不存在的碼,漏網的用空值緩存兜。」

**「擊穿」和「雪崩」在短鏈上不嚴重**(主動暴露,顯得你知道邊界):短鏈的值**永不變、永不過期失效**(一條短鏈建好就終身有效),所以不存在「熱 key 過期瞬間被打穿」的擊穿問題,也不需要給 TTL 加隨機抖動防雪崩。**這正是短鏈比 feed 簡單的地方**——值不可變 = 緩存一致性幾乎是免費的。

### Deep Dive ③ · 301 vs 302 + 點擊統計異步化(跳轉的取捨)

跳轉回的是 3xx,但**回 301 還是 302,直接決定你的統計準不準**——這是短鏈題最容易被忽略、又最能展示細節功力的點。

| | **301 Moved Permanently** | **302 Found(臨時)** |
|---|---|---|
| 語義 | 永久重定向 | 臨時重定向 |
| **瀏覽器行為** | **緩存映射**,下次同一短鏈**不再請求你的服務器**,直接跳 | **不緩存**,每次點都回你的服務器 |
| **你的服務器負載** | 低(熱鏈接後續點擊瀏覽器自己跳了) | 高(每次點都打到你) |
| **點擊統計** | **丟失!** 瀏覽器直接跳,你**收不到後續點擊** | **準確** 每次點都經過你,能計數 |
| **改目標 URL** | 改不動(瀏覽器已緩存舊跳轉) | 隨時能改(每次都問你) |

**黑盒裡發生什麼(這是核心 tradeoff):**

- 選 **301**:瀏覽器把「短碼→長 URL」緩存進本地,**後續點擊根本到不了你的服務器**。好處是**省服務器負載**(熱鏈接的重複點擊瀏覽器自己消化了),壞處是**你的點擊統計從此漏掉這些點**,而且**萬一要改短鏈指向**(改了 long_url)用戶端緩存還指著舊地址。
- 選 **302**:每次點擊都回到你這裡,**統計準確、能隨時改指向**,代價是**所有點擊流量都砸到你**(QPS 上升)——但第 2 步算過讀峰值才 1000 出頭,**你扛得住**。

> **面試話術(把 tradeoff 攤開,`01` Step 7 加分核心):** 「這裡有個經典取捨:301 永久重定向會被瀏覽器緩存,省我服務器負載,但**後續點擊我就統計不到了**,而且改指向也失效;302 每次都回我這、統計準確、可隨時改,代價是流量都過我。**因為點擊統計是這個產品的核心價值(數據變現),而我的讀 QPS 才千級扛得住,所以我選 302。** 如果是純粹追求極致負載、不在乎統計的場景,我會反過來選 301。」

這句「**因為……所以選 X,反過來的場景我會選 Y**」就是 `00` 標準線裡「**擁有這個決策**」的架構師話術。

**點擊統計異步化(選了 302,跳轉路徑就背了統計負擔,得卸掉):**

跳轉是使用者點擊的**關鍵路徑**,P99 要 <100ms。**絕不能**在跳轉時同步 `UPDATE counter SET clicks=clicks+1`——那會:(1) 拖慢跳轉,(2) 熱鏈接的計數行成為**單行熱點**(所有點擊搶同一行的鎖)。

**怎麼卸(從同步到異步,層層遞進):**

| 做法 | 機制 | 取捨 |
|---|---|---|
| 同步 `UPDATE +1` | 跳轉時直接寫 DB | ❌ 拖慢跳轉 + 單行鎖熱點,**禁止** |
| **發 MQ + Worker 消費** | 跳轉只**發一條點擊事件到 Kafka**(極快),Worker 異步聚合計數 | ✅ 跳轉路徑零 DB 寫;統計有秒級延遲(可接受) |
| **本地累加 + 批量刷** | 服務器內存裡攢計數,**每 N 秒批量** flush 到 DB / `INCR` Redis | ✅ 把「每次點一次寫」壓成「批量一次寫」,寫放大暴降;代價是進程崩了丟掉那一窗口的計數 |

> **面試話術:** 「跳轉是關鍵路徑,我絕不在上面同步寫計數——熱鏈接會把計數行變成單行鎖熱點。我把點擊事件**發到 MQ**,Worker 異步聚合;或者更省的話,在跳轉服務本地內存攢一批、**每幾秒批量** flush。統計接受秒級延遲,換來跳轉路徑乾淨。」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:存儲是這題唯一花錢的地方

回到第 2 步的結論——**這題不花在 CPU/QPS(低到可笑),全花在存儲增長上**:

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| 存儲 | 5 年 ~10 TB(含副本),且**只增不減**(短鏈一般不刪) | 主要成本項。**冷熱分離**:90% 的舊短鏈幾乎沒人點(長尾),可下沉到**便宜冷存儲**;熱的留熱層 |
| 緩存 | 熱鏈接 working set 很小(20% 都不到) | Redis 內存成本可控,**別把整庫塞緩存**(沒必要,長尾根本沒人查) |
| 計算 | QPS 千級,幾台機器 | 幾乎不花錢,別過度配 |

> **何時把短鏈當「可過期」資源:** 如果業務允許(行銷活動短鏈),給 TTL **讓老鏈接自動過期回收存儲**,把「只增不減」改成「有進有出」,存儲成本曲線從線性增長壓平。這是用「產品約束」換「成本」的架構決策。

### 容量規劃:短碼空間是有限資源

62⁷ ≈ 3.5 萬億,看著用不完,但要算**用盡的那天**:1 億/月 → 一年 12 億 → **要 ~2900 年才用完 7 位**。**所以 7 位永遠夠**——但如果某天業務量暴漲 100 倍,或改了短碼策略,**短碼長度是要預留升級路徑的**(從 7 位平滑加到 8 位,而不是某天突然發不出碼)。架構師要算「資源池的耗盡時間」,不是只算當下夠不夠。

### 演進路徑:從 MVP 到規模

```
階段 1(MVP):MySQL 單庫 + Redis 緩存
   寫 40 QPS / 讀千級,單庫綽綽有餘,別過度設計
        │  存儲滾到單庫吃力(~TB 級)
        ▼
階段 2:換水平擴展 KV(Cassandra/DynamoDB)或 MySQL 按 short_code 分庫
   分片鍵 = short_code(隨機 → 天然均勻,無熱點分片)
        │  跨地域使用者、要就近低延遲跳轉
        ▼
階段 3:多區域部署 + 全球 CDN/邊緣
   短鏈是「一寫永讀、值不變」→ 極其適合邊緣緩存,跳轉就近命中
```

> **這條演進的關鍵 insight:** 短鏈**從第一天就知道終局架構**(KV + 緩存),但**不該第一天就上**。MVP 用單庫,等存儲真撐不住再分——`01` 紅旗「上來就拆微服務」的反面教材就是在短鏈這種題上堆 Cassandra 多活。**講演進**(「現在單庫,撐到 TB 級我換 KV 分片」)比一上來堆滿組件**更顯資深**。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **Redis 緩存集群掛** | 讀全部回源 DB,DB 瞬間扛 10 倍流量可能被打垮 | 跳轉服務要能**降級回源 + 限流保護 DB**;多 Redis 分片避免單點;緩存預熱 |
| **發號器掛**(選方案 A) | 新短鏈生成失敗(讀不受影響) | 發號器**預取一段號段**到本地(號段模式),掛了還能用緩衝的號;細節錨 `05`/`distribution/分布式id.md` |
| **熱鏈接打爆單片** | 某條爆款短鏈的讀全壓在一個分片/key | 緩存層擋住(這正是 ② 緩存的價值);極端熱點可**多副本緩存 / 本地緩存** |
| **MQ 積壓**(統計) | 點擊計數延遲變大 | **統計是非關鍵路徑,可丟可延遲**——積壓只影響數據新鮮度,不影響跳轉,這是異步化最大的好處 |

> **架構師話術:** 「我的單點是緩存集群——它掛了讀全打到 DB,DB 扛不住 10 倍。所以跳轉服務必須能**降級**:緩存不可用時對 DB 限流,寧可少數請求變慢,也別讓 DB 雪崩。統計鏈路反過來——它**可丟可延遲**,MQ 積壓只影響報表新鮮度,不碰跳轉,這是我把它異步化的核心理由。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **內部系統的臨時短鏈**(量小、活幾天) | 別上 KV 分片 + 布隆 + MQ | 一張 MySQL 表 + TTL 搞定,別過度設計 |
| **需要短鏈攜帶業務語義 / 可讀** | 別用隨機 base62(`aB3xK9` 沒語義) | 用 slug(`/black-friday-sale`),這是另一類問題(衝突 + 人工命名) |
| **短鏈內容要保密 / 一次性** | 別用「永久有效 + 緩存友好」假設 | 加鑑權、一次性 token、禁緩存——這時候 301 緩存反而是漏洞 |
| **強統計實時性**(廣告計費級) | 別用「秒級延遲的異步聚合」 | 那是 `18 廣告點擊實時聚合`(流式 + 精確計數)的題,別把短鏈往那邊過度設計 |

> **這張表是架構師線的精華**——「知道一個方案的邊界、知道何時它是錯的」,比「知道這個方案怎麼做」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 短鏈的容量估算結論一句話? | **QPS 不是瓶頸**(寫 40、讀峰值千級,單機扛)——真矛盾是 **5 年存儲滾到 ~10 TB(冷熱分離)** 和 **流量傾斜、熱鏈接打爆單 key(緩存)**。 |
| 2 | 短碼為什麼選 7 位 base62? | 62⁷ ≈ 3.5 萬億;5 年 ~60 億條佔用 <0.2%,既短又留足餘量。6 位(568 億)餘量不夠,8 位浪費。 |
| 3 | 三種短碼生成方案 + 各自致命傷? | **A 發號器+base62**:無碰撞無查重,但**明文遞增可枚舉**(解:對序號打散)。**B 隨機+查重**:天然不可枚舉,但每寫多一次查重、庫滿碰撞變頻。**C hash 截斷**:自帶去重,但碰撞率高 + 可反推。 |
| 4 | 為什麼自增 ID 明文不能直接當短碼? | 可被**順序枚舉爬光全庫**(隱私+競品扒數據),且暴露業務量。解法:發號器仍發遞增序號,但編碼前對序號做**可逆打散**(置換/異或/Feistel),兼得無碰撞 + 不可枚舉。 |
| 5 | 301 vs 302 怎麼選?核心取捨? | **301** 瀏覽器緩存→省負載但**統計丟失**+改指向失效;**302** 每次回源→**統計準確**+可改指向但流量都過你。因**統計是核心價值 + 讀 QPS 千級扛得住**,選 **302**。 |
| 6 | 點擊統計為什麼不能同步寫?怎麼卸? | 跳轉是關鍵路徑(P99<100ms),同步 `UPDATE +1` 會拖慢 + **單行鎖熱點**。卸法:發**點擊事件到 MQ** 異步聚合,或**本地累加每幾秒批量 flush**。統計接受秒級延遲。 |
| 7 | 短鏈的緩存為什麼比 feed 簡單? | 短鏈值**一寫永讀、永不變、永不過期失效**→ 緩存一致性幾乎免費,**不存在擊穿/雪崩**。只需防**穿透**(隨機碼狂掃):布隆過濾器 + 緩存空值。 |
| 8 | 短鏈何時**不該**套這套方案? | 內部臨時短鏈(MySQL+TTL 即可)、要可讀 slug(衝突命名問題)、保密/一次性(禁緩存+鑑權)、計費級精確統計(走 `18` 流式精確計數)。 |

## 一句話總綱(背起來)

> **短鏈是「QPS 騙局」題——數字一算發現寫 40、讀千級單機全扛,真瓶頸是 5 年 TB 級存儲增長和熱鏈接傾斜。設計重心兩處:短碼生成(發號器 base62 + 對序號打散,兼得無碰撞與不可枚舉;發號器本身錨 `05`)和讀路徑緩存(cache-aside + 布隆防穿透,值不變所以一致性免費)。跳轉選 302 換統計準確,點擊計數靠 MQ/批量異步卸出關鍵路徑。架構師線:存儲冷熱分離、MVP 單庫起步別過度設計、緩存掛了要降級護 DB。**
