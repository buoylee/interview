# 02 · 容量估算數字感

> Step 2 的工具箱。容量估算不是數學題,是**「把規模翻譯成架構決策」**的橋——你算出讀寫 100:1,才有底氣說「上緩存」;算出一年 3.6TB,才有底氣說「要分庫」。
>
> 面試官不要你算到小數點,要你**敢估、估得出數量級、並把數字接到架構上**。這章給你背誦級的基準數 + 一套估算流程 + 5 個 30 秒 drill。

---

## 1. 背誦級基準數(記住這些,臨場不卡)

### 時間
| | 值 | 速記 |
|---|---|---|
| 一天 | 86,400 s | **≈ 10⁵ s**(估算就用 1e5,夠了) |
| 一個月 | ≈ 2.6e6 s | ~250 萬秒 |
| 一年 | ≈ 3.15e7 s | **≈ π×10⁷ s**(經典記法) |

> 最常用就一條:**一天 ≈ 10 萬秒**。`QPS = 日操作數 / 1e5`。

### 數據大小(單條/單字段)
| 東西 | 大小 | 
|---|---|
| 字符(ASCII / UTF-8 英文) | 1 B |
| 字符(中文 UTF-8) | 3 B |
| int / 時間戳(秒) | 4 B |
| long / 時間戳(毫秒) | 8 B |
| UUID | 16 B |
| 一條 tweet / 短消息 | ~100–300 B |
| 一行典型 DB 記錄(含索引) | **~1 KB**(估算常用值) |
| 一張縮略圖 / 小圖 | ~100 KB |
| 一張高清圖 | ~1–5 MB |
| 一分鐘 1080p 視頻 | ~50–100 MB |

### 吞吐量(單機 / 單組件,數量級)
| 組件 | 量級 | 速記 |
|---|---|---|
| 單機 MySQL 寫 | ~**幾千** QPS | 過萬就該想分庫/緩存 |
| 單機 MySQL 讀 | ~1 萬+ QPS | 加從庫/緩存放大 |
| 單機 Redis | ~**10 萬** QPS | KV 點查的主力 |
| 單機 Kafka(分區) | ~10 萬+ msg/s | 順序寫盤快 |
| 單機長連接(IM) | ~**幾萬–10 萬** 連接 | 受內存/FD 限,連接題按這個分機器 |
| 千兆網卡 | ~125 MB/s | 帶寬上限參考 |

> 這些是**數量級**,不是精確值。用途是「算出 50 萬 QPS,對照單 Redis 10 萬 → 我需要 ~5–10 個分片」。

### 延遲(Jeff Dean 數,記相對量級)
| 操作 | 量級 | 
|---|---|
| 內存讀 | ~100 ns |
| Redis / 同機房一次調用 | ~0.5–1 ms |
| SSD 隨機讀 | ~100 µs |
| 磁盤尋道(HDD) | ~10 ms |
| 同城跨機房 RTT | ~1–2 ms |
| 跨地域(跨洲)RTT | ~**100+ ms** |

> 用途:解釋「為什麼要緩存」(內存 ns vs 磁盤 ms,差 10⁵ 倍)、「為什麼異地多活寫延遲高」(跨洲 100ms)。

### 冗餘 / 放大係數(估容量時別忘乘)
- **副本數 ×3**:主流複製因子(算存儲時乘)。
- **峰值 ×2~3**:平均 QPS → 峰值 QPS。
- **索引 / 開銷 ×1.2~2**:DB 實際佔用 > 裸數據。
- **預留 headroom**:別按 100% 算,留 30–50% 餘量。

## 2. 估算流程(照這個順序算,不漏)

```
DAU / 規模
   │  × 人均操作數 ÷ 1e5(一天秒數)
   ▼
平均 QPS ──× 2~3──► 峰值 QPS ──對照單機吞吐──► 要幾個分片/機器
   │
   │  寫 QPS × 單條大小 × 86400
   ▼
每天寫入量 ──× 保留期 × 副本3 × 索引開銷──► 總存儲 ──► 要不要分庫
   │
   │  QPS × 單條 payload
   ▼
帶寬(進/出分開)──► 要不要 CDN / 媒體走對象存儲
```

**四步口訣:** ① DAU→QPS(÷1e5)② QPS→峰值(×3)③ 寫→存儲(×大小×保留×3)④ QPS→帶寬(×payload)。算完每步都接一句「**所以架構上……**」。

## 3. 五個 30 秒 drill(先蓋住答案自己算)

### Drill 1 · Twitter feed
> 2 億 DAU,人均每天讀 feed 100 次、發帖 2 次。算讀/寫峰值 QPS,推一個架構結論。

<details><summary>答案</summary>

- 讀:2e8×100/1e5 = **2e5 平均**,峰值 ×3 ≈ **60 萬 QPS**。
- 寫:2e8×2/1e5 = **4000 平均**,峰值 ≈ **1.2 萬 QPS**。
- **讀寫 ≈ 50:1 → 重點優化讀:feed 預計算 + Redis 緩存,寫路徑可異步扇出。**
</details>

### Drill 2 · URL 短鏈存儲
> 1 億新短鏈/月,每條記錄 ~500 B(含長 URL)。算寫 QPS 和 5 年存儲。

<details><summary>答案</summary>

- 寫 QPS:1e8/2.6e6 ≈ **40 QPS**(很低,單庫輕鬆)。
- 存儲:1e8×500B = 50 GB/月 → 5 年 = 60 個月 ×50GB = **3 TB**,×副本3 ≈ **9 TB**。
- **QPS 低但 5 年到 TB 級 → 重點不是 QPS,是存儲增長與冷熱分離 / 緩存熱鏈接。**
</details>

### Drill 3 · 圖片上傳服務
> 1000 萬張上傳/天,每張平均 2 MB。算每天存儲增量和寫帶寬。

<details><summary>答案</summary>

- 存儲:1e7×2MB = **20 TB/天**(裸量),×副本3 = **60 TB/天** → 一年 ~22 PB。
- 寫帶寬:20TB/天 ÷ 86400 ≈ **230 MB/s**(峰值 ×3 ≈ 700 MB/s,超單千兆網卡 → 多接入點)。
- **PB 級 → 必走對象存儲 + CDN,DB 只存 metadata。**
</details>

### Drill 4 · IM 消息
> 5000 萬 DAU,人均每天收發 40 條,每條 ~200 B。算消息 QPS 和每天存儲。

<details><summary>答案</summary>

- QPS:5e7×40/1e5 = **2 萬平均**,峰值 ≈ **6 萬 QPS**。
- 存儲:5e7×40×200B = **400 GB/天**,×副本3 = 1.2 TB/天。
- 並發長連接:DAU 中同時在線設假設 20% = 1000 萬連接 ÷ 單機 5 萬 = **~200 台接入機**。
- **連接數才是 IM 的關鍵約束,不只是 QPS。**
</details>

### Drill 5 · 視頻直播出口帶寬
> 100 萬人同時看一場直播,碼率 5 Mbps。算出口帶寬,推結論。

<details><summary>答案</summary>

- 出口:1e6 × 5 Mbps = **5 Tbps**(= 625 GB/s)——天文數字。
- **單源絕無可能 → 必須 CDN 邊緣分發 + 多級回源**,源站只服務 CDN 節點(幾百個),放大比百萬→百。
</details>

## 4. 臨場小抄(寫在白板角落)

```
一天 = 1e5 s        峰值 = 平均 ×3        副本 ×3
行記錄 ~1KB         單Redis ~10萬QPS      單MySQL寫 ~幾千QPS
跨洲RTT ~100ms      QPS = 日操作/1e5      存儲 = 寫QPS×大小×86400×保留×3
```

**估完必接一句「所以……」**——這才是估算的意義:
- QPS 高 + 讀寫比大 → **緩存 + 讀寫分離**
- 存儲到 TB/PB → **分庫分表 / 對象存儲**
- 帶寬大 / 媒體 → **CDN**
- 連接數大 → **接入層按連接分機器**
- 峰值尖 → **MQ 削峰 / 限流**

---

## 一句話總綱

> **容量估算 = 用「一天 1e5 秒、峰值 ×3、副本 ×3、行記錄 1KB、單 Redis 10 萬 QPS」這幾個背誦級數,把 DAU 翻成 QPS/存儲/帶寬/連接,再用一句「所以要加緩存/分庫/CDN」把數字焊到架構上。敢估、估出數量級、接上結論——比算得精確重要得多。**
