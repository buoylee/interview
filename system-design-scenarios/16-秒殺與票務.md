# 16 · 秒殺 / 票務(瞬時高並發 + 防超賣)

> **T3 高階,SDI 裡「韌性」與「一致性」雙重 ●● 的題**(`00` 依賴表:秒殺 C●● D●●)。它和短鏈是兩個極端:短鏈是「QPS 騙局」——一算發現流量根本不大;秒殺反過來,是**真·瞬時洪峰**,但洪峰背後藏著一個反直覺的核心 insight。
>
> **這題的命門是一句話:99% 的請求注定失敗,你的全部工作是把這 99% 的洪峰擋在庫存之前。** 幾百萬人搶幾千件,絕大多數人本來就搶不到。如果你讓這幾百萬請求全打到資料庫去搶那幾千個庫存,DB 第一秒就被打死——而它們**根本不該到 DB**,因為命中率只有 0.1%。所以秒殺不是「怎麼讓 DB 扛住百萬 QPS」的題(扛不住,也不該扛),是「**怎麼用一層層漏斗把百萬請求過濾成幾千個有效請求**」的題。
>
> 三條主線:**① 防超賣**(賣出去的絕不能超過庫存——這是會出人命/賠錢的硬約束)、**② 削峰**(把瞬時尖峰拉平成平穩消費)、**③ 防刷與公平**(黃牛/腳本不能把正常人擠掉)。把這三條鑽透,T3 裡最考「韌性 + 一致性」的題就通了。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個秒殺 / 限量搶購系統(像電商整點秒殺、演唱會搶票、12306 春運放票)。在**某個精確時刻**開賣**有限庫存**的商品,瞬間有**遠超庫存數量級**的用戶湧入下單,要求:**不超賣、不少賣、相對公平、系統不被打垮**。

**這題的開放題收斂尤其重要**——「秒殺」三個字底下藏著好幾種差很多的需求,先用 `01` Step 1 問清楚:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 搶到後要不要**立刻付款**?庫存是否要**鎖定**等付款?要不要**排隊頁/答題**?限購幾件? | 搶到 → **預佔庫存 + 給 N 分鐘付款窗口**,逾時釋放;每人限購 1~2 件;答題/驗證碼**用來削峰 + 防腳本** |
| **非功能(SLA)** | 超賣能不能容忍?可用性 vs 一致性怎麼選?延遲? | **絕不超賣**(硬約束,寧可少賣不可超賣);**搶購一致性 > 可用性**(在庫存這一點上);但**展示頁/排隊頁高可用優先**;下單 P99 要快(否則用戶以為卡了狂點) |
| **規模 / 約束** | 多少人搶?多少庫存?峰值持續多久?讀寫比? | **300 萬人搶 1 萬件**(命中率 0.3%);峰值集中在**開賣後 1~3 秒**;**99.7% 的請求是注定失敗的無效請求** |

**面試話術(把假設說出來,並且立刻點出這題的本質):**
> 「我先框範圍,而且秒殺有個關鍵前提我想先講出來——**這題的本質不是讓系統扛住百萬下單,是把百萬請求過濾成幾千個有效下單**。我假設 300 萬人搶 1 萬件,命中率 0.3%,也就是 **99.7% 的人本來就搶不到**。所以我的核心設計目標是:**讓這 99.7% 的請求盡量在離庫存最遠的地方就被擋掉**,真正能消耗庫存的請求拉到幾千的量級,DB 就輕鬆了。超賣是硬約束,寧可少賣不可超賣。搶到後給 N 分鐘付款窗口,逾時釋放庫存。這樣可以嗎?」

**這幾個澄清直接決定後面**:

- 「99.7% 注定失敗」→ 第 4 步架構是**多層漏斗**,不是「怎麼加 DB」。
- 「絕不超賣」→ deep dive ① 整個圍繞庫存扣減的原子性與一致性。
- 「峰值集中在 1~3 秒」→ 第 2 步算的是**瞬時尖峰**而非平均,deep dive ② 削峰把尖峰拉平。
- 「預佔 + 付款窗口 + 逾時釋放」→ 引出**最終一致與庫存回補**(deep dive ①)。
- 「限購 + 答題」→ deep dive ③ 防刷與公平。

**常見陷阱:**
- 🚩 一上來就想「DB 怎麼扛百萬寫」——**方向就錯了**,DB 不該扛,要把流量擋在它前面。
- 🚩 把「秒殺」當成「高 QPS 寫」的題去堆分庫分表——分庫解決不了「同一件商品的庫存是單點熱點」這個核心矛盾(deep dive ③ 的熱點行)。

## 2. 容量估算(瞬時尖峰,不是平均)

> 秒殺的估算和別的題**最大的不同:不要算平均 QPS**。平均沒有意義——一天就秒這麼一場,攤到 86400 秒是個可笑的小數。要算的是**瞬時峰值**和**有效/無效請求比**。

用 `02` 小抄:**單 Redis ~10 萬 QPS、單 MySQL 寫 ~幾千 QPS、峰值是尖峰不是 ×3**。

### ① 瞬時峰值 QPS(這題唯一重要的 QPS)

```
300 萬人,假設都在開賣後 2 秒內湧入(秒殺用戶會卡點狂點)
每人開賣瞬間平均點 3~5 次(搶不到就狂刷)
→ 瞬時請求 ≈ 300萬 × 4 / 2s ≈ 600 萬 QPS(尖峰)
```
> **所以:這是個幾百萬 QPS 的尖峰,持續才 1~3 秒。** 沒有任何單一 DB 能扛——單 MySQL 寫才幾千 QPS,差了三個數量級。**結論不是「加多少 DB」,是「這 600 萬 QPS 裡有多少該到 DB」。**

### ② 有效 vs 無效請求比(這題的靈魂數字)

```
庫存 1 萬件,限購 1 件 → 最多 1 萬個請求能真正扣到庫存
600 萬 QPS 裡,有效請求 ≈ 1 萬,佔比 0.16%
→ 99.84% 的請求是「來了也白來」的無效請求
```
> **所以:核心工作是設計一條漏斗,讓這 99.84% 的無效請求在到達庫存之前的每一層被逐步攔掉。** 庫存賣完的那一瞬間,後面所有請求都該被**前置攔截**(「已售罄」直接返回,連 Redis 都不用查)。能讓多少無效請求死在離 DB 最遠的地方,就是這題的全部技術含量。

### ③ 庫存扣減 QPS(漏斗末端,真正打到庫存的)

```
1 萬件 ÷ 開賣後幾秒搶光 ≈ 幾千 QPS 的扣減操作(打到 Redis 預扣層)
最終落 DB 的訂單:1 萬個,異步消費,可以攤到幾十秒慢慢落
→ DB 寫 QPS:幾百,輕鬆
```
> **所以:只要漏斗設計對,DB 端看到的是「幾百 QPS 的訂單慢慢寫進來」,而不是「600 萬 QPS 砸臉」。** 這就是削峰(deep dive ②)的目標:**把 600 萬/秒的尖峰,拉平成幾百/秒的平穩消費。**

### ④ 讀路徑(商品詳情頁/排隊頁)

```
300 萬人開賣前都在刷詳情頁、倒計時 → 讀峰值也是百萬級
但這是純讀、可緩存、可 CDN、值不變(商品信息開賣前固定)
```
> **所以:讀路徑用 CDN + 多級緩存徹底擋住,根本不碰後端**——這部分是 `06 緩存`/`03 短鏈` 已經會的能力,秒殺只是把它推到極致(靜態化 + 邊緣)。

### 一句話估算結論(面試一定要說出來)

> 「**秒殺的估算不看平均,看尖峰和有效比。瞬時 ~600 萬 QPS、持續 1~3 秒,但庫存只有 1 萬,有效請求佔比 0.16%,99.84% 注定失敗。** 所以我的設計不是『怎麼讓 DB 扛 600 萬』——扛不住也不該扛;是『**設計一條多層漏斗,把 600 萬 QPS 逐層過濾成打到 DB 的幾百 QPS**』。讀路徑(詳情頁/倒計時)百萬級但純讀可緩存,CDN 擋死;寫路徑(下單)才是命門,靠 Redis 原子預扣 + MQ 異步落庫削峰。」

這句「**把尖峰拉平、把無效請求擋在庫存前**」就是秒殺題的資深分水嶺——把它說出來,面試官就知道你抓到了本質。

## 3. API + 數據模型

### API(`01` Step 3:快過,但秒殺有兩個特有設計)

```
GET  /seckill/{itemId}/page          → 靜態化詳情頁(走 CDN,不打後端)

POST /seckill/{itemId}/qualify       # 答題/驗證碼,換一張「秒殺令牌」
  → 200 { seckillToken }             # 削峰 + 防腳本的第一道閘

POST /seckill/{itemId}/order         # 真正的下單
  header: Idempotency-Key, seckillToken
  body:   { userId, qty:1 }
  → 200 { status: "queued", orderToken }   # 注意:不是同步返回「成功」!
       或 409 { status: "sold_out" }       # 售罄前置攔截

GET  /seckill/order/{orderToken}     # 輪詢/長輪詢結果(下單是異步的)
  → { status: "success"|"pending"|"failed", orderId? }
```

**三個秒殺特有要點:**
- **下單接口必須冪等**(`Idempotency-Key`):用戶開賣瞬間狂點,同一人多次點擊不能扣多次庫存、不能生成多張訂單——冪等鍵是底線。錨 `financial-consistency/05-patterns`(冪等鍵)。
- **下單返回 `queued` 而不是 `success`**:因為下單走**異步**(發 MQ 再落庫,deep dive ②),同步等不現實。前端拿 `orderToken` 去**輪詢/長輪詢**結果。這是秒殺和普通下單最大的接口差異——**它故意是異步的**。
- **`qualify`(答題/驗證碼)是一道獨立的閘**:它在 `order` 之前,作用是**削峰**(答題花掉幾秒,把瞬時尖峰在時間軸上拉開)+ **防腳本**(機器難答驗證碼)。沒搶到的人連 `order` 都調不到。

### 數據模型(`01` Step 4:選型 + 為什麼庫存放兩處)

秒殺的數據模型核心是**庫存的「雙層存儲」**——這是這題的標誌性設計:

| 存儲層 | 放什麼 | 為什麼 |
|---|---|---|
| **Redis(預扣層,熱)** | `seckill:stock:{itemId}` → 剩餘庫存數;`seckill:bought:{itemId}` → 已搶用戶 set(限購去重) | 扣減在這發生,**Lua 原子扣**,扛幾千 QPS;DB 扛不住這個扣減速率(deep dive ①) |
| **MySQL(最終扣減層,冷)** | `inventory(item_id PK, total, reserved, sold)`、`order(...)` | **最終真相、可對賬、可恢復**;訂單異步落這,Redis 只是「快取 + 預扣賬本」 |

**核心表:**

| 表 | 關鍵字段 | 說明 |
|---|---|---|
| `inventory` | `item_id PK, total, reserved, sold, version` | DB 端庫存真相;`version` 給樂觀鎖;`reserved`=預佔未付款,`sold`=已成交 |
| `seckill_order` | `order_id PK, item_id, user_id, status, expire_at, idempotency_key UNIQUE` | 訂單;`status`:預佔/已付/已釋放;`expire_at` 付款窗口;**`idempotency_key` 唯一索引**是防重複下單的最後一道 DB 防線 |

**選型(決策樹錨 `system-design/06 存儲選型`):**

> **庫存扣減用 Redis(內存原子操作)+ MySQL 做最終一致賬本,是這題的標準解。** 為什麼不直接用 DB 扣?因為 deep dive ① 要展開的核心問題:**幾千 QPS 搶同一件商品的同一行庫存,DB 的行鎖會把這些請求排成一條長隊,延遲爆炸**。Redis 單線程模型反而適合——所有扣減在一個線程裡串行、無鎖、原子,~10 萬 QPS。

**分片鍵 / 為什麼分片在這題救不了你:**

> 普通海量題靠分片打散熱點。**但秒殺的熱點是「同一件商品的同一個庫存」——它在邏輯上就是一個點,沒法靠分庫分表打散。** 你把訂單表按 user_id 分 100 個庫,但 1 萬個庫存還是一個邏輯計數器,所有扣減都搶它。這就是 deep dive ③ 的「熱點庫存行」問題,解法不是分庫,是**分段庫存(把 1 萬拆成 100 段各 100,扣減請求隨機落到某一段)**——這是秒殺特有的、和普通分片完全不同的打散思路。

## 4. 高層架構(`01` Step 5:多層漏斗,一鏡到底)

**秒殺的架構圖就是一條漏斗——從上到下,流量逐層銳減。** 把「每一層擋掉多少」標出來,就講清了這題的全部精髓。

```
   ┌─────────────────────── 多層削峰漏斗(從上到下流量銳減) ──────────────────────┐

   ① 前端層                  600 萬 QPS  ─┐
   ─────────                              │  按鈕置灰/倒計時/答題/驗證碼
   靜態頁走 CDN(詳情頁不打後端)          │  → 攔掉一大批 + 在時間軸拉開尖峰
   答題/驗證碼換「秒殺令牌」               ▼
                                       ~百萬 QPS

   ② 網關/接入層             ─┐  WAF + 限流(令牌桶)+ 黑名單 + 售罄前置攔截
   ────────────              │  「已售罄」標誌位一立,後面請求直接 409,連 Redis 都不查
   無令牌直接拒                ▼  → 攔掉絕大多數無效請求
                          ~幾萬 QPS

   ③ Redis 預扣層           ─┐  Lua 原子腳本:檢查限購 + DECR 庫存(原子)
   ──────────────           │  stock<=0 → 立刻返回售罄(不再往下)
   秒殺核心:原子扣減          ▼  → 只有「搶到」的請求往下走
                          ~幾千 QPS(扣減) → 搶到的 ~1 萬個

   ④ MQ 削峰                ─┐  搶到的請求發一條「下單消息」到 Kafka/RocketMQ
   ─────────                │  瞬時尖峰 → 進隊列排隊 → 下游按自己節奏消費
                            ▼  把「幾千/秒的尖峰」拉平成「幾百/秒的平穩」
                          MQ 緩衝

   ⑤ 訂單服務 + DB          ─┐  Worker 消費:DB 樂觀鎖最終扣減 + 落訂單 + 起付款窗口計時
   ──────────────           ▼  ~幾百 QPS,DB 輕鬆;成功後回寫結果供前端輪詢
                          DB(最終真相 + 對賬)

   ⑥ 回補閉環                付款逾時 → 釋放預佔:DB reserved-1 + Redis stock+1(INCR)
   ─────────                → 庫存「有進有出」,沒付款的名額還給後面的人(最終一致)
   └──────────────────────────────────────────────────────────────────────────┘
```

**一條「搶到」的請求一鏡到底(`01` Step 5:走完整路徑):**

1. **開賣前** Client 刷詳情頁 → 全走 **CDN 靜態頁**,後端零負載;倒計時到點按鈕才激活。
2. **答題/驗證碼** → 換一張 **秒殺令牌**(削峰 + 擋腳本)。
3. **POST /order**(帶令牌 + 冪等鍵)→ **網關限流 + 售罄標誌檢查**(售罄直接 409)。
4. → **Redis Lua 原子腳本**:檢查這人沒搶過(限購)+ `DECR stock`。`stock>=0` 才算搶到;`<0` 立刻回滾並返回售罄。
5. 搶到 → **發下單消息到 MQ**,接口立刻返回 `queued + orderToken`(**不等 DB!**)。
6. **訂單 Worker** 從 MQ 消費 → DB 樂觀鎖 `UPDATE ... where stock>0` 最終扣減 + 寫訂單 + 起 15 分鐘付款計時 → 回寫結果。
7. Client 拿 `orderToken` **輪詢**結果 → 看到 `success` → 跳付款頁。

**一條「沒搶到」的請求(99.84% 的人):** 在第 1/2/3/4 層的某一層就被擋掉了——大多數人在**第 3 層(售罄標誌)**或**第 4 層(Redis stock<=0)**就被攔下,**根本到不了 MQ 和 DB**。這就是漏斗的意義。

**每加一層都說它解決什麼(`01` Step 5 要點):**

| 層 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| ① CDN 靜態化 + 答題 | 詳情頁百萬讀 + 在時間軸拉開尖峰 | ④ 讀路徑 + ① 尖峰 |
| ② 網關限流 + 售罄前置攔截 | 把無效請求擋在離 DB 最遠處 | ② 99.84% 無效 |
| ③ Redis 原子預扣 | DB 扛不住同一行幾千 QPS 扣減 | ③ 扣減 QPS |
| ④ MQ 異步下單 | 把尖峰拉平成平穩消費 | ① 尖峰 → ③ DB 幾百 |
| ⑤ DB 樂觀鎖最終扣減 | 最終真相、可對賬、可恢復 | 防超賣硬約束 |
| ⑥ 逾時回補 | 預佔未付款的名額要還回去 | 不少賣 |

**這張圖的克制與重點:** 它看著組件多,但**每一層都在做同一件事——過濾流量**。面試時把「**從上到下流量從 600 萬銳減到幾百**」這條主線講出來,比畫多少框都值錢。

## 5. Deep Dive(挑 3 個最會被往死裡追問的點)

### Deep Dive ① · 庫存扣減防超賣(這題的真正命門)

**面試官 100% 會追:「你怎麼保證不超賣?」** 這是整題的核心。先把三種方案攤開對比,再講清為什麼是「Redis 預扣 + DB 最終扣 + 回補」的組合。

#### 為什麼不能直接打 DB?

「直接用 DB 一個原子 `UPDATE inventory SET stock=stock-1 WHERE stock>0`,不就不超賣了嗎?」——**邏輯上對,但扛不住。** 問題不是正確性,是**性能**:

> 幾千個請求**同時搶同一行庫存**,DB 對這一行加**行鎖**,所有請求被排成一條串行隊列,一個一個拿鎖、扣、放鎖。在熱點行上,行鎖的等待 + 事務開銷讓單行的有效扣減速率掉到**幾百 QPS**,延遲飆到秒級。**而且這幾千 QPS 只是「搶到的」,前面還有幾百萬無效請求也想擠進來打 DB——DB 第一秒就被連接數打滿、雪崩。** 所以 DB 的問題是「**熱點單行 + 海量無效請求**」,不是 SQL 寫不對。

#### 三種防超賣方案對比(deep dive 的肉)

| 方案 | 怎麼做 | 優點 | 致命傷 / 適用 |
|---|---|---|---|
| **A. DB 樂觀鎖** | `UPDATE ... SET stock=stock-1, version=version+1 WHERE id=? AND stock>0`(或帶 version);影響行數=0 即搶失敗 | **絕對不超賣**(DB 原子);實現簡單;是最終真相 | 同一行幾千 QPS → **行鎖排隊,延遲爆炸**;**只適合做「最終扣減」,不適合扛前端洪峰** |
| **B. Redis Lua 原子預扣** | Lua 腳本裡 `if stock>0 then DECR` 一氣呵成,**單線程串行、無鎖、原子**,~10 萬 QPS | 扛得住前端洪峰;`stock<=0` 立刻知道售罄,擋住後續 | Redis 是**內存、可能丟**;需要和 DB 對賬;預扣成功 ≠ 訂單一定成交(要回補) |
| **C. 分段庫存** | 把 1 萬拆成 100 段各 100,扣減請求**隨機落到某一段**,把單熱點打散成 100 個 | 把單行鎖熱點分散,**單機 DB 也能扛更高扣減** | 某段先賣空時要**跨段借**(複雜);可能某段沒了但全局還有 → 誤判售罄。見 deep dive ③ |

**生產標準解 = B 扛洪峰 + A 做最終真相(+ C 解熱點):**

> **Redis Lua 原子預扣** 在前面把洪峰扛住、把售罄判斷做掉(99% 的請求死在這);**只有預扣成功的幾千個請求**才異步走到 DB,用 **DB 樂觀鎖最終扣減**落訂單。Redis 負責「快 + 擋」,DB 負責「準 + 對賬」。

**黑盒裡發生什麼——Redis Lua 為什麼能原子防超賣:**

> Redis 是**單線程執行命令**的。一段 Lua 腳本在 Redis 裡是**整體原子執行**——執行期間不會有別的命令插進來。所以 `取庫存 → 判斷 >0 → 扣減` 這三步在腳本裡是不可分割的一個動作,**不存在「兩個請求同時讀到 stock=1 然後都扣成 -1」的競態**(這正是 deep dive 裡最容易被追的點)。對比「先 `GET` 再 `DECR`」兩條命令——中間會被別人插隊,就有 TOCTOU 競態,會超賣。**必須是 Lua 一段腳本,不是兩條命令。** Lua 原子扣減的可運行實作錨 `redis-handson/08-transactions-scripting`(腳本)+ `13-rate-limiting`(同樣的原子扣減模式)。

```lua
-- 秒殺原子預扣(限購 + 扣庫存,一段腳本原子完成)
-- KEYS[1]=stock key, KEYS[2]=bought set, ARGV[1]=userId
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
  return -2          -- 已搶過(限購攔截)
end
local stock = tonumber(redis.call('GET', KEYS[1]))
if not stock or stock <= 0 then
  return -1          -- 售罄
end
redis.call('DECR', KEYS[1])
redis.call('SADD', KEYS[2], ARGV[1])   -- 記錄已搶,防同人重複
return stock - 1     -- 搶到,返回剩餘
```

#### 預扣 + 異步落庫的最終一致與回補(這是區分資深 vs 架構師的點)

引入 Redis 預扣後,**庫存真相被拆成兩處**(Redis 預扣賬本 + DB 最終賬本),於是出現一致性問題:

- **預扣成功 ≠ 訂單一定落庫成功**:Redis 扣了,但下游 MQ 消費或 DB 寫可能失敗 → **庫存「漏」了一個**(Redis 顯示賣了,DB 沒這筆訂單)。
- **付款窗口逾時**:預佔了 15 分鐘沒付款 → 這個名額要**還回去**給後面的人。

**怎麼回補(閉環,缺了就是「少賣」bug):**

| 場景 | 回補機制 |
|---|---|
| **逾時未付款** | 定時掃 `expire_at` 過期的預佔訂單 → DB `reserved-1` + **Redis `INCR stock`**(把名額還回 Redis 預扣池);訂單置「已釋放」 |
| **下游落庫失敗** | MQ 消費保證**至少一次** + 冪等(`idempotency_key` 唯一索引);消費失敗重試;徹底失敗 → **Redis `INCR` 回補** + 告警 |
| **Redis 與 DB 對不上** | 定時**對賬**:以 DB 為準校 Redis(`Redis stock` 應 = `total - reserved - sold`);偏差超閾值告警/修正 |

> **面試話術(把一致性講透):** 「庫存我用 Redis 原子預扣扛洪峰、DB 樂觀鎖做最終真相。但這引入了一個必須講的問題——**預扣成功不等於成交**:付款逾時或落庫失敗,名額要還回去,否則就『少賣』了。所以我有一條**回補閉環**:逾時掃描釋放預佔並 `INCR` Redis、消費保證至少一次 + 冪等、再加一條定時對賬以 DB 為準校 Redis。**超賣靠原子性兜底(Lua + DB 樂觀鎖),少賣靠回補閉環兜底**——兩個方向都要堵。」這條「不只防超賣、還防少賣 + 回補閉環」就是架構師線,庫存一致性深挖錨 `financial-consistency/03-order-payment-inventory`。

### Deep Dive ② · 削峰:把瞬時洪峰拉平成平穩消費(韌性的命門)

第 2 步算出 600 萬 QPS 尖峰、持續 1~3 秒。**削峰的本質:用一系列緩衝把「短促的尖峰」攤開成「較長的平穩」**,讓下游按自己能扛的速率消費,而不是被尖峰瞬間壓垮。**多層漏斗的每一層都在削峰**:

| 層 | 削峰手段 | 黑盒裡發生什麼 |
|---|---|---|
| **前端** | 按鈕置灰防重複點 + **答題/驗證碼** | 答題花掉幾秒,把「所有人同一瞬間點」**在時間軸上拉開**——300 萬人不再擠在 2 秒,攤到 10~30 秒;同時擋掉一部分腳本 |
| **網關** | **限流(令牌桶/漏斗)** + 售罄前置攔截 | 令牌桶按固定速率放行,**超過速率的直接拒**(快速失敗,不排隊);售罄後直接 409,連 Redis 都不查。算法錨 `distribution/限流算法/`、本 track `04 限流器` |
| **Redis 預扣** | `stock<=0` 立刻截斷 | 庫存賣空後,後續請求在 Redis 一條命令就被判售罄,**不往下游放一個無效請求** |
| **MQ** | **異步下單緩衝** | 搶到的請求發消息進隊列**排隊**,下游 Worker 按自己節奏拉取消費——把「幾千/秒湧入」變成「幾百/秒平穩落 DB」。這是削峰的**主力閥門** |

**MQ 削峰的核心 insight(這是面試要講清的):**

> MQ 在這裡不是為了「解耦」(雖然也解耦),核心作用是**蓄洪**——它像水庫:上游瞬間來了一場暴雨(尖峰寫入),水庫先把水蓄住(消息堆積),下游水閘按固定流量慢慢放(Worker 平穩消費)。**沒有 MQ,幾千 QPS 的下單尖峰直接砸 DB,DB 連接池被打滿;有了 MQ,DB 永遠只看到自己舒服的幾百 QPS。** 代價是**下單變異步**——用戶看到的是「排隊中」而非「立即成功」,需要輪詢結果。這個 UX 代價,在秒殺場景是完全可接受的(用戶本來就預期要等)。

> **面試話術(主動暴露 tradeoff,`01` Step 7 加分核心):** 「削峰我用多層漏斗:前端答題在時間軸拉開尖峰、網關限流快速失敗、Redis 截斷售罄、最後 **MQ 蓄洪**把寫入尖峰拉平成平穩消費。MQ 是主力閥門,它的代價是**下單從同步變異步**——用戶看到『排隊中』再輪詢結果,但秒殺場景用戶本來就預期等待,這個代價值。**反過來,如果業務要求下單必須同步秒回(比如金融下單),我就不能用 MQ 異步,得換『預扣後同步落庫 + 嚴格限流把入口卡死在 DB 能扛的範圍』**——那是另一套取捨。」這句「為什麼能異步 / 什麼時候不能」就是擁有決策。

**限流要快速失敗,不要排隊(一個容易踩的坑):** 秒殺限流的關鍵是**超限直接拒(返回「擠爆了請重試」)**,而不是讓請求排隊等。因為排隊會讓請求在系統裡堆積、佔用連接和內存,反而加速雪崩。**快速失敗 = 主動丟棄無效請求,保護核心鏈路**——這是 `system-design/01 韌性`(限流/降級/背壓)的核心思想,秒殺是它最極致的應用場景。

### Deep Dive ③ · 防刷與公平 + 熱點庫存行(防黃牛 + 解熱點)

秒殺有個別的題沒有的維度:**對抗性**——黃牛用腳本、多帳號、機房 IP 來搶,把正常用戶擠掉。公平性本身就是產品價值。

#### 防刷與公平(多道閘,層層遞進)

| 手段 | 怎麼做 | 攔什麼 |
|---|---|---|
| **限購** | Redis set 記已搶用戶(見 ① 的 Lua),同人同商品只能搶 N 件 | 同帳號重複搶 |
| **冪等鍵** | 下單 `Idempotency-Key`,同一次點擊重試不重複扣 | 狂點 + 網絡重試導致的重複下單 |
| **驗證碼/答題** | `qualify` 換令牌,機器難自動通過 | 純腳本批量打請求 |
| **黑名單 / 風控** | 異常 IP / 帳號(高頻、機房段、設備指紋)拉黑 | 黃牛機房 IP、養號 |
| **庫存令牌(token)** | 開賣前按庫存量發等量「令牌」,**無令牌的請求連 Redis 預扣都不調**(更前置的攔截) | 把無效請求擋在更前面,保護 Redis |

> **面試話術:** 「公平是秒殺的產品價值,得專門防黃牛。我分層擋:Redis set 做**限購**(同人限 N 件)、**冪等鍵**防狂點重複、**驗證碼/答題**擋純腳本、**風控黑名單**擋機房 IP 和養號。再激進一點可以用**庫存令牌**——開賣前按 1 萬庫存發 1 萬張令牌,沒令牌的請求連 Redis 都不碰,把攔截點再往前推一層。」

#### 熱點庫存行問題與分段庫存解法(deep dive 的硬核點)

回到 deep dive ① 提過的:**秒殺的熱點是「同一件商品的同一個庫存計數」,它在邏輯上是一個點,普通分庫分表打散不了。** 即使在 Redis,單個 `stock` key 也是一個熱點 key——所有扣減請求都搶它,雖然 Redis 單線程串行不會出錯,但**這一個 key 成了吞吐天花板**(單分片頂多 ~10 萬 QPS),且若該分片故障影響面集中。

**分段庫存(segment / 拆分熱點)——秒殺特有的打散思路:**

> 把 1 萬庫存**拆成 100 段,每段 100**(`stock:{item}:seg0..seg99`),扣減請求**按 userId 或隨機 hash 落到某一段**去扣。這樣:**單一熱點 key 被打散成 100 個 key**,可以分布在不同 Redis 分片上,總吞吐 ×100;單行鎖/熱點 key 壓力降為 1/100。

**分段庫存的代價(必須主動講,否則被追問會卡):**

| 問題 | 後果 | 解法 |
|---|---|---|
| **某段先賣空但全局還有** | 用戶落到空段被判售罄,實際還有貨 → **少賣 + 體驗差** | 落空段時**重試其他段** / **跨段借**(從還有貨的段勻一些過來);或允許「全局還有就放行」的兜底查詢 |
| **段間不均** | 某段被搶光、某段沒人動 | 段數別太多;定期或惰性**再平衡**;hash 要均勻 |

> **面試話術(主動暴露 tradeoff,這是架構師線):** 「秒殺的熱點和別的題不一樣——它是**同一件商品的單個庫存計數**,邏輯上一個點,**分庫分表救不了**(訂單能按 user 分,庫存計數分不了)。解法是**分段庫存**:把 1 萬拆 100 段各 100,扣減隨機落段,把單熱點打散成 100 個,吞吐 ×100。**但這引入新問題:某段先空了但全局還有貨,會誤判售罄、導致少賣。** 所以分段必須配**跨段借 / 重試其他段**的兜底。**只有真到了單 key 扛不住的量級才上分段**——大多數秒殺單 Redis 預扣的幾千扣減 QPS 根本用不到分段,過早分段反而引入跨段複雜度。」這句「**什麼時候才該分段、分段的代價是什麼**」直接推過架構師線。

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛、何時根本不該這麼設計。

### 成本:秒殺的成本是「為極短的尖峰養一支常備軍」

秒殺最反經濟的地方:**為了扛住 1~3 秒的尖峰,你得常備一套能扛百萬 QPS 接入的容量,但它一天就用幾秒,其餘 86397 秒全閒著。**

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **接入/網關** | 要扛百萬 QPS 接入的機器,但只用幾秒 | **彈性擴容**:開賣前按計劃**預擴容**,結束後縮回;雲上用 autoscaling,別常年常備峰值容量(燒錢) |
| **Redis 預扣** | 幾個分片扛扣減,平時閒置 | 可和其他業務共用集群;但秒殺期間**隔離**(別讓秒殺洪峰打垮共用 Redis 的其他業務,見失效模式) |
| **MQ + DB** | 幾百 QPS,廉價 | 幾乎不花錢——**這正是漏斗的價值:把成本壓在便宜的削峰層,不壓在昂貴的 DB 層** |
| **CDN** | 詳情頁百萬讀全靠它 | 按流量付費,但比回源便宜得多;**靜態化越徹底越省** |

> **架構師 insight:** 秒殺的成本曲線是**極度尖峰**的——這決定了「彈性 + 隔離」是核心策略:**用 autoscaling 只在活動窗口付峰值錢**,用**資源隔離**防止秒殺把常駐業務拖垮。把錢花在便宜的削峰層(CDN/限流/MQ),別讓昂貴的 DB 去硬扛——**這就是整個漏斗在成本上的意義**。

### 容量規劃:按「尖峰 + 漏斗收斂比」反推每層容量

秒殺的容量規劃不是算平均,是**算每一層要承接的尖峰,並驗證下一層的收斂比**:

```
入口 600 萬 QPS  ── CDN/前端攔掉 → 網關承接 ~百萬
網關限流(放行速率=保護下游)  ── 攔掉 → Redis 承接 ~幾萬扣減
Redis 售罄截斷  ── 只放搶到的 → MQ 承接 ~幾千
MQ 蓄洪 + Worker 限速  ── 拉平 → DB 承接 ~幾百
```
> **每一層的限流閾值 = 下一層的安全承載量。** 網關放行多少,取決於 Redis 扛多少;MQ 消費多快,取決於 DB 扛多少。**容量規劃 = 從最弱的一環(DB)倒推每一層的閘門開度**,確保任何一層都不會把下一層壓垮。這是架構師做秒殺容量的方法論——`concurrency-capacity/` 的定容思想用在尖峰場景。

### 演進路徑:從一張 MySQL 到完整漏斗

```
階段 0(小規模/低頻搶購):
   MySQL 樂觀鎖 UPDATE ... where stock>0 + 接口限流
   庫存就幾百、搶的人不多 → 單 DB 樂觀鎖完全夠,別過度設計
        │  搶的人多了,熱點行扛不住、DB 被無效請求打滿
        ▼
階段 1(中等秒殺):
   + Redis 原子預扣(扛洪峰)+ 售罄前置攔截 + 網關限流
   DB 退居「最終扣減 + 對賬」,不再扛前端流量
        │  瞬時尖峰太尖,同步下單頂不住
        ▼
階段 2(大型秒殺):
   + MQ 異步下單(蓄洪削峰)+ 下單異步化 + 結果輪詢
   + CDN 靜態化詳情頁、答題削峰、風控防黃牛
        │  單商品扣減 key 成吞吐天花板
        ▼
階段 3(超大型/12306 級):
   + 分段庫存打散熱點 + 多區域 + 完整風控 + 預擴容彈性
```

> **這條演進的關鍵 insight:** 秒殺**最容易被過度設計**。小規模搶購(庫存幾百、搶的人幾千)**一張 MySQL 樂觀鎖 + 限流就夠了**,根本不需要 Redis 預扣 + MQ + 分段庫存那一整套。**講「我會先看量級,小規模單 DB 樂觀鎖,量大了才逐層加 Redis 預扣/MQ/分段」比一上來堆滿全套更顯資深**——這正是 `01` 紅旗「上來就拆微服務」在秒殺題上的化身:上來就分段庫存 + 多活,是過度設計。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **Redis 預扣集群掛** | 庫存扣減失敗,秒殺停擺;若降級回源 DB → DB 被洪峰打死 | **不能降級回源 DB**(會雪崩)!應**快速失敗返回「活動繁忙」** + Redis 主從/集群高可用 + 預熱;以「寧可短暫不可用,不可超賣/打垮 DB」為原則 |
| **超賣真的發生了**(極端:Redis 丟數據/腦裂) | 賣超了,要賠付/補貨/取消訂單 | **DB 樂觀鎖是最後防線**(最終扣減時 `where stock>0` 兜底,絕不會把 DB 賣成負數);+ 對賬發現偏差;業務上備少量超賣緩衝 |
| **MQ 積壓** | 訂單落庫延遲,用戶「排隊中」等更久 | 可接受(用戶預期等待);Worker 擴容消費;**訂單是核心不能丟**——MQ 要持久化 + 至少一次 + 冪等消費 |
| **秒殺洪峰打垮共用基礎設施** | 秒殺把共用的 Redis/DB/網關打滿,**拖垮其他正常業務** | **資源隔離**:秒殺用獨立集群/獨立限流域;`system-design/01 韌性` 的艙壁隔離(bulkhead) |
| **逾時釋放/對賬出 bug** | 少賣(名額沒還回去)或超賣(重複釋放) | 釋放操作**冪等**(訂單狀態機:只有「預佔」能轉「已釋放」);對賬兜底 |

> **架構師話術:** 「我的單點是 Redis 預扣集群——但它掛了**絕不能降級回源 DB**,那等於把洪峰直接放給 DB,必死。正確降級是**快速失敗返回『活動繁忙』**,寧可短暫不可用,守住兩條底線:不超賣、不打垮 DB。**DB 樂觀鎖是超賣的最後防線**——就算 Redis 因為腦裂多扣了,最終落庫 `where stock>0` 也絕不會把庫存寫成負數。另外秒殺洪峰必須**資源隔離**,別讓它把共用的 Redis/網關打滿、拖垮正常下單業務。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套漏斗 | 該怎麼做 |
|---|---|---|
| **小規模/低頻搶購**(庫存幾百、搶的人幾千) | 別上 Redis 預扣 + MQ + 分段庫存 | 一張 MySQL 樂觀鎖 `where stock>0` + 接口限流就夠,別過度設計 |
| **下單必須同步秒回**(金融/不可異步) | 別用 MQ 異步下單 + 輪詢 | 預扣後**同步落庫** + 嚴格限流把入口卡死在 DB 能扛的範圍;放棄極致削峰換同步確定性 |
| **庫存充足、不是「搶」**(普通電商大促) | 別套秒殺漏斗 | 那是「**高並發但不稀缺**」——重點是擴容 + 緩存,不是防超賣漏斗;庫存不是瓶頸 |
| **強公平/實名制配額**(春運放票實名) | 別只靠「先到先得」 | 那是**配額分配 + 排隊公平**的題,可能要排隊機制 + 實名風控 + 候補,複雜度在公平不在吞吐 |
| **超賣可容忍**(如優惠券,多發幾張無所謂) | 別為「絕不超賣」上重型一致性 | 用更鬆的扣減(允許少量超發),換更高吞吐和更簡單的架構 |

> **這張表是架構師線的精華**——秒殺最大的陷阱是「**把所有限量場景都當成 12306 來設計**」。多數「秒殺」其實是小規模,單 DB 樂觀鎖足矣;只有真·百萬尖峰才需要完整漏斗。**知道何時這套是過度設計、何時超賣其實可容忍,比知道怎麼搭漏斗高一個段位**(`00` 標準線:「畫得出」→「擁有決策」)。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 秒殺的核心 insight 一句話? | **99% 的請求注定失敗,核心是把洪峰擋在庫存之前。** 300 萬搶 1 萬件,命中率 0.3%,設計目標是用**多層漏斗**把幾百萬 QPS 過濾成打到 DB 的幾百 QPS——不是讓 DB 扛百萬。 |
| 2 | 秒殺的容量估算為什麼不看平均? | 一天就秒一場,平均沒意義。看**瞬時尖峰**(~600 萬 QPS、持續 1~3 秒)和**有效/無效比**(有效 0.16%,99.84% 注定失敗)。結論:設計漏斗過濾,不是堆 DB。 |
| 3 | 為什麼不能直接用 DB 扣庫存? | 邏輯對(`UPDATE...where stock>0` 不超賣),但**扛不住**:幾千 QPS 搶同一行 → **行鎖排隊、延遲爆炸**;且海量無效請求會把 DB 連接打滿雪崩。DB 只適合做**最終扣減**,不適合扛洪峰。 |
| 4 | Redis 怎麼原子防超賣?為什麼不能 GET+DECR 兩條命令? | **Lua 一段腳本整體原子執行**(Redis 單線程,腳本期間不被插隊):`判 stock>0 → DECR` 不可分割。`GET` 再 `DECR` 兩條會被插隊 → TOCTOU 競態 → 超賣。**必須一段 Lua**。 |
| 5 | 庫存的雙層存儲是什麼?為什麼? | **Redis 預扣(扛洪峰 + 售罄判斷)+ MySQL 最終扣減(真相 + 對賬)**。Redis 快但會丟、DB 準但慢。預扣成功 ≠ 成交,所以要**回補閉環**(逾時釋放 `INCR`、落庫失敗回補、定時對賬以 DB 為準)。**超賣靠原子性、少賣靠回補。** |
| 6 | 削峰的多層漏斗 + MQ 的核心作用? | 前端答題(拉開尖峰)→ 網關限流(快速失敗)→ Redis 截斷售罄 → **MQ 蓄洪**(主力閥門)。MQ 像水庫:上游尖峰先蓄住,下游按固定流量放,把「幾千/秒尖峰」拉平成「幾百/秒平穩」。代價:**下單變異步**(排隊 + 輪詢結果)。 |
| 7 | 熱點庫存行問題?為什麼分庫救不了?怎麼解? | 熱點是**同一件商品的單個庫存計數**,邏輯一個點,訂單能按 user 分但**庫存計數分不了**。解法:**分段庫存**(1 萬拆 100 段各 100,隨機落段,吞吐 ×100)。代價:**某段先空但全局還有 → 誤判售罄**,要配**跨段借/重試**。 |
| 8 | 防黃牛/公平有哪些閘?Redis 掛了怎麼降級? | 閘:**限購**(Redis set)、**冪等鍵**(防狂點)、**驗證碼/答題**、**風控黑名單**、**庫存令牌**(更前置攔截)。Redis 預扣掛了**絕不降級回源 DB**(會雪崩)→ **快速失敗返回『活動繁忙』**,守住「不超賣 + 不打垮 DB」;DB 樂觀鎖是超賣最後防線。 |
| 9 | 秒殺何時**不該**這麼設計? | 小規模搶購(MySQL 樂觀鎖 + 限流即可,別上全套)、下單須同步秒回(不能用 MQ 異步)、庫存充足的普通大促(是擴容題不是防超賣題)、超賣可容忍(優惠券,別上重型一致性)、強實名配額(是公平/候補題)。 |

## 一句話總綱(背起來)

> **秒殺是「99% 注定失敗」題——300 萬搶 1 萬件、瞬時 600 萬 QPS 但有效僅 0.16%,核心不是讓 DB 扛百萬,是用多層漏斗(CDN 靜態化 → 答題拉開尖峰 → 網關限流快速失敗 → Redis Lua 原子預扣截斷售罄 → MQ 蓄洪削峰 → DB 樂觀鎖最終扣減)把洪峰過濾成打到 DB 的幾百 QPS。防超賣靠原子性(Lua 一段腳本 + DB `where stock>0`),防少賣靠回補閉環(逾時釋放 `INCR` + 落庫失敗回補 + 對賬);熱點庫存行分庫救不了,要分段庫存打散(代價是某段先空誤判售罄,配跨段借)。架構師線:成本是為幾秒尖峰養常備軍 → 彈性 + 隔離,小規模別過度設計(單 DB 樂觀鎖足矣),Redis 掛了快速失敗絕不回源 DB。**
