# 15 · 網約車 / 外賣派單(Uber / 滴滴 / 美團)

> **T3 第二題,也是「地理空間 + 實時撮合」這條子模式的母題。** 前面 T1/T2 全是「數據在哪、怎麼存、怎麼扇出給人看」;這題第一次把**第二維坐標(經緯度)**塞進來——數據不再是一個 key,是一個**會動的點**,而且每隔幾秒就動一次。
>
> 這題的真矛盾被 `00` 依賴表標成 **B(存儲)●● + C(韌性流量)+ E(服務化)**:它**不是 QPS 騙局**(對比短鏈),而是**兩股性質完全相反的流量擠在一起**——百萬司機**每 3–5 秒上報一次位置(寫洪峰)**,和乘客叫車時**要在 100ms 內找到附近的人(讀低延遲)**。寫的量級碾壓讀,但延遲敏感的是讀。**設計的全部張力,就是讓寫洪峰不淹死讀路徑。**
>
> 三個 deep dive 各對應一個母題:**地理空間索引怎麼把二維壓成一維可查(GeoHash/QuadTree/S2)**、**位置寫洪峰怎麼扛**、**並發多乘客搶同一司機的撮合一致性**。把這三個鑽透,後面 `16 秒殺`(搶庫存)和任何「LBS / O2O」題都通了。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個網約車 / 外賣派單系統。司機端 App 持續上報自己的位置;乘客下單後,系統要**找到附近的司機並派單**;司機接單後雙方進入行程,乘客能在地圖上看到車在動。

**這題最容易犯的錯是「上來就畫地圖」**——先用 `01` Step 1 的「問三類」把它收斂:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 核心用例?要不要拼車、預約單、路徑規劃、計價、ETA? | 核心:**司機位置上報 → 乘客叫車 → 找附近司機 → 派單撮合 → 行程追蹤**。拼車/預約/動態定價算加分項;**路徑規劃 / 地圖渲染**明確劃到地圖服務(別自己造),這題只用它的「附近查詢」 |
| **非功能(SLA)** | 派單延遲?位置上報頻率?一致性要求?可用性? | **派單要快**(找司機 + 派發目標 P99 < 1–2s,用戶等不了);**位置上報高頻**(每 3–5 秒一次);一致性:**一個司機同一時刻只能接一單**(這是強約束,撮合的命門);可用性:**派不出單比派錯單更糟**(寧可降級也別漏單) |
| **規模 / 約束** | 在線司機數?同時叫車 QPS?覆蓋範圍(全國/全球)? | **~100 萬在線司機**;**叫車峰值 ~1 萬 QPS**;全國,按城市天然分區(跨城不撮合);司機位置**只關心「現在在哪」,歷史軌跡單獨存**(別混進撮合熱路徑) |

**面試話術(把假設說出來讓對方確認):**
> 「我先框範圍——核心鏈路是四段:司機**位置上報**、乘客**叫車**、系統**找附近司機並撮合派單**、雙方**行程追蹤**。地圖渲染和路徑規劃我假設用現成地圖服務,我只調它的『附近查詢』。規模我假設 100 萬在線司機,每 3–5 秒上報一次位置,叫車峰值 1 萬 QPS,按城市分區、跨城不撮合。有個強約束我先點出來:**一個司機同一時刻只能被派一單**,這個並發一致性是我後面 deep dive 的重點。這樣可以嗎?」

**這幾個澄清直接決定後面**:

- 「位置每 3–5 秒上報」→ 第 2 步估算會算出**位置寫 QPS 是叫車 QPS 的幾十倍**,寫洪峰是真瓶頸。
- 「只關心現在在哪、歷史單獨存」→ 撮合熱路徑**只存最新位置、不留歷史**,deep dive ② 的核心減負。
- 「一個司機只能接一單」→ deep dive ③ 直接是**並發搶佔的一致性**題(和 `16 秒殺`同源)。
- 「按城市分區」→ 第 4 步分片鍵 = **地理分區(城市 / GeoHash 前綴)**,天然避免「全國一個大索引」。
- 「派不出比派錯糟」→ 第 6 步失效模式,撮合可降級(放大半徑、放寬條件),但別把單丟了。

**常見陷阱:** 在「要不要做動態定價 / 拼車路徑最優」上糾結。✅ 抓核心鏈路(上報→查附近→撮合),這些是另一類最優化問題,點到為止。

## 2. 容量估算(直接調用 `02` 的數字)

> 這題的估算精神和短鏈相反:**短鏈是「QPS 騙局」(數字小),這題是「兩股流量打架」(寫遠大於讀,但讀更敏感)**。每步照 `01` Step 2 鐵律接一句「所以架構上……」。

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 = 平均 ×3、單 Redis ~10 萬 QPS、單 MySQL 寫 ~幾千 QPS、Redis ~0.5ms vs DB 走盤幾 ms**。

### ① 位置上報寫 QPS(這題的洪峰)

```
100 萬在線司機,每 4 秒上報一次位置
寫 QPS = 1,000,000 / 4 ≈ 25 萬 QPS（持續,不是峰值——這是常態洪峰）
```
> **所以:這是這題最大的數,25 萬持續寫。對照單機 MySQL 寫才幾千 QPS——直接寫關係庫是自殺。這 25 萬必須打進內存(Redis),而且要想辦法削頻(deep dive ②)。注意它不是「峰值尖刺」,是 7×24 的常態地板。**

### ② 叫車 QPS + 附近查詢 QPS

```
叫車峰值 ~1 萬 QPS（用戶澄清給的）
每次叫車要查一次「附近司機」 → 附近查詢 ~1 萬 QPS（範圍查,比點查重）
```
> **所以:讀只有寫的 ~4%(1 萬 vs 25 萬),但讀是延遲敏感的關鍵路徑(用戶在等車)。架構張力出來了——量大的寫不敏感(晚 1 秒上報沒事),量小的讀很敏感。設計要把這兩股分開,別讓寫洪峰搶了讀的資源。**

### ③ 存儲(這題反而不是瓶頸——只存最新位置)

```
司機最新位置 = driverId + 經緯度 + 狀態 + 時間戳 ≈ 50 B/人
100 萬司機 × 50 B = 50 MB  ← 全部在線司機的實時位置才 50 MB!
```
> **所以:這是這題最反直覺的結論——實時位置數據小到一台 Redis 內存塞得下(50 MB),因為我們只存「最新位置」、覆蓋寫、不留歷史。真正大的是『歷史軌跡』(每人每天上報 ~2 萬次 × 100 萬人 = 200 億條/天),但那是離線分析數據,根本不該進撮合熱路徑——它走時序庫 / 對象存儲,和實時撮合徹底隔離。**

### ④ 行程中的追蹤連接

```
行程中的車要實時推位置給乘客 → 長連接（WebSocket）
同時進行的行程數假設 ~30 萬 → ~30 萬並發長連接
對照單機長連接 ~幾萬–10 萬 → ~5–10 台接入機（錨 02 IM 連接估法）
```
> **所以:行程追蹤是個獨立的長連接子系統(和 IM 同源),別和撮合混在一起。撮合是「一次性匹配」,追蹤是「持續推送」,兩種負載特性不同,分開部署。**

### 一句話估算結論(面試一定要說出來)

> 「**這題不是 QPS 騙局——是兩股流量打架:位置上報 25 萬持續寫(洪峰)vs 叫車 1 萬讀(延遲敏感)。寫量碾壓讀,但讀更敏感。三個結論:(1) 25 萬寫必須進 Redis 內存 + 削頻,絕不碰關係庫;(2) 實時位置只存最新、不留歷史,小到 50 MB,歷史軌跡隔離到時序庫;(3) 找附近司機是範圍查,要靠地理空間索引把二維壓成可查的一維。設計重心就在『讓寫洪峰不淹死讀路徑』和『範圍查怎麼快』。**」

這句把「兩股流量的性質差」說清楚,就過了資深線——初級答案只會說「位置上報量很大要扛」,說不出「量大的那股恰恰不敏感」這個關鍵不對稱。

## 3. API + 數據模型

### API(`01` Step 3:快過,點到為止)

```
POST /drivers/location              # 司機高頻上報(25萬 QPS 的入口)
  body: { driverId, lat, lng, ts }
  → 200（極輕,不返回業務數據,只 ack)

POST /rides                         # 乘客叫車
  body: { riderId, pickupLat, pickupLng, rideType }
  → 202 { rideId, status: "MATCHING" }   # 異步撮合,先返回受理

POST /rides/{rideId}/accept         # 司機接單(並發搶佔的入口)
  body: { driverId }
  → 200 ok / 409 already_taken       # 409 = 被別人搶先了

GET  /rides/{rideId}                # 輪詢/長連接拿撮合結果與行程狀態
  → { status, driver?, driverLocation? }
```

**要點:**
- **位置上報接口要極輕**——25 萬 QPS,每個請求多 1ms 都是巨量資源。只 ack、不返回業務數據、不觸發同步邏輯。
- **叫車是異步的**(返回 `202 MATCHING`)——撮合要查附近 + 派發 + 等接受,不可能同步等完。客戶端靠輪詢或長連接拿結果。
- **接單返回 409** 是並發搶佔的關鍵信號(deep dive ③):多個請求搶同一單/同一司機,只有一個成功,其餘 409。

### 數據模型(`01` Step 4:選型 + 分片鍵)

這題有**三類數據,三種存儲**,別混(這是這題數據模型的核心信號):

| 數據 | 存哪 | 為什麼 |
|---|---|---|
| **司機實時位置 + 狀態** | **Redis(GEO + Hash)** | 25 萬寫 / 1 萬範圍查,延遲敏感,只存最新——KV + 地理索引的完美場景。**撮合熱路徑全在這** |
| **行程 / 訂單**(rideId, rider, driver, 狀態機, 金額) | **關係庫(MySQL/PG)** | 有事務、有狀態機、要對賬計費——強一致交易場景。寫量只有叫車的 1 萬級,扛得住 |
| **歷史軌跡**(每秒每車的點) | **時序庫 / 對象存儲**(Cassandra / ClickHouse / S3) | 海量只增、離線分析(熱力圖、計費路徑、調度)——**和撮合徹底隔離** |

**核心結構(Redis 側,撮合熱數據):**

```
# 地理索引:某城市所有可派司機的位置(ZSet,底層 GeoHash 編碼)
GEOADD city:beijing:drivers <lng> <lat> driver:1001

# 司機狀態(Hash,撮合時要查"他現在能不能接單")
HSET driver:1001 status ONLINE  ride null  updated 1699...
```

**選型決策樹錨 `system-design/06 存儲選型`**;Redis GEO 為什麼適合,deep dive ① 鑽底層。

**分片鍵(`01` Step 4 鐵律:海量題必講「按什麼分」,錨 `system-design/07 數據規模化`):**

> **按地理分區分片——城市 / 行政區 / GeoHash 前綴。** 這是地理題的命門:**撮合永遠是局部的**(北京的乘客只會匹配北京的司機,絕不會跨城),所以**按地理分區天然把全國切成互不干擾的片**,每片是一個獨立的撮合域。北京一個 Redis 分片掛了,不影響上海。
>
> **熱點分區的真實風險:** 地理分片有個短鏈沒有的坑——**流量天然不均**。北京/上海的片承載的司機和叫車量是小城市的幾十倍,而且**早晚高峰的 CBD 是熱中熱**(同一個 GeoHash 格子裡擠了大量司機和訂單)。所以分片不能均勻切,要**按城市熱度動態分**(大城市單獨分片甚至多分片,小城市合併),這和短鏈「隨機碼天然均勻」完全相反——deep dive ② 會展開熱點格子的問題。

> **面試話術:** 「我按**地理分區**分片,因為撮合永遠是局部的——跨城不匹配,所以按城市切天然得到互不干擾的撮合域,故障也隔離。但地理分片有個坑:**流量極度不均**,北京 CBD 早高峰一個格子的密度是小城市的幾十倍。所以不能均勻切,要按城市熱度動態分,熱城市單獨分片。這是它和短鏈那種『隨機鍵天然均勻』的本質區別。」

## 4. 高層架構(`01` Step 5:沿一個請求一鏡到底)

**先畫能跑的最簡版,再按需加件。** 這題的關鍵是**讓三條路徑物理分離**(寫洪峰 / 撮合 / 追蹤),別擠在一起。

```
   ① 位置上報(洪峰,25萬 QPS,不敏感)
 司機App ──POST location──► 接入層 ──► 位置寫服務 ──► Redis GEO（覆蓋寫最新位置）
   每4秒一次              （極輕ack）   │ 削頻/採樣（見 DD②）
                                       └─(可選)──► MQ ──► 歷史軌跡Worker ──► 時序庫
                                                          （離線,絕不堵撮合）

   ② 叫車撮合(敏感,1萬 QPS,要快)
 乘客App ──POST /rides──► API GW ──► 撮合服務 ──┬─► Redis GEO: GEORADIUS 查附近司機
   (返回 202 MATCHING)                          │   ├─ 排序(距離/ETA/評分)
                                                │   ├─ 候選派發(搶佔/超時重派, 見 DD③)
                                                │   └─► 寫訂單到 MySQL(狀態機)
                                                ▼
                                          司機App 收到派單推送 ──► accept (409 兜並發)

   ③ 行程追蹤(長連接,獨立子系統)
 乘客App ◄══ WebSocket ══ 推送網關 ◄── 司機位置流（行程中,持續推車的位置）
```

**三條路徑各走一遍(`01` Step 5:把性質差講出來):**

- **位置上報(寫,洪峰路徑):** 司機 App 每 4 秒 POST 一次 → 接入層 → 位置寫服務**只做一件事:覆蓋寫 Redis GEO 的最新位置** → 立即 ack。**這條路徑的設計目標是「便宜」**——25 萬 QPS,每個請求必須極輕,任何同步業務邏輯都禁止。歷史軌跡走旁路 MQ 異步落時序庫,**絕不在這條路徑上同步寫盤**。
- **叫車撮合(讀+寫,敏感路徑):** 乘客 POST 叫車 → 撮合服務 `GEORADIUS` 查附近司機(Redis,~ms 級)→ 按距離/ETA/評分排序 → 取 Top-N 候選**逐個或並發派發** → 司機接受則寫訂單、鎖定司機 → 沒人接超時重派(放大半徑)。**這條路徑的設計目標是「快 + 一致」**。
- **行程追蹤(長連接,獨立):** 撮合成功後,司機的位置流通過推送網關**持續推給乘客**(WebSocket)。這是個 IM 式長連接子系統,和撮合分開部署。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **Redis GEO**(地理索引) | 25 萬寫進內存 + 1 萬範圍查要 ms 級 | ①②「寫進內存 + 範圍查要快」 |
| **位置寫服務(極輕)** | 25 萬 QPS,每請求多 1ms 都是巨量 | ①「洪峰不敏感但量大」 |
| **撮合服務(獨立)** | 撮合是低延遲關鍵路徑,不能被洪峰擠 | ②「讀敏感,和寫分離」 |
| **時序庫(旁路)** | 歷史軌跡海量但離線 | ③「歷史隔離,不進熱路徑」 |
| **推送網關(長連接)** | 行程追蹤是持續推送 | ④「追蹤是 IM 式子系統」 |

**注意這張圖的「分離」:** 三條路徑用**不同的服務、不同的負載特性**——洪峰路徑求便宜、撮合路徑求快和一致、追蹤路徑求連接保持。把它們擠進一個服務,就是讓 25 萬寫洪峰拖垮 1 萬敏感讀。**「分離不對稱的流量」是這題的架構主旋律。**

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · 地理空間索引:怎麼把二維坐標壓成一維可查(這題的真正考點)

「找附近司機」= **範圍查詢**:給一個點(乘客位置)和半徑,查圈內所有司機。問題是**經緯度是二維的**,而數據庫索引(B-Tree / ZSet)是**一維有序**的——你沒法對「(lat, lng) 一對值」建一個既按 lat 又按 lng 有序的索引。**核心難題:把二維的「鄰近」壓成一維的「相鄰」,讓範圍查變成前綴/區間查。**

**最樸素的做法為什麼不行(先講死路,顯得你懂):**

| 樸素做法 | 為什麼不行 |
|---|---|
| `SELECT * WHERE lat BETWEEN .. AND lng BETWEEN ..`(矩形框 + 算距離) | 二維範圍查,**兩個維度的索引只能用一個**,另一個維度全表掃;100 萬司機每次叫車掃一遍,1 萬 QPS 直接打爆 |
| 給每個司機算到乘客的距離再排序 | O(N) 掃全部司機,N=100 萬,不可能 ms 級 |

**三種地理空間索引(這是 deep dive 的肉):**

#### GeoHash:把二維坐標編碼成一維可前綴匹配的字符串

核心思路:**遞歸二分整個地球,用 bit 交錯編碼**。

- 把經度範圍 [-180, 180] 對半分:點在左半 → bit `0`,右半 → bit `1`;再對半,再出一個 bit……
- 緯度 [-90, 90] 同樣對半分出一串 bit。
- **把經度 bit 和緯度 bit 交錯(interleave)**:`經lng₀ 緯lat₀ 經lng₁ 緯lat₁ …`,得到一長串 bit,再按 base32 編碼成字符串,如 `wx4g0`。

**這個編碼的魔法:前綴越長 = 格子越小;前綴相同 = 地理上相鄰。** `wx4g0` 和 `wx4g1` 共享前綴 `wx4g`,它們在地圖上是緊挨著的兩個格子。**於是「查附近」變成了「查前綴相同的 key」——二維鄰近被壓成了一維的字符串前綴匹配。**

| GeoHash 長度 | 格子尺寸(約) | 用途 |
|---|---|---|
| 5 位 | ~5 km × 5 km | 城市級粗篩 |
| 6 位 | ~1.2 km × 0.6 km | 找附近司機的典型粒度 |
| 7 位 | ~150 m × 150 m | 精確定位 |

**GeoHash 的致命邊界問題(必答,顯細節功力):** 編碼是「格子」,但**兩個地理上很近的點,可能落在前綴完全不同的相鄰格子裡**。經典例子:本初子午線或赤道兩側,兩個相距 10 米的點,因為在二分的「分界線」兩側,GeoHash 前綴可能从第一位就不同。**後果:只查「同前綴格子」會漏掉貼著格子邊緣、其實很近的司機。**

> **怎麼解(這是 GeoHash 題的關鍵一招):查中心格子 + 周圍 8 個鄰格(3×3 共 9 格)。** 算出乘客所在格子後,把上下左右 + 四個角共 8 個鄰格的 GeoHash 也算出來,**一起查這 9 個格子**,再對結果按真實距離過濾。這樣邊緣的點也不會漏。Redis GEO 的 `GEORADIUS` 內部就是這麼幹的——你不用自己拼 9 格,它幫你算好。

#### QuadTree:自適應密度,熱的地方分更細

GeoHash 的格子是**固定均勻**的——但現實**密度極不均**:CBD 一個格子裡擠 1 萬個司機,沙漠一個格子裡 0 個。固定格子要嘛城市裡太粗(一格太多人)、要嘛沙漠裡太細(浪費)。

**QuadTree 的思路:遞歸把一個區域分成 4 個象限,但只在「點太多」時才繼續分。** 一個格子裡司機數超過閾值(如 100)就裂成 4 個子格,空曠區域就不裂。**於是密集區自動分得細、稀疏區保持粗——自適應密度。** 代價:樹結構要動態維護(司機移動要在樹上挪節點),比 GeoHash 的「算個字符串」重。

#### Google S2:球面曲線,沒有 GeoHash 的邊界扭曲

GeoHash 把**球面當平面**切,在兩極和子午線附近會扭曲(格子形狀畸變、邊界跳變)。**S2 把地球投影到一個外接立方體的 6 個面,在球面上用希爾伯特曲線(Hilbert curve)編號**——希爾伯特曲線比 GeoHash 的 Z 形交錯**鄰近性更好**(曲線上相鄰的 cell 在空間上也一定相鄰,沒有 Z 形那種「跳一大格」),且能在球面上保持格子大小一致。**S2 是 Uber/Google 真實生產用的**,精度和鄰近性都優於 GeoHash,代價是實現複雜。

**三者對比(面試給結論):**

| 索引 | 核心思路 | 優點 | 代價 / 適用 |
|---|---|---|---|
| **GeoHash** | 二維 bit 交錯 → 一維前綴 | **簡單、可前綴匹配、Redis 原生支持** | 固定格子、有邊界問題(查 9 格兜)、球面扭曲。**面試默認答案 / 中小規模** |
| **QuadTree** | 遞歸四分,密度自適應 | **熱區自動分細**,適合密度極不均 | 樹要動態維護,移動成本高 |
| **S2** | 球面 + 希爾伯特曲線 | 鄰近性最好、無球面扭曲、格子均勻 | 實現複雜。**Uber/大規模生產** |

**黑盒裡發生什麼——Redis GEO 的底層就是 GeoHash + ZSet(這是這題最該講透的內幕,錨 `redis-handson`):**

Redis 的 `GEOADD` / `GEORADIUS` 不是什麼神秘的地理數據結構——**它底層就是一個 ZSet(SortedSet)**:
- `GEOADD key lng lat member` 內部把 (lng, lat) **算成一個 52-bit 的 GeoHash 整數**,當作這個 member 在 ZSet 裡的 **score**。
- ZSet 按 score 有序(底層跳表 + 哈希),**GeoHash 整數有序 = 地理上大致相鄰的點 score 也相近**。
- `GEORADIUS` 查附近 = 算出中心格 + 周圍 8 格的 GeoHash 範圍,在 ZSet 上做**幾段 score 區間查(ZRANGEBYSCORE)**,再按真實距離過濾排序。

> **面試話術(主動暴露底層):** 「找附近司機的核心是把二維壓成一維可查。我會用 **GeoHash——bit 交錯把經緯度編成可前綴匹配的串,前綴相同就地理相鄰**。但有個邊界坑:貼著格子邊緣的近點前綴會差很多,所以要**查中心格 + 周圍 8 格共 9 格**再按真實距離過濾。實作上我直接用 **Redis GEO,它底層就是 ZSet——把 GeoHash 編碼成 52-bit 整數當 score,範圍查就是幾段 ZRANGEBYSCORE**。規模再大、要球面精度,我會換 **Google S2(希爾伯特曲線,Uber 在用)**;密度極不均的場景考慮 QuadTree 自適應。」

> **GeoHash 編碼 / 9 格查詢 / Redis GEO 命令的完整實作和實驗,錨 `redis-handson/03-advanced-types/scenarios/03-geo-nearby.md`,別在白板上手寫 base32 編碼。**

### Deep Dive ② · 位置上報寫洪峰:百萬司機每 4 秒一次怎麼扛(這題的命脈)

第 2 步算出 **25 萬持續寫 QPS**——這是這題的洪峰,扛不住它整個系統就垮。**怎麼扛,分三層,層層遞進(韌性削峰錨 `system-design/01 韌性`):**

#### 第一層:絕不碰關係庫,只進內存覆蓋寫

```
❌ INSERT INTO driver_location ... 每4秒每人一條  → 200億行/天,任何DB當場去世
✅ Redis GEOADD city:xx:drivers <lng> <lat> driver:1001  → 同一 member 覆蓋舊位置
```
**關鍵 insight:位置是「覆蓋語義」不是「追加語義」。** 司機的「最新位置」只有一個,新位置直接覆蓋舊的,**不留歷史**。所以實時位置數據恆定 50 MB(第 2 步算的),不會增長。歷史軌跡是**另一份數據**,走旁路異步落時序庫,和撮合零耦合。

#### 第二層:削頻 / 降採樣(把 25 萬往下壓)

不是每次上報都需要同等對待——**按司機狀態和場景動態調上報頻率**:

| 場景 | 上報頻率 | 為什麼 |
|---|---|---|
| **空閒待單**(沒在行程裡) | 可放到 **10–15 秒**一次 | 待單司機位置變化慢、撮合對精度容忍度高,沒必要 4 秒 |
| **行程中**(載著乘客) | **3–5 秒**(高頻) | 乘客在看車動,要平滑;但這只佔在線司機的一部分 |
| **靜止不動**(停車等單) | 客戶端檢測到沒動就**不報 / 極低頻** | 位置沒變,報了也是覆蓋成一樣的值,純浪費 |

> **黑盒裡發生什麼:** 削頻把「100 萬人全部 4 秒」這個峰值地板,壓成「一部分高頻 + 一部分低頻 + 一部分不報」。25 萬可能直接降到 5–8 萬。**這是用「位置精度的可容忍性」換「寫負載」的架構決策**——和短鏈用「TTL 換存儲成本」同一個思路:找到一個業務上可放鬆的維度去換系統負載。

#### 第三層:就近接入 + 批量 + 分片寫

- **就近接入分區:** 司機上報打到**就近的地理接入點**,直接寫對應城市的 Redis 分片——25 萬被地理分片切成每片幾千到幾萬,單片扛得住。
- **批量寫(可選):** 接入層攢一個小窗口(如 200ms)的位置,`GEOADD` 一批 member 一起寫,把「每次一個網絡往返」壓成「批量一次」,降 Redis 的命令數。
- **背壓:** 洪峰真超載時,位置寫**可以丟**(下一個 4 秒又來了,丟一次位置無傷大雅)——這是位置數據最大的好處:**它是可丟的、自愈的**(下次上報就修正了),所以削峰時優先犧牲它。

> **面試話術:** 「25 萬寫的核心認知是:**位置數據是覆蓋語義、可丟、自愈的**——這三個性質讓我能放開手削。第一,只覆蓋寫 Redis 內存、絕不進關係庫,歷史走旁路時序庫;第二,按狀態削頻——待單司機 10 秒一次、靜止的不報,把 25 萬壓到個位數萬;第三,地理分片把洪峰切到每片可控,超載時直接丟位置,反正下個 4 秒自愈。**對比訂單寫我一個都不能丟,位置寫我丟一打都行——把這兩種寫的『可丟性』分清楚,是這題的關鍵。」

### Deep Dive ③ · 派單撮合:並發多乘客搶同一司機的一致性(撮合的命門)

撮合 = **查附近 → 排序 → 派發 → 確認**。前兩步是 deep dive ① 的範圍查,**真正的難點在後兩步:並發一致性**——多個乘客的撮合同時看上**同一個附近司機**,但「**一個司機同一時刻只能接一單**」(第 1 步的強約束)。這和 `16 秒殺`搶庫存、短鏈的「先查再寫 TOCTOU」是同一類問題。

**樸素做法的競態(先講錯的):**

```
乘客A撮合: GEORADIUS 查到 司機X 空閒 → 派給 X
乘客B撮合: GEORADIUS 同時也查到 司機X 空閒 → 也派給 X   ← 兩單都派給了 X!
```
「先查司機空閒、再派單」在並發下有 **TOCTOU 競態**(check-then-act):兩個撮合都在 X 還空閒時查到了它,然後都派過去。**一個司機被派兩單 = 一致性破裂。**

**怎麼解(層層遞進,這是 deep dive 的肉):**

| 機制 | 怎麼做 | 黑盒裡發生什麼 / 取捨 |
|---|---|---|
| **原子鎖定司機**(核心) | 派單前對司機做**原子搶佔**:`SET driver:X:lock rideA NX`(Redis SETNX)或 DB 行級鎖 `UPDATE drivers SET ride=A WHERE id=X AND ride IS NULL` | **用存儲的原子性兜底,而不是「先查再派」**。只有一個撮合能把 X 的狀態從「空閒」原子翻到「鎖定」,另一個 `WHERE ride IS NULL` 失敗(影響 0 行)→ 它知道 X 被搶了,**換下一個候選**。這就是 API 那個 **409** 的來源 |
| **派發給司機等接受** | 鎖定後**推送派單給司機,司機有 N 秒接受窗口** | 鎖定 ≠ 接單成功。司機可能不接 / 超時。所以鎖是**帶 TTL 的臨時鎖**(如 15 秒),司機不接就自動釋放 |
| **超時重派** | 司機 N 秒沒接 → 釋放鎖 → 撮合**換下一個候選 / 放大搜索半徑**重來 | 派不出比派錯好(第 1 步約束)。重派要**放大半徑、放寬條件**(更遠的車、評分更低的車),寧可遠一點也別讓乘客空等 |
| **避免「驚群」** | 不要一次把單**廣播給附近所有司機搶**(那是另一種模式) | 「廣播搶單」(司機端搶)和「系統派單」(系統選)是兩種撮合範式:搶單模式並發更猛(很多司機搶一單),要用同樣的原子鎖兜底;派單模式系統主導,更可控。**講出這兩種範式的區別是加分點** |

**兩種撮合範式對比(主動暴露,顯得你知道邊界):**

| 範式 | 機制 | 並發一致性挑戰 | 用在 |
|---|---|---|---|
| **系統派單**(push) | 系統選最優司機,定向派發 | 多撮合搶同一司機 → 原子鎖定司機 | 滴滴快車、外賣自動派單 |
| **司機搶單**(pull) | 單廣播給附近司機,先搶先得 | 多司機搶同一單 → 原子鎖定**訂單** | 早期滴滴、貨運 |

> **黑盒裡發生什麼(這是核心一致性):** 兩種範式的本質都是「**一個資源(司機 or 訂單)只能被一個搶佔者拿到**」,解法都一樣——**把搶佔做成存儲層的一次原子寫(CAS / SETNX / 條件 UPDATE),用存儲的原子性消解 TOCTOU,而不是在應用層先查再改**。這和短鏈短碼碰撞用 `SETNX` 兜、和秒殺扣庫存用 `WHERE stock>0` 兜,是**同一個母模式:並發下的唯一性,靠存儲原子操作,不靠先讀後寫**。

> **面試話術(把母模式點破,`01` Step 6 加分核心):** 「撮合的命門是並發一致性——一個司機同時被多個乘客看上,但只能接一單。我**絕不先查空閒再派**,那有 TOCTOU 競態。我用**原子鎖定**:`UPDATE drivers SET ride=A WHERE id=X AND ride IS NULL`,只有一個能成功,失敗的換下一個候選——這就是接單接口返回 409 的來源。鎖帶 TTL,司機超時不接就釋放、放大半徑重派,因為**派不出比派錯更糟**。其實這和秒殺搶庫存、短鏈短碼防撞是**同一個母模式:並發唯一性靠存儲原子操作,不靠應用層先讀後寫**。」

**供需調度(撮合的全局視角,點到加分):** 上面是「一單怎麼派」的微觀一致性;宏觀上還有**供需失衡**——早高峰某區域司機少、訂單多。成熟系統會做**動態定價(漲價吸引司機過來)+ 調度引導(給司機發「去 X 區有單」的引導)**,把撮合從「就近匹配」升級成「全局供需最優化」。這是另一類最優化問題(可提一句,別深陷)。

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:錢花在內存和長連接,不在 CPU

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **Redis 內存** | 實時位置才 50 MB(小),但**地理分片 + 高可用副本**要按城市鋪開 | 內存本身不貴,貴在**為了地理隔離 + 高可用要部署很多分片**。按城市熱度配置,別給小城市單獨開大集群 |
| **位置寫的計算** | 25 萬 QPS 的接入 + 寫——這是常態地板,不是峰值 | 主要算力開銷項。**削頻是省錢的核心手段**(deep dive ②),把 25 萬壓到 5 萬就省 80% 接入機 |
| **長連接(行程追蹤)** | 30 萬並發 WebSocket × 多台接入機 | 和 IM 同源,連接保持的內存 / FD 成本。只有「行程中」的車要長連接,空閒司機不用 |
| **歷史軌跡存儲** | 200 億點/天,海量只增 | 走便宜時序庫 / 對象存儲 + **冷數據過期**(軌跡留 N 天就夠計費對賬,老的歸檔或刪) |

> **成本的關鍵 insight:** 這題的錢花在「**為了把 25 萬寫洪峰扛住 + 把延遲敏感的撮合保護好**」的基礎設施上(接入機 + 地理分片 Redis),不在業務計算。**削頻 + 地理分片是同時降成本和降風險的雙贏手段**——所以它不是優化,是必選項。

### 容量規劃:熱點格子是真正的瓶頸,不是總量

第 2 步算的 25 萬是**全國均攤**,但真實負載**極度傾斜到熱點格子**:北京 CBD 早高峰,一個 GeoHash 格子裡可能擠著上千個司機和幾百個訂單。**容量規劃要算的不是「全國 100 萬司機」,是「最熱的那個格子 / 那個城市分片的峰值」**——它先撐不住。

- **熱點格子的範圍查變慢:** 一格裡上千司機,`GEORADIUS` 返回上千個結果要排序,單次查詢變重。緩解:**熱格子細分**(QuadTree 思路)或**限制返回數**(只要最近 50 個,夠派單了)。
- **熱城市單獨擴容:** 北京/上海分片要遠多於小城市的容量,**按城市維度獨立容量規劃和擴縮容**,別全國一刀切。

> **架構師話術:** 「容量規劃我不看全國總量,看**最熱的城市分片和最熱的格子**——北京 CBD 早高峰一個格子上千司機,範圍查變重、撮合競爭激烈,它先撐不住。所以我按城市維度獨立容量、熱格子細分、範圍查限制返回數。這和短鏈『隨機鍵天然均勻、不用操心熱點』是兩個極端——地理數據的熱點是內生的、躲不掉的。」

### 演進路徑:從單城 MVP 到全國多區域

```
階段 1(MVP):單城,一個 Redis GEO + 一個訂單庫
   撮合服務直接 GEORADIUS,搞定一個城市先跑起來,別過度設計
        │  擴到多城,單 Redis 裝不下 / 互相干擾
        ▼
階段 2:按城市地理分片(Redis 分片 + 訂單庫按城市分庫)
   每城一個獨立撮合域,故障隔離;熱城市單獨分片
        │  跨城市 / 跨地域低延遲 + 容災
        ▼
階段 3:多區域部署 + 就近接入
   位置上報就近寫、撮合就近查;換 S2 提精度;供需調度做全局最優化
```

> **這條演進的關鍵 insight:** 地理題**天生可分**(撮合本來就是局部的),所以**從單城起步是對的、也是安全的**——不像有些題分片鍵難選,這題的分片鍵(地理分區)從第一天就清楚。但**MVP 別上 S2 / QuadTree / 動態定價**,單城用 Redis GEO + GeoHash 9 格查就夠跑。`01` 紅旗「上來就堆複雜度」在這題就是「單城 demo 上來就鋪 S2 多區域多活」。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **位置 Redis 分片掛** | 該城市**找不到附近司機 = 派不出單**(撮合癱瘓) | 分片高可用(主從 + 哨兵);掛了**降級**:放大搜索範圍到鄰近分片、或臨時用稍舊的位置快照。**派不出比派錯糟,降級也要出單** |
| **位置寫洪峰超載** | 寫排隊、Redis 延遲飆升,連累撮合的讀 | **位置寫和撮合讀物理隔離**(第 4 步);洪峰時**丟位置寫**(可丟自愈)保撮合讀。這是把 25 萬不敏感寫和 1 萬敏感讀分開的核心理由 |
| **撮合服務雪崩 / 連環超時** | 大量乘客撮合中卡住,司機鎖不釋放 | 撮合鎖**必帶 TTL**(司機不接自動釋放,別永久鎖死);撮合服務限流 + 熔斷下游;超時快速失敗重派 |
| **司機被重複派單**(一致性失效) | 一車兩單,事故 | 靠 deep dive ③ 的**原子鎖定**從根上杜絕;對賬層兜底發現異常單 |
| **行程追蹤長連接斷** | 乘客看不到車動(體驗降級,但行程還在) | 追蹤是**非關鍵路徑**——斷了不影響撮合和計費,客戶端重連 + 輪詢兜底 |

> **架構師話術:** 「我的單點風險是**城市位置分片**——它掛了該城市派不出單,所以要主從高可用 + 降級(放大半徑用鄰片、退用稍舊位置)。最關鍵的設計決策是**位置寫和撮合讀物理隔離**:25 萬不敏感寫真超載時,我**丟位置寫保撮合讀**——位置可丟自愈,撮合一卡用戶就流失。撮合鎖必帶 TTL 防司機鎖死。行程追蹤反過來是可丟的非關鍵路徑,斷了客戶端重連就好。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **小城市 / 低密度 LBS**(找附近店、量小) | 別上地理分片 + S2 + 削頻洪峰 | 一個 Redis GEO 甚至 PostGIS(Postgres 地理擴展)一張表 + 索引就夠 |
| **不要求實時、可批量的調度**(物流路線規劃) | 別用「實時撮合 + 長連接」 | 那是**離線/準實時最優化**問題(VRP 車輛路徑),走批處理 + 求解器,不是實時匹配 |
| **位置要留完整歷史軌跡的場景**(車隊管理、合規) | 別用「只存最新、覆蓋寫」 | 那時歷史軌跡是核心數據,要設計時序存儲為主,撮合反而次要 |
| **強一致 + 精確計費路徑**(按 GPS 軌跡計費) | 別把「可丟自愈的位置」直接拿來計費 | 計費走**單獨的高保真軌跡通道**(不可丟、補點),和撮合的「可丟位置」分開——別讓削峰丟掉計費依據 |
| **全球級 / 球面精度要求高** | 別用平面 GeoHash(兩極/子午線扭曲) | 用 **S2**(球面希爾伯特曲線),這是 Uber 的選擇 |

> **這張表是架構師線的精華**——「知道一個方案的邊界、知道何時它是錯的」,比「知道這個方案怎麼做」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。尤其「**位置可丟 ≠ 計費可丟**」這條,點出了「同一份位置數據,在不同通道有不同的保真要求」,是真實生產的深坑。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 網約車的容量估算結論一句話? | **不是 QPS 騙局,是兩股流量打架**:位置上報 **25 萬持續寫**(洪峰、不敏感、可丟)vs 叫車 **1 萬讀**(延遲敏感)。寫碾壓讀但讀更敏感。設計重心 = **讓寫洪峰不淹死讀路徑** + 範圍查要快。 |
| 2 | 為什麼實時位置數據只有 ~50 MB? | 只存**最新位置、覆蓋寫、不留歷史**(覆蓋語義)。100 萬司機 × ~50 B = 50 MB,一台 Redis 塞得下。海量的是歷史軌跡(200 億點/天),走時序庫**和撮合徹底隔離**。 |
| 3 | GeoHash 怎麼把二維壓成一維?致命邊界問題? | 經緯度各自二分出 bit,**交錯(interleave)成一串**再 base32 編碼;**前綴相同=地理相鄰**,範圍查變前綴匹配。邊界坑:**貼著格子邊緣的近點前綴差很大**→ 要**查中心格 + 周圍 8 格(3×3 共 9 格)**再按真實距離過濾。Redis `GEORADIUS` 內部就這麼做。 |
| 4 | GeoHash vs QuadTree vs S2? | **GeoHash**:bit 交錯、可前綴匹配、Redis 原生,但固定格子+球面扭曲(默認答案)。**QuadTree**:遞歸四分、密度自適應(熱區自動細分)。**S2**:球面+希爾伯特曲線,鄰近性最好無扭曲(Uber 生產用)。 |
| 5 | Redis GEO 的底層是什麼? | **就是 ZSet**:`GEOADD` 把 (lng,lat) 算成 **52-bit GeoHash 整數當 score**,ZSet 按 score 有序=地理相鄰 score 相近;`GEORADIUS` = 算 9 格範圍做幾段 `ZRANGEBYSCORE` 再按距離過濾。 |
| 6 | 25 萬位置寫洪峰怎麼扛?三層。 | 認知:位置**覆蓋語義 + 可丟 + 自愈**。① 只覆蓋寫 Redis 內存,**絕不進關係庫**,歷史走旁路時序庫;② **按狀態削頻**(待單 10 秒/靜止不報),25 萬→個位數萬;③ 地理分片切到每片可控,**超載直接丟位置**(下個 4 秒自愈)。 |
| 7 | 撮合並發一致性:多人搶同一司機怎麼辦? | **絕不「先查空閒再派」**(TOCTOU 競態,一車兩單)。用**原子鎖定**:`UPDATE drivers SET ride=A WHERE id=X AND ride IS NULL` / `SETNX`,只一個成功,失敗換下個候選(=接單接口的 **409**)。鎖**帶 TTL**,司機超時不接就釋放、**放大半徑重派**(派不出比派錯糟)。 |
| 8 | 撮合一致性和秒殺/短鏈是什麼關係? | **同一個母模式:並發唯一性靠存儲原子操作,不靠應用層先讀後寫**。撮合鎖司機、秒殺扣庫存(`WHERE stock>0`)、短鏈防撞(`SETNX`)——都是「一個資源只能被一個搶佔者拿到」,用 CAS/條件寫消解 TOCTOU。 |
| 9 | 網約車何時**不該**這麼設計? | 小城市低密度 LBS(PostGIS 一張表即可)、離線物流路線規劃(VRP 批處理求解器,非實時撮合)、**計費別用「可丟位置」**(走高保真不可丟軌跡通道)、全球級精度用 S2 不用平面 GeoHash。 |

## 一句話總綱(背起來)

> **網約車是「不對稱流量」題——位置上報 25 萬持續寫(洪峰、不敏感、可丟自愈)vs 叫車 1 萬讀(延遲敏感),設計全部張力是讓寫洪峰別淹死讀路徑(物理隔離 + 削頻 + 超載丟位置)。找附近司機靠地理空間索引把二維壓一維:GeoHash bit 交錯成可前綴匹配的串(邊界查 9 格兜)、底層就是 Redis ZSet,大規模換 S2。撮合的命門是並發搶同一司機,靠存儲原子鎖定(WHERE ride IS NULL / SETNX)消解 TOCTOU、鎖帶 TTL 超時放大半徑重派——和秒殺扣庫存、短鏈防撞是同一個『並發唯一性靠原子寫』母模式。架構師線:實時位置只存最新(50 MB)歷史軌跡隔離、熱點格子才是真瓶頸、單城 MVP 起步別堆 S2、位置可丟但計費不可丟。**
