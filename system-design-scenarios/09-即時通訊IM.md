# 09 · 即時通訊 IM / 群聊

> **T2 第二題,也是 SDI 第二高頻(僅次於 News Feed)。** 前面的題你都在玩「請求—回應」:Client 打一發、Server 回一發、連接用完即拋。IM 把這個模型**整個翻過來**——服務器要**主動推**消息給用戶,於是你被迫面對一個全新的約束軸:**連接**。
>
> 這題的「教學價值」集中在一句反直覺的話:**IM 的關鍵約束不是 QPS,是並發長連接數。** `02 Drill 4` 已經算過——5000 萬 DAU、消息峰值才 6 萬 QPS(對單機 Redis 不算什麼),但**同時在線 ~1000 萬條長連接**,這才是逼你上 200 台接入機、設計路由表、做心跳保活的真矛盾。前面所有題的「無狀態服務 + 水平擴」直覺在這裡**部分失效**:接入層是**有狀態**的(連接黏在某台機器上),這是 IM 最難、也最能分高下的地方。
>
> 用 `00` 的依賴表看:IM 吃 **E(服務化與通信)●● + A/B/C/D**。E 被 ●● 標記,因為 IM 本質是「一條消息怎麼從 A 的連接,路由到 B 的連接」——這是**通信範式**(長連接、推送、扇出)的母題,和 `08 News Feed`、`10 通知系統` 共用同一塊扇出肌肉(`00`:「練一題通三題」)。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個即時通訊系統(像 WhatsApp / 微信 / Slack)。支援一對一聊天和群聊,消息要**即時**(秒級)送達在線用戶,離線用戶上線後能補齊,要有**在線狀態**和**已讀回執**,支援**多端登錄**(手機 + 電腦同時在線)。

**別一上來就畫**——IM 的澄清比短鏈關鍵得多,因為**幾個非功能維度直接決定架構是另一套**(有狀態接入層 vs 無狀態):

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 1對1 / 群聊都要?群多大?已讀回執?多端同步?歷史消息? | 1對1 + 群聊都要;群上限假設 **500 人**(微信級;超大群是另一檔,見第 6 段);**已讀回執要**;**多端同步要**;歷史消息**要存且可拉** |
| **非功能(SLA)** | 送達延遲?可靠性(能丟消息嗎)?消息有序?一致性? | 在線送達 **<1s(P99)**;消息**不能丟**(at-least-once + 客戶端去重);**單會話內必須有序**;在線狀態**可以最終一致**(晚幾秒無所謂) |
| **規模 / 約束** | DAU?人均消息?同時在線比例?群佔比? | **5000 萬 DAU**;人均每天收發 **40 條**;同時在線假設 **20%(1000 萬連接)**;消息均 **200 B** |

**面試話術(把假設說出來讓對方確認):**
> 「我先框範圍——1對1 和群聊都做,群我假設上限 500 人,超大直播群是另一套方案我留到最後講。已讀回執和多端同步都要,這兩個是 IM 的難點我會重點 deep dive。SLA 上最關鍵的兩條:**消息絕對不能丟**(我會做應用層 ACK + 客戶端去重),**單會話內必須有序**(我用單調遞增的 seq)。在線狀態我假設可以最終一致,晚幾秒沒人死。規模上 5000 萬 DAU、同時在線 20% 也就是**1000 萬條長連接**——我先說結論:**這題的關鍵約束是連接數,不是 QPS**,我估算會證明這點。這樣可以嗎?」

**這幾個澄清直接決定後面**:

- 「**1000 萬並發長連接**」→ 第 5 步架構**多一層接入網關(有狀態)**,這是 IM 和前面所有題的本質分野。
- 「**消息不能丟**」→ deep dive ② 整段在講可靠投遞(應用層 ACK、離線補償、去重)。
- 「**單會話有序**」→ deep dive ② 的 seq 設計;注意是**會話內有序**,不是全局有序(全局有序代價極高且沒必要)。
- 「**多端同步 + 已讀回執**」→ deep dive ③ 的扇出成本(已讀回執在大群是 N² 災難)。
- 「**在線狀態可最終一致**」→ presence 不必強一致,給了你用便宜方案(Redis + TTL)的底氣。

**常見陷阱:** 在「表情包怎麼存」「語音消息格式」這種細節上糾結。✅ 抓大放小——IM 的核心是**連接 + 投遞 + 扇出**三件事,媒體消息本質是「先傳對象存儲拿 URL,再把 URL 當文本消息發」,**退化成短鏈/網盤題**,別在這裡展開。

## 2. 容量估算(直接調用 `02 Drill 4`)

> 這題的估算 `02` 的 **Drill 4** 已經算過——這裡擴成完整版,並且**每步都接一句「所以架構上……」**(`01` Step 2 鐵律)。IM 估算的重點和前面題**不一樣**:前面題算 QPS/存儲就夠,IM 必須**多算一條連接數**,而且這條才是主角。

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 = 平均 ×3、副本 ×3、單 Redis ~10 萬 QPS、單機長連接 ~幾萬–10 萬**。

### ① 消息 QPS(其實不是瓶頸)

```
5000 萬 DAU × 40 條/天 = 20 億條/天
20 億 ÷ 1e5 s = 2 萬 QPS(平均)
峰值 ×3 ≈ 6 萬 QPS
```
> **所以:消息 QPS 才 6 萬——單個 Redis/Kafka 集群輕鬆扛,幾個分片就夠。如果這題只看 QPS,你會以為它和短鏈一樣是小題。但這是 IM 最大的認知陷阱——真正的約束在下面那條。**

### ② 並發長連接(IM 的真主角)

```
5000 萬 DAU × 同時在線 20% = 1000 萬條並發長連接
單機長連接上限 ~5 萬(受內存 / FD / 心跳開銷限)
1000 萬 ÷ 5 萬 = ~200 台接入機
```
> **所以:這才是 IM 的命門。** 6 萬 QPS 是「每秒處理 6 萬個消息事件」,而 1000 萬連接是「**同時要握住 1000 萬個 TCP 連接、每個都得定期心跳、每個都可能隨時要被推消息**」。連接是**常駐資源**(占內存、占 FD),不是 QPS 那種「用完即還」的瞬時資源。**這條決定了你需要一個獨立的、有狀態的接入層,並且機器數量由連接數而非 QPS 算出來。**

**為什麼單機只能扛 ~5 萬連接(黑盒裡的成本):** 每條 WebSocket/TCP 連接在內核裡占一個 FD(文件描述符,有上限要調 `ulimit`),在應用層占一塊讀寫緩衝區 + 連接對象(幾 KB ~ 幾十 KB)。1000 萬連接 ÷ 200 台 = 5 萬/台,每台若每連接占 20KB,光連接內存就 1GB。再加上**每連接的心跳定時器**——1000 萬個定時器是巨大的調度壓力,得用時間輪(timing wheel)這種 O(1) 結構管理,不能一連接一 `Timer`。這就是為什麼接入層要用 epoll/Netty/Go netpoller 這種**事件驅動 + 少量線程管海量連接**的模型,而不是一連接一線程(那要 1000 萬線程,內核直接崩)。

### ③ 存儲(可觀但不是最難)

```
5000 萬 × 40 條 × 200 B = 4000 億 B = 400 GB/天(裸量)
× 副本 3 = 1.2 TB/天
一年 ≈ 400 TB(裸)/ 1.2 PB(含副本)
```
> **所以:存儲到 PB 級 → 必走水平擴展 KV(Cassandra/HBase),按會話分片。但存儲在 IM 裡是「已解決的問題」**(就是個寫密集的 KV,後面 deep dive ② 講分片鍵)——它不是這題的難點,難點是連接和投遞。**面試時別在存儲上花太多時間,點出「按會話分片、用寬列 KV」就夠,把白板時間留給連接和扇出。**

### ④ 群消息扇出(隱藏的放大器)

人均 40 條看著小,但**群消息有扇出放大**:一條群消息發到 500 人群 = **1 次寫 → 500 次推送**。假設 30% 消息是群消息、平均群 100 人:

```
群消息寫 QPS:6 萬 × 30% = 1.8 萬
扇出後的推送 QPS:1.8 萬 × 100(平均群人數)= 180 萬次推送/s
```
> **所以:扇出讓「下行推送量」遠大於「上行消息量」**(這裡 30 倍)。這直接決定 deep dive ② 要討論「群消息用寫擴散還是讀擴散」——和 `08 News Feed` 同一個母問題,但 IM 的答案和 feed 不同(下面講)。

### 一句話估算結論(面試一定要說出來)

> 「**這題的數字感和前面題反過來——消息 QPS 才 6 萬、存儲 PB 級都不是最難的,真正的關鍵約束是 1000 萬並發長連接逼出的 ~200 台有狀態接入機。** 所以我的架構重心和 deep dive 全在三處:(1) 接入層怎麼握住百萬連接、消息怎麼從 A 的連接路由到 B 的連接;(2) 消息怎麼保證不丟、有序、不重複;(3) 群扇出和多端同步的成本。存儲就是個按會話分片的寬列 KV,我快速帶過。」

這句「**連接數才是 IM 的關鍵約束,不只是 QPS**」就是 `01` 說的資深 vs 初級的分水嶺——它證明你看穿了 IM 和普通 CRUD 系統的本質區別。

## 3. API + 數據模型

### API(`01` Step 3:IM 的接口分兩類)

IM 的 API 和前面題有個本質區別:**不只有 request/response,還有 server→client 的推送**。所以要分**上行 RPC**(客戶端發起)和**下行推送**(服務端發起)兩類:

```
# ── 連接管理(長連接,非 HTTP)──
WS  CONNECT  /ws?token=...          → 建立 WebSocket,鑑權,登記路由表

# ── 上行(客戶端 → 服務端,走長連接內的應用層協議)──
SEND     { convId, clientMsgId, body }        → { serverMsgId, seq, ts }   # clientMsgId 用於去重冪等
ACK      { convId, seq }                       → ok                         # 客戶端確認已收到,用於投遞可靠性
READ     { convId, seq }                       → ok                         # 已讀回執,上報讀到哪
PULL     { convId, sinceSeq, limit }           → [msg]                      # 離線/翻歷史,按 seq 增量拉

# ── 下行(服務端 → 客戶端,主動推)──
PUSH     { convId, seq, body, senderId }       # 新消息
PUSH_ACK { convId, seq }                        # 對方收到了(投遞回執)
PUSH_READ{ convId, seq, readerId }              # 對方讀了(已讀回執)
PRESENCE { userId, status }                     # 好友上下線
```

**要點:**
- **`clientMsgId`(客戶端生成的去重鍵)是 IM 可靠性的命根子**——它讓「at-least-once 投遞 + 客戶端去重」成為可能(deep dive ②)。為重發鋪路,和 `01` Step 3「寫接口想冪等」是同一個習慣,但在 IM 裡是強制的。
- **`PULL` 用 `sinceSeq` 增量拉**,不用時間戳——因為 seq 單調且無歧義(時間戳會有時鐘問題、會撞)。和 `01` Step 3「分頁用游標不用 offset」同源。
- 上行/下行**共用一條長連接**(全雙工),不是兩條 HTTP——這是 WebSocket 相對輪詢的核心價值。

### 數據模型(`01` Step 4:選型 + 分片鍵)

**核心三張表**(會話、消息、用戶會話列表 / 收件箱):

**① 消息表(message)— 寫密集主表:**

| 字段 | 類型 | 說明 |
|---|---|---|
| `conv_id` | bigint **分片鍵** | 會話 ID(1對1 或群)——**按它分片** |
| `seq` | bigint | **會話內單調遞增**序號,排序 + 去重 + 增量拉的基準 |
| `msg_id` | bigint | 全局消息 ID(Snowflake) |
| `sender_id` | bigint | 發送者 |
| `body` | blob | 消息內容(文本/媒體 URL) |
| `created_at` | ts(ms) | 服務端落庫時間 |

主鍵 `(conv_id, seq)`——天然按會話聚簇、按 seq 有序。

**② 會話成員表(conv_member):** `conv_id, user_id, last_read_seq, joined_at`——存「誰在這個會話、讀到哪了」(已讀回執靠 `last_read_seq`)。

**③ 用戶收件箱 / 會話列表(user_inbox):** `user_id, conv_id, last_msg_seq, unread_count`——存「這個用戶有哪些會話、各自最新到哪、未讀幾條」,是用戶打開 App 時拉會話列表的入口。

**選型(決策樹錨 `system-design/06 存儲選型`):**

| 候選 | 適不適合 | 理由 |
|---|---|---|
| **寬列 KV(Cassandra / HBase)** | ✅ 消息表主力 | 訪問模式是「按 `conv_id` 寫入、按 `(conv_id, seq)` 範圍掃」——寬列的 partition key + clustering key 完美契合;PB 級自動分片;寫密集友好(LSM 樹順序寫) |
| **Redis** | ✅ 路由表 + presence + 最近消息緩存 | 路由表(user→連接)、在線狀態(TTL)、會話最近 N 條熱消息,都是 Redis 的場景(錨 `redis-handson/`) |
| **SQL** | △ 會話/成員元數據可用 | 成員關係、會話設置這種低頻關係數據用 SQL 沒問題;消息表千萬別用 SQL(PB 級 + 寫密集會爆) |

> **面試話術:** 「消息表是寫密集 + PB 級,我用寬列 KV(Cassandra),partition key = `conv_id`、clustering key = `seq`,這樣一個會話的消息物理聚在一起、按 seq 有序,增量拉就是一次範圍掃。路由表和在線狀態這種高頻易變的小數據放 Redis。會話成員元數據可以用 SQL。」

**分片鍵(`01` Step 4 鐵律:海量題必講,錨 `system-design/07 數據規模化`):**

> **消息按 `conv_id` 分片,不是按 `user_id`。** 這是 IM 的關鍵決策:**按會話分**,一個會話的所有消息落同一片,**增量拉 / 範圍掃單片命中**,seq 在單片內生成也最簡單(單調遞增不用跨片協調)。如果按 user_id 分,1對1 消息會被「發送者的片」和「接收者的片」撕成兩份,拉一個會話要跨片合併,seq 也無處安放。

> **熱點分片問題:** 超大群(萬人群)的 `conv_id` 會成為熱點分片(所有人往一個會話寫)。1對1 和小群沒這問題(會話多、天然均勻)。超大群是這題的特殊矛盾,留第 6 段架構師線講「怎麼把熱群打散」。

## 4. 高層架構(`01` Step 5:沿一條消息一鏡到底)

**IM 架構的核心特徵:多一層「接入網關(Gateway)」,它是有狀態的——連接黏在它上面。** 先畫能跑的最簡版,看一條消息怎麼從 A 走到 B:

```
                        ┌─────────────── 接入層(有狀態,~200 台)───────────────┐
                        │                                                        │
  用戶 A ══長連接(WS)══► Gateway-1 ────┐                          ┌──── Gateway-7 ══長連接══► 用戶 B
  (手機)                  │  ▲          │                          │  ▲                       (手機)
                          │  │心跳       │ ① A 發消息上行            │  │ ⑤ 推給 B 的連接
                          │  └保活       ▼                          │  └保活
                          └──登記路由──► 路由表(Redis)             └──查路由表─┘
                             user→gw      user_B → Gateway-7

           ① 上行           ②                ③                      ④                ⑤ 下行推送
   A ──SEND──► Gateway-1 ──► Logic/Message Svc ──► 寫消息(KV)+ 發 seq ──► 查 B 在哪(路由表)──► 投遞:
                                  │                                          │
                                  │                                          ├─ B 在線 → 通過 Gateway-7 推給 B(走長連接)
                                  │                                          └─ B 離線 → 寫 B 的離線消息隊列 + 觸發 APNs/FCM 離線推送
                                  ▼
                            發消息事件到 MQ(Kafka)──► 異步:更新收件箱未讀數 / 已讀統計 / 落歷史庫
```

**一條 1對1 消息一鏡到底(`01` Step 5:核心路徑走完):**

1. **A 上行:** A 的手機通過常駐 WebSocket 把消息 `SEND` 給它連著的 **Gateway-1**(A 連在哪台就走哪台)。Gateway 只負責「收發 + 維持連接」,不含業務邏輯。
2. **路由到 Logic:** Gateway-1 把消息轉給無狀態的 **Message/Logic Service**(這層可隨意水平擴,和普通服務一樣)。
3. **持久化 + 發 seq:** Logic 給這條消息分配**會話內單調 seq**、寫入消息 KV(`conv_id` 分片)、生成 `serverMsgId`。
4. **查 B 在哪:** Logic 查**路由表(Redis)**——`user_B → 它連在哪台 Gateway`。這就是 IM 的核心數據結構:**用戶→連接的映射**。
5. **下行推送(分在線/離線):**
   - **B 在線:** 路由表說 B 連在 **Gateway-7**,Logic 把消息投給 Gateway-7,Gateway-7 通過它和 B 的長連接**主動推**給 B。
   - **B 離線:** 路由表查不到 B(或標記離線)→ 把消息寫進 **B 的離線消息隊列**(B 上線會 PULL),並觸發 **APNs/FCM** 走系統級離線推送(喚醒手機彈通知)。
6. **異步副作用:** 同時發一條事件到 **MQ(Kafka)**,異步更新收件箱未讀數、落歷史庫、做統計——這些不堵主投遞路徑。

**群消息的差別:** 第 4 步從「查一個 B」變成「查群成員列表的每個人在哪」,第 5 步變成「對每個在線成員推、每個離線成員寫離線隊列」——這就是扇出,deep dive ② 講寫擴散/讀擴散的取捨。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **接入網關 Gateway(有狀態)** | 握住 1000 萬長連接、心跳保活、收發消息 | ②「連接是真主角,~200 台」 |
| **路由表(Redis:user→gateway)** | 一條消息要找到「B 連在哪台機器」才能推過去 | ②「連接黏在機器上,要可定位」 |
| **Logic/Message Svc(無狀態)** | 業務邏輯、發 seq、持久化——和連接解耦,可獨立擴 | 把有狀態的連接和無狀態的邏輯分層 |
| **消息 KV(conv_id 分片)** | PB 級寫密集存儲 + 增量拉 | ③「存儲 PB 級」 |
| **離線隊列 + APNs/FCM** | 離線用戶補消息 + 喚醒手機 | deep dive ②「離線補償」 |
| **MQ(Kafka)** | 群扇出削峰 + 異步副作用解耦 | ④「扇出放大 180 萬推送/s」 |

> **這張圖的關鍵 insight(面試一定要點出):** 「IM 架構的精髓是**把有狀態的連接層(Gateway)和無狀態的業務層(Logic)分開**。Gateway 只做一件事:握連接 + 收發字節 + 心跳。所有業務邏輯在無狀態的 Logic 層,可以像普通服務一樣隨便擴。連接和邏輯一旦分層,**路由表就成了它們之間的橋**——這是整個 IM 設計的軸。」

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

> IM 的三個 deep dive 嚴格對應澄清埋的三個雷:**①長連接架構**(連接怎麼握、消息怎麼路由)、**②可靠投遞與有序**(不丟/有序/不重 + 群扇出)、**③在線狀態與多端同步**(presence + 多設備 + 已讀回執成本)。

### Deep Dive ① · 長連接架構(IM 的真正考點)

這是 IM 和前面所有題的本質分野,也是最會被往死裡追問的點。拆成四個小問:**連接怎麼握、路由表怎麼做、心跳怎麼保活、一條消息怎麼從 A 的連接路由到 B 的連接。**

#### (a) 為什麼是長連接,不是輪詢

| 方案 | 機制 | 致命傷 |
|---|---|---|
| **短輪詢(polling)** | 客戶端每 N 秒問一次「有新消息嗎」 | 延遲 = 輪詢間隔(秒級不夠即時);1000 萬人每 3 秒問一次 = **333 萬 QPS 空轉**,絕大多數是「沒有新消息」的浪費 |
| **長輪詢(long-polling)** | 請求 hang 住,有消息才回 | 比短輪詢省,但每條消息一個 HTTP 往返、連接反覆建立,且服務端要 hold 住請求,半吊子 |
| **WebSocket / 長 TCP** | 一條**常駐全雙工**連接,服務端能**主動推** | ✅ 真即時、零空轉;**代價是連接常駐有狀態**——這正是後面所有複雜度的來源 |

> **面試話術:** 「即時送達 + 服務端主動推,輪詢做不到(要嘛延遲高、要嘛海量空轉),所以必須長連接——WebSocket 或裸 TCP。代價是**連接變成有狀態的常駐資源**,後面的路由表、心跳、保活、優雅下線全是為了管這個狀態。」

#### (b) 路由表:user → 連接在哪(IM 的核心數據結構)

消息要從 A 推到 B,必須回答一個問題:**「B 的長連接掛在哪台 Gateway 上?」** 這就是路由表:

```
路由表(Redis):  user_id → { gateway_id, conn_id, ... }
                 user_B  → { gw-7, conn#3391 }
```

| 設計點 | 怎麼做 | 黑盒裡發生什麼 |
|---|---|---|
| **登記** | B 連上 Gateway-7 時,Gateway-7 往 Redis 寫 `user_B → gw-7` | 連接建立的副作用就是寫路由表;斷開時刪除 |
| **多端** | 一個 user 多個設備 → 值是**一個集合**:`user_B → {gw-7:phone, gw-3:pc}` | 推消息要推給**所有設備**(deep dive ③ 多端同步) |
| **過期兜底** | 路由表項帶 **TTL**,靠心跳續期 | Gateway 崩了沒來得及刪,TTL 到期自動清,避免「殭屍路由」把消息推給死連接 |
| **定位下推** | Logic 查到 `gw-7` → 通過**內部 RPC** 把消息發給 gw-7 → gw-7 用 `conn#3391` 找到那條連接推出去 | Gateway 內部維護 `conn_id → 連接對象`(本地內存,因為連接就在這台機器上) |

> **這裡有個容易被追問的陷阱:路由表 Redis 掛了怎麼辦?** 答:路由表是**可重建的軟狀態**——連接的「真相」在 Gateway 的本地內存(它握著真連接),Redis 只是個「索引」。Redis 掛了,新消息暫時找不到 B 在哪(可降級為「先寫離線隊列,B 反正連著,讓 B 靠心跳/重連時 PULL 補」),Redis 恢復後 Gateway 重新登記即可。**「真相在連接、Redis 是索引」這個分層,讓路由表的故障變成可降級而非致命**——這句話講出來就過了資深線。(韌性錨 `system-design/01 韌性-依賴掛了怎麼不崩`)

#### (c) 心跳保活:怎麼知道連接還活著

長連接最坑的地方:**TCP 連接可能「假死」**——對端手機進地鐵、切後台、NAT 超時,連接在你這頭看著還在(沒收到 FIN/RST),其實對面早沒了。你往這條死連接推消息,消息就黑洞了。心跳是唯一解:

| 機制 | 怎麼做 | 取捨 |
|---|---|---|
| **TCP keepalive** | 內核級保活探測 | 默認 2 小時太慢,且**只探活不能帶業務語義**(不知道 App 層是否健康)——不夠 |
| **應用層心跳(主力)** | 客戶端每 ~30s 發一個 `PING`,服務端回 `PONG`;服務端 N 個週期沒收到 PING 就判定連接死、清理 | ✅ 能探 App 層活性;**心跳間隔是個 tradeoff**:太頻繁(10s)→ 1000 萬連接 × 心跳 = 海量空包耗電耗流量耗 CPU;太稀疏(5min)→ 死連接遲遲發現不了、消息黑洞久 |
| **心跳間隔的工程值** | 一般 **30s~60s**;移動端用「自適應心跳」(在不同網絡下動態探測 NAT 超時邊界,在它失效前發心跳) | 微信的「自適應心跳」就是為了在省電和保活間找平衡——這是個能展示深度的細節 |

> **黑盒裡的成本:** 1000 萬連接、每連接每 30s 一次心跳 = **33 萬次心跳/s** 的純開銷。所以心跳處理要極輕(不能每個心跳查 DB),且定時器要用**時間輪**(O(1) 管理千萬級定時器),不能一連接一個 `Timer` 對象(會被 GC 和調度壓垮)。**這就是為什麼接入層要用 Netty/Go netpoller 這種專門的高並發網絡框架,而不是隨便起個 HTTP server。**

> **面試話術:** 「長連接會假死——對端切後台、NAT 超時,你這頭看著連接還在其實已死,往它推消息就黑洞了。所以必須應用層心跳:客戶端 30s 一個 PING,服務端幾個週期沒收到就判死清理 + 刪路由表。心跳間隔是 tradeoff:太頻耗電耗 CPU(千萬連接放大),太疏死連接發現慢。1000 萬連接的心跳要用時間輪管定時器,不然調度壓力扛不住。」

#### (d) 一條消息怎麼從 A 的連接路由到 B 的連接(把上面串起來)

這是這個 deep dive 的收口,面試官最愛讓你**端到端走一遍**:

```
A 在 gw-1,B 在 gw-7,A 發消息給 B:

1. A ──[WS]──► gw-1            A 在它的常駐連接上發 SEND
2. gw-1 ──[RPC]──► Logic       gw-1 不懂業務,轉給無狀態 Logic 層
3. Logic: 發 seq + 寫 KV       會話內單調 seq、落消息庫(conv_id 分片)
4. Logic ──查路由表──► Redis   user_B → {gw-7, conn#3391}
5. Logic ──[RPC]──► gw-7       把消息投給 B 所在的那台 Gateway
6. gw-7 ──[WS]──► B            gw-7 用 conn#3391 找到 B 的連接,推出去
7. B ──ACK──► ...──► A         B 收到回 ACK,最終回給 A 顯示「已送達」(deep dive ②)
```

> **這條鏈的精髓:Gateway 之間不直接通信,全靠「路由表 + 內部 RPC」中轉。** A 不需要知道 B 在哪,Logic 查路由表後幫它定位。**這就是「有狀態接入層 + 無狀態邏輯層 + 軟狀態路由表」三件套的協作**——把它清晰講出來,長連接這個 deep dive 就滿分。

### Deep Dive ② · 消息可靠投遞與有序(不丟 / 有序 / 不重 + 群扇出)

澄清裡埋的雷:「消息不能丟」「單會話有序」。這段拆四個小問:**怎麼保證不丟、怎麼保證有序、怎麼去重、群消息扇出用寫擴散還是讀擴散。**

#### (a) 不丟:應用層 ACK + at-least-once

**為什麼 TCP 的可靠不夠?** TCP 只保證「字節到了對方內核緩衝區」,**不保證 App 處理了**——消息到了 Gateway 緩衝區但 App 還沒寫庫就崩了、或推到 B 手機但 App 還沒渲染就被殺,TCP 都認為「送到了」。所以 IM 必須做**應用層 ACK**(端到端確認),不能依賴傳輸層:

```
A 發送的可靠性(上行):
  A ──SEND──► Server,Server 落庫成功後回 serverAck
  A 沒收到 serverAck → 超時重發(帶同一個 clientMsgId)→ 直到收到 ack

B 接收的可靠性(下行):
  Server ──PUSH──► B,B 收到並落地後回 ACK
  Server 沒收到 B 的 ACK → 認為沒送達 → 重推(B 上線時靠 PULL 補)
```

> **這就是 at-least-once(至少一次):寧可重複,絕不丟失。** 每個環節「沒收到確認就重試」,代價是**消息可能重複**(serverAck 丟了 → A 以為沒成功 → 重發)。重複用下面的去重解決。**選 at-least-once 而不是 at-most-once,是因為 IM 場景「丟消息」比「偶爾重複」嚴重得多**——這是個明確的 tradeoff 決策。

#### (b) 有序:會話內單調 seq(不是全局有序)

**注意:是「會話內有序」,不是「全局有序」。** 全局有序(所有消息一個全局時鐘排)代價極高(全局發號器成瓶頸)且**沒必要**——用戶只關心「同一個會話裡消息順序對」:

| 方案 | 怎麼做 | 取捨 |
|---|---|---|
| **客戶端時間戳排序** | 用消息的客戶端時間排 | ❌ 客戶端時鐘不可信(能改、會偏),且多端時鐘不一致 |
| **服務端落庫時間** | 用 server 落庫的 ms 時間戳 | △ 同一毫秒多條會撞、分散式下不同機器時鐘有偏差 |
| **會話內單調 seq(主力)** | 每個會話維護一個**單調遞增計數器**,消息落庫時分配 seq | ✅ 嚴格單調、無歧義、增量拉的天然游標;在 `conv_id` 分片內生成,單片內單調**不用跨片協調** |

> **seq 怎麼發(黑盒)?** 因為消息**按 `conv_id` 分片**,同一會話的消息都落同一個分片,所以 seq 可以在**會話粒度上用 Redis `INCR conv:{id}:seq`** 或在該分片的單點上原子遞增——**不需要全局發號器**。這是「按會話分片」這個決策的紅利:**有序問題被局部化了**。(對比全局有序要 Snowflake/全局協調,錨 `05 分散式 ID`。)

> **面試話術:** 「我要的是會話內有序,不是全局有序——用戶只在乎同一個會話消息順序對。每個會話一個單調 seq,因為消息按 conv_id 分片,seq 在會話粒度 `INCR` 就行,不用全局發號器。客戶端按 seq 排序展示,seq 還順便當增量拉的游標和去重鍵,一石三鳥。」

#### (c) 去重:客戶端用 seq / clientMsgId 冪等

at-least-once 會帶來重複,去重在**兩端各做一道**:

- **接收端(B)去重:** B 本地記「這個會話已收到的最大 seq」,**收到 seq ≤ 已收到的就丟棄**(重推的舊消息)。seq 單調讓去重變成簡單的數值比較。
- **發送端冪等(防 A 重發造成兩條):** A 重發帶**同一個 `clientMsgId`**,服務端用 `clientMsgId` 做冪等鍵——「同一個 clientMsgId 已經處理過 → 直接回上次的 serverAck,不再插一條」。

> **at-least-once + 客戶端去重 = 業務上的 exactly-once 體感。** 這是分散式系統的經典套路(`financial-consistency` 的冪等同源):**傳輸層做 at-least-once(簡單可靠),冪等性下推到應用層用業務鍵去重**,合起來等效 exactly-once,且比真正的分散式 exactly-once 便宜得多。

#### (d) 群消息:寫擴散 vs 讀擴散(和 News Feed 同母題,但答案不同)

群消息一條要送 N 個人,和 `08 News Feed` 的扇出**是同一個母問題**,但 IM 的最優解和 feed 不同:

| 方案 | 怎麼做 | 優點 | 致命傷 |
|---|---|---|---|
| **寫擴散(push / 寫每個人收件箱)** | 群消息發出時,**給每個成員的收件箱各寫一份**(各自有 seq) | 讀極快(每人讀自己的收件箱);離線補償簡單(收件箱就是離線隊列) | **寫放大 N 倍**;500 人群一條消息寫 500 次;萬人群直接爆;存儲 N 倍 |
| **讀擴散(pull / 只存一份)** | 群消息**只在會話裡存一份**,每個成員讀時自己去拉群會話 | 寫只 1 次,存儲省;超大群友好 | 讀放大(每人都來拉同一個會話);**未讀數/已讀回執難算**;在線推送還是得逐個推 |
| **混合(IM 的實戰答案)** | **小群寫擴散**(成員少、寫放大可接受,換讀快);**超大群讀擴散**(只存一份,成員按需拉) | 兩邊的好處 | 系統要維護兩套路徑、按群大小切換 |

> **IM 和 News Feed 的關鍵差別(主動暴露,顯深度):** News Feed 是「大 V 走讀擴散」因為**粉絲量級懸殊**(千萬粉);IM 群的切換閾值是**群人數**(比如 >1000 人轉讀擴散)。而且 IM 多一個約束:**消息要即時推**——無論寫/讀擴散,**在線成員的實時推送都得逐個查路由表 + 推**(這部分省不掉),寫/讀擴散只影響「存幾份 + 離線怎麼補」。所以 IM 的扇出成本拆成兩塊:**實時推送(逐個推,跑不掉)** + **持久化/收件箱(寫擴散還是讀擴散,可選)**。

> **面試話術:** 「群消息和 News Feed 同一個母問題,但我的答案是**按群人數混合**:小群寫擴散(各寫收件箱,讀快、離線補償簡單),超大群轉讀擴散(只存一份,避免寫放大爆炸)。但 IM 比 feed 多一層——**在線成員的實時推送無論哪種都得逐個查路由表推**,這部分省不掉,寫/讀擴散只決定『離線消息存幾份、怎麼補』。」

#### (e) 離線消息怎麼補

B 離線時消息進了它的離線隊列(寫擴散下就是收件箱),B 上線後:

```
B 上線 → CONNECT → 客戶端帶上「我各會話本地最大 seq」
       → PULL 增量拉:對每個會話 PULL(convId, sinceSeq=本地最大, limit=N)
       → 服務端返回 seq > sinceSeq 的消息 → B 補齊 → 更新本地最大 seq
```

> **這就是 seq 的第三個用途(排序、去重、增量拉游標三合一)。** 離線補償不需要單獨的「離線隊列」數據結構——**收件箱本身 + seq 游標就是離線隊列**:B 記得自己讀到哪(sinceSeq),上線後拉「沒讀到的部分」。**APNs/FCM 只負責「喚醒 B 的手機彈個通知」,真正的消息補齊靠 B 上線後自己 PULL**——這個分工要講清(系統推送通道不可靠且有限額,不能指望它送正文)。

---

### Deep Dive ③ · 在線狀態與多端同步(presence + 多設備 + 已讀回執成本)

澄清埋的雷:「在線狀態可最終一致」「多端同步」「已讀回執」。這三個合在一起是 IM 的「隱性成本中心」——**看著是小功能,扇出成本能拖垮系統。**

#### (a) Presence(在線狀態)怎麼維護

```
presence(Redis):  user_id → online,TTL=45s(略大於心跳間隔)
  用戶連上 / 每次心跳 → SET user:{id}:presence online EX 45
  TTL 到期沒續 → 自動變「離線」(不需要顯式下線事件)
```

| 設計點 | 怎麼做 | 為什麼 |
|---|---|---|
| **靠 TTL 而非顯式下線** | presence 是 `online + TTL`,心跳續期;不續就自動過期成離線 | 顯式下線事件**不可靠**(手機沒網直接斷,發不出「我下線了」)——**TTL 兜底是 presence 的核心技巧** |
| **最終一致即可** | 別人看到你「在線」晚幾秒、或你掉線後幾十秒才顯示離線,**完全可接受** | 澄清裡定的:presence 不需要強一致,這給了用便宜 TTL 方案的底氣 |
| **訂閱推送(可選)** | 好友上下線要不要實時推給對方?| **這是扇出陷阱**:一個大 V 上線要通知千萬「關注者」→ presence 扇出爆炸。**解法:不推,改為「打開聊天窗口時按需查」**——把推(push)改成拉(pull),扇出歸零 |

> **面試話術:** 「在線狀態我用 Redis + TTL:心跳續期,不續就自動過期成離線——因為顯式下線不可靠(沒網直接斷)。presence 最終一致就行,晚幾秒沒人死。**陷阱是『好友上下線實時通知』——大 V 上線要扇出給千萬人,會爆。我的解法是改推為拉:不主動廣播上下線,用戶打開某個聊天窗口時再查對方在不在線。**」

#### (b) 多端同步:一個賬號多設備

多端是 IM 的硬需求(手機 + 電腦同時在線),它把前面所有設計**乘以設備數**:

| 問題 | 怎麼解 |
|---|---|
| **路由表變集合** | `user → {gw-7:phone, gw-3:pc}`,一條消息要推給**所有在線設備** |
| **每設備獨立 seq 游標** | 手機讀到 seq=100、電腦讀到 seq=80 → **各設備記自己的 `sinceSeq`**,各自 PULL 補齊。**不能用一個全局「已讀」游標**(否則電腦會漏掉手機已讀的) |
| **發送方自己的多端同步** | A 用手機發的消息,A 的電腦也要顯示 → A 發消息時,**也要推一份給 A 自己的其他設備**(自己給自己扇出) |
| **已讀狀態多端一致** | 手機把消息標已讀,電腦的紅點也要消 → 已讀回執不只發給對方,**也同步給自己的其他端** |

> **多端同步的本質:把「用戶」當成「一組設備」,消息要扇出給這組設備裡的每一個,每個設備獨立維護自己的同步進度(seq 游標)。** 服務端存的是「用戶讀到哪」的權威 seq,各設備拉取時對齊。**這就是為什麼 seq 必須是服務端權威、單調、按會話的——它是多端對齊的唯一基準。**

#### (c) 已讀回執的扇出成本(大群的 N² 災難)

已讀回執在 1對1 很簡單(A 發、B 讀了,推一個「已讀」給 A)。但**在群裡是災難**:

```
1對1 已讀:B 讀了 → 推 1 個已讀給 A          ✅ 便宜
群已讀:500 人群,每人讀了都要讓"誰讀了"可見
  → 每條消息 × 每個讀者 = 一條已讀回執
  → 500 人各發 100 條、各自讀 → 已讀回執量 ≈ 消息量 × 群人數 = N² 級放大  💥
```

| 場景 | 已讀回執策略 | 取捨 |
|---|---|---|
| **1對1** | 精確已讀:推「對方已讀」+ 雙方狀態同步 | 便宜,做全套 |
| **小群(<100)** | 「已讀 N 人 / 未讀 M 人」**聚合計數**,點進去才展開誰讀了 | 把 N² 的「逐人推」壓成「一個計數」,點開才查明細 |
| **超大群 / 廣播群** | **直接不做已讀回執** | N² 成本不可承受,且產品上沒人關心萬人群誰讀了。**用產品約束砍掉技術難題** |

> **面試話術(主動暴露成本,`01` Step 7 加分核心):** 「已讀回執是個隱性成本炸彈——1對1 很便宜,但群裡是 N² 放大:500 人群每人讀每條都生成一條回執。我的策略是**分級**:1對1 做精確已讀;小群退化成『已讀 N 人』的聚合計數,點開才查明細;**超大群直接砍掉已讀功能**——這個 N² 成本不值得扛,而且產品上萬人群本來也沒人在乎誰讀了。**用產品約束去砍技術難題,比硬扛 N² 聰明。**」

> 這句「**用產品約束砍技術難題**」就是 `00` 標準線「擁有這個決策」的架構師話術——和短鏈「給 TTL 把存儲從只增改成有進有出」同一套思路。

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。IM 的架構師深度集中在**有狀態接入層的運維噩夢**:無狀態服務的那套(隨便擴、隨便殺、滾動發布)在連接層**全部失效**,這是 IM 真正的隱性成本。

### 成本:連接是常駐成本,不是按量付費

回到第 2 步的結論——IM 的成本結構和前面題**根本不同**:

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **接入機(連接)** | ~200 台常駐握連接,**和消息量無關**——哪怕半夜沒人發消息,1000 萬連接還在,200 台不能關 | **最大成本項,且是「常駐固定成本」不是「按 QPS 彈性」**。連接占的是內存/FD,不是 CPU——夜間 CPU 閒著但機器不能縮容 |
| **下行帶寬** | 扇出放大:180 萬推送/s × 200B ≈ 360 MB/s + 協議開銷 | 群扇出是帶寬放大器;媒體消息走對象存儲 + CDN,別過長連接傳 |
| **存儲** | PB 級且只增 | 冷熱分離:近 7 天熱消息留快存儲 + 緩存,老消息下沉冷存儲;法規允許的話老消息可過期 |
| **MQ/計算** | 6 萬 QPS 的邏輯層,幾十台 | 相對便宜,無狀態可彈性 |

> **架構師 insight:** 「IM 的成本反直覺——**最貴的不是處理消息(QPS 才 6 萬),是常駐握住 1000 萬連接的 200 台機器**。而且這 200 台**不能按 QPS 縮容**:夜間沒人發消息,但連接還在,機器關了連接就斷了。連接是**有狀態的常駐成本**,這是 IM 和無狀態服務最大的成本差異。優化方向是**提升單機連接密度**(從 5 萬/台壓到 10 萬+,直接砍一半機器)——這就是為什麼大廠死磕接入層的單機連接數。」

### 容量規劃:單機連接密度是核心指標

IM 的容量規劃不看 QPS,看**單機能扛多少連接**——這個數越高,機器越少,成本越低:

| 瓶頸 | 限制 | 怎麼突破 |
|---|---|---|
| **文件描述符(FD)** | 每連接一個 FD,默認 `ulimit` 1024 太低 | 調 `ulimit -n` 到百萬;內核 `fs.file-max` 配套調 |
| **連接內存** | 每連接讀寫緩衝 + 連接對象,幾 KB ~ 幾十 KB | 調小默認緩衝區、用內存池;Go/Netty 的連接對象精簡 |
| **心跳/定時器調度** | 千萬連接 = 千萬定時器 | **時間輪(timing wheel)** O(1) 管理,不一連接一 Timer |
| **線程模型** | 一連接一線程 = 千萬線程崩內核 | **事件驅動**(epoll/io_uring),少量線程管海量連接(Netty/Go netpoller/C10M) |

> **容量規劃話術:** 「我的容量單位是『單機連接數』不是 QPS。1000 萬連接 ÷ 單機 X = 機器數,所以提升 X 直接省錢。瓶頸依次是 FD(調 ulimit 到百萬)、連接內存(調小緩衝 + 內存池)、定時器(時間輪)、線程模型(事件驅動,C10M/C10K 問題)。我會給每台預留 headroom——不按 5 萬塞滿,留餘量應對**重連風暴**(下面失效模式講)。」

### 演進路徑:從 MVP 到百萬連接

```
階段 1(MVP):單體 + WebSocket,連接和邏輯在一個進程
   幾萬用戶,一台機器握住所有連接,簡單夠用,別過度設計
        │  連接數 / 用戶量上來,單機握不住
        ▼
階段 2:拆出獨立接入層(Gateway 有狀態)+ 無狀態 Logic 層 + 路由表(Redis)
   連接和業務解耦,各自水平擴;這是本章主架構
        │  扇出 / 群消息 / 多端壓力上來
        ▼
階段 3:消息走 MQ 削峰 + 寫/讀擴散混合 + 多端同步 + presence 優化
   扇出放大用 Kafka 緩衝,大群轉讀擴散,已讀回執分級
        │  跨地域用戶、就近接入低延遲
        ▼
階段 4:多地域接入(就近連 Gateway)+ 消息跨地域同步
   用戶連最近的接入點;跨地域消息走專線同步,presence/路由表跨地域協調
```

> **這條演進的關鍵 insight:** IM **比短鏈更該「分階段」**——因為有狀態接入層的複雜度極高,MVP **絕不該**一上來就拆 Gateway/Logic/路由表三層。**「先單體握連接,連接數撐不住再拆接入層」** 比一上來堆滿組件顯資深得多(`01` 紅旗「上來就拆微服務」)。階段躍遷的觸發信號很明確:**單機連接握不住 → 拆接入層**。

### 失效模式(`01` Step 7:IM 特有的「連接洪峰」和「有狀態下線」)

IM 的失效模式和無狀態服務**根本不同**——它有兩個前面題都沒有的、極其要命的失效:

| 失效 | 影響 | 緩解 |
|---|---|---|
| **重連風暴(thundering herd)** | 一台 Gateway 掛 → 上面 5 萬連接**同時重連**到其他機器 → 其他機器瞬間連接洪峰 → 可能連環崩(雪崩) | **客戶端重連加隨機退避抖動**(別都立刻重連);接入層**限流 + 預留 headroom** 吸收洪峰;LB 把重連打散到多機。連接洪峰/降級錨 `system-design/01 韌性-依賴掛了怎麼不崩` |
| **有狀態下線(發布/縮容)** | Gateway 是有狀態的——**直接殺進程 = 5 萬人瞬間掉線 + 重連風暴**;不能像無狀態服務那樣滾動殺 | **連接優雅遷移**:發布前先**停止接受新連接**、通知存量連接**主動重連到別處**、等連接自然遷移完再下線。這是 IM 發布的核心難題,錨 `distribution/优雅下线.md` |
| **路由表(Redis)掛** | 找不到用戶連在哪,新消息暫時投不出去 | 路由表是**軟狀態**(真相在連接),可降級:先寫離線隊列,靠重連 PULL 補;Redis 恢復後 Gateway 重登記(deep dive ①(b)) |
| **單會話熱點(超大群)** | 萬人群的 conv_id 是熱點分片,所有人往一個會話寫 + 扇出 | 超大群**轉讀擴散**(只存一份)+ **熱會話獨立分片/打散** + 扇出走 MQ 削峰 |
| **離線推送通道(APNs/FCM)限流** | 第三方推送有限額、可能延遲/丟 | APNs/FCM **只負責喚醒彈通知,不送正文**;正文靠上線 PULL,通道不可靠不影響消息完整性(deep dive ②(e)) |

> **架構師話術(IM 的招牌難題):** 「IM 最要命的兩個失效是無狀態服務沒有的:**重連風暴**和**有狀態下線**。一台接入機掛,5 萬連接同時重連砸向別的機器,可能連環雪崩——解法是客戶端重連加隨機退避、接入層限流、預留 headroom 吸洪峰。發布更麻煩——Gateway 有狀態,直接殺進程就是 5 萬人掉線,所以要**連接優雅遷移**:先停新連接、通知存量主動遷走、等遷完再下線。**這兩個是 IM 區別於 CRUD 系統的本質難點,也是我設計接入層時最操心的地方。**」(韌性錨 `system-design/01`,優雅下線錨 `distribution/优雅下线.md`)

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **低頻通知 / 系統推送**(不是雙向聊天) | 別上長連接接入層 + 路由表 | 那是 `10 通知系統`——單向 fan-out,可用 APNs/FCM/輪詢,不需要握 1000 萬常駐連接 |
| **超大直播聊天室**(百萬人同一個房間刷彈幕) | 別用「逐個查路由表精確投遞 + 已讀回執」 | 房間級廣播 / 分組廣播 + 抽樣展示 + 不保證每條都送達(刷屏丟幾條無所謂)——是另一套「廣播」模型,不是「精確投遞」 |
| **強一致 / 不可丟一條的場景**(如金融指令) | 別用「at-least-once + 客戶端去重」的最終一致 | 那要事務 + 對賬,走 `14 支付` 的強一致路徑,IM 的可靠性等級不夠 |
| **小團隊內部工具**(幾百人) | 別上 200 台接入機 + 寫讀擴散混合 | 單機 WebSocket + 一張消息表就夠,MVP 別過度設計(演進階段 1) |

> **這張表是架構師線的精華**——「知道一個方案的邊界、知道何時它是錯的」,比「知道這個方案怎麼做」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。**IM 和通知系統、直播彈幕長得像但模型不同**(雙向精確投遞 vs 單向 fan-out vs 廣播),把這三者的邊界講清,就證明你不是在套模板,而是在 reason about 本質。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | IM 容量估算最關鍵的一句結論? | **關鍵約束是連接數不是 QPS**:消息 QPS 才 6 萬(輕鬆),但同時在線 1000 萬條長連接 ÷ 單機 5 萬 = **~200 台有狀態接入機**。連接是常駐資源,這才是真矛盾。 |
| 2 | IM 架構和前面題的本質分野? | 多一層**有狀態接入層(Gateway)**——連接黏在某台機器上。精髓是**有狀態連接層 + 無狀態邏輯層 + 軟狀態路由表(user→gateway)** 三件套;無狀態服務的「隨便擴/殺/滾動發布」在連接層失效。 |
| 3 | 一條消息怎麼從 A 的連接路由到 B 的連接? | A→gw-1→Logic(發 seq+寫 KV)→**查路由表 user_B→gw-7**→投給 gw-7→gw-7 用本地 conn_id 推給 B。Gateway 間不直連,**全靠路由表 + 內部 RPC 中轉**。 |
| 4 | 心跳為什麼必須有?間隔的 tradeoff? | 長連接會**假死**(對端切後台/NAT 超時,你這頭看著還在),往死連接推消息就黑洞。應用層心跳 30s 探活。間隔太頻(千萬連接放大)耗電耗 CPU、太疏死連接發現慢。千萬定時器要用**時間輪**。 |
| 5 | IM 怎麼保證消息不丟 + 不重 + 有序? | **不丟**:應用層 ACK(TCP 不夠)+ at-least-once(沒確認就重發)。**不重**:接收端按 seq 丟舊 + 發送端按 clientMsgId 冪等。**有序**:**會話內單調 seq**(非全局),按 conv_id 分片所以 seq 局部 INCR 即可。三者合起來=業務 exactly-once 體感。 |
| 6 | 群消息用寫擴散還是讀擴散?和 feed 啥區別? | **按群人數混合**:小群寫擴散(各寫收件箱,讀快、離線補簡單),超大群讀擴散(只存一份防寫放大爆炸)。和 feed 差別:IM 多一層**在線成員實時推送逐個查路由表推,跑不掉**,寫/讀擴散只決定「存幾份/離線怎麼補」。 |
| 7 | 在線狀態(presence)怎麼維護?最大陷阱? | Redis + **TTL**(心跳續期,不續自動過期成離線)——因顯式下線不可靠(沒網直接斷)。最終一致即可。**陷阱:好友上下線實時通知會扇出爆炸(大V上線通知千萬人)→ 改推為拉:打開窗口時才查對方在不在線**。 |
| 8 | 已讀回執為什麼是成本炸彈?怎麼解? | 1對1 便宜,但群裡 **N² 放大**(每人讀每條都生成一條回執)。分級:1對1 精確已讀;小群退化成「已讀 N 人」聚合計數;**超大群直接砍掉**——用產品約束砍技術難題。 |
| 9 | IM 特有的兩個要命失效模式? | **①重連風暴**:一台 Gateway 掛,5 萬連接同時重連砸向別機 → 雪崩(解:客戶端隨機退避 + 接入層限流 + headroom)。**②有狀態下線**:Gateway 有狀態,直接殺=5萬人掉線 → 要**連接優雅遷移**(先停新連接、通知存量遷走、再下線)。 |
| 10 | 多端同步的本質? | 把「用戶」當「一組設備」:消息扇出給每個設備,**每設備獨立維護 seq 游標**(手機讀到100/電腦讀到80,各自 PULL 補);發送方自己也要給自己其他端推一份;已讀狀態多端同步。**服務端權威單調 seq 是多端對齊的唯一基準**。 |

## 一句話總綱(背起來)

> **IM 是「連接約束」題——數字一算反直覺:消息 QPS 才 6 萬(不是事兒),真矛盾是 1000 萬並發長連接逼出的 ~200 台有狀態接入機。架構精髓是分三層:有狀態 Gateway(握連接+心跳)+ 無狀態 Logic(發 seq+業務)+ 軟狀態路由表(user→gateway),一條消息靠「查路由表 + 內部 RPC」從 A 的連接中轉到 B 的連接。可靠性靠應用層 ACK(TCP 不夠)+ at-least-once + 客戶端按 seq/clientMsgId 去重 = 業務 exactly-once;有序用會話內單調 seq(非全局,按 conv_id 分片所以局部 INCR);群扇出按人數混合寫/讀擴散,但在線實時推送逐個查路由表跑不掉。presence 用 Redis+TTL(顯式下線不可靠),上下線改推為拉防扇出爆炸;已讀回執群裡 N² 災難,分級到超大群直接砍。架構師線:連接是常駐成本不按 QPS 縮容,單機連接密度是核心指標;兩個招牌失效——重連風暴(隨機退避+限流)和有狀態下線(連接優雅遷移)。**
