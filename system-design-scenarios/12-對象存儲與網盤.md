# 12 · 對象存儲 / 網盤(Dropbox / S3)

> **T2 的「存儲側天花板」題。** 前面 News Feed / IM / 通知練的是「消息怎麼扇出給很多人」(E 包重心);這題切換軸線——它幾乎不考扇出,考的是 **海量二進制怎麼存、怎麼省、怎麼在多端之間同步**(B 包 ●● + D 包同步衝突 + F 包多副本/容災)。用 `00` 依賴表看:網盤吃 **A + B●● + D + F**,是繼支付之後**第二難的存儲題**。
>
> 這題的教學價值高度集中在三個「黑盒」:**分塊 + 內容哈希去重 + 斷點續傳**(三件事咬在一起,是這題的母題)、**metadata 與 blob 分離後的一致性與 GC**(架構師最愛追問)、**多端同步的衝突解決**(直接吃 `distr-tx/` 的版本向量)。把這三個鑽透,網盤就通了——而且這三套機制 `17 視頻點播`、`13 爬蟲`、所有「大文件 + 多副本」的題都復用。
>
> **一句話先定錨:** 網盤的設計重心**不在 QPS**(上傳寫 QPS 很低),而在「**同一份內容全局只存一次(去重)、metadata 與 blob 分兩套存儲、變更只傳增量塊、多端衝突不丟數據**」。記住這四件事,後面全是它的展開。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個文件同步 / 網盤服務(像 Dropbox / Google Drive),底層可能架在對象存儲(像 S3)上。使用者在任意一台設備上傳/修改/刪除文件,其他設備自動同步到最新;支持大文件、斷點續傳、秒傳、歷史版本。

**別一上來就畫對象存儲架構**——先用 `01` Step 1 的「問三類」把這道超大開放題收斂。網盤的功能面極廣(分享、協作、權限、預覽、全文搜索……),**澄清的第一要務是砍範圍**:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 核心用例?要不要分享/協作/權限/在線預覽/全文搜索? | 核心:**上傳、下載、多端同步、版本歷史**。**秒傳 + 斷點續傳**要(它是這題的考點)。分享/權限**做加分項**;在線預覽/全文搜索**砍掉**(那是另一類題,別陷進去) |
| **非功能(SLA)** | 一致性要多強?多端同步延遲容忍多少?可用性?數據持久性? | **數據持久性是第一優先**(丟用戶文件是災難,要 11 個 9 那種 durability);**同步延遲秒級可接受**(改完幾秒後別的設備才更新,沒人死);可用性 > 強一致;**同一文件多端並發改要有衝突策略,不能默默丟一邊** |
| **規模 / 約束** | 用戶量?人均存儲?單文件最大?讀寫比? | **5 億註冊用戶、1 億 DAU**;人均 ~10 GB;單文件上限假設 **幾十 GB**(大文件是考點);**讀(下載/同步拉)> 寫(上傳)**,但寫的數據量巨大 |

**面試話術(把假設說出來讓對方確認):**
> 「我先框範圍——核心是上傳、下載、多端同步、版本歷史四件事,**斷點續傳和秒傳我會重點做**,因為它倆是這題的技術核心。分享和權限我當加分項,在線預覽和全文搜索這次砍掉,免得發散。非功能上我把**數據持久性放第一**——丟用戶文件比短暫不可用嚴重得多;同步延遲我假設秒級可接受。規模我假設 1 億 DAU、人均 10GB、單文件上限幾十 GB。**有一個關鍵決策我想先確認:同一個文件兩台設備同時改,衝突了你希望怎麼處理——是最後寫贏(可能丟改動),還是兩個版本都保留讓用戶選?** 我傾向後者,Dropbox 就是生成一個 conflicted copy。」

> **這句主動問「衝突怎麼處理」是這題的資深信號** —— 初級會默認最後寫贏(悄悄丟數據),資深知道網盤丟用戶數據是紅線,會把這個決策**主動拋給面試官**(`00` 標準線:「擁有這個決策」)。

**這幾個澄清直接決定後面:**

- 「數據持久性第一」→ 第 6 步 blob 存儲必講**多副本 / 跨可用區 / 糾删码**(錨 `system-design/03 複製`)。
- 「秒傳 + 斷點續傳要做」→ deep dive ① 直接是這題主菜(分塊 + 內容哈希)。
- 「多端並發改不能默默丟」→ deep dive ③ 衝突解決(版本向量 / conflicted copy)。
- 「讀 > 寫但寫數據量巨大」→ 第 2 步估算的結論會是**存儲到 EB 級 + 上傳帶寬是真瓶頸**,QPS 反而不是。

**常見陷阱:** 一上來就糾結「在線 Office 預覽怎麼做」「全文搜索索引」——那是把網盤往內容平台過度設計。✅ 抓「存 + 同步 + 去重」這條主幹,5 分鐘收斂。

## 2. 容量估算(直接調用 `02` 的數字)

> 這題的估算思路 `02` 的 **Drill 3(圖片上傳 → PB 級存儲 + 寫帶寬)** 已經給了模板——網盤是它的放大版(從圖片放大到任意大文件 + 加上去重會省多少)。每步都接一句「**所以架構上……**」(`01` Step 2 鐵律)。

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 ×3、副本 ×3、單 Redis ~10萬 QPS、千兆網卡 ~125 MB/s、一行 DB 記錄 ~1KB**。

### ① 存儲總量(這題的第一矛盾)

```
5 億用戶 × 人均 10 GB = 50 億 GB = 5 EB(裸量,去重前)
× 副本 3(或纠删码 ~1.5)≈ 15 EB(三副本)/ ~7.5 EB(EC)
```
> **所以:這是 EB 級存儲——絕無可能塞進任何傳統 DB 或單一存儲。必走對象存儲(S3 那一層),按塊水平打散到成千上萬台機器。而「副本 3 vs 纠删码」直接是幾個 EB 的成本差,架構師線必算(見第 6 步)。**

### ② 去重能省多少(網盤的「省錢機關」)

網盤裡海量重複:同一個 PDF / 安裝包 / 公司共享文件被無數人存了同一份。**塊級去重後實際物理存儲遠小於裸量**:

```
假設去重率 = 物理存儲 / 邏輯存儲 ≈ 0.3~0.5(業界網盤常見)
5 EB 邏輯 × 0.4 ≈ 2 EB 物理(去重後)再 × 冗餘
```
> **所以:去重不是錦上添花,是把存儲成本砍掉一半以上的核心架構決策。這也是 deep dive ① 要鑽的——「秒傳」對用戶是體驗(瞬間完成),對公司是省下 60% 的盤。把這層「省錢」邏輯講出來,就過了架構師線。**

### ③ metadata 數量 vs blob 大小(這題的關鍵反差)

**metadata 條數巨大,但每條極小;blob 數量相對少,但每個巨大** —— 兩者數量級完全不同,這正是為什麼要分兩套存儲:

| | 量級 | 單條大小 | 總量 | 放哪 |
|---|---|---|---|---|
| **metadata**(文件樹/塊清單/版本) | 假設人均 1 萬文件 × 5 億 = **5000 億條** | ~1 KB | ~500 TB | **DB / 分片 KV**(要事務、要查詢、要分片) |
| **blob**(實際文件塊) | 塊數雖多但是「值」 | 4 MB/塊 | **EB 級** | **對象存儲**(只按 hash 點查,不需要查詢) |

> **所以:metadata 是「條數巨大、每條小、要結構化查詢與事務」→ 適合 DB 並要分片(錨 `system-design/07`);blob 是「總量 EB 級、只按 key 取整塊」→ 適合對象存儲。兩者的訪問模式根本不同,硬塞一套存儲必出事——這就是 deep dive ② 的全部理由。**

### ④ 上傳/下載帶寬(真瓶頸之一)

```
假設 1 億 DAU,人均每天上傳 100 MB:
寫入量 = 1e8 × 100 MB = 1e7 GB/天 = 10 PB/天
寫帶寬 = 10 PB / 1e5 s ≈ 100 GB/s(平均)
峰值 ×3 ≈ 300 GB/s —— 遠超任何單接入點
```
> **所以:上傳帶寬是 PB/天、百 GB/s 級——必須多接入點 + 客戶端直傳對象存儲(別讓所有字節都穿過你的應用服務器)。下載更要走 CDN 邊緣(熱門共享文件),源站只服務 CDN 回源,放大比拉滿。這直接決定第 4 步架構裡「客戶端直傳 / CDN 下載」這兩條路徑。**

### ⑤ 上傳 QPS(對比之下低到不重要)

```
1 億 DAU × 人均每天上傳 ~20 個文件 / 1e5 s ≈ 2 萬 QPS(文件級)平均
峰值 ×3 ≈ 6 萬 —— 還要再除以「分塊後一個文件是多次塊上傳」
```
> **所以:控制面(metadata 提交)的 QPS 是萬級,單純的應用層輕鬆扛。這題的難從來不在 QPS,在「字節的洪流」(帶寬 + EB 存儲)和「同步的正確性」。別把優化精力放在 QPS 上。**

### 一句話估算結論(面試一定要說出來)

> 「**這題 QPS 不是瓶頸——控制面才萬級。三個真矛盾是:(1) 存儲到 EB 級,逼出對象存儲 + 去重(去重直接砍掉一半成本);(2) 上傳/下載帶寬 PB/天、百 GB/s 級,逼出客戶端直傳對象存儲 + CDN 下載;(3) metadata 條數巨大但要查詢/事務、blob 總量巨大但只按 key 取,訪問模式根本不同,逼出 metadata 與 blob 分離。我的設計重心就放在這三處,而不是堆 QPS。**」

這句「把估算焊到三個架構決策」就是 `01` 說的資深分水嶺。

## 3. API + 數據模型

### API(`01` Step 3:控制面 / 數據面分開,這是這題的關鍵)

網盤 API 的精髓是**控制面(metadata、走應用服務器)和數據面(blob 字節、直傳對象存儲)分離**——別讓 TB 級字節穿過你的業務邏輯層:

```
# ── 控制面(metadata,走你的應用服務器)──
POST /upload/init        {fileName, fileHash, size, chunkHashes[]}
  → { uploadId, missingChunks: [2,5,7], skipUpload?: true }   # 秒傳:整文件 hash 命中 → skipUpload=true
                                                              # 斷點續傳:只回缺的塊號
POST /upload/commit      {uploadId}
  → 201 { fileId, version }                                   # 所有塊到位後,原子提交 metadata

GET  /files/{fileId}/metadata
  → { chunkList[], version, ... }                            # 拉塊清單,客戶端據此去取 blob

GET  /delta?cursor=<lastSyncCursor>                          # 增量同步:自上次以來的所有變更
  → { changes: [...], nextCursor }

# ── 數據面(blob 字節,直傳/直取對象存儲,通常走預簽名 URL)──
PUT  <presigned-url>     <chunk bytes>                       # 客戶端拿預簽名 URL 直接 PUT 到 S3,不過應用層
GET  <presigned-url / CDN-url>                               # 下載同理,走 CDN/對象存儲
```

**要點(每一條都是考點的伏筆):**
- **`/upload/init` 是這題最妙的接口** —— 客戶端先算好整文件 hash + 每塊 hash 發過來,服務器回「哪些塊我已經有了(去重)、哪些塊你上次傳到一半(斷點續傳)、整個文件我都有(秒傳 → skipUpload)」。**一個接口同時實現秒傳 + 塊級去重 + 斷點續傳**,deep dive ① 全在它身上。
- **上傳是兩階段:`init`(協商) → 傳塊 → `commit`(提交)** —— 先把 blob 塊全部寫進對象存儲,最後才原子提交 metadata。這個順序是 deep dive ② 一致性的命根子(**先寫 blob 再提交 metadata**)。
- **`/delta` 是多端同步的引擎** —— 客戶端帶上次的 cursor,服務器回增量變更。deep dive ③ 全在它身上。
- **數據面走預簽名 URL / CDN** —— 字節不穿應用層,這是第 2 步「百 GB/s 帶寬」的直接落地。

### 數據模型(`01` Step 4:選型 + 分片鍵)

**核心是「三張 metadata 表 + 一個 blob 對象存儲」**,metadata 把「文件樹 / 塊清單 / 塊去重池」拆開:

**① 文件/版本表(file_metadata)—— 用戶的文件樹與版本歷史:**

| 字段 | 類型 | 說明 |
|---|---|---|
| `file_id` | bigint **PK** | 文件唯一 ID |
| `user_id` / `namespace` | bigint | 屬於誰(**分片鍵**) |
| `path` | text | 在用戶文件樹裡的路徑 |
| `version` | int | 版本號(每次修改 +1) |
| `file_hash` | char(64) | 整文件內容 hash(秒傳用) |
| `chunk_list` | json/array | **有序的塊 hash 列表**(文件 = 塊的有序拼接) |
| `is_deleted` | bool | 軟刪除(版本歷史/回收站) |
| `updated_at` | timestamp | |

**② 塊去重池(chunk_store)—— 全局「一個塊只存一份」的引用台賬:**

| 字段 | 類型 | 說明 |
|---|---|---|
| `chunk_hash` | char(64) **PK** | 塊內容 hash(**全局唯一鍵**) |
| `blob_location` | text | 這塊在對象存儲裡的位置 |
| `size` | int | 塊大小 |
| `ref_count` | bigint | **引用計數**(多少個文件/版本引用這塊,GC 用) |

**③ blob 對象存儲 —— 實際字節:**
- key = `chunk_hash`,value = 塊的二進制。只按 hash 點取,**不需要任何查詢能力**。

**選型(決策樹錨 `system-design/06 存儲選型`):**

| 數據 | 選型 | 理由 |
|---|---|---|
| **file_metadata / chunk_store** | **分片 SQL 或分片 KV**(要二級索引 + 事務) | 要按 user_id 查文件樹、要 `commit` 時的原子提交、`ref_count` 要原子增減——需要事務與結構化查詢,KV 純點查不夠 |
| **blob(塊字節)** | **對象存儲**(S3 / MinIO / Ceph) | 純按 hash 點取、EB 級、要 11 個 9 durability——這是對象存儲的完美場景,別用 DB 存大字節(BLOB 進 DB 是經典反模式) |
| ES / 寬列 | ❌(本題範圍內) | 砍掉了全文搜索,不需要 |

> **面試話術:** 「我把存儲分兩層:metadata 走分片 SQL/KV——因為要按用戶查文件樹、要 commit 的事務、要塊的引用計數原子增減,這些 KV 純點查給不了;blob 走對象存儲——它就是『按 hash 取一坨字節』,EB 級且要極高 durability,正是 S3 這類的主場。**把大字節塞進 DB 是反模式**,DB 該存的是『指向 blob 的指針(chunk_hash)』,不是字節本身。」

**分片鍵(`01` Step 4 鐵律,錨 `system-design/07`):**

> **metadata 按 `user_id`(namespace)分片** —— 一個用戶的文件樹、版本、delta 落在同一片,**同步/列目錄都是單片查詢,無跨片**。這和短鏈按隨機 short_code 分片不同:這裡按 user 分,天然把「一個人的所有操作」聚在一起。**熱點?** 個人網盤的單用戶流量有上限(不像大 V 千萬粉),所以 user 分片**沒有大 V 熱點問題**——除非做企業共享空間(一個 namespace 被全公司狂讀),那是後話。
>
> **chunk_store 按 `chunk_hash` 分片** —— hash 高基數,天然均勻,去重查詢「這個 hash 我有沒有」是單片點查。**注意這裡有個張力:metadata 按 user 分、chunk_store 按 hash 分,兩套分片鍵不同**,所以 `commit` 跨這兩套時不是一個本地事務——這個張力 deep dive ② 會展開(引用計數的一致性)。

## 4. 高層架構(`01` Step 5:沿一個請求一鏡到底)

**先畫能跑的最簡版,把「控制面 / 數據面分離」這個骨架立起來**(`01` 鐵律:別堆組件)。

```
                            ┌─────────────────── 控制面(metadata,走應用層,QPS 萬級)
                            │
  ┌────────┐   init/commit  │  ┌──────────────┐      ┌─────────────────┐
  │ Client │───────────────►│  │ Metadata Svc │─────►│ Metadata DB     │
  │(桌面/手機│   /delta       │  │(去重判定/    │      │ (分片:user_id) │
  │ /Web)  │◄───────────────┤  │ 引用計數/    │      │ - file_metadata │
  └───┬────┘   missingChunks │  │ 版本/同步)   │      │ - chunk_store   │
      │                      │  └──────┬───────┘      └─────────────────┘
      │ ① init 回「缺哪些塊」 │         │ 查 chunk_hash 在不在
      │                      └─────────┼──────────────────────────────┐
      │                                                               │
      │ ── 數據面(blob 字節,不穿應用層,帶寬 PB/天)──                  │
      │   ② 只傳缺的塊(預簽名 URL 直傳)                                 ▼
      │   PUT chunk ───────────────────────────────────────►┌──────────────────┐
      │                                                     │  對象存儲(blob)  │
      │   ③ 下載:GET via CDN ◄──── CDN 邊緣 ◄──回源──────────│  key=chunk_hash   │
      │                                                     │  EB 級 / 多副本   │
      │                                                     │  /纠删码 / 多AZ   │
      │   ④ commit:所有塊到位 → Metadata Svc 原子提交版本     └──────────────────┘
      │
      └── ⑤ 其他設備:長輪詢/推送 收到「有變更」→ 調 /delta 拉增量 → 只下載變化的塊
```

**三條典型路徑走一遍(`01` Step 5:這題要走三條——上傳/下載/同步):**

- **上傳(寫,核心路徑):** Client 本地**把文件切塊、算每塊 hash + 整文件 hash** → 調 `init` 把 hash 清單發給 Metadata Svc → Svc 查 chunk_store:**整文件 hash 命中 → 秒傳(skipUpload),0 字節上傳**;否則回**缺的塊號**(已存在的塊去重不傳、上次傳一半的塊斷點續傳不重傳) → Client **只把缺的塊直傳對象存儲**(預簽名 URL,不穿應用層) → 全部到位後調 `commit`,Svc **原子提交 metadata + 增引用計數**。
- **下載(讀):** Client 調 `/metadata` 拿塊清單 → 按 chunk_hash **走 CDN/對象存儲取塊**(熱門共享文件命中 CDN 邊緣) → 本地按順序拼回文件。
- **同步(多端):** 設備 A `commit` 後,服務器標記該 namespace 有變更 → 設備 B 通過**長輪詢/推送**收到通知 → 調 `/delta?cursor=` 拉「自上次以來變了什麼」→ **只下載變化的那幾個塊**(增量同步,不是整文件重下)。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **客戶端分塊 + 算 hash** | 去重/秒傳/斷點續傳全靠塊 hash | ②去重省一半 + ①大文件 |
| **預簽名 URL 直傳對象存儲** | 字節不穿應用層,否則應用層帶寬爆 | ④上傳帶寬 PB/天 |
| **Metadata Svc / DB 分離出 blob** | 條數巨大要查詢的 metadata vs EB 級只取的 blob,訪問模式不同 | ③metadata vs blob 反差 |
| **chunk_store + ref_count** | 全局一個塊只存一份 + 知道何時能 GC | ②去重 |
| **CDN(下載)** | 熱門共享文件下載帶寬,源站扛不住 | ④下載帶寬 |
| **長輪詢/推送 + /delta** | 多端秒級同步,只傳增量 | 同步 SLA |

**注意這張圖的「克制」:** 控制面和數據面**乾淨分離**——應用服務器只碰小小的 metadata(萬級 QPS),海量字節走預簽名 URL 直連對象存儲、走 CDN 下載,**根本不經過你的業務邏輯**。這是這題架構的靈魂:**讓字節繞開你**。一個常見紅旗就是「所有上傳字節都 POST 到應用服務器再轉存 S3」——那你的應用層帶寬瞬間爆,EB 級流量穿過業務機是災難。

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · 分塊 + 內容哈希去重 + 斷點續傳 + 秒傳(這題的母題,四件事咬在一起)

這四個功能對用戶是四種體驗,對系統**是同一套機制的四個面**:把文件切成塊、每塊算內容 hash、用 hash 做全局去重的鍵。鑽透這一個機制,四個功能全解。

**第一步:怎麼切塊(固定 vs 內容定界,這是真考點)**

| 切法 | 怎麼做 | 優點 | 致命傷 |
|---|---|---|---|
| **固定大小分塊** | 每 4 MB 切一刀 | 簡單、塊號好對齊 | **插入位移問題**:文件開頭插入 1 個字節,後面所有塊邊界全部右移 → 每塊 hash 全變 → 去重全部失效,整文件重傳 |
| **內容定界分塊(CDC,如 Rabin 指紋)** | 用滑動窗口算指紋,**內容滿足條件處切刀** | 開頭插入只影響第一塊,**後面塊邊界不動 → hash 不變 → 仍能去重** | 實現複雜、塊大小不固定 |

> **黑盒裡發生什麼:** 固定分塊在「整文件秒傳」和「只改尾部」場景夠用,但**文檔中間插入**會讓固定分塊的去重崩掉(邊界漂移)。Dropbox 早期固定 4MB 塊,對「同步一個改了中間一行的大文件」並不省。**內容定界分塊(CDC)** 用滑動指紋讓切點跟著內容走,插入只擾動局部——這是去重率的關鍵。**面試講出「固定分塊有插入位移問題,生產用內容定界分塊」就過了資深線**,大多數人只知道固定分塊。

**第二步:塊級去重(全局同一塊只存一份)**

```
上傳一塊前:算 chunk_hash(SHA-256) → 問 chunk_store「這個 hash 在不在?」
   ├─ 在  → 不傳字節,只把 ref_count +1,metadata 引用這個既有 blob    ← 去重命中
   └─ 不在 → 傳字節到對象存儲(key=chunk_hash),chunk_store 記一條,ref_count=1
```
- **全局去重**:不是「同一用戶內去重」,是**跨所有用戶**——你和一百萬人都存了同一個 Ubuntu ISO,物理上**只有一份**(chunk_hash 全局唯一)。這就是第 2 步「省一半成本」的來源。
- **內容尋址(content-addressed)**:blob 的 key 就是它內容的 hash。**相同內容必然相同 key,不同內容(幾乎)不同 key**——這是去重的數學基礎,也是 Git/IPFS 同一套思想。

**第三步:秒傳(整文件 hash 命中)**

```
upload/init 帶整文件 file_hash:
   chunk_store / file 索引裡這個 file_hash 已存在
      → 服務器直接回 skipUpload=true,0 字節上傳,瞬間「上傳完成」
```
> 秒傳對用戶是「10GB 文件 1 秒傳完」的魔法,對系統是「`init` 階段整文件 hash 命中 → 連塊都不用協商」。**這就是為什麼 `init` 接口要同時帶整文件 hash 和每塊 hash**:整文件 hash 命中走秒傳,沒命中再用塊 hash 做塊級去重。

**第四步:斷點續傳(靠「已傳塊清單」)**

```
upload/init 帶 chunkHashes[]:
   服務器逐塊查「對象存儲裡這塊在不在 / 這個 uploadId 之前傳到哪了」
      → 回 missingChunks: [缺的塊號]
   客戶端只傳 missingChunks,已傳的(上次斷在這)跳過
```
> 斷點續傳的本質是**上傳天然冪等**——因為塊的 key 是內容 hash,**重複 PUT 同一塊是無害的(覆蓋成相同內容)**,所以「斷了重來」只需要問「哪些塊還沒到」,而不需要複雜的續傳協議。**內容尋址讓冪等免費**,這是把秒傳/去重/續傳統一起來的關鍵 insight。

**安全副題(會被追問):hash 碰撞 / hash 安全性**
- 用 **SHA-256** 而非 MD5/SHA-1(後兩者已被證明可構造碰撞,**攻擊者可故意造一個和你某塊同 hash 但內容不同的塊**,污染去重池 → 別人下載到錯數據)。內容尋址系統的 hash 必須抗碰撞,這是安全考量不只是概率。
- **跨用戶去重的隱私洩漏**:如果「秒傳」基於全局 hash,攻擊者可以**拿一個文件的 hash 試 init,根據是否秒傳推斷「這個文件是否已存在於系統中」**(side-channel)。生產上對敏感場景會**限制跨用戶秒傳**或加 per-user 鹽。講出這個就是架構師細節。

> **面試話術(把四件事串成一個機制):** 「分塊、去重、斷點續傳、秒傳其實是**一套機制的四個面**:把文件切塊、每塊算 SHA-256、用 hash 做全局去重鍵。整文件 hash 命中就**秒傳**;塊 hash 命中就**去重不傳**;沒傳完的塊靠 init 回 missingChunks **斷點續傳**——而因為 key 是內容 hash,**上傳天然冪等,續傳幾乎免費**。切塊我會用**內容定界分塊(CDC)** 而非固定分塊,否則文件中間插入會讓塊邊界漂移、去重失效。hash 用 SHA-256 防碰撞污染,跨用戶秒傳要注意隱私側信道。**發號器之於短鏈,內容尋址之於網盤——是這題的母機制。**」

### Deep Dive ② · metadata 與 blob 分離後的一致性與塊 GC(架構師最愛追問)

第 2 步算出 metadata 和 blob 訪問模式根本不同,所以分兩套存。**但一分兩套,就引入兩個經典難題:跨兩套存儲的一致性、以及去重後塊什麼時候能刪。**

**難題 A:寫順序——先寫 blob 再提交 metadata(絕不能反)**

```
✅ 正確順序:
   1. 先把所有 blob 塊寫進對象存儲(可能耗時,大文件)
   2. 全部成功後,才在 metadata 裡 commit(原子地引用這些塊 + 加 ref_count)

❌ 反過來(先提交 metadata 再寫 blob):
   metadata 說「文件 X 由塊 a,b,c 組成」但 blob 還沒寫完
   → 別的設備同步看到 X 存在 → 去下載塊 → 塊不存在 → 下載失敗,文件損壞
```
> **黑盒裡發生什麼——這是「指針不能先於數據可見」的通用原則:** metadata 是指向 blob 的指針。**指針一旦對外可見(commit),它指向的數據必須已經在那了**。所以順序鐵律是「**先寫數據(blob),最後翻轉指針(metadata commit)**」。commit 那一步是這個文件「對外可見」的原子點——commit 之前傳了一半的塊都是「孤兒塊」(暫時沒人引用),不影響任何人。**這和『先寫 binlog 再可見』『先落盤再返回 ack』是同一個模式**。

**這帶來的好處:斷點續傳/失敗回滾是免費的** —— 因為 metadata 沒 commit 之前,所有已傳的塊都不被任何文件引用,**上傳失敗 = 什麼都不用回滾,那些孤兒塊交給 GC 慢慢清**。沒有「寫了一半要回滾」的分布式事務難題。

**難題 B:引用計數與塊 GC(去重的代價)**

去重的甜頭是「一個塊只存一份」,代價是「**這塊到底還有沒有人用、什麼時候能刪**」變成難題——因為它可能被成千上萬個文件/版本引用。

| 方案 | 機制 | 取捨 |
|---|---|---|
| **引用計數(ref_count)** | 每次有文件引用這塊 +1,刪文件/版本 -1,減到 0 可回收 | 直觀,但**計數的原子性是難點**——commit 增、刪除減,並發下計數錯了就是「還有人用卻被刪」(數據丟失)或「沒人用卻不刪」(漏存儲) |
| **標記-清除 GC(mark-sweep)** | 不維護精確計數,**定期掃所有 metadata,標記「還被引用的塊」,沒被標記的清掉** | 不怕計數漂移(每輪重新算),但**掃 EB 級全量貴**;且要小心「掃的同時有新引用進來」的競態(類似 GC 的並發標記) |

> **黑盒裡發生什麼——為什麼 GC 這麼難:** 問題出在 ②的張力——**metadata 按 user 分片,chunk_store 按 hash 分片**,一個塊的引用散落在不同 user 分片裡。所以「這個塊還有沒有人用」不是一次本地查詢,要跨分片彙總。生產上常用**保守 GC**:寧可晚刪、不可錯刪(刪了還在用的塊 = 用戶文件損壞,是紅線)。具體做法是「**ref_count 做快速路徑,定期 mark-sweep 做兜底校正**」——計數可能漂移,但週期性全量掃描會把它拉回正確。**而且 GC 通常設『延遲刪除』窗口(減到 0 後不立即刪,等 N 天)**,給「並發又有人引用了同一個 hash(去重命中)」留緩衝。

> **這層『去重省了存儲、但 GC 變難了』就是架構師線的精華** —— 初級只會說「去重很好省空間」,資深知道**去重把「刪文件」從本地操作變成了「跨分片引用計數 + 分布式 GC」這道難題**,並能說出「保守刪除 + ref_count 快路徑 + mark-sweep 兜底 + 延遲刪除窗口」這套組合拳。

> **面試話術:** 「分兩套存儲我守兩條鐵律:寫**先 blob 再 commit metadata**——指針不能先於數據可見,否則別人同步到一個下載不了的文件;這也讓失敗回滾免費,沒 commit 的塊就是孤兒塊交給 GC。第二條是**去重後的塊 GC**——一個塊被很多文件引用,我用引用計數做快路徑,但計數在跨分片並發下會漂移,所以加**定期 mark-sweep 兜底 + 延遲刪除窗口**,寧可晚刪不可錯刪——刪了還在用的塊就是用戶文件損壞,這是紅線。」

### Deep Dive ③ · 多端文件同步與衝突解決(吃 `distr-tx/` 的版本向量)

多端同步是這題區別於「純對象存儲(S3)」的地方——S3 沒有「客戶端」「同步」概念,網盤有。核心兩件事:**怎麼高效拉增量、並發改了怎麼不丟數據。**

**機制 A:增量同步(只傳變化的塊)**

```
每個 namespace 維護一個單調遞增的 sync cursor(類似 binlog 位點)
設備 B 帶上次的 cursor 調 /delta:
   服務器回 [cursor 之後該 namespace 的所有變更]
   設備 B 對每個變更的文件:對比本地塊清單 vs 服務器塊清單
      → 只下載「塊 hash 不同」的那幾塊,沒變的塊本地已有,不下載
```
> 增量同步在**兩個層次**省流量:文件級(delta 只告訴你哪些文件變了)+ 塊級(一個改了一行的大文件,只有那幾個塊的 hash 變了,**只下載那幾塊**)。改 10GB 文件裡的一行,可能只傳 4MB——這又是 deep dive ① 內容尋址的紅利。**變更怎麼從服務器推到設備**:長輪詢 / WebSocket / 服務端推送(這塊機制錨 `09 IM` 的長連接,別在這重講)。

**機制 B:衝突檢測與解決(澄清裡埋的雷,這裡引爆)**

兩台設備離線時都改了同一個文件,上線後同時 commit,**誰贏?**

| 策略 | 機制 | 取捨 | 適用 |
|---|---|---|---|
| **最後寫贏(LWW)** | 比時間戳,晚的覆蓋早的 | 簡單,但**默默丟掉一邊的改動** | 不能用於網盤主文件——丟用戶數據是紅線。可用於「無所謂的元數據」 |
| **版本向量 / 版本號檢測** | commit 帶「我基於哪個版本改的」(base version),服務器發現「你基於 v3 改,但現在已經是 v5」→ **檢測到衝突** | 能**檢測**衝突(不默默覆蓋),但檢測到之後還要決定怎麼合 | 網盤主力:先用它**檢測** |
| **保留雙版本(conflicted copy)** | 檢測到衝突 → **兩個版本都留**,生成 `file (conflicted copy from 設備A).docx`,讓用戶自己決定 | **不丟任何數據**(網盤鐵律),代價是用戶要手動處理 | Dropbox 的選擇,網盤的標準答案 |
| **CRDT / OT 自動合併** | 字符級自動合併(像協同編輯) | 能無感合併,但只適用於**結構化可合併內容**(文本),二進制文件(圖片/zip)無法字符級合併 | 那是 `19 協同編輯` 的題,網盤的通用二進制文件用不了 |

> **黑盒裡發生什麼——為什麼網盤選「檢測 + conflicted copy」而非自動合併:** 協同編輯(Google Docs)能用 CRDT/OT **自動合併**,是因為它的內容是**結構化文本**,「在第 3 行插入」和「在第 5 行刪除」可以數學上交換。但網盤面對的是**任意二進制文件**——一個 .psd、一個 .zip,你**無法字符級合併兩個並發版本**(合出來就是壞文件)。所以網盤的策略是:**用版本向量/版本號可靠地檢測到「這倆是並發修改、不是順序修改」,然後不自作主張合併,而是兩個都留(conflicted copy)讓人決定**。`distr-tx/` 講的版本向量(version vector)正是「檢測並發 vs 因果」的工具——A 的版本不是 B 的祖先、B 的也不是 A 的祖先 → 兩者並發 → 衝突。

> **這就是 `00` 依賴表為什麼給網盤標了 D(一致性)** —— 衝突檢測的本質是「判斷兩個寫是因果有序還是並發」,這是分布式一致性的核心問題,深礦在 `distr-tx/`。但網盤**只用它的「檢測」能力,不用「自動合併」**——合併留給協同編輯題。

> **面試話術:** 「同步我分兩層:增量同步靠 namespace 的 sync cursor + 塊級 diff,改一行只傳一個塊。衝突我**絕不用最後寫贏**——那會默默丟用戶數據,是網盤紅線。我用**版本向量檢測並發修改**:commit 帶 base version,服務器發現你基於的版本已經被別人推進過,就判定為衝突;檢測到之後**不自動合併**(二進制文件沒法字符級合),而是**兩個版本都保留,生成 conflicted copy 讓用戶決定**——Dropbox 就是這麼做的。自動合併是協同編輯那道題(CRDT/OT)的事,文本才行,通用文件不行。版本向量的細節錨 `distr-tx`。」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:存儲是這題的全部成本,而成本由「副本策略 + 去重」決定

回到第 2 步——這題不花在 QPS(萬級),全花在 EB 級存儲和 PB/天帶寬上。架構師要算的兩個大槓桿:

| 槓桿 | 賬 | 架構師決策 |
|---|---|---|
| **副本 vs 纠删码** | 三副本 = 200% 額外開銷;纠删码(EC,如 10+4)= 40% 額外開銷,**EB 級下這是天文數字的成本差** | **熱數據三副本(讀快),冷數據降級到纠删码**(省 5 倍冗餘成本)。錨 `system-design/03 複製` |
| **去重率** | 去重把物理存儲砍到邏輯的 30~50% | 去重是**省一半盤**的核心機關,但代價是 GC 複雜度(deep dive ②) |
| **冷熱分層** | 90% 文件上傳後幾乎不再讀(長尾) | 熱層 SSD、溫層 HDD、冷層歸檔(類 S3 Glacier),按訪問頻率自動下沉 |

> **副本 vs 纠删码的取捨(架構師必答):** 三副本讀快(任一副本直接讀)、重建快,但 200% 開銷;纠删码省到 40% 開銷,但**讀要拼分片、重建要算解碼(CPU 貴、重建慢)**。所以**熱數據用副本換讀延遲,冷數據用 EC 換存儲成本**——這是用「訪問頻率」換「冗餘成本」的分層決策,EB 級下省的是真金白銀。

### 容量規劃:帶寬與接入點,不是存儲容量

存儲容量對象存儲層自己會水平擴(加機器);真正要規劃的是**上傳/下載帶寬的接入點**:

> 第 2 步算出上傳峰值 ~300 GB/s。**單接入點(千兆網卡 125 MB/s)根本不可能**——這就是為什麼必須「客戶端直傳對象存儲(預簽名 URL)」,讓字節繞開應用層、由對象存儲的多接入點群承接;下載靠 CDN 把放大比從「百萬用戶→源站」壓成「百萬用戶→CDN 邊緣→少量回源」。**架構師算的是『字節從哪進、從哪出』,不是『應用服務器要幾台』**——應用層只碰 metadata,幾乎不消耗帶寬。

### 演進路徑:從 MVP 到 EB 級

```
階段 1(MVP):整文件上傳 + 單對象存儲 + 單庫 metadata
   先跑通「上傳/下載/基本同步」,不分塊不去重——別過度設計
        │  存儲成本扛不住 + 大文件斷網要重傳
        ▼
階段 2:引入分塊 + 內容哈希去重 + 斷點續傳
   塊級去重砍存儲成本,斷點續傳解大文件;metadata 分片(按 user)
        │  跨用戶共享文件下載熱、多地域用戶延遲高
        ▼
階段 3:CDN 下載加速 + 冷熱分層 + 纠删码冷存儲 + 多地域
   熱文件 CDN 邊緣、冷文件 EC 歸檔、metadata 多地域就近
```

> **這條演進的關鍵 insight:** 網盤**一開始可以不分塊不去重**(整文件存對象存儲就能跑),但**存儲成本會逼你上去重、大文件會逼你上分塊+斷點續傳**。`01` 紅旗「上來就堆滿組件」的反面教材就是在網盤這種題上第一天就上 CDC + 纠删码 + 多地域。**講「MVP 整文件直存,等成本/大文件痛了再上分塊去重」比一上來堆滿機制更顯資深**——你展示的是「知道每個機制解什麼痛、按痛點演進」。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **對象存儲某 AZ/節點掛** | 部分塊讀不到 → 下載失敗 | **多 AZ 多副本 / 纠删码跨機架**:這正是「數據持久性第一」要付的代價;單塊丟失能從副本/EC 分片重建。錨 `system-design/03` |
| **metadata DB 某分片掛** | 該分片用戶**完全無法操作**(連目錄都列不出),但 blob 還在 | metadata 是控制面命門,要主從/多副本 + 快速故障轉移;blob 不受影響(分離的好處) |
| **commit 寫到一半進程崩** | 部分塊已傳、metadata 未提交 | **天然安全**:沒 commit = 文件不可見,已傳的塊是孤兒塊,GC 清掉;客戶端重試 init 會斷點續傳。這是「先 blob 後 metadata」順序的紅利 |
| **引用計數漂移** | 漏刪(浪費存儲)或錯刪(文件損壞) | 錯刪是紅線 → **保守 GC + 延遲刪除 + mark-sweep 兜底**(deep dive ②) |
| **同步衝突默默覆蓋** | 用戶數據被另一端覆蓋丟失 | **版本向量檢測 + conflicted copy**,絕不 LWW(deep dive ③) |

> **架構師話術:** 「我的兩個命門:一是 metadata 分片——它掛了用戶連目錄都列不出,所以要多副本快速切換;但好處是 blob 分離,metadata 掛了字節還在。二是 blob 的持久性——這是『數據持久性第一』的核心,靠多 AZ 多副本/纠删码,單塊壞了能重建。值得強調的是 commit 半途崩潰是**天然安全**的,因為我『先寫 blob 再 commit metadata』,沒提交的塊就是孤兒塊,不會讓任何人下載到壞文件——這個順序把『分布式事務回滾』這道難題消解掉了。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **小文件為主 / 內部知識庫附件** | 別上 CDC 分塊 + 去重池 + 分布式 GC | 整文件直存對象存儲 + 一張 metadata 表,別過度設計 |
| **需要實時協同編輯同一文件** | 別用「conflicted copy 讓用戶手動合」 | 那是 `19 協同編輯`(CRDT/OT 自動合併),網盤的衝突策略反而不適用 |
| **強合規 / 端到端加密的網盤** | 別用「全局跨用戶去重」 | E2E 加密後**每人的塊 hash 都不同(加密過),去重失效**;去重和 E2E 加密本質衝突,要選一個 |
| **純對象存儲服務(賣 S3 本身)** | 別把「同步/版本/conflicted copy」做進去 | 那些是網盤(應用層)的事;對象存儲層只管「按 key 存取一坨字節 + durability」,職責要切乾淨 |
| **媒體點播平台(存的是視頻)** | 別用「塊去重 + 多端同步」這套 | 那是 `17 視頻點播`:轉碼流水線 + 碼率自適應 + 推拉流,關注點完全不同 |

> **這張表是架構師線的精華** —— 尤其「**E2E 加密與去重本質衝突**」這條:加密後相同明文變成不同密文,內容尋址的去重前提(相同內容→相同 hash)被破壞。**知道一個核心機制(去重)在什麼前提下失效,比知道它怎麼實現高一個段位**(`00` 標準線:「畫得出」→「擁有決策」)。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 網盤的容量估算結論一句話? | **QPS 不是瓶頸**(控制面萬級)——三個真矛盾:**EB 級存儲**(逼出對象存儲 + 去重砍半成本)、**上傳/下載帶寬 PB/天百 GB/s**(逼出客戶端直傳對象存儲 + CDN)、**metadata(條多要查詢)vs blob(EB 級只取)訪問模式不同**(逼出兩者分離)。 |
| 2 | 分塊/去重/斷點續傳/秒傳是四個獨立功能嗎? | 不是——是**一套機制的四個面**:切塊 + 每塊算 SHA-256 + 用 hash 做全局去重鍵。整文件 hash 命中=**秒傳**;塊 hash 命中=**去重不傳**;init 回 missingChunks=**斷點續傳**;因 key=內容 hash,上傳**天然冪等**,續傳幾乎免費。 |
| 3 | 固定分塊 vs 內容定界分塊(CDC)? | 固定分塊有**插入位移問題**:文件開頭插一字節,後面所有塊邊界右移、hash 全變、去重失效。**CDC(Rabin 指紋)** 讓切點跟內容走,插入只擾動局部,去重率高。生產用 CDC。 |
| 4 | metadata 與 blob 為什麼分兩套存儲? | 訪問模式根本不同:**metadata** 條數巨大(5000 億)、每條小、要查詢+事務+分片→DB/KV;**blob** 總量 EB、只按 hash 取整塊、要極高 durability→對象存儲。**BLOB 進 DB 是反模式**。 |
| 5 | 為什麼必須「先寫 blob 再 commit metadata」? | metadata 是**指向 blob 的指針**,**指針一旦可見(commit),它指的數據必須已在那**,否則別人同步到一個下載不了的壞文件。好處:沒 commit 的塊是孤兒塊,**上傳失敗免回滾**,交給 GC。 |
| 6 | 去重後塊什麼時候能刪?難在哪? | 一個塊被很多文件/版本引用,刪它要知道「還有沒有人用」。**ref_count 快路徑 + mark-sweep 兜底 + 延遲刪除窗口**;難在 metadata 按 user 分、chunk_store 按 hash 分,引用跨分片彙總。**保守刪除:寧晚刪不錯刪**(錯刪=文件損壞,紅線)。 |
| 7 | 多端並發改同一文件,衝突怎麼解? | **絕不用最後寫贏**(默默丟數據,網盤紅線)。**版本向量檢測並發修改**(base version 已被推進=衝突)→ **不自動合併**(二進制沒法字符級合)→ **保留雙版本 conflicted copy 讓用戶決定**(Dropbox 做法)。自動合併是協同編輯(CRDT/OT)的事。 |
| 8 | 網盤何時**不該**套這套方案? | 小文件為主(整文件直存別上 CDC+去重)、要實時協同(走 `19` CRDT)、**E2E 加密(與全局去重本質衝突——加密後 hash 全不同)**、純對象存儲服務(別把同步/版本做進去)、視頻點播(走 `17` 轉碼推拉流)。 |

## 一句話總綱(背起來)

> **網盤是「字節洪流 + 同步正確性」題——QPS 騙局(控制面才萬級),真瓶頸是 EB 級存儲、PB/天帶寬、metadata 與 blob 的訪問模式撕裂。四個核心機制咬成一團:切塊 + 每塊內容 hash 把秒傳/去重/斷點續傳統一成「內容尋址 + 天然冪等」(切塊用 CDC 防插入位移);metadata(分片 DB,按 user)與 blob(對象存儲,按 hash)分離,守『先寫 blob 再 commit metadata』讓回滾免費、引用計數 + mark-sweep + 延遲刪除做去重後的塊 GC;多端同步靠 sync cursor 拉增量、塊級 diff 只傳變化塊,衝突用版本向量檢測 + conflicted copy 雙版本保留(絕不最後寫贏)。架構師線:熱副本/冷纠删码砍存儲成本、字節繞開應用層直傳對象存儲、E2E 加密與去重本質衝突要二選一。** 去重的細節錨 `system-design/03 複製`、衝突檢測錨 `distr-tx`。
