# 08 · 信息流 News Feed(Twitter / 朋友圈時間線)

> **T2 第一題,也是 SDI 第一高頻題。** 「設計一個 Twitter」幾乎是系統設計面試的國民題——因為它一道題同時考三件最核心的事:**讀寫比極度傾斜下的讀優化**、**消息扇出(fan-out)**、以及一個有標準答案、又能無限往下追問的經典 tradeoff——**寫擴散 vs 讀擴散**。
>
> 這題的教學價值高度集中在一個母題上:**「一個人發的東西,怎麼讓很多人看到」**。它有三種裝法(發帖時推給所有粉絲 / 看時拉所有關注者的最新帖 / 兩者混用),每種在「寫多大代價、讀多大代價、大 V 會不會爆」上各有取捨——這正是面試官要你 reason 的核心。把這個母題鑽透,後面 `09 IM` 的群消息擴散、`10 通知系統` 的 fan-out 全是它的變體(`00` 依賴表把這三題的重心都標在 **E·服務化與通信** ●●,就是這個原因:**練一題通三題**)。
>
> 用 `00` 的依賴表看:News Feed 吃 **A(框架數字感)+ B(存儲)●● + C(韌性)+ E(MQ 扇出)●●**。重心是 **B + E**——存儲(每用戶預計算 feed 放哪)和扇出(發帖怎麼推),這兩處最會被往死裡追問。

## 1. 題目 + 澄清問題清單

**題目:** 設計一個信息流系統(像 Twitter 時間線 / 微信朋友圈 / Instagram feed)。使用者可以**發帖**、**關注別人**,並能看到一條由「自己關注的所有人的帖子」聚合而成的**時間線(home timeline / feed)**。

**別一上來就畫**——先用 `01` Step 1 的「問三類」把這道故意模糊的開放題收斂成有邊界的題。`01` 的範例表用的就是這道 News Feed,這裡把它展開成完整版:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 核心用例?要不要支援哪些? | 核心:**發帖、關注/取關、看 home feed**。**先不做**:搜索、私信、廣告插入、推薦(非關注內容)、轉發/評論樹的深層展開。看的是「**純關注關係的時間線**」 |
| **非功能(SLA)** | 讀寫比?延遲?一致性?排序? | **讀 >> 寫**(~100:1);看 feed P99 要低(<200ms,在用戶刷新的關鍵路徑上);**最終一致可接受**(發帖晚幾秒出現在粉絲 feed 裡沒人死,高可用 > 強一致);排序先做**時間倒序**,算法排序算加分項 |
| **規模 / 約束** | DAU?人均讀寫?關注關係的分佈? | **2 億 DAU**;人均每天讀 feed ~100 次、發帖 ~2 次;關注關係**極度不均勻**——普通人關注幾百個,但**大 V 有千萬級粉絲**(這是這題的命門) |

**面試話術(把假設說出來讓對方確認):**
> 「我先框範圍——核心三件事:發帖、關注、看 home feed。搜索、私信、廣告、推薦這次先不做,我們聚焦在『純關注關係的時間線』。規模上我假設 2 億 DAU,讀寫比大概 100:1,feed 延遲容忍幾秒最終一致。**有一個關鍵約束我想先確認:關注關係是不是極度不均勻?有沒有千萬粉絲的大 V?** 因為這一條直接決定我整個扇出策略——這是這題最核心的 tradeoff,我待會兒會重點講。」

**這幾個澄清直接決定後面**:

- 「**讀 >> 寫(100:1)**」→ 第 5 步架構**重心在讀路徑**:home feed 必須**預計算 + 緩存**,絕不能看一次 feed 就現場去 join 幾百個關注者的帖子。
- 「**最終一致可接受**」→ 扇出可以**異步**(發帖先返回,後台慢慢推給粉絲),這是「寫擴散」能成立的前提。
- 「**大 V 千萬粉絲**」→ 第 5/6 步的核心矛盾:大 V 發一帖要扇出千萬次 → **純寫擴散會爆** → deep dive ① 的混合方案。**這一條是整題的題眼,澄清階段就要主動點出來。**
- 「**時間倒序起步**」→ deep dive ③ 先講純時間序怎麼有序插入/歸併,算法排序作為演進。

**常見陷阱:** 一上來就糾結「要不要支援轉發的轉發的轉發」這種展開細節。✅ 抓大放小,5 分鐘內收斂到「發帖 + 關注 + 看 feed」三件事,**主動把「大 V 扇出」標記為題眼**——這一步就贏在起跑線上。

## 2. 容量估算(直接調用 `02` 的數字)

> 這題的估算 `02 容量估算數字感` 的 **Drill 1** 已經算過讀寫峰值了——這裡把它擴成完整版,並且**每步都接一句「所以架構上……」**(`01` Step 2 的鐵律)。重點不在 QPS 總量,而在**讀寫比**和**大 V 的寫放大**。

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 = 平均 ×3、副本 ×3、單 Redis ~10 萬 QPS、單 MySQL 寫 ~幾千 QPS、一條 tweet ~100–300 B。**

### ① 讀 QPS(看 feed,熱路徑)

```
2 億 DAU × 100 次/天 ÷ 1e5 s = 2e5 QPS(平均)
峰值 ×3 ≈ 60 萬 QPS
```
> **所以:讀 60 萬 QPS——對照單 Redis ~10 萬 QPS,光讀就要 ~6–10 個 Redis 分片只是起步。這個量級下,看 feed 絕不可能現場 join(現場去拉幾百個關注者的最新帖再排序),必須讀『預計算好、放在 Redis 裡』的 feed。讀路徑就是『查一個 key、拿一個已排好的 ID 列表』這麼簡單。**

### ② 寫 QPS(發帖,冷路徑)

```
2 億 DAU × 2 次/天 ÷ 1e5 s = 4000 QPS(平均)
峰值 ×3 ≈ 1.2 萬 QPS
```
> **所以:發帖本身只有 1.2 萬峰值 QPS,單看不嚇人。讀寫比 ≈ 50:1~100:1,鐵律是『重點優化讀』。但——發帖的『1.2 萬 QPS』只是寫『帖子表』的量;真正的炸彈是『發完帖之後要推給多少人的 feed』,那才是寫放大,見下。**

### ③ 寫放大(這題的真炸彈,短鏈題完全沒有的東西)

「發一帖」不是一次寫,而是**「寫一次帖子 + 把這帖 ID 推進所有粉絲的 feed」**。寫放大 = 粉絲數:

| 場景 | 一次發帖 → 多少次 feed 寫入 | 含義 |
|---|---|---|
| 普通人(~200 粉絲) | ~200 次 | 完全可接受,後台異步推一下就完了 |
| 中 V(~10 萬粉絲) | ~10 萬次 | 開始吃力,但攤到時間還行 |
| **大 V(~1000 萬粉絲)** | **~1000 萬次** | **災難** |

把它推到極端算一筆賬:假設全站有 1000 個千萬級大 V,平均每人每天發 5 帖:

```
1000 大 V × 5 帖/天 × 1000 萬粉絲 = 5e13 次 feed 寫入/天
÷ 1e5 s ≈ 5 億次/秒(平均)
```
> **所以:這就是 News Feed 的題眼——『發帖』本身才 1.2 萬 QPS,但如果無腦『發帖就推給所有粉絲』(純寫擴散),光是這 1000 個大 V 就能製造 5 億次/秒的 feed 寫入,是發帖量的 4 萬倍。這個數字直接判了『純寫擴散』死刑(對大 V 而言),逼出 deep dive ① 的混合方案。普通用戶的 200 次扇出沒問題,問題只在超級節點。**

### ④ 存儲

**帖子本身:**
```
2 億 DAU × 2 帖/天 × ~300 B(含 metadata)= ~120 GB/天(裸量)
× 副本 3 ≈ 360 GB/天 → 一年 ~130 TB
```

**feed 列表(預計算的時間線,寫擴散下的額外存儲):** 這是寫擴散「拿存儲換讀速度」的成本。每用戶 feed 只存 **帖子 ID(8 B)**,不存帖子內容(內容回查,見 deep dive ②),且**截斷只留最近 ~1000 條**:
```
2 億用戶 × 1000 條 × 8 B(只存 ID)≈ 1.6 TB
```
> **所以:帖子存儲是 TB→PB 級的長期增長(走分庫/寬列存儲);但 feed 列表因為『只存 ID + 截斷 1000 條』被壓到了 ~1.6 TB,能整個放進 Redis 集群。『只存 ID 不存內容』和『截斷』這兩個決策,是讓寫擴散在存儲上可行的關鍵——deep dive ② 專門講。**

### ⑤ 帶寬

看 feed 返回的是「一頁 ~20 條帖子」,每條 ~300 B + 媒體(圖/視頻走 CDN,不算 feed 帶寬)。文本 feed 帶寬:`60 萬 QPS × 20 條 × 300 B ≈ 3.6 GB/s`——可觀但可由多機分攤,媒體一律走 CDN(`02` Drill 3/5 的結論),不是這題的主瓶頸。

### 一句話估算結論(面試一定要說出來)

> 「**這題有兩個數字定全局:(1) 讀寫比 ~100:1、讀峰值 60 萬 QPS → home feed 必須預計算 + Redis 緩存,看 feed 只是查一個排好的 ID 列表;(2) 大 V 的寫放大——一個千萬粉大 V 發一帖 = 千萬次 feed 寫入,純寫擴散會炸到 5 億次/秒。所以我的設計重心是:讀走預計算 feed,寫走異步扇出,而扇出策略對普通用戶和大 V 必須分開——這就是混合方案。feed 列表靠『只存 ID + 截斷』壓進內存。**」

這句話直接框定了三個 deep dive。

## 3. API + 數據模型

### API(`01` Step 3:快過,點到為止)

```
POST /tweets
  body: { content, mediaIds? }
  → 201 { tweetId, createdAt }

GET  /feed?cursor=<opaque>&limit=20         # home timeline,核心讀接口
  → 200 { tweets: [...], nextCursor }       # cursor 分頁,不用 offset

POST /follow      { targetUserId }  → 200 ok
POST /unfollow    { targetUserId }  → 200 ok

GET  /users/{id}/tweets?cursor=...&limit=20  # 某人的「個人頁時間線」(user timeline,跟 home feed 不同)
```

**要點:**
- 分頁用 **cursor(游標)不用 offset**——feed 是無限下滑流,offset 翻頁在大數據量下越翻越慢,且新帖插入會讓 offset 錯位(翻到重複/漏帖)。cursor 用「最後一條的 (timestamp, tweetId)」編碼,deep dive ② 細講。
- 區分兩種時間線:**home feed**(我關注的所有人的帖子聚合,是這題的主菜)vs **user timeline**(某一個人自己發過的帖,就是按 userId 查帖子表,簡單)。面試官說「feed」通常指前者。
- 寫接口想冪等(帶 `idempotency-key` / 客戶端生成 tweetId),為重試與「重複發帖」防護鋪路(`01` Step 3 習慣)。

### 數據模型(`01` Step 4:選型 + 分片鍵)

**三組核心數據:帖子、關注關係、預計算 feed。**

**(1) 帖子表 `tweets`**

| 字段 | 類型 | 說明 |
|---|---|---|
| `tweet_id` | bigint **PK**(Snowflake) | 全局唯一、**趨勢遞增**——ID 本身帶時間序,排序的命根子(見 deep dive ③) |
| `author_id` | bigint | 誰發的 |
| `content` | text | 帖子內容(媒體存對象存儲,這裡放 mediaId) |
| `created_at` | timestamp | 建立時間 |

> 選型:海量、無複雜事務、按 ID 點查 + 按 author 範圍查 → **NoSQL 寬列(Cassandra)或分庫 MySQL**。**分片鍵 = `author_id`**:讓一個人的帖子落在一起,查「某人發過的帖」(user timeline)單片命中。代價:**大 V 的帖子集中在一個分片 = 熱點分片**(但讀帖內容可被緩存擋住,沒讀 feed 列表那麼致命)。錨 `system-design/07 數據規模化`。

**(2) 關注關係 `follows`**(社交圖,這題的隱藏難點)

關注是**有向**的(我關注你 ≠ 你關注我),且要支援**兩個方向的查詢**:

| 查詢 | 誰要用 | 訪問模式 |
|---|---|---|
| 「**我關注了誰**」(followees) | 讀擴散時拉這些人的帖、UI 顯示關注列表 | 給 `user_id` → 一批 `followee_id` |
| 「**誰關注了我**」(followers) | **寫擴散時要推給這些人** | 給 `user_id` → 一批 `follower_id` |

> 所以關注關係要**存兩份(雙向冗餘)**:一份按「關注者」分片(查我關注誰),一份按「被關注者」分片(查誰關注我)。這跟「鄰接表雙向存儲」是同一招。**分片鍵分別是 `user_id`**(各自那一側)。
>
> **大 V 的 follower 列表是個大問題**:千萬粉絲意味著「誰關注了大 V」這一條會返回千萬行——這既是寫擴散要遍歷的列表(千萬次寫的來源),也是一個熱點 key。錨 `system-design/07` 的「關注關係分片」。

**(3) 預計算 feed `home_feed`(寫擴散的產物,放 Redis)**

```
KEY   feed:{user_id}                       # 每個用戶一個 key
VALUE Redis Sorted Set / List              # 元素 = tweet_id,score = timestamp(或直接用趨勢遞增的 tweet_id 當 score)
      只存 ID,不存內容;截斷只留最近 ~1000 條
```

> 選型:**Redis ZSet**(score 排序、O(log N) 插入、ZREVRANGE 取分頁,天然有序)。**分片鍵 = `user_id`**(每個用戶的 feed 落在它的 user_id hash 到的 Redis 分片)。這就是第 2 步算的「1.6 TB 放得進 Redis」的那份數據。deep dive ② 專講它的結構與截斷。

**面試話術:**
> 「我有三組數據:帖子按 author 分片存寬列/分庫 MySQL;關注關係**雙向冗餘存兩份**——一份查『我關注誰』、一份查『誰關注我』,因為寫擴散時我要快速拿到某人的全部粉絲;最後是預計算的 home feed,放 Redis ZSet,只存 tweet ID + 截斷最近 1000 條,讓它整個能放進內存。」

## 4. 高層架構(`01` Step 5:沿一個請求一鏡到底)

**先畫能跑的最簡版,再按需加件。** News Feed 的架構天然分成**寫路徑(發帖 → 扇出)**和**讀路徑(看 feed)**兩條,讀寫高度分離——這本身就是 CQRS 的雛形(寫的是帖子,讀的是預計算 feed,兩種模型)。

```
   寫路徑(發帖 → 扇出,異步)
 Client ──POST /tweets──► API GW ──► Tweet Svc ──► 1. 寫帖子表(Cassandra/MySQL,按 author 分片)
                                          │        2. 立刻返回 tweetId(發帖者不等扇出!)
                                          ▼
                                   發「新帖事件」到 MQ(Kafka)
                                          │
                                          ▼
                                  ┌─── Fan-out Worker(扇出工人,異步)───┐
                                  │  查作者的 follower 列表               │
                                  │  ├─ 普通作者:遍歷粉絲,逐個          │
                                  │  │   ZADD feed:{follower} tweetId    │  ← 寫擴散
                                  │  └─ 大 V:跳過扇出(標記為大 V)      │  ← 改走讀擴散
                                  └──────────────────────────────────────┘


   讀路徑(看 feed,熱,60 萬 QPS)
 Client ──GET /feed?cursor──► API GW ──► Feed Svc
                                          │
                                          ├─ 1. 讀預計算 feed:ZREVRANGE feed:{me}(普通關注者的帖,已排好)
                                          │
                                          ├─ 2. 拉大 V 最新帖:對「我關注的大 V」逐個查其 user timeline   ← 讀擴散
                                          │       (大 V 數量少,~幾十個,可接受)
                                          │
                                          ├─ 3. 歸併排序(merge):把 1+2 按時間/score 合併成一條流
                                          │
                                          ├─ 4. 回查帖子內容(feed 裡只有 ID):批量 MGET 帖子緩存
                                          │
                                          └─ 5. 返回一頁 + nextCursor
```

**兩條典型路徑走一遍(`01` Step 5:讀 + 寫各一條):**

- **發帖(寫,異步扇出):** Client → GW → Tweet Service → 寫帖子表 → **立刻返回**(發帖者絕不等扇出完成,否則大 V 發帖要卡幾分鐘)→ 發事件到 Kafka → Fan-out Worker 異步消費:**普通作者**把 tweetId 推進每個粉絲的 `feed:{follower}`(寫擴散);**大 V 跳過扇出**(它的帖留在 user timeline,看的時候才拉)。
- **看 feed(讀,熱路徑):** Client → GW → Feed Service → ① `ZREVRANGE feed:{me}` 拿普通關注者推來的帖(已排好序)→ ② 對「我關注的少數大 V」現場拉他們的最新帖(讀擴散)→ ③ 把兩路**歸併排序**成一條時間線 → ④ feed 裡只有 ID,**批量回查帖子內容**(MGET 帖子緩存)→ ⑤ 返回一頁 + cursor。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **預計算 feed(Redis ZSet)** | 讀 60 萬 QPS,不可能現場 join | ①「讀走預計算」 |
| **MQ(Kafka)+ Fan-out Worker** | 發帖要立刻返回,扇出量大且可異步 | ③「寫走異步扇出」 |
| **大 V 走讀擴散分支** | 千萬粉扇出會炸 5 億次/秒 | ③「混合方案」 |
| **帖子內容緩存 + 批量回查** | feed 只存 ID,要補回內容 | ④「只存 ID 省存儲」 |

**注意這張圖的核心:** 它不是一條「直線」,而是**寫擴散和讀擴散兩種扇出在同一張圖裡共存**——這正是混合方案的形態。面試官看到你把「大 V 分支」單獨拉出來,就知道你抓到了題眼。

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · 寫擴散 vs 讀擴散 vs 混合(這題的真正考點,必鑽)

這是整道 News Feed 的核心,也是面試官最愛追問的點。一句話框定問題:**「一個人發的帖,什麼時候『推』給粉絲的 feed?」** 兩個極端 + 一個折中。

#### 三種方案的機制

| | **寫擴散(push / fan-out on write)** | **讀擴散(pull / fan-out on read)** | **混合(hybrid)** |
|---|---|---|---|
| **發帖時做什麼** | **發帖即推**:遍歷作者所有粉絲,把 tweetId 寫進每個粉絲的預計算 feed | 只寫帖子表,**不推任何人** | 普通作者寫擴散;**大 V 不推** |
| **看 feed 時做什麼** | 直接讀自己預計算好的 feed(一個 key 搞定) | **現場拉**:查我關注的所有人 → 拉他們各自的最新帖 → 歸併排序 | 讀預計算 feed(普通關注者)**+** 現場拉我關注的少數大 V → 歸併 |
| **讀性能** | ⚡ 極快(一次 ZREVRANGE) | 🐢 慢(要 join 幾百個關注者 + 排序) | ⚡ 快(主體是預計算,只補幾十個大 V) |
| **寫性能** | 🐢 慢且放大(粉絲數 = 寫次數,大 V 爆炸) | ⚡ 極快(只寫一次帖子) | ⚡ 快(普通用戶扇出小,大 V 不扇出) |
| **存儲成本** | 高(每個粉絲一份 feed 副本) | 低(不存預計算 feed) | 中(只為普通關注存 feed) |

#### 黑盒裡發生什麼——三種方案的真實取捨

**寫擴散為什麼讀快寫爆:**
- 它把「聚合」這件昂貴的事**提前到發帖時做好**——發帖時就把帖子塞進每個粉絲的「收件箱(inbox)」。看 feed 時 inbox 已經是排好序的成品,讀只要 O(1) 拿一頁。**這是空間換時間 + 寫時換讀時**的典型。
- 致命傷在**寫放大 = 粉絲數**。普通人 200 粉,推 200 次無感;但**大 V 千萬粉,發一帖要寫千萬個 inbox**(第 2 步算的 5 億次/秒就是這來的)。而且這千萬次寫**毫無收益**——大 V 的粉絲裡可能 99% 此刻根本不在線、不會刷新,你推給他們的 feed 大部分永遠不會被讀到。**寫了個寂寞。**
- 還有一個隱性炸彈:**新關注的「歷史回填」**。我剛關注一個發了 1 萬帖的人,寫擴散下我的 feed 裡沒有他的舊帖(因為他發帖時我還沒關注,沒推給我)。要嘛回填(又一波寫)、要嘛我的 feed 缺他的歷史——這是寫擴散的尷尬邊界。

**讀擴散為什麼寫省讀爆:**
- 它把聚合**推遲到看 feed 時才做**——發帖只寫一次帖子表,誰都不推。**寫成本是常數,大 V 發帖跟普通人一樣便宜**,完全沒有寫放大問題。
- 致命傷在**讀放大**。我關注 500 個人,看一次 feed 就要:查我關注誰(1 次)→ 拉 500 個人各自的最新帖(500 次查詢)→ 把這 500 路歸併排序取 top 20。**讀 60 萬 QPS × 每次 ~500 路查詢 = 3 億次/秒的後端查詢**,而且大部分是浪費(500 個人裡可能只有幾個最近發了帖,卻要全查一遍才知道)。讀路徑直接被打爆。
- 它的好處場景:**關注數少、發帖頻繁、讀不那麼頻繁**的場景(比如純大 V 廣播、新聞源)讀擴散反而合適。

**混合方案——為什麼是標準答案:**

關鍵洞察:**寫擴散爆只爆在大 V(超級節點)身上;讀擴散爆只爆在「關注了很多普通人」身上。兩者的爆點不重疊**——那就**對症下藥**:

| 用戶類型 | 用哪種 | 為什麼 |
|---|---|---|
| **普通用戶發帖**(粉絲少) | **寫擴散** | 扇出 200 次無痛,換來粉絲讀極快 |
| **大 V 發帖**(粉絲千萬) | **讀擴散** | 不推任何人,避免千萬次寫;反正粉絲看 feed 時順手拉一下 |

於是:
- **發帖**:判斷作者是不是大 V(粉絲數 > 閾值,如 10 萬)。普通作者 → Fan-out Worker 寫擴散;大 V → 跳過扇出,帖子只留在它自己的 user timeline。
- **看 feed**:讀「別人寫擴散推給我的預計算 feed」(主體)**+** 現場拉「我關注的少數大 V 的最新帖」(讀擴散)**+** 歸併排序。因為**一個人關注的大 V 數量很少**(幾個到幾十個),這個讀擴散的補拉成本完全可接受——讀放大從「拉 500 個人」降到「拉幾十個大 V」。

> **面試話術(把決策攤開,`01` Step 7 加分核心):** 「核心 tradeoff 是寫擴散還是讀擴散。純寫擴散讀極快但大 V 發帖會爆——千萬粉一發帖就是千萬次寫,而且大半粉絲根本不在線,寫了個寂寞;純讀擴散寫省但看 feed 要現場拉幾百個關注者再排序,讀路徑被打爆。**所以我用混合:普通用戶寫擴散、大 V 改讀擴散。** 因為兩者的爆點不重疊——寫擴散只爆在超級節點,讀擴散只爆在『關注很多人』。看 feed 時我讀預計算 feed(普通關注者)再現場補拉我關注的少數大 V,因為一個人關注的大 V 很少,這個補拉很便宜。**這就是 Twitter 真實在用的方案。**」

> **架構師補一刀(推過面試線):** 大 V 閾值不是拍腦袋定的——它是**「扇出成本」和「讀時補拉成本」的平衡點**。閾值定低 → 太多人走讀擴散 → 看 feed 補拉變多;閾值定高 → 大 V 扇出量還是大。而且閾值要**動態**:一個用戶從普通漲粉成大 V 的那一刻要切換策略,切換瞬間的「歷史一致性」(切換前已推出去的帖 vs 切換後不推的帖)是個真實的工程坑——這是面試官追問「那用戶漲粉跨過閾值怎麼辦」時的加分答案。

#### 一個常被忽略的邊界:在線狀態裁剪扇出

寫擴散「寫了個寂寞」的問題,還有一個更省的優化:**只給「活躍/在線」粉絲做寫擴散,沉睡用戶按需回填**。一個千萬粉大 V,此刻在線的可能只有幾十萬;給沉睡的幾百萬人推 feed 純屬浪費(他們可能幾個月不登入)。**用「最近活躍」過濾扇出目標**,能把寫放大砍掉一大截。代價是沉睡用戶登入時要**惰性回填**他的 feed(讀擴散補)。這是「在線狀態 × 扇出」的交叉優化,跟 `09 IM`、`10 通知系統` 的「離線/在線決定推不推」是同一招。

### Deep Dive ② · feed 的存儲、緩存與分頁(讀路徑的命門)

寫擴散把 feed 預計算好放 Redis——但「放 Redis」這三個字底下有一堆設計決策,每個都被第 2 步的數字逼出來。

#### (1) 只存 ID,不存內容(回查)

```
feed:{user_id} = ZSet { tweet_id → score }    ← 只有 ID + 排序分數,沒有帖子正文
```

**為什麼不直接把帖子內容也存進 feed?**

| | **feed 存完整帖子內容** | **feed 只存 tweet_id(回查)** ✅ |
|---|---|---|
| 存儲 | 2 億用戶 × 1000 條 × 300 B = **~60 TB**(且每條帖被複製進所有粉絲的 feed) | 2 億 × 1000 × 8 B = **~1.6 TB**(差 ~40 倍) |
| 帖子被改/刪 | **災難**:一條帖被刪/編輯,要去千萬個 feed 副本裡改 | **免費**:feed 裡只有 ID,內容在帖子表改一處,回查時自然拿到新值(或 404) |
| 讀一頁 | 一次 ZREVRANGE 拿到完整內容 | ZREVRANGE 拿 20 個 ID + 一次批量 MGET 回查內容(多一跳,但批量) |

> **黑盒裡發生什麼:** 「只存 ID」是用「讀時多一次批量回查」換「存儲縮 40 倍 + 帖子可變」。回查那一跳不貴——20 個 ID 一次 `MGET` 打帖子內容緩存(帖子內容本身緩存命中率極高,因為熱帖被無數人的 feed 引用)。**而帖子可改可刪這一點是決定性的**:如果 feed 裡存的是內容副本,刪一條帖要去千萬個 feed 改,根本做不到——「存 ID + 回查」讓帖子變成單一真相源(single source of truth),feed 只是「指針列表」。這跟短鏈「值不變所以緩存一致性免費」正好相反:**feed 的值會變,所以絕不存副本、只存引用。**

#### (2) feed 長度截斷(working set 不能無限長)

沒人會往下翻三年前的 feed。**每個 feed 只保留最近 ~800–1000 條**,ZADD 後用 `ZREMRANGEBYRANK` 砍掉超出的尾部。這把每用戶 feed 從「無限增長」釘成「定長」,讓第 2 步的 1.6 TB 成立。想看更老的?**翻到底了走讀擴散現場拉**(冷數據按需查,不佔內存)。

> **黑盒:** 截斷是「用熱數據在內存、冷數據按需查」的冷熱分離,落在 feed 上的具體形態。99% 的刷新只碰最近幾頁,為這 99% 把最近 1000 條釘在內存,長尾交給按需查——這是 working set 思維(`02` 的「working set 能不能放進內存」)。

#### (3) cursor 分頁(為什麼 feed 不能用 offset)

feed 是**無限下滑 + 不斷有新帖插入頂部**的流。用 `LIMIT offset, 20` 翻頁會出兩個 bug:

- **越翻越慢**:offset 1000 要先掃過前 1000 條丟掉,純浪費(`01` Step 3 就點過這個)。
- **錯位**:我在看第 2 頁時,頂部插了 5 條新帖,offset 整體下移 → 第 3 頁會**重複顯示**第 2 頁尾部的帖。

**cursor 分頁**:cursor = 上一頁最後一條的排序錨點(`score` / `tweet_id`)。下一頁 = `ZREVRANGEBYSCORE feed:{me} (last_score -inf LIMIT 0 20)`——**從「上次那條之後」繼續取**,跟總長度無關,O(log N) 定位。新帖插頂部也不影響(我繼續往下取,新帖在我已經翻過的上方,不會重複)。

> **面試話術:** 「feed 分頁我用 cursor 不用 offset——cursor 就是上一頁最後一條的 score,下一頁從那個 score 往後取。這避免了 offset 兩個問題:翻深了要掃過前面浪費,以及看的過程中頂部插新帖導致 offset 錯位、翻頁重複。Redis ZSet 的 ZREVRANGEBYSCORE 天然支援這種『從某個分數往後取 N 條』。」

#### (4) feed 緩存失效與降級

feed 在 Redis,**Redis feed 分片掛了怎麼辦?**——這正是把它跟短鏈緩存區分開的地方:短鏈 miss 回源拿一個 KV 就行,feed 的「源」是「重新跑一遍扇出/讀擴散現場聚合」,重建很貴。緩解:① feed 是**可重建的衍生數據**(帖子表 + 關注關係是真相源,feed 是算出來的),掛了可以**降級為純讀擴散**現場聚合(慢但能用);② 多分片 + 副本避免單點;③ 重建走限流,別把帖子表打垮。錨 `redis-handson/` 的緩存高可用。

### Deep Dive ③ · 排序與多源歸併(時間序 vs 算法序,寫/讀擴散下怎麼有序)

feed 要排序。先把「排序鍵」釘死,再講三個場景下怎麼維持有序。

#### (1) 排序鍵:為什麼用 Snowflake ID 當 score

純時間倒序下,排序鍵就是**時間**。但別用 `created_at` 時間戳當 ZSet 的 score——**同一毫秒多條帖會撞分數,排序不穩定**。用 **Snowflake `tweet_id` 直接當 score**:Snowflake 是「時間戳高位 + 機器位 + 序列號低位」拼出來的**趨勢遞增**整數,它**既是唯一 ID 又自帶全局時間序**,且同毫秒內靠序列號區分,天然全序、無撞分。(Snowflake 細節錨 `05 分散式 ID`,別在這題重講。)

> 這就是第 3 步「tweet_id 用 Snowflake」的回報:**一個趨勢遞增 ID 同時解決了『唯一』和『排序』兩件事**,feed 的 ZSet score 直接用它。

#### (2) 寫擴散下怎麼有序插入

寫擴散時 Fan-out Worker 對每個粉絲 `ZADD feed:{follower} <tweet_id> <tweet_id>`(member 和 score 都用 tweet_id)。ZSet 自動按 score 排好,**插入即有序**,O(log N)。讀時 `ZREVRANGE` 從大到小(最新在前)直接出有序頁——**寫擴散的排序是「插入時順手做掉」的,讀時零排序成本。**

> 一個亂序坑:扇出是異步的,**帖子 A 比 B 先發,但 A 的 Fan-out Worker 比 B 慢**,A 可能晚到某個 feed。但因為 score 用的是 tweet_id(發帖時就定了,趨勢遞增),**ZADD 進去後仍按 tweet_id 排序**——A 雖然晚寫進去,排序位置仍在 B 前面。**用「業務時間(發帖時刻)」而非「寫入時刻」當 score,就免疫了異步扇出的亂序。** 這是個能展示細節功力的點。

#### (3) 讀擴散下怎麼多源歸併(大 V 那一路)

混合方案看 feed 時,要把「預計算 feed(普通關注者)」+「現場拉的 N 個大 V 最新帖」歸併成一條有序流。這是經典的 **k-路歸併(k-way merge)**:

```
N 個大 V 各自的最新帖(各自已按 tweet_id 倒序)+ 預計算 feed(已倒序)
   → 用一個大小為 (N+1) 的最小/最大堆(min-heap on tweet_id desc)
   → 每次彈出當前最大 tweet_id 的那條,取它所在源的下一條補進堆
   → 取夠 20 條(一頁)就停
```

> **黑盒:** 每一路本身有序(都按 tweet_id 排好),歸併只需一個 (N+1) 路的堆,**取一頁 20 條只需 O(20 × log N) 次比較**,N 是我關注的大 V 數(幾十個),極快。這就是為什麼「混合方案在讀時補拉大 V」可行——**補拉的歸併成本被『關注的大 V 很少』這個事實限制住了**。如果是純讀擴散要歸併 500 個普通關注者,同樣的堆要 500 路、且每路都得真去查一次(大部分還沒新帖),那才是讀爆的根源。

#### (4) 從時間序到算法排序(演進)

純時間倒序是起點。真實產品(Instagram/Twitter「For You」)會做**算法排序**:按互動概率、新鮮度、親密度給每條帖打分,再排。這把「排序」從「ZSet 的 score = 時間」變成「**取出候選集 → 過排序模型打分 → 重排**」——score 不再是單一時間,而是模型輸出。

> **面試話術(主動暴露演進與代價):** 「起步我用純時間倒序,排序鍵直接用 Snowflake tweet_id,寫擴散插入即有序、讀時零排序。要升級到算法排序的話,架構會變:feed 不再是『最終順序』,而是『候選集』,看 feed 時把候選集過一個排序模型重排。代價是讀路徑多了一次模型打分(延遲上升)、且預計算 feed 的『順序』失去意義(只剩『候選』意義)。**所以算法排序通常配合『拉一個更大的候選池 + 在線重排』,這是另一套機器學習系統的活了**,面試聚焦時間序版本就夠。」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步 + deep dive ① 的混合方案就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:錢花在「冗餘的 feed 副本」和「扇出計算」上

回到第 2 步——News Feed 的成本結構跟短鏈完全不同,它花在**冗餘存儲**和**扇出計算**兩處:

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **feed 存儲(冗餘)** | 寫擴散把每條帖複製進每個粉絲的 feed,是「N 份副本」 | 用「只存 ID + 截斷 1000 條」壓到 ~1.6 TB(deep dive ②)。**這兩個決策本質是成本控制**,不只是技術選擇 |
| **扇出計算** | 普通用戶扇出量 = ΣΣ 粉絲數,大 V 走讀擴散後被砍掉大頭 | 混合方案 + **在線狀態裁剪**(只推活躍粉絲)把扇出成本砍到可控 |
| **讀時回查 + 歸併** | 60 萬 QPS × (回查內容 + 補拉大 V + 歸併) | 帖子內容緩存命中率拉滿(熱帖被無數 feed 引用);大 V 那路靠「關注的大 V 少」限制成本 |
| **帖子存儲** | TB→PB 級長期增長,只增不刪 | 冷熱分離:老帖下沉冷存儲(沒人翻三年前的帖) |

> **核心成本洞察:** 寫擴散是「**拿存儲和寫計算,換讀延遲**」。對 Twitter 這種讀寫 100:1 的場景,這筆交易划算——少數人寫的代價(扇出),被多數人讀的收益(零排序、O(1) 取 feed)攤平。**但這筆交易的『匯率』是讀寫比**:讀寫比越大,寫擴散越划算;一旦變成寫密集場景,這筆交易就虧了(見「何時不該」)。

### 容量規劃:扇出 worker 池與 MQ 削峰

- **扇出是尖峰負載**:一個中 V(10 萬粉)發帖,瞬間製造 10 萬次 feed 寫——多個中 V 同時發,Fan-out Worker 池要扛得住突發。**Kafka 在這裡是削峰緩衝**:發帖事件先進 Kafka,Worker 按自己節奏消費,扇出延遲(幾秒)被 SLA 允許(最終一致),所以**可以用積壓換平穩**。容量按「平均扇出量 + headroom」配 Worker,尖峰靠 Kafka 積壓吸收,而不是按峰值配滿 Worker(那太貴)。錨 `concurrency-capacity/` 的池容量定容。
- **大 V 閾值是容量旋鈕**:閾值決定「多少扇出量被轉移到讀路徑」。它是平衡寫路徑扇出成本和讀路徑補拉成本的調節閥,要隨流量結構(大 V 數量、人均關注大 V 數)動態調。

### 演進路徑:從 MVP 到混合

```
階段 1(MVP):純寫擴散 + Redis feed
   發帖就推給所有粉絲,看 feed 讀一個 key——簡單、讀極快
        │  出現第一個大 V,扇出開始爆 / 發帖卡頓
        ▼
階段 2:混合方案——大 V 切讀擴散,看 feed 時補拉 + 歸併
   引入大 V 閾值、雙向關注關係、k-路歸併(deep dive ① 的標準答案)
        │  要做個性化、提升停留時長
        ▼
階段 3:算法排序——feed 從「最終順序」變「候選集 + 在線重排」
   接排序模型,讀路徑多一次打分(deep dive ③ 的演進)
```

> **這條演進的關鍵 insight:** **News Feed 跟短鏈相反——它不該一上來就純寫擴散到底,因為純寫擴散在『出現大 V』時必然撞牆**。但也不必第一天就上混合(沒大 V 時混合是過度設計)。`01` 紅旗「過度設計」和「設計不足」在這題是一線之隔:**MVP 純寫擴散是對的,但要在白板上講出『等出現大 V 我切混合』的演進**——能講出這個拐點,比一上來堆滿混合方案更顯資深。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **Redis feed 分片掛** | 一批用戶看不到預計算 feed | feed 是**可重建的衍生數據**(真相源是帖子表+關注表)→ **降級為純讀擴散**現場聚合(慢但能看);多分片+副本;重建限流護帖子表 |
| **Fan-out Worker 積壓** | 新帖延遲出現在粉絲 feed | **最終一致可接受**,積壓只影響 feed 新鮮度(幾秒→幾十秒),不影響發帖和看舊 feed——這是異步扇出最大的好處 |
| **大 V 的 follower 列表熱點** | 寫擴散遍歷千萬粉列表 = 熱點讀 | 大 V 本就走讀擴散**不遍歷粉絲**,從根上避免;follower 列表分片+緩存 |
| **某條爆款帖的回查熱點** | 千萬人 feed 都引用同一 tweet_id,回查打爆帖子緩存 | 熱帖**多副本緩存 / 本地緩存**;熱點打散(錨 `redis-handson/`) |
| **發帖同步等扇出**(設計錯誤) | 大 V 發帖卡幾分鐘,發帖路徑被扇出拖死 | **發帖必須立刻返回**,扇出一律異步——這是架構的鐵律,寫進圖裡 |

> **架構師話術:** 「我的關鍵設計是**把發帖和扇出解耦**——發帖寫完帖子立刻返回,扇出走 MQ 異步。這樣扇出 Worker 積壓只影響 feed 新鮮度(最終一致允許),絕不拖慢發帖。我的單點是 Redis feed 集群,但 feed 是**可重建的衍生數據**,掛了能降級為現場讀擴散——慢,但比掛了強,且重建要限流別把帖子表打垮。最危險的反模式是發帖同步等扇出完成,那大 V 發帖要卡幾分鐘,絕不能這麼做。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **寫密集 / 讀寫接近 1:1**(如某些 IoT/日誌流) | 別用寫擴散(讀沒多到能攤平寫成本) | 倒過來用讀擴散,或乾脆別預計算 |
| **關注關係很少 / 全是廣播**(如新聞訂閱、純大 V 平台) | 別給每個用戶建預計算 feed | 純讀擴散就夠——關注少,讀時拉一下不貴 |
| **強一致 / 順序敏感的流**(如交易流水、聊天消息有序) | 別用「最終一致的異步扇出」 | 那是 `09 IM`(消息有序+已讀回執)或 `14 支付`(強一致)的題,別把 feed 的最終一致假設套上去 |
| **小規模 / 沒大 V**(內部公告、小團隊動態) | 別上混合 + 雙向關注 + k-路歸併 | 純寫擴散一張 Redis List 搞定,別過度設計 |
| **需要精確實時聚合 / 計數**(熱度榜、廣告計費) | 別用 feed 這套近實時最終一致 | 那是 `07 排行榜`(ZSet Top-N)或 `18 廣告實時聚合`(流式精確)的題 |

> **這張表是架構師線的精華**——「知道寫擴散的邊界、知道它在讀寫比變了之後就是錯的」,比「知道寫擴散怎麼做」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。**寫擴散不是『更好的方案』,它是『讀遠多於寫時更好的方案』——這個限定詞就是架構師和工程師的分水嶺。**

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | News Feed 的容量估算結論一句話? | 讀寫 ~100:1、讀峰值 **60 萬 QPS** → home feed 必須**預計算 + Redis 緩存**;真炸彈是**大 V 寫放大**——千萬粉發一帖 = 千萬次 feed 寫,純寫擴散炸到 ~5 億次/秒。所以讀走預計算、寫走異步扇出、且扇出對普通用戶 vs 大 V 分開。 |
| 2 | 寫擴散 vs 讀擴散,各自怎麼炸? | **寫擴散(發帖即推)**:讀快(讀一個 key),但**寫放大=粉絲數**,大 V 千萬粉發一帖寫千萬次(且粉絲多半不在線=寫了個寂寞)。**讀擴散(看時現場拉)**:寫省(只寫一次帖),但**讀放大**=現場拉幾百個關注者再排序,讀路徑被打爆。 |
| 3 | 混合方案怎麼分?為什麼可行? | **普通用戶寫擴散 + 大 V 改讀擴散**。因為兩者爆點不重疊:寫擴散只爆超級節點、讀擴散只爆「關注很多人」。看 feed = 讀預計算 feed(普通關注)+ 現場補拉**少數**大 V + 歸併。補拉便宜因為**人關注的大 V 很少**。這是 Twitter 真實方案。 |
| 4 | feed 為什麼只存 tweet_id 不存內容? | ① 存儲縮 ~40 倍(1.6 TB vs 60 TB);② **帖子可改可刪**——存 ID 則內容改一處(單一真相源)、回查自然拿新值,存副本則要改千萬個 feed。讀時多一次批量 MGET 回查,不貴(熱帖緩存命中率高)。跟短鏈「值不變」正相反:feed 值會變,只存引用。 |
| 5 | feed 排序鍵用什麼?為什麼免疫異步扇出亂序? | 用 **Snowflake tweet_id 當 ZSet score**:趨勢遞增、自帶時間序、同毫秒靠序列號全序無撞分。**用「發帖時刻(tweet_id)」而非「寫入時刻」當 score**,所以即使 Fan-out Worker 異步、晚到的帖 ZADD 進去仍排在正確位置——免疫亂序。 |
| 6 | feed 分頁為什麼用 cursor 不用 offset? | offset 兩個 bug:① 翻深了要掃過前面浪費;② 看的過程中頂部插新帖→offset 錯位→翻頁重複/漏。cursor = 上一頁最後一條的 score,`ZREVRANGEBYSCORE` 從那往後取,跟總長無關、插新帖不影響。 |
| 7 | 大 V 那路讀擴散怎麼歸併保持有序? | **k-路歸併**:每路(預計算 feed + N 個大 V)各自已按 tweet_id 倒序,用 (N+1) 大小的堆每次彈最大 tweet_id、補該源下一條,取夠一頁停。成本 O(20×log N),N=關注的大 V 數(幾十),極快——因為「關注的大 V 少」限制住了歸併路數。 |
| 8 | News Feed 何時**不該**用寫擴散? | 寫密集/讀寫~1:1(讀沒多到攤平寫成本)、關注少/純廣播(讀擴散更省)、強一致順序敏感(走 `09 IM`/`14 支付`)、小規模沒大 V(純寫擴散別上混合)。**寫擴散不是「更好」,是「讀遠多於寫時更好」——限定詞是分水嶺。** |

---

## 一句話總綱(背起來)

> **News Feed 是「扇出母題」——讀寫 100:1、讀峰值 60 萬 QPS 逼出『home feed 必須預計算進 Redis』,而大 V 千萬粉的寫放大(發一帖=千萬次寫,炸到 5 億次/秒)判了純寫擴散死刑。核心答案是混合:普通用戶寫擴散(讀快)、大 V 改讀擴散(免千萬次寫),看 feed 時讀預計算 feed + 現場補拉少數大 V + k-路歸併。feed 靠『只存 ID + 截斷 1000 條』壓進內存(且讓帖子可改可刪),排序鍵用 Snowflake tweet_id(自帶時間序、免疫異步扇出亂序),分頁用 cursor。發帖必須立刻返回、扇出走 MQ 異步(最終一致)。架構師線:寫擴散是『拿存儲換讀延遲』的交易,匯率是讀寫比——讀寫比變小就虧、就該換方案。**
