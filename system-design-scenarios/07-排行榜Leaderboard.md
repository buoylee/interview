# 07 · 排行榜 Leaderboard

> T1 的收官題,也是「**一個零件打天下**」的典型——99% 的排行榜面試,核心答案就一句:**Redis ZSet**。但這題的分水嶺不在「知不知道用 ZSet」,而在後面三個追問:**ZSet 憑什麼能同時做到「按分排序」和「按人查分」?**、**用戶上億時單個 ZSet 為什麼會爆,怎麼救?**、**每日/每週/賽季榜怎麼切、怎麼過期?**
>
> 換句話說:畫得出 `ZADD` + `ZREVRANGE` 是一般工程師;講得清跳表為什麼是 O(log N)、億級為什麼要分桶、時間窗為什麼要拆多個 ZSet——才是資深。這章把這條線走完。
>
> 套用 `01` 的七步框架,調用 `02` 的數字感,深挖三點錨回 `redis-handson/`。蓋住答案、計時 45 分鐘,先自己走一遍。

## 1. 題目 + 澄清問題清單

**題目:** 設計一個遊戲/直播的**實時排行榜**——玩家得分變化後,能查 Top-N、查「我排第幾」、能分頁往下翻,支援每日/每週/賽季榜。

上來別急著喊 ZSet。先用三類澄清把這道開放題收斂成有邊界的題——**不同的答案會把架構推向完全不同的方向**:

| 類 | 要問什麼 | 為什麼這個答案會改架構 |
|---|---|---|
| **功能** | 只看 Top-N,還是**每個人都要查自己排名**? | 只要 Top-N → 一個小 ZSet 就夠;**人人要查排名 → 億級時這是最難的點**(見 deep dive ②) |
| | 要**並列名次**嗎(同分算同名次)?還是嚴格唯一排名? | 並列影響「排名」的定義和算法(見 §5) |
| | 分頁深度?只翻前 100,還是能翻到第 100 萬名? | 深分頁影響是否能純靠 ZSet |
| **非功能 SLA** | 排名要**強實時**(得分一變立刻反映)還是**近實時**(延遲幾秒可接受)? | 強實時 → 同步寫 ZSet;近實時 → 可批量/異步,削寫峰 |
| | 看自己排名:要**精確**到「第 1,234,567 名」,還是「Top 5%」就夠? | **這是億級題的命門**——精確排名 O(log N) 但要全量在一個 ZSet;近似排名可分桶,省一個數量級 |
| | 一致性:短暫看到舊排名能接受嗎? | 能接受 → 緩存 Top-N 隨便加;不能 → 緩存失效要小心 |
| **規模 / 約束** | 總用戶量?**百萬、千萬、還是上億?** | **這是分水嶺**:百萬級單 ZSet 完事;億級單 ZSet 內存爆 + 單分片熱點 → 必須分桶 |
| | 得分更新 QPS?(每次擊殺/打賞/答題都更新?) | 決定寫路徑要不要削峰 |
| | 看榜 QPS?查排名 QPS? | 決定要不要緩存 Top-N |
| | 時間窗:只有總榜,還是日/週/賽季都要? | 多窗 → 多個 ZSet + 過期策略(deep dive ③) |

**面試話術(開場):**
> 「我先收斂範圍。最關鍵的兩個變量是**用戶規模**和**『查自己排名』要不要精確**——如果是百萬級、且只要 Top-N,Redis 一個 ZSet 就解決了,沒什麼可設計的;真正有挑戰的是**上億用戶、且人人都要精確查自己第幾名**,單 ZSet 會內存爆 + 成為熱點。我先按『千萬級總用戶、Top-100 高頻看、人人要查自己近似排名、要日/週/賽季三檔榜』來設計,億級的精確排名我留到 deep dive 講分桶。可以嗎?」

> **這段話的加分點:** 你一句話就把這題的「假題 vs 真題」分清了,並主動把最難的點(億級精確排名)標記出來留到 deep dive——面試官立刻知道你做過這題。

**常見陷阱:**
- ❌ 不問規模直接 `ZADD`——百萬和億級是兩道完全不同的題。
- ❌ 把「並列名次」和「分頁深度」這種影響算法的細節漏掉,deep dive 被問到才現想。
- ✅ 主動點名「查自己排名要不要精確」——這是 90% 候選人想不到、但決定億級架構的那個變量。

## 2. 容量估算(調用 `02` 的數字感)

按澄清結論:**千萬級總用戶、500 萬 DAU**,推一遍。基準數全來自 `02`(一天 1e5 秒、單 Redis ~10 萬 QPS、峰值 ×3)。

### ① 寫:得分更新 QPS

假設活躍玩家人均每天觸發 **50 次**得分變化(擊殺/答題/打賞):

```
寫 QPS = 5e6 DAU × 50 / 1e5 = 2500 平均
峰值 ×3 ≈ 7500 QPS
```

> **所以:** 7500 QPS 的 `ZADD` 對單 Redis(~10 萬 QPS)是毛毛雨——**寫不是瓶頸**。但這是「均攤」數字,賽季結算、整點 boss 戰會有尖峰,可能瞬時 ×10 → 看是否要 MQ 削峰(§6)。

### ② 讀:看榜 + 查自己排名 QPS

看榜遠多於寫(典型讀寫比 10:1~100:1)。假設人均每天**看榜 + 查自己排名 20 次**:

```
讀 QPS = 5e6 × 20 / 1e5 = 1000 平均  →  峰值 3000
```

但**看榜高度集中在 Top-N**(大家都看前 100 名),這是天然的熱點:

> **所以:** Top-N 是「同一份數據被百萬人讀」——**緩存 Top-N(甚至直接讓客戶端定時拉)收益最大**,可把絕大多數看榜 QPS 從 ZSet 上卸掉;`ZREVRANK`(查自己排名)才真正打到 ZSet。

### ③ 內存:ZSet 占用估算(這題的核心約束)

ZSet 在 `skiplist` 編碼下,**每個成員的內存遠大於「裸數據」**——這是錨 `redis-handson` 的關鍵事實。一個成員要存:

| 開銷項 | 大致 |
|---|---|
| member 字符串(玩家 ID,如 16B UUID 或 user:123456) | ~20–40 B |
| score(double,8B)| 8 B |
| **skiplist 節點**(多層前進指針 + 結構頭) | ~30–60 B |
| **額外 dict 條目**(member→score 反向映射,見 deep dive ①) | ~30–50 B |
| Redis 對象/分配器開銷 | ~10–20 B |

**工程估算:每個成員約 100 B**(實機驗證見 deep dive ①——`redis-handson` 實測 200 元素 skiplist 占 17880B ≈ **每元素 ~90 B**,這個數字背下來)。

```
千萬用戶:1e7 × 100 B = 1 GB         → 單 Redis 實例輕鬆(一台 16/32GB 機器)
1 億用戶:1e8 × 100 B = 10 GB        → 開始吃緊,但單機還塞得下
10 億用戶:1e9 × 100 B = 100 GB      → 單 ZSet 必爆(且是單一 key,無法 cluster 分片)
× 時間窗:日/週/賽季 3 個榜 × 3 副本  → 再乘 9
```

> **所以(這句是估算的意義):**
> - **千萬級:單 ZSet,毫無懸念,別過度設計。**
> - **億級:單 ZSet 內存到 10GB+,且因為是「一個 key」無法被 Redis Cluster 自動分片(cluster 按 key 分 slot,一個 ZSet 永遠落在一個分片)→ 該分片成為內存+CPU 熱點 → 必須業務層分桶(deep dive ②)。**

**面試話術:**
> 「我算一下內存。ZSet 每個成員大約 100B,千萬用戶就 1GB,單 Redis 隨便扛。但到**億級就是 10GB 一個 key**——而且關鍵不是內存,是**這是單一 key,Redis Cluster 沒法把它拆到多個分片上**,這個分片會成為熱點。所以億級我不會用一個大 ZSet,我會分桶,這個是這題真正的 deep dive。」

> **這段的價值:** 你不只算了內存,還點出「單 key 無法被 cluster 分片」這個比內存更致命的約束——這正是 §6 架構師深挖和 deep dive ② 的引子。

## 3. API + 數據模型

### API(對外接口)

```
POST /scores            {userId, delta}            → newScore, newRank   # 上報得分變化
GET  /leaderboard/top?window=daily&limit=100       → [{rank, userId, score}]
GET  /leaderboard/rank?userId=...&window=daily      → {rank, score, percentile}
GET  /leaderboard/around?userId=...&radius=5        → [自己 ± 5 名]      # 「我和我前後的人」
GET  /leaderboard/page?window=daily&cursor=...&limit=20  → [...], nextCursor
```

**要點:**
- **`delta` 不是絕對分**:得分通常是「+10」這種增量,對應 `ZINCRBY`(原子加),而不是 `ZADD`(覆蓋)——這避免「讀-改-寫」競態。若是「最高分榜」則用 `ZADD GT`(只在更大時更新,Redis 6.2+)。
- **分頁用 cursor 不用 offset**:這裡 cursor 就是「上一頁最後一名的 (score, member)」,下一頁用 `ZREVRANGEBYSCORE` 帶 `LIMIT`——和 `01` Step 3 講的「深分頁別用 offset」一致,但 ZSet 的 `ZRANGE` 的 offset 也是 O(log N + offset),深翻仍會慢,見 §6。
- **`around`(我前後 N 名)** 是排行榜特有的高頻接口:先 `ZREVRANK` 拿自己排名 r,再 `ZREVRANGE r-5 r+5`。

### 數據模型(核心:ZSet 的鍵設計)

| key 模式 | 結構 | 說明 |
|---|---|---|
| `lb:total` | ZSet `{member=userId, score=分數}` | 總榜 |
| `lb:daily:20260625` | ZSet | 當日榜(key 帶日期) |
| `lb:weekly:2026W26` | ZSet | 當週榜 |
| `lb:season:S5` | ZSet | 賽季榜 |
| `lb:top:daily`(可選)| String/List 緩存 | 預算好的 Top-100 JSON,擋看榜 QPS |

**分片鍵(這題的特殊性):**
> ZSet 的「分片鍵」就是 **key 本身**——`lb:daily:20260625` 整個落在 Redis Cluster 的一個 slot。**這既是優點也是死穴**:優點是排名計算不需要跨片(全在一個 ZSet 裡,`ZREVRANK` 才能 O(log N) 算出全局排名);死穴是**無法水平分片,億級時這個 key 就是熱點**。這個張力是 deep dive ② 的全部。

**面試話術:**
> 「核心數據結構就是 ZSet,member 是 userId、score 是分數。時間窗我用**多個 ZSet**,key 帶日期/週/賽季,這樣切窗和過期都靠 key 解決(deep dive 講)。要注意 ZSet 的『分片鍵』就是 key 自己——這意味著一個榜永遠在一個分片上,排名計算不跨片很爽,但也意味著**它沒法被 cluster 拆開**,億級這是核心矛盾。」

## 4. 高層架構(一鏡到底)

```
                                    ┌──────────────────────────────┐
   上報得分                          │   Redis (ZSet 為核心)          │
 Client ──► Gateway ──► Score Svc ──┼─► ZINCRBY lb:daily:今天        │
                          │         │   ZINCRBY lb:weekly:本週       │
                          │ (近實時可走MQ削峰)  ZINCRBY lb:season:S5  │
                          │         └──────────────────────────────┘
                          │                      ▲
                          ▼                      │ ZREVRANK / ZREVRANGE
                  (異步) MQ ──► Worker            │
                  賽季結算/反作弊/落庫             │
                                                 │
   看榜/查排名                                    │
 Client ──► Gateway ──► Leaderboard Svc ──────────┤
                          │                       │
                          ├─► 先讀 Top-N 緩存 ─────┘ (命中直接回,不打 ZSet)
                          │
                          └─► miss 才 ZREVRANGE 0 99 WITHSCORES

   ┌──── 持久化/數倉(冷路徑)────┐
   │ Redis RDB/AOF + 定期把終榜  │
   │ 落 MySQL/數倉(對賬/補位)    │
   └────────────────────────────┘
```

**沿兩條請求走一遍:**

- **上報得分(寫路徑):** Client → Gateway → Score Service →(同一次)對「今日/本週/賽季」三個 ZSet 各做一次 `ZINCRBY`(原子,O(log N))→ 返回新分數和新排名(順手 `ZREVRANK`)。**近實時場景**可把寫丟進 MQ 由 Worker 批量寫,削掉整點尖峰。
- **看榜(讀路徑):** Client → Gateway → Leaderboard Service → **先查 Top-N 緩存**(命中率極高,因為大家都看前 100)→ 命中直接返回;miss 才 `ZREVRANGE 0 99 WITHSCORES` 回源 ZSet 並回填緩存。
- **查自己排名:** 直接 `ZREVRANK lb:daily:今天 userId`,O(log N) 拿到全局排名。

**要點(每加一個組件說一句為什麼):**
- **Top-N 緩存**:Step 2 算出看榜高度集中在前 100,加緩存把百萬看榜 QPS 從 ZSet 上卸掉。
- **多個 ZSet(按時間窗)**:一次寫更新 N 個榜,換來「每個榜獨立、過期靠 key、互不影響」。
- **MQ(可選)**:只在「近實時可接受 + 有尖峰」時加,削峰;強實時就同步寫。
- **MySQL/數倉(冷)**:Redis 是真相源(快),但賽季結算、發獎、對賬要落庫——Redis 掛了能重建。

**常見陷阱:**
- ❌ 一上來就上 Redis Cluster + 微服務拆 10 個——千萬級一台 Redis 主從就夠,先畫最簡版。
- ❌ 框圖畫了但說不清「看榜為什麼先走緩存」——永遠沿請求講。

## 5. Deep Dive(挑三點往死裡鑽)

> 排行榜的三個必被追問點,正好對應這題的三層深度:**ZSet 憑什麼快(內部結構)→ 億級怎麼不爆(規模化)→ 時間窗怎麼管(運維)**。

### Deep Dive ① · ZSet 底層:跳表 + hash 雙結構,憑什麼同時 O(log N) 排序和 O(1) 查分

這是排行榜面試**最高頻的追問**:「ZSet 內部到底怎麼存的?為什麼 `ZREVRANK` 能 O(log N)、`ZSCORE` 又能 O(1)?」

**核心答案:大 ZSet(skiplist 編碼)同時維護兩個結構,各管一件事。**

```
ZSet (skiplist 編碼) = 跳表 (skiplist) + 字典 (dict)

  跳表 skiplist:按 score 排序            字典 dict:member → score
  ┌─────────────────────────┐           ┌──────────────────┐
  │ L3 ──────────────► n9    │           │ "alice" → 95.0   │
  │ L2 ───► n4 ──────► n9    │           │ "bob"   → 88.0   │
  │ L1 ─► n2 ► n4 ► n6 ► n9  │           │ "carol" → 72.0   │
  │ L0 n1 n2 n3 n4 ...  n9   │           │ ...              │
  └─────────────────────────┘           └──────────────────┘
   負責:按分排序、排名、範圍           負責:O(1) 按 member 查分
   ZRANGE / ZREVRANK / ZRANGEBYSCORE     ZSCORE / ZINCRBY 先查舊值
```

**為什麼要兩個結構?** 因為兩類查詢的「索引方向」相反:

| 查詢 | 走哪個結構 | 複雜度 | 為什麼 |
|---|---|---|---|
| `ZSCORE alice`(查某人分數) | **dict** | **O(1)** | hash 直接命中,不用排序 |
| `ZADD`/`ZINCRBY` | dict 找舊值 + skiplist 調位 | O(log N) | 先 O(1) 查舊 score,再在跳表裡 O(log N) 移動節點 |
| `ZRANGE` / `ZREVRANGE`(Top-N) | **skiplist** | O(log N + M) | 跳表定位起點 O(log N),再順鏈取 M 個 |
| `ZREVRANK alice`(查某人第幾) | **skiplist** | **O(log N)** | 關鍵:跳表節點存了 **span**(跨度),累加跨度即排名 |
| `ZRANGEBYSCORE` | skiplist | O(log N + M) | 按分範圍,跳表二分定位 |

**跳表憑什麼 O(log N)——黑盒裡發生什麼:**

跳表是「**帶多層快速通道的有序鏈表**」。最底層 L0 是完整的有序鏈表(O(N) 掃);往上每層**概率性**地抽稀(典型 1/2 或 1/4),L1 跳過一半節點,L2 再跳過一半……查找時從最高層往右走,走不動就下一層,**像跳格子一樣**跳過大段節點,期望 O(log N) 步到位。

- **為什麼用跳表不用平衡樹(紅黑樹/B 樹)?** 這是面試官最愛的反問。答:① **範圍查詢友好**——跳表 L0 本身就是有序鏈表,`ZRANGE` 定位起點後直接順鏈走,平衡樹要中序遍歷更麻煩;② **實現簡單、無需旋轉**——插入只調指針,不像紅黑樹要旋轉再平衡,代碼量小、並發改造也容易;③ **概率平衡夠用**——不保證最壞 O(log N),但期望 O(log N),工程上足夠。Redis 作者 antirez 明說選跳表就是為了**範圍操作 + 實現簡單**。

- **`ZREVRANK` 的 O(log N) 怎麼來——span 是關鍵(這點 90% 人答不出):** 普通跳表查找是 O(log N),但「我排第幾」需要知道「我前面有多少個」。Redis 在**每個前進指針上額外存了一個 `span`**(這一跳跨過了多少個底層節點)。查排名時沿跳表往目標走,把走過的所有 span 累加,就是排名——所以 `ZRANK`/`ZREVRANK` 也是 **O(log N)**,不需要 O(N) 數。這是 Redis 跳表相對教科書跳表的關鍵改造。

**listpack → skiplist 的轉換(實機數字,背下來):**

小 ZSet 不用 skiplist——元素少時 Redis 用 `listpack`(緊湊數組)省內存。錨 `redis-handson` 02 章**實測**:

| 編碼 | 觸發條件(默認) | 200 元素內存 | 操作複雜度 |
|---|---|---|---|
| `listpack` | 元素 ≤ 128 **且** member ≤ 64B | **1840 B** | O(N) 線性掃(元素少所以快) |
| `skiplist` | 超過任一閾值 | **17880 B**(~9.7 倍) | O(log N) |

> **黑盒事實(來自 `redis-handson` 02 章實跑):** 127 元素 listpack 占 1072B,只多加 2 個到 129 跨過閾值,**內存從 1072 暴漲到 14811(~14 倍)**——不是多存 2 個的開銷,是整個結構換成 skiplist+dict。所以「每個用戶一個幾十元素的小 ZSet」(如個人最近成就)要刻意壓在 128 內留 listpack,能省近一個數量級內存。

**面試話術:**
> 「ZSet 大了是 skiplist + dict 雙結構:dict 管 `ZSCORE` 的 O(1) 按人查分,skiplist 管按分排序。`ZREVRANK` 能 O(log N) 是因為 Redis 在每個跳表指針上存了 **span**——查排名時累加走過的 span,不用 O(N) 數。選跳表不選紅黑樹是為了**範圍查詢友好 + 實現簡單**。小 ZSet 會退化成 listpack 省內存,128 元素是分界線,跨過去內存會瞬間漲近 10 倍。」

> 細節錨 `redis-handson/02-data-structures`(編碼轉換實測)、`03-advanced-types`(ZSet 進階用法)。

### Deep Dive ② · 海量用戶排名:億級單 ZSet 為什麼爆,怎麼救

這是這題**真正的架構題**,也是 §2 估算埋的伏筆:**1 億用戶 = 10GB 一個 key,且這個 key 是單一 slot 無法被 cluster 分片 → 內存熱點 + CPU 熱點。**

#### 問題拆成兩半

| 子問題 | 痛在哪 |
|---|---|
| **內存爆** | 10 億用戶 ×100B = 100GB 一個 ZSet,單機塞不下 |
| **熱點** | ZSet 是一個 key,永遠落單分片;所有讀寫打它,該分片 CPU/網卡先掛(cluster 加機器也沒用——分不開) |
| **深排名貴** | `ZREVRANK` 雖 O(log N),但 N=10 億時 log₂N≈30 跳 × 高 QPS,且**人人都要查自己排名**意味著海量隨機排名查詢 |

#### 救法階梯(按規模升級,別一上來就上最重的)

**方案 A:只精確 Top-N + 近似全局排名(最常用,先給這個)**

關鍵洞察:**「Top-100 要精確,但第 5,234,891 名 vs 第 5,234,890 名沒人在乎」**——絕大多數用戶只想知道「我大概在前 10%」。

- **Top-N** 仍用一個「**只保留高分段**」的小 ZSet(如只存前 1 萬名),精確、小、快。
- **普通用戶查排名** → 走**近似排名**:把分數空間分成 N 個桶(score 區間),每個桶用 `INCR` 計數(「分數落在 [900,1000) 的有多少人」)。查某人排名 = 累加比他分高的桶計數 + 他在自己桶內的位置。**O(桶數)** 而非 O(log N),且計數器可任意分片。

> **這就是把「精確全序」放鬆成「分段統計」**,用一個數量級的精度換一個數量級的可擴展性。錨 `system-design/07 數據規模化-分庫分表` 的核心思想:**當全局有序代價太高,就退到「分桶 + 段內近似」。**

**方案 B:分桶/分片 ZSet(要精確排名但量大)**

- 按 userId hash 或按 score 區間,把用戶拆到 **K 個 ZSet 分片**(`lb:shard:0` … `lb:shard:K-1`)。
- **寫**:`ZINCRBY` 打到對應分片——寫 QPS 被 K 等分,熱點解除。
- **Top-N**:從每個分片取 Top-N,在應用層做 **K 路歸併**(像合併 K 個有序鏈表)取全局 Top-N——O(K × N) 歸併。
- **查全局排名**(精確,代價高):某人的全局排名 = 在自己分片的局部排名 + Σ(其他每個分片中比他分高的人數)。後者要對每個分片做 `ZCOUNT(他的分數, +inf)`,**O(K × log N)**。K 不能太大,否則查排名變慢——**這就是 tradeoff:分片越多寫越散、但精確查排名越貴。**

**方案 C:離線預計算 + 定期刷排名(超大 + 可接受延遲)**

- 億級且排名可接受「分鐘級延遲」:用流/批(Flink/Spark)離線算好每個人的排名,寫進 KV(`rank:userId → 1234567`),查排名變 O(1) 點查。
- 實時榜(Top-N、近期變化)留 Redis ZSet,**全量精確排名走離線**——熱榜與全量榜分離(和 deep dive ③ 的「熱榜/全量榜分離」呼應)。

#### 決策表

| 規模 / 需求 | 方案 |
|---|---|
| ≤ 千萬,任意需求 | **單 ZSet,別分**(過度設計是扣分項) |
| 億級,只要 Top-N + 近似排名 | **方案 A**:小 ZSet 存 Top + 分桶計數近似排名 |
| 億級,要精確全局排名 | **方案 B**:分片 ZSet + K 路歸併 + 跨片 `ZCOUNT`,K 控制在個位數~幾十 |
| 10 億+,排名可分鐘延遲 | **方案 C**:離線預算排名 + 實時榜分離 |

**面試話術:**
> 「億級的死穴不是內存,是**ZSet 是一個 key,Redis Cluster 沒法把它拆開,這個分片成熱點**。我分三檔:量小就一個 ZSet 別折騰;億級但允許近似——這是關鍵洞察:**Top-100 要精確,但第五百萬名沒人在乎**,所以我用小 ZSet 存 Top-N、分桶計數做近似排名,計數器能隨便分片;如果**真要精確排名**,就分片 ZSet + K 路歸併取 Top-N、跨片 ZCOUNT 算精確排名,代價是分片越多查排名越貴;再大就離線預算排名走 O(1) 點查。**核心思想是:全局有序太貴時,退到分桶 + 段內近似,用精度換可擴展。**」

> 錨 `system-design/07 數據規模化`(分桶/分片思想)、`redis-handson/11-cluster`(為什麼單 key 無法分片)。

### Deep Dive ③ · 實時更新 + 持久化 + 時間窗:多 ZSet 切窗、過期、熱榜分離

排行榜不是一個靜態榜,是「日/週/賽季 + 總榜」的一組榜,還要在 Redis 掛了能恢復。這點考的是**運維和演進**思維。

#### 時間窗:按 key 切多個 ZSet

**核心手法:時間窗編進 key,一個窗一個 ZSet。**

```
lb:daily:20260625      ← 今日榜
lb:weekly:2026W26      ← 本週榜
lb:season:S5           ← 賽季榜
lb:total               ← 總榜
```

一次得分變化,**同時對相關窗 `ZINCRBY`**(今日 + 本週 + 賽季,通常不更新總榜或總榜異步)。好處:

- **切窗零成本**:到了明天,寫自然落到 `lb:daily:20260626`,昨天的榜原封不動成為歷史榜可查。
- **窗間隔離**:日榜重置不影響週榜。
- **查歷史**:`lb:daily:20260601` 直接可讀,無需重算。

#### 過期清理:別讓死榜堆滿內存

每天/每週生成新 ZSet,舊的要清,否則內存無限增長:

| 方法 | 做法 | 取捨 |
|---|---|---|
| **TTL 自動過期** | 創建日榜時 `EXPIRE lb:daily:xxx 7*86400`(留 7 天歷史) | 簡單,但 Redis **惰性 + 定期**刪除,大 key 過期可能瞬時釋放大塊內存造成抖動 |
| **定時任務主動刪** | 後台 job 每天掃 `lb:daily:*` 刪 N 天前的 | 可控,能在低峰刪、能 `UNLINK`(異步刪)避免大 key 刪除阻塞 |
| **歸檔後刪** | 過期前先把終榜落 MySQL/數倉,再刪 Redis | 保留歷史用於發獎/對賬,Redis 只當熱數據 |

> **黑盒坑(錨 `redis-handson/04-expiry-eviction`):** Redis 刪一個百萬成員的大 ZSet,`DEL` 是**同步阻塞**的——會卡住主線程造成毛刺。億級榜要用 **`UNLINK`(異步刪除)** 或拆小分批刪。這是大 ZSet 過期的隱藏地雷。

#### 持久化:Redis 掛了排名別丟

- **Redis 自身**:RDB(快照)+ AOF(日誌)。排行榜場景偏向 **AOF everysec**——崩潰最多丟 1 秒得分更新,可接受;純 RDB 可能丟幾分鐘。
- **真相源之爭**:Redis 快但「易失」。生產通常**Redis 為實時計算真相源 + MySQL/數倉為持久備份**——得分變化異步落庫(走 §4 的 MQ→Worker),Redis 全掛了能從庫重建 ZSet。**這是「速度 vs 持久」的經典分工**:讀寫排名走 Redis(ms 級),審計/發獎/重建走 DB。錨 `redis-handson/05-persistence`。

#### 熱榜 vs 全量榜分離

- **熱榜**(Top-N、近期活躍):小、熱、純 Redis,所有看榜 QPS 走這裡(配 §4 的 Top-N 緩存)。
- **全量榜**(所有人的精確排名):大、冷查、可走 deep dive ② 的分片或離線。
- **分離的理由**:99% 看榜流量打 Top-N,不該讓它和「億級全量排名計算」搶同一個 Redis 的內存和 CPU。

**面試話術:**
> 「時間窗我把窗編進 key,一個窗一個 ZSet——切窗零成本、窗間隔離、歷史榜直接可讀。過期我**不只靠 TTL**,因為大 ZSet 同步 `DEL` 會阻塞主線程造成毛刺,我用後台 job 在低峰 `UNLINK` 異步刪,刪前先歸檔到數倉發獎用。持久化上 Redis 是實時真相源、走 AOF everysec,但得分變化我**異步落 MySQL**,Redis 全掛能重建——速度走 Redis、持久和對賬走 DB。最後熱榜(Top-N)和全量榜分離,別讓看榜流量和億級排名計算搶資源。」

> 錨 `redis-handson/04-expiry-eviction`(大 key 刪除 / UNLINK)、`redis-handson/05-persistence`(RDB/AOF 取捨)。

### 附:並列名次怎麼處理(澄清裡問到就答這個)

ZSet 的排名是**嚴格唯一**的(同分按 member 字典序排次序),但業務常要「同分同名次」(競賽排名)。三種處理:

| 需求 | 做法 |
|---|---|
| 同分同名次(1,1,3) | 應用層後處理:取出帶 score 的列表,自己掃一遍合併同分 |
| 同分按時間先到先得 | **score 編碼技巧**:`score = 真實分 × 1e12 - 時間戳`,高分優先、同分早到優先,單一 double 表達複合排序 |
| 嚴格唯一(遊戲常見) | 直接用 ZSet 默認行為即可 |

> **score 編碼技巧**是排行榜進階點:double 有 52 位尾數,可把「主分 + 次序」打包進一個 score,讓 ZSet 一次排序就搞定複合規則——但要注意精度,分數和時間戳的位數不能超 double 精度。

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效 / 何時不該)

IC 線到 §5 就夠拿 offer 了。這一層是「**擁有這個決策**」——錢誰買單、怎麼規劃容量、怎麼演進、會怎麼掛。

### 成本:Redis 內存是按 GB 真金白銀算的

| 規模 | ZSet 內存(×3 窗 ×3 副本) | 成本含義 |
|---|---|---|
| 千萬用戶 | 1GB × 9 ≈ 9GB | 一台中等 Redis,月成本可忽略 |
| 億用戶 | 10GB × 9 ≈ 90GB | 需要大內存實例或分片集群,**內存是主成本** |
| 10 億用戶(全量精確) | 100GB × 9 ≈ 900GB+ | **純 Redis 全量精確排名成本失控** → 必走 deep dive ② 方案 C(離線 + KV),把「全量精確排名」從昂貴的內存挪到便宜的批計算 |

> **架構師判斷:** 「億級全量精確排名 + 純 Redis」是成本陷阱。99% 業務真實需求是「Top-N 精確 + 普通人近似排名」,堅持全量精確等於為 1% 的需求付 10 倍的錢。**先逼問業務:真的要精確到個位排名嗎?** 多數時候答案是不要。

### 容量規劃:寫不是瓶頸,但要防尖峰與大 key

- **寫**:`ZINCRBY` O(log N),單 Redis 10 萬 QPS,絕大多數排行榜寫遠低於此——**寫幾乎永不是瓶頸**。
- **真瓶頸是大 key 操作**:`ZREVRANGE 0 1000000`(深分頁取百萬)、大 ZSet 的 `DEL`、全量 `ZRANGEBYSCORE`——**這些是 O(N) 或大 M,會阻塞單線程 Redis 造成全實例毛刺**。容量規劃要**禁止深分頁**(分頁深度設上限,如最多翻到第 1000 名)、大 key 拆分、刪用 UNLINK。
- **headroom**:Redis 內存留 30% 餘量(`02` 的規則),逼近 `maxmemory` 會觸發淘汰/OOM。

### 演進路徑(別一步到位,按規模長)

```
階段1(<千萬):單 Redis 主從 + 一個 ZSet/窗 + Top-N 緩存。夠用,別分。
   │ 用戶漲、看榜 QPS 漲
   ▼
階段2(千萬~億):看榜 QPS 用本地/CDN 緩存 Top-N(客戶端定時拉),寫加 MQ 削整點尖峰。
   │ 逼近單 key 內存/熱點上限
   ▼
階段3(億級):分片 ZSet(deep dive ②方案B)或退近似排名(方案A);熱榜/全量榜分離。
   │ 全量精確排名成本失控
   ▼
階段4(10億+):全量精確排名離線化(方案C,Flink/批算 → KV O(1) 點查),Redis 只留實時熱榜。
```

> **演進的 trigger 是數字,不是拍腦袋:** 「看榜 QPS 逼近單 Redis」→ 加緩存層;「單 key 內存逼近 10GB / 該分片 CPU 飽和」→ 分片;「全量精確排名月成本 > 收益」→ 離線化。每一步都被 §2 那幾個數驅動。

### 失效模式(會怎麼掛)

| 失效 | 後果 | 緩解 |
|---|---|---|
| **Redis 主掛** | 排名讀寫全停;若無持久化排名丟失 | 主從 + Sentinel 自動切;AOF + 異步落 DB 兜底重建(`redis-handson/10-replication-sentinel`) |
| **大 ZSet 的 DEL/深分頁阻塞主線程** | 單線程卡住 → **整個實例所有 key 都慢**(不只排行榜) | UNLINK 異步刪、禁深分頁、大 key 拆分 |
| **億級單 key 熱點** | 該分片 CPU/網卡飽和,cluster 加機器無效(分不開) | 業務層分片 ZSet,把熱點打散 |
| **賽季結算尖峰** | 結算瞬間海量寫 + 全量讀,打爆 Redis | MQ 削峰 + 結算走離線批,別在線上 ZSet 上跑全量 |
| **緩存與 ZSet 不一致** | 看到舊 Top-N | Top-N 緩存設短 TTL(秒級);排行榜本就近實時,可接受 |

### 何時**不該**用 ZSet / 不該這麼設計(架構師的反向判斷)

- **❌ 不該用 ZSet:全量精確排名 + 億級 + 高頻**——成本失控,該離線預算(方案 C)。ZSet 適合「熱、實時、規模可控」,不適合「冷、全量、超大」。
- **❌ 不該分片:千萬級就分片 ZSet**——憑空增加 K 路歸併和跨片排名的複雜度,純屬過度設計。**單 ZSet 能扛就別分**。
- **❌ 不該強實時:得分更新極高頻但業務只要近實時**——同步寫三個 ZSet 是浪費,該批量/異步,削峰省資源。
- **❌ 不該純 Redis:排名是發獎/結算的法律依據(賭博/競賽獎金)**——Redis 易失,真相源該是 DB(強一致),Redis 只做展示加速。**錢相關的排名,真相在 DB 不在 Redis。**

> **一句架構師判斷:** 「排行榜的設計複雜度**完全由兩個變量決定**:用戶規模、和『查自己排名要不要精確』。絕大多數真實業務是『千萬級 + Top-N 精確 + 個人近似』,答案就是**一個 ZSet + Top-N 緩存 + 多窗按 key 切**,別過度設計;只有當這兩個變量都頂到極限(億級 + 全量精確),才需要分片/離線那套重武器,而那時要先反問業務『真的需要嗎』,因為它的成本是十倍級的。」

## 8. 面試卡(蓋住背面自測)

**卡 1 · 核心答案**
> Q:排行榜怎麼設計?
> A:核心是 **Redis ZSet**,member=userId、score=分數。`ZINCRBY` 更新、`ZREVRANGE` 取 Top-N、`ZREVRANK` 查排名。時間窗按 key 切多個 ZSet(日/週/賽季)。Top-N 加緩存擋看榜 QPS。**複雜度全由「用戶規模 + 排名要不要精確」決定。**

**卡 2 · ZSet 為什麼快**
> Q:ZSet 憑什麼同時 O(log N) 排序、O(1) 查分?
> A:大 ZSet = **skiplist(按 score 排序)+ dict(member→score)** 雙結構。`ZSCORE` 走 dict O(1);`ZREVRANK` 走 skiplist,靠每個指針上存的 **span** 累加得排名 O(log N)。選跳表不選紅黑樹:範圍查詢友好 + 實現簡單。小 ZSet 退化成 listpack(≤128 元素)省內存,跨閾值內存瞬漲近 10 倍。

**卡 3 · 億級怎麼救**
> Q:1 億用戶單 ZSet 為什麼爆?怎麼解?
> A:不是內存(10GB),是**單一 key 無法被 cluster 分片 → 熱點分片**。三檔:① 只要 Top-N+近似排名 → 小 ZSet + 分桶計數;② 要精確排名 → 分片 ZSet + K 路歸併 + 跨片 ZCOUNT(分越多查排名越貴);③ 10億+ → 離線預算排名走 O(1) 點查。核心:**全局有序太貴就退分桶+近似,用精度換可擴展。**

**卡 4 · 時間窗 + 過期**
> Q:日/週/賽季榜怎麼管?過期怎麼清?
> A:窗編進 key,一窗一 ZSet(`lb:daily:20260625`),一次寫更新多窗,切窗零成本、歷史榜直接可讀。過期**別只靠 TTL**——大 ZSet 同步 `DEL` 阻塞主線程,用後台 job 低峰 `UNLINK` 異步刪,刪前歸檔數倉。

**卡 5 · 容量 / 失效**
> Q:排行榜的瓶頸和失效模式?
> A:寫(`ZINCRBY` O(log N))幾乎不是瓶頸;真瓶頸是**大 key 的 O(N) 操作**(深分頁、大 key DEL、全量掃)阻塞單線程 → 全實例毛刺。對策:禁深分頁、大 key 拆分、UNLINK、Top-N 緩存。失效:主掛靠 Sentinel + AOF + 異步落 DB 重建;億級熱點靠分片打散。

**卡 6 · 何時不該這麼設計**
> Q:什麼時候不該用 ZSet / 不該分片?
> A:❌ 億級全量精確排名高頻 → 成本失控,該離線化;❌ 千萬級就分片 → 過度設計,單 ZSet 能扛別分;❌ 錢相關(獎金)排名純靠 Redis → Redis 易失,真相源該是 DB,Redis 只做展示加速。

---

## 一句話總綱

> **排行榜 = Redis ZSet(skiplist 按分排序 + dict 按人查分,靠 span 做 O(log N) 排名),時間窗按 key 切多個 ZSet、Top-N 加緩存擋看榜流量。整題的複雜度只由兩個變量決定——用戶規模 和 排名要不要精確:千萬級就一個 ZSet 別折騰;億級且只要近似就分桶計數;億級要精確才分片歸併;10 億+ 就離線預算排名。核心判斷是『全局有序太貴時退到分桶 + 段內近似,用精度換可擴展』,以及『錢相關的排名真相在 DB 不在 Redis』。**
