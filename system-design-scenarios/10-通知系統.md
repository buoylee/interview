# 10 · 通知系統 Notification(多渠道推送平台)

> **T2 的「扇出三兄弟」收官題。** `08 News Feed`、`09 IM`、`10 通知` 本質都是同一個母題——**一個事件,怎麼可靠地扇出給很多人**(`00` 依賴表三題都吃 E ●●「服務化與通信」)。Feed 扇出的是「帖子寫進誰的時間線」,IM 扇出的是「消息推給哪些在線設備」,而**通知扇出的是「一個業務事件,觸發多少用戶 × 多少渠道的推送」**。
>
> 通知系統的教學價值集中在三處,且**和前兩題不重疊**:**渠道適配層**(同一條通知,怎麼同時走 APNs / FCM / SMS / email 四套完全不同的第三方網關,且任何一家掛了不能拖垮全局——這是 `system-design/01 韌性` 的活靶子)、**去重/限頻/優先級**(防止把用戶轟炸到退訂——這是「為用戶體驗做減法」的工程,別的題都在做加法)、**可靠投遞**(at-least-once + 冪等鍵 + 投遞狀態機,直接吃 `financial-consistency` 的冪等肌肉,但約束比錢弱)。
>
> 用 `00` 依賴表看:通知吃 **A + B + C + E(●●)**——E(MQ 扇出 + 第三方網關)是重心,C(韌性:網關熔斷降級)是它和 Feed/IM 最大的差異點。**這題的靈魂是「我不擁有投遞的最後一公里」**——APNs/SMS 網關是別人的服務,會限流、會抖、會掛,設計的一半篇幅都在處理「依賴一個不可靠的外部系統」。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個多渠道通知系統。上游業務(訂單、風控、社交、營銷)發一個「要通知某用戶」的請求,系統負責把它**渲染成內容、按用戶偏好選渠道(App push / SMS / email / 站內信)、調第三方網關投出去,並追蹤投遞狀態**。

**這題的澄清比短鏈關鍵得多**——「通知」這個詞太大,不收斂範圍會做成四個不同的系統。用 `01` Step 1「問三類」狠狠收斂:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 哪些渠道?要不要模板/個性化?用戶能不能設偏好/退訂?要不要投遞回執? | 渠道:**App push(APNs/FCM)+ SMS + email + 站內信(in-app)** 四種;**模板 + 變量替換**要;**用戶偏好 + 退訂**要(合規剛需);**投遞狀態追蹤**要 |
| **非功能(SLA)** | 可靠性?延遲?哪類通知能丟?能不能重複? | **分優先級**:驗證碼/OTP 是**秒級 + 不能丟 + 不能重**(其實是 IM 級);營銷推送**可延遲幾分鐘、可採樣丟、絕不能轟炸**;整體**at-least-once + 冪等去重**,不追求 exactly-once |
| **規模 / 約束** | 通知量級?扇出放大多大?第三方網關吞吐限制? | **~10 億通知/天**;一個事件可扇出到**百萬用戶 × 多渠道**(營銷群發);**第三方網關有硬吞吐上限和限流**(這是真瓶頸,不是我的機器) |

**面試話術(把假設說出來讓對方確認):**
> 「通知這題範圍很大,我先框一下——我做**四渠道**:App push、SMS、email、站內信;支持**模板渲染 + 用戶偏好/退訂**。可靠性我會**分優先級**:驗證碼這種交易型通知秒級不能丟、要去重;營銷推送可延遲、可採樣,但**絕不能把用戶轟炸到退訂**。規模假設 10 億通知/天,一次營銷群發能扇出到百萬用戶。最關鍵的約束我想先點明:**投遞的最後一公里(APNs/SMS 網關)是第三方,不是我能控制的**——它會限流、會抖、會掛,我的設計有一半在處理這個。這樣可以嗎?」

**這幾個澄清直接決定後面**:

- 「四渠道 + 第三方網關」→ 第 5 步架構**必有一層渠道適配 + provider 抽象**,第 6 步 deep dive ① 鑽它的韌性。
- 「分優先級、驗證碼不能丟不能轟炸」→ deep dive ② 的去重/限頻/優先級隊列是整題的第二根支柱。
- 「at-least-once + 冪等」→ deep dive ③ 的投遞狀態機 + 冪等鍵,錨 `financial-consistency`。
- 「一個事件扇出百萬用戶」→ 第 2 步估算的結論會是**扇出放大 + 削峰才是真矛盾,不是原始 QPS**。

**常見陷阱:** 上來就糾結「email 的 DKIM/SPF 怎麼配」「APNs 的 token 怎麼刷新」這種**第三方協議細節**。✅ 抓大放小——這些是 provider 適配器內部的事,架構層先當黑盒,留時間給扇出/去重/可靠投遞三根主線。

## 2. 容量估算(直接調用 `02` 的數字)

> 通知這題的估算**陷阱在「扇出放大」**——你不能只算「上游發了多少請求」,要算「放大成多少條實際投遞」。這正是 `02` Drill 1 Feed 扇出的同款放大,但通知還多乘一層「渠道數」。

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 = 平均 ×3、單 Kafka 分區 ~10 萬 msg/s、單 Redis ~10 萬 QPS、單機長連接題不適用(通知是短連接 outbound)**。

### ① 投遞 QPS(放大後的真實量)

```
10 億通知/天 ÷ 1e5 s ≈ 1 萬/s(平均,放大後)
峰值 ×3 ≈ 3 萬/s
```
> **所以:平均 1 萬、峰值 3 萬條投遞/s。光看這個數不大——但它是「放大後」的數**,真正的問題不在穩態 QPS,在**尖峰**:營銷活動「雙11 0 點全員推送」,一個事件瞬間生成百萬~千萬條投遞,在幾秒內砸下來。**所以削峰(MQ 緩衝 + 按優先級限速消費)是這題的命門**,不是穩態擴容。

### ② 扇出放大係數(這題的真矛盾)

一個上游事件 → 多少條實際投遞?

```
一個營銷活動事件
  → 收件人解析:1000 萬用戶
  → 每用戶按偏好平均 1.5 個渠道(有人只收 push、有人 push+email)
  → 1000 萬 × 1.5 = 1500 萬條 provider 調用
```
> **所以:放大係數 = 收件人數 × 人均渠道數,輕鬆上千萬倍。** 上游一個請求,下游千萬次第三方網關調用——這就是為什麼**扇出必須異步、必須過 MQ**:同步做的話上游 API 要等千萬次外部調用,根本不可能。這條結論直接定下第 4/5 步的架構骨架(事件入隊 → 扇出 Worker 展開 → 渠道隊列削峰)。

### ③ 第三方網關吞吐(瓶頸不在我這)

這是通知題**獨有的、最該說出口的估算**——瓶頸是別人的限流,不是你的機器:

| 網關 | 典型吞吐限制 | 含義 |
|---|---|---|
| **APNs / FCM** | 單連接 ~數千/s,可多連接並發(較寬鬆) | push 通常不是瓶頸 |
| **SMS 網關**(Twilio 等) | 按賬號限流,常見 **~100–1000 條/s** | **SMS 是最窄的管子**,峰值會被它卡住 |
| **email**(SES/SendGrid) | 按賬號配額,**每秒幾十~幾百**,且有日配額 | email 也窄,且超配額會被封 |

> **所以:我的系統內部能算到峰值 3 萬/s,但 SMS 網關只給我 500/s——這意味著 SMS 渠道天然要排隊、要按 provider 配額限速消費。**「我的吞吐 ≠ 我能投遞的吞吐」,投遞速度被外部配額封頂。這條是 deep dive ① 渠道隊列「按 provider 限速」的估算依據,也是面試官最愛追的點:「你怎麼知道 SMS 會成瓶頸?」——答案就在這。

### ④ 存儲(投遞狀態 + 用戶偏好)

```
投遞狀態日誌:10 億/天 × ~300 B(notif_id/user/channel/狀態/時間)
  = 300 GB/天 → ×副本3 ≈ 1 TB/天(且只增)
  → 熱數據(最近幾天可查/可重試)留快存,老的歸檔/TTL
用戶偏好表:用戶數 × 渠道偏好,相對小(GB 級),Redis 緩存熱用戶
模板表:極小(幾千個模板),全量緩存
```
> **所以:投遞狀態日誌是存儲大頭(1 TB/天,且只增)——但它是「可過期」數據**:通知投出去幾天後,狀態日誌的價值就剩審計/報表,可以 TTL 過期或冷歸檔。和短鏈「只增不減」不同,通知狀態天然可回收。

### 一句話估算結論(面試一定要說出來)

> 「**通知的真矛盾不是穩態 QPS(放大後峰值才 3 萬/s),是三件事:(1) 扇出放大——一個營銷事件展開成千萬條投遞,必須異步過 MQ;(2) 尖峰削峰——0 點全員推送瞬間砸下來,靠 MQ 緩衝 + 按優先級限速消費扛;(3) 第三方網關是瓶頸——SMS/email 網關只給我幾百/s,投遞速度被外部配額封頂,渠道隊列必須按 provider 限速。** 所以我的設計重心是『扇出怎麼削峰』和『不可靠的第三方怎麼兜』,而不是堆內部 QPS。」

這句把估算焊到架構,就是 `01` 說的資深分水嶺。

## 3. API + 數據模型

### API(`01` Step 3:快過,點到為止)

**對上游(業務方)只暴露一個薄入口**——「我要通知這個/這批用戶」,渠道/模板/偏好的複雜度全藏在系統內部:

```
POST /notifications
  body: {
    templateId: "order_shipped",        # 模板 ID,不傳原始文案
    recipients: ["u_123", ...],         # 用戶 ID 列表(或一個 segment 查詢)
    variables: { orderNo: "A001", ... },# 模板變量
    priority: "transactional|marketing",# 優先級(影響隊列與限頻)
    idempotencyKey: "evt_8f3a..."       # 冪等鍵,重發不重複投遞
  }
  → 202 Accepted { requestId }          # 立刻返回!投遞是異步的

GET  /notifications/{requestId}/status  # 查投遞狀態(扇出後的聚合)
  → { total, sent, delivered, failed, perChannel: {...} }

# 用戶偏好(給用戶端/設置頁)
PUT  /users/{uid}/preferences  { channel: "sms", enabled: false, quietHours: ... }
POST /unsubscribe              { uid, category, token }   # 退訂(合規,token 防偽)
```

**要點:**
- 寫接口**只回 202 不回投遞結果**——投遞是異步扇出,API 同步等千萬次外部調用是不可能的(`02` ② 算過)。上游拿 `requestId` 後續查狀態。
- **`idempotencyKey` 是強制的**,不是可選:上游網絡抖動重發、MQ at-least-once 重投,都靠它去重(deep dive ② / ③ 的核心)。
- **傳 `templateId` 不傳原始文案**——渲染在系統內做,好處是改文案不用上游發版、且能做合規審查(deep dive 模板層)。

### 數據模型(`01` Step 4:選型 + 分片鍵)

通知的數據模型有**四張核心表**,訪問模式各不相同——這題不像短鏈只有一張 KV:

| 表 | 關鍵字段 | 訪問模式 | 選型 |
|---|---|---|---|
| **template** | `template_id PK`, `channel`, `locale`, `body_tpl`, `version` | 按 id 點查、極少寫、全量小 | 任意 + **全量緩存** |
| **user_preference** | `user_id PK`, `channel_settings`(push/sms/email 開關)、`quiet_hours`、`unsub_categories` | 按 user_id 點查、讀多寫少 | **KV / Redis 緩存**(熱用戶) |
| **device_token** | `user_id`, `platform`(ios/android)、`push_token`、`updated_at` | 按 user_id 取該用戶所有設備 | KV,按 user_id |
| **delivery_log**(大頭) | `notif_id PK`, `user_id`, `channel`, `status`(狀態機)、`provider_msgid`、`attempts`、`ts` | 按 notif_id 點查(回執回填)、按 user_id+time 查歷史、批量寫 | **寬列(Cassandra)/ KV**,可 TTL |

**選型(決策樹錨 `system-design/06 存儲選型`):**

> **delivery_log 是寫密集 + 海量 + 可過期 + 按 key 點查**——這是寬列存儲(Cassandra)或可 TTL 的 KV 的完美場景,別用單庫 MySQL(1 TB/天寫不動)。template/preference 訪問模式是點查 + 讀多寫少,KV + 緩存即可。

**分片鍵(`01` Step 4 鐵律,錨 `system-design/07 數據規模化`):**

> - **delivery_log 按 `notif_id` 分片**(回執回填要按 notif_id 精確更新狀態,單片命中);需要按用戶查歷史時,另建 `user_id + ts` 的二級索引/物化視圖。
> - **user_preference / device_token 按 `user_id` 分片**——扇出時「解析這批收件人的偏好和設備」是按 user_id 批量取,落在對應片。
> - **熱點考量:** 不像 Feed 有「大 V 熱點分片」——通知的收件人是**離散的一大批普通用戶**,按 user_id 哈希天然均勻,**沒有單片熱點**。唯一的熱點是「同一個營銷事件」本身,但那是**處理側的扇出熱點**(一個事件展開成千萬條),用扇出 Worker 並行展開解決,不是存儲分片問題。

## 4. 高層架構(`01` Step 5:沿一個請求一鏡到底)

**先畫能跑的最簡版,再按需加件**(`01` 鐵律)。這題的骨架是**「事件 → 扇出 → 渠道隊列 → provider 適配 → 回執」**的五段流水線:

```
                          ┌─────────────────────────────────────────────────────┐
  上游業務                │                  通知系統                            │
 (訂單/風控/社交/營銷)    │                                                     │
       │ POST /notifications (202)                                              │
       ▼                  │                                                     │
   ┌─────────┐  入隊      │   ┌──────────────┐                                  │
   │ Notif   │──事件──────┼──►│ 事件 MQ      │ (削峰第一道閘:尖峰先頂在這)    │
   │ API GW  │  (冪等校驗)│   │ Kafka        │                                  │
   └─────────┘            │   └──────┬───────┘                                  │
                          │          ▼                                          │
                          │   ┌──────────────┐  ① 收件人解析(segment→user 列表)│
                          │   │ 扇出 Fan-out │  ② 查用戶偏好/退訂(preference)  │
                          │   │ Worker       │  ③ 渠道路由(該用戶走哪些渠道)   │
                          │   │              │  ④ 模板渲染(templateId+vars)    │
                          │   │              │  ⑤ 去重 + 限頻 check(見 DD②)    │
                          │   └──────┬───────┘                                  │
                          │          │ 展開成「一用戶一渠道」的投遞任務          │
                          │   ┌──────┴──────┬──────────┬──────────┐             │
                          │   ▼             ▼          ▼          ▼             │
                          │ ┌──────┐  ┌──────────┐ ┌──────┐ ┌──────────┐       │
                          │ │push  │  │ sms 隊列  │ │email │ │ in-app   │       │
                          │ │隊列   │  │(優先級)  │ │隊列   │ │ (站內信) │       │
                          │ └──┬───┘  └────┬─────┘ └──┬───┘ └────┬─────┘       │
                          │    │ 按provider│限速消費   │          │             │
                          │    ▼           ▼          ▼          ▼             │
                          │ ┌──────────────────────────────────┐               │
                          │ │   渠道適配層(provider adapter)   │ ← 統一抽象    │
                          │ │  APNs/FCM | Twilio | SES/SGrid    │   重試/熔斷/  │
                          │ └────────┬─────────────────────────┘   降級        │
                          │          │ 寫 delivery_log(pending→sent)          │
                          └──────────┼─────────────────────────────────────────┘
                                     ▼
                          ┌─────────────────────┐
                          │  第三方網關(外部)   │ ──► 用戶設備/手機/郵箱
                          │  APNs/FCM/SMS/SMTP   │
                          └─────────┬───────────┘
                                    │ 異步回執 webhook(delivered/bounced/failed)
                                    ▼
                          ┌─────────────────────┐
                          │ 回執處理 Worker      │ ──► 更新 delivery_log 狀態機
                          └─────────────────────┘
```

**一條典型路徑走一遍(`01` Step 5:這題重點是寫/扇出路徑,讀路徑就是查狀態):**

- **上游發通知(寫 + 扇出,熱路徑):** 業務 → API GW(**冪等鍵校驗**,重發直接返回老 requestId)→ 事件入 Kafka(**削峰第一閘**,202 立刻返回)→ 扇出 Worker 消費:**解析收件人 → 查偏好/退訂 → 渠道路由 → 模板渲染 → 去重限頻 check** → 展開成「一用戶一渠道」的投遞任務,**按渠道進不同隊列** → 渠道隊列**按 provider 配額限速消費** → 渠道適配層調第三方網關,寫 delivery_log(pending→sent)。
- **回執回填(異步,閉環):** 第三方網關投遞後,**異步回 webhook**(delivered / bounced / failed)→ 回執 Worker 消費 → 更新 delivery_log 狀態機 → 失敗的觸發重試或降級到備用渠道。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **事件 MQ(Kafka)** | 扇出必須異步 + 尖峰要削峰 | ①「尖峰」②「扇出放大」 |
| **扇出 Worker** | 一個事件展開成千萬條,要並行展開 | ②「扇出放大千萬倍」 |
| **按渠道分隊列** | SMS/email 網關吞吐天差地別,要分別限速 | ③「第三方網關是瓶頸」 |
| **渠道適配層** | 四套協議統一抽象 + 集中做重試/熔斷/降級 | 韌性(deep dive ①) |
| **去重/限頻 check** | 防止同一通知多觸發 + 轟炸用戶 | 澄清「不能轟炸/不能重」 |
| **回執 Worker + 狀態機** | at-least-once 要追蹤投遞、失敗要重試 | 澄清「投遞狀態追蹤」 |

**注意這張圖的「克制」:** 沒有強一致事務(通知不是錢,at-least-once 夠)、沒有複雜的讀路徑(讀就是查狀態,不是熱點)。**通知的複雜度全在「寫 + 扇出 + 對外部的韌性」一側**——把架構重量壓在這一側,讀側保持樸素,就是這題的正確姿勢。

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · 扇出架構 + 渠道適配層(這題的骨架 + 韌性命門)

扇出本身和 `08 News Feed` 同母題——**一個事件展開給很多收件人**(Feed 是寫進時間線,通知是生成投遞任務),這部分**錨 `08`,不重講扇出 Worker 怎麼並行**。通知**獨有、且最會被追問**的是後半段:**展開後,怎麼把任務可靠地交給一個我不擁有的、會掛的第三方網關。**

**先看完整的扇出展開鏈(事件 → 一用戶一渠道的投遞任務):**

```
事件:{templateId, segment/recipients, vars, priority}
  │
  ① 收件人解析:segment 查詢 → user 列表(百萬級,分批拉,別一次撈進內存)
  │     └─ 大 segment 用游標分批,每批丟回隊列(避免單 Worker 撐爆)
  ② 偏好/退訂過濾:批量查 preference → 砍掉「關了這渠道 / 已退訂 / 在勿擾時段」的
  ③ 渠道路由:剩下的用戶,按偏好決定走哪些渠道(可能一用戶多渠道)
  ④ 模板渲染:templateId + 用戶 locale → 取對應渠道模板 → 變量替換 → 成品文案
  ⑤ 去重/限頻 check(見 DD②)
  │
  ▼ 展開成 N 條「{user, channel, content, notif_id}」投遞任務 → 按 channel 入隊
```

**渠道適配層(provider adapter)——這題的設計核心,直接吃 `system-design/01 韌性`:**

四個渠道、可能多個 provider(SMS 可能同時接 Twilio + 阿里雲短信做主備),但對上層必須是**一個統一接口**:

```
interface ChannelProvider {
    send(task) → providerMsgId | error
    supports(channel) → bool
}
```

**為什麼要這層抽象(別小看,這是架構分):** 上層的扇出 Worker **不該知道** APNs 的 token 怎麼刷、Twilio 的 API 長什麼樣。把每個第三方封裝成一個 adapter,**新增渠道/換 provider 只動一個 adapter,不碰扇出邏輯**——這就是「依賴倒置」在系統設計題裡的落地。

**黑盒裡發生什麼——對「不可靠第三方」的四道韌性防線(這是 deep dive 的肉,逐層遞進):**

| 防線 | 機制 | 黑盒裡發生什麼 | 錨 |
|---|---|---|---|
| **① 限速(rate limit)** | 渠道隊列按 provider 配額**令牌桶限速消費** | 第 2 步算過 SMS 網關只給 500/s——消費端拿不到令牌就**等**,寧可隊列堆積也不超發(超發會被 provider 限流甚至封號) | `04 限流器`(令牌桶) |
| **② 重試(retry)** | 投遞失敗**指數退避 + 抖動重試**,有上限 | 網關 5xx/超時 → 不立即重投(會雪上加霜),**退避 1s→2s→4s…+ 隨機抖動**(防重試風暴同步);重試 N 次仍失敗才標 failed | `system-design/01 韌性` |
| **③ 熔斷(circuit break)** | 某 provider 連續失敗率超閾值 → **熔斷,快速失敗** | Twilio 掛了,不要每條都傻等超時(把 Worker 全卡死)→ 熔斷器跳開,後續直接快速失敗,**保護自己的 Worker 不被外部拖垮** | `system-design/01 韌性` |
| **④ 降級(fallback)** | 主 provider 熔斷 → **切備用 provider 或降級渠道** | Twilio 熔斷 → 切阿里雲短信;整個 SMS 都不行且是重要通知 → **降級到 email/push**;營銷類不重要 → **直接丟棄**(別硬投) | `system-design/01 韌性` |

> **面試話術(把韌性攤開,這是這題的最高加分點):** 「通知的最後一公里是第三方網關,我不擁有它,所以渠道適配層全程在做四件事:**限速**(按 provider 配額令牌桶消費,寧可堆隊列不超發,超發會被封號)、**退避重試 + 抖動**(防重試風暴)、**熔斷**(某 provider 連續失敗就跳開、快速失敗,保護我的 Worker 不被它拖死)、**降級**(主網關掛了切備用 provider,重要通知降級到其他渠道,營銷類直接丟)。這四道防線的細節我錨 `system-design/01 韌性`,但**核心思想是:把『依賴一個會掛的外部系統』當成第一假設來設計,而不是當異常處理。**」

**為什麼分渠道隊列、不混在一個隊列(常被追問):** 四個渠道的吞吐、延遲、失敗模式完全不同——SMS 慢(500/s)、push 快(數千/s)。**混在一個隊列,SMS 的慢會堵住 push 的快**(隊頭阻塞)。分開後各自按自己的 provider 配額消費,**互不拖累**,且某渠道網關掛了只熔斷那一條隊列,其他照常。

### Deep Dive ② · 去重 / 限頻 / 優先級(這題獨有的「為用戶做減法」)

別的扇出題都在「怎麼把消息送出去」做加法,通知題**一半的工程在做減法**——**不要把用戶轟炸到退訂**。這是 deep dive 最能展示產品 sense 的一段。

**(a) 去重(同一通知多次觸發):**

上游可能因為 bug/重試/MQ at-least-once,**對同一個業務事件發多次**(訂單狀態變更觸發兩次「已發貨」)。要保證**同一個 (user, 業務事件) 只投一次**:

| 層 | 機制 | 黑盒裡發生什麼 |
|---|---|---|
| **入口冪等鍵** | API 層按 `idempotencyKey` 去重,重發返回老 requestId | 上游重試/網絡抖動的重發,在門口就擋掉 |
| **投遞級去重鍵** | 展開後每條任務算 `dedup_key = hash(user, eventId, channel)`,投遞前 `SETNX` 到 Redis(短 TTL) | 同一用戶同一事件同渠道,第二次 `SETNX` 失敗 → 跳過。**用存儲原子性兜底,不是先查再寫**(避免 TOCTOU,同 `03` deep dive ① 的教訓) |

**(b) 限頻(per-user rate limit,防騷擾):**

去重防的是「同一條重複」,限頻防的是「不同條但太頻繁」——一個用戶一天收到 50 條營銷推送,他會退訂甚至卸載 App。在**用戶維度**限頻:

| 限頻維度 | 例 | 機制 |
|---|---|---|
| **每用戶每渠道頻率** | 同一用戶 push 每小時 ≤ 3 條、每天 ≤ 10 條 | 滑動窗 / 令牌桶,key = `(user, channel)`,錨 `04 限流器` |
| **勿擾時段(quiet hours)** | 用戶設了 22:00–08:00 不打擾 | 偏好裡帶 quiet_hours,扇出時 ④ 過濾掉(或延遲到窗口外) |
| **類別頻控** | 營銷類每天 ≤ 2 條,交易類不限 | 按 `priority/category` 分別限,**交易型(驗證碼)永不被限頻砍掉** |

> **關鍵 tradeoff:限頻只能砍營銷,不能砍交易。** 驗證碼/安全告警/支付通知是**用戶等著要的**,被限頻誤殺是事故。所以限頻**必須按優先級分桶**——交易型直通,營銷型才進頻控漏斗。這條「分優先級做減法」是這段的靈魂。

**(c) 優先級隊列(高優插隊 + 合併摘要):**

| 機制 | 怎麼做 | 為什麼 |
|---|---|---|
| **優先級隊列** | 驗證碼走**獨立高優隊列**,和營銷物理隔離;高優隊列**永遠先消費** | 0 點營銷群發砸下千萬條,驗證碼**絕不能排在它後面等幾分鐘**——交易型秒級 SLA 靠物理隔離保證,不靠「同隊列裡標個 priority 字段」(那在隊列堆積時沒用) |
| **合併 / 摘要(digest)** | 同類低優通知**攢一批合併成一條**(「你有 5 條新評論」而非 5 條推送) | 既降扇出量、又改善體驗——把「轟炸」變成「摘要」。營銷/社交互動類適合,交易類不合併 |

> **面試話術:** 「通知系統一半在做減法。**去重**靠冪等鍵 + 投遞級 `dedup_key SETNX` 兜原子性;**限頻**在 (user, channel) 維度用滑動窗,但**只砍營銷不砍交易**——驗證碼被限頻誤殺是事故;**優先級**靠**物理隔離的高優隊列**保證驗證碼不被千萬條營銷堵在後面,而不是同隊列標個字段;低優的還能**合併成摘要**,把轟炸變成『你有 N 條新消息』。這套減法的目標只有一個:**別讓用戶退訂。**」

### Deep Dive ③ · 可靠投遞(at-least-once + 冪等 + 投遞狀態機 + 回執)

通知要**「at-least-once + 冪等去重」**——這是 `financial-consistency` 的肌肉,但**約束比錢弱一級**:錢要 exactly-once(多扣一分都是事故),通知**寧可重不可漏**(交易型),且靠去重把「重」收斂掉。

**為什麼是 at-least-once 不是 exactly-once(主動暴露,顯得懂邊界):** exactly-once 跨「我的系統 + 第三方網關」邊界**根本做不到**——你調 APNs 後超時,你不知道它到底投沒投(對方收沒收回執也可能丟)。所以工程上的選擇是:**at-least-once(寧可重投)+ 客戶端/去重鍵把重複收斂**。重發一條 push 體驗上可接受,漏發一條驗證碼不可接受——所以偏向「寧可重」。

**投遞狀態機(每條投遞任務的生命週期):**

```
   created ──► pending ──► sent ──┬──► delivered   (網關回執:成功送達)
      │           │         │     ├──► bounced     (回執:無效設備/號碼 → 不重試,清理 token)
   (展開)     (入隊)    (調網關)   └──► failed      (回執/超時:可重試)
                                          │
                                          ▼
                                    retry(退避)──► sent ... (重試 N 次)
                                          │ 超上限
                                          ▼
                                     dead(進死信,降級/告警/丟棄)
```

| 狀態 | 含義 | 轉移觸發 |
|---|---|---|
| `pending` | 已展開、在渠道隊列等消費 | 入隊 |
| `sent` | 已調網關、等回執 | provider adapter 調用返回 accepted |
| `delivered` | 網關回執確認送達 | webhook 回執 |
| `bounced` | 無效目標(死 token/錯號碼) | 回執——**不重試,要清理該 token**(deep dive 細節:bounce 不清理會一直白投) |
| `failed` | 投遞失敗 | 回執/超時——**可重試** |
| `dead` | 重試耗盡 | 進死信隊列,降級/告警 |

**冪等鍵怎麼保證重投不重複(核心機制):** 每條投遞任務帶**全局唯一 `notif_id`**(冪等鍵)。MQ at-least-once 會重投同一條 → 消費時先查 delivery_log:**已是 sent/delivered 的,直接跳過,不再調網關**。這就是「**消費端冪等**」——用 `notif_id` 做去重鍵,把 at-least-once 的「重」吃掉。錨 `financial-consistency` 的冪等消費模式,**但這裡不需要事務,只需要狀態查驗 + 條件更新**(`UPDATE ... WHERE status='pending'`)。

**回執處理(閉環的最後一段,常被忽略):**

| 回執類型 | 處理 | 為什麼重要 |
|---|---|---|
| **delivered** | 狀態置 delivered,完成 | 投遞成功率報表 |
| **bounced**(死號/死 token) | 狀態 bounced + **清理該 device_token / 標記號碼無效** | **不清理會一直往死號投**,浪費配額 + 拉低送達率,APNs 甚至會懲罰你 |
| **失敗** | 觸發重試 / 降級 | 閉環重投 |
| **回執本身丟了** | 靠超時兜底(sent 超時未收回執 → 當失敗重試) | 回執也是 at-least-once,不能假設一定收到 |

> **面試話術:** 「投遞我用 **at-least-once + 冪等**,不追 exactly-once——因為跨第三方網關的邊界 exactly-once 做不到(調完超時你不知道投沒投)。每條投遞帶唯一 `notif_id` 當冪等鍵,MQ 重投時消費端先查 delivery_log 狀態,**已 sent 的直接跳過**,把『重』吃掉。投遞走一個**狀態機** pending→sent→delivered/bounced/failed,失敗的退避重試、耗盡進死信。**特別注意 bounce**:死號/死 token 的回執必須清理 token,否則一直白投還會被 APNs 懲罰。整套錨 `financial-consistency` 的冪等消費,但約束弱一級——通知寧可重不可漏,靠去重收斂,不需要事務。」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:錢花在「第三方網關」和「狀態存儲」,不在你的機器

通知系統的成本結構**很反直覺**——大頭不是你的算力:

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **第三方網關費(最大頭)** | SMS **每條真金白銀**(幾分錢~幾毛),email/push 便宜或免費 | **SMS 是成本黑洞**:能用 push/email 觸達的**絕不發 SMS**;按渠道成本做路由(免費渠道優先,SMS 兜底);砍掉無效投遞(bounce 清理)直接省錢 |
| **狀態存儲** | delivery_log 1 TB/天且只增 | TTL 過期(投出去 N 天後只留聚合報表,明細歸檔冷存儲) |
| **計算** | 扇出 Worker + 隊列,峰值 3 萬/s | 幾十台機器,相對便宜;按優先級彈性擴(營銷尖峰臨時加 Worker) |

> **這題最重要的成本 insight:你的架構優化(省幾台機器)在第三方網關費面前是零頭。** 真正省錢的決策是**渠道路由策略**:同樣觸達一個用戶,push 免費、SMS 一條幾毛——一次千萬級群發,「能 push 就不 SMS」省的是六位數。架構師在這題的價值是**把成本意識編進渠道路由**,不是調隊列參數。

### 容量規劃:瓶頸是「外部配額」,擴容要先擴配額

短鏈的容量瓶頸是自己的存儲;通知的容量瓶頸**在系統之外**——SMS/email 網關的賬號配額。這意味著:

- **擴容不是加機器,是談配額 / 加 provider。** 你的 Worker 再多,SMS 網關只給 500/s 就是 500/s。要提速得**找 provider 提配額**,或**多 provider 並行**(Twilio + 阿里雲同時發,配額相加)。
- **預留尖峰配額。** 0 點群發前,要提前和 SMS provider 報備峰值,否則被它限流。**容量規劃要把外部依賴的配額算進去**——這是純內部系統不會遇到的維度。
- **隊列堆積是常態,不是異常。** 因為投遞速度被外部封頂,尖峰時隊列**本來就會堆**(這是削峰的正常工作),監控要區分「健康堆積(會慢慢消化)」和「異常堆積(網關掛了消化不掉)」——靠**消費速率 + 隊列年齡**判斷,不是只看隊列長度。

### 演進路徑:從 MVP 到平台

```
階段 1(MVP):一個服務 + 一個隊列 + 直連各 provider
   通知量小,不分渠道隊列、不做熔斷,先跑起來
        │  量漲、營銷尖峰開始堵塞、SMS 堵住 push
        ▼
階段 2:事件 MQ 削峰 + 分渠道隊列 + 渠道適配層(重試/熔斷/降級)
   優先級隊列隔離交易型;去重/限頻防轟炸
        │  業務方越來越多、都要自己接 provider、模板散亂
        ▼
階段 3:通知平台化(notification-as-a-platform)
   統一模板管理 + 偏好中心 + 多租戶配額隔離 + 投遞分析報表
   上游只管「發給誰、用哪個模板」,渠道/合規/韌性全平台兜
```

> **這條演進的關鍵 insight:** 通知系統的終局是**平台**——它的價值是「**把『可靠觸達用戶』這件難事,從每個業務方手裡收上來統一做**」。MVP 階段別上來就建平台(`01` 紅旗「過度設計」),但要知道終局在這:**當第三個業務方來接通知時,就該把渠道適配/合規/偏好抽成平台,而不是讓每個業務各連一遍 Twilio。** 講得出「什麼時候從工具升級成平台」,就過了架構師線。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **某第三方網關掛**(Twilio 全掛) | 該渠道投遞全失敗 | **熔斷 + 切備用 provider + 降級渠道**(DD①);這是設計時的第一假設,不是異常——通知系統**必須假設網關會掛** |
| **扇出 Worker 跟不上尖峰** | 隊列堆積、通知延遲 | MQ 是緩衝(削峰本職);按優先級彈性擴 Worker;**交易型走獨立隊列不受營銷堆積影響** |
| **去重/限頻的 Redis 掛** | 去重失效 → 可能重複投遞;限頻失效 → 可能轟炸 | 去重失效降級成**寧可漏一次去重也別停發**(交易型不能停);限頻失效要**fail-safe 偏保守**(寧可少發,別轟炸)——這兩個降級方向相反,要分清 |
| **回執 webhook 丟失** | delivery_log 停在 sent,狀態不準 | **超時兜底**:sent 超 N 分鐘無回執當失敗重試;回執本身 at-least-once |
| **營銷群發踩到 bug 無限扇出** | 把用戶轟炸 + 燒光 SMS 配額 | **全局熔斷/開關**:單事件扇出量超閾值告警/暫停;**配額硬上限**保護(燒錢類渠道尤其) |

> **架構師話術:** 「我的失效模型和純內部系統不同——**最大的單點是第三方網關,而它根本不歸我管**。所以熔斷/降級不是錦上添花,是設計的第一假設:任何 provider 我都當它隨時會掛,掛了切備用/降級渠道。第二個要小心的是**去重和限頻的降級方向相反**:去重掛了寧可重複別停發(交易型停了是事故),限頻掛了寧可少發別轟炸——同一個 Redis 掛掉,兩個下游要按相反方向 fail-safe。最後,**燒錢渠道(SMS)要有配額硬上限**,防止一個扇出 bug 把預算燒光、把用戶轟炸到退訂。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **小應用、只發一種通知**(只 email 驗證碼) | 別上 MQ + 扇出 + 渠道適配 + 平台 | 直連一個 email provider + 同步發 + 簡單重試,別過度設計 |
| **強實時雙向通訊**(聊天) | 別用「異步扇出 + 第三方推送」 | 那是 `09 IM`——長連接、在線狀態、已讀回執,通知是**單向 best-effort**,IM 是**雙向有序** |
| **金融交易級「必達且僅一次」** | 別用「at-least-once + 去重收斂」 | 那要 exactly-once + 對賬,走 `14 支付`/`financial-consistency` 的強一致路子;但注意:**對外部網關仍做不到真 exactly-once**,只能在自己邊界內保證 |
| **內部系統的低頻告警**(運維群機器人) | 別建通知平台 | 一個 webhook 推 Slack/釘釘即可,殺雞用牛刀 |

> **這張表是架構師線的精華**——「知道一個方案的邊界、知道何時它是錯的」,比「知道這個方案怎麼做」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。尤其要分清**通知 vs IM vs 支付**這三條相鄰的線:通知是**單向 best-effort 多渠道扇出**,IM 是**雙向有序長連接**,支付是**強一致冪等**——三者共享「扇出/可靠投遞」的零件,但 SLA 約束差一個量級。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 通知系統的容量估算結論一句話? | 真矛盾**不是穩態 QPS**(放大後峰值才 3 萬/s),是三件事:**扇出放大**(一事件→千萬條投遞,必須異步過 MQ)、**尖峰削峰**(0 點群發瞬砸,靠 MQ + 限速消費)、**第三方網關是瓶頸**(SMS 只給幾百/s,投遞速度被外部配額封頂)。 |
| 2 | 為什麼通知 API 只回 202、投遞異步? | 一個事件扇出到千萬條 provider 調用,同步等不可能(`02` ② 算過放大千萬倍)。202 + requestId,後續查狀態。 |
| 3 | 渠道適配層對「不可靠第三方」的四道韌性防線? | **限速**(按 provider 配額令牌桶,寧堆隊列不超發,超發被封號)、**退避重試 + 抖動**(防重試風暴)、**熔斷**(某 provider 連續失敗跳開、快速失敗,保護自己 Worker)、**降級**(切備用 provider / 降級渠道 / 營銷直接丟)。錨 `system-design/01 韌性`。 |
| 4 | 為什麼分渠道隊列、不混一個? | 各渠道吞吐天差地別(SMS 500/s vs push 數千/s),混隊列會**隊頭阻塞**(慢 SMS 堵住快 push);分開後各按 provider 配額消費、互不拖累,某網關掛了只熔斷那條隊列。 |
| 5 | 去重 vs 限頻 vs 優先級,各防什麼、怎麼做? | **去重**防「同一條重複」:冪等鍵 + 投遞級 `dedup_key SETNX`。**限頻**防「太頻繁轟炸」:(user,channel) 滑動窗,**只砍營銷不砍交易**。**優先級**保「驗證碼不被營銷堵」:**物理隔離高優隊列**(非同隊列標字段),低優可**合併成摘要**。 |
| 6 | 限頻為什麼必須按優先級分桶? | 驗證碼/安全告警是**用戶等著要的交易型**,被限頻誤殺是事故。所以交易型直通、永不被限頻砍,只有營銷型才進頻控漏斗。 |
| 7 | 投遞為什麼是 at-least-once 不是 exactly-once?怎麼吃掉「重」? | 跨第三方網關邊界 exactly-once 做不到(調完超時不知投沒投)。選 **at-least-once(寧可重不可漏)+ `notif_id` 冪等鍵**:消費端先查 delivery_log,**已 sent 的跳過**,把重收斂掉。錨 `financial-consistency`,但弱一級、不需事務。 |
| 8 | bounce 回執為什麼要特殊處理? | 死號/死 token 的 bounce **必須清理 token / 標記號碼無效**——不清理會一直往死號白投,浪費配額 + 拉低送達率,APNs 還會懲罰你。 |
| 9 | 通知系統的成本大頭在哪?架構師怎麼省? | 大頭是**第三方網關費**(SMS 每條真金白銀),不是你的機器。省錢靠**渠道路由策略**:能 push/email 觸達的絕不發 SMS;砍 bounce 無效投遞。架構優化省的機器費是零頭。 |
| 10 | 通知 vs IM vs 支付,怎麼區分? | 通知=**單向 best-effort 多渠道扇出**(at-least-once);IM=**雙向有序長連接**(`09`);支付=**強一致冪等對賬**(`14`)。共享扇出/可靠投遞零件,但 SLA 約束差一個量級。 |

## 一句話總綱(背起來)

> **通知是「扇出三兄弟」收官題,靈魂是『我不擁有投遞的最後一公里』——第三方網關會限流會掛,所以一半設計在處理不可靠外部依賴。估算的真矛盾是扇出放大(一事件→千萬條,必異步過 MQ)、尖峰削峰、外部網關封頂(SMS 幾百/s)。三根支柱:渠道適配層(統一抽象 + 限速/退避重試/熔斷/降級四道韌性,錨 `01 韌性`)、去重限頻優先級(冪等鍵去重 + per-user 限頻只砍營銷 + 物理隔離高優隊列保驗證碼,做減法別轟炸用戶)、可靠投遞(at-least-once + `notif_id` 冪等消費 + 投遞狀態機 + bounce 清理,錨 `financial-consistency` 弱一級)。架構師線:成本大頭是 SMS 網關費(靠渠道路由省,不是調機器)、容量瓶頸在外部配額(擴容是談配額/多 provider)、終局是平台、去重和限頻掛了要按相反方向 fail-safe。扇出本身錨 `08`,不重講。**
