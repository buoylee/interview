# 17 · 視頻點播(VOD)/ 直播(Live)

> **T3 高階,「帶寬怪獸」題。** 前面 16 題的瓶頸幾乎都在 QPS、存儲或一致性;這題的瓶頸是一個你之前所有題加起來都沒摸到的維度——**出口帶寬**。`02` Drill 5 已經給過你那個讓人窒息的數:100 萬人同看一場 5 Mbps 直播 = **5 Tbps** 出口。單台機器、單個機房,連這個數的零頭都摸不到。所以這題從第一秒起,答案的骨架就被一個物理事實鎖死:**內容必須離開你的源站,推到全球幾千個 CDN 邊緣節點去吐。**
>
> 這題的教學價值集中在三個它獨有的子系統:**轉碼流水線**(一個原片怎麼變成一堆能自適應的小切片)、**CDN 分發與回源收斂**(百萬觀眾的帶寬怎麼在邊緣被吸收、回源怎麼不被打爆)、**直播低延遲**(推流到觀眾屏幕的端到端延遲,以及 HLS 切片固有延遲 vs WebRTC 的取捨)。把這三個鑽透,VOD 和 Live 兩種模式就都通了。
>
> 用 `00` 的依賴表看:視頻/直播吃 **B(存儲,●●)+ F(容災/CDN)**——它不是一致性題(沒人在乎你看到的視頻幀和別人差幾秒),是一道**存儲規模 + 帶寬分發 + 媒體流水線**的題。別把它當交易題去堆強一致組件。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個視頻平台。使用者能**上傳視頻**、別人能**點播觀看**(VOD,像 YouTube/Netflix);進階要求支援**直播**(Live,像 Twitch/抖音直播),觀眾能近實時看到主播畫面。

**這題最大的澄清陷阱:VOD 和 Live 是兩道題,先逼面試官選或分。** 它倆共享「轉碼 + CDN 分發」的下半身,但**上半身完全不同**(VOD 是異步離線處理、Live 是實時流水線),延遲要求差兩個量級。上來先用 `01` Step 1 的「問三類」把它切開:

| 類 | 該問什麼 | VOD 的典型答案 | Live 的典型答案 |
|---|---|---|---|
| **功能** | 核心用例?轉碼/字幕/推薦做不做? | 上傳→轉碼→點播;**先不做**推薦、評論、字幕 | 推流→轉碼→分發→看;**先不做**連麥、禮物、彈幕持久化 |
| **非功能(SLA)** | 延遲容忍?可用性?一致性? | **首屏 <1–2s、卡頓率低**;轉碼可離線(分鐘級);最終一致即可 | **端到端延遲**是核心 KPI(普通直播 3–10s、互動場景要 <1s);高可用 |
| **規模 / 約束** | 上傳量?同看人數?碼率? | 日上傳 X 萬小時;熱門影片 PB 級存儲 | **單場峰值同看百萬**;主播碼率 5 Mbps;觀眾按網絡自適應 |

**面試話術(先切題,再框範圍):**
> 「我先確認一件事——VOD 和 Live 是兩套上半身:VOD 上傳完可以**慢慢離線轉碼**,Live 必須**實時轉碼分發**、延遲是 KPI。它倆共享下半身(多碼率切片 + CDN 分發)。我打算**先把 VOD 的流水線講透**,再講 Live 在實時這條線上多了什麼。規模上 VOD 我假設熱門庫到 PB 級;Live 單場峰值同看百萬、主播 5 Mbps。核心非功能:VOD 首屏 1–2 秒內、卡頓率低;Live 端到端延遲壓到個位數秒。這樣可以嗎?」

**這幾個澄清直接決定後面:**

- 「VOD 可離線轉碼」→ 第 5 步轉碼走**異步流水線**(上傳和轉碼解耦,MQ 驅動),deep dive ① 展開。
- 「同看百萬 + 5 Mbps」→ 第 2 步估算會撞出 **Tbps 級出口 → 必須 CDN**,這是全題的地基(`02` Drill 5)。
- 「觀眾按網絡自適應」→ 必須**多碼率轉碼 + ABR**(自適應碼率),deep dive ① 的「為什麼一個原片要轉成 5 個碼率」。
- 「Live 延遲是 KPI」→ deep dive ③ 的核心矛盾:**HLS 切片固有延遲 vs WebRTC/LL-HLS 低延遲**的取捨。

**常見陷阱:** 不切 VOD/Live 就混著答,結果轉碼流水線講一半被問「那直播怎麼辦」整個亂掉。✅ 先切再答,是這題澄清階段最值錢的一句話。

## 2. 容量估算(直接調用 `02` 的數字)

> 這題的估算 `02` 的 **Drill 5**(直播出口帶寬)和 **Drill 3**(上傳存儲)已經各算過一半——這裡合成完整版,並且**每步都接一句「所以架構上……」**(`01` Step 2 鐵律)。這題的估算結論不是「要不要加緩存」,而是**「單源在物理上不可能,CDN 是入場券」**。

用 `02` 小抄:**一天 ≈ 1e5 s、一分鐘 1080p ≈ 50–100 MB、千兆網卡 ≈ 125 MB/s、副本 ×3**。再背一個視頻題專屬數:**碼率 → 帶寬**,5 Mbps = 0.625 MB/s(8 bit = 1 byte)。

### ① 直播出口帶寬(這題的當頭一棒,`02` Drill 5)

```
100 萬人同看 × 5 Mbps = 5,000,000 Mbps = 5 Tbps = 625 GB/s
```
對照單千兆網卡 125 MB/s:**5 Tbps ÷ 1 Gbps ≈ 5000 張滿載網卡。**

> **所以:單源、單機房絕無可能——這不是「加幾台機器」能解的,是物理碾壓。出口帶寬必須由 CDN 邊緣節點分攤,源站只服務 CDN(幾百個節點回源),把「百萬觀眾」收斂成「幾百個回源連接」,放大比百萬→百。** 這句話是全題的地基,deep dive ② 展開「怎麼收斂」。

### ② VOD 存儲(`02` Drill 3 的視頻版)

假設日上傳 **5 萬小時**視頻(YouTube 量級的零頭),每小時 1080p 原片 ~**3 GB**(以 ~7 Mbps 算):

```
原片:5e4 小時 × 3 GB = 150 TB/天(裸原片)
轉碼放大:每個原片轉成 5 個碼率(240p~4K),總量 ≈ 原片 ×1.5~2
   → ~250 TB/天 的可播文件
× 副本 3(對象存儲多副本)≈ 750 TB/天
一年 ≈ 270 PB
```
> **所以:這是 PB → 數百 PB 級的怪獸,DB 想都別想——必走對象存儲(S3/GCS/Ceph),DB 只存 metadata(videoId、標題、各碼率的切片清單 URL)。** 存儲是這題的最大成本項,架構師深挖層會講冷熱分層。錨 `system-design/06 存儲選型`(blob 走對象存儲)。

### ③ 轉碼算力(容易被漏,但是 VOD 的隱性大頭)

轉碼是 **CPU/GPU 密集**——軟件轉一小時 1080p 可能要幾分鐘到幾十分鐘 CPU 時間,還要 ×5 個碼率:

```
5 萬小時/天上傳,每小時原片轉 5 碼率
若每小時原片總轉碼耗時 ~30 分鐘 CPU(攤平)
   → 5e4 × 0.5 = 2.5 萬 CPU·小時/天
   ÷ 24 ≈ ~1000 核常駐(且上傳有尖峰,要彈性)
```
> **所以:轉碼集群是一大塊算力成本,且上傳量有日內尖峰(晚上多)。這直接推出 deep dive ① 的兩個決策:轉碼必須異步(不能讓使用者等)+ 轉碼集群必須彈性(按隊列深度自動擴縮,尖峰擴、谷時縮)。**

### ④ 上傳帶寬(進口,別只算出口)

```
150 TB/天 ÷ 86400 ≈ 1.7 GB/s(平均上傳帶寬)
峰值 ×3 ≈ 5 GB/s → 必須多接入點 + 分塊並行上傳
```
> **所以:上傳也不是單機能扛的,要分塊上傳 + 多接入點,且大文件要斷點續傳(錨 `12 對象存儲/網盤`的分塊上傳——同一個母題)。**

### 一句話估算結論(面試一定要說出來)

> 「**這題被一個物理事實鎖死:百萬同看 ×5Mbps = 5 Tbps 出口,單源不可能,所以 CDN 是入場券不是選項。存儲是 PB→數百 PB 級怪獸,必走對象存儲、DB 只存 metadata。轉碼是隱性大頭——CPU 密集 ×5 碼率,逼出『異步 + 彈性集群』。我的設計重心三塊:轉碼流水線怎麼異步並行、CDN 怎麼把百萬收斂到幾百回源、直播延遲怎麼壓。**」

這句把估算焊到架構的總結,就是 `01` 說的資深 vs 初級的分水嶺——**估算直接證明了「不上 CDN 這題無解」,而不是拍腦袋說「要 CDN」。**

## 3. API + 數據模型

### API(`01` Step 3:快過,點到為止)

```
# VOD 上傳(分塊 + 異步轉碼)
POST /videos                  {title, meta}        → { videoId, uploadUrls[] }   # 拿到分塊上傳的預簽名 URL
PUT  <uploadUrl>              <chunk bytes>         → 206/200                     # 直傳對象存儲,不過業務服務器
POST /videos/{id}/complete    {parts[]}            → { status: "transcoding" }   # 通知合併 + 觸發轉碼

# VOD 播放(返回的是清單,不是視頻流本身)
GET  /videos/{id}/manifest    → master.m3u8 (HLS) 或 .mpd (DASH)                  # 多碼率清單,指向 CDN
                              # 播放器再按 ABR 自己去 CDN 拉 .ts/.m4s 切片

# Live 直播
POST /streams                 {title}              → { streamKey, ingestUrl }     # 拿推流地址 + 密鑰
# 主播用 OBS 等推流端: rtmp://ingest.../live/{streamKey}  (RTMP 推流,不是 HTTP)
GET  /streams/{id}/manifest   → live.m3u8 (滾動更新的播放清單) 或 WebRTC offer
```

**要點(這題 API 的兩個「反常識」信號,講出來顯細節功力):**

- **播放 API 返回的不是視頻字節,是一份「清單(manifest)」**——`.m3u8`(HLS)或 `.mpd`(DASH)。清單裡列出**各個碼率**和**每個碼率的切片序列 URL**。真正的視頻字節是播放器**直接去 CDN** 拉的,**根本不經過你的業務服務器**。這是視頻系統和普通 API 的本質區別:你的服務器只發「目錄」,CDN 發「貨」。
- **上傳走預簽名直傳對象存儲**——客戶端 `PUT` 分塊**直接進 S3/對象存儲**,不過你的業務服務器(否則 150 TB/天的上傳流量會把業務層撐爆)。業務服務器只發預簽名 URL + 收 complete 通知。錨 `12 對象存儲/網盤` 的同一手法。
- **推流不是 HTTP**——主播端用 **RTMP**(`rtmp://`)或 WebRTC 推,這是長連接推流協議,不是請求-響應。deep dive ③ 講為什麼。

### 數據模型(`01` Step 4:選型 + 分片鍵)

**關鍵認知:視頻本體(blob)和 metadata 分開存,這是這題數據模型的命根子。**

**(a) metadata(存 DB,小而結構化):**

| 字段 | 類型 | 說明 |
|---|---|---|
| `video_id` | bigint **PK** | 視頻唯一 ID(發號器,錨 `05`) |
| `uploader_id` | bigint | 上傳者(按它能查「某人的視頻」) |
| `title / desc` | text | 標題描述(可進 ES 做搜索,另一道題) |
| `status` | enum | `uploading/transcoding/ready/failed` |
| `duration` | int | 時長(秒) |
| `manifest_url` | text | master.m3u8 在 CDN/對象存儲的路徑 |
| `renditions` | json | 各碼率版本清單(240p/480p/720p/1080p/4K 的切片路徑) |
| `created_at` | timestamp | |

**(b) 視頻本體(存對象存儲,海量 blob):**

```
對象存儲布局(一個 video 一個前綴):
  /videos/{video_id}/master.m3u8           ← 主清單(列出所有碼率)
  /videos/{video_id}/1080p/index.m3u8      ← 1080p 的切片清單
  /videos/{video_id}/1080p/seg0.ts ...     ← 1080p 的切片(每片 2~10s)
  /videos/{video_id}/720p/...
  /videos/{video_id}/480p/...
```

**選型(決策樹錨 `system-design/06 存儲選型`):**

| 數據 | 存哪 | 理由 |
|---|---|---|
| **視頻切片(blob)** | **對象存儲**(S3/GCS/Ceph/MinIO) | PB→數百 PB 級;不可變、追加寫;天然多副本、按容量線性擴、單價極低。**DB 存不下也不該存** |
| **metadata** | **SQL 或寬列 KV** | 結構化、要按 uploader/video_id 查;量遠小於 blob(每 video 一行)。MySQL 起步,大了遷 Cassandra |
| **熱門影片的「熱數據」** | **CDN 邊緣 + 對象存儲熱層** | 訪問是冪律分布(20% 影片吃 80% 流量),熱的留 CDN/熱層 |
| 標題/描述全文搜索 | ES | 另一道題,別在這展開 |

> **面試話術:** 「視頻本體是 PB 級不可變 blob,完美匹配對象存儲——線性擴、便宜、多副本免費。DB 只存 metadata(每 video 一行)和指向切片的清單路徑。這個『blob / metadata 分離』是這題數據層的核心,和 `12 網盤` 是同一個母題。」

**分片鍵(`01` Step 4 鐵律,錨 `system-design/07`):**

> metadata 按 **`video_id`** 分片(點查為主、高基數均勻);對象存儲的分片由它自己的一致性哈希/分桶搞定,**不歸你操心**——這是用對象存儲的紅利。**這題沒有「大 V 熱點分片」問題**(對比 News Feed),因為熱點是在**讀流量(帶寬)**上,不在存儲分片上,而讀流量的熱點是 **CDN 解的**,不是分片解的。把這個區分講清楚,就是資深信號。

## 4. 高層架構(`01` Step 5:沿一個請求一鏡到底)

**這題要畫三條線:VOD 上傳→轉碼、VOD 播放、Live。** 先畫 VOD(它是地基),Live 在它上面改。

```
═══ VOD 上傳 + 轉碼流水線(異步,離線)═══════════════════════════

 主播/UP ──分塊直傳──► 對象存儲(原片 raw)
   │ POST /complete                    │ 上傳完成事件
   ▼                                    ▼
 Upload Svc ───────────► MQ(轉碼任務隊列)
                              │
                              ▼   彈性消費(按隊列深度擴縮)
                      ┌── 轉碼集群(Worker,CPU/GPU)──┐
   原片切成分片 → 並行轉成 N 個碼率 → 每碼率切成 HLS/DASH 小片 → 生成 manifest
                      └──────────────┬───────────────┘
                                     ▼
                         寫回對象存儲(各碼率切片 + master.m3u8)
                                     │ 回填 metadata.status=ready
                                     ▼
                                  DB(metadata)


═══ VOD 播放(熱路徑,百萬級,且帶寬怪獸)══════════════════════════

 Player ──GET manifest──► API GW ──► Play Svc ──► DB(取 manifest_url)
   │  ◄──── master.m3u8 ────────────────────────────┘
   │  (播放器解析:有 240p~4K 5 個碼率)
   │
   │  ABR 邏輯:測網速 → 選碼率 → 按序拉切片
   ▼
 Player ──GET seg0.ts──► ★CDN 邊緣節點★
                           │ hit  → 直接吐切片(99% 在這結束)
                           └ miss → 回源(中間層 → 源站對象存儲)→ 緩存 → 吐
   （後續切片同理,播放器邊播邊拉,網速變就切碼率）


═══ Live 直播(實時流水線)══════════════════════════════════════

 主播 OBS ──RTMP 推流──► Ingest(接流) ──► 實時轉碼(轉多碼率)
                                              │ 邊轉邊切成 2~6s 小片
                                              ▼
                                    切片 + 滾動 manifest 寫入 ──► 源站
                                              │
                            ┌─────────────────┴──────────────────┐
                            ▼ CDN 把切片分發到全球邊緣             │
 觀眾 Player ──GET live.m3u8(每幾秒重拉,拿新切片)──► CDN 邊緣 ──┘
   （HLS:固有幾秒延遲 / LL-HLS / WebRTC:亞秒,見 DD③）
```

**沿請求走一遍(`01` Step 5:讀 + 寫 + 實時各一條):**

- **VOD 上傳(寫,冷):** 客戶端分塊**直傳對象存儲**(不過業務層)→ complete 通知 Upload Svc → **發轉碼任務到 MQ**(關鍵:上傳和轉碼解耦,使用者上傳完即返回,不等轉碼)→ 轉碼集群彈性消費 → 切片寫回對象存儲 + manifest 生成 → metadata 標 ready。
- **VOD 播放(讀,熱):** 播放器先要 `manifest`(走你的業務層,輕)→ 拿到 5 個碼率的清單 → 之後**所有視頻切片直接從 CDN 拉,根本不碰你的服務器** → CDN 邊緣命中就吐,miss 才逐級回源。**百萬觀眾的帶寬全被 CDN 吸收。**
- **Live(實時):** 主播 RTMP 推流 → Ingest 接 → **實時轉碼**邊轉邊切小片 → 切片 + 滾動 manifest 推到源站 → CDN 分發到邊緣 → 觀眾每幾秒重拉 manifest 拿新切片。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **CDN 邊緣節點** | 5 Tbps 出口物理上單源無解 | ①「百萬→百回源收斂」 |
| **對象存儲(blob)** | PB→數百 PB 級,DB 存不下 | ②「PB 級必走對象存儲」 |
| **MQ + 彈性轉碼集群** | 轉碼 CPU 密集、有尖峰、不能堵上傳 | ③「異步 + 彈性」 |
| **分塊直傳對象存儲** | 150 TB/天上傳不能過業務層 | ④「上傳帶寬」 |
| **多碼率 + ABR** | 觀眾網絡千差萬別,要自適應 | ①「觀眾按網絡自適應」 |

**注意這張圖的「重心轉移」:** 和前 16 題不同,**這題的數據主體(視頻字節)根本不流經你畫的大部分框**——它從對象存儲經 CDN 直達播放器。你的業務服務器只處理「控制面」(metadata、清單、轉碼編排),「數據面」(GB/s 級的視頻字節)全外包給對象存儲 + CDN。**這個「控制面 / 數據面分離」是視頻架構和普通 CRUD 架構的本質差異**,講出來就過了資深線。

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · 上傳 + 轉碼流水線(VOD 的核心引擎)

第 2 步算出轉碼是 CPU 密集大頭、有尖峰——**這條流水線的每個決策都是被「為什麼異步、為什麼並行、為什麼彈性」逼出來的**。先看一個原片要經歷什麼:

```
原片(mezzanine,~3GB/小時)
  │ ① 切分(split):按時間切成 N 個獨立段(如每 30s 一段)
  ▼
  ┌── 段1 ──┐  ┌── 段2 ──┐  ... 並行!不同 Worker 同時轉
  │ ② 並行轉碼 each 段 → 轉成 5 個碼率(240p/480p/720p/1080p/4K)
  └─────────┘  └─────────┘
  │ ③ 每個碼率再切成 HLS/DASH 小片(seg, 2~10s 一片)
  ▼
  ④ 合併 + 生成 manifest(master.m3u8 列出 5 碼率,每碼率一份 index.m3u8 列切片)
  │
  ▼ 寫回對象存儲 → metadata.status = ready
```

**黑盒裡發生什麼——四個被追問的決策:**

**(1) 為什麼必須異步(上傳 ≠ 轉碼解耦)?**
轉碼一小時影片要幾分鐘到幾十分鐘。若同步——使用者上傳完要乾等到轉碼結束才返回,**體驗災難 + 業務線程被佔死**。所以上傳完只做兩件快事:存原片、**發一條轉碼任務到 MQ**,立刻返回「處理中」。轉碼在後台慢慢跑,完成後改 status + 推送通知。這是 `01` 紅旗「把慢操作放關鍵路徑」的反面——**MQ 把上傳和轉碼在時間上解耦**。

**(2) 為什麼要切段並行轉碼?**
一個 2 小時電影若整條串行轉,要等很久;**把它切成幾十段、發給幾十個 Worker 同時轉,牆鐘時間從幾十分鐘壓到幾分鐘**。轉碼天然可並行(段與段獨立),這是用「分治」換「延遲」。代價:要一個**編排器(orchestrator)**管「切段→分發→收齊→合併」,段全轉完才能拼 manifest——這引入了 fan-out / fan-in 協調(類似 MapReduce)。

**(3) 為什麼一個原片要轉成 5 個碼率(這是 ABR 的物質基礎)?**
觀眾網絡千差萬別:手機 4G 跳到電梯裡剩 240p 才不卡,家裡千兆能吃 4K。**你不知道觀眾是誰,所以全都轉出來**,讓播放器**自適應碼率(ABR)**——播放器邊播邊測網速,網速好拉高碼率切片、網速差**無縫切到低碼率**(因為每個碼率切片在時間上對齊,能逐片切換)。**多碼率轉碼是 ABR 的前提**:沒有多個碼率版本,播放器無從適應。這就是為什麼存儲要 ×1.5~2(②的賬)——**碼率階梯是用存儲換流暢度**。

**(4) 為什麼轉碼集群要彈性(按隊列深度擴縮)?**
上傳有日內尖峰(晚上多),轉碼算力按峰值常駐會浪費(谷時閒置),按均值配又會在尖峰積壓。**正解:Worker 消費 MQ,按隊列深度自動擴縮**——隊列堆積就擴 Worker(甚至開競價實例/Spot)、隊列空就縮。轉碼任務**冪等可重試**(段轉壞了重轉一段即可),所以**容忍 Spot 被回收**,成本能再砍一截。錨 `cloud-native-landscape/`(彈性成本環、按隊列驅動的自動擴縮)。

> **面試話術(把流水線講成一條因果鏈):** 「轉碼流水線的每個設計都是被算力逼出來的:轉碼慢且 CPU 密集,所以**異步**(上傳發 MQ 立刻返回);一個長片要快出片,所以**切段並行轉**(fan-out 給多 Worker,牆鐘從幾十分鐘到幾分鐘);觀眾網絡不可知,所以**轉 5 個碼率**喂 ABR;上傳有尖峰且任務冪等,所以集群**按隊列深度彈性擴縮 + 用 Spot**。產出是一份 master manifest + 一堆對齊的多碼率切片,全進對象存儲。」

### Deep Dive ② · CDN 分發與回源收斂(帶寬怪獸的解法)

第 2 步算出 5 Tbps——**CDN 不是「加速」,是「讓這題在物理上可行」的唯一辦法**。核心要講清兩件事:**邊緣怎麼吸收百萬帶寬**、**回源怎麼不被打爆**。

**(a) 三層回源樹:把百萬觀眾逐層收斂**

```
百萬觀眾  ─┐
          ├─► CDN 邊緣節點(幾千個,離用戶最近)   ← 99%+ 流量在這吐
          │      │ miss(本地沒這切片)
          │      ▼
          ├─► CDN 區域中間層(Mid-tier,幾百個)    ← 把同區域 miss 收斂
          │      │ miss
          │      ▼
          └─► 源站 / 對象存儲(Origin,你的)        ← 只剩幾百個連接回到這
```

**黑盒裡發生什麼——收斂比怎麼來的:**
一條熱門切片,**全球幾千個邊緣節點,每個節點對這個切片只回源一次**(第一個觀眾觸發,之後該節點本地緩存命中,後續觀眾全本地吐)。再加中間層把同區域多個邊緣的 miss 又收斂一次。所以**百萬觀眾 → 幾百個回源連接**,放大比百萬→百(`02` Drill 5 那句「源站只服務 CDN 節點」就是這個意思)。源站要扛的不是 5 Tbps,是**幾百個 CDN 節點各拉一份切片**——這個源站撐得住。

**(b) 為什麼視頻特別吃 CDN 的紅利:**

| 特性 | 為什麼利於緩存 |
|---|---|
| **切片不可變** | 一個 `seg5.ts` 生成後永不變 → 緩存一致性免費(對比動態 API 要操心失效) |
| **熱點冪律** | 20% 熱門影片吃 80% 流量 → 熱切片在邊緣命中率極高,長尾自然回源 |
| **大文件** | 切片是 MB 級大塊 → 攤薄了每請求開銷,帶寬利用率高 |

**(c) 直播的「源站放大」難題(Live 特有,最容易被追問):**

VOD 的切片是**早就轉好躺在對象存儲**裡的,CDN 慢慢預熱即可。但**直播的切片是「剛剛才生成」的**——主播此刻說的話,3 秒後才有對應切片。問題來了:

> **百萬觀眾幾乎同時來拉「最新那一片」,而這片剛生成、所有 CDN 邊緣都還沒有它 → 百萬請求瞬間全 miss → 同時回源 → 把源站打爆。** 這就是直播的「**驚群 / 源站放大(thundering herd / origin shielding)**」問題,VOD 不嚴重(切片是冷的、預熱過的)。

**怎麼收斂(這是 deep dive 的肉):**

| 手段 | 機制 | 黑盒裡發生什麼 |
|---|---|---|
| **回源請求合併(collapsed forwarding)** | 同一個邊緣節點上,同一切片的並發 miss **只發一個回源請求**,其餘請求等這一個的結果 | 一個邊緣節點對「最新切片」的 N 萬個並發請求,合併成**對源站的 1 個請求**,其餘 N 萬-1 個等著一起拿。這是壓平驚群的關鍵一招 |
| **回源屏蔽層(origin shield)** | 在源站前放一層固定的 shield 節點,所有邊緣的 miss 先打 shield,shield 再回源 | 把「幾千邊緣各回源一次」收斂成「shield 回源一次」,源站只面對 shield |
| **直播切片預推(push)** | 轉碼產出切片後,主動 push 到各區域中間層,不等觀眾來 pull 才回源 | 把「百萬同時 pull 觸發 miss」變成「切片提前到位、觀眾來就命中」,從拉模式變推模式 |
| **熱門 VOD 預熱(prewarm)** | 預判要火的內容(新劇集首播),提前推到邊緣 | 首播瞬間百萬湧入時邊緣已有貨,避免冷啟動回源洪峰 |

> **面試話術(主動點出 Live 的源站放大):** 「VOD 的 CDN 很省心——切片是冷的、預熱過的。**直播完全不同:最新切片是剛生成的,百萬觀眾幾乎同時來拉同一片,全 CDN 邊緣都還沒有它,瞬間百萬 miss 同時回源把源站打爆,這是直播的驚群問題。** 我用三招收斂:邊緣**回源請求合併**(同一切片的並發 miss 合成一個回源),加**回源屏蔽層**(所有邊緣 miss 先到 shield、shield 才回源),再對直播切片**主動 push** 到中間層而不是被動 pull。這樣百萬觀眾最終只在源站留下個位數的回源連接。」

### Deep Dive ③ · 直播鏈路與延遲(Live 的核心 KPI)

直播的端到端延遲(主播說話 → 觀眾聽到)是 KPI,要先把延遲**拆成幾段**,才知道優化哪裡:

```
主播說話
  │ ① 採集 + 編碼(主播端)        ~100ms~幾百ms
  │ ② RTMP 推流到 Ingest          ~網絡 RTT
  │ ③ 服務端轉碼(多碼率)         ~幾百ms~1s
  │ ④ 切片(等湊滿一個 segment)   ★ 固有延遲:切片時長 × 緩衝片數 ★
  │ ⑤ CDN 分發到邊緣              ~幾百ms
  │ ⑥ 播放器緩衝(防卡頓)         ★ 固有延遲:播放器預緩衝幾片 ★
  ▼
觀眾聽到
```

**黑盒裡發生什麼——延遲的大頭在 ④ 和 ⑥(切片機制本身):**

普通 HLS/DASH 的延遲大頭不是網絡,是**切片這個機制本身**:
- **④ 切片**:HLS 要**湊滿一個完整 segment**(比如 6s)才能生成那片給觀眾拉。主播說的第 1 秒,要等這 6s 片湊滿才發出去 → 天然欠 ~6s。
- **⑥ 播放器緩衝**:播放器為了防卡頓,通常**預下載 3 個切片**才開播(HLS 建議值)。3 × 6s = **18s 緩衝延遲**。

> **所以標準 HLS 直播延遲常達 ~15–30s**——不是網絡慢,是「切片 + 多片緩衝」這個為了**穩定**而設計的機制,固有地犧牲了延遲。**這是這題最反直覺、也最值錢的 insight:HLS 的延遲是它換穩定性的代價,不是 bug。**

**三條延遲路線的取捨(這是 deep dive 的肉,一張表講透):**

| 方案 | 延遲 | 機制 | 代價 / 何時用 |
|---|---|---|---|
| **標準 HLS/DASH** | ~10–30s | 切大片(6s)+ 播放器緩衝多片;走 HTTP/CDN | ✅ 最穩、最省、CDN 全兼容、海量觀眾。**普通直播(賽事、秀場)夠用**。延遲大但沒人在乎差幾秒 |
| **LL-HLS / LL-DASH**(低延遲版) | ~2–5s | 把片**切更小**(分塊 chunked,亞秒級 part)+ 邊生成邊推 + 減少緩衝 | ✅ 仍走 HTTP/CDN 生態(改造小)、延遲降一個量級。**帶貨、有輕互動的直播**。代價:切片碎、請求多、CDN 要支援 |
| **WebRTC** | **<1s(亞秒)** | 不切片!**UDP 實時流**、點對點/SFU 轉發,為實時通信而生 | ✅ 真互動(連麥、視頻會議、雲遊戲、拍賣)。代價:**不走 HTTP CDN 生態**,要自建 SFU/媒體服務器集群、扩展到百萬觀眾**極貴極難**(每路都是有狀態連接)。**只在「延遲是命」時用** |

**黑盒裡發生什麼——為什麼 WebRTC 低延遲但難規模化:**
HLS 走 **HTTP + CDN**,切片是**無狀態靜態文件**,CDN 隨便緩存隨便扇出,百萬觀眾只是多拉幾個文件——**規模化幾乎免費**。WebRTC 走 **UDP 實時連接**,每個觀眾是一條**有狀態的媒體連接**,要 SFU(選擇性轉發單元)維護、轉發、做擁塞控制——**百萬觀眾 = 百萬條有狀態連接,CDN 幫不上忙,要自建龐大媒體服務器集群**,成本和複雜度爆炸。所以業界默認:**規模直播用 HLS 換延遲,小規模強互動用 WebRTC 換延遲。**

> **面試話術(把延遲取捨攤成決策,`01` Step 7 加分核心):** 「直播延遲的大頭不是網絡,是**切片機制本身**:HLS 要湊滿一個 6s 片才發、播放器又要緩衝 3 片,固有就 ~15–30s。這是它換穩定和 CDN 兼容的代價。所以選型看延遲需求:**普通直播(賽事、秀場)用標準 HLS,延遲十幾秒沒人在乎、CDN 全兼容、便宜;帶貨/輕互動上 LL-HLS,切小片把延遲壓到 2–5s,還能複用 HTTP CDN 生態;只有真互動場景(連麥、拍賣、雲遊戲)才上 WebRTC 做到亞秒,但代價是脫離 CDN 生態、要自建 SFU 集群、百萬觀眾規模化極貴。我會默認 HLS,延遲是命時才為 WebRTC 的成本買單。** 」

這句「**默認 X,只有當……才為 Y 的代價買單**」就是 `00` 標準線裡「**擁有這個決策**」的架構師話術。

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:這題的錢是 CDN 帶寬 + 存儲 + 轉碼三座大山

視頻平台的成本結構和前 16 題完全不同——**CPU/QPS 幾乎不花錢,錢全在帶寬、存儲、轉碼**:

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **CDN 出口帶寬** | **通常是最大成本項**,按 GB 計費,熱門直播一場百萬人就是 PB 級出量 | 多 CDN 廠商**比價 + 智能調度**(按地區/時段選便宜的);自建邊緣節點(量大到一定規模自建比租划算);**P2P 輔助分發**(觀眾互相喂切片,卸載源站帶寬,直播大會場常用) |
| **存儲** | PB→數百 PB,且只增(視頻一般不刪),`02` 算的 270 PB/年 | **冷熱分層**:長尾老視頻(沒人看)下沉到**歸檔冷存儲**(Glacier 類,單價極低,取回慢但長尾本來也不急);熱門留熱層 + CDN。**老視頻可降碼率重轉**省存儲 |
| **轉碼算力** | CPU/GPU 密集,有尖峰 | **Spot/競價實例 + 按隊列彈性**(任務冪等容忍回收);**只轉需要的碼率**(冷門影片先轉 720p,火了再補 4K——按需轉碼省算力) |

> **架構師話術(成本是這題的主旋律):** 「視頻平台不是 CPU 題,是**帶寬 + 存儲 + 轉碼**三座成本大山。帶寬最貴,我會多 CDN 比價調度、量大了自建邊緣、直播加 P2P 卸載;存儲靠冷熱分層把長尾沉到歸檔;轉碼用 Spot + 按需轉碼(火了再補高碼率)。這題的架構決策本質都是**成本決策**——和交易題在乎一致性不同,視頻在乎的是每 GB 多少錢。」

### 容量規劃:按「峰值單場」而不是「平均」配

視頻流量極度尖峰——**一場世界盃決賽 / 頂流首播,單場同看可能是平時全平台的 10 倍**。容量規劃的關鍵不是算平均,是**算「最大單場 + 突發」**:CDN 容量要按峰值買(或用可彈性的 CDN)、源站回源要按「驚群最壞情況」配 shield、轉碼集群要能為大型直播臨時擴容。架構師要算「**最壞那一場的帶寬峰值**」,而不是日均出量。

### 演進路徑:從 MVP 到全球規模

```
階段 1(MVP):同步轉碼 + 單區對象存儲 + 租 CDN
   小量上傳,轉碼可以先串行;直接用商業 CDN(別自建)
        │  上傳量上來、轉碼積壓
        ▼
階段 2:異步轉碼流水線 + 彈性 Worker(MQ 驅動)+ 多碼率 ABR
   切段並行、Spot 集群;存儲冷熱分層
        │  帶寬成本爆炸 / 跨地域延遲
        ▼
階段 3:多 CDN 智能調度 + 回源屏蔽 + 直播 LL-HLS;熱門預熱
        │  帶寬成本到自建臨界點 / 要極致延遲
        ▼
階段 4:自建邊緣 CDN + P2P 輔助 + WebRTC 互動直播
```

> **這條演進的關鍵 insight:** 視頻平台**第一天就該租商業 CDN**(這是入場券,不是優化),但**自建 CDN 要等帶寬成本大到「自建比租便宜」的臨界點**才做——`01` 紅旗「上來就拆」的反面教材就是 MVP 階段自建 CDN。**講「現在租 CDN、量到 X PB/月再自建」比一上來自建更顯資深。**

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **CDN 邊緣節點掛 / 整個 CDN 廠商掛** | 該區域觀眾卡死/黑屏 | **多 CDN + 客戶端 failover**:播放器拉切片失敗能切到另一家 CDN 的域名;這是視頻可用性的命門。錨 `system-design/01 韌性`(多活/容錯) |
| **直播驚群打爆源站**(DD②) | 最新切片回源洪峰,源站雪崩、全場卡 | 回源請求合併 + origin shield + 切片預推(DD② 的三招就是防這個) |
| **轉碼集群積壓**(尖峰/Spot 大量回收) | VOD 上傳的影片遲遲不 ready(讀不受影響) | 任務冪等可重試 + 隊列削峰 + 保底用按需實例兜底;**轉碼是非實時的,積壓只是延遲出片,不影響已有影片播放** |
| **Live 轉碼/Ingest 掛** | 直播中斷(實時鏈路無法重來,比 VOD 嚴重) | Ingest 多副本熱備、主播端**自動重連 + 切備用推流地址**;直播鏈路要比 VOD 鏈路做更高可用 |
| **播放器持續卡頓** | 用戶流失(體驗 KPI) | ABR 自適應降碼率 + 播放器緩衝 + 邊緣就近;卡頓率是要監控的核心 SLO,錨 `observability/` |

> **架構師話術:** 「我的可用性命門是 **CDN**——單 CDN 廠商掛了整個區域黑屏,所以必須**多 CDN + 播放器端 failover**。直播比 VOD 脆弱:VOD 轉碼掛了只是出片延遲、已有影片照播;**直播是實時的,Ingest/轉碼掛了這一刻就斷了重來不了**,所以直播鏈路要做熱備 + 主播端自動重連切備用地址。VOD 的轉碼積壓反而最不慌——它非實時、冪等、可重試,積壓只是延遲出片。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **小規模 / 內部視頻**(企業培訓、幾百人看) | 別上多 CDN + 轉碼集群 + ABR | 一個對象存儲 + 一家 CDN + 單碼率甚至直接發原片,過度設計才是錯 |
| **要極致實時互動**(視頻會議、連麥、雲遊戲) | 別用 HLS 切片(延遲十幾秒不可接受) | 直接上 **WebRTC**,這是另一類「實時通信」題(SFU/MCU、擁塞控制),別硬套點播架構 |
| **短視頻 / Reels(秒級小視頻)** | 多碼率 ABR 收益小、首屏要極快 | 預加載下一條 + 單/雙碼率 + 激進預取;優化重心在**首幀速度和滑動預取**,不在 ABR 階梯 |
| **點播但內容要強版權保護**(付費影院) | 別用「切片公開可緩存」的裸假設 | 加 **DRM(Widevine/FairPlay)+ 加密切片 + token 鑑權**,CDN 緩存的是加密內容,這是另一層複雜度 |

> **這張表是架構師線的精華**——「知道一個方案的邊界、知道何時它是錯的」,比「知道這個方案怎麼做」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。**尤其「小規模別上這套」和「實時互動別套點播」這兩條,是面試官最愛追的反向問題。**

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 視頻/直播的容量估算結論一句話? | **被物理鎖死**:百萬同看 ×5Mbps = **5 Tbps** 出口 → 單源不可能 → **CDN 是入場券**。存儲 **PB→數百 PB** 必走對象存儲、DB 只存 metadata。轉碼是隱性大頭(CPU 密集 ×5 碼率)→ 異步 + 彈性。 |
| 2 | 澄清階段最值錢的一句話? | **先切 VOD / Live**:它倆共享下半身(多碼率切片 + CDN 分發),但上半身不同——VOD 異步離線轉碼,Live 實時流水線,延遲差兩個量級。不切就混著答會亂。 |
| 3 | 為什麼播放 API 返回的不是視頻字節? | 返回的是 **manifest 清單**(.m3u8/.mpd),列出各碼率 + 切片 URL。真正的視頻字節播放器**直接去 CDN 拉,不過業務服務器**。這是「控制面 / 數據面分離」——你發目錄,CDN 發貨。 |
| 4 | 轉碼流水線的四個決策 + 各自的「為什麼」? | **異步**(轉碼慢→上傳發 MQ 立刻返回);**切段並行**(長片快出片,fan-out 給多 Worker);**轉 5 碼率**(觀眾網絡不可知→喂 ABR 自適應);**彈性 Spot 集群**(上傳尖峰 + 任務冪等可重試)。 |
| 5 | 為什麼一個原片要轉成多個碼率? | 觀眾網絡千差萬別,**ABR 自適應碼率**要播放器邊播邊測網速、無縫切碼率,前提是**有多個對齊的碼率版本**。多碼率轉碼 = 用存儲(×1.5~2)換流暢度。 |
| 6 | 百萬觀眾 5 Tbps 怎麼收斂到源站扛得住? | **三層回源樹**:邊緣(幾千,99% 流量)→ 中間層(收斂同區 miss)→ 源站。每個邊緣對一切片只回源一次。放大比**百萬→百**,源站只面對幾百個 CDN 回源連接。 |
| 7 | 直播的「源站放大/驚群」是什麼?怎麼解? | 最新切片**剛生成、全 CDN 邊緣都沒有**,百萬觀眾幾乎同時來拉 → 百萬 miss 同時回源打爆源站(VOD 不嚴重,切片是冷的)。三招:**回源請求合併**(並發 miss 合成一個回源)+ **origin shield** + **切片主動 push**。 |
| 8 | 為什麼標準 HLS 直播延遲十幾秒?延遲大頭在哪? | 不是網絡,是**切片機制本身**:要湊滿一個 6s 片才能發 + 播放器預緩衝 3 片 = ~15–30s。這是換**穩定 + CDN 兼容**的代價,不是 bug。 |
| 9 | HLS / LL-HLS / WebRTC 怎麼選? | **HLS**(~10–30s):最穩最省、CDN 全兼容、海量觀眾,普通直播默認。**LL-HLS**(~2–5s):切小片、仍走 HTTP CDN,帶貨/輕互動。**WebRTC**(<1s):UDP 實時、脫離 CDN、要自建 SFU、百萬規模極貴,只在真互動(連麥/拍賣/雲遊戲)用。 |
| 10 | 視頻平台何時**不該**套這套方案? | **小規模/內部**(一個對象存儲+一家 CDN 即可);**實時互動**(直接 WebRTC,別套點播切片);**短視頻**(優化首幀+滑動預取,ABR 收益小);**付費版權**(加 DRM + 加密切片 + 鑑權)。 |
| 11 | 這題的可用性命門是什麼? | **CDN**——單廠商掛了整區黑屏,必須**多 CDN + 播放器端 failover**(拉切片失敗切另一家)。直播比 VOD 脆弱:Live 實時鏈路斷了重來不了,要 Ingest 熱備 + 主播端自動重連切備用地址。 |
| 12 | 這題的錢花在哪?和交易題有何本質不同? | 錢在 **CDN 帶寬(最大)+ 存儲 + 轉碼**三座山,CPU/QPS 幾乎不花。架構決策本質是**成本決策**(每 GB 多少錢),不是交易題的一致性。省法:多 CDN 比價 / 自建邊緣 / P2P;存儲冷熱分層;轉碼 Spot + 按需轉碼。 |

## 一句話總綱(背起來)

> **視頻/直播是「帶寬怪獸」題——百萬同看 ×5Mbps = 5 Tbps 出口把單源物理碾死,CDN 是入場券不是優化。控制面(你的服務器發 manifest 清單)和數據面(視頻字節從對象存儲經 CDN 直達播放器)分離是它和普通 CRUD 的本質差異。三個核心子系統:轉碼流水線(異步 + 切段並行 + 轉多碼率喂 ABR + Spot 彈性),CDN 回源收斂(三層回源樹把百萬→百,直播靠請求合併 + origin shield + 切片預推壓驚群),直播延遲(HLS 切片機制固有十幾秒換穩定,LL-HLS 壓到 2–5s,WebRTC 亞秒但脫離 CDN 極貴,按延遲需求選)。架構師線:錢全在帶寬/存儲/轉碼三座山,決策本質是成本;CDN 是可用性命門要多家 failover;先租 CDN,量到臨界點再自建。**
