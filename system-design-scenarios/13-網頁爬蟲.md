# 13 · 網頁爬蟲 Web Crawler

> **T2 收尾題,也是「分散式協調」的母題。** 它表面是「下載網頁」這種誰都會的小事,但一旦把規模拉到**百億網頁、跑幾天到幾週、上千台機器**,它就變成一道**純粹的分散式系統題**:不是算法難,是**怎麼讓上千台 worker 既跑得快、又不互相打架、還對外不擾民**。
>
> 這題的「教學價值」高度集中在三處:**URL frontier**(怎麼調度「下一個爬誰」——優先級 + 禮貌性同時滿足)、**海量 URL 去重**(百億 URL 全量存不下 → 布隆過濾器 + 誤判取捨)、**分散式禮貌性協調**(robots.txt、per-domain 限速、同域路由到同 worker、爬蟲陷阱規避)。前兩個是數據結構功力,第三個是**系統設計的真正分水嶺**——很多人能畫出抓取流水線,但講不清「為什麼不能讓 10 台機器同時打同一個域名」。
>
> 用 `00` 的依賴表看:爬蟲吃 **A(框架數字感)+ B(存儲)+ C(韌性/限流)+ E(隊列扇出)**——它是**第一道「四包齊吃」的題**(短鏈只吃 A+B)。但它**不吃 D(強一致)**:爬慢一點、漏幾頁、重複爬幾頁,天塌不下來,**最終一致就夠**。認清「這題不要強一致」是答對它的前提。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個大規模網頁爬蟲(像 Googlebot),從一批種子 URL(seed)出發,不斷下載網頁、解析出頁面裡的鏈接、把新鏈接加回待爬隊列,持續擴張,最終覆蓋整個(可達的)互聯網,把網頁內容存下來供搜索引擎建索引。

**別一上來就畫爬取流水線**——先用 `01` Step 1 的「問三類」把這道極度開放的題收斂。爬蟲的澄清**特別關鍵**,因為「給誰用」直接決定整個架構:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 爬什麼?只要 HTML 還是要圖片/JS 渲染?要不要遵守 robots.txt?要不要增量重爬? | 主爬 **HTML 文本**(圖片/視頻只記 URL 不下載,或交專門管線);**必須遵守 robots.txt + 禮貌限速**(這是合法爬蟲的底線);**要增量重爬**(網頁會變);**JS 渲染**先不做(是另一個量級的成本,單獨一句帶過) |
| **非功能(SLA)** | 規模多大?多快爬完?可用性?容錯? | **百億級(10¹⁰)網頁**;**幾天到幾週**跑一輪;**禮貌性 > 速度**(寧可慢,絕不能把人家小站打掛);**可中斷可續跑**(跑幾週的任務必須能 checkpoint 恢復) |
| **規模 / 約束** | 抓取速率上限?去重要多準?存儲預算? | 目標 **~萬級頁/秒**;去重**允許極小誤判**(布隆漏掉幾頁無所謂,**多爬比漏爬可接受**);內容存 **PB 級** |

**面試話術(把假設說出來讓對方確認):**
> 「我先框範圍——核心是一個閉環:下載 → 解析 → 提取新 URL → 去重 → 調度回去爬。規模我假設**百億網頁、幾週跑一輪、目標萬級頁/秒**。有兩條我想先釘死的設計原則:**第一,禮貌性是硬約束**——必須遵守 robots.txt、對每個域名限速,寧可慢也不能把小站打垮,這既是合規也是不被封 IP 的前提;**第二,這題我不要強一致**——漏爬幾頁、重複爬幾頁都能接受,所以去重我會用布隆過濾器這種**允許極小誤判**的結構換內存。JS 渲染和圖片下載先不做,那是另一個量級。這樣可以嗎?」

**這幾個澄清直接決定後面**:

- 「百億網頁 + 萬級頁/秒」→ 第 2 步估算的結論會是:**單機絕無可能,必須分散式;且 URL 去重的內存放不下全量,逼出布隆**。
- 「禮貌性 > 速度」→ deep dive ③ 的整個「per-domain 限速 + 同域路由到同 worker」都是為這條服務,也是這題**最容易被追問、最能拉開差距**的點。
- 「可中斷可續跑」→ 第 6 步架構師線:frontier 的狀態必須**持久化**,不能只在內存,否則一台機器掛了那段隊列就丟了。
- 「不要強一致 + 允許多爬」→ deep dive ② 直接選布隆而不是「全量精確去重」,並讓「分散式去重的弱一致」變成可接受。

**常見陷阱:** 上來糾結「怎麼解析 JS 渲染的 SPA」「怎麼識別語言」這種**內容處理**細節。✅ 抓大放小——這題的難點在**調度與協調**(frontier + 去重 + 禮貌),不在解析。解析就是「提取 `<a href>`」一句帶過,把時間留給 frontier。

## 2. 容量估算(直接調用 `02` 的數字)

> 爬蟲的估算和短鏈相反——短鏈是「QPS 騙局」(數字一算發現很小);爬蟲是**每個維度都大**,而且**逼出的設計決策最多**。用 `02` 小抄:一天 ≈ 1e5 s、單機網卡 ~125 MB/s、行記錄 ~1KB、副本 ×3。

### ① 抓取速率(頁/秒)——逼出「要幾台機器」

百億網頁、目標幾週(算 ~10 天 ≈ 1e6 s)跑完一輪:

```
1e10 頁 ÷ 1e6 s = 1e4 頁/秒(平均)= 1 萬頁/秒
```
> **所以:目標就是穩定 ~1 萬頁/秒。** 單台下載器受網絡 IO + 解析 CPU 限,樂觀算一台能扛 ~幾百頁/秒(大量時間花在等遠端響應,靠高並發異步 IO 撐),那就需要 **幾十到上百台下載器**。**這題從第一個數字就否定了單機方案**——和短鏈的「單機全扛」是兩個世界。

### ② URL 去重的內存(布隆 vs 全量)——這題的命門

爬一輪會**見到**的 URL 遠多於最終下載的網頁(一個頁面提取出幾十個鏈接,大量是已見過的)。假設累計見過 **~1000 億(1e11)個 unique URL**,要對每個「見過沒」做判重:

| 方案 | 每個 URL 存什麼 | 1e11 個 URL 要多少內存 | 結論 |
|---|---|---|---|
| **全量存原始 URL** | 平均 ~100 B 字符串 | 1e11 × 100B = **10 TB** | ❌ 放不進任何單機內存,要查盤,慢 |
| **全量存 URL 的 hash** | 一個 64-bit(8B)指紋 | 1e11 × 8B = **800 GB** | ❌ 仍放不進單機內存;分散式 KV 可以但每次判重一次網絡 RTT |
| **布隆過濾器** | 每個 URL ~**幾 bit**(目標誤判率算 ~8–10 bit) | 1e11 × 10 bit ÷ 8 = **~125 GB** | ✅ 一個量級的壓縮;且純內存位運算,判重 ~ns 級 |

> **所以:這就是這題的真矛盾——百億 URL 的精確去重要 TB 級內存/存儲,放不下也太慢;布隆過濾器用 ~125 GB(可以放進一台大內存機,或分片到幾台)把它壓掉一個量級,代價是允許極小誤判(把沒見過的誤判成見過 → 漏爬那頁)。** 因為澄清裡釘了「漏爬可接受」,這個代價**正好被吃掉**。這條結論要在 deep dive ② 展開。

### ③ 存儲(網頁內容)——PB 級,必走對象存儲

百億網頁,每頁壓縮後均 ~**50 KB**(HTML 去掉媒體,gzip 後):

```
1e10 頁 × 50 KB = 500 TB(裸量,單輪)
× 副本 3 ≈ 1.5 PB
（多輪歷史快照 / 增量版本 → 還要再乘）
```
> **所以:內容是 PB 級**——和短鏈的「TB 級 metadata」又差一個量級。網頁正文這種**大 blob、只追加、順序寫、批量讀**的數據,典型走**對象存儲**(S3/HDFS),DB 只存「URL → 內容地址 + 元數據(抓取時間、狀態碼、content hash)」。錨 `system-design/06 存儲選型`。

### ④ 帶寬——這題真有帶寬問題(對比短鏈沒有)

1 萬頁/秒 × 每頁原始 ~100 KB(下載時還沒壓縮、含一些媒體):

```
1e4 頁/s × 100 KB = 1 GB/s 入站帶寬
```
> **所以:~1 GB/s 入站**——遠超單張千兆網卡(125 MB/s),需要**多台下載器分攤 + 分佈在多個出口**。帶寬是真實成本項(雲上出入站流量要錢),也是「為什麼要分散式」的又一個理由。

### 一句話估算結論(面試一定要說出來)

> 「**爬蟲每個維度都大,而且每個數字都逼出一個設計決策:1 萬頁/秒 → 單機不可能、要上百台下載器;1000 億 URL 去重 → 全量 TB 級放不下、逼出布隆(~125 GB)換極小誤判;內容 PB 級 → 走對象存儲、DB 只存指針;入站 ~1 GB/s → 多出口分攤帶寬。所以這是一道徹頭徹尾的分散式協調題,設計重心在 frontier 調度、布隆去重、和禮貌性的分散式協調——不在『怎麼下載一個網頁』。**」

這句「把每個數字焊到一個決策」的總結,就是 `01` 說的資深分水嶺。

## 3. API + 數據模型

> 爬蟲是**系統內部**為主、幾乎沒有對外 API 的題——它的「接口」是內部組件之間的契約。所以這步**重心在數據模型(frontier 的結構),API 一句帶過**。

### API(`01` Step 3:點到為止)

爬蟲對外只有**運維/控制面**接口,不是業務 API:

```
POST /seeds          { urls: [...] }         # 灌入種子 URL
GET  /stats                                  # 抓取進度、速率、隊列深度、去重命中率
POST /domains/{domain}/recrawl  { priority } # 手動觸發某域名重爬
GET  /url?u=...                              # 查某 URL 的抓取狀態(運維排查)
```

> **要點:** 爬蟲的「請求」不是用戶發來的,是**自己對外網站發起的**——所以真正的「接口設計」是**對外的禮貌契約**(User-Agent 標識自己、遵守 robots.txt、限速),這個放 deep dive ③。內部組件間用**消息隊列**解耦(錨 `system-design/04 服務化與通信範式`)。

### 數據模型(`01` Step 4:選型 + 分片鍵)

爬蟲有**三類核心數據**,選型各不同——這是這題數據模型的重點:

| 數據 | 存什麼 | 選型 | 為什麼 |
|---|---|---|---|
| **URL Frontier**(待爬隊列) | 待爬的 URL + 優先級 + 所屬 host | **持久化隊列 / 分片 Kafka / 自建分片隊列** | 海量、要按優先級和 host 組織、要可持久化續跑——見 deep dive ① |
| **已見 URL 集合**(去重) | 「這個 URL 見過沒」 | **布隆過濾器**(主)+ 可選精確 KV(兜底) | 1000 億判重、允許誤判、要 ns 級——見 deep dive ② |
| **網頁內容庫** | URL → 內容地址、content hash、抓取時間、狀態碼 | **對象存儲(blob)+ KV/寬列(metadata)** | PB 級 blob 走對象存儲,索引/查詢走 KV,錨 `system-design/06` |

**URL Frontier 的記錄結構(核心):**

| 字段 | 說明 |
|---|---|
| `url` | 待爬 URL |
| `host` | 域名(`example.com`)——**禮貌性 + 路由的分片依據** |
| `priority` | 優先級(PageRank 估值 / 更新頻率 / 新鮮度需求) |
| `discovered_at` | 何時發現 |
| `next_crawl_at` | 最早可爬時間(per-host 限速用) |

**分片鍵(`01` Step 4 鐵律:海量題必講「按什麼分」,這題的分片鍵是整道題的靈魂):**

> **按 `host`(域名)分片,而不是按 URL。** 這是爬蟲區別於其他海量題的**最關鍵決策**:如果按 URL 隨機分片,同一個域名的 URL 會散落到所有 worker 上,於是**多台 worker 會同時打同一個域名**——直接違反禮貌性、還會被對方封 IP。**按 host 分片 → 同一域名的所有 URL 永遠路由到同一個 worker → 那個 worker 一台機器串行控速,天然實現 per-domain 限速。** 這條在 deep dive ③ 是核心,務必講出來。

> **面試話術:** 「frontier 的分片鍵我選 **host(域名)不選 URL**——這是爬蟲最反直覺的一點。按 URL 分會均勻,但會導致**多台機器同時轟一個網站**,違反禮貌性。按 host 分,同域名 URL 全落到一台 worker,那台機器自己串行限速,禮貌性就**天然落在分片策略裡**,不需要全局協調。代價是熱門大站(維基、新浪)的 host 會成為**熱點分片**——這個我留到 deep dive 講怎麼解。」

## 4. 高層架構(`01` Step 5:沿一個 URL 一鏡到底)

**先畫能跑的最簡閉環,再按需加件**(`01` 鐵律)。爬蟲的架構**本質是一個閉環**(crawl loop),不是線性管線——畫的時候要把「環」畫出來:

```
   ┌──────────────────────── 爬取閉環(crawl loop)────────────────────────┐
   │                                                                       │
   ▼                                                                       │
 種子 URL ──► ┌─────────────────┐                                          │
 (seed)      │   URL Frontier   │  按 host 分片 + 優先級隊列 + per-host 限速 │
             │  (待爬隊列,持久化)│                                          │
             └────────┬─────────┘                                          │
                      │ 調度器取「下一個該爬的 URL」(挑優先級高且到限速窗口的) │
                      ▼                                                     │
              ┌───────────────┐   先查 robots.txt(緩存)是否允許             │
              │  Downloader    │ ──► 允許才下載,帶 User-Agent,遵守限速       │
              │  (下載器集群)   │                                            │
              └───────┬────────┘                                            │
                      │ 拿到 HTML                                            │
                      ▼                                                     │
              ┌───────────────┐                                            │
              │   Parser       │ 解析 HTML、抽正文、提取所有 <a href> 新鏈接   │
              │  (解析器集群)   │                                            │
              └───┬───────┬────┘                                           │
        正文+meta │       │ 提取出的新 URL 列表                              │
                  ▼       ▼                                                 │
        ┌──────────────┐  ┌──────────────────┐                            │
        │ 內容去重      │  │  URL 去重(布隆)   │ ──見過?丟棄                │
        │ (near-dup)   │  │  沒見過 → 加回 ────┼────────────────────────────┘
        └──────┬───────┘  └──────────────────┘    (回到 Frontier,閉環)
               ▼
        ┌──────────────┐
        │ 內容存儲      │  正文 → 對象存儲;metadata → KV;喂給下游建索引
        │ (對象存儲+KV) │
        └──────────────┘
```

**一條 URL 的生命週期走一遍(`01` Step 5):**

1. **調度:** 調度器從 **Frontier** 挑「下一個該爬的 URL」——既要**優先級高**(重要/該更新的頁先爬),又要**到了該 host 的限速窗口**(沒到就跳過,換別的 host)。
2. **禮貌檢查:** 下載前先查該 host 的 **robots.txt**(緩存著,別每次都拉)——不允許爬的路徑直接丟棄。
3. **下載:** Downloader 帶**識別性 User-Agent** 發起請求,遵守 per-host 限速,拿到 HTML(失敗則按退避重試)。
4. **解析:** Parser 抽正文 + 提取頁面裡所有 `<a href>` 新鏈接,把相對路徑補成絕對 URL、規範化(normalize)。
5. **去重(兩層):** 提取的新 URL 過 **布隆**——見過就丟,**沒見過才加回 Frontier**(這就是閉環的「回」);同時正文過**內容去重(near-dup)**——和已存內容近似就不重複存。
6. **存儲:** 新內容正文進對象存儲、metadata 進 KV,喂給下游索引管線。

**每加一個組件都說一句它解決什麼問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **URL Frontier(分片 + 優先級 + 限速)** | 調度「先爬誰、別打爆誰」——這題的大腦 | ①速率 + 禮貌性硬約束 |
| **Downloader 集群** | 1 萬頁/秒單機不可能 | ①「上百台下載器」 |
| **布隆去重** | 1000 億 URL 全量放不下 | ②「布隆 ~125 GB」 |
| **內容去重(near-dup)** | 鏡像站/同文不同 URL,別重複存 PB | ③存儲省錢 |
| **對象存儲 + KV** | 內容 PB 級、metadata 要可查 | ③「對象存儲 + 指針」 |

**注意這張圖的兩個「克制」:**(1) 解析器只畫一個框、不展開——這題難點不在解析;(2) 沒有複雜的強一致組件、沒有事務——因為這題**不吃 D(一致性)**,加上去就是 `01` 紅旗的「過度設計」。

## 5. Deep Dive(挑 3 個最會被追問的點往死裡鑽)

### Deep Dive ① · 整體閉環與 URL Frontier(這題的大腦)

爬蟲畫得出閉環的人不少,但能講清 **Frontier 怎麼同時滿足「優先級」和「禮貌性」這兩個會打架的目標**的人不多——這是這題最值錢的 deep dive。

**Frontier 要同時做兩件互相衝突的事:**

| 目標 | 想要 | 衝突點 |
|---|---|---|
| **優先級**(prioritization) | 重要的頁(高 PageRank、新聞首頁、該更新的)**先爬** | 如果只按優先級,可能一連串高優先級 URL 都來自**同一個熱門域名** → 短時間猛打一個站 |
| **禮貌性**(politeness) | 對**每個域名**控速,兩次請求間隔夠長,絕不並發轟炸 | 如果只按禮貌性排隊,又可能讓重要頁等太久 |

**經典解法(Mercator 設計,業界標準,值得畫出來):兩級隊列**

```
            提取出的新 URL
                  │
                  ▼
        ┌───────────────────┐
        │  前端隊列(優先級)  │  按 priority 分成 N 條隊列(F1 高 … Fn 低)
        │  F1  F2  ...  Fn   │  優先級高的 URL 進高隊列
        └─────────┬─────────┘
                  │  優先級路由器:按權重從前端隊列抽(高隊列抽得勤)
                  ▼
        ┌───────────────────┐
        │  後端隊列(禮貌性)  │  按 host 分成 M 條隊列,每條只裝一個域名的 URL
        │  B(host1) B(host2) │  + 一個「host → 哪條後端隊列」的映射
        │  ...  B(hostM)     │  + 一個「最早可取時間」堆(per-host 限速)
        └─────────┬─────────┘
                  │  限速堆:挑「已到限速窗口」的 host,從它的隊列取一個 URL
                  ▼
              交給 Downloader
```

**黑盒裡發生什麼(兩級的分工):**

- **前端隊列管「先爬誰」**(優先級):新 URL 按優先級丟進不同隊列,路由器按權重抽——高優先級被抽中的概率高,但低優先級也不會餓死。
- **後端隊列管「別打爆誰」**(禮貌性):每條後端隊列**只屬於一個 host**,且配一個 **per-host 的「下次最早可取時間」**。一個小頂堆按這個時間排序,調度器**只從「已經到時間」的 host 取 URL**——這就強制了「同一域名兩次請求之間有間隔」。
- **兩級之間的解耦**:前端決定「重要性順序」,後端決定「節奏」——把兩個打架的目標**拆到兩級**分別解決,這就是 Mercator 的精髓。

**Frontier 必須持久化(架構師線的伏筆):** 這個隊列裝著上億待爬 URL、跑幾週,**絕不能只在內存**——一台機器掛了那段隊列就全丟了,幾週白跑。所以後端隊列要落到**持久化存儲 / 分片 Kafka**,內存裡只放「當前正在處理的窗口」。這條是第 6 步「可中斷續跑」的根。

> **面試話術(把衝突講出來再給解法):** 「Frontier 要同時滿足兩個打架的目標:**優先級**要重要頁先爬,**禮貌性**要對每個域名控速。如果只按優先級,一串高優先級 URL 可能全來自同一個熱門站,瞬間把它打爆。經典解法是 **Mercator 的兩級隊列**:前端按優先級分隊列、決定『先爬誰』;後端按 host 分隊列、配一個 per-host 的最早可取時間堆、決定『什麼節奏爬』。兩級解耦,各管一頭。而且這個隊列要**持久化**——跑幾週的任務,機器掛了不能丟隊列。」

### Deep Dive ② · 海量 URL 去重(布隆過濾器 + 誤判取捨)

第 2 步算過:1000 億 URL 精確去重要 ~800 GB(存 hash)到 10 TB(存原串),放不進單機內存。**這正是布隆過濾器的教科書場景**——而且**布隆的原理在 `06 分散式緩存` 的緩存穿透防護裡已經講過**(同一個結構,那邊用它擋「不存在的 key」,這邊用它擋「見過的 URL」),這裡只講爬蟲用法和取捨,原理錨過去。

**為什麼是布隆,不是別的:**

| 方案 | 1000 億 URL | 判重延遲 | 取捨 |
|---|---|---|---|
| **內存 HashSet(原串/hash)** | 800 GB–10 TB,**放不下** | ns 級 | ❌ 內存爆 |
| **分散式 KV 精確去重** | 存得下(分片) | 每次一個**網絡 RTT(~1ms)** | ❌ 1 萬頁/秒 × 幾十鏈接 → 每秒幾十萬次判重,RTT 受不了 |
| **布隆過濾器** | **~125 GB**(壓一個量級) | **ns 級**(純內存位運算) | ✅ 內存省、判重快,**代價:有誤判** |

**布隆的誤判方向——這是這題能不能用它的關鍵(務必講清):**

布隆過濾器**只會「假陽性」,不會「假陰性」**:

- **說「沒見過」→ 一定沒見過**(可靠)→ 放心加進 Frontier 去爬。
- **說「見過」→ 可能其實沒見過**(誤判)→ **誤把一個新 URL 當成見過 → 丟棄 → 漏爬那一頁**。

> **這個誤判方向正好和爬蟲匹配:** 布隆的錯誤後果是「**漏爬幾頁**」,而澄清裡釘死了「漏爬可接受」(那一頁多半還會從別的鏈接被發現)。如果反過來(假陰性 → 重複爬),頂多浪費點帶寬也不致命。**所以布隆的「不對稱誤判」對爬蟲是良性的**——這就是為什麼這題能用它、而強一致場景不能用它。調誤判率:給更多 bit/URL(從 8 調到 12)能把誤判率壓到 0.1% 以下,代價是更多內存,**面試官常追問這個 tradeoff,要會說「bit 數 ↔ 誤判率 ↔ 內存」三角」**。

**分散式去重的一致性(這題的進階追問):**

布隆 125 GB 可以放一台大內存機,但**上百台 worker 都要判重**,怎麼共享?三種做法:

| 方案 | 怎麼做 | 取捨 |
|---|---|---|
| **中心化布隆服務** | 一個(或主從)布隆服務,所有 worker 來問 | 一致性好,但**每次判重一個 RTT**,又回到 KV 的延遲問題;且是單點 |
| **按 host 分片布隆**(推薦) | 布隆也**按 host 分片**,和 frontier 同樣的分片鍵 → 同一域名的去重永遠在同一台 | 判重落在本地、無跨機 RTT;天然和「同域路由到同 worker」對齊 |
| **每 worker 本地布隆 + 弱一致** | 每台各維護自己見過的;偶爾跨機重複 | 最快,但**跨 worker 會重複爬**(兩台各自第一次見到同一 URL);因為「重複可接受」,弱一致 OK |

> **面試話術:** 「百億 URL 精確去重放不下、查 KV 又每次一個 RTT 撐不住每秒幾十萬次判重。**布隆過濾器**用 ~125 GB 把它壓一個量級、判重 ns 級。關鍵是布隆的誤判方向**正好良性**:它只會『假陽性』,後果是**漏爬幾頁**,而我們澄清過漏爬可接受;它永遠不會『假陰性』,所以說沒見過的就放心爬。布隆原理我在緩存穿透那題講過,這裡是同一個結構。分散式我會**讓布隆和 frontier 用同一個 host 分片鍵**,判重就落在本地、不用跨機。」

### Deep Dive ③ · 禮貌性與分散式協調(這題真正的分水嶺)

前兩個 deep dive 是數據結構功力,**這一個才是系統設計的真分水嶺**——很多人能畫流水線、能說布隆,但講不清「為什麼不能讓 10 台機器同時打同一個域名,以及怎麼從架構上保證不會」。

**禮貌性的四件事,逐個拆:**

**(1) robots.txt 遵守——合法爬蟲的底線**

每個域名根目錄有 `/robots.txt`,聲明哪些路徑禁爬、建議的 `Crawl-delay`。

| 做法 | 黑盒裡發生什麼 |
|---|---|
| **首次訪問某 host 先拉 robots.txt 並緩存** | 別每爬一個 URL 都拉一次(那是對人家的二次騷擾);緩存它,設個 TTL(如 24h)定期刷新 |
| **解析出 Disallow 規則,過濾 Frontier** | URL 入 Frontier 前(或出隊前)先比對規則,禁爬的直接丟 |
| **尊重 Crawl-delay** | 如果 robots 聲明了間隔,per-host 限速取「我的默認」和「它要求的」的較大值 |

**(2) per-domain QPS 限速——別打垮小站**

> 對**每個域名**而不是全局限速。大站(維基)能扛你猛爬,小站(個人博客)你每秒兩個請求就可能把它打掛、還連累你被封 IP。**限速粒度必須是 per-host**。算法錨 `distribution/限流算法/`——令牌桶 / 漏桶都行,**每個 host 一個獨立的桶**(或最簡單:per-host 記「下次最早可取時間 = 上次抓取時間 + delay」)。

**(3) 同域 URL 路由到同一個 worker——分散式限速的關鍵招**

這是把「per-host 限速」從「需要全局協調的難題」變成「單機本地問題」的**關鍵設計**:

```
   ❌ 按 URL 分片(均勻但闖禍)            ✅ 按 host 分片(禮貌天然成立)
   example.com/a → Worker 1              example.com/* → 全部 Worker 3
   example.com/b → Worker 5              ──────────────────────────────
   example.com/c → Worker 2              Worker 3 一台機器本地串行限速,
   → 三台同時打 example.com,違規!         不需要任何跨機協調,天然禮貌
```

> **這就是第 3 步分片鍵選 host 的回報:** 同一域名的所有 URL 都路由到同一個 worker(用 `hash(host) % N` 路由),那台 worker **在本地就能串行控速**——根本不需要「全局分散式限速器」這種重武器。**把分散式協調問題消解成本地問題,是這題最漂亮的一招**,務必講出來。代價是**熱點分片**(熱門大站的 host 落到某台 worker,它累死),解法:對超大站再做**二級拆分**(按 host + 路徑前綴),或給大站單獨配更多下載並發(它本來就扛得住)。

**(4) 爬蟲陷阱(crawler trap)規避 + 去環**

互聯網不是樹是圖,還有惡意/無心的無限結構,不防會把 frontier 撐爆、worker 困死:

| 陷阱 | 長什麼樣 | 怎麼防 |
|---|---|---|
| **無限日曆** | `/calendar?date=2025-01-01` → 頁面有「下一天」鏈接,永遠點不完 | **限制每域名的最大爬取數 / URL 深度上限**;檢測同 host 下 URL 模式高度相似 |
| **動態無限 URL** | session id、排序參數造出無窮 URL(`?sort=a&sort=b…`) | **URL 規範化(normalize)**:去掉無關 query 參數、排序參數、fragment、統一大小寫/結尾斜杠 → 把「看起來不同實則同頁」的 URL 收斂成一個 |
| **去環(cycle)** | A→B→A→B 互相鏈接 | **去重(布隆)本身就是去環**——A 第二次被發現時布隆說「見過」直接丟,環就斷了 |
| **深度爆炸** | 一個站無限深的目錄樹 | **per-host 爬取預算**(這個站最多爬 N 頁就停),保證不會被一個站吸乾 |

> **URL 規範化是這裡的隱形重點:** `http://A.com/p`、`http://a.com/p/`、`http://A.com/p?utm=x` 其實是同一頁——不規範化就會被當成三個 URL 各爬一遍,既浪費又繞不出陷阱。規範化(統一 scheme/host 大小寫、去 fragment、去追蹤參數、排序 query、補全相對路徑)**是去重的前置**:先 normalize 再進布隆,否則布隆判重形同虛設。

> **面試話術(把「協調消解成本地」這招講出來,直接過架構師線):** 「禮貌性是這題最容易翻車的點。核心招數是**按 host 分片 + 同域路由到同 worker**:這樣 per-host 限速就變成**一台機器的本地串行問題**,不需要全局分散式限速器。robots.txt 我緩存著按 host 過濾,限速用 per-host 令牌桶。陷阱和去環靠三道防線:**URL 規範化**(把假裝不同的同頁收斂,這是去重前置)、**布隆去重天然去環**、**per-host 爬取預算**防止被無限日曆這種陷阱吸乾。代價是熱門大站成熱點分片,對超大站再按路徑二級拆分。」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步就夠拿 offer;這一段是 `00` 的**架構師決策線**——「代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛」。

### 成本:存儲 + 帶寬是大頭,且增量重爬讓它「越滾越多」

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **內容存儲** | 單輪 ~1.5 PB(含副本);**增量重爬要存歷史版本** → 多輪疊加 | 主成本項。**內容去重(near-dup)** 直接省錢——鏡像站、轉載、同文不同 URL 佔比很高,去掉它能省 20–40% 存儲。冷數據(老快照)下沉廉價冷存儲 |
| **帶寬** | 入站 ~1 GB/s,雲上出入流量要錢 | 多出口分攤;**增量重爬靠條件請求省帶寬**(`If-Modified-Since` / ETag → 304 沒變就不下載正文) |
| **去重內存** | 布隆 ~125 GB(可控),但**隨爬取量持續增長** | 布隆**滿了會誤判率飆升**——要監控 fill ratio,接近閾值就**擴容/重建更大的布隆**(或用可擴展布隆 scalable bloom) |
| **計算** | 上百台下載 + 解析 | 下載多是 IO 等待(高並發異步省機器);解析吃 CPU,可彈性伸縮 |

> **增量重爬的成本智慧:** 不是所有頁都同頻重爬——新聞首頁可能每小時、靜態文檔可能每月。按**頁面變化頻率自適應調整重爬週期**(觀測到某頁老不變就拉長間隔),把有限的抓取預算花在「真會變」的頁上。這是用「觀測」換「成本」的架構決策。

### 容量規劃:Frontier 深度和布隆容量是兩個要盯的池

- **Frontier 深度**:提取出的新 URL 速度 vs 下載消費速度——如果**入 > 出**,frontier 無限膨脹(陷阱常導致這個)。要監控隊列深度,失控就是陷阱沒防住的信號。
- **布隆容量**:按「預計總 unique URL 數 + 目標誤判率」**提前算好 bit 數和 hash 函數個數**(這是布隆的兩個設計參數)。布隆**不能刪元素**,只增不減,所以要按「整輪的峰值 unique 數」一次性配足,或用 scalable bloom 分段擴。

### 演進路徑:從單機腳本到分散式爬蟲

```
階段 1(MVP):單機 + 內存隊列 + 內存 HashSet 去重
   爬幾萬頁夠用,別過度設計
        │  規模上萬頁/秒、去重內存放不下、要跑幾週
        ▼
階段 2:分散式 — frontier 按 host 分片 + 持久化隊列 + 布隆去重 + 下載器集群
   核心轉變:去重從 HashSet 換布隆、隊列從內存換持久化、單機換集群
        │  要持續重爬保新鮮、多數據中心、反爬對抗
        ▼
階段 3:增量重爬調度 + 多 DC 部署 + JS 渲染管線(headless 瀏覽器,貴)
   按變化頻率自適應重爬;就近抓取;渲染 SPA(另一個量級成本)
```

> **這條演進的關鍵 insight:** 爬蟲和短鏈一樣**從第一天就知道終局**(分散式 + 布隆 + 持久化 frontier),但**MVP 用單機內存版完全夠驗證閉環邏輯**。一上來就上 Kafka + 分片布隆 + 多 DC,是 `01` 紅旗的過度設計。**講「現在內存版,規模到了我換布隆 + 持久化分片」比一開口堆滿組件更顯資深。**

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **某下載 worker 掛** | 它負責的那些 host 暫停爬取 | frontier 持久化 → URL 沒丟;worker 重啟/重分配後從持久化隊列**續爬**(這就是 frontier 必須持久化的理由) |
| **布隆過濾器丟失/重建** | 去重狀態沒了 → 大量已爬 URL 被當「沒見過」**重爬**(浪費,但不致命) | 布隆**定期 checkpoint 到持久化**;重建期間靠「重複可接受」兜底——這正是選「弱一致去重」的回報 |
| **熱點分片(大站)worker 過載** | 維基/新浪這類大 host 落在一台,它累死、其他閒置 | 對超大站按路徑二級拆分;給大站更多並發(它扛得住);監控 per-worker 負載再平衡 |
| **被目標站封 IP / 觸發反爬** | 整段抓取被拒(429/403) | 嚴格遵守 robots + 限速(根本預防);出口 IP 池輪換;遇 429 **指數退避**;這是「禮貌性 = 自我保護」的體現 |
| **爬蟲陷阱撐爆 frontier** | 隊列無限膨脹、worker 困在一個站 | per-host 爬取預算 + URL 規範化 + 深度上限(deep dive ③ 的三道防線) |

> **架構師話術:** 「我的關鍵韌性設計是**frontier 持久化**——它是跑幾週任務的命根子,worker 掛了能從持久化隊列續爬,不丟進度。布隆丟了不致命,因為『重複爬可接受』,重建期間多爬一點就是了——這正是我選弱一致去重的回報。最危險的外部失效是被封 IP,而**最好的預防恰恰是禮貌性本身**:嚴格限速 + 遵守 robots,既是合規也是自我保護。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **只爬幾個已知站、量小**(爬競品價格、抓特定 API) | 別上 frontier + 布隆 + 集群 | 一個帶限速的腳本 + 一張表記去重就夠,別過度設計 |
| **要爬重 JS 的 SPA**(內容靠前端渲染) | 別用「下載 HTML 抽 `<a>`」的純文本管線 | 要 **headless 瀏覽器渲染**(Puppeteer/Playwright)——但它**慢一個量級、貴一個量級**,只對必要的站開,別全量上 |
| **聚焦爬蟲 / 垂直爬蟲**(只要某主題的頁) | 別無腦廣度優先爬全網 | 加**主題相關性判斷**(爬到的頁算分,低分的鏈接不擴展),是另一類問題(focused crawling) |
| **實時 / 近實時抓取**(秒級要拿到新內容) | 別用「幾週一輪」的批量爬蟲 | 那是 **push / 訂閱**(RSS、sitemap ping、PubSubHubbub)或事件驅動,不是 pull 爬蟲的場景 |

> **這張表是架構師線的精華**——「知道一個方案的邊界、知道何時它是錯的」,比「知道這個方案怎麼做」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。尤其「小規模別上這套」和「JS 渲染是另一個量級」這兩條,最能體現你對成本的判斷。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 爬蟲的容量估算結論一句話? | **每個維度都大且各逼一個決策**:1 萬頁/秒 → 上百台下載器(單機不可能);1000 億 URL 去重 → 布隆 ~125 GB 換極小誤判(全量 TB 級放不下);內容 PB 級 → 對象存儲 + DB 存指針;入站 ~1 GB/s → 多出口分攤。**徹頭徹尾的分散式協調題。** |
| 2 | Frontier 為什麼要兩級隊列?各管什麼? | 優先級和禮貌性會打架。**前端隊列按優先級** → 管「先爬誰」;**後端隊列按 host + per-host 最早可取時間堆** → 管「什麼節奏爬、別打爆誰」。兩級解耦(Mercator 設計)。 |
| 3 | 為什麼布隆過濾器適合 URL 去重?誤判方向是關鍵? | 1000 億精確去重放不下(TB 級)、查 KV 每次一 RTT 撐不住。布隆 ~125 GB、ns 級。**誤判只會假陽性(說見過其實沒見 → 漏爬幾頁),永不假陰性**——而漏爬可接受,所以良性匹配。原理錨 `06` 緩存穿透。 |
| 4 | frontier / 布隆的分片鍵為什麼選 host 不選 URL? | 按 URL 分會均勻**但導致多台機器同時打同一域名**,違反禮貌性。按 **host 分** → 同域 URL 全落同一 worker → **per-host 限速變成本地串行問題**,不需要全局分散式限速器。代價:大站熱點分片(二級拆分解)。 |
| 5 | 禮貌性怎麼從架構上保證(而不是靠約定)? | **按 host 分片 + 同域路由同 worker** → 限速本地化;**robots.txt 緩存按 host 過濾**;**per-host 令牌桶/最早可取時間**;陷阱靠 **URL 規範化 + 布隆去環 + per-host 爬取預算** 三道防線。 |
| 6 | 爬蟲陷阱有哪些?怎麼防? | **無限日曆 / 動態 URL**(session、排序參數)→ **URL 規範化**(去追蹤參數、排序 query、補相對路徑)收斂;**A↔B 環** → 布隆去重天然斷環;**深度爆炸** → per-host 爬取預算。規範化是去重前置,不做布隆判重形同虛設。 |
| 7 | frontier 為什麼必須持久化? | 跑幾週、裝上億 URL,只在內存則機器掛了那段隊列全丟、幾週白跑。持久化(分片 Kafka/落盤)→ worker 掛了能**續爬**。布隆反而可弱一致(重建期多爬可接受)。 |
| 8 | 爬蟲何時**不該**套這套方案? | 只爬幾個已知小站(腳本 + 一張去重表即可);爬重 JS 的 SPA(要 headless 渲染,慢/貴一個量級,別全量上);聚焦爬蟲(加主題相關性判斷);近實時(走 push/RSS/sitemap,不是 pull 批量爬)。 |

## 一句話總綱(背起來)

> **網頁爬蟲是「分散式協調」的母題——它不難在算法,難在讓上千台 worker 既快又不互相打架還對外不擾民。三個命門:(1) URL Frontier 用兩級隊列(前端優先級、後端 host + per-host 限速堆)化解「先爬誰 vs 別打爆誰」的衝突,且必須持久化以續跑;(2) 海量去重用布隆過濾器(125 GB 壓掉 TB 級全量),靠它「只假陽性 = 只漏爬」的良性誤判方向,正好吃掉「漏爬可接受」的約束;(3) 禮貌性靠『按 host 分片 + 同域路由到同 worker』把分散式限速消解成本地串行問題,配 robots 緩存、URL 規範化、per-host 爬取預算防陷阱去環。架構師線:frontier 持久化保續跑、內容/near-dup 去重省 PB 存儲、增量重爬按變化頻率自適應、MVP 單機別過度設計。**
