# 14 · 支付 / 錢包

> **T3 第一題,也是整個 track 的「正確性旗艦」。** 前面所有題的母題都是「**規模**」——QPS 多大、存儲多大、扇出多寬。支付反過來:**QPS 小到可笑,難在一分錢都不能錯。** 它考的不是你能不能扛流量,是你能不能在「跨服務、會重試、會超時、會崩」的真實世界裡,讓**錢守恆**。
>
> 這題的教學價值集中在三個詞:**冪等**(同一筆支付重試多少次都只扣一次)、**一致性**(扣款 + 記帳跨服務怎麼不丟不重)、**對賬**(就算前面都做對了,還要有一張網兜住漏網的差錯)。把這三個鑽透,T3 的「強一致交易」這一大類(`16 秒殺`、`15 派單` 的庫存/餘額扣減)就通了一半。
>
> 用 `00` 的依賴表看:支付吃 **D(一致性)●●** 是絕對重心,A/B/C/E/F 都沾但都不是命門——**這是全 track 唯一一題,你講不清 D 就直接出局**(`00` 第 3 節原話)。所以本章的打法和短鏈截然不同:短鏈是「克制,別過度設計」;支付是「**克制地把正確性講透**」——把冪等/一致性/對賬講清,而不是堆組件。
>
> **本章是「重錨」章。** 一致性/事務/對賬的細節,你的 repo 裡有 `financial-consistency/`(~80 檔的世界級深礦)整套講透了。本章**只教「整題面試打法 + 怎麼指進」**——把零件**裝**成一道 45 分鐘能講完的支付題,並在每個深挖點精準錨回深礦,**絕不在這裡重寫本地消息表/TCC/Saga 的實作細節**(那是搶 `financial-consistency` 的戲、也炸掉白板時間)。

## 1. 題目 + 澄清問題清單

**題目:** 設計一個支付 / 錢包系統。使用者在 App 裡發起一筆支付(買東西 / 轉帳 / 充值提現),系統要扣款、記帳、調用第三方渠道(銀行 / 微信 / Stripe),並保證**錢不多扣、不少扣、不重複扣、最終對得上**。

**這題上來最忌諱「開畫」**——支付是強一致題,澄清沒做清楚,後面整個一致性模型就建錯了。用 `01` Step 1 的「問三類」收斂:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 支付的「種類」是什麼?內部錢包扣減,還是接外部渠道?要不要退款、轉帳、充值提現? | 核心:**發起支付 → 扣款 → 調渠道 → 回結果**。**內部錢包扣減**(餘額)+ **外部渠道**(銀行/三方)都要;**退款**要(逆向流程);轉帳/充值提現算錢包子場景。**對賬**必做。 |
| **非功能(SLA)** | 一致性要多強?延遲容忍?可用性?**錢能不能錯?** | **正確性 > 一切**——一分錢都不能錯(這是支付的第一公理)。一致性:**核心扣款強一致,跨服務/跨渠道最終一致 + 對賬兜底**。延遲:秒級可接受(用戶等得起,但別卡死)。可用性:高,但**寧可拒絕也不能扣錯**(fail-safe 偏保守)。 |
| **規模 / 約束** | 支付 QPS?帳務記錄量?峰值在哪? | 峰值 QPS **不大**(對比 feed 的幾十萬,支付撐死幾千~上萬,且多數場景幾百)。但**帳務記錄只增不刪、要存很多年**(合規/審計)。峰值在大促/發薪日。 |

**面試話術(把「正確性優先」這個定調說出來——這是支付題的開場核心):**
> 「支付題我先定個調:**它和前面的題不一樣,難點不在規模在正確性——一分錢都不能錯**。所以我會把 SLA 框成『**扣款這一步強一致,跨服務和跨渠道用最終一致 + 對賬兜底**』。規模上我假設峰值幾千 QPS、帳務記錄存多年只增不減。澄清清楚這個定調,後面我所有設計都圍繞『**錢守恆**』來做——冪等防重複扣、狀態機防亂跳、對賬兜住漏網差錯。這樣框可以嗎?」

**這幾個澄清直接決定後面**(支付題的澄清「殺傷力」比任何題都大):

- 「正確性 > 一切」→ 整個架構的取捨方向是 **fail-safe(寧拒絕勿錯扣)**,不是 fail-open。
- 「會重試 / 會超時」(這是支付的隱含約束,你要主動點出)→ deep dive ① **冪等是不可繞過的地基**。
- 「跨服務扣款 + 記帳」→ deep dive ② **分布式事務怎麼選**(本地消息表 / TCC / Saga,**不是** 2PC)。
- 「就算前面都對,還會有差錯」→ deep dive ③ **對賬是最後一道網**,不是可選項。
- 「QPS 不大」→ 第 2 步估算的結論會是**「規模不是矛盾,別堆分片堆緩存秀肌肉」**——這正是支付題最容易答歪的地方。

**常見陷阱(支付題專屬):**
- 🚩 **把它當高並發題做**——上來算幾十萬 QPS、堆緩存堆分片。錯,支付的矛盾是正確性,你秀規模反而暴露你沒抓到題眼。
- 🚩 **餘額直接 `UPDATE balance = balance - amount`** 就完事——沒提冪等、沒提流水、沒提對賬,直接掉到初級線。
- ✅ 開場第一句就把「正確性 > 規模」說出來,把面試官的預期錨到 D(一致性)上。

## 2. 容量估算(這題的估算結論是「反高潮」)

> 支付的估算很特別:**算出來是要證明「QPS 不是瓶頸」**,好讓你理直氣壯地把精力全投在正確性上。`01` Step 2 鐵律「每步接一句所以」在這題尤其重要——因為結論是反直覺的。

用 `02` 小抄:**一天 ≈ 1e5 s、峰值 = 平均 ×3、單 MySQL 寫 ~幾千 QPS、行記錄 ~1KB**。

### ① 支付 QPS(寫,核心路徑)

假設一個中型支付平台:**1 億筆支付/天**(已經是很大的量了,雙 11 級別才更高):

```
1 億/天 ÷ 1e5 s = 1000 QPS(平均)
峰值 ×3 ≈ 3000 QPS（大促可能再 ×3~5,按 1 萬 QPS 上限規劃)
```
> **所以:核心支付 QPS 撐死幾千、極端大促上萬——單機 MySQL 寫(幾千)就在臨界,稍微分庫就綽綽有餘。** 規模**根本不是這題的矛盾**。對比短鏈讀峰值千級、feed 幾十萬——支付的數字小得「無聊」。**這個「無聊」就是答案:把腦力從規模挪到正確性。**

### ② 帳務記錄存儲(真正會增長的東西,但增長也不快)

支付不是只記一條「支付單」。**複式記帳**下,一筆支付要落多條帳務流水(借/貸至少各一條,加上手續費、渠道分潤可能更多)。一條帳務記錄估 **~1 KB**:

```
1 億筆/天 × ~4 條流水/筆 × 1 KB ≈ 400 GB/天
× 副本 3 ≈ 1.2 TB/天
存 5 年(合規)≈ 365×5 × 400GB ≈ 700 TB（裸量,×副本更大）
```
> **所以:帳務是只增不減、要存很多年的「冷重資產」。** 但和短鏈一樣——**這是存儲增長問題,不是 QPS 問題**。架構上:**熱帳(近期)走在線庫,老帳歸檔到便宜冷存儲 / 數據倉**(對賬和審計從歸檔讀)。這條在第 6 步架構師層展開。

### ③ 對賬批量規模(支付獨有的一條估算)

對賬是**離線批量**——每天日切後,把**自己這邊的帳**和**第三方渠道對帳單**逐筆比對:

```
1 億筆/天 → 對賬要掃 1 億 × 多個數據源(內部帳 + 渠道帳單)
這是離線批處理,不在線上關鍵路徑——
但 1 億級逐筆比對,單機跑不完 → 要分片並行 / 走大數據(Spark)
```
> **所以:對賬的瓶頸不是延遲是吞吐**——它是 T+1 的離線任務,但數據量大到要分而治之。這引出一個架構師級的點:**對賬規模隨業務線性漲,日切窗口是有限的**,對賬本身就是個要做容量規劃的系統(第 6 步)。

### 一句話估算結論(面試一定要說出來,而且要「反高潮」地說)

> 「**支付的估算結論是反直覺的:QPS 不是瓶頸——核心才幾千、極端大促上萬,稍微分庫就夠了。真正會增長的是帳務記錄(只增不減、存多年,要冷熱分離歸檔),以及對賬的批量規模(1 億級逐筆比對要分片並行)。所以我接下來的設計重心,不放在『扛流量』,而放在『一分錢不錯』——冪等、跨服務一致性、對賬。這是支付題和前面所有題最大的不同。**」

這句「**主動宣告我不在這題秀規模、要秀正確性**」,就是 `00` 標準線裡「擁有這個決策」的開場——你**主動拒絕了一個錯誤的優化方向**,比答對更顯資深。

## 3. API + 數據模型

### API(`01` Step 3:支付的 API 有一個別題沒有的硬要求——冪等鍵)

```
POST /payments
  Header: Idempotency-Key: <client 生成的唯一鍵>   ← 支付 API 的命根子,不是可選
  body:   { orderId, payerId, payeeId, amount, currency, channel }
  → 200 { paymentId, status: "success|paying|failed" }

GET  /payments/{paymentId}            # 查支付單狀態(客戶端輪詢/回調補償用)
  → { paymentId, status, ... }

POST /payments/{paymentId}/refund     # 退款(逆向流程,本身也要冪等)
  Header: Idempotency-Key: <唯一鍵>
  → 200 { refundId, status }

POST /payments/callback               # 接收第三方渠道的異步回調(支付結果通知)
  → 200（回調本身也會重複投遞,所以處理也要冪等)
```

**要點(每一條都是支付的「信號」,面試官在聽你提不提):**
- **`Idempotency-Key` 是 Header 級的硬約束,不是「最好有」**——客戶端對**同一筆支付意圖**生成一個唯一鍵,重試時帶**同一個鍵**。服務端據此保證「同一鍵只執行一次」。這是 deep dive ① 的入口。冪等鍵本身怎麼保證全局唯一,**錨本 track `05 分散式 ID`**,別在這重講。
- **狀態是顯式的有限集合**(`paying / success / failed`),不是 bool——因為支付有「**中間態**」(已發起、等渠道回調),這是 deep dive ① 狀態機的入口。
- **回調 `/callback` 必須冪等**——第三方渠道的回調**會重複投遞**(網絡重試),你收兩次「支付成功」不能記兩次帳。
- **退款是獨立逆向流程**,也要冪等(防重複退款比重複收款更危險——是真金白銀流出)。

### 數據模型(`01` Step 4:支付的數據模型是「三張表撐起正確性」)

支付的數據模型不是一張表,是**三類結構協同**——這本身就是 deep dive 的鋪墊:

**① 支付單(payment_order)——記「一筆支付意圖的生命週期」:**

| 字段 | 類型 | 說明 |
|---|---|---|
| `payment_id` | varchar **PK** | 支付單號(全局唯一,錨 `05`) |
| `idempotency_key` | varchar **UNIQUE** | 冪等鍵,**唯一索引**——deep dive ① 的兜底防線 |
| `order_id` | varchar | 業務訂單號 |
| `payer_id` / `payee_id` | bigint | 付款方 / 收款方 |
| `amount` | **decimal / bigint(分)** | 金額——**絕不用 float**(浮點誤差會吃錢),用定點 decimal 或「以分為單位的整數」 |
| `status` | enum | `created/paying/success/failed/refunded`——狀態機 |
| `channel` | varchar | 走哪個渠道 |
| `created_at`/`updated_at` | timestamp | |

**② 帳戶餘額(account)+ 餘額流水(account_log)——錢包的「錢」:**

| account 字段 | 說明 |  | account_log 字段 | 說明 |
|---|---|---|---|---|
| `account_id` PK | 帳戶 |  | `log_id` PK | 流水號 |
| `balance` | **餘額(以分為單位整數)** |  | `account_id` | 哪個帳戶 |
| `version` | **樂觀鎖版本號**(防並發扣錯) |  | `change_amount` | +/- 變動 |
| | |  | `balance_after` | 變動後餘額(對賬用) |
| | |  | `biz_id` UNIQUE | 關聯業務號,**唯一**——同一筆業務只能記一次帳(冪等) |

> **複式記帳(double-entry)的核心思想(架構師信號,提一句就過資深線):** 嚴肅的錢包**不是直接改 balance**,而是**每筆變動記一條流水(借/貸),餘額是流水的累加結果**。「錢從不憑空產生或消失,只在帳戶間流動」——這個**不變量(invariant)**讓對賬有了數學基礎(全系統借方總額 = 貸方總額,恆等)。**複式記帳的完整模型錨 `financial-consistency/02-payment-recharge-withdraw/02-invariants.md`,別在這展開。**

**③ 防重表 / 冪等表(dedup)——deep dive ① 的主角:**

| 字段 | 說明 |
|---|---|
| `idempotency_key` **PK / UNIQUE** | 冪等鍵 |
| `payment_id` | 首次執行產生的結果(再次請求直接回它) |
| `status` | 處理狀態(防「正在處理」的並發重複) |
| `created_at` | 用於過期清理 |

**選型(決策樹錨 `system-design/06 存儲選型`):**

| 維度 | 選擇 | 理由 |
|---|---|---|
| 核心帳務 | **SQL(MySQL/PG)+ 事務** | 餘額扣減和記流水**必須在一個本地事務裡原子完成**——這是 SQL 的主場,NoSQL 的最終一致在這裡是災難 |
| 金額類型 | **decimal / 整數分**,**絕不 float** | 浮點誤差 = 吃錢 = 對不上賬 |
| 並發控制 | **樂觀鎖(version)** 或 **行鎖** | 防兩個請求同時扣同一帳戶扣穿(餘額扣成負) |
| 帳務歷史歸檔 | 冷存儲 / 數倉 | 老帳只增不刪,對賬從歸檔讀 |

> **面試話術:** 「支付的核心數據——餘額和帳務流水——我堅持放 **SQL + 本地事務**,因為『扣餘額 + 記流水』必須**原子**,NoSQL 的最終一致在這一步是不可接受的。金額我一律用 **decimal 或以分為單位的整數,絕不用 float**,浮點誤差會直接吃掉錢、對不上賬。並發我用**樂觀鎖 version** 防扣穿。然後我有一張**獨立的冪等表 / 防重表**,冪等鍵打**唯一索引**——這是後面 deep dive 防重複扣款的兜底。」

**分片鍵(`01` Step 4 鐵律,但支付的分片有講究):**

> **按 `account_id`(或 `user_id`)分片**——讓**一個帳戶的所有流水和餘額落在同一片**,這樣「扣某帳戶餘額 + 記它的流水」能在**單片內的本地事務**完成,不用跨片事務(跨片事務 = 分布式事務 = 你最想避免的東西)。**這是支付分片的關鍵 insight:選分片鍵的目標,是讓核心強一致操作落在單片,把分布式事務的範圍壓到最小。** 對比短鏈按隨機 short_code 分(只為均勻)——支付分片的首要目標是**事務局部性**,均勻是次要的。

## 4. 高層架構(`01` Step 5:沿一筆支付一鏡到底)

**先畫能跑的最簡版,克制地加件**(`01` 鐵律)。支付的圖不複雜,但每個組件都在解一個「正確性」問題:

```
   一筆支付的完整生命週期(同步 + 異步兩段)

 Client ──POST /payments (Idempotency-Key)──► API GW ──► Payment Service
                                                              │
                                              ┌───────────────┤
                                   ① 冪等檢查  │   查防重表:這個 Idempotency-Key 處理過嗎?
                                              │   ├─ 處理過 → 直接回上次結果(不重複執行)  ◄── 防重複扣款
                                              │   └─ 沒處理 → 寫入防重表(唯一索引兜底),繼續
                                              ▼
                                   ② 創建支付單(status=created → paying)  ◄── 狀態機
                                              │
                                  ┌───────────┴────────────┐
                       内部錢包扣减(强一致)        外部渠道(最终一致)
                                  │                        │
                     ③ 本地事务原子:                 ④ 调第三方渠道(银行/三方)
                        - 扣余额(乐观锁 version)        - 同步返回「受理中」
                        - 记账务流水(biz_id 唯一)       - 真正结果走【异步回调】
                        - 同一事务里写「消息表」一行   ◄── 本地消息表(跨服务不丢)
                                  │ commit                  │
                                  ▼                         ▼
                        ⑤ 投递消息到 MQ ──► 下游(记账/通知/积分…)异步消费
                                                            │
                          ┌─────────────────────────────────┘
                          ▼
                 ⑥ 渠道异步回调 /callback(也要幂等)
                          │  更新支付单 status → success/failed(状态机:只能正向流转)
                          ▼
                 ⑦ 回调失败/超时? → 主动【对账/查询补偿】兜底
                                                            
   ─────────────────────────────────────────────────────────
   ⑧ 日切对账(离线批,T+1):自己的帐 ⇄ 渠道对帐单 逐笔比对 → 差错(长款/短款)→ 补偿
```

**一筆支付走一遍(`01` Step 5:這題只有「寫」路徑,但分同步段和異步段):**

1. **冪等檢查(同步,進門第一件事):** 拿 `Idempotency-Key` 查防重表——**處理過就直接回上次結果**,根本不重複執行。這是防重複扣款的**第一道閘**。
2. **創建支付單 + 狀態機:** `created → paying`,顯式記錄「這筆支付走到哪了」。
3. **內部扣款(強一致,本地事務):** 在**一個本地事務**裡原子完成「扣餘額(樂觀鎖)+ 記帳務流水(biz_id 唯一)+ 寫一行消息表」。**三件事要嘛全成功要嘛全回滾**——這是支付正確性的核心。
4. **外部渠道(最終一致):** 調第三方,**同步只拿到「受理中」**,真正結果走異步回調(銀行不會秒回)。
5. **投遞消息:** 把第 3 步寫進消息表的消息**投到 MQ**,下游(積分/通知/二次記帳)異步消費——**本地消息表保證這條消息不丟**(deep dive ②)。
6. **回調(異步):** 渠道回調帶來最終結果,**更新狀態機**(`paying → success/failed`),回調本身**冪等**(會重複投遞)。
7. **回調兜底:** 回調可能丟 / 超時 → **主動查詢渠道狀態**補償(別乾等)。
8. **日切對賬(離線):** 就算前面全做對,還是用對賬**兜住一切漏網差錯**(deep dive ③)。

**每加一個組件都說它解決什麼正確性問題(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應哪個正確性問題 |
|---|---|---|
| **防重表 + 冪等鍵唯一索引** | 同一筆支付重試/重複提交,只能扣一次 | 防重複扣款(deep dive ①) |
| **狀態機(created/paying/success/failed)** | 支付有中間態,只能正向流轉,防亂跳 | 防重複扣款 + 防錯誤狀態(①) |
| **本地事務(扣款+記流水原子)** | 餘額和流水不能一個成一個敗 | 單機強一致 |
| **本地消息表 + MQ** | 跨服務「扣款 + 通知下游」不丟不重 | 跨服務一致性(deep dive ②) |
| **回調 + 查詢補償** | 第三方異步,結果要可靠拿到 | 與外部渠道的最終一致 |
| **日切對賬** | 兜住所有漏網差錯 | 最後一道網(deep dive ③) |

**注意這張圖的「克制」:** 沒有花哨的緩存層(支付不靠緩存——你敢緩存餘額嗎?)、沒有秀分片(QPS 不大)、**沒有 2PC**(deep dive ② 會講為什麼)。**支付的複雜度全在「正確性語義」上,不在組件數量上**——把圖畫簡單、把每條邊的一致性語義講透,才是這題的資深答法。

## 5. Deep Dive(挑 3 個最會被往死裡追問的點——支付的三大命門)

> 這三個 deep dive 是**重錨區**。本章只講「面試怎麼把它講清、怎麼指進」,**一致性/事務/對賬的實作細節全錨 `financial-consistency/`**,不重寫。

### Deep Dive ① · 冪等:為什麼支付必須冪等,怎麼做到

**先講「為什麼非冪等不可」(這是支付的第一公理,要講到面試官點頭):**

支付的世界裡,**「重複」是常態不是異常**:
- 客戶端網絡超時 → 用戶**手抖再點一次** / 客戶端**自動重試**。
- 服務調用超時 → 上游**不知道下游到底成沒成**,只能重試(這是分布式系統的「**兩將軍問題**」:你發了請求沒收到回應,你**無法區分**是「請求沒到」還是「回應丟了」)。
- MQ **至少投遞一次**(at-least-once)→ 同一條消息**必然會重複消費**。
- 第三方渠道回調**重複投遞**。

> **核心 insight:** 在分布式系統裡,**你無法消除重複,只能讓重複無害**。讓「執行 N 次 == 執行 1 次」的性質就是**冪等**。支付每一個會被重試的環節(下單、扣款、回調、退款)**都必須冪等**,否則重試 = 重複扣錢。

**怎麼做到(層層遞進的四道防線,這是 deep dive 的肉):**

| 防線 | 機制 | 黑盒裡發生什麼 | 防住什麼 |
|---|---|---|---|
| **① 冪等鍵 + 防重表** | 客戶端對「同一筆支付意圖」生成唯一 `Idempotency-Key`,服務端進門先查防重表 | 鍵存在 → **直接回上次結果**,根本不執行業務 | 大多數重複(應用層攔截) |
| **② 唯一索引兜底** | 冪等鍵 / `biz_id` 在 DB 打**唯一索引** | 並發下兩個請求同時過了「查防重表沒有」→ 都去寫 → **第二個被唯一索引拒絕**(`Duplicate key`),靠 DB 原子性兜底 | TOCTOU 競態(查和寫之間的窗口) |
| **③ 狀態機正向流轉** | 支付單狀態只能 `created→paying→success`,**用條件更新** `UPDATE ... WHERE status='paying'` | 重複的「成功回調」第二次來,`WHERE status='paying'` 匹配不到(已是 success)→ **更新 0 行,自動忽略** | 重複的狀態變更(回調重投) |
| **④ 業務流水唯一** | 帳務流水的 `biz_id` 唯一 | 同一筆業務重複記帳 → 唯一索引拒絕 | 重複記帳(餘額被扣兩次) |

> **這四道防線的精髓(架構師信號):** **不要「先查再寫」**——「先 `SELECT` 看有沒有,沒有就 `INSERT`」在並發下有 **TOCTOU 競態**(兩個請求同時查到「沒有」,然後都寫)。正確姿勢是**讓存儲的原子性兜底**:`INSERT ... ON DUPLICATE KEY` / 唯一索引 / 條件 `UPDATE` 的 affected-rows。應用層的防重表是「快路徑」(攔下絕大多數),**唯一索引是「慢路徑兜底」(攔下漏網的並發競態)——兩層都要,缺一不可。** 這句話講出來,冪等這個點就過架構師線了。

> **面試話術(主動給完整防線):** 「支付必須冪等,因為重試在分布式裡是常態——超時你分不清是請求沒到還是回應丟了,只能重試。我用**四層冪等**:客戶端冪等鍵 + 應用層防重表攔下大多數;**冪等鍵打唯一索引**靠 DB 原子性兜住並發競態;**狀態機用條件 UPDATE 正向流轉**,重複回調自動變成更新 0 行;帳務流水 `biz_id` 唯一防重複記帳。關鍵是**絕不先查再寫**,那有 TOCTOU 競態,我靠存儲的唯一性兜底。冪等鍵本身的全局唯一生成,是分散式 ID 那道獨立的題。」

**錨點(不重寫):** 冪等的完整模式(冪等鍵設計、防重表生命週期、與狀態機的配合)錨 `financial-consistency/05-patterns/` 與 `02-payment-recharge-withdraw/03-state-machine.md`;冪等鍵的唯一 ID 生成錨本 track **`05 分散式 ID`**。

### Deep Dive ② · 一致性:扣款 + 記帳跨服務,怎麼不丟不重(本地消息表 vs TCC vs Saga)

**先講清問題(這是支付最硬的點):** 一筆支付要做的事往往**跨多個服務 / 多個庫**——扣帳戶餘額(帳戶服務)、生成支付單(支付服務)、通知下游(積分/通知)、調外部渠道。這些**不在一個本地事務裡**,怎麼保證「要嘛都成、要嘛都不成,不會扣了款但下游沒收到」?

**第一個要主動回答的問題:為什麼不用 2PC(兩階段提交)?**

> **面試話術(主動排除 2PC——這是加分點):** 「跨服務一致性最教科書的答案是 2PC,但**支付我不用 2PC**,原因有三:**① 同步阻塞**——準備階段所有參與者**鎖住資源等協調者**,支付鏈路上鎖住帳戶 = 吞吐崩塌;**② 協調者單點**——協調者在提交階段掛了,參與者**卡在不確定狀態**(鎖死);**③ 跨外部渠道根本不可能**——你沒法讓銀行/第三方加入你的 2PC。所以支付走的是**『本地事務 + 異步消息 + 最終一致 + 對賬兜底』**這條路,不是 2PC 的強同步。」

**三個主流方案對比(這是 deep dive 的核心,但細節錨深礦):**

| 方案 | 一句話機制 | 適用場景 | 代價 / 致命傷 | 在支付的角色 |
|---|---|---|---|---|
| **本地消息表(Outbox)** | 業務操作和「待發消息」**寫在同一個本地事務**裡;事務提交了消息一定在表裡;再由投遞器掃表發 MQ,發成功才標記 | **「本地強一致 + 通知下游」**的最常見場景(扣款後通知積分/記帳) | 消息**最終才到**(有延遲);下游必須**冪等消費**(消息會重發) | **支付的主力方案**——扣款和「要發的消息」原子落庫,絕不丟消息 |
| **TCC(Try-Confirm-Cancel)** | 業務自己實現三個操作:Try 預留資源(凍結額度)、Confirm 真扣、Cancel 解凍 | **需要「資源預留 + 較強一致」**的場景(轉帳、需要先凍結餘額) | **侵入業務**(每個服務寫三個方法)、要處理空回滾/懸掛 | 用於**強一致的餘額凍結/轉帳**——比本地消息表強一致,但成本高 |
| **Saga** | 一串本地事務,每步配一個**補償操作**;某步失敗就**反向補償**前面已做的 | **長流程、多步驟**(訂單→支付→發貨→…),允許中間態可見 | **無隔離性**(中間態能被看到)、補償邏輯複雜、要防補償失敗 | 用於**跨多服務的長業務流程**編排 |

> **怎麼選(架構師決策樹,主動給):** 「我的選法是:**核心扣款記帳走本地事務(單片強一致);跨服務通知下游用本地消息表(最常見、最簡單、消息不丟);需要資源預留的強一致場景(轉帳凍結)用 TCC;多步長流程用 Saga 編排。** 大多數支付場景,**本地消息表就夠了**——它最簡單、侵入最小。TCC/Saga 是當你真的需要更強一致(預留)或更長流程(編排)時才升級。」

**為什麼這套能保證「不丟不重」(把語義講透):**
- **不丟**:消息和業務在**同一本地事務**——事務成功 = 消息一定在表裡 = 投遞器一定會發出去(本地消息表的核心保證)。
- **不重(其實是「重了也無害」)**:消息**至少一次**投遞(會重),靠**下游冪等消費**(deep dive ①)讓重複無害。**最終一致 = at-least-once 投遞 + 冪等消費。**
- **最終一致 + 對賬兜底**:就算消息表投遞器掛了一陣、或某條消息卡住,**對賬(deep dive ③)是最後一道網**——把所有「沒對上的」揪出來補。**這就是支付的整體一致性哲學:強一致守住核心扣款,最終一致處理跨服務,對賬兜住一切漏網。**

> **錨點(重錨,絕不重寫):** 本地消息表的完整實作(消息表結構、投遞器、與 CDC 的對比)錨 `financial-consistency/05-patterns/02-outbox-local-message-table.md` 與 `distr-tx/本地事务表-无MQ蓝图.md`、`financial-consistency/11-outbox-publisher/`(可跑的投遞器原型);TCC/AT 錨 `distr-tx/SEATA/`(`tcc-at区别.md`)與 `financial-consistency/05-patterns/04-tcc.md`;Saga 錨 `financial-consistency/05-patterns/03-saga.md`;三方案的**決策圖和框架對比**錨 `05-patterns/01-decision-map.md` 與 `07-framework-comparison.md`。一致性模型的理論地基錨 `system-design/00 理論基礎-CAP與共識` 與 `02 一致性模型與時鐘`。**這幾檔是這題真正的深礦,面試前重點刷。**

### Deep Dive ③ · 對賬:就算前面都對了,還要有一張網

**先講「為什麼必須對賬」(這是支付區別於一切普通系統的地方):**

> **核心 insight:** 前面的冪等、本地消息表、狀態機,都是**「盡力把它做對」**。但支付有第三方、有網絡、有 bug、有極端時序——**你永遠無法 100% 保證每一筆都對**。所以支付有一條鐵律:**「事前控制 + 事後對賬」雙保險。對賬不是補救措施,是設計的一部分**——它假設「前面一定會有漏網的」,然後用一個**獨立的、離線的、數據驅動的**過程把差錯揪出來。**沒有對賬的支付系統,是不能上線的。**

**對賬在對什麼(兩類):**

| 類型 | 對誰和誰 | 找什麼差錯 |
|---|---|---|
| **內部對賬** | 自己系統內**多個帳戶 / 多張表**互相對(支付單 ⇄ 帳務流水 ⇄ 餘額) | 內部數據不一致(扣了款沒記帳、記了帳餘額沒動) |
| **外部對賬(渠道對賬)** | **自己的帳** ⇄ **第三方渠道的對帳單**(銀行/微信/Stripe 每日下發) | 你以為成功渠道說失敗、渠道扣了你沒記、金額不符 |

**差錯的兩種長相(長款 / 短款,要能脫口而出):**

| 差錯 | 含義 | 典型成因 | 補償方向 |
|---|---|---|---|
| **長款(多了)** | **渠道有、我方無**,或我方多收 | 回調丟了(渠道成功我沒記)、重複收款 | 補記帳 / 退還多收 |
| **短款(少了)** | **我方有、渠道無**,或我方多付 | 我以為成功實際渠道失敗、重複付款 | 衝正 / 追回 / 人工核查 |

**日切對賬流程(`01` Step 5 的「沿流程走一遍」,但這是離線批):**

```
日切(T+1 凌晨,昨日帳已封板)
   │
   ① 拉取数据源:自己的帐务流水(昨日)+ 各渠道下发的对帐单文件
   │
   ② 逐笔比对(按订单号/流水号 join)：
   │     ├─ 双方都有 + 金额一致 → 平(99%+)
   │     ├─ 我有渠道无 → 短款差错
   │     ├─ 渠道有我无 → 长款差错
   │     └─ 都有但金额不符 → 金额差错
   │
   ③ 差错分类 → 自动补偿(能补的)/ 挂起人工核查(补不了的)
   │
   ④ 差错归档 + 告警(差错率超阈值 → 报警,可能是上游 bug)
```

> **黑盒裡發生什麼(架構師關注點):**
> - **對賬是「冪等不變量」的驗證**——複式記帳下「借方總額 = 貸方總額」恆等,對賬就是在驗這個數學不變量有沒有被破壞。
> - **日切窗口是硬約束**——對賬必須在下一個業務日開始前跑完。1 億筆逐筆比對,**單機跑不完 → 按帳戶/時間分片並行 / 走 Spark 批處理**(回到第 2 步的估算)。
> - **對賬本身要可審計**——對賬結果、差錯、補償動作全要留痕(合規)。

> **面試話術(把對賬講成「設計的一部分」而非補丁):** 「支付我一定會做對賬,而且**對賬是設計的一部分不是補救**——因為有第三方、有網絡、有極端時序,我**無法保證每筆都對**,所以用『事前冪等控制 + 事後對賬兜底』雙保險。對賬分內部對賬(自己多張表互對)和渠道對賬(和銀行對帳單逐筆比),找**長款(渠道有我無,通常回調丟了)和短款(我有渠道無,通常我誤判成功)**,能自動補的補、補不了的掛人工。對賬是 T+1 離線批,1 億級要分片並行,差錯率超閾值就告警——它本身就是個要做容量規劃的系統。」

> **錨點(重錨,絕不重寫):** 對賬的完整模型(數據源、對賬類型、差錯分類、修復工作流、可審計)是 `financial-consistency/07-reconciliation/` 整個子目錄(8 檔)——`01-reconciliation-model.md`(模型)、`03-reconciliation-types.md`(內部/渠道對賬類型)、`04-difference-classification.md`(長款短款差錯分類)、`05-repair-workflow.md`(修復流程)、`07-verification-and-audit.md`(可審計)。**這是對賬的權威深礦,本章只是它的「面試打法摘要 + 入口」。**

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步講清三大命門就夠拿 offer;這一段是 `00` 的**架構師決策線**——支付的架構師深度,核心是「**正確性的代價怎麼算、邊界在哪**」。

### 成本:支付不花在算力,花在「正確性的稅」和「合規存儲」

支付和短鏈一樣**算力便宜**(QPS 小),但它有兩筆別題沒有的「稅」:

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **計算** | QPS 幾千,幾台機器 | 幾乎不花錢,**別過度配 / 別秀分片** |
| **正確性的稅** | 冪等表、消息表、對賬批處理、補償任務——都是「為了不出錯」多花的 IO 和算力 | 這是**必要成本不是浪費**——但要控制:防重表要**過期清理**(否則無限增長)、對賬要**分片並行**控窗口 |
| **合規存儲** | 帳務只增不刪、存多年(700 TB+ 級) | 主要成本項。**冷熱分離**:熱帳在線庫,老帳歸檔冷存儲 / 數倉,對賬和審計從歸檔讀 |
| **人工核查** | 補不了的差錯要人工 | 差錯率是**運營成本**——差錯率高 = 上游有 bug,要從源頭治,別靠人海 |

> **架構師話術:** 「支付的成本不在算力——QPS 才幾千。成本在兩處:**正確性的稅**(冪等表、消息表、對賬批處理是必要開銷,但防重表要過期清理別讓它無限長);和**合規存儲**(帳務只增不刪存多年,熱帳在線、老帳歸檔到冷存儲,對賬從歸檔讀)。差錯率是運營成本——差錯多說明上游有 bug,要從源頭治。」

### 容量規劃:對賬窗口和帳務增長是兩條要算的線

| 維度 | 規劃 |
|---|---|
| **核心 QPS** | 幾千~上萬,分庫即可,**永遠不是瓶頸**——但大促要預留 headroom(`02` 留 30–50%) |
| **對賬窗口** | T+1 必須跑完。對賬規模**隨業務線性漲**,要算「業務漲到 N 倍時,日切窗口還夠不夠」——不夠就**分片並行 / 上 Spark** |
| **帳務存儲** | 線性增長、只增不減,要算「幾年後在線庫放不下 → 歸檔策略」 |

### 演進路徑:從 MVP 到規模(支付的演進是「正確性逐層加碼」)

```
階段 1(MVP):单库 + 本地事务,扣款记账原子,加幂等表
   QPS 几百,单库强一致全搞定;先把「一分钱不错」做对,别碰分布式
        │  跨服务(通知下游/外部渠道)出现
        ▼
階段 2:本地消息表 + MQ + 渠道异步回调 + 对账
   核心强一致不变,跨服务用最终一致 + 对账兜底
        │  单库扛不住 / 账户量大
        ▼
階段 3:按 account_id 分库,核心事务落单片;对账分片并行
   分片键选「事务局部性」,把分布式事务范围压到最小
        │  转账等需要资源预留的强一致场景
        ▼
階段 4:TCC / Saga 编排长流程;多活容灾(RPO=0 是支付的硬要求)
```

> **這條演進的關鍵 insight:** 支付**和短鏈相反**——短鏈是「終局架構第一天就知道,但別第一天上」;支付是「**第一天就要把正確性做對(冪等+本地事務+對賬),分布式是後加的**」。**先把單庫的『一分錢不錯』做扎實,再逐層加分布式一致性**——一上來就 TCC/Saga 全家桶是 `01` 紅旗「過度設計」在支付上的變體。**而且支付有個別題沒有的硬約束:RPO 必須 = 0(不能丟任何一筆已確認的支付)**,所以多活/容災的複製方案要比一般系統嚴格(錨 `system-design/08 容災-RPO-RTO`)。

### 失效模式(`01` Step 7:支付的失效要問「錢會不會錯」)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **扣款成功但回調丟了** | 用戶被扣錢但狀態還是 paying(體驗差 + 看著像沒付) | **主動查詢補償**(別等回調)+ **對賬兜底**(長款差錯會被揪出) |
| **消息表投遞器掛** | 下游收不到通知(積分/記帳延遲) | 投遞器**重啟後續投**(消息在表裡不丟);**對賬最終兜底** |
| **並發扣同一帳戶** | 餘額可能扣穿(扣成負) | **樂觀鎖 version / 行鎖** + 餘額不足校驗;扣款必須**條件更新** `WHERE balance >= amount` |
| **第三方渠道掛 / 超時** | 支付卡在 paying | 超時**主動查詢**確認最終態;渠道長時間不可用 → **熔斷 + 切備用渠道**(錨 `system-design/01 韌性`) |
| **重複扣款(最嚴重)** | 直接吃用戶的錢 | **四層冪等**(deep dive ①)+ 對賬發現重複收款 → 退還 |

> **架構師話術:** 「支付的失效我全部問同一個問題:**錢會不會錯?** 扣款成功回調丟了——靠主動查詢 + 對賬把長款揪出來;並發扣同帳戶——樂觀鎖 + 條件更新 `WHERE balance >= amount` 防扣穿;最嚴重的是重複扣款——四層冪等 + 對賬雙保險。**注意我所有兜底最後都收斂到對賬**——這就是為什麼對賬不是可選項。我的取捨方向永遠是 **fail-safe:寧可拒絕一筆、卡在 paying 讓用戶重試,也絕不錯扣一分錢**。」

### 何時**不該**這麼設計(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **純內部積分 / 虛擬幣**(不涉真金、可容錯) | 別上對賬 + 多渠道 + RPO=0 全家桶 | 一張帳戶表 + 流水 + 冪等即可,正確性要求降一檔 |
| **超高並發的庫存扣減**(秒殺) | 別硬套支付的強一致本地事務 | 那是 `16 秒殺` 的題——**Redis 預扣 + 異步落庫 + 超賣防護**,正確性語義不同(可少賣不可超賣 vs 一分不錯) |
| **金額無關的狀態流轉**(訂單狀態) | 別上複式記帳 + 對賬 | 普通狀態機 + 冪等即可,沒有「錢守恆」不變量 |
| **能接受強同步阻塞的小範圍兩庫** | 不是不能用 2PC,只是支付鏈路不適合 | 小範圍、低並發、無外部參與者時 2PC/XA 也是選項——**但支付的鏈路長 + 有外部渠道,所以排除** |

> **這張表是架構師線的精華**——「知道支付這套方案的邊界、知道**秒殺的正確性語義和支付不同**(可少賣 vs 一分不錯)」,比「會做支付」高一個段位(`00` 標準線:「畫得出」→「擁有決策」)。**特別是「秒殺 ≠ 支付」這個區分,是 T3 兩道強一致題的分水嶺,講出來直接過架構師線。**

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 支付題的估算結論一句話? | **QPS 不是瓶頸**(核心幾千、大促上萬,分庫即可)——難在**正確性,一分錢不能錯**。真正增長的是帳務(只增不刪存多年,冷熱分離)和對賬批量(1 億級逐筆比對要分片並行)。**主動拒絕「秀規模」這個錯誤方向。** |
| 2 | 為什麼支付必須冪等? | 分布式裡**重試是常態**:超時分不清「請求沒到」還是「回應丟了」(兩將軍問題),MQ at-least-once 必重複,回調重投。**無法消除重複,只能讓重複無害**——讓「執行 N 次 == 1 次」。 |
| 3 | 防重複扣款的四道防線? | ① 冪等鍵 + 防重表(攔大多數)② 冪等鍵**唯一索引**兜底(攔 TOCTOU 並發競態)③ 狀態機**條件 UPDATE** 正向流轉(重複回調=更新 0 行)④ 帳務 `biz_id` 唯一(防重複記帳)。**絕不先查再寫,靠存儲原子性兜底。** |
| 4 | 跨服務扣款+記帳為什麼不用 2PC?用什麼? | 2PC:**同步阻塞鎖資源、協調者單點卡死、外部渠道無法加入**。改用**本地事務(核心強一致)+ 本地消息表/TCC/Saga(跨服務最終一致)+ 對賬兜底**。 |
| 5 | 本地消息表 vs TCC vs Saga 怎麼選? | **本地消息表**:扣款+待發消息同事務,最簡單最常用(主力);**TCC**:要資源預留的強一致(轉帳凍結),侵入業務;**Saga**:多步長流程,補償回滾,無隔離性。大多數支付場景**本地消息表夠用**。 |
| 6 | 「不丟不重」怎麼保證? | **不丟**=消息和業務同本地事務(成功則消息一定在表裡);**不重**=at-least-once 投遞 + **下游冪等消費**(重了無害);**漏網的**=**對賬兜底**。 |
| 7 | 為什麼必須對賬?對什麼?找什麼? | 無法 100% 保證每筆對(有第三方/網絡/bug)→ **事前冪等 + 事後對賬雙保險,對賬是設計的一部分**。對:**內部多帳互對 + 渠道對帳單**。找:**長款**(渠道有我無,回調丟了)、**短款**(我有渠道無,誤判成功)、金額不符。 |
| 8 | 支付的失效取捨方向?和秒殺有何不同? | 一律 **fail-safe:寧拒絕/卡 paying 讓重試,絕不錯扣一分**;所有兜底收斂到**對賬**。和秒殺不同:**秒殺可少賣不可超賣**(Redis 預扣+異步落庫),**支付一分不錯**(本地事務+複式記帳)——正確性語義不同,別互套(秒殺見 `16`)。 |

## 一句話總綱(背起來)

> **支付是「反規模」題——QPS 算出來才幾千、分庫就夠,難點全在『一分錢不能錯』,所以開場就主動拒絕秀規模、把腦力全投到正確性。三大命門:① 冪等(重試是常態,四道防線——防重表+冪等鍵唯一索引+狀態機條件更新+流水唯一,絕不先查再寫);② 跨服務一致性(不用 2PC,核心走本地事務強一致、跨服務走本地消息表/TCC/Saga 最終一致,不丟靠同事務、不重靠冪等消費);③ 對賬(無法保證每筆都對,事前控制+事後對賬雙保險,找長款短款差錯補償)。架構師線:正確性的稅 + 合規存儲是成本、RPO=0 是硬約束、所有失效兜底收斂到對賬、fail-safe 寧拒絕勿錯扣、秒殺≠支付(可少賣 vs 一分不錯)。一致性/事務/對賬的實作細節全錨 `financial-consistency/`,本章只教打法與指進。**
