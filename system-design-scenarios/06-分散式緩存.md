# 06 · 分散式緩存

> 題目:**設計一個分散式緩存系統**(Redis Cluster / Memcached 那一類)。
>
> 這題的特殊之處:**它本身就是別的題的零件**。News Feed 緩存 feed、限流器把計數放 Redis、排行榜用 ZSet——它們都在「用」緩存。這一章是唯一一道把鏡頭轉過來,**設計緩存自己**的題:數據怎麼分到多台、擴容怎麼搬、命中率怎麼撐住後端、熱點 key 怎麼不打爆單機。
>
> 面試裡它有兩種出法:① 直接「設計一個分散式緩存」(考數據分布 + 擴縮容);② 藏在別的題裡當 follow-up——「你說加 Redis,那**緩存擊穿了怎麼辦**?**緩存和 DB 不一致怎麼辦**?」第二種更高頻,而且**緩存四件套是 SDI 被追問次數最多的單一知識點**。這章兩種都覆蓋。

---

## 第 1 段 · 題目 + 澄清問題清單

**一句話收斂:** 設計一個**對應用透明、橫向可擴、單點不致命**的 KV 緩存層,坐在應用和 DB 之間,把熱數據放內存,扛住讀流量、保護後端。

上來別開畫,先問三類:

| 類 | 問什麼 | 為什麼這問題改變設計 |
|---|---|---|
| **功能** | 純 KV 點查(get/set/del)還是要 ZSet/Hash 等結構?要不要 TTL 過期?要不要持久化(掛了能恢復 vs 純揮發)? | 純 KV → Memcached 心智;要結構/原子操作 → Redis 心智。要持久化 → 多一條 AOF/RDB 路徑 |
| **非功能(SLA)** | 命中率目標?P99 延遲?可用性(掛了能不能降級回 DB)?**緩存是「加速層」還是「事實源」**? | 加速層 → 丟數據可接受、優先可用性;事實源(如 session)→ 要持久化 + 高可用,設計完全不同 |
| **規模 / 約束** | 數據總量(working set 多大)?QPS?key/value 平均大小?讀寫比?有沒有熱點(明星/秒殺)? | working set 決定要幾台機器裝得下;QPS 對照單機 10 萬決定分幾片;有熱點 → 必須提前設計打散 |

**面試話術(澄清):**
> 「我先確認兩件最影響設計的事:第一,**這個緩存是加速層還是事實源**?如果是加速層,我可以接受偶爾丟 key、優先做可用性和降級;如果存的是 session 這種事實源,我要上持久化和強一致複製,那是另一套。第二,**有沒有熱點 key**?如果有秒殺商品、明星這種單 key 打爆單機的場景,我會在一開始就把熱點打散設計進去,而不是等掛了再補。」

**這題的隱藏陷阱(資深會主動點破):**
- **「緩存」這個詞太泛**——它至少有三種角色:旁路加速(Cache-Aside,絕大多數)、寫穿透(Write-Through,緩存即事實源)、會話存儲(session,丟了用戶掉線)。**先問清角色,後面一致性方案才不會講錯。**
- **本機緩存 vs 分散式緩存不是二選一**——成熟方案是**多級緩存**(L1 進程內 + L2 分散式),這在熱點 deep dive 裡是核心武器,澄清時可以埋一句。

## 第 2 段 · 容量估算(調用 `02` 的數字感)

緩存題的估算和別的題不同——**核心約束不是 QPS,是「working set 放不放得進內存」**。內存比磁盤貴 10–20 倍,放不下就退化成「緩存抖動」(cache thrashing),命中率崩盤。所以這題要算三樣:**內存夠不夠 → 分幾片(QPS) → 命中率對 DB 的放大**。

> 用 `02` 的基準數:**一天 ≈ 1e5 秒、單 Redis ≈ 10 萬 QPS、行記錄 ~1KB、副本 ×3、峰值 ×3**。

### 假設一組規模(典型電商緩存)

| 參數 | 假設 |
|---|---|
| 要緩存的對象數(熱數據) | 1 億條(商品、用戶、session…) |
| 平均 value 大小 | 1 KB |
| 讀 QPS(平均) | 50 萬 |
| 讀寫比 | 10:1 |
| 命中率目標 | 95% |

### ① working set 放不放得進內存(這題的命根子)

```
裸數據 = 1e8 × 1KB = 100 GB
× Redis 對象開銷(每 key 約 50–100B 元數據 + 數據結構開銷)≈ ×1.5 → 150 GB
× 副本(主從各一份)×2 → 300 GB
留 headroom(別超過 70%,否則觸發淘汰/碎片)÷ 0.7 → ~430 GB
```

> **所以:** 單機 Redis 通常配 16–64 GB(超過後 fork 做 RDB/主從同步會卡、故障恢復慢),按 32 GB/實例算,430 GB ÷ 32 ≈ **需要 ~14 個分片(主)**,加從庫共 28 個實例。**這一步就否決了「單機 Redis」,直接進入分散式。**

### ② QPS 對照單機,確認分片數夠

```
讀峰值 = 50 萬 × 3 = 150 萬 QPS
單 Redis ~10 萬 QPS → 150萬 / 10萬 = 需要 ~15 個分片扛讀
```

> **所以:** 內存算出 14 片、QPS 算出 15 片,**兩條約束撞在一起 ≈ 15 片**——這是巧合也是常態:數據量和訪問量通常同向增長。取較大者,**定 16 片**(湊 2 的冪,擴容好對半切)。

### ③ 命中率對後端 DB 的放大(緩存的核心價值,必算)

這是緩存題最該算、最能體現「站在數字上」的一步:

```
讀峰值 150 萬 QPS,命中率 p:
  打到 DB 的 QPS = 150萬 × (1 - p)

p = 95%  → DB 承受 150萬 × 0.05 = 7.5 萬 QPS
p = 99%  → DB 承受 150萬 × 0.01 = 1.5 萬 QPS
p = 90%  → DB 承受 150萬 × 0.10 = 15 萬 QPS   ← 單 MySQL 寫幾千、讀萬級,撐不住
```

> **面試話術(把命中率焊到架構上):**
> 「命中率每掉 1%,後端壓力是**非線性**放大的。從 99% 掉到 95%,DB 的 QPS 從 1.5 萬漲到 7.5 萬——**5 倍**。所以緩存題裡命中率不是個漂亮指標,它直接決定後端要不要再分庫。這也是為什麼緩存雪崩(命中率瞬間歸零)這麼致命——DB 要在一瞬間承受平時 20 倍的流量,基本必崩。」

### 估算小結(寫白板角落)

| 結論 | 數字 | 接到架構 |
|---|---|---|
| working set | ~430 GB(含副本+headroom) | **單機裝不下 → 必須分片** |
| 分片數 | ~16 片(內存 14 + QPS 15 取大湊 2 冪) | 16 主 + 16 從 |
| 命中率敏感度 | 99%→95% 時 DB 壓力 ×5 | **命中率是 SLA,雪崩=DB 必崩** |

## 第 3 段 · API + 數據模型 + 分片鍵

### 對外 API(就是緩存協議本身)

緩存的「API」就是它的命令集。設計時要明確**哪些是原子的**(這決定了能不能用它做分散式鎖、計數器):

```
GET    key                         → value | nil
SET    key value [EX ttl] [NX]     → ok        # NX = 不存在才設(原子,做鎖用)
DEL    key                         → count
INCR   key                         → newVal    # 原子自增(計數器/限流)
MGET   key1 key2 ...               → [values]  # 批量,省 RTT
EXPIRE key ttl                     → ok
# 結構化(Redis 心智):
ZADD / ZRANGE  (排行榜)  HSET / HGET (對象)  EVAL script (Lua 原子複合操作)
```

**要點:**
- **`SET key value NX EX ttl` 是一等公民**——它是「原子地『不存在才設且帶過期』」,分散式鎖、緩存擊穿互斥、冪等去重全靠它。面試提到鎖一定要說 `NX`,否則是錯的(check-then-set 有競態)。
- **`EVAL`(Lua 腳本)= 服務端原子複合操作**——多個命令要原子執行(如「讀計數→判斷→自增」)時用,不會被別的客戶端命令插隊。限流器的滑動窗就靠它(錨 `04 限流器`)。

### 數據模型 / 內部結構(緩存「自己」怎麼存)

緩存沒有業務表,它的「數據模型」是**內部如何組織 key-value + 過期 + 淘汰**:

| 內部結構 | 作用 | 黑盒裡是什麼 |
|---|---|---|
| 主哈希表(dict) | key → value 的索引 | 開鏈哈希,擴容時**漸進式 rehash**(分多次搬,不一次卡死) |
| 過期字典(expires dict) | key → 過期時間戳 | 獨立於主表;過期采**惰性 + 定期**雙策略刪除(見 deep dive) |
| 淘汰元數據 | LRU/LFU 排序信息 | Redis 用**近似 LRU**(隨機採樣,非全局鏈表,省內存) |

### 分片鍵(這題的命根子之一)

**分片鍵就是 key 本身**——按 key 的哈希決定它落在哪個分片。關鍵設計問題:

1. **怎麼把 key 映射到分片?** → 一致性哈希 vs Redis 16384 slot(**deep dive ①** 的全部內容)。
2. **怎麼保證「該在一起的 key 落同一片」?** → Redis 的 **hash tag**:`{user123}:profile` 和 `{user123}:cart` 用 `{}` 裡的內容算哈希,保證同片,讓 `MGET`/事務/Lua 能跨這幾個 key 工作(否則 cluster 下跨片命令直接報錯)。

**面試話術(分片鍵):**
> 「分片鍵就是 key。但有個坑:Redis Cluster 下,**跨分片的多 key 命令(MGET、事務、Lua)會直接拒絕**。所以如果業務上有『一個用戶的多個 key 要一起讀/原子操作』,我會用 **hash tag**——把 `{userId}` 用花括號框起來,讓這組 key 強制落同一片。代價是:如果某用戶是超級大戶,他的所有 key 擠在一片,**可能造成數據傾斜**——這個我留到 deep dive。」

## 第 4 段 · 高層架構(一鏡到底)

```
                                ┌─────────────── 控制面 ───────────────┐
                                │  集群管理(Gossip / 協調器)          │
                                │  · 維護 slot→node 映射                │
                                │  · 故障檢測(PFAIL→FAIL 投票)        │
                                │  · 主從切換 / slot 遷移調度           │
                                └───────────────────────────────────────┘
   App                                          ▲ (節點間 Gossip 心跳)
    │                                            │
    │  ┌── L1 本地緩存(進程內,可選,熱點用)──┐
    │  │   miss ↓                              │
    ▼  ▼                                       │
 ┌─────────────┐   key 算 slot   ┌──────────────────────────────────┐
 │ Cache Client│ ──────────────► │  分片集群(L2 分散式緩存)        │
 │ (smart      │                 │                                  │
 │  client /    │  ◄── MOVED ──── │  Shard0 [slot 0-5460]  主─從     │
 │  proxy)      │   (路由糾正)    │  Shard1 [slot 5461-10922] 主─從  │
 └─────────────┘                 │  Shard2 [slot 10923-16383] 主─從 │
       │ miss / 回源              └──────────────────────────────────┘
       ▼
 ┌─────────────┐         寫:先寫 DB,再刪緩存(Cache-Aside)
 │   後端 DB    │ ◄────── (一致性方案見 deep dive ②)
 └─────────────┘
       │
       └─► (可選)CDC / binlog → 訂閱者刪緩存(解耦一致性)
```

**沿著請求走兩條路徑(讀 + 寫):**

- **讀(get user:123):**
  1. App 先查 **L1 本地緩存**(若啟用)——命中直接返回(ns 級,連網絡都省)。
  2. miss → Cache Client 對 key 算 CRC16 → 取模 16384 得 slot → 查本地 slot 映射表 → 直連對應分片主節點。
  3. 分片命中 → 返回;若客戶端映射過期(剛擴容過),節點回 **MOVED 重定向**,客戶端糾正映射並重試。
  4. 分片 miss → App 回源 DB,讀到後 **set 回緩存**(填充)→ 返回。

- **寫(update user:123):**
  1. App 先寫 DB(事實源)。
  2. 再 **刪除**緩存 key(不是更新,見 deep dive ②)。
  3. 下次讀自然 miss → 回源 DB → 重新填充最新值。

**要點(每個組件解決什麼,一句話):**
- **Smart Client / Proxy**:把 key 路由到正確分片。Redis Cluster 用 smart client(客戶端持有 slot 映射,直連,少一跳);Twemproxy/Codis 用 proxy(客戶端無腦,proxy 路由,多一跳但客戶端簡單)。**這是一個經典 tradeoff,deep dive ① 細講。**
- **控制面**:Redis Cluster **去中心化**(無單獨協調器,節點間 Gossip 自己選主/搬 slot);Codis/老派方案**有中心協調器**(ZK/etcd 存映射)。
- **L1 本地緩存**:不是必選,但它是**熱點 key 的終極解藥**(deep dive ③)——熱 key 在每台 App 進程內各存一份,單 key 流量根本不出 App 進程。

## 第 5 段 · Deep Dive(三個最會被追問的點)

---

### Deep Dive ① · 數據分布:一致性哈希 vs Redis 16384 slot

**問題的本質:** 怎麼把 key 映射到 N 個分片,使得**擴縮容時只搬一小部分 key**(而不是全量重新洗牌)?

#### 為什麼不能用「取模」(`hash(key) % N`)

最樸素的做法,但**一加機器就災難**:N 從 4 變 5,`% 4` 和 `% 5` 對幾乎所有 key 都給出不同結果——**約 80% 的 key 要搬家**,緩存瞬間大面積 miss → 雪崩。所以取模直接出局,這也是一致性哈希存在的理由。

#### 方案 A:一致性哈希(Memcached 客戶端 / Cassandra / DynamoDB 心智)

**黑盒裡發生什麼:**
- 把哈希空間想成一個 **0 ~ 2³²-1 的環**。每個節點算哈希,落在環上一個點。
- 一個 key 算哈希,落環上一點,**順時針找到的第一個節點**就是它的歸屬。
- **加一台機器**:它插進環上某點,**只有「它到上一個節點」這段弧的 key 需要搬**(從下一個節點手裡接過來),其餘節點紋絲不動。搬遷量 ≈ `1/N`,不是 80%。

**但有個致命缺陷——數據傾斜:**
- 節點少時,它們在環上隨機分布**很不均勻**,可能一個節點負責半個環、另一個只負責一小段 → 負載嚴重不均。
- **解法:虛擬節點(virtual nodes)。** 每個物理節點不是在環上放 1 個點,而是放 **150~300 個虛擬點**(`hash(node#1)`, `hash(node#2)`…)。物理節點的負載 = 它所有虛擬點負責的弧長之和。虛擬點一多,**大數定律拉平了分布**,每個物理節點分到的弧長趨近 `1/N`。
- **副作用紅利:** 摘除一個故障節點時,它的負載**被打散分攤給多個其他節點**(因為它的虛擬點散布在環各處),而不是全壓給「環上下一個」——避免了故障轉移時的二次雪崩。

> **面試話術(虛擬節點):**
> 「裸一致性哈希在節點少時傾斜很嚴重——可能一個節點扛一半。所以工程上一定配**虛擬節點**:一個物理節點在環上放幾百個虛擬點,用大數定律把負載拉平。額外好處是節點掛掉時,它的負載是**打散**分攤的,不會把壓力全甩給鄰居造成連環雪崩。」

#### 方案 B:Redis Cluster 的 16384 slot(預分槽)

**黑盒裡發生什麼:**
- 不用連續的哈希環,而是**預先切成 16384 個固定的槽(slot)**。`slot = CRC16(key) % 16384`。
- **slot → node 的映射**是一張顯式的表,存在每個節點裡(通過 Gossip 同步)。比如 slot 0-5460 歸 NodeA,5461-10922 歸 NodeB……
- **加節點**:不是搬「一段弧」,而是**把若干完整的 slot 從舊節點遷給新節點**(連同 slot 裡的所有 key)。遷移以 slot 為原子單位,粒度可控、進度可見。

**為什麼是 16384 不是更大?** 兩個工程權衡:① 節點間用 Gossip 交換「我負責哪些 slot」的 bitmap,16384 位 = 2KB,心跳包能塞下;65536 位 = 8KB,心跳開銷翻 4 倍。② Redis Cluster 設計上限約 1000 節點,16384 / 1000 ≈ 每節點 16 個 slot,粒度已足夠細。**這是「心跳包大小」和「遷移粒度」之間的平衡點**——能答出這個,就是把黑盒講透了。

#### 兩者對比(白板上畫這張表)

| 維度 | 一致性哈希(+虛擬節點) | Redis Cluster 16384 slot |
|---|---|---|
| 映射方式 | 哈希環,順時針歸屬 | 固定 16384 槽,顯式 slot→node 表 |
| 均勻性 | 靠虛擬節點(幾百個/物理節點)拉平 | 槽預切均勻,天然較均 |
| 擴容搬遷單位 | 環上一段弧的 key | 整數個 slot(連帶其 key) |
| 搬遷量 | ≈ 1/N | 可精確控制搬幾個 slot |
| 映射存哪 | 客戶端算(無中心表) | 顯式表,Gossip 同步到所有節點 |
| 遷移進度 | 不易觀測(弧是連續的) | **可見**(剩幾個 slot 沒搬) |
| 跨 key 操作 | 難保證同片 | hash tag `{}` 強制同 slot |
| 代表 | Memcached 客戶端、Cassandra、DynamoDB | Redis Cluster |

**何時選哪個:**
- **純客戶端分片、無中心協調**(如 Memcached 多實例)→ 一致性哈希 + 虛擬節點,簡單、無服務端集群協議。
- **要原子多 key、要結構化、要可觀測的遷移**→ Redis Cluster slot,代價是要跑集群協議(Gossip + 故障投票)。

> 一致性哈希與故障檢測的完整原理(環、虛擬節點、Quorum、SWIM/Gossip 故障檢測)在 **`system-design/03 複製-Quorum-一致性哈希-故障檢測`**——這裡只講它在緩存場景的取捨,原理回那章。

#### 擴縮容遷移:只搬部分 key 的原理(這是面試常追問的)

不管哪種方案,擴容的核心都是「**只搬該搬的那部分,且搬遷期間不停機**」。Redis Cluster 的遷移黑盒:

1. 協調指令:`CLUSTER SETSLOT <slot> MIGRATING`(源)+ `IMPORTING`(目標)——標記這個 slot 正在搬。
2. 逐個 key 搬:`MIGRATE` 命令把 key **原子地**從源序列化發到目標。
3. **搬遷期間的讀寫怎麼辦?**(關鍵)——客戶端請求一個正在遷移的 slot:
   - key 還在源 → 源正常服務。
   - key 已搬到目標 → 源回 **ASK 重定向**(臨時的,告訴客戶端「這個 key 這次去目標問」),客戶端帶 `ASKING` 標誌去目標取。
   - **ASK vs MOVED 的區別**:MOVED 是永久的(映射變了,更新本地表);ASK 是臨時的(只這一次,別更新表,因為遷移還沒完)。能答出這個區別 = 真懂遷移。
4. 全部 key 搬完 → `SETSLOT NODE`,正式把 slot 歸屬改到目標,廣播 MOVED。

> **面試話術(擴容不停機):**
> 「擴容時我不需要停服。Redis Cluster 以 slot 為單位逐個搬,**搬遷中的 slot 對讀寫用 ASK 臨時重定向**——key 在哪邊就去哪邊問,客戶端無感。和 MOVED 的區別是:ASK 是『就這一次』的臨時跳轉,不更新本地路由表;遷移徹底完成後才發 MOVED 讓客戶端永久更新。所以整個擴容過程**只有正在搬的那個 slot 有一瞬間的多一跳,其餘流量零影響**。」

### Deep Dive ② · 緩存四件套 + 緩存與 DB 一致性

這是**整個 SDI 被追問次數最多的單點**。四件套必須能脫稿背 + 講清每個方案的代價。先記一句:**穿透/擊穿/雪崩 = 三種「miss 風暴打到 DB」的場景,差別在「miss 的是什麼 key、有多少」。**

#### 先定基:Cache-Aside(旁路緩存)

絕大多數緩存都用這個模式,它定義了讀寫流程,四件套都是在它上面打補丁:

```
讀:  查緩存 → 命中返回 / miss → 查 DB → 寫回緩存 → 返回
寫:  寫 DB → 刪除緩存 key(不更新!)
```

> Cache-Aside 的完整流程、和 Write-Through/Write-Back 的對比在 **`redis-handson/06-caching-patterns`**,這裡聚焦它的失效場景。

#### 三件套對照(背這張表)

| | 緩存穿透 | 緩存擊穿 | 緩存雪崩 |
|---|---|---|---|
| **是什麼** | 查**根本不存在**的 key,緩存永遠 miss,每次都打 DB | **一個熱 key** 過期的瞬間,大量並發同時 miss,一起打 DB | **大量 key 同時過期**(或緩存集群宕機),命中率瞬間歸零,洪流打 DB |
| **典型觸發** | 惡意攻擊(刷不存在的 id)、爬蟲 | 熱門商品 key 剛好到期 | 大批 key 用了同一個 TTL 一起到期;或整個 Redis 掛 |
| **打到 DB 的量** | 持續的、針對不存在 key | 瞬間的、針對單個熱 key | 海量的、針對所有 key |
| **解法 1** | **布隆過濾器**(不存在直接擋) | **互斥鎖**(只放一個請求回源) | **隨機 TTL**(打散過期時間) |
| **解法 2** | **緩存空值**(null 也緩存,短 TTL) | **邏輯過期**(永不真過期,後台異步刷新) | **多級緩存** + 緩存集群高可用 + 限流降級兜底 |

#### 黑盒裡發生什麼(逐個講透)

**穿透 · 布隆過濾器(Bloom Filter):**
- 把「所有存在的 key」預先灌進一個布隆過濾器(一個位數組 + 多個哈希函數)。查 key 前先問布隆:**「絕對不存在」就直接擋掉,不查緩存也不查 DB。**
- 黑盒:布隆**有假陽性無假陰性**——它說「不存在」一定不存在(可放心擋);說「可能存在」才可能誤判(放行去查,最壞就是查一次)。用 ~10 bit/key 可把假陽性壓到 ~1%。
- 代價:① 新增 key 要同步寫布隆;② 標準布隆**不支持刪除**(刪 key 後布隆仍說「可能存在」)→ 要刪就上 Counting Bloom 或 Cuckoo Filter。
- **兜底搭配:緩存空值**——查到 DB 確實沒有,就把 `key→null` 也緩存(短 TTL,如 60s),擋住同一個不存在 key 的重複查詢。布隆擋「沒見過的」,空值擋「查過確實沒有的」,兩者互補。

**擊穿 · 互斥鎖 vs 邏輯過期:**
- **互斥鎖**:熱 key miss 時,**只有搶到鎖(`SET lock NX EX`)的那個請求去回源 DB**,其餘請求短暫等待/重試讀緩存。回源的那個填好緩存、釋放鎖,大家就都命中了。
  - 黑盒代價:其餘請求被**阻塞**了一小段(等鎖)——犧牲了一點延遲,換來 DB 只被打一次。鎖要帶過期(防回源請求掛了死鎖),這又引出鎖的可靠性(錨 `redis-handson/07-distributed-locks`)。
- **邏輯過期**:緩存**永不設真 TTL**(物理上不過期),而是在 value 裡塞一個「邏輯過期時間」字段。讀到發現邏輯已過期 → **返回舊值的同時,異步起一個線程去刷新**。
  - 黑盒代價:**返回的是舊數據**(刷新完成前所有人讀到舊值)——犧牲一致性,換來**永不阻塞、永不擊穿**。適合「能容忍短暫陳舊、但絕不能打爆 DB」的場景(如首頁熱門榜)。
- **二選一**:要強一致 → 互斥鎖(可能阻塞);要絕對不阻塞、能忍舊數據 → 邏輯過期。

**雪崩 · 隨機 TTL + 多級兜底:**
- **隨機 TTL**:設 TTL 時加抖動,`ttl = base + random(0, jitter)`(如 30min ± 5min)。讓 key 的過期時間**散開**,不會在同一秒集體到期。**這是一行代碼級的防雪崩,必說。**
- 但隨機 TTL 只防「集體過期」型雪崩,防不了「整個 Redis 集群宕機」型雪崩——那要靠:
  - **緩存集群高可用**(主從 + 故障自動切換,deep dive ③ 的主從部分)。
  - **多級緩存**:L1 本地緩存頂一陣(Redis 掛了 App 還有進程內緩存)。
  - **限流 + 降級兜底**:緩存全掛時,對 DB 加限流(只放一部分流量),其餘請求直接返回降級內容/錯誤——**保住 DB 不被打死**比「服務每個請求」更重要。這是韌性的核心思想(錨 `system-design/01 韌性`)。

> **面試話術(四件套串講):**
> 「我把這三個一起講,因為它們本質都是『miss 打到 DB』,差別在規模。**穿透**是查不存在的 key——我用布隆過濾器擋『沒見過的』、緩存空值擋『查過確實沒有的』。**擊穿**是單個熱 key 過期瞬間並發 miss——要強一致我用互斥鎖只放一個回源,能忍舊數據我用邏輯過期永不真過期、異步刷。**雪崩**是大批 key 同時失效或集群宕——基礎防護是隨機 TTL 打散過期,但真正的兜底是緩存集群高可用 + 多級緩存 + 對 DB 限流降級。**記一句:雪崩最致命,因為命中率歸零時 DB 要瞬間扛平時 20 倍流量,所以雪崩防的不是緩存,是 DB。**」

#### 第四件:緩存與 DB 一致性(寫的問題)

四件套前三個是「讀」的問題,這個是「寫」的問題,而且**沒有完美解,只有取捨**。核心矛盾:寫 DB 和操作緩存是**兩個不能原子完成的動作**,中間任何順序都有不一致窗口。

**先回答一個高頻陷阱:寫時是「更新緩存」還是「刪除緩存」?**

| | 更新緩存(寫 DB 後寫新值進緩存) | 刪除緩存(寫 DB 後刪掉 key) |
|---|---|---|
| 並發寫覆蓋 | **有 bug**:兩個寫並發,DB 是 B 贏,但緩存可能被慢的 A 後寫覆蓋成 A → 永久不一致 | 無此問題,刪了下次讀自然取最新 |
| 計算浪費 | 寫多讀少時,每次寫都算新值塞緩存,可能塞了沒人讀 | 惰性,讀了才算 |
| 結論 | ❌ 一般不用 | ✅ **標準做法:寫 DB → 刪緩存** |

**但「刪緩存」也有不一致窗口,於是有了一階一階的方案:**

| 方案 | 怎麼做 | 解決什麼 | 殘留問題 / 代價 |
|---|---|---|---|
| 先刪緩存再寫 DB | del → update DB | — | **更糟**:刪後寫前,有讀進來會把舊值重新填回緩存 → 不一致更久 |
| **先寫 DB 再刪緩存**(標準) | update DB → del | 多數情況一致 | 仍有罕見窗口:讀請求在「你寫 DB 前讀到舊值」、在「你刪緩存後才寫回」→ 舊值殘留(概率低,需讀比寫慢) |
| **延遲雙刪** | del → 寫 DB → **睡幾百 ms** → 再 del | 補刪掉「殘留的舊值回填」 | 引入延遲;睡多久難定(要 > 一次讀回填耗時);仍非 100% |
| **訂閱 binlog / CDC 刪緩存** | 寫 DB → DB binlog → CDC(Canal/Debezium)消費 → 刪緩存 | **解耦**:應用只管寫 DB,刪緩存由 binlog 驅動,不會漏刪 | 引入 CDC 組件 + 異步延遲(緩存在 binlog 消費前還是舊的);這是**最終一致**,不是強一致 |

> **面試話術(一致性,主動暴露取捨):**
> 「緩存和 DB 強一致基本做不到,因為這是兩個系統、兩個不能原子的動作。我的默認方案是**先寫 DB 再刪緩存 + 給緩存設 TTL 兜底**——TTL 保證『就算某次刪漏了,最多陳舊 N 秒也會自動過期』。對一致性要求再高一點,上**延遲雙刪**補殘留窗口;要徹底解耦、不怕漏刪,就上 **CDC 訂閱 binlog 來刪緩存**——應用只管寫庫,刪緩存交給 binlog 流,代價是引入 Canal/Debezium 這類組件和一個異步延遲。**關鍵是先問業務:能容忍多大的不一致窗口?** 能容忍秒級,TTL + 先寫後刪就夠;不能容忍,才付 CDC 的複雜度。」

### Deep Dive ③ · 熱點 key 與大 key

分片解決了「總量」和「總 QPS」,但解決不了**單個 key 的流量**——熱點 key 把一個分片打爆,擴再多片也沒用(那個 key 永遠只在一片)。

#### 熱點 key:單 key 打爆單分片

**問題本質:** 秒殺商品、明星微博、首頁 banner——這些 key 的流量可能佔全站 50%,但它只 hash 到**一個** slot、一個分片主節點。這個分片被打到 CPU 100%,其餘 15 個分片閒著。**水平擴容對單熱 key 完全無效。**

**三層解法(從輕到重,面試按這個順序講):**

| 解法 | 黑盒做法 | 代價 |
|---|---|---|
| **L1 本地緩存(多級)** | 熱 key 在**每台 App 進程內**各存一份(Caffeine/Guava),短 TTL(如 5s)。熱 key 流量根本不出 App 進程,連 Redis 都不碰 | 多級一致性更難(本地緩存陳舊窗口 = 它的 TTL);內存佔用 |
| **熱 key 多副本打散** | 把一個熱 key 拆成 N 個物理 key:`hotkey#1`…`hotkey#N`,寫時全寫,讀時**隨機選一個**。N 個副本 hash 到不同分片 → 流量被打散到 N 個分片 | 寫放大 N 倍;讀到的可能是不同副本(要保證副本內容一致) |
| **讀走從庫副本** | 熱 key 所在分片掛多個從庫,讀請求分流到從庫 | 從庫有複製延遲(可能讀到略舊);僅緩解讀熱點 |

> **面試話術(熱點 key):**
> 「熱點 key 最反直覺的點是:**加分片救不了它**——它永遠只在一個分片。所以我分三層打:第一層也是最強的一層,**本地緩存(多級)**——把熱 key 在每台 App 進程內各存一份、設個幾秒的短 TTL,這個 key 的流量根本不出進程,Redis 完全不碰。第二層,**把熱 key 複製成 N 份**(`key#1`…`key#N`)散到不同分片,讀時隨機選一份,把單片流量攤成 N 片。代價分別是:本地緩存帶來幾秒的陳舊窗口、多副本帶來 N 倍寫放大。我會優先用本地緩存,因為熱 key 通常恰好是『讀爆但變化慢』的數據,正適合本地短 TTL。」

**配套:熱 key 怎麼發現?** 不能等掛了才知道。手段:① Redis 4.0+ 的 `--hotkeys`(基於 LFU 採樣);② 客戶端/proxy 層做 key 訪問計數採樣上報;③ 監控分片間 QPS 不均(某片明顯高)。**架構師會把『熱 key 自動發現 + 自動晉升到本地緩存』做成閉環**,而不是人肉救火。

#### 大 key:單 key 體積過大

另一類隱形殺手——**value 特別大**(一個 List/Hash 存了百萬元素,或一個 String 幾 MB)。

**大 key 的黑盒危害(為什麼比想象的嚴重):**
- **Redis 是單線程處理命令**——操作一個大 key(如 `HGETALL` 一個百萬字段的 Hash)會**長時間佔住唯一的命令線程**,阻塞所有其他請求 → P99 飆升。
- **網絡阻塞**:一次傳幾 MB,佔滿網卡、撐爆客戶端緩衝。
- **刪除也危險**:`DEL` 一個大 key 也是同步的,會卡住線程 → 要用 `UNLINK`(異步惰性釋放)。
- **遷移災難**:擴容搬 slot 時,`MIGRATE` 一個大 key 是原子阻塞的 → 搬它時源節點卡死。
- **內存不均**:大 key 讓它所在分片內存暴漲,破壞分片均衡。

**解法:**

| 問題 | 解法 |
|---|---|
| value 太大 | 拆分:大 Hash 按字段哈希拆成多個小 Hash(`bigkey:0`…`bigkey:n`);大 String 壓縮或挪去對象存儲、緩存只存 metadata |
| 大集合全量讀 | 別 `HGETALL`/`LRANGE 0 -1`,改用 `HSCAN`/分頁游標增量讀 |
| 刪除阻塞 | 用 `UNLINK` 代替 `DEL`,異步釋放 |

> **面試話術(大 key):**
> 「大 key 比熱 key 更隱蔽,因為它平時不報錯,但 Redis 是單線程,一個 `HGETALL` 百萬字段的大 Hash 就能把整個分片卡住,把所有其他 key 的 P99 拖垮。而且擴容搬 slot 時,`MIGRATE` 大 key 是原子阻塞的,會讓源節點卡死。所以我會在設計階段就立規矩:**單 key 不超過幾 KB、集合元素不超過幾千**,超了就拆分或挪對象存儲;刪大 key 一律用 `UNLINK` 異步釋放。」

## 第 6 段 · 架構師深挖層(推過面試線)

IC 線到 deep dive 為止已經能拿高分。架構師線要回答的是:**這套緩存的代價誰買單、容量怎麼規劃、怎麼演進、會怎麼掛、何時根本不該上分散式緩存。**

### 6.1 成本:緩存是「拿錢買延遲」,要算清這筆賬

| 成本項 | 黑盒裡的真實開銷 |
|---|---|
| **內存錢** | 內存比磁盤貴 10–20×。430 GB 內存(含副本)的緩存集群,月成本可能是同容量磁盤 DB 的數倍。**緩存命中率每提升要多買多少內存,是條收益遞減曲線**——從 90% 到 95% 很划算(DB 壓力減半),從 99% 到 99.9% 可能要翻倍內存換 0.9% DB 流量,不值。 |
| **複製錢** | 主從 ×2 內存。要不要從庫?從庫只為高可用(故障切換)還是也分擔讀?純加速層 + 能降級回 DB 的場景,**從庫甚至可以不要**——省一半內存,代價是主掛了那個分片短暫不可用(回源 DB)。 |
| **運維錢** | 集群協議(Gossip/故障投票)、slot 遷移、熱 key 救火都是人力。Memcached 簡單但功能少;Redis Cluster 功能全但運維重。 |

> **架構師判斷:** 「命中率目標不是越高越好,是**買到 DB 不需要再分庫的那個點就停**。」算清「再加 X GB 內存能讓 DB 少扛多少 QPS」對「再加一個 DB 分片要多少錢」,取交叉點。

### 6.2 容量規劃:盯三條曲線,別只盯 QPS

| 監控維度 | 為什麼盯它 | 觸發動作 |
|---|---|---|
| **內存水位** | 超 70% → 淘汰開始頻繁 / 碎片 / fork 卡頓 | 加分片(這題擴容主因常是內存不是 QPS) |
| **命中率** | 掉了 → DB 壓力非線性漲;突然掉可能是有人灌了冷數據沖刷熱數據 | 查是否 working set 變大 / 有掃描型流量污染緩存 |
| **分片間 QPS 偏差** | 某片明顯高 → 熱點 key 或數據傾斜 | 觸發熱 key 打散 / hash tag 檢查 |

> **架構師的反直覺點:** **緩存的擴容信號通常是「內存滿」而不是「QPS 滿」**——因為訪問量還沒到瓶頸,working set 先撐爆內存,觸發淘汰、命中率下跌。盯 QPS 會漏掉這個。

### 6.3 演進路徑(從小到大,別一上來就 Cluster)

```
階段1:單 Redis(主從)          ── working set < 單機內存,QPS < 10萬
   │   撐不住:內存超 ~32GB 或 QPS 近 10萬
   ▼
階段2:客戶端分片 / Proxy 分片   ── 一致性哈希,簡單,無集群協議
   │   撐不住:要原子多 key / 要可觀測遷移 / 要自動故障切換
   ▼
階段3:Redis Cluster            ── slot + Gossip + 自動主從切換 + hash tag
   │   撐不住:單熱 key 打爆 / 跨地域
   ▼
階段4:多級緩存(L1本地+L2集群) ── 熱 key 進程內,集群只扛溫數據
   │
   ▼
階段5:跨地域多活緩存            ── 每地域一套,寫穿透 + 失效廣播(複雜,慎上)
```

> **架構師話術:** 「我不會一上來就 Redis Cluster——集群協議的運維成本是真金白銀。**單機主從能撐就單機**,內存或 QPS 撞線了才上分片,要原子多 key 和自動切換了才上 Cluster,有單熱 key 了才上多級緩存。**每一步都是被一個具體瓶頸逼出來的,不是為了架構好看。**」

### 6.4 失效模式(會怎麼掛,怎麼防)

| 失效 | 黑盒裡發生什麼 | 防護 |
|---|---|---|
| **緩存雪崩** | 命中率歸零,DB 瞬間扛平時 20× 流量必崩 | 隨機 TTL + 多級 + **對 DB 限流降級**(保 DB 比保服務重要) |
| **主從切換丟數據** | 異步複製下,主寫了還沒同步到從就掛,切換後新主缺這些寫 → 緩存丟 key | 加速層可接受(回源 DB 補);事實源(session)→ 要更強複製或別用純緩存存 |
| **腦裂(split-brain)** | 網絡分區,舊主和新主同時接寫 → 兩份不一致數據 | Redis `min-replicas-to-write` 配置:主寫前要求至少 N 個從在線,否則拒寫(犧牲可用換一致) |
| **緩存擊穿連鎖** | 熱 key 過期 → 互斥鎖回源 → 回源那個請求慢/掛 → 鎖超時 → 又一批衝進來 | 鎖帶過期 + 邏輯過期兜底 + 回源加超時熔斷(錨 `system-design/01 韌性`) |
| **熱 key 遷移卡死** | 擴容搬 slot 時恰好搬到大 key/熱 key,`MIGRATE` 阻塞源節點 | 遷移前掃出大 key 先拆;低峰期遷移;限制遷移併發 |

### 6.5 淘汰策略:LRU vs LFU(黑盒裡的近似算法)

內存滿了淘汰誰?面試常追問「Redis 的 LRU 是真 LRU 嗎?」——**不是,是近似 LRU**,這是個漂亮的工程取捨:

| 策略 | 淘汰誰 | 黑盒裡怎麼實現 | 適用 |
|---|---|---|---|
| **LRU**(最近最少使用) | 最久沒被訪問的 | **真 LRU 要全局雙向鏈表,每次訪問都移動節點 → 太貴**。Redis 用**近似 LRU**:每個 key 記一個訪問時間戳,淘汰時**隨機採樣 N 個(默認 5),挑其中最舊的踢**。採樣越多越接近真 LRU,但越慢 | 大多數場景;熱數據有時間局部性 |
| **LFU**(最少使用頻率) | 訪問次數最少的 | 每個 key 一個**計數器**,但用**對數遞增 + 隨時間衰減**(不是裸計數,否則老 key 計數永遠最高踢不掉)。同樣隨機採樣淘汰 | 有「偶發掃描沖刷」風險的場景——LRU 會被一次全表掃描沖掉所有熱 key,LFU 不會(掃描的 key 頻率低) |

> **架構師話術(淘汰策略選擇):** 「Redis 的 LRU 是**近似**的——真 LRU 要全局鏈表每次訪問都移動節點,太貴,所以它隨機採樣幾個挑最舊的踢,用一點精度換大量內存和 CPU。**選 LRU 還是 LFU 看怕不怕『掃描污染』**:如果業務有定時全表掃描/報表類流量,LRU 會被它沖掉所有熱 key 導致命中率暴跌,這時 LFU 更穩(掃描的 key 頻率低,不會頂掉真熱 key)。淘汰策略的細節錨在 `redis-handson/04-expiry-eviction`。」

### 6.6 何時不該上分散式緩存(架構師最該答的「反向」問題)

| 不該上的場景 | 為什麼 | 該用什麼 |
|---|---|---|
| working set 能塞進**單機進程內存** | 分散式緩存多一次網絡跳(0.5–1ms);本地緩存是 ns 級 | 純本地緩存(Caffeine),零網絡、零運維集群 |
| **強一致要求**的數據(賬戶餘額) | 緩存與 DB 不可能強一致,加緩存等於引入不一致窗口 | 直接讀 DB / 讀寫走主庫;真要快就上 DB 自己的 buffer pool |
| 寫多讀少 | 緩存收益來自「寫一次讀多次」,寫多讀少時緩存命中率低、刪緩存頻繁,**淨虧** | 別緩存,直接 DB;或只緩存讀路徑的派生視圖 |
| 數據變化極快(實時行情) | TTL 還沒到數據就變了,命中即陳舊 | 推送 / 流式,別緩存 |

> **架構師話術(何時不該):** 「加緩存不是免費的——它引入了一致性窗口、多一個會掛的組件、多一筆內存錢。**如果 working set 能塞進單機進程內存,我直接用本地緩存,連 Redis 都不上**,省掉整個分散式的網絡跳和運維。如果數據要強一致(餘額、庫存的最終值),我寧可讀 DB 也不加緩存,因為緩存和 DB 強一致根本做不到。**緩存是『讀多寫少 + 能容忍短暫陳舊』場景的解,把它用到強一致或寫密集場景就是負優化。**」

## 第 8 段 · 面試卡(抽認卡)

**卡 1 · 數據分布兩條路**
> Q:一致性哈希 vs Redis 16384 slot,怎麼選?
> A:純客戶端分片無中心 → 一致性哈希 + **虛擬節點**(幾百個/物理節點,大數定律拉平傾斜,故障負載打散)。要原子多 key / 可觀測遷移 / 自動切換 → Redis slot(`CRC16%16384`,顯式表 Gossip 同步,hash tag `{}` 強制同片)。取模 `%N` 直接出局——加機器 80% key 搬家。

**卡 2 · 擴容不停機**
> Q:Redis Cluster 擴容時搬遷中的 slot 怎麼讀寫?MOVED vs ASK?
> A:以 slot 為單位逐 key `MIGRATE`。搬遷中:key 在源用源、已搬走回 **ASK 臨時重定向**(就這次去目標,**不更新本地表**)。搬完發 **MOVED 永久重定向**(更新表)。所以擴容零停機,只有正在搬的 slot 多一瞬間的一跳。

**卡 3 · 緩存四件套**
> Q:穿透/擊穿/雪崩各是什麼、怎麼解?
> A:都是「miss 打 DB」,差在規模。**穿透**(查不存在 key)→ 布隆過濾器擋沒見過的 + 緩存空值擋查過沒有的。**擊穿**(單熱 key 過期瞬間並發)→ 互斥鎖只放一個回源(強一致但阻塞)/ 邏輯過期異步刷(不阻塞但舊數據)。**雪崩**(大批同時失效/集群宕)→ 隨機 TTL 打散 + 多級 + 對 DB 限流降級。記:雪崩防的是 DB,不是緩存。

**卡 4 · 緩存與 DB 一致性**
> Q:寫時更新還是刪除緩存?怎麼保證一致?
> A:**刪**不更新(更新有並發覆蓋 bug)。**先寫 DB 再刪緩存 + TTL 兜底**為默認;要更強上**延遲雙刪**(刪→寫→睡→再刪)補殘留窗口;徹底解耦上 **CDC 訂閱 binlog 刪緩存**(最終一致)。先問業務:能容忍多大不一致窗口?能容忍秒級就別付 CDC 的複雜度。

**卡 5 · 熱點 key & 大 key**
> Q:單熱 key 打爆單分片怎麼辦?大 key 為什麼危險?
> A:**熱 key 加分片無效**(永遠在一片)→ ① 本地緩存(進程內存一份,流量不出進程,最強)② 拆成 `key#1..N` 散到多片讀時隨機選(寫放大 N×)。**大 key**:Redis 單線程,`HGETALL` 百萬字段卡住整片 P99;遷移時 `MIGRATE` 阻塞;刪用 `UNLINK` 不用 `DEL`。設計階段立規矩:單 key 幾 KB、集合幾千元素。

**卡 6 · 容量估算 + 何時不該**
> Q:緩存擴容的主因是 QPS 嗎?何時不該上分散式緩存?
> A:**主因常是內存不是 QPS**——working set 先撐爆內存(70% 水位)觸發淘汰、命中率跌。命中率 99%→95% 時 DB 壓力 ×5(非線性)。**不該上**:working set 塞得進單機 → 用本地緩存(省網絡跳+運維);強一致數據(餘額)→ 讀 DB;寫多讀少 → 緩存淨虧。緩存是「讀多寫少+能忍陳舊」的解。

---

## 一句話總綱

> **設計分散式緩存 = 先用「working set 放不放得進內存」算出要分幾片(內存常比 QPS 先撞線),用一致性哈希(虛擬節點防傾斜)或 16384 slot(hash tag 保同片、ASK/MOVED 保擴容不停機)把 key 分到多台,再用緩存四件套(布隆+空值防穿透、互斥鎖/邏輯過期防擊穿、隨機 TTL+多級+限流防雪崩、先寫 DB 再刪緩存+CDC 保一致)和熱點打散(本地緩存+多副本)把它撐成一個命中率扛得住後端、單 key 打不爆、擴容不停機的加速層;而架構師的加分句是『緩存是拿內存錢買延遲、引入一致性窗口的取捨,working set 塞得進單機或數據要強一致時根本不該上』。**
