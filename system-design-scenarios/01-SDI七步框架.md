# 01 · SDI 七步框架

> 這是你跑**任何一道**系統設計題的腳本。背下這條脊柱,上白板就不會慌——你永遠知道「下一步該做什麼、現在該說什麼」。
>
> 17 道題長得千差萬別,但**流程只有一套**。練的就是這套流程的肌肉記憶:看到題 → 自動進入第 1 步,而不是直接開畫。

---

## 七步脊柱(先背這張表)

| 步 | 一句話 | 時間 | 核心產出 |
|---|---|---|---|
| 1 **需求澄清** | 先問清楚要做什麼、多大 | ~5 min | 功能清單 + 非功能 SLA + 規模數字 |
| 2 **容量估算** | 把規模算成 QPS/存儲/帶寬 | ~5 min | 幾個關鍵數量級 |
| 3 **API 設計** | 對外接口長什麼樣 | ~3 min | 幾個核心 endpoint |
| 4 **數據模型** | 數據怎麼存、分片鍵是什麼 | ~5 min | 核心表/KV + 選型 + 分片鍵 |
| 5 **高層架構** | 請求一鏡到底走完 | ~10 min | 一張架構圖(主菜) |
| 6 **Deep dive** | 挑 2–3 個點往死裡鑽 | ~12 min | 細節 + 算法 |
| 7 **瓶頸與取捨** | 主動暴露代價與演進 | ~5 min | tradeoff 清單(加分核心) |

> 記法:**「澄清 → 估算 → API → 數據 → 架構 → 鑽 → 取捨」**。前 4 步是「鋪墊」(快),第 5 步是「主菜」,第 6/7 步是「分高下」的地方。

---

## Step 1 · 需求澄清(別一上來就開畫)

**目的:** 把一道開放題收斂成一道**有邊界**的題。面試官給的題故意模糊(「設計一個 Twitter」),就是要看你會不會問。

**問三類:**

| 類 | 問什麼 | 例(News Feed) |
|---|---|---|
| **功能需求** | 核心用例有哪些?砍掉哪些? | 發帖、看 feed、關注——**先不做**搜索/廣告/私信 |
| **非功能(SLA)** | 可用性、延遲、一致性、規模 | 高可用 > 強一致;feed 延遲幾秒可接受;讀遠多於寫 |
| **規模 / 約束** | DAU、讀寫比、峰值、數據量 | 200M DAU,讀:寫 ≈ 100:1,大 V 千萬粉 |

**面試話術:**
> 「我先確認範圍——核心場景我打算聚焦在發帖和看 feed,搜索和廣告這次先不做,可以嗎?規模上我假設 2 億 DAU、讀寫比大概 100:1,峰值按平均的 3 倍估,這樣可以嗎?」

**常見陷阱:**
- ❌ 一句沒問直接開畫 → 大概率做錯題,後面全白費。
- ❌ 問太細(「密碼要不要加鹽?」)→ 浪費時間,抓大放小。
- ✅ 把「你的假設」**說出來並讓面試官確認**——這既澄清了題,也展示了你知道哪些維度重要。

---

## Step 2 · 容量估算(把規模變成數字)

**目的:** 後面所有架構決策都要「站在數字上」——要不要分庫、要不要加緩存、單機夠不夠,全靠這幾個數。

**算四樣**(詳細數字感和 drill 在 `02`):

1. **QPS**:`平均 QPS = DAU × 人均操作數 / 86400`;`峰值 ≈ 平均 × 2~3`(或更高,看是否有尖峰)。
2. **存儲**:`每天寫入量 = 寫 QPS × 單條大小 × 86400`;再 × 保留期 × 副本數(通常 ×3)。
3. **帶寬**:`= QPS × 單條 payload 大小`(進出分開算,媒體類重點)。
4. **連接 / 內存**:長連接題(IM)算並發連接數;緩存題算 working set 能不能放進內存。

**面試話術:**
> 「2 億 DAU,人均每天讀 100 次 feed,讀 QPS ≈ 2e8×100/8.6e4 ≈ 23 萬,峰值算 70 萬。寫(發帖)人均 2 次,寫 QPS ≈ 5000。**讀寫比 ~100:1,這直接決定我要重點優化讀路徑、上緩存。**」

**常見陷阱:**
- ❌ 跳過估算直接拍架構 → 「你怎麼知道單庫扛不住?」答不上來。
- ❌ 算一堆用不到的數 → 只算**會影響架構決策**的那幾個。
- ✅ 估完馬上**接一句「所以……」**,把數字翻譯成架構含義(像上面那句「所以要優化讀」)。

---

## Step 3 · API 設計(對外長什麼樣)

**目的:** 定義系統邊界。API 一出來,第 5 步的架構就有了「入口」和「出口」。

**怎麼做:** 給核心用例各寫一個 endpoint,標出關鍵參數和返回。REST 或 RPC 都行,風格一致即可。

```
POST /tweets            {content}          → tweetId
GET  /feed?cursor=...&limit=20             → [tweet], nextCursor
POST /follow            {targetUserId}      → ok
```

**要點:**
- 分頁用 **cursor(游標)** 不用 offset——大數據量下 offset 翻頁會越來越慢。
- 寫接口想冪等(帶 requestId / idempotency key),為重試鋪路。

**常見陷阱:** 在字段細節上糾結(要不要駝峰命名)。API 這步**快過**,點到為止,留時間給架構。

---

## Step 4 · 數據模型(怎麼存 + 分片鍵)

**目的:** 決定「數據放哪種存儲、按什麼鍵分片」——這是規模化的命根子。

**三問:**
1. **選型**:強一致交易 → SQL;海量 KV / 寬列 → NoSQL;全文搜索 → ES;排行/集合 → Redis。(決策樹錨 `system-design/06`)
2. **核心實體**:列出主要表/集合和關鍵字段、關係。
3. **分片鍵(sharding key)**:按什麼分?能不能避免熱點和跨片查詢?(錨 `system-design/07`)

**面試話術:**
> 「tweet 按 `userId` 分片,讓一個人的帖子落在一起;但這會讓**大 V 成為熱點分片**——這個我留到 deep dive 講怎麼解。」

**常見陷阱:** 不提分片鍵。海量題不講「按什麼分」等於沒講規模化,這是資深 vs 初級的分水嶺。

---

## Step 5 · 高層架構(主菜:一鏡到底)

**目的:** 把一個請求**從進到出**走完,讓面試官看到完整數據流。這是評分的主體。

**怎麼畫(從左到右一條線):**

```
Client → LB / API Gateway → Service → Cache → DB
                                  │
                                  └→ MQ → 異步 Worker → (索引/通知/...)
```

**走兩條典型路徑**(讀 + 寫各一條):
- 「**發帖**:Client → Gateway → Tweet Service → 寫 DB → 發消息到 MQ → 扇出 Worker 更新粉絲 feed 緩存。」
- 「**看 feed**:Client → Gateway → Feed Service → 讀 Redis(預計算好的 feed)→ miss 才回源 DB。」

**要點:**
- **先畫能跑的最簡版**,再按需加組件(緩存、MQ、CDN)。不要一開口就拆 20 個微服務。
- 每加一個組件,**說一句它解決什麼問題**(「這裡加 Redis 是因為 Step 2 算出讀寫 100:1」)。

**常見陷阱:**
- ❌ 過度設計:一上來就 microservices / k8s / service mesh,把簡單題搞複雜。
- ❌ 框圖沒數據流:畫了一堆框但說不清請求怎麼走。✅ 永遠**沿著一個請求講**。

---

## Step 6 · Deep Dive(挑 2–3 點往死裡鑽)

**目的:** 證明你不只會畫框,還懂細節。面試官會挑點追問,或讓你自己選。

**怎麼選要鑽的點:** 每道題都有「**最會被追問的點**」——通常是這題的 ●● 重心(見 `00` 依賴表)。提前準備:

| 題型 | 最該鑽的點 |
|---|---|
| 讀密集(feed/緩存) | 緩存一致性、熱點打散、寫擴散 vs 讀擴散 |
| 寫密集/海量 | 分片鍵與熱點、分庫分表後的跨片查詢 |
| 強一致(支付) | 冪等、對賬、分布式事務 vs 最終一致 |
| 長連接(IM) | 連接保持、消息有序、離線推送 |

**面試話術(主動引導):**
> 「這裡最值得深入的是大 V 的扇出問題,我來講講為什麼純寫擴散會爆,以及怎麼用『寫擴散 + 大 V 走讀擴散』的混合方案……」

**常見陷阱:** 被問到才被動回答,且只停在表面。✅ **主動挑一個點深入**,展示你知道這題的難點在哪。

---

## Step 7 · 瓶頸與取捨(分高下的地方)

**目的:** 這是**資深和架構師的加分核心**。面試官等的不是完美架構,是你**知不知道自己方案的代價**。

**做三件事:**
1. **找瓶頸 / 單點**:哪個組件先撐不住?掛了會怎樣?(單點 DB、熱點分片、同步調用鏈)
2. **給演進**:現在的設計撐到多大?下一步怎麼擴(加緩存層 / 再分片 / 多活)?
3. **主動暴露 tradeoff**:「我選了 X,換來 A 但犧牲了 B,在這個業務場景值得,因為……」

**面試話術(主動暴露=加分):**
> 「我這套用了寫擴散,寫放大很厲害——一個大 V 發帖要寫千萬個 feed。代價是寫成本和延遲;好處是讀極快。對 Twitter 這種**讀遠多於寫**的場景值得,但如果是寫密集場景我會反過來。另外單點是 feed 緩存集群,它掛了要能降級回源 DB。」

**常見陷阱:** 假裝方案完美。✅ **主動說缺點**比被面試官挑出來強十倍——前者顯得你 reason 得動,後者顯得你沒想到。

## 一個 90 秒微範例(把七步串起來)

> 題:**設計一個 Pastebin(貼文本,給短鏈,別人憑鏈接看)**

1. **澄清**:功能=存文本+生成鏈接+讀;非功能=讀多寫少、鏈接不可枚舉;規模=100 萬新 paste/天。
2. **估算**:寫 QPS ≈ 1e6/8.6e4 ≈ 12;讀按 10:1 ≈ 120 QPS——**很小,單庫能扛,重點是存儲增長**。文本均 10KB → 每天 10GB → 一年 ~3.6TB。
3. **API**:`POST /pastes {text,ttl}→id`、`GET /pastes/{id}→text`。
4. **數據模型**:`paste(id PK, text, created, ttl)`;id 用 base62 隨機 7 位(不可枚舉);文本大可放對象存儲、DB 只存 metadata。
5. **架構**:Client → Gateway → Paste Service → 寫 metadata 到 DB + 文本到對象存儲;讀走 CDN/緩存,miss 回源。
6. **Deep dive**:id 怎麼發(隨機 vs 發號器、碰撞重試);TTL 過期怎麼清(惰性 + 定時掃)。
7. **取捨**:文本放對象存儲(便宜、可擴)換來一次額外網絡跳;隨機 id 換來偶發碰撞需重試。讀多寫少,加 CDN 收益最大。

> 看到了嗎——**同一套七步**,套哪道題都成立。17 道題就是把這套腳本跑 17 遍,每遍在第 5/6 步填不同的零件。

---

## 通用話術模板(背下來臨場套)

- 開場:「我先澄清範圍和規模……」
- 估算後:「讀寫比是 X,**所以**我要重點優化……」
- 架構時:「我沿著一個請求走一遍……這裡加 Y 是為了解決 Z。」
- Deep dive:「這題最難的點是……我深入講一下。」
- 收尾:「我的方案 tradeoff 是……單點是……撐到 N 倍後我會……」

## 紅旗清單(踩中就扣分)

- 🚩 沒澄清就開畫　🚩 沒估算就拍架構　🚩 不提分片鍵　🚩 上來就拆微服務　🚩 框圖講不出數據流　🚩 假裝方案沒缺點

---

## 一句話總綱

> **七步是一套腳本——澄清把題收斂、估算把規模變數字、API/數據定邊界與存儲、高層架構沿請求一鏡到底、deep dive 鑽 2–3 個難點、最後主動暴露 tradeoff。背下脊柱,看到任何題都自動進第 1 步;17 道題只是把這套腳本跑 17 遍。**
