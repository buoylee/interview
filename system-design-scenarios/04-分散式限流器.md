# 04 · 分散式限流器 Rate Limiter

> T1 基本功第二題。它**短小但密度極高**——演算法選型、原子性、分散式一致性三個硬核點全擠在一道題裡,是面試官「看你懂不懂分散式並發」的試金石。
>
> 這題的能力包是 **A(框架)+ C●●(韌性與流量)**——見 `00` 依賴表。限流是 `system-design/01 韌性`「六件套」(超時/熔斷/重試/隔離/限流/降級)之一,但前五件防的是「**被慢下游拖死**」,限流防的是**另一個方向**:「被上游/洪峰打死」。這道整題,就是把那顆零件**裝成一個能扛百萬 QPS、跨多節點還算得準**的系統,並在白板上講清每個取捨。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個分散式限流器,保護後端服務不被流量洪峰打死。

別一上來就背「令牌桶」。先問清楚**限什麼、限多嚴、放哪層**——這三個問題的答案直接決定你選哪種演算法、用不用 Redis。

| 類 | 要問什麼 | 為什麼這問題改變設計 |
|---|---|---|
| **功能** | 限流**粒度**?(全局 / 按 user / 按 IP / 按 API / 按 user×API 組合) | 粒度決定**限流鍵的基數**——基數從幾千到幾億,是「本地內存」還是「中心 Redis」的分水嶺 |
| **功能** | 超限後怎麼處理?(拒絕 429 / 排隊等待 / 降級) | 「拒絕」用計數器類演算法即可;「排隊平滑」要漏桶;這決定演算法選型 |
| **非功能** | **精度要求**?能容忍短暫超額(突刺)嗎? | 能容忍 → 固定窗口最省;不能容忍任意 1 秒超額 → 滑動窗口;要限均速+容突發 → 令牌桶 |
| **非功能** | 限流器自己的**延遲預算**?(每請求能花幾 ms?) | 每請求一次 Redis 往返 ≈ +0.5~1ms,百萬 QPS 下這既是延遲也是 Redis 的負載 |
| **非功能** | 限流器掛了,**fail-open 還是 fail-closed**? | 限流器是旁路件——它掛了該「放行」(保業務)還是「全拒」(保後端)?業務性命攸關 |
| **規模** | 多少 QPS?多少獨立限流鍵?(user 數 × API 數) | 決定中心 Redis 扛不扛得住、要不要兩級 |

**面試話術(把假設說出來讓對方確認):**
> 「我先確認範圍——我假設這是**閘道前置**的限流,粒度按 `user × API` 組合,超限直接回 429 不排隊。精度上我先按『**允許合理突發、但限均速**』設計,所以主演算法選令牌桶。規模我假設要扛**百萬級 QPS**、約 **1000 萬獨立限流鍵**。限流器掛了我傾向 **fail-open**——寧可漏放也不能讓限流器本身變成新的單點把業務全打死,可以嗎?」

**常見陷阱:**
- ❌ 不問粒度就開畫——「全局限流」和「按 user 限流」是兩個完全不同的系統(後者鍵基數爆炸)。
- ❌ 不問 fail-open/closed——這是限流題**最會被追問**的失效模式,沒主動提會被當沒想過。
- ✅ 主動把「精度 vs 成本」這對 tradeoff 點出來——它貫穿整道題。

## 2. 容量估算(調用 `02` 的數字感)

限流題的估算**不在存儲**,在三件事:**Redis 往返成本、限流鍵基數、單機 Redis 夠不夠**。用 `02` 的背誦數:一天 ≈ 1e5 s、單 Redis ≈ 10 萬 QPS、同機房一次調用 ≈ 0.5~1ms。

**① 流量 → Redis 負載(這題的命門)**

假設閘道前置,峰值 **100 萬 QPS**。若**每請求都打一次中心 Redis**:
- Redis QPS = 100 萬(且每次是「讀+判斷+寫」的 Lua,不是純讀)。
- 對照單 Redis ≈ 10 萬 QPS → **需要 ~10+ 個分片**(還沒算 headroom)。
- **所以**:純中心 Redis 在百萬 QPS 下,光限流就要一個 Redis 集群,且每請求 +0.5~1ms 延遲打在**所有**業務請求的關鍵路徑上。這直接逼出第三個 deep dive 的「兩級方案」。

**② 限流鍵基數 → 存不存得下**

| 粒度 | 鍵基數估算 | 落地選擇 |
|---|---|---|
| 全局(整個 API 一個閥) | O(1)~O(API 數) | 本地都嫌重,一個計數器搞定 |
| 按 IP | 活躍 IP ~百萬級 | 中心 Redis,鍵不算多 |
| 按 user | DAU 級,千萬~億 | 中心 Redis;熱 user 可本地兜 |
| **按 user × API** | 千萬 user × 數十 API = **數億組合**(但同時活躍的遠少) | 中心存活躍鍵 + TTL 自動淘汰;基數靠 TTL 收斂 |

> **關鍵 insight:** 限流鍵雖然**理論基數**上億,但每個鍵帶 TTL(窗口一過就被 Redis 自動清),所以**同時駐留**的鍵 ≈ 「一個窗口內活躍的鍵數」,遠小於理論值。單個令牌桶狀態 ~50–100 B(`tokens` + `last` 兩個欄位),1000 萬活躍鍵 × 100B ≈ **1GB 內存**——單台 Redis 放得下,瓶頸是 QPS 不是內存。

**③ 延遲預算**

每請求一次 Redis 往返 ≈ 0.5~1ms。若後端目標 P99 < 50ms,這 1ms 可接受;但若是 P99 < 5ms 的高頻內部調用,**限流的 1ms 就是 20% 預算**——這時候本地限流(納秒級)比中心 Redis 更合適。**所以**:延遲敏感場景 → 本地;精度敏感場景 → 中心。

**面試話術:**
> 「100 萬 QPS 每請求一次 Redis Lua,單 Redis 才 10 萬 QPS,我需要十幾個分片,而且 1ms 延遲打在每個請求上。**所以純中心 Redis 在這個量級是奢侈的**——我會在 deep dive ③ 講『本地預扣配額 + 中心對賬』的兩級方案,把絕大多數判斷收斂在本地內存(納秒級),只在配額用完時才碰 Redis。」

## 3. API + 數據模型

### 對外接口

限流器通常**不是一個獨立 service**,而是**閘道/SDK 裡的一個攔截器**。對外的「API」其實是一個判定函數:

```
allow(key, cost=1) → {allowed: bool, remaining: int, retryAfter: ms}
```

- `key` = 限流鍵(如 `u:123:GET:/feed`)。
- `cost` = 本次消耗幾個令牌(批量/重接口可 >1)。
- 返回 `retryAfter` → 寫進 HTTP `429` 的 `Retry-After` header,告訴客戶端**多久後再來**(這是體面降級的關鍵,別讓客戶端盲目重試形成風暴)。

### 數據模型(中心存儲)

不同演算法的 Redis 結構**完全不同**——這正是 deep dive ① 的核心:

| 演算法 | Redis 結構 | 鍵內容 |
|---|---|---|
| 固定窗口 | `String`(計數器) | `rl:u123:<當前秒>` → count;帶 EXPIRE |
| 滑動日誌 / 滑動窗口 | `ZSet`(時間戳日誌) | member=請求唯一 id,score=時間戳ms |
| 令牌桶 | `Hash` 或兩個欄位 | `{tokens: 當前令牌數, last: 上次補充時間}` |
| 漏桶 | `Hash` | `{water: 當前水量, last: 上次漏水時間}` |

**分片鍵:** 限流鍵本身就是分片鍵——`u:123:...` 按 `key` hash 到某個 Redis 分片。**好處**:同一個 user 的所有限流操作落同一片,天然原子(同片才能跑 Lua);**代價**:熱 user(大 V、被攻擊的 IP)會讓某片變熱點 → 留到架構師層講「熱點打散」。

## 4. 高層架構(一鏡到底)

```
                 ┌─────────────────────── 控制面 ───────────────────────┐
                 │  規則配置(每 key 的 limit/rate)→ 推送到各閘道節點   │
                 └──────────────────────────┬──────────────────────────┘
                                            │ (熱更新,別重啟)
   Client ──► LB ──► 閘道節點(限流攔截器)───┼─────────────────────┐
                          │                                        │
                  ① 算限流鍵 key                                    │
                          │                                        ▼
                  ② 本地令牌桶(L1)──夠?──► 放行 ──► 後端 Service
                          │ 不夠/配額耗盡                           ▲
                          ▼                                        │
                  ③ 中心 Redis(L2,Lua 原子)──夠?──► 放行 ───────┘
                          │ 還是不夠
                          ▼
                  ④ 拒絕:429 + Retry-After
```

**沿一個請求走一遍(白板版):**
1. 請求進閘道,攔截器**算出限流鍵** `key`(如 `u:123:GET:/feed`)。
2. 先查**本地令牌桶 L1**(這台閘道分到的本地配額)——命中且有令牌,**直接放行,不碰網絡**(納秒級)。這是百萬 QPS 下的主路徑。
3. L1 配額耗盡 → 打**中心 Redis L2**,跑一段 Lua 原子判斷全局是否還有額度,有就**領一批配額**回本地。
4. 全局也超了 → 返回 `429` + `Retry-After`,客戶端按提示退避。
5. **控制面**把每個 key 的規則(limit/rate)熱推到所有閘道節點——改限流閥值不能重啟服務。

> **先畫最簡版再加:** 白板上先只畫「Client → 閘道 → 中心 Redis(Lua)→ 後端」這條能跑的最簡單版,講清原子性;**再**說「百萬 QPS 下中心 Redis 扛不住,所以加 L1 本地層」——分兩步畫,讓面試官看到你的演進思路,而不是一上來就堆兩級。

**常見陷阱:** 把限流器畫成一個獨立微服務,讓每個請求多一跳 RPC 去問它——那等於**自己給每個請求加了一次網絡往返**,還製造新單點。限流邏輯應**內嵌在閘道/SDK**,狀態才放外部 Redis。

## 5. Deep Dive

挑這三個——它們是限流題**100% 會被追問**的點。

### Deep Dive ①:五種演算法,差在「邊界」和「突發語義」

面試官最愛的開場:「你說用令牌桶,那固定窗口為什麼不行?」答這題要能講清**每種演算法的黑盒裡到底在數什麼**。

| 演算法 | 黑盒裡在做什麼 | 邊界突刺 | 突發語義 | 內存 | 何時用 |
|---|---|---|---|---|---|
| **固定窗口** | 按整秒分桶,桶內 `INCR`,超 N 拒 | **有,2 倍** | 不控 | O(1) | 極簡、能忍邊界 2 倍 |
| **滑動日誌** | 存每個請求時間戳,每次數「最近 1 秒內」有幾條 | **無** | 不控 | O(QPS×窗口) | 要精確、量不大 |
| **滑動窗口計數器** | 把窗口切成小格,按比例加權前後兩格 | 近似消除 | 不控 | O(格數) | 滑動日誌的省內存近似 |
| **令牌桶** | 按速率**匀速加令牌**,請求取一個;桶有存量 | 無 | **允許突發**(攢的令牌能一次用) | O(1) | 最常用——限均速+容突發 |
| **漏桶** | 請求進桶,桶底**恆速漏出**;桶滿則拒 | 無 | **強制平滑**(出口恆速) | O(1) | 要嚴格整流、保護恆速下游 |

**(a) 固定窗口的「邊界雙倍突刺」——這是面試必考的坑**

固定窗口限的是「**每個整秒桶 ≤ N**」,**不是**「任意滑動 1 秒 ≤ N」。問題就出在這:

```
limit = 5/秒
  桶[00:00.0~00:00.999]:                          ●●●●●  ← 後 5 個擠在桶尾(0.5~1.0s)
  桶[00:01.0~00:01.999]:  ●●●●●                          ← 前 5 個擠在桶頭(1.0~1.5s)
                              └──── 真實的 1 秒窗 [0.5~1.5] 內 = 10 個!────┘
```

桶 N 末尾放 5 個 + 桶 N+1 開頭放 5 個,這 10 個落在「跨桶的真實 1 秒」內 → **瞬時通過 2 倍限額**。`redis-handson` 的 sc01 **實測**:limit=5/秒,0.45 秒內通過 10 個(2 倍突刺)。

**(b) 滑動窗口怎麼解**

不分死桶,改盯「**從現在往回數 1 秒**」這個一直在滑的區間:用 ZSet 存每個請求的時間戳,每次來先 `zremrangebyscore` 刪掉窗口外的,再 `zcard` 數窗口內有幾個。因為窗口跟著「現在」滑,**永遠不會出現跨桶拼接** → 突刺消失。sc02 實測:同樣的 burst 被堵住,任意 1 秒嚴格 ≤5。代價:每個請求存一條 ZSet 記錄,**內存隨 QPS×窗口增長**——高 QPS 下用「滑動窗口計數器」(細分小格加權)折中。

**(c) 令牌桶 vs 漏桶——語義差異是這題的高頻區分題**

兩者都限速,但**對突發的態度相反**:

```
令牌桶:閒時令牌攢著(桶裡有 100 個)→ 流量來了能「一次用一批」→ 允許突發
        ┌──匀速加令牌──┐
        │ ▓▓▓▓▓▓▓▓ 桶 │ ← 攢滿 100,突發來了瞬間放 100,之後回到匀速
        └──────────────┘

漏桶:請求進桶,出口「恆速漏」→ 不管進得多猛,出去永遠是恆速 → 強制平滑
        進:▓▓▓▓▓▓ ┐
                  ▼
        ┌─────────┐
        │  桶     │
        └────┬────┘ 出:▓ ▓ ▓ ▓  ← 恆速,削平所有突發
```

> **一句話差異(背起來):** 漏桶**強制恆速**(平滑優先),令牌桶**允許突發**(靈活優先)。
>
> **業務上為什麼令牌桶更常用?** 真實流量天然有合理突發(用戶連點、批量),漏桶一刀切平滑會把正常突發也誤殺;令牌桶「閒時攢、忙時放一批」更貼合真實需求。**漏桶只在「下游必須恆速」**(如保護一個吞吐固定的老系統、做匀速排隊)時才更優——Sentinel 的「匀速排隊」、Nginx 的 `limit_req` 用的就是漏桶。

**面試話術(主動引導 deep dive):**
> 「我主推令牌桶。固定窗口有個經典坑——它限的是整秒桶而不是滑動 1 秒,跨桶邊界能放過 2 倍流量;滑動窗口能解但要存每個時間戳費內存。令牌桶 O(1) 狀態、限均速又容突發,大多數場景最合適。除非下游是個吞吐恆定的老系統,我才換漏桶強制整流。」

### Deep Dive ②:分散式原子實現——為什麼**必須** Redis + Lua

這題的真正硬核點。面試官會問:「你這計數器,並發下準嗎?」

**(a) 為什麼非原子會放過超額**

三種演算法的核心都是同一個三段式:**讀狀態 → 判斷 → 改狀態**。如果這三步不原子,並發下就會出事:

```
limit=1,兩個請求同時來:
  請求A: GET count → 0     │  請求B: GET count → 0      ← 都讀到 0
  請求A: 0<1,放行          │  請求B: 0<1,放行           ← 都判斷可放
  請求A: INCR → 1          │  請求B: INCR → 2            ← 兩個都過了!超額
```

這就是經典的 **check-then-act 競態**(和秒殺超賣同一個根因)。`GET` 和 `INCR` 之間有窗口,兩個並發請求都讀到舊值 → 都放行。**所以讀-判-改必須是一個不可分割的原子操作。**

**(b) Redis Lua 為什麼能保證原子**

Redis 是**單線程執行命令**的,而一段 `EVAL` 的 Lua 腳本在 Redis 裡**作為一個整體執行,中間不會插入任何其他命令**——天然把「讀-判-改」焊成一個原子塊。這是 Lua 在限流裡不可替代的原因(`redis-handson/08 transactions-scripting` 講透了 Lua 的原子模型)。

> ⚠️ 注意:`MULTI/EXEC` 事務**不能**替代——它只保證命令**連續執行不被打斷**,但**不能在事務中間根據讀到的值做分支判斷**(事務裡讀不到中間結果)。而限流的本質就是「**讀到 count 再決定放不放**」這個分支——只有 Lua 能在腳本裡 `if` 判斷後再寫。

**(c) 令牌桶的 Lua(讀狀態→補令牌→判斷→扣減,一次原子)**

```lua
-- KEYS[1]=桶key  ARGV: rate(每秒補), capacity(桶容量), now(ms), cost(本次消耗)
local b = redis.call('HMGET', KEYS[1], 'tokens', 'last')
local tokens = tonumber(b[1]) or capacity      -- 首次=滿桶
local last   = tonumber(b[2]) or now
-- 1) 按流逝時間補令牌(不超 capacity)
local filled = math.min(capacity, tokens + (now-last)/1000*rate)
-- 2) 判斷 + 扣減
local allowed = filled >= cost
local newTokens = allowed and filled-cost or filled
redis.call('HMSET', KEYS[1], 'tokens', newTokens, 'last', now)
redis.call('EXPIRE', KEYS[1], math.ceil(capacity/rate)+1)  -- 閒置自動淘汰,收斂鍵基數
return allowed and 1 or 0
```

關鍵:**令牌不是用定時任務真的「加」進去的**——那樣要為上億個桶跑定時器,不可能。而是**惰性計算**:每次請求來,用 `(now-last)×rate` 算出「這段時間本該補多少」,一次補上。`EXPIRE` 讓閒置的桶自動消失,正是第 2 段「鍵基數靠 TTL 收斂」的落地。

**(d) 滑動窗口的 Lua(ZSet)**

```lua
-- 刪窗口外 → 數窗口內 → 沒超就加當前請求,全程原子
redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, now-window)
if redis.call('ZCARD', KEYS[1]) < limit then
  redis.call('ZADD', KEYS[1], now, now..':'..math.random())  -- 唯一 member
  redis.call('EXPIRE', KEYS[1], window/1000 + 1)
  return 1
else return 0 end
```

**面試話術:**
> 「限流的核心是『讀計數 → 判斷 → 改計數』,這是個 check-then-act,並發下非原子會放過超額,跟秒殺超賣一個根因。我用 Redis Lua——Redis 單線程跑命令,一段 Lua 整體執行中間不插別的命令,把三步焊成原子。注意不能用 MULTI/EXEC,因為它不能在事務中間根據讀到的值做 if 分支,而限流恰恰要『讀到 count 再決定』。」

> **別手撸的工程提醒:** 生產上 Java 直接用 Redisson 的 `RRateLimiter`、Go 用 `redis_rate`,演算法和 Lua 都封好了;`distribution/限流算法` 裡 Guava `RateLimiter`(`SmoothBursty` 容突發 / `SmoothWarmingUp` 預熱)是**單機**令牌桶的標準件。手寫 Lua 只在面試白板和定制需求時。

---

### Deep Dive ③:多節點同步——純中心 Redis 的單點與延遲 → 兩級方案

第 2 段算出:百萬 QPS 每請求一次中心 Redis,要十幾個分片 + 每請求 1ms。這引出限流題**最有深度**的追問:「怎麼既全局準、又不讓中心 Redis 變瓶頸?」

**(a) 純中心 Redis 的三個問題**

| 問題 | 細節 |
|---|---|
| **延遲** | 每請求 +0.5~1ms 打在**所有**業務的關鍵路徑上 |
| **吞吐** | 百萬 QPS → Redis 集群,光限流就是一筆不小的基建 |
| **單點 / 失效** | Redis 抖一下、網絡分區 → 限流判斷全卡。**這時 fail-open 還是 closed?**(必須事先想好) |

**(b) 兩級方案:本地預扣配額(L1)+ 中心對賬(L2)**

核心思想:**別每個請求都問中心**,而是**本地一次性領一批配額,在本地內存裡扣**,扣完了再去中心領。把 99% 的判斷收斂在納秒級的本地內存。

```
全局 limit = 1000/秒,3 個閘道節點:

  控制面/中心 Redis(全局 1000 額度池)
        │ 各節點「批量領配額」(如一次領 100)
        ├──► 節點A: 本地桶 100 ── 本地扣到 0 ──► 再去領 100
        ├──► 節點B: 本地桶 100 ── 本地扣 ──► ...
        └──► 節點C: 本地桶 100 ── 本地扣 ──► ...

  絕大多數請求只碰「本地桶」(納秒)→ 只有「本地配額耗盡」才打中心 Redis(1ms)
```

**配額怎麼分?** 兩種策略,各有取捨:

| 策略 | 做法 | 取捨 |
|---|---|---|
| **靜態均分** | 全局 1000 ÷ 3 節點 = 各 333 | 簡單;但流量傾斜時(LB 不均、某節點熱)會**誤殺**——這節點 333 用完被拒,別的節點還有餘 |
| **動態批量領取** | 節點按需向中心「領一批」,用快了多領、用慢了少領 | 更準、自適應流量傾斜;代價是中心要維護全局池 + 領取邏輯複雜 |

**(c) 兩級方案的代價(主動暴露)**

| 收益 | 代價 |
|---|---|
| 99% 請求走本地(納秒),中心 QPS 降 1~2 個數量級 | **精度下降**:本地預扣的配額若沒用完就「漏」了(或節點掛了配額丟失)→ 全局實際放行可能略低於 limit(保守誤殺)或在配額回收延遲窗口內略超 |
| 中心 Redis 不再是每請求單點 | **複雜度**:多了配額分配、回收、節點增減時的再平衡 |

> **這正是「精度 vs 成本」這對貫穿全題的 tradeoff 的終點:** 要 100% 精確全局限流 → 純中心 Redis(貴、慢、單點);能容忍**近似**全局限流 → 兩級(快、便宜、但略不準)。**百萬 QPS 場景幾乎一定選後者**——限流本來就是個「兜底保護」,差幾個請求無傷大雅,延遲和成本才是命門。

**面試話術(把近似性說成優點):**
> 「純中心 Redis 全局最準,但百萬 QPS 下它又是瓶頸又是單點。我會上兩級:本地預扣一批配額在內存裡扣,扣完才碰中心,把 99% 判斷收斂到納秒級。代價是全局計數變成**近似**——但限流是兜底保護,放過或誤殺幾個請求遠比給每個請求加 1ms、給系統加個單點划算。配額我用動態批量領取應對流量傾斜,而不是靜態均分。」

## 6. 架構師深挖層(推過面試線)

IC 線到第 5 段就夠拿 offer。這一段是把「面試答法」升級成「**真實決策**」——成本誰買單、容量怎麼規劃、怎麼演進、會怎麼掛、何時**根本不該**自建。

### 成本:限流器自己的開銷誰買單

- **延遲稅**:中心 Redis 方案給**每個**業務請求加 0.5~1ms。在百萬 QPS、P99 預算緊的系統,這是實打實的用戶體驗成本——兩級方案本質是「用一點精度,買回這 1ms」。
- **基建成本**:純中心方案的 Redis 集群(十幾個分片)是限流**專屬**的硬成本。兩級把它降一兩個數量級。
- **隱性成本——誤殺**:限太緊或配額分配不當 → 誤殺正常用戶 = 直接的業務損失(轉化率、留存)。限流閥值是**業務決策不是技術決策**,要可灰度、可回滾、可觀測(配合 `observability/` 的限流拒絕率告警)。

### 容量規劃:閥值怎麼定、怎麼隨容量漲

- 限流閥值應**錨定後端真實容量**(壓測出來的安全 QPS),而不是拍腦袋。後端擴容了,限流閥值要跟著漲——否則限流變成「人為的容量天花板」。
- **預留 headroom**(`02` 的數字感):別把閥值設成後端極限,設成 ~70% 安全線,留突發緩衝。
- 熱點鍵容量:某個 user/IP 被 ●● 打(攻擊、大 V),其限流鍵會讓對應 Redis 分片變熱點 → 對熱鍵做**本地降級**(這台閘道直接本地限,不再每次問中心)或**鍵打散**。

### 演進路徑(從小到大,別一步到位)

```
單機本地令牌桶(Guava/x-time-rate)   ← 單實例 / 全局不準也能忍
        │ 多實例後全局不準(10 個實例各限 100 = 實際 1000)
        ▼
中心 Redis + Lua(全局一致)           ← 中等規模,精度優先
        │ QPS 上百萬,中心成瓶頸 + 單點
        ▼
本地預扣 + 中心對賬(兩級,近似全局)  ← 大規模,延遲/成本優先
        │ 跨地域多活
        ▼
分區域獨立限流 / 區域配額          ← 跨洲 RTT 100ms,不可能全球一個中心
```

> **架構師的 insight:** 限流的演進主軸是「**全局精度** 和 **延遲/成本** 的天平」。規模越大,天平越往「犧牲精度換性能」傾——這不是妥協,是對「限流本就是兜底件」的正確理解。

### 失效模式(面試官最愛追的)

| 失效 | 後果 | 對策 |
|---|---|---|
| **中心 Redis 掛 / 網絡分區** | 限流判斷卡住 | **事先定 fail-open(放行,保業務)還是 fail-closed(全拒,保後端)**——多數場景 fail-open + 本地兜底限流(降級到本地寬鬆閥值),別讓限流器成為新的單點 |
| **時鐘漂移** | 滑動窗口/令牌桶補令牌靠 `now`,各節點時鐘不齊 → 算錯 | 中心方案用 **Redis 的 `TIME`** 作統一時鐘(別用各閘道本地時鐘);兩級方案容忍小漂移 |
| **熱點鍵** | 某 user/IP 把一個 Redis 分片打爆 | 熱鍵本地降級 / 鍵打散 / 對攻擊 IP 走更上游(LB、WAF)直接擋 |
| **配額丟失**(兩級) | 節點掛,它預扣未用的配額丟了 | 配額帶短 TTL,中心定期回收;接受短暫的保守誤殺 |
| **限流自己成為單點** | 把限流做成獨立 RPC service,它掛了全鏈斷 | 限流邏輯**內嵌閘道/SDK**,狀態才外置;限流件必須比被保護的後端更可靠 |

### 何時**不該**自建分散式限流(推過面試線的關鍵)

- **流量沒到百萬級**:單機本地令牌桶(Guava / `x/time/rate`)就夠,多實例不準也常能忍——**別為了「分散式」上 Redis**,徒增延遲和單點。
- **粗粒度全局擋洪峰**:雲廠商的 API Gateway、Nginx `limit_req`、WAF 自帶限流足矣,自建是重複造輪子。
- **粒度極細又要強精度又要超低延遲**:這三個要求**互相打架**——強精度要中心(慢)、超低延遲要本地(不準)。這時該回頭跟業務方談「哪個要求能放鬆」,而不是硬上一個又貴又複雜的系統。
- **真正該自建的場景**:百萬 QPS+、按 user/API 細粒度、且能接受近似全局——這才是兩級方案發光的地方。

> **架構師話術:** 「在動手設計前我會先問——**真的需要分散式限流嗎?** 沒到百萬 QPS,單機令牌桶 + 雲閘道粗限流就夠,自建 Redis 限流是給自己加延遲和單點。只有當細粒度 + 大規模 + 能容忍近似這三條同時成立,兩級方案才值這個複雜度。」

## 8. 面試卡(抽認自檢)

**Q1 · 固定窗口為什麼有突刺?滑動窗口怎麼解?**
> 固定窗口限的是「每個整秒桶 ≤N」不是「任意滑動 1 秒 ≤N」。桶 N 尾放 N 個 + 桶 N+1 頭放 N 個,落在跨桶的真實 1 秒內 = 2N(實測 0.45s 過 10 個)。滑動窗口用 ZSet 存時間戳,每次數「從現在往回 1 秒」內有幾個,窗口跟著現在滑,不會跨桶拼接 → 無突刺,代價是存每個時間戳費內存。

**Q2 · 令牌桶 vs 漏桶,差在哪?為什麼業務上令牌桶更常用?**
> 漏桶**強制恆速**(出口恆速漏,削平所有突發,平滑優先);令牌桶**允許突發**(閒時攢令牌,忙時一次放一批,靈活優先)。真實流量有合理突發,漏桶會誤殺,所以令牌桶更常用;漏桶只在「下游必須恆速」時(保護吞吐固定的老系統、匀速排隊)才更優。

**Q3 · 為什麼限流必須用 Redis Lua?MULTI/EXEC 行不行?**
> 限流核心是「讀計數→判斷→改計數」的 check-then-act,非原子並發下兩請求都讀到舊值都放行 → 超額(同秒殺超賣根因)。Redis 單線程,一段 Lua 整體執行中間不插別的命令,把三步焊成原子。MULTI/EXEC **不行**——它不能在事務中間根據讀到的值做 if 分支,而限流恰恰要「讀到 count 再決定放不放」。

**Q4 · 純中心 Redis 限流有什麼問題?兩級方案怎麼做?**
> 純中心:每請求 +1ms 延遲打在所有業務上、百萬 QPS 要十幾個 Redis 分片、且是單點。兩級:本地一次性領一批配額在內存裡扣(納秒),扣完才碰中心,把 99% 判斷收斂本地。代價是全局計數變**近似**(預扣未用會漏/丟)。限流是兜底件,近似可接受,延遲和成本才是命門。

**Q5 · 限流鍵基數上億,Redis 存得下嗎?**
> 理論基數(user×API)上億,但每個鍵帶 TTL,窗口一過 Redis 自動清,**同時駐留**的只有「一個窗口內活躍的鍵」。單個令牌桶狀態 ~100B,1000 萬活躍鍵 ≈ 1GB,單機放得下——**瓶頸是 QPS 不是內存**。

**Q6 · 限流器自己掛了怎麼辦?**
> 事先定 **fail-open(放行保業務)還是 fail-closed(全拒保後端)**,多數場景 fail-open + 本地兜底寬鬆限流。限流邏輯**內嵌閘道/SDK**(別做成獨立 RPC service 製造新單點),狀態才外置 Redis。限流件必須比它保護的後端更可靠——否則它就成了你親手裝的單點。

## 一句話總綱

> **分散式限流器 = 選令牌桶(限均速 + 容突發,優於有突刺的固定窗口和費內存的滑動日誌)→ 用 Redis Lua 把「讀-判-改」焊成原子(MULTI/EXEC 不夠,要在腳本裡 if 分支)→ 百萬 QPS 下純中心 Redis 太貴太單點,改「本地預扣配額 + 中心對賬」兩級,用一點全局精度換回延遲和成本。貫穿全題的天平是『精度 vs 成本』,規模越大越往成本側倒——因為限流本就是兜底件。**
