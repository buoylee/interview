# 19 · 協同編輯(Google Docs)

> **T3 收官題,也是整個 track 最反直覺的一題。** 前面 16 題你練的肌肉都是「規模怎麼扛」——QPS、存儲、扇出、削峰。協同編輯把這套肌肉**整個關掉**:它的規模小得可笑(一篇文檔同時編輯的人很少超過 50),真正的難點是一個你前面從沒正面碰過的問題——**多個人同時改同一段文字,怎麼讓所有人最後看到一模一樣的結果。**
>
> 用 `00` 依賴表看:協同編輯吃 **D(一致性/複製)●● + A + B + E**。它是 **D 包的壓軸**——支付/秒殺講的是「錢不能算錯」的強一致,協同編輯講的是另一條路:**最終一致 + 因果順序 + 並發收斂**。這正是 `system-design/02 一致性模型與時鐘` 那套邏輯時鐘/因果關係的**最具體落地**。
>
> 一句話定調:**這題難在「並發正確收斂」,不在「規模」。** 你要是上來就算 QPS、堆分片,就答錯題了——面試官等的是你能不能講清 **OT vs CRDT** 這對冤家,以及為什麼「兩個人同時在第 5 個字後面插字」這件小事,能難倒一整個領域 20 年。

---

## 1. 題目 + 澄清問題清單

**題目:** 設計一個多人實時協同文檔編輯器(像 Google Docs / Notion / 騰訊文檔)。多個使用者**同時**編輯同一篇文檔,每個人的每一次按鍵都要**近實時**同步到所有其他人的螢幕,且不管網絡延遲、不管誰先誰後,**所有人最終必須收斂到完全相同的內容**。

**這題的澄清比別題更關鍵**——因為「協同編輯」這四個字底下藏著好幾個難度天差地別的產品:

| 類 | 該問什麼 | 這題的典型答案(也是你的假設) |
|---|---|---|
| **功能** | 是純文字還是富文本(格式/圖片/表格)?要不要光標/選區共享(presence)?要不要版本歷史?要不要評論? | **富文本**(段落/粗體/列表,先不做表格嵌套);**presence 要**(看到別人光標在哪是協同的核心體驗);**版本歷史要**(能回滾);評論先不做 |
| **非功能(SLA)** | 同步延遲多低?要不要支援**離線編輯**?衝突怎麼算「對」?可用性? | 打字同步要**近實時**(<200ms 體感無延遲);**要支援離線**(斷網繼續打,重連自動合併);**最終一致**(允許短暫不一致,但必須收斂);高可用 > 強一致 |
| **規模 / 約束** | 單篇文檔同時幾人編?文檔總量?一個人打字多快? | **單篇並發編輯通常 <50 人**(Google Docs 硬上限 ~100);文檔總量可上億篇(但這是**冷存儲問題**,不是難點);打字 ~5–8 字/秒/人 |

**面試話術(把題收斂 + 主動定調難點):**
> 「我先框範圍——核心是富文本的多人實時編輯,每次按鍵近實時同步,要支援離線編輯後重連合併,要有 presence 和版本歷史。**我想先強調一點:這題的難點不在規模**——單篇文檔同時編輯的人很少超過幾十個,QPS 算下來很小。**真正難的是並發衝突的正確收斂**:兩個人同時在同一個位置插字,系統怎麼保證所有端最後看到一樣的內容。所以我會把大部分時間花在 **OT 和 CRDT 這兩種衝突解決模型的取捨**上,規模那塊我快速帶過。可以嗎?」

**這段話術本身就是加分項**——它展示了你**認得出這題的題眼**(`01` Step 1 的精髓:把開放題收斂到正確的難點上)。多數人會習慣性地開始算 QPS、畫分片,在協同編輯這題上那是**走錯方向**。

**這幾個澄清直接決定後面:**

- 「要離線編輯」→ deep dive 直接把 **CRDT** 推到前台(OT 依賴中心服務器排序,離線場景天生吃力)。
- 「最終一致 + 必須收斂」→ 整題的核心契約變成 `system-design/02` 的**因果一致 + 並發操作收斂**,不是強一致。
- 「presence 要」→ 復用 `09 IM` 的長連接 + 在線狀態能力包,**不重造輪子**。
- 「單篇 <50 人」→ 第 2 步估算的結論會是「**規模根本不是約束**」,把注意力逼回正確性。

**常見陷阱:** 把它當成「Dropbox 文件同步」(`12` 那題)來答——上來就講分塊、內容哈希、斷點續傳。❌ 那是**文件級**同步(整檔覆蓋,衝突就是「保留兩份」);協同編輯是**字符級**並發,衝突必須**自動合併到同一篇**,不能讓使用者選「保留我的還是他的」。這兩題的衝突模型是**兩個世界**,別混。

## 2. 容量估算(這題的估算是用來「證明規模不是問題」的)

> 別題的估算是為了推出「要分庫/要緩存/要 CDN」;這題的估算**反過來**——是為了**證明規模小到可以忽略**,好讓你理直氣壯地把全部精力投到正確性上。`01` Step 2 鐵律「估完接一句所以」,在這題每句「所以」都指向同一個結論:**瓶頸不在這。**

用 `02` 小抄:一天 ≈ 1e5 s、單條操作 ~50–100 B、單機長連接 ~幾萬–10 萬。

### ① 操作(按鍵)QPS

協同編輯的「寫」單位不是「一篇文檔」,是**一次編輯操作(一次按鍵 / 一次刪除)**。算單篇熱門文檔的峰值:

```
單篇同時 50 人編輯,每人 ~5 次按鍵/秒(打字算快的)
= 50 × 5 = 250 ops/s（單篇峰值）
```
> **所以:單篇文檔的寫操作峰值才 ~250 ops/s——一台機器一根手指就扛住。** 真正的放大在「廣播」:這 250 ops 每一條都要**扇出給文檔內所有 50 個連接**,所以下行是 `250 × 50 = 12,500 msg/s`(單篇)。這是**扇出題**(`00` 能力包 E),但扇出範圍極小(同篇 ≤50 人),和 News Feed 的千萬粉扇出完全不是一個量級。

### ② 全站操作 QPS

```
假設 10 萬篇文檔「同時」處於活躍協同（很高估了）
平均每篇 ~5 ops/s（多數文檔沒 50 人在打）
= 10 萬 × 5 = 50 萬 ops/s（全站）
```
> **所以:全站 50 萬 ops/s——數量級和一個中型寫密集系統相當,但每條操作極小(~50 B)、處理極輕(就是把 op 應用到內存裡的文檔模型再廣播)。按**文檔分片**(`docId` 做 sharding key),一篇文檔的所有操作落在同一個節點上**串行處理**,天然均勻、無跨片。50 萬 ÷ 單機幾萬 ≈ 幾十台協同節點,不誇張。

### ③ 連接數(這才是基礎設施的真約束)

協同是長連接題(復用 `09 IM`):每個正在看/編輯文檔的人佔一條 WebSocket。

```
活躍協同用戶 ~500 萬同時在線（樂觀估）
單機長連接 ~5 萬
= 500 萬 ÷ 5 萬 = ~100 台接入機
```
> **所以:和 IM 一樣,連接數而非 QPS 決定接入層機器數**——但這套長連接管理(心跳、斷線重連、連接路由)`09 IM` 已經講透,**這題直接錨過去復用,不重講**。

### ④ 存儲(op 日誌 + 快照)

```
單篇文檔:基礎內容 ~10–100 KB
op 日誌:一篇活躍文檔一天可能幾萬次編輯 × ~50 B = 幾 MB/篇/天（會膨脹）
全站上億篇文檔的歷史 → TB~PB 級，但 99% 是冷數據
```
> **所以:存儲的矛盾不是「大」,是「op 日誌會無限增長」**——這逼出「**定期快照 + 截斷舊 op 日誌**」(像數據庫的 checkpoint),deep dive ③ 會展開。文檔本體 + 冷歷史是普通的冷存儲問題,不是難點。

### 一句話估算結論(面試一定要說出來)

> 「**估算的結論很反常——這題沒有規模瓶頸。** 單篇 250 ops/s、全站 50 萬 ops/s、扇出範圍只有同篇 ≤50 人,按 docId 分片天然均勻。唯一像樣的基礎設施約束是長連接數(~100 台接入機,和 IM 同款,直接復用)。**所以我不會在規模上多花時間——這題 90% 的設計難度集中在『50 人同時改一段話,怎麼正確收斂到同一結果』,我把時間都投在那。**」

這句「**估算證明規模不是瓶頸,把難度逼回正確性**」就是這題資深 vs 初級的分水嶺——初級會繼續算分片、算緩存,資深知道**該停在哪、該往哪鑽**。

## 3. API + 數據模型

### API(`01` Step 3:協同編輯的 API 是「雙向流」,不是請求-響應)

協同編輯的對外接口和 REST 很不一樣——核心是一條**雙向 WebSocket**,在上面跑增量操作協議:

```
WS  /docs/{docId}/connect          # 建立協同會話,雙向長連接

  ── 上行(client → server)──
  { type: "op",       op: <一次編輯操作>, baseVersion: 42 }   # 我基於版本42做了這個改動
  { type: "cursor",   pos: <光標位置/選區> }                  # presence:我光標移到這
  { type: "sync_req", sinceVersion: 100 }                     # 離線重連:給我100之後的所有op

  ── 下行(server → client)──
  { type: "op",       op: <已變換/已排序的操作>, version: 43 } # 廣播給所有人
  { type: "cursor",   userId: "u9", pos: ... }                # 別人的光標
  { type: "ack",      version: 43 }                           # 你的op已被接受,服務器版本到43
  { type: "snapshot", doc: <完整文檔>, version: 100 }         # 首次連入/差太遠時給全量快照
```

```
# 輔助 REST(非實時路徑)
GET  /docs/{docId}            → 文檔當前快照 + version(首次載入)
GET  /docs/{docId}/history    → 版本歷史列表
POST /docs/{docId}/revert     {toVersion}  → 回滾到某版本
```

**要點(把協同 API 的特殊性說出來):**
- 核心不是「POST 一個改動」,是**在長連接上持續交換增量 op**——每個 op 帶 `baseVersion`(我是基於哪個版本改的),服務器靠它判斷有沒有並發衝突需要處理。
- **首次載入給 snapshot,之後只傳 op**——傳全文太貴,傳「在第 5 個字後插入 'a'」這種增量才實時得起來。
- 光標(presence)走**同一條連接但獨立通道**,且**不需要持久化、不需要強一致**(光標飄一下沒人死),可以盡力而為地廣播。

### 數據模型(`01` Step 4:選型 + 分片鍵)

協同編輯的數據模型有**三層**,各管一件事:

| 層 | 存什麼 | 選型 | 為什麼 |
|---|---|---|---|
| **文檔快照** | 某個版本的完整文檔內容 | 對象存儲 / 文檔型 DB | 整檔讀寫,訪問模式簡單;快照是「checkpoint」,不頻繁 |
| **操作日誌(op log)** | 從某快照之後的**每一條增量操作**(有序、append-only) | 順序寫存儲(類 Kafka / WAL 表) | append-only + 按 version 順序讀,正是日誌型存儲的完美場景;是「重放出當前狀態」的真相源 |
| **文檔元數據** | 文檔 ID、權限、協作者、當前 version | SQL / KV | 結構化、要查權限 |

> **核心 insight:文檔的「當前狀態」= 最近一個快照 + 之後的所有 op 重放。** 這和數據庫的 **快照 + WAL(redo log)** 是同一個模式——快照讓你不用從頭重放,op 日誌記錄增量。協同編輯本質上是把一篇文檔當成一個「**狀態機**」,op 是輸入,所有端按相同順序餵相同的 op,就到達相同狀態(這正是 `system-design/03` 共識/狀態機複製的思路,只是這裡用 OT/CRDT 而非 Raft 來定序)。

**分片鍵(`01` Step 4 鐵律):**

> **按 `docId` 分片。** 一篇文檔的所有連接、所有 op、所有狀態,都路由到**同一個協同節點**(它在內存裡持有這篇文檔的「權威狀態」)。這是這題分片的**命根子**——因為 OT 的衝突變換**必須在單點串行做**(見 deep dive ①),所以「一篇文檔 = 一個 owner 節點」不只是分片策略,是**正確性的前提**。docId 隨機高基數 → hash 後天然均勻,且**永遠單片命中**(協同從不跨文檔)。

> **面試話術:** 「按 docId 分片有個比『均勻』更深的理由——OT 模型要求同一篇文檔的並發 op 在**單一權威節點上串行變換**才能保證收斂,所以『一篇文檔由一個節點獨佔』是正確性的硬約束,不只是負載均衡。這也意味著**單篇文檔的吞吐有天花板**(單節點),但我們估算過單篇才 250 ops/s,離天花板遠得很。」

## 4. 高層架構(`01` Step 5:沿一次按鍵一鏡到底)

**先畫能跑的最簡版**(`01` 鐵律:別堆組件)。協同編輯的架構主幹其實不複雜,複雜的是**節點內部那個「協同引擎」黑盒**(deep dive 展開):

```
                                  ┌─────────────────────────────────────────┐
                                  │        協同節點(owner of docId)         │
   多個 client(同一篇 doc)        │                                          │
                                  │   ┌──────────────────────────────┐       │
 用戶A ──WS──┐                    │   │     協同引擎(OT / CRDT)       │       │
 用戶B ──WS──┼──► 接入層 ───路由──┼──►│  · 收 op、定序/變換             │       │
 用戶C ──WS──┘    (長連接,       │   │  · 應用到內存裡的文檔權威狀態   │       │
                  心跳/重連,      │   │  · version++                  │       │
                  錨 09 IM)       │   └───────┬──────────────────────┘       │
                                  │           │ 廣播已定序的 op               │
                                  │           ▼                              │
                                  │      扇出給本文檔所有連接 ──────► A/B/C 螢幕更新 │
                                  └───────────┼─────────────┬────────────────┘
                                              │             │
                                  op 日誌(append)    定期快照
                                              ▼             ▼
                                       ┌────────────┐  ┌──────────┐
                                       │  op log     │  │ 快照存儲  │
                                       │ (WAL/Kafka) │  │(對象存儲) │
                                       └────────────┘  └──────────┘
                                              │
                          presence(光標)─────┘ 盡力廣播,不持久化
```

**兩條典型路徑走一遍(`01` Step 5):**

- **一次按鍵(寫,實時路徑):** 用戶 A 在第 5 個字後插入 'x' → client 生成一個 op `{insert 'x' @ pos5, baseVersion: 42}` → WS 上行到接入層 → 路由到**這篇 doc 的 owner 節點** → 協同引擎做**並發變換/定序**(若同時有 B 的 op,要先變換,見 deep dive)→ 應用到內存權威狀態、version→43 → **廣播給 A/B/C 所有連接** + **append 到 op 日誌** → A 收到 ack、B/C 收到變換後的 op 在各自螢幕重放。**全程目標 <200ms。**

- **離線重連(合併路徑):** 用戶 C 斷網時本地照常編輯(本地暫存 op),重連後發 `sync_req{sinceVersion: 100}` → 服務器把 100→當前的所有 op 補發給 C(讓 C 追上)+ C 把離線期間積壓的本地 op 上傳(服務器對它們做變換後合入)→ **雙向追平,收斂到同一狀態**。這條路徑是 CRDT 比 OT 漂亮的地方(deep dive ②)。

**每加一個組件都說一句它解決什麼(`01` Step 5 要點):**

| 組件 | 為什麼加 | 對應第 2 步哪個結論 |
|---|---|---|
| **接入層長連接** | 每次按鍵雙向實時推送,輪詢扛不住延遲 | ③「連接數是約束」(錨 `09 IM`) |
| **doc owner 節點(內存權威狀態)** | OT 並發變換必須單點串行 | ①「按 docId 分片是正確性前提」 |
| **op 日誌(append-only)** | 真相源、重放出狀態、做版本歷史 | ④「op 日誌 + 快照」 |
| **定期快照** | 截斷無限增長的 op 日誌、加速載入 | ④「op 日誌會無限膨脹」 |

**注意這張圖的「克制」:** 沒有複雜緩存層(權威狀態就在內存)、沒有重型分片路由(按 docId 一跳到位)、沒有 CDN。**協同編輯的架構主幹本該樸素**——它的全部複雜度**集中在那個「協同引擎」黑盒裡的算法**,不在分布式拓撲。這正是它和前 16 題的根本區別:**別題的難在系統拓撲,這題的難在一個盒子裡的數據結構與算法。**

## 5. Deep Dive(這題的全部價值在這——三個點都往死裡鑽)

> 前面四步在協同編輯這題是「鋪墊」,**這一段才是肉**。面試官問協同編輯,90% 是想聽你能不能講清 **OT vs CRDT**。這兩個 deep dive 講不透,前面畫得再漂亮也過不了線。

### Deep Dive ① · OT(Operational Transformation):把並發建模成「可變換的操作」

**先看問題本身——「並發插入」為什麼難:** 文檔當前是 `"HELO"`(version 42)。兩個人**同時**(都基於 v42)操作:

- 用戶 A:在位置 3 插入 `"L"`(想修成 `"HELLO"`)→ op_A = `insert("L", pos=3)`
- 用戶 B:在位置 0 插入 `"O"`(想加個前綴)→ op_B = `insert("O", pos=0)`

兩個 op 同時到服務器。**如果服務器把 op_A 原封不動轉發給 B、op_B 原封不動轉發給 A**,會發生什麼?

```
A 的視角:本地已是 "HELLO"(pos3 插了 L),再收到 B 的 op_B=insert("O",pos0)
         → 在 pos0 插 O → "OHELLO" ✓(碰巧對了,因為 B 的位置在 A 改動之前)

B 的視角:本地已是 "OHELO"(pos0 插了 O),再收到 A 的 op_A=insert("L",pos3)
         → 在 pos3 插 L → "OHLELO" ✗  應該是 "OHELLO"!
```

**B 的結果錯了**——因為 op_A 的「pos=3」是相對於**原始 "HELO"** 算的,但 B 本地已經在前面插了一個字,**位置全部右移了一格**,pos=3 現在指向了錯誤的地方。**兩端發散了(A 看到 "OHELLO",B 看到 "OHLELO"),這就是並發編輯不收斂的本質。**

**OT 的解法——變換(transform):** 在把 op_A 應用到 B 的上下文之前,先用一個 **transform 函數** `T(op_A, op_B)` 把 op_A「改寫」成「在 op_B 已經生效的上下文裡，能達到相同效果」的新 op:

```
T(op_A=insert("L",pos3), op_B=insert("O",pos0))
  → 因為 op_B 在 pos0 插了 1 個字,op_A 的位置要右移 1
  → op_A' = insert("L", pos4)
B 應用 op_A':在 "OHELO" 的 pos4 插 L → "OHELLO" ✓ 收斂!
```

**OT 的核心契約(這是面試要講出來的一句話):** transform 函數必須滿足 **`apply(apply(s, op_A), T(op_B, op_A)) == apply(apply(s, op_B), T(op_A, op_B))`**——即「先 A 後 B'」和「先 B 後 A'」必須到達**同一個最終狀態**(這叫 **TP1 性質**,transformation property 1)。**這就是收斂的數學保證。**

**黑盒裡發生什麼——OT 的三個真實取捨:**

| 維度 | OT 的真相 |
|---|---|
| **必須有中心服務器** | transform 的順序敏感:服務器要**為所有並發 op 定一個全局順序**,再對每個 op 相對其他並發 op 做變換。**這個定序 + 變換必須在單點串行做**——這就是第 4 步「一篇文檔一個 owner 節點」的根本原因。OT 是**中心化**的協同模型。 |
| **變換函數是地獄** | 兩個操作類型(insert/delete)兩兩之間都要寫一個 transform,富文本加上格式操作後組合爆炸;且要證明滿足 TP1(甚至多人並發要 TP2),**歷史上 Google Wave 的 OT 實現以「極難寫對」聞名**,業界踩過無數收斂 bug。 |
| **離線是軟肋** | OT 依賴「服務器持續定序」。離線久了,客戶端積壓一大堆基於舊版本的 op,重連後服務器要把它們逐一相對中間發生的所有 op 做變換,**變換鏈很長、容易出錯**。OT 不是不能離線,是離線越久越痛。 |

> **面試話術(把 OT 講完整):** 「OT 的核心是:把每次編輯建模成一個帶位置的 operation,中心服務器給所有並發 op 定一個全局順序,然後在把別人的 op 應用到你的上下文前,用 transform 函數把它的位置『改寫』成適配你當前狀態的版本,數學上保證 `T(A,B)` 滿足 TP1——不管誰先誰後都收斂到同一結果。**它的優點是文檔模型簡單(就是一個字符串 + 位置),缺點是 transform 函數極難寫對、強依賴中心服務器定序、離線合併吃力。** Google Docs 早期就是 OT。」

### Deep Dive ② · CRDT(無衝突複製數據類型):讓數據結構本身保證收斂

> **這是這題的重錨深礦** —— CRDT/yjs 的同步算法細節在 `distr-tx/CRDT/yjs-sync-algorithm.md` 講透,**這裡不重寫**,只講「為什麼 CRDT 是另一條路、它和 OT 的本質區別、代價是什麼」,細節錨過去。

**CRDT 的根本思路和 OT 正相反:** OT 靠**中心服務器的變換 + 定序**來消解衝突;CRDT 靠**重新設計數據結構本身**,讓「並發操作以任意順序、任意次數應用,都自動收斂到同一結果」——**衝突在數據結構層面就不存在了**,所以叫 conflict-free。

**關鍵招數——給每個字符一個全局唯一、可全序排序的 ID,而不是用「位置」:** OT 的痛全來自「位置會因為別人的插入而漂移」。CRDT 釜底抽薪——**不用位置,用稳定 ID**:

- 每個插入的字符分配一個**唯一 ID**(通常是 `(siteId, 邏輯時鐘)` 或一個能在任意兩個 ID 之間生成新 ID 的「分數位置 / fractional index」)。
- 插入「在字符 X 和字符 Y 之間」——這是個**相對關係,不是絕對位置**,別人在別處插字**不影響** X 和 Y 之間的關係。
- 所有字符按 ID 全序排列 = 文檔內容。**任意兩端只要收到了相同的操作集合(不管什麼順序),按 ID 排序的結果必然相同。**

**這直接給了三條 OT 沒有的好處:**

| 性質 | CRDT 怎麼做到 | 對協同的意義 |
|---|---|---|
| **交換律(順序無關)** | op 帶稳定 ID,應用順序不影響最終排序 | **不需要中心服務器定序**——可以去中心化、P2P 直接同步 |
| **幂等(重複無害)** | 同一個 op ID 應用兩次 = 一次(靠 ID 去重) | 重連後**重發整批 op 不會出錯**——離線合併天然簡單 |
| **天然離線/去中心** | 不依賴中心定序,兩個離線分叉重連後合併必收斂 | **離線編輯是 CRDT 的主場**,正好命中我們「要支援離線」的需求 |

**黑盒裡發生什麼——CRDT 的代價(主動暴露,別只吹優點):**

| 代價 | 細節 |
|---|---|
| **元數據膨脹(tombstone)** | 刪除不能真刪——若真刪了,一個並發的「在被刪字符後面插字」的 op 就找不到錨點了。所以 CRDT **刪除靠打墓碑(tombstone)標記**,字符還在、只是不顯示。**墓碑會累積**,文檔內部數據遠大於可見內容,長期需要 GC/壓縮(這正是 `yjs-sync-algorithm.md` 要解的工程難點之一)。 |
| **per-字符元數據** | 每個字符都背一個唯一 ID + 因果元數據,**內存/存儲開銷比 OT 的純字符串大得多**(yjs 等成熟庫做了大量結構優化才把它壓到可用,比如用「塊」而非「逐字符」存)。 |
| **實現仍然複雜** | CRDT「概念簡單」但「寫對一個生產級的富文本 CRDT 一樣難」——富文本的格式、嵌套、撤銷(undo)在 CRDT 下都是硬骨頭。**別把 CRDT 說成『簡單』,它只是把難點從『變換函數』搬到了『數據結構設計 + 元數據治理』。** |

> **面試話術(OT vs CRDT 的決策,這是這題的最高分點):**
> 「OT 和 CRDT 是兩條根本不同的路。**OT** 用中心服務器對並發 op 做變換來收斂——文檔模型簡單(字符串+位置),但變換函數極難寫對、強依賴中心、離線吃力,適合**已有中心服務器、主要在線協同**的場景(Google Docs 早期)。**CRDT** 把收斂性做進數據結構本身——每個字符有稳定 ID,操作滿足交換律和幂等,所以**任意順序應用都收斂、天然支持離線和去中心化**;代價是**墓碑導致的元數據膨脹和 per-字符開銷**,實現的難點從變換搬到了數據結構治理。**我會選 CRDT**——因為我們明確要求離線編輯,而離線正是 CRDT 的主場;Yjs、Automerge 這些成熟庫也讓 CRDT 落地成本大降。細節我可以錨到 CRDT 的同步算法那塊展開。**如果是純在線、且團隊已有重型 OT 基礎設施,OT 也是合理的——這是個有上下文的決策,不是非黑即白。**」

這段「**我選 X 因為需求是離線,但 Y 在純在線場景也合理**」的話術,就是 `00` 標準線裡架構師的「**擁有這個決策**」——不是背「CRDT 更好」,是知道**在什麼約束下選什麼、代價是什麼**。

### Deep Dive ③ · 實時同步鏈路(把算法接回工程:連接、廣播、離線、存儲)

OT/CRDT 解決「收斂」,但要讓它**實時跑起來**,還有一整條工程鏈路。這四件事是「算法之外」最會被追問的:

**(1) 增量廣播(WebSocket 上只傳 op,不傳全文):**
- 首次連入:服務器給 **snapshot**(完整文檔 + version),client 渲染。
- 之後每次編輯:只在連接上推**那一條增量 op**(~50 B),client 在本地文檔上重放。**這是實時的前提**——50 人同時打字,每秒上萬條 op,只有「傳增量」才扛得住延遲;傳全文每次幾十 KB,延遲和帶寬都崩。

**(2) presence(光標/選區同步,錨 `09 IM` 的在線狀態):**
- 光標位置走**同一條連接的獨立通道**,廣播給文檔內其他人。
- **關鍵取捨:presence 不持久化、不保證可靠、不參與收斂**——光標飄一下、丟一幀,沒人在意。所以它走「盡力而為」廣播,**絕不能讓 presence 的可靠性要求拖累 op 的關鍵路徑**。這和 `09 IM` 的「在線狀態」是同一類弱一致數據,直接復用那套思路。

**(3) 離線重連的 op 補發與合併(這是離線需求的落地):**
- client 本地維護「**已確認 version**」和「**本地未確認 op 隊列**」。
- 斷網時:本地 op 照常應用到本地副本(用戶無感),積壓在未確認隊列。
- 重連時:① client 發 `sync_req{sinceVersion}`,服務器補發它缺的 op(讓它追上別人的改動);② client 上傳積壓的本地 op,服務器合入(**OT 要逐一變換,CRDT 直接按 ID 合併**)。
- **這一步正是 CRDT 完勝 OT 的地方**:CRDT 的 op 幂等 + 交換,重連後**無腦把兩邊的 op 集合並起來**就收斂;OT 要把積壓 op 相對中間所有 op 做變換鏈,長且易錯。

**(4) 版本快照 + op 日誌的存儲(接回第 2 步④的「op 無限增長」):**
- **真相源是 op 日誌**(append-only,有序)。「當前狀態 = 最近快照 + 之後的 op 重放」。
- **為什麼要快照:** op 日誌無限增長,若每次載入都從頭重放幾百萬條 op,既慢又佔存儲。所以**定期把當前狀態固化成一個 snapshot,並截斷該 snapshot 之前的 op 日誌**(像 DB 的 checkpoint / Redis 的 RDB+AOF 重寫)。
- **版本歷史 = 快照 + op 日誌的回放**:想看「昨天下午 3 點的版本」,就取最近的快照、重放到那個時間點的 op。回滾 = 生成一個「把內容改回某版本」的新 op(**不是刪歷史**,協同編輯的歷史只增不改,符合 op 日誌 append-only 的本性)。

> **面試話術:** 「算法之外,實時鏈路有四個工程點:**只傳增量 op 不傳全文**(實時的前提)、**presence 走弱一致盡力廣播不拖累關鍵路徑**(錨 IM 的在線狀態)、**離線靠『本地未確認隊列 + 重連雙向補發』合併**(CRDT 在這裡完勝 OT,因為 op 可交換可幂等)、**存儲用快照 + op 日誌**(快照截斷無限增長的日誌,版本歷史靠重放,和數據庫 checkpoint+WAL 一個模式)。」

## 6. 架構師深挖層(推過面試線:成本 / 容量 / 演進 / 失效模式 / 何時不該)

> IC 線到第 5 步講清 OT vs CRDT 就能拿 offer;這一段是 `00` 的**架構師決策線**——把「我選 CRDT」升級成「這個選擇的代價誰買單、撐到多大、會怎麼掛、什麼時候根本不該上協同編輯」。

### 成本:這題不花在規模,花在「per-用戶有狀態節點」和「元數據膨脹」

回到第 2 步——規模小,所以成本結構和別題完全不同:

| 維度 | 賬 | 架構師決策 |
|---|---|---|
| **有狀態節點內存** | 每篇活躍文檔的權威狀態 + 所有連接的會話都常駐 owner 節點內存 | 主要成本項。協同節點是**有狀態**的(不像無狀態 web 機可隨意擴),要按「活躍文檔數 × 單篇狀態大小 + 連接數」配內存;**冷文檔要能被驅逐**(沒人編時把狀態卸載到存儲,有人來再載入) |
| **CRDT 元數據膨脹** | 墓碑 + per-字符 ID,內部數據可達可見內容的數倍 | 用成熟庫(yjs)的塊化結構壓開銷;**定期 GC 墓碑**(在「所有副本都已確認某刪除」後才能安全清,需因果追蹤,錨 `yjs-sync-algorithm.md`) |
| **op 日誌存儲** | append-only,無快照會無限漲 | **快照 + 截斷**把線性增長壓平;冷文檔歷史下沉冷存儲 |
| **計算/帶寬** | 全站 50 萬 ops、每條 50 B | 幾乎不花錢,別過度配——這題沒有 CDN/轉碼這種重成本項 |

> **架構師視角的成本本質:** 協同編輯是**「有狀態 + 元數據重」**的系統,它的成本曲線和「無狀態 + 規模驅動」的 News Feed 完全不同——**貴在每篇活躍文檔都要一個常駐內存的權威狀態,以及 CRDT 為了收斂付出的元數據稅。** 規模反而便宜。

### 容量規劃:單篇文檔吞吐有天花板(這是別題沒有的約束)

別題的擴容是「加機器分流量」;協同編輯有個**結構性天花板**——**一篇文檔的所有 op 必須由單一 owner 節點串行處理**(OT 是正確性硬約束;CRDT 雖可去中心,但生產上仍常用單 owner 簡化)。所以:

- **單篇文檔的吞吐 = 單節點處理能力**,加機器**救不了單篇熱點**。
- 好在估算過單篇才 250 ops/s,離單節點天花板遠;但架構師要知道**這個天花板存在**——如果有一天要做「一萬人同時編輯同一篇」(像超大型協作白板),單 owner 模型會撐不住,要換**分區協同**(把文檔切成多個獨立協同的子區域)或**分層 CRDT**,那是另一個量級的設計。
- **擴容方向是「文檔數」而非「單篇並發」**:加協同節點 = 能承載更多**篇**文檔,不是讓單篇更快。

### 演進路徑:從 MVP 到生產

```
階段 1(MVP):單協同節點 + OT(或直接用 yjs CRDT 庫)+ WebSocket
   單篇 250 ops、按 docId 分片,先把「兩人同步」跑通,別碰離線
        │  要支援離線編輯 + 多端
        ▼
階段 2:切 CRDT(yjs/automerge)+ 本地未確認隊列 + 重連合併
   離線成主場,client 端做本地副本,op 可交換可幂等
        │  活躍文檔多、單機內存吃緊
        ▼
階段 3:有狀態節點集群 + 冷文檔驅逐/載入 + 快照截斷 + 墓碑 GC
   按 docId 一致性哈希路由,冷熱分離,op 日誌 checkpoint
        │  跨地域協作、要就近低延遲
        ▼
階段 4(極致):多區域 + 文檔狀態就近,CRDT 的去中心特性讓跨區同步更自然
```

> **這條演進的關鍵 insight:** **「上來就用成熟 CRDT 庫(yjs)」往往比「自己寫 OT」更省**——OT 自己寫變換函數是著名的坑,而 yjs 把 CRDT 的元數據治理都封裝好了。架構師的判斷是「**這題別造輪子**」:協同算法是 20 年的學術 + 工程積累,自研 OT/CRDT 的 ROI 極低,**用 yjs/automerge 把難題外包給庫,把精力放在業務(權限、富文本 schema、版本 UI)上**。這比「我能手寫 OT」更顯成熟。

### 失效模式(`01` Step 7:哪個組件先撐不住,掛了怎樣)

| 失效 | 影響 | 緩解 |
|---|---|---|
| **owner 節點掛(有狀態!)** | 該文檔所有人的編輯中斷,內存裡未持久化的 op 可能丟 | **op 必須先持久化到日誌再 ack**(WAL 順序);owner 掛了,從**最近快照 + op 日誌重放**在另一節點重建狀態,連接重連過去。**這是有狀態服務的容災核心——狀態能從日誌重建,節點才能死** |
| **網絡分區 / 客戶端離線久** | 兩端長時間分叉 | **CRDT 主場**:重連後 op 集合並起來必收斂,墓碑保證刪除也對齊;OT 則變換鏈長、風險高 |
| **時鐘/邏輯時鐘問題** | 並發 op 的因果順序判斷錯,可能不收斂 | 用**邏輯時鐘(Lamport / 向量時鐘)**而非物理時鐘定因果序——錨 `system-design/02 一致性模型與時鐘`,**這是協同收斂的時鐘地基** |
| **墓碑無限累積** | 文檔越用越臃腫,內存/同步變慢 | 因果追蹤下安全 GC(所有副本確認後清),錨 `yjs-sync-algorithm.md` |
| **presence 風暴** | 50 人狂動光標,廣播量大 | presence 可**節流/合併**(每 50ms 才廣播一次光標),且**可丟**——不拖累 op 關鍵路徑 |

> **架構師話術:** 「我的核心風險是 **owner 節點是有狀態的**——它掛了文檔編輯就斷。所以鐵律是 **op 先落日誌再 ack**,狀態永遠能從『快照 + op 日誌』重放重建,在別的節點復活、連接重連過去——這是有狀態服務和無狀態 web 機容災的根本區別。第二個地基是**邏輯時鐘**:並發 op 的因果序判斷錯了就不收斂,所以我用 Lamport/向量時鐘而非物理時鐘,這塊錨一致性模型那章。CRDT 在網絡分區下反而最穩,因為收斂性是數據結構保證的。」

### 何時**不該**用協同編輯這套(推過面試線的關鍵一問)

| 場景 | 別套這套方案 | 該怎麼做 |
|---|---|---|
| **文件級同步(整檔覆蓋)** | 別上 OT/CRDT 字符級協同 | 那是 `12 Dropbox/網盤`——內容哈希 + 版本向量 + 衝突就「保留兩份讓用戶選」,**不需要字符級收斂** |
| **協作但不要求實時/同段並發** | 別上長連接 + 協同引擎 | 加**鎖 / 段落級簽出(check-out)** 讓一次只有一人改一段,衝突直接避免——很多企業 wiki 就這麼做,簡單可靠 |
| **強一致業務數據(訂單/餘額)** | 別用「最終一致 + 收斂」 | 那是 `14 支付`——錢必須強一致、不能「兩個改動都保留」,走事務 + 冪等,**和協同編輯是相反的一致性世界** |
| **超大規模單文檔(萬人同編)** | 別用單 owner 模型 | 單節點吞吐天花板會爆,要分區協同 / 分層 CRDT,是另一個量級的題 |
| **只有兩三個人偶爾協作** | 別上 CRDT 全套元數據稅 | 簡單的「最後寫入勝出 + 提示衝突」可能就夠,別為小場景付收斂的全部複雜度 |

> **這張表是架構師線的精華**——協同編輯最容易被**過度套用**:很多需求其實只是「文件同步」或「段落鎖」,根本不需要字符級收斂的全套複雜度。**知道『這個需求不需要 CRDT』比『我會寫 CRDT』高一個段位**(`00` 標準線:「畫得出」→「擁有決策、知道邊界」)。

## 8. 面試卡(抽認卡,蓋住背面自測)

| # | 正面 | 背面 |
|---|---|---|
| 1 | 協同編輯的容量估算結論一句話? | **規模不是瓶頸**——單篇 ~250 ops/s、全站 ~50 萬 ops、扇出範圍僅同篇 ≤50 人、按 docId 分片天然均勻。唯一基礎設施約束是長連接數(~100 台接入機,錨 IM)。**90% 難度在『50 人同改一段怎麼正確收斂』,不在規模。** |
| 2 | 「並發插入」為什麼難?(一句話講清不收斂) | 兩人同時基於同一版本在不同位置插字,別人的插入會讓你 op 的「絕對位置」漂移 → 原樣轉發會讓兩端發散(如 A 看到 "OHELLO"、B 看到 "OHLELO")。**位置會漂移 = 並發編輯不收斂的本質。** |
| 3 | OT 怎麼解決收斂?核心契約是什麼? | **中心服務器**對並發 op 定全局序,把別人的 op 用 **transform 函數**改寫成「在你當前上下文能達到相同效果」的新 op。契約:`T` 滿足 **TP1**——「先 A 後 B'」與「先 B 後 A'」到同一狀態。**優點模型簡單,缺點變換函數極難寫對、強依賴中心、離線吃力。** |
| 4 | CRDT 怎麼解決收斂?和 OT 本質區別? | 把收斂做進**數據結構本身**:每個字符給**稳定唯一 ID**(不用會漂移的位置),op 滿足**交換律 + 幂等** → **任意順序/任意次數應用都收斂**。本質區別:OT 靠**中心變換+定序**,CRDT 靠**數據結構天然無衝突** → 天然支持**離線 + 去中心**。代價:**墓碑/per-字符元數據膨脹**。錨 `yjs-sync-algorithm.md`。 |
| 5 | OT vs CRDT 怎麼選?有上下文的決策 | **要離線 → 選 CRDT**(離線是 CRDT 主場,op 可交換幂等,重連並集即收斂;yjs/automerge 讓落地成本大降)。**純在線 + 已有 OT 基礎設施 → OT 也合理**(模型簡單)。不是非黑即白,看「是否需要離線/去中心」這個約束。 |
| 6 | 為什麼刪除要打墓碑(tombstone)而不真刪? | 真刪會讓「並發地在被刪字符後插字」的 op 找不到錨點 → 不收斂。所以 CRDT **刪除=打墓碑、字符還在只是不顯示**。代價:墓碑累積、文檔內部 >> 可見內容,需在「所有副本確認刪除後」安全 GC。 |
| 7 | 協同編輯的存儲模型?op 日誌為何要快照? | **快照 + op 日誌**(append-only):當前狀態 = 最近快照 + 之後 op 重放(同 DB 的 checkpoint+WAL)。**op 日誌無限增長**,所以定期固化快照並截斷舊日誌;版本歷史 = 快照 + 重放;回滾 = 生成新 op(歷史只增不改)。 |
| 8 | owner 節點是有狀態的,掛了怎麼辦? | 鐵律:**op 先落日誌再 ack**。節點掛了從**最近快照 + op 日誌重放**在另一節點重建狀態,連接重連過去。**狀態能從日誌重建,節點才能死**——這是有狀態服務容災的核心。 |
| 9 | 協同編輯何時**不該**用? | 文件級整檔同步(走 `12` 內容哈希+版本向量)、不要求同段並發(用段落鎖/簽出)、強一致業務數據(走 `14` 事務,錢不能「兩改都留」)、萬人同編單文檔(單 owner 天花板,要分區協同)、小場景(別付 CRDT 全套元數據稅)。 |

## 一句話總綱(背起來)

> **協同編輯是 track 裡唯一「規模不是瓶頸」的題——單篇 250 ops、按 docId 分片天然均勻,難點全在『50 人同改一段怎麼收斂』。兩條路:OT 靠中心服務器對並發 op 做 transform 定序收斂(模型簡單但變換函數地獄、依賴中心、離線吃力),CRDT 靠給每字符稳定 ID 讓 op 交換+幂等、數據結構天然無衝突(天生支持離線去中心,代價是墓碑元數據膨脹,錨 yjs)。因需求要離線 → 選 CRDT。工程鏈路:只傳增量 op、presence 走弱一致盡力廣播(復用 IM)、離線靠本地未確認隊列+重連並集合並、存儲用快照+op 日誌(同 DB checkpoint+WAL)。架構師線:owner 是有狀態節點,靠『op 先落日誌再 ack + 從快照重放重建』容災;邏輯時鐘是收斂地基;別把『文件同步/段落鎖』需求過度設計成字符級 CRDT。**
