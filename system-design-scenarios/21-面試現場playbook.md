# 21 · 面試現場 playbook

> 前面所有章教你「知道怎麼設計」。這章教你「**在 45 分鐘、有人盯著、還會被打斷的壓力下,把它演出來**」。
>
> 真相:系統設計面試**不是筆試,是一場有觀眾的即興演出**。同樣的設計能力,會演的人拿 strong hire,不會演的人——架構畫對了卻被判「溝通差、抓不住重點」。這章是那套表演劇本。

---

## 1. 時間分配(45 分鐘的現場版)

`01` 給過理想分配,但現場永遠不夠用。記住這條**節奏線**和「時間不夠怎麼砍」:

| 階段 | 理想 | 累計 | 時間緊時怎麼砍 |
|---|---|---|---|
| 澄清 + 估算 | 10 min | 10 | 估算只留「決定架構的那 1–2 個數」,其餘口頭帶過 |
| API + 數據模型 | 8 min | 18 | API 列 2–3 個核心的,數據模型直接講分片鍵 |
| 高層架構 | 10 min | 28 | **這步不能砍**——主菜 |
| Deep dive | 12 min | 40 | 砍到只深入 1 個點,但要深 |
| 取捨收尾 | 5 min | 45 | **這步不能砍**——加分核心 |

> **現場鐵律:** 時間到一半(~22 min)還沒畫出高層架構 = 危險信號,立刻收住澄清/估算往前推。**寧可架構畫完留時間 deep dive,也別在前戲耗死。**

**主動報時:** 「我先花兩分鐘把規模估出來,然後直接進架構。」——讓面試官知道你有時間意識,也讓他能喊停重定向。

---

## 2. 畫圖順序(白板怎麼長出來)

別想一次畫完美。架構圖是**長出來的**,每加一塊都帶一句理由:

```
① 先畫兩端:Client ──────────────── DB
② 中間填主幹:Client → Gateway → Service → DB
③ 按估算結論加組件:
   - 讀多 → Service 和 DB 間插 Cache
   - 寫尖峰 → Service 後插 MQ → Worker
   - 媒體/靜態 → Client 前插 CDN
④ 標數據流:沿「寫」走一條線,再沿「讀」走一條線
```

**口訣:兩端 → 主幹 → 按需加料 → 標數據流。** 每加一個框,說一句「這裡加 X 是因為估算出 Y」——讓面試官看到你的決策鏈,而不是憑空變出組件。

---

## 3. 怎麼接 follow-up(面試官的追問套路)

面試官的追問就那幾種,認出來就不慌:

| 追問類型 | 他想看什麼 | 你怎麼接 |
|---|---|---|
| 「如果流量 ×10 呢?」 | 你的擴展性思路 | 指出當前瓶頸 → 給下一級方案(加分片/加緩存層/多活) |
| 「這裡掛了會怎樣?」 | 失效模式意識 | 講降級/容錯,不要嘴硬說「不會掛」 |
| 「為什麼不用 X?」 | 你是否真懂取捨 | 承認 X 的好處,再說你的場景為什麼選了當前方案 |
| 「一致性怎麼保證?」 | 你對 CAP 的掌握 | 先問「這業務能接受最終一致嗎」,再給對應方案 |
| 「資料量再大十倍?」 | 分片/冷熱分離 | 分片鍵 + 歸檔冷數據 |

> **核心心法:** 追問不是攻擊,是**遞話筒**。面試官追問的點,往往就是這題的加分點——他在給你機會展示深度。**接住,別防守。**

---

## 4. 主動暴露 tradeoff(加分話術庫)

這是資深和架構師的**分水嶺動作**。背幾個句式,臨場套:

- 「我選了 A,**換來** X,但**犧牲**了 Y。在這個讀多寫少的場景,我認為值得,因為……」
- 「這套的**單點**是 Z,它掛了我會降級到……」
- 「現在的設計**撐到大約 N 倍**流量,再往上我會……」
- 「如果業務需求反過來變成寫密集,我會**改用**……」
- 「這裡我**故意不做** XX,因為當前規模不需要,過度設計反而增加複雜度。」

> 最後一句尤其加分——**主動說「我不做什麼」**展示你懂 YAGNI、不會過度設計,這是高級信號。

---

## 5. 不會的題怎麼推(沒見過也能扛)

碰到沒準備過的題(設計一個 XX),別僵。用**第一性原理 + 類比**推:

1. **拆成讀路徑 + 寫路徑**——任何系統都是這兩條。
2. **問自己「它本質像哪道練過的題」**:
   - 「要把一條消息發給很多人」→ 像 News Feed / 通知(扇出)
   - 「要在海量數據裡按某維度查」→ 像搜索 / 排行(索引)
   - 「要保證錢不錯」→ 像支付(冪等 + 對賬)
   - 「要很多人同時連著」→ 像 IM(長連接)
3. **套七步框架**——框架本身就是「不會時的腳手架」,照走一定有結構。

> 面試官常故意出沒有標準答案的題,**看的就是你沒準備時的推理過程**,不是背答案。能套框架、能類比,就贏了一半。

---

## 6. 紅旗 vs 加分信號(對照自查)

| 🚩 紅旗(扣分) | ✅ 加分信號 |
|---|---|
| 沒澄清就開畫 | 先收斂範圍 + 確認假設 |
| 沒估算就拍架構 | 估完接「所以要……」 |
| 一上來拆 20 個微服務 | 先最簡能跑,再按需加料 |
| 框圖講不出數據流 | 沿一個請求一鏡到底 |
| 假裝方案沒缺點 | 主動暴露 tradeoff 和單點 |
| 被問才被動答,停在表面 | 主動挑難點深入 |
| 悶頭畫不出聲 | 邊畫邊講思路(think out loud) |
| 面試官重定向了還在原地 | 立刻 pivot,順著他走 |

---

## 7. 溝通:這是「協作」不是「考試」

- **出聲思考**(think out loud):你的思考過程比結論值錢。沉默 = 面試官看不到你會什麼。
- **驅動但不固執**:你主導流程(driving),但面試官一重定向就立刻跟(「好,那我們聚焦在 X」)。被 led 著走 = 扣分;不聽指揮硬走 = 更扣分。
- **確認再深入**:「我打算深入講緩存一致性,可以嗎?」——把方向盤遞一下,避免在他不關心的點上耗。
- **承認不知道**:「這個我不確定,但我的直覺是……」比瞎編強。

---

## 8. 不同級別,面試官在看什麼(校準你的發力點)

| 級別 | 重心 | 你要多花力氣的地方 |
|---|---|---|
| 中級 | 能畫出合理、能跑的架構 | 完整性、基本組件選對 |
| **資深(你的目標)** | **能 reason about tradeoff**、主動暴露代價 | 第 6/7 步:deep dive 深度 + tradeoff |
| 架構師 | **擁有決策**:成本/容量/演進/組織 | 每題第 6 段「架構師深挖層」那套 |

> 同一道題,中級講「我用 Redis 緩存」就行;資深要講「為什麼是 Cache-Aside 不是 Write-Through、一致性窗口多大、能不能接受」;架構師還要講「這緩存層每月多少成本、撐到多大要再分片、誰來維護」。**往上一級,就是多回答一層 why 和 who pays。**

---

## 9. 臨場急救(卡住了怎麼辦)

- **腦子空白** → 回到七步,從「我重新確認一下需求」開始,邊說邊找回節奏。
- **時間爆了** → 「時間關係我快速說一下剩下的設計和取捨」,直接跳到收尾講 tradeoff(保住加分項)。
- **被問倒** → 「好問題,我現場推一下」+ 第一性原理拆,別裝懂。
- **畫亂了** → 「我重新整理一下這張圖」,擦掉重畫比將就強。

---

## 一句話總綱

> **系統設計面試是一場即興演出:用節奏線守住「半場必須畫完架構」,白板按『兩端→主幹→按需加料→標數據流』長出來,邊畫邊出聲思考,把面試官的追問當遞話筒接住,主動暴露 tradeoff 和單點,沒見過的題就套七步 + 類比已知題。記住——他不是在考你背了多少,是在看你怎麼 reason、怎麼協作、怎麼面對不確定。**
