# 第一個閉環 —— 親手跑出你自己的 P99 曲線

> 這份不是讓你讀的,是讓你**跑**的。30–45 分鐘,你會親手讓 P99 在眼前爆炸、把一個服務壓到飽和、找到瓶頸、修掉、再測。
>
> **為什麼這一次比讀十遍重要**:你已經有比多數工程師都多的文檔,卻還是覺得缺。瓶頸從不是缺知識,是這些從沒變成**親手做過一次**。下面每條命令的真實輸出我都先跑過貼在這裡了——你照著跑,看到的數字應該和這裡一致(機器不同會有出入,形狀一樣)。

每一步都用同一格式:**跑什麼 → 你會看到什麼 → 為什麼 → 它證明 6 原語裡的哪一條**。

## 第一幕 · 純模擬:看 P99 曲棍球桿(10 分鐘)

零基建、確定性。先在最乾淨的環境看 Little 定律的拐點,不被任何真實噪音干擾。

### 步驟 1 — 跑一次基準

**跑什麼**
```bash
python3 lab/sim/little.py --lam 200 --w 0.05
```

**你會看到什麼**
```
L = λ·W = 200.0 × 0.05 = 10.00 in flight

ρ      R=S/(1-ρ)   (bar ∝ R) — watch it explode toward ρ=1
0.1        11.1ms  #
0.3        14.3ms  #
0.5        20.0ms  ##
0.7        33.3ms  ###
0.8        50.0ms  #####
0.9       100.0ms  ##########
0.95      200.0ms  ###################
0.99     1000.0ms  ############################################################
```

**為什麼**
- 第一行就是 **Little 定律** `L = λ × W`:到達率 200/s × 停留 0.05s = 同時 10 個在途。
- 下面那張表是**飽和曲線**:利用率 ρ 從 0.9 → 0.99(只多用了 9% 的資源),延遲卻從 100ms 炸到 **1000ms(10 倍)**。這就是「曲棍球桿」——延遲在拐點是**暴漲**,不是線性。

**它證明**:⑤ Little 定律 + 飽和曲線 · ① 延遲(看的是隨負載變化的尾巴)。

### 步驟 2 — 自己撥旋鈕,把直覺撥出來

**跑什麼**(改服務時間 W,看天花板怎麼變)
```bash
python3 lab/sim/little.py --lam 200 --w 0.02     # 更快的服務
python3 lab/sim/little.py --lam 400 --w 0.05     # 更高的到達率
```

**你會看到什麼**:L(在途數)跟著 `λ×W` 變;但那條曲棍球桿的**形狀不變**——任何排隊系統,逼近 ρ=1 都會這樣爆。

**為什麼**:讓你親手確認「拐點是系統的普遍性質,不是某個數字的巧合」。這就是為什麼**沒人把目標利用率定在 95%**——那是站在懸崖邊。

**它證明**:⑤ + ④(利用率/飽和度是同一件事的兩面)。

## 第二幕 · 真服務:把它壓到飽和,修瓶頸,再測(25 分鐘)

現在換真東西:一個 FastAPI 服務,`/slow` 接口背後有一個**有界池**(`POOL_SIZE`),池滿就回 503。你要親手把它壓爆,然後修。

> 開**兩個終端**。終端 A 跑服務,終端 B 打流量。兩個都先 `cd concurrency-capacity/lab/service`。

### 場景 1 — 壓到飽和,看吞吐天花板

**跑什麼**

終端 A(起服務,池開 8):
```bash
cd concurrency-capacity/lab/service
POOL_SIZE=8 uvicorn app:app --port 8000 --workers 1
```
終端 B(階梯加壓,每個請求佔池 50ms):
```bash
cd concurrency-capacity/lab/service
python3 drive.py --url 'http://127.0.0.1:8000/slow?ms=50' --steps 50,100,150,200,400 --seconds 4
```

**你會看到什麼**(我實跑的結果)
```
rps=50    qps=49.6    p50=53.3 ms  p99=61.3 ms   errors=0
rps=100   qps=98.9    p50=52.1 ms  p99=58.0 ms   errors=0
rps=150   qps=146.9   p50=52.2 ms  p99=58.8 ms   errors=5      ← 膝點
rps=200   qps=143.9   p50=51.8 ms  p99=71.9 ms   errors=217
rps=400   qps=150.3   p50=51.5 ms  p99=61.5 ms   errors=991    ← qps 卡死 ~150
```
再讀一次計數器:
```bash
curl -s http://127.0.0.1:8000/stats        # → rejected≈1213, pool_size=8, max_in_flight≈12
```

**為什麼**
- **吞吐天花板 = 池大小 ÷ 服務時間 = 8 ÷ 0.05 = 160 rps**(這就是 Little 定律反過來用!)。壓到 200、400,實際 qps 卡在 ~150 上不去,**多出來的全被甩成 503**(`rejected` 一路漲)。
- 注意一個反直覺點:**p50/p99 反而穩住在 ~52/60ms**,沒有像第一幕那樣爆。為什麼?因為這個池**有界 + 丟棄**:它不讓請求無限排隊,而是把進不來的直接拒絕(背壓)。**這是拿可用性(503)換有界延遲**——和第一幕「無界佇列→延遲爆炸」正好是兩種設計的對照。

**它證明**:② 吞吐天花板 · ③ RED(你同時看到 Rate 卡頂、Errors 暴漲、Duration 穩定)· ④ USE(`rejected`/`max_in_flight` 就是飽和度讀數)· ⑤ Little 定律。

### 場景 2 — 修瓶頸,再測(閉環的關鍵一下)

瓶頸是「池太小」。修法:把池調大,讓天花板抬到負載之上。

**跑什麼**

終端 A:`Ctrl-C` 停掉,重起,池開 32:
```bash
POOL_SIZE=32 uvicorn app:app --port 8000 --workers 1
```
終端 B:**跑一模一樣的階梯**(才能對比):
```bash
python3 drive.py --url 'http://127.0.0.1:8000/slow?ms=50' --steps 50,100,150,200,400 --seconds 4
```

**你會看到什麼**(我實跑的結果)
```
rps=50    qps=49.6    p50=53.1 ms  p99=64.7 ms     errors=0
rps=100   qps=98.9    p50=52.2 ms  p99=83.3 ms     errors=0
rps=150   qps=148.2   p50=52.6 ms  p99=127.6 ms    errors=0     ← 原本這裡開始錯,現在乾淨
rps=200   qps=197.5   p50=51.8 ms  p99=149.8 ms    errors=0     ← 原本 217 個 503,現在 0
rps=400   qps=37.6    p50=370.8 ms p99=17673.1 ms  errors=844   ← ⚠️ 新問題,見場景 3
```
跑完**立刻再讀一次計數器**(這就是場景 3 那 844 個錯到底是不是 503 的證據):
```bash
curl -s http://127.0.0.1:8000/stats        # → rejected≈20, pool_size=32, max_in_flight≈32
```
對比場景 1 的 `rejected≈1213`:這次服務端幾乎沒拒。errors=844 但 rejected≈20 → 那 ~824 個失敗服務端從沒拒過,死在客戶端(10s timeout)。

**為什麼**
- 天花板從 160 抬到 **32 ÷ 0.05 = 640 rps**,所以 200 rps 現在游刃有餘——**之前的 217 個 503 全消失了**。
- 你剛剛完成了一次完整閉環:**壓測 → 觀察異常(503)→ 定位根因(池=8 天花板 160)→ 改(池→32)→ 復測(503 歸零)**。這就是「會排查」和「只看懂」的分界線。

**它證明**:閉環本身(觀察→假設→修→驗證)· ④ USE(改的是資源的飽和點)。

### 場景 3 —(加碼)瓶頸會搬家

看到 400 rps 那行了嗎?池夠大了(天花板 640),它卻**崩得更慘**:qps 暴跌到 37,p99 飆到 **17 秒**,844 個錯。

**為什麼**(這是資深和普通的分水嶺)
- 證據就在上面那條 `curl /stats`:`rejected` 只有 ~20(場景 1 是 1213)——也就是**這 844 個錯幾乎不是服務端 503,是客戶端超時**(`drive.py` 的 10s timeout)。`errors`(壓測端數的)和 `rejected`(服務端數的)是兩個不同來源的計數器,差額就是死在客戶端的量。
- 瓶頸**搬家了**:從服務端的池,搬到了**壓測端**——單一 `httpx` client 的連線池 + 單 uvicorn worker 的 event loop 扛不住 400 rps 的開環壓力。
- 兩個鐵律:**(1) 修掉一個瓶頸,下一個就冒出來**(性能優化是打地鼠);**(2) 你的壓測工具自己常常才是瓶頸**——這正是 `drive.py` 註解裡提的 coordinated omission 的近親。下次看到「服務端指標都正常但延遲很爛」,先懷疑你的測量工具。

**它證明**:④ USE(換個資源看)· 以及一個元技能:**永遠不要無條件相信你的壓測數字**。

> 想接著挖場景 3:這就是 [`../concurrency-capacity/`](../concurrency-capacity/) 的主場(多 worker、隔離、背壓),以及 [`07-load-testing/`](./07-load-testing/) 教你怎麼讓壓測端不成為瓶頸。

下一步：把 SELF-DIAGNOSTIC 右欄(能在真機上做)對應的格子改成 ✓;
