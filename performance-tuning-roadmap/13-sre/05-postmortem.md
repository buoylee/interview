# 事故复盘

事故复盘（Postmortem）是 SRE 实践中最重要的学习机制。**不复盘的团队注定重蹈覆辙。** 但复盘做得不好比不做更糟——流于形式的复盘浪费时间，追责式的复盘让人隐瞒真相。本文介绍如何做一次真正有价值的事故复盘。

---

## 一、无责文化（Blameless Postmortem）

### 为什么无责

传统的事故处理方式是找到「犯错的人」然后处罚。这种方式有两个致命问题：

1. **人会隐瞒** —— 如果犯错意味着处罚，工程师会隐瞒关键信息，导致无法找到真正的根因
2. **系统才是根因** —— 如果一个人的操作失误就能导致严重事故，说明系统缺乏必要的安全防护

```
❌ 有责思维："小王误删了数据库，应该扣绩效"
✅ 无责思维："为什么一个工程师能在没有二次确认的情况下直接删除生产数据库？"
```

### 无责的边界

无责 ≠ 无底线。以下情况不在无责保护范围内：

| 受保护 | 不受保护 |
|--------|---------|
| 操作失误（手滑、命令写错） | 故意破坏 |
| 判断失误（选错了方案） | 明知故犯（明确违反已知规程） |
| 知识不足（不知道某个影响） | 反复犯同样的错误且拒绝学习 |
| 流程漏洞导致的错误 | 蓄意绕过安全机制 |

### 如何营造无责氛围

```
1. 复盘开场明确声明："这次复盘不追究个人责任"
2. 使用被动语态描述事件："配置被误改"而非"小王改错了配置"
3. 聚焦系统改进："如何让这种错误不可能发生"而非"谁犯了错"
4. 领导以身作则：领导公开复盘自己参与的事故
5. 奖励主动上报：对主动发现和上报问题的行为给予正面反馈
```

## 二、事故复盘模板

以下是一份可以直接使用的复盘文档模板：

```markdown
# 事故复盘报告

## 基本信息

| 字段 | 内容 |
|------|------|
| 事故编号 | INC-2024-042 |
| 事故等级 | P1 |
| 事故标题 | 商品查询接口 P99 延迟突增至 5s |
| 影响时间 | 2024-03-15 14:23 ~ 15:47（共 84 分钟） |
| 影响范围 | 所有用户的商品查询功能，约 12 万用户受影响 |
| 影响程度 | 查询成功率从 99.95% 降至 78%，P99 从 200ms 升至 5000ms |
| 复盘时间 | 2024-03-18 14:00 |
| 复盘主持 | 张三 |
| 参与人 | 李四（值班）、王五（DBA）、赵六（架构师） |

## 事故时间线

| 时间 | 事件 | 操作人/系统 |
|------|------|------------|
| 14:20 | 上线新版本，包含商品查询逻辑变更 | CI/CD 系统 |
| 14:23 | Grafana 告警：商品查询 P99 > 1s | Prometheus |
| 14:25 | 值班工程师收到 PagerDuty 通知 | 李四 |
| 14:30 | 开始排查，初步怀疑是新版本导致 | 李四 |
| 14:35 | 查看 Jaeger，发现 MySQL 查询耗时从 5ms 升至 3s | 李四 |
| 14:40 | 检查 MySQL 慢查询日志，发现全表扫描 | 王五 |
| 14:50 | 定位根因：新代码的查询条件导致索引失效 | 李四 + 王五 |
| 15:00 | 决定回滚，开始执行回滚流程 | 李四 |
| 15:15 | 回滚完成，P99 恢复至 200ms | CI/CD 系统 |
| 15:30 | 确认所有指标恢复正常 | 李四 |
| 15:47 | 关闭事故，进入复盘流程 | 李四 |

## 根因分析

### 直接原因
新上线的商品查询代码中，WHERE 条件对 `price` 字段使用了 `CAST(price AS CHAR)`
进行字符串匹配，导致 `price` 列上的索引失效，查询退化为全表扫描。

### 根本原因（5 Whys）

1. **Why**: 为什么查询变慢了？
   → 因为索引失效，变成了全表扫描

2. **Why**: 为什么索引失效了？
   → 因为代码中对索引列做了类型转换（CAST）

3. **Why**: 为什么这样的代码能上线？
   → 因为没有 SQL 审核流程，代码审查时没有关注 SQL 变更

4. **Why**: 为什么代码审查没有发现？
   → 因为团队没有 SQL 变更的审查清单，审查者不了解索引失效规则

5. **Why**: 为什么没有在预发环境发现？
   → 因为预发环境数据量只有 1 万行，全表扫描也很快，掩盖了问题

### 根因分类
- [x] 代码缺陷
- [ ] 配置错误
- [ ] 基础设施故障
- [x] 流程缺失
- [x] 测试覆盖不足

## 改进项（Action Items）

| 编号 | 改进项 | 负责人 | 优先级 | 截止日期 | 状态 |
|------|--------|--------|--------|---------|------|
| AI-1 | 修复查询代码，避免对索引列做类型转换 | 李四 | P0 | 2024-03-19 | ✅ 完成 |
| AI-2 | 建立 SQL 变更审查清单，加入代码审查流程 | 张三 | P1 | 2024-03-25 | 进行中 |
| AI-3 | 预发环境导入生产级数据量（100 万+）| 王五 | P1 | 2024-04-01 | 待开始 |
| AI-4 | 接入 SQL 审核工具（如 Yearning/Archery）| 王五 | P2 | 2024-04-15 | 待开始 |
| AI-5 | 新增慢查询告警（> 500ms 自动告警）| 赵六 | P1 | 2024-03-22 | ✅ 完成 |

## 经验教训

### 做得好的
- 告警及时，从异常发生到工程师响应仅 2 分钟
- Jaeger 链路追踪帮助快速定位到 MySQL 层
- 回滚流程顺畅，15 分钟内完成

### 需要改进的
- 缺少 SQL 变更审查机制
- 预发环境数据量与生产差距太大，无法暴露性能问题
- 从发现到回滚用了 50 分钟，应该在确认回滚方向后更快执行
```

## 三、根因分析方法

### 方法 1：5 Whys（五个为什么）

最简单最实用的根因分析方法。对每个答案继续追问「为什么」，直到找到系统性的根因。

```
关键原则：
1. 每个 "Why" 的答案必须是事实，不是猜测
2. 不要停在人的层面（"因为他犯了错"），要追到系统/流程层面
3. 一般 3-5 层就能到达根因，超过 5 层可能在绕圈
4. 可能有多条因果链，每条都要追
```

**常见陷阱**：

```
❌ 停得太早：
   "为什么出问题？" → "因为工程师配置错了" → 停止
   （这只找到了人，没有找到系统问题）

✅ 继续追问：
   → "为什么能配置错？" → "因为没有配置校验"
   → "为什么没有配置校验？" → "因为从来没有建立配置变更流程"
   → 改进项：建立配置变更审批流程 + 自动校验
```

### 方法 2：鱼骨图（Ishikawa Diagram）

适合原因复杂、涉及多个领域的事故。按类别组织可能的原因：

```
                                        ┌─ 代码审查未覆盖 SQL
                         ┌── 人员 ──────┤
                         │              └─ DBA 未参与上线评审
                         │
                         │              ┌─ 无 SQL 审核工具
                         ├── 流程 ──────┤
                         │              └─ 上线无灰度发布
     商品查询延迟 ───────┤
                         │              ┌─ 索引列类型转换
                         ├── 代码 ──────┤
                         │              └─ 缺少慢查询检测
                         │
                         │              ┌─ 预发数据量不足
                         └── 环境 ──────┤
                                        └─ 缺少性能压测环节
```

**使用步骤**：

1. 确定问题（鱼头）
2. 画出主要类别（人员/流程/代码/环境/工具/外部因素）
3. 团队头脑风暴，将可能的原因归入各类别
4. 对每个原因评估是否为根因
5. 确定改进项

### 方法 3：故障树分析（FTA, Fault Tree Analysis）

适合分析严重事故，用逻辑门（AND/OR）组织因果关系：

```
                    商品查询超时
                         │
                    ─────┴───── OR
                   │              │
            SQL 全表扫描     连接池耗尽
                   │              │
              ─────┴────         ...
             │           │
        索引失效    统计信息过期
             │
        ─────┴───── AND
       │              │
  代码类型转换    无 SQL 审核
```

**何时用哪种方法**：

| 方法 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 5 Whys | 因果链清晰的简单事故 | 快速、易用 | 单一因果链，可能遗漏 |
| 鱼骨图 | 原因复杂、多因素交叉 | 全面、适合团队讨论 | 耗时较长 |
| 故障树 | 严重事故、需要精确分析 | 逻辑严密、可量化 | 学习成本高 |

## 四、改进项（Action Items）管理

### SMART 原则

每个改进项必须满足 SMART 原则：

| 原则 | 含义 | 反例 | 正例 |
|------|------|------|------|
| **S**pecific | 具体的 | "加强代码审查" | "建立 SQL 变更审查清单，包含索引失效 6 种场景" |
| **M**easurable | 可度量的 | "提高系统稳定性" | "慢查询数量从每天 50 条降至 5 条以下" |
| **A**chievable | 可实现的 | "消除所有 bug" | "接入 SQL 审核工具，覆盖 80% 的变更" |
| **R**elevant | 与根因相关的 | "升级服务器硬件" | "预发环境导入生产级数据量" |
| **T**ime-bound | 有截止日期的 | "以后再说" | "2024-04-01 前完成" |

### 改进项分级

```
P0（立即）: 直接修复导致事故的问题（如修复代码 bug）
   → 事故关闭前必须完成

P1（本周）: 防止同类问题再次发生的改进（如添加告警、审查流程）
   → 一周内完成

P2（本月）: 提升系统整体韧性的改进（如接入审核工具、改善测试环境）
   → 一个月内完成

P3（本季度）: 架构级改进（如灰度发布、自动回滚）
   → 纳入季度规划
```

### 改进项跟踪

改进项写了不跟踪 = 没写。建立跟踪机制：

```yaml
# 跟踪方式建议
跟踪工具: Jira / Linear / GitHub Issues（不要用文档跟踪，会被遗忘）

定期检查:
  - 每周站会花 2 分钟过一遍未完成的改进项
  - 每月复查一次所有事故的改进项完成率

度量指标:
  - 改进项完成率（目标 > 90%）
  - 平均完成时间（P0 < 1天，P1 < 7天）
  - 同类事故复发率（目标 0%）

升级机制:
  - 逾期 7 天未完成 → 自动通知负责人的 manager
  - 逾期 14 天 → 升级到技术总监
```

## 五、知识沉淀与分享

### 事故知识库

```
建立方式：
1. 所有复盘报告存入统一的知识库（Confluence / Notion / 内部 Wiki）
2. 打标签分类（按服务/按根因类型/按事故等级）
3. 建立搜索索引，让工程师能快速查找类似事故

目录结构示例：
postmortems/
├── 2024/
│   ├── INC-2024-042-商品查询超时.md
│   ├── INC-2024-041-Redis-OOM.md
│   └── ...
├── patterns/                  # 按模式归类
│   ├── 索引失效类.md
│   ├── 连接池耗尽类.md
│   ├── GC-暂停类.md
│   └── 超时传播类.md
└── checklists/                # 排查清单
    ├── 数据库慢查询排查清单.md
    ├── JVM-内存问题排查清单.md
    └── 网络超时排查清单.md
```

### 分享机制

```
团队内分享：
  - 每次 P0/P1 事故复盘后，在团队周会做 10 分钟分享
  - 重点讲：发生了什么、根因是什么、我们学到了什么

跨团队分享：
  - 每月一次「事故分享会」，各团队分享有价值的事故
  - 建立跨团队的事故订阅机制（如 Slack channel）

新人培训：
  - 将经典事故案例加入新人培训材料
  - 新人入职第一周阅读本团队的 Top 10 事故复盘
```

## 六、好的复盘 vs 坏的复盘

### 坏的复盘（反模式）

```
反模式 1：追责大会
  症状：花大量时间讨论「谁犯了错」
  后果：工程师不敢暴露问题，事故被隐瞒

反模式 2：表面功夫
  症状：时间线写得很详细，但改进项只有「加强监控」
  后果：同类事故一再发生

反模式 3：改进项黑洞
  症状：写了 10 条改进项，3 个月后一条没完成
  后果：团队对复盘失去信心，认为是浪费时间

反模式 4：只关注直接原因
  症状：修复了 bug 就结束，不追问为什么 bug 能上线
  后果：修了这个 bug，下个 bug 还是同样的方式上线

反模式 5：复盘延迟
  症状：事故发生一个月后才复盘
  后果：细节被遗忘，当事人记不清关键决策的原因
```

### 好的复盘特征

```
1. 时效性：事故恢复后 48 小时内完成复盘
2. 参与度：所有相关人都参加，而非只有值班工程师
3. 深度：追到系统/流程层面的根因，而非停在人的层面
4. 改进项质量：每条改进项都是 SMART 的
5. 跟踪闭环：改进项有人负责、有截止日期、有跟踪机制
6. 知识传播：复盘结果被分享给更大范围的团队
7. 持续改进：定期回顾改进项完成率和事故趋势
```

### 案例对比

**坏的复盘摘要**：

```
事故：Redis OOM 导致服务不可用
根因：数据量增长导致内存不足
改进项：
  1. 扩大 Redis 内存（完成）
  2. 加强监控（待定）
  3. 优化数据结构（待定）
```

**好的复盘摘要**：

```
事故：Redis OOM 导致商品查询服务不可用 30 分钟
根因：
  - 直接原因：商品详情缓存使用 String 类型存储完整 JSON（平均 15KB/key），
    在大促期间缓存命中率下降导致大量 key 被写入，触发 maxmemory 限制
  - 根本原因：缺少 Redis 内存容量预警，缺少大 Key 检测机制，
    缓存数据结构未经评审

改进项：
  1. [P0] 将商品详情缓存从 String 改为 Hash，预计内存降低 60%
     负责人：李四 | 截止：2024-03-20 | 状态：已完成
  2. [P1] 新增 Redis 内存使用率告警（> 70% 预警，> 85% 告警）
     负责人：王五 | 截止：2024-03-22 | 状态：已完成
  3. [P1] 建立大 Key 周扫描机制（redis-cli --bigkeys + 报告）
     负责人：王五 | 截止：2024-03-25 | 状态：进行中
  4. [P2] 引入缓存容量规划流程，大促前必须做缓存容量评估
     负责人：张三 | 截止：2024-04-01 | 状态：待开始
  5. [P2] 建立 Redis 数据结构评审机制（新增缓存需 DBA 评审）
     负责人：赵六 | 截止：2024-04-15 | 状态：待开始
```

## 七、复盘流程 Checklist

### 复盘前准备

```
□ 确定复盘时间（事故恢复后 24-48 小时内）
□ 邀请所有相关人（值班工程师、受影响团队、管理者）
□ 收集事故时间线（从告警、日志、沟通记录中整理）
□ 准备监控数据截图（Grafana、Jaeger、日志）
□ 指定复盘主持人（最好不是当事人）
```

### 复盘中

```
□ 主持人开场声明无责原则
□ 回顾事故时间线（当事人补充细节）
□ 逐段讨论：每个决策点当时为什么这么做？还有什么选择？
□ 讨论做得好的地方（不要只关注问题）
□ 进行根因分析（5 Whys 或鱼骨图）
□ 讨论并确定改进项（每项指定负责人和截止日期）
□ 确认改进项的优先级
```

### 复盘后

```
□ 24 小时内发出复盘报告
□ 改进项录入跟踪系统（Jira/Linear）
□ 在团队会议上分享复盘结果
□ 将复盘报告归档到知识库
□ 一周后检查 P0/P1 改进项进度
□ 一个月后检查所有改进项完成情况
```

## 八、度量与持续改进

### 事故相关指标

```yaml
事故指标:
  MTTD (Mean Time To Detect):     # 平均发现时间
    定义: 从事故发生到被发现的平均时间
    目标: < 5 分钟
    计算: avg(发现时间 - 发生时间)

  MTTR (Mean Time To Resolve):    # 平均恢复时间
    定义: 从事故发现到恢复的平均时间
    目标: P1 < 30 分钟, P2 < 2 小时
    计算: avg(恢复时间 - 发现时间)

  MTBF (Mean Time Between Failures):  # 平均故障间隔
    定义: 两次事故之间的平均时间
    目标: 持续增长
    计算: 总运行时间 / 事故次数

复盘指标:
  复盘完成率: 100%（所有 P0/P1 事故必须复盘）
  复盘时效: 48 小时内完成复盘
  改进项完成率: > 90%
  同类事故复发率: 0%
```

### 趋势分析

```
每季度做一次事故趋势分析：

1. 事故数量趋势（是否在减少？）
2. 事故类型分布（什么类型最多？是否有新类型？）
3. MTTD/MTTR 趋势（是否在改善？）
4. 改进项完成率趋势
5. 同类事故复发情况

将分析结果分享给管理层和团队，用数据证明复盘机制的价值。
```

## 九、PerfShop 复盘练习

### 练习场景

模拟一次 PerfShop 的 P1 事故，走完完整的复盘流程：

```
场景：PerfShop Java 版在压测期间，商品查询接口 P99 从 200ms
突增至 8s，持续 15 分钟后自动恢复。

你的任务：
1. 根据 Grafana 和 GC 日志数据，还原事故时间线
2. 使用 5 Whys 方法找到根因
3. 写一份完整的复盘报告（使用本文的模板）
4. 列出至少 5 条改进项，每条满足 SMART 原则
5. 设计一个跟踪机制确保改进项落实
```

### 评估标准

| 维度 | 优秀 | 合格 | 不合格 |
|------|------|------|--------|
| 时间线 | 精确到分钟，包含关键决策点 | 有主要事件，但遗漏部分细节 | 时间线模糊或不完整 |
| 根因分析 | 追到系统/流程层面，有多条因果链 | 找到直接技术原因 | 停在表面现象 |
| 改进项 | 全部 SMART，覆盖短期和长期 | 部分 SMART，以短期为主 | 笼统、不可执行 |
| 复盘报告 | 格式规范，信息完整 | 包含基本信息 | 缺少关键部分 |
