# 自我診斷 —— 你到底缺哪一格

> 「我好像缺了什麼」是最折磨人的焦慮,因為它**沒有邊界**。這份自測把模糊的焦慮變成一張具體的座標圖。
>
> 用法:對著下面 6 條原語 × 3 層的每一題,誠實問自己「現在能不能做到」,在 [§ 計分網格](#計分網格) 填 ✓ / ✗。**15 分鐘。** 「能算」的題附了折疊答案,先自己算再展開對。
>
> 先讀過 [`README.md`](./README.md) 的「6 條核心原語」會更好答,但不讀也行——答不出來本身就是有用的信號。

三層的意思:

| 層 | 問的是 | 為什麼分開 |
|---|---|---|
| **能講** | 你能用話把概念講清楚嗎 | 看懂 ≠ 會用,但這是底線 |
| **能算** | 給你數字,你能推出答案嗎 | 工程判斷要靠數字,不是名詞 |
| **能在真機上做** | 你能在一個真服務上把它做出來 / 量出來嗎 | **這一欄才是「真的懂系統」** |

---

## ① 延遲 = 百分位

- **能講**:為什麼**不能**把兩台機器各自的 P99 求平均,當成整體 P99?
- **能算**:某服務 avg=50ms、P50=40ms、P99=800ms、QPS=2000。**每分鐘有多少個請求慢於 800ms?**
  <details><summary>答案</summary>P99 表示最慢的 1%。2000 × 60 × 1% = <b>1200 個/分鐘</b>。「1% 聽起來小」在高 QPS 下是巨量的人。</details>
- **能在真機上做**:用壓測工具(wrk / locust / `../concurrency-capacity/lab/service/drive.py`)壓一個接口,從輸出**讀出 P50 與 P99**,並說出 P99 落在哪個量級。

## ② 吞吐 = 工作單位 / 時間

- **能講**:同一個用戶「下單」動作,為什麼用 TPS、RPS、QPS 數出來的數字不同?各自數的是哪一層?
- **能算**:一個業務事務平均打 6 條 SQL,系統跑 200 TPS。**DB 端的 QPS 大約多少?**
  <details><summary>答案</summary>200 × 6 = <b>1200 QPS</b>。同一筆流量,TPS / RPS / QPS 是不同焦段的鏡頭,不矛盾。</details>
- **能在真機上做**:壓測時同時讀出**目標 RPS vs 實際 QPS**,並判斷系統是否已撞到吞吐天花板(實際 qps 不再跟著目標漲)。

## ③ RED(對流量)

- **能講**:RED 三個信號是什麼?它們合起來回答「這個服務健不健康」的哪三問?
- **能算**:某接口一分鐘 60000 個請求,其中 600 個回 5xx。**錯誤率多少?**
  <details><summary>答案</summary>600 / 60000 = <b>1%</b>。錯誤率要看比例不是絕對數,且通常排除 4xx(那是客戶端的錯)。</details>
- **能在真機上做**:在 Grafana / PromQL(或本 lab 的 `/stats`)上,**同時調出** Rate、Errors、Duration 三條線。

## ④ USE(對資源)

- **能講**:USE 三個是什麼?為什麼**飽和度(saturation)往往比利用率更早報警**?
- **能算**:一個有 8 個並發槽的池,每個請求佔住槽 50ms。**這個池的吞吐天花板是多少 rps?**
  <details><summary>答案</summary>8 / 0.05 = <b>160 rps</b>。這正是 `FIRST-LOOP` 第二幕 POOL_SIZE=8 壓出來的天花板。</details>
- **能在真機上做**:對一個資源(CPU 或連接池),**同時讀出利用率與飽和度**(隊列深度 / 等待中的請求數 / `max_in_flight`)。

## ⑤ Little 定律 + 飽和曲線

- **能講**:為什麼利用率逼近 100% 時,延遲是**暴漲(曲棍球桿)**而不是線性上升?
- **能算**:QPS=2000、平均延遲 50ms。**系統內同時在途的請求約多少個?**
  <details><summary>答案</summary>L = λ×W = 2000 × 0.05 = <b>100 個在途</b>。這就是你定線程 / 連接池大小的起點。</details>
- **能在真機上做**:跑 `../concurrency-capacity/lab/sim/little.py`(或壓一個真服務),**親眼看 P99 在拐點爆炸**,並指出膝點大概在哪個利用率。

## ⑥ SLI / SLO + error budget

- **能講**:SLI / SLO / SLA 三者差在哪?error budget 是什麼,拿來做什麼決策?
- **能算**:SLO 訂「可用率 99.9%」。**一個月(30 天)的 error budget 是多少分鐘?**
  <details><summary>答案</summary>30×24×60 × 0.1% = 43200 × 0.001 ≈ <b>43 分鐘/月</b>。預算花得起就拿去換發版速度,花超就凍結上線。</details>
- **能在真機上做**:給一個你**做過**的服務,寫出一條可量測的 SLO(挑一個 SLI + 一個目標 + 一個時間窗)。

## 怎麼讀你的網格

**最可能的結果**:左邊兩欄(能講 / 能算)大半是 ✓,**最右欄(能在真機上做)大半是 ✗。**

如果是這樣——**這正是好消息,也是這份診斷的重點**:

> 你的焦慮不該是「我什麼都不懂」,而是「**我懂很多,只是從沒親手做過**」。差距不在知識,在手感。手感只有一種補法:做一次。

按你的網格決定下一步:

| 你的網格長相 | 下一步 |
|---|---|
| 右欄(真機)大量 ✗,左兩欄還行 | 去 [`FIRST-LOOP.md`](./FIRST-LOOP.md) —— 一次就把右欄點亮好幾格 |
| 某條原語**整行**空 | 去 [`README.md`](./README.md) 那條原語的「深礦」連結補概念 |
| 「能算」普遍空 | 重點補 ⑤①②④ 的公式(都在 README §3,各一行就記得住) |
| 三欄都挺滿 | 你其實不缺——去 [`SENIOR-RUBRIC.md`](./SENIOR-RUBRIC.md) 對資深表達標準,或直接刷 [`INTERVIEW-MATRIX.md`](./INTERVIEW-MATRIX.md) |
