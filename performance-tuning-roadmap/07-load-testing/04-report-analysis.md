# 性能报告解读

## 为什么要会读性能报告

跑完压测只是第一步。真正的价值在于**从数据中提取洞察**——系统的瓶颈在哪？能承载多少流量？应该优化什么？不会读报告的人只会说"QPS 是 5000"，会读报告的人能说"在 3000 QPS 时 P99 从 50ms 陡增到 500ms，瓶颈是数据库连接池，扩到 50 后 P99 回到 80ms"。

---

## 一、百分位数详解

### 什么是百分位数

百分位数（Percentile）表示有多少比例的请求在这个延迟以内完成。

```
P50 = 20ms   → 50% 的请求在 20ms 内完成（中位数）
P95 = 80ms   → 95% 的请求在 80ms 内完成
P99 = 200ms  → 99% 的请求在 200ms 内完成
P999 = 1.2s  → 99.9% 的请求在 1.2s 内完成
```

### 为什么不看平均值而看 P99

```
场景：1000 个请求
- 990 个请求：10ms
- 10 个请求：5000ms（超时）

平均值 = (990 × 10 + 10 × 5000) / 1000 = 59.9ms  ← 看起来还行
P50    = 10ms     ← 大多数人体验很好
P99    = 5000ms   ← 每 100 个用户就有 1 个等 5 秒

平均值掩盖了长尾延迟。
```

**为什么 P99 比平均值重要**：

1. 一个日活百万的服务，P99 = 5s 意味着每天有 **1 万次请求** 超过 5 秒
2. 一个用户的一次页面加载可能触发 10 个后端请求，只要其中 1 个慢，整个页面就慢
3. 微服务架构中，一个请求链路有 5 个服务，P99 会累积放大

```
链路放大效应：
单服务 P99 = 100ms
5 个串行服务的 P99 ≈ 远大于 100ms

如果每个服务的延迟独立：
P(全部 < 100ms) = 0.99^5 = 0.951
即链路 P99 实际上只有 ~P95 的水平
```

### 该看哪个百分位

| 百分位 | 适用场景 |
|-------|---------|
| P50 | 了解"典型"用户体验 |
| P95 | SLO 常用指标 |
| P99 | 后端服务常用 SLO |
| P999 | 支付等关键路径、金融系统 |

## 二、吞吐量 vs 延迟的权衡

### 经典的 QPS-延迟曲线

```
延迟
  ^
  │                          ╱
  │                        ╱
  │                      ╱     ← 延迟急剧上升
  │                    ╱
  │               ···╱
  │          ····
  │     ····           ← 线性增长区间（系统健康）
  │ ···
  └─────────────────────────── QPS
                    ↑
               拐点 (Knee Point)
```

### 各阶段特征

| 区间 | QPS 变化 | 延迟变化 | 系统状态 |
|------|---------|---------|---------|
| 低负载 | 线性增长 | 基本不变 | 资源充裕 |
| 中负载 | 增长放缓 | 缓慢增加 | 接近饱和 |
| 拐点 | 增长停滞 | 急剧上升 | 某资源饱和 |
| 过载 | 开始下降 | 极高/超时 | 排队严重 |

**拐点识别方法**：

1. 画 QPS vs P99 延迟的散点图
2. 找到延迟斜率突然变大的点
3. 那个点对应的 QPS 就是系统的实际容量
4. 生产环境应该运行在拐点的 **60-70%** 以下

## 三、协调遗漏问题（Coordinated Omission）

### 什么是协调遗漏

这是性能测试中最隐蔽也最重要的问题之一，由 Gil Tene 提出。

```
正常情况（期望每 10ms 发一个请求）：
时间:  0    10   20   30   40   50   60
请求:  R1   R2   R3   R4   R5   R6   R7
延迟:  5ms  8ms  6ms  7ms  5ms  6ms  7ms

出现慢请求时（R3 花了 50ms）：
时间:  0    10   20   ............  70   80
请求:  R1   R2   R3                 R4   R5
延迟:  5ms  8ms  50ms              6ms  5ms

问题：R3 花了 50ms，但 R4、R5 根本没发出去！
      本该在 30ms、40ms、50ms、60ms 发出的请求都"消失"了
      这些"消失"的请求没被统计，延迟被严重低估
```

### 核心矛盾

大多数压测工具（wrk、ab、JMeter 默认模式）使用"闭环"模型：发一个请求，等响应回来，再发下一个。当服务变慢时，请求发送速率自动降低，那些"本该发出但没发出"的请求的等待时间没有被统计。

**真实用户不会因为服务慢就不来了**——他们还在那里等着。

### 修正方案

| 工具 | 是否修正 | 方式 |
|------|---------|------|
| wrk | 不修正 | 闭环模型 |
| wrk2 | 修正 | 恒定速率 + 延迟修正 |
| Vegeta | 修正 | 恒定请求速率 |
| k6 | 部分修正 | constant-arrival-rate executor |
| hey | 不修正 | 闭环模型 |

```bash
# wrk2：恒定 2000 QPS
wrk2 -t4 -c100 -d60s -R2000 http://localhost:8080/api

# k6 恒定到达率
export const options = {
  scenarios: {
    constant_rate: {
      executor: 'constant-arrival-rate',
      rate: 1000,           // 每秒 1000 个请求
      timeUnit: '1s',
      duration: '1m',
      preAllocatedVUs: 50,
      maxVUs: 200,
    },
  },
};

# Vegeta 天然恒定速率
echo "GET http://localhost:8080/api" | vegeta attack -rate=1000/s -duration=60s
```

## 四、HDR Histogram

### 为什么需要 HDR Histogram

普通直方图用固定大小的桶（如每 1ms 一个桶），在高延迟范围精度不够。HDR Histogram（High Dynamic Range Histogram）使用对数分桶，能在 1 微秒到 1 小时的范围内保持恒定的精度。

### 关键特性

```
传统直方图桶划分：
[0-1ms] [1-2ms] [2-3ms] ... [999-1000ms]
→ 1000 个桶，但 0-1ms 和 999-1000ms 精度一样（不合理）

HDR Histogram 桶划分（对数）：
[0-1µs] [1-2µs] ... [1-2ms] [2-4ms] ... [1-2s] [2-4s]
→ 在每个量级内保持一定精度，总桶数可控
→ 精确记录从微秒到秒级的延迟分布
```

### 使用 HDR Histogram

```bash
# Vegeta 输出 HDR 格式
echo "GET http://localhost:8080/api" | \
  vegeta attack -rate=500/s -duration=30s | \
  vegeta report -type=hdrplot

# 输出可以上传到 http://hdrhistogram.org/ 在线分析
# 或使用本地工具解析

# Java 中使用 HdrHistogram 库
# <dependency>
#   <groupId>org.HdrHistogram</groupId>
#   <artifactId>HdrHistogram</artifactId>
# </dependency>
```

## 五、瓶颈识别方法

压测结果告诉你"有问题"，但你需要进一步判断**瓶颈在哪里**。

### 四种瓶颈类型及判断依据

```
┌──────────────┬─────────────────────────────────────────┐
│  瓶颈类型     │  判断依据                                │
├──────────────┼─────────────────────────────────────────┤
│  CPU bound   │  CPU 使用率 > 80%                        │
│              │  us% 高（用户态计算密集）                    │
│              │  sy% 高（系统调用频繁/上下文切换多）           │
│              │  load average 高于 CPU 核数                │
├──────────────┼─────────────────────────────────────────┤
│  IO bound    │  iowait (wa%) 高                         │
│              │  iostat %util 接近 100%                    │
│              │  CPU 使用率不高但 QPS 上不去                 │
│              │  await 高（IO 请求排队）                    │
├──────────────┼─────────────────────────────────────────┤
│  Memory bound│  频繁 GC（GC 时间占比高）                   │
│              │  swap 使用量增加                           │
│              │  RSS 持续增长                              │
│              │  Page fault 频繁                          │
├──────────────┼─────────────────────────────────────────┤
│  Network     │  网络带宽打满                              │
│  bound       │  TCP 重传率高                              │
│              │  连接数到达上限                             │
│              │  ss -s 显示大量 TIME_WAIT                  │
└──────────────┴─────────────────────────────────────────┘
```

### 快速排查命令

```bash
# CPU 状态
top -bn1 | head -5        # 看 %us %sy %wa
mpstat -P ALL 1 3         # 每核 CPU 使用率

# 内存状态
free -h                   # 总体内存使用
vmstat 1 5                # si/so 看 swap 活动

# IO 状态
iostat -xz 1 5            # 磁盘 IO 统计
iotop -obn1               # 按进程看 IO

# 网络状态
sar -n DEV 1 3            # 网络流量
ss -s                     # 连接统计
netstat -s | grep -i retrans  # 重传统计

# Java 特有
jstat -gcutil <pid> 1000  # GC 统计，每秒刷新
```

## 六、性能报告模板

一份专业的性能报告应包含以下部分：

```markdown
# 性能测试报告

## 1. 测试目标
- 验证 v2.1.0 版本在 5000 QPS 下 P99 < 200ms
- 确认系统能承受预期大促流量（10000 QPS）

## 2. 测试环境
- 服务配置：4C8G × 3 实例，JDK 17，-Xmx4g
- 数据库：MySQL 8.0, 8C32G, 数据量 500 万行
- Redis：6.2, 4C8G, 集群模式 3 主 3 从
- 网络：压测机与服务同机房，RTT < 1ms

## 3. 测试方法
- 工具：k6 v0.47
- 场景：混合场景（查询 60% / 写入 30% / 复杂查询 10%）
- 加压方式：阶梯加压，每阶段 2 分钟
  - 1000 → 2000 → 3000 → 5000 → 8000 → 10000 QPS

## 4. 测试结果

| QPS  | P50   | P95    | P99    | 错误率  | CPU  | 内存  |
|------|-------|--------|--------|--------|------|------|
| 1000 | 8ms   | 15ms   | 25ms   | 0%     | 20%  | 45%  |
| 3000 | 12ms  | 35ms   | 60ms   | 0%     | 55%  | 52%  |
| 5000 | 18ms  | 80ms   | 150ms  | 0.1%   | 78%  | 58%  |
| 8000 | 45ms  | 250ms  | 800ms  | 2.5%   | 95%  | 72%  |
| 10000| 200ms | 2000ms | 5000ms | 15%    | 99%  | 85%  |

## 5. 分析
- **拐点**：5000 QPS，P99 = 150ms，系统仍在健康范围
- **瓶颈**：CPU bound，8000 QPS 时 CPU 达到 95%
- **数据库**：慢查询在 8000 QPS 时开始出现
- **内存**：浸泡 4 小时无泄漏

## 6. 结论与建议
- v2.1.0 满足 5000 QPS / P99 < 200ms 的目标 ✓
- 大促 10000 QPS 需要扩容到 6 实例（当前 3 实例）
- 建议优化 /api/search 接口的数据库查询（慢查询源头）
- 建议开启连接池预热（冷启动时前 10 秒错误率偏高）
```

## 总结

性能报告解读的核心能力：

1. **看 P99 不看平均值**——平均值会欺骗你
2. **找拐点**——那是系统真实容量的边界
3. **警惕协调遗漏**——闭环压测的延迟数据天然偏低
4. **用 HDR Histogram**——精确记录长尾延迟
5. **判断瓶颈类型**——CPU/IO/内存/网络，对症下药
6. **写结构化报告**——让压测结果转化为可执行的决策
