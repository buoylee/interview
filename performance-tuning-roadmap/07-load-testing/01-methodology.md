# 压测方法论

## 为什么需要系统化的压测方法论

压测不是"起个工具打一打"这么简单。没有明确目标的压测等于浪费时间——你不知道在测什么、不知道结果说明什么、更不知道该怎么做决策。系统化的方法论确保每次压测都能产出可执行的结论。

---

## 一、压测目标定义：你到底要测什么

在启动压测之前，必须明确回答一个问题：**这次压测的目标是什么？**

常见的四类目标：

| 目标类型 | 问题 | 产出 |
|---------|------|------|
| 最大吞吐量 | 系统最多能处理多少 QPS？ | 峰值 QPS 数值 |
| 延迟基线 | 正常负载下 P99 是多少？ | 各百分位延迟数据 |
| 容量上限 | 需要多少实例才能扛住预期流量？ | 实例数 / 资源规格建议 |
| 稳定性 | 长时间运行会不会出问题？ | 内存/连接/线程泄漏报告 |

**反面案例**：团队说"压一下看看"，跑了 10 分钟 wrk，得到一个 QPS 数字就完事了。没人知道这个数字是在什么条件下得到的，也不知道离系统极限有多远。

## 二、五种压测类型详解

### 1. 基线测试（Baseline Test）

**目的**：在低并发、无压力的情况下，测量系统的"正常表现"。

```
负载模式：
并发 ──┐
       │  ████████████████████████
       │  ████████████████████████   ← 固定低负载（如 10 并发）
       └──────────────────────────── 时间
```

- 并发数：5-10
- 持续时间：5-10 分钟
- 关注指标：P50/P95/P99 延迟、错误率
- 意义：建立性能基准线，后续所有优化都跟这个对比

### 2. 阶梯加压测试（Step Load / Ramp-up Test）

**目的**：逐步增加并发，找到系统性能拐点。

```
负载模式：
并发 ──┐
       │              ████████
       │         █████████████
       │    ██████████████████   ← 每分钟增加 50 并发
       │  ████████████████████
       └──────────────────────── 时间
                         ↑
                    拐点在这里
```

- 每阶段持续：1-3 分钟（让系统达到稳态）
- 每阶段增加：根据预期容量的 10-20%
- 关注指标：QPS 不再线性增长的点、延迟开始陡增的点
- **拐点识别**：当 QPS 增长放缓但延迟开始急剧上升时，就是系统的性能拐点

### 3. 浸泡测试（Soak Test / Endurance Test）

**目的**：在中等负载下长时间运行，检测资源泄漏。

- 负载：系统容量的 60-80%
- 持续时间：4-24 小时
- 关注指标：内存使用趋势、GC 频率和耗时、连接数变化、线程数变化
- 常见发现：内存泄漏（缓慢增长直到 OOM）、连接泄漏、文件描述符泄漏

### 4. 尖峰测试（Spike Test）

**目的**：模拟突发流量，验证系统的弹性和恢复能力。

```
负载模式：
并发 ──┐
       │     ██
       │     ████
       │     ██████      ← 突然 10x 流量
       │  ████████████
       │  ████████████████
       └────────────────── 时间
```

- 流量突增倍数：5-10 倍
- 持续时间：1-5 分钟
- 关注指标：错误率、恢复时间、自动扩容是否生效
- 典型场景：秒杀、热点事件、定时任务集中触发

### 5. 混合场景测试（Scenario Test）

**目的**：模拟真实用户行为比例，最接近生产的压测方式。

```
真实流量比例模型：
┌──────────────────────────┐
│  浏览商品列表    50%      │
│  查看商品详情    30%      │
│  加入购物车      12%      │
│  下单支付         5%      │
│  搜索商品         3%      │
└──────────────────────────┘
```

- 每个接口按生产实际比例施压
- 用户行为有先后顺序（先浏览再下单）
- 需要维护测试数据（用户账号、商品库存等）

## 三、压测环境要求

### 环境隔离

压测环境**必须与生产环境隔离**，同时在硬件配置上尽可能对齐。

| 维度 | 要求 | 常见问题 |
|------|------|---------|
| 机器规格 | 与生产保持一致或按比例缩放 | 压测环境 2C4G，生产 8C16G，结果完全不可参考 |
| 网络拓扑 | 压测客户端和服务端之间的网络延迟要已知 | 跨机房压测，网络延迟 50ms，把网络延迟算进了服务延迟 |
| 中间件版本 | 与生产一致 | 压测环境 MySQL 5.7，生产 MySQL 8.0 |
| 数据量级 | 表数据量级与生产对齐 | 空表压测 QPS 很高，生产数据量大后性能完全不同 |

### 数据准备

```bash
# 典型数据准备步骤
# 1. 从生产脱敏导出数据
mysqldump --where="1 LIMIT 1000000" prod_db users | \
  sed 's/[0-9]\{11\}/13800138000/g' > test_data.sql

# 2. 或使用数据生成工具
# 确保数据量级、分布、索引与生产一致
```

关键原则：**数据量级决定查询性能**。1 万行数据和 1 亿行数据的 SQL 性能天差地别。

## 四、容量模型验证

压测的核心产出之一是**容量模型**：生产需要多少资源来承载预期流量。

### 推算公式

```
单实例能力（压测得出）：
  - 4C8G 单实例：处理 2000 QPS，P99 < 50ms

生产需求：
  - 预期峰值：10000 QPS
  - 安全系数：1.5（预留 50% 余量）

所需实例数 = (10000 × 1.5) / 2000 = 7.5 → 8 台
```

### 验证方法

1. 先单实例压测，得到单机容量
2. 再多实例压测，验证线性扩展性（实际很少完美线性，通常 N 台机器只能达到 N × 80% 的性能）
3. 考虑安全系数：日常 1.3-1.5 倍、大促 2-3 倍

## 五、压测流程 Checklist

完整的一次压测应覆盖以下流程：

### 压测前

- [ ] 明确压测目标（测什么、期望值是什么）
- [ ] 确认压测环境就绪（机器、网络、中间件、数据）
- [ ] 确认压测数据准备完毕且数据量级合理
- [ ] 确认监控系统就绪（能看到 CPU/内存/IO/网络/应用指标）
- [ ] 确认压测不会影响其他环境（特别是生产）
- [ ] 确认压测工具客户端不会成为瓶颈
- [ ] 通知相关方（DBA、运维、相关团队）

### 压测中

- [ ] 先跑基线测试，确认系统正常
- [ ] 按计划逐步加压，记录每个阶段的指标
- [ ] 实时观察监控面板（不要光盯 QPS 数字）
- [ ] 出现异常及时记录现象和时间点
- [ ] 每个阶段保持足够时间让系统达到稳态（至少 1 分钟）

### 压测后

- [ ] 整理原始数据和监控截图
- [ ] 分析瓶颈点和拐点
- [ ] 输出结构化的压测报告
- [ ] 给出容量建议和优化方向
- [ ] 归档压测脚本和配置（下次可复现）

## 六、常见压测误区

1. **只测单接口不测链路**：生产中一个用户请求可能触发 5-6 个内部调用，单接口 QPS 高不代表整体没问题。

2. **客户端成为瓶颈**：一台 2C4G 的压测机跑 Locust，自身 CPU 打满了，得到的 QPS 其实是客户端极限而非服务端极限。解决方法：分布式压测或用资源消耗更低的工具（如 wrk）。

3. **不看错误率**：QPS 很高但有 10% 的 500 错误——这不叫"抗住了"。**有效 QPS = 总 QPS × (1 - 错误率)**。

4. **预热不充分**：JVM 应用需要 JIT 编译预热，前几分钟的性能不代表稳态性能。连接池也需要预热。

5. **忽略依赖方**：服务本身没问题，但下游数据库连接池满了、Redis 达到带宽上限——这些都要一起观察。

---

## 总结

压测方法论的核心是三个字：**目标、控制、分析**。

- **目标明确**：知道测什么，才知道怎么测
- **变量控制**：环境对齐、数据对齐、负载模型对齐
- **结果分析**：不只看 QPS，要看延迟分布、错误率、资源利用率的全貌

没有方法论指导的压测，只是在制造噪音数据。
