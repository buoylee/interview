# RED 方法与黄金指标

## RED 方法

RED 方法由 Tom Wilkie（Grafana Labs）提出，专门针对**微服务和请求驱动型系统**的性能监控。与 USE 方法关注底层资源不同，RED 方法关注的是用户可感知的服务质量。

三个核心指标：

- **R**ate（速率）：每秒处理的请求数（QPS/RPS）
- **E**rrors（错误）：每秒失败的请求数，或错误率
- **D**uration（耗时）：请求的延迟分布（P50/P95/P99）

```
                   ┌─────────────────────────────────┐
                   │         用户请求                  │
                   └────────────┬────────────────────┘
                                │
                                ▼
               ┌────────────────────────────────┐
               │          你的服务               │
               │                                │
               │  Rate:     1200 req/s          │
               │  Errors:   12 req/s (1.0%)     │
               │  Duration: P50=45ms P99=320ms  │
               └────────────────────────────────┘
```

### Rate（速率）

速率反映系统的负载水平和业务流量。

关注点：
- **趋势变化**：QPS 突然下降可能说明上游出问题了（用户访问不进来），而不是你的系统变好了
- **与基线对比**：工作日 vs 周末、白天 vs 夜间、促销期 vs 平时
- **流量分布**：各接口的请求占比，找出热点接口

```bash
# 用日志快速统计 QPS（按秒聚合）
$ awk '{print $4}' access.log | cut -d: -f1-3 | sort | uniq -c | tail -5
   1156 15/Jan/2024:10:30:01
   1243 15/Jan/2024:10:30:02
   1189 15/Jan/2024:10:30:03
   2847 15/Jan/2024:10:30:04   ← 流量突增
   2934 15/Jan/2024:10:30:05
```

### Errors（错误）

错误率是最直接反映服务健康度的指标。

需要区分的错误类型：
- **显式错误**：HTTP 5xx、gRPC 错误码
- **隐式错误**：HTTP 200 但业务逻辑返回错误（需要应用层埋点）
- **超时**：请求未在规定时间内完成（很多监控系统不把超时算作错误，但它对用户来说就是失败）

```bash
# 统计 HTTP 状态码分布
$ awk '{print $9}' access.log | sort | uniq -c | sort -rn
  45231  200
   3421  304
    892  499   ← 客户端主动断开（通常是超时）
    234  502   ← 上游服务不可用
    156  503   ← 服务过载
     43  500   ← 内部错误
```

499 (client closed) 经常被忽略，但大量 499 通常意味着你的服务响应太慢，客户端等不及了。

### Duration（耗时）

延迟分布是 RED 方法中最重要的指标。

**永远不要只看平均值。** 平均值会掩盖真实的用户体验：

```
场景 A：1000 个请求，全部 100ms          → 平均 100ms
场景 B：990 个请求 50ms + 10 个请求 5000ms → 平均 99.5ms

平均值几乎相同，但场景 B 有 1% 的用户等了 5 秒。
```

**正确做法**：看分位数。

```
P50  = 45ms    ← 一半用户的体验
P95  = 180ms   ← 95% 用户的体验
P99  = 850ms   ← 99% 用户的体验
P999 = 3200ms  ← 最差的 0.1% 用户的体验
```

实际中，关注 P99 就够了。如果你的服务每秒 10000 请求，P99 = 850ms 意味着每秒有 100 个请求超过 850ms。

## Google SRE 四大黄金指标

Google SRE 团队在《Site Reliability Engineering》一书中提出了四大黄金指标（Four Golden Signals）：

```
┌──────────────────────────────────────────────┐
│              四大黄金指标                       │
├──────────┬──────────┬──────────┬──────────────┤
│  延迟     │   流量   │   错误   │    饱和度     │
│ Latency  │ Traffic  │ Errors  │  Saturation   │
│          │          │         │              │
│ 请求耗时  │ 请求速率  │ 错误率   │ 资源使用程度  │
│ (分位数)  │ (QPS)   │ (5xx率) │ (队列长度等)  │
└──────────┴──────────┴──────────┴──────────────┘
```

### Latency（延迟）

与 RED 的 Duration 对应。Google 特别强调要**区分成功请求和失败请求的延迟**：

- 一个返回 500 的请求可能只用了 5ms（快速失败），如果把它混在延迟统计里会拉低整体延迟，让你以为性能变好了
- 正确做法：成功请求和失败请求的延迟分开统计

### Traffic（流量）

与 RED 的 Rate 对应。具体指标取决于服务类型：
- Web 服务：HTTP 请求数/秒
- 流式系统：每秒处理的消息数
- 存储系统：每秒读写操作数（IOPS）
- CDN：每秒传输的字节数

### Errors（错误）

与 RED 的 Errors 对应。Google 补充了一层："即使状态码是 200，如果返回了错误的内容，也算错误"。

### Saturation（饱和度）

这是四大黄金指标相比 RED 多出的维度。

饱和度关注的是"服务还有多少余量"：
- CPU 利用率、内存利用率
- 线程池使用率、连接池使用率
- 队列长度、排队等待时间

饱和度是一个**预警指标**——在服务崩溃之前，饱和度会先升高。如果你只看 Rate/Errors/Duration，可能直到服务崩溃才发现问题。

## USE vs RED 对比

```
┌─────────┬────────────────────────┬─────────────────────────┐
│         │       USE 方法          │       RED 方法           │
├─────────┼────────────────────────┼─────────────────────────┤
│ 关注对象 │ 基础设施资源            │ 用户可见的服务             │
│         │ (CPU/内存/磁盘/网络)    │ (API/微服务)             │
│ 视角     │ 自底向上（从资源看）     │ 自顶向下（从请求看）      │
│ 回答问题 │ "哪个资源是瓶颈？"      │ "服务表现如何？"          │
│ 适用角色 │ 运维/SRE/基础设施团队   │ 开发/SRE/产品团队        │
│ 适用阶段 │ 资源级排查/容量规划     │ 服务监控/告警/SLO 定义    │
└─────────┴────────────────────────┴─────────────────────────┘
```

### 什么时候用 USE

- "服务器变慢了" → USE 逐一排查 CPU、内存、磁盘、网络
- 容量规划 → 用 USE 找到最先到达上限的资源
- 新机器上线前的健康检查

### 什么时候用 RED

- "用户反馈某个功能慢了" → 用 RED 看这个服务的延迟分布
- 设置监控告警 → 基于 RED 指标定义 SLO（如 P99 < 500ms）
- 版本发布后的性能回归检测

### 实际中两者结合使用

通常的排查路径是：

```
RED 发现问题 → USE 定位根因

例：RED 告诉你 "订单服务 P99 从 200ms 涨到 2s"
    USE 告诉你 "订单服务所在机器的磁盘 %util = 99%"
    进一步排查发现是日志写入量太大导致磁盘饱和
```

## 实际案例：用 RED 指标发现 API 性能退化

### 背景

一个电商平台，周三下午 3 点发布了新版本。周四上午产品经理反馈："搜索功能好像变慢了，但又说不出来哪里慢。"

### 用 RED 指标定量分析

**Rate 对比：**
```
搜索 API QPS:
  周二（发布前）: 平均 850 req/s
  周四（发布后）: 平均 820 req/s  → 基本持平，流量没变
```

**Errors 对比：**
```
搜索 API 错误率:
  周二: 0.02%
  周四: 0.03%  → 基本正常
```

**Duration 对比（关键发现）：**
```
搜索 API 延迟:
          周二        周四
P50:     35ms        38ms      ← 差别不大
P95:     120ms       450ms     ← 明显退化!
P99:     280ms       2100ms    ← 严重退化!
```

P50 几乎没变，这就是为什么"大部分时候感觉还行"。但 P95 和 P99 严重退化，说明有一部分请求变得很慢。

### 进一步分析

按请求参数拆分延迟：

```
搜索 API 延迟 (周四), 按搜索类型:
                    P50      P99
简单关键词搜索:     36ms     150ms    ← 正常
带筛选条件搜索:     42ms     180ms    ← 正常
带排序的搜索:      380ms    2800ms    ← 问题在这里!
```

定位到"带排序的搜索"是问题所在。查看代码变更：

```diff
- // 旧代码：数据库排序
- SELECT * FROM products WHERE ... ORDER BY price LIMIT 20

+ // 新代码：应用层排序（"为了支持更灵活的排序逻辑"）
+ results = db.query("SELECT * FROM products WHERE ...")
+ results.sort(key=lambda x: x.price)
+ return results[:20]
```

新版本把排序从数据库移到了应用层，但没有 LIMIT 下推，导致查询返回全量数据再在内存中排序。数据量大的分类（如"服装"有 50 万条）就会特别慢。

### 修复

恢复数据库排序，或者在数据库层做分页。P99 恢复到 300ms 以内。

### 复盘

如果不看 RED 指标，可能的情况是：
- 产品经理说"好像慢了" → 开发回复"我试了没问题啊"（因为 P50 确实没变）
- 问题持续到有大量用户投诉才被重视

RED 指标的价值在于把模糊的"感觉慢了"变成精确的"P99 从 280ms 退化到 2100ms，具体是带排序的搜索接口"。

## 小结

| 方法 | 核心指标 | 适用场景 |
|------|----------|----------|
| USE | Utilization / Saturation / Errors | 基础设施排查、容量规划 |
| RED | Rate / Errors / Duration | 服务监控、API 性能分析 |
| 黄金指标 | Latency / Traffic / Errors / Saturation | SRE 全面监控、SLO 定义 |

实际工作中，建议每个服务至少暴露 RED 指标，每台机器至少采集 USE 指标。两者结合，从服务层发现问题，到资源层定位根因。
