## 本阶段心智模型

阶段 1 要把排查流程固定成下面的顺序：

```text
1. 先描述现象：谁慢了？慢多少？什么时候开始？
2. 再看服务指标：QPS、错误率、P95/P99 延迟是否异常？
3. 再看资源指标：CPU、内存、磁盘、网络是否利用率高、饱和或报错？
4. 提出可证伪假设：瓶颈可能在哪里，为什么？
5. 采集证据：用指标、日志、Trace、profile 验证或推翻假设。
6. 只改一个变量：修复后用同一套指标复测。
```

这个阶段不要急着掌握所有工具细节。工具会在阶段 2 以后系统学习。

## RED 与 USE 的分工

| 方法 | 视角 | 先回答的问题 | 常用场景 |
|------|------|--------------|----------|
| RED | 服务/用户 | 请求有没有变慢、变错、变少或变多？ | API 慢、错误率升高、用户投诉 |
| USE | 资源/机器 | 哪个资源忙、排队或出错？ | 服务器卡顿、CPU 高、I/O 慢、网络异常 |

推荐起手式：

```text
用户说接口慢 → 先 RED → 再 USE → 再选择具体工具
机器告警资源高 → 先 USE → 再回到 RED 判断用户影响
```

## 本阶段练习

选择一个正在运行的服务，完成下面记录：

```text
问题描述：
影响范围：
开始时间：

RED：
- Rate：
- Errors：
- Duration：

USE：
- CPU Utilization / Saturation / Errors：
- Memory Utilization / Saturation / Errors：
- Disk Utilization / Saturation / Errors：
- Network Utilization / Saturation / Errors：

当前假设：
需要采集的下一份证据：
```

不要追求一次填得完美。关键是建立“先写清楚，再查证据”的习惯。

## 完成标准

学完阶段 1 后，应该能做到：

- 能解释为什么“先改代码再看指标”是错误路径
- 能写出一个可证伪的性能假设
- 能区分 RED 和 USE 的适用场景
- 能用 Little 定律粗略估算并发、吞吐和延迟关系
- 能写出一份最小排查记录，而不是只说“可能是数据库慢”

---

## 常见误区

| 误区 | 正确做法 |
|------|----------|
| 看到 CPU 高就开始优化代码 | 先判断 CPU 是利用率高还是已经饱和，再 profile |
| 只看平均响应时间 | 至少看 P95/P99 和错误率 |
| 一次改多个参数 | 每次只改一个变量，否则无法归因 |
| 找到一个异常就认定根因 | 检查异常是否与用户影响在时间线上吻合 |
| 只看服务内部指标 | 同时看依赖服务、数据库、中间件和主机资源 |

## 下一阶段衔接

阶段 1 解决“怎么想”。阶段 2 解决“怎么看”。

进入阶段 2 后，你会把本阶段的 RED/USE 清单落到具体命令：

- `top` / `mpstat` / `pidstat` 看 CPU
- `free` / `vmstat` / `pmap` 看内存
- `iostat` / `iotop` 看磁盘
- `ss` / `tcpdump` / `ip -s link` 看网络
