# 性能定律

性能优化不只是调参数和改代码，它背后有严格的数学规律。理解这些定律能帮你：判断优化的理论上限、做容量规划、避免"加机器就能解决"的思维陷阱。

## Amdahl 定律（并行加速上限）

### 公式

```
                    1
Speedup = ─────────────────────
           (1 - P) + P / N

P = 可并行化比例
N = 处理器/线程数
```

### 直觉理解

如果一个任务有 5% 的部分必须串行执行，那么无论你用多少个核心并行处理剩下的 95%，加速比永远不会超过 1/0.05 = **20 倍**。

```
串行比例    最大加速比(N→∞)
50%        2x
25%        4x
10%        10x
5%         20x
1%         100x
```

### 实际意义 —— 为什么 8 核不能快 8 倍

假设一个 Web 请求的处理流程：

```
┌────────────┬─────────────────────────────────────┬────────────┐
│  解析请求   │         业务逻辑（可并行）            │  返回响应   │
│  串行 10ms │            80ms                     │  串行 10ms │
└────────────┴─────────────────────────────────────┴────────────┘

总时间 = 100ms, 串行比例 = 20/100 = 20%
```

用 8 个核心：
```
Speedup = 1 / (0.2 + 0.8/8) = 1 / (0.2 + 0.1) = 1 / 0.3 = 3.33x
```

8 核只快了 3.33 倍，不是 8 倍。即使用 1000 核：
```
Speedup = 1 / (0.2 + 0.8/1000) ≈ 1 / 0.2008 ≈ 4.98x
```

上限约 5 倍。串行部分决定了加速的天花板。

### 工程启示

- **优化前先测量串行比例**，如果串行比例高，加核心没用，应该减少串行部分
- 常见串行瓶颈：全局锁、共享数据结构的同步、串行化/反序列化、I/O 等待
- Go 中常见例子：`sync.Mutex` 保护的共享 map 就是串行部分，GOMAXPROCS 再大也没用

## Little 定律（排队系统基本定律）

### 公式

```
L = λ × W

L = 系统中的平均请求数（并发数）
λ = 请求到达率（吞吐量）
W = 平均停留时间（延迟）
```

这个公式简单到令人怀疑，但它适用于**任何**稳态排队系统，无需假设分布。

### 实际应用

#### 应用一：计算线程池大小

已知条件：
- 目标吞吐量 λ = 1000 req/s
- 每个请求平均耗时 W = 50ms = 0.05s

```
L = λ × W = 1000 × 0.05 = 50
```

至少需要 50 个线程才能处理这个负载。这是数学推导出来的结果，不是凭感觉。

#### 应用二：计算数据库连接池大小

已知条件：
- 应用每秒发出 500 个数据库查询
- 每个查询平均耗时 20ms

```
L = 500 × 0.02 = 10
```

连接池至少需要 10 个连接。实际中会设置为 15-20 个，留一些余量应对波动。

#### 应用三：反向推算 — 已知并发数和延迟，算吞吐量

线程池 200 个线程，平均请求耗时 100ms：
```
λ = L / W = 200 / 0.1 = 2000 req/s
```

这就是系统的理论最大吞吐量。如果实际吞吐量远低于这个值，说明有其他瓶颈（比如锁竞争、I/O 等待）。

### 常见误用

Little 定律假设系统是**稳态**的——到达率 ≤ 服务率。如果流量超过系统处理能力，队列会无限增长，Little 定律不再适用。

## USL 通用可扩展性定律

### 问题：为什么加机器有时反而变慢

理想情况下，系统吞吐量应该随节点数线性增长。但实际往往是这样的：

```
吞吐量
  │         理想线性
  │        ╱
  │       ╱    ← 实际曲线
  │      ╱  ·····
  │     ╱ ··
  │    ╱··        ← 加到这里后开始下降!
  │   ╱·
  │  ╱·
  │ ╱·
  │╱·
  └──────────────── 节点数
```

USL（Universal Scalability Law）解释了这个现象。

### 公式

```
                      N
C(N) = ──────────────────────────────
        1 + σ(N-1) + κN(N-1)

N = 节点/线程数
σ = 串行化系数（contention，类似 Amdahl 的串行比例）
κ = 一致性开销系数（coherency，节点间通信/同步的开销）
```

### σ 串行化（Contention）

与 Amdahl 定律类似，系统中需要串行执行的部分。例如：
- 分布式锁
- 数据库主节点写入
- 全局序列号生成

σ 导致吞吐量增长放缓（亚线性）但不会下降。

### κ 一致性开销（Coherency）

这是 USL 比 Amdahl 定律多出的维度。当节点增多时，节点间需要互相通信来保持一致性，这个通信开销是 **O(N^2)** 的。

实际例子：
- 分布式缓存失效通知：每加一个节点，需要通知所有其他节点
- 数据库多主复制：每个写操作需要同步到所有主节点
- 微服务间的服务发现和健康检查

κ 导致吞吐量在节点数超过某个临界点后**开始下降**。

### 工程启示

```
场景                     σ (串行化)    κ (一致性)    建议
────────────────────────────────────────────────────────────
无状态 Web 服务          低           低            可以线性扩展
有共享 Session 的 Web    中           低            Session 外置到 Redis
强一致性分布式数据库      低           高            节点数不宜过多
全局锁保护的服务          高           低            减少锁粒度
```

## 排队论基础

### M/M/1 队列模型

最基础的排队模型，描述一个单服务器的排队系统：

```
到达 → [ 等待队列 ] → [ 服务器 ] → 离开
       ←─────────────────────→
              停留时间 W
```

- M/M/1：到达过程是泊松分布（M），服务时间是指数分布（M），1 个服务器
- λ = 到达率，μ = 服务率
- ρ = λ/μ = 利用率（必须 < 1，否则队列无限增长）

### 利用率与响应时间的非线性关系

M/M/1 队列的平均响应时间：

```
         1
W = ──────────
     μ(1 - ρ)

ρ = 利用率 = λ/μ
```

这就是为什么 **CPU 80% 时延迟就开始飙升**：

```
利用率 ρ    响应时间 W（相对于空闲时）
0%         1.0x
30%        1.4x
50%        2.0x
70%        3.3x
80%        5.0x     ← 延迟已经是空闲时的 5 倍
85%        6.7x
90%        10.0x    ← 延迟是空闲时的 10 倍!
95%        20.0x
99%        100.0x
```

画成图：

```
响应时间
  │                                          ╱
  │                                        ╱
  │                                      ╱
  │                                   ╱
  │                                ╱
  │                            ·╱
  │                        ··╱
  │                    ···
  │              ·····
  │     ·········
  └──────────────────────────────── 利用率
  0%    20%    40%    60%    80%  100%
```

### 工程启示

- **不要让关键资源的利用率超过 70-80%**。在这个区间之后，延迟会急剧增加
- 容量规划时要留 headroom（余量），不能按 100% 利用率来规划
- 这也是为什么"平时没问题，流量一涨就崩"的数学解释——你平时利用率 60%（延迟 2.5 倍），流量翻倍到 90%（延迟 10 倍），不是翻倍而是 4 倍的延迟增长

## 容量模型构建

将以上定律综合应用来构建容量模型：

### 步骤

1. **测量基线**：在当前负载下测量 RED 指标和资源利用率
2. **用 Little 定律计算理论吞吐量上限**
3. **用排队论预估不同负载下的延迟**
4. **用 USL 预估扩展后的实际吞吐量**

### 示例：计算一个服务需要多少实例

已知条件：
- 单实例：100 线程，每个请求平均 50ms
- 单实例最大吞吐量（Little）：L/W = 100/0.05 = 2000 req/s
- 但排队论告诉我们 70% 利用率时延迟开始明显上升
- 所以单实例安全吞吐量 = 2000 × 0.7 = 1400 req/s
- 目标：支撑 10000 req/s

```
需要实例数 = 10000 / 1400 ≈ 7.1 → 8 个实例
```

再考虑 USL 的一致性开销（假设 κ 很小，因为是无状态服务），不需要额外补偿。但如果服务有共享状态（如分布式 Session），可能需要 10-12 个实例来补偿一致性开销。

## 小结

| 定律 | 核心公式 | 核心启示 |
|------|----------|----------|
| Amdahl | S = 1/(1-P+P/N) | 串行部分决定并行加速上限 |
| Little | L = λW | 并发数 = 吞吐量 × 延迟 |
| USL | C(N) = N/(1+σ(N-1)+κN(N-1)) | 节点间一致性开销会让扩展出现负收益 |
| 排队论 | W = 1/μ(1-ρ) | 利用率超过 70% 后延迟急剧增长 |

这些定律不是理论空谈，而是你做容量规划、线程池配置、扩展策略时的数学基础。凭感觉设置的参数，大概率不是最优的。
