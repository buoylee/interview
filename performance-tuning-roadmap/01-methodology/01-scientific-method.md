# 性能问题的科学方法

## 为什么直觉不可靠

在性能排查中，依赖直觉是最常见也最危险的做法。一个有 5 年经验的后端工程师可能会说"肯定是数据库慢了"，然后花两天优化 SQL，最后发现瓶颈其实在序列化层。

直觉不可靠的根本原因：

1. **现代系统太复杂**：一个请求可能经过 LB → API Gateway → 微服务 A → 缓存 → 微服务 B → 数据库，任何一层都可能是瓶颈
2. **性能问题是非线性的**：CPU 从 70% 到 85% 可能导致延迟翻倍（排队论），但直觉告诉我们应该是线性增长
3. **经验可能过时**：上次的瓶颈在数据库，这次的瓶颈可能在 GC。系统在变化，上一次的经验不等于这一次的答案

一项 Google 内部的统计发现，工程师在没有数据支持的情况下，对性能瓶颈的首次猜测准确率不到 30%。

## 科学方法四步

性能排查应该遵循科学方法，而不是"猜测→修改→祈祷"：

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  1. 提出假设  │────▶│  2. 设计实验  │────▶│  3. 收集数据  │────▶│  4. 得出结论  │
│              │     │              │     │              │     │              │
│ "我认为瓶颈   │     │ 确定观测指标   │     │ 运行实验      │     │ 数据是否支持   │
│  在 X 组件"   │     │ 设计对照实验   │     │ 记录结果      │     │  原始假设？    │
└──────────────┘     └──────────────┘     └──────────────┘     └──────┬───────┘
       ▲                                                              │
       │                        假设被推翻，提出新假设                    │
       └──────────────────────────────────────────────────────────────┘
```

### 第一步：提出假设

假设必须是**可证伪的**。不好的假设："系统有点慢"。好的假设："P99 延迟从 200ms 上升到 800ms 是因为数据库连接池耗尽导致请求排队"。

好的假设具备三个特征：
- 明确指出嫌疑组件
- 说明因果机制
- 可以通过数据验证或推翻

### 第二步：设计实验

确定需要采集什么数据来验证或推翻假设。例如：
- 假设是"数据库连接池耗尽" → 观察连接池活跃连接数、等待队列长度
- 假设是"GC 停顿" → 观察 GC 日志、停顿时间与请求延迟的相关性

### 第三步：收集数据

用工具采集数据，而不是人眼盯着日志看。关键原则：
- **同时采集**：CPU、内存、I/O、网络指标要同时采集，才能看到相关性
- **保留原始数据**：不要只看平均值，保留分位数（P50/P95/P99）
- **注意采样率**：1 秒采样和 1 分钟采样看到的世界完全不同

### 第四步：得出结论

数据要么支持假设，要么推翻假设。如果被推翻，回到第一步提出新假设。这不是失败，而是进展——你排除了一个可能性。

## 控制变量法在性能分析中的应用

控制变量法的核心：**每次只改变一个因素**。

这是工程师在性能调优时最容易违反的原则。一个典型的错误场景：

```
"我同时升级了 JDK 版本、调大了堆内存、换了 GC 算法，性能好了 20%"
```

问题是你不知道这 20% 的提升来自哪个改动。下次遇到类似问题时，你无法复用这个经验。

**正确做法：**

```
基准测试 (JDK 11, 4G heap, G1GC)          → P99 = 800ms
实验1 (JDK 17, 4G heap, G1GC)             → P99 = 750ms   ← JDK 升级贡献 50ms
实验2 (JDK 17, 8G heap, G1GC)             → P99 = 720ms   ← 堆内存贡献 30ms
实验3 (JDK 17, 8G heap, ZGC)              → P99 = 400ms   ← GC 算法贡献 320ms
```

现在你知道：GC 算法是最大的改善因素。这个认知比"结果好了"有价值得多。

实际操作建议：
- 每次改动前先跑基准测试，记录基线数据
- 改一个变量，重新测试，记录数据
- 对比基线和实验结果
- 然后再改下一个变量

## 常见认知偏差

### 确认偏误（Confirmation Bias）

**表现**：只看支持自己猜测的数据，忽略矛盾的证据。

> 你怀疑是数据库慢查询导致的，于是只看慢查询日志。确实找到几条慢查询，就认定找到了原因。但你没有看到：慢查询只占总请求的 0.1%，根本不是 P99 延迟高的主因。真正的原因是 GC 停顿。

**对策**：主动寻找**反面证据**。问自己："如果不是数据库的问题，还可能是什么？数据是否也能被其他原因解释？"

### 幸存者偏差（Survivorship Bias）

**表现**：只看成功的请求或正常的指标，忽略失败的和异常的。

> 你看到平均响应时间是 100ms，觉得没问题。但实际上有 5% 的请求超时被 LB 直接丢弃了，它们根本不在你的统计里。如果把这些超时请求算上，P99 其实是 30 秒。

**对策**：
- 同时看成功和失败的请求
- 看错误率和超时率，不只看成功请求的延迟
- 用分位数（P50/P95/P99/P999）替代平均值

### 过早优化（Premature Optimization）

**表现**：没有数据就开始改代码。凭感觉觉得某段代码"看起来慢"就去优化。

> 你花了三天把一个 O(n^2) 算法改成 O(n log n)，但 n 最大只有 50，这段代码的执行时间从 0.01ms 变成 0.005ms。而真正的瓶颈——一个没有索引的数据库查询——每次耗时 500ms，你根本没看。

**对策**：先 profile，再优化。Donald Knuth 说的"过早优化是万恶之源"不是让你不优化，是让你先找到值得优化的地方。

## 案例：如何用科学方法排查「接口偶尔慢」的问题

### 背景

一个电商系统的"查询订单详情"接口，监控显示 P50 = 50ms（正常），但 P99 = 2000ms，且没有明显规律。

### 第一步：提出假设

根据现象"偶尔慢"，列出可能的假设：
- 假设 A：数据库慢查询（某些订单数据量大）
- 假设 B：JVM GC 停顿
- 假设 C：网络抖动（微服务间调用）

### 第二步 & 第三步：逐一验证

**验证假设 A — 数据库慢查询**
```bash
# 查看慢查询日志
$ mysqldumpslow -s t -t 10 /var/log/mysql/slow-query.log

Count: 12  Time=0.15s   Lock=0.00s  Rows=5.0
  SELECT * FROM order_detail WHERE order_id = 'N'
```

慢查询平均只有 150ms，不到 2000ms。且慢查询与 P99 尖刺的时间点不吻合。→ **假设 A 被推翻**。

**验证假设 B — GC 停顿**
```bash
# 查看 GC 日志（JDK 11+）
$ grep "pause" gc.log | awk '{print $1, $NF}'

2024-01-15T10:23:15  1.82s
2024-01-15T10:25:43  1.95s
2024-01-15T10:28:01  1.76s
```

发现每隔 2-3 分钟有一次 Full GC，停顿时间约 1.8 秒。与接口 P99 = 2000ms 高度吻合！

进一步确认——对比 GC 时间点和慢请求时间点：

```bash
# 慢请求日志
$ grep "duration>1500" access.log | awk '{print $1}'
10:23:15
10:25:43
10:28:02

# GC 停顿时间
10:23:15  ← 吻合
10:25:43  ← 吻合
10:28:01  ← 吻合（1秒误差，在 GC 停顿时间范围内）
```

时间点完全吻合。→ **假设 B 被确认**。

### 第四步：得出结论并验证

结论：P99 延迟尖刺由 Full GC 停顿导致。

验证手段：
- 调整 GC 策略（从 G1GC 切换到 ZGC），Full GC 停顿降至 < 10ms
- 上线后 P99 从 2000ms 降至 120ms

### 复盘

如果一开始按直觉去优化数据库，可能白忙几天。科学方法的价值不在于每一步都猜对，而在于：
- 每一步都有明确的数据支撑
- 错误的假设能被快速排除
- 最终的结论可以被验证
- 排查过程可以被复用和教给其他人

## 小结

| 原则 | 说明 |
|------|------|
| 先观测，后行动 | 不采集数据就不要改代码 |
| 每次只改一个变量 | 否则不知道哪个改动有效 |
| 用数据说话 | 用分位数，不用平均值；用时间线相关性，不用直觉 |
| 假设要可证伪 | "可能是网络问题"不是好假设 |
| 记录过程 | 写下每一步的假设、数据和结论，方便复盘 |
