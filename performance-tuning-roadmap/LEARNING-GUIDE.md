# 性能调优路线学习指南

> 目标：保证学习者学习的系统性和流畅性，降低学习曲线，让这套路线从“知识库”变成“训练系统”。

---

## 1. 学习原则

### 1.1 先闭环，再扩展

性能调优不是靠记工具名学会的。学习者必须先完整经历一次闭环：

```
发现异常 → 建立基线 → 提出假设 → 采集证据 → 定位根因 → 修改方案 → 验证效果 → 记录复盘
```

因此，前期不要追求覆盖全部语言和全部组件。先用一门语言、一个接口、一个问题，把完整流程跑通。

### 1.2 每个阶段只增加一种主要认知负担

学习曲线陡峭，通常是因为同一阶段同时引入了太多新东西：新概念、新工具、新环境、新语言、新问题类型。

本路线建议这样控制难度：

| 阶段 | 新增负担 | 不强求 |
|------|---------|--------|
| P | 运行环境 | 不要求理解所有配置 |
| 0 | OS 心智模型 | 不要求立即调内核参数 |
| 1 | 分析方法 | 不要求熟练所有工具 |
| 2 | Linux 工具 | 不要求修复复杂问题 |
| 3 | 可观测性 | 不要求一次设计完美 Dashboard |
| 3.5 | 负载生成 | 不要求完整压测报告 |
| 4/5/6 | 语言运行时 | 不要求三语言同时深入 |
| 7 | 压测设计 | 不要求 CI 门禁一次到位 |
| 8-10 | 生产排查场景 | 不要求覆盖所有根因 |
| 11-13 | 工程化闭环 | 不要求团队制度一步到位 |
| 14 | 综合演练 | 不要求第一次就满分 |

### 1.3 每次学习都要留下产物

没有产物，学习很容易变成“看懂了但不会做”。建议每个阶段至少留下一个文件或截图：

- 命令输出摘录
- Grafana 截图
- 火焰图或 profile 文件
- 慢查询 EXPLAIN 结果
- 压测报告
- 排查记录
- 复盘记录

## 2. 三层学习路线

### 2.1 核心主线：必须掌握

这是所有方向都应该完成的路线：

```
P 实践环境
→ 0 OS 基础
→ 1 性能方法论
→ 2 Linux 工具链
→ 3 可观测性
→ 3.5 压测快速上手
→ 7 压测方法论
→ 8 网络与 I/O
→ 9a 数据库性能
→ 10 分布式排查
→ 13 SRE 与生产实践
→ 14 综合实战
```

这条线解决“能不能系统排查”的问题。

### 2.2 主语言专项：先选一门

学习者应该先选择一门工作中最常用的语言作为主线专项：

| 主语言 | 学习文件夹 | 目标 |
|--------|------------|------|
| Java | 04a + 04b | 能排查 JVM、GC、线程、锁、Heap、Netty 问题 |
| Go | 05a + 05b | 能排查 pprof、goroutine、GC、网络客户端、逃逸问题 |
| Python | 06a + 06b | 能排查 CPU、内存、GIL、asyncio、Web 框架部署问题 |

建议：主语言先完整学完，再补另外两门的“profiling 工具 + 常见反模式”。

### 2.3 进阶扩展：按场景触发

这些内容不要一开始强行学习，遇到场景再深入：

| 场景 | 建议学习 |
|------|---------|
| 需要更深 Linux 观测 | 02/06 eBPF |
| TLS、证书、握手慢 | 08/04 TLS 排查 |
| Redis 高频故障 | 先做 `labs/perfshop-p0` 的 P1-mini Redis slow / big key，再学 9b 中间件性能 |
| Kafka consumer lag | 9b 中间件性能，当前需要手工环境或未来 P2 扩展 |
| 架构改造和容量提升 | 11 架构级性能优化 |
| 容器或 K8s 资源问题 | 12 容器与云原生性能 |
| 团队级稳定性建设 | 13/03-13/05 |

### 2.4 按面试目标选择路线

所有学习者先完成 P0 最小闭环，再完成 P1-mini 的 Redis/downstream/retry 闭环，然后按目标选择 track：

| 面试目标 | 推荐入口 | 学习重点 |
|----------|----------|----------|
| 资深后端工程师 | [TRACKS.md](./TRACKS.md) 的 Backend Senior Track | runtime、数据库、缓存、连接池、分布式调用 |
| SRE / 平台工程师 | [TRACKS.md](./TRACKS.md) 的 SRE / Platform Track | 可观测性、容量、K8s、告警、事故响应 |
| Staff / Tech Lead | [TRACKS.md](./TRACKS.md) 的 Staff / Tech Lead Track | 架构权衡、性能治理、容量模型、跨团队落地 |

如果目标不确定，先走 Backend Senior 的 4 周版本，因为它最贴近日常服务排查；之后补 SRE 的可观测性、容量和事故复盘能力。

## 3. 每个学习单元的统一学习法

建议每篇文章都按同一套流程学习，避免读完忘记怎么用。

### 3.1 读前

先回答三个问题：

1. 这篇解决什么现象？
2. 我现在是否具备前置条件？
3. 学完后我要产出什么证据？

示例：

| 文件 | 现象 | 产出 |
|------|------|------|
| 04a/04-async-profiler-flamegraph.md | CPU 高但不知道热点方法 | 一张 CPU 火焰图和 top 3 热点解释 |
| 09a/01-slow-query-analysis.md | 接口慢且数据库耗时高 | 一份慢查询日志、EXPLAIN、优化前后对比 |
| 10/01-timeout-circuit-breaker.md | 下游慢拖垮上游 | 一张 Trace、超时链路说明、修复方案 |

### 3.2 学中

按固定顺序推进：

1. 先理解现象，不急着记命令
2. 再理解指标，明确正常和异常的区别
3. 然后运行命令，拿到真实输出
4. 最后解释输出，而不是只保存输出

### 3.3 学后

用 5 句话完成阶段记录：

```text
本节问题：
观察到的现象：
使用的工具：
证据和结论：
下一步：
```

这份记录可以很短，但必须有证据和结论。

## 4. 阶段检查点

阶段检查点用于控制学习节奏。更细的逐阶段 Definition of Done 见 [STAGE-DOD.md](./STAGE-DOD.md)：它规定每个阶段必须交付的证据、可进入下一阶段的条件，以及对应的面试表达标准。

### Checkpoint 1：方法论成型

完成阶段：P、0、1、2

必须能做到：
- 解释 load average、CPU 使用率、iowait、RSS、TIME_WAIT 的基本含义
- 面对“接口慢了”，能列出 USE + RED 检查清单
- 能用 `top`、`vmstat`、`mpstat`、`pidstat`、`iostat`、`ss` 做第一轮定位

产物：
- 一份 5 分钟主机巡检记录
- 一份 PerfShop 接口 USE/RED 检查清单

### Checkpoint 2：观测闭环

完成阶段：3、3.5

必须能做到：
- 服务有指标、日志、Trace
- 能用 wrk 或 Locust 施加稳定负载
- 能在 Grafana 看见 QPS、错误率、P95/P99 延迟、CPU、内存

产物：
- 一张 RED Dashboard
- 一条 trace_id 关联日志或 Jaeger Trace
- 一份 30 秒压测输出

### Checkpoint 3：主语言排查

完成阶段：4a/4b 或 5a/5b 或 6a/6b

必须能做到：
- 在压测期间采集 profile
- 解释 profile 中最明显的热点
- 提出一个可验证的优化动作

产物：
- 一份 profile 证据
- 一份优化前后对比

### Checkpoint 4：生产场景排查

完成阶段：7、8、9a、9b、10

必须能做到：
- 设计一次有目标的压测
- 定位一次慢查询或中间件问题
- 用 Trace 解释一次跨服务延迟
- 区分根因、诱因、放大器

产物：
- 一份压测报告
- 一份数据库或中间件排查记录
- 一份分布式链路分析记录

### Checkpoint 5：完整闭环

完成阶段：13、14

必须能做到：
- 从告警或现象开始排查
- 给出缓解、修复、验证、复盘
- 用数据证明问题已经解决

产物：
- 一份完整性能报告
- 一份事故复盘或演练复盘
- 自测题评分记录

### 4.6 面试化产物要求

每个 checkpoint 的产物都要能转化成面试表达：

| Checkpoint | 面试化产物 | 面试中要讲清楚 |
|------------|------------|----------------|
| 方法论成型 | 一份 USE/RED 排查记录 | 如何从现象提出假设，而不是直接猜根因 |
| 观测闭环 | 一张 dashboard 或 PromQL 证据 | 哪个指标异常，为什么它支持某个假设 |
| 主语言排查 | 一份 profile、火焰图或 runtime 证据 | 热点、等待或 GC 的机制是什么 |
| 生产场景排查 | 一份数据库、中间件或 Trace 分析 | 根因、诱因、放大器分别是什么 |
| 完整闭环 | 一份报告或复盘 | 如何修复、验证、回滚和预防复发 |

回答是否达到资深水平，使用 [SENIOR-RUBRIC.md](./SENIOR-RUBRIC.md) 自检。

## 5. PerfShop 落地优先级

PerfShop 是这套路线的主轴。它不必一开始就完整实现三语言、微服务、Kafka、全套故障注入，但必须尽早具备最小可运行闭环。

P0 / P1-mini 可运行实验闭环的详细标准见：[labs/perfshop-p0](./labs/perfshop-p0/)。

### 5.1 P0：最小实验闭环

优先保证：

- 一个后端服务（Java、Go、Python 任一门）
- 一个 HTTP 接口
- MySQL
- Prometheus 指标
- Grafana Dashboard
- wrk 或 Locust 压测
- 至少 3 个可控问题：
  - 慢 SQL
  - CPU 热点
  - 连接池或下游超时

P0 的目标不是系统完整，而是让学习者能完成：

```
压测 → 观察异常 → 采集证据 → 定位根因 → 修改 → 复测
```

### 5.2 P1-mini：当前可运行的多组件排查

P0 跑通后，直接在同一个 `labs/perfshop-p0/` 中完成 P1-mini：

- Redis cache-aside
- Redis slow 和 Redis big key
- 独立 downstream 推荐服务
- App/downstream `X-Trace-Id` 透传和结构化日志关联
- Downstream delay、App timeout 和稳定 502
- Retry storm 和 `app_downstream_retries_total`

P1-mini 的目标是让学习者证明：缓存、下游依赖和重试策略如何把局部问题放大成跨组件延迟或错误率上升。

### 5.3 P1：后续多组件扩展

P1-mini 之后再补：

- Kafka consumer lag
- OpenTelemetry Trace 和 Jaeger
- 日志聚合
- 连接池耗尽
- 更完整的降级、限流和熔断策略

这些内容可以先作为 `Runnable with manual setup` 或 `Target-state example` 学习，不应阻塞第 0 轮闭环。

### 5.4 P2：三语言对称实验

最后补齐：

- Java / Go / Python 三套同构 API
- 三语言 runtime 指标
- 三语言 profiling 对比
- 同一业务问题在不同语言下的排查差异

这样能避免学习者一开始被环境复杂度挡住。

## 6. 学习者常见卡点与处理

| 卡点 | 表现 | 处理方式 |
|------|------|---------|
| 环境跑不起来 | Docker、端口、依赖问题耗尽耐心 | 只保留 P0 最小环境，先跳过非核心组件 |
| 工具太多 | 记住了命令但不知道何时使用 | 回到“现象 → 假设 → 工具”的选择逻辑 |
| 指标看不懂 | Grafana 有图但无法解释 | 先只看 RED + USE，其他指标暂时隐藏 |
| 火焰图看不懂 | 不知道宽度、高度、栈顶含义 | 只要求找 top 3 热点并解释调用路径 |
| 压测结果不稳定 | 每次 QPS 差异大 | 固定数据、固定并发、先跑预热，再记录结果 |
| 学到后面忘前面 | 知识断裂 | 每个 checkpoint 回写一份短报告 |

## 7. 文档维护规则

后续新增或修改学习单元时，尽量保持同一结构：

```markdown
# 标题

## 本节解决什么问题
## 前置条件
## 真实现象
## 核心概念
## 工具和命令
## 在 PerfShop 上练习
## 验收标准
## 常见误区
## 下一步
```

不是每篇都必须完全一样，但至少要让学习者知道：

- 为什么学
- 怎么做
- 怎么判断自己学会了
- 下一步接哪里

## 8. 学习产物模板

为了让学习者每次练习都留下可复盘的证据，本路线提供以下模板：

| 模板 | 用途 |
|------|------|
| [performance-investigation-record.md](./templates/performance-investigation-record.md) | 记录一次完整性能排查 |
| [load-test-report.md](./templates/load-test-report.md) | 输出一次压测报告 |
| [profiling-report.md](./templates/profiling-report.md) | 记录一次 profiling 分析 |
| [postmortem.md](./templates/postmortem.md) | 记录一次事故或演练复盘 |

建议从第 0 轮最小闭环开始就使用模板。模板可以简写，但不要跳过“假设、证据、结论、验证”这几个字段。
