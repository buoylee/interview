# Wheel of Misfortune(故障转盘):把演练变成压力测试

> Google SRE 的经典练习:有人扮演「事故」,你扮演 on-call,在**有人盯着、计时**的压力下排查。
> 看着答案抄命令 ≠ 会;**被丢一台不知道哪坏的机器、自己决定先敲什么**,才是面试和真事故的样子。这个练习专门补这一块。

---

## 一、为什么要做这个(补的是哪部分「大厂经验」)

[道场总纲的 gap map](../README.md#二大厂真实经验拆解--这个道场能补多少) 里,「**压力下的识别速度**」只能靠这种练习补:

- 平时照着场景文档走,你**知道答案**,练的是熟练度;
- Wheel of Misfortune 让你在**不知道答案 + 计时 + 要边查边讲**的状态下做——这才逼出真实水平,暴露你「卡在哪一步、哪句话讲不清」。

## 二、玩法 A:双人(最接近真实)

1. **出题人**:挑一个场景(或用 [`blind-test.sh`](./blind-test.sh) 随机注入),不告诉答题人是哪个。
2. **答题人(on-call)**:边查边**出声讲**你的推理——「我先看 load……是 CPU 型还是 I/O 型……」。出题人只回答你「观察到的事实」(像监控/日志),不给方向。
3. **计时**:记 **MTTD**(多久定位到根因)和你**讲清因果链**的程度。
4. **复盘**:用 [`postmortem-template.md`](../postmortem-template.md) 填一份,出题人挑刺:哪步多余、哪步漏证据、哪句话含糊。

## 三、玩法 B:单人(用脚本自测)

```bash
bash drills/blind-test.sh
```
脚本随机注入一个故障并计时,你按 [07](../../07-troubleshooting-playbook/) 的路径自己查,回来按 Enter 看答案和用时。**规则**:查的过程把每一步的「假设 + 验证命令 + 结论」写在纸上,揭晓后对照场景的「破案点」找差距。

## 四、给自己打分(每局问这几条)

- [ ] **方向**:是否在前 1 分钟就分清了 CPU / IO / 内存 / 网络?(没分清就乱抓 = 减分)
- [ ] **证伪**:每个假设是否用一条命令验证后才往下走?(凭感觉跳步 = 减分)
- [ ] **锁定**:是否从「机器」一路锁到「进程 / 线程 / fd / 根因」?
- [ ] **讲清**:能否一句话说清因果链(现象→根因)?
- [ ] **止血意识**:有没有先想「怎么先恢复」,而不是一头扎进根因?

## 五、加压方式(进阶)

- **限时**:给自己定 5 分钟必须说出方向、10 分钟必须到根因。
- **盲打**:不许翻场景文档,只许用 `man` 和机器本身。
- **连击**:连续抽 3 个不同场景,中间不许查资料——练「场景切换」的反射。

> 攒下来的每一局复盘,就是你自己的「战争故事」。面试问「讲一次你排查问题的经历」,这些就是合法、且能讲出方法论的素材。
