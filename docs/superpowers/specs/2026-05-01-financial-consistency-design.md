# 金融级分布式事务与一致性学习项目设计

日期：2026-05-01

## 目标

建立一套贴近真实金融系统的学习与实现项目，用 Java 主线系统化掌握事务、一致性、幂等、补偿、对账、审计和可扩展架构。

这个项目不是单体事务 demo，也不是只写理论笔记。它要从最简单的转账场景开始，但从第一天就按分布式服务边界设计，逐步扩展到充值、提现、支付回调、电商交易、退款、机票酒店组合预订、跨境和多币种结算等现实场景。

最终能力目标：

- 能解释真实系统里为什么不能靠一个数据库事务解决所有问题。
- 能判断每类业务场景的一致性边界、可逆动作、不可逆动作和补偿方式。
- 能设计可重试、可审计、可对账、可恢复的交易链路。
- 能先定义资金系统的不变量，再用模型验证、属性测试、故障注入、历史检查和对账验证实现。
- 能用 Java/Spring、关系型数据库、Kafka、Redis、Kubernetes 等主流技术实现可 scale 的原型。
- 能把这些设计转化成面试、架构评审和真实工程中的表达。

## 核心决策

### 1. 技术栈

主线使用 Java，而不是 Go。

原因：

- Java/Spring 仍是国际金融、银行、支付、清结算和大型企业交易系统的主流后端生态。
- Spring 生态对事务、数据库、消息、可观测性、微服务治理的支持完整。
- 大型金融机构公开招聘和工程案例中，Java、Kafka、Kubernetes、云平台和关系型数据库组合非常常见。
- Go 后续保留为辅助语言，用于压测、故障注入、轻量 worker、sidecar 或基础设施工具。

默认栈：

- Java 21
- Spring Boot / Spring Cloud 作为服务开发与治理基础
- PostgreSQL 或 MySQL
- Kafka 作为事件流、Outbox 投递和异步解耦基础
- Redis
- Docker / Kubernetes
- OpenTelemetry + Prometheus + Grafana
- Outbox / Debezium / CDC
- Temporal 作为国际主线的 durable workflow / Saga Orchestrator

框架选型结论：

- 主实现路线采用 `Spring Boot + Kafka + Outbox + Temporal`。
- 第一阶段允许手写最小 Saga Orchestrator，用来学习状态机、补偿、幂等、重试和恢复的底层机制。
- 第二阶段引入 Temporal，对比它如何承接 workflow 状态持久化、activity retry、timeout、worker crash recovery 和补偿编排。
- Seata 不作为主实现框架。它放入模式对比章节，用来理解 AT/TCC/Saga 思想，以及为什么国际金融主线更偏显式业务状态机、Outbox、消息、补偿和对账。
- Camunda 放入后续对比章节。它更适合 BPMN、审批、人审、合规流程和企业流程自动化；本项目核心是资金一致性与 durable execution，因此 Temporal 优先。

数据库选择不在第一版强制锁死。学习阶段优先让模型、事务边界和故障恢复清楚；实现阶段再根据本机环境和已有笔记选择 PostgreSQL 或 MySQL。

### 2. 架构形态

不采用模块化单体作为主实现。

第一阶段从简单转账业务开始，但服务边界从一开始就是分布式：

- `transaction-orchestrator`：交易流程编排，负责 Saga/TCC/补偿流程推进。
- `account-service`：账户余额、冻结、解冻、扣减、入账。
- `ledger-service`：不可随意修改的会计分录、资金流水、审计账本。
- `payment-service`：支付请求、支付单状态、渠道回调、幂等处理。
- `risk-service`：交易前风险检查、限额、风控拒绝。
- `notification-service`：通知用户和业务方，不参与强一致核心链路。
- `reconciliation-service`：准实时/日终对账、差错识别、修复工单。
- `message-broker`：Kafka 作为事件流和异步解耦基础。

转账只是最小业务场景，不是简化架构。

### 3. 学习路线

采用“场景从简单到复杂，架构保持真实”的路线。

每个阶段都采用同一个科学学习循环：

```text
业务场景
-> 正确性不变量
-> 状态机和事件模型
-> 形式化/半形式化推演
-> 真实代码实现
-> 属性测试和集成测试
-> 故障注入和历史检查
-> 对账闭环
-> 面试和架构评审表达
```

这个循环比“先写代码再补测试”更贴近金融系统。学习者必须先知道系统永远不能违反什么，再讨论用什么事务模式和框架。

第一层：资金内核

- 余额

- 可用余额
- 流水
- 会计分录

- 对账

- MQ 消息重复
- 服务超时但实际成功

## 真实性标准

项目必须避免玩具化实现。

### 资金模型

- 钱相关数据不能只靠一个 `balance` 字段表达。
- 账户必须区分可用余额、冻结金额和总余额。
- 每一次资金变化都必须产生流水或会计分录。
- 账本记录默认 append-only，不允许业务代码随意覆盖历史记录。
- 所有资金动作必须能被追踪到请求、交易、消息、分录和操作者。

### 幂等与重试

- 所有写接口必须有幂等键。
- 外部请求、内部 RPC、消息消费、回调处理都默认可能重复。
- 重试不能造成重复扣款、重复入账、重复退款或重复通知。
- 幂等记录必须能区分处理中、成功、失败、超时和可重试状态。

### 分布式一致性

- 本地事务只保护单服务、单数据库内的强一致边界。
- 跨服务不假设可以共享数据库事务。
- 跨服务一致性通过 Saga、TCC、Outbox、事务消息、补偿和对账组合实现。
- 消息系统默认至少一次投递，因此消费者必须幂等。
- 不能把“调用返回成功”当作最终事实，必须以状态机、事件和对账闭环为准。

### 外部系统

- 银行、支付渠道、航司、酒店等外部系统默认不可控。
- 外部系统不参与我们的数据库事务。
- 外部调用可能成功但超时，失败但稍后回调成功，或者重复回调。
- 外部动作如果不可逆，系统只能做业务补偿、退款、赔付或人工处理，不能伪装成数据库回滚。

### 可观测性与恢复

- 每个交易必须有全局 trace id / transaction id。
- 状态变化、消息发布、消息消费、补偿动作、对账结果都要可追踪。
- 系统必须能从 Orchestrator 宕机、消费者宕机、消息重复、消息延迟、DB 短暂不可用中恢复。
- 对账不是补充功能，而是金融级可靠性的核心闭环。

### 可验证性

金融级实现不能只依赖人工 code review 或 happy-path 测试。每个核心场景都必须明确不变量，并使用多层验证手段持续检查。

转账场景的基础不变量：

- 系统总金额不能凭空增加或减少。
- 同一个幂等键只能产生一次业务效果。
- 成功转账必须同时存在付款方借记分录和收款方贷记分录。
- 账户余额必须能由分录或流水重新计算并校验。
- 冻结金额最终必须进入扣减、释放或异常处理状态。
- 终态交易不能从 `SUCCESS`、`FAILED`、`CANCELLED` 非法跳回处理中状态。
- 消息可以重复、延迟、乱序，但业务结果不能重复扣款、重复入账或漏记分录。

验证手段按学习阶段递进：

- 形式化/半形式化模型：用 TLA+ / PlusCal 或等价模型描述状态机、重试、补偿和故障路径，先找设计级 bug。
- 属性测试：随机生成转账、重试、取消、补偿和并发序列，持续检查不变量。
- 集成测试：用 Testcontainers 启动真实数据库、Kafka、Redis，验证真实中间件语义。
- 故障注入：模拟 RPC 超时、消费者宕机、ack 丢失、重复消息、DB 死锁、网络中断、Orchestrator 重启。
- Jepsen-like 历史检查：记录并发操作历史，检查最终结果是否满足资金守恒、幂等和状态机约束。
- 运行时校验：在服务内记录审计事件，定期扫描非法状态、长时间悬挂交易和未消费 outbox。
- 对账验证：通过余额、流水、分录、交易单、渠道账单多方交叉校验，发现并修复差错。

这不是为了追求“证明没有 bug”。目标是用科学方法把最危险的事务 bug 提前暴露，并确保线上出现异常时可以发现、定位、止损和修复。

目录职责：

- `02-payment-recharge-withdraw/`：充值、提现、支付回调、渠道差错。
- `03-order-payment-inventory/`：电商交易链路。
- `04-travel-booking-saga/`：机票、酒店、租车、保险的组合预订和补偿。

- `08-interview-synthesis/`：面试表达、架构评审话术、常见追问。
- `references.md`：归档当前仓库已有笔记的路径，不搬乱旧文档。

## 第一阶段范围

第一阶段只做分布式转账，但按真实金融系统拆服务和建模。

必须覆盖：

- 用户发起转账请求。
- 请求幂等。
- 风控检查。
- 扣减付款方可用余额或冻结金额。
- 生成付款方分录。
- 给收款方入账。
- 生成收款方分录。
- 发布交易事件。
- 处理消息重复和乱序。
- 交易状态机推进。
- 失败补偿。
- 转账不变量定义。
- 最小 TLA+ / PlusCal 或等价状态模型。
- 属性测试和故障注入用例。
- 对账发现并修复异常。

第一阶段不覆盖：

- 高并发撮合交易。
- 复杂监管报送。
- 多币种汇率。
- 真实第三方支付渠道接入。
- 完整 Kubernetes 生产部署。

这些内容会在后续阶段扩展，第一阶段只保留能支撑真实一致性思维的最小闭环。
