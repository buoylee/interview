# 04 旅行组合预订 Saga 一致性设计

日期：2026-05-02

## 目标

创建 `financial-consistency/04-travel-booking-saga/`，把学习主线从电商订单、支付、库存推进到旅行组合预订。

这个阶段的核心不是“多个服务一起成功或一起失败”，而是理解机票、酒店、租车和保险这些外部供应商动作不可控、价格会漂移、部分动作不可逆时，系统如何用 Saga、状态机、付款策略、补偿、对账和人工处理让组合订单进入可解释终态。

完成后，学习者应该能解释：

- 为什么旅行组合交易通常不能做严格全有全无。
- 为什么核心项和附加项要分层处理。
- 为什么补偿不是数据库回滚，而是释放授权、取消供应商保留、退款、罚金、补差、替代供应商或人工处理。
- 为什么付款策略必须按子产品建模，而不是全局统一。
- 为什么供应商确认、出票、保险生效和支付扣款都必须是可审计事实。
- 为什么组合订单必须支持部分成功、附加项失败和人工处理状态。

## 背景

`01-transfer` 建立了内部资金事实和账本基础。

`02-payment-recharge-withdraw` 引入了外部支付渠道：

- 外部系统不参与本地事务。
- 超时不是失败。
- 回调、查询、账单和本地事实必须能对账。

`03-order-payment-inventory` 引入了订单、库存、支付、取消和退款的业务资源一致性：

- 库存必须预留、确认或释放。
- 支付成功不能被普通取消覆盖。
- 已支付取消进入退款或人工处理。
- 对账是恢复链路，不是报表。

`04-travel-booking-saga` 在此基础上引入更难的现实约束：

- 供应商动作可能不可逆或有罚金。
- 搜索价、锁价和最终成交价可能不同。
- 航班座位、酒店房态、租车库存和保险生效规则来自不同供应商。
- 核心项成功但附加项失败是常见业务结果。
- 用户、运营、供应商和支付渠道都可能参与最终收敛。

## 范围

第一版采用完整组合范围：机票 + 酒店 + 租车 + 保险。

业务分层：

- 核心项：机票 + 酒店。核心项必须成功，组合订单才能算核心成功。
- 附加项：租车 + 保险。附加项允许失败，不拖垮核心行程。

覆盖 7 个核心场景：

1. 搜索报价、锁价和报价过期。
2. 创建组合订单和子订单。
3. 预授权、扣款、释放授权和退款。
4. 机票出票和酒店确认的核心 Saga。
5. 租车和保险附加项处理。
6. 核心失败、附加失败、部分成功和人工处理。
7. 供应商、支付、退款、罚金、补差和本地订单对账。

这个阶段构成的闭环是：

```text
搜索报价
-> 创建组合订单
-> 识别每个子产品的付款策略和可逆性
-> 预授权或扣款
-> 锁座/出票、酒店保留/确认、租车确认、保险投保
-> 成功、失败、未知或部分成功裁决
-> 释放授权、取消供应商保留、退款、罚金、补差、替代供应商或人工处理
-> 供应商/支付/退款/账本/事件
-> 对账与差错修复
```

## 非目标

第一版不覆盖：

- 真实航空 GDS、酒店 CRS、租车或保险供应商 SDK。
- 航司复杂退改签规则的完整实现。
- 多城市、多航段、联程票、不同 PNR 拆分。
- 酒店超售后的真实换房和赔付运营流程。
- 保险理赔。
- 会员积分、优惠券、礼品卡和复杂价格引擎。
- 商户结算、供应商账期和佣金分润。
- 国际多币种清结算、汇率锁定和监管报送。

这些场景后续可以单独展开。第一版先把组合交易的 Saga、可逆性、付款策略和对账讲清楚。

## 业务策略

### 核心项和附加项

第一版采用现实 OTA 常见策略：

- 机票和酒店是核心项。
- 租车和保险是附加项。
- 核心项全部成功时，组合订单进入核心成功。
- 核心项失败时，组合订单不能直接成功，必须补偿、退款、替代或人工处理。
- 附加项失败时，保留核心成功，附加项进入退款、替代、用户确认或人工处理。

典型总订单状态：

- `QUOTE_SELECTED`
- `BOOKING_CREATED`
- `PAYMENT_AUTHORIZED`
- `CORE_BOOKING_IN_PROGRESS`
- `CORE_CONFIRMED`
- `CONFIRMED`
- `CONFIRMED_WITH_ADDON_FAILURES`
- `PARTIALLY_CONFIRMED`
- `COMPENSATING`
- `CANCELLED`
- `REFUNDING`
- `MANUAL_REVIEW`

### 付款策略

付款策略按子产品建模，不做全局假设。

| 策略 | 现实含义 | 成功处理 | 失败处理 |
| --- | --- | --- | --- |
| `PRE_AUTH_THEN_CAPTURE` | 先预授权或冻结用户资金，供应商确认后再扣款 | 核心项成功后 capture | 供应商失败则 void 或 release 授权 |
| `IMMEDIATE_CAPTURE_REQUIRED` | 供应商要求先扣款，或出票/确认时立即扣款 | 记录外部资金事实，继续推进订单 | 后续失败不能回滚，只能取消、退款、改签、补差或人工处理 |
| `PAY_AT_SUPPLIER` | 用户到店、到柜台或取车时支付，平台只担保或登记卡信息 | 平台订单确认，但平台账本不扣款 | 标记供应商侧责任，可能产生担保费、罚金或 no-show 费用 |
| `NON_REFUNDABLE_OR_PENALTY` | 不可退或有罚金产品 | 成功后标记半不可逆或不可逆 | 失败补偿只能部分退款、收取罚金、补差或人工处理 |

底层原则：

> 旅行组合 Saga 的补偿不是“回滚”，而是根据每个供应商动作的可逆性，选择释放授权、取消供应商保留、退款、收取罚金、补差、替代供应商或人工处理。

## 场景卡

第一版必须包含 7 张场景卡。

### 搜索报价、锁价和报价过期

场景边界：

- 用户搜索机票、酒店、租车和保险。
- 系统展示组合报价。
- 用户选择后，系统尝试锁价或生成报价快照。
- 报价可能过期、涨价、降价或库存消失。

核心风险：

- 搜索价被当作成交价。
- 报价过期后仍按旧价扣款。
- 价格变动没有用户确认。
- 报价快照和最终供应商确认事实无法对账。

### 创建组合订单和子订单

场景边界：

- `booking-service` 创建组合订单。
- 为机票、酒店、租车和保险创建子订单。
- 每个子订单记录供应商、报价、付款策略、可逆性和幂等键。

核心风险：

- 组合订单创建成功但子订单缺失。
- 同一用户请求重复创建多个组合订单。
- 子订单没有独立状态机，导致补偿不可定位。
- 付款策略和可逆性没有快照。

### 预授权、扣款、释放授权和退款

场景边界：

- 系统按子产品付款策略决定预授权、立即扣款、到店付或不可退规则。
- 资金动作必须和供应商动作解耦。
- 捕获、释放授权、退款、罚金和补差都要进入账本。

核心风险：

- 供应商失败后忘记释放授权。
- 已扣款后供应商失败但没有退款或人工处理。
- 同一子订单重复 capture 或重复退款。
- 罚金和补差没有分录。

### 机票出票和酒店确认的核心 Saga

场景边界：

- 机票和酒店是核心项。
- 机票可能先锁座再出票，酒店可能先占房再确认。
- 核心项都成功后，组合订单进入核心成功。
- 任一核心项失败，必须考虑另一个核心项是否已经不可逆。

核心风险：

- 机票已出票但酒店失败。
- 酒店已确认但机票失败。
- 机票或酒店供应商返回未知状态。
- 一端成功、一端失败时错误地把整单标记为失败而不处理已成功事实。

### 租车和保险附加项处理

场景边界：

- 租车和保险是附加项。
- 核心项成功后，附加项可以继续确认。
- 附加项失败不取消核心行程。
- 附加项失败进入退款、替代供应商、用户确认或人工处理。

核心风险：

- 保险投保失败但订单显示已投保。
- 租车失败却取消了机票酒店。
- 保险已生效后被当作可删除记录。
- 附加项退款和组合订单状态不一致。

### 核心失败、附加失败、部分成功和人工处理

场景边界：

- 组合订单可能进入 `PARTIALLY_CONFIRMED`、`CONFIRMED_WITH_ADDON_FAILURES` 或 `MANUAL_REVIEW`。
- 系统必须解释每个子订单最终状态。
- 用户或运营可能参与接受替代、补差、取消或人工退款。

核心风险：

- 只有总订单状态，没有子订单事实。
- 部分成功状态无法对用户、客服或财务解释。
- 人工处理直接改历史。
- 用户决策没有审计证据。

### 供应商、支付、退款、罚金、补差和本地订单对账

场景边界：

- 对比本地组合订单、子订单、供应商订单、支付单、退款单、罚金、补差、会计分录和 Outbox 消费记录。
- 每个差错必须有分类、owner、修复动作和审计记录。

核心风险：

- 本地确认但供应商没有订单。
- 供应商确认但本地订单未推进。
- 供应商取消产生罚金但本地没有分录。
- 附加项失败已退款但组合订单仍显示完整成功。

## 服务边界

第一版采用这些逻辑服务：

- `booking-service`：组合订单、子订单、组合状态机、用户决策、幂等键。
- `supplier-adapter-service`：机票、酒店、租车、保险供应商请求、回调、查询和供应商事实。
- `payment-service`：预授权、capture、void/release、扣款、支付未知状态。
- `refund-service`：退款、部分退款、罚金退差、退款未知状态。
- `ledger-service`：支付、退款、罚金、补差、冲正和差错修复分录。
- `reconciliation-service`：本地订单、供应商账单、支付、退款和账本对账。
- `manual-review-service`：高风险差错、用户决策、运营审批和 maker-checker。
- `message-broker`：Outbox 事件投递和异步解耦，不代表业务事实本身。

关键边界：

- `booking-service` 拥有组合订单状态，但不拥有供应商事实或资金事实。
- `supplier-adapter-service` 记录供应商事实，但不直接决定资金入账或退款。
- `payment-service` 和 `refund-service` 记录资金渠道事实，但不改写供应商订单历史。
- `ledger-service` 只追加分录或冲正分录，不删除历史。
- `reconciliation-service` 只分类、路由和审计差错，不直接修改领域事实。
- `manual-review-service` 记录人工决策，不绕过状态机和审计。

## 状态机

### 组合订单状态

```text
QUOTE_SELECTED
-> BOOKING_CREATED
-> PAYMENT_AUTHORIZED
-> CORE_BOOKING_IN_PROGRESS
-> CORE_CONFIRMED
-> CONFIRMED
```

部分成功路径：

```text
CORE_CONFIRMED
-> ADDON_BOOKING_IN_PROGRESS
-> CONFIRMED_WITH_ADDON_FAILURES
```

失败和人工路径：

```text
CORE_BOOKING_IN_PROGRESS
-> PARTIALLY_CONFIRMED
-> COMPENSATING
-> REFUNDING
-> MANUAL_REVIEW or CANCELLED
```

关键规则：

- 核心项没有全部成功时，组合订单不能进入 `CONFIRMED`。
- 附加项失败不能把核心成功订单改成 `FAILED`。
- `PARTIALLY_CONFIRMED` 必须能解释每个子订单状态和资金状态。
- `MANUAL_REVIEW` 不能直接覆盖供应商事实或资金事实。

### 子订单状态

```text
CREATED
-> QUOTE_LOCKED
-> SUPPLIER_PENDING
-> SUPPLIER_CONFIRMED
-> APPLIED_TO_BOOKING
```

失败和补偿状态：

```text
SUPPLIER_UNKNOWN
SUPPLIER_FAILED
CANCEL_REQUESTED
CANCELLED
CANCEL_UNKNOWN
REFUND_REQUIRED
MANUAL_REVIEW
```

关键规则：

- `SUPPLIER_UNKNOWN` 不是失败，不能直接释放资金或取消成终态。
- `SUPPLIER_CONFIRMED` 不能被失败回调覆盖。
- 半不可逆或不可逆子订单失败后只能取消、退款、罚金、补差或人工处理。
- 每个子订单必须记录付款策略和可逆性快照。

### 付款状态

```text
PAYMENT_CREATED
-> AUTHORIZED
-> CAPTURE_REQUESTED
-> CAPTURED
```

释放和退款路径：

```text
AUTHORIZED
-> VOID_REQUESTED
-> VOIDED
```

```text
CAPTURED
-> REFUND_REQUIRED
-> REFUNDING
-> REFUNDED
```

异常状态：

```text
PAYMENT_UNKNOWN
VOID_UNKNOWN
REFUND_UNKNOWN
MANUAL_REVIEW
```

关键规则：

- 预授权成功不等于扣款成功。
- capture、void 和 refund 都必须幂等。
- 已 capture 的资金不能通过 void 释放，只能退款或人工处理。
- `REFUND_UNKNOWN` 不能重复提交外部退款。

## 不变量

组合订单：

- 同一幂等键只能创建一个组合订单。
- 每个组合订单必须有可解释的子订单集合。
- 组合订单 `CONFIRMED` 必须要求机票和酒店核心项成功。
- 附加项失败只能进入附加失败、替代、退款或人工处理路径，不能覆盖核心成功事实。
- `PARTIALLY_CONFIRMED` 必须有明确 owner、下一步动作和审计记录。

子订单和供应商：

- 每个子订单必须记录供应商请求号、供应商事实、付款策略和可逆性。
- `SUPPLIER_UNKNOWN` 不能直接当失败。
- 供应商成功事实不能被失败回调覆盖。
- 不可逆动作不能被数据库字段回滚掩盖。

资金：

- 预授权、capture、void、refund、罚金和补差必须有幂等键。
- 同一授权只能 capture 或 void 一次。
- 已 capture 资金不能 void，只能 refund、冲正或人工处理。
- 退款金额、罚金和补差必须能被原支付、供应商规则和会计分录解释。
- 资金渠道事实必须最终有本地订单状态和会计分录对应。

事件和对账：

- Outbox 事件必须和本地事实在同一服务事务中提交。
- broker offset 不能证明供应商确认、付款、退款或补偿完成。
- 组合订单、子订单、供应商事实、支付、退款、罚金、补差和会计分录最终必须能对账。
- 修复动作必须有幂等键、owner、审批或审计记录。

## 失败矩阵重点

第一版必须覆盖这些高风险失败：

- 报价过期但继续按旧价扣款。
- 预授权成功但机票锁座失败。
- 机票已出票但酒店确认失败。
- 酒店已确认但机票出票失败。
- 供应商请求超时或未知。
- 核心项部分成功后用户取消。
- 租车失败但核心行程成功。
- 保险投保失败或投保未知。
- 保险已生效后收到失败回调。
- capture 成功但供应商确认失败。
- void/release 授权失败。
- 退款提交超时或未知。
- 罚金或补差分录缺失。
- Outbox 发布失败。
- 消费者重复消费。
- 人工修复失败或重复提交。

恢复原则：

- 先确认供应商事实和资金事实，再决定补偿动作。
- 未知状态只能查询、等待回调、对账或人工处理，不能直接失败。
- 核心项失败时，必须检查另一核心项是否已经成功或不可逆。
- 附加项失败时，默认保留核心成功，处理附加退款、替代或人工流程。
- 已发生的供应商、支付、退款和账本事实不能删除，只能追加解释事实。

## 对账设计

对账类型：

- 组合订单对账：组合状态能否解释所有子订单状态。
- 供应商对账：本地子订单和供应商订单、出票、酒店确认、租车确认、保险生效是否一致。
- 支付对账：预授权、capture、void、refund 和支付渠道账单是否一致。
- 账本对账：支付、退款、罚金、补差和冲正分录能否解释业务状态。
- 事件对账：Outbox 事件和消费者处理记录是否覆盖本地事实。

典型差错：

- 本地核心成功，但供应商缺少机票或酒店确认。
- 供应商确认成功，但本地组合订单未推进。
- 附加项失败已退款，但组合订单没有附加失败状态。
- 用户已扣款，但核心项失败后未退款。
- 供应商取消产生罚金，本地缺少罚金分录。
- 预授权长期未 capture 或 void。
- 退款成功但订单仍处于退款中。

修复原则：

- `reconciliation-service` 只分类和路由，不直接改订单、供应商、支付或账本历史。
- 自动修复必须通过领域服务的幂等命令。
- 高风险修复需要 maker-checker 或人工复核。
- 修复完成后必须重新对账。

## 验证路线

模型验证：

- 核心项未全部成功时，组合订单不能 `CONFIRMED`。
- 附加项失败不能取消核心成功事实。
- `SUPPLIER_UNKNOWN`、`PAYMENT_UNKNOWN`、`REFUND_UNKNOWN` 不能被直接当失败。
- 已 capture 资金不能 void。
- 不可逆供应商成功事实不能被失败回调覆盖。
- 补偿不能删除历史事实。

属性测试：

- 随机生成报价过期、供应商成功/失败/未知、支付授权/capture/void/refund、用户取消、附加项失败、人工修复等序列。
- 检查每个组合订单最终进入确认、附加失败、取消、退款或人工处理中的可解释终态。
- 检查每个子订单最多产生一次供应商业务效果。
- 检查每个资金动作最多产生一次业务效果。
- 检查罚金、补差和退款金额能被供应商规则和原支付解释。

集成测试：

- 幂等键和唯一约束。
- 状态转换版本号或条件更新。
- 本地事务和 Outbox 同提交。
- 供应商回调、查询和人工修复重复到达。
- 核心项部分成功后的补偿路径。
- 附加项失败后的保留核心成功路径。

故障注入：

- 预授权后供应商失败。
- 机票出票成功后酒店失败。
- 酒店确认成功后机票失败。
- 供应商超时但稍后成功。
- capture 成功但本地订单未推进。
- refund unknown 后重复修复请求。
- Outbox 发布失败和消费者重复消费。
- 对账账单缺失或金额不一致。

## 面试和架构评审表达

核心表达：

> 旅行组合预订不能靠一个大事务保证一致性。机票、酒店、租车、保险来自不同供应商，动作可逆性和付款策略不同。系统必须把核心项和附加项分层，用 Saga 编排供应商动作，用状态机记录每个子订单事实，用付款策略决定预授权、扣款、释放授权和退款，用对账和人工处理收敛不可逆或未知状态。

高频追问：

- 为什么不能全有全无？
- 机票已出票但酒店失败怎么办？
- 核心成功但租车或保险失败怎么办？
- 预授权、capture、void 和 refund 分别解决什么？
- 供应商超时能不能当失败？
- 不可退酒店或已生效保险怎么补偿？
- Temporal 在这里解决什么？

回答底线：

- 不接受只用总订单成功/失败表达组合状态。
- 不接受把搜索价当作最终成交价。
- 不接受供应商超时直接失败。
- 不接受不可逆动作被本地字段回滚掩盖。
- 不接受已扣款失败后没有退款、罚金、补差或人工处理路径。
- 不接受附加项失败拖垮核心成功行程。
- 不接受没有供应商、支付、退款、账本和本地订单对账。

## 技术路线

主线仍采用 Java 生态：

- Spring Boot 表达服务边界和本地事务。
- Kafka + Outbox 表达跨服务事实传播。
- Temporal 表达长流程 Saga、定时查询、超时、重试和人工处理入口。
- 数据库唯一约束和条件更新表达幂等和状态机裁决。
- 属性测试、模型验证、故障注入和对账验证作为核心学习目标。

Temporal 在本阶段解决的是长流程编排，不替代：

- 供应商事实记录。
- 子订单状态机。
- 付款和退款幂等。
- Outbox。
- 会计分录。
- 对账和人工修复。

## 成功标准

本阶段完成后，学习者应该能：

- 解释旅行组合预订为什么比电商订单更难。
- 区分核心项和附加项的业务含义。
- 按子产品说明付款策略和可逆性。
- 画出组合订单、子订单、供应商请求、支付和退款状态机。
- 说明核心失败、附加失败和部分成功如何处理。
- 说明为什么补偿不是回滚。
- 设计供应商、支付、退款、罚金、补差和账本对账。
- 设计验证方法捕获未知状态、部分成功、重复外部动作和人工修复错误。
