# 设计 spec · `concurrency-capacity/` 并发容量工程

> 状态：已 brainstorm 对齐,待写实现计划。
> 日期：2026-06-23
> 来源对话：用户痛点——「我懂并发原理了,但不知道大公司高并发系统到底怎么用这些技术」。

---

## 1. 一句话定位

把一个业务数字(*"10k RPS, P99 < 200ms"*)翻译成**站得住脚的配置**(这个并发模型、这种规格的机器、这几个池各多大、把这两类活隔离、盯这三个信号)——并且**知道每个数字为什么是这个数字**。

这是「我会用并发原语」到「我能给系统定容」之间那一层工程学。

## 2. 为什么是一门新课(不是扩已有课)

仓库已有的相邻材料各差一截,**这一层散落在三门课之间、从未被拼成一条决策管线**:

| 已有 | 教什么 | 为什么不是这门 |
|---|---|---|
| `python-concurrency/09-patterns-tuning` | 工具:Semaphore/池/重试熔断/舱壁/fan-out | 给**武器**不给**用兵法**;说"池大小最终压测定",给了旋钮没给「事前怎么算出这个数」 |
| `performance-tuning-roadmap/` | 剖析、压测、调试(py-spy/pprof/wrk) | **反应式**——出事后怎么找瓶颈;用户问的是**事前**怎么定容 |
| `system-design/` | 宏观分布式架构(L0–L9) | 高一个高度——分片/共识/CAP,不是「这一个服务开几个线程」 |
| `golang/concurrency`、`java/concurrent` | 语言原语 | 用户自陈这层已掌握 |

缺的正是中间这层:**容量与并发工程**。

## 3. 受众与深度(硬要求)

- **等级**:资深工程师 → 架构师。与 `system-design/`、`software-architecture/` 同一寄存器。
- **语言**:**语言中立的主线**;每个具体旋钮给 **Python(领衔,面试向)+ Go + Java** 三语对照。
- **深度契约(可证伪,不是口号)**:对每条「人人会背的经验法则」,都补上**它底下的排队论 / 运行时机理**那一层,让用户能从第一性原理**推导**出数字,而不是记忆启发式。这是这门课的核心价值,**不许 defer 到「以后」**——底层进正文(narrative),不丢给问答题。

### 3.1 深度契约逐章清单

| 章 | 人人会背的经验法则 | 本课要补的「内部层」 |
|---|---|---|
| 01 | "P99 一上量就飙" | **M/M/1 响应时间 `R = S/(1−ρ)`**——`1/(1−ρ)` 这一项**就是**那根曲棍球杆;**Kingman/VUT**——为什么方差(C²)让你在均值还没到时就饱和;**coordinated omission**——为什么你的压测低报了尾延迟 |
| 02 | "async 管 I/O,线程更贵" | 上下文切换开销 + 每单位内存(线程栈 vs goroutine vs 协程帧),**C10k** 源流,GIL 的精确释放点,Go 的 **M:N** park-the-M 机理 |
| 03 | "估一下你的 QPS" | 峰均比,**fan-out 尾部放大**(Dean & Barroso *tail-at-scale*:1 请求=N 调用 → 最慢那个的 P99 主导),重尾服务时间(为什么均值骗人) |
| 04 | "线程数 = 2×核数+1" | 从 Little 推导:**`线程数 = 核数 × (1 + 等待/计算)`** @ 目标 CPU——2N+1 只是其特例;**Universal Scalability Law**——σ(争用)+ κ(协调)项,为什么吞吐会**回退**而非仅是平台 |
| 05 | "用连接池" | 池 = **M/M/c 队列**;**HikariCP / PostgreSQL「小池更快」** 结论及其原因(DB 自身并行度才是真瓶颈);`×worker×副本` vs `max_connections` 的总账 |
| 06 | "用舱壁隔离" | **队头阻塞(HoL)** → 线程池耗尽级联;**信号量隔离 vs 线程隔离** 取舍(Hystrix);**shuffle sharding**(AWS)作为进阶形态 |
| 07 | "加重试和超时" | 有界队列 *shed vs block*,**重试风暴的乘性放大**,**deadline 传播**,过载下 LIFO,**metastable failure**(近年分布式系统文献) |
| 08 | "监控 CPU" | 利用率在尾部**会骗人**(方差);**饱和度=队列深度**才是领先指标;**USE vs RED**;并发 ≠ 速率;为什么不能对 P99 求平均 |
| 09 | "不够就加机器" | **USL** 决定纵向天花板;纵向 vs 横向的经济学;按**并发/队列**而非 CPU 自动扩缩;每 RPS 成本曲线 |

每条都在正文里讲;大多数在 `lab/sim/` 里有可跑的证明(如 `little.py` 画出 `R=S/(1−ρ)` 爆炸、`saturate.py` 演示 USL 回退)。**深度可跑,不止可读。**

## 4. 主线:Little 定律作为总钥匙

每一章都挂在 **Little 定律 `L = λ × W`** 上。用户不是遇到 12 个互不相干的话题,而是遇到**一个**方程,看着每个生产问题从它身上掉下来:

- *测算并发需求* → `λ × W` 直接给出
- *池多大 / 多少线程* → 所需并发 ÷ 单 worker 容量
- *单机配置* → 利用率在哪饱和(USE)
- *是否隔离* → 哪些工作负载共享一个池而互相饿死
- *是否监控* → 利用率/饱和度/队列深度就是这个方程的实时读数

一条主线 = 系统性(用户要求一)+ 阅读流畅性(用户要求二)。

| 章 | 标题 | 干掉的问题 |
|---|---|---|
| **00** | 全景:从需求到配置的决策管线 | *框定整条管线*——需求→模型→单机→池→隔离→过载→监控→扩容 |
| **01** | Little 定律:这门课的总钥匙 | *你自陈缺的「系统性认知」*——`L=λW`、利用率 ρ、饱和曲线、为什么 P99 会拐 |
| **02** | 三种并发模型 | **正确的并发模型怎么选**——thread-per-req / event-loop / 多进程worker;GIL/runtime 如何逼你选 |

| **04** | 单机配置与线程/worker 数 | **多少线程 + 单机配多大**——I/O 等待比公式,2×核+1/≈核 的由来,USE 找饱和点 |
| **05** | 池:哪些要池、池多大、×worker 总账 | **哪些需要 pool、pool 多大**——池=稀缺资源前的队列,连接池=对资源套 Little,总账坑 |

### 5.1 排序理由(流畅性)

先给法(01)→ 给设计空间(02)→ 给输入 λ,W(03)→ 单节点定容(04 worker、05 池、06 隔离)→ 过载行为(07)→ 在线观测(08)→ 然后才把单节点乘成机群(09)。先彻底配好一个节点,再去乘。

## 6. 学习环境(两层 lab)

用户明确要「构建学习环境」。本主题尤其需要**看见**——饱和与 Little 定律是看着 P99 曲棍球杆和吞吐平台才信的,读公式不一样。

```
concurrency-capacity/lab/
  sim/                    # 离线、纯 stdlib、确定性 —— 「无基建地板」
    little.py             # 拨 λ,W → 打印 L, ρ, 队列, P99 曲线            (ch01)
    saturate.py           # 扫负载 → 吞吐平台 + 延迟拐点 + USL 回退        (ch01/04)
    starve.py             # 池=N → 等待时间爆炸                          (ch05)
    bulkhead.py           # 共享池 vs 隔离池                            (ch06)
  service/                # 真实 FastAPI,旋钮走 env(模型、池大小)
    app.py                # /fast /slow /cpu 端点
    drive.py              # 拉升 λ,记录 P50/P99/QPS
    compose.yaml          # 服务 + 压测 + 指标(复用 fastapi-ops 基建)
  experiments/            # 每章一份 runbook——跑什么、会看到什么、为什么
    e02-model-shootout.md
    e04-find-saturation.md
    e06-bulkhead.md
    e07-overload-collapse.md
    ...
```

- **floor 层(sim/)**:纯 Python、零基建、确定性,每章都能引用,day-one 可跑。
- **real 层(service/)**:一个带旋钮的小 FastAPI,真负载下跑脚本化实验,复用仓库已有的 `fastapi-ops/` 与 `performance-tuning-roadmap/03x-load-gen-quickstart/` 基建。
- 每章正文以 **「动手:跑这个实验」** 盒子收尾,指进 `lab/`,把「读」和「看见」焊在一起。
