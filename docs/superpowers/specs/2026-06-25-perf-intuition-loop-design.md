# 性能直覺養成閉環 — 設計 spec

> 日期:2026-06-25 · 分支:`worktree-perf-domain-map`(base `23e633e`)
> 一句話:把「我整個職涯都缺這些」的焦慮,用 **診斷 → 親手跑通 → 框架** 的閉環打掉;不是再加一份讓人讀的文檔。

## 緣起與真正要解決的問題

對話從「P99 是什麼」「QPS/TPS/RPM 區別」開始,升級成使用者的真實困惑:*「我整個 IT 職涯都很缺這些,學校也沒教,這些到底在哪學的?我想要真的能解決問題的教程。」*

反思後鎖定:這份困惑有三層,而**地圖只能解第二層**。

| 層 | 困惑 | 誰能解 |
|---|---|---|
| 1 表層 | 名詞是什麼意思(P99/QPS) | 對話已解 |
| 2 中層 | 有一整門學科我不知道它的形狀 | 領域地圖能解 |
| 3 真正的焦慮 | 我到底**是不是真的懂系統**?還是只在收集名詞? | **只有「親手做過一次」能解** |

**關鍵診斷(使用者自己早就發現過):** 他有比多數工程師都多的文檔,卻還是覺得缺——瓶頸從來不是缺文檔。再加一份完美文檔,只會餵養製造焦慮的循環:讀→「看懂了」→還是不會做→焦慮。他自己的 `LEARNING-GUIDE`(「没有产物…看懂了但不会做」)、`roadmap_practical` 方案A(「真跑出数字…文档只回填锚点」)、多個 track 的「待用真 key 跑/实机待跑」都印證了這個模式:**鷹架搭到完美,在「真跑一次」前停下。**

所以本設計刻意把「讀」降為配角,把「做」放正中央。

## 成功標準

- 使用者**親手跑出自己的 P99 曲線**(`little.py` 已驗證:ρ 0.9→0.99 時 R 100ms→1000ms 曲棍球桿)。
- 焦慮從「我什麼都不懂」搬到「我懂很多,只是沒做過」——靠診斷的具體座標達成。
- 留下產物(五句話記錄 + 自己的曲線),不是「看完就忘」。

## 設計:三件式,全部放 `performance-tuning-roadmap/`

複用現有 lab 與 rubric,**不重寫、不造新課程**。三件薄連接件。新意只在「3 層診斷視角」+「單次引導閉環」。

### 件 1 — 診斷自測 `SELF-DIAGNOSTIC.md`
- 6 條原語 × 3 欄:**能講 / 能算 / 能在真機上做**,約 18 題,自評 ✓/✗ 填成 6×3 網格。
- 設計假設=療效:他會看見「能講」幾乎全 ✓、「能在真機上做」幾乎全 ✗。
- 連回 `SENIOR-RUBRIC.md` / `14-capstone/03-self-assessment.md`,不重複造。

### 件 2 — 引導實戰閉環 `FIRST-LOOP.md`(心臟)
- 一次實驗,30–45 分鐘,產出**使用者自己的 P99 曲線**。
- 每步格式:`跑什麼 / 你會看到什麼(附已驗證真實輸出) / 為什麼 / 證明 6 原語裡哪一條`。
- **第一幕**:`concurrency-capacity/lab/sim/little.py`(零基建、已驗證可跑)→ 落原語 1/2/5。
- **第二幕**:真 FastAPI 服務 `concurrency-capacity/lab/service/`(需 Docker)+ 壓測 → 看 RED/USE、找一個瓶頸、修、再測 → 落原語 3/4 + 閉合「觀察→假設→修→驗證」。建構時須親手先跑通並捕捉輸出。
- 收尾:用 `LEARNING-GUIDE` 的五句話格式留產物。

### 件 3 — 領域地圖 `README.md`(框架,降級為腳手架)
- 6 原語敘事(why / 心智模型 / 最常見的坑 + 連深礦):①延遲=百分位 ②吞吐=工作單位/時間 ③RED ④USE ⑤Little定律+飽和曲線 ⑥SLI/SLO+error budget。分組:1+2 測什麼 · 3+4 怎麼看 · 5 為什麼爆 · 6 多少算違約。
- 「人們在哪學的」+ 正典書單(Gil Tene「How NOT to Measure Latency」/ DDIA Ch.1 / Google SRE Book / Brendan Gregg《Systems Performance》)。
- 4-track 分工表(perf-roadmap / concurrency-capacity / observability / fastapi-ops)。
- **路由門面**:想要全貌→讀地圖;想知道缺哪→做診斷;想真的會→跑 FIRST-LOOP。

## 使用順序

診斷(15min,焦慮落地)→ FIRST-LOOP(30–45min,親手畫曲線)→ 地圖(把經驗掛回整門學科)。

## 範圍邊界(YAGNI)

- 不重教各 track 細節、不造新課程、不複製 rubric/self-assessment。
- 6 原語維持 6 條(成本/緩解模式分別住 `09-scaling-out`/`11-architecture`,地圖指過去即可)。
- 語言:三件全繁體(使用者選定,雖與 track 既有簡體文檔混用)。

## 決策記錄

- 放 `performance-tuning-roadmap/` 頂層(它原本連 README 都沒有,順手補上門面)。
- 隔離:在 worktree `worktree-perf-domain-map` 上做,避開另一並發任務直接提交本地 `main` 的 git add 陷阱。
- 流程:輕量。本 spec 當錨點;不另起多步驟 implementation plan(對三份 markdown 過度,且違使用者偏好)。
