# 金融级分布式事务场景矩阵设计

日期：2026-05-01

## 目标

在 `financial-consistency/` 下新增一个场景矩阵，系统回答“真实世界常见的分布式事务场景有哪些，以及每类场景应该用什么一致性思路分析”。

这个矩阵不是目录清单，也不是框架名词表。它要把业务场景、正确性风险、事务模式、补偿能力、外部系统不确定性、对账要求和后续学习模块连接起来，让学习者先建立全局地图，再逐个展开完整模块。

## 背景

当前 `financial-consistency/` 已经有完整展开的 `01-transfer` 模块，也在总入口中列出了后续方向：

- 充值、提现、支付回调。
- 电商下单、支付、库存、取消、退款。
- 机票、酒店、租车、保险组合预订。
- Outbox、TCC、Saga、Temporal、事务消息、CDC。
- 验证实验、对账和人工补偿。

但这些内容目前主要存在于路线图中，尚未形成可横向比较的场景地图。用户当前目标是从真实场景出发，系统学习金融级一致性实现，而不是只掌握单一转账例子。

## 文件设计

```text
financial-consistency/00-scenario-matrix.md
```

更新文件：

```text
financial-consistency/README.md
```

`00-scenario-matrix.md` 放在 `01-transfer` 之前，作为所有具体场景模块的导航层。`README.md` 在阶段路线中加入该文件链接，并说明矩阵和后续模块的关系。

## 矩阵结构

每个场景按同一组字段分析：

| 字段 | 目的 |
| --- | --- |
| 场景 | 真实业务问题，而不是技术名词 |
| 典型业务动作 | 这个事务链路实际做了什么 |
| 一致性难点 | 最危险的资金、库存、状态或外部系统风险 |
| 外部系统 | 是否依赖银行、支付渠道、航司、酒店、物流等不可控系统 |
| 可逆性 | 动作是否能自然撤销，还是只能业务补偿 |
| 推荐模式 | Outbox、Saga、TCC、Temporal、对账、人工处理等组合 |
| 核心验证 | 必须检查的不变量、故障注入或对账项 |
| 后续模块 | 应该展开到哪个学习模块 |

这个结构的重点是强迫每个场景回答两个问题：

1. 这个业务到底怕什么错？
2. 这个错误靠事务、补偿、对账、人工处理还是验证体系解决？

## 第一版覆盖场景

第一版矩阵覆盖 14 类场景：

1. 账户转账。
2. 充值。
3. 提现。
4. 支付回调。
5. 电商下单、支付、库存扣减。
6. 订单取消。
7. 退款。
8. 优惠券、积分、余额组合支付。
9. 机票、酒店、租车组合预订。
10. 外部渠道超时或未知状态。
11. 跨境汇款。
12. 多币种结算。
13. 对账发现单边账。
14. 人工补偿与运营修复。

覆盖顺序遵循两个原则：

- 从资金内核到外部渠道，再到组合交易和跨境结算。
- 从可强约束的内部服务，到只能最终一致和人工兜底的外部系统。

## 场景分层

矩阵中的场景按复杂度分为四层。

### L1 资金内核

代表场景：

- 账户转账。
- 充值。
- 提现。
- 退款。

目标是掌握账户、冻结、扣减、入账、分录、幂等、状态机、Outbox 和对账。

### L2 外部渠道

代表场景：

- 支付回调。
- 外部渠道超时或未知状态。
- 渠道对账。

目标是掌握外部系统不可控、回调重复、查询补偿、未知状态收敛和渠道账单核对。

### L3 业务资源组合

代表场景：

- 电商下单、支付、库存扣减。
- 订单取消。
- 优惠券、积分、余额组合支付。
- 机票、酒店、租车组合预订。

目标是掌握 Saga、TCC、资源预留、可补偿动作、不可逆动作和长流程编排。

### L4 清结算与运营闭环

代表场景：

- 跨境汇款。
- 多币种结算。
- 对账发现单边账。
- 人工补偿与运营修复。

目标是掌握汇率、轧差、清算、差错处理、人工审核、审计和监管友好的证据链。

## 模式选择原则

矩阵不把任何框架当作银弹。每个场景都按业务属性选择模式组合：

- 单服务内部一致性优先使用本地数据库事务和唯一约束。
- 跨服务事实发布优先使用 Outbox 或等价的可靠事件发布机制。
- 短时可预留资源使用 TCC 思维，例如冻结、Try/Confirm/Cancel。
- 长流程、可重试、需要定时器和人工处理入口的流程使用 Saga/Temporal。
- 外部系统动作不可逆时，不承诺技术回滚，只设计业务补偿、退款、赔付、人工处理和对账。
- 所有资金相关链路都必须有对账，不能把对账当作上线后的补丁。
- 所有核心场景都必须定义不变量，并在后续模块中映射到属性测试、集成测试、故障注入或历史检查。

## 与现有模块的关系

`00-scenario-matrix.md` 是导航层，不替代具体模块。

已经完成的 `01-transfer` 继续作为第一个完整展开案例。矩阵中的“账户转账”行应链接到 `01-transfer/README.md`。

后续模块从矩阵中逐个展开：

- `02-payment-recharge-withdraw/` 展开充值、提现、支付回调、渠道未知状态。
- `03-order-payment-inventory/` 展开电商下单、支付、库存、取消、退款。
- `04-travel-booking-saga/` 展开机票、酒店、租车和保险组合预订。
- `05-patterns/` 对比 Outbox、TCC、Saga、Temporal、Seata、Camunda、事务消息和 CDC。
- `06-verification-lab/` 把矩阵中的核心验证项变成可执行验证路线。
- `07-reconciliation/` 展开对账、差错分类和人工修复。

## 非目标

第一版不写每个场景的完整状态机、事件流和失败矩阵。

这些内容属于具体场景模块。矩阵只负责横向比较和优先级导航，避免一个文件膨胀成大而全的教程。

第一版不引入真实代码实现。

这个步骤仍然是学习文档组织层，代码实现应在具体模块的设计、计划和验证路线明确后推进。

## 验收标准

完成后应满足：

- `financial-consistency/00-scenario-matrix.md` 存在。
- `financial-consistency/README.md` 能从总入口链接到场景矩阵。
- 矩阵至少覆盖 14 类真实场景。
- 每个场景都有一致性难点、外部系统、可逆性、推荐模式、核心验证和后续模块。
- 矩阵明确说明它不是完整教程，而是后续模块的导航层。
- 文档内容完整，不留下空白段落或含糊要求。
