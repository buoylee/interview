# 07 统一对账闭环设计

日期：2026-05-02

## 目标

创建 `financial-consistency/07-reconciliation/`，把前面所有阶段里反复出现的对账、差错处理、人工修复和审计闭环系统化。

这一阶段不是只讲支付渠道账单，也不是直接实现一个 `reconciliation-service`。它的目标是让学习者理解真实金融系统中“发现事实不一致并把它收敛成可解释事实”的完整过程：准实时对账、日终对账、T+N 对账、差错分类、修复命令、maker-checker 审批、复核、关闭原因、审计和报表。

完成后，学习者应该能回答：

- 对账为什么是金融系统核心链路，而不是实现完成后的补丁？
- 对账系统读取哪些事实源，哪些东西不能当事实源？
- 准实时对账、日终对账、T+N 对账和专项重跑分别解决什么问题？
- 差错应该如何分类，而不是简单写成“金额不一致”？
- 修复为什么不能直接改平数据，而必须追加修复事实、审批和复核？
- 转账、充值提现、电商下单、旅行组合预订分别如何进入统一对账闭环？
- 如何验证对账系统本身不会制造新的不可解释事实？

## 背景

已有阶段已经反复触及对账：

- `01-transfer`：内部转账需要账户流水、account movement、ledger posting 和人工冲正。
- `02-payment-recharge-withdraw`：充值、提现、支付回调和退款必须依赖渠道查询、渠道账单和对账差错收敛 unknown。
- `03-order-payment-inventory`：订单、支付、库存、退款、发货和商户结算之间需要业务事实和资金事实互相解释。
- `04-travel-booking-saga`：供应商迟到成功、迟到失败、已出票、已生效保险、罚金和补差都需要对账和人工处理闭环。
- `05-patterns`：已经明确对账不是一致性模式失败后的补丁，而是金融级设计的一部分。
- `06-verification-lab`：已经定义 Fact-first 验证、不变量、oracle、异常 History、故障注入和历史回放。

`07-reconciliation` 的作用是把这些分散在不同场景里的对账概念统一起来，形成一个真实工程可用的对账闭环模型。

## 范围

第一版覆盖：

- 对账模型：Fact、Statement、Difference、Case、Repair、Review、Close。
- 对账数据源：领域事实、账本事实、渠道账单、供应商账单、库存事实、订单事实、清结算事实、人工审批事实。
- 对账类型：准实时对账、日终对账、T+N 对账、专项重跑、人工复核。
- 差错分类：本地少记、本地多记、外部多扣、外部少扣、状态不一致、账本不平、退款 unknown、供应商迟到、库存与订单不一致、结算差异。
- 修复闭环：差错工单、证据快照、修复命令、maker-checker、修复事实、复核、关闭原因。
- 场景矩阵：内部转账、充值提现、电商下单、旅行组合预订。
- 验证与审计：如何验证对账系统本身，如何输出监控、报表和审计证据。
- 面试和架构评审表达。

## 非目标

第一版不做这些事情：

- 不实现 Java、Go 或其他可运行服务。
- 不创建数据库表、调度器、消息消费者或管理后台。
- 不接入真实支付渠道、供应商、Kafka、Temporal、Flink、Spark、数据库或报表系统。
- 不把对账简化成 SQL diff 工具教程。
- 不提供具体厂商或框架 API 教程。
- 不允许对账脚本直接改写领域事实、账本事实或外部事实。
- 不把对账替代 06 阶段的不变量、oracle、故障注入和历史回放。

后续如果进入工程实现，可以单独设计 `reconciliation-service`、差错工单表、修复命令表、审计表、任务调度和可观测性。

## 核心模型

### 对账对象

| 对象 | 含义 | 示例 |
| --- | --- | --- |
| Fact | 系统已经持久化、可审计的业务或资金事实 | 账本分录、支付单、供应商订单、库存流水、人工审批 |
| Statement | 外部或内部对账来源提供的陈述 | 渠道账单、供应商账单、清算文件、库存快照 |
| Difference | Fact 与 Statement 或不同 Fact 之间的差异 | 渠道成功但本地 unknown、账本借贷不平 |
| Case | 差错工单，承载证据、分类、责任边界和处理状态 | `CASE_PAYMENT_LOCAL_MISSING` |
| Repair | 修复命令和修复事实 | 补记账本、退款、冲正、调整、补差、罚金 |
| Review | maker-checker 审批和复核 | 审批通过、复核失败、重新打开 |
| Close | 差错关闭事实 | 已修复关闭、无需修复关闭、长期挂起 |

对账的主语不是“把两张表查平”，而是“把一组事实差异变成可解释、可审计、可复核的处理闭环”。

### 不可变边界

对账系统必须继承 06 阶段的 Fact-first 原则：

- 日志、workflow history、broker offset、consumer offset 和任务运行记录只能说明执行过程，不能替代业务事实、账本事实、渠道事实、供应商事实或人工审批事实。
- 对账不能删除历史事实，也不能用本地状态覆盖外部成功或外部失败。
- `PAYMENT_UNKNOWN`、`WITHDRAW_UNKNOWN`、`REFUND_UNKNOWN` 和 `SUPPLIER_UNKNOWN` 不能被对账脚本直接判失败。
- 已 capture 资金不能 void；只能通过 refund、reversal、adjustment posting、差错工单或人工处理闭环。
- 已出票、已确认供应商订单和已生效保险不能被本地失败、取消字段或对账脚本删除。
- 修复必须追加 Repair Fact、审批 Fact、复核 Fact 和关闭 Fact。

## 数据源设计

`02-reconciliation-sources.md` 必须把数据源分成可判定事实源和辅助线索。

可判定事实源：

- 领域事实：订单、支付请求、退款请求、提现请求、供应商请求、库存预留、发货、售后。
- 账本事实：account movement、ledger posting、调整分录、冲正分录、日切批次。
- 外部事实：支付渠道流水、退款流水、提现结果、渠道账单、供应商订单、供应商账单。
- 清结算事实：商户结算、平台手续费、供应商结算、资金划拨、清算文件。
- 人工事实：差错工单、证据快照、maker-checker 审批、修复命令、复核结果、关闭原因。

辅助线索：

- 日志、trace、workflow history、broker offset、consumer offset、任务实例、报表生成记录。

辅助线索可以帮助定位问题，但不能单独证明资金到账、退款成功、供应商确认、库存释放、账本平衡或人工修复完成。

## 对账类型

`03-reconciliation-types.md` 必须区分不同对账时效和用途：

| 类型 | 触发 | 主要用途 | 典型风险 |
| --- | --- | --- | --- |
| 准实时对账 | 回调、查询、事件流、短周期任务 | 尽早发现本地和外部状态分叉 | 把临时延迟误判成差错 |
| 日终对账 | 日切后批量账单和账本汇总 | 发现金额、笔数、状态和清结算差异 | 只看汇总不看明细 |
| T+N 对账 | 延迟账单、供应商账单、清算文件到达后 | 处理迟到成功、迟到失败、手续费和结算差异 | 本地提前关闭 unknown |
| 专项重跑 | 修复后、事故后、版本修复后 | 重新解释一批事实和差错 | 重跑产生重复修复 |
| 人工复核 | 自动规则无法收敛或金额风险较高 | 决策、审批、补证和关闭 | 人工直接改数据 |

每种类型都要说明输入事实、匹配键、时间窗口、容忍延迟、输出差错和关闭条件。

## 差错分类

`04-difference-classification.md` 必须建立可执行的分类，而不是只写“金额不一致”。

第一版分类：

- 本地少记：外部成功或账单成功，但本地缺少领域事实或账本事实。
- 本地多记：本地成功，但外部账单不存在或明确失败。
- 外部多扣：渠道或供应商多扣、多结算或重复收费。
- 外部少扣：本地认为成功，但渠道、供应商或清算侧没有对应资金事实。
- 账本不平：ledger posting 借贷不平、缺业务来源或与 account movement 不可解释。
- 状态不一致：业务状态、渠道状态、供应商状态和账本状态互相冲突。
- unknown 未收敛：支付、提现、退款或供应商长期 unknown，缺少查询、账单、对账或人工挂起路径。
- 重复业务效果：重复回调、重复消息、重复修复造成重复入账、退款、出票、释放库存或结算。
- 清结算差异：商户、平台、供应商、手续费、罚金、补差或汇率差异不可解释。
- 人工修复缺证据：修复命令缺审批、证据快照、复核结果或关闭原因。

每个分类都必须写清：

1. 读取哪些事实。
2. 如何判断差错成立。
3. 允许进入哪些处理路径。
4. 禁止哪些危险修复。
5. 如何关闭。

## 修复闭环

`05-repair-workflow.md` 必须强调：对账不能直接改平数据。

合格修复流程：

1. 创建 Difference。
2. 生成 Case，并保存证据快照。
3. 分类差错和风险等级。
4. 自动规则能处理的，生成幂等 Repair Command。
5. 需要人工判断的，进入 maker-checker 审批。
6. 执行修复，追加 Repair Fact，例如补记账本、冲正、调整、退款、补差、罚金或供应商人工确认。
7. 复核修复结果，确认领域事实、账本事实、外部事实和人工事实互相解释。
8. 关闭 Case，并记录关闭原因。

危险做法：

- 对账 SQL 直接 update 业务状态。
- 为了报表平衡删除原始账本分录。
- 用人工备注替代审批和复核。
- 对已 capture 资金执行 void。
- 删除已出票、已生效保险或已确认供应商订单。
- 修复命令不幂等，重复执行产生第二次资金效果。
- 关闭差错但没有证据说明为什么可关闭。

## 场景矩阵

`06-scenario-matrix.md` 必须覆盖四类主线场景：

| 场景 | 对账重点 | 典型差错 |
| --- | --- | --- |
| 内部转账 | account movement、ledger posting、冻结、扣减、入账、冲正、调整 | 借贷不平、入账缺失、重复冲正、账本与账户流水不一致 |
| 充值提现 | 渠道流水、回调、查询、账单、退款、提现、手续费 | 渠道成功本地 unknown、重复回调、退款 unknown、提现受理后本地释放资金 |
| 电商下单 | 订单、支付、库存、退款、发货、商户结算 | 已支付未发货、库存释放重复、退款缺账本、结算金额不一致 |
| 旅行组合预订 | 供应商订单、出票、酒店确认、保险生效、罚金、补差、退款 | 供应商迟到成功、已出票后本地失败、已生效保险被取消覆盖、退款和罚金不可解释 |

每个场景都要说明：

- 读取的事实源。
- 匹配键和时间窗口。
- 差错分类。
- 自动修复路径。
- 需要人工处理的边界。
- 不合格修复示例。

## 验证与审计

`07-verification-and-audit.md` 必须回答：如何验证对账系统本身可靠。

必须覆盖：

- 差错生成幂等：同一事实差异不能无限生成重复 Case。
- 修复命令幂等：重复提交不能产生第二次资金效果。
- 关闭条件可解释：关闭 Case 必须有证据、修复结果或长期挂起原因。
- 重跑安全：日终重跑、T+N 重跑和事故重跑不能重复修复。
- 审计完整：每个差错从发现到关闭都有操作者、时间、证据、命令、结果和复核。
- 报表一致：报表只能读取事实和差错状态，不能反向修改事实。
- 监控指标：差错数量、金额暴露、unknown 年龄、自动修复率、人工处理时长、重开率、复核失败率。

这一章必须和 06 阶段连接：用异常 History、故障注入、历史回放和 oracle 检查对账闭环是否会制造新的不可解释事实。

## 面试与架构评审表达

`08-interview-synthesis.md` 必须让学习者能清楚表达：

- 对账不是事后补丁，而是金融一致性的最后一道事实闭环。
- 对账不是直接改数据，而是发现 Difference、创建 Case、执行 Repair、Review 和 Close。
- 对账读取的是业务事实、账本事实、外部事实和人工事实，不是日志或 offset。
- 自动修复必须幂等，高风险修复必须 maker-checker。
- 对账系统自己也要被验证，否则它会成为新的风险来源。

高频问题必须覆盖：

- 为什么有了分布式事务、Outbox、Saga、TCC 或 Temporal 还需要对账？
- 渠道回调成功但本地失败怎么办？
- 本地成功但渠道账单没有怎么办？
- 对账能不能直接把状态改成成功？
- 日终对账和准实时对账有什么区别？
- 人工修复如何避免变成不可审计后门？
- 如何验证对账系统本身？

## 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 章节变成支付渠道对账，覆盖面过窄 | 强制覆盖账本、供应商、库存、订单、清结算和人工修复 |
| 章节变成数据库表设计 | 保持文档阶段，只保留未来工程映射，不创建实现细节 |
| 对账被写成“查差异后 update 状态” | 每章都强调追加事实、修复命令、审批、复核和关闭原因 |
| 差错分类太抽象 | 每类差错必须绑定读取事实、判断规则、处理路径和禁止行为 |
| 与 06 验证实验室断裂 | 在验证与审计章节复用 History、Invariant、Oracle、Replay 和 Fault Injection |
| 人工修复被美化成万能手段 | 要求 maker-checker、证据快照、幂等修复、复核和可审计关闭 |
