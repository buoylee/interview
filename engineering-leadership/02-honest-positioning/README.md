# 02 · 「沒正式帶過人」的誠實定位法

> 這章是為**你的處境**量身寫的:8 年資深 IC,沒帶過下屬、沒有 leader 頭銜,最多指點一下、偶爾 code review。
>
> 被問「**你帶過人嗎?**」時,你有兩條死路和一條活路。這章教你走活路:**不假裝、不妄自菲薄,把你真有的東西翻譯成 leadership 訊號。**

---

## 1. 先拆掉一個誤解:leadership ≠ 頭銜

你一聽「帶過人嗎」就心虛,是因為你把 leadership 等於「**當過主管、有下屬、背績效**」。但對你投的開發崗,面試官要的根本不是這個(那是 EM 的事,見 00 章)。他們要的是:

> **leadership = 影響力(influence)。** 你能不能讓你周圍的人和事,因為有你而變得更好——哪怕你對他們**沒有任何職權**。

這個定義一換,局面全變了。因為「有頭銜的領導」靠的是權力(我是你主管,你得聽我);而**「沒頭銜還能影響別人」反而是更稀缺、更高級的訊號**——它證明你靠的是專業、判斷和信任,不是位置。

**所以你的處境不是劣勢,是「需要翻譯」。** 你 8 年裡一定做過一堆有影響力的事,只是你沒把它們**叫做** leadership。

---

## 2. 你一定有的三種「準領導」訊號

幾乎每個資深 IC,身上都有這三類訊號。對著想你的經歷,一定挖得出來:

### ① Mentoring(帶人 / 教人)—— 對應你的「指點一下 + code review」
你說你「最多指點一下、偶爾 review」。**這就是 mentoring,別小看它。**
- 帶過新人上手專案 / 熟悉 codebase?
- 有人卡住來問你,你幫他理清楚?
- code review 時你不只說「這裡改」,還說了「為什麼這樣更好」?(這是**教**,不是**挑**)
- 寫過文件 / wiki 讓別人少踩坑?

→ 這些全是「讓別人變強」的訊號(Collaboration + Influence)。

### ② Technical leadership(主導技術)—— 對應你的「把事做深」
- 主導過一個模組 / 服務的設計?
- 定過介面、定過技術方案,讓別人照著接?
- 推動過一個技術改進(換框架、加監控、改流程)?
- 做過技術選型、寫過設計文件被採納?

→ 這些是「在沒有頭銜時主導了技術方向」(Ownership + Judgment + Influence)。**這是你最該主打的一類**,因為它最接近 Tech Lead 的訊號,而你作為資深 IC 又真的做過。

### ③ Influence without authority(無頭銜推動)—— 對應你的「跨人協調」
- 推動過一件「沒人負責但會出事」的事?
- 說服過 PM / 別的工程師改方案?(07 章那個對齊三件套就是!)
- 跨團隊對齊過一個介面 / 約定?

→ 這些是「沒有權力,靠講清楚利弊把事推成」(Influence 的最強形態)。

> **金句**:你不是「沒帶過人」,你是「**帶過事、影響過人,只是沒人給你頭銜**」。面試要做的,就是把後者講出來。

---

## 3. 怎麼從你的 IC 經歷裡「挖」訊號?(自我盤點提問)

對著這幾個問題回憶,每想到一個就記下來(這些就是你故事銀行的原料):

- 有沒有**新人因為你**而更快上手?(誰、什麼專案)
- 有沒有**別人卡住、你 unblock** 了他?(怎麼幫的)
- 有沒有一個**模組 / 決定是你主導**的?別人照你定的做?
- 有沒有你**說服了誰**改變做法?(用什麼說服的)
- 有沒有你**主動補了一個沒人管的坑**?
- code review 裡,有沒有你**教會別人**一個東西的時刻?

> 別追求「驚天動地」。面試官要的不是你扭轉乾坤,是你**身上有沒有那個「會主動讓周圍變好」的傾向**。小事也算數,關鍵是你怎麼講(STAR/SBI,見 01 章)。

---

## 4. 「你帶過人嗎?」完整應答

### 4.1 誠實開場框架(三件事缺一不可)

一個好的開場,同時做到:**① 誠實**(不假裝帶過團隊)+ **② 給訊號**(亮出準領導經歷)+ **③ 展現意願**(我想往這方向走)。骨架:

> 「**正式來說,我沒有帶過下屬、沒有 leader 頭銜**(①誠實)。**不過實際工作裡,我經常承擔 [mentoring / 主導模組 / 跨人推動] 的角色**——比如 [一個具體 STAR 故事](②訊號)。**我滿享受這種把事情理清楚、讓旁邊的人也能順順做下去的狀態,這也是我接下來想多承擔的方向**(③意願)。」

### 4.2 before / after

- **❌ 死路一(假裝)**:「帶過啊,我帶過一個小團隊。」→ 面試官一追問「幾個人?你怎麼分配?怎麼處理績效?」**立刻穿幫**,而且留下「會吹牛」的致命印象。
- **❌ 死路二(妄自菲薄)**:「沒有,我就是個寫程式碼的。」→ 主動把自己貶成沒有 leadership 潛力,**白白放棄整片得分區**。
- **✅ 活路(誠實翻譯)**:用 4.1 框架。例:「正式沒帶過下屬。不過上個專案,有兩個新同事是我帶上手的——我幫他們梳理了 codebase、定了我們模組的介面約定讓他們照著接,他們有問題也都來找我。我也習慣在 code review 裡多寫一句『為什麼』而不只是『改這裡』。我蠻喜歡這種讓團隊整體更順的感覺,也想之後多承擔一點這方面的責任。」

→ 活路版**全程沒撒謊**,但面試官聽到的是:mentoring ✓、technical leadership ✓、leadership-readiness ✓。

### 4.3 常見追問接法

- **「那你想帶人 / 當 leader 嗎?」**(測意願和自我認知)
  → 別說「都行」(顯得沒想法),也別過度承諾。誠實版:「我想往**技術影響力**更大的方向走,帶人是其中一塊我願意嘗試的——我喜歡把複雜的事理清楚、讓團隊少走彎路。具體是走純技術深度還是帶團隊,我還在體會,但『讓周圍變好』這件事我是真的有動力。」

- **「你 mentor 別人的時候,具體怎麼做的?」**(挖細節驗真假)
  → 給一個真實的 STAR:誰、卡在哪、你怎麼帶的(注意:是「教他釣魚」還是「替他釣」?好的 mentor 是前者)。例:「我不直接給答案,先問他卡在哪、他怎麼想的,再點關鍵的那一步,讓他自己走通——下次他就不用問我了。」

- **「沒帶過人,你怎麼證明你能 lead?」**(壓力追問)
  → 回到「leadership = 影響力」:「頭銜我確實沒有,但 lead 的核心是讓事情和人往對的方向走。我舉個沒有頭銜卻推動成的事……」接 07 那種對齊三件套故事。

### 4.4 紅線

- 🚫 **假裝帶過團隊**——追問必穿幫,且毀掉誠信印象。
- 🚫 **妄自菲薄**(「我就是個 coder」)——主動棄權。
- 🚫 **把 mentoring 講成「替別人做完」**——好 mentor 是讓別人變強,不是當別人的拐杖。
- 🚫 **空有意願沒有實例**(「我很想帶人」但舉不出一件影響別人的事)——意願要用故事撐。

## 本章小結

- **leadership = 影響力,不是頭銜。** 沒頭銜還能影響人,反而是更稀缺的訊號——你的處境是「需要翻譯」,不是劣勢。
- 你一定有三種準領導訊號:**mentoring(你的指點/review)、technical leadership(你主導的模組/決策,最該主打)、influence without authority(你推動成的事)**。
- 「你帶過人嗎」活路 = **誠實 + 給訊號(STAR 實例)+ 展現意願**,三件齊。
- 兩條死路:假裝(穿幫)、妄自菲薄(棄權)。
