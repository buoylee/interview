# 05 · 工作流程

> 面試官問「你們的工作流程是怎樣的?」——很多人就開始背:「我們用兩週一個 sprint,每天站會,兩週一次 review……」**背完了,面試官心裡打的分是『初級』。**
>
> 因為他真正想聽的不是你**記得**哪些儀式,是你**懂不懂這些儀式為什麼存在、有沒有自己的觀點、會不會改進它**。這章教你看穿所有流程背後的兩個根本目的,從此對流程有發言權。

---

# Part A · 方法論(工作真戰力)

## A0 收口地圖:所有流程,都在解決這兩個問題

不管 Scrum、Kanban、code review、CI/CD……名目再多,**所有工程流程的存在理由就兩個**:

> **① 縮短反饋環**(讓「做錯了」儘早、儘便宜地暴露)
> **② 降低協作成本**(讓「誰在做什麼、做到哪」對大家透明,減少對齊和等待)

把這兩條記死,你就有了一把「照妖鏡」:**任何一個流程/儀式,問它『縮短了什麼反饋?降低了什麼協作成本?』答不出來,它就是形式主義(cargo cult),該砍或該改。** 這就是你對流程「有觀點」的底氣。

5 條派生原語:

| # | 原語 | 一句話 |
|---|---|---|
| 1 | **流程為反饋與協作而生** | 任何儀式都要能回答「它解決什麼問題」 |
| 2 | **可見性(visibility)** | 讓工作狀態對所有人透明(看板/站會的本質) |
| 3 | **小批量 + 頻繁集成** | 批量越小、集成越勤,風險越低 |
| 4 | **質量內建(shift left)** | 質量靠過程保證,不靠末端把關 |
| 5 | **持續改進** | 流程是團隊自己調的活物,不是聖旨 |

## A1 黑盒內幕:為什麼是「敏捷」而不是「瀑布」?

### 瀑布的致命假設
傳統瀑布流程(需求→設計→開發→測試→上線,一條道走到黑)有個假設:**需求一開始就完全清楚且不變。** 但你在 07 章已經知道——需求是**有損信道**,從來不會一開始就清楚,而且一定會變。瀑布把驗證(測試/上線)推到最後,等於把 03 章說的「風險憋到最後才爆」放大到整個專案尺度。

### 敏捷(Agile)在賭什麼
敏捷承認「需求會變、會錯」,於是不賭「一次規劃到位」,改成**短迭代 + 頻繁反饋**:做一小塊→給人看→根據反饋調整→再做一小塊。**這就是 03 章「拆風險、早暴露」在團隊流程尺度的版本。** 所有敏捷實踐(sprint、站會、demo)都是在縮短那個反饋環。

> 看穿了嗎:**03 章拆活、07 章對齊、05 章流程,底層是同一件事——縮短反饋環、讓風險早而便宜地暴露。** 這是貫穿整個工程的元原則。

### Cargo cult 陷阱(資深 vs 初級的分水嶺)
很多團隊照抄儀式——每天站會、兩週 sprint、燃盡圖——卻忘了為什麼,於是站會變成對老闆的工作彙報、retro 變成走過場。這叫 **cargo cult(草包族崇拜)**:模仿形式,丟了實質。**資深的標誌,就是能看穿儀式背後的目的、能批判性地裁剪流程**,而不是當儀式的搬運工。

> 30 秒 primer:**cargo cult** 典故——二戰時太平洋島民見過飛機送來物資,戰後他們用稻草搭假飛機假跑道,以為這樣物資就會再來。比喻「照搬形式卻不懂原理,自然沒效果」。

## A2 把常見流程「翻譯成它解決的問題」

別背儀式,記住每個儀式**對應哪個問題**:

### Scrum(時間盒迭代)
| 儀式 | 它解決什麼 |
|---|---|
| Sprint planning | 對齊「這個迭代做什麼、DoD 是什麼」(降協作成本) |
| 每日站會 standup | 同步進度 + **暴露阻塞**(讓卡住的事被看見,早 unblock) |
| Sprint review / demo | 向干系人**收反饋**(縮短反饋環) |
| Retrospective 回顧 | **改進流程本身**(原語 5) |
> 站會的精髓**不是彙報給主管**,是團隊互相同步、把 blocker 攤出來。一旦變成對著老闆背昨天幹了啥,它就死了。

### Kanban(持續流 + 限制 WIP)
核心是**限制在製品(WIP, work in progress)**:看板每一列規定最多幾張卡。為什麼?**同時做太多 = 哪個都做不完 + 上下文切換成本爆炸。** 限流逼你「做完一個再開下一個」,反而更快交付。
> 30 秒 primer:**WIP 限制**——你以為同時推 5 件事效率高,其實人腦切換上下文極貴(撿起一件事要重新載入腦內狀態)。限制同時進行的數量,是用「少開戰線」換「快速結案」。

### Code review
**不只是抓 bug**。它同時解決:① 知識共享(別人懂你的代碼了)② 風格/標準一致 ③ 集體 ownership(代碼是「我們的」不是「我的」)④ 教學(資深藉 review 帶新人,見 06 章)。
> 好的 review 講「為什麼這樣更好」,不只「改這裡」——這是 06 章的影響力槓桿。

### CI / CD(持續集成 / 持續交付)
- **CI(持續集成)**:每次提交都自動合併+跑測試。本質:讓「合併衝突、集成 bug」這種風險**小步、頻繁、便宜地暴露**,而不是憋到上線前大爆炸(又是「早暴露」)。
- **CD(持續交付/部署)**:自動化發布。本質:**小批量發布**,每次改動小→出事好定位好回滾→發布不再是膽戰心驚的大事。
> 30 秒 primer:**shift left(質量左移)**——把測試、檢查從流程末端(上線前)往前移到開發過程中(寫代碼時就有 CI 跑測試)。越早發現問題越便宜,這是 CI 的靈魂。

## A3 走一遍:你發現一個流程問題,怎麼推動改

**情境**:你們每天站會開成 30 分鐘的「對著 leader 彙報」,大家輪流背流水帳,沒人真在聽,blocker 也沒人提。

用收口地圖診斷:**站會該解決的是「同步 + 暴露 blocker」,現在這兩個都沒做到,純儀式。**

你的行動(這就是 leadership):
1. **不指責,先講問題**:「我發現站會我們花了半小時但好像沒幫到彼此,blocker 也沒真的被解決。」
2. **回到目的提方案**:「站會的目的是互相同步和暴露卡點。要不要試試:每人只講『卡在哪、需要誰幫』,進度同步挪到看板上看?控制在 10 分鐘。」
3. **小範圍試 + 看數據**:「先試兩週,不行我們再調。」
4. **retro 復盤**:兩週後看效果,留下好的。

→ 你沒有頭銜,但你**看穿了流程的目的、提了改進、推動團隊試**——這是面試官最想看到的「對流程有觀點 + influence」。

---

## A4 自評量表(rubric)

判斷「你對流程是不是真懂」:
- ☐ 對每個你在用的儀式,我能說出**它解決哪個問題**(反饋 or 協作)?
- ☐ 我能指出我們流程裡哪個環節是**cargo cult**(走形式、沒效果)?
- ☐ 我有沒有**推動 / 嘗試改進**過某個流程,而不只是抱怨或服從?
- ☐ 我理解 CI/code review/小批量背後的共同原則(**早暴露、質量內建**)?

# Part B · 面試轉化(應答戰力)

## B1 這題在考什麼

「你們的工作流程是?」表面是描述題,實則考 **Ownership(對流程有沒有主人翁意識)+ Judgment(懂不懂為什麼)+ Influence(會不會改進)**。**只描述儀式 = 初級;有觀點 + 改進過 = 資深。**

## B2 答題結構:描述 → 觀點 → 改進(三段)

別只停在描述。好答案三段:
1. **描述**(簡短):「我們用兩週 sprint + 看板。」
2. **觀點**(關鍵):「我比較認同的是它讓反饋環短——每兩週能對齊一次方向,避免悶頭做錯。不過我覺得儀式不該僵化,比如站會我們就……」
3. **改進**(加分):「我推動過一個改進……」(接 A3 那種 STAR)

## B3 STAR:薄 vs 厚

- **❌ 薄版**:「我們就是標準 Scrum,兩週 sprint,每天站會,我按流程做。」(純描述、零觀點、像流程的螺絲)
- **✅ 厚版**:
  - **S**:「我們用 Scrum,但站會開成了半小時的彙報,效率很低。」
  - **T**:「我想讓站會回到它該有的作用——同步和暴露 blocker。」
  - **A**:「**我**在 retro 上提了這個觀察,沒指責誰,而是回到『站會是為了什麼』,提議改成只講 blocker、進度看板上看、限時 10 分鐘,先試兩週。」
  - **R**:「站會從 30 分鐘降到 10 分鐘,blocker 真的開始被當場解決,團隊都覺得清爽。後來我們也養成了『流程不對就在 retro 調』的習慣。」

**5 維度**:Ownership ✓、Judgment ✓(懂目的)、Influence ✓(推動改進)、Communication ✓。

## B4 常見追問 & 接法
1. **「你覺得敏捷有什麼問題?」**(測你有沒有獨立思考)→ 「敏捷容易淪為形式——儀式照做但忘了目的,比如為敏捷而敏捷、估點變成 KPI。我關注的是它的內核(短反饋環),儀式該為內核服務,不該倒過來。」
2. **「如果團隊流程很爛但你改不動呢?」** → 「我會先在我能影響的範圍做(比如我自己的 PR 拆小、review 寫清楚 why),用效果說話,而不是一上來要求改全公司。影響力是從小範圍的可信度積累的。」(接 06 章)

## B5 紅線
- 🚫 只會背儀式名詞,問「為什麼」答不出。
- 🚫 把流程當聖旨(「公司規定這樣我就這樣」)——零 ownership。
- 🚫 光抱怨流程爛,但從沒推動過任何改進。
- 🚫 顯得教條(「必須嚴格按 Scrum」)——資深要的是務實裁剪,不是原教旨。

# Part C · 你的 drill

```
【故事卡 · 流程類】
標題:____________________________________________
S 情境:你們哪個流程有問題 / 你觀察到什麼低效?
T 任務:你想讓這個流程恢復什麼作用?
A 行動:你怎麼診斷(回到它該解決的問題)?怎麼提的?怎麼推動試行?
R 結果:改進效果?有沒有變成團隊習慣?
覆蓋維度:Ownership / Judgment / Influence
```

沒推動過流程改進?那就從下週做一件小的:在你自己的 PR 上實踐「小批量 + review 寫清 why」,或在下次 retro 提一個你觀察到的問題。**做了,就有故事;也順手讓你的團隊好一點。**

## 本章小結

- 所有流程只為兩件事:**縮短反饋環 + 降低協作成本**。這是你的照妖鏡——儀式答不出「解決什麼問題」就是 cargo cult。
- 敏捷 vs 瀑布:瀑布賭「需求一次清楚」,敏捷承認「需求會變」,用短迭代頻繁反饋適應——**就是 03 章拆風險在團隊尺度的版本**。
- 每個儀式 = 一個問題:站會=同步+暴露 blocker,demo=收反饋,retro=改進流程,WIP 限制=少開戰線,CI=集成風險早暴露,code review=知識共享+集體 ownership。
- 面試答流程要 **描述 → 觀點 → 改進** 三段,只描述是初級。
