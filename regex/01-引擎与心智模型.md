# 01 · 引擎与心智模型(治本的一章)

> 这章不教符号,教**正则引擎怎么思考**。读完你应该能在脑子里"放慢播放"一个匹配过程。
> 这是让正则「记得住」的关键——符号是查的,模型是理解的。

---

## 1. 正则到底是什么:一个匹配引擎

把正则想成一个**小程序**,你写的模式是源码,引擎是解释器,输入字符串是数据。引擎做一件事:

> 从字符串某个位置出发,**逐字符**地拿模式去对,对上了就推进,对不上就换策略,直到整个模式走完(匹配成功)或所有策略用尽(失败)。

关键词是「**位置**」和「**逐字符**」。引擎心里始终有两根指针:一根指向字符串当前读到哪(text pointer),一根指向模式当前走到哪(pattern pointer)。匹配就是这两根指针协同往前走的过程。

`re.search` 找不到时会怎样?引擎从位置 0 试,失败 → 从位置 1 重新试整个模式 → 位置 2……直到某个起点成功,或扫到末尾彻底失败。这就是为什么 `search` 能在任意位置找到,而 `match` 只在位置 0 试一次(下章细讲)。

---

## 2. 两种引擎:回溯 NFA vs DFA(这解释了一切"怪现象")

世界上的正则引擎分两大流派,**先记结论**:

| 流派 | 代表 | 怎么跑 | 代价 |
|---|---|---|---|
| **回溯 NFA** | Python `re`、Java、PCRE、JS、Perl | 一条路走到黑,撞墙了**回头换路**(回溯) | 功能强(前瞻、反向引用),但**可能指数级爆炸**(ReDoS) |
| **DFA / RE2** | Go `regexp`、grep、RE2、awk | 同时"想象"所有可能的路,一遍扫完 | **永远线性时间**,但**放弃**前瞻、反向引用 |

这一条解释了你将来遇到的几乎所有"为什么":
- 为什么 Python 正则能写 `(?=...)` 前瞻、`\1` 反向引用,Go 不行?→ 那些功能**依赖回溯**,DFA 做不到。
- 为什么一个看着人畜无害的 Python 正则能把 CPU 跑满几秒钟(ReDoS)?→ 回溯可能指数爆炸。Go 因为不回溯,**天然免疫**。
- 为什么 Go 正则"慢"在编译、"快"在匹配且稳定?→ DFA 前期构造代价换运行期线性。

> **面试一句话:** "Python/Java 是回溯型引擎,功能强但有灾难性回溯风险;Go 的 RE2 是非回溯型,牺牲了前瞻和反向引用,换来线性时间保证,适合处理不可信输入。"

本章后面讲的「回溯」「贪婪」「占有量词」全是**回溯流派**的概念。Go 用户也该懂,因为它解释了 Go 为什么"少了那些功能"。

---

## 3. 回溯实战:放慢播放一次匹配

光说"回溯"太抽象。我们拿 `a.*b` 去匹配字符串 `axbxb`,一步步看引擎在干嘛。

记住 `.*` = 桶①的 `.`(任意字符)+ 桶②的 `*`(零或多个),**默认贪婪**。

```
模式: a  .*  b
输入: a  x  b  x  b
      0  1  2  3  4
```

1. `a` 对 `a`(位置0)✅,text 指针推进到 1。
2. `.*` 贪婪 —— **能吃多少吃多少**:它一口气吃掉 `xbxb`(位置 1~4),text 指针冲到字符串末尾(位置5)。
3. 模式还剩一个 `b` 要匹配,但 text 已经到末尾,没字符了 ❌。
4. **回溯!** `.*` 不情愿地**吐回一个字符**:现在它只吃了 `xbx`(位置1~3),text 退回位置 4。
5. 再拿 `b` 对位置4 的 `b` ✅。模式走完,**匹配成功**。

最终匹配到的是整个 `axbxb`。注意第 2 步的"吃过头"和第 4 步的"吐回来"——**这一吃一吐就是回溯**。模式越复杂、量词越多,回溯的组合越多,慢就慢在这里。

### 换成懒惰:`a.*?b`

`*?` 是懒惰量词(`*` 后加 `?`)—— **能少吃就少吃**:

1. `a` 对 `a` ✅,推进到 1。
2. `.*?` 懒惰 —— 先**一个都不吃**,直接试下一步。
3. `b` 对位置1的 `x` ❌。懒惰量词被迫**吃一个** `x`(位置1),text 到 2。
4. `b` 对位置2的 `b` ✅。模式走完,**匹配成功**。

最终匹配到 `axb`。

> **同一个串,贪婪给你 `axbxb`,懒惰给你 `axb`。** 这就是为什么爬 HTML 标签内容、提取最短片段时要用 `.*?`——你想要"到第一个结束符为止",而不是"贪到最后一个"。这是日常最高频的坑,记牢。

## 4. 三种量词行为:贪婪 / 懒惰 / 占有

同一个量词 `*` `+` `?` `{n,m}`,有三种"性格":

| 性格 | 写法 | 行为 | 会回溯吗 |
|---|---|---|---|
| **贪婪**(默认) | `*` `+` `{n,m}` | 先吃到最多,不够再吐 | 会 |
| **懒惰**(加 `?`) | `*?` `+?` `{n,m}?` | 先吃到最少,不够再多吃 | 会(反方向) |
| **占有**(加 `+`) | `*+` `++` `{n,m}+` | 吃到最多,**绝不吐**(吐不出来就直接失败) | **不会** |

占有量词和它的兄弟「原子组 `(?>...)`」是**回溯流派里用来"关掉回溯"的开关**,专治灾难性回溯(见 `03`)。

> **版本注意(Python):** 占有量词 `*+ ++ ?+` 和原子组 `(?>...)` 是 **Python 3.11+** 才加进标准库 `re` 的。更早的版本想用,得装第三方 `regex` 模块。Java 一直支持占有量词和原子组。Go(RE2)不回溯,所以**根本不需要**这俩概念。

## 5. 零宽:为什么有些东西"匹配但不占字符"

桶③(位置)里的符号有个共同点:它们**匹配的是"位置"而不是"字符"**,匹配成功后 **text 指针不前进**。这叫**零宽(zero-width)**。

- `^` `$`:行/串的开头、结尾这个"缝隙"。
- `\b`:单词边界 —— `\w` 和非 `\w` 之间的那条"缝"。`\bcat\b` 能匹配 `a cat sat` 里的 `cat`,但不匹配 `category` 里的 `cat`(因为 `cat` 后面紧跟 `e`,没有边界缝)。
- 前瞻 `(?=...)` `(?!...)`、后顾 `(?<=...)` `(?<!...)`:"我后面/前面**得是 / 不能是**某个模式",但那段模式**不计入匹配结果**。

零宽的威力在于**加条件而不消耗字符**。经典例子——给数字加千分位逗号:

```python
re.sub(r'(?<=\d)(?=(\d{3})+$)', ',', '1234567')   # -> '1,234,567'
```

这里两个零宽断言:`(?<=\d)`(前面是数字)+ `(?=(\d{3})+$)`(后面正好是 3 的倍数个数字直到结尾),它们**只定位插入点**,一个字符都不吃,所以能在数字"缝隙"里插逗号。看不懂没关系,记住"零宽 = 加条件不占位"这个概念即可,配方在 `03`。

## 6. 四个桶,展开(呼应 README 总纲)

README 给了四个桶的速记表,这里说清**每个桶的边界和易混点**,速查具体符号去 `02`。

**桶① 匹配什么字符**
- 单字符:字面量、`.`(除换行外任意,加 `re.S` 后连换行)、`\d \w \s` 及大写取反版 `\D \W \S`。
- 字符集合:`[abc]`(三选一)、`[a-z]`(范围)、`[^abc]`(取反)。
- ⚠️ 易混:`[.]` 里的 `.` 是**字面点**,不是"任意字符"——方括号内大部分元字符失去魔力。

**桶② 匹配多少个**
- `*`(≥0)、`+`(≥1)、`?`(0 或 1)、`{n}`(恰好 n)、`{n,}`(≥n)、`{n,m}`(n 到 m)。
- 三种性格见第 4 节。
- ⚠️ 易混:`?` 有两个身份——**量词**(`a?` = 0或1个a)和**懒惰修饰符**(`*?`)。看它前面贴着谁。

**桶③ 匹配什么位置(零宽)**
- 锚点 `^ $ \b \B \A \Z`,断言 `(?=) (?!) (?<=) (?<!)`。见第 5 节。
- ⚠️ 易混:`^` 在 `[^...]` 里是"取反",在外面是"行首"。位置决定含义。

**桶④ 分组与复用**
- `(...)` 捕获组(可回头引用、可在结果里单独取出)。
- `(?:...)` 非捕获组(只为分组,不占编号,性能略好)。
- `(?P<name>...)` 命名组(Python 写法;Java 是 `(?<name>...)`)。
- `|` 或(`cat|dog`)。⚠️ `|` 的优先级很低,`^cat|dog$` 是 `(^cat)|(dog$)` 不是 `^(cat|dog)$`,要加组。
- `\1 \2` 反向引用:"再来一个和第1组**一模一样**的"。`(\w)\1` 匹配叠字(`aa` `bb`)。

> **回到用法:** 写正则时永远先归桶——*我现在要表达字符、数量、位置、还是分组?* 这个习惯一旦养成,你不会再"忘",只会"查"。

## 7. 三语言桥(Java/Go 背景的你重点看)

同一套心智模型,三语言的差异主要在**引擎流派**和**少数语法**:

| 维度 | Python `re` | Java `java.util.regex` | Go `regexp`(RE2) |
|---|---|---|---|
| 引擎 | 回溯 NFA | 回溯 NFA | DFA 式,线性时间 |
| 前瞻 `(?=)(?!)` | ✅ | ✅ | ❌ |
| 后顾 `(?<=)(?<!)` | ✅(**需定长**) | ✅(可变长) | ❌ |
| 反向引用 `\1` | ✅ | ✅ | ❌ |
| 占有量词 / 原子组 | ✅(3.11+) | ✅ | 不需要(不回溯) |
| 命名组语法 | `(?P<n>...)` / 引用 `(?P=n)` | `(?<n>...)` / 引用 `\k<n>` | `(?P<n>...)` |
| 字符串里写正则 | 用 `r"..."` 原始串免双重转义 | 必须双重转义 `"\\d"` | 用反引号 `` `\d` `` 免转义 |
| "整串匹配" | `re.fullmatch` | `matcher.matches()` | `regexp.MatchString` 配 `^...$` |
| "从头匹配" | `re.match` | `matcher.lookingAt()` | (无直接对应,用 `^`) |
| "任意位置找" | `re.search` | `matcher.find()` | `regexp.FindString` |

> 三个最容易让你栽跟头的差异:
> 1. **Python 后顾必须定长**:`(?<=ab)` 行,`(?<=a+)` 不行。要变长后顾得用第三方 `regex` 模块。
> 2. **Java 双重转义**:Python `r"\d+"` 在 Java 里要写成 `"\\d+"`(因为 `\` 在 Java 字符串本身要转义)。Python 的 `r"..."` 原始串就是为了免这层。
> 3. **Go 没有前瞻/反向引用**:别在 Go 里找 `(?=...)`,找不到。需要时只能改用代码逻辑或多次匹配。

## 本章最小 takeaway(只记这些也够用)

如果上面的细节一时没全吃透,**至少把这三条刻进脑子**,它们撑得起 80% 的日常:

1. **引擎从左到右扫,量词默认贪婪,撞墙就回溯。** —— 解释一切匹配结果和性能问题。
2. **任何符号都归四个桶之一:字符 / 数量 / 位置 / 分组。** —— 写正则先归桶,再查符号,从此不"忘"。
3. **回溯型(Python/Java)功能强但会 ReDoS;非回溯型(Go RE2)线性安全但没前瞻/反向引用。** —— 解释三语言差异和性能取舍。
