# 06 · 隔离 / 舱壁 —— 为什么一个慢依赖能拖垮整个服务

> 池配好了,但所有请求共用一个池时,一个慢下游能把池占满,让**毫不相干**的快请求全部饿死。这一章讲这种级联是怎么发生的、怎么用舱壁挡住它、以及大厂的进阶玩法 shuffle sharding。
>
> 前置:`05`。动手:`lab/sim/bulkhead.py`、`lab/experiments/e06-bulkhead.md`。

---

## 1. 一个真实事故的形状

线上典型故障:推荐服务(下游,平时 20ms)突然变慢到 5 秒。结果**整个主站挂了**——连不依赖推荐的「查订单」也 502。运维一脸懵:订单跟推荐有什么关系?

关系在于:**它们共用同一个线程池 / 连接池 / worker 池。** 推荐慢了 5 秒,处理推荐的请求**长时间占着池里的资源不放**;新来的请求(包括查订单)发现池被占满,只能排队;队列越堆越长,直到整个服务的资源全被「等推荐」的请求占据——**一个非关键下游的慢,通过共享池,放大成了全站故障。**

这就是需要**隔离(bulkhead,舱壁)**的场景:像船舱用隔板分区,一个舱进水不会沉整条船。

---

## 2. 底层①:队头阻塞 → 池耗尽级联

机制叫**队头阻塞(Head-of-Line blocking)**:共享池/队列里,前面那个慢请求卡住,后面所有请求(哪怕自己很快)都得等。具体的级联链条:

```
慢下游(5s) → 处理它的请求长时间占住池资源
           → 池被这类慢请求逐渐占满(offered load a 暴涨,c 不变 → 05 的 ρ→1)
           → 快请求拿不到资源,在队列里堆积
           → 队列无界增长,延迟雪崩,健康检查也超时
           → 整个实例被判死,流量打到其他实例,连锁打挂          ← 级联
```

注意它怎么和前几章咬合:慢下游让某类请求的占用时长 `S` 暴涨 → `05` 的 offered load `a=λS` 冲破池容量 `c` → `01` 的 `1/(1−ρ)` 让等待爆炸 → 整个池被拖死。**隔离的本质,是不让一类请求的 `a` 暴涨殃及另一类。**

```bash
# 亲眼看共享池里慢负载饿死快负载,以及隔离如何挡住:
cd concurrency-capacity
python lab/sim/bulkhead.py
# SHARED   fast: rejected=427 ...   ← 快请求被慢请求拖死,大量被拒
# ISOLATED fast: rejected=0   ...   ← 隔离后快路毫发无伤,慢的只坑自己
```

## 3. 底层②:两种隔离 —— 信号量 vs 线程池(Hystrix 的核心抉择)

怎么隔离?Netflix Hystrix(熔断/隔离的鼻祖)给了两种,各有取舍,这是面试深水区:

| | **信号量隔离**(semaphore) | **线程池隔离**(thread-pool) |
|---|---|---|
| 怎么做 | 给每类下游一个独立的 `Semaphore(N)`,限它的并发数 | 给每类下游一个独立的线程池,调用在它的线程里跑 |
| 开销 | 极轻(就一个计数器) | 重(额外线程 + 上下文切换) |
| 能否中断慢调用 | **不能**——调用还在调用方线程里,卡住就卡住 | **能**——可以给线程池调用设超时,超时丢弃 |
| 适合 | 调用快、量大、信任下游不会卡死 | 调用可能卡很久、需要强制超时保护 |

**关键区别在「能不能中断」**:信号量隔离只是限制了「同时有几个请求打这个下游」(挡住了 offered load 暴涨),但如果下游真卡死 5 秒,那 N 个名额还是被占 5 秒——它防的是「慢下游占满**整个**池」,但占满它**自己那份**配额还是会发生。线程池隔离能再加一道超时,把卡死的调用直接砍掉。代价是每类一个线程池的资源开销。

> 现代实践(Resilience4j、服务网格)多用**信号量隔离 + 独立超时**作为默认(轻量),线程池隔离留给「下游可能长时间卡死且无法设客户端超时」的情况。无论哪种,**每个隔离单元的大小,回到 `05` 的池定容**——独立算各自的 offered load。

## 4. 底层③:进阶 —— shuffle sharding(大厂的概率隔离)

普通隔离按「下游类型」拆。但多租户场景(你的服务给 1000 个客户用)有个更难的问题:**一个客户发疯(打爆某个资源),怎么不影响其他 999 个?**

按客户开 1000 个池不现实。AWS 的解法是 **shuffle sharding(洗牌分片)**:

- 把后端资源分成若干「虚拟分片」,**给每个客户随机分配一个小子集**(比如从 8 个 worker 里随机分 2 个)。
- 某个客户打爆了,只波及**和他共享那 2 个 worker 的少数客户**;绝大多数客户分到的是**不同的随机组合**,概率上几乎不重叠,完全不受影响。
- 用很少的资源,得到「近似每客户独立」的隔离效果。

这是从「按类型硬隔离」升级到「按概率软隔离」,是大规模多租户系统(AWS、CDN、API 网关)的标配思路。知道它存在、知道它解决「多租户爆炸半径」问题,就够撑起架构师面试的深度了。

---

## 5. 怎么决定拆不拆、按什么拆

不是所有东西都要隔离(隔离有管理成本)。**该隔离的信号**:不同工作负载**共享资源**、且**故障特征不同**(一个慢/不稳,会殃及另一个)。常见的拆分轴:

| 拆分轴 | 例子 |
|---|---|
| **快 / 慢** | 快查询和慢报表分开池,别让报表占满 |
| **CPU / I/O** | CPU 密集进程池、I/O 密集线程/协程池分开(呼应 `02/04`) |
| **关键 / 非关键** | 支付(关键)和推荐(可降级)分开,推荐挂了不影响支付 |
| **租户 / 来源** | 大客户 vs 小客户、内部 vs 外部流量,防一个打爆全部 |

定容动作:识别出共享资源 + 故障会互相传染的负载 → 拆成独立单元 → **每个单元用 `05` 单独定容**(各自的 λ、S、池大小)。

**三语速记**:

- **Java**:Resilience4j `Bulkhead`(信号量)/ `ThreadPoolBulkhead`(线程池),声明式注解即可。
- **Go**:每类下游一个独立的 `semaphore`(`golang.org/x/sync/semaphore`)或独立 worker 池 + `context` 超时。
- **Python**:每类下游独立的 `asyncio.Semaphore`;CPU 活用专用 `ProcessPoolExecutor`、I/O 活用专用线程池(`run_in_executor` 指定 executor)。

## 6. 一句话收口

共享池让一个慢下游通过**队头阻塞**把池占满、级联拖垮整个服务;隔离(舱壁)按 快/慢、CPU/IO、关键/非关键、租户 把资源拆成独立单元,一个进水不沉全船。两种实现——信号量(轻、不能中断)vs 线程池(重、能超时砍断)——按下游会不会卡死来选;多租户的爆炸半径用 shuffle sharding 概率隔离。每个隔离单元仍按 `05` 各自定容。**隔离挡住了「内部传染」,`07` 接着处理「整体被打爆」时怎么办。**
