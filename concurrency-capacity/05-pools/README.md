# 05 · 池:哪些要、多大、×worker 总账 —— 连接池为什么是定容里最容易爆的雷

> 池大小不是「越大越能扛」。这一章讲池本质是个 M/M/c 队列、为什么小连接池反而更快、以及那个把数据库打挂的经典坑:`单池大小 × worker 数 × 副本数`。
>
> 前置:`04`(各下游 λ)。动手:`lab/sim/starve.py`、`lab/experiments/e05-pool-starvation.md`。

---

## 1. 哪些资源该上池(不是所有东西都要池)

池(pool)= 预先建好一批可复用的资源,借出去用完还回来。值得上池的资源同时满足三个条件:

- **建立昂贵**:DB 连接要 TCP 握手 + 认证 + TLS;线程要分配栈、登记内核——每次新建都肉疼。
- **数量稀缺**:数据库的 `max_connections` 是有限的;线程受内存限。
- **可复用**:一个连接服务完这个请求,可以接着服务下一个。

典型该上池的:**数据库连接、HTTP/RPC 客户端连接、线程**。不该上池的:无状态的廉价对象(直接 new)、本身就有内部池的客户端(别在外面再套一层)。

定容问题变成:**这几个池,每个开多大?** 答案藏在「池其实是个排队系统」这件事里。

---

## 2. 底层①:一个池就是一个 M/M/c 队列

把池大小记作 `c`(服务台数量)。请求要用资源时:有空闲就借走(被服务),没空闲就**排队等**别人归还。这正是 `01` 提过的 M/M/c 排队模型——`c` 个服务台 + 一条等待队列。

它的稳定条件和等待行为,和 M/M/1 是同一家:

- **供给到达的负载**(offered load)`a = λ × S`(erlang),`λ` 是借池的速率,`S` 是每次占用时长。`a` 的含义是「平均同时有多少个请求在占着资源」。
- **稳定的前提是 `c > a`**——池子至少要比「平均同时占用数」大,否则队列无限增长。
- **当 `c` 逼近 `a`,等待时间爆炸**(又是 `1/(1−ρ)` 那一家)。

```bash
# 亲眼看池太小时等待如何爆炸:offered load a = 80×0.1 = 8 erlang
cd concurrency-capacity
python lab/sim/starve.py --lam 80 --service-time 0.1
#   c=9  → 等待 65ms   （c 只比 a 大 1,挤,等待巨大)
#   c=12 → 等待 3.5ms  （留了余量,顺畅)
```

**池大小的下限**:`c` 必须 `> a = λ×S`。这把抽象的「池配多大」变成了一个可算的数:量出借池速率 λ 和每次占用时长 S(就是下游响应时间),算出 offered load `a`,池子配到 `a` 之上 + 余量,让等待落在可接受范围。**这是池定容的主公式。**

> 注意 `S` 是「资源被占用的时长」,对 DB 连接池就是**查询往返时间**。`03` 的 fan-out 在这里回来咬你:如果某下游被每个请求调用,它的池 λ = 前端 λ × 扇出次数,`a` 跟着放大,池要配更大。

## 3. 底层②:反直觉 —— 小连接池往往更快

下限是 `c > a`,那上限呢?直觉是「越大越保险」。**对连接池,这个直觉是错的**,这是 HikariCP 作者和 PostgreSQL 社区反复强调的经典结论:

**连接数超过数据库自身的并行处理能力后,再加连接只会让一切更慢。**

为什么?数据库能真正并行干活的程度,受限于它的 **CPU 核数 + 磁盘 iops**——比如一台 8 核 DB,同一时刻最多 ~几十个查询真在推进。你给它灌 500 个连接:

- 这 500 个连接对应 500 个并发查询,但 DB 只有 8 核,于是它们**在 DB 内部疯狂上下文切换、抢锁、抢缓存**——又是 USL 的 κ 项(`04`),吞吐不升反降。
- 连接还各自吃 DB 的内存(每连接 work_mem 等),太多连接直接压垮 DB。

PostgreSQL 的经验公式常被引为:`连接数 ≈ ((核数 × 2) + 有效磁盘数)`——一个**小得惊人**的数(8 核机可能就配 ~20)。结论:

> **池大小不是「能开多大开多大」,是「刚好喂饱下游的并行能力」**:下限 `c > offered load a`(别让自己排队),上限 ≤ 下游真实并行度(别把下游搞垮)。两者之间留余量。如果 `a` 已经超过下游并行度,那是**下游容量不够**,该扩下游或加缓存,而不是把池开更大——开大只会把排队从你这儿挪到 DB 里,还更慢。

## 4. 底层③:把数据库打挂的经典坑 —— ×worker 总账

这是生产事故里**最常见**的一类,也是面试必考。

你给连接池配了 `max=20`,自测没问题。上线后数据库 `too many connections` 挂了。为什么?因为**每个 worker 进程有自己独立的连接池**,你要算的是**总账**:

```
对数据库的总连接数 = 单池 max × worker 数 × 服务副本数(机器/Pod 数)

例:max=20 × 每机 16 个 worker × 8 台机 = 2560 连接
    数据库 max_connections = 500  →  当场打挂
```

**单池看着很合理的 20,乘上 worker 和副本,轻松冲到几千。** 这个坑在 `02` 选了「多进程 worker」之后尤其致命——Python 多 worker 部署几乎必踩。

防法:

- **算总账**:`单池 × worker × 副本 ≤ 数据库 max_connections × 安全系数(留 30%)`。反推单池大小。
- **上连接池中间件**:PgBouncer/ProxySQL 这类**外置连接池**,在应用和 DB 之间收口。应用侧开很多连接,中间件用少量真实连接复用——彻底解耦「应用并发」和「DB 连接数」。大规模多副本部署的标配。
- **别让单池配置脱离部署拓扑**:配池大小时,脑子里必须同时装着 worker 数和副本数。

> 这一条把 `02`(多 worker)、`04`(worker 数)、`09`(副本数)全串起来了:**池大小是个全局约束,不能只在单实例里想。** `python-concurrency/09` 有这个坑的语言落地,这里给的是定容公式。

## 5. 池定容的完整流程

```
对每个要上池的下游资源:
1. 量 offered load a = λ_借池 × S占用时长      （λ 含 03 的扇出放大)
2. 下限:c > a,加余量让等待可接受          （starve.py 找这个点)
3. 上限:c ≤ 下游真实并行度                  （别超过 DB 的核/iops 能力)
4. 总账:单池 c × worker 数 × 副本数 ≤ 下游连接上限 × 安全系数
5. 若 2 和 3 冲突(a > 下游并行度)→ 扩下游/加缓存/上 PgBouncer,而非硬开大池
```

**三语速记**:

- **Java**:HikariCP(`maximumPoolSize`,作者明确建议小池);连接池是 Java 生态最成熟的。
- **Go**:`database/sql` 的 `SetMaxOpenConns`/`SetMaxIdleConns`;`pgxpool`。默认无上限,**必须手动设** `SetMaxOpenConns`,否则高并发下连接数失控(Go 的常见坑)。
- **Python**:`asyncpg.create_pool(min_size, max_size)`、SQLAlchemy `pool_size`+`max_overflow`;多 worker 下尤其要算总账。

---

## 6. 一句话收口

池是 M/M/c 队列:大小下限是 `c > offered load a=λS`(否则排队爆炸,`starve.py` 可见),上限是下游的真实并行度(小池往往更快,HikariCP/PG 反直觉结论),两者之间留余量;而真正容易爆的雷是**总账** `单池 × worker × 副本`,它能把合理的 20 放大成打挂数据库的几千。**池配对了,`06` 才轮到问:这些池要不要互相隔离。**
