# 01 · Little 定律 —— 这门课的总钥匙

> 一个方程 `L = λ·W`,加上它的「阴暗面」`R = S/(1−ρ)`。认下这两个,后面 8 章都是它们的应用。这一章不配任何东西,只把钥匙磨亮。
>
> 前置:`00`。动手:`lab/sim/little.py`。

---

## 1. 一个几乎显然、却能定容整个系统的方程

```
L = λ × W
```

- **λ(lambda)**= 到达率,每秒进来多少请求(就是你说的 QPS)。
- **W** = 每个请求在系统里**停留**的总时间(从进来到出去,就是端到端延迟)。
- **L** = 任一时刻**在途**的请求数(正在被处理 + 在排队的总数)。

直觉:一家奶茶店每秒进 2 个客人(λ=2),每个客人从进门到拿到奶茶停留 5 分钟=300 秒(W=300),那么店里**随时**约有 `2 × 300 = 600` 个客人。你不需要知道客人怎么来的、做奶茶快慢怎么分布——**只要系统稳态(进出速率相等),这个等式就成立**。这就是 Little 定律的厉害:它**不依赖任何分布假设**,几乎是个恒等式。

把它对应到你的服务:

| 奶茶店 | 你的服务 | 它具体是什么 |
|---|---|---|
| λ 进店速率 | QPS | 每秒到达的请求数 |
| W 停留时间 | 端到端延迟 | P50/P99 latency |
| **L 店里人数** | **在途请求数** | **= 忙碌的线程数 / 被借出的连接数 / 在飞的 goroutine 数** |

最后一行是关键:**L 就是你要定容的那个东西**。「我需要多少线程」「连接池多大」「几台机器」,问的全是「L 会有多大」。

## 2. 把方程倒过来,就回答了三个定容问题

同一个等式,挪一下,就分别回答了管线上的三件事:

```
L = λ·W      →  我需要准备多大并发?      （给定流量和延迟,算在途数）
λ = L / W    →  这套配置能扛多少 QPS?     （给定并发上限和延迟,算吞吐天花板）
W = L / λ    →  并发受限时延迟会多差?      （给定并发上限和流量,算排队恶化）
```

**算例**:你的服务 P99 延迟 W=200ms,峰值 λ=5000 QPS。
→ 在途请求 `L = 5000 × 0.2 = 1000`。
→ 这 1000 个「同时在处理」的请求,就是你后面要分配给线程/协程/连接池/机器的总量。一台机器扛不下 1000 个并发的阻塞线程(光内存就爆),于是你立刻知道:要么换并发模型(`02`),要么多开机器(`09`)。**一个乘法,就把「凭感觉」变成了「有数」。**

> 这里藏着一个常被忽略的事实:**降低 W 会直接降低你需要的并发 L**。把一个下游调用从 200ms 优化到 50ms,不只是「快了」,是让同样的 QPS 下在途数掉到 1/4 ——线程、连接、内存需求全都跟着掉。延迟优化是容量优化。

## 3. 阴暗面:为什么利用率一高,延迟不是变差,是「炸」

Little 定律告诉你**稳态**下的 L,但没告诉你一件要命的事:**当系统快被填满时,W 自己会失控地涨**。这是定容里最反直觉、也最重要的一课。

先定义**利用率 ρ(rho)**:

```
ρ = λ / μ = λ × S      （μ=每秒能服务完多少个;S=单个请求的纯服务时间）
```

ρ=0.5 就是「一半时间在干活,一半空着」;ρ=1 就是「满负荷,一刻不停」。你的直觉可能是:ρ 越高越省钱,配到 90%、95% 最划算。**这个直觉会让你上线就炸。**

### 3.1 M/M/1:`R = S / (1 − ρ)`

对一个最简单的排队系统(请求泊松到达、服务时间指数分布、1 个服务台,记作 M/M/1),平均响应时间是:

```
R = S / (1 − ρ)
```

`S` 是纯干活时间(固定),关键全在 `1/(1−ρ)` 这一项。它在 ρ 接近 1 时**爆炸**——这是一个 30 秒就能记住的函数行为:分母 `1−ρ` 趋近 0,整个值趋近无穷。代进去看:

| ρ(利用率) | 1−ρ | R = S/(1−ρ) | 相对空载放大 |
|---|---|---|---|
| 0.5 | 0.5 | 2·S | ×2 |
| 0.8 | 0.2 | 5·S | ×5 |
| 0.9 | 0.1 | 10·S | ×10 |
| 0.95 | 0.05 | 20·S | ×20 |
| 0.99 | 0.01 | 100·S | **×100** |

**从 90% 推到 99% 利用率,你只多榨了 10% 的机器,延迟却涨了 10 倍。** 这就是 P99 那根「曲棍球杆」的数学来源——不是玄学,就是 `1/(1−ρ)`。

```bash
# 动手现在就看一眼:
python lab/sim/little.py --service-time 0.01
# ρ=0.9 → 100ms,ρ=0.99 → 1000ms,bar 一下顶到底
```

**定容上的直接结论**:别把目标利用率配到 90% 以上。生产里 CPU/池的**稳态目标通常留在 60–70%**,把那段陡峭的尾巴让出来给突发。配得太满 = 把自己顶在曲棍球杆的拐点上,一点抖动就翻车。这也是为什么后面每一章算池/线程,都会**留余量**,而不是卡着算出来的最小值。

### 3.2 为什么是「指数感」而不是线性

很多人记成「负载越高越慢」,但漏掉了**慢的速度本身在加速**。线性的世界里,负载 +10% 延迟 +10%;真实排队的世界里,负载从 80%→90% 延迟翻倍,90%→95% 再翻倍。原因很物理:利用率越高,新请求来时**前面有人没干完**的概率越大,得排队;而队列一旦开始堆,后面的人等的是「前面所有人 + 自己」,恶化是复利的。`1/(1−ρ)` 就是这个复利的闭式解。

## 4. 第二个阴暗面:真实流量不平滑,你比公式以为的更早饱和

M/M/1 假设请求像节拍器一样泊松到达。**真实流量不是**——它一阵一阵的(整点定时任务、营销推送、重试风暴)。排队论里用 **Kingman 公式(VUT 形式)** 描述这件事,平均排队等待约为:

```
            ρ        C²a + C²s
W_q  ≈   ─────  ×  ───────────  ×  S
          1−ρ          2
         （利用率项）   （变异性项）   （服务时间）
```

中间多出来的 **`(C²a + C²s)/2`** 是**变异性项**:`C` 是变异系数(标准差 ÷ 均值),衡量「到达」和「服务」有多不平滑。

- 完全平滑(每个请求间隔一样、耗时一样)→ C²=0 → 几乎不排队。
- 泊松到达 → C²=1 → 退回 M/M/1。
- **突发流量 / 重尾服务时间** → C² 远大于 1 → 排队等待**成倍放大**。

**这解释了一个让无数人翻车的现象:监控显示 CPU 平均才 50%,服务却已经在超时。** 因为「平均 50%」掩盖了「峰值瞬间冲到 100%」——变异性大,你在均值还很低时就已经周期性饱和了。**定容不能只看均值,要看峰值和分布。** 这也是 `03 测需求` 要专门讲峰均比、`08 监控` 要盯饱和度而不是平均利用率的原因。

> 一句话:`1/(1−ρ)` 告诉你「满了会炸」,Kingman 告诉你「不平滑的流量会让你**提前**炸」。两个一起,才是真实系统的延迟模型。

## 5. 第三个坑:你的压测数字本身在骗你(coordinated omission)

定容要靠测量(量出 λ、W、ρ),但**最常用的压测姿势会系统性地低报尾延迟**,这个坑叫 **coordinated omission(协同遗漏)**,Gil Tene 提出。

机制:压测工具通常是「发一个请求 → 等它回来 → 再发下一个」。当服务器卡住 1 秒时,这种工具**自己也跟着停了 1 秒不发新请求**——于是那 1 秒里「本该发生但被卡住」的几百个慢请求,**根本没被记录**。结果:真实世界里用户疯狂踩到的慢请求,在你的压测报告里被「协同遗漏」掉了,P99 看着很美,上线就打脸。

修正:用**按固定速率发送、不等回包**的压测模式(open-model load),并用 **HdrHistogram** 这类能做 coordinated-omission 修正的工具记录延迟。`07 load-testing` 那门课的工具就支持这个。

**对定容的意义**:你算 ρ、估 W 时,确认手上的延迟数字不是被遗漏污染过的——否则你会基于一个偏乐观的 W 去定容,留的余量根本不够。

## 6. 三语锚点:L 在你的运行时里具体是什么

`L`(在途数)不是抽象概念,它在每个运行时里都有一个你能直接监控的对应物:

| 运行时 | L = 在途数,具体看 | 它的天花板(满了就排队/拒绝) |
|---|---|---|
| **Java** | 线程池 active count;或借出的连接数 | 线程池 `maximumPoolSize`、连接池 max(`04`/`05`) |
| **Go** | 在飞的 goroutine 数(`runtime.NumGoroutine` 粗看) | 你自己用 semaphore/worker pool 设的并发上限 |
| **Python** | event loop 里在途的 Task/请求数 | worker 进程数 × 每进程并发(`04`) |

注意一个关键差异(`02` 细讲):Java 一个在途请求**占一个线程**(贵,L 受线程数硬限);Go/Python-async 一个在途请求只占一个 goroutine/协程(便宜,同样的 L 内存小得多)。**同样的 `L=λW` 算出来的在途数,在不同模型下成本天差地别**——这正是「选并发模型」(`02`)要解决的。

## 7. 一句话收口

`L = λ·W` 把「我要准备多大并发」变成一个乘法;`R = S/(1−ρ)` 告诉你利用率超过 ~80% 后延迟会以 `1/(1−ρ)` 的速度炸,所以要留余量;Kingman 告诉你流量不平滑会让你**提前**炸,所以要看峰值不看均值;coordinated omission 提醒你**测量本身可能在骗你**。这四件事,是后面每一章定容时的共同底座。
