# 08 · 在线监控并发 —— 盯对了信号,才能在雪崩前看见它

> 定容是上线前的事,但流量会变、依赖会抖。这一章讲上线后该盯哪几个信号——以及为什么「CPU 利用率」这个所有人都在看的指标,恰恰是最会骗你的那个。
>
> 前置:`01`(L、ρ、饱和)、`07`(过载)。动手:`lab/experiments/e08-read-the-signals.md`。

---

## 1. 监控并发,本质是给 `L=λW` 装上活体读数

整门课的主线 `L=λW` 不只是定容时算一次的公式,它的每个量在线上都是一个**实时信号**。监控并发,就是把这个方程的四个量画出来:

| 信号 | 是方程里的 | 它告诉你 |
|---|---|---|
| **到达率 λ** | λ | 进来多少(QPS)——需求侧 |
| **延迟 W** | W | 每个请求停留多久(P50/P99)——体验侧 |
| **在途数 L** | L | 此刻有多少在处理(忙线程/借出连接/在飞协程)——负载侧 |
| **利用率 ρ + 队列深度** | ρ | 离饱和有多近——**最关键,见下** |

如果只能加一个新指标,加 **L(在途数)和队列深度**——大多数团队盯着 λ 和 W,却没盯 L,而 L 才是「容量用了多少」的直接读数。

---

## 2. 底层①:利用率会骗你 —— 50% 也可能在过载

几乎所有人上来就看 **CPU 利用率**。但它是定容监控里**最会骗人**的指标,原因有二:

**其一,它是平均值,掩盖峰值。** 「CPU 平均 50%」可能是「半秒 100% 满载 + 半秒空闲」的平均。`01` 的 Kingman 讲过:流量变异性(高 C²)大时,你在均值很低时就已经周期性饱和、周期性丢请求了。**均值 50% ≠ 安全**,它可能每秒都在尖峰处超时。

**其二,它是滞后指标。** 等 CPU 利用率涨上来,排队**早就开始**了。利用率告诉你「现在多忙」,但你需要的是「快撑不住了」的**领先**信号。

> 这解释了那个经典困惑:「监控显示 CPU 才 60%,服务为什么在超时?」——因为利用率掩盖了峰值、且落后于排队。光看利用率定容/告警,你永远是在事后救火。

---

## 3. 底层②:饱和度(队列深度)才是领先指标

`01`/`07` 反复出现的「队列」,在监控里是**最该盯的领先信号**。

- **利用率** = 资源有多忙(0–100%,封顶,滞后)。
- **饱和度(saturation)** = **排队等待的工作量**(无上限,领先)。run-queue 长度、连接池等待数、线程池队列长度、消息积压。

队列深度的好处:它**在延迟飙升之前就开始涨**。利用率到 100% 封顶后就不动了(你看不出「过载了多少」),但队列深度会继续增长,直接量出过载的程度。**排队一开始堆积,就是过载的第一声警报**——比延迟飙升早,比错误率早。

**告警要建在饱和度上,不是利用率上。** 「连接池等待数 > 0 持续 N 秒」「线程池队列 > 阈值」「消息积压增长」——这些在用户感知到慢之前就会响。

## 4. 底层③:两套互补的方法论 —— USE 与 RED

业界两套标准监控框架,定容要**两个都用**,因为它们是两个视角:

| | **USE**(Brendan Gregg) | **RED**(Tom Wilkie) |
|---|---|---|
| 视角 | **资源**(机器/池) | **请求**(服务) |
| 三个量 | **U**tilization 利用率 / **S**aturation 饱和度 / **E**rrors 错误 | **R**ate 速率 / **E**rrors 错误 / **D**uration 延迟 |
| 回答 | 「哪个资源先扛不住」(`04` 的 USE 选机型) | 「用户体验如何」 |
| 对应本课 | ρ、队列深度(§2/§3) | λ、错误率、W |

- **USE** 用来做容量诊断:逐个资源(CPU/内存/网络/磁盘/池)看 U-S-E,找出**先饱和的那个**(就是 `04` 选机型的方法,只是搬到线上持续看)。
- **RED** 用来做服务 SLO:每个服务看 Rate-Errors-Duration,守住 P99 和错误率的承诺。

两者交叉:RED 告诉你「服务慢了」,USE 告诉你「因为哪个资源饱和了」。缺一个,你要么知道病了不知道病因,要么盯着资源却不知道用户疼不疼。

## 5. 底层④:聚合陷阱 —— 你不能对 P99 求平均

一个会让监控数据**直接错掉**的坑:**百分位数不能跨实例/跨时间简单平均或相加。**

「10 台机器,每台 P99=100ms,所以整体 P99=100ms」——**错**。也不能把 10 个 P99 加起来除以 10。P99 是分布的一个点,不同实例/时间窗的分布不同,算术平均出来的数没有任何统计意义,通常**严重低估**真实的整体尾延迟。

正解:监控系统要在**原始直方图(histogram)层面合并**,再从合并后的分布重新算百分位。Prometheus 的 `histogram_quantile` 作用于 bucket、HdrHistogram 能无损合并——用对工具,别在仪表盘上手动平均百分位。

> 这和 `01` 的 coordinated omission 同源:**尾延迟的测量与聚合都有陷阱,拿到一个偏乐观的假 P99,会让你以为容量够、其实不够。** 定容依赖的数字,本身得是对的。

## 6. 该上墙的一组信号(定容监控清单)

把前面收成一块实战仪表盘:

```
需求侧(RED-R):  λ / QPS,按端点拆
体验侧(RED-D):  P50 / P99 / P999 延迟(直方图合并,别手算)
体验侧(RED-E):  错误率、503/429(shed)计数
负载侧(L):       在途请求数、各池的借出数 / 使用率
饱和度(USE-S):  ★ 池等待数、线程池队列长、run-queue、消息积压 —— 告警建这儿
资源侧(USE-U):  CPU / 内存 / 网络 / fd,逐个看,找先饱和者
过载动作:        重试率、断路器状态、限流触发数(呼应 07)

告警优先级:饱和度 > 错误率 > 延迟 > 利用率   （越靠前越领先)
```

`lab/service` 的 `/stats` 端点(`in_flight`、`max_in_flight`、`rejected`)就是这套的最小实现——把 L、饱和、shed 暴露出来给监控抓。

**三语速记**(怎么把这些指标吐出来):

- **Java**:Micrometer + Prometheus,线程池/HikariCP 自带 metrics(活跃数、队列长、等待),直接接。
- **Go**:`prometheus/client_golang` + `expvar`;goroutine 数、池等待自己埋点。
- **Python**:`prometheus-client` 或 OpenTelemetry;event-loop 滞后、池使用率埋点(`asyncio` debug 模式可测 loop 卡顿)。

## 7. 一句话收口

监控并发就是给 `L=λW` 装上活体读数:**利用率会骗你**(平均掩盖峰值、且滞后),**饱和度/队列深度才是领先指标**,告警要建在它上面;**USE(资源视角)+ RED(请求视角)两套都要**,一个找病因一个守体验;**百分位不能手动平均**,要在直方图层面合并。盯对了信号,你能在雪崩前看见它。**线上信号也是 `09` 自动扩缩容的触发依据。**
