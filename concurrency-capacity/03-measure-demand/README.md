# 03 · 测算真实并发需求 —— λ 和 W 到底从哪来

> `01` 给了公式 `L=λW`,`02` 选了模型。现在要把 λ 和 W 这两个数**从真实系统里量出来**——而且量对。这一章讲的全是「为什么你凭直觉估的那个数,几乎总是偏低」。
>
> 前置:`01`、`02`。动手:`lab/sim/little.py`、`lab/experiments/e04-find-saturation.md`。

---

## 1. 两个输入:λ 怎么拿,W 怎么拿

定容的起点是两个数,都来自**测量**,不是拍脑袋:

| 量 | 从哪拿 | 注意 |
|---|---|---|
| **λ 到达率** | 访问日志 / 网关 / APM 的 QPS 曲线 | 取**峰值**那一段,不是日均(见 §2) |
| **W 停留时间** | APM / trace 的端到端延迟 | 取**分布**(P50/P99),不是只看均值(见 §4) |

拿到后 `L = λ_peak × W` 就是你要准备的并发量。听起来简单,但**这一步最容易翻车**,因为三个地方都会让你低估:峰均比、fan-out 放大、重尾延迟。这一章就是逐个拆这三个坑。

---

## 2. 坑①:用日均 QPS 定容 = 给自己挖坑(峰均比)

最常见的错误:「我们一天 8640 万次请求,86400 秒,那就是 1000 QPS,按 1000 配。」

**错。** 真实流量从来不是均匀摊开的:

- 有日内波峰(午高峰、晚高峰),峰值常是日均的 **3–10 倍**。
- 有突发尖刺:整点定时任务、营销推送、缓存集体过期、上游重试风暴——瞬时能冲到日均的几十倍。

**定容必须按峰值,而且按「峰值 + 突发余量」。** 一个实用做法:

```
λ_设计 = 日均QPS × 峰均比 × 突发系数
       = 1000   × 5(看历史曲线)× 1.5(留给尖刺)
       = 7500 QPS
```

峰均比怎么来?**看你自己的历史监控曲线**,量出「峰值小时 QPS ÷ 全天平均 QPS」。没有历史数据的新服务,先按行业经验取 3–5,上线后用真实曲线回填。

> 回扣 `01`:为什么不能卡着峰值配、还要留突发系数?因为 `R=S/(1−ρ)`——配到刚好等于峰值,意味着峰值时 ρ→1,延迟进入曲棍球杆。留余量不是浪费,是不让自己坐在拐点上。

---

## 3. 坑②:fan-out 尾部放大 —— 1 个请求在内部裂变成 N 个

这是大公司高并发系统里**最被低估**的一个放大器,也是面试高频深水区。

你的服务收到 1 个用户请求,内部往往要调用 N 个下游(查用户、查库存、查价格、查推荐……),等它们全回来才能响应。两个后果:

**后果一:下游的 λ 被乘 N。** 你前端 7500 QPS,某个被每个请求都调用的下游,它的 λ 是 7500×(调用次数)。**给下游定容,要用放大后的 λ,不是前端的 λ。** 漏算这个,下游池/线程一定配小。

**后果二(更阴险):你的 W 被「最慢的那个」绑架(tail-at-scale)。** 这是 Google 的 Dean & Barroso 在《The Tail at Scale》里点破的:你要等**所有** N 个下游都回来,所以你的延迟 = N 个里**最慢的那个**。哪怕每个下游单独看只有 1% 的请求慢(P99),N 个一扇出:

```
至少命中一次慢调用的概率 = 1 − (0.99)^N
  N=1   →  1%
  N=10  →  9.6%
  N=100 →  63%      ← 扇出 100,有 63% 的用户请求会踩到至少一个 P99 慢下游
```

**也就是:每个下游都「P99 很好」,但你的前端 P99 可能惨不忍睹**——因为扇出把「小概率慢」放大成了「大概率慢」。

对定容的意义:

- 扇出大的服务,**前端 W 要按「被最慢下游拉高后」估**,不能拿单个下游的 P50 乐观估。
- 这也是为什么大厂砸资源做**对冲请求(hedged requests)**、**下游 P99 压降**——因为扇出会把任何一个下游的尾延迟放大成全局问题。

> 一句话:扇出同时放大了**下游的 λ**(乘 N)和**你的 W**(被最慢者绑架)。两个都让你需要的并发 L 变大,且都容易被漏算。

---

## 4. 坑③:用均值估 W = 系统性低估(重尾)

第三个坑在 W 本身。延迟分布几乎总是**右偏重尾**的:大多数请求很快,少数请求慢得离谱(GC、慢查询、冷缓存、网络重传)。这种分布里:

- **均值被长尾拉高,却又掩盖了尾巴有多长**——P99/P999 可能是 P50 的几十倍。
- 用均值 W 算 L,会**双重出错**:既没反映尾部用户的真实体验,又因为忽略了 `01` 讲的 Kingman 变异性(高 C²),低估了排队恶化。

**正解:用分布定容,不用均值。**

- 容量按 **P99 的 W** 算并发(保证尾部请求也有资源)。
- 监控盯 P99/P999 而非均值(`08` 细讲)。
- 警惕「均值看着没事」的假象——它最会骗人。

> 还记得 `01` 的 coordinated omission 吗?如果你的 W 数据来自「发一个等一个」的压测,尾巴本身就被遗漏污染了,你拿到的 P99 是偏乐观的假数。**测 W 之前,先确认数据采集姿势对**(开环压测 + HdrHistogram)。

## 5. 把三个坑合进一个估算流程

给一个服务定容,需求测算的完整动作:

```
1. 拿日均 QPS                      （日志/APM）
2. × 峰均比 × 突发系数 → λ_设计     （看历史曲线,坑①）
3. 数清扇出:每个下游被调几次       （trace,坑②后果一)
4. W 按「P99 + 扇出最慢者」估       （坑②后果二 + 坑③)
5. L = λ_设计 × W,并对每个下游单独算 λ_下游 = λ_设计 × 扇出次数
6. 目标利用率压到 60–70%,反推所需容量（坑①余量 + 01 的 1/(1−ρ))
```

这个 L(和每个下游的 λ)就是 `04`(线程/单机)、`05`(池)的输入。**测错需求,后面配得再精细都是错的**——所以这一步值得花时间量准。

**三语锚点(这些数从哪个工具拿)**:不强语言相关,关键是有 APM/trace 在跑:

- **Java**:Micrometer + Prometheus 出 QPS/延迟分布;trace 用 SkyWalking/Zipkin 数扇出。
- **Go**:`expvar`/prometheus client 出指标;OpenTelemetry trace 数扇出。
- **Python**:`prometheus-client` 或 OTel 出指标 + trace。

---

## 6. 一句话收口

λ 和 W 都来自测量,但三个放大器让你的直觉估值几乎总偏低:**峰均比**(按峰值不按日均)、**fan-out 尾部放大**(下游 λ 乘 N、自己的 W 被最慢下游绑架,`1−0.99^N`)、**重尾延迟**(按 P99 不按均值)。把这三个都算进去、再压个目标利用率留余,得到的 L 才是 `04/05` 能放心拿去定容的真实需求。

> **动手:跑这个实验**
> ```bash
> # 用「测出来的」λ 和 W 算在途数:
> cd concurrency-capacity
> python lab/sim/little.py --lam 7500 --w 0.2     # L=1500,这就是你要分配的并发量
> ```
> 然后用 `lab/experiments/e04-find-saturation.md` 把真实服务压到饱和,**反过来测量**它实际能扛的 λ 和此时的 W——对照你估的数,校准你的峰均比和余量。
