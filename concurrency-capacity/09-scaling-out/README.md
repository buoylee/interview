# 09 · 横向扩容与成本 —— 把定好容的单机,乘成一个机群

> 单机定容(04–08)做完了,最后一步:乘起来。但「几台」不是简单除法——纵向加机器有 USL 天花板,横向加机器有协调成本,自动扩缩容该看的信号也不是 CPU。这一章把单节点接到机群。
>
> 前置:`04`(单机容量)、`08`(监控信号)。动手:`lab/sim/saturate.py`。

## 1. 从单机到机群:一个除法,三个修正

最朴素的算法:

```
机器数 = 总需求 L / 单机容量 L_node
```

但这个除法要打三个补丁,才是能上线的数:

1. **按峰值,不按均值**(回 `03`):用 `λ_设计`(含峰均比 + 突发系数)。
2. **留冗余(N+1 / N+2)**:挂一台、滚动发布、单 AZ 故障时,**剩下的机器仍要扛住峰值**。不是按「刚好够」配,是按「死掉一个故障域还够」配。
3. **算上扩容滞后**:自动扩容需要时间(拉起实例 + 预热),要么留足缓冲,要么提前扩。

所以真实的机群数 ≈ `(峰值需求 / 单机容量) × 冗余系数`,且要能容忍一个故障域整体消失。**这是「几台机器」的完整答案,不是一个除法。**

---

## 2. 底层①:为什么不能无限「加核」—— USL 的纵向天花板

定容有两个方向:**纵向**(scale up,单机加核/加内存)和**横向**(scale out,加机器)。先说纵向为什么有上限——又是 `01`/`04` 的 USL。

给单机加核,吞吐**不会线性涨**:

- **σ(争用)**:程序里的串行部分(锁、单点)不会因为加核变快——Amdahl 定律,吞吐趋于 `1/σ` 封顶。
- **κ(协调)**:核越多,跨核同步、缓存一致性流量(cache coherency)越贵——这一项让吞吐**过峰后回退**。

```bash
# 单机纵向扩展的天花板:加并发到 N≈31 见顶,再加反而降
cd concurrency-capacity
python lab/sim/saturate.py --sigma 0.03 --kappa 0.001
```

**结论:纵向扩展到 USL 峰值附近就该停,再加是负优化。** 过了这个点,唯一的出路是**横向**——多台中等机器,而不是一台巨型机。这给了「纵向还是横向」一个量化判据:**纵向到 USL 峰,之后横向。**

## 3. 底层②:纵向 vs 横向的经济学

| | 纵向(scale up) | 横向(scale out) |
|---|---|---|
| 扩展性 | 有 USL 天花板,且大机器单价**非线性贵** | 近似线性扩(在协调成本内) |
| 新增成本 | 越大越贵(高端 CPU/大内存溢价) | 加同款中等机器,边际成本平 |
| 代价 | 简单(不用改架构) | 加了协调成本:负载均衡、状态外移、网络、数据一致性 |
| 适合 | 有状态、难拆的组件(单机数据库主库) | 无状态服务(Web/API 层) |

**每 RPS 成本**是定容的经济视角:把「机群月成本 ÷ 支撑的峰值 RPS」算出来,你会发现——

- 无状态服务横向扩,每 RPS 成本基本是平的(线性),所以**优先把服务做成无状态**,扩容才便宜。
- 纵向扩的每 RPS 成本随规模上升(大机器溢价 + USL 边际递减),所以巨型机只在「实在拆不开」时用。
- 这也是为什么云原生时代默认「无状态 + 横向 + 自动扩缩」——它把容量变成了一条**线性可预测的成本曲线**,而不是一道纵向的墙。

## 4. 底层③:自动扩缩容该看什么信号(不是 CPU)

K8s HPA 默认按 CPU 利用率扩容。**但 `08` 讲过,CPU 利用率是滞后 + 会骗人的指标**——等它涨上来,你已经在排队、在丢请求了,这时才扩,实例还要几十秒才就绪,黄花菜都凉了。

**更好的扩容信号是领先指标**(呼应 `08`):

| 信号 | 为什么好 |
|---|---|
| **在途并发数 / 队列深度** | 领先于延迟和 CPU,直接反映「容量用了多少」 |
| **RPS / 每实例并发** | 直接对应 `L=λW`,可预测 |
| **P99 延迟 / 饱和度** | 贴近 SLO,延迟要破线就扩 |

现代实践:HPA on **custom metrics**(并发数、队列长、RPS),而不是 CPU;或用 KEDA 按消息积压、请求数扩。**按领先信号扩,才能赶在过载前扩出来。** 配合 §1 的「留冗余 + 算扩容滞后」,不至于扩容追着过载跑。

> 这把整门课闭环了:`08` 盯的那些领先信号(饱和度、在途数),正是这里自动扩缩容的触发器。**监控对了,扩容才对。**

## 5. 平台中立的落地 + 边界

- **K8s**:HPA(按 custom metric)/ VPA(纵向)/ KEDA(事件驱动);Pod 副本数 = §1 的机群数。
- **云 ASG**:Auto Scaling Group 按目标跟踪(target tracking)扩缩。
- **故障域**:跨 AZ/Region 部署,冗余要能扛住整个 AZ 挂掉(§1 的 N+ 冗余落到故障域上)。

## 6. 一句话收口

机群数 ≈ `峰值需求 / 单机容量 × 冗余`,且要扛得住一个故障域消失;**纵向扩有 USL 天花板**(到峰即止,`saturate.py` 可见),之后走**横向**;无状态 + 横向让每 RPS 成本变成平的线性曲线,这是云原生默认姿势的经济学根因;**自动扩缩容要按领先信号**(并发/队列/RPS)而非滞后的 CPU,才能赶在过载前扩出来。再往上就交给 `system-design`。**至此,从需求数字到机群配置的整条管线走完了——去 `99` 把它收成面试答案。**
