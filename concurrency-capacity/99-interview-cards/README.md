## 深题卡:练资深深度(每张 = 问题 + 资深答法 + 证据)

### Q1 · 为什么负载一高,P99 延迟不是线性变差,而是「拐」上去?

**答**:因为响应时间服从排队论的 `R = S/(1−ρ)`(M/M/1),`S` 是固定服务时间,`ρ` 是利用率。关键是 `1/(1−ρ)` 这一项:ρ 从 0.9→0.99,只多榨 10% 的机器,`1/(1−ρ)` 从 10 涨到 100,延迟翻 10 倍。物理直觉:利用率越高,新请求来时前面有人没干完的概率越大,排队恶化是复利的。**所以容量目标利用率压在 60–70%,把陡峭的尾巴留给突发。**

> 证据:[01 §3](../01-littles-law/);亲手跑 `python lab/sim/little.py --service-time 0.01`。

### Q2 · 利用率才 50%,为什么服务已经在超时?

**答**:两个原因。①利用率是**平均值**,掩盖峰值——Kingman 公式里的变异性项 `(C²a+C²s)/2`,流量越突发(高 C²),你在均值还很低时就已经周期性饱和、周期性丢请求了。②利用率是**滞后指标**,等它涨上来排队早就开始了。**所以监控/告警要看饱和度(队列深度)这个领先指标,而不是利用率。**

> 证据:[01 §4](../01-littles-law/) + [08 §2-3](../08-monitoring-concurrency/)。

### Q3 · 为什么把连接池开大反而更慢?

**答**:连接池是 M/M/c 队列,大小下限是 `c > offered load`(否则排队爆炸)。但上限受**数据库自身的并行处理能力**(核数+iops)限制:连接数超过它,这些连接只会在 DB 内部疯狂抢锁、抢缓存、上下文切换(USL 的 κ 项),吞吐不升反降——HikariCP/PostgreSQL 的经典结论,推荐池小到 `(核数×2)+磁盘数`。**池要"刚好喂饱下游并行度",不是越大越好;`a` 超过下游并行度时该扩下游/加缓存,而非硬开大池。**

> 证据:[05 §2-3](../05-pools/)。

### Q4 · 一个下游短暂抖动,怎么会演变成全面雪崩?

**答**:重试风暴的正反馈。下游抖一下 → 上游失败 → 每个失败重试 N 次 → 打到下游的负载 ×N → 本来小故障被几倍负载彻底打死 → 更多失败 → 更多重试。**单看「重试 3 次」很合理,乘上成千上万客户端就是一场自发 DDoS。** 驯服:指数退避+抖动(避免同时重试)、重试预算(限重试流量占比)、断路器(连续失败就快速失败给下游喘息)、只重试幂等操作。更阴险的是 **metastable failure**——触发器消失了,系统靠重试/缓存击穿的正反馈仍卡在坏稳态,必须主动降载才能打破。

> 证据:[07 §3-4](../07-overload-backpressure/)。

### Q5 · 「线程数 = 2×核数+1」对吗?什么时候不对?

**答**:它只是真公式 `核数 × (1+等待/计算)` 在「等待≈计算」时的特例(+1 是补位冗余)。CPU 密集(等待=0)应该 ≈核数;I/O 重(等待是计算的 9 倍)该开 10×核数;极端 I/O(50 倍)该开几百——这时 thread-per-request 的内存/切换成本撑不住,该换 event-loop。**所以没有万能线程数,得先量出"等待/计算"比值;且上限受 USL 峰封顶(线程太多吞吐回退),最终压测定准。**

> 证据:[04 §2](../04-sizing-one-node/);跑 `python lab/sim/saturate.py` 看过峰回退。

### Q6 · 一个慢下游怎么把不相干的接口也拖垮?怎么防?

**答**:队头阻塞 + 共享池耗尽。慢下游让处理它的请求长时间占住共享池资源(占用时长 S 暴涨 → offered load 冲破池容量),新请求(包括不依赖该下游的)拿不到资源、队列无界增长、整个实例被拖死并级联到其他实例。**防法:舱壁隔离**——按 快/慢·关键/非关键 给不同负载独立的池/信号量,一个进水不沉全船;多租户用 shuffle sharding 限制爆炸半径。

> 证据:[06](../06-isolation/);跑 `python lab/sim/bulkhead.py` 看 fast rejected 从 427→0。

### Q7 · 给你一个新服务「10k RPS,P99<200ms」,你怎么端到端定容?

**答**(把整条管线串起来,这是压轴):
1. **测需求**:λ=10k×峰均突发余量;W 按 P99 且算 fan-out(下游 λ 乘扇出次数、自己 W 被最慢下游绑架)→ `L=λW`。
2. **选模型**:I/O 密集选 event-loop/async,CPU 密集多进程;典型 Web → 多进程×async。
3. **单机**:线程数 `核数×(1+等待/计算)`,USL 峰封顶;USE 找先饱和资源选机型。
4. **池**:每个下游 `c>offered load` 且 ≤下游并行度,算 ×worker×副本 总账防打挂 DB。
5. **隔离**:关键/非关键、快/慢拆池。
6. **过载**:有界队列+shed、退避重试+预算+断路器、deadline 传播。
7. **监控**:盯饱和度(队列)领先指标,USE+RED,告警建在饱和度。
8. **扩容**:`峰值/单机容量×冗余`,纵向到 USL 峰后横向,自动扩缩按并发/队列。
**每个数字都能说出第一性原理依据——这就是"会定容"和"背口诀"的区别。**

> 证据:全九章;速答表是它的索引。
