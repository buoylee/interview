# 04 · 单机配置与线程/worker 数 —— 「2×核数+1」到底从哪来

> 你一直背的那个「线程数 = 2×核数+1」是个**特例**,不是公式。这一章把真公式推出来,告诉你它什么时候成立、什么时候差十倍,再讲单机该配多大规格、为什么线程不是越多越好。
>
> 前置:`03`(拿到 L 和各下游 λ)。动手:`lab/sim/saturate.py`、`lab/experiments/e04-find-saturation.md`。

---

## 1. 从「需要多大并发」到「开几个线程」

`03` 给了你所需并发 L。单机定容就是回答:**这台机器开几个线程/worker,才能撑住分给它的那部分 L?**

直觉链条:

```
单机要扛的并发 L_node  =  总 L / 机器数        （机器数见 09,先假设已知或=1)
单个线程能同时扛的并发 = 1                      （一个阻塞线程同时只处理一个请求)
→ 阻塞模型下:线程数 ≈ L_node
```

但「线程数 ≈ L_node」对 I/O 密集会大得离谱(L_node 可能上千),这时要么换 event-loop(`02`),要么理解线程到底该配多少——这就要那个真公式。

---

## 2. 底层①:推导真公式,顺便看穿「2N+1」

先问一个根本问题:**一个线程处理一个请求时,有多少时间在真干活(占 CPU),多少时间在干等(等 I/O)?**

设一个请求:计算耗时 `C`(占 CPU),等待耗时 `Wait`(等 DB/网络,不占 CPU)。在它等待的那段时间里,**这个核是空的,可以让另一个线程上来干活**。所以要喂饱一个核,你需要的线程数是:

```
每核线程数 = (C + Wait) / C = 1 + Wait/C
总线程数  = 核数 × (1 + 等待时间/计算时间)
```

这就是真公式。代几个数感受一下:

| 负载类型 | Wait/C | 每核线程数 | 8 核该开 |
|---|---|---|---|
| 纯 CPU(无 I/O) | 0 | 1 | **8**(≈核数) |
| 计算 = 等待(各一半) | 1 | 2 | 16 |
| 典型 Web(等 DB 为主,Wait≈C) | ~1 | ~2 | ~17 = **2×8+1** ← 「2N+1」 |
| I/O 重(等待是计算的 9 倍) | 9 | 10 | **80** |
| 极端 I/O(等待是计算 50 倍) | 50 | 51 | 400 → 该换 event-loop 了 |

**看穿了吗:**

- **「2×核数+1」只是「Wait≈C」这一个特例**(那个 +1 是给某个线程偶尔缺页/让出时补位的经验冗余)。把它当万能公式用,在 I/O 重的服务上会配小 5–10 倍,白白让核空转。
- **CPU 密集 → 线程数≈核数**(再多只是徒增切换)。
- **I/O 越重,线程数该越大**——直到大到 thread-per-request 的内存/切换成本撑不住,这就是 `02` 说的「该换 event-loop」的临界点。

> 这个公式也解释了 `03` 为什么要分别量计算和等待:**没有 Wait/C 这个比值,你根本算不出线程数**,只能套口诀。资深和背口诀的差别就在这。

## 3. 底层②:线程不是越多越好 —— USL 天花板

公式说「I/O 重就多开线程」,但**有上限**,而且过了上限不是「不再变好」,是**变差**。这是 `01` 提过的 USL(通用扩展定律)在单机上的体现。

加线程,吞吐先涨,但每多一个线程都带来两笔隐性成本:

- **争用(σ)**:线程抢同一把锁、同一个连接池、同一段临界区——串行部分越占比,加线程的收益越快见顶(Amdahl)。
- **协调/相干(κ)**:线程越多,上下文切换、CPU 缓存被互相刷掉(cache thrash)、调度开销越大。这一项让吞吐**不只是平台,而是回退**。

```bash
# 亲眼看它回退:
cd concurrency-capacity
python lab/sim/saturate.py --sigma 0.03 --kappa 0.001 --max-n 64
# 吞吐爬到 N≈31 见顶,然后掉头向下——线程开太多,总吞吐反而降
```

**所以「压测找拐点」不是偷懒,是在找 USL 的峰值 N。** 真公式给你一个**起点估值**,USL 告诉你**别越过那个峰**,压测告诉你**峰具体在哪**。三者配合,而不是只靠一个。

定容结论:线程/worker 数 = `min(公式估值, USL峰值)`,再留点余量。配到峰值以上纯属花钱买负优化。

## 4. 底层③:单机配多大规格 —— USE 找先饱和的资源

「线程数」定了「软件并发」,但机器本身能不能扛,取决于**哪个硬件资源先饱和**。用 **USE 方法**(Utilization 利用率 - Saturation 饱和度 - Errors 错误)逐个资源体检:

| 资源 | 看什么 | 先饱和 = 你需要的是 |
|---|---|---|
| **CPU** | 利用率、run-queue 长度(饱和度) | 更多核 / 计算型机型 |
| **内存** | 用量、swap、OOM | 更大内存机型(线程栈 × 线程数也算在内!) |
| **网络** | 带宽、连接数、`TIME_WAIT` 堆积 | 更高带宽 / 调内核参数 |
| **磁盘 I/O** | iops、await、队列深度 | SSD / 更高 iops |
| **文件描述符** | 打开的 fd 数 vs `ulimit` | 调 `ulimit -n`(高并发连接必踩) |

**关键洞察:不同服务被不同资源卡。** 一个算法服务被 CPU 卡(选高主频多核机);一个网关被网络+fd 卡(选高带宽、调内核);一个缓存被内存卡(选大内存机)。**选机型 = 找出你的先饱和资源,然后买它。** 拿统一规格套所有服务,要么浪费要么瓶颈。

> USE 的精髓是**饱和度**那一栏:CPU 利用率 70% 看着没满,但 run-queue 已经排长队(饱和),说明实际已经过载——这呼应 `01` Kingman 和 `08` 监控的主题:**别只看利用率,要看饱和度。**

## 5. Python 特例:为什么是「多进程 worker × 每进程 async」

Java/Go 一个进程就能吃满多核(`02`),线程公式直接用。**Python 因为 GIL,一个进程的纯 Python 代码只能用一个核**——所以 Python 的单机定容是两层:

```
进程数(worker)  ≈ 核数        （每进程一个 GIL,吃一个核;这样吃满多核)
每进程内并发     = async 协程   （I/O 密集,单线程 event-loop 扛大 L_node / worker)
```

`gunicorn`/`uvicorn` 的经验配法,正是这个的落地:

- **同步 worker**(每请求一个阻塞线程/进程):`workers ≈ 2×核数 + 1`——你认出来了,这就是 §2 的「Wait≈C」特例。
- **异步 worker**(uvicorn,每进程一个 event-loop):`workers ≈ 核数`,并发靠协程,不靠堆 worker。

```
gunicorn app -k uvicorn.workers.UvicornWorker -w <核数> ...
```

> 这把 `02`(选了多进程×async)和 `03`(算出 L_node)接上了:**进程数定多核利用,进程内 async 定 I/O 并发,两层各管一件事。** 别用同步 worker 配置去跑异步服务(白瞎了 event-loop),也别给异步 worker 开 `2N+1`(进程间抢核、徒增内存)。落地细节见 `python-concurrency/07`。

**三语速记**:

- **Java**:一个 JVM 吃满多核;线程池 `corePoolSize` 用 §2 公式;虚拟线程(Loom)让你「按 L 开线程」而不担心栈成本。
- **Go**:`GOMAXPROCS`=核数(默认),goroutine 数按 L 走、无需手算线程池;CPU 密集自动并行。
- **Python**:`worker 进程数≈核数 × 每进程 async`,两层组合;CPU 密集另走 multiprocessing/C 扩展。

---

## 6. 一句话收口

线程数的真公式是 `核数 × (1 + 等待/计算)`,「2N+1」只是「等待≈计算」的特例;CPU 密集≈核数,I/O 越重数越大,直到 USL 峰值(`saturate.py` 看它过峰回退)封顶、压测定准。单机规格靠 USE 找出先饱和的资源再选机型。Python 因 GIL 多一层:进程数≈核数 吃多核,进程内 async 扛 I/O 并发。**这些数定了,`05` 才能算每个池配多大。**
