## 习题 1 · 主干:从需求到机群(踩「绑定约束不是并发,是瓶颈资源」)

**题面**:读多写少的 API,峰值 **16,000 RPS**。每个请求 = 5ms CPU 计算 + 45ms 等一次 DB 查询(W_mean=50ms)。机器是 8 核。问:用什么并发模型?单机能扛多少?要几台?

<details><summary>先自己算,再展开</summary>

**① 在途数(03 + Little)**:`L = λ × W_mean = 16000 × 0.05 = 800` 个并发在途。
> 注意用 **mean W** 算 L(Little 定律对均值精确);尾部余量靠后面压目标利用率留,别把 P99 直接塞进 Little。

**② 并发模型(02)**:`等待/计算 = 45/5 = 9`,强 I/O 密集。Thread-per-request 要 `8×(1+9)=80` 线程/机,还算扛得住,但扇出更高就爆——**选 event-loop/async**(Python:多进程×async;Go:goroutine)。

**③ 单机能扛多少 —— 找绑定约束(04 + USE)**:
- **CPU 天花板**:每请求 5ms CPU → 一个核 `1/0.005 = 200` req/s → 8 核 `= 1600` req/s/机。
- **并发是否是瓶颈?** 800 在途 ÷ 机器数,后面会算到每机只几十个在途——async 轻松。**所以绑定约束是 CPU 吞吐,不是并发数。** 这是关键洞察:别默认「并发」是瓶颈,USE 逐资源找真正先饱和的那个。

**④ 几台(09)**:目标利用率 ρ=0.65(留尾部余量,回 01)。
`机器数 = 16000 / (1600 × 0.65) = 16000 / 1040 ≈ 15.4 → 16 台`。
再加 N+2 冗余 + 跨 3 个 AZ(挂一个 AZ 仍扛峰值)→ **约 18 台**。
- 校验并发:800 在途 / 16 ≈ 每机 50 个在途,async 毫无压力,印证 CPU 才是绑定约束。

**收口**:L=800、async 模型、CPU 是瓶颈、~18 台跨 3 AZ。**先找绑定资源,再算机群。**
</details>

## 习题 2 · fan-out:你的尾延迟被下游绑架(踩「P99 ≈ 下游的 P99.95」)

**题面**:聚合服务,峰值 **5,000 RPS**。每个请求**并行**扇出到 **20 个**下游,每个下游各调一次,等全部回来才响应。单个下游延迟:P50=10ms,P99=30ms。问:每个下游要按多大 λ 定容?你自己的 W 该按多少估?

<details><summary>先自己算,再展开</summary>

**① 下游的 λ 被放大(03 后果一)**:每个下游被每个请求各调一次 → 每个下游承受 `5000 RPS`,下游侧总调用量 `5000 × 20 = 100,000 calls/s`。**给下游定容用 5000,不是把前端 5000 当总量。**

**② 你的 W 被「最慢的那个」绑架(03 后果二,tail-at-scale)**:你等 20 个里最慢的。设单下游延迟分布为 F,则你的延迟 = max of 20:
- 你的 **P50**:`F(x)²⁰ = 0.5 → F(x) = 0.5^(1/20) = 0.966` → **= 单下游的 P96.6**。
- 你的 **P99**:`F(x)²⁰ = 0.99 → F(x) = 0.99^(1/20) = 0.9995` → **= 单下游的 P99.95**。

**翻译**:**你的中位数 ≈ 下游的 P97,你的 P99 ≈ 下游的 P99.95**——你整个被下游的**深尾**绑架。下游「P99=30ms 看着不错」,但你的 P99 落在它的 P99.95,可能是 50–80ms。

**③ 定容结论**:
- W 按「下游深尾」估(≈下游 P99.95),别拿单下游 P50=10ms 乐观估;据此 `L = 5000 × W_放大`。
- 真正的解药是**压下游的尾**(下游 P99.95 降一点,你的 P99 大降)或上 **hedged requests**(对冲请求:慢了就并发再发一个,取先回的),这是大厂砸资源治尾延迟的原因。

**收口**:扇出同时放大下游 λ(×20)和你的 W(被 P99.95 绑架)。**两个都漏算,你的池和延迟预算一定配小。**
</details>

## 习题 3 · ×worker 总账:把 DB 打挂(踩「单池合理 ≠ 总账合理」)

**题面**:服务跑在 **12 台**机器,每台 8 核跑 **8 个** uvicorn worker,每个 worker 的 DB 连接池 `max_size=20`。Postgres `max_connections=500`。打 DB 的总查询量 8,000 queries/s,单查询 10ms。问:上线会怎样?怎么修?

<details><summary>先自己算,再展开</summary>

**① 总账(05 §4)**:`总连接 = 单池 20 × worker 8 × 机器 12 = 1920`。vs `max_connections=500` → **当场 `too many connections` 打挂**(超 3.8 倍)。单池 20 看着无比合理,乘起来要命。

**② 反推单池上限**:可用预算 = `500 × 0.7(安全系数) = 350`。
`单池 = 350 / (8 × 12) = 3.6 → 向下取 3`。总连接 = `3×8×12 = 288 < 350` ✓。

**③ 池=3 够用吗?查 offered load(05 §2)**:
- 每机查询量 `8000/12 ≈ 667/s` → 每 worker `667/8 ≈ 83/s`。
- 单池 offered load `a = λ_pool × S = 83 × 0.01 = 0.83 erlang`。
- 池=3 > 0.83(ρ≈0.28),**舒适,不饿死** ✓。两个约束(总账 ≤350、池 > offered load)同时满足。

**④ 如果冲突呢?** 假设查询量 10 倍(80k/s)→ offered load `a≈8.3` > 池上限 3 → **池太小会饿死,但总账又不允许开大**。这时正解**不是硬开大池**(会打挂 DB),而是:
- 上 **PgBouncer/ProxySQL** 外置连接池,解耦「应用并发」与「DB 连接数」;
- 或减 worker 数 / 加 DB 副本 / 加缓存降 DB λ。

**收口**:池大小是**全局约束**——脑子里必须同时装着 worker 数和副本数。冲突时改架构(外置池/扩 DB),不是改大数字。
</details>

## 习题 4 · 隔离决策:别让推荐拖垮支付(踩「非关键路径杀死关键路径」)

**题面**:一个服务里两个端点共用一个 64 槽的处理池:**支付**(关键,500 RPS,20ms)+ **推荐**(非关键,2000 RPS,平时 15ms,但模型重载时偶发飙到 **5s**)。问:有什么风险?要不要隔离?怎么落地?

<details><summary>先自己算,再展开</summary>

**① 风险 —— 队头阻塞级联(06 §2)**:推荐飙到 5s 时,它的 offered load `= 2000 × 5 = 10,000 erlang` 灌进 64 槽的共享池 → **瞬间占满**。支付请求(本来 `500×0.02=10` erlang,毫无压力)**拿不到槽**,排队、超时、失败。**一个可降级的推荐,通过共享池,杀死了不可降级的支付。**

**② 决策 —— 必须隔离**(满足「共享池 + 故障会传染」两个条件):
- 支付独立池:offered load 10 → 池 ~16(留余量),它永远摸不到推荐的抖动。
- 推荐独立池:平时 offered load `2000×0.015=30` → 池 ~48;飙 5s 时**它自己那份**被占满,**但支付完全不受影响**。

**③ 落地(06 §3)**:推荐会**卡死 5s**,所以不能只用信号量(信号量限了并发数,但卡死的调用仍占满它自己的配额 5s)——要么 **线程池隔离 + 200ms 超时**砍断卡死调用,要么 **信号量 + 客户端 deadline**。再加 **断路器**:推荐连续慢/失败就跳闸、直接降级返回兜底推荐,给它喘息(呼应 07)。

**收口**:识别「共享 + 会传染」→ 按关键/非关键拆池 → 每池按 05 各自定容 → 给会卡死的那个加超时+断路器。**牺牲非关键,保住关键。**
</details>

## 习题 5 · 过载:闪购冲垮 + 重试风暴(踩「等流量回落救不回来」)

**题面**:服务容量(拐点)= **10,000 RPS**。一次闪购把流量冲到 **15,000 RPS**(1.5× 容量)。客户端配了「失败重试 3 次」。问:不做保护会怎样?怎么扛?闪购结束后能自动恢复吗?

<details><summary>先自己算,再展开</summary>

**① 不保护 → 塌方 + 重试风暴(07 §1/§3)**:
- `λ=15k > μ=10k` → 队列每秒净增 5k → 无界增长 → 延迟雪崩 → 客户端超时。
- 超时触发重试:失败的那部分 ×3 打回来 → 有效负载冲向 **~25–30k** → 把本来「只是过载 1.5 倍」彻底打成「过载 3 倍」→ 全面崩溃。

**② 扛法(07 §2/§3/§4)**:
- **边缘限流 + shed**:令牌桶把入口卡在 10k(=容量),多出的 5k 立刻 `429/503` **快速失败**,而不是排队烂掉。有界,绝不无界排队。
- **重试预算**:限重试流量 ≤ 总量 10% → 重试最多把负载抬到 ~11k,掐断乘性放大。
- **断路器 + deadline 传播**:连续失败跳闸快速失败;每跳传剩余 deadline,杀掉注定超时的僵尸请求,别让 CPU 空耗。

**③ 闪购结束能自动恢复吗 —— 警惕 metastable(07 §5)**:
- 如果已进入坏稳态(重试风暴/缓存击穿自我维持),**流量回到正常也救不回来**——正反馈不依赖触发器。
- 必须**主动降载到坏稳态维持不住的水平以下**:临时关掉重试、强制 shed 大部分、预热缓存,等队列彻底排空再放量。

**收口**:过载靠 **shed 卡在容量 + 重试预算掐放大 + deadline 杀僵尸**;最危险的是 metastable,**打破它要主动降载,不是干等**。
</details>

## 一句话收口

五道题 = 把管线套五遍,每道踩一个真坑:**绑定约束是瓶颈资源不是并发**(1)、**fan-out 把你的 P99 绑到下游 P99.95**(2)、**单池合理但总账打挂 DB**(3)、**非关键路径经共享池杀死关键路径**(4)、**过载叠重试风暴会进 metastable**(5)。能把这五个推导链当场讲出来,你就从「读得懂」到「考得过」了。
