# 实验 e04 · 找到饱和点 —— 亲手摸到那根 P99 曲棍球杆

> 对应章节 [04 单机定容](../../04-sizing-one-node/)、[01 Little 定律](../../01-littles-law/)。证明:容量有一个明确的「拐点」,过了它 P99 暴涨、吞吐塌方。

## 目的

把服务从健康压到过载,**实测**它的饱和点,亲眼验证 `R=S/(1−ρ)`:利用率逼近 1 时延迟非线性爆炸。这个拐点就是单机容量 `L_node`。

## 跑什么

```bash
cd concurrency-capacity/lab/service
# POOL_SIZE=8,每请求占用 50ms → 池能力 = 8/0.05 = 160 RPS,拐点应在这附近
POOL_SIZE=8 uvicorn app:app --port 8000 --workers 1 &

python drive.py --url 'http://127.0.0.1:8000/slow?ms=50' --steps 40,80,160,320,640 --seconds 4
```

## 你会看到什么(实测形状)

| 目标 RPS | p50 | p99 | errors | 状态 |
|---|---|---|---|---|
| 40 | ~52ms | ~63ms | 0 | 健康(L=2 < 池 8) |
| 80 | ~52ms | ~58ms | 0 | 健康(L=4) |
| 160 | ~52ms | ~70ms | 少量 | **拐点**(L=8 = 池容量,开始拒绝) |
| 320 | **数秒** | **数秒** | 大量 | 过载:延迟雪崩 + qps 塌方 |

## 为什么

- 池能力 = `POOL_SIZE / 每请求占用时长` = 8/0.05 = **160 RPS**,这就是 offered load 等于池容量(ρ=1)的点。
- 在拐点前,延迟 ≈ 纯服务时间 50ms(没排队);逼近拐点,`1/(1−ρ)` 让 P99 抬头;过了拐点,λ>μ,队列无界增长,延迟以秒计、有效 qps 反而下降(`07` 的过载塌方)。
- **把 POOL_SIZE 当作"线程/worker 数"看**:这就是 `04` 说的——容量是个有拐点的曲线,定容要配在拐点**之前**留余量,而不是卡在拐点上。

> 收尾:`pkill -f "uvicorn app:app --port 8000"`
