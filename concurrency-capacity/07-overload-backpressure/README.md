# 07 · 过载与背压 —— 撑爆那一刻,系统是优雅降级还是雪崩

> 前面都在让系统**别**过载。但流量总有超出设计容量的一天。这一章是 3am 救命章:过了饱和点会发生什么、为什么无界队列是慢性自杀、重试如何把小抖动放大成全面崩溃、以及系统怎么会卡在「坏稳态」里出不来。
>
> 前置:`01`(饱和)、`05/06`(池/隔离)。动手:`lab/experiments/e07-overload-collapse.md`。

---

## 1. 过了饱和点,延迟不是变差,是「崩」

`01` 讲过 `R=S/(1−ρ)`:ρ→1 时延迟趋于无穷。但真实系统更糟——一旦到达率 λ **持续超过**服务能力 μ(即 ρ≥1),系统进入**没有稳态**的区域:

```
λ > μ  →  队列每秒净增 (λ−μ) 个请求  →  队列无界增长
       →  排队时间随之线性增长,且越积越多
       →  到请求被处理时,客户端早就超时了(白干)
       →  CPU 全花在处理「已经没人要的结果」上  →  有效吞吐趋近 0
```

你在 `lab` 里已经见过这一幕:`drive.py` 把服务压到 320 RPS(超过 pool 能力)时,P50 从 52ms 跳到 **6 秒**,QPS 从 140 **崩到 28**。**过载的特征不是「慢一点」,是「吞吐塌方 + 延迟雪崩同时发生」。** 这一章就是教系统在这一刻别躺平。

---

## 2. 底层①:无界队列是慢性自杀 —— shed vs block

很多人下意识给队列/池配「无限大」或很大的缓冲,觉得「能接住更多请求总是好的」。**恰恰相反**:

**无界队列 = 把延迟无限累积 + 把内存当垃圾场。** 请求进了一个无限长的队,要么排到天荒地老(客户端早超时,你还在为它排队),要么队列吃光内存直接 OOM。它把一个「快速失败」问题,拖成了「全部慢死 + 崩溃」问题。

正解:**有界队列 + 满了之后明确的策略**,二选一:

| 策略 | 做法 | 适合 |
|---|---|---|
| **Shed(卸载/拒绝)** | 队列满 → 立刻返回 503/429,**快速失败** | 在线请求:迟到的响应没价值,早拒绝早让客户端重试/降级 |
| **Block(阻塞/背压)** | 队列满 → 阻塞生产者,让压力**回传**给上游 | 流水线/消费者:让上游自然减速,别丢数据 |

**背压(backpressure)的精髓就是「把"我扛不住了"这个信号往上游传」**:要么拒绝(让上游知道并退避),要么阻塞(让上游慢下来)。最怕的是中间态——默默全收下、慢慢烂掉。你在 `lab/service` 里那个 `POOL_SIZE` 满了返回 503,就是 shed;TCP 的滑动窗口满了让发送方停,就是 block。

> 回扣 `05`:有界,就是给池/队列一个明确的 `c` 和最大队长,而不是放任 offered load 无限堆积。**「有界 + 明确拒绝策略」是过载保护的地基。**

## 3. 底层②:重试风暴 —— 把小抖动乘成全面崩溃

这是分布式系统最危险的正反馈,面试必考。

下游抖动了一下(短暂变慢/少量失败)。每个上游都配了「失败重试 3 次」。于是:

```
下游轻微故障 → 上游请求失败 → 每个失败重试 3 次
            → 打到下游的负载 ×3(甚至 ×N 层,层层放大)
            → 本来只是小抖动的下游,被 3 倍负载彻底打死
            → 更多失败 → 更多重试 → 负载更高 …          ← 正反馈,雪崩
```

**重试在下游健康时是良药,在下游已经过载时是毒药**——它恰好在最不该加压的时候加压。缓解手段(组合使用):

- **指数退避 + 抖动(jitter)**:重试间隔 1s、2s、4s… 且加随机抖动,避免所有客户端**同时**重试形成尖刺(thundering herd)。
- **重试预算(retry budget)**:限制「重试流量占总流量的比例」(如 ≤10%)。预算用完就不再重试——从根上掐断放大。Google SRE 的标准做法。
- **断路器(circuit breaker)**:下游连续失败 → 断路器跳闸 → 一段时间内**直接快速失败、根本不打下游**,给它喘息恢复的窗口;半开状态试探性放行,好了再合闸。
- **只重试幂等操作**:非幂等的重试会重复扣款/下单。重试的前提是幂等。

> 关键心智:**重试要有「全局预算」意识,不能每个客户端各自闷头重试。** 单看「重试 3 次」很合理,乘上成千上万的客户端,就是一场 DDoS——又是一个「单点合理、总账要命」的模式(呼应 `05` 的 ×worker 总账)。

## 4. 底层③:超时即背压 —— deadline 必须传播

超时不只是「别等太久」,它是**背压的一种形式**:超时砍掉的是「注定要失败的等待」,把资源还给还有救的请求。但有个常被忽略的点——**deadline 要在调用链上传播**。

场景:用户请求总预算 1 秒。它扇出调 A(慢,花了 0.9s)再调 B。等轮到 B 时,只剩 0.1s,但如果不传播 deadline,B 仍按自己的 5s 超时吭哧吭哧地算——**算完也没用,用户早超时走了**,纯属浪费下游资源。

正解:**每一跳传递「剩余 deadline」**,下游发现剩余时间不够就立刻放弃,别做无用功。这是 gRPC 的 deadline propagation、context 取消的核心价值:

```
入口设总 deadline=1s → 每跳传剩余时间 → 超了就全链路一起放弃
```

**对过载的意义**:过载时大量请求注定要超时,传播 deadline 能让系统**尽早放弃这些死亡请求**,把 CPU/连接还给有希望成功的请求,避免「全在处理僵尸请求」。这是 `01` coordinated omission 的另一面——别让系统为已经放弃的客户端继续埋头苦干。

**额外一招:过载时用 LIFO 而非 FIFO。** 队列满时,后进先出能救活更多请求——因为队头那些老请求大概率已经超时了(处理它们=白干),先服务刚到的新请求,成功率更高。反直觉但有效。

## 5. 底层④:metastable failure —— 为什么故障源消失了,系统还起不来

最阴险的一种过载,近年才被系统总结(《Metastable Failures in Distributed Systems》):**触发故障的原因已经消失,系统却卡在「坏稳态」里自己出不来。**

形状:

```
触发器(一次流量尖刺)→ 把系统推过临界点 → 进入坏稳态
坏稳态靠「自我维持的正反馈」续命,即使触发器早就没了:
  - 重试风暴:过载→失败→重试→维持过载(§3)
  - 缓存被打穿:过载→缓存超时失效→全打 DB→DB 过载→维持
系统就这么一直崩着,流量恢复正常也救不回来          ← metastable
```

**打破它,光「等流量回落」没用**,必须主动把负载降到**坏稳态维持不住的水平以下**:临时关掉重试、强制 shed 大部分流量、预热缓存、甚至重启清空积压。理解这点能避免一类经典误判——「上游流量都正常了,为什么还不恢复?」因为系统已经进入了不依赖触发器的自维持坏稳态。

> 这把前面的工具串成了一个防御体系:**有界队列 + shed**(§2)防止进入坏稳态,**重试预算 + 断路器**(§3)掐断维持坏稳态的正反馈,**deadline 传播**(§4)减少僵尸负载。过载保护不是单点措施,是一套互相配合的护栏。

---

## 6. 一句话收口

过了饱和点,系统会吞吐塌方 + 延迟雪崩;**无界队列是慢性自杀**,要换成「有界队列 + 明确的 shed/block 背压」;**重试在过载时会乘性放大故障**,靠指数退避+抖动+重试预算+断路器+幂等来驯服;**超时即背压**,deadline 要全链路传播以杀掉僵尸请求;最阴险的 **metastable failure** 会让系统在触发器消失后仍自我维持地崩着,必须主动降载才能打破。**这些是系统被打爆时的求生术,`08` 教你怎么提前看见它要爆了。**
