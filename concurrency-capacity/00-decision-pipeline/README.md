# 00 · 决策管线全景 —— 从需求到配置,到底要决定哪些事

> 你已经会用线程、协程、goroutine、连接池了。但面试官问「这个服务你打算怎么配」,或者上线前要你给个机器数,你心里没有一条系统的路径——这章给你那条路径。后面每一章是路径上的一站。

---

## 1. 真正卡住你的不是「怎么并发」,是「配多少」

回想你卡住的那些时刻,几乎都不是语法问题:

- 「这个服务该开几个线程 / 几个 worker?」——你只知道有个「2×核数+1」的口诀,但不知道为什么,也不知道它什么时候不成立。
- 「连接池配多大?」——抄了个 20,撞上数据库 `max_connections` 才发现不对。
- 「要几台机器?」——按 CPU 利用率估,结果一上量 P99 就炸,CPU 看着才 60%。
- 「要不要加熔断 / 隔离?」——知道有这些东西,但不知道**什么情况下非加不可**。

这些都不是「不会并发」,是**不会给并发定容**。定容是一门有方法的工程,不是拍脑袋。这门课就是把这套方法摊开。

## 2. 一张图:你要依次决定的 8 件事

把「给一个服务定容」拆开,本质是顺着一条管线,依次做 8 个决定。**每个决定就是后面的一章:**

```
业务需求 (RPS, P99 目标, 流量形状)
      │
      ▼
 ① 测算需求   ──→  需要多大并发?         L = λ·W            (→ 03)
      │
      ▼
 ② 选并发模型 ──→  thread / event-loop / 多进程?            (→ 02)
      │
      ▼
 ③ 单机定容   ──→  几个线程/worker?单机什么规格?           (→ 04)
      │
      ▼
 ④ 池定容     ──→  哪些资源要池?每个池多大?               (→ 05)
      │
      ▼
 ⑤ 隔离       ──→  哪些活要拆开,别互相饿死?               (→ 06)
      │
      ▼
 ⑥ 过载/背压  ──→  撑爆了怎么办?限流?降级?               (→ 07)
      │
      ▼
 ⑦ 监控       ──→  上线后盯哪几个信号?                     (→ 08)
      │
      ▼
 ⑧ 扩容       ──→  单机不够,几台?何时横向?               (→ 09)
```

注意这条管线的**顺序是有讲究的**:先量需求,再选模型,再把**单台机器**彻底配好(③④⑤),处理好它撑爆时的行为(⑥),装上观测(⑦),**最后才把单台乘成一个机群**(⑧)。先把一个节点吃透,再去复制——而不是一上来就谈「几台机器」。

## 3. 为什么这 8 件事能用一条主线串起来

这门课不想给你 8 套互不相干的口诀(那样你还是记不住、也不会迁移)。它们其实都从**同一个方程**推出来:

$$L = \lambda \times W$$

> 在途数 L = 到达率 λ × 每个请求停留时间 W。

- ① 测需求,就是量出 λ 和 W,算出 L。
- ③ 多少线程,就是 L 除以「单个线程能扛的并发」。
- ④ 池多大,就是把这个方程套到「连接」这种稀缺资源上。
- ⑦ 监控,就是把 λ、L、利用率、队列实时画出来——它们是这个方程的**活体读数**。

所以下一章(`01`)不急着配任何东西,先把 `L=λW` 这把钥匙彻底认下来,以及它的「阴暗面」——为什么利用率一高,延迟就不是线性变差、而是**指数炸**。认下这一章,后面 7 章都是它的应用。

## 6. 一句话收口

给系统定容,就是顺着「需求 → 模型 → 单机 → 池 → 隔离 → 过载 → 监控 → 扩容」这条管线,依次做 8 个有依据的决定;而这 8 个决定全都从一个方程 `L=λW` 长出来。下一章先认这把钥匙。

> **动手:跑这个实验**
> 还没正式开始,先热个身,直观感受一下主线方程:
> ```bash
> cd concurrency-capacity
> python lab/sim/little.py --lam 200 --w 0.05
> ```
> 你会看到 `L = 10 in flight`(200 req/s × 50ms),以及一张「利用率→延迟」的表:ρ 从 0.9 涨到 0.99,延迟从 100ms 跳到 1000ms。**这个非线性,就是整门课要你随时记住的东西。** `01` 讲它为什么这样。
