## 📖 讲解

### 订阅者 Subscriber

订阅者 = 一个节点,盯着某个话题,**一来消息就执行一个回调函数**。

代码骨架还是课 5 那七步套路,差别在第 3 步:发布者用 `create_publisher`,订阅者用 `create_subscription`,而且要给它一个回调函数:

```python
class MinimalSubscriber(Node):
    def __init__(self):
        super().__init__('minimal_subscriber')
        self.subscription = self.create_subscription(
            String,                 # 消息类型(必须和发布者一致)
            'topic',                 # 话题名(必须和发布者一致)
            self.listener_callback,  # 收到消息就调它
            10)                      # 队列长度

    def listener_callback(self, msg):
        self.get_logger().info('I heard: "%s"' % msg.data)
```

`main` 部分和发布者一模一样(`init` → `spin` → `shutdown`)。

### 关键观念:发布者和订阅者互不认识

发布者不知道有谁在收,订阅者不知道是谁在发。**它们俩唯一的共识只有两样:话题名 + 消息类型。** 对上了就能通话。

这叫**解耦**。它的好处:你可以随时加一个新订阅者,发布者完全不用改;某个节点挂了,别的不受影响。这正是 ROS 2 能把机器人拆成几十个小节点、还能协同工作的根基。

类比:消息队列的发布/订阅模式。

### `rclpy.spin()` 在干嘛

`spin` = "让这个节点一直转下去,有消息/有定时器到点就触发对应回调"。没有它,程序跑完 `main` 就退出了,啥也收不到。

类比:web server 的"一直监听、来请求就处理"的那个循环。

## 🛠️ 动手

继续课 5 那个官方教程 **"Writing a simple publisher and subscriber (Python)"**,这次做**订阅者**部分:
<https://docs.ros.org/en/jazzy/Tutorials/Beginner-Client-Libraries/Writing-A-Simple-Py-Publisher-And-Subscriber.html>

1. 在课 5 的 `py_pubsub` 包里,新建订阅者的 `.py` 文件(代码见上面骨架)。
2. **`setup.py` 再加一个 `entry_points`** —— 登记订阅者的可执行名(现在这个包有两个可执行了)。
3. 构建:
   ```bash
   cd ~/ros2_ws
   colcon build --packages-select py_pubsub
   source install/setup.bash
   ```
4. **两个终端各跑一个**(都记得先 `source install/setup.bash`):
   - 终端 1:`ros2 run py_pubsub <发布者可执行名>`
   - 终端 2:`ros2 run py_pubsub <订阅者可执行名>`

订阅者终端应持续打印 "I heard: ..."。

## ⚠️ 踩坑预警

- **订阅者收不到**:发布者和订阅者的**话题名**或**消息类型**不一致 —— 这是头号原因。回到代码逐字对。
- **第二个 `entry_points` 漏了**:`ros2 run` 会找不到订阅者。
- **改完没 build/source**:老规矩,每次都要。
- 用 `ros2 topic list` / `rqt_graph`(课 18 会细讲)能直观看到两个节点有没有连上同一个话题。
