# 05 · 配置讓它真的吐(Py/Go/Java 各自的坑)

> 治的症狀:**「沒有日誌」**(程式明明 log 了卻沒輸出)、**「等級看不到」**(門檻擋掉)、**「同一條重複」**(配置造成重複發射)。
> 前面四章是「**寫**對」,這章是「**配**對」—— 寫得再好,配置沒接上,一行都吐不出來。三語言各有一個經典陷阱。

---

## 一、共同心智模型:日誌怎麼從一行程式碼走到你眼前

不管哪個語言,一條日誌都要經過這條管線:

```
log.info(...)  →  [logger:檢查 level 門檻]  →  [handler/appender:格式化 + 寫到哪]  →  輸出(stdout/檔案/遠端)
```

「沒輸出」一定是這條管線斷在某處:**要嘛被 level 擋,要嘛根本沒接 handler,要嘛 handler 寫到你沒在看的地方。** 「重複」則是這條管線**接了兩條**。三語言的差異,就在「logger 怎麼組織」和「handler 怎麼掛」。

---

## 二、Python:`logging` 階層 + propagate(雙重發射的元兇)

### 階層與傳播

Python 的 logger 是一棵**樹**,用名字的點號分層:`getLogger("a.b")` 是 `getLogger("a")` 的子,最上面是 root。

```python
log = logging.getLogger(__name__)   # 慣例:用模組名,自動形成階層
```

關鍵機制 **propagate(預設 True)**:一條紀錄在某個 logger 產生後,會**往上冒泡**,依序交給**每一個祖先的 handler** 處理,直到 root。

### 🔬 坑 1:沒有 handler → 你的 INFO/DEBUG 靜默消失(「沒日誌」)

如果整棵樹**都沒有任何 handler**,Python 會啟用一個 `lastResort` 後備 handler —— 但它**只印 WARNING 以上到 stderr**。於是:

```python
log = logging.getLogger("myapp")
log.info("started")     # ❌ 沒配 handler → lastResort 不收 INFO → 人間蒸發
log.warning("careful")  # 這條才會(以陽春格式)出現
```

這就是無數人「為什麼我的 INFO 都不見」的答案:**不是程式錯,是根本沒配輸出。**

### 🔬 坑 2:handler 掛兩處 / 重複註冊 → 同一條印兩三次(「重複」)

因為 propagate 會走訪祖先的 handler,如果你**同時在子 logger 和 root 掛 handler**,一條 log 就被印兩次:

```python
logging.getLogger("myapp").addHandler(h1)   # 子
logging.getLogger().addHandler(h2)          # root
logging.getLogger("myapp").info("hi")       # ❌ h1 印一次,冒泡到 root 又被 h2 印一次
```

更常見的是**重複執行 setup**(在每個請求、模組重載、notebook 重跑時又 `addHandler`),handler 越疊越多,一條變 N 條。

### 正解:一次設好,只在 root 掛唯一 handler

```python
import logging.config

logging.config.dictConfig({
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {"json": {"()": "myapp.log.JsonFormatter"}},
    "handlers": {"stdout": {"class": "logging.StreamHandler", "formatter": "json"}},
    "root": {"handlers": ["stdout"], "level": "INFO"},   # ✅ 唯一 handler 在 root
    "loggers": {
        "myapp": {"level": "DEBUG"},   # ✅ 子 logger 只設 level,不掛 handler,靠 propagate
        "uvicorn.access": {"level": "WARNING"},  # 馴服框架 logger
    },
})
```

口訣:**業務 logger 只 `getLogger(__name__)` + 設 level,不要自己 `addHandler`;handler 統一掛 root。** 這樣杜絕雙重發射,也保證一定有輸出。

> `basicConfig()` 的陷阱:它**只在 root 還沒有 handler 時**生效一次,第二次呼叫是 no-op(除非 `force=True`)。所以混用 `basicConfig` + 自己 `addHandler` 很容易出意外。生產用 `dictConfig` 一次到位。

## 三、Go:從「沒等級」的老 `log` 到 `slog`

### 🔬 坑:標準庫老 `log` 套件**沒有等級概念**

```go
import "log"
log.Println("something")   // 就一行,沒有 level,無法「prod 只看 WARNING 以上」
```

這是很多 Go 服務「等級不夠 / 沒法過濾」的根源 —— 用了一個**根本沒有 level** 的工具。正解是 Go 1.21(2023)才進標準庫的 **`log/slog`**。

### 正解:全域設好一個 leveled handler

```go
import (
    "log/slog"
    "os"
)

func init() {
    h := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{Level: slog.LevelInfo})
    slog.SetDefault(slog.New(h))   // ✅ 設全域:之後任何 slog.Info 都走這個 handler
}
```

`slog.SetDefault` 還有一個好處:它會把**老 `log` 套件的輸出也接管**到 slog,讓第三方庫用老 `log` 打的東西也統一格式。

### Go 為什麼幾乎不會有 Python 那種「重複」

Go 的 slog **沒有階層、沒有 propagate** —— 一個 logger 就一個(或一串顯式包裝的)handler,不會自動往上冒泡。所以 Go 的「重複」通常是你自己造的:**多次 `SetDefault`、或把 handler 包了兩層都會寫**、或你又 `slog` 又用老 `log` 各寫一份。排查方向跟 Python 不同。

> 另一個 Go 專屬坑:用 `slog.Info(...)`(不帶 ctx)時,你那個「從 context 取 request_id」的自訂 handler **拿不到 ctx** → request_id 不見。要用 `slog.InfoContext(ctx, ...)`(呼應 `03`)。

## 四、Java:SLF4J 是門面,真正輸出要綁一個後端

### 🔬 坑 1:沒綁 backend → 全部靜默(「沒日誌」)

SLF4J(`org.slf4j`)**只是介面(facade)**,自己不會輸出任何東西。真正幹活的是後端實作(**Logback** / Log4j2)。classpath 上:

- **沒有任何 provider** → 啟動印 `SLF4J: No SLF4J providers were found`,所有 log 走 NOP(什麼都不做)→ 全部靜默。
- **有多個 provider**(同時混進 logback + log4j + slf4j-simple)→ 印 `Class path contains multiple SLF4J providers`,隨機挑一個 → 行為詭異。

這就是 Java「明明 log 了卻沒輸出 / 格式怎麼變了」的頭號原因:**binding 沒接好。** 正解:只留**一個**後端(Spring Boot 預設給你 `logback-classic`,別再手動拉別的進來)。

### 🔬 坑 2:additivity → 同一條印兩次(「重複」)

Logback 的 logger 也是**階層 + 冒泡**,機制和 Python propagate 一模一樣,叫 **additivity(預設 true)**:一條 log 會交給自己的 appender,**再交給祖先(root)的 appender**。如果你在某個 logger 和 root **都掛了 appender**:

```xml
<logger name="com.myapp" level="INFO">
    <appender-ref ref="STDOUT"/>     <!-- 自己印一次 -->
</logger>
<root level="INFO">
    <appender-ref ref="STDOUT"/>     <!-- 冒泡到 root 又印一次 ❌ -->
</root>
```

→ 每條 `com.myapp` 的 log 印兩次。修法:要嘛**只在 root 掛 appender**,要嘛在該 logger 設 `additivity="false"` 切斷冒泡。

### 正解(Spring Boot):`logback-spring.xml`,appender 只掛 root

```xml
<configuration>
  <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>  <!-- 輸出 JSON -->
  </appender>
  <logger name="com.myapp" level="INFO"/>   <!-- 只設 level,不掛 appender,靠冒泡 -->
  <root level="WARN">
    <appender-ref ref="STDOUT"/>            <!-- ✅ 唯一 appender 在 root -->
  </root>
</configuration>
```

## 五、把「重複」一次講清楚:兩個完全不同的根因

你說的「同一條重複」其實有**兩個來源,住在兩章**,面試常考你能不能分清:

| 重複的種類 | 根因 | 在哪章 | 怎麼認 |
|---|---|---|---|
| **配置重複發射** | propagate(Py)/ additivity(Java)/ 多次 SetDefault(Go):handler 掛多處 | 本章 `05` | 重複的內容**一模一樣**(同一行、同 timestamp),只是輸出兩三份 |
| **程式碼 cascade** | 每層 `catch → log → rethrow` | `04` | 重複的是**堆疊**,出現在不同 logger、訊息不同(`dao failed` / `service failed`) |

看到重複先分這兩類:內容完全一樣 → 查配置;堆疊在不同層 → 查 catch-rethrow。

## 六、🔬 內幕:階層 + 冒泡的精確機制(Python/Java 共通)

一條 log 產生時的判定順序(以 Python 為例,Java additivity 同理):

1. **產生與否**:看「你 log 在哪個 logger」的**有效等級**(沿樹往上找第一個有設 level 的祖先)。被擋就此打住,連 record 都不建。
2. **若產生**:交給**該 logger 自己的 handler**;
3. **若 propagate=True**:再依序交給**每個祖先的 handler**(一路到 root)。
4. 注意:冒泡時**不再重新檢查祖先 logger 的等級**,但**每個 handler 自己**還有一層 level 過濾。

所以「沒輸出」可能卡在:logger 有效等級擋掉(第 1 步)、或沒有任何 handler(第 2/3 步沒人收)、或 handler 自己的 level 又擋一層(第 4 步)。「重複」則是第 2 步和第 3 步**都有 handler**。Go 沒有第 3 步,所以結構簡單、也少這類坑。

## 七、落地清單

- [ ] **Python**:用 `dictConfig` 一次設定;handler **只掛 root**;業務 logger 只 `getLogger(__name__)` + 設 level,不 `addHandler`;馴服框架 logger(uvicorn/gunicorn)。
- [ ] **Go**:用 `log/slog`(不用老 `log`);啟動時 `slog.SetDefault` 設好 leveled JSON handler;一律 `InfoContext(ctx, ...)`。
- [ ] **Java**:classpath 只有**一個** SLF4J 後端;appender **只掛 root**(或 `additivity="false"`);Spring 用 `logback-spring.xml`。
- [ ] 遇到「重複」先分類:**內容全同**→查配置(本章);**堆疊分層**→查 cascade(`04`)。
- [ ] 確認 prod 真的有輸出:啟動後 `curl` 打一條,確認 stdout 有結構化 JSON。
