# 03 · 結構化 + 關聯 ID(一行帶你找到整串)

> 治的症狀:**「只有一行」**(撈到一條卻看不出前後文)和**「一長串撈不到」**(有資料但沒法 filter)。
> 解法是兩件事合起來:**結構化**(讓機器能 filter/aggregate)+ **關聯 ID**(讓你撈一個 id 就拉出整個請求的所有行)。

---

## 一、原理一:結構化日誌 —— 寫給機器讀,不是寫給人讀

傳統字串日誌:

```
2026-06-22 10:03:11 INFO order placed for user 1001, amount 250, status ok
```

問題:這是給**人眼**讀的。當你有一億行、要問「`user=1001` 且 `status=failed` 的訂單有幾筆」時,字串日誌只能用正則硬撈,既慢又脆(訊息格式一改,正則就壞)。

結構化日誌:每行是一組 **key=value**(通常輸出成 JSON):

```json
{"ts":"2026-06-22T10:03:11Z","level":"INFO","msg":"order placed","user_id":1001,"amount":250,"status":"ok","request_id":"a1b2c3"}
```

好處是**可被機器查詢與聚合**:

```
user_id=1001 AND status=failed        ← 精確 filter
count(*) by status                     ← 聚合
avg(latency_ms) where path=/checkout   ← 算指標
```

這就是為什麼「一長串撈不到」會被治好:**有了欄位,你是在「查資料庫」而不是「肉眼掃文字」。**

### 該帶哪些標準欄位

一條生產級日誌的「最小欄位集」:

| 欄位 | 為什麼 |
|---|---|
| `ts` / timestamp | 時間(用 UTC + ISO8601) |
| `level` | 等級(`01`) |
| `logger` / source | 哪個模組發的 |
| `msg` | 人讀的事件描述(**固定字串**,變數放別的欄位) |
| `request_id` / `trace_id` | 關聯 ID(下一節) |
| 業務鍵 | `user_id`、`order_id` 等,你最常拿來 filter 的 |
| 錯誤資訊 | `error`、堆疊(`04`) |
| 耗時 | `latency_ms`(邊界,`02`) |

> 關鍵:**`msg` 要是固定字串**(`"order placed"`),變數放成欄位(`order_id=...`)。
> 別寫成 `msg="order 1001 placed"` —— 那等於把變數又拼回字串,失去結構化的意義(你沒法 `count by msg`,因為每條 msg 都不一樣)。

---

## 二、原理二:關聯 ID —— 撈一個 id,拉出整串故事

「只有一行」的另一個根因:**每行日誌是一座孤島**。你看到 `payment failed`,但不知道是哪個請求、那個請求之前做了什麼、之後又如何。

解法:給每個請求一個 **關聯 ID**,讓它**出現在這個請求產生的每一行日誌上**。然後你只要:

```
request_id="a1b2c3"
```

就把這個請求從進來到出去的**所有行**一次撈齊 —— 一行帶你找到整串。

### 關聯 ID 從哪來

1. **入口生成**:請求進到第一個服務時,生成一個 UUID(或接受上游傳來的 `X-Request-ID`)。
2. **跨服務傳遞**:呼叫下游時,把 ID 放進 header 傳下去 —— 這樣**整條呼叫鏈**共用同一個 ID。
3. 業界標準是 **W3C Trace Context**:`traceparent` header 帶 `trace_id`,這正是接到 `observability/` 分散式追蹤的鉤子。

> `request_id`(服務內,自己生成)vs `trace_id`(W3C,跨服務,給 trace 系統用)。小系統 request_id 夠用;要做分散式追蹤就用 trace_id,並讓 log 也帶上它 —— **這樣才能「從一條 trace 跳到對應的 logs」**(`observability/` 三支柱串聯的核心)。

### 怎麼讓 ID「自動」出現在每一行(不用每行手動傳)

這是三語言差最大的地方:**上下文怎麼隨請求流動**。

---

## 三、三語言並排:結構化 + 關聯 ID

**Python —— `contextvars` 帶 request_id + JSON 輸出**

```python
import logging, contextvars, uuid

request_id_var = contextvars.ContextVar("request_id", default="-")

# 一個 filter,把 contextvar 注入每筆 record
class RequestIdFilter(logging.Filter):
    def filter(self, record):
        record.request_id = request_id_var.get()
        return True

# 在請求入口(中介層)設定
def on_request():
    request_id_var.set(uuid.uuid4().hex[:12])

log.info("order placed", extra={"order_id": 1001, "status": "ok"})
# 配上 JSON formatter 後輸出:{"msg":"order placed","order_id":1001,"request_id":"a1b2c3",...}
```

> 為何用 `contextvars` 不用全域變數?因為 `contextvars` **async/協程安全**:每個請求的協程有自己的副本,並發不會互相污染。(對照下方 🔬)

**Go —— `slog` 原生結構化 + `context` 帶 request_id**

```go
import "log/slog"

// slog 天生就是 key=value 結構化,不用額外套件
logger.InfoContext(ctx, "order placed", "order_id", 1001, "status", "ok")

// 讓 request_id 自動出現:自訂 Handler,從 ctx 取出注入每筆
type ctxHandler struct{ slog.Handler }
func (h ctxHandler) Handle(ctx context.Context, r slog.Record) error {
    if rid, ok := ctx.Value(ridKey{}).(string); ok {
        r.AddAttrs(slog.String("request_id", rid))
    }
    return h.Handler.Handle(ctx, r)
}
// 入口:ctx = context.WithValue(ctx, ridKey{}, newID())
```

> Go 沒有隱式上下文 —— `context.Context` **必須顯式傳遞**(所以用 `InfoContext(ctx, ...)`)。這是 Go 的設計哲學:沒有魔法,代價是每個函式簽名都要帶 `ctx`。

**Java —— `MDC` 帶 request_id + Logback JSON encoder**

```java
import org.slf4j.MDC;

// 入口(filter)塞進 MDC,MDC 底層是 ThreadLocal
MDC.put("requestId", UUID.randomUUID().toString().substring(0, 12));
try {
    log.info("order placed, orderId={} status={}", 1001, "ok");
    // logback 用 %X{requestId} 取出,或 logstash-encoder 自動把整個 MDC 變成 JSON 欄位
} finally {
    MDC.clear();   // 一定要清!見 🔬
}
```

```xml
<!-- logback:把 MDC 的 requestId 印進每行 -->
<pattern>%d %level %X{requestId} %logger - %msg%n</pattern>
```

三種原語(`contextvars` / `context.Context` / `MDC`),解的是**同一件事**:讓 request_id 不用手動傳就出現在每一行。

## 四、🔬 內幕:contextvars vs ThreadLocal vs 顯式 context

這三個原語的差異,正好對應三語言的並發模型,面試常考:

| | Python `contextvars` | Java `MDC`(ThreadLocal) | Go `context.Context` |
|---|---|---|---|
| 載體 | 協程/任務的 context 副本 | 執行緒區域變數 | 顯式參數傳遞 |
| async 安全? | ✅ 天生支援 | ⚠️ **危險**:async/執行緒池會出錯 | ✅(因為顯式傳) |
| 要手動清理? | 否(隨 context 結束) | ✅ **必須 `MDC.clear()`** | 否 |

**Java MDC 的大坑**:MDC 是 ThreadLocal,在**執行緒池**裡執行緒會被重用 —— 如果你忘了 `clear()`,**下一個請求會看到上一個請求殘留的 requestId**,日誌張冠李戴。而且如果你把工作丟到別的執行緒(`@Async`、reactor),MDC **不會自動跟過去**(因為換執行緒了),requestId 就丟了。這是 Java 日誌最常見的「為什麼 requestId 是別人的 / 不見了」。

**Python contextvars** 沒這問題:它跟著協程的 context 走,`await` 跳來跳去也不會串味,任務結束自然回收 —— 這正是 Python async 時代取代舊 thread-local 寫法的原因。

**Go** 最「笨」也最不會錯:沒有隱式儲存,context 是函式參數,你傳到哪它就在哪,換 goroutine 也是你顯式 `go func(){ ...ctx... }`。代價是簽名囉嗦,好處是沒有「魔法消失」。

## 五、反例

**反例 1:把變數拼進 msg。**
```python
log.info(f"order {oid} placed for user {uid}")   # ❌ 不能 filter、不能 count by
```
✅ `log.info("order placed", extra={"order_id": oid, "user_id": uid})` —— msg 固定,變數成欄位。

**反例 2:沒有關聯 ID。** 每行是孤島,看到一條 error 撈不出同請求的前後文。→ 入口生成 request_id,隨上下文帶。

**反例 3:把整個物件倒進日誌。**
```java
log.info("request: {}", request);   // ❌ 可能含一大包、還可能含密碼(→ 06 衛生)
```
✅ 只記你會拿來 filter/debug 的幾個關鍵欄位。

**反例 4:Java 忘了 `MDC.clear()`。** → 執行緒池下 requestId 串味。✅ 用 `try/finally` 清,或用框架的 MDC filter(`08` 示範)。

## 六、落地清單

- [ ] 日誌輸出是**結構化 JSON**(prod),不是純字串。
- [ ] `msg` 是**固定字串**,變數一律放成欄位。
- [ ] 有**最小欄位集**:ts / level / logger / msg / request_id / 業務鍵。
- [ ] 入口生成或接收 **request_id / trace_id**,並讓它**自動出現在每一行**(contextvars / context / MDC)。
- [ ] 跨服務呼叫**把關聯 ID 傳下去**(header);要做分散式追蹤就用 W3C `traceparent`(接 `observability/`)。
- [ ] Java 確認 **`MDC.clear()`**(或用 MDC filter),避免執行緒池串味。
