# 08 · 生產級完整範例(三語言)

> 前面七章是一條條紀律;這章把它們**全部組裝**進同一個服務,Python / Go / Java 各寫一份。
> 三份程式碼都在 `examples/`,而且**本機真的跑得起來**(下面的輸出都是實跑擷取,不是手寫的)。

---

## 一、共同規格:同一個服務,三語言各做一遍

端點:`GET /users/{id}`,刻意設計四種情境,正好對應前面的紀律:

| 情境 | 觸發 | 預期等級 | 回應 | 考的紀律 |
|---|---|---|---|---|
| 成功 | `id=42` | INFO | 200 | 邊界 + 業務里程碑(`02`) |
| 參數非法 | `id=0` | **INFO**(不是 ERROR) | 400 | 等級判準(`01`) |
| 查無此人 | `id=13` | **INFO**(不是 ERROR) | 404 | 等級判準(`01`) |
| 下游故障 | `id=99` | **ERROR**(完整堆疊) | 500 | 異常 log-or-throw(`04`) |

每份實作都做到:
- **每行帶 `request_id`**(從 `X-Request-ID` header 接,沒有就生成)—— `03`
- **請求進/出**各一條,帶 method/path/status/latency —— `02`
- **資料層只包裝往上拋,頂層記一次完整堆疊 + cause** —— `04`
- **可預期業務狀況(400/404)記 INFO,真故障(500)記 ERROR** —— `01`
- **結構化 JSON,寫 stdout** —— `03`/`06`

## 二、Python:FastAPI + dictConfig + contextvars

**技術棧**:FastAPI + 標準庫 `logging`(自寫 `JsonFormatter`)+ `contextvars`。
**關鍵檔案**(`examples/python/`):`log_config.py`(設定 + JSON + filter)、`app.py`(中介層 + 端點 + 全域異常處理器)。

核心三段(完整見檔案):

```python
# 03:contextvars 帶 request_id,filter 注入每筆 record
request_id_var = contextvars.ContextVar("request_id", default="-")
class RequestIdFilter(logging.Filter):
    def filter(self, record):
        record.request_id = request_id_var.get(); return True

# 04:資料層只包裝往上拋(from e 保留 cause),不記
raise RepositoryError(f"load user failed, uid={uid}") from e

# 04:頂層唯一記完整堆疊的地方
@app.exception_handler(Exception)
async def on_unhandled(request, exc):
    log.exception("unhandled error", extra={"path": request.url.path})
    return JSONResponse(status_code=500, content={"error": "internal"})
```

**跑法**:`pip install -r requirements.txt` 然後 `python smoke_test.py`(免起 server)。

**實跑輸出**(節選,`id=42` 成功 + `id=99` 故障):

```json
{"ts":"...","level":"INFO","logger":"myapp","msg":"request started","request_id":"req-42","method":"GET","path":"/users/42"}
{"ts":"...","level":"INFO","logger":"myapp","msg":"user fetched","request_id":"req-42","uid":42}
{"ts":"...","level":"INFO","logger":"myapp","msg":"request finished","request_id":"req-42","status":200,"latency_ms":1.8}
{"ts":"...","level":"INFO","logger":"myapp","msg":"request started","request_id":"req-99","method":"GET","path":"/users/99"}
{"ts":"...","level":"INFO","logger":"myapp","msg":"request finished","request_id":"req-99","status":500,"latency_ms":0.3}
{"ts":"...","level":"ERROR","logger":"myapp","msg":"unhandled error","request_id":"req-99","path":"/users/99",
 "error":"Traceback (most recent call last): ... ConnectionError: connection reset by peer\n\nThe above exception was the direct cause of the following exception: ... app.RepositoryError: load user failed, uid=99"}
```

> 注意:`id=99` 那條 ERROR 的 `error` 欄位裡,Python 自動印出**雙段堆疊**——「`ConnectionError`(原因)→ The above exception was the direct cause →`RepositoryError`(包裝)」。這就是 `from e` 的回報。堆疊**只出現這一次**(log-or-throw)。

**升級**:想要更省事的結構化 + 自動 contextvars 綁定,把 `logging` 換成 `structlog`(API 更順、processor 管線好做遮罩)。原理不變。

## 三、Go:gin + slog + context handler

**技術棧**:gin(國際主流框架)+ 標準庫 `log/slog`。
**關鍵檔案**(`examples/go/`):`main.go`(全部)、`main_test.go`(httptest 驗證)。

核心三段:

```go
// 03:自訂 handler 從 context 取 request_id 注入每筆 record
func (h contextHandler) Handle(ctx context.Context, r slog.Record) error {
    if rid, ok := ctx.Value(ridKey{}).(string); ok {
        r.AddAttrs(slog.String("request_id", rid))
    }
    return h.Handler.Handle(ctx, r)
}

// 04:資料層用 %w 包裝往上回傳,不記
return nil, fmt.Errorf("load user failed, uid=%d: %w", uid, downstream)

// 04:頂層用 errors.Is 分流;真故障記一次,err 帶完整 %w 鏈
if errors.Is(err, errUserNotFound) { logger.InfoContext(ctx, "user not found", "uid", uid); ... }
logger.ErrorContext(ctx, "request failed", "uid", uid, "err", err)
```

**跑法**:`go mod tidy` 然後 `go test -v`(或 `go run .` 起 server)。

**實跑輸出**(節選):

```json
{"time":"...","level":"INFO","msg":"request started","method":"GET","path":"/users/:id","request_id":"req-42"}
{"time":"...","level":"INFO","msg":"user fetched","uid":42,"request_id":"req-42"}
{"time":"...","level":"INFO","msg":"request finished","status":200,"latency_ms":0,"request_id":"req-42"}
{"time":"...","level":"ERROR","msg":"request failed","uid":99,"err":"load user failed, uid=99: connection reset by peer","request_id":"req-99"}
{"time":"...","level":"INFO","msg":"request finished","status":500,"latency_ms":0,"request_id":"req-99"}
```

> 兩個 Go 專屬細節:① `path` 是 `/users/:id`(`c.FullPath()` 回傳**路由樣板**而非實際路徑)—— 低基數,適合拿來聚合/做指標。② `err` 欄位就是整條 `%w` 鏈展開的字串(`load user failed, uid=99: connection reset by peer`),不像 Java/Python 有多行堆疊;要堆疊得上 `zap`/`zerolog` 或自己包。

**升級**:超高吞吐要 zero-allocation,把 handler 換成 `zap` 或 `zerolog`(可讓 slog 當前台 API、底層接它們)。

## 四、Java:Spring Boot + Logback(JSON)+ MDC filter

**技術棧**:Spring Boot + SLF4J/Logback + `logstash-logback-encoder`。
**關鍵檔案**(`examples/java/`):`RequestLoggingFilter`(MDC)、`GlobalExceptionHandler`(`@RestControllerAdvice`)、`UserService`(包裝)、`logback-spring.xml`(JSON + appender 只掛 root)。

核心三段:

```java
// 03:filter 設 MDC,finally 一定 clear(避免執行緒池串味)
MDC.put("requestId", rid);
try { chain.doFilter(req, res); }
finally { log.info("request finished, status={} ...", res.getStatus(), ...); MDC.clear(); }

// 04:service 包裝往上拋(e 當 cause),不記
throw new RepositoryException("load user failed, id=" + id, e);

// 04:@RestControllerAdvice 頂層記一次,e 當最後一參(印完整堆疊+cause)
log.error("unhandled error, path={}", req.getRequestURI(), e);
```

**跑法**:`mvn test`(或 `mvn spring-boot:run` 起 server)。

**實跑輸出**(節選,堆疊已截短):

```json
{"ts":"...","level":"INFO","logger":"com.example.RequestLoggingFilter","msg":"request started, method=GET path=/users/42","requestId":"req-42"}
{"ts":"...","level":"INFO","logger":"com.example.UserController","msg":"user fetched, id=42","requestId":"req-42"}
{"ts":"...","level":"INFO","logger":"com.example.RequestLoggingFilter","msg":"request finished, status=200 latency_ms=17","requestId":"req-42"}
{"ts":"...","level":"ERROR","logger":"com.example.GlobalExceptionHandler","msg":"unhandled error, path=/users/99","requestId":"req-99",
 "stack_trace":"com.example.RepositoryException: load user failed, id=99\n\tat com.example.UserService.fetch(...) ... \nCaused by: java.lang.RuntimeException: connection reset by peer\n\t... 110 common frames omitted\n"}
```

> Java 的 `stack_trace` 欄位裡有 `Caused by: ... connection reset by peer` 和 `... 110 common frames omitted` —— 這就是 Java 的 cause chain 顯示(對應 `new RepositoryException(msg, e)` 的 `e`)。同樣**只出現一次**。

**升級**:Logback 已是生產級;高吞吐加 `AsyncAppender`(注意 `06` 的丟失取捨);要動態調級用 `/actuator/loggers`。

## 五、壓軸對照:同一條請求,三語言日誌長一樣

這是整個 track 的 payoff。看 **`id=99` 那條 ERROR**,三種語言**用完全不同的原語**,卻收斂成**同形的結構化日誌**:

| | Python | Go | Java |
|---|---|---|---|
| 帶 request_id 的原語 | `contextvars` + filter | `context.Context` + 自訂 handler | `MDC`(ThreadLocal)+ encoder |
| 包裝原因的原語 | `raise ... from e` | `fmt.Errorf("...: %w", err)` | `new X(msg, e)` |
| 頂層記異常的原語 | `log.exception(...)` | `slog.Error(..., "err", err)` | `log.error("...", e)` |
| 分流業務 vs 真錯 | `except UserNotFound` | `errors.Is(err, sentinel)` | `@ExceptionHandler(類型)` |
| **輸出的 ERROR 行** | `level=ERROR` · `request_id=req-99` · 帶 cause 鏈 | `level=ERROR` · `request_id=req-99` · 帶 cause 鏈 | `level=ERROR` · `requestId=req-99` · 帶 cause 鏈 |

三份輸出共同做到、你可以逐條對照驗收的事:

1. **`request_id`(req-99)出現在該請求的每一行** —— 撈一個 id 拉出整串(`03`)。
2. **故障只記一次 ERROR + 完整 cause 鏈** —— 沒有 cascade 重複(`04`)。
3. **400 / 404 是 INFO 不是 ERROR** —— 不會半夜誤叫醒人(`01`)。
4. **每行是結構化 JSON,寫 stdout** —— 可 filter/aggregate、交給平台收集(`03`/`06`)。
5. **請求進/出各一條,帶 latency** —— 邊界錨點齊全(`02`)。

> 換句話說:**你學的是一套紀律,不是三套 API。** 哪天換語言,把這張對照表的原語換掉,紀律照搬。

## 六、怎麼用這三份

- **當模板**:新服務直接抄對應語言那份的 `log_config` / `initLogger` / `logback-spring.xml` + 中介層 + 全域異常處理器,五分鐘就有生產級日誌骨架。
- **接 observability/**:把 `request_id` 換成 W3C `trace_id`(從 `traceparent` header 接),這三份的 log 就能和 metrics/traces 在 `observability/` 那條管線裡用同一個 id 串起來 —— 從一條告警點到出問題的那一行。
- **對照舊 `log/`**:`log/` 是更早、Python/FastAPI 專屬的落地深切,可當 Python 那份的延伸閱讀。
