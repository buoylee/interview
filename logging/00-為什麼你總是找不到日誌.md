# 00 · 為什麼你總是找不到日誌(總綱地圖)

> 這章不教任何語言原語,只給你一個**心智模型**和一張**診斷流程圖**。
> 後面每一章都是在補這張圖的某一格。

---

## 一、先承認:日誌會從兩個相反方向壞掉

大多數人以為「打日誌」的問題是「記得不夠多」,於是拼命加 `log`,結果線上變成一片雜訊海,**還是找不到**要的那行。

真相是:日誌有**兩個失敗方向**,而且它們會同時存在於同一個系統:

```
        太少 / 太弱  ←─────────  剛剛好  ─────────→  太多 / 太吵
   沒有日誌                                              一長串撈不到
   只有一行                  在對的高度                  同一條重複 N 次
   被等級擋掉                給對的訊號                  把祕密都印出來
```

你列的五個症狀,正好落在兩端:

| 你的原話 | 落在哪端 | 一句話根因 |
|---|---|---|
| 「沒有日誌」 | 太少 | 該記的沒記,或配置根本沒讓它輸出 |
| 「只有一行」 | 太少 | 記了 `e.getMessage()` 而非完整堆疊與上下文 |
| 「等級不夠看不到」 | 太少 | 等級判準錯,prod 門檻把它擋了 |
| 「一長串不好找」 | 太多 | 過度記錄、倒大物件、在迴圈裡刷 |
| 「同一條重複」 | 太多 | 傳播重複發射,或異常逐層 log 又 rethrow |

**關鍵體悟:加 log 只能救「太少」那端,反而會惡化「太多」那端。** 真正的功夫是把訊號放到「對的高度」—— 這就是整個 track 的主軸:

> **right signal at the right altitude**:該被看到的事,用對的等級、在對的位置、帶足夠的上下文記一次;不該吵的,降級或不記。

## 二、診斷流程圖:下次找不到日誌,照這個順序查

別再亂加 log 了。按這條決策樹走,每一步都指向一個具體的章節去修:

```
找不到我要的日誌
│
├─ 這支服務「任何」日誌都沒有?
│     → 配置壞了:沒 handler / level 門檻太高 / binding 沒接上     →【05 配置】
│
├─ 別的日誌有,就「這條」沒有?
│     ├─ 那個程式點根本沒寫 log                                    →【02 該在哪記】
│     └─ 寫了但被 level 擋掉(prod 設 WARN,它是 DEBUG)           →【01 等級】
│
├─ 有這條,但「只有一行」、看不出前因後果?
│     ├─ 異常只印了 message,沒堆疊                                →【04 異常】
│     └─ 沒帶上下文(哪個 user?哪個 request?)                    →【03 結構化+關聯ID】
│
├─ 日誌「太多」,要的那行淹在裡面?
│     ├─ 沒結構化、沒 request_id,沒法 filter                      →【03 結構化+關聯ID】
│     └─ 同一個動作刷一堆(迴圈裡記 / 倒大物件)                   →【02 該在哪記】+【06 效能衛生】
│
└─ 「同一條」重複印兩三次?
      ├─ 一條 log 出現在多個輸出 / 重複 N 次                       →【05 配置:propagate】
      └─ 同一個錯在 stack 每一層都被記                             →【04 異常:log-or-throw】
```

> 把這張圖記起來,你 debug 日誌問題的速度會直接拉開跟同事的差距 —— 大多數人遇到「沒日誌」只會無腦加 log,你能直接定位是「配置沒吐」還是「該記的沒記」還是「被等級擋」。

## 三、六條紀律(本 track 的骨架)

把上面那張圖的每個出口收口,就是六條紀律。後面六章一章一條:

| # | 紀律 | 一句話 | 治哪端 |
|---|---|---|---|
| ① | **等級判準** | 「3am 會不會為這條叫醒我?」決定它是不是 ERROR | 太少(被擋)+ 太多(亂分級) |
| ② | **該在哪記 / 記什麼** | 在**邊界**(請求進出、外部呼叫)和**決策點**(選了哪條分支、retry、fallback)記;別在迴圈裡記 | 太少(沒記)+ 太多(刷屏) |
| ③ | **結構化 + 關聯 ID** | 用 key=value/JSON,每行帶 `request_id`/`trace_id`,一行就能撈出整串 | 太少(只有一行)+ 太多(撈不到) |
| ④ | **異常 log-or-throw** | 要嘛記、要嘛往上拋,**不要兩者都做**;記就記完整堆疊 + cause chain | 太少(一行/吞掉)+ 太多(cascade 重複) |
| ⑤ | **配置讓它真的吐** | 各語言「為什麼明明 log 了卻沒輸出 / 卻印兩次」的坑 | 太少(沒輸出)+ 太多(重複) |
| ⑥ | **效能與衛生** | 惰性求值(別 debug 也算昂貴字串)、取樣、截斷、不印祕密/PII | 太多 |

## 四、同一批紀律,三語言只是換原語

這是這份文件最想讓你帶走的一句話 —— **你不是在學三套日誌系統,是在學一套紀律的三種方言。** 先給你一張總覽,細節在各章:

| 紀律 | Python | Go | Java |
|---|---|---|---|
| 分級 | `logging`:DEBUG/INFO/WARNING/ERROR/CRITICAL | `slog`:Debug/Info/Warn/Error(整數刻度) | SLF4J:TRACE/DEBUG/INFO/WARN/ERROR |
| 結構化 | `structlog` 或 `logging` 的 `extra` + JSON formatter | `slog` **原生**就是結構化(key/value) | Logback + `logstash-logback-encoder` 出 JSON |
| 上下文隨請求走 | `contextvars`(async 安全) | `context.Context`(顯式傳遞) | `MDC`(底層是 ThreadLocal) |
| 異常帶完整堆疊 | `logging.exception(...)` / `exc_info=True` | wrap:`fmt.Errorf("...: %w", err)`,**頂層記一次** | `log.error("msg", e)` —— 例外當**最後一個參數**,別塞進字串 |
| 配好才會輸出 | `dictConfig` + 一個 root,走 logger 階層 | `slog.SetDefault(...)` 設全域 handler | `logback-spring.xml` + **唯一一個** SLF4J binding |
| 惰性求值 | `log.debug("user=%s", u)`(**別**用 f-string) | `slog` 天生惰性;昂貴值用 `LogValuer` | `log.debug("user={}", u)` 或 `if (log.isDebugEnabled())` |

三語言各有一個「**最容易害你沒日誌**」的特產坑,先點名,各章詳解:

- **Python**:logger 沒配 handler 時,預設只透過 `lastResort` 印 **WARNING 以上**到 stderr —— 你的 `INFO`/`DEBUG` 就這樣**靜默消失**。(→ `05`)
- **Go**:標準庫老的 `log` 套件**完全沒有等級概念**,很多人用它打「日誌」結果無法分級過濾。正解是 Go 1.21 才進標準庫的 `log/slog`。(→ `05`)
- **Java**:SLF4J 只是**門面(facade)**,真正輸出要綁一個後端(Logback/Log4j2);classpath 上**沒綁**或**綁了多個**都會讓日誌消失或報衝突。(→ `05`)

---

## 五、這章的最小takeaway

就算後面都還沒讀,先帶走這三句:

1. **日誌兩個方向都會壞** —— 別只會加 log,要分辨是「太少」還是「太多」。
2. **遇到問題照第二節那張流程圖走** —— 先判「沒輸出 / 沒記 / 被擋 / 太吵 / 重複」,再去對應章。
3. **三語言同一套紀律** —— 學原理,不是背 API。
