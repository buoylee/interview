# 04 · 異常怎麼記(log-or-throw,完整堆疊)

> 治的症狀:**「只有一行」**(只印了 message 沒堆疊)、**「沒有日誌」**(異常被吞)、**「同一條重複」**(每層都記一次)。
> 這三個你列的症狀**一半以上**都出在「異常記錯了」。這章是整個 track 的重點。

---

## 一、核心原則:log-or-throw(要嘛記,要嘛拋,別兩者都做)

一個異常往上冒泡時,每一層都面臨一個選擇:**我能處理它嗎?**

```
能處理(我知道怎麼補救/降級)        →  處理掉,在這裡記一次,不再往上拋
不能處理(我只是路過)              →  往上拋(可加上下文),不要記
最終沒人能處理                     →  在「最頂層」(請求邊界)記一次完整的
```

一句話:**log-or-throw —— 要嘛你處理它(就記),要嘛你拋給能處理的人(就別記)。同一個異常不要又記又拋。**

違反它的兩個方向,正好是你的兩個症狀:

| 違反方式 | 後果(你的症狀) |
|---|---|
| **又記又拋**(每層 `catch → log → rethrow`) | 一個錯在 stack 每層各記一次 → **「同一條重複」**(cascade) |
| **吞掉**(`catch {}` 不記也不拋) | 錯誤靜默消失 → **「沒有日誌」**(最難 debug 的一種) |

---

## 二、記就要記「完整」:堆疊 + cause chain

「只有一行」最常見的寫法:

```java
catch (SQLException e) {
    log.error("save failed: " + e.getMessage());   // ❌ 只有一句話,沒有堆疊
}
```

你看到 `save failed: connection reset`,但**不知道是哪一行、經過哪些呼叫、根本原因是什麼**。`getMessage()` / `str(e)` 丟掉了異常最值錢的東西:**堆疊軌跡(stack trace)** 和 **cause chain(誰引起誰)**。

正解:**把異常物件本身交給 logger**,讓它印出完整堆疊:

| 語言 | 正確寫法 | 錯誤寫法 |
|---|---|---|
| Python | `log.exception("save failed")` 或 `log.error("save failed", exc_info=True)` | `log.error(f"save failed: {e}")` |
| Go | `slog.Error("save failed", "err", err)`(err 用 `%w` 一路包著上下文) | `slog.Error(err.Error())` |
| Java | `log.error("save failed", e)`(**e 當最後一個參數**) | `log.error("save failed: " + e.getMessage())` |

> **cause chain** 就是 `python/08-exceptions.md` 教的 `raise X from e`、Go 的 `fmt.Errorf("...: %w", err)`、Java 的 `new BizException(msg, e)` —— 包裝成業務異常時保留原始原因。記日誌時,logger 會把整條鏈印出來(`Caused by:` / `__cause__` / `%w` 展開),你一眼看到「業務層失敗 ← 因為 DAO 失敗 ← 因為連線重置」。

## 三、三語言並排:同一個「外部呼叫失敗」怎麼處理

情境:`service` 呼叫 `dao`,`dao` 出錯。`dao` **不能處理**(只是 IO 失敗),`service` 也**不能處理**(它不知道怎麼補),最終由**請求邊界**統一記。

**Python**

```python
# dao 層:不能處理 → 包裝加上下文,往上拋,不記
def get_user(uid):
    try:
        return db.query_one("SELECT ... WHERE id=%s", uid)
    except DBError as e:
        raise RepositoryError(f"load user failed, uid={uid}") from e   # from e 保留原因

# service 層:也不能處理 → 直接往上冒泡,連 try 都不用寫,不記

# 頂層(FastAPI exception handler):能決定回什麼 → 在這裡記一次,完整堆疊
@app.exception_handler(Exception)
async def on_error(request, exc):
    log.exception("unhandled error, path=%s", request.url.path)  # exception() 自帶 exc_info
    return JSONResponse(status_code=500, content={"error": "internal"})
```

**Go**(沒有 try/catch,錯誤是回傳值)

```go
// dao 層:包裝加上下文,往上回傳,不記
func GetUser(ctx context.Context, uid int) (*User, error) {
    u, err := db.QueryUser(ctx, uid)
    if err != nil {
        return nil, fmt.Errorf("load user failed, uid=%d: %w", uid, err)  // %w 包裝
    }
    return u, nil
}

// service 層:也不能處理 → 繼續往上回傳(可再 %w 包)
//   if err != nil { return fmt.Errorf("get profile: %w", err) }

// 頂層(http handler):能決定回什麼 → 記一次,完整鏈
func handler(w http.ResponseWriter, r *http.Request) {
    u, err := svc.GetProfile(r.Context(), uid)
    if err != nil {
        slog.ErrorContext(r.Context(), "request failed", "path", r.URL.Path, "err", err)
        http.Error(w, "internal", 500)
        return
    }
    // ...
}
```

**Java**

```java
// dao 層:包裝加上下文,往上拋,不記
public User getUser(long uid) {
    try {
        return jdbc.queryForObject(SQL, userMapper, uid);
    } catch (DataAccessException e) {
        throw new RepositoryException("load user failed, uid=" + uid, e);  // e 當 cause
    }
}

// service 層:不能處理 → 不 catch,讓它往上冒泡

// 頂層(@RestControllerAdvice):統一記一次,完整堆疊
@RestControllerAdvice
class GlobalHandler {
    private static final Logger log = LoggerFactory.getLogger(GlobalHandler.class);

    @ExceptionHandler(Exception.class)
    ResponseEntity<?> onError(Exception e, HttpServletRequest req) {
        log.error("unhandled error, path={}", req.getRequestURI(), e);  // e 最後一參
        return ResponseEntity.status(500).body(Map.of("error", "internal"));
    }
}
```

三語言的關鍵共識:**中間層只「包裝 + 往上傳」,不記;頂層記一次完整的。** 這同時殺掉「重複(cascade)」和「只有一行」。

## 四、🔬 內幕:為什麼 cascade 會讓你「以為有 N 個錯」

看這個反模式(每層都 catch-log-rethrow):

```java
// dao
catch (SQLException e) { log.error("dao failed", e); throw e; }      // 印第 1 次完整堆疊
// service
catch (Exception e) { log.error("service failed", e); throw e; }     // 印第 2 次(幾乎一樣的堆疊)
// controller
catch (Exception e) { log.error("controller failed", e); throw e; }  // 印第 3 次
```

一個請求、一個根本原因,卻在日誌裡產生**三段又臭又長、80% 重疊的堆疊**,分散在不同時間點/不同 logger。線上 debug 時你會:

1. 以為發生了三個錯(其實是一個);
2. 對著三段堆疊比對「到底哪個是源頭」;
3. 日誌量被異常路徑灌爆。

這就是你說的「**同一條重複**」的根因之一(另一個是 `05` 的 propagate/additivity 配置問題 —— 那是「配置重複發射」,這是「程式碼重複記錄」,面試要能分清這兩種重複)。

**破法**:中間層只 `throw`(可 `%w`/`from e`/`new X(msg, e)` 加上下文),**只有頂層記**。一個請求 = 一條完整堆疊。

## 五、反例(逐一對應你的症狀)

**反例 1 —— 吞掉(你的「沒有日誌」)**
```python
try:
    risky()
except Exception:
    pass          # ❌ 最邪惡的一行:錯誤人間蒸發
```
✅ 至少記下來;真的要忽略,也要 `log.debug("ignored: ...", exc_info=True)` 留痕並寫清楚為何能忽略。

**反例 2 —— 只記 message(你的「只有一行」)**
```python
log.error(f"failed: {e}")     # ❌ 沒堆疊
log.error("save failed", e)   # ❌ Python 這樣 e 被當成 % 參數,不會印堆疊!
```
✅ `log.exception("save failed")` 或 `log.error("save failed", exc_info=True)`。
> 注意 Python 和 Java 相反:Java 是 `log.error("msg", e)` 把 e 當最後一參(會印堆疊);Python 要用 `exc_info=True`,直接傳 e 當參數**不會**印堆疊。

**反例 3 —— 又記又拋(你的「重複」cascade)**(見 🔬)→ 中間層別記。

**反例 4 —— 把可預期業務狀況當 ERROR 異常記**
```python
except UserNotFound:
    log.exception("error!")    # ❌ 查無此人是正常業務,不是該叫醒人的 ERROR(呼應 01)
```
✅ `log.info("user not found, uid=%s", uid)`,回 404。不要動用 exception-logging。

## 六、落地清單

- [ ] 心裡有 **log-or-throw**:中間層只包裝往上拋,**只在頂層記一次**。
- [ ] 有一個**全域/頂層異常處理器**(FastAPI handler / http 頂層 / `@RestControllerAdvice`)記未捕獲異常。
- [ ] 記異常一律帶**完整堆疊**:`log.exception()`(Py)/ `"err", err`(Go,err 用 `%w` 包)/ `log.error("msg", e)`(Java)。
- [ ] **包裝時保留 cause**:`raise X from e` / `%w` / `new X(msg, e)`。
- [ ] 沒有任何**空 catch**(`except: pass` / `catch (e) {}`);要忽略也要留痕 + 註明原因。
- [ ] 可預期的業務狀況(查無資料、參數非法)**不走 exception-logging**,用 INFO/WARN(呼應 `01`)。
