# 02 · 該在哪記 / 記什麼(邊界 × 決策點)

> 治的症狀:**「沒日誌」**(該記的地方沒記)和**「一長串撈不到」**(到處亂記)。
> 這兩個是同一個病的兩面:**不知道哪裡該記**。記太散 → 重點被淹;記太少 → 關鍵點空白。

---

## 一、原理:只在兩種地方記 —— 邊界 與 決策點

新手記 log 的方式是「想到就記」「進函式記一條、出函式記一條」,結果到處是 `enter foo` / `leave foo`,真正有用的訊號淹沒在裡面。

資深的做法:**日誌是系統行為的敘事**,只在兩種位置記:

### ① 邊界(boundary):資料/控制權跨過一條線的地方

| 邊界 | 記什麼 |
|---|---|
| **請求進 / 出**(HTTP / RPC / MQ 消費) | 進:method、path、關鍵參數、request_id;出:status、耗時 |
| **外部呼叫進 / 出**(DB、第三方 API、cache、MQ 生產) | 目標、關鍵參數、耗時、成功/失敗 |
| **程序生命週期** | 啟動(版本、設定摘要)、關閉、健康檢查狀態變化 |
| **排程 / 批次任務** | 開始(參數)、結束(處理 N 筆、耗時) |

邊界是 debug 的錨點:**任何問題第一個問題都是「請求有沒有進來?有沒有出去?在哪個外部呼叫卡住?」** 把邊界記好,80% 的「服務沒反應」類問題當場定位。

### ② 決策點(decision point):程式「做了選擇」或「狀態變了」的地方

> 口訣:**記決策,不記資料(log the decision, not the data)。**

| 決策點 | 記什麼 |
|---|---|
| 走了哪條分支 | 「命中快取 / 未命中,回源」「feature flag X 開啟,走新路徑」 |
| retry / fallback | 「第 2 次重試」「主庫失敗,切讀庫」 |
| 限流 / 熔斷 | 「限流觸發,拒絕 request」「熔斷器 open」 |
| 狀態轉換 | 「訂單 PENDING → PAID」 |
| 業務里程碑 | 「訂單成立 order_id=...」 |

決策點是 debug 的另一半:**「為什麼這個請求走了奇怪的路?」** 答案就在你有沒有把「選擇」記下來。

### 不該記的地方(這些就是你的「一長串」)

- ❌ getter / setter、純資料搬運、DTO 轉換
- ❌ 框架已經幫你記的(HTTP access log、ORM 的 SQL —— 別重複記)
- ❌ 每個函式的進入/離開(那是 trace 的工作,不是 log,→ `observability/`)
- ❌ **迴圈的每一圈**(見下)

## 二、關鍵技巧:記聚合,不記逐筆

「一長串撈不到」最大的元兇是**在迴圈裡記**:

```python
for row in rows:                      # rows 可能上萬筆
    process(row)
    log.info("processed row %s", row.id)   # ❌ 一個請求噴一萬行
```

正解:**迴圈外記聚合摘要**,迴圈內最多在「異常的那幾筆」記:

```python
ok, failed = 0, 0
for row in rows:
    try:
        process(row)
        ok += 1
    except Exception:
        failed += 1
        log.warning("row failed, id=%s", row.id)   # 只記壞的那幾筆
log.info("batch done, total=%d ok=%d failed=%d", len(rows), ok, failed)  # ✅ 一條摘要
```

一條好的聚合日誌(`total=10000 ok=9998 failed=2`)比一萬行逐筆有用得多 —— 它同時給你**規模感**和**異常率**。

## 三、三語言並排:邊界與決策點怎麼記

> 完整的「請求進出中介層」三語言實作在 `08` 壓軸章,這裡只示意「同一個點、三種寫法」。

**邊界:外部呼叫包一層,記目標 + 耗時 + 結果**

```python
# Python
import time
start = time.monotonic()
try:
    resp = httpx.get(url, timeout=3)
    log.info("downstream ok, url=%s status=%d ms=%.0f",
             url, resp.status_code, (time.monotonic() - start) * 1000)
except httpx.HTTPError as e:
    log.warning("downstream failed, url=%s ms=%.0f err=%s",
                url, (time.monotonic() - start) * 1000, e)
    raise
```

```go
// Go
start := time.Now()
resp, err := client.Get(url)
if err != nil {
    logger.WarnContext(ctx, "downstream failed",
        "url", url, "ms", time.Since(start).Milliseconds(), "err", err)
    return err
}
logger.InfoContext(ctx, "downstream ok",
    "url", url, "status", resp.StatusCode, "ms", time.Since(start).Milliseconds())
```

```java
// Java
long start = System.nanoTime();
try {
    var resp = client.get(url);
    log.info("downstream ok, url={} status={} ms={}",
             url, resp.statusCode(), (System.nanoTime() - start) / 1_000_000);
} catch (HttpException e) {
    log.warn("downstream failed, url={} ms={}",
             url, (System.nanoTime() - start) / 1_000_000, e);  // e 當最後一參
    throw e;
}
```

**決策點:記「選了什麼」**

```python
if cached := cache.get(key):
    log.debug("cache hit, key=%s", key)        # 命中是常態,DEBUG 就好
    return cached
log.info("cache miss, key=%s, querying db", key)  # 未命中走 DB,值得 INFO
```

三語言的差別只在語法,**「在邊界和決策點記」這條紀律一模一樣**。

## 四、反例

**反例 1:迴圈裡逐筆記。**(見第二節)→ 改記聚合。

**反例 2:記資料不記決策。**
```python
log.info("user=%s data=%s", user, big_dict)   # ❌ 倒了一堆資料,但看不出「發生了什麼」
```
✅ `log.info("user %s checkout failed: insufficient balance", user.id)` —— 記事件與原因。

**反例 3:enter/leave 每個函式都記。**
```python
def foo():
    log.info("enter foo")    # ❌ 這是 trace 的工作,不是 log
    ...
    log.info("leave foo")
```
✅ 函式進出的時序圖交給分散式追蹤(`observability/`);log 只記有意義的邊界與決策。

**反例 4:重複記框架已經記的。**
→ HTTP access log、SQL log 框架都有;你再手動記一遍只是製造雜訊。先確認框架記了沒,別疊。

## 五、落地清單

- [ ] 每支服務的**請求進/出**有日誌(method、path、status、耗時、request_id)。
- [ ] 每個**外部呼叫**(DB/API/cache/MQ)有「目標 + 耗時 + 成敗」。
- [ ] 重要**決策與狀態轉換**有記(分支、retry、fallback、限流、里程碑)。
- [ ] **迴圈裡沒有 log**;改成迴圈外記聚合(`total/ok/failed`),只在壞的那幾筆記。
- [ ] 沒有 `enter/leave` 式噪音;沒有重複記框架已記的東西。
