## 先把主线串起来：一段代码到底怎么执行

先不要从 syscall、上下文切换这些大词开始。面试里更稳的讲法，是从一段普通代码开始，逐层讲到 CPU、内存、内核：

```c
int add(int a, int b) {
    int c = a + b;
    return c;
}

write(fd, buf, len);
```

这段代码在源码里很简单，但机器真正执行时，会经过下面这条链：

```text
源码
  → 编译成机器指令
  → PC/rip（指令指针）指向下一条指令
  → 指令把值装进寄存器
  → ALU（算术逻辑单元）在寄存器上做加法/比较/地址计算
  → 需要内存数据时，通过地址访问 L1/L2/L3/内存
  → 函数调用时用栈帧保存返回地址、局部变量、溢出的寄存器
  → syscall 时按 ABI/内核约定把参数放寄存器，执行 syscall/svc（陷入内核的指令）
  → CPU 进内核，内核保存用户态寄存器现场到内核栈
  → 如果要调度，内核保存当前任务恢复线索，切到下一个任务
  → 恢复时还原 PC/SP（指令指针/栈指针）和通用寄存器，线程从原来的机器指令继续跑
```

先把这条链里的缩写和硬件名词翻译成人话。后面看到它们时，不要先背英文，全都代回“CPU 下一步去哪”“当前栈用到哪”“算术在哪里做”“怎么进内核”这几个问题。

| 名词 | 英文全称/原词 | 中文抓手 | 在哪里 | 本章里怎么用 |
|---|---|---|---|---|
| CPU | Central Processing Unit | 中央处理器，真正执行机器指令的硬件 | 处理器核心/多个核心 | 取指令、解码、执行、访问寄存器和内存 |
| `PC` | Program Counter | 指令指针，CPU 下一步要执行哪条机器指令 | CPU 寄存器状态 | 上下文切换必须保存它，否则恢复后不知道从哪条指令继续跑 |
| `rip` | x86-64 的 instruction pointer register 名字；可理解成 64-bit Instruction Pointer | x86-64 上的指令指针 | CPU 寄存器状态 | 和通用说法 `PC` 是同一类东西，只是架构命名不同 |
| `SP` | Stack Pointer | 栈指针，当前线程栈顶地址 | CPU 寄存器状态 | 它不是整条栈，只是指向栈内存的地址；保存它就能重新接回调用链 |
| `rsp` | x86-64 的 stack pointer register 名字；`R` 表示 64-bit register 形态 | x86-64 上的栈指针 | CPU 寄存器状态 | 和通用说法 `SP` 是同一类东西，只是架构命名不同 |
| `ALU` | Arithmetic Logic Unit | 算术逻辑单元，真正做加减、比较、位运算的执行部件 | CPU 执行单元内部 | `a + b` 这类计算通常是先把值放进寄存器，再由 ALU 算 |
| L1/L2/L3 cache | Level 1 / Level 2 / Level 3 cache | 第一级/第二级/第三级缓存 | CPU 内部或 CPU 封装内 | 访问内存时硬件先查 cache，没命中才逐层往 DRAM 找 |
| `syscall` | system call；x86-64 上也有同名 `syscall` 指令 | 系统调用/进入内核的特殊入口 | CPU 指令 + Linux 内核入口约定 | 用户态不能直接操作内核资源，要通过它切到内核态 |
| `svc #0` | Supervisor Call | aarch64 上进入内核的特殊指令 | CPU 指令 + Linux 内核入口约定 | 和 x86-64 的 `syscall` 是同类动作：从用户态陷入内核态 |
| `ABI` | Application Binary Interface | 二进制调用约定，规定参数/返回值/寄存器保存规则 | 编译器、库、内核、CPU 架构之间的约定 | 解释为什么 `write(fd, buf, len)` 编译后参数会出现在固定寄存器里 |
| trap frame | trap frame；trap 指 CPU 从当前执行流陷入内核/异常入口 | 陷入内核时保存的一份用户态 CPU 现场 | 通常在当前任务的内核栈上 | 里面保存用户态 PC、SP、通用寄存器等，内核返回用户态时按它恢复 |
| `CR3` | Control Register 3 | x86 上保存页表基址的控制寄存器 | CPU 特权寄存器 | 进程间切换可能要改它；线程共享同一地址空间时通常不用改 |
| `TTBR` | Translation Table Base Register | ARM 上保存页表基址的寄存器 | CPU 特权寄存器 | ARM/aarch64 上承担类似 `CR3` 的角色 |
| `TLB` | Translation Lookaside Buffer | 地址翻译缓存，缓存“虚拟地址 → 物理地址”的翻译结果 | CPU 内部缓存 | 换页表后很多翻译失效，重新填充 TLB 是上下文切换成本来源之一 |

`rax`、`rdi`、`rsi`、`rdx`、`x0`、`x8` 这类名字也是寄存器名，不是每个都要拆成英文单词背。更好的记法是按用途记：`rax` 常放返回值或 syscall 编号，`rdi/rsi/rdx` 常放 x86-64 前几个参数；`x0/x1/x2` 常放 aarch64 前几个参数，`x8` 常放 aarch64 syscall 编号。

再补几个会在执行过程中出现的内存层次词：

| 名词 | 英文全称/原词 | 中文抓手 | 怎么理解 |
|---|---|---|---|
| cache line | cache line | cache 一次搬运/缓存的最小块，常见 64 字节 | CPU 不是只缓存一个 int，而是把相邻一小段内存一起带进 cache |
| DRAM | Dynamic Random-Access Memory | 主内存，平时说的“内存条” | 容量大但远；L1/L2/L3 都没命中时才去这里取 |
| store buffer | store buffer | 写缓冲 | CPU 写内存时通常先把写入放到缓冲/缓存里，之后再逐层落到更下层 |
| FPU | Floating-Point Unit | 浮点计算单元 | 处理浮点计算；上下文切换时可能也要保存它的状态 |
| SIMD | Single Instruction, Multiple Data | 单指令多数据，向量化批量计算 | 一条指令同时处理多个数据，常用于向量、加密、压缩、多媒体 |
| link register / `LR` | Link Register | 链接寄存器，常见于 aarch64 的 `x30` | 保存函数返回地址；如果函数还要继续调用别人，通常会把它保存到栈上 |

现在再看几个最容易混的概念：寄存器、栈、cache。先把三者底层身份分清：

| 概念 | 本质 | 谁管理 | CPU 怎么用 |
|---|---|---|---|
| 寄存器 | CPU 核心内部的少量硬件存储格子 | 编译器 + ABI + CPU 指令显式使用 | 指令直接点名，如 `rax`、`rsp`、`x0`、`sp` |
| 栈 | 当前线程虚拟地址空间里的一段内存 | 编译器、ABI、运行时、内核共同约定 | 通过 `rsp` / SP 这个寄存器指向，按地址读写 |
| L1/L2/L3 cache | 内存内容的高速副本 | CPU 硬件自动管理 | 程序通常不能直接点名，只是访问内存时自动命中/未命中 |

所以一句话先定住：**寄存器不是 cache，栈也不是寄存器。寄存器在 CPU 里；栈在内存里；cache 是 CPU 为了加速内存访问而自动维护的内存副本。**

### 1. CPU 不是执行源码行，而是执行机器指令

源码里的：

```c
c = a + b;
```

编译后可能变成类似这样的机器级动作（不是固定汇编，只看概念）：

```text
把 a 放进寄存器 R1
把 b 放进寄存器 R2
在寄存器里执行 R3 = R1 + R2
把 R3 作为 c 的值继续使用，或写回某个内存位置
```

CPU 真正关心的是 `PC` / `rip`。它不是“源码第几行”，而是“下一条机器指令的地址”。CPU 取到这条指令，解码，再让执行单元去做对应动作。加法、比较、位运算、地址计算这类操作，通常发生在寄存器上。机器指令可以访问内存，但大多数计算指令的核心输入输出仍然是寄存器。

这就是为什么“变量”不能直接等同于“栈上的一个格子”。`a`、`b`、`c` 在源码里是变量名；编译后可能变成：

- 直接放在寄存器里；
- 常量折叠后根本没有运行时变量；
- 因为寄存器不够，被 spill 到栈上的某个栈槽；
- 因为代码要取地址，比如 `&c`，必须真的有一个内存位置；
- 因为优化、内联、尾调用，源码里的函数边界和变量生命周期被改写。

面试可以这样说：**源码变量是编译器给人看的抽象；CPU 执行的是机器指令，机器指令直接操作寄存器和内存地址。**

### 2. 寄存器、L1/L2 cache、内存的执行顺序

假设某个局部变量已经在栈上，指令要把它读出来相加，过程大致是：

```asm
mov rax, [rsp+8]
add rax, rbx
mov [rsp+8], rax
```

可以按下面理解：

```text
rsp 是寄存器，里面放一个地址：当前线程栈顶附近的位置
[rsp+8] 是一个内存地址：栈上的某个槽位

mov rax, [rsp+8]
  CPU 要读这个内存地址
  先查 L1 cache 有没有这个地址所在的 cache line
  L1 没有再查 L2，L2 没有再查 L3，最后才去 DRAM
  读到的值放进 rax 这个寄存器

add rax, rbx
  加法在寄存器里完成
  不需要再访问内存

mov [rsp+8], rax
  把寄存器里的结果写回这个栈地址
  写入通常先进入 store buffer / cache，之后再逐层同步到更下层
```

这里能看出三层的关系：

- **寄存器保存 CPU 正在使用的值**：如 `rax`、`rbx`、`rsp`。指令能直接点名它们。
- **cache 保存内存地址对应内容的副本**：如果栈上的 `[rsp+8]` 很热，它所在的 cache line 可能在 L1/L2；但它本质仍是内存位置，不是寄存器。
- **内存保存程序的大状态**：栈、堆、代码段、mmap 区域都在虚拟地址空间里。CPU 通过地址访问它们。

因此不要说“栈在 L1 里”或者“寄存器在 cache 里”。更准确的说法是：**栈是内存区域；栈上某些热数据的副本可能正在 L1/L2 cache 里；CPU 真正计算前，会把需要的值装进寄存器。**

### 3. 为什么寄存器和栈里会放类似内容

你会觉得寄存器和栈里放的东西差不多，是因为它们都可能出现“参数、返回值、局部变量、中间结果”。但这只是内容相似，不是本质相同。

根本原因有三个：

1. **寄存器数量很少**：CPU 能直接操作的寄存器有限。活跃变量太多、表达式太复杂、函数调用太深时，编译器会把暂时放不下的值写到栈上，这叫 spill。之后需要时再从栈 load 回寄存器。

2. **函数调用需要遵守 ABI**：ABI 会规定参数先放哪些寄存器、返回值放哪里、哪些寄存器由 caller 保存、哪些由 callee 保存。比如 x86-64 SysV 函数调用前几个整数/指针参数放 `rdi`、`rsi`、`rdx`、`rcx`、`r8`、`r9`；返回值常放 `rax`。参数太多、结构体太大、或者 ABI 要求栈上传参时，才会继续用栈。

3. **调用前后要保护现场**：如果调用方某个寄存器里的值之后还要用，但被调函数可能覆盖它，调用方就要先把它保存起来，常见位置就是栈。被调函数如果用了 callee-saved registers，也要在函数开头保存、返回前恢复。这些保存动作会让“寄存器里的值”临时出现在栈上。

所以可以用这句话面试回答：**寄存器是 CPU 当前执行的工作区，栈是函数调用链和寄存器后备区。两者会存类似内容，是因为 ABI 和编译器会在二者之间搬运值。**

### 4. 栈不是 CPU 内部结构，`SP/rsp` 才是寄存器

栈经常被讲得像一个硬件容器，但在 Linux 进程里，用户栈本质上就是当前线程虚拟地址空间里的一段内存。每个 OS 线程有自己的用户栈；进入内核后，同一个任务还会使用自己的内核栈。

真正被 CPU 特别认识的是 `SP` / `rsp` 这个寄存器。它里面不是“整条栈”，而是一个地址：当前栈用到哪里了。以 x86-64 的传统函数调用为例，编译器生成的 prologue/epilogue 会移动 `SP`：

```text
call foo
  返回地址进入栈，CPU 跳到 foo

foo prologue
  保存旧 frame pointer
  建立新 frame pointer
  移动 SP，给局部变量/临时值留空间

foo epilogue
  恢复 SP
  弹出返回地址
  跳回调用方
```

aarch64 的形状不同：返回地址通常先放在 `x30`（link register）里；如果函数还要继续调用别的函数，才常把 `x29/x30` 保存到栈上。概念不变：栈帧里可能有返回地址或 link register 的保存值、保存的 frame pointer、保存的寄存器、局部变量、spill 临时区、给下一层函数的参数区。这些内容仍然是内存里的字节。函数返回时，通常也不会把这些字节清零，只是把 `SP` 调回去；旧内容还在，但不再属于当前有效栈帧，之后会被覆盖。

这点对上下文切换非常关键：**内核切换任务时不复制整条栈，只保存 `SP` 和其他寄存器。** 因为栈内容本来就在这个线程的地址空间里，没必要搬走。恢复时只要把 `SP` 放回原来的地址，CPU 就能继续按那条栈找到当前函数、上一层函数、返回地址和局部状态。

### 5. syscall 是一条特殊的“进内核”路径

普通函数调用是在用户态内部跳转；syscall 是用户态主动请求内核服务。比如：

```c
write(fd, buf, len);
```

用户代码看到的是 libc 函数 `write()`，但真正进内核前，libc 或运行时要按平台约定准备寄存器：

```text
x86-64 Linux syscall:
  rax = syscall 编号，比如 write 是 1
  rdi = fd
  rsi = buf
  rdx = len
  执行 syscall 指令

aarch64 Linux syscall:
  x8 = syscall 编号
  x0 = fd
  x1 = buf
  x2 = len
  执行 svc #0
```

CPU 执行 `syscall` / `svc #0` 后，会从用户态切到内核态，跳到内核预先设置好的入口。内核入口代码不会去读 C 语言参数名，它只按约定读寄存器。进入内核后，用户态的 `PC`、`SP`、通用寄存器等现场会被保存到当前任务的内核栈上，形成 trap frame。这样内核处理完 `sys_write` 后，才能把这些寄存器恢复，再回到用户态下一条指令继续执行。

所以 syscall 可以这样讲：**它不是“调用一个很慢的函数”这么简单，而是 CPU 特权级切换 + 按寄存器约定传参 + 保存用户态现场 + 进入内核入口 + 再恢复现场返回。**

### 6. 上下文切换为什么保存寄存器，不复制栈

如果线程 A 跑到 `bar()` 中间被暂停，CPU 里最关键的现场是：

```text
PC/rip: 下一条机器指令在哪里
SP/rsp: 当前线程的栈顶在哪里
通用寄存器: 当前计算到一半的值、参数、返回值、临时指针
FPU/SIMD 寄存器: 浮点、向量计算状态（如果用到）
```

内存里的状态是：

```text
A 的用户栈: main → foo → bar 的栈帧还在
A 的堆: 对象、malloc 出来的内存还在
A 的代码段/mmap 区域: 仍在进程地址空间里
```

切到线程 B 时，内核不需要复制 A 的栈、堆、代码段。它只要保存“下次怎么恢复 A”：

```text
保存 A 的 PC/SP/寄存器
保存 A 的内核栈恢复线索到 task_struct.thread
如果切到另一个进程，可能还要切页表（CR3/TTBR）
加载 B 的恢复线索
恢复 B 的 PC/SP/寄存器
CPU 开始执行 B 下一条机器指令
```

再切回 A 时，把 A 的 `PC`、`SP`、寄存器恢复回来，CPU 就能从 `bar()` 中间继续跑。A 的栈内容没有被复制过，只是一直留在内存里。真正会变的是 cache：A 之前热在 L1/L2 里的数据，可能被 B 的执行挤掉了。切回 A 后，寄存器恢复很快，但 A 重新访问自己的栈/堆数据时，可能要重新从 L2/L3/内存把 cache warm up 回来。这就是上下文切换成本里很大一部分来自 cache 冷效应。

### 7. 面试时可以这样串

如果面试官问“寄存器、栈、cache、上下文切换到底什么关系”，不要先背定义，可以按执行过程回答：

> CPU 执行的不是源码行，而是 `PC/rip` 指向的机器指令。指令做计算时优先用寄存器，比如加法通常是在寄存器里完成。变量如果在栈或堆里，CPU 会按地址去读内存；读内存时硬件先查 L1/L2/L3 cache，命中就很快，未命中才去 DRAM。栈本身不是 CPU 内部结构，而是线程虚拟地址空间里的一段内存，`SP/rsp` 这个寄存器指向当前栈顶。函数调用会在栈上留下返回地址或 link register 的保存值、保存的寄存器、局部变量或 spill 区。寄存器和栈会放类似内容，是因为 ABI 和编译器会在二者之间搬运参数、返回值和临时状态。发生 syscall 或上下文切换时，内核保存的是 PC、SP 和寄存器这些 CPU 现场，栈内容本身留在内存里，不会整段复制；切回来恢复这些寄存器后，线程就能从原来的机器指令继续执行。

这段话能覆盖本章后面的六个原语：syscall 解释“怎么进内核”，用户态/内核态切换解释“为什么要保存现场”，中断解释“异步打断从哪里来”，上下文切换解释“任务怎么停下再继续”，栈帧解释“函数调用链怎么保存”，进程 vs 线程解释“内核调度的对象是什么”。

## 原语一：syscall（系统调用）

### ① 你视角

Java 里你调 `FileOutputStream.write(buf)`，Go 里你调 `os.File.Write(buf)`——底层最终都落到同一件事：**用一条特殊指令让 CPU 跳进内核**。这条路叫 syscall（系统调用）。libc 的 `write()`、`read()`、`mmap()` 函数只是入口登记台，真正的门在 `syscall` 指令这一步。

这里的 **libc** 是「C 标准库 + POSIX/Linux 常用函数」在用户态的一套实现，不是 Linux 内核本身。常见实现有 GNU/Linux 上的 `glibc`、Alpine Linux 常见的 `musl`、Android 的 `bionic`。它给程序员一个普通函数接口，比如 `write(fd, buf, len)`；函数内部再按 CPU/内核约定把参数和 syscall 编号放进寄存器，最后执行真正进内核的那条指令。

#### 寄存器最小模型

寄存器可以先理解成 CPU 内部少量、极快的「工作格子」。CPU 正在用的值会临时放在这些格子里：函数参数、返回值、计算到一半的中间值、下一条指令地址、当前栈顶地址。内存像桌面上的大文件柜，容量大但远；寄存器像 CPU 手边的便签，数量少但拿取最快。

所以 syscall 里说「把参数放进寄存器」，本质是用户程序和内核之间有一套调用约定：x86-64 上 `write(fd, buf, len)` 的三个参数放在 `rdi` / `rsi` / `rdx`，syscall 编号放在 `rax`；aarch64 上参数放在 `x0` / `x1` / `x2`，编号放在 `x8`。内核入口代码不会去读 C 语言里的参数名，而是按这套约定直接从寄存器里取值。

上下文切换也离不开寄存器。一个任务被暂停时，CPU 当前的 `rip` / PC（下一条指令地址）、`rsp` / SP（当前栈顶）、通用寄存器（临时值、参数、返回值）都要保存起来。切回来时恢复这些寄存器，CPU 才能像刚才只是暂停了一下那样继续执行。栈和堆的大块内存不会被复制，保存的是「从哪里继续」和「CPU 手上那点现场」。

对资深工程师/架构师来说，寄存器至少要理解到这几层：

| 层次 | 要抓住的点 | 为什么重要 |
|---|---|---|
| 指令位置 | `rip` / PC 指向下一条机器指令 | 解释程序为什么能从中断、syscall、上下文切换后继续跑 |
| 栈位置 | `rsp` / SP 指向当前线程栈顶 | 解释函数调用链、局部变量、返回地址如何找回来 |
| 通用寄存器 | 放整数、指针、参数、返回值、中间结果 | 解释普通代码、函数调用、syscall 参数如何传递 |
| 标志寄存器 | 保存比较/进位/中断开关等状态（x86 的 `rflags`、aarch64 的 `PSTATE`） | 解释条件跳转、抢占/中断控制为什么也是 CPU 现场 |
| FPU/SIMD 寄存器 | 放浮点、向量、加密/压缩等批量计算状态 | 解释为什么大量 SIMD 工作负载的上下文保存更重 |

#### 源码变量不等于内存格子

`a = 1; b = 1; c = a + b;` 这种源码看起来像三个变量都在内存里，但编译后不一定。编译器会做寄存器分配和优化：

```text
源码变量 a/b/c
  → 可能直接放在寄存器里
  → 可能因为常量折叠变成一个立即数 2
  → 可能因为寄存器不够被 spill 到栈上
  → 如果变量需要被取地址，才更可能真的落到内存里
```

所以排查性能或理解底层时，不要把「源码变量」直接等同于「栈上的一个格子」。CPU 执行的是机器指令，机器指令直接操作的是寄存器和内存地址；变量名只是编译器前端和调试信息里的抽象。

#### ABI：寄存器和栈之间的约定

函数调用和 syscall 都不是随便把值塞进哪里，而是遵守 ABI（Application Binary Interface，二进制接口约定）。ABI 规定「参数放哪些寄存器」「返回值放哪里」「哪些寄存器调用方负责保存」「哪些寄存器被调用方负责恢复」。

以常见 64 位平台为例：

| 场景 | 参数主要放哪里 | 返回值主要放哪里 | 重点 |
|---|---|---|---|
| x86-64 SysV 函数调用 | `rdi`、`rsi`、`rdx`、`rcx`、`r8`、`r9` | `rax` | Linux 用户态 C ABI，参数太多才继续放栈上 |
| x86-64 Linux syscall | `rdi`、`rsi`、`rdx`、`r10`、`r8`、`r9`，编号在 `rax` | `rax` | 注意第 4 个参数用 `r10`，不是函数调用 ABI 的 `rcx` |
| aarch64 函数调用/syscall | 参数常用 `x0`–`x7`，syscall 编号在 `x8` | `x0` | `x30` 是 link register，常用于保存返回地址 |

这也是为什么同样是 `write(fd, buf, len)`，C 代码里有参数名，内核入口却只看寄存器。参数名在编译后不存在，ABI 才是用户态和内核、调用方和被调用方之间真正的契约。

还有一个面试常问点：**caller-saved / callee-saved**。有些寄存器调用函数前如果还想保留，调用方自己先保存；有些寄存器被调用函数用了以后必须恢复给调用方。这个约定让不同编译单元、不同库、甚至不同语言编译出来的二进制能互相调用。

### ② 黑盒内部

**syscall 是用户态程序进入内核的唯一合法通道。**

调用链：

```
你的代码  →  libc write()  →  syscall 指令  →  内核 sys_write()  →  返回
```

每一步拆开看：

1. **libc 是薄封装**：对 `write()` 这类系统调用包装函数来说，函数体核心只有几步——把参数放进寄存器（x86-64: `rdi`/`rsi`/`rdx`；aarch64: `x0`/`x1`/`x2`），把 syscall 编号放进 `rax`（x86-64）或 `x8`（aarch64），然后执行一条指令。这条指令就是真正的触发器。`printf()`、`malloc()` 这种库函数会更复杂，但它们一旦需要让内核做 I/O、映射内存等事情，最后仍然要走 syscall。

2. **触发指令**：x86-64 叫 `syscall`，aarch64 叫 `svc #0`（Supervisor Call）。CPU 收到这条指令，立刻：
   - 把当前用户态指令地址（PC）和栈指针存到特定寄存器/内核栈
   - x86-64 从 `IA32_LSTAR` MSR 直接读取内核入口地址，**不经过 IDT**——IDT/异常向量表是硬件中断与 CPU 异常（缺页、除零）的入口，不是 `syscall` 指令的路径；aarch64 上 `svc #0` 跳到 `VBAR_EL1` 指向的异常向量基址
   - 把 CPU 特权级从 Ring 3（用户态）切到 Ring 0（内核态）
   - 跳到内核 `entry_SYSCALL_64`（x86-64）或 `el0_svc`（aarch64）入口函数

3. **内核里**：根据 syscall 编号查系统调用表（`sys_call_table[]`），dispatch 到真正的实现（`sys_write`、`sys_read`...），执行完后执行 `sysret`/`eret` 返回用户态。

4. **libc 不是 syscall 本身**：syscall 是那条 CPU 指令，不是 libc 函数。你完全可以绕过 libc 用汇编直接发 syscall。Go 的 `syscall` 包正是这样做的——它直接用汇编包裹 `SYSCALL` 指令，不走 glibc。

常见 syscall 编号（x86-64，`/usr/include/asm/unistd_64.h`）：

| 编号 | 名字 | 作用 |
|------|------|------|
| 0 | `read` | 从 fd 读 |
| 1 | `write` | 向 fd 写 |
| 9 | `mmap` | 建立内存映射 |
| 231 | `exit_group` | 进程退出 |

### ③ 砸实

`strace` 是拦截进程所有 syscall 的工具。下面跑 `echo hi`，把它发出的 syscall 序列完整抓出来：

```bash
docker run --rm --cap-add=SYS_PTRACE arm64v8/ubuntu:22.04 bash -c '
apt-get update -qq && apt-get install -y -qq strace >/dev/null 2>&1
strace echo hi 2>&1
'
```

真实输出（截取关键部分，省略中间动态链接过程）：

```
execve("/usr/bin/echo", ["echo", "hi"], 0xfffff4e94068 /* 6 vars */) = 0  # 进程启动
brk(NULL)                               = 0xaaab024a8000                  # 查堆顶
mmap(NULL, 8192, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0) = 0xffffab268000  # 内核数据结构
openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY|O_CLOEXEC) = 3              # 打开动态链接缓存
close(3)                                = 0
# ... 动态链接 libc.so ...
write(1, "hi\n", 3)                     = 3                               # 真正的 write syscall
close(1)                                = 0
close(2)                                = 0
exit_group(0)                           = ?                                # 退出
+++ exited with 0 +++
```

**看点**：`write(1, "hi\n", 3) = 3`——fd=1（stdout），内容 `"hi\n"`（3字节），返回 3（实际写入字节数）。`exit_group(0)` 是进程退出 syscall（不是 C 的 `exit()` 库函数）。`echo hi` 这么简单一个命令，真正跑起来有几十个 syscall——这就是「libc 是薄封装、真实工作在内核」的体现。

## 原语二：用户态↔内核态切换

### ① 你视角

你知道 Go 的 goroutine 比传统 Java platform thread（一个 Java 线程通常对应一个 OS 线程）轻——部分原因就是 goroutine 调度在用户态完成，不走内核调度器。但这不等于 Go 不进内核：但凡涉及文件 I/O、网络 I/O、进程创建、内存映射等内核能力，无论 goroutine 还是线程，都得走 syscall。Go 省下来的主要是「大量 goroutine 等待/恢复/互相切换」时，不必把每个等待单元都交给内核线程调度。

两条路径分开看：

```text
需要内核能力：
  goroutine → Go runtime → syscall/epoll/read/write → Linux 内核

只是在 goroutine 之间切换：
  goroutine A 暂停 → Go runtime 记录状态 → 同一个 OS 线程继续跑 goroutine B
  （这一段通常不需要 syscall，也不触发内核 context switch）
```

**到底省在哪里？** 不是把必须的内核工作省掉，而是把「等待者的管理」从内核线程调度里搬到 Go runtime。

传统阻塞线程模型里，一个线程读网络但暂时没数据：

```text
OS 线程 A 调 read(fd)
  → 进内核
  → 内核发现没数据，把线程 A 挂起
  → 内核 schedule() 另一个线程
  → fd 可读后，内核唤醒线程 A
  → 线程 A 重新被内核调度上 CPU
```

这里每个等待者都是一个内核调度对象。10 万个等待连接，就意味着内核要知道并管理大量睡眠/唤醒/调度对象。

Go 网络 I/O 的常见路径是非阻塞 fd + runtime netpoller（Linux 上基于 `epoll`）：

```text
goroutine G1 想读 fd
  → Go runtime 尝试非阻塞 read
  → 没数据时返回 EAGAIN
  → runtime 把 G1 park 在用户态等待队列
  → 同一个 OS 线程继续跑 G2/G3/G4

少量 OS 线程通过 epoll_wait 批量等很多 fd
  → 一次进内核可以等成千上万个 fd
  → fd 可读后，runtime 把对应 G 放回 runnable 队列
```

所以减少的是：

| 省下来的东西 | 为什么能省 |
|---|---|
| goroutine A → goroutine B 的切换 syscall | runtime 自己保存/恢复 goroutine 状态，不调用内核 `schedule()` |
| 每个等待连接一个阻塞 OS 线程 | goroutine 只是用户态对象，fd 等待由 `epoll` 批量管理 |
| 大量内核线程睡眠/唤醒/抢占成本 | 内核只看见少量 OS 线程，Go runtime 在用户态调度大量 goroutine |

没省掉的是：真正 `read`/`write` 数据、`epoll_wait` 等待事件、创建进程、`mmap` 内存这些内核能力仍然要 syscall。Go 的收益来自「用较少的内核交互管理大量等待任务」，不是来自「I/O 不用内核」。

### ② 黑盒内部

**每次 syscall/中断都包含一次用户↔内核态切换，成本不可忽视。**

先把「保存现场」分三层看：

| 层次 | 先怎么理解 | 需要记住什么 |
|---|---|---|
| CPU 手上的现场 | CPU 正在用的那点值 | `PC/rip` 表示下一条指令在哪，`SP/rsp` 表示当前栈顶在哪，通用寄存器保存临时值/参数/返回值 |
| 还在内存里的程序状态 | 栈、堆、代码段、mmap 区域 | 这些大块内存本来就在进程地址空间里，切换时不会整块复制 |
| 内核保存恢复线索的地方 | 内核栈 + `task_struct` | 用户态现场通常先保存在内核栈上的 trap frame；调度器再把恢复这个任务所需的内核栈指针等信息记到 `task_struct.thread` |

所以「上下文切换保存状态」不是把整个进程搬走，而是保存两类关键线索：**CPU 手上正在用的值**，以及**下次从哪里把这个任务恢复回来**。

切换时 CPU/内核必须做的事：

1. **保存 CPU 现场**：最核心的是 `PC/rip`、`SP/rsp` 和通用寄存器。普通整数、地址、函数参数、返回值都在通用寄存器里。若任务使用了浮点或向量计算，还可能要保存 FPU/SIMD 状态（x86 上如 XMM/YMM 寄存器）。入门先抓住一句话：保存的是「CPU 手上那点现场」，不是保存整个进程。
2. **特权级切换**：CPU 从 Ring 3 → Ring 0，涉及修改 CS 段寄存器、切换栈指针（用户栈 → 内核栈）。aarch64 等效为 EL0 → EL1。
3. **地址空间切换**（部分场景）：如果启用了 KPTI（内核页表隔离，Meltdown 缓解措施），用户态和内核态用不同 CR3（页表基址），切换时需要刷 TLB——代价极高（几百纳秒）。
4. **Cache 冷效应**：内核代码和用户代码在不同虚拟地址、不同物理页，进内核后 L1 指令缓存里全是内核代码，出内核后用户代码重新进缓存，冷了一批。

**量化参考**：一次 syscall 往返（含切换）大约 **50–500 ns**，与具体 CPU 和 KPTI 开启状态有关。相比之下，L1 缓存访问约 1 ns，内存访问约 100 ns——一次 syscall 的成本相当于若干次内存访问。

**这个代价催生了几类架构优化：**

| 优化手段 | 动机 | 原理 |
|---|---|---|
| `writev` / `sendmsg` | 合并多次 `write` 为一次 syscall | 减少切换次数，同样数据量更少 syscall |
| `io_uring`（Linux 5.1+） | 异步、批量提交，减少每次 I/O 都 syscall | 用户态 ring buffer 共享给内核；setup/唤醒仍要 syscall，开启 SQPOLL 等模式时可进一步避开每次提交的 syscall |
| `vDSO`（虚拟 DSO） | 某些只读 syscall（如 `gettimeofday`）在用户态完成 | 内核把数据映射到用户空间，读时无需切换 |

> 本原语无 ③（无干净 5 行复现能直接量化单次切换成本；真正的基准测试需要 perf/BPF 等工具，超出本章边界）。

→ **回链**：`linux-handson/03-process-model`（strace 计数、进程模型里的 syscall 频率讨论）；`os-for-architects/`（io_uring 作为架构决策的完整讨论）

## 原语三：中断 IRQ

### ① 你视角

你的 Java/Go 代码在顺序执行——但磁盘读完了数据、网卡收到了包、定时器到期了，CPU 怎么知道？不是靠你的代码轮询，而是**硬件主动打断 CPU**。这个打断机制叫中断（Interrupt Request，IRQ）。

### ② 黑盒内部

**中断是硬件异步打断 CPU 当前执行流的机制，与 syscall（软件主动发起）有本质区别。**

**IRQ vs syscall 对比：**

| 维度 | syscall（软件陷入·同步） | IRQ（硬件中断·异步） |
|---|---|---|
| 发起方 | 用户程序主动调用 `syscall` 指令 | 硬件设备（网卡/磁盘/时钟）发 IRQ 信号线 |
| 时机 | 同步，程序执行到那一行才触发 | 异步，可能在任意指令之间插入 |
| 处理上下文 | 当前进程的内核态，有进程上下文 | 中断上下文（interrupt context），无进程归属 |

**中断上下文为什么不能睡眠？**

中断处理程序运行在「中断上下文」：没有关联的 `task_struct`（没有「当前进程」），调度器不知道该把 CPU 还给谁。如果在中断处理器里调用 `schedule()`（主动睡眠），内核会直接 panic。这个约束影响内核驱动开发的每一行代码——中断处理里只能做原子操作（加减、位操作、内存读写），不能做任何可能阻塞的事（不能加 mutex、不能 `kmalloc(GFP_KERNEL)`）。

**Top half / Bottom half（上半部/下半部）设计：**

为了最小化「关中断时间」，内核把中断处理分两段：

- **Top half（上半部）**：中断处理程序本体，越快越好——只做「确认硬件、把数据放到队列里」，立即返回。
- **Bottom half（下半部）**：用 `tasklet`、`workqueue`、`softirq` 等机制把真正的处理推迟到「中断上下文结束后、普通进程上下文里」执行——这里可以睡眠、可以申请内存。

类比：网卡中断到来 → Top half 把数据包 DMA 地址记下来就返回（微秒级）→ Bottom half 的 `NAPI`/软中断把包从网卡 buffer 取出、走协议栈（可以睡眠等锁）。

> **一句破误解:** 这里的「软中断(softirq)」**不是硬件触发的中断**,而是内核自己一套**固定几类的高优先级待办队列**(`NET_RX`/`NET_TX`/`TIMER`…)。硬中断的 top half 只是去队列上「打个标记:有活」,内核在硬中断返回时、或在专门的 `ksoftirqd` 内核线程里把这些待办跑掉。名字里的「中断」是历史包袱——读成「内核的延后处理任务」就不会和硬件中断混了。

> **本原语无 ③**：在 Docker Desktop / macOS aarch64 宿主机上，容器内 `/proc/interrupts` 为空文件（Docker 虚拟化层不向容器暴露宿主机中断表）。真实 Linux 裸机上 `cat /proc/interrupts` 会看到每个 CPU 核上各中断线的累计触发次数。若在真实 Linux 上运行，可用 `watch -n1 cat /proc/interrupts` 观察网卡 IRQ 在收包时实时递增。

→ **回链**：`linux-handson/03-process-model`（驱动模型/设备 I/O 时中断的角色）

## 原语四：上下文切换（context switch）

### ① 你视角

Java 的 `Thread.sleep()` 让出 CPU，JVM 底层调 `futex`，OS 把这个线程「暂停」——但 CPU 接下来要跑另一个线程，它得先把「当前线程跑到哪了」保存下来，再把「下一个线程上次暂停在哪」恢复出来。这个保存+恢复的过程就是**上下文切换**。

### ② 黑盒内部

**上下文切换保存的是 CPU「跑某个任务时的完整状态」。**

Linux 里进程和线程都是 `task_struct`（下一个原语会细说），切换时操作系统做：

1. **保存当前任务的恢复线索**：
   - **用户态现场**：`PC/rip`（下一条指令）、`SP/rsp`（用户栈顶）、通用寄存器（临时值、参数、返回值）。这是线程以后能从原位置继续跑的关键。
   - **扩展现场**：如果任务用了浮点或向量计算，还可能保存 FPU/SIMD 状态（x86 上如 XMM/YMM 寄存器）。入门先知道它也是 CPU 现场的一部分，不用先背每个寄存器名。
   - **内核恢复线索**：每个 `task_struct` 都有自己的内核栈。调度器切走当前任务时，会把“下次怎么找回这条内核执行路径”的信息（尤其是内核栈指针）记到 `task_struct.thread`。
   - **保存在哪里**：用户态寄存器现场通常在进入内核时保存到内核栈上的 trap frame；调度器再把内核栈指针等少量恢复线索存在 `task_struct.thread`。可以先记成：大现场在内核栈，恢复入口线索在 `task_struct`。

2. **切换内存上下文**（进程间切换才有，同进程线程间可省略）：
   - 修改 CR3 寄存器（x86-64）→ 切换页表 → TLB 失效（极贵）
   - 线程共享进程地址空间，同进程线程间切换不换 CR3

3. **恢复下一个任务的状态**：根据下一个任务的 `task_struct.thread` 找回它的内核栈，再沿着内核栈上的现场恢复寄存器，最后回到它原来暂停的位置继续跑。

#### 寄存器 + 栈指针：为什么能从原地继续跑

把一个线程暂停后再恢复，核心不是把整段代码「重新跑一遍」，而是把 CPU 当时的**现场**存下来。现场最关键的是三类东西：

| 状态 | 里面是什么 | 恢复时起什么作用 |
|---|---|---|
| `rip` / PC（指令指针） | 下一条要执行的机器指令地址 | CPU 知道从哪一行机器码继续跑 |
| `rsp` / SP（栈指针） | 当前线程调用栈的栈顶位置 | CPU 知道当前函数的局部变量、返回地址、调用链从哪里找 |
| 通用寄存器 | 当前计算的中间值、函数参数、返回值、临时指针 | CPU 知道刚才算到一半的值是什么 |

关键点:**栈的内容没有被拷贝走。** 每个线程自己的栈本来就在内存里,函数调用链、局部变量、返回地址都还躺在那里。上下文切换时真正要保存的是「从哪里恢复」和「CPU 手上那点现场」:

```text
线程 A 跑到 bar() 中间:

内存里的 A 栈:
  [main 的栈帧]
  [foo  的栈帧]
  [bar  的栈帧]  ← rsp 指向这里附近

CPU 寄存器:
  rip = bar() 里下一条指令
  rsp = A 栈当前栈顶
  rax/rbx/... = bar() 算到一半的临时值

切到线程 B:
  保存 A 的 rip/rsp/寄存器
  载入 B 的 rip/rsp/寄存器

再切回线程 A:
  恢复 A 的 rip/rsp/寄存器
  CPU 从 bar() 那条指令继续执行,就像刚才只是暂停了一下
```

所以「栈指针」不是整条栈,只是指向栈顶的一个地址;「寄存器快照」也不是保存整个进程,只是保存 CPU 手上正在用的少量状态。完整的程序数据仍在进程地址空间里。

#### 用 `a = 1; b = 1; c = a + b;` 校准直觉

刚开始可以把指令指针理解成「现在执行到哪了」，但要注意：它指向的是**下一条机器指令的地址**，不是源码里的第几行。比如：

```c
a = 1;
b = 1;
c = a + b;
```

这 3 行源码编译后可能变成多条机器指令，也可能被编译器优化成更少的指令。CPU 不认识「第 1 行、第 2 行、第 3 行」这种源码概念，它只按 `rip` / PC 指向的机器指令一条条执行。

栈也不是「专门用来处理每一行计算」的地方。真正的加法、赋值地址计算，优先发生在**寄存器**里；栈更多负责保存函数调用现场：局部变量、返回地址、调用链、以及需要暂存的寄存器值。

##### 栈指针指向的栈到底是什么

先把栈从神秘名词还原成普通东西：**栈就是当前线程地址空间里的一段虚拟内存**。它之所以叫栈，不是因为硬件里有一个特殊容器，而是因为这段内存按「后进先出」的方式使用：后进入的函数先返回，后压进去的值先弹出来。

它从哪里来：

- **进程启动时**，内核执行 `execve()` 后会为主线程准备用户栈，把 `argc`、`argv`、环境变量、auxv 等启动信息放在栈上，然后把用户态 `SP` 设置到这段栈的栈顶附近，再跳到程序入口开始跑。
- **创建线程时**，`pthread` / 运行时会先在进程地址空间里分配一段内存当新线程的用户栈，再通过 `clone()` 把新线程的初始 `SP` 交给内核。所以每个 OS 线程都有自己的用户栈。
- **进入内核时**，同一个任务还会使用一条内核栈。用户栈保存用户态函数调用链，内核栈保存 syscall / 中断后内核态执行路径；二者不是同一条栈。

它怎么工作：

- 栈通常**向低地址增长**。以 x86-64 为例，`push`、`call`、函数开头的 `sub rsp, N` 会让栈指针往低地址移动；`pop`、`ret`、函数结尾的 `add rsp, N` 会让栈指针往高地址移动。aarch64 没有完全同名的 `push`/`call` 指令，但函数开头同样会移动 `SP`，把需要保存的寄存器写到栈上。
- 每次函数调用都会形成一段**栈帧**：里面可能有返回地址、上一层帧指针、被保存的寄存器、局部变量、临时空间。
- `SP` 永远指向当前线程「正在使用到哪里了」。编译器生成的机器指令会以 `SP`（有时配合 frame pointer）为基准，找到当前函数的局部变量、返回地址和保存的寄存器。
- 函数返回时不是把栈内存清零，而是把 `SP` 调回去。旧字节还在内存里，但已经不属于当前有效栈帧，之后会被新的函数调用覆盖。

它用来干嘛：

- 让函数能一层层调用，再一层层返回：`main()` 调 `foo()`，`foo()` 调 `bar()`，每层的返回地址和现场都能找回来。
- 存放当前函数需要但寄存器放不下、或必须有内存地址的局部数据。
- 在寄存器不够用、函数调用约定要求保存寄存器、或需要把参数/返回现场暂存时，提供一块有序的临时空间。
- 上下文切换时，内核只要保存当前线程的 `SP` 和寄存器，就能在恢复时重新接上这条调用链；栈内容本身仍留在内存里，不需要整段复制。

可以用下面这个简化过程理解 `c = a + b`：

```text
1. rip/PC 指向“读取 a”的机器指令
2. CPU 把 a 的值读到寄存器
3. CPU 把 b 的值读到另一个寄存器
4. add 指令在寄存器里完成 a + b
5. 结果写回 c 所在的位置（可能是寄存器，也可能是栈/内存）
```

所以更稳的模型是：

| 概念 | 先怎么理解 |
|---|---|
| 指令指针 `rip` / PC | CPU 下一步要执行哪条机器指令 |
| 栈指针 `rsp` / SP | 当前线程的栈顶地址 |
| 通用寄存器 | CPU 手上正在计算、传参、保存返回值的少量高速格子 |
| 栈 | 保存函数调用现场、局部变量、返回地址和调用链的内存区域 |

如果你从 Java 角度理解过「操作数栈」，要分清层次：Java bytecode 的操作数栈是 JVM 执行模型的一部分；这里讲的 `rsp` / SP 是 CPU/OS 层面的线程栈指针。JVM 最终仍要落到 native thread 的寄存器、机器指令和 OS 栈上执行。

再精确一点:上下文切换发生在**内核态**。用户线程进入内核时,入口代码会把用户态的 `rip`/`rsp`/寄存器保存到内核栈上的 trap frame 里;调度器切换时再保存当前任务的**内核栈指针**到 `task_struct.thread`。下次排到这个任务时,内核先恢复它的内核栈,再一路返回用户态,最后把用户态 `rip`/`rsp` 等寄存器还原。面试或排查时可以先用简化模型:「保存 PC + SP + 寄存器,栈内容留在内存」。

**主动切换 vs 抢占：**

| 切换类型 | 触发条件 | 典型场景 |
|---|---|---|
| 主动切换（voluntary） | 任务主动调 `schedule()`：睡眠/`yield`/等锁 | `read()` 等数据、`mutex_lock()` 被占 |
| 抢占切换（preemptive） | 时间片耗尽、高优先级任务就绪，内核强制切换 | 实时任务抢占、定时器中断触发调度 |

**切换成本主要来自 Cache 冷效应**：寄存器保存/恢复本身 < 1 µs，但切换后新任务的数据不在 L1/L2 缓存里（前一个任务的数据刚把缓存占满），重新 warm up 才是大头——这也是为什么同进程线程切换（不换页表）比跨进程切换代价小。

## 原语五：栈帧（stack frame）

### ① 你视角

Java 里 `StackOverflowError` 是递归太深导致的，Go 里 goroutine 的栈会自动扩容。两者都涉及「函数调用用的那块内存」——这块内存的基本单元就是**栈帧（stack frame）**。

### ② 黑盒内部

**每次函数调用在栈上分配一帧，帧里存局部变量、参数、返回地址。**

函数调用时，硬件指令和编译器生成的函数序言/结尾共同完成（以 x86-64 为例）：

```
调用方执行 call 指令:
  1. 把返回地址（下一条指令地址）push 到栈上
  2. 跳到被调函数起始地址

被调函数 prologue（函数开头）:
  3. push rbp（保存调用方帧基址）
  4. mov rbp, rsp（建立本帧基址）
  5. sub rsp, N（N字节局部变量区域）← 栈向下增长，SP 往低地址移

函数执行结束 epilogue（函数结尾）:
  6. mov rsp, rbp（回收局部变量空间）
  7. pop rbp（还原调用方帧基址）
  8. ret（从栈上取返回地址，跳回调用方）
```

栈在内存里是**向低地址方向增长**的（每次 push 地址减小），与堆的增长方向相反。

#### 栈帧内部到底有什么

一个栈帧可以理解成「某次函数调用在当前线程栈上占用的一小段区域」。不同架构、ABI、编译优化级别会改变细节，但入门先抓住这个简化模型：

```text
高地址
┌────────────────────────────┐
│ 调用方 caller 的栈帧        │
├────────────────────────────┤
│ 返回地址 return address    │  call/blr 后函数返回要跳回哪里
├────────────────────────────┤
│ 上一层帧指针 saved rbp/fp  │  可选；用于串起调用链和调试回溯
├────────────────────────────┤
│ 被调用方要保存的寄存器      │  callee-saved registers
├────────────────────────────┤
│ 局部变量 locals            │  放不进寄存器、或必须有地址的局部数据
├────────────────────────────┤
│ spill 临时区 / 对齐填充     │  寄存器不够时临时写回栈；满足 ABI 对齐要求
├────────────────────────────┤
│ 给下一层函数的栈上传参区    │  参数太多、结构太大、或 ABI 要求时使用
└────────────────────────────┘
低地址  ← rsp/SP 通常在当前有效栈帧的低地址边界附近
```

这张图不要背成固定格式。真实机器码里，编译器会尽量少用栈：

- 能放寄存器的局部变量通常不会进栈。
- 优化编译时可能省略 frame pointer（x86-64 的 `rbp`、aarch64 的 `x29`），直接用 `SP` 加偏移访问栈槽；这会让调试回溯更依赖 unwind 信息。
- 叶子函数（不再调用别的函数）可能完全不建立栈帧，只用寄存器完成工作。
- 函数被内联后，这次调用的栈帧会消失，逻辑上的函数边界不再对应真实栈帧。
- 尾调用优化会让当前函数复用调用方/被调用方的返回路径，栈上看不到普通的「A 调 B 再返回 A」形状。

#### `SP`、frame pointer、返回地址怎么配合

`SP` 是硬件和 ABI 都必须认识的「当前栈顶」。frame pointer（x86-64 常见 `rbp`，aarch64 常见 `x29`）是编译器可选使用的「当前栈帧基准点」。返回地址则告诉 CPU：这个函数结束后跳回哪里。

以 x86-64 的传统帧为例：

```text
call foo
  CPU 把返回地址压到栈上
  CPU 跳到 foo

foo prologue
  push rbp        ; 保存调用方的帧指针
  mov rbp, rsp    ; rbp 成为 foo 这一帧的稳定基准
  sub rsp, N      ; 给局部变量/临时空间留位置

foo epilogue
  mov rsp, rbp    ; 丢弃 foo 的局部区域
  pop rbp         ; 恢复调用方帧指针
  ret             ; 弹出返回地址并跳回调用方
```

aarch64 的形状不完全一样：返回地址通常先在 `x30`（link register）里，函数如果还要调用别的函数，就会把 `x29`/`x30` 成对保存到栈上，避免下一次调用覆盖返回地址。概念不变：**返回地址决定回哪，`SP` 决定当前栈用到哪，frame pointer 帮助稳定地定位这一帧。**

这也是为什么崩溃时能看到 stack trace：调试器/运行时从当前 `PC`、`SP`、frame pointer 或 unwind metadata 出发，一层层还原「谁调用了谁」。如果编译器省略帧指针、做了内联或尾调用，stack trace 就可能和源码调用关系不完全一致。

**栈溢出的两种来源：**

1. **递归无终止（最常见）**：每层递归分配一帧，帧没有回收，栈指针一直往低走，越过栈底边界（`[stack]` 区域边界）→ 访问没有映射的页 → 缺页，但缺页异常处理器发现这不是合法扩展请求 → 发 SIGSEGV（段错误）。这就是 Java `StackOverflowError` / C `Segmentation fault` 的根本来源。
2. **巨大局部变量**：单个函数里 `char buf[1000000]` 这种大数组直接在栈上分配，一帧就能越界。

**每线程一个栈 → 线程数和内存的直接关系：**

每个 OS 线程（或 Go 的 M，即 OS thread）都有独立的栈空间，默认 8 MB（Linux 默认）。1000 个线程 → 8 GB 仅用于栈——这就是「线程池为什么不能无限大」的最直接原因。

Go goroutine 的初始栈只有 2–8 KB，可动态扩展（Go 运行时在堆上分配新栈段，复制旧内容），这是 goroutine 能开几十万个而不耗尽内存的关键。

**Java 对照：JVM stack 和 OS stack**

HotSpot 的普通 Java platform thread 基本对应一个 OS native thread。Java 方法调用产生的 Java frame 通常落在这条 native thread 的 OS stack 上,只是 frame 的格式由 JVM 定义:局部变量表、操作数栈、动态链接、返回地址等。OS/CPU 只看到 native stack、`rsp`/SP、返回地址和寄存器保存;JVM 才知道哪些字节是 Java frame。完整的 Java Heap / Stack 如何映射到 OS 进程虚拟地址空间,见 [`../../performance-tuning-roadmap/04a-java-profiling/01-jvm-memory-model.md`](../../performance-tuning-roadmap/04a-java-profiling/01-jvm-memory-model.md)。

### ③ 砸实

`ulimit -s` 显示当前 shell（及子进程）的栈大小软限制：

```bash
docker run --rm arm64v8/ubuntu:22.04 bash -c 'ulimit -s'
```

真实输出：

```
8192
```

单位 KB，即 **8 MB**。这是 Linux 每个线程的默认栈大小上限（软限制，`setrlimit(RLIMIT_STACK, ...)` 可调）。用 `ulimit -s unlimited` 可以去掉软限制，但物理内存和虚拟地址空间才是真正的上界。

## 原语六：进程 vs 线程（内核视角）

### ① 你视角

Java 里 `Process` 和 `Thread` 是完全不同的类，Go 里有 goroutine 和 `os/exec.Cmd`。在用户态这两者差别很大——但进到 Linux 内核，两者**底层是同一个数据结构**，区别只是「共享了多少」。

### ② 黑盒内部

**Linux 内核里进程和线程都是 `task_struct`，区别来自 `clone()` 的标志位。**

`fork()` 和 `pthread_create()` 的内核路径都经过 `clone()` 系统调用，区别在于传入的标志：

| 操作 | 关键 clone 标志 | 效果 |
|---|---|---|
| `fork()` | 无 CLONE_VM、无 CLONE_FILES | 复制地址空间（写时复制）、复制 fd 表 → 完全独立进程 |
| `pthread_create()` | `CLONE_VM \| CLONE_FILES \| CLONE_SIGHAND \| CLONE_THREAD` | 共享地址空间、共享 fd 表、共享信号处理 → 同进程线程 |

所以「线程」在内核视角就是「共享了地址空间和 fd 表的 task_struct」，没有任何专属的「线程」数据结构。

**内核线程调度看 LWP**：`ps` 显示的 PID 是进程 ID，LWP（Light Weight Process）是内核分配给每个 task_struct 的 ID。同一进程的多个线程有相同 PID 但不同 LWP。

**Go goroutine ≠ 内核线程**：goroutine 是 Go runtime 在用户态实现的 **M:N 调度**——N 个 goroutine（G）映射到 M 个 OS 线程（M），中间由 P（processor）控制可并行执行 Go 代码的额度。`GOMAXPROCS` 控制的是 P 的数量，通常等于 CPU 核数，不是 OS 线程 M 的精确数量。Go runtime 有自己的调度器（`runtime.schedule()`），goroutine 的切换不走内核 `schedule()`，不产生上下文切换的内核代价。Go goroutine 的暂停（如 channel 等待）是 Go runtime 把 goroutine 从 OS 线程上摘下来、换上另一个 goroutine 来跑，OS 层面看到的那个 OS 线程一直在跑，没有切换；如果某个 goroutine 真正阻塞在 syscall 里，runtime 可以把 P 挪给别的 OS 线程继续跑其他 goroutine。

### ③ 砸实

编译一个启动 2 个工作线程的小程序，用 `ps -eLf` 看 LWP（每个 task_struct 对应一行）：

```bash
docker run --rm arm64v8/ubuntu:22.04 bash -c '
apt-get update -qq && apt-get install -y -qq gcc procps >/dev/null 2>&1
cat > /tmp/threads.c <<"EOF"
#include <pthread.h>
#include <unistd.h>
void* w(void* a){ sleep(10); return 0; }
int main(){ pthread_t t1,t2; pthread_create(&t1,0,w,0); pthread_create(&t2,0,w,0); sleep(5); }
EOF
gcc /tmp/threads.c -o /tmp/threads -lpthread
/tmp/threads &
sleep 0.3
ps -eLf | grep -E "^UID|threads" | grep -v grep
'
```

真实输出：

```
UID          PID    PPID     LWP  C NLWP STIME TTY          TIME CMD
root         770       1     770  0    3 07:23 ?        00:00:00 /tmp/threads
root         770       1     772  0    3 07:23 ?        00:00:00 /tmp/threads
root         770       1     773  0    3 07:23 ?        00:00:00 /tmp/threads
```

**看点**：同一个 PID（770），三行 LWP（770=主线程、772/773=两个工作线程）。`NLWP=3` 表示该进程共 3 个线程。内核为每个 task_struct 分配了独立的 LWP ID——这就是「Linux 线程是 clone 出来的 task_struct」的直接证据。

→ **回链**：`linux-handson/03-process-model`（`ps -eLf` 在线上排查「线程泄漏」时的用法；`/proc/[pid]/task/` 目录下每个子目录对应一个 LWP）

|---|---|---|
| syscall | 用户态进内核的唯一合法门，libc 是薄封装 | `write(1,"hi\n",3)=3`，编号 1（x86-64） |

## 面试回答模板：把整章讲成一条链

### 60 秒版

如果面试官问“程序执行时寄存器、栈、cache、syscall、上下文切换怎么串起来”，可以这样答：

> CPU（Central Processing Unit）执行的是机器指令，不是源码行。`PC`（Program Counter）/`rip` 是指令指针，指向下一条机器指令；`SP`（Stack Pointer）/`rsp` 是栈指针，指向当前线程栈顶。指令做计算时主要操作寄存器，真正的加法、比较、位运算由 ALU（Arithmetic Logic Unit，算术逻辑单元）完成。寄存器是 CPU 内部少量最快的工作区；栈是线程虚拟地址空间里的一段内存；L1/L2 cache 是硬件自动维护的内存副本。函数调用时，ABI（Application Binary Interface）规定参数和返回值优先走寄存器，放不下或需要保存现场时才写到栈上，所以寄存器和栈里会出现类似内容，但本质不同。syscall 时，用户态按约定把 syscall 编号和参数放进寄存器，x86-64 执行 `syscall`，aarch64 执行 `svc #0`（Supervisor Call）进内核。内核会保存用户态的 PC、SP 和寄存器现场。上下文切换时也是保存这些 CPU 现场和恢复线索，不复制整条栈；恢复后线程从原来的机器指令继续跑。

### 3 分钟版

更完整的回答可以按时间线展开：

1. **先讲普通指令执行**：源码会被编译成机器指令，CPU（Central Processing Unit）用 `PC`（Program Counter）/`rip` 找到下一条指令。像 `c = a + b` 这种语句，运行时通常不是直接在栈上相加，而是把值装进寄存器，然后 ALU（Arithmetic Logic Unit，算术逻辑单元）在寄存器里完成加法。

2. **再讲寄存器、cache、内存的层次**：寄存器在 CPU 内部，指令能直接点名使用。栈、堆、全局变量都属于进程虚拟地址空间里的内存。CPU 访问内存地址时，硬件会先查 L1/L2/L3 cache，cache 命中就快，没命中才去 DRAM。cache 是内存数据的副本，不是寄存器的副本。

3. **再讲栈和函数调用**：栈不是 CPU 里的结构，而是每个线程的一段内存；`SP`（Stack Pointer）/`rsp` 是栈指针寄存器，它指向当前栈顶。函数调用会形成栈帧，里面可能有返回地址或 link register / `LR`（链接寄存器）的保存值、保存的 frame pointer、保存的寄存器、局部变量、spill 区和栈上传参区。优化后很多变量不会进栈，可能一直在寄存器里。

4. **再讲为什么寄存器和栈内容相似**：ABI（Application Binary Interface）规定函数参数和返回值优先放寄存器，参数太多或结构太大才放栈。寄存器不够时，编译器会 spill 到栈。调用函数前后，为了保护 caller-saved / callee-saved registers，也会把寄存器值暂存到栈。所以内容可能相似，但角色不同：寄存器是执行工作区，栈是调用链和后备存储。

5. **再讲 syscall**：`write(fd, buf, len)` 这种调用最终要进入内核。libc 或运行时会把 syscall 编号和参数放进约定好的寄存器，比如 x86-64 上 `rax` 放编号，`rdi/rsi/rdx` 放前三个参数，然后执行 `syscall` 指令。CPU 切到内核态，跳到内核入口，内核按寄存器取参数，执行 `sys_write`，最后恢复用户态现场返回。

6. **最后讲上下文切换**：线程暂停时，内核保存的是 PC、SP、通用寄存器、必要的 FPU/SIMD 状态，以及内核栈恢复线索。栈、堆、代码段仍在内存里，不会整块复制。恢复线程时，把 PC/SP/寄存器还原，CPU 就能从原来的机器指令继续执行。切换成本不只是保存寄存器，更大的成本常来自 cache/TLB 变冷。

### 容易被追问的点

| 追问 | 回答重点 |
|---|---|
| 栈是在 CPU 里吗？ | 不是。栈是线程地址空间里的内存，`SP/rsp` 是指向栈顶的寄存器。 |
| L1/L2 cache 会保存寄存器吗？ | 不会。cache 保存内存地址对应的数据副本；寄存器是 CPU 显式状态。 |
| 上下文切换会复制整个栈吗？ | 不会。只保存 `SP` 和寄存器等现场；栈内容留在内存里。 |
| 为什么变量有时在寄存器，有时在栈？ | 编译器按优化、寄存器数量、ABI、是否取地址等因素决定。 |
| ALU 是什么？ | CPU 里做算术/逻辑运算的执行单元；加法通常先把值放进寄存器，再由 ALU 算。 |
| `svc #0` 是什么？ | aarch64 上进入内核的指令，和 x86-64 的 `syscall` 是同类动作。 |
| syscall 和普通函数调用最大区别是什么？ | 普通函数调用仍在用户态；syscall/`svc #0` 触发特权级切换，进入内核入口，并保存/恢复用户态现场。 |
