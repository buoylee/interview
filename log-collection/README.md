# 容器化之後,日誌怎麼被收起來(Log Collection on k8s)— 一張地圖

> 面試導向 · **架構師視角** · 容器 / k8s 為主場,**EKS 作貫穿樣例** · 雲中立原則優先
>
> 這個 track 只解決一件事:**應用把日誌吐出來之後,在容器 / k8s 上它是怎麼被收走的 —— 以及這件事的最佳實踐與取捨。**

## 這個 track 落在哪(先認位置)

你 repo 裡有四個相鄰的地方,這份補的是它們之間**沒人管的那層縫**:

```
logging/          ★ log-collection/(這份)        observability/
怎麼「寫」  ───►   容器化後怎麼「收」      ───►    用什麼工具 / 存 / 查
日誌離開程式前        日誌離開程式後、進後端前         日誌進了後端之後
                  ↑ 坐在 cloud-native/ 的 k8s 機制之上
```

`logging/` 教你把日誌**寫對**(寫 stdout、結構化、帶 trace_id),然後它**明確 defer**:收集 / 存 / 查請見 `observability/`。但 `observability/` 是「**用哪個工具、信號怎麼分階段**」的選型視角,對「**在 k8s 上、那行日誌到底怎麼從容器走到後端**」這層機制只給了一行。這個 track 專攻那一層。

## 一個轉變,推出大半最佳實踐

容器化最佳實踐之所以「長那樣」,根子是**一個被逼出來的轉變**:

| | 傳統 VM 時代 | 容器 / k8s 時代 |
|---|---|---|
| 日誌的形態 | **日誌即檔案**:app 寫 `/var/log/app.log` | **日誌即事件流**:app 寫 stdout,平台收 |
| 誰管輪轉 | app 自己 / logrotate | 平台(kubelet)管,app 別碰 |
| 機器 | 長命、固定 IP,你 ssh 上去 `grep` | **短命、唯讀根 fs、會水平擴、不知落在哪台 node** |
| 找日誌 | ssh 進去翻檔 | 進不去也翻不了 —— 必須**先收到集群外** |

→ 這不是風格選擇,是**被環境逼的**:容器短命、又不知道落在哪台 node,你**必須在它消失前把日誌推出容器、收到外面**,否則 pod 一掛 / 一擴你就再也找不到。12-factor App 第 XI 條把這件事定成公理:**程式把日誌當事件流寫到 stdout,不自己管檔案。**

一旦接受這個轉變,大半「最佳實踐」其實是**推論**:寫 stdout、app 內別做 rotation、收集 / 傳輸交給平台、輸出結構化 JSON(讓管線好解析)。

---

## 脊柱(整個 track 的 through-line)

> **容器化把「日誌即檔案」變成「日誌即事件流」。** 一旦內化這個轉變,基本最佳實踐(寫 stdout、node agent 收、app 別管 rotation)就是推論;
> 真正的難點是:**不是所有日誌源都是 app-stdout —— 不同的源處在不同的層,需要不同的收集機制。**

第一句回答「容器化後的最佳實踐」;第二句回答「不同日誌要不要不同收法」。

---

## 真正的難點:日誌源是分層的(`03` 的預告)

「寫 stdout 交給平台」只覆蓋了**應用容器日誌**這一類。一個 k8s 集群裡至少有 **6 類日誌源**,各在不同的層、各有各的收法 —— 這就是你那句「不同日誌不同收法」的答案:

| 日誌源 | 怎麼收(一句話) |
|---|---|
| 應用 stdout/stderr | node agent 讀 `/var/log/containers`(主力) |
| 應用寫**檔案**(legacy / 多檔) | 改 stdout / streaming sidecar / 共享 emptyDir |
| **Node / 系統**(kubelet、containerd、kernel) | agent 另讀 journald,**不同 input** |
| **K8s 控制平面**(apiserver / scheduler / etcd) | **託管**(EKS → CloudWatch),你不自己收 |
| **K8s 審計日誌(audit)** | 控制平面日誌的一種,**獨立管道 + 不可變存儲** |
| **K8s Events**(物件事件) | event-exporter,**不是容器日誌** |

## 一句話總綱(背起來)

> 容器化把**日誌即檔案**變成**日誌即事件流**:app 只負責**產生(寫 stdout)**,平台負責**搬運**。
> 那行日誌離開 app 就成了 **node 上一個 CRI 格式檔案**,pod 一沒就沒 —— 所以**必須**有東西在它被刪前搬到集群外。
> 「那東西放哪、幾個」= **node agent(預設)/ sidecar(例外)/ 直推(別)**;
> 而**不同的日誌源(app / 系統 / 控制平面 / 審計 / events)分在不同的層,需要不同的收法** —— 這才是真正考你的地方。
