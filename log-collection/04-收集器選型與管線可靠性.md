# 04 · 收集器選型與管線可靠性

> `02`/`03` 講了「收集進程放哪、收哪些源」。這章講剩下兩件**架構師真正會被追問**的事:**用哪個收集器**,以及**這條管線在哪裡會丟日誌、怎麼不丟**。
> 前者是選型(別綁死一家),後者是 `01` 埋的兩個坑(輪轉搶刪、buffer)在這裡集中還債。

---

## 一、收集器選型:別把工具當信仰

主流收集器,按「你會在 k8s 上真的遇到」排:

| 收集器 | 語言 | 資源 | 定位 / 何時選 |
|---|---|---|---|
| **Fluent Bit** | C | 極輕(~幾十 MB) | **k8s 日誌 DaemonSet 的事實預設**。輕、快、k8s 原生 filter 完善。先選它。 |
| **OTel Collector** | Go | 中 | **要統一 logs + metrics + traces 一條管線**時。未來方向,AWS 的 ADOT 即它。 |
| **Vector** | Rust | 中(高吞吐) | 要**高性能 + 複雜轉換**(VRL 轉換語言強)。Datadog 出品。 |
| Fluentd | Ruby | 重 | 老牌、插件生態最廣,但重;**正被 Fluent Bit 取代**(很多場景 Fluent Bit 直接夠) |
| Logstash | JVM | 很重 | ELK 的重型轉換層,吃記憶體;輕量收集已交給 Beats / Fluent Bit |
| Promtail / Alloy | Go | 輕 | Loki 專用採集;Promtail 已被 **Grafana Alloy** 取代(Alloy 也能當通用 OTel 管線) |

> 🔬 **生態平衡的一句話**:**Fluent Bit 是當前最穩的默認**(DaemonSet agent 首選);**OTel Collector 是你要把三種信號統一、面向未來**時的選擇;**Vector 是你要高吞吐 / 複雜轉換**時的選擇。三者不互斥 —— 常見組合是 **Fluent Bit 當 node agent + OTel Collector 當中心 gateway**。別讓「我們是 X 廠所以只能用 Y」綁死選型,看的是**輕量採集 vs 統一信號 vs 高吞吐轉換**這三個軸。

## 二、兩層管線:agent + gateway 🔬

小集群可以 agent 直推後端;**規模一上來,標準做法是兩層**:

```
node1 [Fluent Bit DaemonSet] ┐
node2 [Fluent Bit DaemonSet] ┼──► [Gateway:OTel/Vector Deployment ×N] ──► 後端(ES/Loki/CloudWatch)
node3 [Fluent Bit DaemonSet] ┘        (中心聚合層,可水平擴)
   就近採集、輕量                   聚合 / 緩衝 / 路由 / 限流 / 統一出口
```

**為什麼加 gateway**(每條都是面試點):
- **聚合連接數**:100 台 node **直連**後端 = 後端扛 100 條連接 + 100 份認證;經 gateway → 後端只見**幾條**。後端(尤其 ES / 第三方 SaaS)的連接與寫入壓力驟降。
- **緩衝後端抖動**:後端慢 / 掛,gateway 用大 buffer 頂住,node agent 不受影響(否則背壓直接打到 node)。
- **統一路由 / 限流 / 轉換**:分流(`03` 的 `log_type`)、限速、脫敏集中在 gateway 一處,node agent 保持輕量。
- **解耦換後端**:換後端只改 gateway,不動每台 node 的 DaemonSet。

**取捨**:小集群 / 單後端,gateway 是過度設計,agent 直推即可;**多後端 / 大規模 / 後端不穩**才上兩層。

## 三、日誌在哪裡會丟 🔬(這節是這章的命脈)

「日誌收集」聽起來只是搬運,但**搬運鏈上每一跳都可能丟**。把丟失點攤開,你才知道哪些旋鈕在保命:

```
app ─①─► node 檔案 ─②─► agent buffer ─③─► gateway buffer ─④─► 後端
     ①寫太快/輪轉搶刪   ②agent 讀太慢/重啟   ③背壓/積壓        ④後端抖動/拒收
```

1. **輪轉搶先刪(回扣 `01`)**:agent 讀得比 app 寫得慢,老檔案在被讀走前就被 kubelet 輪轉刪掉(預設 `10Mi×5`)→ **無聲丟失**。對策:agent 跟得上 + node 檔案保留足夠大。
2. **agent buffer 滿**:agent 收到後、送出去前先進 buffer。
   - **記憶體 buffer**(`Mem_Buf_Limit`)滿 → 要嘛**背壓**(暫停讀取,但 node 檔案在漲、回到風險 1),要嘛**丟棄**。
   - **檔案系統 buffer**(filesystem buffering)能撐更久(落 node 磁碟),但也有上限、且吃磁碟。
3. **agent 重啟丟未 flush 的**:agent 掛/升級時,記憶體裡還沒送出的就沒了。
   - 🔬 對策:**tail position checkpoint**(Fluent Bit 的 `DB`,SQLite 記到底讀到哪)。重啟後從斷點續讀,**不重讀也不漏讀**。
4. **後端抖動 / 拒收**:後端慢、429、掛。對策:**retry + 退避 + buffer**;retry 耗盡前要有足夠 buffer 頂住。

**at-least-once vs at-most-once**:多數日誌管線是 **at-least-once** —— 寧可**重複**(後端去重 / 容忍),不可**丟**。重複來自「送了但沒收到 ack 就重送」;靠 checkpoint 把「重複」壓到最小。**要 exactly-once 代價極高,日誌一般不追求。**

> 🔬 **背壓(backpressure)的本質取捨,和 `logging/06` 異步 appender 是同一道題**:下游慢了,上游**要嘛阻塞(慢但不丟)要嘛丟棄(快但丟)**。你必須**主動決定「滿了之後丟誰」** —— 通常丟 DEBUG/INFO、保 WARN/ERROR。沒想過這題的管線,真出大流量時就是**隨機丟、丟得最狠的時候(故障時日誌暴漲)恰恰是你最需要日誌的時候**。

## 四、多行堆疊合併(multiline)🔬 —— `01` 的坑在這裡還

一個異常,在你眼裡是「一件事」,在管線裡可能是 **N 條記錄**。有**兩個層次**的「多行」,要分開看:

1. **運行時切的 `P`/`F`(回扣 `01`)**:單行超過 containerd 16KB buffer,被切成多段(前 `P` 後 `F`)。收集器要先**按 tag 把同一行拼回**。
2. **應用本來就是多物理行**:Java stacktrace、Python traceback 本身就是幾十行,每行被當成獨立日誌 → 後端裡你只看到「半個堆疊」,`level` 還可能解析錯。收集器要用 **multiline parser**(規則如「下一行以空白 / `at ` / `Caused by` 開頭就視為續行」)把它們**合成一條**。

> 🔬 **治本不在管線,在源頭**:讓**應用輸出結構化單行 JSON,把整個 stacktrace 塞進一個 `exception` 欄位**(回扣 `logging/04`「異常帶完整堆疊」)。這樣一個異常 = 一行 = 一條記錄,**multiline 問題在源頭就消失了**,管線不用猜行邊界。這正是 `logging/` 一路強調結構化的回報在收集側兌現。沒治本時才在收集器配 multiline —— 但它**跨行邊界容易出錯、耗 CPU**,是下策。

## 五、高基數與標籤爆炸(cardinality)🔬

給日誌打標籤要命的一個坑,尤其用 **Loki**(按 label 建索引)或 ES(欄位過多):

- **把高基數值當 label**:`user_id`、`request_id`、`pod_id`、`trace_id` 這種**取值近乎無限**的維度,如果拿去當索引 label → **索引爆炸、查詢變慢、存儲暴漲、甚至打爆後端**。
- **規則**:**label 只放低基數維度**(`namespace`、`app`、`level`、`env`、`region` —— 取值有限);**高基數信息放進日誌正文 / 欄位**(全文搜得到,但**不進索引**)。

> 🔬 這和 metrics 的高基數戒律是**同一條原理**(Prometheus 把 `user_id` 當 label 也會爆),回扣 `observability/`。一句話:**索引維度要低基數,高基數的東西當「內容」不當「索引鍵」。**

## 六、解析鏈小結:一條日誌進管線後被做了什麼

把前面拼起來,一條日誌從 node 檔案到後端,典型經過:

```
node CRI 檔
  → ① cri parser     剝信封(時間戳/stream/F-P/payload)        ← 01
  → ② json parser    把 payload 解成結構化欄位
  → ③ multiline      把被切 / 多物理行的堆疊拼回一條            ← 01/本章四
  → ④ kubernetes filter  補 pod/namespace/label 元數據          ← 02
  → ⑤ 分流 / 限流 / 脫敏   按 log_type 路由、丟 DEBUG、遮 PII    ← 03 / logging06
  → ⑥ output         送後端(retry + buffer + checkpoint)       ← 本章三
```

面試裡能把這條「parser 鏈」順下來,基本就證明你真配過、不是只會講概念。

---

## 反模式

- ❌ parser 配成 `docker` 但運行時是 containerd → 日誌解不出(`01`)。
- ❌ 沒 checkpoint(`DB`),agent 一重啟就重讀整個檔 / 或漏讀。
- ❌ 把 `request_id` 當 Loki label → 索引爆。✅ 當欄位。
- ❌ 靠管線 multiline 救一切。✅ 源頭輸出結構化單行 JSON(`logging/04`)。
- ❌ 沒想過 buffer 滿丟誰 → 大流量時隨機丟、丟在最需要的時候。
- ❌ 大集群 agent 全直連後端 → 後端被連接數壓垮。✅ 加 gateway。

## 交叉引用

- **輪轉搶刪、CRI 的 `P`/`F`、node 檔案** → `01`
- **agent / gateway 放哪、kubernetes filter 補元數據** → `02`
- **分流(log_type)、不同保留** → `03`
- **EKS 上具體用哪個收集器、Firehose 緩衝** → `05`
- **源頭結構化、異常單行、PII 脫敏、異步丟失取捨** → `logging/04`、`logging/06`
- **高基數同理見於 metrics** → `observability/`

## 本章小結

- 選型三軸:**輕量採集(Fluent Bit,默認)/ 統一信號(OTel Collector)/ 高吞吐轉換(Vector)**;常見組合 Fluent Bit agent + OTel/Vector gateway。別綁死一家。
- **gateway 兩層**:聚合連接、緩衝抖動、統一路由限流、解耦換後端;小集群可省。
- 🔬 **丟失點**:輪轉搶刪 / buffer 滿(背壓 vs 丟棄)/ agent 重啟(靠 checkpoint)/ 後端抖動(retry+buffer)。日誌管線是 **at-least-once**,主動決定「滿了丟誰」(保 ERROR)。
- 🔬 **multiline 治本在源頭**:結構化單行 JSON 把堆疊塞一個欄位(`logging/04`),管線 multiline 是下策。
- 🔬 **高基數當內容不當索引鍵**:label 只放低基數(ns/app/level),`user_id`/`request_id` 進欄位。
