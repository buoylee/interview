# 06 · 壓軸:反思與審計一套現有方案

> 回到你的開場:**「我們服務都在 EKS 上,日誌都走 sidecar 收集 —— 這對嗎?」**
> 這章把前五章變成**一套能套到真實方案上的工具**:9 問審計 checklist → 遷移決策樹 → 怎麼不停機遷 → 最後把整件事**煉成一個 STAR 面試故事**。讀完你不只會答「對不對」,而是能拿著它去**反思你經手過的任何一套日誌收集方案**。

---

## 一、先把問題問對:「sidecar 對嗎」不是 yes/no

`02` 已經給過判準:**Sidecar 是用「N 份成本」換「per-pod 隔離 / 定制」。** 所以「對不對」取決於兩個事實:

1. **你的 app 寫 stdout 還是寫檔案?**(`01`/`03`)
2. **當初為什麼選 sidecar?** 通常是三種動機之一:

| 當初的動機 | 評價 | 怎麼辦 |
|---|---|---|
| (a) app 寫檔案、改不動 stdout | **合理** | 留著(streaming sidecar),或推動改 stdout |
| (b) 多租戶 / 每團隊獨立憑證與後端,真有強隔離需求 | **合理** | 留著 |
| (c) 抄了某個模板 / 沒想過,服務其實寫 stdout 又無隔離 | **可優化** | 算帳 → 遷 DaemonSet |

> **反思的價值在於分清你是哪一種** —— 而不是預設「sidecar 一定錯」。很多人前公司是 (c),但也有不少是 (a)。

---

## 二、審計 checklist:9 問診斷一套現有收集方案

拿這 9 問去審任何一套 k8s 日誌方案(括號是回扣的章):

1. **app 寫 stdout 還是檔案?**(`01`/`03`)→ 若**寫 stdout 還全員 sidecar**,大概率是 (c),優化空間最大。
2. **當初為什麼選 sidecar?隔離需求現在還真實嗎?**(`02`)→ 需求可能早就不在了。
3. **算成本帳:sidecar 記憶體 × pod 數 vs DaemonSet × node 數。**(`02`)
   > 例:200 個 pod,每個 sidecar ~50 MB → **~10 GB** 常駐;同集群 20 台 node 的 DaemonSet → **~1 GB**。**差約 10 倍**,還沒算每個 sidecar 的 CPU、buffer、配置維護。
4. **控制平面 / 審計日誌有沒有單獨收?**(`03`/`05`)→ **最常見的漏項**:只顧應用日誌,EKS control plane logging 根本沒開,出合規 / 排查事件時抓瞎。
5. **node / 系統日誌收了嗎?**(`03`)→ node NotReady、OOM、磁碟滿時有沒有數據?還是只有應用日誌?
6. **多行堆疊處理了嗎?**(`01`/`04`)→ 後端裡一個 Java 異常是**一條**還是**碎成 N 條**?有沒有在源頭結構化?
7. **高基數標籤爆了嗎?**(`04`)→ Loki / ES 查詢慢、貴?是不是把 `request_id`/`pod_id` 當了 label?
8. **丟日誌的點想過嗎?**(`01`/`04`)→ 輪轉搶刪、buffer 滿丟誰、後端抖動有沒有 retry+buffer?故障(日誌暴漲)時還收得住嗎?
9. **成本 / 保留 / PII / 合規?**(`05`/`logging06`)→ access 和 audit 同樣保留期?CloudWatch ingest 量失控?PII 遮了嗎?

> 把這 9 問答一遍,你對「這套方案哪裡對、哪裡是技術債」就有了**可量化**的判斷 —— 這比「我覺得 sidecar 不好」有說服力一百倍。

## 三、遷移決策樹:留 sidecar 還是遷 DaemonSet

```
                  app 寫 stdout 嗎?
                  ├─ 否(寫檔,改不動) ─► 留 streaming sidecar,或推動改 stdout(治本)
                  └─ 是
                       │
              有真實的多租戶 / per-app 隔離需求嗎?
              ├─ 是 ─► 留 sidecar(這就是它存在的理由)
              └─ 否 ─► ★ 遷 DaemonSet(省成本 + 省運維)
                          │
                  這些 pod 在 Fargate 上嗎?
                  ├─ 是 ─► 本來就用 log router,不動(05)
                  └─ 否(EC2)─► Fluent Bit DaemonSet,灰度遷(下一節)
```

**注意**:遷移不是「全 sidecar」換「全 DaemonSet」的二選一 —— 而是**把無隔離需求的大多數遷到 DaemonSet,保留少數真有理由的 sidecar**。混合是常態。

---

## 四、怎麼不停機遷(灰度)🔬

直接拔掉 sidecar 換 DaemonSet,中間可能丟日誌。安全做法是**並行雙收 → 驗證 → 逐步撤**(借 `distribution/zero-downtime-release/` 的灰度思路):

```
階段1  sidecar 照收 ──┐
       DaemonSet 並行收 ┴──► 後端(短期接受重複,at-least-once 本來就容忍,04)
階段2  對比兩條來源:DaemonSet 的數據完整嗎?欄位 / 元數據對嗎?多行堆疊對嗎?
階段3  一個 namespace 先撤 sidecar,監控有沒有缺口
階段4  逐 namespace 推開,撤完;保留 (a)(b) 類真需要 sidecar 的
```

**為什麼這樣**:`04` 說過日誌管線是 **at-least-once**,並行期的重複是可接受代價;**先確認新路徑數據完整,再撤舊路徑**,任何一步發現缺口都能回退。**先撤後驗 = 賭;先驗後撤 = 工程。**

## 五、把它煉成一個面試故事(STAR 骨架)🔬

這正好進你的**故事銀行**(`engineering-leadership/`)。下面是骨架,把數字換成你的真實情況:

**S(情境)**:前司服務跑在 EKS,**所有**服務都掛了 Fluent Bit **sidecar** 收 stdout 日誌,約 N 個服務 / M 個 pod。

**T(任務)**:這套**成本高**(每 pod 一份收集器)且**運維重**(每個 deployment 都要帶 sidecar、升級收集器要動所有 workload)。我要評估**這套是否合理、該不該改**。

**A(行動)**——用上面的 checklist:
- 審計發現 **~90% 服務寫 stdout、無隔離需求**(屬 (c)),只有少數多租戶服務真需要 sidecar;
- **算帳**:M 個 sidecar × 50 MB ≈ X GB 常駐,遷 DaemonSet 後降到 node 數 × 50 MB,**省約 10 倍**;
- 設計 **DaemonSet(aws-for-fluent-bit)+ 灰度遷移**(並行雙收 → 逐 namespace 撤),**保留**少數真有隔離需求的 sidecar;
- 順手**補上之前漏掉的**:EKS control plane logging(尤其 `audit`)、node 系統日誌、event-exporter(`03`/`05` 的漏項)。

**R(結果)**:記憶體省 X GB;運維從「改 N 個 workload」變成「改 1 個 DaemonSet」;**補齊了審計 / 控制平面日誌的合規缺口**;遷移過程零丟失(並行雙收驗證)。

**面試官會追問什麼 + 怎麼接**(列出來,這是資深信號):
- 「**為什麼不直接全 DaemonSet?**」→ 因為少數服務真有多租戶隔離需求(`02`),一刀切會破壞隔離 —— **我是按需求分類,不是換信仰**。
- 「**怎麼保證遷移不丟日誌?**」→ 並行雙收 + 先驗後撤 + at-least-once 容忍重複(`04`)。
- 「**Fargate 上的服務怎麼辦?**」→ 那部分本來就用 log router,不在這次遷移範圍(`05`)。
- 「**怎麼證明省了成本?**」→ sidecar 記憶體 × pod 的實測 + CloudWatch ingest 帳單前後對比。

> 🔬 **誠實提示**:如果你前司其實是 **(a) app 寫檔案才上 sidecar**,別硬編「我把它遷成 DaemonSet」的故事。改成:**「我評估後確認 sidecar 在這裡是對的(因為寫檔 / 隔離),但審計時發現控制平面與審計日誌被漏收了,我補上了那塊。」** —— **反思的價值在「你會系統地審一套方案」這個過程,不在「結論是改還是不改」。** 面試官要看的是你的判斷框架,不是你推翻了什麼。

## 六、面試自檢問答(綜合全 track)

1. 「容器化之後收集日誌的最佳實踐是什麼?」——用「日誌即事件流」這個轉變開頭,一路講到分層收集。(README/`01`/`03`)
2. 「不同日誌要不同收集方式嗎?」——畫 6 類源的分層圖,每層說為什麼不能共用收法。(`03`)
3. 「sidecar 收日誌好不好?」——不是 yes/no:說清它換什麼、什麼時候對、什麼時候是純浪費。(`02`/本章)
4. 「寫 stdout 之後日誌去哪了?」——CRI 檔 → agent → 後端,以及 pod 沒了為什麼就沒了。(`01`)
5. 「日誌會在哪裡丟?」——輪轉搶刪 / buffer 滿 / agent 重啟 / 後端抖動,各自對策。(`04`)
6. 「日誌太貴怎麼辦?」——ingest 量 × 保留 × 單價,從分流 / 差異保留 / 抽樣下手。(`05`)
7. 「你怎麼審一套現有的日誌方案?」——9 問 checklist。(本章二)
8. **終極綜合題**:「給你一個 EKS 集群,從零設計日誌收集」——node agent(EC2)/ log router(Fargate)+ 多 input(容器 + journald)+ control plane logging(含 audit 獨立保留)+ event-exporter + 少數 sidecar,gateway 聚合,分流差異保留控成本。一口氣把全 track 串起來。

## 本章小結

- 「sidecar 對嗎」**不是 yes/no**:看 app 寫 stdout 還是檔、看當初動機 (a)(b) 合理 /(c) 可優化。
- **9 問審計 checklist** 把方案的對與債變成**可量化**判斷(成本帳、漏收項、丟失點、合規)。
- **遷移是分類不是一刀切**:無隔離需求的遷 DaemonSet,真需要的留 sidecar;Fargate 走 log router。
- **不停機遷 = 並行雙收 → 先驗後撤**(at-least-once 容忍重複),不是直接拔換。
- 🔬 **面試故事的價值在「系統審一套方案」的框架,不在結論** —— 寫檔 / 隔離場景就講「我確認 sidecar 對、但補了漏收的審計日誌」。

---

> **全 track 收口(背這段)**:容器化把**日誌即檔案**變成**日誌即事件流** —— app 只負責**寫 stdout**,平台負責**搬運**。那行日誌離開 app 就是 **node 上一個 CRI 檔**,pod 一沒就沒,所以**必須**在被刪前搬到集群外。「搬運的進程放哪」= **node agent(預設)/ sidecar(寫檔·強隔離的例外)/ 直推(別)**;而**不同的源(應用 / 系統 / 控制平面 / 審計 / Events)分在不同層,需要不同收法**,成熟方案是**分層組合**。最後,能拿這套去**反思你經手過的方案**(算帳、找漏收、設計遷移)—— 那就是從「會配」到「能決策」的那一步。
