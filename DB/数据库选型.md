# 数据库选型 — 面试向思维框架

> 关系型 / MongoDB / Elasticsearch 怎么选。核心不是「谁最好」，而是「数据形状 + 访问模式」匹配引擎的取舍。
> 这是一篇可背诵的面试资产；落地一致性的细节见文末「关联阅读」。

---

## 一、先扔掉「哪个 DB 最好」的思路

面试官问选型，**考的不是你背没背过功能表**，而是你会不会**从需求反推**。

- 资浅：「Mongo 灵活、ES 搜索强、MySQL 稳」——功能列表，一追问就崩。
- 资深：反问 **数据长什么样？怎么访问？一致性要多强？多大规模？**

一句立住分水岭的话：

> **关系型是默认。你需要一个「理由」才能不用它，而不是反过来。**

这句话说明你知道：专用 DB 是来解决关系型解决不了的**特定问题**的，不是用来「显得高级」的。

---

## 二、三者的本质（不是功能，是「它到底是个什么东西」）

| | 本质 | 核心能力 | 真实短板 |
|---|---|---|---|
| **关系型**<br>(Postgres/MySQL) | 带 schema 的二维表 + 外键关系 | **事务(ACID)、JOIN、强一致、约束** | 水平扩写难(分库分表是体力活)、schema 改动重 |
| **MongoDB**<br>(文档型) | 一堆 JSON 文档，无固定 schema | **整篇文档读写、schema 灵活、原生分片好扩** | 跨文档事务/JOIN 是它的弱项；"schemaless" 是个陷阱 |
| **Elasticsearch**<br>(倒排索引) | 为「全文检索 + 聚合分析」建的索引引擎 | **模糊/分词/相关性排序(BM25)、海量日志聚合** | **它不是数据源(source of truth)**；近实时、无事务、会丢数据 |

**ES 单独强调一句：它永远是「二级索引」，不是主库。**
数据源头存在 Postgres/Mongo，再**同步**一份进 ES 专做搜索。因为 ES 近实时（默认 ~1s refresh 才可见）、没有真事务、极端情况会丢数据。谁说「订单直接存 ES」，面试官立刻知道他没做过生产。

## 三、决策树：拿到需求先问这 4 句

```text
1. 数据形状?
   实体之间有大量关系(用户-订单-商品)  → 关系型
   自包含的文档,一次读写一整篇         → Mongo
   要按文本内容搜/排序                 → ES

2. 访问模式?
   点查 + 事务(转账、扣库存)          → 关系型
   整篇文档读写、没什么 JOIN           → Mongo
   全文/模糊/相关性/聚合分析           → ES

3. 一致性?
   必须强一致(钱、库存)              → 关系型
   最终一致能接受                     → Mongo/ES

4. 规模与写入?
   写入巨大、要靠加机器扛             → Mongo 分片
   读多 + 要搜索                      → 主库 + ES
```

## 四、真实系统几乎都是「多种混用」(polyglot persistence)

别把它当单选题——这才是成熟度的体现。一个电商搜索场景：

```text
商品 / 订单(要事务、强一致)   → Postgres        ← 系统源头(source of truth)
搜索框(打错字也能搜、带筛选)   → Elasticsearch   ← 从 Postgres 同步过来的索引
购物车 / 会话                  → Redis           ← 缓存
```

「ES vs 关系型」常常是**假对立**。正解是：**Postgres 当源头，ES 当从它同步出来的搜索索引**（CDC / 双写同步）。能说出这句，等于告诉面试官你做过真系统。

---

## 五、2026 年的现实：Postgres 吃掉了很多专用 DB 场景

现代选型的潜台词是「**先问 Postgres 够不够**」：

- `JSONB` → 能干 Mongo 大部分文档型的活（中小规模）
- 内置全文检索 / `pg_trgm` → 能干 ES 的轻量搜索
- `pgvector` → 向量检索（做 RAG 正用得上）

撞到它的真实天花板（全文相关性排序、千万级写入分片）再上专用 DB。一上来就 Mongo+ES 三件套，反而会被问「为什么不先用一个 Postgres？」

## 六、规模上来后：分片如何改变「关系功能」

**关键认知：「分布式」≠「分片」。** JOIN/外键失效的是**水平分片**这一层，不是分布式本身：

```text
单机 Postgres                → JOIN/FK/事务 全在,扛百万级用户没问题
  ↓ 加从库(主从复制)        → 还是分布式! JOIN 照常,只是读扩展 + 从库最终一致
  ↓ 垂直拆分(按业务拆库)    → 跨库 JOIN 没了,但「库内」JOIN 还在
  ↓ 水平分片(sharding-jdbc) → 跨分片 JOIN 才真的没了
```

**绝大多数系统根本走不到最后一步**（单机 + 读写分离扛几百万用户是常态）。真分片了，处理「需要关联」靠这几招（按优先级）：

| 招式 | 做法 | 对应 sharding-jdbc 概念 |
|---|---|---|
| **① 同居** | 选对分片键，让要 JOIN 的表按同一个键分片，落同一分片 → JOIN 照用 | **绑定表 (binding table)** |
| **② 复制小表** | 小且少变的维度表全量复制到每个分片，本地 JOIN | **广播表 (broadcast table)** |
| **③ 冗余字段** | 写时把 `用户名` 拷进订单行，读时不 JOIN（反范式） | — |
| **④ 应用层 join** | 查 A 拿 id，`IN(...)` 批量查 B，内存拼（微服务日常） | — |
| **⑤ CQRS 宽表** | Binlog/CDC → 同步出一张预拼好的扁平读模型 / 灌进 ES | — |
| **⑥ 分布式事务** | 跨分片**写**用 Saga / TCC / outbox（Java 圈 Seata） | — |

**想「鱼和熊掌都要」→ NewSQL**：TiDB / CockroachDB / OceanBase / Spanner，把分布式事务做进引擎，对上层就是「一个能跨节点 JOIN 的大 MySQL」，代价是延迟和运维复杂度。

**Punchline（最值钱的一句）：**
> 分片会干掉 JOIN，这对**所有数据库**都成立——不是关系型独有的弱点。Mongo 分片后 `$lookup` 一样废，ES 压根没有 JOIN。所以「关系型分片后丢了 JOIN」**不能作为转投 NoSQL 的理由**——NoSQL 从来没有 JOIN 给你丢。选型的真正分界线是**「一个分片内部那一坨数据」的形状和事务需求**，不是「大规模下谁还有 JOIN」（都没有）。

## 七、面试话术骨架

被问到时**别直接报答案，按顺序走**：

1. **先反问澄清**：数据形状？访问模式？一致性？规模？——光这步就赢一半
2. **给默认**：「没有特殊理由我先选关系型，事务和强一致是多数业务的底线」
3. **给偏离理由**：「自包含文档、关系少、schema 快速迭代 → Mongo；核心是全文搜索/相关性 → ES，但只作为从主库同步的二级索引」
4. **点出混用**：「实际生产通常是 Postgres 源头 + ES 搜索 + Redis 缓存的组合」
5. **规模追问**：分片后用绑定表/广播表/冗余/应用层 join/CQRS 处理关联；要分布式 ACID 就上 NewSQL

**看场景给方案，而不是给一个万能答案**——这本身就是资深的标志。
