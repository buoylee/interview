# 第 2 章 · 从 docker-compose 到 Kubernetes

> 🔬 单机跑几个容器,`docker-compose` 就够了。但生产要跑在**多台机器**、要**自愈**、要**扩缩容**、要**滚动发布**——compose 全做不到,这正是 k8s 出场的理由。这章讲清「单机编排的天花板」和「k8s 到底解决什么」,让你回答「为什么用 k8s」时是讲清了需求,而不是「因为流行」。

---

## Part A · docker-compose:单机多容器编排

一个 `docker-compose.yml` 描述多个容器(服务/网络/卷),一条 `docker compose up` 全起来:

```yaml
services:
  web:   { build: ., ports: ["8080:8080"], depends_on: [db, redis] }
  db:    { image: postgres:16, volumes: ["pgdata:/var/lib/postgresql/data"] }
  redis: { image: redis:7 }
```

**适合**:本地开发、单机小部署、CI 里拉起依赖。**它的边界:只管一台机器。**

---

## Part B · 单机编排的天花板 🔬

生产要的这些,compose 都给不了:

| 生产需求 | compose | 对应 system-design |
|---|---|---|
| 跨多台机器调度 | ❌ 只管一台 | — |
| **自愈**(容器/机器挂了自动重建到别处) | ❌ | L4 可用性 |
| **扩缩容**(按负载加减实例) | ❌ 手动 | L3 扩展 |
| **滚动发布 / 回滚** | ❌ | L6 发布 |
| 跨机**服务发现 + 负载均衡** | ❌ | L1 |
| 配置/密钥统一管理 | 弱 | L1/L7 |

> 一句话:compose 是「**单机**的 docker 批处理」,生产要的是「**集群级**的自愈 + 编排」——这中间的鸿沟就是 k8s 填的。

---

## Part C · Kubernetes 解决什么 🔬

> **k8s = 容器编排系统**:把「一堆容器跑在一堆机器上」管成一个**声明式、自愈、可扩缩、可发布**的集群。

| 能力 | k8s 怎么给 |
|---|---|
| **自愈** | Pod 挂了自动重建,节点挂了重新调度到别的节点 |
| **扩缩容** | 改 `replicas`,或 HPA 按指标自动伸缩(第 4 章) |
| **滚动发布/回滚** | Deployment 内建(第 3 章 + L6) |
| **服务发现 + LB** | Service 给稳定 IP/DNS(第 5 章 + L1) |
| **配置/密钥** | ConfigMap/Secret(第 7 章) |
| **调度** | Scheduler 决定容器放哪台(第 4 章) |

**灵魂:声明式(declarative)** 🔬:你只描述「**想要的状态**」(我要 3 个副本、用这个镜像),k8s 持续让现实**趋近**它;而不是命令式地「你一步步去启动、去重启」。这就是它能自愈的根本——下一章详讲控制器 reconcile。

---

## Part D · 什么时候不该上 k8s(架构师视角)

回扣选型思想(「用对工具 vs 少一个依赖」):k8s **复杂度高、运维重**。

- **别上**:单机能跑、团队没人懂 k8s、小项目——硬上 k8s 是给自己挖坑。
- **替代**:compose(单机)、Nomad(更轻)、云托管(Cloud Run / ECS Fargate,免运维控制面)。
- **值回票价的时机**:**规模和复杂度上来了**(多服务、多机、要自愈和弹性),k8s 才划算。

## 本章小结

- **compose** = 单机多容器一键起,适合开发/CI;**天花板是「只管一台机器」**。
- 生产要的**跨机调度、自愈、扩缩、滚动发布、服务发现**,compose 都没有——这就是 k8s 的价值。
- **k8s = 声明式 + 自愈 + 可扩缩 + 可发布**的集群编排;灵魂是「描述期望状态,系统持续趋近」。
- **别盲目上 k8s**:复杂度高,规模/复杂度到了才值得;小场景用 compose/Nomad/云托管。
- 下一章:`03` k8s 架构与核心对象——控制面组件 + 一个 Pod 的诞生 + 声明式 reconcile。
