# 分库分表与扩展性

## 1. 核心问题

单库单表到了某个规模就会遇到三面墙：存储墙（磁盘放不下）、读墙（QPS 超单机上限）、写墙（写入锁竞争 + binlog 复制延迟）。本章解决四件事：

**(a)** 什么时候该分、分到什么粒度；
**(b)** 分片策略怎么选、分片键怎么挑；
**(c)** 分完之后的副作用（全局 ID、跨库 JOIN、跨库事务、分页）怎么处理；
**(d)** 已有数百亿行的库，怎么做在线不停服迁移。

---

## 2. 直觉理解

把数据库想象成图书馆：

> **初始**：一个房间、一排书架，图书管理员一个人搞定。
>
> **书太多**：书架放不下 → 加书架、把书分类放（**分表**）：科技类第 1-100 号书架、文学类 101-200 号。
>
> **读者太多**：一个管理员排队严重 → 多开分馆，每个分馆服务不同的读者（**分库**）：西城区分馆、东城区分馆。
>
> **书太多 + 读者太多**：每个分馆里还有多排书架（**分库 + 分表**）。

分表解决的是**单表行数过多**（B+ 树太高、统计慢）；分库解决的是**单机资源瓶颈**（CPU、连接数、磁盘 IO）。两件事可以独立做，也可以同时做。

---

## 3. 原理深入

### 3.1 何时该分（三条线 + 具体数字）

**线 1：存储/容量**

- 单表行数 > **5000 万**：InnoDB B+ 树通常三层，再往上加一层树高，每次查询磁盘 IO 从 3 次变 4 次。
- 单表数据文件（`.ibd`）> **50 GB**：DDL（加列、加索引）时间以小时计，Online DDL 期间对写压力明显。
- 单机磁盘利用率 > **70%**：留余量不足，写放大 + page merge 频繁。

**线 2：读 QPS**

- 单实例读 QPS 超 **1 万~3 万**（取决于是否有覆盖索引、内存是否够 buffer pool）：慢查开始拖累 P99。
- 只读从库负载 > 80%：从库追主延迟增大 → 读从结果可能过期。

**线 3：写瓶颈**

- 单实例写 QPS 超 **5000~8000**：row-level locking 热点、binlog 写盘成为瓶颈。
- 主从延迟持续 > **1 秒**：从库 SQL 线程积压，主库 semi-sync 模式下写入也开始阻塞。

> 经验值：单表 ≤ 2000 万行、单库 ≤ 1 TB、写 QPS ≤ 5000 时，先做索引优化、读写分离、缓存前置，不要急着分。**能不分就不分**——分库分表引入的复杂度是线性的，运维成本是指数的。

---

### 3.2 分库 / 分表 / 都分 三种组合

| 组合 | 解决什么 | 什么时候用 | 副作用 |
|---|---|---|---|
| **只分表**（垂直/水平） | 单表行数过多 → 降树高、加速统计 | 表大但单机资源还够 | 连接数、CPU 还在一台机上 |
| **只分库**（垂直拆库） | 把不同业务模块的表拆到独立实例 | 不同业务耦合太重，连接数告警 | 跨库 JOIN 消失，事务跨库变复杂 |
| **分库分表**（水平切分） | 单表行数 + 单机资源都到瓶颈 | 大促高并发、用户/订单亿级 | 全局 ID、跨库 JOIN、分布式事务、分页全中 |

**垂直分表** vs **水平分表**：

- 垂直分表：把一张「胖表」按列拆成两张表，冷列（如大 TEXT/BLOB、不常查的）挪到扩展表，热列留主表。索引更小、buffer pool 命中更高。
- 水平分表：同一张表的行按某个规则打散到 N 张表（`orders_00`~`orders_31`），每张表结构相同。行数分摊，适合超大单表。

### 3.3 分片策略四种

#### Hash 分片

把分片键（如 `user_id`）取模：`shard = user_id % N`。

- **优点**：数据分布均匀，热点少。
- **缺点**：扩容时 N 变了，几乎所有行都要迁移（一致性 Hash 缓解这个问题，见下）。
- **适用**：用户表、订单表等按用户维度均匀查询的场景。

```sql
-- 32 张分表，user_id = 123456
shard_index = 123456 % 32 = 16  →  orders_16
```

#### Range 分片

按分片键的值域范围划分：如 `user_id 1~1000000 → shard_0`、`1000001~2000000 → shard_1`。

- **优点**：扩容只需加新分片，不迁移老数据；范围查询可以定位到少数分片。
- **缺点**：分布不均匀——早期注册的用户 ID 小，全在 shard_0；如果按时间分片，最新分片是「热点写」。
- **适用**：按时间归档（日志表按月/年）、按地理区域（城市编码）。

```sql
-- 按下单时间分表
orders_202501  ← 2025-01-01 ~ 2025-01-31
orders_202502  ← 2025-02-01 ~ 2025-02-28
```

#### List 分片

把枚举值映射到指定分片：如大区 `city_code IN ('BJ','TJ') → shard_north`。

- **优点**：语义清晰，可按业务边界切分。
- **缺点**：枚举值改变时需要重新配置路由；枚举值分布不均匀时分片大小差异大。
- **适用**：多租户（按租户 ID 分片）、地理多区（按大区归并）。

#### 一致性 Hash

把分片键 hash 到一个虚拟环（0~2^32），每个物理节点负责一段弧。扩容时加新节点只需迁移相邻弧段上的数据，迁移量从「全量迁移」降到约 `1/N`。

- **缺点**：实现复杂；若虚拟节点（vnodes）数量不够，分布仍然倾斜；需要中间件或客户端库支持。
- **适用**：分布式缓存（Redis Cluster 内部即用此思想）；MySQL 场景下通常由 ShardingSphere 或 Vitess 托管逻辑。

```
虚拟环 0 ─────────────────────────── 2^32
         ↑shard_0  ↑shard_1  ↑shard_2
新增 shard_3 时只需迁移它「前驱节点」到它之间那段数据
```

### 3.4 分片键选择三原则

选错分片键是事后最痛苦的事，基本上只能重新迁移。

**原则 1：高频访问列**

业务最常用的查询 WHERE 条件是什么？分片键必须在里面，否则每次查询都要广播到所有分片（scatter-gather），性能等于没分。

```sql
-- 电商订单，最常查询：SELECT * FROM orders WHERE user_id = ?
-- → 分片键选 user_id，每次查询定位单个分片
-- 如果选 order_id 做分片键，查「某用户的所有订单」就要 scatter 32 个分片
```

**原则 2：分布均匀**

分片键的值域要足够分散，避免热点分片。自增 ID、UUID、高基数枚举（如城市）都行；性别、状态这类低基数字段绝对不行。

**原则 3：不变（或极少变）**

分片键变更意味着行要从一个分片搬到另一个分片——UPDATE 变成 DELETE + INSERT，且跨分片事务。用户手机号可能换、邮箱可能换，`user_id`（自增或雪花）通常不变。

> **复合分片键**（如 `(tenant_id, user_id)`）：SaaS 多租户场景常见，先按 tenant 路由到库，再按 user_id 路由到表。缺点是两维都要带入查询。

### 3.5 全局 ID 四方案对比

分库分表后，不同分片上的表不能再用各自的 `AUTO_INCREMENT`（会重复），必须有全局唯一 ID 生成机制。

#### 方案对比表

| 方案 | 优点 | 缺点 | 适用 |
|---|---|---|---|
| **UUID** | 无中心化、无网络调用、实现零成本 | 36 字节字符串；随机插入导致 B+ 树页分裂频繁；不含时间信息无法排序 | 低频小表；不做主键，做业务流水号 |
| **Snowflake（雪花）** | 64 位整数、有序、高性能（单节点 400 万 ID/s）、含时间信息 | 依赖系统时钟（时钟回拨导致重复）；机器 ID 需要注册管理（Zookeeper/etcd） | 主流选择：订单、用户、消息 |
| **号段模式（Leaf-segment）** | 实现简单；批量取段减少数据库压力；可高可用（双机房双 buffer） | 重启或扩容时当前号段剩余 ID 浪费；步长设不好造成 ID 稀疏 | 中小规模；内部系统 |
| **数据库自增（步长模式）** | 无需额外组件 | N 个实例各自起点 1/2/…/N、步长 N，扩缩容要停服重新配步长；单点压力仍在 | 小规模测试；不推荐生产 |

#### Snowflake 64 位分配详解

```
 63       62      21    11    0
  │        │       │     │    │
  0   │   41 位   │ 10 │ 12  │
 符号  毫秒时间戳   机器ID  序列号
 (固定=0)
```

- **1 位符号位**：固定 0，保证 ID 为正数。
- **41 位毫秒时间戳**：可表示 `2^41 - 1 = 2199023255551` 毫秒 ≈ **69.7 年**（从自定义 epoch 起算，通常设为 2020-01-01，能用到 2089 年）。
- **10 位机器 ID**：可部署最多 `2^10 = 1024` 个节点（可拆成 5 位 datacenter + 5 位 worker）。
- **12 位序列号**：同一毫秒同一机器最多生成 `2^12 = 4096` 个 ID；超过则等到下一毫秒。

> **时钟回拨问题**：NTP 校时可能让系统时间倒退几毫秒。解决方案：(a) 检测到回拨时拒绝生成，等时钟走回来（等待几毫秒可接受）；(b) 用逻辑时钟 + 上次生成时间的最大值；(c) 美团 Leaf 的做法：向 ZooKeeper 汇报上次时间，启动时校验防止大幅回拨。

#### 号段模式（Leaf-segment）工作原理

```sql
-- 号段表
CREATE TABLE id_alloc (
  biz_tag    VARCHAR(64) NOT NULL,
  max_id     BIGINT      NOT NULL DEFAULT 0,  -- 已分配到哪
  step       INT         NOT NULL DEFAULT 1000,  -- 每次取多少
  version    INT         NOT NULL DEFAULT 0,
  PRIMARY KEY (biz_tag)
);

-- 取一段号（原子 UPDATE + SELECT）
UPDATE id_alloc SET max_id = max_id + step, version = version + 1
WHERE biz_tag = 'order' AND version = #{version};
SELECT max_id, step FROM id_alloc WHERE biz_tag = 'order';
-- 本次可用范围：[max_id - step + 1, max_id]
-- 用完再来取下一段
```

双 buffer 优化：快用完当前号段时，提前异步取下一段，消除取号等待。

### 3.6 中间件方案 vs 客户端方案

#### 中间件方案（Proxy 层）

```
应用 → [ShardingSphere-Proxy / MyCAT] → 多个 MySQL 实例
```

- **ShardingSphere-Proxy**：以 MySQL 协议暴露，应用不改代码，仅改连接串。支持分片、读写分离、数据加密、影子库（压测）。
- **MyCAT**：老牌中间件，配置复杂，社区活跃度下降，新项目不推荐。

优点：应用无感知，语言无关（Java/Go/Python 都行）。
缺点：多一跳网络延迟（~1ms）；Proxy 本身是新单点，需要高可用部署（两台 + VIP）；SQL 兼容性问题（某些复杂 SQL 中间件不支持）。

#### 客户端方案（SDK 层）

```
应用（内嵌 ShardingSphere-JDBC / Sequelize 插件）→ 多个 MySQL 实例
```

- **ShardingSphere-JDBC**：Java JDBC 实现，内嵌到应用 jar，无额外网络跳。
- **GORM Sharding 插件**（Go）：中间件语义，自动路由，支持 hash/range。

优点：无额外网络跳，延迟更低；不需要额外部署维护 Proxy。
缺点：只支持特定语言/框架；升级时每个服务都要改依赖。

> **选择建议**：微服务异构（多语言）→ Proxy 方案；Java 单体或纯 Java 微服务 → JDBC 客户端方案。

### 3.7 跨库 JOIN 三种处理

分库后，两张表可能在不同实例上，`SELECT a.*, b.* FROM orders a JOIN users b ON a.user_id = b.id` 直接炸。

#### 方案 A：应用层多次查询 + 内存 JOIN

```
1. SELECT * FROM orders WHERE user_id = ?  →  shard_3
2. 取出 order 列表，提取所有 user_id
3. SELECT * FROM users WHERE id IN (id1, id2, ...)  →  可能多个 shard
4. 在应用代码里 Map 对应合并
```

- **适用**：关联数据量小（IN 列表 ≤ 1000 条）；两次查询各自走索引。
- **不适用**：需要 JOIN 后再 ORDER BY / GROUP BY / HAVING；数据量大时内存和网络开销不可控。

#### 方案 B：全局表（广播表）

小字典表（城市表、类目表、配置表，通常 ≤ 10 万行）在**每个分片上冗余一份完整副本**。

```
orders 在 shard_0~shard_31，category 表在每个 shard 上都有 → 可以本地 JOIN
```

- 写入时同步写所有分片（由中间件代理透明处理，或应用层广播）。
- **适用**：稳定的小字典表；更新频率低。
- **不适用**：频繁变更的表（每次写都要 32 次 INSERT/UPDATE）。

#### 方案 C：ER 分片（关联表同键分片）

把关联表用**同一个分片键**做水平切分，保证有关联关系的数据落在同一个分片上，就可以做本地 JOIN。

```
orders 按 user_id % 32 分片
order_items 也按 user_id % 32 分片（而不是按 order_id）
→ 同一个 user 的 orders + order_items 一定在同一个 shard → 可以 JOIN
```

- **适用**：有明确父子关系、且总是通过父键查子表的场景（订单-订单明细、用户-用户地址）。
- **不适用**：无法共享分片键的表（如 users JOIN products，两者维度完全不同）。

### 3.8 跨库事务方案概述

分库后，一个业务操作跨了两个 MySQL 实例，单机 ACID 事务失效。常见方案：

| 方案 | 一致性 | 性能 | 实现复杂度 | 适用 |
|---|---|---|---|---|
| **XA（2PC）** | 强一致 | 低（阻塞协调） | 中 | 低并发、金融核心 |
| **本地消息表** | 最终一致 | 高 | 低 | 主流业务场景 |
| **TCC** | 最终一致（补偿） | 中 | 高 | 需要明确 Try/Confirm/Cancel 语义 |
| **Saga** | 最终一致（正向+补偿） | 中 | 中~高 | 长事务（多步骤业务流程） |

> 本章只作概述，分布式事务完整原理和实战见 **`11-distributed-transactions`** 章。

### 3.9 在线迁移 SOP

不停服把单库单表迁移到分库分表，核心是**双写 + 校验 + 渐进切流量**，分五个阶段。

#### 整体时序

```
阶段 1：存量数据迁移（全量同步）
──────────────────────────────────────────────────────────
  旧库（单表） ──全量导出──→ 新分片集群

阶段 2：双写开启（存量迁完后）
──────────────────────────────────────────────────────────
  写请求 ──同时写──→ 旧库（主）
                └──→ 新分片集群（影子写）

  增量数据追平：Canal/Maxwell 监听旧库 binlog，
               把旧库增量同步到新分片集群

阶段 3：数据校验
──────────────────────────────────────────────────────────
  抽样或全量对比 旧库 vs 新分片集群
  diff 工具确认：行数一致、核心字段一致

阶段 4：切读流量（灰度）
──────────────────────────────────────────────────────────
  按 user_id % 100 切 1% 读流量 → 新分片集群
  逐步放量：1% → 5% → 20% → 50% → 100%
  观察延迟、错误率

阶段 5：切写流量 + 旧库降级
──────────────────────────────────────────────────────────
  全量写请求切到新分片集群
  旧库保持只读，保留 72 小时作为回滚后备
  72 小时无异常 → 关闭旧库
```

#### 双写的注意事项

- **写顺序**：先写旧库（已有事务保障），成功后异步写新分片；旧库失败则整体失败，不写新分片。
- **不一致窗口**：双写期间新分片可能短暂滞后，所以阶段 4 切读时要先观察，不能直接切 100%。
- **校验策略**：行数一致只是第一步，还要对核心字段（金额、状态）做 checksum 校验，发现不一致要补写。

#### 回滚预案

```
读流量回滚：把路由配置改回指向旧库（配置中心热更新，秒级生效）
写流量回滚：切回旧库写，停止双写，新分片只接增量追追 binlog
数据不一致：旧库 binlog 全程保留，可重放到任意时间点
```

## 4. 日常开发应用

**建表 / 设计阶段**

- 提前想好分片键：用户体系选 `user_id`，订单体系选 `user_id`（以查询维度为准，不要选 `order_id`）。
- 主键用 Snowflake ID，不要 `AUTO_INCREMENT`（分库后重复）。
- 小字典表（类目、城市、配置）建成全局表，交给中间件广播写。
- 关联表（订单-明细）设计成 ER 分片，确保父子数据同分片。

**写 SQL 时**

- 每条 SQL 的 WHERE 里**必须带分片键**，否则 scatter-gather 到所有分片，效果比没分更差。
- 分页查询（LIMIT offset, size）在多分片下代价极高，见 §5 Case B。
- 跨库 JOIN 走方案 A/B/C，不要直接写跨库 JOIN 语法丢给中间件解析。
- 批量写入时，按分片键预先分组，减少跨分片事务。

**上线前**

- 用 `EXPLAIN` / ShardingSphere 的 SQL 解析日志确认路由是否正确（是否定向到单分片）。
- 压测时模拟真实分片分布，不要只压单分片。

## 5. 调优实战

### Case A：分片键选错了导致热点

**症状**：32 个分片里有 2 个 CPU 100%，其余空闲。慢查日志里大量出现在同两个实例。

**根因**：业务按 `city_id` 分片，但用户集中在北京（`city_id=1`）和上海（`city_id=2`），80% 的流量打到这两个分片。

**诊断**：
```sql
-- 在中间件层统计各分片的 QPS
SHOW STATUS LIKE 'Com_select';  -- 在每个分片上分别执行，对比差异
-- 或者用 ShardingSphere 的 metrics 接口
```

**解决**：
1. 短期：把热点分片（city_id=1/2）拆成多个子分片，配置中间件做二次 hash。
2. 根本：更换分片键为 `user_id`（高基数、均匀分布）。这需要在线迁移（见 §3.9）。
3. 如果无法迁移：在热点分片前加 Redis 缓存层，降低直达 MySQL 的请求量。

---

### Case B：分页 LIMIT 100000, 20 在 32 个分片下

**问题**：单库时 `LIMIT 100000, 20` 已经慢（要扫 100020 行）；分 32 个分片后，每个分片都要返回前 100020 行给协调节点，协调节点合并排序后取第 100001~100020 条。

**代价**：32 × 100020 行 = 320 万行网络传输 + 内存排序。

**方案一：禁止深翻页，改用游标分页（推荐）**

```sql
-- 第一页
SELECT * FROM orders WHERE user_id = ? ORDER BY id ASC LIMIT 20;
-- 记录最后一条 id = 1234567

-- 第二页（带游标）
SELECT * FROM orders WHERE user_id = ? AND id > 1234567 ORDER BY id ASC LIMIT 20;
```

游标分页的前提是有稳定的排序列（通常是 Snowflake ID，天然有序）。

**方案二：搜索引擎兜底（Elasticsearch）**

把需要深翻页的查询（如用户消息列表、全站搜索）的索引同步到 ES，ES 做分页排序，只返回主键列表，再回 MySQL 按主键批量取数据。

**方案三：中间件协调（分片有序归并）**

ShardingSphere 等中间件支持 `ORDER BY + LIMIT` 的跨分片归并：每个分片取前 `offset + size` 条，协调节点归并取最终 `size` 条。但代价仍是 O(N × (offset + size))，不根本解决深翻页问题。

### Case C：在线迁移 100 亿行历史数据

**场景**：订单表 100 亿行，单库 8 TB，需要在不停服的前提下迁到 32 分片。

**关键约束**：
- 业务不能停（24/7 在线电商）
- 迁移期间写入继续
- 允许最终一致（迁移窗口内短暂延迟可接受）

**具体步骤**：

**Step 1：准备新分片集群**（1 周）
- 搭建 32 个 MySQL 实例（或用 RDS 托管），验证连通性
- 在 ShardingSphere 配置分片规则（`user_id % 32`），但不切流量

**Step 2：全量数据导入**（3~7 天，视带宽）

```bash
# 用 mydumper 并行导出，按 user_id 范围分块
mydumper \
  --host old-db --user root --password xxx \
  --database orders --tables-list orders \
  --where "user_id BETWEEN 1 AND 100000000" \
  --outputdir /data/dump/chunk_01 \
  --threads 8

# 每个 dump chunk 按分片键路由，写入对应分片
# 可用 Canal + 自定义 consumer 或 DataX 做路由写入
```

使用 DataX（阿里开源）的 reader/writer 插件，支持并行分片写入，速度约 **5000 万行/小时**（视磁盘 IO）。100 亿行约需 200 小时 ≈ 8 天。

**Step 3：开启 CDC 增量同步**（全量导完后立即开启）

```
Canal Server 监听旧库 binlog
  → 解析 INSERT/UPDATE/DELETE
  → 按 user_id % 32 路由
  → 写入对应新分片
```

**Step 4：双写开启**（Canal 追平延迟 < 1 秒后）

应用层改为双写模式（配置中心热更新）：

```java
// 伪代码
if (featureFlag.isEnabled("double_write")) {
    oldDb.write(order);           // 主写
    newShards.write(order);       // 影子写（异步，失败告警不影响主链路）
}
```

**Step 5：数据校验**（并行进行，不阻塞其他步骤）

```bash
# pt-table-checksum 对比旧库和新分片数据一致性
pt-table-checksum --host old-db \
  --databases orders --tables orders \
  --chunk-size 10000

# 自定义脚本：对比核心字段的 SUM（如 amount、count）
SELECT COUNT(*), SUM(amount) FROM orders WHERE created_at BETWEEN ? AND ?;
-- 在旧库和每个新分片分别执行，对比汇总值
```

**Step 6：灰度切流量**（1~2 周）

```
Day 1: 读流量 1% → 新分片（观察 P99 延迟、错误率）
Day 3: 读流量 10%
Day 7: 读流量 50%
Day 10: 读流量 100%
Day 14: 写流量切到新分片，旧库降为只读备份
Day 21: 确认无异常，下线旧库
```

**可能踩的坑**：

- **Canal 追不上**：旧库写 QPS 高时 Canal 延迟增大 → 在低峰期开始全量导入，Canal 追平窗口选在凌晨。
- **新分片写入慢**：全量导入期间关闭新分片的二级索引，导完再重建（`ALTER TABLE ... DISABLE KEYS`）。
- **双写不一致**：记录双写失败的 `(shard_id, primary_key, timestamp)` 到补偿表，异步重试。
- **回滚**：旧库只读保留 30 天，配置中心随时可以切回。

## 6. 面试高频考点

### 必考对比

| 维度 | Hash 分片 | Range 分片 |
|---|---|---|
| 数据分布 | 均匀 | 可能热点（时间序列尤其明显） |
| 扩容代价 | 高（一致性 Hash 可降低） | 低（只加新分片）|
| 范围查询 | 需 scatter 所有分片 | 可定位到少数分片 |
| 适用场景 | 用户/订单（均匀查询） | 日志/归档（按时间查） |

### "分库分表之后主键怎么保证唯一" — 三句话答法

1. 每个分片不能各自用 `AUTO_INCREMENT`，因为不同分片会生成相同数字。
2. 主流方案是 Snowflake：64 位整数、有序、机器 ID 保证全局唯一、单节点 400 万/s。
3. 次选号段模式：从中心数据库批量取号段（默认 1000 个），减少中心数据库压力，重启浪费最多 1000 个 ID 可接受。

### "在线迁移不停服怎么做" — 五步回答

双写 → 增量同步（Canal binlog）→ 数据校验（pt-table-checksum）→ 灰度切读流量 → 切写流量 + 旧库只读保留。

### "跨库 JOIN 怎么办" — 三种方案

(a) 应用层多次查询 + 内存 JOIN（小数据量）；(b) 全局表广播冗余（小字典表）；(c) ER 分片（关联表用同一分片键）。

### 易错点

- **选了低基数字段做分片键**（如 status、gender）：取模后所有数据只分到 2~5 个分片，相当于没分。
- **忘记全局 ID**：分完表后发现 ID 冲突，事后改代价极高。
- **分片键和索引设计分离考虑**：分片键必须是查询最常用的 WHERE 条件列，否则每次查询都是 scatter-gather。
- **认为中间件能解决所有问题**：`ORDER BY col LIMIT 100000, 20` 丢给中间件只是把问题放大了，根本解法是游标分页或搜索引擎。
- **忽略事务范围**：分库后单机事务范围缩小，跨库操作要意识到一致性级别降到了最终一致，对账/补偿逻辑不能省。

---

## 7. 一句话总结

分库分表是**用复杂度换容量和吞吐**：单表 > 5000 万行、写 QPS > 5000 才应该考虑；分片键选高频 + 均匀 + 不变的列；全局 ID 首选 Snowflake（64 位、有序、4096 ID/ms/节点）；跨库 JOIN 用应用层拼装 / 全局表 / ER 分片三种手段；在线迁移走双写-增量同步-校验-灰度切流量五步 SOP，旧库只读保留 30 天作回滚预案。
