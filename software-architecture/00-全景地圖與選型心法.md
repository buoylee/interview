# 第 0 章 · 全景地圖與選型心法

> 🔬 這章不教任何單一模式,教**整張地圖**:DDD、Clean、Hexagonal、CQRS、Event Sourcing……這些詞分別站在哪個高度、彼此什麼關係、什麼時候該用、什麼時候用了就是過度設計。
>
> **最常見的露怯**:把這些當成「越多越高級」的技術選型,新項目一上來就 DDD 全套 + CQRS + Event Sourcing,結果一個本質是 CRUD 的後台被搞成沒人維護得動的怪物。資深的標誌相反——**知道什麼時候「不上」**。這章先把判斷框架給你,後面每章才填細節。

---

## Part A · 為什麼有這條軸:你天天在做,只是沒有名字

先把兩條軸分清,否則你會一直把這條軸的問題拿去 `system-design/` 找答案(找不到):

| | `system-design/`(那條軸) | **本主線(這條軸)** |
|---|---|---|
| 在問 | 系統**運行時**怎麼不崩 | 一個服務**裡面** code 和業務怎麼組織 |
| 關鍵詞 | CAP、共識、複製、容災、限流 | DDD、限界上下文、分層、聚合、CQRS |
| 壞掉的樣子 | 半夜宕機、數據不一致、雪崩 | 改一個需求要動五個檔、沒人說得清某個 service 管到哪 |
| 誰最痛 | SRE / 運維 | **寫業務的你自己** |

**潑個現實**:你寫了 5 年 Java/Go,這條軸的事其實天天在做——分 controller/service/dao 是「分層架構」,寫 `interface OrderRepository` 是「Repository 模式」,`OrderService.createOrder()` 把規則全塞進去是「Transaction Script」。**你不是不會,是做得無意識**:框架擺好位子你就往裡填,從沒問過「為什麼是這個位子、換個位子代價是什麼」。

> 這條軸要把你從「**會用**」升級到「**講得出 why 和取捨**」。面試問「你怎麼組織業務邏輯」,答「Spring 三層」是工程師,答「這塊是貧血+事務腳本因為它就是 CRUD;那塊核心計費我用充血模型把不變量守在聚合裡」——這是資深。

## Part B · 四個高度(全書骨架,務必記住)

把所有概念按「**它在回答哪個問題**」分到四個高度。拿一個電商例子貫穿,你會發現它們根本不在同一層、不該互相比較。

```
H1 切系統   ← 「訂單」「庫存」「支付」該不該是不同服務?邊界畫哪?
   │
H2 依賴方向  ← 訂單服務「裡面」,業務邏輯要不要依賴 MySQL / Spring?
   │
H3 建模      ← 「訂單」這個概念,寫成一個帶行為的胖物件,還是一張表 + 一堆函數?
   │
跨切面      ← 讀和寫要不要拆成兩個模型?狀態用「當前值」存還是「事件序列」存?
```

### H1 · 切系統:邊界比技術難

問題:把系統切成幾塊、每塊管什麼、團隊怎麼對應。
- **DDD 戰略設計**站這層:用**限界上下文(bounded context)**找邊界——同一個「商品」在目錄上下文是描述+圖片,在庫存上下文只是 SKU+數量,**別做一個滿足所有人的 God Model**。
- 同層還有:微服務按 business capability 拆、**Conway's Law**(架構長成組織的樣子)、**Team Topologies**。
- 🔬 內幕:微服務怎麼切,90% 是組織+業務問題,披著技術外衣。→ 細節在 `01`,並回扣 `system-design/04`。

### H2 · 依賴方向:讓業務別黏著框架

問題:在**一個服務內部**,業務邏輯能不能不依賴「它存在 MySQL」「它跑在 Spring 裡」。
- **分層 → Hexagonal → Onion → Clean** 全站這層,核心思想只有一個:**依賴反轉**——讓 DB、框架、外部 API 依賴你的業務,而不是反過來。
- 🔬 內幕:這四個其實是**同一個 idea 的演化**,不是四選一。新人最常見的錯就是把 Clean Architecture 當成 DDD 的競品。→ `04` 一次講透。

### H3 · 建模:業務規則放哪、物件胖還是瘦

問題:「訂單」這個東西,寫成**帶行為的胖物件**(`order.cancel()` 自己檢查能不能取消),還是**一張貧血的數據結構 + 一個 `OrderService` 把規則全包**。
- **充血模型(Rich Domain Model)= DDD 戰術設計**站這層:Entity / Value Object / **Aggregate**。
- 對立面:**貧血模型 + Service**、**Transaction Script**、**Active Record**(Rails 風)。
- 🔬 內幕:**這層才是 DDD 真正有爭議、最容易過度設計的地方**。Java 圈愛充血,Go 圈幾乎全用貧血+service(且視為優點)。→ `02`(怎麼做)+ `03`(什麼時候別做)。

### 跨切面 · 讀寫分離 / 事件溯源

- **CQRS**:讀模型和寫模型拆開(在 code 層,不是只在 DB 主從)。
- **Event Sourcing**:不存「訂單現在是已付款」,而存「下單→付款→發貨」一串事件,當前狀態用事件重放算出來。
- 🔬 內幕:這兩個能**脫離 DDD 單獨用**,只是和 DDD 氣味相投常被綁一起。代價很高,別默認上。→ `06`、`07`,落地回扣 `financial-consistency/`。

## Part C · 選型心法:複雜度驅動,DDD 是工具箱不是宗教

**唯一的選型原則**:用**業務複雜度**決定投資多少,不是用「聽起來高不高級」。

```
業務複雜度低 ──────────────────────────────► 業務複雜度高
(CRUD、後台、報表)                      (計費、風控、履約、交易撮合)

H3 建模:  貧血 + Transaction Script  ──►  充血 + Aggregate(完整戰術 DDD)
H2 依賴:  傳統分層就夠           ──►  Hexagonal/Clean(隔離 + 好測)
H1 切分:  單體 / 模塊化單體      ──►  按限界上下文拆服務
跨切面:   一個模型讀寫共用       ──►  CQRS,甚至 Event Sourcing
```

**DDD 自己內部也分輕重**——這是全書最重要的一句:

| | 戰略設計(H1) | 戰術設計(H3) |
|---|---|---|
| 成本 | 低(畫圖、對齊語言) | 高(建模儀式、學習曲線) |
| 回報 | 高、幾乎總是值得 | 只在**核心域**值得 |
| 業界 | 廣泛採用 | 挑著用,常被批過度設計 |
| 你的策略 | **無腦用** | **只在核心域用** |

「核心域」哪來?DDD 把業務分三類,決定你的**頂尖工程師往哪放**:

| 子域 | 是什麼 | 怎麼做 |
|---|---|---|
| **核心域(Core)** | 你公司賺錢、競爭力所在(電商的履約定價、風控規則) | **上戰術 DDD,放最好的人** |
| **支撐域(Supporting)** | 必要但不獨特(後台管理、審批流) | 簡單寫,貧血+CRUD |
| **通用域(Generic)** | 誰都需要、沒差異(認證、發信、發票) | **直接買 / 用開源,別自己造** |

> **一句話心法**:**戰略 DDD 無腦用,戰術 DDD 只進核心域,通用域直接買。** 把 Aggregate 那套儀式擺到一個 CRUD 後台上,就是把架構複雜度當成了業務複雜度——這是反模式。

## Part D · 三大誤解校正(高頻面試陷阱)

**誤解 1:Clean Architecture 是 DDD 的「升級版 / 替代品」。**
> ❌ 錯。它們在**不同高度**。DDD 管「業務怎麼建模(H3)+ 系統怎麼切(H1)」,Clean/Hexagonal 管「依賴往哪指(H2)」。**正解:兩個一起用**——DDD 決定領域層長怎樣,Clean 決定領域層別依賴 DB/框架。

**誤解 2:不用 DDD 就「沒有架構」。**
> ❌ 錯。**Transaction Script / 貧血模型是正當的、而且常常是更好的選擇**(簡單、好懂、適合 CRUD)。它才是戰術 DDD 真正的對立方案。不是「DDD vs 沒架構」,是「DDD vs 事務腳本」,按複雜度選。

**誤解 3:CQRS 和 Event Sourcing 必須綁在一起、且必須配 DDD。**
> ❌ 錯。三者**正交**,可獨立採用。CQRS 可以只是「讀走一個快查表、寫走領域模型」;Event Sourcing 是另一個獨立決策(代價很大:版本演進、重放、調試都難)。**別打包上。**

## 面試滿分答法

> **Q:「你怎麼看 DDD?項目裡會用嗎?」**(這題在篩「是不是只會背名詞」)
>
> 滿分骨架:
> 1. **先拆兩半**:「DDD 分戰略和戰術。戰略(限界上下文、統一語言、防腐層)我**幾乎都會用**,因為便宜、回報高,也是我們劃服務邊界的依據。」
> 2. **戰術看複雜度**:「戰術那套(聚合、實體、值對象)我**只在核心域用**——複雜的計費、履約值得;一個 CRUD 後台上這套就是過度設計,我會用貧血模型加 service。」
> 3. **撇清誤解**:「而且 DDD 跟 Clean Architecture 不是二選一,它們在不同層,我通常一起用:DDD 管建模,Clean 管讓領域層不依賴框架。」
>
> 講到這裡,面試官知道你**不是背書,是有判斷**。

## 本章小結

- **兩條軸**:`system-design/` 管運行時不崩;本主線管應用內部怎麼建模/分層/演化。你天天做後者,只是沒有名字。
- **四個高度**:H1 切系統(戰略 DDD)、H2 依賴方向(Clean/Hexagonal)、H3 建模(戰術 DDD vs 貧血)、跨切面(CQRS/ES)。**它們不在同一層,別互相比較。**
- **選型 = 複雜度驅動**:業務越複雜,投資越多;反之保持簡單。
- **DDD 是工具箱不是宗教**:戰略無腦用,戰術只進核心域,通用域直接買。
- **三大誤解**:Clean ≠ DDD 替代(不同層,常一起用);DDD 的對立面是事務腳本不是「沒架構」;CQRS/ES 正交、別打包。
