# 第 8 章 · EventStorming 實作(流程 · 怎麼發現模型)

> 🔬 前面幾章是**結果**(限界上下文、聚合、領域事件長什麼樣),這章是**怎麼得到結果**。EventStorming(事件風暴,Alberto Brandolini 發明)是一個**工作坊方法**:把業務專家和開發拉進一個房間,用**便利貼貼滿一面牆**,在幾小時內把一個複雜領域的事件流、邊界、痛點全挖出來。
>
> 這章偏 **tech lead 的實操工具**(面試較少直接考,但「你怎麼和業務一起建模」「怎麼劃邊界」答得出 EventStorming 是加分項)。重點:**色卡語法 + 工作坊流程 + 怎麼主持 + 常見坑**。

---

## Part A · 為什麼用它(解決什麼)

傳統需求:業務寫文檔 → 開發讀文檔猜 → 建模 → 做出來發現理解錯了。**慢、且翻譯層厚**(回扣 `01` 統一語言要消滅的就是這個)。

EventStorming 的賭注:**把所有人拉到同一面牆前,用同一套便利貼**,當場對齊。它的三個威力 🔬:
- **以「事件」為起點**,而不是以「資料庫表/UI」為起點——事件是業務語言,業務專家天然會講「然後就……了」。
- **可視化整個流程**,矛盾和空白一眼暴露(「這裡到底誰負責?」紅卡一貼,焦點就找到了)。
- **當場產出統一語言**:貼上牆的詞,就是之後寫進代碼的詞(`01`)。

**三個層次(由粗到細)**:
| 層次 | 目的 | 誰參加 |
|---|---|---|
| **Big Picture** | 摸清整個業務全貌、找子域/上下文邊界 | 所有相關方,人越多越好 |
| **Process Level** | 細化某條業務流程的事件/命令/策略 | 該流程的業務+開發 |
| **Design Level** | 逼近代碼:聚合、命令、事件一一對應 | 開發為主 |

## Part B · 色卡語法(這套「視覺文法」是核心)

EventStorming 用便利貼顏色表達不同概念,順序也有講究。記住這套對應,你就讀得懂任何一面 EventStorming 牆:

| 顏色 | 概念 | 寫什麼 | 對應前面章節 |
|---|---|---|---|
| 🟧 橙 | **領域事件 Domain Event** | 過去式:「訂單已提交」 | `07` 領域事件 |
| 🟦 藍 | **命令 Command** | 動作:「提交訂單」(誰觸發了事件) | 觸發聚合方法 |
| 🟨 黃 | **聚合 Aggregate** | 命令打在哪個聚合上:「訂單」 | `02` 聚合 |
| 🟪 紫 | **策略 / Policy** | 「每當 X 事件,就要 Y」的業務規則 | Saga/事件訂閱 `07` |
| 🟫 粉/淺紅 | **外部系統 External System** | 「支付網關」「物流 API」 | `01` 上下文邊界/ACL |
| 🟥 紅 | **熱點 Hotspot** | 疑問、衝突、卡點:「這裡規則不清?」 | 要解決的問題 |
| 🟩 綠 | **讀模型 Read Model** | 「用戶看到的訂單列表」 | `06` CQRS 讀模型 |
| 👤 | **角色 Actor** | 誰發起命令(客戶、客服) | |

**標準微觀結構**(一個業務步驟長這樣):
```
👤角色 ──► 🟦命令 ──► 🟨聚合 ──► 🟧事件 ──► 🟪策略 ──► (觸發下一個🟦命令)
 客戶      提交訂單     訂單      訂單已提交   每當下單成功    扣減庫存
```

> 🔬 為什麼這個結構重要:它**幾乎一比一對應你的代碼**——命令=聚合方法、聚合=`02` 的聚合、事件=`07` 的領域事件、策略=事件訂閱/Saga。所以 Design Level 的牆,**可以直接翻譯成代碼骨架**。

## Part C · 工作坊流程(Big Picture 為例)

1. **只貼事件(發散)**:給橙色便利貼,所有人把想到的「已發生的業務事實」貼上牆,**過去式**,先不管對錯、不管順序。牆很快貼滿——這是好事。
2. **排時間線(收斂)**:把事件按**時間順序**從左到右排好;重複的合併、矛盾的並排。業務流程的骨架浮現。
3. **找命令與角色**:每個事件**是誰、用什麼命令觸發的**?藍卡、角色補上。
4. **找策略與外部系統**:「每當某事件就要做某事」貼紫卡;碰到外部系統(支付、物流)貼粉卡——**粉卡聚集處往往是上下文邊界**(`01`)。
5. **標熱點**:規則不清、有爭議、沒人能答的地方貼**紅卡**——這些是**最有價值的產出**(找到了真正要澄清的問題)。
6. **劃邊界**:看牆上「語言換意思」「一簇緊密相關的事件」的地方畫線——**這就是限界上下文 / 子域的邊界**(`01` 的「語言接縫」在這裡變得肉眼可見)。

## Part D · 怎麼主持 + 常見坑(tech lead 視角)

**主持要點**:
- **業務專家是主角,開發是記錄/提問者**——別讓開發主導,否則又滑回技術視角。
- **無限長的牆 + 大量便利貼**:空間不夠是最常見的物理失敗;線上用 Miro/EXcalidraw 等白板。
- **時間盒**:Big Picture 半天到一天;別追求一次完美,**迭代**。

**常見坑** 🔬:
| 坑 | 症狀 | 解 |
|---|---|---|
| 一上來糾結技術 | 開始討論用什麼資料庫、什麼框架 | 拉回來:**先只貼事件,技術最後談** |
| 事件不用過去式 | 貼「提交訂單」(那是命令) | 過去式紀律:事件是**已發生**,「訂單已提交」 |
| 紅卡(熱點)被忽略 | 大家繞開難題只貼簡單的 | **紅卡才是金礦**,專門花時間攻 |
| 只做一次就完事 | 牆拍照存檔再沒人看 | 它是**活文檔**,隨理解更新;Design Level 直接導出代碼骨架 |

> **和後面章節的接力**:EventStorming 產出的邊界 → `01` 限界上下文;聚合黃卡 → `02` 聚合;策略紫卡 → `07` 事件/Saga;紅卡熱點 → 需要澄清的業務決策(可寫成 ADR,`10`)。

## 本章小結

- **EventStorming = 把業務專家和開發拉到一面牆前,用便利貼幾小時挖出整個領域**;以「事件」為起點(業務語言),當場產出統一語言、暴露矛盾、找到邊界。
- **三層次**:Big Picture(全貌/找邊界)→ Process Level(細化流程)→ Design Level(逼近代碼)。
- **色卡語法**:橙=事件、藍=命令、黃=聚合、紫=策略、粉=外部系統、紅=熱點、綠=讀模型;微觀結構 `角色→命令→聚合→事件→策略` **幾乎一比一對應代碼**。
- **流程**:只貼事件→排時間線→找命令角色→找策略/外部系統→標熱點→劃邊界。
- **主持**:業務當主角、紅卡是金礦、技術最後談、它是活文檔。產出直接接 `01`/`02`/`07`/`10`。
- 下一章 `09`:**漸進引入**——你 EventStorming 出了理想模型,但現實是一坨 legacy,怎麼用防腐層 + Strangler Fig 不靠大重寫地遷過去。
