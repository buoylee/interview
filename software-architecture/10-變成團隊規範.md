# 第 10 章 · 變成團隊規範(落地 · 併入 engineering-handbook)

> 🔬 前 9 章給了你**判斷力**;但判斷力只活在你腦子裡,團隊不會自動照做。這章是**收官**:把這套東西沉澱成你 `<20` 人團隊的**開發規範**(目錄、命名、PR 檢查點、決策記錄),讓「複雜度驅動、戰略無腦用戰術只進核心域」這些判斷**內建進日常流程**,而不是靠你一個人盯。
>
> **這章的反模式**:① 把 DDD 當**教條全員強推**(所有模塊都聚合+CQRS,團隊崩潰);② 只寫一份沒人看的規範文檔。好的規範是**默認值 + 檢查點 + 決策記錄**,不是聖經。產出併入你的 [engineering-handbook]。

---

## Part A · 第一原則:規範要「複雜度分級」,不是一刀切

把 ch0 的核心判斷直接寫進團隊規矩——**不同子域,不同規範**:

| 子域(`01`) | 默認規範 | 對應章 |
|---|---|---|
| **核心域** | 充血聚合 + Clean 分層 + Repository + 領域事件;PR 走架構評審 | `02` `04` `07` |
| **支撐域** | 貧血 + Transaction Script + 按功能分包就好,別上聚合 | `03` `05` |
| **通用域** | **禁止自研**,用開源/買;只寫 ACL 接入 | `01` |

> 🔬 為什麼這條最重要:團隊翻車最常見的原因不是「不會 DDD」,是**把核心域的重型規範套到支撐域 / CRUD 上**,結果人人覺得「DDD 又重又煩」。**規範第一條就要寫清:先判子域,再選重量。**

## Part B · 目錄與命名規範(可直接抄進 handbook)

**目錄(回扣 `05`)**:按業務功能 / 限界上下文分頂層包,不按技術層。
```
src/
├── order/           ← 一個限界上下文 = 一個頂層模塊
│   ├── domain/          聚合、VO、Repository 接口、領域事件
│   ├── application/     應用服務 / use case
│   └── infrastructure/  Repository 實現、ACL、MQ adapter
├── payment/
└── shared/          ← 真正跨上下文共享的(慎放,易變共享內核 `01`)
```
規矩:**`order` 不准直接 import `payment.domain`**——跨上下文只能透過對外接口 / 事件(模塊化單體紀律,`05`)。

**命名(回扣 `01` 統一語言)**:
- 類別/方法名 = **業務術語**,不准翻譯:`policy.endorse()` 不是 `policyService.updateRecord()`。
- 維護一份**團隊術語表(glossary)**,新人入職第一份讀物;術語表的詞 = 代碼裡的詞 = 和業務溝通的詞。
- 事件用**過去式**(`OrderPlaced`,`07`),命令用**祈使句**(`PlaceOrder`)。

## Part C · PR 檢查清單(把判斷力變成 checklist)

規範落地的真正抓手是 **PR review checklist**——把前 9 章的判斷變成幾個可勾的問題,貼進 PR 模板:

```markdown
## 架構檢查(改到業務邏輯時勾)
- [ ] 這塊屬於哪個子域?重量級規範用對了嗎?(核心域才上聚合,CRUD 別上)  → ch0/03/10
- [ ] 業務邏輯放對地方了嗎?(該充血的進聚合,別散在 service;該貧血的別硬塞行為)→ ch02/03
- [ ] 有沒有跨限界上下文直接 import?(只能走接口/事件)  → ch01/05
- [ ] 領域層有沒有依賴框架/DB?(JPA 註解、Spring 爬進領域對象沒?)  → ch04
- [ ] 一個事務有沒有改多個聚合?(應該一事務一聚合 + 事件最終一致)  → ch02
- [ ] 接外部/舊系統,有沒有走防腐層?(別讓外部模型滲進來)  → ch01/09
- [ ] 命名用業務術語了嗎?(對得上 glossary)  → ch01
```

> 🔬 這份 checklist 才是規範的「牙齒」。文檔沒人讀,但 **PR 模板裡的勾選框人人都會碰到**——把架構判斷嵌進**已有的高頻流程**,比開會宣講有效得多。

## Part D · 架構決策記錄(ADR):讓「為什麼」留下來

團隊最大的知識流失:**「當初為什麼這麼設計」沒人記得**。用 **ADR(Architecture Decision Record)** 治:每個重要架構決策寫一頁,進 git。

```markdown
# ADR-007: 訂單與庫存做成最終一致,不放一個事務
- 狀態:已採納
- 背景:下單要扣庫存,兩者是不同聚合/服務
- 決策:下單成功發 OrderPlaced 事件,庫存異步扣減(Saga 補償)
- 理由:一事務一聚合(ch02);跨服務 2PC 不可取(ch07)
- 代價:存在短暫不一致窗口,需 UI 配合提示
```

> ADR 的價值:① 新人讀 ADR 就懂演進脈絡;② 半年後想改時,知道**當初的約束還在不在**(回扣記憶裡的「recalled 決策要先驗證還成不成立」);③ EventStorming(`08`)的紅卡熱點,澄清後正好沉澱成 ADR。`<20` 人團隊,輕量 ADR(一頁、markdown、進 repo)足矣,別上重流程。

## Part E · 怎麼推行(tech lead 的節奏)

別想著一次到位。推行順序 🔬:
1. **先寫 glossary + 目錄規範**(`01`/`05`)——最便宜、零爭議、立刻見效。
2. **在一個新的核心域模塊試點**完整套路(`02`/`04`),做成團隊的**樣板工程**,讓人照著抄。
3. **PR checklist 上線**,先當「建議」再變「必須」。
4. **存量 legacy 用 `09` 漸進絞殺**,不強制立刻重構。
5. **ADR 養成習慣**,從下一個大決策開始記。

> 一句話收官:**規範的目的是讓團隊默認做出好判斷,不是讓團隊膜拜 DDD。** 你作為 tech lead 的價值,是把這 10 章的「what + why」翻譯成團隊能無痛執行的「默認值 + 檢查點」——這正是從資深工程師走向架構師/技術管理的那一步。

## 本章小結

- **規範要複雜度分級**:核心域(充血+Clean+評審)、支撐域(貧血+事務腳本)、通用域(禁自研、只寫 ACL);團隊翻車多因「重型規範套到 CRUD」。
- **目錄按上下文分**(`05`)、**命名用統一語言 + glossary**(`01`)、事件過去式命令祈使句。
- **PR checklist 是規範的牙齒**:把前 9 章判斷變成幾個勾選框,嵌進高頻流程,勝過開會宣講。
- **ADR**:每個架構決策一頁進 git,留住「為什麼」;EventStorming 紅卡熱點澄清後沉澱成 ADR;`<20` 人用輕量版。
- **推行有節奏**:glossary/目錄先行 → 核心域樣板工程 → PR checklist 由建議到必須 → legacy 漸進絞殺 → ADR 習慣。
