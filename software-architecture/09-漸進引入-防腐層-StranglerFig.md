# 第 9 章 · 在 legacy 裡漸進引入(流程 · 防腐層 + Strangler Fig)

> 🔬 前面教的都是「理想模型怎麼設計」,但你 90% 的職業生涯面對的是**已經存在的一坨大泥球 / 分布式單體**(`01` Context Map 的 Big Ball of Mud)。這章回答最現實的問題:**怎麼不靠「推倒重寫」,把一個爛系統漸進地遷到乾淨架構。**
>
> **最常見的露怯 / 最大的職業陷阱**:① 主張**大爆炸重寫(big-bang rewrite)**——這是軟體史上最著名的失敗模式;② 知道防腐層(`01`)卻不知道它正是漸進遷移的引擎;③ 答不出「怎麼安全地一塊一塊換」。這章把 **Strangler Fig + 防腐層 + 配套技法**講成一套可操作的遷移劇本。

---

## Part A · 先說為什麼不能大重寫

「這代碼太爛了,我們重寫一個吧」——聽起來爽,**幾乎總是災難** 🔬:

- **舊系統的「醜代碼」裡藏著多年的業務知識**(無數 edge case、補丁、合規規則),重寫會把這些**全部丟失再痛苦重新發現**。
- **重寫期間,業務不會停**:你要同時維護舊系統 + 追趕它的新需求 + 寫新系統,**兩倍工作量**,新系統永遠追不上。
- **「最後一刻才切換」的巨大風險**:一切賭在某個上線日。

> 一句話:**重寫是用「已知的爛」換「未知的險」。** 正解是**漸進絞殺**——讓新舊並存,一塊一塊遷,每一步都可上線、可回退。

---

## Part B · Strangler Fig(絞殺榕)模式

名字來自一種植物:絞殺榕纏繞在宿主樹上生長,**慢慢取代它,最後宿主枯死、榕樹獨立成形**。Fowler 借來形容遷移:

```
階段 1:  [外觀/路由層 Facade]                    ← 所有流量先進這層
              │
       ┌──────┴──────┐
       ▼             ▼
   [新系統(一小塊)]  [舊系統(其餘全部)]          ← 新代碼只接管一小塊功能

階段 2..n: 新系統那塊越長越大,舊系統越縮越小       ← 一塊一塊遷,每步都上線

最終:    [新系統(全部)]   舊系統枯死、刪除        ← 宿主被「絞殺」
```

**關鍵:前面要有一個「外觀 / 路由層」(Facade)** 🔬——它攔截所有請求,**按功能決定路由到新還是舊**。客戶端完全不知道後面在換引擎。常見載體:API 網關(回扣 `system-design/04`)、反向代理、或一個 facade 服務。

---

## Part C · 防腐層是遷移的引擎(接 `01`)

遷移時新舊**必然要共存、要互相調用**,而舊系統的爛模型不能污染你的新乾淨上下文——這正是**防腐層(ACL,`01`)**的主戰場:

```
[新的乾淨上下文]  ──►  [ACL 防腐層]  ──►  [舊系統 / 舊資料庫]
  用統一語言、聚合        翻譯:把舊模型              c_nm, c_sts...
  (01/02)             翻成新模型,擋在門外           爛但別碰它
```

- 新上下文**只透過 ACL** 跟舊系統說話,舊系統的概念一個都不漏進來。
- 等某塊舊功能完全遷完、舊系統那部分刪掉,對應的 ACL 也跟著退役。
- **ACL 是「臨時的、會死的」**——它的存在就是為了讓遷移期新舊隔離,遷完即拆。

---

## Part D · 配套技法(讓每一步都安全)

漸進遷移要安全,靠這幾個和 `system-design/09`(發布變更)聯動的技法 🔬:

| 技法 | 幹什麼 |
|---|---|
| **Branch by Abstraction(抽象分支)** | 在舊實現上先抽一層接口(`04` 依賴反轉),新實現實現同一接口,切換只換 wiring——**不長期拉 git 分支**就能並行開發 |
| **Parallel Run(並行運行)** | 新舊邏輯同時跑,**比對結果**,確認新的算得和舊的一致才切流量(金融遷移常用) |
| **Feature Flag(`system-design/09`)** | 用開關控制流量走新還是舊,**出問題秒回退**,灰度放量 |
| **先讀後寫 / 先旁路後核心** | 先遷風險低的查詢/旁路功能練手,核心寫路徑最後遷 |
| **資料庫怎麼辦** | 最難。先共享舊庫(ACL 直連)→ 雙寫過渡(`system-design/07` 遷移雙寫)→ 新庫獨立。**數據遷移是漸進遷移裡最硬的骨頭** |

**遷移劇本(綜合)**:
1. 用 EventStorming(`08`)畫出目標邊界,選**一個小而低風險的限界上下文**先動。
2. 前面架好 Facade / 路由層。
3. 新上下文用乾淨架構(`02`/`04`/`05`)實現這塊,透過 **ACL** 跟舊系統/舊庫交互。
4. 用 **Feature Flag + Parallel Run** 灰度切流量、比對、放量。
5. 這塊穩了,刪掉舊代碼和對應 ACL;**重複**,絞殺下一塊。

## 面試滿分答法

> **Q:「給你一個又大又爛的單體,怎麼現代化 / 拆微服務?」**(高頻,直擊判斷力)
> 「**絕不大爆炸重寫**——那會丟失藏在舊代碼裡的業務知識、且要兩倍工作量追趕業務。我用 **Strangler Fig**:前面架一個路由層,選一個低風險的限界上下文先抽出來用乾淨架構重做,新模塊透過**防腐層**跟舊系統隔離,用 **Feature Flag + 並行比對**灰度切流量;這塊穩了就刪舊代碼,再絞殺下一塊。**每一步都能上線、能回退**,風險可控。最硬的是資料庫,走共享→雙寫→獨立。」

---

## 本章小結

- **絕不大爆炸重寫**:它丟失舊代碼裡的業務知識、要兩倍工作量、賭一個上線日——用「已知的爛」換「未知的險」。
- **Strangler Fig**:前置 Facade/路由層,新系統一塊塊接管功能,舊系統一塊塊縮小直到枯死刪除;每步可上線可回退。
- **防腐層(ACL)是遷移引擎**:新舊共存期用它隔離,舊模型不漏進新上下文;遷完即拆,是臨時的。
- **配套技法**:Branch by Abstraction(接口先行)、Parallel Run(比對)、Feature Flag(灰度+回退)、先旁路後核心、資料庫共享→雙寫→獨立。
- 劇本:EventStorming 定邊界 → 選小塊 → Facade → 新架構+ACL → Flag+比對放量 → 刪舊 → 重複。
- 下一章 `10`(收官):**變成團隊規範**——把這 9 章的判斷力沉澱成你 `<20` 人團隊的開發規範,併入 `engineering-handbook`。
