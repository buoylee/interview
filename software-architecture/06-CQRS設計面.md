# 第 6 章 · CQRS(跨切面 · 設計面的讀寫分離)

> 🔬 CQRS 是個被嚴重誤解的詞。它的本質是**在 code/模型層**把「寫(命令)」和「讀(查詢)」分成兩套模型——**不是**資料庫主從讀寫分離(那是 `system-design/07` 的數據面),**不**強制要兩個資料庫,**不**強制配 Event Sourcing。
>
> **最常見的露怯**:① 把 CQRS 等同於「讀走從庫、寫走主庫」(那只是部署層,和 CQRS 是兩回事);② 以為 CQRS 必須配 Event Sourcing 或必須兩個庫;③ 不在 CRUD 上也硬拆讀寫模型,徒增複雜。這章把 CQRS 的**層次**和**何時值得**講清。

---

## Part A · 從 CQS 到 CQRS

**CQS(命令查詢分離,Bertrand Meyer)** 是個方法級原則:**一個方法,要麼改狀態(命令,返回 void),要麼查狀態(查詢,返回數據),不要兩者都做。**

```java
order.cancel();              // 命令:改狀態,不返回業務數據
Money total = order.total(); // 查詢:返回數據,不改狀態(調它前後系統狀態一樣)
```
> 好處:查詢無副作用 → 可隨意調用、緩存、重試;命令意圖明確。

**CQRS(命令查詢職責分離,Greg Young)** 把這個原則**上升到架構**:不只方法分,**整個模型分兩套**——
- **寫模型(Command side)**:就是你的領域模型 / 聚合(`02`),負責守不變量、執行業務規則。
- **讀模型(Query side)**:為查詢**量身定做**的數據結構,可以高度反范式、直接拼成頁面要的形狀,**繞過聚合**。

```
       命令(下單/取消)         查詢(訂單列表/詳情頁)
            │                        │
       ┌────▼─────┐            ┌─────▼──────┐
       │ 寫模型     │            │ 讀模型       │
       │ 聚合+規則  │  ──同步──► │ 反范式視圖   │
       │(守不變量)│   (事件/   │(直接喂 UI) │
       └──────────┘    投影)   └────────────┘
```

## Part B · CQRS 的三個層次(別一步到位)

CQRS 是**光譜**,不是開關。按複雜度逐級上,**絕大多數系統停在第 1 級就夠**:

| 級 | 做法 | 何時用 |
|---|---|---|
| **L1 同庫雙模型** | 同一個資料庫,但寫走領域模型、讀走專門的查詢(直接 SQL/視圖,返回 DTO 不經聚合) | **最常見、推薦起點**;讀寫需求開始分化時 |
| **L2 獨立讀存儲** | 寫進主庫,**異步投影**出一份讀專用存儲(如 ES/Redis/反范式表) | 讀負載大、查詢形狀和寫模型差很遠(搜索、報表) |
| **L3 全分離管線** | 讀寫完全獨立的模型+存儲+服務,常配 Event Sourcing | 極端讀寫不對稱、超高規模;**代價最大,慎用** |

> 🔬 **關鍵內幕**:L2/L3 一旦讀寫**異步同步**,讀模型就是**最終一致**的——下單後立刻查可能還沒更新。這是 CQRS 最大的隱性代價,UI/產品要能接受。L1 同庫同事務就沒這問題。**別默認跳到 L2/L3。**

## Part C · CQRS vs 資料庫讀寫分離(面試高頻區分)

這兩個最容易混,一張表釘死:

| | **CQRS(本章,設計面)** | **讀寫分離(`system-design/07`,數據面)** |
|---|---|---|
| 分的是 | **模型/代碼**(寫模型 vs 讀模型) | **物理庫**(主庫寫、從庫讀) |
| 目的 | 讀寫**邏輯/形狀**差太多,各自優化 | 分攤讀**負載**、提升讀吞吐 |
| 模型 | 兩套不同模型 | **同一套表結構**,只是多副本 |
| 層次 | 架構/應用層 | 部署/基礎設施層 |

> 它倆**正交、可疊加**:你可以同時做 CQRS(讀寫兩套模型)+ 讀模型放只讀從庫(讀寫分離)。**但概念別混**:讀寫分離是「同一個模型多副本扛流量」,CQRS 是「兩個不同模型各司其職」。

---

## Part D · 何時值得 / 何時是過度設計

**值得(✅)**:
- 讀和寫的**需求嚴重分化**:寫端是複雜聚合+規則,讀端是花式報表/聚合查詢/搜索,硬用一套模型兩邊都難受。
- 讀寫**負載極不對稱**(讀遠多於寫),要分別擴展。
- 核心域,複雜度撐得起。

**過度設計(❌)**:
- 普通 CRUD:讀寫就是同一張表的增刪改查,**一個模型搞定**,拆成兩套純屬自找麻煩。
- 團隊不熟、又上 L2/L3 的最終一致,徒增 bug。

> 🔬 一句話(回扣 ch0):**CQRS 的 L1(同庫雙模型)成本低、隨時可用,讀寫一分化就上;L2/L3(獨立讀庫、最終一致)成本高,只在讀寫真的撕裂時上。** 和 Event Sourcing 是**獨立決策**——可以只 CQRS 不 ES,反之 ES 幾乎必然要配 CQRS(`07` 講)。

**生態**:.NET 圈 CQRS + MediatR 很流行(命令/查詢各一個 handler,配 `05` Vertical Slice);Java 用 Axon 或手寫 command/query 分離;Go/Python 多手寫——**本質是「兩套模型」的紀律,不依賴框架**。

## 面試滿分答法

> **Q:「CQRS 和資料庫讀寫分離是一回事嗎?」**
> 「**不是。** 讀寫分離是**數據面**——同一套表多副本,主庫寫從庫讀,目的是分攤讀負載。CQRS 是**設計面**——把寫模型(領域聚合,守規則)和讀模型(反范式、直接喂 UI)拆成**兩套不同模型**,目的是讀寫需求分化時各自優化。兩者正交,可以疊加,但別混。」

> **Q:「CQRS 一定要配 Event Sourcing / 兩個資料庫嗎?」**
> 「不用。CQRS 是光譜:最輕的 L1 是同一個庫、同一事務,只是讀走專門查詢、寫走領域模型——這級沒有最終一致問題,推薦起點。要獨立讀存儲(L2)、要 ES(L3)是更重的、獨立的決策,只在讀寫真的撕裂或要極致審計時才上。」

## 本章小結

- **CQS**(方法級:命令改狀態、查詢返數據,別混)→ **CQRS**(架構級:寫模型=聚合守規則,讀模型=反范式喂 UI,兩套)。
- **CQRS 是光譜**:L1 同庫雙模型(推薦起點、無最終一致問題)→ L2 獨立讀存儲(異步投影、最終一致)→ L3 全分離+ES(代價最大)。**別默認跳到 L2/L3。**
- **CQRS ≠ 讀寫分離**:前者分模型(設計面)、後者分物理庫(數據面);正交可疊加,概念別混。
- **何時值**:讀寫需求/負載嚴重分化的核心域;**何時別**:普通 CRUD。
- **CQRS 和 ES 是獨立決策**:可只 CQRS;ES 幾乎必配 CQRS。
- 下一章 `07`:**Event Sourcing + 領域事件 + Saga**——把「狀態當事件存」「跨聚合最終一致怎麼編排」一次講透。
