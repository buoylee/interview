# 第 7 章 · Event Sourcing + 領域事件 + Saga(跨切面 · 事件與編排)

> 🔬 這章三個東西常被混成一團,先釘死它們是**三個獨立的概念**:**領域事件**(通信:宣布「發生了一件事」)、**Event Sourcing**(存儲:用事件序列當真相,而非當前狀態)、**Saga**(編排:跨聚合/服務的長事務怎麼保證最終一致 + 補償)。可以只用領域事件而不用 ES,可以用 Saga 而不用 ES。
>
> **最常見的露怯**:① 以為 DDD 必須配 Event Sourcing;② 講不出 ES 的真實代價(版本演進、重放、查詢難),把它當銀彈;③ Saga 只知道名字,分不清編排式(orchestration)和協作式(choreography)、不知道為什麼不用 2PC。這章把三者 + 取捨講清,落地細節接你的深礦 `financial-consistency/`、`MQ/`、`distr-tx/`。

---

## Part A · 領域事件(Domain Event):最該先掌握的

**定義**:領域裡**已經發生**的、業務關心的事實,用**過去式**命名:`OrderPlaced`、`PaymentCaptured`、`InventoryReserved`。

**它解決什麼**:`02` 的鐵律說「一個事務只改一個聚合,聚合間最終一致」——靠什麼最終一致?**靠領域事件**。下單聚合提交後發 `OrderPlaced`,庫存、積分、通知各自訂閱去做自己的事,**彼此解耦**(回扣 `system-design/04` 的事件驅動)。

```java
class Order {
    private final List<DomainEvent> events = new ArrayList<>();
    void place() {
        // ...守不變量、改狀態
        events.add(new OrderPlaced(id, lines, total));  // 記錄「發生了什麼」
    }
}
// 應用服務提交事務後,把 events 發布出去(進程內 or MQ)
```

> 🔬 **可靠發布的坑**:「改聚合(DB)」和「發事件(MQ)」是**兩個系統**,中間崩了就會「庫改了事件沒發」或反之。解法是 **Outbox 模式**(事件先和業務數據寫進同一個 DB 事務的 outbox 表,再由另一進程轉發到 MQ)——**這就是你深礦 `financial-consistency/` 的核心內容,這裡不重寫,指過去。** 領域事件是 ES、Saga、跨上下文集成(`01` Context Map)的共同地基。

## Part B · Event Sourcing(ES):用事件當真相

**常規存儲**:存**當前狀態**——`account` 表一行 `balance = 80`。改了就覆蓋,歷史沒了。

**Event Sourcing**:**不存當前狀態,存發生過的事件序列**;當前狀態靠**重放(replay)事件**算出來。

```
常規:        balance = 80                      (只剩結果,過程丟了)

Event Sourcing:
  AccountOpened(0) → Deposited(+100) → Withdrawn(-20)   ← 這串才是真相
  當前餘額 = 把它們 fold 一遍 = 0+100-20 = 80           ← 狀態是「算」出來的
```

| ES 的好處 ✅ | ES 的代價 🔬(面試重點,別當銀彈) |
|---|---|
| **完整審計**:每次變化都有記錄,天然合規 | **事件版本演進**:`Deposited` 改了結構,老事件怎麼讀?要 upcasting |
| **時間旅行**:能重建任意歷史時刻的狀態 | **查詢難**:不能直接 `WHERE balance > 100`,**必須配 CQRS** 投影出讀模型 |
| **重建投影**:換讀模型?重放事件即可 | **快照(snapshot)**:事件多了重放慢,要定期存快照 |
| **業務洞察**:事件流本身是寶貴數據 | **最終一致 + 調試難**:狀態不直觀,排查靠讀事件流 |

> 🔬 **何時值(很窄)**:強審計/合規(賬本、金融、交易)、業務本質就是「一連串事件」(履約軌跡)、需要時間旅行/重放。**何時別(絕大多數)**:普通業務上 ES = 用核彈打蚊子,版本演進和查詢的長期成本會壓垮團隊。**ES 幾乎必然要配 CQRS**(查詢得靠投影),所以它是「CQRS L3」那一級的東西。**DDD 完全不要求 ES**——這是最該破的迷思。

## Part C · Saga:跨聚合/服務的長事務,不用 2PC

**問題**:下單要「扣庫存 + 扣款 + 創建發貨」,橫跨三個聚合/服務。傳統想法是分布式事務(2PC/XA),但 **2PC 在微服務裡基本不用** 🔬:它要鎖住所有參與者直到協調者拍板,**長時間持鎖 → 吞吐崩、可用性差、協調者單點**。

**Saga 的思路**:把長事務拆成**一串本地事務 + 對應的補償操作**;任一步失敗,**反向執行補償**(而不是回滾整個分布式事務),做成**最終一致**。

```
正向:  扣庫存 → 扣款 → 創建發貨
失敗(扣款失敗):
補償:  ← 釋放庫存  (扣庫存的補償)        ← 沒有「回滾」,只有「做一個反向操作」
```

**兩種協調方式(高頻區分)**:

| | **協作式 Choreography** | **編排式 Orchestration** |
|---|---|---|
| 怎麼推進 | 各服務**訂閱事件**自行反應(無中心) | 一個 **Saga 協調者**發命令、收結果、決定下一步 |
| 優點 | 鬆耦合、無單點 | 流程**集中可見**、好調試/改流程 |
| 缺點 | 流程**散在各處**,難看清全貌、易循環 | 協調者是**焦點**(可能變胖、單點) |
| 適合 | 步驟少、簡單 | 步驟多、流程複雜、要可視化 |

> **Process Manager** ≈ 有狀態的編排式 Saga(記住「走到哪一步了」)。落地的可靠性(冪等、去重、補償順序)→ 你的深礦 `distr-tx/`、`financial-consistency/`、`MQ/`,這裡只給骨架。

## Part D · 三者關係與選型(一張圖收束)

```
領域事件 ──── 是地基,獨立可用(解耦、跨上下文集成)
   │
   ├──► Event Sourcing:把事件當「存儲真相」(重,窄場景,必配 CQRS)
   │
   └──► Saga:用事件/命令「編排跨聚合長事務 + 補償」(替代 2PC)
```

| 概念 | 是什麼類別 | 默認該不該上 |
|---|---|---|
| 領域事件 | 通信/解耦 | **常用**,聚合間最終一致的基礎 |
| Event Sourcing | 存儲策略 | **默認不上**,窄場景(審計/金融)才上 |
| Saga | 分布式編排 | **跨服務長事務時上**(替代 2PC) |

> 一句話(回扣 ch0):**領域事件無腦用;Saga 在跨服務長事務時用;Event Sourcing 是高成本特化武器,別默認、別和 DDD 綁定。**

## 面試滿分答法

> **Q:「Event Sourcing 有什麼代價?DDD 一定要用嗎?」**
> 「**DDD 完全不要求 ES,這是常見誤解。** ES 的好處是完整審計、時間旅行、可重建投影;代價很實在:事件版本演進(老事件怎麼讀)、查詢必須配 CQRS 投影、要快照防重放變慢、狀態不直觀調試難。所以我只在強審計/金融賬本這種窄場景用,普通業務用常規存儲 + 領域事件就夠。」

> **Q:「跨服務的長事務怎麼保證一致?為什麼不用 2PC?」**
> 「用 **Saga**:拆成一串本地事務 + 補償,失敗就反向補償,做成最終一致。不用 2PC 是因為它要長時間鎖住所有參與者,吞吐和可用性都崩、協調者還是單點。Saga 分協作式(訂閱事件、無中心、適合簡單流程)和編排式(中央協調者、流程可見、適合複雜流程),步驟多我傾向編排式。」

## 本章小結

- **三個獨立概念**:領域事件(通信)、Event Sourcing(存儲)、Saga(編排),別混、別綁。
- **領域事件**:過去式命名的業務事實,是聚合間最終一致、跨上下文集成的地基;可靠發布靠 **Outbox**(→ `financial-consistency/`)。
- **Event Sourcing**:存事件序列而非當前狀態,狀態靠重放算;好處審計/時間旅行/重建投影,代價版本演進/查詢難(必配 CQRS)/快照/調試難;**窄場景才上,DDD 不要求**。
- **Saga**:跨服務長事務 = 本地事務串 + 補償,替代 2PC(2PC 長持鎖、單點、不可用);分協作式(無中心、簡單)vs 編排式(中央協調者、複雜流程)。
- 選型:領域事件常用、Saga 按需、ES 默認不上。落地可靠性接深礦 `distr-tx/`、`financial-consistency/`、`MQ/`。
