# 第 4 章 · 分層 → Hexagonal → Onion → Clean(H2 · 依賴方向)

> 🔬 這章治一個流行病:把 **Hexagonal、Onion、Clean 當成「三選一」的技術選型**,糾結「該用哪個」。真相是——**它們是同一個 idea 的三張畫法**,核心都是一句話:**讓依賴指向業務,而不是指向資料庫和框架**(依賴反轉)。
>
> **最常見的露怯**:① 把 Clean Architecture 當成 DDD 的升級版/替代品(它倆在不同高度);② 講得出名字卻說不清「依賴反轉到底反轉了什麼」;③ 不知道上一章的**防腐層、Repository 在架構圖上到底擺哪**。這章一次講透,並把 `01` 的零件安回架構圖。

---

## Part A · 起點:傳統分層,和它的「原罪」

你最熟的結構,Spring 三層:

```
Controller  ──►  Service  ──►  Repository/DAO  ──►  MySQL
 (Web 框架)      (業務邏輯)        (JPA/MyBatis)       (資料庫)

依賴方向:從上往下,一路指向資料庫 ▼▼▼
```

它能跑,但有個**原罪**:**業務邏輯(Service)依賴了基礎設施(DAO/JPA/MySQL)**。後果:

- **測業務邏輯要連資料庫** 🔬:想測「訂單金額算對沒」,卻得起一個 DB(或 H2),因為 Service `import` 了 Repository 的具體實現。業務和存儲焊死了。
- **換存儲 = 動業務**:MySQL 換 Mongo、JPA 換 MyBatis,Service 跟著改。
- **框架綁架領域**:JPA 註解 `@Entity`、`@Column` 爬進了你的領域對象,領域對象變成「資料庫表的鏡像」,而不是「業務概念」。

> 一句話:**分層架構的依賴方向錯了——最重要的東西(業務)依賴了最不穩定的東西(DB/框架)。** 後面三個變體,做的都是同一件事:**把這個箭頭掉過來。**

---

## Part B · 核心思想:依賴反轉(整章只有這一個重點)

**依賴反轉(DIP,SOLID 的 D)**:不讓業務依賴存儲,而是**業務定義一個接口(它需要什麼),存儲去實現這個接口**。箭頭就反過來了——存儲依賴業務。

```java
// 領域層:業務「聲明它需要什麼」,但不知道怎麼實現。這是一個「埠(port)」
package domain;
public interface OrderRepository {       // ← 接口住在領域層
    Optional<Order> findById(OrderId id);
    void save(Order order);
}

// 基礎設施層:具體實現,它 import 領域層(依賴指向內) ▲▲▲
package infra;
class JpaOrderRepository implements OrderRepository {  // ← 實現住在外層
    // JPA/SQL 的髒活全在這,領域層完全不知道 MySQL 的存在
}
```

```
傳統分層:  Service ──► JPA ──► MySQL          (業務依賴 DB,錯)
依賴反轉:  Service ──► OrderRepository(接口)  ◄── JpaOrderRepository  (DB 依賴業務,對)
                         ↑ 領域層擁有它              ↑ 基礎設施實現它
```

**🔬 內幕:運行時 vs 編譯時**——這是最容易卡住的點。
> 運行時,`Service` 當然還是調到了真的 JPA 代碼;**但在源代碼/編譯層面,領域層不 import 任何基礎設施**。靠誰把 `JpaOrderRepository` 塞進去?**依賴注入(DI 容器 / 構造函數注入)**在啟動時 wiring。所以「反轉」反轉的是**源碼依賴方向**,不是運行時調用方向。
>
> **回報立刻兌現**:測業務邏輯時,塞一個 `InMemoryOrderRepository`(一個 HashMap)進去,**不碰資料庫就能測**。領域層成了一塊**可以脫離框架、脫離 DB 獨立存在和測試**的純業務。

Go/Python 同理,且更輕:Go 慣例「**接受接口、返回結構體**」,接口往往定義在**使用方**的 package;Python 用 `Protocol` 或構造函數傳入依賴。儀式比 Java 少,idea 一樣。

## Part C · 三個變體 = 同一個 idea 的三張畫法

把「依賴反轉 + 業務在中心」這一個 idea,三個人畫了三張圖:

### Hexagonal / Ports & Adapters(Cockburn, 2005)
中心是應用,四周開**埠(port,就是接口)**,外面插**適配器(adapter)**。六邊形只是「畫得下多個埠」,**和 6 沒關係**。關鍵是**兩側對稱**:

```
   主動側(driving)            被動側(driven)
   誰來驅動我                  我去驅動誰
  ┌─────────┐              ┌─────────┐
  │ REST 控制器│──┐      ┌──│ JPA 倉儲  │
  │ CLI       │  │ port │  │ Kafka 發送│
  │ 測試       │──┼─►[應用]◄┼──│ 第三方 ACL│
  └─────────┘  │ 核心  │  └─────────┘
   adapter        埠       埠    adapter
```
> 洞察:**輸入和輸出都走埠**。REST、CLI、測試對核心是平等的「驅動者」,可隨意替換;DB、MQ、外部 API 是平等的「被驅動者」,也可隨意替換。核心對它們**一無所知**。

### Onion 洋蔥(Palermo, 2008)
同一個 idea 畫成**同心圓**,並把「裡面」也分層:

```
   ┌──────────────────────────┐  外:UI / 基礎設施 / 測試
   │   ┌──────────────────┐   │
   │   │  應用服務(用例編排) │   │  依賴只能
   │   │  ┌────────────┐  │   │  「由外向內」▼
   │   │  │ 領域服務     │  │   │
   │   │  │ ┌────────┐ │  │   │
   │   │  │ │領域模型  │ │  │   │  ← 最穩定、最核心
   │   │  │ └────────┘ │  │   │
   │   └──────────────────┘   │
   └──────────────────────────┘
```
> 規矩一條:**依賴只能向內指**。比 Hexagonal 多畫了「核心內部還能再分層」。

### Clean 整潔(Uncle Bob, 2012)
還是同心圓,換套詞:**Entities(企業規則)→ Use Cases(應用規則)→ Interface Adapters(控制器/Presenter/Gateway)→ Frameworks & Drivers(DB/Web/UI)**。一條**依賴規則(The Dependency Rule)**:**源碼依賴只能指向圓心**。比 Onion 多強調了**顯式的 Use Case** 和**跨邊界用 DTO 轉換**。

> 🔬 **三者本質對照**:都是「業務在中心 + 依賴向內 + 用接口隔離外部」。差別只在**詞彙和強調點**,不是架構差異:

| | 提出者/年 | 中心隱喻 | 多強調了什麼 |
|---|---|---|---|
| Hexagonal | Cockburn 2005 | 埠與適配器 | 輸入輸出**對稱**(driving/driven) |
| Onion | Palermo 2008 | 同心圓 | 核心**內部**再分層 |
| Clean | Martin 2012 | 同心圓 + 依賴規則 | 顯式 **Use Case** + 跨邊界 DTO |

**面試只要記**:它們是「依賴反轉、保護領域」的同一個 idea 的不同畫法;真要說差別,Hexagonal 強調對稱的埠、Clean 多了顯式用例和依賴規則的命名。**沒有「哪個更高級」**。

## Part D · 把 `01` 的零件安回架構圖(關鍵連結)

上一章的東西,在這張圖上都有固定位子:

| `01` 的概念 | 在架構圖上是什麼 | 住哪層 |
|---|---|---|
| **Repository(接口)** | 一個**被動埠(driven port)** | 領域層(接口);實現在基礎設施層 |
| **防腐層 ACL** | 一個**被動適配器**,翻譯外部上下文 | 基礎設施層 / 邊界 |
| **REST Controller** | 一個**主動適配器(driving)** | 最外層 |
| **領域模型 / 聚合**(下一批 `02`) | 圓心 | 領域層核心 |
| **應用服務 / Use Case** | 編排:取聚合→調方法→存回 | 應用層 |

> 所以 ch0 的四個高度在這裡交匯:**H2(這章的依賴方向)是「房間格局」,H3(`02`/`03` 的建模)是「房間裡擺什麼家具」。** 你可以有 Clean 的格局 + 貧血的家具,也可以 + 充血的家具——**正交**。這也再次說明:**Clean ≠ DDD**,Clean 管格局,DDD 管家具,常一起用。

## Part E · 取捨:它不是免費的

別被「乾淨」騙了,這套有**實打實的成本**:

| 收益 | 成本 |
|---|---|
| 領域脫離框架/DB,**可獨立測試** | **更多接口、更多間接層**(埠/適配器) |
| 換 DB/框架/外部供應商,只改適配器 | **更多映射**:領域對象 ↔ DTO ↔ DB 實體,層層轉換 |
| 多種交付方式(REST/CLI/MQ)共用核心 | 簡單 CRUD 上這套 = 為了開門造了三道閘 |

**什麼時候值(回扣 ch0 複雜度驅動)**:
- ✅ 核心域、業務複雜、生命週期長、要高可測、有多個交付入口 → **值得**。
- ❌ 支撐域、本質 CRUD、改動少 → **傳統分層就夠,甚至 Active Record(`03` 講)更省**。
- 🔬 生態差異:**Java/Spring 做得最儀式化**(一堆接口 + MapStruct 映射);**Go 做輕量版**(包邊界放接口,少 DTO 層);**Python** 介於之間。別把 Java 的重儀式當成「唯一正確」。

## 面試滿分答法

> **Q:「Clean Architecture 和 Hexagonal 有什麼區別?」**(在篩你是不是只背了名詞)
> 滿分:「**本質是同一個 idea——依賴反轉、業務在中心、用接口隔離外部。** 三張不同的畫法:Hexagonal 強調輸入輸出對稱的埠與適配器;Onion 畫成同心圓並把核心內部分層;Clean 多了顯式的 Use Case 和『源碼依賴只能向內』的依賴規則。**所以不是選哪個的問題,選一個團隊看得懂的詞就行。**」

> **Q:「為什麼要依賴反轉?直接 Service 調 DAO 不好嗎?」**
> 「分層架構讓**最重要的業務依賴了最不穩定的 DB/框架**——測業務要連庫、換存儲要動業務。依賴反轉讓業務只依賴自己定義的接口,DB 反過來實現它;業務層就能脫離框架獨立測試和演進。代價是更多接口和映射,所以**只在核心域這麼做,CRUD 不值得**。」

## 本章小結

- **傳統分層的原罪**:依賴向下指向 DB/框架,業務被存儲焊死(難測、難換、被框架綁架)。
- **唯一重點:依賴反轉**——業務定義接口、基礎設施實現它,源碼依賴反向指進業務;靠 DI 在運行時 wiring。立刻兌現**可獨立測試**。
- **Hexagonal / Onion / Clean = 同一個 idea 的三張畫法**:依賴反轉 + 業務在中心。差別是詞彙與強調點,**沒有高下、不是三選一**。
- **Clean ≠ DDD**:Clean 管「房間格局」(H2 依賴),DDD 管「家具」(H3 建模),**正交、常一起用**。Repository=埠、ACL=適配器,各有定位。
- **有成本**(接口 + 映射 + 間接),**只在核心域值得**,生態上 Java 最重、Go 最輕。
- 下一章(打底批收尾)`03`:**充血 vs 貧血 + Transaction Script**——進到 H3,看「房間裡擺什麼家具」,以及**什麼時候根本不該上 DDD**。
