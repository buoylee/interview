# 第 1 章 · DDD 戰略設計(H1 · 切系統)

> 🔬 戰略設計是 DDD **便宜又高回報的那一半**,也是面試「設計一個 X 系統」時,你劃服務邊界、接 legacy 的理論武器。它不教你怎麼寫 class,教你**系統怎麼切、語言怎麼統一、上下文之間怎麼相處**。
>
> **最常見的露怯**:① 一上來就講聚合、實體(那是戰術,H3),卻從沒做過邊界劃分;② 把「微服務 = 戰略 DDD」劃等號,講不出「限界上下文」到底是什麼、和子域差在哪;③ 接第三方/舊系統時,讓對方的爛模型直接滲進自己代碼,不知道有「防腐層」這個東西。這章把這四個核心武器補齊:**統一語言 → 限界上下文 → 子域 → Context Map**。
>
> 回扣:`system-design/04` 講過「按限界上下文拆服務」的**輕量版**,這裡是**正文**。

---

## Part A · 統一語言(Ubiquitous Language):一切的地基

**問題**:業務專家說「保單」「核保」「批改」,開發說 `PolicyDTO`、`underwriteFlag`、`updateRecord`。中間隔了一層**翻譯**——每次需求溝通都要翻一次,翻錯一次就埋一個 bug,而且這個翻譯**沒寫進代碼,只活在某個人腦子裡**。

**統一語言**就一條規矩:**業務說的詞,直接變成代碼裡的名字。** 業務說「核保」,代碼裡就有 `underwrite()`,不叫 `processCheck()`;業務說「批改」,類別就叫 `Endorsement`,不叫 `PolicyUpdateRequest`。

```java
// ❌ 翻譯過的:讀代碼的人不知道這對應業務什麼動作
policyService.updateRecord(id, flag, data);

// ✅ 統一語言:業務專家來看也懂,溝通零翻譯
policy.endorse(endorsement);   // 「給保單做一次批改」
```

> 🔬 **為什麼這是地基**:統一語言**不是文檔,是活在代碼裡的**。一旦代碼用業務的詞,需求討論、白板、代碼三者用同一套詞彙——翻譯層消失,理解偏差就無處藏身。**而且它的作用域 = 一個限界上下文**(下面講):同一個詞在不同上下文可以是不同意思,這正常,不要強求全公司一個詞典。
>
> **這一步是免費的、回報最高的**。不需要任何框架,只需要你和業務坐下來把詞對齊,然後在代碼裡誠實地用。Go/Python 一樣適用:`func (p *Policy) Endorse(...)`、`policy.endorse(...)`。

## Part B · 限界上下文(Bounded Context):DDD 對行業最大的貢獻

如果這條軸只能記一個詞,就是它。

**核心洞察**:**同一個業務名詞,在不同上下文裡是不同的模型。** 拿電商的「商品(Product)」:

| 上下文 | 「商品」是什麼 | 關心的字段 |
|---|---|---|
| 商品目錄 | 給人看的東西 | 標題、描述、圖片、分類、SEO |
| 庫存 | 一個可扣減的數量 | SKU、倉庫、可用量、鎖定量 |
| 定價 | 一個算價對象 | 成本、折扣規則、會員價 |
| 物流 | 一個要搬的箱子 | 長寬高、重量、易碎 |

**新手的災難**:做一個 `Product` 類滿足所有人——它有 60 個字段,目錄改個圖片要碰到物流關心的重量,沒人敢動。這就是 **God Model**。

**限界上下文的解法**:**每個上下文各有各的 `Product` 模型,互不共享**。它們之間是不同的「世界」,邊界上用明確的契約溝通(API / 事件 / ID 引用)。

```
┌──────────────┐   商品上架事件   ┌──────────────┐
│  商品目錄上下文  │ ───────────────► │   庫存上下文    │
│  Product       │                  │  Product       │
│ {標題,圖片...}  │  (只傳 SKU + 必要 │ {SKU,可用量...} │
└──────────────┘   字段,不傳整個對象)└──────────────┘
   各自的 Product,同名不同模型 —— 這就是「限界」
```

**怎麼找邊界(這是 H1 最難、最值錢的技能)** 🔬:
- **語言的接縫**:當同一個詞**換了一套意思**,那裡就是邊界。「商品」從「有圖片標題」變成「一個數量」,接縫到了。
- **高內聚**:總是一起變化的概念放一起;很少一起變的拆開。
- **團隊/組織邊界**:Conway's Law——一個上下文最好由一個團隊擁有(回扣 H1)。
- **拆錯的信號**(回扣 `system-design/04`):分布式單體(改一個要一起發版)、多個上下文共享一張表(偷偷耦合,假邊界)。

> **限界上下文 ≈ 微服務,但不等於**:一個限界上下文**可以**是一個微服務,也可以只是單體裡的一個模塊(modular monolith)。**「先用模塊邊界,扛不住再拆成服務」**是更穩的路——邊界是邏輯概念,服務是部署概念,別混。面試講得出這句就贏一半人。

## Part C · 子域(Subdomain):問題空間,決定你把好鋼用在哪

**和限界上下文的關係**(高頻面試陷阱)🔬:

| | 子域(Subdomain) | 限界上下文(Bounded Context) |
|---|---|---|
| 屬於 | **問題空間**(業務現實長怎樣) | **解空間**(你怎麼設計軟體) |
| 是什麼 | 業務的一塊領域 | 一個模型生效的邊界 |
| 類比 | 「我們要解決的問題有哪些塊」 | 「我寫的代碼怎麼分塊」 |

理想是一個子域對一個限界上下文,現實常錯位(一個爛 legacy 可能橫跨三個子域)。

**三類子域,決定資源投放**(ch0 預告過,這裡是決策表):

| 子域 | 定義 | 例(電商) | 策略 |
|---|---|---|---|
| **核心域 Core** | 你**賺錢/競爭力**所在,別人抄不走 | 履約調度、動態定價、風控 | **上戰術 DDD,放最強的人,自己造** |
| **支撐域 Supporting** | 必要,但不獨特、沒壁壘 | 後台管理、審批流、商家入駐 | 簡單寫(貧血+CRUD),普通人力 |
| **通用域 Generic** | 誰都要、毫無差異 | 認證、發信、發票、對象存儲 | **直接買 / 用開源,別自己造** |

> **一句話**:**戰術 DDD 的儀式只值得花在核心域。** 把好鋼(資深 + 複雜建模)用在通用域(自己造一套登錄),是資源錯配——這也是「DDD 是工具箱不是宗教」在 H1 的體現。

## Part D · Context Map:上下文之間怎麼相處(含防腐層)

系統一定不只一個上下文,**它們之間的關係**才是大型系統的真實複雜度。DDD 給了 9 種關係模式。別死背,按「**權力結構**」分四組記:

### ① 對等合作(雙方都有話語權)
| 模式 | 意思 | 取捨 |
|---|---|---|
| **Partnership 合作** | 兩個上下文同生共死,協同演進 | 溝通成本高,只在真的綁定時用 |
| **Shared Kernel 共享內核** | 共享一小塊模型/代碼 | **耦合最高,慎用**——改它要兩邊同意 |

### ② 上下游(一方說了算)
| 模式 | 意思 | 取捨 |
|---|---|---|
| **Customer-Supplier 客戶-供應商** | 上游排期會**考慮**下游需求 | 健康的上下游關係 |
| **Conformist 跟隨者** | 下游**沒得談**,只能順從上游模型 | 接大廠 API(微信支付)時的現實 |

### ③ 防護與開放(關鍵)
| 模式 | 意思 | 取捨 |
|---|---|---|
| **Anti-Corruption Layer 防腐層(ACL)** | 下游建一層**翻譯**,把上游的爛模型擋在外面 | **接 legacy / 第三方的標配**,核心防禦武器 |
| **Open Host Service 開放主機服務(OHS)** | 上游提供標準化公共 API,給一堆下游用 | 你是被很多人依賴的上游時用 |
| **Published Language 發布語言(PL)** | 整合用一套公開、規範的語言(如標準 JSON/事件 schema) | 常和 OHS 配對 |

### ④ 放棄與失敗
| 模式 | 意思 | 取捨 |
|---|---|---|
| **Separate Ways 各行其道** | 不整合,各做各的 | 整合成本 > 重複成本時,**故意重複是對的** |
| **Big Ball of Mud 大泥球** | 沒有邊界的一團亂 | 不是選項,是**要識別並隔離**的現實 |

**防腐層到底是什麼(最該掌握的一個)** 🔬:它就是下游邊界上的**一層翻譯適配器**,把外部模型翻成你自己的乾淨模型,**外部的概念一個都不准漏進你的核心**。

```java
// 接一個爛的舊 CRM,它的「客戶」叫 CustInfo,字段是 c_nm / c_sts(誰看得懂)
// 防腐層:唯一知道 CRM 長什麼樣的地方
class CrmCustomerAcl {
    private final LegacyCrmClient crm;
    Customer fetch(CustomerId id) {                 // 對內:吐出你自己的乾淨 Customer
        CustInfo raw = crm.getCustInfo(id.value());  // 對外:髒活只發生在這裡
        return new Customer(                          // 翻譯:c_nm -> name, c_sts -> status
            id, raw.getCNm(), mapStatus(raw.getCSts()));
    }
}
```

> **為什麼非要這層**:沒有它,`c_nm`、`c_sts` 這種爛概念會像藤蔓爬進你整個系統;哪天換掉這個 CRM,你要改幾百個地方。**有了它,外部世界只透過這一個窗口進來**,換供應商只改這一個類。Strangler Fig(絞殺榕,漸進替換 legacy)就是靠 ACL 撐起來的——細節在 `09`。

## 大廠怎麼用(誠實版)

- **Amazon**:Bezos 的「API Mandate」+ two-pizza team——每個小團隊擁有一個服務,**團隊邊界 = 限界上下文 = 服務邊界**(Conway's Law 的主動運用)。但他們講 "service ownership" / "business capability",不掛 DDD 招牌。
- **共同模式**:戰略概念(邊界、統一語言、防腐層)**拿來就用**,因為便宜高回報;戰術那套(聚合、實體)挑著用。
- **一句話**:你面 senior/staff,面試官想聽的「怎麼劃邊界、怎麼接 legacy、團隊怎麼對齊架構」——**全是戰略 DDD**,而不是背值對象定義。

## 面試滿分答法

> **Q:「設計這個系統,你怎麼劃服務邊界?」**
> 骨架:① **先別急著拆**(單體扛得住別拆,回扣 `system-design/04`);② 用**限界上下文找接縫**——「語言換意思的地方就是邊界」,舉商品在目錄/庫存的例子;③ **先模塊邊界、扛不住再拆服務**(邏輯邊界 vs 部署邊界);④ 用 **Conway's Law** 讓團隊和邊界對齊;⑤ 警惕**分布式單體**。

> **Q:「要接一個又舊又爛的第三方系統,怎麼辦?」**
> 一個詞:**防腐層(ACL)**。「我會在邊界建一層翻譯適配器,把它的模型翻成我自己的乾淨模型,外部概念一個都不漏進核心;將來換供應商只改這一層。」再加一句 Strangler Fig 就滿分。

## 本章小結

- **統一語言**:業務的詞直接進代碼,消滅翻譯層;作用域 = 一個限界上下文。**免費、回報最高。**
- **限界上下文**(本軸最重要的詞):同名概念在不同上下文是**不同模型**,別做 God Model;靠**語言接縫**找邊界;**≈ 微服務但不等於**,先模塊邊界再拆服務。
- **子域**(問題空間)分核心/支撐/通用,決定資源投放:**戰術 DDD 只值得花在核心域,通用域直接買**。
- **Context Map** 9 模式按權力結構記四組;最該掌握**防腐層(ACL)**——接 legacy/第三方的核心防禦,外部模型只透過一個窗口進來。
- 大廠普遍用戰略 DDD(常不掛名);面試的邊界題、legacy 題,答案都在這章。
- 下一章 `04`(打底批):**分層 → Hexagonal → Onion → Clean**,H2 依賴方向,看 ACL/Repository 這些「邊界」在架構圖上到底擺哪。
