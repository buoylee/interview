# 應用架構與領域建模 · master 索引

> 面試導向 · 資深 → 架構師視角 · **這是地圖,也是教學**
>
> 你 repo 裡的 `system-design/`(L0–L9)走的是一條軸:**系統「運行時」怎麼不崩**——CAP、共識、複製、存儲選型、韌性、容災、可觀測。這條新主線走**另一條正交軸**:**一個服務「裡面」的 code 和業務怎麼建模、怎麼分層、怎麼演化**。兩條軸是工程師的兩隻手,面試與真實工作都同時考。

---

## 先解開心結:你不是「不會」,是「沒有名字」

你 5 年 Java/Go,其實天天在做這條軸上的事——分層、寫 `@Repository`、塞 service、用 MVC。問題不是不會做,是**做得「無意識」**:

| 角色 | 在這條軸上的要求 | 一句話 |
|---|---|---|
| 一般工程師 | 跟著框架擺 code | 「Spring 叫我這樣放我就這樣放」 |
| **資深工程師(你的目標)** | **知道每個擺法的名字 + why + 取捨** | 「這裡用貧血模型是因為它就是 CRUD,上 DDD 是過度設計」 |
| 架構師 | **擁有「整個系統怎麼建模/分層」的決策** | 「核心域上戰術 DDD、邊界用防腐層接 legacy,這是我定的」 |

> 這條軸面試最致命的露怯:**Clean / Hexagonal / DDD / CQRS 名詞都聽過,但講不清「差在哪、什麼時候用、代價是什麼」**,或把它們當成互斥的「技術選型」(其實很多是搭配關係)。這份地圖就是把每個概念的 **why / 取捨 / 何時別用** 給你補上。

## 全景:四個高度 + 跨切面(這是全書的骨架)

DDD 不是「單一層」的東西,它橫跨好幾個高度。**先記住這張高度圖,後面所有章節都掛在它上面**:

| 高度 | 在問什麼 | 本書章節 | 同層的其他知識 |
|---|---|---|---|
| **H1 切系統** | 系統拆幾塊?團隊怎麼對應? | `01` DDD 戰略設計 | 微服務 business-capability 拆分、Team Topologies、Conway's Law、EDA |
| **H2 依賴方向** | 業務邏輯怎麼跟 DB/框架解耦? | `04` 架構四連、`05` Vertical Slice | 傳統分層、MVC |
| **H3 建模** | 業務規則放哪?物件胖還瘦? | `02` 戰術 DDD、`03` 充血vs貧血 | Transaction Script、Active Record、Table Module |
| **跨切面** | 讀寫怎麼分?狀態怎麼存?怎麼編排? | `06` CQRS、`07` Event Sourcing/Saga | 領域事件、最終一致 |
| **落地流程** | 怎麼帶團隊真的用起來? | `08` EventStorming、`09` 漸進引入、`10` 團隊規範 | Strangler Fig、防腐層 |

**三個高頻誤解(全書反覆校正)** 🔬:
1. **Clean / Hexagonal ≠ DDD 的替代品**——它們管 H2「依賴方向」,DDD 管 H1+H3,**經常一起用**。
2. **真正和戰術 DDD 對立的是 Transaction Script / 貧血模型**(H3),這才是「另一個方案」,而且是多數團隊的實際默認。
3. **CQRS / Event Sourcing 可以脫離 DDD 單用**,只是氣味相投常被綁一起講。

## 怎麼讀 / 格式約定

- **閱讀順序 = 編號順序**(00 全景打底 → H1 切系統 → H2 依賴 → H3 建模 → 跨切面 → 落地)。
- 每章格式固定,沿用 `system-design/` 房子:**🔬 黑盒內幕(底層到底發生什麼)+ 取捨表 + 決策幾問 + 面試滿分答法 + Java/Go/Python 錨點 + 章末問答(只複習,不放新知)**。
- **語言錨點生態平衡**:Java 為主(你最熟、OOP 重,DDD 戰術層最自然)+ Go 對照(反 OOP 儀式,戰術層常退化成 service,這個視角很有教學價值)+ Python 點到(你面試在轉)。
