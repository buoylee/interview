# 轮询（Polling）：异步等待的朴素方案与规模化优化

更新时间：2026-04-22

目标读者：在做"前端等待异步任务完成"、"订单/任务/报告状态查询"类需求的开发者。读完你应该能：

- 破除"正经公司都用长连接优雅等待"的 myth
- 回答面试 / 设计评审时"为什么不用 WebSocket"
- 给一个朴素轮询做规模化优化，不慌

轮询不是协议，是**基于 HTTP 做异步等待**的一种应用模式。本文只讲这个模式：怎么用、怎么优化、什么时候别用。

---

## 1. 起点：一个常见误解

很多人以为"正经公司的支付/订单/任务等待页用的是长连接（WebSocket/SSE）做优雅推送，轮询是 low 的/玩具项目才用的"。

事实正好反过来：

| 公司 | 场景 | 技术 |
|-----|-----|-----|
| **Stripe** | Payment intent 状态查询 | 短轮询（官方文档推荐） |
| **AWS** | 所有 `Describe*`、长任务 API | 轮询（`GetOperation`，5-10s 间隔） |
| **Google Cloud** | Long-Running Operation（Vertex AI、Cloud Build） | 轮询（LRO 官方协议） |
| **GitHub** | Actions workflow、PR check 状态 | 轮询（devtools 可见） |
| **支付宝 / 微信支付** | 商户端查订单 | 短轮询 + notify webhook 双通道 |
| **Shopify** | Bulk operation / fulfillment 状态 | 轮询 |

这些都是百万~亿级用户系统。**如果轮询不能 scale，它们早挂了。**

WebSocket 在这些系统里主要承担：IM、协作编辑、实时行情、司机/骑手位置 —— 真正需要 **双向 + 低延迟 + 高频** 的场景。支付等待、报告生成、任务完成这类"一次状态变更"场景，几乎清一色轮询。

**核心原因**：轮询是 **stateless** 的，长连接是 **stateful** 的。在分布式系统里，stateless 永远更容易水平扩展、更容易容错、更容易运维。

## 2. 概念区分：短轮询 vs 长轮询

两者都叫"轮询"，但机制完全不同。

### 2.1 短轮询（Short Polling）

客户端每隔 N 秒主动发一个请求查状态。

```
Client            Server
  │ GET /status?id=X │
  ├────────────────→ │
  │ ← 200 {pending} │
  │                  │
  │  (等 N 秒)       │
  │                  │
  │ GET /status?id=X │
  ├────────────────→ │
  │ ← 200 {pending} │
  │                  │
  │  (等 N 秒)       │
  │                  │
  │ GET /status?id=X │
  ├────────────────→ │
  │ ← 200 {ready}   │
  │                  │
```

- 每个请求都是完整、独立的 HTTP 往返
- 服务端完全无状态
- 延迟 = 最坏情况下 1 个轮询间隔
- **这是本文的主角**

### 2.2 长轮询（Long Polling）

客户端发一个请求，服务端**故意 hold 住**，直到有结果或超时。

```
Client                     Server
  │ GET /status?id=X&wait=30 │
  ├────────────────────────→ │
  │                           │ (服务端订阅 pub/sub
  │                           │  或内部 polling DB，
  │    ...  30s 内等待  ...    │  等状态变化)
  │                           │
  │ ← 200 {ready}            │
  │                           │
  │ GET /status?id=X&wait=30 │ ← 客户端立即再发下一次
  ├────────────────────────→ │
```

- 单次请求 TTL 长（10-60s）
- 服务端有状态（至少在请求存活期间）
- 延迟更低（事件一到就返回）
- 客户端代码反而更简单（一次请求等结果）
- 成本转嫁到服务端：连接占用、需要信号机制（Redis pubsub / condition variable）

### 2.3 对比

| 维度 | 短轮询 | 长轮询 |
|-----|-------|-------|
| 服务端状态 | 无 | 有（连接保持） |
| 单次请求时长 | ms 级 | 最多几十秒 |
| 延迟 | 取决于间隔 | 近实时 |
| 实现复杂度 | 简单 | 中等（需 pubsub） |
| LB / 代理友好度 | ✅ | ⚠️ idle timeout |
| 水平扩展 | ✅ 完全 stateless | 要么 sticky session，要么全局 pubsub |
| 失败恢复 | 下一次自然恢复 | 需手动处理断连 |

**何时选长轮询**：当短轮询间隔必须 <500ms，但又不值得上 SSE/WebSocket 时。实践中很少见，80% 场景短轮询够。

## 3. 成本分析：为什么轮询比长连接便宜

### 3.1 直觉计算

假设 **10 万并发等待用户**，短轮询 2s 一次：

```
QPS = 100,000 / 2 = 50,000
```

看起来吓人。但每个 poll：
- 请求体 ~200 B
- Redis 缓存查 → <1ms
- 响应体 ~500 B
- HTTP/2 连接复用，零 TCP 握手

### 3.2 等价 WebSocket 的成本

同样 10 万用户用 WebSocket 常驻：

- 内核 fd × 100,000
- 每连接 ~8 KB 最小内存 = 800 MB（业务数据还没算）
- 心跳包每 30s 一次 ≈ 3333 QPS（小但真实）
- **服务重启 → 10 万连接同时重连风暴**
- 跨 server 的用户要么 sticky session（破坏弹性），要么全局 pubsub

### 3.3 真正会"无限变大"的是谁？

| 方案 | 会无限变大的是 |
|-----|--------------|
| 短轮询 | QPS（但有客户端数上限，可 cache，可限流） |
| 长轮询 | 长连接数（= 用户数） |
| WebSocket | 长连接数（= 用户数） |
| SSE | 长连接数（= 用户数） |

**用户数上限就是长连接数上限，每个连接吃内存 + fd，这是硬成本。** QPS 是软成本，可压可缓。

所以"担心轮询压力大"其实搞反了对象：**大规模系统偏爱轮询，正是因为 stateless 才能线性扩容**。

### 3.4 Slack 的公开案例

Slack 曾公开：维护几百万 WebSocket 连接需要**专门的 connection gateway** 层，与业务服务隔离。原因：

- 业务逻辑迭代快、频繁重启 → 不能让几百万连接每次都断
- Gateway 层独立，只负责维持连接、转发消息、心跳
- 业务层保持 stateless，照常扩缩容

也就是说：**支撑长连接本身就是一个独立子系统**，不是免费的。业务团队之所以没感觉到这份成本，是因为它被 gateway 团队吸收了。

轮询没有这个代价：nginx + app server + Redis，三件套搞定。

## 4. 规模化优化：7 个技巧

朴素轮询在 MVP 阶段完全够用；真碰到流量问题时按以下顺序加优化，**绝大多数场景只需前 3 条**。

### 4.1 分层缓存

```
Client → CDN → Edge cache → Redis → DB
```

状态类查询（订单状态、任务进度）是极度可缓存的：
- "pending" 状态可缓存 1-2s
- Redis 命中率 > 99%
- **99% 的轮询压根到不了 DB**

### 4.2 指数退避（客户端实现）

```
阶段         间隔     次数      累计耗时
前 5s        1s      5 次      5s
5-15s        2s      5 次      15s
15-30s       5s      3 次      30s
30s+         10s     ...       超时上限
```

朴素"每 2s 一次"在 5 分钟内 150 次；指数退避 ~40 次。长尾变瘦。

大部分任务在前 5-10s 内完成，高频轮询应该集中在"有希望"的窗口，之后就该退。

### 4.3 Conditional GET（ETag / If-None-Match）

```http
GET /order/123 HTTP/1.1
If-None-Match: "v17"

HTTP/1.1 304 Not Modified
(响应体 0 字节)
```

状态没变 → 304，几乎零流量。**真实 egress 远小于表面 QPS**。

配合 CDN 边缘节点，304 可以在边缘返回，不碰源站。

### 4.4 `Retry-After` 响应头（服务端反向调度）

```http
HTTP/1.1 202 Accepted
Retry-After: 3
```

服务端告诉客户端："下次 3 秒后再来"。过载时延长 Retry-After，是**服务端控制节奏**，不是客户端。客户端实现时尊重这个头就够了。

### 4.5 Rate Limit

- 单用户单会话 max N 次/分钟
- 超出直接 429 + `Retry-After`

防止恶意/错误循环打爆服务。

### 4.6 冷热分离

业务层把"轮询目标"分池：

| 池 | 条件 | 轮询间隔 |
|---|-----|---------|
| 热池 | 刚创建 / 有进展 | 1-2s |
| 温池 | 5 分钟无进展 | 10s |
| 冷池 | 1 小时无进展 | 30s，或标记超时停止 |

### 4.7 极端场景上长轮询兜底

如果真要压榨延迟，应用层做长轮询：

```
Client: GET /order/status?wait=30
Server: 订阅 Redis pubsub "order:123"
        - 30s 内收到事件 → 立即返回
        - 30s 无事件 → 返回当前状态
Client: 拿到响应后立即再发一次
```

本质上是**把客户端的轮询移到服务端**。代价：

- 服务端要做 pubsub 信号分发
- LB / 代理的 idle timeout 要调高于 wait 值
- 连接状态管理（客户端断连时释放 pubsub 订阅）

**只在"短轮询间隔必须 <500ms" 时才值得**。金融撮合、高频交易这类场景才见。

## 5. 什么时候别用轮询

轮询不是万能的。以下场景换协议：

| 场景特征 | 推荐 | 为什么 |
|---------|-----|-------|
| 双向、低延迟（<100ms） | WebSocket | 轮询延迟下限太高 |
| 一次请求多个事件流（LLM token） | SSE | 多次轮询开销大、延迟高 |
| 服务器主动推送、无用户触发 | SSE / WebSocket | 轮询需要用户在线才有意义 |
| 离线通知 | Web Push / FCM | 浏览器关了还要送达 |
| 高频数据（>1 Hz） | WebSocket / SSE | 轮询 QPS 爆炸 |
| 广播模式（一对多） | WebSocket + pubsub | 轮询每个用户独立查询 |

**经验法则**：默认轮询；能证明"用户体验上 1s 延迟不可接受"再升级。

## 6. 决策框架

```
是双向通信吗？
├─ 是 → WebSocket（别犹豫）
└─ 否
   ├─ 单次请求多个事件流？
   │  ├─ 是 → SSE
   │  └─ 否
   │     ├─ 延迟要求 <500ms？
   │     │  ├─ 是 → 长轮询 或 SSE
   │     │  └─ 否 → 短轮询 ✅
```

默认路径：**短轮询**。升级的阻力应该大于不升级的阻力，因为每升级一层都要多一套基础设施（重连、心跳、信号、鉴权、监控）。

## 7. 双通道架构（支付场景的经典模式）

**核心洞察**：真实支付系统的异步等待**不是**用长连接做的，是"双通道"。

```
┌──────────────────┐       ┌──────────────────┐
│  原页面（待付款）  │       │   新窗口（收银台） │
│                  │       │                  │
│  轮询 /order/     │       │ 用户完成支付动作   │
│  status 每 2-3s  │       │         ↓         │
│         ↓        │       │ 收银台跳转        │
│  status=paid?   │       │ /payment/done    │
│  → 跳转成功页    │       │ (窗口自动关闭)     │
└──────────────────┘       └──────────────────┘
         ↑
         │
    ┌────┴─────┐
    │ 支付宝/微信/Stripe │
    │ 调 notify_url     │  ← webhook（server → server）
    │ 更新我方后端状态   │
    └──────────────────┘
```

### 两条通道的分工

| 通道 | 角色 | 技术 | 特征 |
|-----|-----|-----|------|
| **权威通道** | 决定状态真相 | Webhook（第三方 → 我方后端） | 准确、可重试、不面向用户 |
| **观察通道** | 让前端感知变化 | 前端短轮询 /order/status | 最终一致、用户可见 |

### 为什么前端不能直接订阅 webhook

- 用户随时可能关掉收银台窗口 → postMessage 不可靠
- 用户网络抖 / 切 Tab / 手机锁屏 → 长连接挂掉
- **短轮询每一次都是独立快照**，任何时候恢复都对

支付金额涉及真金白银，**稳定性 > 延迟 > 优雅**，所以选轮询作为观察通道。

### 实战原则（从支付系统偷来）

1. **后端状态是唯一真相源**，前端任何时候刷新、重连都能恢复
2. **成功 redirect 参数不可信**（可伪造），必须二次调后端确认
3. **多级 timeout 降级 UI**：15s → "请稍等..." / 30s → "点击刷新" / 60s → "标记超时"
4. **渐进渲染**：能先展示的立即展示（比如已完成的子任务），剩下的继续轮询

## 8. 面试 / 评审常见问答

**Q：为什么不用 WebSocket 做等待页？**

A：等待页是**单次状态变更**场景，不是"双向 + 高频 + 低延迟"。上 WebSocket 要引入连接管理、心跳、重连、鉴权二次适配、代理/CDN 兼容等一整套复杂度，收益只是省掉几次轮询的延迟（1-2s 级别）。性价比不合。

**Q：短轮询会不会把数据库打爆？**

A：不会，原因是分层缓存。99% 的轮询在 Redis 层就返回了，数据库只在状态真正变化时才被写入，读取可以完全走缓存。真正的压力是 Redis QPS，而 Redis 单实例十几万 QPS 起步，远未到瓶颈。如果真到了瓶颈，上 Conditional GET + CDN 边缘缓存，或者走冷热分离。

**Q：长轮询和 SSE 的区别是什么？**

A：长轮询是"单次请求，一次响应，客户端再发下一次"；SSE 是"单次请求，多次响应（事件流），保持连接直到结束"。SSE 更适合 **单次任务产生多个事件**（如 LLM token 流），长轮询适合 **单次任务只产生一个最终结果**（如订单状态）。

**Q：为什么大公司的支付 API 文档里都推荐轮询而不是 webhook？**

A：Webhook 是**权威通道**（server → server），前端用不了 —— 浏览器没法被 webhook 主动调用。所以前端必须走轮询观察后端状态。Webhook 和轮询不是二选一，是互补关系：webhook 让后端知道真相，轮询让前端知道真相。

**Q：你们 MVP 上这套，流量真上来了怎么办？**

A：按本文 §4 的顺序逐条上：缓存 → 指数退避 → ETag → Retry-After → rate limit → 冷热分离 → 长轮询。实际经验是前 3 条就能覆盖 95% 的流量增长场景，不用一次全做。
