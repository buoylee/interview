## 0. 导读：这是一条纵切线

`network/` 里别的文档**按层切**——每篇讲自己那层的协议或机制。这篇相反，它**纵着切一刀**：把一个请求穿过所有层的完整路径串成一条线，只干两件别人没干的事——

1. 把**整条链路画成一张图**，并给一套**二分逐跳定位**的纪律；
2. 补一张**「症状 → 哪一跳」鉴别诊断表**。

**命令怎么敲、字段怎么读、TCP 状态机、怎么抓包**——这些已经写透了，本文一律**指过去不重写**。你要的「排查能力」= 这条线 + 那些散件，本文负责把它们接起来。

## 1. 全链路图：一个 HTTPS 请求经过的每一跳

以浏览器/客户端 `GET https://api.example.com/x` 为例。**先看清这条线有多长——正因为长，才不能从头一跳跳查。**

```
 ┌─ 本机（你这端）──────────────────────────────────────────────┐
 │ ① 应用 / 缓存层    强缓存命中 / HSTS / 连接复用(keep-alive)    │  ← 命中就根本不出网
 │ ② DNS 解析         host → IP：stub→本地缓存→hosts→resolv.conf │  L7(但出网第一步)
 │ ③ 路由+ARP+出网    查路由表选网关 → ARP 拿 MAC → 本机防火墙    │  L3/L2
 └───────────────────────────┬──────────────────────────────────┘
                             │
 ┌─ 网络路径（中间）─────────▼──────────────────────────────────┐
 │ ④ 公网/内网多跳     路由器逐跳转发；丢包/延迟/MTU 在这发生     │  L3
 └───────────────────────────┬──────────────────────────────────┘
                             │
 ┌─ 对端（服务那侧）─────────▼──────────────────────────────────┐
 │ ⑤ TCP 握手         SYN/SYN-ACK/ACK 打到监听端口；队列         │  L4
 │ ⑥ TLS 握手         ClientHello(SNI/ALPN)→证书链→密钥协商      │  L5/6
 │ ⑦ (CDN 边缘→回源)  公网才有：边缘命中直接回；未命中回源       │  （子链路重来一遍 ②~⑥）
 │ ⑧ LB / 反向代理    Nginx/网关：自己的超时·健康检查·连接池      │  L4 或 L7
 │ ⑨ 应用进程         accept → 业务处理（首字节就在这产生）       │  L7
 │ ⑩ 下游 DB/RPC/缓存 应用又当客户端，发起一条新链路（②~⑥重来）  │  L7→L4…
 └───────────────────────────┬──────────────────────────────────┘
                             │
 ⑪ 回程：响应数据沿 TCP 流回来（大响应才暴露的坑：MTU 黑洞/带宽/反代缓冲）
```

**三段记忆**：本机（①②③）→ 网络路径（④）→ 对端（⑤~⑩）→ 回程（⑪）。
每一跳都可能是「网络问题」的真凶，而**症状常常长得一样（都叫"慢"或"连不上"）**——所以需要探针把它们区分开。

> ⑦⑩ 是「链路里套链路」：CDN 回源、应用调下游，本质都是**在某一跳里又完整跑一遍 ②~⑥**。排查时把它当成一条独立的子链路，同一套方法递归用。

## 2. 二分逐跳定位法（这才是"能力"）

链路有 ~11 跳。新手的错误是**从①开始一跳一跳往下试**——O(N)、容易在某跳钻牛角尖、还把现场搅乱。资深的做法是**二分**：每个探针在某跳上问一句「**到这还通 / 还快吗？**」，答案直接把嫌疑范围**砍掉一半**。

### 2.1 一个探针 = 一刀切

- 探针**通** → 故障在这跳**之后**（更靠对端），前半段全部排除。
- 探针**不通 / 慢** → 故障在这跳**及之前**，后半段先不用看。

几刀下去，O(log N)，故障被夹到某一跳。**别在没切到的那半段花时间。**

### 2.2 三个万能切割点（先打这三刀）

为什么是这三个：它们都**一条命令可观测**，且每个都干净地切在「网络/下层」和「应用/上层」的接缝上，不需要先把别的搞清楚。

| 刀 | 问什么 | 探针 | 通了排除掉 | 不通指向 |
|---|---|---|---|---|
| **第一刀:能解析吗** | host→IP 对不对 | `dig +short` / `getent hosts`（[03 §3](../cli-toolbox/03-network-triage.md)） | ② DNS 整段 | DNS 配置 / hosts / resolv.conf / ndots |
| **第二刀:能建连吗** | TCP 端口通不通（不管应用） | `nc -zv host port`（[03 §2](../cli-toolbox/03-network-triage.md)） | ③④⑤ 网络路径+端口 | 路由 / 防火墙 / 端口没监听 / 队列满 |
| **第三刀:应用回什么** | L7 真实响应 | `curl -v` / `curl -w`（[03 §4](../cli-toolbox/03-network-triage.md)） | ⑥~⑩ 应用+后端 | TLS / 5xx / 后端慢 |

打完这三刀，故障已经被夹进「DNS 段 / 网络段 / 应用段」之一，再在那一段里继续二分。**这就是为什么 `ping` 通也不能停**——`ping` 只切了 L3 那一刀，L4/L7 还没问。

### 2.3 还有一个维度：空间二分（两端对抓）

时间维度（沿链路前后）切完还定不下，就切**空间维度**：同一个探针在**客户端**跑一遍、上**服务端**再跑一遍。

- 客户端发不出，服务端没收到 → 卡在中间（④ 网络 / 防火墙）。
- 客户端发出了，服务端收到了却没回 → 卡在对端（⑤~⑩）。
- 两端抓包对齐（`tcpdump -nn`，[03 §5](../cli-toolbox/03-network-triage.md)）是这一步的终极裁判，**但放最后**——能用 `nc`/`curl` 夹出来就别抓包。

## 3. 每跳：判定探针 + 坏掉的样子 + 和邻跳怎么区分

锁定大致区段后，照这张表在具体跳上确认。**「探针」列只说在这跳用它判什么 + 看哪个信号**；命令完整用法点链接。

| 跳 | 这跳干嘛 | 判定探针（看什么信号） | 坏了的典型样子 | 和邻跳怎么区分 |
|---|---|---|---|---|
| ① 缓存/复用 | 命中就不出网 | 抓不到包 / `curl` 正常但浏览器异常 | 改了后端但还看到旧结果 | 清缓存/换 `curl` 复现就排除浏览器层 |
| ② DNS | host→IP | `dig +short`（有没有 A 记录）、`getent`（系统真实解析） | 解不出、解到旧/错 IP | **`dig` 通但 `getent` 不通** = hosts/nsswitch 拦截（[03 §3](../cli-toolbox/03-network-triage.md)） |
| ③ 路由/出网 | 选网关+ARP+本机防火墙 | `ip route get IP`、本机 `iptables -L OUTPUT` | 包根本没出门 | 客户端抓包看有没有 SYN 发出 |
| ④ 网络路径 | 多跳转发 | `mtr host`（哪一跳开始丢包/变慢） | 中间丢包、某跳 RTT 飙高 | 某一跳起持续丢包 = 这跳/这段（[08 丢包延迟](../performance-tuning-roadmap/08-network-io/03-packet-loss-latency.md)） |
| ⑤ TCP 握手 | SYN/SYN-ACK/ACK + 队列 | `nc -zv`：`succeeded` / `refused` / `timeout` | refused=没监听；timeout=丢包；偶发=队列满 | **refused vs timeout** 是关键分诊（[06 §2.8](../linux-handson/06-networking/README.md)） |
| ⑥ TLS 握手 | SNI/证书/密钥 | `curl -v`（握手报错）、`openssl s_client -connect host:443 -servername host` | 证书过期/域名不符/CA 不信/版本不匹配 | TCP 已连上(⑤通)但握手失败 = 在 TLS（[08 TLS 调试](../performance-tuning-roadmap/08-network-io/04-tls-debugging.md)） |
| ⑦ CDN | 边缘命中/回源 | 响应头 `X-Cache` / `Age` / `CF-Ray` 之类 | 只有未命中(回源)时慢/错 | 命中正常、回源出错 = 源站或回源链路 |
| ⑧ LB/反代 | 转发+自己的超时 | `curl -v` 看是谁回的 5xx；反代日志 | 502/503/504（见 §5） | 502/504 多半是反代↔上游，不是客户端↔反代（[../nginx/](../nginx/)、[../gateway/](../gateway/)） |
| ⑨ 应用 | 业务处理 | `curl -w '%{time_starttransfer}'` | 首字节慢、5xx | **首字节慢但传输快 = 后端处理慢**（线程池/GC/锁） |
| ⑩ 下游 | DB/RPC/缓存 | 应用侧 `ss -tnp` 看到下游连接堆积/CLOSE_WAIT | 连接池耗尽、下游慢拖垮上游 | 一条新子链路，递归用 ②~⑥（[06 CLOSE_WAIT](../linux-handson/06-networking/README.md)） |
| ⑪ 回程 | 响应流回来 | 大请求复现 + 抓包看有没有卡在某个 seq | 大响应卡死、小请求正常（见 §6） | 首字节到了、后续断 = 回程/传输层 |

> 慢的总账：`curl -w 'dns=%{time_namelookup} tcp=%{time_connect} tls=%{time_appconnect} first=%{time_starttransfer} total=%{time_total}'` 一行就把 ②⑤⑥⑨ 各段耗时拆开——**这是"慢在哪一跳"最快的一刀**（[03 §4](../cli-toolbox/03-network-triage.md)）。

## 4. 症状 → 哪一跳：鉴别诊断决策树

现实里你拿到的不是"哪跳坏了"，而是一句模糊的**症状**。这张表把症状反向映射到最可能的跳 + 第一个该打的探针。

| 症状 | 最可能的跳 | 第一个探针 | 深挖 |
|---|---|---|---|
| **全部请求都连不上**（connect timeout） | ④⑤ 网络/端口 | `nc -zv`：refused or timeout？ | refused→服务/绑错；timeout→防火墙 DROP（[06 §2.8](../linux-handson/06-networking/README.md)） |
| **全部都能成但都慢** | 看哪段 | `curl -w` 拆段：哪段大 | dns→②；tcp→④；tls→⑥；first→⑨（[03 §4](../cli-toolbox/03-network-triage.md)） |
| **偶发超时/失败**（大多数 OK） | ⑤ 队列满 / ⑧ 某个坏后端 / ⑩ 池耗尽 | `ss -lnt` 看 Recv-Q 是否逼近 Send-Q | 全连接队列满（[06 §2.6](../linux-handson/06-networking/README.md)）；LB 摘除坏节点 |
| **只有部分用户/某地区** | ② 解到不同 IP / ④ 某条公网路径 / ⑧ 某个后端 | 让对方 `dig +short` 看解到哪、`mtr` 看路径 | 地理调度/CDN 节点差异；某后端异常（[08 丢包](../performance-tuning-roadmap/08-network-io/03-packet-loss-latency.md)） |
| **时快时慢、抖动** | ④ 丢包重传 / ⑨ GC / ⑩ 下游抖 | `mtr` 看丢包率 + 应用 GC 日志 | 拥塞/重传 vs 应用停顿，两边都要看 |
| **首字节慢、传输快** | ⑨ 后端处理慢 | `curl -w '%{time_starttransfer}'` | 线程池/慢查询/锁（[08 连接问题](../performance-tuning-roadmap/08-network-io/02-connection-issues.md)，read timeout） |
| **首字节快、传输卡/大响应才卡** | ⑪ MTU 黑洞 / 带宽 / 缓冲 | 大小请求对比 + 抓包看停在哪 | 见 §6（[08 丢包延迟](../performance-tuning-roadmap/08-network-io/03-packet-loss-latency.md)，MTU） |
| **502 / 503 / 504** | ⑧ 反代↔上游 | `curl -v` 看谁回的 + 反代日志 | 502=上游连不上/RST；503=过载·熔断·无健康后端；504=上游超时（[http.md](./http.md)、[../nginx/](../nginx/)） |
| **TLS 报错家族** | ⑥ TLS | `openssl s_client -servername host` | 证书过期/域名不符(SNI)/CA 不信/版本·cipher（[08 TLS](../performance-tuning-roadmap/08-network-io/04-tls-debugging.md)） |
| **间歇 connection reset（RST）** | 对端主动 RST | 抓包确认是谁发的 RST | 超时/空闲被 LB 关、上游崩、SO_LINGER（[basic.md §13](./basic.md)） |

> 用法：**先用这张表定"该往哪跳看"，再回 §2 二分确认，再去 §3/深挖文档查命令**。三张表是一条流水线，不是各看各的。

## 5. 502 / 503 / 504：三个最常被混的状态码

它们都来自 ⑧ 反向代理/LB，但指向不同的跳，**别一律当"服务挂了"**：

- **502 Bad Gateway**：反代**连上游失败**或上游**回了非法响应/RST**。→ 上游进程没起、绑错地址、刚崩、或 ⑤ 层连不上。
- **503 Service Unavailable**：反代**主动拒绝**——过载保护、熔断打开、或**后端池里没有健康节点**（健康检查全挂）。→ 容量/熔断/健康检查问题，不是单跳连通性。
- **504 Gateway Timeout**：反代**连上了上游，但在 `proxy_read_timeout` 内没等到响应**。→ ⑨ 应用处理太慢，或 ⑩ 下游拖死。**和客户端自己的 read timeout 是两码事**——看是谁先超时。

> 排查起手：`curl -v` 看 `Server:`/`Via:` 头确认是哪一层回的码 + 翻**反向代理**日志（它同时看得见"客户端←→自己"和"自己←→上游"两段）。

## 6. 回程：最常被忘的一跳

请求发出去了、服务也处理了、首字节也回来了——然后**响应传到一半卡死**。这类问题排查时容易只盯"去程"，但根因在 ⑪：

- **MTU / 分片黑洞**：典型特征是 **小请求成功、大响应（或大上传）卡死**。路径上某跳 MTU 偏小又丢弃了"不可分片"的大包，ICMP 又被防火墙吞了 → 发送方一直重传同一段。→ [08 丢包延迟](../performance-tuning-roadmap/08-network-io/03-packet-loss-latency.md)（MTU/PMTUD）。
- **非对称路由**：去程和回程走不同路径，某一方向单边丢包/限速，去程探针全绿但体验差。
- **反代 read timeout 早于上游**：上游还在吐数据，反代先超时切断 → 客户端拿到截断的响应。调 `proxy_read_timeout`（[../nginx/](../nginx/)）。
- **出口带宽 / 接收窗口**：大响应被带宽或对端窗口限速，`time_starttransfer` 正常但 `time_total` 大。

> 判定要点：**首字节到了 vs 没到**把问题劈成两半——没到看 ⑤~⑨，到了却断在后面看 ⑪。

## 7. 公网 vs 内网：同一条线，切点不同

你不确定在意公网还是内网——**不用选，链路是同一条**，只是某些跳多出东西、坏法和深挖文档不同：

| 跳 | 公网访问多出来的 | 内网 / K8s 多出来的 | 坏了去哪查 |
|---|---|---|---|
| ② DNS | 递归→权威、TTL、地理调度 | CoreDNS + **ndots:5** + search 域（非 FQDN 多次查询） | [06 §2.7](../linux-handson/06-networking/README.md)、[../service-governance-on-k8s/](../service-governance-on-k8s/) |
| ④ 路径 | 运营商多跳、公网拥塞 | CNI overlay、**ClusterIP→iptables/IPVS DNAT** | `mtr`；[../cloud-native/](../cloud-native/) |
| ⑦/⑧ 入口 | CDN 边缘→回源、WAF | Ingress/Gateway、**sidecar(Envoy) 拦截所有进出**、mTLS | [../gateway/](../gateway/)、[../service-governance-on-k8s/](../service-governance-on-k8s/) |
| ① 客户端 | 浏览器缓存、HSTS、预连接 | 服务间多为长连接池/HTTP2，复用度更高 | [06 CLOSE_WAIT/池](../linux-handson/06-networking/README.md) |

> 内网最容易踩的两个独有坑：**ndots 把短域名拼出一堆失败查询**（DNS 莫名慢），**sidecar 把所有流量劫持**（少一跳没算进去，抓包发现包先进了 Envoy）。

**一句话记忆点**：
> 一个 URL 请求是一条 ~11 跳的链路（本机→网络→对端→回程）；排查不要从头线性猜，要**二分**——先用「能解析 / 能建连 / 应用回什么」三刀把故障夹进 DNS 段、网络段、应用段，再在那段里继续切；拿到模糊症状先查「症状→哪一跳」表反推切点；`refused` 是没监听、`timeout` 是没回应、`502/503/504` 分别是上游连不上/没健康后端/上游超时，命令和抓包统统有现成文档。

**面试高频追问**：

- **「访问一个网址慢，你怎么查？」** → `curl -w` 拆 DNS/TCP/TLS/首字节四段，哪段大查哪跳；别空谈"可能是网络"。
- **「连不上一个服务，第一步做什么？」** → 不是抓包，是 `nc -zv` 分 refused（端口/服务）还是 timeout（防火墙/路由），再决定往哪切。
- **「只有部分用户访问异常？」** → 先怀疑 ② DNS 地理调度解到不同 IP、或 ⑧ 某个后端节点坏，让对方 `dig`+`mtr` 把"他的链路"和"你的链路"对齐。
