# 实时通信协议选型与排障总览：SSE / WebSocket / WebRTC

更新时间：2026-04-17

这份文档不重复讲协议本身（请分别看 [`sse.md`](./sse.md) / [`websocket.md`](./websocket.md) / [`webrtc.md`](./webrtc.md)），只回答两个问题：

- **选型**：新需求落到哪个协议（或组合）最合理
- **排障**：线上出问题时，第一件事看什么，第二件事看什么

把这篇当索引用：遇到问题先来这里定位"去哪个文档的哪节"。

## 1. 一分钟决策树

```
你要做的是什么？
│
├── 只从服务端推给客户端（单向）
│   │
│   ├── 浏览器主导，不需要二进制      → SSE
│   │   （例子：LLM token 流、通知、进度、实时行情）
│   │
│   └── 需要二进制 / 要客户端也发命令 → WebSocket
│
├── 服务端↔客户端双向消息（不是音视频）
│   │
│   ├── 文本/JSON 即可，中等频率       → WebSocket
│   │   （例子：IM、协同编辑、交易撮合、游戏大厅）
│   │
│   └── 极低延迟 + 可允许丢包（UDP 语义）→ WebRTC DataChannel
│       （例子：联机 FPS、实时白板笔触）
│
├── 需要音视频
│   │
│   ├── 直播一对多、延迟可 3–10s   → HLS / DASH（不在本系列）
│   │
│   ├── 互动通话 < 500ms            → WebRTC
│   │   （信令配 WebSocket；会议用 SFU）
│   │
│   └── 低延迟直播（1s 内）+ 互动 → WebRTC SFU 或 WebRTC→HLS 桥接
│
└── 不确定 / 原型阶段
    → 从 SSE 或 WebSocket 起步，够用再升级。别一上来就上 WebRTC。
```

## 2. 八维对比

| 维度 | SSE | WebSocket | WebRTC |
| --- | --- | --- | --- |
| 方向 | S → C | 双向 | 双向（P2P）|
| 底层 | HTTP/1.1 chunked 或 HTTP/2 DATA | TCP（握手走 HTTP 101）| UDP + SRTP / SCTP over DTLS |
| 经过服务器 | 是 | 是 | 通常否；TURN 时经服务器中继 |
| 典型延迟 | 中 | 中 | 最低 |
| 二进制 | ❌（要 Base64）| ✅ | ✅ |
| 自动重连 | ✅ 浏览器内置 | ❌ 要自己写 | ICE restart 机制，非傻瓜 |
| 消息续传 | ✅ `Last-Event-ID` | ❌ | ❌（媒体用 FEC/NACK/RTX）|
| 内置音视频编解码 / 采集 | ❌ | ❌ | ✅ |
| NAT 穿透 | 不需要 | 不需要 | **核心问题**（STUN/TURN）|
| 浏览器 API | `EventSource` | `WebSocket` | `RTCPeerConnection` + 一堆 |
| 运维复杂度 | ⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ |
| 成本主要在哪 | 服务端长连接数 | 服务端长连接数 | 信令 + TURN 带宽 + SFU CPU |

## 3. 常见组合

真实产品常常不是"三选一"，而是混搭。记这几种：

### 3.1 LLM 产品（典型：ChatGPT / Claude Web）

- **SSE** 承担"流式输出 token"
- 普通 HTTP 承担非流式请求
- 不需要 WebSocket / WebRTC

为什么不是 WS？LLM 输出是单向、纯文本、低频互动（用户隔几十秒才发一句），SSE 完美匹配，且天然过 CDN。

### 3.2 IM / 聊天产品（微信、Slack）

- **WebSocket** 主干：消息、已读、输入中...
- **HTTP** 做历史消息分页拉取
- 离线推送走厂商通道（FCM / APNs）

为什么不是 SSE？要双向 + 二进制文件传输指令 + 更紧凑的二进制帧。

### 3.3 实时视频会议（Zoom / 腾讯会议 Web / Google Meet）

- **WebSocket** 做信令：房间加入、SDP 交换、ICE trickle、控制指令（静音/举手）
- **WebRTC** 传音视频 + 屏幕共享
- **SFU**（人多时）做转发
- **STUN + TURN** 做穿透兜底

### 3.4 低延迟互动直播（电商带货、在线课堂）

- 主路：**WebRTC SFU** 拉流，< 500ms 延迟
- 兜底：**HLS/LL-HLS** 回退，兼容所有浏览器
- **WebSocket** 广播弹幕、连麦请求、互动指令

### 3.5 协同编辑（Google Docs、Figma）

- **WebSocket** 承载 CRDT / OT 操作流
- 偶尔用 **WebRTC DataChannel** 做局部加速（比如光标位置同步，不重要丢了就丢）

### 3.6 AI Agent 调用 Tool 的 streaming

- 主响应：**SSE**（Anthropic `messages` streaming、OpenAI `responses` streaming 都走这个）
- MCP server 可以用 stdio / SSE / WebSocket，看实现

## 4. 排障：三协议第一眼分别看什么

出问题时先分层：**是连不上，还是连上后数据有问题？** 然后查对应协议的工具。

### 4.1 SSE

**第一眼**：`curl -N` 直接请求，看字节是不是一行行吐出来。

- 能吐 → 问题在浏览器 / 代理 / LB
- 不能吐 → 服务端没 flush、被 gzip 缓了、或逻辑 bug

**第二眼**：DevTools → Network → 该请求 → `EventStream` 面板，看浏览器实际收到了几条事件。

详见 [`sse.md §7`](./sse.md)。

### 4.2 WebSocket

**第一眼**：看握手的 HTTP status。

- 不是 101 → 代理没透传 Upgrade，或鉴权/路由问题
- 是 101 但马上断 → 多半是心跳缺失或 LB 空闲超时

**第二眼**：DevTools → Network → 选中 WS 连接 → `Messages` 面板看真实帧。

**本地复现**：`wscat -c wss://...` 或 `websocat`。

详见 [`websocket.md §7`](./websocket.md)。

### 4.3 WebRTC

**第一眼（一定要先做）**：打开 `chrome://webrtc-internals`（Firefox 是 `about:webrtc`），让用户复现一次，保存 dump 发给你。

**第二眼**：看 `iceConnectionState` 停在哪个状态：

- 卡在 `checking` → `failed` → 90% 是没 TURN / TURN 不通
- 连上后很快 `disconnected` → 网络切换或防火墙端口回收
- `connected` 但没画面 → 看 `inbound-rtp.packetsReceived`，是 0 还是在涨

**第三眼**：看选中的 ICE candidate pair 是什么类型。`relay`（TURN）多了说明直连成功率低。

详见 [`webrtc.md §7`](./webrtc.md)。

## 5. 常见问题快速对号

| 症状 | 最可能的协议层原因 | 去哪查 |
| --- | --- | --- |
| 浏览器一直转圈没数据 | SSE 被 nginx 缓冲 | [sse.md §7.1](./sse.md) |
| 过 60 秒就断 | LB idle timeout；缺心跳 | 对应协议 §7 |
| WS 握手 200 / 400 / 不 101 | 代理吃了 Upgrade 头 | [websocket.md §7.1](./websocket.md) |
| WS close code 1006 频繁 | TCP 被中间人杀了；缺心跳 | [websocket.md §7.2](./websocket.md) |
| WebRTC 有时连不通 | 没 TURN / 企业网封 UDP | [webrtc.md §7.3](./webrtc.md) |
| WebRTC 连通但听不到 | SDP 方向；autoplay；track 没加 | [webrtc.md §7.4](./webrtc.md) |
| 远端画面卡顿花屏 | 丢包 / 上行带宽不够 / CPU | [webrtc.md §7.5](./webrtc.md) |
| LLM 流式响应没出字 | SSE 被压缩或缓冲 | [sse.md §7.1](./sse.md) |

## 6. 学习建议路径

如果你是按"面试 + 真开发"的角度来学的，推荐顺序：

1. **SSE**（1–2 小时）——最简单的长连接模型，先把"HTTP chunked + 一个长响应"这个直觉建立起来
2. **WebSocket**（半天）——握手、帧、心跳、代理配合，这些概念在面试里最常考
3. **WebRTC**（1–2 天）——难度和前两个不是一个量级。**别先读规范**，先照 §4 跑起来一个本地 demo，再回头看各个概念

每个文档里的 §7 排障章节建议**每学完一个就动手制造一次问题**：

- SSE：给 nginx 加上 `proxy_buffering on` 看会发生什么
- WS：故意停掉心跳，观察多久会被 LB 踢
- WebRTC：用 `iceTransportPolicy: 'relay'` 强制走 TURN，看 `webrtc-internals` 里发生什么

这种"看到症状 → 想到是哪一层"的肌肉记忆，就是线上排障速度的来源。
