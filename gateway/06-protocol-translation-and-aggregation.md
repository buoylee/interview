# 06 · 協議轉換與聚合:BFF、gRPC↔HTTP、WebSocket 🔬

> 一句話:管線的 **⑥請求改寫 + ⑧回應過濾** 兩段的「重活」版本。外部世界(瀏覽器、App、第三方)說的協議,和內部服務(gRPC、內網約定)說的協議常常不同;網關在邊緣**翻譯、改寫、聚合**,讓兩邊各自舒服。

通信範式(同步 RPC vs 異步消息)、BFF 概念在 `system-design/04` 有;協議本身(WebSocket/SSE/HTTP)在 `network/`。本章講網關**怎麼在邊緣做這些翻譯**。

---

## 1. 🔬 協議轉換:外面 REST/JSON,裡面 gRPC

典型矛盾:**外部**客戶端(尤其瀏覽器)最順手的是 **HTTP/1.1 + JSON**;**內部**服務間為了性能/強類型,常用 **gRPC(HTTP/2 + protobuf)**。瀏覽器又不能直接發 gRPC(它連不上原生 gRPC 的 framing)。網關來當翻譯:

- **gRPC-JSON transcoding**:網關接收 `POST /v1/orders` + JSON,**按 proto 的註解**(`google.api.http`)把它翻成對內的 gRPC 調用,再把 gRPC 回應翻回 JSON。Envoy 的 `grpc_json_transcoder`、gRPC-Gateway(Go)都是幹這個。好處:後端只寫一套 gRPC,外部自動得到 REST。
- **gRPC-Web**:瀏覽器發 gRPC-Web(HTTP/1.1 可承載的 gRPC 變體),網關翻成原生 gRPC。
- **HTTP 版本轉接**:外部 HTTP/2 或 HTTP/3(QUIC,基於 UDP,解決隊頭阻塞),內部可能還是 HTTP/1.1。網關**終止外部協議、用內部協議連後端**(回扣 ch01 TLS 終止——終止了才能轉)。

> **面試點**:「瀏覽器要調 gRPC 後端怎麼辦?」→ 網關做 gRPC-JSON transcoding 或 gRPC-Web 轉換,後端無感。

---

## 2. 🔬 改寫:加減頭、改路徑、脫敏

掛在管線 ⑥(請求向後)和 ⑧(回應向前):

- **請求改寫**:注入內部可信頭(`X-User-Id`,回扣 ch04 offload)、注入 `traceparent`(ch07)、**剝掉外部不該傳進來的頭**(如客戶端偽造的 `X-User-Id`——這條很重要,否則 offload 模式被繞過)、改寫 path(`/api/v1/x` → 後端的 `/x`)。
- **回應改寫**:加 CORS 頭、加安全頭(`Strict-Transport-Security`)、**脫敏**(把後端多回的敏感欄位刪掉)、壓縮(gzip/br)。

**🔬 流式 vs 緩衝(回扣 ch02 的 header_filter/body_filter 分段)**:只改 header 不需要把整個 body 讀進記憶體——在 `header_filter` 階段流式處理即可,省記憶體、低延遲。只有要改 body(脫敏、轉碼)才需要緩衝。網關設計上盡量**能流式就別緩衝**,因為緩衝大 body 是記憶體與延遲殺手。

---

## 3. 🔬 API 聚合 / BFF:一次外部請求,扇出多個後端

行動端打開一個頁面常要 5 份數據(用戶、訂單、推薦…)。若讓 App 發 5 個請求,**移動網路的多次往返(RTT)**很傷體驗。

- **API 聚合(aggregation)**:網關收**一個**外部請求,**並行**扇出到多個後端,合併結果一次返回。把 5 次客戶端往返,變成「1 次客戶端往返 + 5 次內網並行調用」(內網 RTT 極低)。
- **BFF(Backend For Frontend)**:給**每種前端**(iOS / Android / Web)各做一個專屬網關層,各自裁剪/聚合出最適合自己的數據形狀。對比「一個通用網關服務所有端」。
  - **BFF 取捨**:專屬 → 每端體驗最優、前端團隊能自主改自己的 BFF;代價是**多套要維護**、邏輯可能重複。通用 → 一套省事,但容易變成誰都不敢動的「上帝網關」。團隊多、端差異大選 BFF;否則通用足矣。
> 🔗 BFF 與通信範式在 `system-design/04-服務化與通信範式.md`。

## 4. 🔬 長連線:WebSocket / SSE 穿透網關

普通網關是「一來一回」的請求/回應;長連線(雙向實時)過網關有額外講究:

- **WebSocket**:靠 HTTP `Upgrade: websocket` 握手把連線升級成雙向通道。網關要**識別並放行 Upgrade**、之後**保持連線**(不能像普通請求那樣處理完就關)、設好**空閒超時**(別把活躍長連線當死連線砍掉)。
- **SSE(Server-Sent Events)**:伺服器單向持續推(基於普通 HTTP 流,`text/event-stream`)。網關**不能緩衝**整個回應(否則客戶端永遠收不到——它是無盡流),要**邊收邊轉**、關掉對這條路由的 response buffering。
- **和限流/鑑權的衝突**:長連線「一次握手、長期存在」,按**請求數**限流對它沒意義(它就一個請求);鑑權只在握手時做一次,連線期間 token 過期了怎麼辦(要不要踢)是設計點。長連線還佔住連線額度(回扣 ch01 連線池、ch05 並發上限)。

> **面試點**:「SSE/WebSocket 過網關要注意什麼?」→ 放行 Upgrade、關 response buffering、保持連線+合理空閒超時、限流/鑑權模型要為「長連線」單獨考慮。

---

## 本章小結

- **協議轉換**:gRPC-JSON transcoding / gRPC-Web 讓瀏覽器能用 gRPC 後端;HTTP/1↔2↔3 在網關終止轉接。
- **改寫**:注入可信頭/traceparent、**剝掉外部偽造頭**、脫敏、壓縮;**能流式就別緩衝**(只改 header 不必讀 body)。
- **聚合 / BFF**:把多次客戶端往返收斂成「一次外部 + 多次內網並行」;BFF 是「每端一個專屬網關」,團隊多/端差異大才划算。
- **長連線**:WebSocket 放行 Upgrade + 保連線;SSE 關 buffering 邊收邊轉;限流/鑑權要為長連線單獨設計。
