## 白板大題:「設計一個 API 網關」

別東一句西一句,**沿管線一條龍**講(這結構本身就是高分點):

1. **先畫管線(ch02)**:接收 → 路由匹配 → 認證 → 授權 → 限流 → 改寫 → 轉發 upstream → 回應過濾 → 埋點。「所有功能都掛在這條按序、可短路的管線上。」
2. **引擎(ch01)**:底層是事件驅動的反向代理(epoll + 多 worker),L7 才能認 path/header;TLS 在邊緣終止。
3. **路由 + LB(ch03)**:host/path 匹配(radix tree)、健康檢查摘壞後端、按權重做灰度。
4. **鑑權(ch04)**:邊緣驗 JWT(無狀態、公鑰驗簽),後端信任注入的身份頭。
5. **流量控制(ch05)**:per-consumer 限流 +(叢集下)Redis+Lua 共享計數;熔斷保護後端;過載 load shedding。
6. **可觀測(ch07)**:統一日誌 + traceparent 注入 + RED 指標,網關是天然觀測點。
7. **網關自己是叢集(ch08)**:多節點 + 前置 L4 + **無狀態**(計數/ session 外移),控制面動態下發配置;**別讓它成 SPOF**。
8. **(若問 LLM)AI 網關(ch10)**:token 限流、語義快取、模型路由、護欄。

收尾一句:「網關的本質是**邊緣橫切關注點的集中地**,設計就是把這些橫切能力按正確順序縫進一條高吞吐、可熱更新、無狀態的管線。」

## 高頻陷阱卡(問題 → 一句要點 → 回扣章)

**Q1. 限流該放鑑權前還是後?**
→ 放前省 CPU 但只能粗粒度(IP/全域);放後能按用戶配額但鑑權自己會被打。大流量**兩道都做**(邊緣粗限流護鑑權,鑑權後細配額)。〔ch02〕

**Q2. 網關會不會變成單點(SPOF)?怎麼破?**
→ 會,它是全站入口。破法:多節點 + 前置 L4 分流 + **無狀態化**(計數/ session 外移到 Redis/JWT),任一節點可替換、可水平擴。〔ch08〕

**Q3. 有了 API 網關,為什麼還要 Service Mesh?**
→ 網關管**南北向**(入口、面向用戶);mesh 管**東西向**(服務間),把 mTLS/重試/熔斷下沉到 sidecar,業務碼零侵入。互補不互斥。〔ch09〕

**Q4. JWT 為什麼能在網關「無狀態」校驗?**
→ 簽名自證真偽 + claims 自帶 `exp`/`aud`,網關有公鑰(JWKS）就能驗,**不查任何存儲**。代價:撤銷難(靠短 exp / 黑名單)。〔ch04〕

**Q5. 5 台網關怎麼實現「每用戶全域 100/s」?**
→ 本地計數最快但總量會飄;**Redis 集中計數 + Lua 原子**最準(代價:每請求一跳 + Redis 單點);令牌預取批量租約兼顧。Lua 是為了讓「INCR+判斷+EXPIRE」原子化防競態。〔ch05,lab 已驗〕

**Q6. L4 和 L7 代理差在哪?各用在哪?**
→ L4 只看 IP:port(快但瞎,做不了懂業務的事),L7 解析 HTTP(能按 path/header 路由、能鑑權改寫,但有解包成本)。API 網關必為 L7;生產常 L4(扛量)+ L7(懂業務)兩層疊用。〔ch01〕

**Q7. LLM 網關和普通 API 網關差在哪?**
→ 限流按 **token** 不按請求數;快取是**語義**(embedding 相似度)不是精確;路由要**跨供應商容錯**;護欄要懂 **prompt 注入/PII**;回應是 **SSE 流式**。〔ch10,lab 已驗〕

**Q8. 改網關配置怎麼「不斷現有連線」?**
→ Nginx reload:新 worker 接新連線、舊 worker 處理完舊連線再退;Envoy:xDS **動態下發**無需重啟。〔ch08〕

**Q9. 網關怎麼做灰度發布?藍綠和金絲雀差在哪?**
→ 按權重/header/用戶 hash 切流量。**藍綠是開關**(兩套全量、整體切),**金絲雀是旋鈕**(漸進放量 5%→100%)。〔ch03〕

**Q10. 瀏覽器要調 gRPC 後端怎麼辦?**
→ 網關做 gRPC-JSON transcoding 或 gRPC-Web 轉換,後端只寫 gRPC、外部自動得到 REST。〔ch06〕

**Q11. 過載了網關該怎麼辦?**
→ load shedding:按優先級**主動丟**一部分(`429`/`503`),保住剩下的正常——丟一部分好過全死(背壓);別無界緩衝。〔ch05〕

**Q12. 網關 5xx 飆升怎麼查?**
→ 先看訪問日誌區分**網關自身 vs upstream**(`status` vs `upstream_status`)、用 `latency - upstream_latency` 分「網關慢 vs 後端慢」,再往對應方向深挖。〔ch07/ch11〕

## 一句話總綱

**網關 = 邊緣的可程式化代理,是橫切關注點(路由/鑑權/限流/熔斷/觀測)的集中地;沿 反向代理 → API 網關 →〔mesh 邊界〕→ AI 網關 演化,後面每層多認識一層業務。設計它 = 把這些橫切能力按正確順序縫進一條高吞吐、可熱更新、無狀態、別成 SPOF 的管線。**
