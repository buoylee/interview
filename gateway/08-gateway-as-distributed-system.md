# 08 · 網關自己是個分散式系統 🔬

> 一句話:前面七章把網關當成「一台會做很多事的代理」。但生產裡網關是**一個叢集**——它有自己的控制面、要熱更新配置、要高可用、要無狀態、最怕變成全站的單點。這章講的是「**怎麼治理網關本身**」,是資深/架構師面試最容易拉開差距的地方。

這部分倉庫其它 track 幾乎沒覆蓋,本章自己擁有。

---

## 1. 🔬 控制面 vs 資料面:配置怎麼到網關

把網關拆成兩塊看(這個區分到 ch09 會擴展到整個 mesh):

- **資料面(data plane)**:真正轉發流量的那些網關進程(Nginx/Envoy worker)。
- **控制面(control plane)**:管「路由規則、upstream、限流策略」這些**配置**從哪來、怎麼下發到資料面。

三種配置下發方式,從靜態到動態:

| 方式 | 代表 | 改路由要怎樣 |
|---|---|---|
| **靜態文件 + reload** | Nginx | 改 `nginx.conf` → `nginx -s reload` |
| **聲明式配置** | Kong(DB-less)、APISIX | 一份 yaml/API 描述全部,推給節點 |
| **動態 API 下發** | **Envoy xDS** | 控制面通過 gRPC 流**實時推**,網關不重啟即時生效 |

**🔬 Envoy xDS(面試高頻、現代網關/mesh 的核心機制)**:xDS 是一組「discovery service」API,控制面通過它把配置**動態推**給 Envoy:
- **LDS**(Listener)= 監聽哪些埠;**RDS**(Route)= 路由規則;**CDS**(Cluster)= 有哪些 upstream 服務;**EDS**(Endpoint)= 每個 cluster 下具體有哪些健康實例的 IP。
- 意義:**改路由、加後端、調灰度權重,控制面推一下就生效,資料面零重啟、零斷連**。這就是為什麼 Envoy 成了 API 網關和 service mesh(ch09)資料面的事實標準——它的配置是「活的」。

**🔬 Kong DB vs DB-less / APISIX + etcd**:
- Kong 早期靠 Postgres 存配置(有狀態、DB 是依賴);後來推 **DB-less**(聲明式 yaml,無資料庫,配置即代碼,更易 GitOps)。
- APISIX 把配置存 **etcd**,各網關節點 **watch** etcd,配置一變所有節點秒級感知(回扣 `cloud-native/03b` 的 etcd/watch 機制)。
- 取捨:有 DB → 動態增刪方便但多一個有狀態依賴;DB-less/聲明式 → 簡單、易版本控制,但改配置要重新 apply 整份。

## 2. 🔬 熱更新:改配置怎麼「不斷現有連線」

改了路由/憑證,怎麼讓新配置生效又**不掐斷正在處理的請求**?

- **Nginx `reload` 的優雅之處**:`nginx -s reload` 時,master **fork 出一批新 worker** 用新配置接**新**連線,**舊 worker 繼續處理完手上的舊連線**才退出。所以 reload 期間不斷流。(回扣 ch01 master-worker)
- **Envoy 動態下發**:根本不用重啟——xDS 推新配置,Envoy 在內部原子切換到新路由表,進行中的請求用舊規則走完。
- **憑證輪換**:TLS 證書到期換新,要能**不斷連地換**(reload 或動態下發),否則換證書時全站握手失敗。

## 3. 🔬 高可用 + 無狀態:別讓網關自己成為瓶頸/單點

網關是全站入口,**它掛 = 全站掛**。所以:

- **多節點 + 前置分流**:跑 N 個網關節點,前面用 **L4 LB(LVS/NLB)/ DNS 輪詢 / VIP(keepalived)** 把流量分到各節點(回扣 ch01「L4 扛量 + L7 懂業務」兩層疊用)。任一節點掛了,前置分流摘掉它。
- **🔬 無狀態化是 HA 的前提**:網關節點之間**不能各自存著只有自己有的狀態**,否則一掛就丟、且無法水平擴展。對策:把狀態**外移**——
  - 限流計數 → Redis(**這正是 ch05 lab 證明的:狀態在 Redis,停掉一個節點配額照樣全域生效**)。
  - session / 鑑權結果 → 外部存儲或乾脆用無狀態 JWT(回扣 ch04:JWT 無狀態,網關不必存 session)。
  - 一旦無狀態,網關就能**隨意水平擴縮容**,任一節點可被替換。
- **別把重邏輯堆網關**:網關是共享的關鍵路徑,塞太多重 CPU 邏輯(複雜轉換、大 body 緩衝)會讓它成為**全站瓶頸**。原則:**網關只做橫切的、輕量的事;重業務邏輯留給後端。**

---

## 4. 🔬 網關自身的灰度/藍綠:升級網關怎麼不斷流

前面 ch03 講的是「用網關給**後端**做灰度」。這裡反過來:**網關自己**要升級版本(換 Envoy 版本、改核心配置),怎麼不出事?

- 把網關節點當普通服務做**滾動升級 / 藍綠**:前置 L4 LB 先摘掉一個節點 → 升級它 → 健康檢查通過再放回 → 下一個。
- 配置變更走**金絲雀**:新限流規則先在一個節點/一小撮流量生效,觀察沒問題再全量(避免一條錯規則全站誤殺)。
- 這就是「網關是分散式系統」的完整含義:**它既是治理別人的工具,自己也要被同一套治理手段(滾動/藍綠/金絲雀/健康檢查)管理。**

---

## 本章小結

- 網關 = **控制面(配置從哪來)+ 資料面(轉發流量)**。下發從靜態 reload → 聲明式 → **Envoy xDS 動態推**(LDS/RDS/CDS/EDS,零重啟生效)。
- **熱更新**:Nginx reload 用「新 worker 接新連線、舊 worker 處理完舊連線」不斷流;Envoy 動態下發無需重啟;憑證也要不斷連輪換。
- **HA = 多節點 + 前置 L4 分流 + 無狀態**;狀態外移到 Redis/JWT(ch05 lab 已證),才能隨意擴縮容、任一節點可替換。
- **別把重邏輯堆網關**,否則它成全站瓶頸/單點。
- 網關自己也要被滾動/藍綠/金絲雀治理——它治理別人,也被治理。
