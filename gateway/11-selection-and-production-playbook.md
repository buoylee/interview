# 11 · 選型與生產 playbook

> 一句話:前面把「網關該做什麼、怎麼做」講完了,這章回答工程問題——**選哪個、怎麼上線不出事、出事了怎麼查**。

---

## 1. 選型對照:六個候選,按維度比

| | **Nginx** | **Envoy** | **Kong** | **APISIX** | **Spring Cloud Gateway** | **雲託管**(AWS API Gateway / ALB 等) |
|---|---|---|---|---|---|---|
| 定位 | 反向代理 / 輕量網關 | 資料面標準 | 插件型 API 網關 | 插件型 API 網關 | Java 微服務內嵌網關 | 全託管 |
| 動態配置 | 弱(reload) | **強(xDS 動態下發)** | 中(DB / DB-less) | **強(etcd watch)** | 中(配置中心) | 廠商控制台/API |
| 插件生態 | 少(需 Lua/OpenResty) | 中(C++/WASM filter) | **豐富** | **豐富(Lua)** | Java filter | 廠商功能 |
| 性能 | 高 | 高 | 高(基於 OpenResty) | **高** | 中(JVM) | 看廠商 |
| 運維成本 | 低 | 中高(要控制面) | 中 | 中 | 低(隨服務走) | **最低(不自運維)** |
| 語言生態綁定 | 無 | 無 | 無 | 無 | **綁 Java** | 綁雲廠商 |
| 逃生票(可自托管) | ✅ | ✅ | ✅ | ✅ | ✅ | ❌(鎖定) |

> 註:Spring Cloud Gateway 只在「**團隊就是 Java/Spring 生態、網關想和服務同生命週期**」時才是首選;否則它的 JVM 成本與語言綁定不划算。別因為「我會 Java」就把網關也綁死在 Java——網關是基礎設施,該按基礎設施選型(回扣:框架對照要生態平衡)。

---

## 2. 何時用哪個(決策樹)

- **只要 L7 路由 + TLS + 基本限流,團隊小** → **Nginx**(或雲廠商 ALB)。別上重型網關(回扣 ch00「何時不需要」)。
- **k8s 原生、要動態配置 / 做 mesh 資料面** → **Envoy**(配 Istio/控制面)或 **APISIX**。
- **要豐富插件生態、按 API 維度治理(鑑權/限流/轉換開箱即用)** → **Kong / APISIX**。
- **就是 Java/Spring 微服務、想內嵌** → **Spring Cloud Gateway**。
- **不想自運維、能接受廠商鎖定與成本** → **雲託管**。
- **LLM 流量** → 在上面之上加 AI 網關層(LiteLLM / Kong AI / Envoy AI Gateway,ch10)。

**「關鍵 × 好換」原則**:網關是**極關鍵**的基礎設施(掛了全站掛)。越關鍵,越要保留**逃生票**——優先選能**自托管的開源**(Nginx/Envoy/APISIX/Kong),這樣即使你現在用雲託管圖省事,哪天要遷移/降本/合規,也隨時能搬到自托管,不被某個 SaaS 綁死。雲託管不是不能用,而是要**心裡有 B 計畫**。

## 3. 遷移路徑:別大爆炸,要絞殺

從一台 Nginx 漸進演化到 APISIX/Envoy(典型路徑):

1. **並排**:新網關(APISIX)和舊 Nginx 並存,前面的 L4 LB 先把流量全給 Nginx。
2. **絞殺者模式(strangler)**:**按路由**逐條切——先把 `/v2/*` 切到 APISIX,觀察指標,沒問題再切下一批。
3. **能力對齊**:把舊 Nginx 的限流/重寫規則在新網關上重建並驗證(灰度比對)。
4. **收尾**:流量全切走後,Nginx 退役。
- 原則:**永遠保留回滾路徑**(改 L4 LB 權重就能切回),別一次性整體切換(回扣 ch03 灰度 vs 藍綠)。

## 4. 上線生產清單

- [ ] **超時**:對 upstream 的連線/讀寫超時都設,值用鏈路預算反推(ch03)。沒有無限等待。
- [ ] **重試**:只對冪等、退避+抖動、限次(ch03);確認不會放大成重試風暴。
- [ ] **限流**:全域 + per-route + per-consumer 默認值都配;分散式計數的後端(Redis)做了 HA;明確 Redis 掛了 fail-open 還 fail-close(ch05)。
- [ ] **健康檢查**:主動探活 + `/health` 是「深健康」(查依賴,不只查進程)(ch03)。
- [ ] **鑑權**:JWT 公鑰輪換(JWKS)驗證過;offload 模式下後端確認**不可從外網直連**(否則偽造 `X-User-Id`)(ch04)。
- [ ] **可觀測**:結構化訪問日誌、traceparent 注入、RED 指標、告警接好(ch07)。
- [ ] **容量 + 壓測**:壓到飽和點,確認 load shedding 生效而非雪崩(ch05);算好單節點容量再定叢集規模。
- [ ] **HA / SPOF 演練**:殺一個網關節點,確認前置分流摘除、流量無感;殺 Redis,確認限流降級行為符合預期(ch08)。
- [ ] **配置變更**:走灰度(一個節點/一小撮流量先生效),有回滾(ch08)。

## 5. 故障模式 debug playbook

| 症狀 | 先看什麼 | 常見根因 |
|---|---|---|
| **網關 5xx 飆升** | 區分網關自身 5xx vs upstream 5xx(`status` vs `upstream_status`,ch07) | upstream 掛/熔斷打開(ch05);若是 502/504 多半後端慢或不可達;若 503 多半網關限流/卸載 |
| **延遲毛刺(p99 高)** | `latency - upstream_latency`(ch07)定位是後端慢還是網關慢 | 後端 GC/慢查詢;或網關側:TLS 握手、限流排隊、大 body 緩衝、連線池打滿 |
| **限流誤殺(正常用戶被 429)** | 限流維度與閾值;分散式計數是否傾斜 | 本地計數叢集下不均(ch05);閾值拍太低;固定窗口臨界突刺;某用戶共享了 IP 維度配額 |
| **配置下發失敗 / 不生效** | 控制面狀態、節點是否拉到新配置 | etcd/xDS 推送失敗;節點未 watch 到;配置語法錯被拒(ch08) |
| **連線池打滿 / 拒絕新連線** | upstream keepalive 連線數、後端響應時間 | 後端變慢 → 連線都在忙 → 新請求排隊/拒絕(ch01);要麼擴後端要麼降並發 |
| **熱更新後斷連** | reload/下發方式 | reload 配置語法錯導致 worker 沒起;或非優雅重啟掐了連線(ch08) |

**通用排查順序**:先用訪問日誌的 `status / upstream_status / latency / upstream_latency` 四個欄位**定位是網關還是後端**(ch07 那招),再往對應方向深挖。**別一上來就猜**——網關天然有最全的觀測數據,先看數據。

## 本章小結

- 選型按**動態配置、插件生態、性能、運維成本、語言綁定**比;小團隊 Nginx、k8s 原生 Envoy/APISIX、插件生態 Kong/APISIX、Java 棧 SCG、省運維雲託管。
- **「關鍵×好換」**:網關極關鍵,優先選能自托管的開源,保留逃生票,別被 SaaS 或單一語言鎖定。
- 遷移用**絞殺者模式**按路由漸進切,永遠留回滾。
- 上線過一遍清單(超時/重試/限流/健檢/鑑權/觀測/壓測/HA 演練)。
- 出事先用訪問日誌四欄位定位「網關 vs 後端」,再深挖。
