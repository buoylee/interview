# 04 · 邊緣鑑權:認證與授權落在網關 🔬

> 一句話:管線的 **③認證 + ④授權** 兩段。網關在邊緣先回答「**你是誰**(authn)」和「**你能不能做這件事**(authz)」,驗通過了才把請求放給後端——後端因此可以「假設請求已可信」,不必各自重寫一套登入校驗。

認證 vs 授權的**概念基礎**、RBAC/ABAC 模型在 `system-design/10` 已講透。本章只補一個增量:**為什麼這件事該下沉到網關、以及網關怎麼無狀態地做。**

---

## 0. 先分清:authn vs authz(高頻陷阱)

- **認證 authentication(authn)**:**你是誰?** 驗明身份。產出「這是用戶 #123」。
- **授權 authorization(authz)**:**你能不能?** 驗明權限。產出「#123 能不能刪這個訂單」。
- 順序固定:**先 authn 再 authz**(不知道你是誰,談不上你能不能)。HTTP 狀態碼也對應:身份沒過 → `401 Unauthorized`(其實是「未認證」);身份過了但沒權限 → `403 Forbidden`。

---

## 1. 🔬 為什麼鑑權要下沉到網關

沒有網關時,每個後端服務都得自己:解析 token、驗簽、查用戶、判權限。結果是同一套邏輯抄 N 份,改一次密鑰輪換策略要動 N 個服務,還容易有人寫錯漏驗。

把 authn(以及粗粒度 authz)收到網關:

- **統一**:一處驗,全站一致;改策略改一處。
- **後端簡化**:後端服務收到的請求**已經是可信的**,可以信任網關注入的身份頭(見 §4),自己只做細粒度業務權限。
- **減少攻擊面**:無效 token 在邊緣就被擋,打不到後端。
- **回扣 ch02**:正因為鑑權在管線靠前,**短路**得早(`401` 直接返回),省下後面所有關和後端的成本。

---

## 2. 🔬 JWT:網關為什麼能「無狀態」驗身份

JWT(JSON Web Token)是邊緣鑑權的主力,關鍵就在**無狀態**——網關**不查資料庫、不查 session 存儲**就能驗一個 token 真不真、過沒過期。

**結構**:`header.payload.signature` 三段 base64,用 `.` 連接。

```
eyJhbGciOiJSUzI1NiJ9 . eyJzdWIiOiIxMjMiLCJleHAiOjE3...} . <簽名>
   header(算法)            payload(claims:sub/exp/aud...)      signature
```

- **header**:用了什麼簽名算法(`alg`,如 `RS256`)。
- **payload(claims)**:`sub`(誰)、`exp`(過期時間)、`iat`(簽發時間)、`aud`(給誰用)、`iss`(誰簽的),加自定義(角色、租戶)。
- **signature**:用簽發方的密鑰對 `header.payload` 簽出來的。**這是無狀態的核心**——只要驗簽通過,就證明這 token 確實是可信簽發方發的、且內容沒被篡改。

**🔬 網關驗 JWT 做這幾步(全程不查 DB)**:
1. 驗**簽名**:用密鑰重算簽名,對得上 → 沒被篡改、確是可信方簽的。
2. 驗 **`exp`**:過期了 → 拒。驗 **`nbf`/`iat`**(生效時間)。
3. 驗 **`aud` / `iss`**:這 token 是不是發給「我這個服務」、是不是「我信任的簽發方」簽的——防止把 A 系統的 token 拿來用在 B 系統。

**🔬 對稱 vs 非對稱簽名**(關鍵取捨):
- **HS256(對稱)**:簽和驗用**同一把密鑰**。簡單,但網關和簽發方得共享密鑰——洩漏一處全完,難多方擴展。
- **RS256 / ES256(非對稱)**:簽發方用**私鑰**簽,網關用**公鑰**驗。網關只拿公鑰(洩漏也只能驗不能偽造),公鑰可公開分發。**生產主流。**
- **公鑰怎麼來:JWKS**。簽發方暴露一個 `/.well-known/jwks.json`(JWKS endpoint),列出當前公鑰(含 `kid` 鍵 ID)。網關按 token header 的 `kid` 取對應公鑰來驗。**密鑰輪換**時簽發方換 `kid` 發新公鑰,網關拉新 JWKS 即可,無需重啟——這就是無狀態 + 可輪換的關鍵。

> **面試點**:「JWT 為什麼能在網關無狀態校驗?」→ 簽名自證真偽 + claims 自帶過期/受眾,網關有公鑰就能驗,不必查任何存儲。「為什麼用 RS256 而非 HS256?」→ 非對稱讓網關只持公鑰、不能偽造,密鑰管理更安全、易多方擴展。

**JWT 的軟肋:撤銷難**。無狀態的代價是「簽出去就生效到 `exp`」,想提前作廢(用戶登出/被封)沒法直接撤——對策:短 `exp` + refresh token、或維護一個小的「黑名單/撤銷列表」(這就又引入了一點狀態)。

## 3. 🔬 其它認證方式

- **不透明 token(opaque token)+ introspection**:token 本身是個無意義隨機串(不像 JWT 自包含)。網關**沒法自己驗**,得拿去問授權伺服器的 **introspection endpoint**(`POST /introspect`)「這 token 有效嗎、是誰」。**有狀態、每次多一跳**,但**能即時撤銷**(授權伺服器說無效就無效)。和 JWT 正好是「無狀態 vs 可撤銷」的取捨。
- **OAuth2 / OIDC**:授權框架。常見 **authorization code flow**——用戶在授權伺服器登入拿到 code,換成 access token(OAuth2,授權)+ id token(OIDC,認證,通常是 JWT)。網關作為 **resource server**,校驗請求帶的 access token(JWT 驗簽,或 opaque introspection)。
- **API Key**:給機器/第三方的長期憑證,放 header。簡單,但要配額/輪換管理(常和 ch05 的 per-consumer 限流綁定)。
- **mTLS(雙向 TLS)**:不只伺服器出證書,**客戶端也出證書**,網關驗客戶端證書來認身份。常用於**服務間 / 合作方**的強認證(回扣 ch01 的 TLS 終止——mTLS 在網關終止並驗客戶端證書)。

## 4. 🔬 鑑權的兩種收尾:offload vs 委派,以及 authz 落點

網關驗完之後,身份怎麼傳給後端?

- **auth offload(卸載 + 注入可信頭)**:網關驗完,把身份結果寫進**內部 header**(`X-User-Id: 123`、`X-Roles: admin`),原始 token 不再往後傳。後端**信任**這些頭(前提:後端不直接暴露在外,只能經網關進來,否則有人偽造 `X-User-Id` 就完了)。後端最省事。
- **token 透傳(委派)**:把原始 token 原樣傳給後端,後端自己再驗一次/取細粒度 claims。後端要重複驗簽,但不必信任網關注入。零信任場景用這個。

**authz 落點**:
- **粗粒度放網關**:「這個角色能不能訪問這條路由」「免費用戶不能調 `/admin/*`」——這類**與具體資源無關**的判斷適合在網關按 RBAC 做。
- **細粒度留後端**:「#123 能不能刪**這一張**訂單」要查「這訂單是不是他的」,網關不該懂業務數據,**留給後端**(ABAC/業務規則)。

> 🔗 RBAC vs ABAC、認證 vs 授權的概念與模型,`system-design/10-安全-網路分區與認證授權.md` 已講透,本章不重述。

## 本章小結

- **先 authn(你是誰,過不了 401)再 authz(你能不能,過不了 403)。**
- 鑑權**下沉網關**:統一、後端簡化、減攻擊面、能早短路省成本。
- **JWT 無狀態**:簽名自證真偽 + claims 自帶 `exp`/`aud`,網關有公鑰就能驗,不查存儲;**RS256 非對稱**讓網關只持公鑰更安全,公鑰經 **JWKS + kid** 輪換。軟肋是**撤銷難**。
- **opaque token + introspection** 是反面:有狀態、多一跳,但**可即時撤銷**。
- 收尾:**offload(注入可信頭,後端信任)vs 透傳(後端再驗)**;authz **粗粒度放網關、細粒度留後端**。
