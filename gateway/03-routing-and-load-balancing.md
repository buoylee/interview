# 03 · 路由與負載均衡:把請求送對地方 🔬

> 一句話:管線的 **②匹配路由 + ⑦轉發 upstream** 兩段展開。網關先用一張**路由表**決定「這個請求歸哪個後端組」,再在組內**挑一台健康的實例**打過去——挑的過程要會摘除壞實例、要能按權重做灰度、要對慢後端設超時與克制的重試。

算法本身(輪詢/最少連線/一致性哈希)在 `system-design/05` 已講透;本章只講**網關側**的增量:路由怎麼匹配、健康檢查怎麼摘恢復、灰度怎麼切、邊緣超時重試怎麼設。

---

## 1. 🔬 路由匹配:一張表怎麼快速找到目標

一個請求進來,網關要在**成百上千條 route 規則**裡找出該走哪條。匹配維度:

| 維度 | 例 | 備註 |
|---|---|---|
| **host** | `api.foo.com` vs `admin.foo.com` | 多租戶/多站常用 |
| **path** | 前綴 `/v1/orders/*`、精確 `/health`、正則 `~ ^/u/\d+` | 最常用 |
| **header / method / query** | `X-Canary: true`、`POST`、`?ver=2` | 灰度、版本路由 |
| **權重** | 90% → v1,10% → v2 | 灰度放量(見 §3) |

**🔬 匹配優先級與衝突**:多條規則同時命中怎麼辦?通用規則是**「越具體越優先」**:精確 path > 最長前綴 > 正則 > 兜底。`/v1/orders/123` 同時匹配 `/v1/orders/*` 和 `/v1/*` 時,選更長更具體的那條。設計路由表時,**衝突要靠明確的優先級規則消解,而不是靠書寫順序撞運氣**(雖然有些網關確實按定義順序匹配——這正是坑)。

**🔬 路由表怎麼組織**:幾千條前綴規則若逐條 `startsWith` 掃,是 O(N) 線性,慢。生產網關用 **trie / radix tree(基數樹)** 存 path,按字元逐層下降,一次查找就能拿到**最長前綴匹配**,複雜度約 O(path 長度) 而與規則總數無關。這就是為什麼網關能在上萬條路由下仍然微秒級匹配。(你若好奇 radix tree,Go 的 `httprouter`、gin 的路由就是這套。)

## 2. 🔬 健康檢查:自動摘除壞後端,再自動接回

挑後端前,得先知道**哪些後端是活的**。兩種健康檢查,分工不同:

**被動健康檢查(passive,從真實流量學)**
- 機制:統計轉發給某後端的**真實請求**結果,連續失敗 N 次(`max_fails`)就把它標記 **down**,隔離一段時間(`fail_timeout`),期間不再分流量給它;過了窗口再試探性放流量,成功就恢復。
- 優點:零額外探測成本。缺點:**得先犧牲幾個真實請求**才發現後端壞了;且若後端徹底沒流量,永遠不會被探測到恢復。

> 🔗 **這正是 ch01 的 nginx lab 看到的現象**:剛啟動時 echo2 還沒就緒,nginx 試了一次失敗就把它標記 down ~10s(`fail_timeout` 默認),那 10 秒流量全打 echo1;窗口過後 echo2 恢復,輪詢才乾淨交替。當時的「bug」其實就是被動健康檢查在工作。

**主動健康檢查(active,定期探活)**
- 機制:網關**定期**(如每 5s)主動發探測請求(`GET /health`),連續成功才標記 up、連續失敗才標記 down。
- 優點:**在真實流量受影響前**就摘除壞後端;沒流量也能探測恢復。缺點:要額外探測流量;`/health` 要設計好(別只查進程活著,要查依賴——「淺健康」vs「深健康」)。

**生產常兩者疊用**:主動探活做主、被動兜底快速反應。一句話面試答法:**被動是「摔了才知道」,主動是「定期體檢」;主動更及時但有探測成本,生產通常都開。**

## 3. 🔬 灰度 / 金絲雀:讓新版本只承接一小撮流量

把新版本(canary)先放出去接**一小部分**流量,觀察指標再決定放量還是回滾。網關是做這件事的天然位置:

- **按權重切分**:`v1: 95%, v2: 5%`,逐步調到 `50/50`、`100% v2`。最常見。
- **按 header / cookie 定向**:`X-Canary: true` 或內部員工的 cookie 先打到 v2——**定向灰度**,讓特定人群先試,風險可控。
- **按用戶屬性 / 百分比一致性**:用 `hash(userId) % 100 < 5` 決定,保證**同一個用戶穩定落在同一版本**(避免一個用戶一會兒 v1 一會兒 v2,體驗錯亂)。

**灰度 vs 藍綠(blue-green)的區別**(高頻):
- **藍綠**:兩套**全量**環境(blue 舊、green 新),流量**一次性整體切換**過去,出問題整體切回。切換快、回滾快,但**沒有「只放 5%」的中間態**,且要雙倍資源。
- **金絲雀**:**漸進**放量(5%→50%→100%),問題只影響一小撮,但過程慢、要持續盯指標。
- 一句話:**藍綠是「開關」,金絲雀是「旋鈕」。** 求穩、要逐步驗證選金絲雀;求快速整體切換、有完整預發驗證選藍綠。

## 4. 🔬 邊緣的超時與重試:對慢後端要克制

網關直接面對不可靠的後端,必須設好兩道保護(否則一個慢後端能拖垮整個網關的連線):

- **超時**:對 upstream 的**連線超時**(TCP 建連多久算失敗)與**讀寫超時**(後端多久不回算失敗)。沒有超時 = 慢後端把網關的 worker/連線全佔住,雪崩。超時值要用**鏈路預算**反推(整條鏈路允許多少延遲,逐層分配),別拍腦袋。
- **重試**:後端偶發失敗時重試**下一台健康實例**——但有三條紀律:
  1. **只對冪等請求重試**(GET 可以,沒有冪等保證的 POST 不能,否則重複下單)。
  2. **退避 + 抖動(backoff + jitter)**,否則所有客戶端同時重試 = **重試風暴**,把抖動放大成全面崩潰。
  3. **限制重試次數 + 配合熔斷**,別把下游已經過載的情況越打越死。

> 🔗 超時的鏈路預算、退避抖動、重試風暴、冪等性這些**機制細節**在 `system-design/01-韌性-依賴掛了怎麼不崩.md` 已深講,本章只強調「它們在網關這一層怎麼配」。重試風暴的量化也見 `performance-tuning-roadmap/10-distributed/02-retry-ratelimit.md`。

## 5. 跨生態對照

| | 路由 | 後端/LB | 灰度 |
|---|---|---|---|
| **Nginx** | `location` 前綴/正則 | `upstream` + 被動健康(商業版有主動) | `split_clients` 按比例 |
| **Envoy** | `route_config`(virtual host + 最長前綴) | 動態 `cluster`,主動+被動健康 | route `weighted_clusters` |
| **APISIX** | `uri`/`host`/`vars` 條件 + radix tree | `upstream` + 服務發現,主動健康 | `traffic-split` plugin |
| **Spring Cloud Gateway** | `Predicate`(Path/Host/Header/Weight) | `lb://` 走 LoadBalancer | `Weight` predicate |

要點:**路由都是「條件匹配 → 命中 route → 選 upstream」**;差別在健康檢查強弱與灰度做法。Envoy/APISIX 因為配置能**動態下發**(ch08),灰度放量可以「改個權重即時生效」,不必 reload。

## 本章小結

- **路由匹配**:host/path/header/權重多維;衝突按「越具體越優先」消解;路由表用 **radix tree** 做最長前綴,規則再多也微秒級。
- **健康檢查**:被動(摔了才知道、零成本)+ 主動(定期體檢、更及時);生產常疊用。ch01 lab 的 echo2 下線就是被動健檢。
- **灰度/金絲雀**:按權重/header/用戶 hash 切;**藍綠是開關(整體切),金絲雀是旋鈕(漸進放)**。
- **邊緣超時重試**:超時用鏈路預算反推;重試只對冪等 + 退避抖動 + 限次,否則重試風暴。
