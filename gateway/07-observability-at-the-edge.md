# 07 · 可觀測性:網關是天然的觀測點 🔬

> 一句話:管線的 **⑨埋點** 段。所有南北向流量**必經網關這一點**——所以它是放監控的最佳位置:一處埋點,覆蓋全站入口的請求數、錯誤、延遲,還能在這裡給每個請求**蓋上 traceId**,讓後面整條鏈路的日誌能串起來。

可觀測性的通用體系(metrics/logs/traces 三本柱、OTel)在 `observability/`,日誌怎麼寫好在 `logging/`。本章只講**網關視角**:為什麼觀測該放這、網關特有的觀測能力。

---

## 1. 🔬 為什麼網關是「天然觀測點」

- **必經性**:每個進出系統的請求都穿過網關(回扣 ch00 南北向)。在任何後端服務埋點都只看到局部;**在網關埋點,一處看全站入口**。
- **位置中立**:網關不關心業務,適合採集**與業務無關的通用信號**(延遲、狀態碼、流量),不需要侵入每個服務的程式碼。
- **回扣 ch02 的 pre/post**:可觀測 filter 在 **pre 段**記下開始時間、注入 trace context,在 **post 段**算出 latency、寫訪問日誌——一個 filter 跨越轉發前後,這就是它能算出端到端耗時的原因。

---

## 2. 🔬 統一訪問日誌:結構化,不是一行文本

網關的訪問日誌是排障第一現場。關鍵是**結構化**(JSON,而非難解析的文本),欄位至少含:

| 欄位 | 為什麼要 |
|---|---|
| `latency`(總耗時)/ `upstream_latency`(後端耗時) | 兩者一減 = 網關自己花的時間,定位「是後端慢還是網關慢」 |
| `status` / `upstream_status` | 網關回的 vs 後端回的(網關 503 但後端沒收到 = 網關熔斷/限流) |
| `route` / `upstream` | 哪條路由、打到哪台後端 |
| `consumer` / `user_id` | 哪個用戶/租戶(回扣 ch04 鑑權結果) |
| `trace_id` | 串起整條鏈路(見 §3) |

**🔬 一個高頻排障技巧**:`latency - upstream_latency` 能把「後端慢」和「網關自己慢(如限流排隊、TLS、緩衝大 body)」分開——這是只有在網關才拿得到的視角。

---

## 3. 🔬 traceId 注入與傳播:讓一個請求的全鏈路日誌串起來

分散式追蹤的核心問題:一個用戶請求穿過 網關→服務A→服務B→DB,**怎麼把這幾段的日誌認成「同一個請求」**?答案:給它一個全局唯一的 **trace ID**,一路傳下去。

- **誰生成**:網關是**最自然的生成點**(它是入口)。請求進來若沒帶 trace context,網關**生成一個**;若帶了(來自上游/前端),就**沿用**。
- **怎麼傳:W3C `traceparent`**。標準 header `traceparent: 00-<trace-id>-<span-id>-<flags>`。網關把它注入(回扣 ch06 請求改寫),後端服務讀它、繼續往下傳、各自上報自己的 span,後端的可觀測系統(OTel Collector)就能把所有 span 按同一 `trace-id` 拼成一棵調用樹。
- **網關自己也是一個 span**:它記錄「在網關停留了多久」,成為調用樹的根。

> 🔗 trace/span/context 傳播、OTel 的完整模型在 `observability/` 已講透。本章記住:**網關是 trace 的起點與注入點。**

---

## 4. 🔬 邊緣指標:RED,按 route/consumer 切

網關天然適合採集 **RED** 三個黃金信號(對請求驅動的系統最對口):

- **Rate**:每秒請求數(按 route、consumer、status 切維度)。
- **Errors**:錯誤率(4xx/5xx 分開;網關自身錯 vs 後端錯分開)。
- **Duration**:延遲分布——**看 p50/p95/p99,不要只看平均**(平均會被掩蓋,p99 才反映尾部體驗)。

為什麼 golden signals 在邊緣最全:網關看得到**所有**入口流量的這三個信號,且能按用戶/路由切;後端各自只看到自己那段。**容量規劃、限流閾值、SLO 監控**的數據源頭都在這。

---

## 5. 🔬 取樣:全量太貴,怎麼取捨

每個請求都生成完整 trace 並上報,在高 QPS 下**存儲和傳輸成本爆炸**。取樣是必要取捨:

- **頭部取樣(head-based)**:在**入口(網關)**就決定「這條 trace 採不採」(如 1% 機率)。簡單、便宜,但**可能漏掉那條出錯的慢請求**(因為出錯與否在請求結束才知道)。
- **尾部取樣(tail-based)**:**等請求結束**再決定——「凡是出錯的、慢的,全採;正常的抽 1%」。能保住有價值的異常 trace,但要先把所有 span 暫存起來才能決策,**成本和複雜度高**(通常在 Collector 做,不在網關)。
- 一句話:**頭部便宜但會漏異常,尾部能抓異常但貴**;生產常頭部為主、對關鍵路徑上尾部。

---

## 本章小結

- 網關是**天然觀測點**:必經、位置中立、pre/post 能算端到端耗時——一處埋點覆蓋全站入口。
- **結構化訪問日誌**:`latency - upstream_latency` 區分「後端慢 vs 網關慢」,這是網關獨有視角。
- **traceId**:網關是 trace 的**生成與注入點**,用 W3C `traceparent` 往後傳,讓全鏈路日誌串成一棵樹。
- **RED**(Rate/Errors/Duration,看 p99 不看平均)在邊緣最全,是限流閾值/SLO 的數據源。
- **取樣**:頭部便宜會漏異常,尾部抓異常但貴。
