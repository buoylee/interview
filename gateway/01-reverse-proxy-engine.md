# 01 · 反向代理:網關的引擎 🔬

> 一句話:在談「網關多管什麼事」之前,先看清**代理本身這台引擎**——它在 socket 與事件層到底做了什麼。後面所有章節(路由、鑑權、限流)都是往這台引擎上掛邏輯;引擎撐不住,掛再多功能也白搭。

這一章用 Nginx 當解剖標本(因為它最透明、最經典),講清四件內幕:**它怎麼用很少的進程扛住海量連線、怎麼復用到後端的連線、怎麼卸載 TLS、L4 和 L7 到底差在哪**。

---

## 0. 先擺正:正向代理 vs 反向代理

- **正向代理(forward proxy)**:替**客戶端**出面,客戶端知道有代理(如公司翻牆代理、爬蟲代理池)。代理「對外」代表你。
- **反向代理(reverse proxy)**:替**伺服器**出面,客戶端**不知道**後面有幾台機器,它以為自己在跟一台伺服器說話。網關永遠是**反向**代理——它對外假裝成整個系統的單一入口。

記憶法:**「反向」是站在伺服器這側反過來迎接客戶端。** 後面講的全是反向代理。

---

## 1. 🔬 Master-Worker:它怎麼用幾個進程扛住幾萬連線

你 Java/Go 背景熟悉的入口模型大概是「**一個連線一個線程**」(Tomcat BIO 時代)或線程池。Nginx 反過來:**進程數極少(= CPU 核數),靠事件驅動讓每個進程同時服務上萬連線。**

```
        ┌─────────────┐
        │   master    │   root 啟動:讀配置、bind+listen 80/443、管理 worker 生命週期
        └──────┬──────┘   (它自己不處理請求)
        ┌──────┼──────┐
        ▼      ▼      ▼
   [worker] [worker] [worker]   = CPU 核數,降權到普通用戶,真正處理連線
```

- **master 進程**:以 root 啟動(因為要綁 <1024 的特權埠),負責**讀配置、`bind()`+`listen()` 監聽 socket、fork/管理 worker、平滑重載**(見 ch08)。它**不碰業務請求**,所以攻擊面小。
- **worker 進程**:數量設 `worker_processes auto;`(= CPU 核數),降權到非特權用戶。每個 worker 是**單線程**,但靠事件迴圈同時服務海量連線。worker 之間**無共享記憶體的請求狀態**(限流計數這類共享狀態要外移,見 ch05/ch08),所以幾乎無鎖、無 context-switch 抖動。

**關鍵內幕:多個 worker 都從 master 繼承了同一個 listen socket 的 fd,誰來 `accept()` 新連線?**

兩種機制,對應兩個時代:

1. **`accept_mutex`(舊默認)**:worker 搶一把鎖,搶到的才去 `accept()`。為什麼要鎖?——**驚群(thundering herd)**:不加控制時,一個新連線到達會喚醒**所有**阻塞在 `accept()` 的 worker,但只有一個能成功,其餘白醒一場,浪費 CPU。鎖讓同一時刻只有一個 worker 去接,避免驚群。缺點:鎖本身是序列化點,高並發下成瓶頸,且負載可能不均。
2. **`SO_REUSEPORT`(現代默認,Linux 3.9+)**:每個 worker 各自 `bind()` 同一埠(內核允許多個 socket 共用埠),**由內核用四元組 hash 把新連線直接分發到某個 worker 的佇列**。沒有驚群、沒有用戶態鎖、負載更均。代價:同一時刻關閉某 worker 可能讓它佇列裡的連線被丟(滾動重啟時的細節)。

> **面試點**:「Nginx 怎麼避免驚群?」→ 早期用 `accept_mutex` 序列化 accept,現代用內核 `SO_REUSEPORT` 直接分發。能說出「驚群是什麼、為什麼浪費」就到位了。

## 2. 🔬 事件迴圈:一個單線程 worker 為什麼能服務上萬連線

這是 Nginx「快」的真正原因,也是 thread-per-connection 模型撞牆的地方。

**thread-per-connection 的牆**:一個連線一個線程,1 萬連線 = 1 萬線程。線程要棧記憶體(MB 級),要被內核調度(context switch 成本),大部分線程其實**卡在等 I/O(等客戶端發完、等後端回)什麼都沒做**。連線數一上來,記憶體和調度開銷先崩。

**Nginx 的解法:單線程 + epoll + 非阻塞 I/O。**

```
worker 進程裡的事件迴圈(簡化):
  while (true) {
    events = epoll_wait(epfd, ...)   // 阻塞,直到「有 fd 就緒」才返回(就緒 = 可讀/可寫)
    for (fd in events) {
      handle(fd)                      // 處理這個就緒的 fd,絕不阻塞;沒數據就掛起這個請求,去處理下一個
    }
  }
```

- **非阻塞**:對每個 socket 設 `O_NONBLOCK`,讀不到數據立刻返回(而非卡住整個線程)。
- **epoll(Linux 的 I/O 多路復用)**:一次 `epoll_wait` 就能拿到「**這批 fd 裡哪些就緒了**」。關鍵是它的成本是 **O(就緒數)而非 O(總連線數)**——1 萬連線裡只有 100 個此刻有數據,`epoll_wait` 只返回那 100 個。這就是它能撐住海量**空閒長連線**的根本。
- **edge-triggered(邊緣觸發,ET)**:Nginx 用 ET 模式——fd 從「無數據」變「有數據」時**只通知一次**,所以拿到通知後要**一次把數據讀乾**(讀到 `EAGAIN`)。對比 level-triggered(LT,只要還有數據就反覆通知)。ET 通知少、效率高,但程式必須讀乾,寫起來更講究。

> 🔗 epoll 的內核實現(紅黑樹存 fd、就緒鏈表、ET vs LT 的喚醒細節)在 **`performance-tuning-roadmap/00-os-fundamentals/`** 與 **`python-concurrency/00-execution-model/`** 有深挖,本章不重講——你只要記住:**epoll 讓「一個線程盯著上萬連線、只為就緒的那幾個幹活」成為可能。**

一句話收口:**thread-per-conn 把錢花在「等」上(每個等待者佔一個線程);event-loop 只為「就緒」的連線幹活(等待者只是一個 fd)。** 這就是 C10K 問題的答案,也是所有現代網關/代理(Nginx、Envoy、HAProxy)的共同底座。

## 3. 🔬 upstream 連線池:別對後端重複握手

代理把請求轉給後端(upstream)時,如果每個請求都新建一條 TCP 連線、用完就關,就要反覆付 **TCP 三次握手 + 慢啟動 + (若 HTTPS)TLS 握手** 的成本——後端 QPS 高時這是純浪費。

**解法:對後端維持一個長連線池,復用。**

```nginx
upstream backend {
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
    keepalive 32;                 # 每個 worker 對這個 upstream 保留最多 32 條空閒長連線
}
server {
    location / {
        proxy_pass http://backend;
        proxy_http_version 1.1;   # keepalive 需要 HTTP/1.1
        proxy_set_header Connection "";   # 清掉逐跳的 Connection: close,才能復用
    }
}
```

- `keepalive 32`:**每個 worker** 對該 upstream 保留的空閒連線數(不是全局數——4 worker 就是 4×32)。
- **為什麼要 `proxy_http_version 1.1` + 清空 `Connection` 頭**:HTTP/1.0 默認短連線;且客戶端的 `Connection: close` 是**逐跳(hop-by-hop)**頭,代理若原樣轉給後端就會讓後端關連線、池子失效。清掉它,代理才能維持自己到後端的長連線。
- **連線池打滿會怎樣**:空閒連線不夠時新建,超過上限的空閒連線會被關。若後端慢、連線都在忙,新請求要麼等、要麼新建——這時連線池其實是一道**隱性的並發上限**,和限流/背壓相關(見 ch05、`concurrency-capacity/05-pools`)。

> **面試點**:「代理到後端為什麼要連線池?」→ 省掉每請求的 TCP+TLS 握手與慢啟動;「為什麼要清 Connection 頭?」→ 它是逐跳頭,不清會讓後端關連線、池子白做。

## 4. 🔬 TLS 終止:把加解密收在邊緣

**TLS 終止(TLS termination)= 客戶端的 HTTPS 在網關這一層解密,網關到後端走明文或重新加密。**

為什麼要在邊緣終止:

- **握手很貴**:TLS 握手要做**非對稱加密**(RSA/ECDHE,昂貴)協商出一個對稱密鑰,之後的數據傳輸才用便宜的**對稱加密**(AES)。非對稱握手是 CPU 大戶。把它集中在網關,可以用專門優化(硬體加速、session resumption、OCSP stapling)統一處理,後端服務不必各自背 TLS 開銷。
- **證書集中管理**:憑證、輪換、SNI(一個 IP 多域名)都在網關一處,後端不碰。
- **後端怎麼辦**:
  - **明文轉發(TLS offload)**:網關→後端走 HTTP。簡單、快,前提是內網可信。
  - **重新加密(re-encrypt / TLS bridging)**:網關解密看內容(才能路由/改寫/鑑權),再用一條新 TLS 連到後端。零信任內網用這個。
  - **TLS passthrough**:網關**不解密**,直接按 SNI 把整條 TLS 流轉給後端(這其實是 L4 行為,見下節)——網關看不到 HTTP 內容,也就做不了 L7 的路由/鑑權。

> **取捨一句話**:要做 L7 的事(按 path 路由、鑑權、改寫),就**必須**終止 TLS(解密才看得到 HTTP);只想當啞管道轉發,才用 passthrough。

## 5. 🔬 L4 vs L7:這台引擎工作在哪一層,決定它能做什麼

這是反向代理最本質的一條分界,也是高頻面試題。

| | **L4 代理**(傳輸層) | **L7 代理**(應用層) |
|---|---|---|
| 看得到什麼 | 只看 TCP/UDP 的 **IP:port**(四元組) | 解析完整 **HTTP**:method/path/header/body |
| 能做的路由 | 按目標 IP/port 轉發 | 按 host/path/header/cookie 路由 |
| 能不能改內容 | 不能(不解包) | 能(改 header、改 body、轉碼) |
| 能不能鑑權/限流到用戶 | 不能(不認識身份) | 能(看得到 token、看得到 API) |
| 成本 | 低(不解 HTTP,近乎轉發位元組) | 高(要解包、緩衝、再封包) |
| 典型 | LVS、雲廠商 NLB、HAProxy(tcp 模式) | Nginx、Envoy、所有 API 網關 |

**本質**:L4 早(在 TCP 層就決定轉給誰),**快但瞎**——它不認識 HTTP,做不了任何「懂業務」的事。L7 晚(等 HTTP 解析出來),**慢一點但全能**——能路由、能鑑權、能改寫。

**回扣脊柱(ch00)**:這正是為什麼 API 網關一定是 **L7**——它要「認識用戶/API/配額」,就**必須**解析到應用層。L4 永遠停在反向代理的最左端。

> 生產裡常見**兩層疊用**:外層一個 L4(LVS/NLB)做超高吞吐的入口分流 + 抗 DDoS,後面再接一排 L7(Nginx/Envoy)做真正的 API 治理。L4 扛量、L7 懂業務。

🔗 負載均衡演算法本身(輪詢/最少連線/一致性哈希)、L4/L7 在 LB 語境的對比,在 **`system-design/05-服務治理設施.md`** 已深講,本章只講「代理引擎」視角。

## 6. 跨生態對照:同一台引擎的三種長相

| | **Nginx** | **HAProxy** | **Envoy** |
|---|---|---|---|
| 並發模型 | master + 多 worker(進程)+ epoll | 事件驅動(單/多進程) | 多線程 + 事件驅動 |
| 強項 | 反代/靜態/L7,生態廣、配置成熟 | L4/L7 負載均衡、極致穩定 | **動態配置(xDS)**、雲原生、可觀測 |
| 配置 | 文件 + reload(較靜態) | 文件 + reload | **API 動態下發**(不重啟改路由,見 ch08) |
| 在脊柱哪 | 反向代理 / 輕量 API 網關 | L4/L7 LB | API 網關 / mesh 資料面的事實標準 |

要點:**底座都是「事件驅動 + 多路復用」**(第 2 節那套),差別在**配置是靜態文件還是動態 API**——這條差別到 ch08(網關作為分散式系統)會變成核心。Envoy 之所以成為現代網關/mesh 的資料面標準,關鍵就是它的配置能被控制面**動態推送**。

## 本章小結

- 網關永遠是**反向代理**:對外假裝成單一入口。
- **master-worker**:master 綁埠/管理,worker(= CPU 核)單線程處理;靠 `accept_mutex` 或 `SO_REUSEPORT` 避免驚群。
- **事件迴圈(epoll + 非阻塞 + ET)**:一個線程只為「就緒」的 fd 幹活,所以能用極少進程扛海量連線——這是 C10K 的答案,也是所有現代代理的共同底座。
- **upstream 連線池**:復用到後端的長連線,省握手;記得 `proxy_http_version 1.1` + 清 `Connection` 頭。
- **TLS 終止**:握手貴(非對稱)→ 集中在邊緣;要做 L7 就必須終止 TLS。
- **L4(快但瞎)vs L7(慢但全能)**:API 網關必為 L7;生產常 L4+L7 兩層疊用。
