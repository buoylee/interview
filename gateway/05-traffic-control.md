# 05 · 流量控制:限流、熔斷、降級、卸載 🔬

> 一句話:管線的 **⑤限流** 段,加上對後端的保護。這章是你最初問的「**併發控制**」的正主——網關怎麼保護自己和後端不被打死:**限流**(主動設上限)、**熔斷**(後端壞了快速失敗)、**load shedding**(過載了主動丟)、**降級**(保核心棄枝節)。

限流**算法本身**(令牌桶/漏桶/滑動窗口)在 `system-design/01` 講透;**Lua 原子實作**在 `redis-handson/13` 有可跑範例;**過載/背壓**在 `concurrency-capacity/07`。本章的增量是一個你在那些地方找不到的問題:**N 台網關怎麼合起來限流?**

---

## 1. 限流落在哪、按什麼維度

落點在管線 ⑤(回扣 ch02 的「限流放鑑權前還後」取捨)。維度從粗到細:

| 維度 | 例 | 何時用 |
|---|---|---|
| 全域 | 整個 API 每秒 10 萬 | 保護後端總容量 |
| per-route | `/search` 限得比 `/health` 嚴 | 貴的接口單獨護 |
| per-consumer | 金牌 1000/s、免費 10/s | 按用戶/API Key 配額(需先鑑權,ch04) |
| per-IP | 每 IP 100/s | 擋匿名洪水(鑑權前) |

**算法一句話回顧**(細節去 `system-design/01`):**令牌桶**(勻速發令牌、允許突發,最常用)、**漏桶**(勻速漏出、削峰平滑)、**固定窗口**(簡單但有臨界突刺)、**滑動窗口**(平滑但要更多計數)。網關默認多用**令牌桶**。

## 2. 🔬 本章核心:N 台網關怎麼合起來限流

單機限流很簡單:本地一個計數器。但網關是**叢集**(ch08),前面掛個 LB,請求隨機落到任一台。問題來了:

> 你要「**每個用戶全域** 100/s」。但 5 台網關各自只看到自己收到的流量。各算各的,怎麼保證**合起來**是 100/s?

這是分散式限流的核心難題。三種解法,沿「準確 ↔ 成本」光譜:

### 解法 A:本地限流(local counter)—— 各算各的
- 做法:全域 100/s 平攤,每台限 `100/5 = 20/s`,各自本地計數。
- 優點:**零協調、零延遲、零外部依賴**,極快。
- 缺點:**總量會飄**。流量不均時(LB hash 傾斜、某台掛了剩 4 台),要麼超(實際放過 >100)要麼欠(誤殺)。且擴縮容要重算配額。
- 適用:對精度不敏感、只要「大致擋住洪水」的粗保護。

### 解法 B:集中計數(global counter)—— 共享一個存儲
- 做法:計數放 **Redis**,每台網關每個請求去 `INCR` 同一個 key,讀回判斷。
- 優點:**準**。全域一個真相源。
- 缺點:每請求多一次網路往返(延遲 + Redis 成為熱點/單點,要 ch08 的 HA);Redis 掛了限流怎麼辦(fail-open 放過 vs fail-close 全擋,要明確決策)。
- **🔬 為什麼必須用 Lua 原子**:`INCR` 然後「讀回來判斷要不要 `EXPIRE`」是**讀-改-寫兩步**,高並發下兩台網關同時跑會競態(都看到計數=99,都放行,實際變 101)。把「INCR + 首次設過期 + 判斷」打包成一段 **Lua 腳本**,Redis **單線程**內**原子**執行,杜絕競態。
  > 🔗 可跑的 Lua 限流實作在 `redis-handson/13-rate-limiting/`,**ch05 的 lab(`gateway/lab/distributed-rate-limit/`)也親手驗證**:2 台節點共用一個 Redis 計數,合起來才是全域配額。

### 解法 C:令牌預取 / 批量租約(近似,兼顧準與快)
- 做法:每台網關**一次性向中央領一批配額**(如領 10 個令牌的「租約」),本地消耗,用完再領。
- 優點:把「每請求一次往返」攤薄成「每批一次往返」,**準確度接近 global、成本接近 local**。
- 缺點:實作複雜;批量大小是準確/性能的旋鈕(批大→快但飄,批小→準但往返多)。
- 這是大規模網關(如 Envoy 全域限流服務、雲廠商網關)的常見做法。

**面試一句話答法**:單機本地計數最快但叢集下總量會飄;要準就用 **Redis 集中計數 + Lua 原子**(代價是每請求一跳和 Redis 單點);要兼顧就用**令牌預取**批量租約。

## 3. 🔬 熔斷:後端壞了,別再往火坑裡送請求

限流是「保護**我自己**不被上游打死」;熔斷是「保護**下游**不被我打死、也保護我自己不被慢下游拖死」。

- **三態機(白板必畫)**:**Closed**(正常放行,統計失敗率)→ 失敗率超閾值 → **Open**(直接快速失敗,**完全不打**後端,給它喘息)→ 過冷卻期 → **Half-Open**(放幾個試探請求)→ 成功則回 Closed,失敗則回 Open。
- 為什麼關鍵:後端已經過載/掛了時,繼續打它只會讓它更死、讓你的連線/線程全卡在等它(回扣 ch01 連線池、ch03 超時)。熔斷 = **快速失敗**,把資源讓給還能服務的路徑。
- 與 ch03 健康檢查的分工:健康檢查摘**整台**壞實例;熔斷針對**某條依賴/某個 upstream 服務**的整體健康做開關。
> 🔗 三態機細節、失敗率窗口在 `system-design/01`。

## 4. 🔬 Load shedding(負載卸除):過了飽和點,主動丟

限流是「按預設規則擋」;load shedding 是「**系統實時快撐爆了,主動丟掉一部分請求**」,是最後一道保命閘。

- **為什麼要主動丟**:過了飽和點還硬接,延遲不是線性變差而是**崩**(排隊無限增長);無界佇列是慢性自殺。**丟掉一部分、讓剩下的正常**,好過全部一起慢死(metastable failure——故障源消失了系統還起不來)。
- **怎麼判斷該丟**:按**佇列長度 / 實時延遲 / CPU** 觸發;按**優先級**丟(先丟可重試的、低價值的;保住核心交易)。
- 這就是**背壓(backpressure)**在邊緣的體現:網關感到後端扛不住,把壓力反推回客戶端(`429`/`503`),而不是無腦緩衝。
> 🔗 shed vs block、無界佇列、重試風暴、metastable failure 在 `concurrency-capacity/07-overload-backpressure/` 深講。

## 5. 降級:保核心,棄枝節

- **主動降級**:預判高峰(大促),提前關掉非核心功能(推薦、評論),把資源讓給核心(下單、支付)。
- **被動降級**:依賴掛了,返回**兜底**——快取的舊數據、默認值、友好的「稍後再試」,而非把錯誤直接拋給用戶。
- 網關層常見:某後端熔斷後,網關直接返回一個 fallback 回應(快取/靜態),用戶無感或弱感。

---

## 本章小結

- 流量控制四件套:**限流**(主動設上限)、**熔斷**(下游壞了快速失敗)、**load shedding**(實時過載主動丟)、**降級**(保核心棄枝節)。
- **分散式限流是本章核心**:本地計數(快但飄)→ Redis 集中計數 + **Lua 原子**(準但一跳 + 單點)→ 令牌預取(近似,兼顧)。
- 熔斷三態機 **Closed→Open→Half-Open**;與健康檢查分工(摘實例 vs 對依賴開關)。
- load shedding 是最後保命閘:**丟一部分好過全死**,按優先級丟,是背壓在邊緣的體現。
