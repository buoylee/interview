# 00 · 決策地圖:網關到底是什麼、何時不需要

> 一句話:**網關 = 站在系統邊緣的「可程式化代理 programmable proxy」。** 外部請求打不到你的業務碼,中間一定先落到一個代理上;你讓這個代理多管事,它就一級一級進化成 API 網關、Service Mesh、AI 網關。

這一章不教任何具體技術,只給你**整體觀念**——看完你就知道後面 11 章彼此什麼關係,以及更重要的:**什麼時候根本不該上網關**。

---

## 1. 演化脊柱:一張圖看懂「網關」這個詞的所有意思

「網關」是個被講爛的詞,同一個字能指四種差很多的東西。它們其實是同一根脊柱上的不同段:

```
                     認識的東西越來越「懂業務」 →
┌─────────────┬──────────────────┬───────────────────┬─────────────────┐
│ 反向代理      │   API 網關         │  Service Mesh      │  AI / LLM 網關    │
│ Nginx/HAProxy│  Kong/APISIX/      │  Istio+Envoy/      │  LiteLLM/Portkey/ │
│              │  Envoy/SC Gateway  │  Linkerd (sidecar) │  Envoy AI Gateway │
│ host/path 路由│ +認證/授權/限流     │ 治理下沉到 sidecar  │ +模型路由/token   │
│ TLS/LB/靜態  │ +熔斷/協議轉換/聚合 │ 管東西向(服務間)   │ +語義快取/護欄    │
│ (config 驅動) │ 認識用戶/API/配額   │ (網關管南北向)      │ 認識 token/模型   │
└─────────────┴──────────────────┴───────────────────┴─────────────────┘
```

**核心洞察(整個 track 的靈魂):後面每一層 = 前一層 + 更多「認識業務」的橫切能力。**

- **反向代理(Nginx)** 認識 `host / path / IP`——它不知道「誰」在請求、這是哪個 API。所以它能路由、能限流,但限流只能到 IP 粒度,鑑權只能看 header 有沒有某個值。
- **API 網關** 認識 `用戶 / API / 配額`——所以它能做「**這個用戶**每秒只能調 **這個 API** 10 次」「這個 token 過期了,打回」。這就是反向代理升級成 API 網關的本質:**從認 IP 到認身份**。
- **Service Mesh** 是同一批治理能力(mTLS、重試、熔斷),但**下沉**到每個服務旁的 sidecar,管服務**之間**的流量。
- **AI 網關** 認識 `token / 模型 / prompt / 成本`——它是 API 網關針對 LLM 流量的特化版。

> **給 Java/Go 背景的橋接**:你寫過的 Spring `Filter`/`HandlerInterceptor`、Go 的 `middleware` 鏈,就是「在請求到業務碼之前先過一串橫切邏輯」。網關把這件事**從進程內提升成獨立的網路元件**——跨服務、跨語言、跨團隊共用一套。心智模型一樣,只是搬到了網路邊緣。

## 2. 南北向 vs 東西向:先分清你在說哪種流量

這是面試對網關**最常見的混淆點**,也是「網關和 Service Mesh 到底什麼區別」這道題的答案核心。先把流量分兩類:

```
        外部 client(瀏覽器 / App / 第三方)
                  │   ← 南北向 North-South(入口流量)
                  ▼
        ┌──────────────────┐
        │   API 網關         │   ← 管南北向
        └──────────────────┘
                  │
   ┌──────────────┼──────────────┐
   ▼              ▼              ▼
[服務 A]  ←──→  [服務 B]  ←──→  [服務 C]
        ↑ 東西向 East-West(服務間流量)← Service Mesh 管這個
```

- **南北向(north-south)**:外部 ↔ 系統入口。**API 網關**的地盤。一個系統通常一個(邏輯上的)入口網關。
- **東西向(east-west)**:服務 ↔ 服務(內網)。**Service Mesh** 的地盤(透過每個 Pod 旁的 sidecar)。

**為什麼非先分不可**:很多治理能力(限流、重試、熔斷、mTLS、追蹤)南北東西**都要**,但**放的位置不同、做法不同**。把這條軸畫清楚,後面 ch09 才能把「有了網關為什麼還要 mesh」講透。記住一句話面試答法:**網關管入口(南北),mesh 管服務間(東西),它們是互補不是二選一。**

---

## 3. 網關到底解決什麼問題:橫切關注點的集中化

沒有網關時,每個後端服務都得自己實現:驗 token、限流、記訪問日誌、注 traceId、處理 CORS……結果是**重複、不一致、難治理**(改一條限流規則要動 N 個服務)。

網關的本質價值就一句:**把這些「每個服務都要、又跟業務無關」的橫切關注點,從各服務裡抽出來,收到邊緣一處統一做。**

| 橫切關注點 | 收到網關後的好處 |
|---|---|
| 認證 / 授權 | 後端服務可以「假設請求已經是可信的」,不必各自驗 |
| 限流 / 配額 | 一處定義「每個租戶每秒多少」,全站一致 |
| 路由 / 灰度 | 改流量切分不動業務碼 |
| 協議轉換 | 外部 REST、內部 gRPC,翻譯在邊緣完成 |
| 可觀測性 | 所有入口流量必經一點,埋點一次覆蓋全站 |

這也是為什麼面試愛考它:**它是分散式系統裡橫切知識點最密集的單一元件**——答好它等於把限流、熔斷、鑑權、路由、觀測一次性串起來。

## 4. 何時**不該**上網關(避免過度設計)

網關不是免費的。它是**新的一跳延遲、新的 SPOF、新的待運維元件**。下面這些情況,別急著上 Kong/APISIX 全家桶:

- **單體或極少數服務、內網直連夠用**:沒有「跨服務的橫切需求」,網關只是徒增一跳。
- **只需要 L7 路由 + TLS 終止**:一台 Nginx 或雲廠商的 ALB/Application Gateway 就夠,不必上重型 API 網關。**先用脊柱左端(反向代理),別一步跳到中段。**
- **團隊扛不住額外運維**:網關掛了全站掛——它本身需要 HA、容量規劃、配置治理、故障演練(見 ch08)。團隊還沒這個成熟度時,過早引入等於給自己埋一個不懂的黑盒。
- **能力已有更合適的歸宿**:服務間的重試/熔斷,可能用 mesh(ch09)或客戶端庫更合適,不必硬塞進南北向網關。

**判斷準則一句話**:你有沒有**跨多個服務的、與業務無關的橫切需求**(統一鑑權、統一限流、統一灰度)?**有,才值得引入網關;沒有,就是過度設計。** 而且要**沿脊柱按需漸進**:先反向代理,真有 API 治理需求了再升 API 網關,真有大規模服務間治理了再上 mesh。

---

## 本章小結

- 網關 = 邊緣的可程式化代理;**反向代理 → API 網關 → Mesh → AI 網關** 是同一根脊柱,後面每層多「認識」一層業務。
- **南北向(入口)= 網關;東西向(服務間)= mesh**——這是「網關 vs mesh」的答題核心。
- 網關的本質價值 = **橫切關注點集中化**;這也是它成為面試密集考點的原因。
- 不是所有系統都該上網關;**有跨服務橫切需求才值得**,且要沿脊柱漸進,別過度設計。
