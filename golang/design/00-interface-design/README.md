# 00 · 接口设计哲学:小接口、消费方定义、接受接口返结构体

> Go 的接口设计和 Java 截然不同:**接口要小、由消费方定义、函数「接受接口、返回结构体」**。这是 idiomatic Go 的核心,也是架构师面试爱挖的设计题。机制在 [`type-system/01`](../../type-system/01-interface-internals/README.md),这里讲**怎么设计**。
>
> 桥接锚点:Java 习惯「先定义大接口 `UserService`,再 `UserServiceImpl` 实现」(生产方定义、接口往往很大);Go 反过来——**使用方按需定义小接口,实现方根本不知道这个接口存在**(隐式实现)。

---

## 1. 核心问题

- Go 接口该多大?为什么 `io.Reader` 只有一个方法?
- 接口该由谁定义——提供实现的人,还是使用它的人?
- 为什么 idiom 是「**接受接口,返回结构体**」而不是反过来?

---

## 2. 直觉理解

### 接口越大,抽象越弱(Go proverb)

> *"The bigger the interface, the weaker the abstraction."* —— Rob Pike

一个 10 方法的接口,几乎没人能简单实现、也没人能简单替换它,抽象力极弱;而 `io.Reader` 只要一个 `Read([]byte)(int,error)`,**无数类型能满足它**(文件、网络、内存、加密流……),组合力极强。Go 标准库的威力正来自这些单方法接口(`Reader`/`Writer`/`Closer`/`Stringer`)。

准则:**接口尽量小,一两个方法最佳;大接口拆成小接口再组合**(`io.ReadWriter = Reader + Writer`)。

### 接口由消费方定义,而非生产方

Java:`UserService`(接口)和 `UserServiceImpl`(实现)成对出现,接口由**提供方**定义、往往很大。

Go:**谁用接口,谁定义接口**——而且只定义「我用到的那几个方法」。因为接口是**隐式实现**的,实现方(如 `*sql.DB`)根本不需要知道你的接口存在:

```go
// 消费方:我只需要 "能按 id 查名字" 这一个能力
package report
type nameQuerier interface { QueryName(id int) (string, error) }   // 小、私有、按需

func Generate(q nameQuerier, id int) { ... }    // *sql.DB、fake 都能传进来
```

好处:解耦(不依赖庞大的真实类型)、好测(fake 只需实现一个方法,见 [`testing/01`](../../testing/01-mock-interfaces/README.md))、好演进。

### 接受接口,返回结构体

> *"Accept interfaces, return structs."*

- **参数用接口**:让调用方能传任意满足的实现(灵活、可测)。
- **返回值用具体结构体**(`*Client`、`*Server`):返回接口会**隐藏具体类型的字段和方法**、强迫调用方只能用接口那点能力,还让 API 难演进(加方法即破坏接口)。

```go
func NewClient(opts ...Option) *Client          // ✅ 返回具体类型
func Process(r io.Reader) (*Result, error)      // ✅ 接受接口
```

## 3. 原理深入

### 3.1 为什么「返回接口」常是反模式

- **泄漏抽象 / 丢失能力**:返回 `io.Reader`,调用方就用不了你具体类型的其它方法,即使它有。
- **难演进**:给返回的接口加方法 = 破坏所有实现者;返回结构体加方法则向后兼容。
- **逃逸/装箱**:返回接口会装箱(见 [`type-system/01`](../../type-system/01-interface-internals/README.md))。
- 例外:确实要返回多态结果时(如工厂返回不同实现)才返回接口;且接口应小。标准库 `net.Conn`、`error` 是合理的「返回接口」例子(本质就是多态)。

### 3.2 接口隔离:大接口拆小再组合

```go
type Reader interface { Read(p []byte) (int, error) }
type Writer interface { Write(p []byte) (int, error) }
type Closer interface { Close() error }
type ReadWriteCloser interface { Reader; Writer; Closer }   // 嵌入组合(type-system/04)
```

需要什么能力就声明什么接口:只读的函数收 `io.Reader`,别收 `io.ReadWriteCloser`。这让函数签名**精确表达它真正需要的最小能力**。

### 3.3 「先有实现,后有接口」

Go 的演进顺序常是:先写具体类型把事做成,**等出现第二个实现 / 测试需要替换时,才在消费方抽出小接口**。过早定义接口(YAGNI 反例)会得到「只有一个实现的接口」——纯增噪音。Go 文化:**不要为了接口而接口**。

### 3.4 对照 Java

| | Java | Go |
|---|---|---|
| 接口大小 | 常较大(Service 全集) | 尽量小(1-2 方法) |
| 定义方 | 生产方(随实现成对) | **消费方**(按需) |
| 实现声明 | 显式 `implements` | 隐式 |
| 返回 | 常返回接口 | **返回结构体** |
| 何时定义 | 先定义接口再实现 | 先实现,需要时再抽接口 |

## 4. 日常开发应用

- **函数参数收最小接口**:只读就 `io.Reader`,别收大接口或具体类型。
- **接口定义在使用它的包**,只声明用到的方法;别建一个 `interfaces.go` 把所有接口堆一起(那是 Java 习惯)。
- **构造函数返回 `*T`**(具体类型),不返回接口。
- **不要过早抽接口**:一个实现就先用具体类型,出现第二实现/测试需求再抽。
- **大接口拆小 + 嵌入组合**(`io.ReadWriteCloser`)。

---

## 5. 生产&调优实战

- **小接口 = 可演进 + 可测 + 可组合**:大型代码库里,小接口让你替换实现、注入 fake、加装饰器都轻松;大接口是耦合和僵化的根源。
- **「返回结构体」利于库演进**:对外库返回具体类型,后续加方法不破坏兼容;返回接口则加方法即 break。
- **避免接口污染**:每个具体类型都配一个同名接口(`Foo`+`FooIface`)是 Java 式过度设计,徒增维护;只在真需替换处抽接口。
- **接口装箱成本**:热路径上把具体类型硬转成接口会逃逸(见 [`data-structures/03`](../../data-structures/03-escape-analysis/README.md));参数用接口没问题,但别无谓地到处转接口。

---

## 6. 面试高频考点

- **Go 接口该多大?** 尽量小(1-2 方法)。「接口越大,抽象越弱」——`io.Reader` 是典范;大接口拆小再嵌入组合。
- **接口由谁定义?** **消费方**(谁用谁定义,只声明用到的方法),靠隐式实现解耦;不像 Java 由生产方定义大接口。
- **为什么「接受接口返结构体」?** 参数用接口=灵活可测;返回结构体=不隐藏能力、易演进(加方法不破坏兼容)、免装箱。返回接口常是反模式(泄漏抽象/难演进)。
- **什么时候才定义接口?** 出现第二个实现 / 测试要替换时再抽;别过早抽出「只有一个实现的接口」。
- **接口隔离怎么做?** 函数只收它需要的最小能力(只读收 Reader);大接口用嵌入拆组合。
- **和 Java 接口设计区别?** Java 大接口、生产方定义、显式实现、返回接口、先接口后实现;Go 小接口、消费方定义、隐式、返结构体、先实现后接口。

---

## 7. 一句话总结

> **idiomatic Go 接口设计四原则:① 接口尽量小**(「越大越弱」,`io.Reader` 单方法是典范,大接口拆小再嵌入组合);**② 由消费方定义**(谁用谁定义、只声明用到的方法,靠隐式实现解耦,不像 Java 生产方定义大接口);**③ 接受接口、返回结构体**(参数用接口=灵活可测,返回具体类型=不隐藏能力、易演进、免装箱,返回接口多是反模式);**④ 不要过早抽接口**(先写具体实现,出现第二实现或测试需替换时再抽,别造「只有一个实现的接口」)。这套让 Go 代码可组合、可测、可演进。
