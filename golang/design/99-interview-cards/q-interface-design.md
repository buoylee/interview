# 接口设计:小接口·消费方定义·接受接口返结构体

## 一句话回答

idiomatic Go 接口四原则:**① 接口尽量小**(「接口越大,抽象越弱」,`io.Reader` 单方法是典范,大接口拆小再嵌入组合);**② 由消费方定义**(谁用谁定义、只声明用到的方法,靠隐式实现解耦,不像 Java 由生产方定义大接口);**③ 接受接口、返回结构体**(参数用接口=灵活可测,返回具体类型=不隐藏能力·易演进·免装箱,返回接口多是反模式);**④ 不要过早抽接口**(先写具体实现,出现第二实现或测试需替换时再抽)。

## 为什么「返回结构体」

- 返回接口会**隐藏具体类型的字段/方法**、强迫调用方只用接口那点能力。
- **难演进**:给返回的接口加方法 = 破坏所有实现者;返回结构体加方法向后兼容。
- 返回接口还**装箱**(逃逸)。
- 例外:确实多态(工厂返回不同实现,如 `net.Conn`/`error`)才返回接口,且接口要小。

## 易追问的延伸

- **为什么消费方定义?** 隐式实现 → 使用方只声明需要的方法,不耦合庞大真实类型;别建 `interfaces.go` 堆所有接口(那是 Java 习惯)。
- **接口隔离?** 函数只收需要的最小能力(只读收 `io.Reader`),大接口拆小嵌入组合。
- **接口污染?** 每个类型配一个同名接口是过度设计,只在真需替换处抽。
- **和 Java?** Java 大接口/生产方定义/显式实现/返接口/先接口后实现;Go 反之。
