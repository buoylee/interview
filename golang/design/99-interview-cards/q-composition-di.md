# 组合与依赖注入(wire vs 手动)

## 一句话回答

Go 没有继承,复用靠**组合**(嵌入 + 方法提升,但无多态重写——多态用接口);没有 Spring 容器,依赖装配靠**显式构造注入**:依赖声明为**接口字段**(可注入 fake)、`NewXxx(deps...) *Xxx` 传入、`main` 作组装根。依赖图大时用 **wire**(google/wire)——它**编译期生成普通 Go 装配代码、无运行时反射**(本质区别于 Spring 的运行时容器),但默认手动 DI 更透明、中小项目够用。

## wire vs Spring

| | wire | Spring |
|---|---|---|
| 时机 | 编译期生成代码 | 运行时反射注入 |
| 失败 | 编译期报错 | 可能运行时才报「找不到 bean」 |
| 可见性 | 生成的代码可读可跳转 | 容器内部魔法 |

## 关键原则

- 依赖是**接口字段**不是具体类型 → 可测(注入 fake,见 [`testing/01`](../../testing/01-mock-interfaces/README.md))。
- **「一点拷贝优于一点依赖」**:为小复用引大依赖不划算,宁可拷几行;警惕依赖膨胀。
- 装饰器/中间件 = 嵌入接口 + 重写部分方法转发 / 函数包装。

## 易追问的延伸

- **没继承怎么复用通用逻辑?** 嵌入 BaseRepo/BaseModel(方法提升);要多态用接口。
- **手动 DI 不累吗?** 中小项目就是 main 里一串 New,透明;大了再 wire 减样板。
- **wire 生成代码要提交吗?** 要(见 [`engineering/02`](../../engineering/02-toolchain/README.md))。
- **别滥用嵌入做继承**:嵌入会提升内嵌的全部导出方法成你的 API,可能泄漏;不想暴露用具名字段+转发。
