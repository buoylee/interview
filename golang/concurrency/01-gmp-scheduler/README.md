# 01 · GMP 调度器

> 调度器内核三章之一，本课的**核心**。`00` 给了你一张「G 靠 runtime 复用到 M 上」的全链路图，但中间冒出个 **P** 没解释。本章把这张图拆开：**G/M/P 各是什么、为什么非要有 P、一个 goroutine 是怎么被排上队、被偷、被跑的。**
>
> 桥接锚点：Java 的 **ForkJoinPool 工作窃取**（Loom 的调度器就是它）——P 的本地队列 ≈ ForkJoinPool worker 的双端队列。

---

## 1. 核心问题

`00` 说了：G 是任务，M 是 OS 线程，runtime 把 G 复用到 M 上。那直接「G 排个全局队列，哪个 M 空了就来全局队列拿一个跑」不就行了？为什么还要插一个 **P**？

这正是 Go 调度器**最值得讲清**的一点。答案是：**只有 G/M + 一个全局队列，会被一把全局锁卡死。** 每个 M 每次找活、放活都要抢这把锁，核越多抢得越凶，根本扩展不上去。

**P 就是为了打碎这把全局锁而生的。** 它给每个「执行槽」配一个**本地队列**，让绝大多数调度操作变成**无锁的本地操作**。这一章回答：P 是什么、三个队列怎么配合、一个 G 从创建到被执行经历了什么。

---

## 2. 直觉理解

### G / M / P 三件套

| | 全称 | 是什么 | 类比 |
|---|---|---|---|
| **G** | Goroutine | 一个任务（带自己的栈、PC、状态） | 一份待办工单 |
| **M** | Machine | 一根真 OS 线程，真正执行代码 | 一个工人 |
| **P** | Processor | 「执行 Go 代码的许可证」+ 一条**本地任务队列** | 一个**工位**（带自己的待办筐） |

铁律（务必记住）：**M 必须先占到一个 P，才能执行 G。**

把它想成工厂：**工人（M）要先坐到一个工位（P）上，才能从工位旁边的筐（本地队列）里拿工单（G）来做。** 工位数量 = `GOMAXPROCS`（默认 = CPU 核数），是固定的；工人数量是动态的（一个工人去仓库搬货卡住了，可以再雇个工人顶上）。

### 为什么这个设计能并行又省锁

- **并行**：有几个 P，就能有几个 M 同时在跑 Go 代码 → 吃满 `GOMAXPROCS` 个核。
- **省锁**：每个 P 有自己的本地筐，工人拿自己工位的工单**不用和别人抢锁**。只有本地筐空了，才去公共仓库（全局队列）或**偷别人的筐**。

### Java 人的最佳镜子：ForkJoinPool

这套「每个 worker 一条双端队列 + 工作窃取」你其实见过——**Java 的 `ForkJoinPool` 就是这么干的**，而 Java 21 虚拟线程（Loom）的默认调度器**正是一个 ForkJoinPool**。对应关系：

| Go GMP | Java ForkJoinPool / Loom |
|---|---|
| P 的本地队列 | 每个 worker 线程的 work-stealing deque |
| work stealing（偷一半） | ForkJoinPool 的 work stealing |
| GOMAXPROCS（P 数量） | pool 的 parallelism |
| M（承载 G 的 OS 线程） | carrier / 平台线程 |

所以**你不是在学一个全新东西**——你是在 Go runtime 里重新认出 ForkJoinPool 的骨架。

> 反向对照 asyncio：asyncio **没有 P 这一层**，就一根线程、一个事件循环、一条就绪队列，没有工作窃取、没有并行。GMP 比它多出来的复杂度，换来的就是「真多核 + 自动负载均衡」。

## 3. 原理深入

### 3.1 为什么有 P：一段真实的历史

- **Go 1.0：GM 模型。** 只有 G 和 M，配**一个全局运行队列 + 一把全局锁（`Sched.Lock`）**。每个 M 找 G、放 G、创建 G 都要抢这把锁。核数一多，锁竞争把吞吐压垮，且缓存局部性差（G 在 M 之间乱跳）。
- **Go 1.1：引入 P（Dmitry Vyukov 重设计）。** 在 G 和 M 之间插入 P，每个 P 自带本地队列。从此**绝大多数调度走本地、无锁**，全局锁只在「访问全局队列 / 偷工作」时才用。这是 Go 调度器能扩展到多核的根本一步。

记住这条因果：**P = 为了把全局锁拆成 per-P 本地无锁。** 面试问「为什么要 P」，答这个。

### 3.2 三个「活儿的来源」

一个 M（坐在 P 上）找下一个要跑的 G，有三个来源：

1. **P 的本地运行队列（LRQ）**：一个**容量 256 的环形数组** + 一个特殊的 **`runnext` 单槽**。
   - `runnext` 放「最近刚被创建/唤醒、最该马上跑」的那个 G（局部性优化，下面 3.4 细说）。
   - 本地队列满了（超过 256），会把**一半（128 个）移到全局队列**，避免一个 P 囤太多。
2. **全局运行队列（GRQ）**：所有 P 共享，**需要加全局锁**。本地放不下的溢出、以及一些场景（如刚从系统调用回来）产生的 G 会进这里。
3. **netpoller**：网络 I/O 就绪后被唤醒的 G 从这里来（`03` 章主题）。

### 3.3 调度循环：一个 G 是怎么被选中的

每根 M 跑的其实是一个**调度循环**（在 M 的 `g0` 系统栈上执行）：

```
schedule():
    G = findRunnable()      // 找一个能跑的 G（可能阻塞直到有活）
    execute(G)              // 切到 G 的栈，跑它，直到它让出/结束
    // 回到 schedule()，再找下一个
```

核心是 `findRunnable()` 的**查找顺序**（白板常考）：

```
findRunnable():
  1. 每 61 次调度，先瞄一眼全局队列    ← 防止全局队列被饿死（magic number 61）
  2. 看自己 P 的本地队列（含 runnext）  ← 最常走这条，无锁
  3. 本地空 → 去全局队列拿一批         ← 加锁
  4. 还没有 → 查 netpoll（网络就绪的 G）
  5. 还没有 → work stealing：随机挑别的 P，偷它本地队列的一半
  6. 实在没有 → M 进入休眠（park），交还 P
```

两个值得记住的设计：

- **「每 61 次先看全局队列」**：如果永远先吃本地队列，全局队列里的 G 可能永远轮不到。用一个 61 的计数兜底，保证全局队列不被饿死。
- **work stealing 偷「一半」**：偷太少则很快又得再偷（开销），偷一半是均衡点——和 ForkJoinPool 思路一致。

### 3.4 `runnext`：为什么「刚 go 出来的」往往最先跑

当你 `go f()`，新建的 G **不是**塞到本地队列尾部，而是优先放进当前 P 的 **`runnext` 单槽**，下一轮调度**优先**跑它。原因是**局部性**：刚创建的 G 通常和当前 G 数据相关（缓存还热），马上跑它能减少 cache miss。原来 `runnext` 里的 G 才被挤到队列尾。

> 这解释了一个常见现象：`go f(); g()` 时，谁先真正开始跑没有保证，但 `f` 借 `runnext` 往往很快被调度——不过**绝不能依赖这个顺序**写正确性逻辑。

### 3.5 M 数量、g0、spinning M

- **M 数量是动态的**，可以**超过 GOMAXPROCS**：当 M 阻塞在系统调用时，它带不走 P 干活，runtime 会创建/复用别的 M 来接管这个 P（`03` 章 syscall handoff）。默认 M 上限 10000（`debug.SetMaxThreads` 可调）。
- **每个 M 有一个 `g0`**：一个特殊的「系统 goroutine」，带一个较大的系统栈。**调度逻辑本身跑在 g0 上**，而不是在你的业务 G 上——切换 G 时其实是「业务 G → g0 → 下一个业务 G」。
- **spinning M（自旋）**：为降低延迟，runtime 会保持少量 M 处于「自旋找活」状态而不立即休眠，这样新活一来能立刻被捡起，省掉「唤醒一个睡着的 M」的延迟。代价是一点点 CPU 空转——延迟 vs 功耗的权衡。

### 3.6 把 `00` 的全链路图补全

```
go f()
  │  新 G 进当前 P 的 runnext（局部性）
  ▼
┌─────────────────── P0 ───────────────────┐   ┌─────── P1 ───────┐
│ runnext:[G_new]  本地队列:[G,G,G,…256]    │   │ 本地队列:[G,G]    │
└───────────┬──────────────────────────────┘   └─────────┬────────┘
            │ M0 占着 P0，从队列取 G 执行                  │ M1 占着 P1
            │ 本地空了就：全局队列 → netpoll → 偷 P1 一半  │
            ▼                                             ▼
          M0(OS线程)                                    M1(OS线程)
            │ 被内核调度到核                               │
            ▼                                             ▼
          CPU core 0                                    CPU core 1

全局运行队列(GRQ, 带锁) ──── 溢出/兜底，所有 P 共享
sysmon(独立监控线程) ──── 巡逻：抢占跑太久的 G(02章)、抢回阻塞的 P(03章)、跑 netpoll、强制 GC
```

## 4. 日常开发应用

你几乎**不直接操作 GMP**，但理解它能直接解释你会遇到的现象：

- **GOMAXPROCS 决定并行度**：CPU 密集任务想吃满 N 核，就需要 ≥N 个能跑的 goroutine 和 `GOMAXPROCS≥N`。设成 1，再多 goroutine 也只用一个核（但**不会**饿死——`02` 抢占保证轮转）。
- **goroutine 顺序无保证**：`runnext`/队列/窃取让调度顺序不可预测。任何依赖「先 go 的先跑」的逻辑都是错的——要顺序就用 channel/WaitGroup 显式同步（`05`/`06`）。
- **别长时间占着 M 不让出**：一个 goroutine 闷头跑 CPU，虽然 `02` 的抢占会 10ms 切它，但你若有大量这种 G，会把 P 都占满、挤掉 I/O 型 G 的及时响应。CPU 密集活该限并发度（`08` worker pool）。

```go
import "runtime"

func main() {
    println(runtime.NumCPU())        // 机器逻辑核数
    println(runtime.GOMAXPROCS(0))   // 当前 P 数量（传 0 = 只读不改）
    println(runtime.NumGoroutine())  // 当前存活 G 数
}
```

## 5. 生产&调优实战

- **观察调度器**：`GODEBUG=schedtrace=1000`（每秒一行）看 P 数、各运行队列长度、空闲/自旋 M 数；加 `scheddetail=1` 看每个 P/M/G 细节。
  ```bash
  GODEBUG=schedtrace=1000 ./app
  # 输出含: gomaxprocs=8 idleprocs=.. threads=.. runqueue=<全局队列长> [<各P本地队列长>]
  ```
  - **全局 `runqueue` 持续很长** = 活儿堆在全局队列、本地消化不过来，常见于 G 数远超 P 数的突发负载。
  - **idleprocs 长期 >0 而活还堆着** = 可能有 G 卡在阻塞调用（看 `03`）。
- **GOMAXPROCS 容器坑**（呼应 `00`）：Go ≤1.24 默认按**宿主机**核数设 P，而非 cgroup 限额——4 核 limit 的 Pod 跑在 64 核机器上会 `GOMAXPROCS=64`，制造大量无谓的 M 切换和调度开销。对策：`go.uber.org/automaxprocs`（import 即生效），或 Go 1.25+ 关注 runtime 对 cgroup 的感知。细节见 `09`。
- **`go tool trace`** 能可视化每个 P 上 G 的执行/阻塞/被偷时间线，排查「为什么某些 G 延迟高」最直观（`09` 章）。

## 6. 面试高频考点

- **讲一下 GMP / Go 调度模型？** G=任务、M=OS 线程、P=执行许可证+本地队列；M 必须占 P 才能跑 G；P 数=GOMAXPROCS；本地队列无锁、全局队列带锁、工作窃取均衡负载。能画出 `00`/`3.6` 那张图最好。
- **为什么需要 P（不能只有 G/M）？**（高频）GM 模型靠一把全局锁 + 全局队列，多核下锁竞争严重、局部性差；Go 1.1 加 P 给每个执行槽配本地无锁队列，全局锁只在访问全局队列/窃取时用。
- **work stealing 是什么？** 某个 P 本地队列空了，随机挑别的 P 偷它队列的一半来跑，让负载在 P 间自动均衡。和 ForkJoinPool 同思路。
- **GOMAXPROCS 是什么？默认多少？** 能同时执行 Go 代码的 P 数量，默认=逻辑核数；决定并行度（不是 goroutine 上限）。
- **M 的数量能超过 GOMAXPROCS 吗？** 能。P 固定=GOMAXPROCS，但 M 动态——M 阻塞在 syscall 时会创建/复用别的 M 接管 P，所以 M 数可远多于 P。
- **全局队列什么时候用？** 本地队列溢出（>256 移一半进去）、从 syscall 返回找不到原 P 时等；并用「每 61 次调度先查全局」防止它被饿死。
- **G 的调度顺序有保证吗？** 没有。runnext/本地/全局/窃取共同决定，不可依赖。

---

## 7. 一句话总结

> **GMP = 给每个「执行槽」（P）配一条本地无锁队列，把 GM 模型那把全局锁打碎。** M（OS 线程）占住 P 才能跑 G，优先吃本地队列（含 runnext 局部性优化），空了再去全局队列、netpoll、偷别人一半；P 数=GOMAXPROCS 决定并行度，M 数动态可超 P。这套「per-P 队列 + 工作窃取」就是 Java ForkJoinPool/Loom 的同款骨架——它换来了 asyncio 给不了的「真多核 + 自动负载均衡」。
