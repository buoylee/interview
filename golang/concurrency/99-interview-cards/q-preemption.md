# Go 的抢占式调度怎么实现

## 一句话回答

Go 1.14 起用**异步抢占**：后台监控线程 `sysmon` 发现某个 goroutine 连续跑超过 **10ms**，就向它所在的 M（OS 线程）发一个 **`SIGURG` 信号**，信号处理器在「异步安全点」劫持执行流、注入 `asyncPreempt` 保存现场，再切回调度器把这个 G 丢回队列。**核心：用 OS 信号在用户态复刻了时钟中断**，于是连「没有任何让出点的死循环」也能被强行切走。

## 协作 → 抢占的进化

| 模型 | 靠什么让出 | 无停止点的死循环 |
|---|---|---|
| asyncio | 只有 `await` | 永不让出，冻死循环 |
| Go ≤1.13 | 函数调用安全点（栈检查 `stackguard0`） | **无函数调用的循环抢不了**（旧坑） |
| Go ≥1.14 | sysmon 发 SIGURG 强制打断 | 跑超 10ms 被切走 ✅ |
| OS 线程 | 硬件时钟中断 | 时间片轮转 |

## 实现链路（白板默写）

```
sysmon（不占 P 的监控线程）
  → 发现 G 跑 > 10ms（forcePreemptNS）
  → preemptM：向 M 发 SIGURG
  → 信号处理器 doSigPreempt：检查是否在异步安全点
  → 注入 asyncPreempt：保存全部寄存器（现场快照）
  → 切回调度器：G 回运行队列，M 去捞别的 G
```

## 三层论证

1. **为什么 1.14 之前不行**：协作式只在函数调用序言检查抢占哨兵，无函数调用的紧密循环没有安全点 → 抢不了 → 饿死其他 G，甚至让 STW GC 无限挂起。
2. **为什么用信号**：信号是 OS 提供的「异步打断正在跑的线程」能力，`SIGURG` 平时几乎没人用、不易冲突。
3. **为什么是「异步安全点」而非任意指令**：抢占要能正确保存栈/寄存器供 GC 扫描，runtime 把几乎所有指令做成安全点，仅持 runtime 锁/写屏障等极少数窗口不可抢占。

## 易追问的延伸

- **Windows 没信号怎么办？** `SuspendThread`+`GetThreadContext`/`SetThreadContext` 改指令指针注入抢占。
- **GOMAXPROCS=1 一个死循环会饿死别人吗？** 1.14+ 不会（10ms 后抢占）；≤1.13 且循环无函数调用：会。
- **抢占带来什么代价？** 任意点可切 → 共享状态必须同步，否则 data race（[06](../06-sync-memory-model/README.md)）。
- **和 asyncio 本质差异？** asyncio 无强制打断机制，靠你手写让出；Go runtime 强制打断，不用 await、无函数染色。
