# 02 · 抢占式调度

> 调度器内核三章之二。`01` 讲了 GMP 怎么把 goroutine 排上 OS 线程跑；本章回答一个更尖锐的问题：**一个 goroutine 不主动让出（没有 channel 操作、没有函数调用、没有 I/O），runtime 凭什么也能把它切走？**
>
> 这是 Go 区别于 `asyncio`/Rust/Node 等**协作式**模型的根本一脚，也是「Go 写并发为什么省心」的底层答案。

---

## 1. 核心问题

看这段代码：

```go
func main() {
    runtime.GOMAXPROCS(1)              // 只给 1 个 P，逼出单执行槽
    go func() { for {} }()             // 一个永不让出的死循环 goroutine
    time.Sleep(time.Millisecond)
    fmt.Println("我还能被打印出来吗？")   // Go 1.14+：能！
}
```

那个 `for {}` 里**没有任何让出点**——没有 `await`、没有函数调用、没有 channel、没有 I/O。在**协作式**模型里（asyncio、老 Go），它会**永久霸占**那唯一的执行槽，后面的 `Println` 永远跑不到。

但在 **Go 1.14+**，`Println` 照样打印。核心问题就是：**没有代码层面的停止点，Go 是怎么强行夺回 CPU 的？**

答案一句话剧透：**runtime 用操作系统信号（`SIGURG`）从外部打断它**——这就叫**抢占（preemption）**。

---

## 2. 直觉理解

### 协作式 vs 抢占式：让出主动权在谁手里

- **协作式（cooperative）**：让出的主动权在**代码自己**手里。代码必须主动跑到某个让出点（asyncio 的 `await`、老 Go 的函数调用）才会交出 CPU。不跑到 → 永不让出。
- **抢占式（preemptive）**：让出的主动权在 **runtime/OS** 手里。它能从外部强行打断正在跑的任务，无需代码配合。

你说的那句话就是抢占的定义：**「即使代码里没有任何主动让出点，runtime 也能把它切走。」**

### 让出能力的进化阶梯（把你两门母语放进去）

| 模型 | 靠什么让出 | 无停止点的死循环 `for {}` |
|---|---|---|
| **Python asyncio** | 只有显式 `await` | **永不让出**，冻死整个事件循环 |
| **Go ≤ 1.13**（协作式） | 函数调用的安全点 | 循环里没函数调用 → **也不让出**（Go 早年真实的坑） |
| **Go ≥ 1.14**（异步抢占） | runtime **发信号强行打断** | 跑超过 ~10ms 被**强行切走** ✅ |
| **OS 线程**（Java 平台线程） | 硬件时钟中断 | 一直被时间片轮转 |

看出关键了吗：**Go 1.14 之前，其实和 asyncio 是同一档「协作式」**——都依赖代码自己跑到某个点。区别只是 asyncio 的点是 `await`，老 Go 的点是函数调用。**两者都对付不了「闷头死循环」。** 真正让 Go 跨过这道坎的，是 1.14 的**异步抢占**。

> 对 Java 人：JVM 的线程是 OS 线程，天生被内核**抢占式**调度（硬件时钟中断），所以你在 Java 里从没担心过「一个死循环饿死别的线程」。Go 在用户态**自己复刻了这套抢占**，让 goroutine 也享受到同样的待遇。

---

## 3. 原理深入

### 3.1 协作式时代（Go ≤ 1.13）：函数调用安全点

早期 Go 的抢占完全是协作式的，机制是**栈检查**：

- 编译器在**每个函数的序言（prologue）**插一段检查：比较当前栈指针和 `g.stackguard0`。
- runtime 想抢占某个 G 时，把它的 `stackguard0` 设成一个特殊哨兵值 `stackPreempt`。
- 这个 G **下一次进入函数调用**时，序言检查发现哨兵值，就跳进 runtime 让出（借的是「栈不够、需要扩栈」`morestack` 那条路）。

问题很明显：**让出点绑死在「函数调用」上。** 一段没有函数调用的紧密循环——

```go
for i := 0; i < 1e18; i++ {
    sum += i      // 没有任何函数调用 → 没有安全点 → 永远检查不到哨兵
}
```

——**没有任何安全点**，于是**永远无法被抢占**。更糟的是：Go 的 GC 需要「Stop The World」对齐所有 G，而一个不可抢占的 G 会让 **STW 无限期挂起**，拖垮整个程序。这是 Go 多年的著名痛点。

### 3.2 异步抢占（Go 1.14+）：信号强行打断

1.14 引入**异步抢占（asynchronous preemption）**，彻底解决「无停止点」。机制是**借用 OS 信号在用户态复刻时钟中断**，链路如下：

```
sysmon（后台监控线程，不占 P）
   │  周期性扫描所有 P 上正在跑的 G
   │  发现某个 G 连续跑了 > 10ms（forcePreemptNS）
   ▼
preemptM(mp) ── 向那根 M（OS 线程）发送 SIGURG 信号
   │
   ▼
信号处理器（sighandler → doSigPreempt）在那根线程上触发
   │  检查当前是否停在「异步安全点」（栈/寄存器状态能被 GC 正确扫描）
   ▼
若安全：劫持执行流，注入调用 asyncPreempt
   │  保存全部寄存器状态（相当于一次「被中断的现场快照」）
   ▼
切回调度器（gopreempt_m → goschedImpl）
   │  把这个 G 重新丢回运行队列，M 去捞别的 G
   ▼
（稍后）G 被重新调度，从被打断的那条指令继续
```

四个关键角色，记住它们就能在白板上画出这条链：

1. **sysmon**：一个特殊的监控线程，**不绑定 P** 也能跑。它周期性巡逻，是抢占的「发起者」。它还顺手干别的：跑 netpoll、抢回阻塞在 syscall 的 P、强制 GC。（`01`/`03` 会再遇到它。）
2. **10ms 阈值（`forcePreemptNS`）**：一个 G 连续跑超过 10ms 就被盯上。这就是 Go 的「时间片」。
3. **SIGURG**：选这个信号是因为它平时几乎没人用，不容易和应用冲突。这是「从外部打断」的硬手段。
4. **asyncPreempt + 异步安全点**：信号随时可能到，但 runtime 只在**能安全保存现场的点**真正执行抢占——它把**几乎所有指令**都做成了「异步安全点」。

### 3.3 一个重要精度：不是「任意一条指令」

严格说，抢占不是在任意机器指令处发生，而是在**异步安全点（async-safe point）**。runtime 做了大量工作把绝大多数指令变成安全点，但仍有少数**不可抢占区**：

- 持有某些 runtime 内部锁时；
- 写屏障（write barrier）等 GC 敏感序列中；
- 某些寄存器状态不便保存的窗口。

但从你的心智模型看，结论不变：**「无函数调用的死循环也会被切走」是对的**，那点例外不用纠结。

### 3.4 Windows 没有信号怎么办

POSIX 信号是 Unix 的东西，Windows 没有。Go 在 Windows 上用另一套等价手段实现抢占：`SuspendThread` 挂起目标线程 → `GetThreadContext` 拿到它的寄存器现场 → 改写指令指针指向抢占逻辑 → `SetThreadContext` 写回 → `ResumeThread`。**机制不同，目的一样：从外部强行打断。**

### 3.5 和 asyncio 对照：同一套思想，谁来负责合作

把这章接回你刚学完的 asyncio，整条线就闭环了：

| | 怎么让出 | 谁来负责 |
|---|---|---|
| Python `asyncio` | 只在 `await` 协作让出 | **你**（漏写 await 就卡死） |
| Go ≥ 1.14 | sysmon 发 SIGURG 强制让出 | **runtime**（你什么都不用做） |

**asyncio 根本没有「强行打断一个正在跑的协程」的机制**，所以「别在一段代码里闷头跑太久」的责任只能压在你身上（手动塞 `await asyncio.sleep(0)`）。Go 用 SIGURG 把这份责任**从你身上接管到了 runtime**。这就是「Go 写并发心智负担小」的最底层原因。

## 4. 日常开发应用

抢占对你意味着什么——以及不意味着什么：

**它替你兜住了这些**：
- **不用手动插让出点**。Go 里没有 `await asyncio.sleep(0)` 这种「我让一下贤」的样板代码。
- **一个 CPU 死循环不会饿死别人**。哪怕 `GOMAXPROCS=1`，10ms 后就被切走，别的 goroutine 照样轮到。
- **一个忙 goroutine 不会卡死 GC**。STW 不再被不可抢占的 G 无限挂起。

**但它没有替你兜住这个（重要）**：
- **抢占带来「任意点可切」→ 数据竞争**。正因为一个 goroutine 可能在**任何一条语句中间**被切走，任何被多个 goroutine 共享的可变状态都必须正确同步（锁/atomic/channel），否则就是 data race。这正是协作式单线程 asyncio **不需要**操心、而 Go **必须**操心的代价——在 asyncio 里「两个 await 之间是原子的」，在 Go 里没有这种保证。详见 `06 内存模型`。

```go
// 错示范：counter 被多个 goroutine 在「任意点」并发改 → data race
var counter int
for i := 0; i < 1000; i++ {
    go func() { counter++ }()   // counter++ 不是原子的，抢占可切在读-改-写中间
}
// 正解：atomic.AddInt64 / Mutex / channel，见 06 章
```

> 一句话权衡：**抢占让「调度」变健壮（没人能饿死），却让「共享状态正确性」变难（到处可能竞争）。** 这就是为什么 Go 有 `-race` 检测器、文化上推「用通信共享内存」。

## 5. 生产&调优实战

- **观察调度器行为**：`GODEBUG=schedtrace=1000` 每秒打印一次调度器状态（P 数、运行队列长度、被抢占次数等），排查「为什么某些 goroutine 迟迟跑不上」时有用。
  ```bash
  GODEBUG=schedtrace=1000 ./your-app
  # 加 scheddetail=1 看每个 P/M/G 的细节
  ```
- **抢占失败的历史包袱**：runtime 自己拦不住的阻塞，抢占也无能为力——
  - **长 cgo 调用 / 阻塞 syscall**：M 真的陷在内核或 C 代码里，信号打断不了那段。runtime 的应对不是抢占，而是**把 P 从这根 M 上摘走交给别的 M**（`03` 章的 syscall handoff），保证全局不卡。
  - **极个别不可抢占区**（见 3.3）：持锁/写屏障窗口，会等到离开该区再抢占。
- **抢占与 STW/GC**：现代 Go 的 GC 标记终止、栈扫描都依赖能可靠抢占所有 G。如果你在 trace（`go tool trace`）里看到异常长的 STW，历史上常和「某个 G 迟迟无法到达安全点」有关——1.14 后基本绝迹，但这是理解 trace 的背景知识。
- **不要靠抢占做「公平性」**：10ms 粒度只能防饿死，不能保证低延迟调度。需要严格优先级/低延迟的场景，靠的是架构（拆 CPU 密集到单独 worker、限并发度），不是指望调度器抢占。

## 6. 面试高频考点

- **Go 的抢占式调度是怎么实现的？**（白板高频）画出 `sysmon → 发现 G 跑超 10ms → 发 SIGURG → 信号处理器 → asyncPreempt 保存现场 → 切回调度器`。点出关键词：异步抢占、SIGURG、异步安全点、10ms。
- **协作式和抢占式的区别？** 让出主动权在代码（协作，靠 await/函数调用安全点）还是在 runtime/OS（抢占，靠信号/中断强行打断）。
- **Go 1.14 之前有什么问题？** 协作式只在函数调用安全点让出，**无函数调用的紧密循环无法被抢占**，会饿死其他 G、甚至无限挂起 STW GC。1.14 用异步抢占（信号）解决。
- **为什么 Go 不需要像 Python 那样写 `await`？** 因为 Go 的让出点由 runtime 抢占式插入（信号 + 调度器），而不是由你在代码里手写。这也意味着 Go **没有函数染色**。
- **GOMAXPROCS=1 时，一个死循环会不会饿死别的 goroutine？** Go 1.14+ 不会——sysmon 10ms 后发信号抢占它。（Go ≤1.13 且循环无函数调用：会。）
- **（对标题）和 asyncio 的本质差异？** asyncio 纯协作、无强制打断机制，靠你手写让出；Go 抢占式，runtime 强制打断。代价是 Go 有数据竞争问题（任意点可切），asyncio「两 await 之间原子」反而更易写对共享状态。

---

## 7. 一句话总结

> **抢占 = 让出的主动权从「代码自己」转移到「runtime」。** Go 1.14 用 `sysmon` 巡逻 + `SIGURG` 信号 + `asyncPreempt`，在用户态复刻了 OS 的时钟中断，于是连「无停止点的死循环」也能被强行切走——这是 Go 区别于 asyncio/Rust/Node 等一切协作式模型的根本一脚，也是「Go 写并发省心」的底层来源。代价：任意点可切 → 必须认真对待数据竞争（`06`）。
