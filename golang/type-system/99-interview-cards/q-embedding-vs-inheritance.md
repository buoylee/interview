# 嵌入 vs 继承:Go 为什么没有继承

## 一句话回答

Go **故意不要类继承**,用**嵌入(embedding)**做复用:把类型匿名放进 struct,它的字段/方法被「**提升**」到外层、可直接访问。但这是 **has-a 组合 + 自动转发**,不是 is-a 继承——**没有虚函数重写、没有多态覆盖**。外层定义同名方法只是「**遮蔽(shadow)**」;把内嵌值取出来调,走的仍是内嵌类型自己的方法。**Go 的多态只通过接口实现,不通过嵌入。**

## 关键差异(对照 Java)

```go
type Dog struct{ Animal }              // 嵌入,不是 extends
func (d Dog) Speak() string { ... }    // 遮蔽 Animal.Speak,不是 override

var a Animal = d.Animal
a.Speak()    // 调 Animal.Speak —— 没有动态派发到 Dog!(Java 会派发到 Dog)
```

| | Java 继承 | Go 嵌入 |
|---|---|---|
| 关系 | is-a + 可重写虚方法 | has-a 组合 + 方法提升,无重写 |
| 多态 | 虚函数动态派发 | 只靠接口 |
| 向上转型后调方法 | 仍调子类(覆盖) | 调内嵌类型自己的 |
| super | 有 | `d.Animal.Speak()` 显式 |

## 嵌入的威力

- **方法提升 → 外层自动满足接口**:内嵌类型满足的接口,外层也满足(复用 + 满足接口的主力手法)。
- **嵌入接口**:组合接口(`io.ReadWriter` 嵌 `Reader`+`Writer`);struct 嵌接口做**装饰器/中间件**(默认转发 + 只重写想改的方法)。

## 易追问的延伸

- **同名方法冲突?** 浅层(外层自定义)遮蔽深层;同深度两个嵌入类型同名 → ambiguous **编译错误**,需 `d.A.X` 显式。
- **嵌入的坑?** 把内嵌的导出方法全提升成你的公共 API(可能泄漏)、内嵌接口/指针为 nil 时调提升方法 panic、JSON 序列化字段平铺。
- **怎么只复用不暴露全部方法?** 别嵌入,用具名字段 + 显式转发。
- **一句话记忆?** 嵌入为复用,接口为多态,两件事。
