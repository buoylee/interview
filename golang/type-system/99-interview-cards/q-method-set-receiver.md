# 方法集与接收者:为什么 `*T` 能满足接口而 `T` 不能

## 一句话回答

接收者是方法的**第一个隐藏参数**:值接收者 `(t T)` 操作**拷贝**(改不到原值、每次调用拷贝),指针接收者 `(t *T)` 操作**原值**(能改、免拷)。方法集规则:**`T` 只含值接收者方法,`*T` 含值接收者 + 指针接收者方法(全部)**。所以接口要求的方法若是**指针接收者**,只有 `*T` 的方法集覆盖它——`T` 的值赋给该接口会编译失败,必须用 `*T`。

## 为什么直接调能、装接口不能

```go
c := Counter{}
c.Inc()                       // ✅ 局部变量可寻址,编译器自动 (&c).Inc()
var x Incrementer = Counter{} // ❌ 接口里的值不可寻址,无法自动取址满足指针方法
var y Incrementer = &Counter{}// ✅ *Counter 方法集含 Inc
```

根因是**可寻址性**:局部变量、`*p`、`s[i]`、可寻址 struct 的字段可寻址,编译器能自动 `&`;而**接口里的值、map 元素、函数返回值、字面量不可寻址**,不能自动取址去调指针接收者方法。

## 选择准则

- **默认指针接收者**:要改状态、类型大、含锁/含 `sync.Mutex`。
- **值接收者**:类型小且不可变、方法只读(如 `time.Time`),且想让 `T` 和 `*T` 都满足接口。
- **一致性铁律**:**有一个方法要指针接收者,就把所有方法都用指针接收者**(心智简单 + 避免拷贝含锁结构体的 bug)。

## 易追问的延伸

- **map 元素为什么不能调指针方法/改字段?** map 元素不可寻址(扩容会搬迁);取出改写回,或用 `map[K]*V`。
- **slice 元素呢?** `s[i]` 可寻址,可调指针方法;但 `range` 出来的 `v` 是副本,改不到原值。
- **值接收者 + 含锁 struct?** 拷贝锁→互斥失效,`go vet copylocks` 报。
- **和 Java 区别?** Java 方法 `this` 永远是引用、无值/指针接收者之分;Go 接收者按值还是指针直接决定能否改原值 + 谁能进接口。
