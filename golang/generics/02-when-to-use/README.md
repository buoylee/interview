# 02 · 工程取舍:泛型 vs 接口 vs 代码生成

> 泛型不总更快(`01`)、还可能伤可读性,所以「**该不该用泛型**」本身是面试题。Go 团队(Ian Lance Taylor 的《When To Use Generics》)给过明确指引。一句话:**接口为行为,泛型为类型**——本章把判据讲清。
>
> 桥接锚点:Java 程序员习惯「能泛型就泛型」;Go 文化更克制——泛型是 1.18 才加的,社区刻意强调「先问需不需要」。

---

## 1. 核心问题

```go
// 同一件事的三种写法
func PrintAll(xs []any)            { for _, x := range xs { fmt.Println(x) } }  // 接口
func PrintAll[T any](xs []T)       { for _, x := range xs { fmt.Println(x) } }  // 泛型
//go:generate ... 生成 PrintAllInt / PrintAllString                              // 代码生成
```

- 这三种到底什么时候用哪个?有没有判据,而不是凭感觉?
- 「我要写个通用函数」就一定该用泛型吗?
- 什么时候泛型是**错误**选择?

---

## 2. 直觉理解

### 一句总纲:接口为行为,泛型为类型

- **你关心「这东西能做什么」(行为)→ 接口**。例:`io.Writer`——我不在乎你是文件还是网络,只要你能 `Write`。运行期多态、异构集合,都是接口。
- **你关心「对任意类型做同样的结构操作」(类型)→ 泛型**。例:`Map[T,U]`、`slices.Sort`、栈/队列——逻辑对所有元素类型一模一样,只是类型不同。

判别窍门:**如果泛型函数体里你只是「调用类型参数的某个方法」,那其实用接口就够了**(还更简单、更直接,见下)。泛型的价值在「逻辑不依赖具体方法、纯靠类型结构」时才凸显。

---

## 3. 原理深入

### 3.1 该用泛型的信号(ILT 指引)

1. **操作任意元素类型的 slice/map/channel**:`Map`/`Filter`/`Keys`/`Sort` 这类——逻辑与元素类型无关。
2. **通用数据结构**:类型安全的栈、队列、集合、链表、有序树——以前只能 `interface{}` 存(丢类型 + 装箱),泛型让它类型安全且免装箱。
3. **多个类型上的函数实现完全相同**:以前要 `MaxInt`/`MaxFloat`… 一堆重复,现在一个 `Max[T cmp.Ordered]`。

### 3.2 不该用泛型的信号

1. **只是调用类型参数的方法** → **用接口**。
   ```go
   // ❌ 没必要泛型
   func Write[T io.Writer](w T, b []byte) { w.Write(b) }
   // ✅ 接口更简单、更直接(还省字典间接)
   func Write(w io.Writer, b []byte) { w.Write(b) }
   ```
   规则原话:*"If you are calling a method on the type argument, use an interface."*
2. **不同类型的实现逻辑不同** → 用**方法 + 接口**(让各类型自己实现),别硬塞进一个泛型体里 `switch` 类型。
3. **只有一个类型用** → 不用泛型,直接写具体类型。
4. **能用反射更自然**(如处理任意结构体字段)→ 反射;泛型表达不了「运行期任意结构」。
5. **会显著降低可读性**(一堆类型参数 + 复杂约束)而收益不大 → 别用。

### 3.3 代码生成(go:generate)还有位置吗?

泛型出来后,**多数「为类型重复代码」的场景被泛型取代**。代码生成仍适用于:

- 泛型表达不了的(如根据 struct tag 生成序列化代码、生成 mock、protobuf/stringer);
- 需要生成**方法**而非函数、或生成跨类型的样板(泛型方法不能带类型参数,见 [`00`](../00-basics/README.md));
- 性能极敏感、要完全单态化避免字典间接的(罕见)。

口诀:**泛型替代了「为类型而生成」,代码生成留给「为元数据/样板而生成」。**

### 3.4 泛型 vs 接口的成本对照

| | 接口 | 泛型 |
|---|---|---|
| 多态时机 | 运行期(动态派发) | 编译期(参数化) |
| 类型安全 | 弱(`any` 要断言) | 强(编译期检查) |
| 值类型 | 装箱 + 逃逸 | 不装箱、直接存 |
| 调 T 方法 | 直接虚派发 | 经字典间接(可能更慢) |
| 异构集合 | ✅ `[]io.Writer` 各种实现 | ❌ `[]T` 只能同一类型 |
| 适合 | **行为**抽象、运行期多态 | **类型**参数化、容器/算法 |

---

## 4. 日常开发应用

- **写通用容器/slice·map 算法 → 泛型**(类型安全 + 免装箱)。
- **抽象行为、要异构集合、要运行期多态 → 接口**。
- **泛型体里只调方法 → 退回接口**(更简单)。
- **各类型逻辑不同 → 接口 + 各自实现**,别泛型 + 类型 switch。
- **先标准库**:`slices`/`maps`/`cmp`/`min`/`max` 覆盖了大量常见泛型需求,别自己造轮子。
- **拿不准就先不用泛型**:Go 文化偏保守,先写具体/接口版本,真出现「多类型重复」再泛型化。

---

## 5. 生产&调优实战

- **泛型用于「免装箱」是真收益**:高频数值/值类型容器,泛型 `[]T` 比 `[]any` 省大量分配(见 [`data-structures/03`](../../data-structures/03-escape-analysis/README.md));这是性能正当理由。
- **泛型用于「调方法」常是负收益**:字典间接 + 难内联,可能比接口慢(见 [`01`](../01-implementation/README.md));这种场景退回接口。
- **API 设计**:对外暴露的库 API,接口往往比泛型更好演进(泛型签名一旦定下、约束变更是破坏性的);内部实现可大胆泛型。
- **可读性是硬约束**:三四个类型参数 + 嵌套约束的函数,后人难维护;团队代码里克制使用,复杂泛型集中在少数基础库。

---

## 6. 面试高频考点

- **泛型 vs 接口怎么选?** **接口为行为(运行期多态、异构集合、行为抽象),泛型为类型(对任意类型同样的结构操作、容器/算法、免装箱)**。
- **一个判据?** 泛型体里**只调类型参数的方法 → 用接口**(更简单直接);逻辑不依赖具体方法、纯类型结构 → 泛型。
- **什么时候别用泛型?** 只调方法、各类型实现不同、只有一个类型、反射更自然、显著伤可读性。
- **泛型出来后代码生成还有用吗?** 有——泛型替代「为类型重复」,代码生成留给「为元数据/样板生成」(序列化、mock、stringer、生成方法)。
- **泛型用于性能的正当场景?** 免装箱的值类型容器/算法;反例是「调 T 方法」(字典间接可能更慢)。
- **为什么 Go 社区对泛型克制?** 1.18 才加、可能伤可读性、不总更快;文化是「先问需不需要」。

---

## 7. 一句话总结

> **接口为行为,泛型为类型**:关心「能做什么」(运行期多态、异构集合、行为抽象)用接口;关心「对任意类型做同样的结构操作」(容器、slice/map 算法、免装箱)用泛型。**一条硬判据:泛型体里若只是调用类型参数的方法,就该用接口**(更简单,还省字典间接)。别用泛型的情形:各类型实现不同(用接口 + 各自实现)、只有一个类型、反射更自然、严重伤可读性。代码生成没被淘汰,只是退到「为元数据/样板生成」(序列化、mock、生成方法)。泛型的真收益是**类型安全 + 免装箱 + 少重复**,不是速度——先标准库 `slices`/`maps`/`cmp`,拿不准就先别泛型。
