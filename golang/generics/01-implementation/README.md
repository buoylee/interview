# 01 · 底层实现:GCShape stenciling + 字典

> 「`Max[T]` 对 int、string、各种指针,编译器生成几份代码?」——这是泛型最有区分度的面试题。答案既不是 C++ 的「每种类型一份」(完全单态化),也不是 Java 的「全擦成 Object 一份」(擦除),而是 Go 独有的 **GCShape stenciling + 字典(dictionaries)**,介于两者之间。理解它也解释了「泛型为什么有时比接口/手写还慢」。
>
> 桥接锚点:C++ 模板 = 完全单态化(每个具体类型生成一份机器码,极致性能但代码膨胀);Java 泛型 = 类型擦除(`List<String>` 运行期就是 `List`,一份代码 + 装箱 + 强转);Go = **按 GCShape 共享实例 + 字典补类型信息**的折中。

---

## 1. 核心问题

```go
func Max[T cmp.Ordered](a, b T) T { if a > b { return a }; return b }
Max(1, 2)            // int
Max(1.5, 2.5)        // float64
Max("a", "b")        // string
var p, q *Node
Max2(p, q)           // *Node —— 和别的指针类型共享代码吗?
```

- 编译器为这些调用生成**几份** `Max` 的机器码?
- 如果共享代码,不同类型大小不同(int 8 字节、string 16 字节),一份代码怎么处理?
- 泛型函数里调用类型参数的方法,函数怎么知道调哪个具体实现?

---

## 2. 直觉理解

### 两个极端,Go 走中间

| 方案 | 做法 | 代价 |
|---|---|---|
| **C++ 模板(单态化)** | 每个具体类型生成一份专用机器码 | 最快,但**代码膨胀**(编译慢、二进制大) |
| **Java 泛型(擦除)** | 全擦成 `Object`,一份代码 | 代码小,但**装箱 + 强转 + 丢失值类型性能** |
| **Go(GCShape + 字典)** | **按「内存形状」分组**生成代码,类型差异用字典补 | 折中:代码不爆炸,但**字典间接有成本** |

### GCShape = 「内存形状一样的类型,共享一份代码」

Go 的关键洞察:很多类型在**运行时/GC 看来内存布局是一样的**,可以共用同一份机器码。一个「GCShape」就是一组「内存形状相同」的类型。最重要的规则:

> **所有指针类型共享同一个 GCShape**(它们都是「一个指针」,GC 都按指针扫描)。而每个不同的值类型(int、float64、各种 struct)各自一个 GCShape。

所以:`Max[*Node]`、`Max[*User]`、`Max[*int]` ——**全部共享一份** stenciled 代码(都是指针形状);而 `Max[int]`、`Max[string]`、`Max[float64]` 各生成一份(形状不同)。

### 字典 = 补上「同形状但不同类型」的差异

但「共享代码」有个问题:`*Node` 和 `*User` 形状一样、共享代码,可代码里若要调 `T` 的方法、或要知道 T 的具体类型信息,光靠共享的机器码不够。于是编译器给每次泛型调用**额外传一个隐藏参数——字典(dictionary)**,里面装着这次具体类型的信息:类型描述符、要调的方法地址、派生类型信息等。

**一句话:GCShape 决定「生成几份代码」(按内存形状分组),字典补上「同组内不同类型」的运行期差异。**

---

## 3. 原理深入

### 3.1 stenciling(模板印刷)按 GCShape 进行

编译器为每个用到的 **GCShape** 把泛型函数体「印」一份(stencil):

- `Max[int]`、`Max[float64]`、`Max[string]` → 三份(三种值形状)。
- `Max[*Node]`、`Max[*User]`、`Max[*int]`… → **共一份**(指针形状)。

这样既避免 C++ 那种「每个类型一份」的爆炸,又保留了值类型的直接操作(不像 Java 全装箱)。

### 3.2 字典里有什么、怎么传

字典是编译器生成的、每个 (泛型函数, 具体类型实参) 一份的只读数据,作为**隐藏的第一参数**传进泛型函数。它包含:

- 具体类型的 **`_type` 描述符**(运行期需要类型信息时用,如反射、`new(T)`、比较);
- **方法表/函数指针**:若泛型体调用了 T(或约束)的方法,通过字典找到该具体类型的实现来调;
- 派生类型信息(如 `[]T`、`*T` 的类型)。

> 共享代码 + 字典的组合:同形状的指针类型共用机器码,但各自传不同的字典,从而行为正确。

### 3.3 为什么泛型有时更慢(反直觉的重点)

- **方法调用走字典间接**:泛型体里调 `T` 的方法,不是直接调用,而是**从字典里取函数指针再调**——多一层间接,且**难以内联**。这和接口动态派发的成本类似,有时甚至因为优化器看不穿而**比接口还慢**。
- **失去单态化的优化**:C++ 单态化后能针对具体类型内联、常量折叠;Go 共享代码 + 字典则牺牲了部分这种优化。
- 结论:**泛型主要赢在「类型安全 + 免装箱 + 少写代码」,不一定赢在速度**。性能热点要 benchmark,别假设「泛型 = 更快」。这是 Go 泛型最反直觉、也最显资深的一点。

### 3.4 对照 Java/C++(白板表)

| | C++ 模板 | Java 泛型 | Go 泛型 |
|---|---|---|---|
| 实现 | 完全单态化 | 类型擦除 | GCShape stenciling + 字典 |
| 代码量 | 每类型一份(膨胀) | 一份 | 按形状分组(指针共享) |
| 值类型 | 直接操作 | 装箱成 Object | 直接操作(不装箱) |
| 运行期类型信息 | 编译期定死 | 擦除(运行期没了) | **字典携带**(有运行期类型信息) |
| 方法调用 | 内联,最快 | 虚派发 | 经字典间接(可能慢) |

## 4. 日常开发应用

- **知道「指针类型共享实例」**:大量不同指针类型用同一泛型,代码不会膨胀——可放心对 `*T` 用泛型。
- **值类型多了会各生成一份**:对很多不同值类型实例化同一个大泛型函数,会增加二进制体积(通常可接受,极端时留意)。
- **泛型体里频繁调 T 的方法 = 字典间接成本**:这种场景可能用**接口参数**更直接(见 [`02`](../02-when-to-use/README.md))。
- **要运行期类型信息**(`new(T)`、反射 T)在泛型里是可行的——字典带着 `_type`。

---

## 5. 生产&调优实战

- **别迷信泛型性能**:把热路径从接口/具体类型改成泛型,**先 benchmark**(`-benchmem`);字典间接 + 失去内联可能让它不升反降。
- **泛型 vs 接口的性能不是一边倒**:纯结构操作(不调 T 方法,如数值计算、slice 搬运)泛型通常更快(免装箱);频繁调 T 方法时泛型的字典间接可能抹掉优势。按场景实测。
- **二进制体积**:对海量不同**值**类型实例化大泛型函数会增大二进制;指针类型不影响(共享)。一般无需担心,monorepo 极端情况才量化。
- **编译时间**:比 C++ 模板友好得多(不爆炸),但比无泛型代码略增。

---

## 6. 面试高频考点

- **Go 泛型怎么实现的?生成几份代码?** GCShape stenciling + 字典:按「内存形状(GCShape)」分组生成代码——**所有指针类型共享一份**,各值类型各一份;类型差异由作为隐藏参数传入的**字典**(含 `_type` + 方法地址)补上。
- **和 C++ 模板、Java 擦除区别?** C++ 完全单态化(每类型一份、最快、膨胀);Java 擦除(一份 + 装箱 + 丢运行期类型);Go 折中(按形状分组 + 字典,不装箱值类型、保留运行期类型信息)。
- **什么是 GCShape?** 内存布局(对 GC/运行时而言)相同的一组类型;最关键:所有指针类型同一个 GCShape。
- **字典是干什么的?** 每个(泛型函数,具体类型)一份的隐藏参数,携带具体类型描述符和方法地址,让共享代码对不同类型行为正确。
- **泛型一定比接口快吗?** 不一定。泛型体调 T 的方法走字典间接、难内联,可能和接口一样甚至更慢;纯值操作(免装箱)才通常更快。要 benchmark。
- **泛型会让二进制膨胀吗?** 值类型多实例化会增体积;指针类型共享、影响小。远比 C++ 模板克制。

---

## 7. 一句话总结

> **Go 泛型 = GCShape stenciling + 字典,介于 C++ 单态化和 Java 擦除之间。** 编译器按「内存形状(GCShape)」生成代码——**所有指针类型共享一份**、各值类型各一份(避免 C++ 式爆炸,又不像 Java 装箱值类型);同形状不同类型的差异,靠每次调用传入的隐藏参数**字典**(含类型描述符 + 方法地址)补上。代价:泛型体里调类型参数的方法要**经字典间接、难内联**,所以**泛型不保证比接口/手写更快**——纯值操作通常更快(免装箱),频繁调方法时可能反慢。泛型的真正收益是**类型安全 + 免装箱 + 少写代码**,性能要 benchmark 而非假设。
