# 泛型 vs 接口:怎么选

## 一句话回答

**接口为行为,泛型为类型。** 你关心「这东西能做什么」(运行期多态、异构集合、行为抽象)→ **接口**;你关心「对任意类型做同样的结构操作」(容器、slice/map 算法、避免 `interface{}` 装箱)→ **泛型**。一条硬判据(Go 团队原话):**如果泛型函数体里你只是调用类型参数的某个方法,就该用接口**——更简单、更直接,还省掉泛型的字典间接成本。

## 成本对照

| | 接口 | 泛型 |
|---|---|---|
| 多态时机 | 运行期(动态派发) | 编译期(参数化) |
| 类型安全 | 弱(any 要断言) | 强(编译期检查) |
| 值类型 | 装箱 + 逃逸 | 不装箱、直接存 |
| 调 T 方法 | 直接虚派发 | 经字典间接(可能更慢) |
| 异构集合 | ✅ `[]io.Writer` 各种实现 | ❌ `[]T` 只能同类型 |
| 适合 | 行为抽象、运行期多态 | 类型参数化、容器/算法 |

## 该用泛型的信号

1. 操作任意元素类型的 slice/map/channel(`Map`/`Filter`/`Sort`)。
2. 通用数据结构(类型安全的栈/队列/集合,免装箱)。
3. 多个类型上实现完全相同(`Max[T]` 取代一堆 `MaxInt`/`MaxFloat`)。

## 别用泛型的信号

只调 T 的方法(用接口)、各类型实现不同(接口 + 各自实现)、只有一个类型、反射更自然、显著伤可读性。

## 易追问的延伸

- **代码生成还有用吗?** 有——泛型替代「为类型重复」,codegen 留给「为元数据/样板生成」(序列化、mock、stringer、生成方法)。
- **泛型用于性能的正当场景?** 免装箱的值类型容器/算法;反例是「调 T 方法」(字典间接可能更慢)。
- **库 API 用泛型还是接口?** 对外 API 接口更易演进(泛型约束变更是破坏性的);内部实现可大胆泛型。
- **为什么 Go 社区克制?** 1.18 才加、可能伤可读性、不总更快;文化是「先问需不需要」。
