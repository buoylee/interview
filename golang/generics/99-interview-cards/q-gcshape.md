# 底层实现:GCShape stenciling + 字典

## 一句话回答

Go 泛型用 **GCShape stenciling + 字典(dictionaries)**,介于 C++ 完全单态化和 Java 类型擦除之间。编译器按「**GCShape(内存形状)**」给泛型函数体生成代码:内存布局相同的类型共享一份——**所有指针类型共享同一份**,各不同值类型各生成一份;同形状不同类型的差异(类型描述符、要调的方法地址),由作为**隐藏参数**传入的**字典**补上。代价:泛型体里调类型参数的方法要**经字典间接、难内联**,所以泛型不保证比接口/手写更快。

## 几份代码?

```go
func Max[T cmp.Ordered](a, b T) T { ... }
Max[int]    Max[float64]    Max[string]    // 三份(三种值形状)
Max[*A]     Max[*B]         Max[*C]         // 共一份(都是指针形状)
```

## 三方对照

| | C++ 模板 | Java 泛型 | Go 泛型 |
|---|---|---|---|
| 实现 | 完全单态化 | 类型擦除 | GCShape stenciling + 字典 |
| 代码量 | 每类型一份(膨胀) | 一份 | 按形状分组(指针共享) |
| 值类型 | 直接操作(最快) | 装箱成 Object | 直接操作、不装箱 |
| 运行期类型信息 | 编译期定死 | 擦除丢失 | 字典携带(有) |
| 调方法 | 内联最快 | 虚派发 | 经字典间接(可能慢) |

## 易追问的延伸

- **GCShape 精确定义?** 对 GC/运行时而言内存布局相同的一组类型;最关键规则:所有指针类型同一个 GCShape。
- **字典里有什么?** 具体类型 `_type` 描述符、方法函数指针、派生类型(`[]T`/`*T`)信息;每个(泛型函数,具体类型)一份,隐藏传入。
- **泛型一定更快吗?** 不一定——调 T 方法走字典间接、难内联,可能和接口一样甚至更慢;纯值操作(免装箱)才通常更快。要 benchmark。
- **会让二进制膨胀吗?** 值类型多实例化会增体积;指针共享、影响小;远比 C++ 克制。
