# 逃逸分析:什么在栈、什么在堆

## 一句话回答

一个值分配在**栈**还是**堆**,由**编译期的逃逸分析**决定:编译器能证明它的生命周期不超过当前函数 → **栈**分配(随函数返回自动回收、零 GC 成本);否则 → **堆**分配(由 GC 回收)。Go 默认尽量栈分配。逃逸 = 堆分配 = GC 压力,所以减少逃逸是常见优化。用 `go build -gcflags='-m'` 看判定、`go test -benchmem` 看 `allocs/op`。

## 典型逃逸场景

1. **返回局部变量的指针** `return &x`(最经典)。
2. **装进接口/`any`**:接口 data 是指针,值常搬到堆;`fmt.Println(x)` 收 `...any` 是逃逸大户。
3. **被逃出的闭包捕获**:`go func(){ use(x) }()`、返回的闭包引用 x。
4. **编译期大小不定/过大的对象**:`make([]T, n)`(n 变量且大)、超大局部数组。
5. **地址存进已逃逸的结构**(slice/map/struct 字段)。

## 关键澄清

- **取地址 `&` 不必然逃逸**——地址不逃出函数就能留栈。
- **返回 `&x` 安全**:编译器发现逃逸自动改放堆,**没有 C 的悬垂指针**。
- **小对象传值常比传指针好**:传指针可能反而触发逃逸(承接 [`type-system/00`](../../../type-system/00-values-layout/README.md))。
- **栈会增长**(goroutine 栈 2KB 起可扩),所以栈也能放不小的东西。

## 易追问的延伸

- **怎么观察?** `go build -gcflags='-m'`(`-m -m` 更详细原因);基准 `-benchmem` 看 `allocs/op`、`B/op`。
- **怎么减少逃逸?** 热路径少装接口/少 `fmt`、小对象传值、预分配 slice/map、`sync.Pool` 复用不得不堆分配的临时对象。
- **和 JVM 区别?** JVM 对象默认在堆、逃逸分析做标量替换(运行期、不可见);Go 默认尽量栈、编译期决定且 `-m` 可见。
- **逃逸分析准吗?** 编译期保守——拿不准就判逃逸(宁上堆不悬垂),有些看似不该逃的是保守结果。
