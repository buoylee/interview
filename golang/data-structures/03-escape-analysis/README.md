# 03 · 逃逸分析与栈/堆

> 前面反复说「装接口会逃逸」「子串持有大数组」。这章正面讲:一个值到底分配在**栈**还是**堆**,谁决定的(编译期的**逃逸分析**),什么会逃逸,以及为什么这关乎性能(堆要 GC、栈不用)。
>
> 桥接锚点:Java 对象**默认都在堆**,JVM 靠逃逸分析做「标量替换」把没逃逸的优化掉(对你不可见);Go 反过来——**默认尽量栈分配**,逃逸分析在**编译期**决定谁上堆,而且 `-gcflags=-m` 能直接看到结果。

---

## 1. 核心问题

```go
func newOnStack() int {
    x := 42                 // x 在栈还是堆?
    return x
}
func newOnHeap() *int {
    x := 42                 // 这个 x 呢?返回了它的地址……
    return &x               // 函数返回后栈帧没了,&x 还能用吗?
}
```

- Go 没有「new 一定在堆」的规则——那一个变量在栈还是堆,谁说了算?
- 返回局部变量的地址 `&x` 为什么不会出问题(不像 C 的悬垂指针)?
- 哪些写法会让值「逃逸」到堆?逃逸为什么影响性能?

---

## 2. 直觉理解

### 栈便宜、堆要 GC

- **栈**:每个 goroutine 自带的栈,分配/释放就是移动栈指针,**随函数返回自动回收**,零 GC 成本。
- **堆**:全局共享,分配要找空间、释放靠 **GC** 扫描回收——有成本。

所以原则:**能放栈就放栈**。Go 默认尽量栈分配,只有「编译器无法证明这个值的生命周期不超过当前函数」时,才把它放堆——这个判定就是**逃逸分析**。

### 逃逸 = 「我活得比这个函数久 / 编译器拿不准」

`newOnHeap` 里 `x` 的地址被 `return &x` 带出了函数——函数返回后调用方还要用它,它必须**活得比函数久**,栈帧回收它就没了。于是编译器把 `x` **分配到堆**(逃逸),返回的指针指向堆。这也是 Go **没有 C 悬垂指针问题**的原因:编译器发现地址逃逸就自动改放堆,GC 负责回收。

一句话:**值的「引用」逃出了它所在的函数作用域(或编译器无法在编译期确定它不会逃),它就被分配到堆。**

---

## 3. 原理深入

### 3.1 用 `-gcflags=-m` 看逃逸判定

```bash
go build -gcflags='-m' main.go
# ./main.go:7:6: moved to heap: x          ← x 逃逸到堆
# ./main.go:3:2: x does not escape          ← 没逃逸,留栈上
```

`-m` 是面试和调优都该会的工具:它把编译器的每个逃逸决策打印出来。`-m -m`(两次)给更详细的原因。

### 3.2 典型逃逸场景(高频考点)

1. **返回局部变量的指针**:`return &x`——最经典。
2. **装进接口**:`var i any = x` / `fmt.Println(x)`——接口 data 是指针,值常被搬到堆(承接 [`type-system/01`](../../type-system/01-interface-internals/README.md))。`fmt` 系列因为收 `...any` 是逃逸大户。
3. **被闭包捕获且闭包逃出**:`go func(){ use(x) }()`、返回的闭包引用 x。
4. **大小在编译期不确定**:`make([]T, n)`(n 是变量且较大)、超大局部数组(放栈会爆)→ 上堆。
5. **存进逃逸的结构**:把局部变量地址塞进一个已在堆上的 slice/map/struct 字段。
6. **指针被传给「编译器看不穿」的函数**(无法内联分析的调用)。

### 3.3 不逃逸的常见情况

- 纯局部值、只在函数内用、不取地址或地址不外传 → 栈。
- 取了地址但**地址没逃出函数**(`p := &x; *p = 1` 本地用)→ 仍可栈。
- 小对象、生命周期清晰、编译器能内联看穿 → 栈。

> 关键:**取地址 `&` 不必然逃逸**;逃逸看的是「地址有没有逃出作用域 / 编译器能否证明」。

### 3.4 栈会增长,所以栈也能放不小的东西

Go 的 goroutine 栈是**可增长**的(2KB 起,不够就拷贝到更大的栈,见 [`concurrency/04`](../../concurrency/04-goroutine/README.md))。所以栈不是只能放小东西;只有「逃逸」或「编译期大小不定的超大对象」才被迫上堆。

---

## 4. 日常开发应用

- **想减少分配,先 `-gcflags=-m` 看哪逃逸**,再针对性改。别凭感觉。
- **热路径少装接口**:大量 `any` 参数(尤其 `fmt`)会逃逸;高频日志用结构化、避免 `fmt.Sprintf` 满天飞。
- **小对象传值常比传指针好**:传指针反而可能让对象逃逸;小 struct 直接传值能留栈(承接 [`type-system/00`](../../type-system/00-values-layout/README.md))。
- **复用对象用 `sync.Pool`** 减少堆分配(高频临时大对象,如 buffer)。
- **预分配 slice/map** 减少扩容带来的堆分配(见 `00`/`01`)。
- 别为「避免逃逸」过度优化代码可读性——先 profile 确认是热点再动。

---

## 5. 生产&调优实战

- **逃逸 = 堆分配 = GC 压力**:高频逃逸的小对象会推高分配速率,触发更频繁 GC、拉高 CPU 和延迟。`pprof` 的 `alloc_space`/`alloc_objects` + `-gcflags=-m` 一起定位。
- **`fmt`/接口装箱是最常见的意外逃逸源**:把热路径里的 `fmt.Sprintf`、`interface{}` 参数换成具体类型或 `strconv`,常能显著降分配。
- **`sync.Pool` 回收逃逸对象**:把不得不堆分配的临时对象池化(注意 Pool 里的对象可能被 GC 清掉,只做缓存别当持久存储)。
- **基准里看分配**:`go test -bench . -benchmem` 输出 `allocs/op`、`B/op`——优化逃逸就盯这两个数降下来。
- **逃逸分析是编译期、保守**:它拿不准就判逃逸(宁可上堆也不能错放栈导致悬垂);所以有些「看起来不该逃逸」的是编译器保守,未必能消除。

---

## 6. 面试高频考点

- **栈和堆怎么分配的?谁决定?** 编译期**逃逸分析**决定:能证明生命周期不超过函数就栈分配(随返回自动回收、无 GC),否则堆分配(GC 回收)。Go 默认尽量栈。
- **返回 `&x`(局部变量地址)为什么安全?** 编译器发现地址逃逸,把 x 分配到堆;所以没有 C 的悬垂指针。
- **哪些情况会逃逸?** 返回局部指针、装进接口/`any`(含 `fmt`)、被逃出的闭包捕获、编译期大小不定的大对象、地址存进已逃逸的结构。
- **取地址 `&` 一定逃逸吗?** 不一定——地址不逃出函数就可留栈。
- **怎么观察逃逸?** `go build -gcflags='-m'`(`-m -m` 更详细);基准 `-benchmem` 看 `allocs/op`。
- **逃逸为什么影响性能?** 堆分配 + GC 扫描成本,高频逃逸推高 GC 频率和延迟。
- **和 JVM 逃逸分析区别?** JVM 对象默认在堆、逃逸分析做标量替换(对你不可见、运行期);Go 默认尽量栈、逃逸分析编译期决定且可见。

---

## 7. 一句话总结

> **一个值在栈还是堆,由编译期的逃逸分析决定:能证明它活不过当前函数就栈分配(随返回自动回收、零 GC),否则逃逸到堆(GC 回收)。** 典型逃逸:返回局部指针、装进接口/`any`(`fmt` 是大户)、被逃出的闭包捕获、编译期大小不定的大对象、地址存进已逃逸的结构——但**单纯 `&` 取地址不必然逃逸**,关键看地址是否逃出作用域。返回 `&x` 安全正是因为编译器自动把它改到堆(无悬垂指针)。逃逸=堆分配=GC 压力,用 `go build -gcflags=-m` 看判定、`go test -benchmem` 看 `allocs/op`,热路径少装接口、小对象传值、必要时 `sync.Pool`。
