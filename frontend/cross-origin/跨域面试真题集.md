## 基础题

### Q1：什么是跨域？为什么浏览器要限制跨域？

**难度**：⭐

**参考答案**：

跨域是指浏览器不能执行其他网站的脚本，这是由浏览器的同源策略造成的，是浏览器对 JavaScript 施加的安全限制。

**同源的定义**：协议、域名、端口三者完全相同。

**为什么要限制**：
1. 防止恶意网站读取其他网站的敏感数据
2. 保护用户隐私和数据安全
3. 防止 CSRF（跨站请求伪造）攻击

**举例**：
如果没有同源策略，当你登录网银后，访问了一个恶意网站，该网站就可以通过 JavaScript 获取你的网银页面内容，包括账户余额、交易记录等敏感信息。

**评分标准**：
- 能说出定义（3分）
- 能说出同源三要素（3分）
- 能说出限制原因（2分）
- 能举出实际例子（2分）

### Q2：哪些情况算跨域？

**难度**：⭐

**参考答案**：

当前页面：`https://www.example.com:443/page.html`

| URL | 是否跨域 | 原因 |
|-----|---------|------|
| `https://www.example.com/api` | ❌ 不跨域 | 协议、域名、端口都相同 |
| `http://www.example.com/api` | ✅ 跨域 | 协议不同（https vs http） |
| `https://api.example.com/data` | ✅ 跨域 | 域名不同（www vs api） |
| `https://www.example.com:8080/api` | ✅ 跨域 | 端口不同（443 vs 8080） |
| `https://www.example.com/other.html` | ❌ 不跨域 | 路径不同不影响同源 |

**特殊情况**：
- `http://localhost:8080` 和 `http://127.0.0.1:8080` 也算跨域（域名不同）
- `http://example.com` 和 `http://www.example.com` 也算跨域（子域名不同）

### Q3：同源策略限制了哪些行为？

**难度**：⭐⭐

**参考答案**：

**受限制的**：
1. **DOM 访问**：无法读取跨域 iframe 的 DOM
2. **AJAX/Fetch**：无法向不同源的地址发起请求
3. **Cookie/LocalStorage/IndexedDB**：无法读取不同源的存储数据

**不受限制的**：
1. **资源嵌入**：
   - `<script src="跨域URL">`
   - `<img src="跨域URL">`
   - `<link href="跨域URL">`
   - `<iframe src="跨域URL">`（可以加载，但无法访问内容）
   - `<video>`、`<audio>` 等

2. **表单提交**：可以向跨域 URL 提交表单

**代码示例**：
```javascript
// ✅ 可以加载跨域资源
<script src="https://cdn.example.com/jquery.js"></script>

// ❌ 但无法通过 AJAX 获取
fetch('https://api.example.com/data')
  .then(res => res.json())
// Error: CORS policy

// ❌ 无法访问跨域 iframe 内容
const iframe = document.querySelector('iframe');
iframe.contentWindow.document.body.innerHTML;
// DOMException: Blocked a frame
```

### Q4：常见的跨域解决方案有哪些？

**难度**：⭐⭐

**参考答案**：

1. **CORS**（跨域资源共享）
   - 最常用、最标准的方案
   - 服务端设置响应头：`Access-Control-Allow-Origin`
   - 支持所有 HTTP 方法、可携带 Cookie

2. **代理服务器**
   - 开发环境：webpack-dev-server 代理
   - 生产环境：Nginx 反向代理
   - 通过同源服务器中转请求

3. **JSONP**
   - 利用 `<script>` 标签不受限制的特性
   - 只支持 GET 请求
   - 已过时，不推荐

4. **postMessage**
   - 用于窗口/iframe 通信
   - HTML5 新增 API
   - 安全可控

5. **WebSocket**
   - 不受同源策略限制
   - 支持双向通信
   - 适合实时场景

6. **document.domain**（已废弃）
   - 仅用于同主域的子域
   - 需要双方设置相同的 domain
   - 不推荐使用

**推荐程度**：
- CORS > 代理 > postMessage > WebSocket > JSONP > 其他

### Q5：CORS 是什么？如何配置？

**难度**：⭐⭐

**参考答案**：

**定义**：CORS（Cross-Origin Resource Sharing）跨域资源共享，是 W3C 标准，允许服务器声明哪些源可以访问资源。

**原理**：通过设置 HTTP 响应头告诉浏览器允许跨域访问。

**基本配置（服务端）**：

```javascript
// Node.js Express
app.use((req, res, next) => {
  // 允许的源
  res.header('Access-Control-Allow-Origin', 'https://www.example.com');
  // 允许的方法
  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE');
  // 允许的请求头
  res.header('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  // 允许携带凭证
  res.header('Access-Control-Allow-Credentials', 'true');
  // 预检请求缓存时间
  res.header('Access-Control-Max-Age', '86400');

  if (req.method === 'OPTIONS') {
    return res.sendStatus(200);
  }
  next();
});
```

**前端配置**：
```javascript
fetch('https://api.example.com/data', {
  method: 'POST',
  credentials: 'include', // 携带 Cookie
  headers: {
    'Content-Type': 'application/json'
  }
});
```

## 进阶题

### Q6：简单请求和预检请求的区别是什么？

**难度**：⭐⭐⭐

**参考答案**：

**简单请求**满足以下**所有**条件：

1. **请求方法**：
   - `GET`
   - `POST`
   - `HEAD`

2. **Content-Type**（仅针对 POST）：
   - `text/plain`
   - `multipart/form-data`
   - `application/x-www-form-urlencoded`

3. **请求头**：只能包含简单头
   - `Accept`
   - `Accept-Language`
   - `Content-Language`
   - `Content-Type`（仅限上述三个值）

**预检请求**（Preflight）：

不满足简单请求条件的，会先发送 OPTIONS 请求：

```http
OPTIONS /api/data HTTP/1.1
Host: api.example.com
Origin: https://www.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, Authorization
```

服务端响应：
```http
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://www.example.com
Access-Control-Allow-Methods: POST
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400
```

如果服务端同意，浏览器才会发送真正的请求。

**对比表格**：

| 项目 | 简单请求 | 预检请求 |
|------|---------|---------|
| 触发条件 | 满足所有简单请求条件 | 不满足简单请求条件 |
| 请求次数 | 1次 | 2次（OPTIONS + 实际请求） |
| 自定义头 | ❌ 不允许 | ✅ 允许 |
| Content-Type | 仅3种 | 任意（如 application/json） |
| 性能 | 更好 | 稍差（多一次请求） |

**示例**：
```javascript
// 简单请求
fetch('https://api.com/data', {
  method: 'GET'
});

// 预检请求（因为 Content-Type）
fetch('https://api.com/data', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json' // 触发预检
  },
  body: JSON.stringify({ key: 'value' })
});

// 预检请求（因为自定义头）
fetch('https://api.com/data', {
  headers: {
    'X-Custom-Header': 'value' // 触发预检
  }
});
```

### Q7：如何在跨域请求中携带 Cookie？

**难度**：⭐⭐⭐

**参考答案**：

**需要同时配置前端和后端**：

**前端**：
```javascript
// fetch
fetch('https://api.example.com/data', {
  credentials: 'include' // 关键配置
});

// axios
axios.defaults.withCredentials = true;

// XMLHttpRequest
const xhr = new XMLHttpRequest();
xhr.withCredentials = true;
```

**后端**：
```javascript
// 允许携带凭证
res.header('Access-Control-Allow-Credentials', 'true');

// ⚠️ 重要：携带凭证时，Origin 不能用 *
res.header('Access-Control-Allow-Origin', 'https://www.example.com');
```

**三个重点**：
1. 前端必须设置 `credentials: 'include'`
2. 后端必须设置 `Access-Control-Allow-Credentials: true`
3. 后端的 `Access-Control-Allow-Origin` 不能是 `*`，必须指定具体源

**错误示例**：
```javascript
// ❌ 错误：携带凭证时用了 *
res.header('Access-Control-Allow-Origin', '*');
res.header('Access-Control-Allow-Credentials', 'true');

// 浏览器报错：
// The value of the 'Access-Control-Allow-Origin' header must not be '*'
// when the request's credentials mode is 'include'
```

**Cookie 的 SameSite 属性影响**：
```javascript
// 如果 Cookie 设置了 SameSite=Strict，跨站完全不发送
Set-Cookie: id=a; SameSite=Strict

// SameSite=Lax，部分跨站场景发送（默认值）
Set-Cookie: id=a; SameSite=Lax

// SameSite=None，跨站也发送（需要 Secure）
Set-Cookie: id=a; SameSite=None; Secure
```

### Q8：Cookie 的 SameSite 属性有什么作用？

**难度**：⭐⭐⭐

**参考答案**：

**作用**：防止 CSRF（跨站请求伪造）攻击。

**三个值对比**：

| 值 | 限制 | 使用场景 | 安全性 |
|----|------|---------|--------|
| **Strict** | 完全禁止跨站发送 | 高安全性要求 | 最高 |
| **Lax** | 部分允许跨站（安全导航） | 平衡安全和体验（默认） | 中等 |
| **None** | 允许所有跨站 | 第三方登录、跨域 API | 最低 |

**详细说明**：

**1. Strict（严格模式）**
```javascript
Set-Cookie: sessionId=abc; SameSite=Strict
```
- 完全禁止跨站发送
- 用户从外站链接访问也不发送
- **示例**：用户从 Google 点击链接到你的网站，Cookie 不会发送

**2. Lax（宽松模式，浏览器默认）**
```javascript
Set-Cookie: sessionId=abc; SameSite=Lax
```
- 允许安全的跨站请求（导航）
- 允许的场景：
  - `<a href="...">`
  - `<link rel="prerender" href="...">`
  - `<form method="GET">`
- 禁止的场景：
  - `<form method="POST">`
  - `fetch/axios POST`
  - `<iframe>`、`<img>`

**3. None（无限制）**
```javascript
Set-Cookie: sessionId=abc; SameSite=None; Secure
```
- 允许所有跨站请求
- **必须配合 Secure 使用**（HTTPS）
- 用于：第三方登录、跨域 API

**CSRF 攻击示例**：
```html
<!-- 恶意网站 evil.com -->
<img src="https://bank.com/transfer?to=hacker&amount=10000">

<!-- 如果 bank.com 的 Cookie 没有设置 SameSite，
     用户访问 evil.com 时会自动发起转账请求，
     并携带 bank.com 的登录 Cookie -->

<!-- 防御：设置 SameSite=Lax 或 Strict -->
Set-Cookie: sessionId=abc; SameSite=Lax
<!-- 这样 img 标签的请求不会携带 Cookie -->
```

### Q9：什么是预检请求缓存？如何优化？

**难度**：⭐⭐⭐

**参考答案**：

**问题**：预检请求（OPTIONS）增加了额外的网络开销。

**解决**：使用 `Access-Control-Max-Age` 缓存预检结果。

**配置**：
```javascript
// 服务端设置缓存时间（秒）
res.header('Access-Control-Max-Age', '86400'); // 缓存 24 小时

// Firefox 上限：24 小时
// Chrome/Safari 上限：2 小时（实际测试）
// 建议设置：3600（1小时）到 86400（24小时）
```

**效果**：
```javascript
// 第一次请求：
// 1. OPTIONS 预检
// 2. 实际请求（POST）

// 后续 24 小时内的同类请求：
// 直接发送实际请求（跳过预检）
```

**注意**：
- 缓存是针对**特定的 URL + 方法 + 请求头组合**
- 不同的请求头会触发新的预检
- 浏览器重启会清除缓存

**其他优化方法**：
1. **避免触发预检**：使用简单请求
2. **合并请求**：减少请求次数
3. **使用 WebSocket**：不需要预检

### Q10：为什么 localhost 无法获取 .example.com 域的 Cookie？

**难度**：⭐⭐⭐

**参考答案**：

**Cookie 域名匹配规则**：

```javascript
Cookie: Domain=.example.com

匹配规则：
  - 精确匹配：example.com ✅
  - 子域名匹配：*.example.com ✅
  - 其他域名：❌

具体示例：
  ✅ example.com
  ✅ www.example.com
  ✅ api.example.com
  ✅ local.example.com
  ❌ localhost（不是 *.example.com 的子域）
  ❌ 127.0.0.1
  ❌ other.com
```

**核心原因**：
- `localhost` 是一个独立的域名
- 不属于 `*.example.com` 的范畴
- 浏览器不会将 `.example.com` 的 Cookie 发送给 `localhost`

**解决方案**：

**方法 1：使用本地子域名**
```bash
# 1. 配置 hosts
echo "127.0.0.1 local.example.com" | sudo tee -a /etc/hosts

# 2. 生成 HTTPS 证书
mkcert local.example.com

# 3. 配置 webpack
https: {
  key: fs.readFileSync('local.example.com-key.pem'),
  cert: fs.readFileSync('local.example.com.pem')
}

# 4. 访问
https://local.example.com:8080
# 现在可以获取 .example.com 的 Cookie ✅
```

**方法 2：使用测试用户（开发环境）**
```javascript
// 本地开发时直接使用测试用户，跳过登录
if (location.hostname === 'localhost') {
  this.user = { id: 'test', name: 'Test User' };
}
```

**知识点**：
- **同源策略**：严格（localhost ≠ example.com）
- **Cookie 域名策略**：灵活（允许父子域共享）
- 这是两个不同的安全机制

## 场景题

### Q11：单点登录（SSO）如何实现跨域？

**难度**：⭐⭐⭐⭐

**参考答案**：

**场景**：
- 公司有多个子系统：`app1.company.com`、`app2.company.com`、`app3.company.com`
- 用户在 `login.company.com` 登录一次，访问所有子系统都自动登录

**实现方案**：

**方案 1：共享 Cookie（同主域）**

```javascript
// 登录服务（login.company.com）
app.post('/login', (req, res) => {
  const user = authenticateUser(req.body);
  const token = generateToken(user);

  // ✅ 关键：设置 Cookie 的 Domain 为主域
  res.cookie('sso_token', token, {
    domain: '.company.com', // 所有子域都能访问
    httpOnly: true,
    secure: true,
    sameSite: 'lax',
    maxAge: 7 * 24 * 60 * 60 * 1000 // 7天
  });

  res.json({ success: true });
});

// 各个子系统（app1.company.com, app2.company.com）
app.use((req, res, next) => {
  const token = req.cookies.sso_token; // ✅ 可以获取到
  const user = verifyToken(token);

  if (!user) {
    return res.redirect('https://login.company.com');
  }

  req.user = user;
  next();
});
```

**方案 2：CAS（中央认证服务）**

```javascript
// 1. 用户访问 app1.company.com
// 2. 检测到未登录，重定向到 login.company.com?service=app1.company.com
// 3. 用户在 login.company.com 完成登录
// 4. 重定向回 app1.company.com?ticket=xxx
// 5. app1.company.com 拿 ticket 去 login.company.com 验证
// 6. 验证通过，建立 app1.company.com 的登录态
```

**方案 3：JWT Token（跨主域）**

```javascript
// 登录后返回 Token
const token = jwt.sign({ userId: user.id }, SECRET);

// 前端存储到 LocalStorage
localStorage.setItem('token', token);

// 每次请求带上 Token
fetch('https://api.company.com/data', {
  headers: {
    'Authorization': 'Bearer ' + token
  }
});
```

**对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 共享 Cookie | 简单、自动携带 | 仅同主域 | 同主域多子系统 |
| CAS | 标准、安全 | 实现复杂 | 企业级系统 |
| JWT Token | 跨域无限制 | 需手动携带 | 前后端分离 |

### Q12：微前端中主应用和子应用如何跨域通信？

**难度**：⭐⭐⭐⭐

**参考答案**：

**场景**：
- 主应用：`https://main.company.com`
- 子应用：`https://sub1.company.com`、`https://sub2.company.com`
- 需要互相通信（传递用户信息、事件等）

**方案 1：postMessage（iframe 模式）**

```javascript
// ============ 主应用 ============
const microApp = document.querySelector('#microAppIframe');

// 发送消息给子应用
function sendToMicroApp(type, data) {
  microApp.contentWindow.postMessage(
    { type, data },
    'https://sub1.company.com' // 目标源
  );
}

// 接收子应用消息
window.addEventListener('message', (event) => {
  // 验证来源
  if (!['https://sub1.company.com', 'https://sub2.company.com'].includes(event.origin)) {
    return;
  }

  switch (event.data.type) {
    case 'LOGOUT':
      handleLogout();
      break;
    case 'ROUTE_CHANGE':
      updateRoute(event.data.data);
      break;
  }
});

// 示例：发送用户信息
sendToMicroApp('UPDATE_USER', { id: 123, name: '张三' });

// ============ 子应用 ============
// 接收主应用消息
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://main.company.com') return;

  switch (event.data.type) {
    case 'UPDATE_USER':
      store.commit('setUser', event.data.data);
      break;
  }
});

// 发送消息给主应用
function notifyMainApp(type, data) {
  window.parent.postMessage(
    { type, data },
    'https://main.company.com'
  );
}

// 示例：通知登出
notifyMainApp('LOGOUT');
```

**方案 2：自定义事件总线（qiankun）**

```javascript
// 主应用
import { initGlobalState } from 'qiankun';

const actions = initGlobalState({
  user: null,
  theme: 'light'
});

// 监听变化
actions.onGlobalStateChange((state, prev) => {
  console.log('状态变化:', state, prev);
});

// 修改状态
actions.setGlobalState({ user: { id: 123, name: '张三' } });

// 子应用
export async function mount(props) {
  props.onGlobalStateChange((state, prev) => {
    console.log('收到状态变化:', state);
  });

  props.setGlobalState({ theme: 'dark' });
}
```

**方案 3：SharedWorker（同浏览器）**

```javascript
// 创建 SharedWorker
const worker = new SharedWorker('worker.js');

// 发送消息
worker.port.postMessage({ type: 'UPDATE_USER', data: user });

// 接收消息
worker.port.onmessage = (event) => {
  console.log('收到消息:', event.data);
};

// worker.js
const ports = [];

onconnect = (event) => {
  const port = event.ports[0];
  ports.push(port);

  port.onmessage = (e) => {
    // 广播给所有连接
    ports.forEach(p => {
      p.postMessage(e.data);
    });
  };
};
```

### Q13：手写一个 JSONP 函数

**难度**：⭐⭐⭐

**参考答案**：

```javascript
/**
 * JSONP 跨域请求
 * @param {string} url - 请求地址
 * @param {object} params - 请求参数
 * @param {string} callbackKey - 回调函数名参数（默认 'callback'）
 * @returns {Promise}
 */
function jsonp(url, params = {}, callbackKey = 'callback') {
  return new Promise((resolve, reject) => {
    // 1. 生成唯一的回调函数名
    const callbackName = `jsonp_${Date.now()}_${Math.random().toString(36).substr(2)}`;

    // 2. 创建全局回调函数
    window[callbackName] = (data) => {
      resolve(data);
      cleanup();
    };

    // 3. 构建 URL 参数
    params[callbackKey] = callbackName;
    const queryString = Object.keys(params)
      .map(key => `${encodeURIComponent(key)}=${encodeURIComponent(params[key])}`)
      .join('&');
    const finalUrl = `${url}?${queryString}`;

    // 4. 创建 script 标签
    const script = document.createElement('script');
    script.src = finalUrl;
    script.type = 'text/javascript';

    // 5. 错误处理
    script.onerror = () => {
      reject(new Error(`JSONP request failed: ${url}`));
      cleanup();
    };

    // 6. 超时处理
    const timer = setTimeout(() => {
      reject(new Error(`JSONP request timeout: ${url}`));
      cleanup();
    }, 10000); // 10秒超时

    // 7. 清理函数
    function cleanup() {
      clearTimeout(timer);
      delete window[callbackName];
      if (script.parentNode) {
        script.parentNode.removeChild(script);
      }
    }

    // 8. 发起请求
    document.body.appendChild(script);
  });
}

// 使用示例
jsonp('https://api.example.com/data', { id: 123 })
  .then(data => {
    console.log('成功:', data);
  })
  .catch(error => {
    console.error('失败:', error);
  });
```

**服务端示例**：
```javascript

  const callbackName = req.query.callback;
  const data = { name: '张三', age: 25 };
```

### Q14：手写一个简单的 CORS 中间件

**难度**：⭐⭐⭐

**参考答案**：

```javascript
/**
 * CORS 中间件
 * @param {object} options - 配置选项
 * @returns {Function} Express 中间件
 */
function cors(options = {}) {
  const {
    origin = '*',
    methods = ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS'],
    allowedHeaders = ['Content-Type', 'Authorization'],
    credentials = false,
    maxAge = 86400
  } = options;

  return function(req, res, next) {
    // 1. 设置 Access-Control-Allow-Origin
    if (origin === '*') {
      res.header('Access-Control-Allow-Origin', '*');
    } else if (typeof origin === 'string') {
      res.header('Access-Control-Allow-Origin', origin);
    } else if (Array.isArray(origin)) {
      // 多个允许的源
      const requestOrigin = req.headers.origin;
      if (origin.includes(requestOrigin)) {
        res.header('Access-Control-Allow-Origin', requestOrigin);
      }
    } else if (typeof origin === 'function') {
      // 动态判断
      const allowed = origin(req.headers.origin);
      if (allowed) {
        res.header('Access-Control-Allow-Origin', req.headers.origin);
      }
    }

    // 2. 设置允许的方法
    res.header('Access-Control-Allow-Methods', methods.join(', '));

    // 3. 设置允许的请求头
    if (allowedHeaders) {
      res.header('Access-Control-Allow-Headers', allowedHeaders.join(', '));
    } else {
      // 动态允许所有请求头
      const requestHeaders = req.headers['access-control-request-headers'];
      if (requestHeaders) {
        res.header('Access-Control-Allow-Headers', requestHeaders);
      }
    }

    // 4. 设置是否允许携带凭证
    if (credentials) {
      res.header('Access-Control-Allow-Credentials', 'true');
    }

    // 5. 设置预检请求缓存时间
    res.header('Access-Control-Max-Age', maxAge);

    // 6. 处理预检请求
    if (req.method === 'OPTIONS') {
      return res.sendStatus(204);
    }

    next();
  };
}

// 使用示例
const express = require('express');
const app = express();

// 方式 1：允许所有源
app.use(cors());

// 方式 2：指定单个源
app.use(cors({
  origin: 'https://www.example.com',
  credentials: true
}));

// 方式 3：指定多个源
app.use(cors({
  origin: [
    'https://www.example.com',
    'https://app.example.com'
  ],
  credentials: true
}));

// 方式 4：动态判断
app.use(cors({
  origin: (requestOrigin) => {
    // 允许所有 .example.com 的子域
    return /\.example\.com$/.test(requestOrigin);
  },
  credentials: true
}));
```

### Q15：分析这段代码为什么报跨域错误，如何修复？

**难度**：⭐⭐⭐⭐

**题目**：
```javascript
// 页面地址：https://www.example.com

// 前端代码
fetch('https://api.example.com/users', {
  method: 'POST',
  credentials: 'include',
  headers: {
    'Content-Type': 'application/json',
    'X-Custom-Token': 'abc123'
  },
  body: JSON.stringify({ name: '张三' })
});

// 后端代码（Node.js）
app.post('/users', (req, res) => {
  res.header('Access-Control-Allow-Origin', '*');
  res.header('Access-Control-Allow-Methods', 'POST');
  res.json({ success: true });
});

// 浏览器报错：
// Access to fetch at 'https://api.example.com/users' from origin
// 'https://www.example.com' has been blocked by CORS policy:
// The value of the 'Access-Control-Allow-Origin' header must not be '*'
// when the request's credentials mode is 'include'.
```

**参考答案**：

**问题分析**：

1. **错误原因**：
   - 前端设置了 `credentials: 'include'`（携带凭证）
   - 后端设置了 `Access-Control-Allow-Origin: *`
   - 当携带凭证时，Origin 不能是 `*`

2. **其他潜在问题**：
   - 自定义请求头 `X-Custom-Token` 会触发预检请求
   - 后端没有设置 `Access-Control-Allow-Headers`
   - 后端没有设置 `Access-Control-Allow-Credentials`
   - 后端没有处理 OPTIONS 预检请求

**完整修复**：

```javascript
// 前端代码（不变）
fetch('https://api.example.com/users', {
  method: 'POST',
  credentials: 'include',
  headers: {
    'Content-Type': 'application/json',
    'X-Custom-Token': 'abc123'
  },
  body: JSON.stringify({ name: '张三' })
});

// 后端代码（修复后）
app.use((req, res, next) => {
  // ✅ 修复 1：指定具体的 Origin（不能用 *）
  res.header('Access-Control-Allow-Origin', 'https://www.example.com');

  // ✅ 修复 2：允许携带凭证
  res.header('Access-Control-Allow-Credentials', 'true');

  // ✅ 修复 3：允许的方法
  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');

  // ✅ 修复 4：允许自定义请求头
  res.header('Access-Control-Allow-Headers', 'Content-Type, X-Custom-Token');

  // ✅ 修复 5：预检请求缓存
  res.header('Access-Control-Max-Age', '86400');

  // ✅ 修复 6：处理 OPTIONS 预检请求
  if (req.method === 'OPTIONS') {
    return res.sendStatus(204);
  }

  next();
});

app.post('/users', (req, res) => {
  res.json({ success: true });
});
```

**关键点总结**：
1. **携带凭证时，Origin 必须指定具体值**
2. **必须设置 `Allow-Credentials: true`**
3. **自定义头需要在 `Allow-Headers` 中声明**
4. **必须处理 OPTIONS 预检请求**
