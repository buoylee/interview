## 1. 基础概念

### 1.1 什么是跨域？

**定义**：浏览器从一个域名的网页去请求另一个域名的资源时，如果域名、协议、端口任一不同，都算跨域。

### 1.2 同源的判断标准

**协议 + 域名 + 端口** 三者完全相同才算同源。

```javascript
// 当前页面：https://www.example.com:443/page.html

✅ 同源：
https://www.example.com/api/data
https://www.example.com:443/other.html

❌ 跨域（协议不同）：
http://www.example.com/api/data

❌ 跨域（域名不同）：
https://api.example.com/data
https://www.other.com/data

❌ 跨域（端口不同）：
https://www.example.com:8080/data

❌ 跨域（子域名不同）：
https://sub.example.com/data
```

### 1.3 为什么会有跨域限制？

**安全原因**：防止恶意网站读取其他网站的敏感数据。

**典型攻击场景**：
```javascript
// 假设没有同源策略
// 用户登录了 bank.com
// 然后访问了恶意网站 evil.com

// evil.com 的代码：
fetch('https://bank.com/api/account')
  .then(res => res.json())
  .then(data => {
    // 恶意网站可以获取用户的银行账户信息！
    sendToHacker(data);
  });

// ❌ 同源策略阻止了这种攻击
```

## 2. 同源策略详解

### 2.1 同源策略限制的内容

| 类型 | 是否受限 | 说明 |
|------|---------|------|
| **DOM 访问** | ✅ 受限 | 无法读取跨域 iframe 的 DOM |
| **AJAX/Fetch** | ✅ 受限 | 无法发起跨域请求（除非 CORS） |
| **Cookie** | ✅ 受限 | 无法读取跨域 Cookie（但有例外） |
| **LocalStorage** | ✅ 受限 | 无法读取跨域 LocalStorage |
| **IndexedDB** | ✅ 受限 | 无法读取跨域 IndexedDB |
| **图片/CSS/JS** | ❌ 不受限 | 可以跨域加载（但无法读取内容） |
| **iframe 加载** | ❌ 不受限 | 可以跨域加载（但无法访问内容） |
| **表单提交** | ❌ 不受限 | 可以跨域提交 |

### 2.2 同源策略的三个层面

#### (1) DOM 同源策略

**禁止对不同源的 DOM 进行操作**

```javascript
// 页面：https://a.com/index.html
const iframe = document.createElement('iframe');
iframe.src = 'https://b.com/page.html';
document.body.appendChild(iframe);

// ❌ 跨域错误
iframe.contentWindow.document.body.innerHTML;
// DOMException: Blocked a frame with origin "https://a.com"
// from accessing a cross-origin frame.
```

#### (2) XMLHttpRequest 同源策略

**禁止向不同源的地址发起 HTTP 请求**

```javascript
// 页面：https://a.com
fetch('https://b.com/api/data')
  .then(res => res.json())
// ❌ CORS error: No 'Access-Control-Allow-Origin' header
```

#### (3) Cookie、LocalStorage 同源策略

**禁止读取不同源的存储数据**

```javascript
// 页面：https://a.com
// ❌ 无法读取 b.com 的 Cookie
document.cookie; // 只能看到 a.com 的 Cookie

// ❌ 无法读取 b.com 的 LocalStorage
localStorage.getItem('key'); // 只能读取 a.com 的数据
```

### 2.3 同源策略的例外

#### (1) 资源嵌入允许跨域

```html
<!-- ✅ 可以跨域加载 -->
<script src="https://cdn.example.com/jquery.js"></script>
<link href="https://cdn.example.com/style.css" rel="stylesheet">
<img src="https://cdn.example.com/image.png">
<video src="https://cdn.example.com/video.mp4"></video>
<iframe src="https://other.com/page.html"></iframe>

<!-- ❌ 但是无法通过 JS 读取内容（canvas 污染） -->
```

#### (2) 表单提交允许跨域

```html
<!-- ✅ 可以跨域提交 -->
<form action="https://other.com/api/submit" method="POST">
  <input name="data" value="test">
  <button type="submit">提交</button>
</form>
```

## 3. 跨域解决方案

### 3.1 CORS（跨域资源共享）★★★★★

**最常用、最标准的跨域解决方案**

#### 原理

服务端设置响应头，告诉浏览器允许跨域访问。

#### 简单请求 vs 预检请求

**简单请求（Simple Request）**：

满足以下所有条件：
1. 请求方法：`GET`、`POST`、`HEAD`
2. Content-Type：
   - `text/plain`
   - `multipart/form-data`
   - `application/x-www-form-urlencoded`
3. 请求头：只能包含简单头（Accept、Accept-Language、Content-Language 等）

**预检请求（Preflight Request）**：

不满足简单请求条件的，会先发送 OPTIONS 请求：

```http
OPTIONS /api/data HTTP/1.1
Host: api.example.com
Origin: https://www.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type
```

#### 服务端配置

**Node.js (Express)**

```javascript
// 方法 1：手动设置响应头
app.use((req, res, next) => {
  // 允许的源（* 表示所有）
  res.header('Access-Control-Allow-Origin', 'https://www.example.com');
  // 允许的方法
  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
  // 允许的请求头
  res.header('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  // 允许携带凭证（Cookie）
  res.header('Access-Control-Allow-Credentials', 'true');
  // 预检请求缓存时间（秒）
  res.header('Access-Control-Max-Age', '86400');

  // 处理 OPTIONS 预检请求
  if (req.method === 'OPTIONS') {
    return res.sendStatus(200);
  }
  next();
});

// 方法 2：使用 cors 中间件
const cors = require('cors');
app.use(cors({
  origin: 'https://www.example.com', // 允许的源
  credentials: true, // 允许携带 Cookie
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization']
}));
```

**Nginx**

```nginx
location /api/ {
    add_header 'Access-Control-Allow-Origin' 'https://www.example.com' always;
    add_header 'Access-Control-Allow-Methods' 'GET, POST, OPTIONS' always;
    add_header 'Access-Control-Allow-Headers' 'Content-Type, Authorization' always;
    add_header 'Access-Control-Allow-Credentials' 'true' always;

    if ($request_method = 'OPTIONS') {
        return 204;
    }

    proxy_pass http://backend;
}
```

#### 前端配置

```javascript
// fetch
fetch('https://api.example.com/data', {
  method: 'POST',
  credentials: 'include', // 携带 Cookie
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({ key: 'value' })
});

// axios
axios.defaults.withCredentials = true; // 全局配置
axios.post('https://api.example.com/data', { key: 'value' });
```

#### 注意事项

⚠️ **重要**：
- `Access-Control-Allow-Origin` 为 `*` 时，不能设置 `credentials: true`
- 携带 Cookie 时，必须指定具体的 Origin，不能用 `*`
- 预检请求不会携带 Cookie

### 3.2 JSONP ★★★☆☆

**利用 `<script>` 标签不受同源策略限制的特性**

#### 原理

```html
<!-- 1. 定义回调函数 -->
<script>
function handleData(data) {
  console.log('接收到数据:', data);
}
</script>

<!-- 2. 动态创建 script 标签 -->
<script src="https://api.example.com/data?callback=handleData"></script>

<!-- 3. 服务端返回 -->
handleData({"name": "张三", "age": 25})
```

#### 前端实现

```javascript
function jsonp(url, callbackName, callback) {
  // 1. 创建全局回调函数
  window[callbackName] = function(data) {
    callback(data);
    // 清理
    delete window[callbackName];
    document.body.removeChild(script);
  };

  // 2. 创建 script 标签
  const script = document.createElement('script');
  script.src = `${url}?callback=${callbackName}`;
  document.body.appendChild(script);
}

// 使用
jsonp('https://api.example.com/data', 'handleData', (data) => {
  console.log(data);
});
```

#### 服务端实现

```javascript
// Node.js
app.get('/api/data', (req, res) => {
  const callbackName = req.query.callback;
  const data = { name: '张三', age: 25 };

  // 返回 JavaScript 代码
  res.send(`${callbackName}(${JSON.stringify(data)})`);
});
```

#### 优缺点

**优点**：
- ✅ 兼容性好（支持 IE6+）
- ✅ 实现简单

**缺点**：
- ❌ 只支持 GET 请求
- ❌ 安全性差（容易遭受 XSS 攻击）
- ❌ 错误处理困难
- ❌ 已过时，不推荐使用

### 3.3 代理服务器 ★★★★★

**通过同源的服务器中转请求**

#### 原理

```
浏览器（同源）→ 代理服务器 → 目标服务器
  ✅                ✅
```

#### 开发环境：webpack-dev-server

```javascript
// vue.config.js / webpack.config.js
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'https://api.example.com', // 目标服务器
        changeOrigin: true, // 改变请求头中的 Origin
        pathRewrite: {
          '^/api': '' // 重写路径：/api/users → /users
        }
      }
    }
  }
}

// 前端请求（同源）
fetch('/api/users') // 实际请求：https://api.example.com/users
```

#### 生产环境：Nginx 反向代理

```nginx
server {
    listen 80;
    server_name www.example.com;

    # 前端静态资源
    location / {
        root /var/www/html;
        index index.html;
    }

    # 代理 API 请求
    location /api/ {
        proxy_pass https://api.example.com/;
        proxy_set_header Host api.example.com;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

#### Node.js 中间件

```javascript
const express = require('express');
const { createProxyMiddleware } = require('http-proxy-middleware');

const app = express();

// 代理配置
app.use('/api', createProxyMiddleware({
  target: 'https://api.example.com',
  changeOrigin: true,
  pathRewrite: {
    '^/api': ''
  }
}));

app.listen(3000);
```

#### 优缺点

**优点**：
- ✅ 支持所有 HTTP 方法
- ✅ 可以携带 Cookie
- ✅ 服务端无需配置 CORS
- ✅ 可以做请求/响应拦截和修改

**缺点**：
- ❌ 需要额外的服务器
- ❌ 增加了请求链路

### 3.4 postMessage ★★★★☆

**用于不同窗口/iframe 之间的跨域通信**

#### 使用场景

- 页面与 iframe 通信
- 页面与 window.open 打开的页面通信
- 多个 tab 页之间通信

#### 示例

**父页面（https://a.com）**

```javascript
const iframe = document.getElementById('myIframe');

// 发送消息
iframe.contentWindow.postMessage(
  { type: 'greeting', data: 'Hello' },
  'https://b.com' // 目标源（安全考虑）
);

// 接收消息
window.addEventListener('message', (event) => {
  // 安全检查：验证来源
  if (event.origin !== 'https://b.com') return;

  console.log('收到消息:', event.data);
});
```

**子页面（https://b.com）**

```html
<iframe id="myIframe" src="https://b.com/page.html"></iframe>

<script>
// 接收消息
window.addEventListener('message', (event) => {
  // 安全检查
  if (event.origin !== 'https://a.com') return;

  console.log('收到消息:', event.data);

  // 回复消息
  event.source.postMessage(
    { type: 'reply', data: 'Hi' },
    event.origin
  );
});
</script>
```

#### 安全注意事项

```javascript
window.addEventListener('message', (event) => {
  // ⚠️ 必须验证来源
  if (event.origin !== 'https://trusted.com') {
    return; // 拒绝不可信来源
  }

  // ⚠️ 验证消息内容
  if (typeof event.data !== 'object' || !event.data.type) {
    return; // 拒绝格式错误的消息
  }

  // 处理消息
  handleMessage(event.data);
});
```

### 3.5 WebSocket ★★★☆☆

**不受同源策略限制的全双工通信协议**

#### 特点

- ✅ 不受同源策略限制
- ✅ 支持双向通信
- ✅ 实时性强

#### 使用示例

**前端**

```javascript
// 连接任意域名的 WebSocket 服务
const ws = new WebSocket('wss://api.example.com/socket');

ws.onopen = () => {
  console.log('连接成功');
  ws.send('Hello Server');
};

ws.onmessage = (event) => {
  console.log('收到消息:', event.data);
};

ws.onerror = (error) => {
  console.error('错误:', error);
};

ws.onclose = () => {
  console.log('连接关闭');
};
```

**后端（Node.js）**

```javascript
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', (ws) => {
  console.log('客户端连接');

  ws.on('message', (message) => {
    console.log('收到:', message);
    ws.send('Hello Client');
  });

  ws.on('close', () => {
    console.log('客户端断开');
  });
});
```

### 3.6 document.domain ★★☆☆☆

**用于同一主域下的不同子域之间通信**

#### 使用场景

仅限于：`a.example.com` 和 `b.example.com` 这种同主域的情况。

#### 示例

**父页面（http://a.example.com）**

```javascript
// 设置 document.domain 为主域
document.domain = 'example.com';

const iframe = document.getElementById('myIframe');
// ✅ 现在可以访问 iframe 的内容
iframe.contentWindow.document.body.innerHTML;
```

**子页面（http://b.example.com）**

```javascript
// 也要设置相同的 domain
document.domain = 'example.com';
```

#### 限制

- ❌ 只能用于同主域的子域
- ❌ 必须是 HTTP 协议（HTTPS 不支持）
- ❌ 端口必须相同
- ⚠️ 已被废弃，不推荐使用

## 4. Cookie 跨域专题

### 4.1 Cookie 的域名策略

**重要区分**：同源策略 ≠ Cookie 域名策略

#### 同源策略（严格）

```javascript
https://a.com !== https://b.com  ❌ 跨域
```

#### Cookie 域名策略（灵活）

```javascript
Cookie: Domain=.example.com

可以访问此 Cookie 的域名：
✅ example.com
✅ www.example.com
✅ api.example.com
✅ 任何 *.example.com

不能访问的：
❌ other.com
❌ localhost
```

### 4.2 Cookie 属性详解

```http
Set-Cookie: sessionId=abc123; Domain=.example.com; Path=/; Expires=Wed, 09 Jun 2024 10:18:14 GMT; Secure; HttpOnly; SameSite=Lax
```

| 属性 | 说明 | 示例 |
|------|------|------|
| **Domain** | Cookie 的域名 | `.example.com` |
| **Path** | Cookie 的路径 | `/` 或 `/api` |
| **Expires** | 过期时间 | `Wed, 09 Jun 2024 10:18:14 GMT` |
| **Max-Age** | 有效期（秒） | `3600`（1小时） |
| **Secure** | 仅 HTTPS 传输 | 无值，存在即生效 |
| **HttpOnly** | 禁止 JavaScript 访问 | 无值，存在即生效 |
| **SameSite** | 跨站请求限制 | `Strict`/`Lax`/`None` |

### 4.3 SameSite 详解

**作用**：防止 CSRF 攻击

```javascript
// Strict：严格模式
Set-Cookie: id=a3fWa; SameSite=Strict
// 完全禁止跨站发送 Cookie
// 用户从 google.com 点击链接到 bank.com，Cookie 不会发送

// Lax：宽松模式（默认）
Set-Cookie: id=a3fWa; SameSite=Lax
// 允许安全的跨站 GET 请求（导航）
// <a href>, <link rel="prerender">, <form method="GET">

// None：无限制
Set-Cookie: id=a3fWa; SameSite=None; Secure
// 允许所有跨站请求
// ⚠️ 必须配合 Secure 使用（HTTPS）
```

### 4.4 跨域携带 Cookie

**前端配置**

```javascript
// fetch
fetch('https://api.example.com/data', {
  credentials: 'include' // 携带 Cookie
});

// axios
axios.defaults.withCredentials = true;

// XMLHttpRequest
xhr.withCredentials = true;
```

**后端配置**

```javascript
// 允许携带凭证
res.header('Access-Control-Allow-Credentials', 'true');

// ⚠️ 携带 Cookie 时，Origin 不能是 *
res.header('Access-Control-Allow-Origin', 'https://www.example.com');
```

### 4.5 第三方 Cookie

**定义**：当前页面和 Cookie 的域名不同

```html
<!-- 页面：https://a.com -->
<iframe src="https://b.com/tracker.html"></iframe>

<!-- b.com 设置的 Cookie 就是第三方 Cookie -->
```

**浏览器限制**：
- Chrome：默认阻止第三方 Cookie（需要 SameSite=None; Secure）
- Safari：完全阻止第三方 Cookie
- Firefox：Enhanced Tracking Protection

**解决方案**：
- 使用第一方 Cookie（同域名）
- 使用 localStorage + postMessage
- 使用服务端 Session

### 5.1 案例 1：单点登录（SSO）

**场景**：用户在 `login.company.com` 登录后，访问 `app1.company.com` 和 `app2.company.com` 都自动登录

**实现方案**

// 1. 登录服务（login.company.com）
app.post('/login', (req, res) => {

  const user = authenticateUser(req.body);

  res.cookie('sessionId', generateSessionId(), {

### 5.2 案例 2：微前端跨域通信

**场景**：主应用和子应用在不同域名，需要通信

**实现方案：postMessage**

```javascript
// 主应用（https://main.com）
const microApp = document.getElementById('microAppIframe');

// 发送消息给子应用
function sendToMicroApp(data) {
  microApp.contentWindow.postMessage(
    { type: 'UPDATE_USER', data },
    'https://micro.com'
  );
}

// 接收子应用消息
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://micro.com') return;

  if (event.data.type === 'LOGOUT') {
    handleLogout();
  }
});

// 子应用（https://micro.com）
// 接收主应用消息
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://main.com') return;

  if (event.data.type === 'UPDATE_USER') {
    updateUser(event.data.data);
  }
});

// 发送消息给主应用
function notifyMainApp(data) {
  window.parent.postMessage(
    { type: 'LOGOUT' },
    'https://main.com'
  );
}
```

## 6. 高频面试题

### Q1：什么是跨域？为什么会有跨域问题？

**答**：

跨域是指浏览器从一个域名的网页去请求另一个域名的资源，由于浏览器的同源策略限制而被阻止。

**同源的判断标准**：协议、域名、端口三者完全相同。

**为什么有跨域限制**：
- 安全考虑，防止恶意网站读取其他网站的敏感数据
- 例如：防止恶意网站通过 AJAX 读取用户在银行网站的账户信息

### Q2：同源策略限制了哪些行为？

**答**：

1. **DOM 访问**：无法读取跨域 iframe 的 DOM
2. **AJAX/Fetch 请求**：无法向不同源发起请求
3. **Cookie/LocalStorage/IndexedDB**：无法读取不同源的存储数据

**不受限制的**：
- 资源嵌入：`<script>`、`<img>`、`<link>`、`<iframe>` 等标签
- 表单提交

### Q3：常见的跨域解决方案有哪些？

**答**：

1. **CORS**（最常用）：服务端设置 `Access-Control-Allow-Origin` 响应头
2. **代理服务器**：通过同源服务器中转请求
3. **JSONP**：利用 `<script>` 标签不受限制的特性（仅 GET）
4. **postMessage**：用于窗口/iframe 通信
5. **WebSocket**：不受同源策略限制
6. **Nginx 反向代理**：生产环境常用

### Q4：CORS 的简单请求和预检请求有什么区别？

**答**：

**简单请求**：
- 方法：GET、POST、HEAD
- Content-Type：`text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`
- 只包含简单请求头
- **直接发送请求**

**预检请求**（Preflight）：
- 不满足简单请求条件
- 先发送 OPTIONS 请求询问服务器是否允许
- 服务器同意后再发送真正的请求

**示例**：
```javascript
// 简单请求
fetch('https://api.com/data', {
  method: 'GET'
});

// 预检请求（因为自定义了 Content-Type）
fetch('https://api.com/data', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json' // 触发预检
  },
  body: JSON.stringify({ key: 'value' })
});
```

### Q5：如何在跨域请求中携带 Cookie？

**答**：

**前端**：
```javascript
// fetch
fetch('https://api.example.com/data', {
  credentials: 'include' // 携带 Cookie
});

// axios
axios.defaults.withCredentials = true;
```

**后端**：
```javascript
// 允许携带凭证
res.header('Access-Control-Allow-Credentials', 'true');

// ⚠️ 必须指定具体的 Origin，不能用 *
res.header('Access-Control-Allow-Origin', 'https://www.example.com');
```

### Q6：JSONP 的原理是什么？有什么缺点？

**答**：

**原理**：利用 `<script>` 标签的 src 属性不受同源策略限制的特性，动态创建 script 标签加载跨域脚本。

**流程**：
1. 前端定义回调函数
2. 动态创建 `<script src="https://api.com/data?callback=handleData">`
3. 服务端返回 `handleData({"data": "value"})`
4. 浏览器执行，调用回调函数

**缺点**：
- ❌ 只支持 GET 请求
- ❌ 安全性差（易受 XSS 攻击）
- ❌ 错误处理困难
- ❌ 已过时，不推荐使用

### Q7：postMessage 如何使用？有哪些安全注意事项？

**答**：

**使用**：
```javascript
// 发送消息
targetWindow.postMessage(data, targetOrigin);

// 接收消息
window.addEventListener('message', (event) => {
  // 处理消息
  console.log(event.data);
});
```

**安全注意事项**：
1. **验证来源**：
```javascript
if (event.origin !== 'https://trusted.com') return;
```

2. **验证数据**：
```javascript
if (typeof event.data !== 'object' || !event.data.type) return;
```

3. **指定目标源**：
```javascript
// ✅ 好的做法
window.postMessage(data, 'https://trusted.com');

// ❌ 不安全
window.postMessage(data, '*');
```

### Q8：Cookie 的 SameSite 属性有什么作用？

**答**：

**作用**：防止 CSRF（跨站请求伪造）攻击

**三个值**：

1. **Strict**（严格）：
   - 完全禁止跨站发送 Cookie
   - 用户从外站链接访问也不发送
   - 最安全，但用户体验差

2. **Lax**（宽松，默认）：
   - 允许部分跨站请求（安全的导航，如链接点击）
   - 禁止跨站 POST、AJAX
   - 平衡安全和体验

3. **None**（无限制）：
   - 允许所有跨站请求
   - 必须配合 `Secure` 使用（HTTPS）
   - 用于需要跨站的场景（如第三方登录）

### Q9：开发环境和生产环境分别如何处理跨域？

**答**：

**开发环境**：
```javascript
// webpack-dev-server 代理
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'https://api.example.com',
        changeOrigin: true
      }
    }
  }
}
```

**生产环境**：

方案1：CORS
```javascript
// 服务端配置
app.use(cors({
  origin: 'https://www.example.com',
  credentials: true
}));
```

方案2：Nginx 反向代理
```nginx
location /api/ {
    proxy_pass https://api.example.com/;
}
```

### Q10：为什么 localhost 无法获取 .example.com 域的 Cookie？

**答**：

**Cookie 域名匹配规则**：
```javascript
Cookie: Domain=.example.com

匹配：
✅ example.com
✅ www.example.com
✅ api.example.com
✅ local.example.com

不匹配：
❌ localhost（不是 *.example.com 的子域）
❌ 127.0.0.1
```

**解决方案**：
1. 配置本地域名：`local.example.com → 127.0.0.1`
2. 生成 HTTPS 证书
3. 通过 `https://local.example.com` 访问

## 7. 进阶知识

### 7.1 预检请求优化

**问题**：每次请求都发送 OPTIONS 预检，影响性能

**解决**：使用 `Access-Control-Max-Age` 缓存预检结果

```javascript
// 服务端
res.header('Access-Control-Max-Age', '86400'); // 缓存 24 小时

// 24 小时内，同样的跨域请求不会再发送 OPTIONS
```

### 7.2 CORS 与认证

**场景**：跨域请求需要携带 Token

**方案 1：Authorization 头**
```javascript
fetch('https://api.example.com/data', {
  headers: {
    'Authorization': 'Bearer ' + token
  }
});

// 服务端
res.header('Access-Control-Allow-Headers', 'Authorization');
```

**方案 2：自定义头**
```javascript
fetch('https://api.example.com/data', {
  headers: {
    'X-Access-Token': token
  }
});

// 服务端
res.header('Access-Control-Allow-Headers', 'X-Access-Token');
```

### 7.3 CORS 错误排查

**常见错误 1**：
```
No 'Access-Control-Allow-Origin' header is present
```
**原因**：服务端未设置 CORS 响应头
**解决**：添加 `Access-Control-Allow-Origin`

**常见错误 2**：
```
The 'Access-Control-Allow-Origin' header contains multiple values
```
**原因**：重复设置了响应头
**解决**：检查是否在多处设置了 CORS 头

**常见错误 3**：
```
Credentials flag is 'true', but 'Access-Control-Allow-Credentials' is not true
```
**原因**：前端设置了 `credentials: 'include'`，但服务端未设置
**解决**：服务端添加 `Access-Control-Allow-Credentials: true`

**常见错误 4**：
```
The value of the 'Access-Control-Allow-Origin' header must not be '*' when credentials flag is true
```
**原因**：携带凭证时，Origin 不能是 `*`
**解决**：指定具体的 Origin

### 7.4 Canvas 跨域图片污染

**问题**：跨域图片绘制到 Canvas 后无法导出

```javascript
const img = new Image();
img.src = 'https://other.com/image.png';
img.onload = () => {
  ctx.drawImage(img, 0, 0);
  canvas.toDataURL(); // ❌ Tainted canvas error
};
```

**解决**：
```javascript
const img = new Image();
img.crossOrigin = 'anonymous'; // 启用 CORS
img.src = 'https://other.com/image.png';

// 服务端需要返回 CORS 头
// Access-Control-Allow-Origin: *
```

### 7.5 Service Worker 与跨域

**特点**：Service Worker 可以拦截跨域请求

```javascript
// service-worker.js
self.addEventListener('fetch', (event) => {
  if (event.request.url.includes('api.example.com')) {
    event.respondWith(
      fetch(event.request, {
        mode: 'cors' // 跨域模式
      })
    );
  }
});
```

### 7.6 iframe 沙箱与跨域

```html
<!-- 沙箱 iframe，限制跨域行为 -->
<iframe
  src="https://untrusted.com"
  sandbox="allow-scripts allow-same-origin"
></iframe>

<!-- sandbox 属性值：
  - allow-scripts: 允许脚本执行
  - allow-same-origin: 允许同源访问
  - allow-forms: 允许表单提交
  - allow-top-navigation: 允许导航顶层窗口
-->
```
