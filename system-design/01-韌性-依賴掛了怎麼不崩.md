# 第 1 章 · 韌性:依賴掛了怎麼不崩

> 🔬 這是資深後端面試最愛問的一塊:「**你的服務依賴的下游掛了、或變慢了,會發生什麼?你怎麼辦?**」普通人答「加個重試」就結束了——而且答錯了(重試常常是補刀)。能把**級聯失敗**講清、把**超時/重試/熔斷/隔離/限流/降級**六件套各自防什麼、它們怎麼配合畫在白板上,你在「分散式系統穩定性」的印象分會立刻拉滿。

---

## 先認識敵人:級聯失敗(cascading failure)🔬

所有韌性手段都是為了對付同一個敵人。先把敵人講清,六件套才不是死記。

**場景**:你的「訂單服務」每次下單要呼叫「支付服務」。某天支付服務**不是掛掉,而是變慢**——回應從 50ms 變成 5s。

關鍵在底層發生了什麼:

```
正常時:                              支付變慢後(沒設超時):
請求進來 → 拿一個執行緒 → 呼叫支付      請求進來 → 拿一個執行緒 → 呼叫支付
50ms 拿到結果 → 還執行緒               執行緒卡在這裡等 5s,不還!
                                      ↓ 請求持續湧入,每個都卡 5s
執行緒池: [空閒][空閒][空閒]           執行緒池: [卡住][卡住][卡住]...全滿
                                      ↓ 新請求拿不到執行緒
                                      你自己的訂單服務開始拒絕/超時
                                      ↓ 呼叫訂單服務的「上游」也被拖住
                                      ↓ 一路往上,整條鏈雪崩
```

**這就是級聯失敗**:一個下游的故障,順著「呼叫方等待 → 資源被占住 → 自己也耗盡 → 再拖垮上游」一路擴散。

> **最反直覺、也最該記住的一句:殺死你的往往不是「下游掛了」,而是「下游變慢」。**
> 下游**乾脆掛掉**(連線立刻被拒)反而好處理——你瞬間知道、瞬間放手。**慢**才是隱形殺手,因為它**占著你的執行緒/連線不放**,把你的資源一點點吸乾。下面六件套,本質上全是在回答同一個問題:**「當下游變慢或掛掉,我怎麼不讓我的資源被它拖死、怎麼不被它補刀拖垮?」**

---

## ① 超時 timeout:所有韌性的地基

沒有超時 = 無限等待 = 資源永遠不釋放 = 級聯失敗的引信。**這是六件套裡最基礎、也最常被漏掉的一件。**

### 兩種超時要分清

```
你 ──①連線超時──► [建立 TCP 連線]  ──②讀取超時──► [等對方回資料]
   connect timeout                    read / socket timeout
   「撥號撥不通」的上限               「接通了但對方一直不開口」的上限
```

- **連線超時**:建 TCP 連線的等待上限(對方機器宕了、網路不通 → 撥不通)。
- **讀取超時**:連上了、請求發出去了,等回應的上限(對方活著但處理很慢)。
- 兩個都要設。只設一個是常見坑——很多「服務 hang 住」是漏了讀取超時。

### 超時值怎麼定?用「鏈路預算」反推,別拍腦袋 🔬

面試官問「你超時設多少?」答「3 秒」是普通人,答「**看鏈路預算**」是資深。

```
閘道(預算 3s)
  └ 訂單服務(必須 < 3s,留給自己邏輯 0.5s → 給下游 2.5s)
      ├ 庫存服務 超時設 800ms
      └ 支付服務 超時設 1.2s        ← 800ms + 1.2s + 餘量 < 2.5s ✓
```

**鐵律:上游的超時,必須 > 它所有下游超時之和。** 否則會出現最蠢的浪費:下游其實 1.2s 能回,但你上游只給了 1s,你**還沒等到結果就自己超時放棄了**——下游白做工,你也失敗。從最外層的使用者可接受延遲(P99)往內一層層切預算,才是對的。

### Java/Go 錨點
- **Go**:`ctx, cancel := context.WithTimeout(ctx, 1200*time.Millisecond)`,把 `ctx` 一路傳給下游呼叫,到點自動 cancel。這是 Go 控制超時的慣用法。
- **Java**:`HttpClient` 的 `connectTimeout` / 請求的 `timeout`;用 Feign/RestTemplate 要分別設連線與讀取超時;`Future.get(timeout)`。

## ② 重試 retry:別把小故障放大成大故障

下游偶發抖動(一次網路丟包、一次 GC 停頓)時,重試一下確實能救。但**重試是把雙面刃,用錯會補刀**。

### 前提:冪等性(idempotency)——先過這關才談重試

**冪等 = 同一個請求執行一次和執行 N 次,結果一樣。**

- 查詢(GET)、刪除(DELETE 同一個 id)天生冪等 → 可放心重試。
- **「扣款」「下單」這種寫操作不冪等** → 你以為超時失敗了去重試,其實下游第一次已經成功了,結果**重複扣款**。
- 解法:寫操作要先做成冪等——靠**冪等鍵**(idempotency key,如訂單號)讓下游識別「這是同一個請求,我處理過了,直接回上次結果」。

> **面試點:** 被問「你怎麼做重試」,先反問/先講「**前提是冪等**」,層次立刻不一樣。沒講冪等就談重試,等於沒答到點。

### 退避 + 抖動:避免「重試風暴」與「驚群」🔬

```
❌ 固定間隔重試:失敗→等100ms→重試→等100ms→重試...
   下游正喘不過氣,所有客戶端還用固定節奏一起捶它 = 補刀

✅ 指數退避 exponential backoff:等 100ms → 200ms → 400ms → 800ms...
   給下游越來越多喘息時間

✅ 再加抖動 jitter(隨機化):等 100±隨機 → 200±隨機...
   為什麼要隨機?否則「1000 個客戶端同時失敗、同時退避、同時重試」
   會形成一波波同步的尖峰(驚群 thundering herd),等於沒退避
```

### 三條重試紀律(面試話術)
1. **只重試可重試的錯誤**:超時、5xx、連線失敗可重試;**4xx(參數錯、無權限)重試一萬次也沒用**,別重試。
2. **限次 + 退避 + 抖動**:不是無限重試。
3. **重試要配熔斷**:下游真的掛了(不是偶發),就該停止重試、快速失敗——這就引出下一件。

## ③ 熔斷 circuit breaker:快速失敗,給下游喘息 🔬

類比家裡的**斷路器/保險絲**:電流過載時自動跳閘,保護整個電路,而不是讓電線燒起來。

### 核心:一個三態機(白板必畫)

```
        失敗率超過閾值(例:10s 內 >50% 失敗)
   ┌──────────────────────────────────────────┐
   ▼                                           │
[CLOSED 閉合]                              [OPEN 打開]
 正常放行,                                直接「快速失敗」,
 統計成功/失敗率        ◄──────┐           根本不打下游!
   ▲                          │            (省掉等超時的 5s)
   │ 試探成功                  │ 試探失敗      │
   │                          │              │ 過了冷卻時間(例:30s)
   │                  [HALF-OPEN 半開]◄───────┘
   └──────────────────  放幾個請求去試探下游活了沒
```

- **CLOSED(閉合/正常)**:請求正常通過,後台統計失敗率。
- **失敗率超閾值 → OPEN(打開)**:後續請求**直接秒拒**(快速失敗 / 走 fallback),**完全不打下游**。
- **冷卻一段時間 → HALF-OPEN(半開)**:放幾個「試探請求」過去。成功 → 回 CLOSED;還失敗 → 回 OPEN 繼續冷卻。

### 為什麼這招關鍵?
下游掛了的時候,與其讓**每個請求**都傻等 5s 超時(浪費 5s + 占住執行緒 + 給下游補刀),不如**直接秒拒**:
- **保護自己**:不再有請求卡在等待裡,執行緒立刻釋放,不會被拖到級聯失敗。
- **保護下游**:不再有流量砸過去,給它喘息與恢復的空間(否則它剛要爬起來又被流量打趴)。

### Java/Go 錨點
- **Java**:**Resilience4j**(現在的主流;Netflix Hystrix 已停止維護,面試提到 Hystrix 要知道「它是經典但已 EOL」)、阿里 **Sentinel**(熔斷 + 限流一體,中文圈常見)。
- 配置三要素:**滑動窗口**怎麼統計(計數 or 時間)、**失敗率閾值**、**OPEN 持續時間**。

## ④ 隔離 bulkhead:別讓一個爛依賴拖垮全部 🔬

**Bulkhead = 船的艙壁**:船分成多個密封艙,一個艙進水,不會淹沉整艘船。

### 問題:共用資源 = 故障會擴散
你的服務同時呼叫 A、B、C 三個下游,如果它們**共用同一個執行緒池**:

```
共用一個池(20 執行緒):              每個下游一個獨立池:
C 掛了,呼叫 C 的請求全卡住,          [A 池:8][B 池:8][C 池:4]
慢慢把 20 個執行緒占滿               C 掛了只占滿「C 池」的 4 個,
↓                                   A、B 各自的池不受影響,照常服務
連呼叫 A、B 的請求也沒執行緒用了
= C 的故障擴散到了 A 和 B
```

**解法:給每個下游獨立的資源池**(執行緒池或連線池),把故障**關進它自己的艙**。C 再爛,最多占滿 C 自己的池,A、B 毫髮無傷。

### 兩種隔離,取捨不同
| | **執行緒池隔離** | **信號量隔離(semaphore)** |
|---|---|---|
| 做法 | 每個依賴一個獨立執行緒池 | 用一個計數器限制「同時呼叫數」 |
| 優點 | 能加超時**中斷**、能異步;隔離最徹底 | 輕量,幾乎無額外開銷 |
| 缺點 | 有執行緒切換 + 排隊開銷 | **不能中斷**正在執行的呼叫 |
| 何時用 | 下游慢、不可控(外部 API) | 下游快、只想限並發 |

## ⑤ 限流 rate limiting:保護自己不被打死 🔬

前四件防的是「**被慢下游拖死**」;限流防的是另一個方向——「**被上游/流量洪峰打死**」(秒殺、爬蟲、重試風暴、突發熱點)。

### 四種演算法(令牌桶最常用)

```
固定窗口計數:每秒最多 100 個。簡單,但有「臨界突刺」——
            0.9s 來 100 個 + 1.1s 來 100 個,跨窗口邊界等於 0.2s 內放了 200 個。

滑動窗口:    把窗口切細 + 滑動統計,消掉臨界突刺。

漏桶 leaky:  請求進桶,桶底以「恆定速率」漏出。流量被徹底削平,
            但「不允許突發」——即使現在很閒,也只能恆速處理。

令牌桶 token:以恆定速率往桶裡放令牌,桶滿為止;每個請求消耗一個令牌,
  ★最常用   沒令牌就被限流。閒時令牌會攢著 → 「允許一定程度的突發」(攢的能一次用)。
```

**漏桶 vs 令牌桶 的一句話差異**:漏桶**強制恆速**(平滑優先),令牌桶**允許突發**(靈活優先)。大多數業務要容忍合理突發,所以**令牌桶用得最多**。

### 單機限流 vs 分散式限流 🔬
- **單機**:本地令牌桶就行(Java 的 Guava `RateLimiter`、Go 的 `golang.org/x/time/rate`)。但你有 10 個實例,每個限 100 → 全局其實放了 1000,不準。
- **分散式**:要讓所有實例**共享一份配額**。常見做法:**Redis + Lua 腳本**(Lua 保證「讀計數→判斷→扣減」原子執行,不會併發超賣),或直接用 **Sentinel** 這類組件。
- **限流放哪層**:閘道層做粗粒度全局限流(擋洪峰),應用層做細粒度(按使用者/介面)限流。

## ⑥ 降級 degradation:保核心,棄枝節

當系統真的扛不住,**主動關掉非核心功能,把資源讓給核心鏈路**——這叫「有損服務」:寧可少做點事,也不要整個垮掉。

```
雙 11 流量洪峰,系統快撐不住:
  關掉 →「商品評價」「歷史訂單查詢」「猜你喜歡」(非核心,可損)
  保住 →「瀏覽商品 + 下單 + 支付」(核心鏈路,不能損)
```

### 主動 vs 被動
- **主動降級**:靠**開關**(feature flag / 配置中心動態推送),人為或自動關掉某功能。預案式,雙 11 前就配好。
- **被動降級 / fallback**:熔斷打開後,呼叫不再打下游,而是走一個**兜底邏輯**——回**預設值**、回**快取舊資料**、或回**友善提示**(「系統繁忙,稍後再試」)。降級和熔斷常常是搭配出現的。

## 把六件套串起來:面試滿分答法 🔬

被問「**你依賴的服務掛了/變慢,你怎麼辦?**」——別只說一個詞。按下面這條「一個請求的生命線」講,層層遞進,就是資深答案:

```
請求要呼叫一個變慢/掛掉的下游:

①超時   到點就放手,不無限等          ← 地基,先保證資源會釋放
②隔離   它在自己的艙裡,波及不到別的下游 ← 防故障橫向擴散
③熔斷   連續失敗就跳閘,後續請求秒拒    ← 不再傻等、不再補刀
④降級   熔斷後走 fallback,回兜底/快取   ← 核心功能仍有(有損)結果
⑤重試   僅對偶發抖動 + 冪等 + 退避抖動   ← 救「真·偶發」,不是救「真·掛了」
⑥限流   同時守住自己入口,別被洪峰打死  ← 另一個方向的防護
```

**一句話收束(背起來)**:

> **超時兜底、隔離防擴散、熔斷防雪崩、降級保核心、重試救偶發、限流保自己。**
> 六者不是並列選一個,而是**協同**:超時是地基,熔斷+隔離防止級聯,降級給熔斷一個體面的退路,重試只在冪等+偶發時補位,限流守住另一道門。

## 回扣可觀測性:這些開關都要「看得見」

韌性手段全都要配監控,否則你不知道它有沒有在生效(見 `observability/`):
- **熔斷器狀態**(CLOSED/OPEN/HALF-OPEN)要打成 metric + 告警——熔斷器跳開往往是故障的第一個信號。
- **限流拒絕數、降級開關狀態**要可觀測——不然「為什麼這個功能沒反應」會查到天亮。
- 重試次數、超時率,都是判斷下游健康的領先指標。

## 本章小結

- **敵人是級聯失敗**:殺死你的常常不是「下游掛了」,而是「下游變慢」占住你的資源不放。
- **① 超時**是地基:連線超時 + 讀取超時都要設;超時值靠**鏈路預算**反推(上游 > 下游之和)。
- **② 重試**前提是**冪等**;要退避 + 抖動避免重試風暴/驚群;只重試 5xx/超時,不重試 4xx。
- **③ 熔斷**是三態機(CLOSED→OPEN→HALF-OPEN):快速失敗,既保自己執行緒、又給下游喘息。Java 用 Resilience4j / Sentinel。
- **④ 隔離**(艙壁):每個下游獨立資源池,把故障關進自己的艙;執行緒池 vs 信號量各有取捨。
- **⑤ 限流**:令牌桶最常用(允許突發);分散式限流靠 Redis+Lua 共享配額。
- **⑥ 降級**:有損服務,保核心棄枝節;主動(開關)vs 被動(熔斷 fallback)。
- 下一章:撐住流量的另一半——**橫向擴展**(為什麼無狀態才能水平擴展、分庫分表的取捨)。
