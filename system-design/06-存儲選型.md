# 第 2 章 · 存儲選型:面對需求該選哪種資料庫(L2-a)

> 🔬 「設計一個 X 系統」面試的**第二大菜**(第一是 L1 服務通信)。給你一個需求——該用 MySQL 還是 MongoDB?要不要上 ES?Redis 能當主存嗎?數據量爆了怎麼辦?——答不出**選型理由**,就只能背「聽說 X 快」。這章給你一棵**存儲選型決策樹** + 各類存儲的本質/適合/代價/跨生態對照。**這跟我們做的「Redis 該不該用」(`redis-handson/00-when-to-use`)是同一招,放大到「全部存儲」。**

---

## Part A · 先定性:選型看的是「數據 × 訪問」的幾個維度

選型不是背產品,是看你的**數據和訪問模式**落在哪幾個維度上:

| 維度 | 問自己 |
|---|---|
| **數據模型** | 結構化(表/關係)?半結構化(文檔)?KV?寬列?圖?時序?全文? |
| **一致性** | 強一致(錢)?還是能最終一致(社交/統計)?(回扣 L0) |
| **訪問模式** | 讀多寫少還是寫多?點查 / 範圍 / join / 聚合 / 全文? |
| **規模** | 數據量多大?QPS?單機放得下嗎?增長多快? |

> **一句話:選型 = 把〔數據模型 × 訪問模式 × 一致性 × 規模〕映射到最合適的存儲。** 下面逐類給你「它為何這樣設計 → 適合什麼 → 什麼時候別用」。

---

## Part B · 各類存儲:本質 / 適合 / 別用 / 代表 🔬

### ① 關係型 SQL(RDBMS)—— 默認選擇
- **本質**:表 + 關係 + **事務(ACID)** + SQL;B+ 樹索引。
- **適合**:強一致、要事務、複雜查詢/join、schema 明確。**沒有特別理由,就用它**——別為了「顯得高級」上 NoSQL。
- **別用**:超大規模水平擴展難(要分庫分表,見 L2-b)、schema 極頻繁變、純非結構化。
- **代表**:**MySQL**(互聯網主流)、**PostgreSQL**(功能強,近年新項目首選)。

### ② 文檔型 NoSQL
- **本質**:存 JSON/BSON 文檔,schema 靈活,弱 join。
- **適合**:schema 多變、層級數據(一個對象一次取出)、快速迭代。
- **別用**:複雜跨文檔 join、強事務(MongoDB 雖支持事務,但不是強項)。
- **代表**:**MongoDB**。

### ③ KV
- **本質**:key→value,O(1) 點查,極簡極快。
- **適合**:緩存、會話、簡單高速點查。
- **別用**:複雜查詢、範圍掃描。
- **代表**:**Redis**(內存,深挖見 `redis-handson`)——它的「該不該用」就是這一層的活樣本。

### ④ 寬列 / 列族(Wide-column)
- **本質**:按列族存,**LSM 樹(寫優化)**,為**海量寫 + 線性水平擴展**而生。
- **適合**:海量時序/日誌/寫多、要線性擴展、能接受最終一致。
- **別用**:複雜查詢、強一致事務。
- **代表**:**Cassandra**(AP、無主、quorum——回扣 L0 `03`)、**HBase**(CP)、ScyllaDB。

### ⑤ 搜索引擎
- **本質**:**倒排索引**,為全文檢索 + 複雜聚合而生。
- **適合**:全文搜索、多條件過濾、日誌分析、聚合統計。
- **別用**:**當主庫**(不保證強一致/事務)、高頻精確更新。
- **代表**:**Elasticsearch**(深挖見 `elasticsearch/`)、OpenSearch。
- **關鍵姿勢** 🔬:ES 常作「**派生視圖**」——主數據在 MySQL,通過 CDC/雙寫把數據同步進 ES 供搜索(回扣 L2-b + L1 事件驅動)。

### ⑥ 時序資料庫(TSDB)
- **本質**:為「時間戳 + 指標」優化(超高寫入、按時間範圍查、降採樣)。
- **適合**:監控指標、IoT、metrics。
- **代表**:**Prometheus**(深挖見 `observability/`)、InfluxDB、TimescaleDB。

### ⑦ 圖資料庫
- **本質**:節點 + 邊,為「**關係多跳遍歷**」優化。
- **適合**:社交關係、推薦、風控、知識圖譜(「朋友的朋友買過什麼」這種多跳查詢)。
- **別用**:普通業務(殺雞用牛刀)。
- **代表**:**Neo4j**、NebulaGraph。

### ⑧ NewSQL / 分布式資料庫
- **本質**:「**既要 SQL + 事務,又要水平擴展**」——分布式 + 強一致,底層常用 **Raft/Paxos**(回扣 L0 共識!)。
- **適合**:數據量大到單機 MySQL 扛不住,**但又不想要分庫分表的痛、還要事務和 SQL**。
- **代價**:架構複雜、有分布式事務延遲。
- **代表**:**TiDB**(MySQL 兼容,Go,Raft)、**CockroachDB**(PG 兼容,Go,**用 HLC**——回扣 L0 `02`)、**OceanBase**(螞蟻)、Google **Spanner**(**TrueTime**——回扣 L0)。

## Part C · 存儲選型決策樹 🔬

把上面收斂成一棵能上白板的樹:

```
需求來了
  ├─ 要強事務 + 複雜查詢 + 中等規模? ──► SQL(MySQL/PG)← 默認,先想它
  │     └─ 但數據量爆了、還要事務和 SQL? ──► NewSQL(TiDB/CockroachDB)
  ├─ 要極速點查 / 緩存? ───────────────► KV(Redis)
  ├─ schema 多變 / 層級文檔? ──────────► 文檔(MongoDB)
  ├─ 海量寫 / 時序 / 日誌 + 線性擴展 + 能最終一致? ─► 寬列(Cassandra)
  ├─ 全文搜索 / 複雜聚合? ─────────────► 搜索(ES,作派生視圖)
  ├─ 監控指標? ───────────────────────► 時序(Prometheus)
  └─ 多跳關係(社交/風控/推薦)? ───────► 圖(Neo4j)
```

> 口訣:**「先問能不能用 MySQL,不行再往外跳。」** 大多數業務 SQL 就夠;跳出去一定要有明確理由(規模/模型/訪問模式對不上)。

## Part D · 多存儲是常態:Polyglot Persistence 🔬

**現實**:大系統從來不是「選一個」,而是「**每種數據用最合適的存儲**」(polyglot persistence,多語言持久化)。

```
一個電商系統:
  訂單/賬戶 → MySQL(事務、強一致)
  商品搜索   → Elasticsearch(全文 + 聚合)
  購物車/緩存 → Redis(KV、快)
  監控指標   → Prometheus(時序)
  推薦關係   → Neo4j(圖)
```

**最關鍵的架構原則** 🔬:**一個真相源(source of truth)+ 多個派生視圖**。
- 主數據放**能保事務的真相源**(通常 MySQL);ES/Redis 是**派生的、可重建的**視圖。
- 回扣 Redis 選型那句:「**不能從真相源重建的東西,別只放在派生存儲裡。**」
- 派生視圖怎麼跟主數據同步?CDC / 雙寫 / 事件驅動(→ L2-b + L1 `04`)。

**代價(架構師要權衡)**:多存儲 = 多運維 + 派生數據的延遲/不一致挑戰。**「用對工具」和「少一個依賴」要放天平上**——不是存儲種類越多越高級。

## 本章小結

- **選型 = 〔數據模型 × 訪問模式 × 一致性 × 規模〕→ 最合適的存儲**,不是背產品。
- **SQL 是默認**:強事務 + 複雜查詢 + 中等規模先想 MySQL/PG;**跳出去要有明確理由**。
- 各類本質一句話:文檔(schema 靈活)、KV(極速點查)、寬列(海量寫+線性擴展,LSM)、搜索(倒排索引,作派生視圖)、時序(指標)、圖(多跳關係)、**NewSQL(SQL+事務+水平擴展,底層靠共識)**。
- **多存儲是常態(polyglot)**:一個**真相源** + 多個**可重建的派生視圖**;同步靠 CDC/事件;代價是運維與一致性,別為「高級」堆存儲。
- 大量回扣 L0:Cassandra 的 quorum、NewSQL 的共識、Spanner 的 TrueTime、CockroachDB 的 HLC——**理論地基在這裡變現**。
- 下一章 L2-b:數據**規模化**——分庫分表、讀寫分離、遷移雙寫、CQRS。
