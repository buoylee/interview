# 第 4 章 · 可用性與容災:RPO/RTO 與多活

> 🔬 你一開始問的「**雙活資料中心、RPO<15min、RTO<30min**」就是這章。面試問「你怎麼保證高可用」「機房掛了怎麼辦」「資料會不會丟」——考的全在這。把可用性算清、把 RPO/RTO 講成**可推理的取捨**、把「單機房 → 同城雙活 → 異地多活」的階梯講出來,這塊就不虛了。繞了一圈,終於正面回答它。

---

## Part A · 可用性怎麼量:幾個 9 + 串並聯 🔬

**可用性 = 正常時間 / 總時間**,用「幾個 9」表示:

| 可用性 | 年停機 | 場景 |
|---|---|---|
| 99.9%(3 個 9) | ~8.76 小時 | 一般內部系統 |
| 99.99%(4 個 9) | ~52.6 分鐘 | 重要業務 |
| 99.999%(5 個 9) | ~5.26 分鐘 | 核心金融/電信 |

**串聯 vs 並聯(這是關鍵直覺)** 🔬:
```
串聯(依賴鏈):A → B → C,總可用 = 各自相乘
   99.9% × 99.9% × 99.9% ≈ 99.7%   ← 依賴越多越脆!(回扣 L3 韌性)

並聯(冗餘):兩個副本,只要一個活就行
   1 − (1−99.9%)² ≈ 99.9999%        ← 冗餘大幅拉高可用性
```

> **高可用的本質 = 消除單點 + 冗餘 + 快速故障轉移。** 串聯告訴你「為什麼要減少依賴」,並聯告訴你「為什麼要冗餘」。

## Part B · RPO 與 RTO:容災的兩個核心指標 🔬

```
                    ⚡災難發生
   ──────●─────────────│─────────────●──────►  時間
        最近一次可用備份              服務恢復
        │◄──── RPO ────►│◄──── RTO ────►│
         能容忍丟多少數據    能容忍多久恢復
```

- **RPO(Recovery Point Objective)= 能容忍丟多少數據**(時間度量)。RPO<15min = 災難時最多丟 15 分鐘數據。**取決於複製/備份頻率**;RPO=0 必須**同步複製**(回扣 L0 `03`、L2-b)。
- **RTO(Recovery Time Objective)= 能容忍多久恢復服務**。RTO<30min = 災難後 30 分鐘內必須恢復。**取決於故障轉移的速度與自動化**。

> **架構師的活,就是按業務定這兩個數,再反推方案**:RPO 越小越貴(同步複製→寫延遲),RTO 越小越貴(熱備/自動切換→成本)。「轉帳系統 RPO=0」和「日誌系統 RPO=1 天」對應的錢差幾個數量級。

| RPO/RTO 要求 | 對應方案 | 成本 |
|---|---|---|
| 鬆(小時級) | 冷備(定期備份,災難時恢復) | 低 |
| 中(分鐘級) | 溫備(備用環境待命,異步複製) | 中 |
| 嚴(秒級) | 熱備 / 雙活(實時複製,自動切換) | 高 |

## Part C · 高可用的階梯:單機 → 雙活 → 多活 🔬

| 級別 | 能力 | RPO/RTO | 難點 / 代價 |
|---|---|---|---|
| **單機** | 無容災,掛了就完 | 災難=數據全丟 | — |
| **主從 + 故障轉移** | 主掛從上(回扣 L0 故障檢測+quorum 防腦裂) | RTO 看切換速度;RPO 看同步/異步 | 主從延遲、切換要防腦裂 |
| **同城雙活** | 兩機房都活、都扛流量 | 同城延遲<2ms → 可同步複製 → **RPO≈0** | 機房級容災,但防不了城市級災難 |
| **異地災備(兩地三中心)** | 異地多一份備份,防區域災難(地震) | 異地延遲大 → 異步 → **RPO>0** | 備份平時不對外,利用率低 |
| **異地多活(終極)** | 多個異地機房**都對外服務** | 接近 0,但最複雜 | **跨地域一致性**(回扣 CAP)+ 單元化 |

> **「雙活」難的不是部署兩套,是數據雙向同步的一致性** 🔬:同城雙活靠低延遲還能同步複製;**異地多活**因為跨地域延遲大,只能異步 + 最終一致 + **單元化(unitization,把用戶分片到就近單元、避免跨地域寫衝突)**+ 衝突合併(回扣 L0 向量時鐘/CRDT)。這就是為什麼異地多活是頂級難題。

---

## Part D · 故障轉移、腦裂與演練 🔬

- **故障轉移 failover**:健康檢查觸發(回扣 L0/L1),自動 vs 手動切換。切換瞬間最容易**腦裂**(新舊主同時對外)。
- **防腦裂**:靠 **quorum / 仲裁節點 / fencing**(隔離舊主)——回扣 L0 `00/03`「少數派湊不齊 quorum 就退位」。
- **容災不演練 = 沒有容災** 🔬:備用環境**沒真切過,永遠不知道能不能切**。要定期做**災備演練 / 切換演練 / 混沌工程**(回扣 `perf/13-sre`)。很多「有雙活」的公司真出事才發現切不過去。

---

## 交叉引用

- **同步/異步複製 → RPO** → L0 `03` + L2-b `07`
- **故障轉移 / 腦裂 / quorum** → L0 `00`/`03`
- **依賴串聯降低可用性** → L3 `01-韌性`
- **異地多活的一致性 / 衝突合併** → L0 `02`(向量時鐘/CRDT)+ CAP `00`
- **災備演練 / 混沌工程** → `perf/13-sre`(深礦)

---

## 本章小結

- **可用性用「幾個 9」量**:串聯(依賴)相乘越串越低 → 減依賴;並聯(冗餘)大幅提升 → 消單點。
- **RPO = 能丟多少數據**(看複製頻率,RPO=0 要同步);**RTO = 多久恢復**(看切換速度)。**越小越貴,架構師按業務定值反推方案。**
- **高可用階梯**:單機 → 主從故障轉移 → 同城雙活(RPO≈0)→ 異地災備 → 異地多活(終極,難在跨地域一致性 + 單元化)。
- **故障轉移要防腦裂**(quorum/fencing);**容災必須演練**,沒切過的備用等於沒有。
- 下一章 L6:**發布與變更管理**(藍綠/金絲雀/灰度/回滾)。
