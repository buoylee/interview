# 第 0 章 · 分布式理論基礎:CAP 與共識(Raft)

> 🔬 這是分布式面試的**地基題**。CAP 幾乎必問;共識(Raft)是 ZooKeeper / etcd / Kafka / TiKV 這類中間件的內核。最常見的翻車是:講不清 P 到底是什麼、誤以為「CA 也是個選項」、或把「最終一致」跟「CAP 的 C」混為一談——一張口就暴露是背的。這章把地基夯實到能上白板、能被追問。
>
> 這也是整條 L0–L9 主線的第 0 層:上面所有「一致性 / 容災 / 複製」的取捨,根都在這裡。

# Part A · CAP:不是「三選二」

## 先把三個詞的定義釘死(多數人栽在定義不準)

| 字母 | 精確定義 | 最常見的誤解 |
|---|---|---|
| **C** Consistency | **線性一致性**(linearizability):任何一次讀,都能讀到「最近一次寫」的結果,全局看起來像只有一份數據 | ❌ 不是 ACID 的 C!(見下方陷阱) |
| **A** Availability | 每個**打到非故障節點**的請求,都能拿到**非錯誤**響應(但不保證是最新的) | ❌ 不是「系統在線/沒宕機」那種泛泛的可用 |
| **P** Partition tolerance | 網路分區(節點間消息丟失/延遲/斷開)發生時,系統**仍能繼續運行** | ❌ 不是「能不能容忍機器掛」,是「能不能容忍**網路斷開**」 |

## 核心洞察:P 是必選項,真正的取捨是「分區時選 C 還是 A」🔬

CAP 最大的誤導,就是「三選二」這個說法。真相是:

> **在分布式系統裡,網路分區不是一個選項,是一個必然**——網路一定會抖、會斷、會延遲。既然 P 躲不掉,你就**不能放棄 P**。

所以 CAP 真正的取捨只發生在**分區那一刻**:

```
                  網路分區發生了!(節點 A、B 互相通不到)
                            │
              ┌─────────────┴─────────────┐
        選 C(一致性)                 選 A(可用性)
              │                            │
   少數派那邊「拒絕服務」              兩邊都繼續讀寫
   (寧可報錯,也不返回可能            (保證有響應,但兩邊
    過期/不一致的數據)                數據可能暫時分叉)
              │                            │
          = CP 系統                    = AP 系統
       ZK / etcd / HBase          Cassandra / Eureka / Dynamo
```

> **所以「CA 系統」基本是個偽命題**——只有「假設網路永不分區」(即單機、或非分布式)才談得上 CA。**面試時若有人說「我們選 CA」,這是危險信號**,正確說法是「我們是 CP / AP」。

## CP vs AP:具體對照

| | **CP**(分區時保一致、犧牲可用) | **AP**(分區時保可用、犧牲強一致) |
|---|---|---|
| 分區時行為 | 少數派拒絕服務(報錯/不可寫) | 各分區照常讀寫,事後合併 |
| 一致性 | 線性一致 | 最終一致 |
| 代表 | ZooKeeper、etcd、HBase、傳統主從DB(強一致配置) | Cassandra、DynamoDB、Eureka、Redis(主從異步) |
| 適合 | 配置、選主、分布式鎖、金額(不能讀到舊值) | 購物車、點讚數、社交 feed(短暫不一致可接受) |

## PACELC:把 CAP 沒講完的補上 🔬

CAP 只回答了「**分區時**(Partition)」怎麼辦,卻沒說「**沒分區的正常時候**」呢?PACELC 補全:

```
if (P)  Partition 時 ─► 在 C 和 A 之間選       ← 這就是原本的 CAP
else(E) 正常時     ─► 在 L(延遲)和 C(一致)之間還要再選
```

正常時也有取捨:要強一致(等多數派確認,慢)還是低延遲(就近返回,可能讀到舊副本)。

| 系統 | PACELC 分類 | 白話 |
|---|---|---|
| Cassandra / Dynamo | **PA / EL** | 分區時保可用;正常時也優先低延遲 |
| HBase / BigTable | **PC / EC** | 分區時保一致;正常時也優先一致 |
| MongoDB(默認) | **PA / EC** | 分區時偏可用;正常時偏一致 |

> 能在「CAP」之後補一句 PACELC,面試印象分立刻拉開——因為它證明你知道**「一致性的代價在正常時也存在」**,不是只會背三個字母。

## 兩個必踩的面試陷阱 🔬

1. **「最終一致」屬於 CAP 的哪一邊?**
   → **AP**。最終一致 = 放棄「實時強一致(C)」、保住「可用(A)」,分區/延遲消失後各副本**最終**收斂到一致。注意:**最終一致 ≠ 不要一致性**,它要的是「最後會一致」,只是不保證「每次讀都最新」。

2. **CAP 的 C,和 ACID 的 C 是同一個嗎?**
   → **完全不同**,這是高頻陷阱:
   - **CAP-C** = 線性一致性(分布式多副本之間,讀得到最新寫)。
   - **ACID-C** = Consistency(一個事務不破壞數據庫的**完整性約束**,如外鍵、唯一、餘額非負)。
   - 一個講「多副本同不同步」,一個講「事務有沒有違反業務約束」,撞名而已。

# Part B · 共識(Consensus)與 Raft

## 共識問題是什麼:CP 系統的內核

> **共識** = 一組節點,對「**某個值 / 一系列操作的順序**」達成一致,**即使有節點宕機、消息延遲丟失**。

選主、配置下發、分布式鎖、複製日誌——所有 CP 系統的底層都是它。

## 為什麼需要共識 = 複製狀態機(replicated state machine)🔬

多副本要保持一致,最乾淨的辦法是:

```
讓每個副本,按【完全相同的順序】,執行【完全相同的操作日誌】
      ↓
副本 1:  [set x=1][set x=2][del x] ─► 狀態:x 不存在
副本 2:  [set x=1][set x=2][del x] ─► 狀態:x 不存在   ← 順序相同 → 狀態必然相同
副本 3:  [set x=1][set x=2][del x] ─► 狀態:x 不存在
```

**所以共識算法的全部工作,就是:讓所有副本對「日誌的第 N 條到底是什麼」達成一致。** 只要日誌一致,狀態自然一致。Raft / Paxos / ZAB 幹的都是這件事。

## Raft 的設計哲學:為「可理解」而生

Paxos 是理論鼻祖,但出了名的難懂、難工程化。Raft 的賣點就是**可理解性**:它把共識拆成三個能分開講的子問題——**① Leader 選舉、② 日誌複製、③ 安全性**。

### ① Leader 選舉 🔬

Raft 是**強 Leader**模型:同一時刻只有一個 Leader 處理寫,Followers 跟隨。

```
三種角色 + 任期(term,一個遞增的邏輯時鐘,每個 term 最多一個 Leader):

   Follower ──選舉超時沒收到心跳──► Candidate ──拿到多數票(quorum)──► Leader
      ▲                                  │                              │
      │                                  │ 選舉失敗/發現更高 term         │ 持續發心跳
      └──────────────────────────────────┴──────────────────────────────┘
                          (收到合法 Leader 心跳就退回 Follower)
```

- **任期 term**:每次選舉 term+1。term 充當邏輯時鐘,**誰的 term 舊誰服從**,用來識別過期的 Leader。
- **觸發**:Follower 在「選舉超時」內沒收到 Leader 心跳 → 自認 Leader 掛了 → 變 Candidate,term+1,給自己投一票,向所有人拉票。
- **當選**:拿到**多數派(N/2+1)**選票即成為 Leader,開始發心跳壓制其他人。
- **隨機選舉超時防選票瓜分(split vote)** 🔬:若所有 Follower 同時超時、同時拉票,選票會被瓜分誰也不過半 → 卡死。Raft 讓每個節點的選舉超時**隨機化**(如 150~300ms),錯開發起時間。**這正是第 1 章「重試加抖動 jitter」的同一個思想**——用隨機打散「同步驚群」。

### ② 日誌複製 🔬

```
客戶端寫請求
   │
   ▼
Leader: append 到自己日誌(尚未 commit)
   │  並行發給所有 Follower
   ├──────────► Follower 1: append + 回 ACK
   ├──────────► Follower 2: append + 回 ACK
   │
   ▼ 收到「多數派」ACK(含自己)
Leader: 標記該條為 committed → apply 到狀態機 → 回客戶端「成功」
   │  後續心跳順帶告訴 Followers:這條可以 apply 了
   ▼
所有副本最終 apply,狀態一致
```

> **commit 的定義 = 被多數派持久化。** 這一條是整個容錯的關鍵:只要多數派寫下了,就算少數派(包括原 Leader)掛掉,數據也不會丟。

### ③ 為什麼是「多數派 quorum = N/2+1」🔬

這是 Raft(以及一切 quorum 系統)的數學核心:

> **任意兩個多數派,必然有交集。**

```
5 個節點,多數派 = 3:
  舊 Leader 把某條日誌 commit 到了 {A,B,C}(3 個,多數派)
  選新 Leader 需要 {?,?,?}(任意 3 個)→ 一定碰到 A/B/C 中至少一個
  → 新 Leader 一定「見過」這條已 commit 的日誌 → 不會丟
```

- 配合 **Raft 的選舉限制**:Candidate 只有在「日誌至少和投票者一樣新」時才拿得到票 → 保證**選出的新 Leader 一定包含所有已 commit 的日誌**。這就是「安全性」。
- **容錯能力**:3 節點容忍 1 掛,5 節點容忍 2 掛(N 個容忍 ⌊(N−1)/2⌋ 個)。
- **為什麼集群通常是奇數(3/5/7)** 🔬:4 個節點的多數派也是 3,容錯也只有 1——和 3 節點一樣的容錯,卻多一台機器、還增加分區時兩邊都湊不齊/更易出問題的概率。**偶數不划算**。

## 共識天然防腦裂(split brain)🔬

> 回扣 `distribution/脑裂.md`(目前只有 6 行,這裡補上原理):

網路分區把 5 個節點切成 {3} 和 {2} 兩邊:

```
多數派那邊 {3}:  湊得齊 quorum → 能選出/保住 Leader → 繼續正常服務
少數派那邊 {2}:  湊不齊 quorum(需要 3)→ 選不出 Leader → 自動拒絕寫,降級
```

**結果:任何時刻最多只有一個分區能寫,不可能出現兩個 Leader 同時寫**——這就是 quorum 對腦裂的天然免疫。對比之下,沒有 quorum 機制的主從(如純異步複製 + 手動切換)才會腦裂出「雙主」。

## Raft vs Paxos vs ZAB(面試常問區別)

| | Paxos | **Raft** | ZAB |
|---|---|---|---|
| 定位 | 理論鼻祖 | 為可理解性重新設計 | ZooKeeper 專用 |
| Leader | (Multi-Paxos 才有) | **強 Leader** | **強 Leader** |
| 核心 | quorum + 兩階段 | 選舉 + 日誌複製 + 安全性 | 主備原子廣播 + 崩潰恢復 |
| 用在哪 | Chubby、理論 | **etcd / Consul / TiKV** | **ZooKeeper** |

> ZAB 和 Raft 高度相似(都是強 Leader + quorum + 日誌),差異在細節(ZAB 圍繞「原子廣播 + 主備崩潰恢復」設計)。更細的對比見既有的 `distribution/raft-zab-区别.md`(44 行)。

## 共識的代價 / 邊界(架構師視角)

- 每次寫都要等**多數派往返確認** → 有固有延遲,**不是越多副本越好**(副本多 → quorum 大 → 更慢)。
- 所以 **AP 系統的寫路徑根本不走強共識**(它要低延遲),靠最終一致 + 衝突合併。
- **共識 ≠ 分布式事務**:共識解決「多副本對同一份日誌達成一致」;分布式事務解決「跨多個服務/資源的原子性」。二者相關但不同,事務的深挖見 `financial-consistency/`(你已有的 ~80 檔深礦)。

## 本章小結

- **CAP 不是三選二**:P(網路分區)是必然、不能放棄;真正的取捨是**分區時選 C(CP,少數派拒服務)還是 A(AP,繼續服務但可能不一致)**。「CA 系統」是偽命題。
- **C 的兩個陷阱**:CAP-C(線性一致)≠ ACID-C(完整性約束);「最終一致」屬於 **AP**。
- **PACELC** 補全:正常時還要在**延遲 vs 一致**之間選(Cassandra=PA/EL,HBase=PC/EC)。
- **共識 = 複製狀態機**:讓所有副本按相同順序執行相同日誌,狀態自然一致。
- **Raft 三件事**:選舉(term + 隨機超時防瓜分)、日誌複製(多數派 ACK 即 commit)、安全性(選舉限制保證新 Leader 含全部已提交日誌)。
- **quorum N/2+1**:任兩個多數派必相交 → 不丟已提交數據、天然防腦裂;集群用奇數(3/5/7)。
