# 第 1 章續 · 服務治理設施(L1-b)

> 🔬 L1-a 講了服務怎麼拆、怎麼通信。但微服務一多,三個運維問題立刻冒出來:服務這麼多、實例還在動態增減,調用方**怎麼找到對方**(服務發現)?配置散在幾十個服務裡**怎麼統一改**(配置中心)?流量**怎麼分**到多個實例(負載均衡)?這章把這三件「讓微服務真正跑得起來」的基礎設施講清,最後**輕量帶過服務網格**(把這些治理下沉到基礎設施的思路;深入留給 L9 容器 track)。
>
> 框架對照一律生態平衡,雲原生/語言無關優先,Java 在主流處才用。

---

## Part A · 服務發現 Service Discovery 🔬

**問題**:服務 A 要調服務 B,但 B 有**多個實例、會擴縮容、會掛會重啟,IP 不固定**。A 怎麼拿到「**現在活著的 B 實例列表**」?

**核心 = 註冊中心(registry)**:

```
B 實例啟動 ──register──► [註冊中心] ◄──discover── A 要調用前查詢
   │  定期 heartbeat 續約      │           拿到 [B1,B2,B3] 活實例列表
   └──────────────────────────┘
   B 掛了/不續約 → 註冊中心把它摘除 → A 下次查不到它
```

### client-side vs server-side(高頻區分)🔬
| | **client-side** 客戶端發現 | **server-side** 服務端發現 |
|---|---|---|
| 誰查註冊中心 | 調用方自己查 + 自己做負載均衡 | 調用方只打一個固定入口(LB/網關),由它查並轉發 |
| 例 | Spring Cloud LoadBalancer、Ribbon(舊) | **k8s Service + kube-proxy**、Nginx |
| 優點 | 少一跳、客戶端控制力強 | 客戶端簡單、**語言無關** |
| 缺點 | 每種語言要實現一遍 | 多一跳 |

### 註冊中心該選 CP 還是 AP?——CAP 的經典實戰題 🔬(回扣 L0)
- **CP**(ZooKeeper、etcd):強一致,分區時少數派**不可用**。
- **AP**(Eureka、Nacos 的 AP 模式):分區時還能讀到**稍舊但能用**的實例列表。
- **多數觀點偏 AP**:服務發現場景下,「**拿到一個稍舊但大概率能用的列表**」比「**整個查不到、誰都調不了**」好得多。這就是 CAP 落到真實選型的一道經典題,面試能講出「服務發現我傾向 AP,因為可用性比強一致更重要」很加分。

### 跨生態對照
| 註冊中心 | 生態 | 一致性 | 一句話 |
|---|---|---|---|
| **k8s Service + CoreDNS** | 雲原生(Go) | — | **k8s 環境默認**,DNS 即發現,語言無關 |
| **Nacos** | Java/阿里 | AP/CP 可切 | 國內 Java 主流,發現+配置一體 |
| **Consul** | Go/HashiCorp | CP(可調) | 通用、跨平台、自帶健康檢查 |
| **etcd** | Go | CP | k8s 的底層存儲,也可直接做發現 |
| ZooKeeper | Java | CP | 老牌,但做發現偏重 |
| Eureka | Java/Netflix | AP | 經典但已停更 |

## Part B · 配置中心 Configuration Center 🔬

**問題**:幾十個服務,配置(功能開關、限流閾值、降級開關、資料庫連接)散在各自的配置文件裡——改一個要**重新發版**,還做不了灰度。

**解法**:配置**集中存儲 + 動態推送**(不重啟就生效)。核心能力:

- **集中管理** + **環境隔離**(dev/test/prod)
- **動態刷新**:配置變了,推送(push 或 long-poll)給服務,**運行時生效**
- **版本 + 回滾 + 灰度**:改錯能回退,能只對部分實例先生效

> 回扣 L3:`01-韌性` 裡的**降級開關、限流閾值**就是放配置中心、運行時動態推的——這就是「主動降級」能在不發版的情況下秒級生效的原因。

### 跨生態對照
| 配置中心 | 生態 | 一句話 |
|---|---|---|
| **Nacos** | Java/阿里 | 配置 + 發現一體,國內常用 |
| **Apollo** | Java/攜程 | 灰度/權限/審計強 |
| **Consul KV / etcd** | Go | 雲原生通用 KV,也存配置 |
| **k8s ConfigMap / Secret** | 雲原生 | k8s 原生;但**默認不動態推**,要配合 reloader 或掛載刷新 |

## Part C · 負載均衡 Load Balancing 🔬

**為什麼**:多個實例,要把流量**分攤**,避免某台過載、某台閒置。兩個關鍵維度:

### L4 vs L7 🔬
```
L4(傳輸層):只看 IP + 端口轉發,不看內容
   → 快、通用(任何 TCP/UDP)、盲(不懂業務)
   → LVS、F5、k8s kube-proxy

L7(應用層):看 HTTP 內容(URL/header/cookie)再轉發
   → 智能(按路徑/灰度/會話保持路由)、稍慢
   → Nginx、Envoy、API 網關
```
> 取捨:L4 快但盲,L7 智能但慢一點。**大流量常兩級**:LVS(L4)扛入口超大流量 → Nginx(L7)做精細路由。

### 客戶端 LB vs 服務端 LB
- **服務端**:流量先到 LB,LB 轉發(Nginx/LVS)。客戶端無感、語言無關。
- **客戶端**:調用方自己拿實例列表自己選(= client-side discovery)。少一跳、靈活,但侵入客戶端、每語言一套。

### 負載均衡算法(面試必問)
| 算法 | 怎麼分 | 適合 |
|---|---|---|
| 輪詢 / 加權輪詢 | 依次/按權重 | 簡單通用,實例同構 |
| 最少連接 least-conn | 給當前連接最少的 | 長連接、處理時長不均 |
| 隨機 / 加權隨機 | 擲骰子 | 簡單、大規模下也均勻 |
| **一致性哈希** | 同一來源固定打同一台 | **會話保持 / 緩存命中**(回扣 L0 `03`) |
| ip_hash | 按來源 IP | 簡單會話保持 |

> 配合**健康檢查**:LB 要主動剔除掛掉的實例(回扣 Part A 健康檢查 + L3 故障處理)。

## Part D · 服務網格 Service Mesh(輕量)🔬

**一個觀察**:前面這些治理能力——服務發現、負載均衡,加上 L3 的熔斷/重試/超時、L5 的鏈路追蹤——**每個服務、每種語言都要用 SDK 自己實現一遍**,重複、且難統一升級(升級一次要推動所有團隊改代碼)。

**Service Mesh 的核心思想**:把這些「服務治理」能力**從應用裡抽出來,下沉到一個 sidecar 代理**——一個和應用一起部署、攔截所有進出流量的小代理。**應用只管業務,治理交給 sidecar。**

```
   ┌─────────────┐        ┌─────────────┐
   │  App A       │        │  App B       │     ← 應用只寫業務
   │  + sidecar 🔁│◄──────►│  + sidecar 🔁│     ← 數據面:Envoy 代理,
   └─────────────┘        └─────────────┘        真正轉發 + 執行規則
            ▲                     ▲
            └────────┬────────────┘
                [ 控制面 Istio ]   ← 下發規則(路由/重試/熔斷/mTLS)給所有 sidecar
```

- **數據面(data plane)**:sidecar 代理(**Envoy**),真正轉發流量、執行規則。
- **控制面(control plane)**:**Istio**,把策略統一下發給所有 sidecar。
- **好處**:治理邏輯**語言無關、統一升級、應用零侵入**。
- **代價**:多一跳延遲、運維複雜、資源開銷(每個 pod 多一個代理)。

> 一句話:**mesh = 把 L1/L3 的治理能力,從「每個服務的 SDK」搬到「基礎設施的 sidecar」。** 生態上 **Istio + Envoy** 是雲原生事實標準(Linkerd 更輕);它們都跑在 k8s 上——**深入留給 L9 容器 track**。

## 本章小結

- **服務發現**:靠註冊中心(註冊+心跳+摘除);**client-side**(自查自選,語言各一套)vs **server-side**(打固定入口,語言無關,k8s 默認);**註冊中心多偏 AP**(可用 > 強一致,CAP 實戰題)。
- **配置中心**:集中存儲 + **動態推送**(不發版生效),支持灰度回滾;**降級開關/限流值就放這**。
- **負載均衡**:**L4**(快、盲、通用)vs **L7**(智能、看內容);服務端 vs 客戶端;算法裡**一致性哈希**做會話保持/緩存命中。
- **服務網格**:把治理能力**下沉到 sidecar**(數據面 Envoy + 控制面 Istio),換來語言無關/統一升級/零侵入,代價是延遲與運維複雜度。
- **L1 服務化層完整**(L1-a 通信範式 + L1-b 治理設施)。下一步 **L2 數據架構與存儲選型**。
