# 第 0 章續 · 一致性模型與邏輯時鐘

> 🔬 上一章的 CAP 把「C」當一個點(線性一致)講,但真實世界的一致性是**一條光譜**。面試問「強一致和最終一致差在哪」「什麼是因果一致」「Spanner 怎麼做全球強一致」——答不清就露怯。而要看懂這條光譜、要搞清「順序」到底指什麼,得先接受一個反直覺的底層事實:**分布式系統裡沒有「現在幾點」這種全局時鐘**。本章把「一致性光譜」和「時間順序(邏輯時鐘)」兩件事一次打通——它倆其實是一體兩面。

---

# Part A · 一致性模型:一條光譜,不是兩個點

多副本 + 並發 → 不同客戶端可能看到不同的新舊、不同的順序。**一致性模型 = 系統對「你能看到什麼」給的契約。** 越強的契約越好用,但越貴。從強到弱:

## ① 線性一致性 Linearizability(最強)🔬

- **定義**:每個操作看起來在「調用與返回之間的某個瞬間」**原子生效**;而且**尊重真實時間**——一旦某次寫返回成功,之後**所有**節點的讀都必須看到它(或更新的)。
- **直覺**:整個系統表現得像「只有一份數據 + 一個全局時鐘」。**這就是 CAP 的 C。**
- **代價**:要全局協調(共識 / quorum 往返)→ 慢。
- **例**:etcd / ZooKeeper 的線性化讀、單 key 的 Raft、`AtomicLong` 的分布式版。

## ② 順序一致性 Sequential Consistency

- **定義**:存在一個**全局總順序**,所有節點都同意;且每個進程自己的操作在這個順序裡**保持它的程序順序**。但——**不要求符合真實時間**。
- **與線性一致的關鍵差別**:
  - 線性一致 = 順序一致 **+ 尊重跨進程的真實時間順序**。
  - 順序一致允許「整體重排」,只要不打亂各進程自己的先後。
- **白話**:大家對「一個全局劇本」達成一致,但這個劇本可以和真實鐘錶對不上(你 10:00 寫的,劇本裡可能排在別人 10:01 寫的後面,只要不違反你自己的順序)。

## ③ 因果一致性 Causal Consistency 🔬

- **定義**:**有因果關係**的操作(A happens-before B),所有節點都按 A→B 看到;**沒有因果關係的並發操作**,不同節點可以看到不同順序。
- **例**:論壇回覆——你必須先看到原帖、才看到對它的回覆(因果必須保序);但兩條互不相關的帖子誰先顯示,無所謂。
- **地位**:它是「**能保住直覺正確性**的最弱實用一致性」,是很多系統的**甜點**(比最終一致安全,比線性一致便宜)。它靠的就是 Part B 的**邏輯時鐘 / 向量時鐘**來追蹤「誰因果先於誰」。

## ④ 最終一致性 Eventual Consistency(最弱常用)

- **定義**:停止寫入後,**最終**所有副本收斂到同一狀態;期間可能讀到舊值、可能亂序。
- **例**:DNS、Cassandra 默認、購物車、點讚數。
- **兩個坑**:① 「最終」**沒有時間保證**(可能 1ms 也可能 10s);② 它**不保證因果**——可能出現「看得到回覆卻看不到原帖」的怪象。所以實務常用「因果一致」來補最終一致的直覺漏洞。

## 客戶端中心的「會話保證」——最終一致系統的補丁 🔬

純最終一致太弱會出怪事(自己改了暱稱,刷新卻看到舊的)。於是有一組**針對單個客戶端**的承諾,常單獨考:

| 保證 | 承諾 | 違反的例子 |
|---|---|---|
| **Read-your-writes**(讀己之寫) | 你自己寫的,後面一定讀得到 | 改完暱稱刷新還是舊的 |
| **Monotonic reads**(單調讀) | 讀到過新值,就不會再讀回舊值 | 列表一會兒有、刷新又沒了 |
| **Monotonic writes** | 你的多次寫按發出順序生效 | 先改A後改B,結果B被A覆蓋 |

> **落地手法**:通常靠「同一會話**黏同一副本**」(sticky session),或「請求**帶上你已知的版本號**,副本不滿足就等同步」。面試能說出這兩招就到位了。

## 一張總表(強 → 弱,背下來)

| 模型 | 承諾 | 代價 | 典型 |
|---|---|---|---|
| 線性一致 | 像單機單時鐘,讀到最新 | 最貴(全局協調) | etcd/ZK 讀、Raft 單key |
| 順序一致 | 全局總序 + 各自程序序,但不管真實時間 | 貴 | 部分分布式內存模型 |
| 因果一致 | 因果保序、並發可亂 | 中(靠向量時鐘) | 協作編輯、社交 feed |
| 最終一致 | 最終收斂,期間可舊可亂 | 便宜 | DNS、Cassandra、購物車 |
| (會話保證) | 對單客戶端的局部承諾 | 低 | 黏副本 / 帶版本號 |

# Part B · 邏輯時鐘:分布式裡沒有「現在幾點」🔬

要理解上面「順序」「因果」到底怎麼定義出來的,得先承認一個底層事實。

## 物理時鐘靠不住

- 每台機器的牆上時鐘(wall clock)會**漂移**、會被 NTP **回撥**(甚至倒退)。
- 所以**用 timestamp 比較跨機器事件的先後是不可靠的**:可能 A 的時間戳比 B 大,但其實 A 先發生。
- 經典生產事故:用 `System.currentTimeMillis()` 給分布式事件 / 日誌排序 → NTP 一校準回撥,順序就亂、ID 甚至重複或倒退。

> 結論:分布式系統需要一種**不依賴物理時間、只靠「誰導致了誰」**的順序 —— 這就是邏輯時鐘。核心概念叫 **happens-before(→)**:A→B 意思是「A 可能影響了 B」。

## Lamport 邏輯時鐘(標量計數器)🔬

規則簡單到三條:

```
每個節點維護一個計數器 C:
  ① 本地發生事件      → C = C + 1
  ② 發送消息          → 帶上當前 C
  ③ 收到消息(帶 C_msg)→ C = max(C, C_msg) + 1
```

```
節點P1: e1(1) ──msg──┐        e4(4)
                     │
節點P2:        e2(1) ▼e3(2) ──msg──┐
                                   │
節點P3:                     e5(1)  ▼e6(3)
        (括號內是 Lamport 時間戳)
```

- **保證**:若 A→B(有因果),則 **C(A) < C(B)**。
- **局限(必考)**:**反過來不成立**——C(A) < C(B) **推不出** A→B,它倆可能只是**並發**、碰巧計數器有大小。Lamport 能排出一個**全序**(再用節點 id 打破平手),但**分不清「真因果」還是「碰巧」**。

## 向量時鐘 Vector Clock(能識別並發)🔬

為了補上「分不清因果 vs 並發」這個洞,把標量升級成**向量**(每個節點佔一格):

```
N 個節點,每個維護向量 V = [c1, c2, ..., cN]:
  ① 本地事件   → 自己那格 +1
  ② 發消息     → 帶整個向量
  ③ 收消息     → 逐格 max,再自己那格 +1

比較兩個向量:
  V(A) ≤ V(B) 每格都 ≤,且至少一格 <  ⟺  A → B(因果先後)
  互相都不 ≤(各有一格更大)          ⟺  A、B 並發(無因果)
```

- **威力**:**能精確判斷「因果」還是「並發」**——這正是「因果一致性」的引擎,也是 **Dynamo/Cassandra 檢測寫衝突(siblings)**、**CRDT 合併**的底層(見 `distr-tx/CRDT/`)。
- 代價:向量長度 = 節點數,節點多時開銷大(有變體優化)。

```
Lamport:  只給一個數 → A=3, B=3 → 看不出是因果還是並發
Vector :  A=[2,1,0], B=[2,0,1] → 互不 ≤ → 確定是「並發」!
```

## HLC 與 TrueTime(進階,面試能提一句就贏)🔬

- **HLC(Hybrid Logical Clock)**:把物理時間和邏輯計數器**揉在一起**——既接近真實時鐘(方便人看、方便和外部對齊),又保住因果單調。**CockroachDB** 用它。
- **TrueTime(Google Spanner)**:用 **GPS + 原子鐘**把時鐘誤差壓進一個**已知區間** [earliest, latest];寫事務提交時**主動等待這段誤差過去**(commit-wait),從而實現**全球範圍的線性一致**。**本質是用錢和硬體,把「全局時鐘」買回來了。**

> 面試金句:「分布式默認沒有全局時鐘,所以靠邏輯時鐘定因果;Spanner 是例外——它用原子鐘把物理時鐘的不確定性縮到可控,才敢做全球強一致。」

## 交叉引用

- **線性一致 = CAP 的 C** → 回扣 `00-理論基礎-CAP與共識`
- **因果一致 / 向量時鐘 → 衝突合併**(AP 系統 siblings、CRDT)→ `distr-tx/CRDT/`
- **happens-before / 邏輯時鐘** 是一切「分布式排序」的根:日誌順序、事件流、甚至 Raft 的 **term 也是一種邏輯時鐘**
- **一致性的事務級落地**(轉帳/支付怎麼保不變量)→ `financial-consistency/`(深礦)

## 本章小結

- 一致性是**光譜不是開關**,從強到弱:**線性一致 > 順序一致 > 因果一致 > 最終一致**;另有一組**會話保證**(讀己之寫 / 單調讀)是對單客戶端的局部承諾。
- **線性一致 = CAP 的 C** = 尊重真實時間;**順序一致**只保各自程序序、不管真實時間;**因果一致**只保有因果的操作,是性價比甜點;**最終一致**最便宜但不保因果。
- 分布式**沒有可靠的全局物理時鐘**(漂移 / NTP 回撥)→ 用**邏輯時鐘**定因果。
- **Lamport 標量鐘**:A→B ⟹ C(A)<C(B),但反之不成立(分不清並發);**向量鐘**:能精確區分**因果 vs 並發**,是因果一致 / 衝突檢測的引擎。
- **HLC** 揉物理+邏輯(CockroachDB);**TrueTime** 用原子鐘買回全局時鐘(Spanner 全球強一致)。
- 下一步(L0 收尾):複製模式、quorum(NWR)、一致性哈希、故障檢測 gossip。
