# 第 0 章續 · 複製、Quorum、一致性哈希與故障檢測

> 🔬 L0 理論層收尾。前兩章講「怎麼達成一致」(CAP/共識)和「能看到什麼/怎麼定順序」(一致性模型/時鐘)。這章講最後一塊:**數據怎麼鋪開到多台機器、節點怎麼知道彼此死活**——複製、Quorum(NWR)、一致性哈希、故障檢測。這四個是上層 **L4 容災(RPO/RTO)、L1 負載均衡、L2 分片** 的共同地基,補完它們,理論地基就齊了。

---

# Part A · 複製 Replication:三種拓撲

**為什麼要複製**:① 容錯(機器一定會掛)② 擴讀(讀分散到多副本)③ 降延遲(就近讀)。代價是——**多副本就有「一致性」問題**(這就是前兩章的根源)。三種拓撲:

| 模式 | 怎麼寫 | 衝突 | 代表 |
|---|---|---|---|
| **主從(single-leader)** | 只寫主,讀主或從 | 無寫衝突(只有一個寫入點) | MySQL 主從、Redis 主從 |
| **多主(multi-leader)** | 多個寫入點(常見多機房各一主) | **有寫衝突**,要合併(靠向量時鐘/CRDT,回扣 `02`) | 跨機房 DB、CouchDB |
| **無主(leaderless,Dynamo 風)** | 客戶端直接寫多個副本 | 靠 **quorum 讀寫 + 讀修復/反熵** 解決 | Cassandra、DynamoDB |

## 同步 vs 異步複製 → 直接決定 RPO 🔬(預告 L4)

主從複製,主寫完要不要等從確認?

```
同步複製:  主寫 → 等從確認 → 才回客戶端「成功」
            ✓ 從一定有數據,主掛了不丟(RPO=0)
            ✗ 慢(等一個往返),從掛了還會卡住主

異步複製:  主寫 → 立刻回「成功」 → 之後才慢慢同步給從
            ✓ 快
            ✗ 主掛且還沒同步 → 這段數據丟了(RPO>0)

半同步:    至少等「一個」從確認(折中,生產常用)
```

> 這就是 **L4 容災 RPO 的來源**:`RPO`(能容忍丟多少數據)本質就是「你選同步(RPO≈0)還是異步(RPO>0)複製」。回扣 CAP——同步複製偏 CP,異步偏 AP/低延遲。

# Part B · Quorum / NWR:把一致性做成可調旋鈕 🔬

無主複製靠 **Quorum** 保證讀到新值。三個字母:

- **N** = 副本總數　**W** = 一次寫要幾個副本確認　**R** = 一次讀要讀幾個副本

> **黃金不等式:`W + R > N` → 讀集合與寫集合必有交集 → 讀一定能讀到最新的寫(強一致讀)。**

```
N=3, W=2, R=2:  W+R=4 > 3 ✓
  寫到 {副本1,副本2};讀 {副本2,副本3} → 交集 副本2 → 讀到最新 ✓
  還能容忍 1 台掛(剩 2 台仍湊得齊 W=2/R=2)
```

調 W/R 就是調「一致性 vs 延遲/可用」(回扣 **PACELC 的 EL/EC**):

| 配置 | 效果 |
|---|---|
| W=N, R=1 | 寫慢讀快;讀強一致;但任一副本掛就不能寫 |
| W=1, R=N | 寫快讀慢;寫可用性高 |
| W=R=2, N=3 | **均衡**:容 1 故障 + 強一致(最常用) |
| W+R ≤ N | 弱一致(可能讀到舊值),換更高可用/更低延遲 |

> **與 Raft quorum 的區別**(常考):Raft 是 **leader-based 固定多數派**(N/2+1,為了線性一致);NWR 是 **leaderless 可調**(W/R 自己配,換一致性等級)。一個是「強一致的內核」,一個是「一致性旋鈕」。

# Part C · 一致性哈希 Consistent Hashing 🔬

**問題**:把數據分片到 N 台機器,最直覺的 `hash(key) % N`——**一旦加/減一台,N 變了,幾乎所有 key 的映射全變** → 緩存全失效、數據幾乎全遷移。災難(回扣 Redis Cluster reshard、分庫分表擴容)。

**解法**:把哈希空間做成一個**環**(0 ~ 2³²−1),節點和 key 都映射到環上,**key 順時針找到的第一個節點**就是它的歸屬。

```
        0/2³²
          │
   節點C ─┤        ●key1 → 順時針撞到 → 節點A
          │      ╱
   ───────┼────●─────── 節點A
          │   key1
   節點B ─┤
          │
  加一台「節點D」插在 A 和 B 之間:
  → 只有「原本屬於 A、現在落在 D 區間」的那一小段 key 要遷移
  → 其他 key 全不動!遷移量 O(K/N) 而非全部
```

- **虛擬節點(virtual nodes)** 🔬:每個物理節點在環上放多個「分身」→ ① 數據更均勻(避免傾斜)② 支持異構(強機器多放幾個分身)。
- **用在哪**:Cassandra、一致性哈希負載均衡、CDN 選源站。
- **Redis Cluster 是變體,要講清** 🔬:它**不是**經典一致性哈希,而是 **16384 個固定 hash slot**(`CRC16(key) % 16384`),節點認領 slot,遷移**以 slot 為單位**。好處是 slot 數固定、映射可控、遷移粒度清晰(見 `redis-handson/11-cluster`)。

# Part D · 故障檢測 Failure Detection 🔬

> 回扣 `distribution/脑裂.md` 空殼,也回扣第 1 章「慢比死更難處理」。

**問題**:節點 A 怎麼知道節點 B 死了?分布式裡最難的是——**「真死」和「只是慢/網路抖」分不清**。判錯了:把活的當死的 → 誤切主 → 腦裂;把死的當活的 → 一直往黑洞發請求。

| 機制 | 怎麼做 | 取捨 |
|---|---|---|
| **心跳 heartbeat** | 固定週期 ping,超時判死 | 最簡單;但閾值難定(太短誤判、太長發現慢) |
| **Gossip 協議** 🔬 | 每個節點隨機找幾個 peer 交換「我所知的成員存活表」,像八卦擴散 → 最終全集群收斂 | 去中心、可擴展、抗單點;用於 Cassandra/Consul/Redis Cluster |
| **phi-accrual** 🔬 | 不給「死/活」二元,給一個**「懷疑度」分數** φ,由上層自己定閾值 | 更魯棒(自適應網路抖動);Cassandra/Akka 用 |

> **核心洞察**:故障檢測天生不可靠(慢≈死),所以**不能靠「檢測準」來防腦裂,要靠 quorum**(回扣 `00`:少數派湊不齊 quorum 就自動退位)。檢測只負責「觸發懷疑」,**最終誰能服務由 quorum 拍板**。

## 本章小結

- **複製三拓撲**:主從(無寫衝突,最常見)、多主(跨機房,有衝突要合併)、無主(Dynamo,靠 quorum)。
- **同步 vs 異步複製直接決定 RPO**:同步 RPO≈0 但慢、異步快但可能丟——這是 L4 容災的物理根源。
- **Quorum:`W+R>N` ⟹ 強一致讀**;調 W/R = 調一致性 vs 延遲(PACELC 旋鈕)。N=3/W=2/R=2 是均衡甜點。
- **一致性哈希**:哈希環讓加減節點只遷移 O(K/N) 數據,虛擬節點解決傾斜;**Redis Cluster 是 16384 slot 的變體,不是經典一致性哈希**。
- **故障檢測**:心跳(簡單)、Gossip(去中心可擴展)、phi-accrual(懷疑度分數);**「慢≈死」不可靠 → 防腦裂最終靠 quorum 不靠檢測**。
- **L0 理論層到此完整**(CAP/共識 → 一致性模型/時鐘 → 複製/quorum/哈希/檢測)。下一步往上走 **L1 服務化與通信**。
