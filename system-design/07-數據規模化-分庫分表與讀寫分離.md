# 第 2 章續 · 數據規模化:分庫分表、讀寫分離、遷移雙寫(L2-b)

> 🔬 L2-a 選好了存儲,但當 MySQL 單庫**扛不住了**怎麼辦?這章講「數據長大了怎麼擴」:讀壓力大 → 讀寫分離;寫/容量到頂 → 分庫分表;換存儲/上分片 → 遷移雙寫;讀寫需求差異大 → CQRS(輕)。這些是面試「**你們資料庫怎麼扛住的 / 億級數據怎麼存**」的標準答案。
>
> 一條主線:**這些都是「不得已」的手段,複雜度都很高——能不用就別用,先窮盡簡單方案。**

---

## Part A · 讀寫分離 Read-Write Split 🔬

**問題**:讀遠多於寫(電商常見 10:1 以上),單庫讀壓力先到頂。

**解法**:**主庫寫,多個從庫讀**(基於主從複製,回扣 L0 `03`)。讀分攤到從庫,主庫只扛寫。

```
       寫 ──► [主庫] ──複製──► [從庫1] ◄─讀
                  └──複製──► [從庫2] ◄─讀
```

**致命坑:主從延遲(replication lag)** 🔬:
- 寫完主庫**立刻讀從庫**,從庫可能還沒同步到 → **讀到舊值**。這正是 L0 講的 **read-your-writes 被破壞**。
- 三種解法:① **強制讀主**(剛寫完的這次讀走主庫)② 等從庫追上再讀 ③ 用緩存兜這一下。

> 讀寫分離只解決**讀**壓力,**不解決寫瓶頸和容量瓶頸**——那要靠分庫分表。

---

## Part B · 分庫分表 Sharding 🔬(核心)

**為什麼**:單表數據量太大(億級)→ 索引變深、查詢變慢;單庫的寫入/連接數/磁碟容量到頂。

### 垂直 vs 水平 🔬
```
垂直分庫:按業務拆庫(訂單庫 / 用戶庫 / 商品庫)
垂直分表:把大表的冷字段、大字段(text)拆到另一張表
水平分庫分表:同一張表的數據,按規則散到多個庫/表
            user 表 → user_0, user_1, ... user_15   ← 「分庫分表」通常就指這個
```

### 分片鍵 Sharding Key —— 最關鍵的決策 🔬
**按什麼字段分?** 選錯代價極大:
- **數據傾斜**:分布不均 → 熱點全壓一個分片(等於沒分)。
- **跨分片查詢**:查詢條件不是分片鍵 → 要掃**所有**分片再聚合,慢且複雜。
- **好的分片鍵**:高基數、分布均勻、**是最常用的查詢條件**(如電商按 `user_id` 分,因為大多查詢帶 user_id)。

**分片算法**:
| 算法 | 優點 | 缺點 |
|---|---|---|
| 範圍 range | 易擴容(加範圍即可)、範圍查詢友好 | 易熱點(新數據全落最後一片) |
| 哈希 hash | 分布均勻 | 擴容要大遷移 |
| 一致性哈希 | 擴容只遷一小段(回扣 L0 `03`) | 實現複雜 |

### 分庫分表帶來的「四個新麻煩」🔬
1. **跨分片 join / 查詢**:難。對策——冗餘/反範式、或中間件聚合、或避免這類查詢。
2. **分布式事務**:跨庫寫不再是本地事務 → 要 Saga/TCC/Outbox(→ 你的深礦 `financial-consistency/`)。
3. **全局唯一 ID**:不能再用單庫自增 → **雪花算法 snowflake / 號段模式 / UUID**(回扣 `distribution/分布式id`)。
4. **擴容 / 再分片**:加機器要遷移數據(一致性哈希 / 預分片 / 翻倍擴容 緩解)。

### 工具與「免分片」選項
- **中間件**:ShardingSphere(Java,見 `Sharding-Sphere/`)、Vitess(Go,MySQL 分片)、MyCAT。
- **或者直接上 NewSQL(TiDB)**,讓資料庫自己分片,**免去手動分庫分表的痛**——回扣 L2-a「數據量爆了就上 NewSQL」。

> **關鍵洞察**:分庫分表是**最後手段**、複雜度最高。先窮盡:加索引 → 讀寫分離 → 冷數據歸檔 → 緩存 → NewSQL,**真不行再分**。面試能說出這個「升級階梯」很顯成熟。

## Part C · 數據遷移與雙寫 🔬

**場景**:換存儲(MySQL→TiDB)、分庫分表上線、加 ES 派生視圖——都要把**存量 + 增量**數據遷過去,**還不能停服**。

**平滑遷移四步(經典套路)** 🔬:
```
① 雙寫:新舊兩邊都寫,讀還讀舊      ← 保證增量不丟
② 存量遷移:後台慢慢把歷史數據搬到新庫
③ 校驗對賬:比對新舊一致(→ financial-consistency/07-reconciliation)
④ 切讀:灰度把讀切到新庫 → 穩定後下線舊庫、停雙寫
```

- **雙寫的坑** 🔬:雙寫**不是原子的**(寫舊成功、寫新失敗 → 不一致),所以**第③步對賬修復是必須的**,不能省。
- **增量同步**常用 **CDC(訂閱 binlog)** 而非應用層雙寫——更可靠、解耦(回扣 L1 `04` 事件驅動)。
- **影子表/影子庫**:遷移前用來壓測、驗證新庫扛不扛得住。

## Part D · CQRS(輕)🔬

**思想**:**Command(寫)和 Query(讀)用不同的模型/存儲**。寫模型保一致(MySQL),讀模型為查詢優化(ES/緩存/寬表),兩者**異步同步**。

```
寫 ─► MySQL(規範化、事務)──投影/CDC──► ES(供搜索)
                                  └──► Redis(供詳情頁)   ← 讀走這些
```

- **為什麼**:讀寫需求差異大時——寫要事務+規範化,讀要快+靈活+聚合。**一個模型迁就兩邊,兩邊都彆扭**。
- **代價**:讀寫**最終一致**(投影有延遲)、複雜度上升。
- **事件溯源(event sourcing)一句話**:不存「當前狀態」,而存「**所有發生過的事件**」,當前狀態由事件**回放**得出;常和 CQRS 搭配。深入(Saga/Temporal/事件流)→ `financial-consistency/05`。
- **別濫用**:簡單 CRUD 上 CQRS = 過度設計。它是「讀寫差異大」時才划算。

## 交叉引用

- **讀寫分離主從延遲 → L0 `03` 複製 + `02` read-your-writes**

- **全局 ID → `distribution/分布式id`**
- **跨庫事務 / 遷移對賬 → `financial-consistency/`(深礦)**

## 本章小結

- **讀寫分離**:主寫從讀,分攤讀壓力;**主從延遲**會破壞 read-your-writes,靠強制讀主/等待/緩存解;只解決讀。
- **分庫分表**:垂直(按業務/字段)vs 水平(同表散多庫);**分片鍵是最關鍵決策**(高基數+均勻+常用查詢條件),選錯=傾斜+跨片查詢;帶來四麻煩(跨片 join、分布式事務、全局 ID、再分片)。
- **它是最後手段**:先加索引/讀寫分離/歸檔/緩存/NewSQL,真不行才分。
- **遷移四步**:雙寫 → 存量遷移 → 對賬 → 切讀;雙寫非原子,**對賬必須**;增量用 CDC。
- **CQRS(輕)**:讀寫模型分離,寫保一致、讀求性能,異步投影;最終一致+複雜度是代價,別濫用;深入見 `financial-consistency/`。
- **L2 數據架構層完整**(選型 + 規模化)。下一步 **L4 可用性與容災**(回到你最初的心結:RPO/RTO、多活)。
