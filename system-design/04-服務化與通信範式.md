# 第 1 章 · 服務化與通信範式(L1-a)

> 🔬 這是「設計一個 X 系統」面試的**主菜**。考的是三件事:服務**怎麼拆**、服務之間**怎麼說話**(同步 RPC?異步消息?事件驅動?)、用**什麼協議/框架**。最常見的露怯是:答不清「什麼時候用 RPC、什麼時候用消息」,或一上來就堆框架名卻講不出取捨。這章給你**範式判斷**(骨架)+ **跨生態框架對照**(落地,不綁死語言)。
>
> 框架對照一律**生態平衡**:預設偏雲原生/語言無關,Java 在真行業主流處才用,並補 Go/Python 等價物。

---

## Part A · 微服務拆分:邊界比技術更難(DDD 輕量版)

**先潑冷水**:拆微服務的第一原則是「**單體扛得住就別拆**」。拆分換來獨立部署/獨立擴展/團隊自治/技術異構,代價是**一整套分布式複雜度**(網路、一致性、運維、調試)——也就是這條 L0–L9 主線的全部內容。

**怎麼劃邊界**(這是難點,不是技術問題):
- **按業務能力 / 限界上下文(bounded context,DDD)**:一個服務 = 一塊高內聚的業務領域(訂單、庫存、支付),而不是按技術分層(不要「DAO 服務」「Controller 服務」)。
- **康威定律(Conway's Law)**:系統架構會長成組織溝通結構的樣子。**反過來用**:想要什麼架構,先調整團隊邊界。
- **拆錯的信號** 🔬:
  - **分布式單體**(distributed monolith):服務間強耦合、改一個要一起發版——拿了分布式的苦、沒拿到解耦的甜,**最壞情況**。
  - 鏈式調用過長(A→B→C→D→E,一個慢全鏈崩,回扣 L3 級聯失敗)。
  - 多個服務共享同一個資料庫(偷偷耦合,沒有真正邊界)。

> 一句話:**拆分是組織 + 業務問題,披著技術外衣。** 面試講得出「先別拆、拆要按限界上下文、警惕分布式單體」就到位了。

## Part B · 通信範式:同步 RPC vs 異步消息 vs 事件驅動 🔬(核心)

這是 L1 最該講清的判斷。三種範式,本質是「**耦合度**」的光譜:

| 範式 | 怎麼說話 | 耦合 | 優點 | 代價 |
|---|---|---|---|---|
| **同步 RPC**(請求-響應) | 調用方**等**結果 | 緊(時間+空間都耦合) | 簡單、低延遲、即時拿到結果 | 下游掛=你掛(級聯失敗)、強依賴 |
| **異步消息**(MQ,點對點) | 丟進佇列就走,稍後處理 | 鬆 | **解耦、削峰、緩衝** | 最終一致、要處理丟失/重複/順序 |
| **事件驅動**(發布-訂閱/事件流) | 生產者**不知道**誰在消費 | 最鬆 | 擴展性最好、加消費者零改動 | 「事件風暴」、調試難、最終一致 |

**怎麼選(決策三問)**:
1. 調用方**需要立即拿到結果**嗎?(要 → 傾向同步 RPC)
2. 能**容忍最終一致**嗎?(能 → 可用異步)
3. 要**解耦 / 削峰 / 一份事件多方消費**嗎?(要 → 異步消息 / 事件驅動)

> **關鍵洞察** 🔬:同步是「**命令式 + 強耦合**」(我叫你做事,等你做完),異步是「**事件式 + 鬆耦合**」(我宣布發生了一件事,誰關心誰處理)。**成熟系統都是混合**:核心鏈路(下單→扣款)走同步保即時與一致,旁路(發通知、更新統計、寫搜索索引)走異步解耦削峰。
>
> 異步/事件的落地(怎麼保不丟、不重、有序)→ 你的深礦 `financial-consistency/`(outbox/事件流)+ `MQ/`。

## Part C · RPC 怎麼實現 + 跨生態框架對照 🔬

**RPC 本質**:讓「遠程調用」**看起來像本地函數調用**。底層五步:

```
①定義接口(IDL) → ②生成樁代碼 stub → ③參數序列化 → ④網路傳輸 → ⑤對端反序列化執行
   .proto 文件      client/server 樁    對象→字節      TCP/HTTP2     字節→對象
```

### 跨生態框架對照(這就是你要的 B 層)

| 框架 | 主生態 | 協議 / 序列化 | 一句話定位 |
|---|---|---|---|
| **gRPC** | **語言無關**(Go/Python/Java/C++…) | HTTP/2 + Protobuf | **雲原生默認、跨語言首選**,流式好,Go 世界的標配 |
| **Dubbo** | **Java**(有 Dubbo-go) | Triple/Dubbo + Hessian2/PB | **國內 Java 微服務主流**,服務治理(路由/權重/降級)強 |
| **Thrift** | 跨語言 | 自有二進制 | Facebook 系老牌跨語言,性能好 |
| **REST + JSON** | 語言無關 | HTTP/1.1 + JSON | 簡單通用、**對外開放 API 首選**,性能/體積一般 |
| **Kitex / Hertz** | **Go**(字節) | gRPC/Thrift | Go 高性能微服務,字節生態 |
| FastAPI(REST)| **Python** | HTTP + JSON | Python 服務常用,或直接上 gRPC |

**哪個生態主導、怎麼選** 🔬:
- **跨語言 / 雲原生 / Go 為主** → **gRPC**(HTTP/2 多路復用 + Protobuf 高效 + 原生流式)。
- **國內 Java 微服務** → **Dubbo**(治理能力是賣點;這類確實 Java 主導,照用)。
- **對外 / 第三方開放 API** → **REST**(通用、可讀、好調試)。

### 序列化對照(面試常問)
| | Protobuf / Thrift | JSON | Hessian / Avro |
|---|---|---|---|
| 體積/速度 | 小、快 | 大、慢 | 中 |
| 可讀性 | 差(二進制) | 好 | 差 |
| 需要 schema | 要(.proto) | 不要 | Avro 要 |
| 用在 | 內部高性能 RPC | 對外 API/調試 | Java/大數據 |

> 取捨一句話:**對內求性能用二進制(Protobuf),對外求通用可讀用 JSON。** 序列化性能深挖見 `perf/11-07-protocol-serialization`。

## Part D · API 網關 / BFF 🔬

**為什麼要網關**:客戶端不該直連一堆微服務(要知道幾十個地址、各自認證、跨域……)。**網關 = 統一入口**,把橫切關注點收口。

**網關職責**(典型):路由轉發、認證鑑權(→ L7)、**限流/熔斷**(→ L3)、協議轉換、響應聚合、日誌/監控。

**BFF(Backend for Frontend)**:給不同前端(Web / App / 小程序)各做一個**定制聚合層**,避免一個通用 API 遷就所有端。

### 跨生態對照
| 網關 | 生態 | 底層 |
|---|---|---|
| **Spring Cloud Gateway** | Java | Reactor/Netty(Java 微服務體系內) |
| **Kong** | 語言無關 | OpenResty(Nginx + Lua) |
| **APISIX** | 語言無關(Apache,國產) | OpenResty + etcd |
| **Envoy** | 語言無關(C++) | **雲原生標配**,也是服務網格的數據面 |

> **網關 vs 服務網格**(高頻區分,預告 L1-b):**網關管「南北向」**(集群外→集群內的入口流量);**服務網格管「東西向」**(集群內服務之間的流量)。兩者互補不衝突。

## 本章小結

- **拆分**:單體扛得住別拆;要拆按**限界上下文(DDD)**,用**康威定律**反向設計團隊;警惕**分布式單體**(最壞情況)。
- **三種通信範式是耦合光譜**:同步 RPC(緊耦合、即時、有級聯風險)、異步消息(解耦削峰、最終一致)、事件驅動(最鬆耦合、擴展最好、調試難)。**成熟系統混合用**:核心同步、旁路異步。
- **RPC 本質** = 遠程調用像本地;跨生態:**gRPC(雲原生/Go 默認)、Dubbo(Java 主流)、REST(對外)**;序列化**對內二進制、對外 JSON**。
- **API 網關**收口橫切關注點(路由/認證/限流/聚合);**BFF** 為不同前端定制;網關管**南北向**、網格管**東西向**。
- 下一章 L1-b:服務治理設施——**服務發現、配置中心、負載均衡、服務網格**。
