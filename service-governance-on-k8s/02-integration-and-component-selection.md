# 02 · 整合與選型:同層選一,以及特性差怎麼補

> 🔬 上 k8s 後第一個現場問題:同一件治理事(比如服務發現),Spring Cloud 一套(Eureka)、k8s 一套(CoreDNS),**留哪個?** 「選一個」是上半場、相對好答;真正考你的是下半場 —— **k8s-native 常常特性更窮**(CoreDNS 給不了權重 / 元數據 / 推送 / 灰度),那**缺的那塊去哪補**?這章給一套可複用的判斷:同層選一的理由,加一個「特性差補法矩陣」。

---

## Part A · 為什麼同層不能兩套並行(選一)

直覺上「Eureka 也留著、k8s DNS 也用」像是穩妥,其實是**最糟的選擇**。核心病:**雙重真相源(dual source of truth)**。

```
            ┌─ Eureka 的活實例列表 ─┐   ← B 自己註冊 + 心跳;B 崩了但沒來得及註銷 → 還在
  A 要調 B ─┤                       ├─ 兩份列表不一致時,A 該信誰?
            └─ k8s Endpoints 列表 ──┘   ← readiness 探針判定;k8s 已摘除 → 不在
```

兩套發現各有**自己的健康判定、自己的摘除時機**:

- k8s 靠 **readiness 探針**(探針失敗就從 Endpoints 摘掉,秒級)。
- Eureka 靠 **心跳續約 + 自我保護**(默認租約 90 秒才過期,還有「大面積失聯就索性都不摘」的自我保護機制)。

兩者**不可能同步**。於是出現經典故障:**pod 已被 k8s 摘除(滾動發布 / readiness 掛),但 Eureka 列表裡還在 → 用 Eureka 發現的調用方把流量打進一個已經沒了的實例,黑洞**。反過來,Eureka 自我保護期間不摘、k8s 已重建,也對不齊。

> 一句話:**同一層只能有一個「誰活著」的權威**。兩套並行 = 兩套都不權威。所以 `01` 那句「同層選一」不是潔癖,是避免雙真相源。

### 選哪一套?——一張決策表

「選一個」要選哪個,看你的處境,不是看哪個技術新:

| 你的處境 | 傾向 | 為什麼 |
|---|---|---|
| **全新專案、全在 k8s** | **k8s Service DNS(CoreDNS)** | 平台自帶、零額外組件、語言無關;greenfield 默認 |
| **混合 VM + k8s**(一半服務還在虛機) | **保留 registry(Nacos/Consul/Eureka)** | k8s DNS 只認 k8s 內的 Service;虛機側那半 k8s 看不見,得有個跨環境的 registry |
| **強客戶端 LB 需求**(zone-aware、權重) | 保留 registry + 客戶端 LB,或上 mesh | 見 Part B 的 LB 行 |
| **國內 Java、已重度用 Nacos** | **保留 Nacos** | Nacos 發現 + 配置一體、生態成熟;Spring Cloud Alibaba 跟到 Boot 4.x |
| **想留 Spring 開發體驗但用 k8s 做真相源** | **`spring-cloud-kubernetes`** 橋 | `DiscoveryClient` 直接讀 k8s API,既不雙真相也不丟 Spring 抽象(但這條是少數派中間路線)|

## Part B · 同定位、不同特性:補法矩陣 🔬

這是這章的核心,也是你問題②的正解。「選 k8s-native」聽起來乾脆,但你會發現它**做的事比 Spring Cloud 那套少**:

> CoreDNS 能回答的只有「**這個 Service 名 → 哪些 ready 的 pod IP**」。而 Eureka / Nacos 還能回答:**每個實例的權重多少、帶什麼元數據(版本 / 機房)、實例變化時主動推給你、按元數據做灰度路由。** 這些 CoreDNS 一概沒有。

所以遷移不是「換個組件」,是**先盤點你到底用了哪些特性,再決定每個特性去哪**。判斷三步:

1. **列出你*真正用到*的特性** —— 不是組件能做的全部,是你**現在真的依賴**的那幾個(很多「特性」盤一盤發現根本沒人用)。
2. **看 k8s-native 等價物有沒有、語義同不同。**
3. **缺的那塊,落到四種補法之一:**

```
            ┌──────────────────────────────────────────────┐
  缺的特性 ─┤ (a) 保留富 registry  ── 只為這個特性留著它      │
            │ (b) 下沉到 mesh      ── 灰度/權重/重試/熔斷     │
            │ (c) app 層自己實現   ── 讀 Endpoints 自己算     │
            │ (d) 砍掉             ── 盤完發現沒人真用         │
            └──────────────────────────────────────────────┘
   兩邊都有但「語義不同」→ 別並存,選一(否則又回到雙真相源)
```

### 逐層具體例子(面試能逐欄說 = 你真的懂遷移)

| 治理層 | k8s-native 給的 | k8s-native **缺**什麼 | 缺的怎麼補 |
|---|---|---|---|
| **服務發現** | CoreDNS:name→ready-IP | 權重、實例元數據、主動推送、按 metadata 灰度 | 要灰度按版本路由 → **(b) mesh**(Istio 按 label 路由)或 **(a) 留 Nacos** |
| **配置** | ConfigMap / Secret:掛載 + 滾動重啟生效 | 動態推送、監聽、灰度發布、配置變更審計 | 要免重啟熱更 → **spring-cloud-kubernetes 的 ConfigMap watch**,或 **(a) 留 Nacos/Apollo** |
| **負載均衡** | kube-proxy:連接級隨機 / 輪詢(L4) | zone-aware、權重、按請求(L7)、會話保持 | 要 locality / 按請求 → **(b) mesh L7** 或客戶端 LB;長連接見 `03` |
| **熔斷 / 重試** | (k8s 本身不做) | —— | **(b) mesh**(網路級)+ **app 內 Resilience4j**(業務級)→ **不是選一,是分工** |

最後一行特別重要:**熔斷不是「mesh 取代 Resilience4j」的非此即彼。** mesh 做的是**網路級**熔斷(這個 pod 連不上就跳過),語言無關、粒度粗;Resilience4j 做的是**業務級**(這個*方法*失敗率高就降級、返回兜底數據),在 app 進程裡、懂業務、粒度細。成熟系統**兩層都要**,各管一段。把它當「補法矩陣」的一個特例:有些特性 k8s/mesh 補不全,就**留在 app**。

### 🔬 黑盒內幕:為什麼 CoreDNS 給不了權重、ConfigMap 給不了推送?

不是 k8s 偷懶,是**這兩個東西的本質決定的**:

- **CoreDNS 就是個 DNS。** DNS 的 A 記錄只有「名字→IP」,**語義裡根本沒有權重**。理論上 SRV 記錄帶 `priority/weight`,但絕大多數客戶端解析器**不讀** SRV 的權重,只把它當地址列表。所以「按權重分流」這件事,DNS 這個協議層面就表達不了 —— 必須往上挪到懂應用語義的 L7(mesh / 網關)。
- **ConfigMap 默認不推送,是因為它只是 etcd 裡的一個對象。** app 看到的配置是 **kubelet 把 ConfigMap 內容掛載成文件**,kubelet 定期(默認約分鐘級)同步更新文件 —— 這是 **kubelet 的 pull,不是給 app 的 push**,而且很多 app 啟動時讀一次就不再讀文件了。所以「配置一改、app 秒級生效」要嘛靠 sidecar/reloader 監聽後觸發、要嘛靠 app 自己 watch k8s API(spring-cloud-kubernetes 就是後者)、要嘛你乾脆留一個本來就為「動態推送」而生的 Nacos。

> 看懂這兩個內幕,你就不會問「為什麼 k8s 不直接做」——因為**那不是它那層該做的事**。特性差的根源是「k8s 原語停在基礎設施層,應用語義的特性本就該在更上面補」。

## Part C · 一個反模式 + 一條決策樹收口

**反模式(別這麼幹):** Eureka 和 k8s DNS 並行,服務既向 Eureka 註冊、又被 k8s Service 選中,兩套發現各跑各的。→ 雙真相源,滾動發布時必出黑洞流量。要嘛純 Eureka(把 k8s 當純調度)、要嘛純 k8s 發現、要嘛 spring-cloud-kubernetes 讓 Eureka 抽象**讀 k8s 真相** —— 三選一,別並列。

**決策樹(遷移任一層時套):**

```
要把某治理層搬上 k8s?
  ├─ 1. 列出你真正用到的特性
  ├─ 2. k8s-native 夠用嗎?
  │       夠 ──────────────► 用 k8s-native(選一,退掉 Spring Cloud 那套)
  │       不夠 ↓
  ├─ 3. 缺的特性是「網路級 / 路由級」嗎(灰度/權重/重試/mTLS)?
  │       是 ──────────────► (b) 下沉 mesh
  │       否 ↓
  ├─ 4. 是混合環境 / 國內已重度 Nacos?
  │       是 ──────────────► (a) 保留富 registry,只為那個特性
  │       否 ↓
  └─ 5. 這特性 app 自己算得出來嗎(讀 Endpoints)?
          算得出 ──────────► (c) app 層實現      算不出/不值得 ─► (d) 砍掉,接受沒有
```

## 本章小結

- **同層選一,因為兩套並行 = 雙真相源**:Eureka(心跳 + 自我保護)和 k8s(readiness)健康判定不同步,滾動發布必出黑洞。
- **選哪套看處境**:greenfield→CoreDNS;混合 VM+k8s / 國內 Nacos→保留 registry;想留 Spring 抽象又要 k8s 真相→spring-cloud-kubernetes。
- **特性差補法矩陣(核心)**:k8s-native 常更窮(CoreDNS 無權重/推送/灰度)。盤出*真正用到*的特性 → 落到 **(a) 保留富 registry / (b) 下沉 mesh / (c) app 自實現 / (d) 砍掉**。
- **熔斷是「分工」不是「選一」**:mesh 做網路級、Resilience4j 做業務級,兩層都要。
- **🔬 根源**:CoreDNS 是 DNS(協議無權重語義)、ConfigMap 是 etcd 對象(kubelet pull 非 push)——特性差是「應用語義本就該在更上層補」的必然。
- **下一章**:LB 下沉到 kube-proxy 後,長連接(gRPC)會出大問題 —— 為什麼,怎麼修。
